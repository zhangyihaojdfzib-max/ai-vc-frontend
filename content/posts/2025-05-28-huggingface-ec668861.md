## 🤔 智能体动作的演进

AI智能体需要在世界中执行动作——无论是调用API、处理数据还是推理复杂问题。智能体表达这些动作的方式经历了几个范式的演进：

**传统JSON智能体**：智能体生成结构化JSON来调用工具。

```json
{"tool": "get_weather", "arguments": {"city": "Paris"}}
```

这类智能体通过从预定义工具列表中选择并生成JSON格式的调用来运作。这种调用工具的方法因OpenAI的function calling API而流行，并自此成为最广泛使用的工具调用方法。

它可靠，但受限于：
*   **有限的动作集**：智能体能执行的动作仅通过预定义工具表达，这限制了其功能。
*   **缺乏可组合性**：如果任务需要组合来自多个来源的信息，JSON智能体会很吃力，因为它们缺乏在工具调用间维护中间状态的支持。虽然有些模型支持并行工具调用，但它们难以轻松处理一个工具的输出决定下一个动作，或需要将结果进行比较和共同处理的场景。
*   **结构僵化**：在处理工具不完全匹配所需操作的情况时非常受限。

**代码智能体**：智能体利用其内在的编码能力，直接编写可执行的Python代码。

```python
temperature_sum = 0
for city in ["Paris", "Tokyo", "New York"]:
    temp = get_weather(city)
    temperature_sum += temp
    
print(f"Average temperature: {temperature_sum / 3:.1f}°C")
```

这一转变首次在论文《Executable Code Actions Elicit Better LLM Agents》中以CodeAct的形式提出，它赋予了AI智能体除了工具调用之外，还能编写任意可执行Python代码的灵活性。

这里的关键洞见是，**工具直接从代码内部调用**，使得变量和状态管理更加可靠。智能体可以在循环、函数和条件语句中调用工具——本质上是在每个动作中生成一个动态的工具执行图！

使用CodeAgent的优点：
*   **智能工具使用**：智能体根据当下情况决定使用哪些工具。
*   **无限灵活性**：可以使用任何Python功能来实现目标。
*   **测试想法的能力**：智能体可以提出假设并进行测试，从而使其行动更具灵活性。

然而，从Markdown中解析代码容易出错，这引出了我们的提议：**为什么不使用结构化生成来生成代码动作呢？**

### ➡️ 为代码智能体添加结构化输出

通过结构化输出，你可以强制LLM将明确的思考和代码生成为一个JSON块：

```json
{
  "thoughts": "I want to find the average temperature across 3 cities.",
  "code": "temperature_sum = 0\nfor city in [\"Paris\", \"Tokyo\", \"New York\"]:\n    temp = get_weather(city)\n    temperature_sum += temp\n\nprint(f\"Average temperature: {temperature_sum / 3:.1f}°C\")"
}
```

关键区别在于生成是**强制性的**：基本上，现在不仅仅是提示它输出思考然后代码，**结构化输出的使用强制它遵守这个结构**。

这种方法将结构化生成的可靠性，与代码执行的灵活性相结合，从而实现了两全其美。
*   **明确的推理**：`thoughts`字段强制智能体在采取行动前进行推理。
*   **可靠的解析**：JSON结构消除了Markdown解析错误。
*   **完整的代码表达能力**：`code`字段保留了代码智能体的所有灵活性。
*   **更好的分离**：规划与执行之间有清晰的分离。

## 💡 为何结构化（通常）有帮助

### 解析问题是真实存在的

我们在smolagents中对CodeAgent的实现是从LLM输出中提取Python代码，这在以下情况下可能失败：
*   Markdown中的代码块格式不完整或格式错误
*   单个响应中出现多个代码块

结构化生成通过可靠的JSON解析消除了这些问题。

为了理解结构化生成为何重要，我们分析了基准测试中的15,724条智能体轨迹。结果令人震惊：
*   **2.4%** 的轨迹在首次调用时出现解析错误
*   **首次调用有解析错误的轨迹**：成功率 **42.3%**
*   **首次调用无解析错误的轨迹**：成功率 **51.3%**

没有解析错误的智能体轨迹成功率比有解析错误的高出 **21.3%**。

这不仅仅是方便性的问题——解析错误会引发一连串的失败，显著影响智能体的整体性能。当智能体因格式错误的代码而无法执行其第一个动作时，它往往难以恢复，导致次优的问题解决路径。

**此外：强制的推理过程**

使用结构化生成和明确的`thoughts`字段，不仅仅是提示，而是**强制**智能体在行动前阐明其推理。这导致：
*   **更好的规划**：智能体更系统地思考问题
*   **更高的可靠性**：明确的推理能及早发现逻辑错误

### 结构化税

我们的结果也揭示了一个明确的能力阈值：模型需要足够的指令遵循能力，以及预训练数据中足够的JSON覆盖，才能从结构化生成中受益。这表明结构化方法在以下情况下效果最佳：
*   大型、训练良好的模型
*   具有强大指令遵循能力的模型
*   在结构化生成上进行过微调的模型

#### 当结构化失效时：一个真实例子

当较小的模型（例如`mistralai/Mistral-7B-Instruct-v0.3`）尝试生成结构化代码时会发生什么——认知负荷变得过大：

```json
{
  "thought": "I need to find the height...",
  "code": "web_search(query=\"Eiffel Tower height\")\", "
}
```

模型生成了语法错误的Python代码：`web_search(query="Eiffel Tower height")",` ——注意格式错误的字符串，多了一个引号和逗号。这导致立即出现`SyntaxError`和执行失败。

这说明了“**结构化税**”：较小的模型难以同时处理JSON格式化、Python语法和实际的问题解决逻辑。结构化生成的认知开销可能会压垮那些原本使用更简单的基于Markdown的代码生成就能表现相当不错的模型。

## 🚀 何时使用结构化CodeAgents

✅ 在以下情况使用结构化CodeAgents：

- 使用能力强大的模型（320亿参数以上或前沿模型）
- 任务需要复杂推理和代码执行
- 需要对Agent（智能体）输出进行可靠解析

⚠️ 在以下情况请考虑替代方案：

- 使用较小模型时，其结构化生成能力不足
- 简单、预定义的工作流已足够

### 如何在 smolagents 中使用：

非常简单！只需通过 `use_structured_outputs_internally` 启用：

```python
from smolagents import CodeAgent, InferenceClientModel, GoogleSearchTool


agent = CodeAgent(
    tools=[GoogleSearchTool(provider="serper")],
    model=InferenceClientModel("Qwen/Qwen3-235B-A22B", provider='nebius'),
    use_structured_outputs_internally=True 
)

result = agent.run("计算猎豹跑过金门大桥所需的时间")

```

LLM（大语言模型）将生成类似这样的内容：

```json
{
  "thoughts": "我需要找到金门大桥的长度和猎豹的最高速度，然后计算时间。",
  "code": "bridge_info = web_search('Golden Gate Bridge length meters')\ncheetah_speed = web_search('Cheetah top speed') ..."
}

```

然后，"code" 部分会像往常一样由Agent（智能体）执行：这是标准的 CodeAgent，但现在它拥有 100% 的解析可靠性！
