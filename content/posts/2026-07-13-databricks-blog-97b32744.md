- 外部引擎（包括 Apache Spark、Apache Flink 和 DuckDB）现在可以创建并写入 UC 管理的 Delta 表，并通过 Unity Catalog 实现集中治理
- UC 管理表使用预测优化来提升查询性能并降低存储成本，同时保持与外部引擎的完全互操作性。现有外部表可通过 `ALTER TABLE SET MANAGED` 就地升级，无需重写数据且保持互操作性
- 外部访问基于开源 Unity Catalog (UC OSS) 项目中的开放 UC Delta API 构建，因此支持的引擎可同时与 Databricks UC 和 UC OSS 配合使用。

Unity Catalog 专为大规模互操作性而设计。企业可以灵活地在单一数据副本上运行任何引擎，并通过集中治理实现管控。

今天，我们将互操作性提升到了新高度。我们很高兴地宣布，对 Unity Catalog (UC) 管理的 Delta 表的外部访问现已进入公开预览阶段。

过去一年中，越来越多的引擎（从 Apache Spark、Flink 到现在的 Starburst 和 DuckDB）已集成 UC 管理的 Delta 表，且生态系统持续壮大。企业可以兼得两者优势：UC 管理表的价格性能优势，以及使用适合各工作负载的引擎的灵活性。Unity Catalog 仍作为集中治理层，确保跨引擎一致执行访问策略。

此前，需要多引擎访问的团队只能使用外部表，但外部表缺乏性能优化和治理保障。随着外部访问进入公开预览，这一权衡已不复存在。在价格、性能和互操作性方面，UC 管理表是湖仓的明确选择。

## 团队为何选择 Unity Catalog 管理表

UC 管理表将 Delta Lake 和 Apache Iceberg 的互操作性与自动内置功能相结合，可提升查询性能、降低存储成本，并长期保持表健康。它们也是平台功能（如灾难恢复和零字节摄取）的基础。

Databricks 通过预测优化自动调优管理表。预测优化可清理存储、收集查询统计信息，并自动选择 Liquid 聚类列，以随着查询模式变化而演进表布局。这些优势综合起来，可节省高达 50% 的存储成本，并将查询速度提升 20 倍，同时数据团队无需手动调优表。

![预测优化](/images/posts/536ef4e77752.png)

我们近期宣布了目录提交的正式可用性，这使得 Unity Catalog 成为 UC 管理表的提交协调器。外部客户端不再直接向云存储提交写入，而是通过目录协调对 UC 管理 Delta 表的写入。这使得 Unity Catalog 成为表状态的唯一真实来源，从而解锁了安全外部写入、多语句事务以及外部操作审计等关键优势。

## 从外部引擎创建、读取和写入

现在处于公开预览阶段，外部引擎可以：

- 通过 Apache Spark（批处理和流处理）、Apache Flink、Starburst、DuckDB 和 Streamnative，使用目录提交创建、读取和写入 UC 管理的 Delta 表。
- 将所有引擎的治理集中保留在 Unity Catalog 中。
- 受益于预测优化，该优化可降低所有引擎的成本并提升性能。

外部访问基于用于元数据、凭证分发和提交协调的开放 API 构建。凭证分发根据 UC 权限颁发短期、作用域限定的凭证，而 Unity Catalog Delta API 为 Delta 引擎提供了用于创建、读取和写入操作的版本化契约。由于这些 API 是开放的，并在 Unity Catalog OSS 项目中实现，因此任何引擎都可以集成创建、读取和写入支持。其结果是形成了一个不断壮大的引擎生态系统，这些引擎在 Databricks UC 和 UC OSS 中对同一受治理数据副本进行操作。

合作伙伴已开始以不同方式围绕这些 API 进行构建。例如，Starburst 与 Unity Catalog 和 Delta Lake 集成，以支持创建、读取和写入 UC 管理的 Delta 表。

## 通过 Delta Kernel 扩展生态系统

对 UC 管理表的外部访问是 Delta Lake 和 Unity Catalog OSS 生态系统更广泛投资的一部分。目标是使 Delta 与 Iceberg 首创的目录管理模型保持一致，从而使企业能够从目录的性能和一致性保证中受益，同时保持与任何引擎的互操作性。

我们正在通过 Delta Kernel 扩展 Delta 生态系统，Delta Kernel 提供了用于读取、写入和提交 Delta 表的库，无需从头重新实现协议。结合 UC OSS 中的 UC Delta API，Delta Kernel 允许引擎与 Databricks 管理的目录或自托管的 UC OSS 部署无缝集成。

例如，DuckDB 基于 Delta Kernel 构建了其 Delta 和 Unity Catalog OSS 扩展。从 DuckDB v1.5.1 开始，它可以直接读取和写入 UC 管理的 Delta 表。这反映了我们对开放、互操作生态系统的承诺：随着 Delta 协议的发展，基于 Kernel 构建的引擎会自动继承最新功能，因此无论客户选择哪种引擎，都能始终访问最新、最强大的能力。

我们正在继续与社区合作，开发更多基于 Kernel 的 UC 管理 Delta 表集成。例如，我们正在取得重大进展，通过一个新的基于 Kernel 的 Flink 连接器，将 Flink 流处理工作负载引入 UC 管理表。

## 集中治理

集中治理对于多引擎管道至关重要。流处理应用可能将数据写入管理表，Spark 可能对其进行转换，而 Starburst 或 DuckDB 可能对其进行查询——无论使用何种引擎或查询背后的身份，都必须遵守访问控制。

Unity Catalog 是跨所有引擎的单一治理层。平台团队通过 UC 权限控制哪些主体可以外部访问表以及他们可以执行哪些操作。跨引擎 ABAC 进一步扩展了这一点，允许 Unity Catalog 在从外部引擎读取时过滤行或掩码列。细粒度访问控制针对 UC 管理表集中执行，无需自定义变通方法。

## 立即开始使用 UC 管理表

对于在开放湖仓上构建的团队来说，UC 管理表是可用的最强大基础。预测优化自动调优数据布局并执行表维护，无需手动操作即可提供最佳价格性能。互操作性也是内置的：管理表以 Delta 或 Iceberg 格式存储，可从任何引擎访问，因此团队无需在性能和灵活性之间做出选择。

使用外部表的团队可以通过 `ALTER TABLE SET MANAGED` 就地升级现有表，从而在保持其已依赖的开放多引擎访问的同时，获得预测优化和目录提交。

我们的愿景很明确：客户应该能够为每个工作负载使用最佳引擎，同时在 Unity Catalog 中保留一份数据副本，并享受最佳价格性能和统一治理。

## 如何开始

对 UC 管理 Delta 表的外部访问已进入公开预览阶段，包括支持从外部客户端进行创建、读取和写入操作。

要开始使用：

1. 在您的 Unity Catalog 元存储上启用外部数据访问。
2. 在包含您希望外部引擎访问的 UC 管理 Delta 表的模式上授予 `EXTERNAL USE SCHEMA` 权限。
3. 使用 Delta 4.3 和 Unity Catalog 0.5.1 客户端连接您的引擎。请参阅外部访问文档。
4. 测试从外部引擎对启用了目录提交的 UC 管理 Delta 表进行创建、读取和写入操作（或将现有外部表转换为管理表）。
