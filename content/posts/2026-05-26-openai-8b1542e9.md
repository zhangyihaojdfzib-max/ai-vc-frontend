# 解锁 Codex 框架：我们如何构建 App Server

作者：Celia Chen，技术团队成员

OpenAI 的编程 Agent（智能体）Codex 存在于多种不同的界面中：网页应用⁠(在新窗口中打开)、CLI⁠(在新窗口中打开)、IDE 扩展⁠(在新窗口中打开)以及全新的 Codex macOS 应用。在底层，它们都由同一个 Codex 框架驱动——即支撑所有 Codex 体验的 Agent（智能体）循环和逻辑。它们之间的关键连接是什么？是 Codex App Server⁠(在新窗口中打开)，一个对客户端友好的、双向的 JSON-RPC¹ API。

在这篇文章中，我们将介绍 Codex App Server；我们将分享迄今为止学到的关于将 Codex 能力集成到你的产品中以帮助用户提升工作流程的最佳实践。我们将涵盖 App Server 的架构和协议，以及它如何与不同的 Codex 界面集成，同时提供利用 Codex 的技巧，无论你是想将 Codex 变成一个代码审查员、一个 SRE Agent（智能体），还是一个编程助手。

## App Server 的起源

在深入探讨架构之前，了解 App Server 的背景故事会很有帮助。最初，App Server 是一种在不同产品间复用 Codex 框架的实用方式，后来逐渐演变成我们的标准协议。

Codex CLI 最初是一个 TUI（终端用户界面），意味着通过终端访问 Codex。当我们构建 VS Code 扩展（一种更符合 IDE 习惯的与 Codex Agent（智能体）交互的方式）时，我们需要一种方法来使用相同的框架，以便从 IDE 界面驱动相同的 Agent（智能体）循环，而无需重新实现它。这意味着要支持超越简单请求/响应的丰富交互模式，例如浏览工作区、在 Agent（智能体）推理时流式传输进度，以及生成差异。我们首先尝试将 Codex 作为 MCP 服务器⁠(在新窗口中打开) 暴露出来，但事实证明，以对 VS Code 有意义的方式维护 MCP 语义非常困难。相反，我们引入了一个镜像 TUI 循环的 JSON-RPC 协议，这成为了 App Server 的非官方第一版⁠(在新窗口中打开)。当时，我们并未预料到其他客户端会依赖 App Server，因此它并非作为稳定 API 来设计。

随着 Codex 在接下来的几个月里被更广泛地采用，内部团队和外部合作伙伴希望能够将相同的框架嵌入到他们自己的产品中，以加速其用户的软件开发工作流程。例如，JetBrains 和 Xcode 想要一个 IDE 级别的 Agent（智能体）体验，而 Codex 桌面应用则需要并行编排多个 Codex Agent（智能体）。这些需求促使我们设计一个平台界面，使得我们的产品和合作伙伴的集成能够长期安全地依赖它。它需要易于集成且向后兼容，这意味着我们可以在不破坏现有客户端的情况下演进协议。

接下来，我们将介绍我们如何设计架构和协议，以便不同的客户端能够使用相同的框架。

## Codex 框架内部

首先，让我们聚焦于 Codex 框架内部包含什么，以及 Codex App Server 如何将其暴露给客户端。在我们上一篇 Codex 博客中，我们分解了协调用户、模型和工具之间交互的核心 Agent（智能体）循环。这是 Codex 框架的核心逻辑，但完整的 Agent（智能体）体验远不止于此：

1. 线程生命周期与持久化。线程是用户与 Agent（智能体）之间的一次 Codex 对话。Codex 创建、恢复、分叉和归档线程，并持久化事件历史，以便客户端能够重新连接并呈现一致的时间线。

2. 配置与认证。Codex 加载配置、管理默认值，并运行诸如“使用 ChatGPT 登录”之类的认证流程，包括凭据状态。

3. 工具执行与扩展。Codex 在沙箱中执行 shell/文件工具，并连接诸如 MCP 服务器和技能之类的集成，以便它们能够在一致的政策模型下参与 Agent（智能体）循环。

我们在此提到的所有 Agent（智能体）逻辑，包括核心 Agent（智能体）循环，都存在于 Codex CLI 代码库中一个名为“Codex core⁠(在新窗口中打开)”的部分。Codex core 既是一个包含所有 Agent（智能体）代码的库，也是一个可以启动以运行 Agent（智能体）循环并管理一个 Codex 线程（对话）持久化的运行时。

为了发挥作用，Codex 框架需要能被客户端访问。这就是 App Server 的用武之地。

![标题为“App server 进程流程”的图表。客户端发送 JSON-RPC 消息给 stdio 读取器，后者将请求分派给 Codex 消息处理器。处理器通过查找线程、线程句柄、已提交请求和事件/更新与线程管理器和核心线程交互，然后将响应返回给客户端。](/images/posts/ab37acf107e3.jpg)

App Server 既是客户端与服务器之间的 JSON-RPC 协议，也是一个托管 Codex 核心线程的长期运行进程。从上图可以看出，一个 App Server 进程有四个主要组件：stdio 读取器、Codex 消息处理器、线程管理器和核心线程。线程管理器为每个线程启动一个核心会话，然后 Codex 消息处理器直接与每个核心会话通信，以提交客户端请求并接收更新。

一个客户端请求可能导致许多事件更新，正是这些详细的事件使我们能够在 App Server 之上构建丰富的 UI。此外，stdio 读取器和 Codex 消息处理器充当客户端与 Codex 核心线程之间的转换层。它们将客户端的 JSON-RPC 请求转换为 Codex 核心操作，监听 Codex 核心的内部事件流，然后将这些底层事件转换为一小组稳定的、UI 就绪的 JSON-RPC 通知。

客户端与 App Server 之间的 JSON-RPC 协议是完全双向的。一个典型的线程包含一个客户端请求和许多服务器通知。此外，当 Agent（智能体）需要输入（例如批准）时，服务器可以发起请求，然后暂停该轮次，直到客户端响应。

## 对话原语

接下来，我们将分解对话原语，即 App Server 协议的构建块。为 Agent（智能体）循环设计 API 很棘手，因为用户/Agent（智能体）交互并非简单的请求/响应。一个用户请求可能会展开成一个结构化的动作序列，客户端需要忠实地呈现这些动作：用户的输入、Agent（智能体）的增量进度、过程中产生的工件（例如差异）。为了使这种交互流易于集成并能适应不同的 UI，我们确定了三个具有明确边界和生命周期的核心原语：

1. 项目：项目是 Codex 中输入/输出的原子单元。项目是有类型的（例如，用户消息、Agent（智能体）消息、工具执行、审批请求、差异），并且每个项目都有一个明确的生命周期：
   - 项目/开始：当项目开始时
   - 可选的项目/*/增量事件：当内容流式传输时（针对流式项目类型）
   - 项目/完成：当项目以其最终载荷结束时

   这个生命周期允许客户端在“开始”时立即开始渲染，在“增量”时流式传输增量更新，并在“完成”时最终确定。

2. 轮次：轮次是由用户输入发起的一个 Agent（智能体）工作单元。它从客户端提交输入（例如，“运行测试并总结失败”）开始，到 Agent（智能体）完成对该输入的输出生成时结束。一个轮次包含一系列项目，这些项目代表了过程中产生的中间步骤和输出。

3. 线程：线程是用户与 Agent（智能体）之间正在进行的 Codex 会话的持久容器。它包含多个轮次。线程可以被创建、恢复、分叉和归档。线程历史被持久化，以便客户端能够重新连接并呈现一致的时间线。

现在，我们将看一个客户端与 Agent（智能体）之间的简化对话，其中对话由原语表示：

![图表标题为“客户端-服务器协议消息流：初始化握手”。客户端向服务器发送包含客户端信息的初始化请求。服务器回复一个包含用户代理字符串“my_client/1.0”的结果事件。](/images/posts/ea7c71a55613.jpg)

在对话开始时，客户端和服务器需要建立初始化握手。客户端必须在任何其他方法之前发送一个单一的初始化请求，服务器通过响应进行确认。这使服务器有机会通告其能力，并让双方在实际工作开始前就协议版本、功能标志和默认设置达成一致。以下是来自OpenAI的VS Code扩展的一个示例负载：

```
1{2  "method": "initialize",3  "id": 0,4  "params": {5    "clientInfo": {6      "name": "codex_vscode",7      "title": "Codex VS Code Extension",8      "version": "0.1.0"9    }10  }11}
```

这是服务器返回的内容：

```
1{2  "id": 0,3  "result": {4    "userAgent": "codex_vscode/0.94.0-alpha.7 (Mac OS 26.2.0; arm64) vscode/2.4.22 (codex_vscode; 0.1.0)"5  }6}
```

![图表标题为“客户端-服务器协议消息流：线程与轮次生命周期”。客户端向服务器发送线程/开始和轮次/开始请求。服务器发出事件——线程/已开始、轮次/已开始、项目/已开始和项目/已完成——展示了一个轮次，其中用户消息为“运行测试并总结失败”。](/images/posts/41f236088830.jpg)

当客户端发出新请求时，它首先会创建一个线程，然后创建一个轮次。服务器会发送进度通知（线程/已开始和轮次/已开始）。它还会将注册为项目的输入发送回来，例如这里的用户消息。

![图表标题为“客户端-服务器协议消息流：带可选审批的工具执行”。在工具调用期间，服务器发出项目/已开始，然后发出项目/命令执行/请求审批，并附带原因（“运行测试”）。客户端返回一个审批事件（允许/拒绝）。然后服务器发出项目/已完成，显示命令执行结果（“pnpm test”）。](/images/posts/8b972050f896.jpg)

工具调用也会作为项目发送回客户端。此外，服务器在运行某个操作前，可能会通过发送服务器请求来请求客户端审批。审批会暂停轮次，直到客户端回复“允许”或“拒绝”。以下是VS Code扩展中审批流程的展示：

![图表标题为“客户端-服务器协议消息流：流式Agent消息流”。服务器分部分流式传输助手消息：项目/已开始，两个agentMessage/delta事件（“运行了3个测试。”，“全部通过”），然后是项目/已完成。轮次以轮次/已完成结束。](/images/posts/d83c7f221e2b.jpg)

最后，服务器发送一条Agent消息，然后以轮次/已完成结束该轮次。Agent消息的delta事件会流式传输消息的片段，直到消息通过项目/已完成最终确定。

图表中的消息为了可读性进行了简化。如果你想查看完整轮次的JSON，可以从Codex CLI仓库运行测试客户端：

```
1codex debug app-server send-message-v2 "run tests and summarize failures"
```

## 与客户端集成

现在，让我们看看不同的客户端界面如何通过App Server嵌入Codex。我们将涵盖三种模式：本地应用和IDE、Codex Web运行时以及TUI。

![图表标题为“通过应用服务器与Codex框架集成的Codex客户端”。第一方客户端（Codex桌面应用、TUI/CLI、Web运行时）和第三方集成（JetBrains IDE、VS Code、Xcode）通过JSON-RPC调用与Codex框架通信。](/images/posts/95adfa9deb97.jpg)

在所有这三种模式中，传输方式都是基于stdio（JSONL）的JSON-RPC。JSON-RPC使得用你选择的语言构建客户端绑定变得简单直接。Codex界面和合作伙伴集成已经用包括Go、Python、TypeScript、Swift和Kotlin在内的语言实现了App Server客户端。对于TypeScript，你可以通过运行以下命令直接从Rust协议生成定义：

```
1codex app-server generate-ts
```

对于其他语言，你可以通过运行以下命令生成一个JSON Schema包，并将其输入到你首选的代码生成器中：

```
1codex app-server generate-json-schema
```

#### 本地应用与IDE

![VS Code运行Codex扩展的截图。一个Rust测试文件已打开，其下方Codex面板描述了仅运行fmt和cargo test -p codex-app-server，报告格式化和测试正在进行中，同时等待最终的通过/失败结果。](/images/posts/73ebd99d53c8.png)

本地客户端通常会捆绑或获取一个特定平台的App Server二进制文件，将其作为长期运行的子进程启动，并保持一个双向的stdio通道用于JSON-RPC。例如，在我们的VS Code扩展和桌面应用中，发布的制品包含了特定平台的Codex二进制文件，并固定到一个经过测试的版本，这样客户端始终运行我们验证过的确切代码。

并非每个集成都能频繁地推送客户端更新。一些合作伙伴（如Xcode）通过保持客户端稳定，并允许其在需要时指向更新的App Server二进制文件，来解耦发布周期。这样他们就可以采用服务器端的改进（例如，Codex核心中更好的自动压缩或新支持的配置键）并推出错误修复，而无需等待客户端发布。App Server的JSON-RPC接口设计为向后兼容，因此旧客户端可以安全地与新服务器通信。

#### Codex Web

![Codex Web界面的截图，显示了一个标题为“更新登录成功消息”的更新。左侧面板总结了更改、测试和修改的文件，而右侧面板显示了login.rs的代码差异，其中登录成功消息的措辞已更新。](/images/posts/2cbfbf8c2185.png)

Codex Web使用Codex框架，但将其运行在容器环境中。一个工作进程会为已检出的工作区配置一个容器，在其中启动App Server二进制文件，并维护一个基于stdio的长期JSON-RPC通道。Web应用（运行在用户的浏览器标签页中）通过HTTP和SSE与Codex后端通信，后者流式传输由工作进程产生的任务事件。这使得浏览器端的UI保持轻量，同时仍能在桌面和Web上提供一致的运行时。

由于Web会话是短暂的（标签页关闭、网络断开），Web应用不能作为长期运行任务的事实来源。将状态和进度保存在服务器端意味着即使标签页消失，工作仍会继续。流式协议和保存的线程会话使得新会话能够轻松地重新连接，从中断处继续，并在无需在客户端重建状态的情况下赶上进度。

#### TUI/Codex CLI

历史上，TUI是一个“原生”客户端，与Agent循环运行在同一进程中，并直接与Rust核心类型通信，而不是通过应用服务器协议。这使得早期迭代很快，但也使TUI成为一个特例界面。

既然App Server已经存在，我们计划重构TUI⁠(opens in a new window)以使用它，使其行为像任何其他客户端一样：启动一个App Server子进程，通过stdio使用JSON-RPC通信，并渲染相同的流式事件和审批。这解锁了工作流，使得TUI可以连接到运行在远程机器上的Codex服务器，使Agent靠近计算资源，即使笔记本电脑休眠或断开连接也能继续工作，同时仍在本地提供实时更新和控制。

## 选择合适的协议

Codex App Server将是我们未来维护的一等集成方法，但也存在其他功能更有限的方法。默认情况下，我们建议客户端使用Codex App Server与Codex集成，但值得看看不同的集成方法并理解它们的优缺点。以下是驱动Codex的最常见方式以及每种方式可能适用的场景。

### JSON-RPC协议

#### Codex作为MCP服务器

Runcodex mcp-server⁠（在新窗口中打开）并从任何支持 stdio 服务器的 MCP 客户端（例如 OpenAI Agents SDK⁠（在新窗口中打开））进行连接。如果你已有基于 MCP 的工作流并希望将 Codex 作为可调用工具来调用，这将非常合适。其缺点在于你只能获得 MCP 暴露的功能，因此依赖更丰富会话语义（例如差异更新）的 Codex 特定交互可能无法通过 MCP 端点清晰映射。

#### 跨提供商 Agent 编排协议

某些生态系统提供可移植接口，能够面向多个模型提供商和运行时。如果你希望用一个抽象层来协调多个 Agent，这将非常合适。其权衡在于这些协议通常收敛于通用功能子集，这使得更丰富的交互难以表达，尤其是在提供商特定的工具和会话语义至关重要时。这一领域正在快速发展，我们预计随着我们找到表示真实世界 Agent 工作流的最佳原语，更多通用标准将会出现（技能⁠（在新窗口中打开）就是一个很好的例子）。

#### Codex App Server

当你希望将完整的 Codex 编排层作为稳定、对 UI 友好的事件流暴露时，请选择 App Server。你将获得 Agent 循环的完整功能以及其他支持特性，例如通过 ChatGPT 登录、模型发现和配置管理。主要成本在于集成工作，因为你需要用你的语言构建客户端 JSON-RPC 绑定。然而在实践中，如果你向 Codex 提供 JSON 模式和文档，它能够完成大量繁重工作。我们合作的许多团队都使用 Codex 快速实现了可用的集成。

### 嵌入 Codex 的其他方式

#### Codex Exec⁠（在新窗口中打开）

一种轻量级、可脚本化的 CLI 模式，适用于一次性任务和 CI 运行。它非常适合自动化和流水线场景，你希望单个命令以非交互方式运行至完成，流式输出结构化日志，并以明确的成功或失败信号退出。

#### Codex SDK⁠（在新窗口中打开）

一个 TypeScript 库，用于从你自己的应用程序中以编程方式控制本地 Codex Agent。当你希望为服务器端工具和工作流提供原生库接口，而无需构建单独的 JSON-RPC 客户端时，这是最佳选择。由于它比 App Server 更早发布，目前支持的语言较少且覆盖范围较小。如果有开发者的兴趣，我们可能会添加额外的 SDK 来封装 App Server 协议，以便团队无需编写 JSON-RPC 绑定即可覆盖更多编排层功能。
