## 梅赛德斯-奔驰韩国在“Talk to Data”中的愿景：BI和AI的统一语义

梅赛德斯-奔驰是高端豪华汽车领域的市场领导者，拥有全球销售网络，其中数据驱动的、针对特定市场的决策始终是优先事项。“Talk to Data”自助式分析正是为进一步支持这一优先事项而探索的能力之一。

梅赛德斯-奔驰韩国拥有成熟的数据基础。随着时间推移，梅赛德斯-奔驰韩国在Databricks的Lakehouse和Unity Catalog中建立了黄金层报表数据、主KPI目录和共享定义。该基础作为BI报表、自动化和其他数据产品的单一事实来源，涵盖销售、产品、营销、客户服务和财务等业务领域的500多个KPI。基于这一基础，梅赛德斯-奔驰韩国被选为“Talk to Data”方法的试点单位。

与此同时，梅赛德斯-奔驰韩国很大一部分业务语义是在Power BI中定义的。作为AI用例准备的一部分，这些定义通过Lakehouse中一个开放、AI就绪的语义层得到了补充。

梅赛德斯-奔驰韩国对“Talk to Data”的更广泛愿景是，为企业决策建立一个统一、AI就绪且受治理的语义基础，使其能够基于一致的业务定义集支持报表、自助式分析和AI体验。与此愿景一致，梅赛德斯-奔驰韩国并未将“Talk to Data”视为从Power BI的迁移，而是追求三个关键目标：

- **为AI提供一致的上下文**：业务逻辑和KPI已在两个层面定义：在Power BI的DAX语言中用于报表，以及在Lakehouse中经过策划的银层和金层表中，可供AI使用。下一步是将语义上下文从BI报表迁移到Unity Catalog，以丰富Lakehouse中现有的数据产品。这使得Genie和其他AI Agent能够在一个地方访问所有KPI定义，从而确保同一个问题（例如“我们按车辆类别划分的当月至今零售总额是多少？”）在AI体验中产生一致的答案。
- **架构向Agent AI演进**：梅赛德斯-奔驰韩国拥有成熟的BI技术栈，将用于数据工程和仓库的Databricks与用于语义建模和报表的Power BI相结合。这一演进过程的下一步是，基于BI报表中的业务逻辑，扩展一个统一、AI就绪的语义层，以便下游BI工具和AI Agent能够基于相同的受治理KPI运行。
- **从报表用户到基于角色的Agent**：针对表级和报表级最终用户的治理已作为现有数据基础设施的一部分就位。下一步是通过Unity Catalog中基于角色的访问控制和角色Agent的编排规则来扩展该治理，从而使CFO或销售副总裁等角色能够拥有针对其领域定制的Agent体验，而无需更改底层业务语义。

遵循这些原则有助于实现未来AI和BI工具能够使用相同已验证业务逻辑的目标。这有助于实现企业级应用所需的一致性、可解释性和答案质量。

## 数据、语义和AI的统一架构

![image12.png](/images/posts/7e774e802be1.png)

为了实现AI就绪的语义，梅赛德斯-奔驰韩国在Databricks数据智能平台上采用统一架构实施了“Talk to Data”，从而大规模提供可信的AI。

该解决方案依赖于Databricks不同能力的协同工作：

1. **Lakeflow和Lakehouse**从各种源系统摄取企业数据，并为BI和AI工作负载准备数据。
2. **Unity Catalog业务语义**作为KPI的单一事实来源，将Power BI DAX度量转换为指标视图：源、连接、度量、维度、注释和同义词都与数据共存，并受与底层表相同的权限治理。
3. **Genie空间**让业务团队能够“与数据对话”。Genie空间按业务领域组织，每个空间都由一组经过策划的指标视图支持。由于指标视图中的KPI直接定义在金层数据之上，Genie无需猜测或复杂的连接即可找出正确答案，从而提高了其响应的速度和准确性。
4. **Agent Bricks**在多个Genie空间之上组合了基于角色的Agent，因此CFO、销售副总裁和营销主管各自获得一个针对其角色定制的“Talk to Data”体验。
5. **Databricks Apps**为组合后的Agent提供自定义前端、与外部服务的连接以及其他扩展能力，其记忆和状态存储在Lakebase中。

在此次试点之前，报表数据存储在Hive元存储中，并且“Talk to Data”是通过其他基于Power BI报表中报表中心语义的AI解决方案进行探索的。在这种设置下，BI和AI的语义上下文分布在多个组件中，这使得AI层更难跨用户和工作负载一致地捕获语义。基于KPI的、基于角色的访问控制也尚未实现，并且其他AI解决方案需要比Genie更明确的指导和提示词调优，而Genie直接基于Unity Catalog中受治理的KPI层运行。

在Databricks上采用此架构为业务用户提供了简化的“Talk to Data”体验基础。

## 加速试点：自动化DAX到Metric View转译器

将 Power BI DAX 中的业务逻辑也引入 Unity Catalog 指标视图，是构建开放语义层的关键一步。梅赛德斯-奔驰韩国公司在 DAX 中定义了超过 500 个 KPI，因此需要一种高效且标准化的方法。为此，Databricks 为梅赛德斯-奔驰韩国团队构建了一个自动化的 DAX 到指标视图的转译器。

![image3.png](/images/posts/dd72e755a58b.png)

该转译器作为一个管道运行，将 Power BI DAX 度量转换为可部署的 Databricks 指标视图。它能够：

- 解析 Power BI 语义模型并提取每个 DAX 度量。
- 为从 Power BI 语义模型中提取的每个度量创建元数据目录，例如事实表、维度表、连接关系、连接键等。
- 将每个度量的源表映射到 Unity Catalog 中的对应表。
- 生成指标视图定义草案（源、连接、维度、度量），将 DAX 语义转换为指标视图度量以及用于创建指标视图的、可立即运行的 SQL 语句。
- 标记不可自动化的度量（通常涉及复杂的 DAX 特定功能，如行上下文操作），以供人工审查。
- 在编译指标视图之前验证语法和聚合逻辑。

![image9.png](/images/posts/0052db26a1b8.png)

当管道完成时，它会生成一份评估报告，其中包含转换统计信息、差距以及针对不可自动化的 DAX 度量转换的补救策略。

其输出是一个强大的起点，为可自动化的 DAX 度量提供了即用型指标视图，从而在语义迁移中节省了数百小时的人工工作。在此基础上，团队根据相应的 Power BI 报告验证每个度量，并使用 Genie Code 进行迭代，Genie Code 可根据自然语言输入优化和改进指标视图定义。

在此转译器初步成功之后，Databricks 将这些功能整合到一个新的、用于 Power BI 迁移的 Genie Code 技能中。该技能目前处于私有预览阶段，可直接在 Genie Code 中访问，无需额外工具，并支持在梅赛德斯-奔驰进一步推广。

## 构建可信、AI 就绪的语义

对于梅赛德斯-奔驰韩国公司“与数据对话”的业务用户而言，答案质量是重中之重。将 KPI 作为指标视图引入 Genie 空间是一个关键步骤，但这本身并不能保证来自 Agent（智能体）AI 的答案可靠。只有当 Genie 的响应得到验证后，指标视图才算 AI 就绪。梅赛德斯-奔驰韩国内部目标是 Genie 的答案与相应的 Power BI 报告完全对齐——范围内每个 KPI 的匹配率达到 100%。

在试点期间，梅赛德斯-奔驰韩国公司与 Databricks 共同记录了策划、集成和优化指标视图及 Genie 空间的最佳实践，包括使用 Agent（智能体）元数据和基准测试。这些实践支持了 AI 就绪的业务语义，并为与企业数据交互的业务团队提供一致的答案。

遵循试点期间记录的迭代五阶段流程，梅赛德斯-奔驰韩国公司在 Databricks 数据智能平台上构建了业务用户可依赖的 AI 就绪业务语义。Databricks 还在开发一个应用程序解决方案来自动化此流程，梅赛德斯-奔驰可将其用于进一步推广。

**阶段 1：准备。** 选择要引入的 KPI，并将每个 KPI 映射到 Unity Catalog 中的源表。对于 Power BI 语义迁移，识别相关的 DAX 度量和语义模型。此阶段确定范围和事实来源。

**阶段 2：构建语义层。** 使用数据源、维度、度量、注释和 Agent（智能体）元数据创建 Unity Catalog 指标视图。在添加下一个 KPI 之前，单独验证每个 KPI。对于跨多个事实表的 KPI，首先构建一个基础视图，然后在其上叠加指标视图。始终在指标视图、维度和度量级别添加描述；Genie 会综合推理这三者。

**阶段 3：按领域组织，而非按报告。** 围绕业务领域（例如“市场营销”）构建 Genie 空间，并围绕子领域内的 KPI 组（例如“在线营销转化指标”）构建指标视图。将每个 Genie 空间限制在 30 个 Unity Catalog 项目内，并且始终包含空间描述，以便多 Agent（智能体）系统能够正确路由问题。

**阶段 4：增量测试。** 增量地引入度量。使用样本问题验证每个度量。将验证过的查询保存为示例 SQL。然后构建基准测试：每个问题的措辞变体，配以真实 SQL，用于系统性地衡量答案准确性。启用提示词匹配，以便 Genie 能够将用户语言映射到实际数据值。

**阶段 5：验证并发布。** 每次更改后运行回归测试；如果先前通过的基准测试失败，则最近添加的内容最可能是原因。一旦稳定，向一小部分业务用户发布以获取真实反馈。使用“监控”选项卡在一个地方跟踪、审查和处理所有用户反馈。回归失败和用户反馈都直接反馈到阶段 4。

其输出是一个语义层，其中 AI 答案在范围内的每个 KPI 上都与相应的 BI 报告对齐，从而支持业务用户所期望的可靠性。

## 从单个 Genie 空间到具有内置治理的多 Agent（智能体）系统

单个 Genie 空间可以回答关于单个领域的问题。梅赛德斯-奔驰市场真正的“与数据对话”体验需要涵盖销售、产品、市场营销、客户服务、财务等多个领域——同时根据每个角色的身份和权限定制其可见内容。

![image11.png](/images/posts/30017a671316.png)

这就是 Agent Bricks 的用武之地。梅赛德斯-奔驰韩国公司部署的架构：

- 一个父级监督 Agent（智能体）将每个问题路由到具有特定角色指令的角色 Agent（智能体）。
- 角色 Agent（智能体）（例如“CFO”、“销售经理”、“市场分析师”）整合来自相关领域内适当 Genie 空间的见解，选择正确的指标视图范围来回答问题。
- Unity Catalog 治理策略强制执行行级和列级访问控制，因此区域经理即使使用自然语言提问，也只能看到其角色允许的数据。
- 这些 Agent（智能体）通过 Databricks 应用程序呈现，具有自定义前端和扩展功能，如自定义可视化、Agent（智能体）记忆和基于 Lakebase 的事务数据库。该应用程序还允许组合后的 Agent（智能体）与业务用户已使用的工具和服务集成，从而使“与数据对话”融入他们的日常工作流程，而不是作为一个独立的目的地。

由于指标视图位于 Unity Catalog 中，因此它们是开放的，并且可以在所有 Genie 空间、Agent（智能体）和应用程序中重复使用。每个答案都完全受治理、可审计，并且其血缘关系可追溯到源数据。这种多 Agent（智能体）模式可以作为将“与数据对话”推广到更多梅赛德斯-奔驰市场的参考。
