# SmolLM3：小巧、多语言、长上下文推理模型

随着用户寻求能够高效部署的、性能强大的模型，小型语言模型正变得越来越重要。社区已经开发出了一系列引人入胜的强大小型模型，每个模型都在不断突破这个规模下可能性的边界。我们很高兴通过 SmolLM3 贡献一个全新的、具有竞争力的、完全开源的 3B 模型：

- 基础模型：https://hf.co/HuggingFaceTB/SmolLM3-3B-Base
- 指令与推理模型：https://hf.co/HuggingFaceTB/SmolLM3-3B

SmolLM3 处于效率的甜点区。我们的 3B 模型性能优于 Llama-3.2-3B 和 Qwen2.5-3B，同时与更大的 4B 替代模型（Qwen3 和 Gemma3）保持竞争力。除了性能数据，我们还分享了如何利用公开数据集和训练框架构建它的完整方法。

![](/images/posts/51552a40018b.png)

模型概要：

- **3B 模型**：在 11T Token 上训练，在 3B 规模上达到 SOTA，并与 4B 模型竞争
- **指令模型**：具备**双模式推理**，支持`思考`/`不思考`模式
- **多语言支持**：支持 6 种语言：英语、法语、西班牙语、德语、意大利语和葡萄牙语
- **长上下文**：通过 NoPE 和使用 YaRN，支持高达 128k 的上下文窗口

完整配方：我们随同工程蓝图一起发布 SmolLM3。它包括架构细节、精确的数据混合比例（展示了我们如何通过三阶段预训练方法逐步提升各领域性能），以及构建混合推理模型的方法论。通常，要取得这些成果需要数月的逆向工程。相反，我们提供了完整的方法论。

![](/images/posts/947395ff78e8.png)

无论您是在构建自己的模型，还是想了解在这个规模上驱动性能的因素，这份蓝图都揭示了实现有竞争力的 3B 性能背后的工程故事。

让我们来看看预训练阶段。

# 预训练

SmolLM3 在其前代模型的基础上，既改变了架构，也改变了数据混合比例。我们先来看看架构和训练配置！

## 架构与训练细节

![](/images/posts/1d328eebc2f6.png)

SmolLM3 遵循 Transformer 解码器架构，采用与 SmolLM2 类似的嵌入共享，基于 Llama 架构，并针对效率和长上下文性能进行了一些关键优化修改。

**分组查询注意力（GQA）**：我们使用 4 个分组的分组查询注意力取代了多头注意力。我们在使用来自 FineWeb-Edu 的 100B Token 训练的 3B 模型上进行的消融实验表明，GQA 在性能上与多头注意力相当，同时显著减少了推理时的 KV 缓存大小。

**NoPE**：我们实现了《RoPE to NoRoPE and Back Again: A New Hybrid Attention Strategy》（Yang 等人，2025）中的 NoPE，选择性地从每第 4 层中移除旋转位置嵌入。我们的消融实验证实，这种方法在不影响短上下文能力的情况下，提升了长上下文性能。

**文档内掩码**：遵循《Analysing The Impact of Sequence Composition on Language Model Pre-Training》，在训练期间，我们使用注意力掩码来确保同一训练序列中来自不同文档的 Token 不会相互关注。与 Llama 3 类似，这有助于实现更快、更稳定的长上下文训练，同时保持短上下文性能。

**训练稳定性**：遵循 OLMo 2，我们从嵌入层移除了权重衰减，以提高训练稳定性。在我们的消融实验中，这一修改有助于更稳定的训练动态，嵌入范数在训练过程中自然地稳定在更健康的值，而不影响整体性能。

所有这些更改都通过使用相同 3B 架构、在 FineWeb-Edu 的 100B Token 上训练的消融实验进行了验证，确保每项修改要么提升了性能，要么在保持性能的同时带来了其他好处。

**训练配置**：我们使用全局批大小为 236 万 Token，序列长度为 4096，学习率为 2e-4，使用 AdamW 优化器（beta1: 0.9, beta2: 0.95），权重衰减为 0.1，梯度裁剪为 1。我们使用 WSD（预热-稳定-衰减）调度器，预热步数为 2000 步，并在最后 10% 的训练步数中线性衰减至 0。我们使用 `nanotron` 框架进行训练，`datatrove` 进行数据处理，`lighteval` 进行评估。该模型在 384 块 H100 GPU 上训练了 24 天。您可以在下图中看到分布式训练设置。

![](/images/posts/95171fdb549b.png)

除了架构更改，我们还对训练配方进行了消融实验和改进。让我们更仔细地看看。

## 数据混合与训练阶段

遵循 SmolLM2 的多阶段方法，我们使用三阶段训练策略在 11.2T Token 上训练 SmolLM3，该策略混合了网络、数学和代码数据，且比例不断变化。我们在使用 50B 到 100B Token 训练的 3B 模型上进行了广泛的消融实验，以确定数据混合和比例。

![](/images/posts/c261a4db663a.png)

预训练包含以下阶段，如上图所示：

- **阶段 1：稳定阶段（0T → 8T Token）**
    这个基础阶段使用我们的核心数据集混合建立了强大的通用能力：
    - 网络：85%（12% 多语言）- FineWeb-Edu, DCLM, FineWeb2 和 FineWeb2-HQ
    - 代码：12% - The Stack v2（16 种编程语言）、StarCoder2 pull requests、Jupyter 和 Kaggle 笔记本、GitHub issues 和 StackExchange。
    - 数学：3% - FineMath3+ 和 InfiWebMath3+

- **阶段 2：稳定阶段（8T → 10T Token）**
    我们引入了更高质量的数学和代码数据集，同时保持良好的网络覆盖：
    - 网络：75%（12% 多语言）
    - 代码：15% - 添加 Stack-Edu
    - 数学：10% - 引入 FineMath4+、InfiWebMath4+ 和 MegaMath（包括 Qwen Q&A、Pro 合成重写和文本-代码交错块）

- **阶段 3：衰减阶段（10T → 11.1T Token）**
    最后阶段进一步上采样数学和代码数据：
    - 网络：63%（12% 多语言）
    - 代码：24% - 上采样高质量代码数据
    - 数学：13% - 上采样数学数据并引入指令和推理数据集，如 OpenMathReasoning

通过这些阶段和混合比例，我们为基础模型实现了非常有竞争力的性能。更多细节请参见评估部分。包含精确数据权重的 nanotron 训练配置可以在[此处](here)找到。我们也将分享我们的训练日志以及中间检查点。

在主预训练之后，我们在一个中期训练阶段对模型进行了长上下文和推理能力的改进。

# 中期训练

我们将长上下文适应和推理适应称为“中期训练”。它们比主预训练短得多，但仍然具有一定的通用性，旨在提升模型在这两个领域的能力。我们先来看看长上下文训练。

## 长上下文扩展

![](/images/posts/fbc07752f636.png)

在主预训练之后，我们使用额外的1000亿个Token对SmolLM3进行训练，以扩展其上下文长度。我们分两个阶段依次扩展上下文窗口，每个阶段使用500亿个Token：首先通过将RoPE的theta值增加到150万，将上下文从4k过渡到32k；然后通过将RoPE的theta值增加到500万，将上下文从32k扩展到64k。两个阶段都上采样了数学、代码和推理数据。在消融实验中，我们发现上采样特定的长上下文数据（如代码仓库、书籍和长网页，这些数据超出了我们混合数据中自然存在的长样本）并未进一步提升模型在RULER和HELMET基准测试上的性能。使用NoPE并在包含更长序列和更高RoPE theta值的衰减混合数据上进行训练，足以使模型在长达64k的上下文长度上获得有竞争力的性能。

遵循Qwen2.5的做法，我们使用YARN来外推超出训练上下文长度的范围。在推理过程中，模型最多可以处理128k的上下文（即在64k训练长度基础上扩展2倍）。

## 中期推理训练

在扩展了模型的上下文长度后，我们在中期训练阶段对其进行训练，以融入推理能力。中期训练阶段与预训练及后训练阶段的主要区别在于，我们旨在培养一种通用能力，而非专注于特定领域。在我们的案例中，我们希望训练模型进行推理，但不针对特定领域，如数学或计算机代码。

我们的中期训练数据集包含350亿个Token，来源于Open Thought的OpenThoughts3-1.2M以及NVIDIA的Llama-Nemotron-Post-Training-Dataset-v1.1中带有R1推理轨迹的一个子集。我们使用了ChatML聊天模板和包装打包技术，以避免向模型提供过多结构信息。我们对模型进行了4个周期（约1400亿个Token）的训练，并使用该检查点进行后续的监督微调阶段。

# 后训练

像DeepSeek R1和Qwen3这类推理模型的发布，展示了当模型能够进行显式推理时所展现的强大能力。然而，社区仍然缺乏完全开放的、使用公开数据集的完整方案来构建同时支持推理和非推理模式的双重指令模型。大多数现有方法涉及复杂的强化学习过程和专有数据集，使得研究人员难以复现并基于这些成果进行构建。

在本节中，我们将解释我们如何应对这些挑战，并分享我们构建双重指令模型的完整方案。我们将详细说明如何通过精心设计的训练流程来平衡推理模式和非推理模式之间的性能，该流程包括用于通用推理能力的中期训练、使用合成数据生成的监督微调，以及使用锚定偏好优化（APO）——一种DPO的最新变体——进行对齐。

![](/images/posts/12d22735b89e.png)

## 构建聊天模板

在深入探讨训练方法之前，建立用户如何与我们的双模式模型交互至关重要。聊天模板作为接口，使用户能够在推理和非推理模式之间无缝切换，其设计直接影响我们的训练数据格式和模型行为。SmolLM3的聊天模板允许用户在对话过程中控制推理模式。用户可以通过在系统提示词中分别包含`/think`和`/no_think`标志来激活推理或非推理模式。在非推理模式下，我们会在模型的响应中预填充空的思考块，类似于Qwen3的做法，以确保直接给出答案而不进行显式推理。

SmolLM3支持工具调用，其聊天模板包含两个独立的工具描述部分：XML工具和Python工具。在我们的实验中，这种特定的分类被证明有助于模型准确解释每种格式的工具定义。

聊天模板为两种推理模式提供了默认的系统消息，以及一个包含日期、知识截止日期和当前推理模式的元数据部分。用户可以通过提供带有`system`角色的消息来替换默认系统消息。通过在系统提示词中使用`/system_override`标志，可以排除元数据部分，为特定用例提供灵活性。

## 监督微调

在中期推理训练阶段（我们在该阶段使用1400亿个通用推理数据的Token对模型进行了训练）之后，我们继续进行监督微调，以融入数学、代码、通用推理、指令遵循、多语言和工具调用等方面的能力，涵盖推理和非推理两种模式。训练双模式模型需要仔细平衡数据混合比例，以在所有目标领域中保持两种模式的强大性能。为了评估SmolLM3在整个训练过程中的性能，我们跟踪了以下领域：数学、代码、通用推理、指令遵循和多语言。

我们在构建推理模式数据集时遇到的主要挑战是，包含某些领域推理轨迹的数据集稀缺。为了弥补这一差距，我们通过使用现有非推理数据集中的提示词，在推理模式下提示Qwen3-32B来生成合成数据。这使得我们能够提升模型在推理模式下最初表现不佳的领域的性能，例如多轮对话、多语言和日常对话。

![](/images/posts/4c9ec879b73c.png)

我们最终的数据混合比例是经过大量消融实验的结果，这些实验考察了推理与非推理Token的最佳比例以及每种模式内的数据构成。最终的SFT数据集包含18亿个Token：非推理模式10亿个，推理模式8亿个，由12个非推理数据集和10个包含推理轨迹的数据集组成。我们使用BFD（最佳适应递减）打包方法训练了4个周期（约80亿个Token），并在用户回合和工具调用结果上对损失进行了掩码处理。

我们将发布此数据混合比例以及完整的训练脚本，以使社区能够复现并基于我们的工作进行构建。

## 使用锚定偏好优化进行离策略模型对齐

在SFT步骤之后，我们使用Tulu3偏好数据集（用于非推理模式）和新的合成偏好对（用于推理模式，由Qwen3-32B和Qwen3-0.6B生成）的组合进行了一轮模型对齐。为了确保完全覆盖非思考数据集中的所有领域，我们生成了补充的思考模式偏好对。我们选择Qwen3-32B的生成作为“被选中的”响应，选择Qwen3-0.6B的响应作为“被拒绝的”响应，用于锚定偏好优化对齐。

![](/images/posts/213721cadfc4.png)

锚定偏好优化是直接偏好优化的一个变体，它提供了一个更稳定的优化目标。在DPO中，奖励函数 r_θ(x,y) 衡量的是训练期间序列概率相对于训练开始时模型（即参考模型）概率的对数比：

![](/images/posts/7b415cf4b1a7.png)

这里的 β 控制着被优化的模型相对于参考模型可以改变的程度。DPO损失优化的是提示词 x、被选中的响应 y_w 和被拒绝的响应 y_l 的三元组：

![](/images/posts/1f635ae481af.png)

APO目标已被证明更加稳定，我们在内部消融实验中也观察到了更高的下游性能。

![](/images/posts/0d45bba3d446.png)

尽管下游评估显示在数学、科学、指令遵循、编码、聊天和多语言任务上都有所改进，但我们观察到在RULER等长上下文基准测试上性能有所下降。我们将这种性能下降追溯到中期推理训练阶段，该阶段对推理能力的关注影响了长上下文性能。此外，由于我们绝大多数的推理数据集长度都低于24k个Token，APO训练数据被限制在24k个Token以内。

为了减轻这种性能下降，我们探索了模型合并作为一种解决方案。

## 模型合并

模型合并是一种流行且强大的技术，它允许结合不同模型的优势，而无需集成带来的计算开销，也无需额外的训练。我们使用 `MergeKit` 库来执行模型合并，因为它包含多种合并方法，包括线性和非线性合并。

我们的合并方案包含两个步骤：

1.  获取每个 APO 检查点并创建一个模型“汤”。
2.  将模型汤与一个在长内容处理方面表现强劲的中期训练检查点相结合。为 APO 模型汤和中期训练检查点分别赋予 0.9 和 0.1 权重的线性合并，取得了最佳性能。我们能够在长达 128k Token 的上下文上恢复基础模型的 RULER 分数。

由此产生的模型就是我们今天发布的检查点。它在广泛的任务中保持了性能。那么，让我们来看看这个模型以及基础模型的评估结果。

# 评估

我们在推理和非推理两种模式下评估基础模型和指令模型。首先介绍基础模型的性能！

## 基础模型

下图展示了 SmolLM3 在评估知识、推理、数学和编码能力的 12 个流行基准测试中的胜率。SmolLM3 始终优于其他 3B 模型，并与包括 Qwen3-4B 和 Gemma3-4B 在内的更大 4B 模型取得了有竞争力的性能。

用于计算胜率的评估基准：HellaSwag, ARC, Winogrande, CommonsenseQA, MMLU-CF, MMLU Pro CF, PIQA, OpenBookQA, GSM8K, MATH, HumanEval+, MBPP+

![](/images/posts/19f709d2f992.png)

SmolLM3 在知识和推理基准测试（HellaSwag, ARC, BoolQ）中取得了第一或第二名的成绩，展示了在这些核心能力上的强劲表现。数学和编码性能在 3B 级别中具有竞争力。在 Ruler 64k 上的长上下文性能表明该模型能有效处理扩展序列。

![](/images/posts/a8fd99579b7e.png)

在多语言基准测试（包括 Global MMLU, MLMM HellaSwag, Flores-200, Belebele，测试知识、常识推理、文本理解和翻译）中，该模型在五种主要欧洲语言上表现出强大的多语言性能。这表明 SmolLM3 在英语之外也保持了一致的性能。

![](/images/posts/d6270c750d10.png)

总之，基础模型在许多领域都表现出非常强劲的性能。让我们看看这如何转化为指令模型的性能。

## 双重指令/推理模型

由于 SmolLM3 同时具备指令和推理模式，我们需要在两种模式下评估该模型，并与具有相同能力的模型进行比较。

### 无扩展思维评估

我们将 SmolLM3 与其他 3B 非推理模型进行评估，并在多个基准测试中将其与处于无思维模式的 Qwen3 推理模型进行比较。如性能图所示，SmolLM3 优于其他 3B 非推理模型，包括 Llama3.2 3B Instruct 和 Qwen2.5 3B Instruct，并且处于推理模型之间的效率最佳点，显著优于 Qwen3 1.7B，同时以更低的计算成本接近 4B 模型的性能。

![](/images/posts/49d8cc126df4.png)

因此，指令模型正好处于性能与成本的帕累托前沿。让我们看看推理模型的表现如何！

### 扩展思维评估

在启用扩展思维评估 SmolLM3 的推理能力时，与其非推理版本相比，该模型在大多数基准测试中显示出显著的改进。我们观察到在具有挑战性的任务上取得了显著提升，例如 AIME 2025（36.7% 对 9.3%）、LiveCodeBench 上的竞争性编程（30.0% 对 15.2%）以及 GPQA Diamond 上的研究生水平推理（41.7% 对 35.7%）。

虽然 Qwen3 4B 在思维和非思维模式下通常都取得了最高分，但 SmolLM3 在 3B 参数级别中展示了有竞争力的性能，尤其在数学推理和复杂问题解决任务方面表现出色。该模型的双模式能力允许用户在无需推理的快速推理和带有扩展思维的更彻底分析之间进行选择。

# 如何在本地运行

SmolLM3 的建模代码已在 transformers v4.53.0 中提供，因此请确保升级你的 transformers 版本。你也可以使用最新版本的 `vllm` 加载模型，它使用 transformers 作为后端。

pip install -U transformers

```python
from transformers import AutoModelForCausalLM, AutoTokenizer

model_name = "HuggingFaceTB/SmolLM3-3B"
device = "cuda" 


tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
).to(device)


prompt = "Give me a brief explanation of gravity in simple terms."
messages_think = [
    {"role": "user", "content": prompt}
]

text = tokenizer.apply_chat_template(
    messages_think,
    tokenize=False,
    add_generation_prompt=True,
)
model_inputs = tokenizer([text], return_tensors="pt").to(model.device)


generated_ids = model.generate(**model_inputs, max_new_tokens=32768)


output_ids = generated_ids[0][len(model_inputs.input_ids[0]) :]
print(tokenizer.decode(output_ids, skip_special_tokens=True))

```

我们建议在采样参数中设置 `temperature=0.6` 和 `top_p=0.95`。

### 启用和禁用扩展思维模式

我们默认启用扩展思维，因此上面的示例会生成带有推理轨迹的输出。要在启用之间进行选择，你可以通过系统提示词提供 `/think` 和 `/no_think` 标志，如下面的代码片段所示（用于禁用扩展思维）。生成带有扩展思维的响应的代码将是相同的，只是系统提示词应包含 `/think` 而不是 `/no_think`。

```python
]

    messages,
    tokenize=False,
    add_generation_prompt=True,
)

```

### Agent（智能体）使用

SmolLM3 支持工具调用！只需将你的工具列表作为 `xml_tools` 参数传递（用于标准工具调用），或作为 `python_tools` 参数传递（用于调用像 `<code>` 片段中的 Python 函数这样的工具）。

```python
from transformers import AutoModelForCausalLM, AutoTokenizer

checkpoint = "HuggingFaceTB/SmolLM3-3B"

tokenizer = AutoTokenizer.from_pretrained(checkpoint)
model = AutoModelForCausalLM.from_pretrained(checkpoint)

tools = [
    {
        "name": "get_weather",
        "description": "Get the weather in a city",
        "parameters": {"type": "object", "properties": {"city": {"type": "string", "description": "The city to get the weather for"}}}}
]

messages = [
    {
        "role": "user",
        "content": "Hello! How is the weather today in Copenhagen?"
    }
]

inputs = tokenizer.apply_chat_template(
    messages,
    enable_thinking=False, 
    xml_tools=tools,
    add_generation_prompt=True,
    tokenize=True,
    return_tensors="pt"
)

outputs = model.generate(inputs)
print(tokenizer.decode(outputs[0]))

```
