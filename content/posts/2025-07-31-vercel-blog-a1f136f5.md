## 重新设计的聊天功能

在AI SDK 5中，我们从头重建了聊天功能。我们采用了开发者喜爱的强大LLM基础原语，并在此基础上构建了世界级的UI集成，为整个应用提供端到端的类型安全。从服务器到客户端，每一块数据、工具调用和元数据都完全类型化。这代表了Web AI库的下一代演进。

### 分离UI消息与模型消息

开发者在先前版本的AI SDK中面临的最大挑战之一是管理不同的消息类型，以及如何正确持久化聊天历史。

这是重建`useChat`时的核心考量，从而催生了不同类型的消息：

- **UIMessage**：这是您应用状态的**单一事实来源**，包含所有消息、元数据、工具结果等。我们建议使用UIMessage进行持久化，以便始终能恢复正确的面向用户的聊天历史。
- **ModelMessage**：这是为发送给语言模型而优化的精简表示。

我们已在API中明确区分了这两种类型：

```
12const uiMessages: UIMessage[] = [  ]3const modelMessages = convertToModelMessages(uiMessages);4
5const result = await streamText({6  model: openai('gpt-4o'),7  8  9  messages: modelMessages,10});11
1213return result.toUIMessageStreamResponse({14  originalMessages: uiMessages,15  onFinish: ({ messages, responseMessage }) => {16    17    saveChat({ chatId, messages });18    19    20    saveMessage({ chatId, message: responseMessage })21  },22});
```

UI消息与模型消息的这种分离使得持久化变得简单直接。`onFinish`回调以保存所需的格式提供所有消息，无需显式转换。

有关使用AI SDK实现消息持久化的完整示例，请查看我们的聊天机器人[持久化文档](https://sdk.vercel.ai/docs/ai-sdk-ui/chat/persistence)和[持久化模板仓库](https://github.com/vercel/ai-chatbot/tree/main/templates/persistence)。

### 数据部件

现代AI应用需要从服务器向客户端发送的不仅仅是LLM的纯文本响应（例如，从状态更新到部分工具结果的任何内容）。如果没有适当的类型化，流式传输自定义数据可能会使您的前端代码变成一堆运行时检查和类型断言的混乱组合。**数据部件**通过提供一种一流的方式，将任意类型安全的数据从服务器流式传输到客户端，解决了这个问题，确保您的代码在应用增长时仍保持可维护性。

在服务器端，您可以通过指定部件类型（例如`data-weather`）然后传递数据来流式传输一个数据部件。您可以通过指定ID来更新同一个数据部件：

```
123const stream = createUIMessageStream<MyUIMessage>({4  async execute({ writer }) {5    6    7    const dataPartId = 'weather-1';8
9    10    writer.write({11      type: 'data-weather', 12      id: dataPartId,13      data: { city: 'San Francisco', status: 'loading' },14    });15
16    17    writer.write({18      type: 'data-weather',19      id: dataPartId,20      data: { city: 'San Francisco', weather: 'sunny', status: 'success' }, 21    });22  },23});
```

在客户端，您可以渲染这个特定的部件。当您使用相同的ID时，AI SDK会用新的数据部件替换现有的部件：

```
12const { messages } = useChat<MyUIMessage>();3{4  messages.map(message =>5    message.parts.map((part, index) => {6      switch (part.type) {7        case 'data-weather':8          return (9            <div key={index}>10              {}11              {part.data.status === 'loading'12                ? `Getting weather for ${part.data.city}...`13                : `Weather in ${part.data.city}: ${part.data.weather}`}14            </div>15          );16      }17    }),18  );19}
```

还有一些情况，您希望发送不希望持久化的数据，而是用于通信状态更新或对UI进行其他更改——这时就需要用到**临时数据部件**和`onData`钩子。

临时部件会发送到客户端，但不会添加到消息历史中。它们只能通过`useChat`的`onData`处理程序访问：

```

### 类型安全的工具调用

`useChat`中的工具调用已重新设计，采用了特定类型的部件标识符。现在每个工具都会创建一个类似`tool-TOOLNAME`的部件类型，而不是使用通用的`tool-invocation`部件。

AI SDK 5在此基础上进行了三项改进：

- **类型安全**：通过在自定义消息类型中定义工具的形状，您可以为输入（工具的`inputSchema`）和输出（工具的`outputSchema`）获得端到端的类型安全。
- **自动输入流式传输**：工具调用输入现在默认进行流式传输，在模型生成时提供部分更新。
- **明确的错误状态**：工具执行错误仅限于该工具，并且可以重新提交给LLM。

这些功能共同使您能够构建可维护的UI，向用户清晰展示工具执行过程中的每一步——从初始调用、流式更新到最终结果或错误：

```

```

聊天也支持动态工具（详见下文）。动态工具（例如来自 MCP 服务器的工具）在开发时是未知的，可以使用 `dynamic-tool` 部件进行渲染：

## 链接到标题智能体循环控制

构建可靠的 AI 智能体需要对执行流程和上下文进行精确控制。借助 AI SDK 5，我们引入了原语，让您可以完全控制智能体的运行方式以及它们在每一步拥有的上下文和工具。

AI SDK 5 引入了三个用于构建智能体的功能：

- **stopWhen**：定义何时停止工具调用循环。
- **prepareStep**：调整每个步骤的参数
- **智能体抽象**：使用预定义设置的 `generateText` 和 `streamText`

**stopWhen**：定义何时停止工具调用循环。

**prepareStep**：调整每个步骤的参数

**智能体抽象**：使用预定义设置的 `generateText` 和 `streamText`

### 链接到标题stopWhen

当您使用 `generateText` 和 `streamText` 发出请求时，默认情况下它只运行一个步骤。`stopWhen` 参数将您的单个请求转换为一个工具调用循环，该循环将持续运行，直到：

- 满足 `stopWhen` 条件，或者
- 模型生成文本而不是工具调用（始终是一个停止条件）

满足 `stopWhen` 条件，或者

模型生成文本而不是工具调用（始终是一个停止条件）

常见的停止条件包括：

- **步骤限制**：`stepCountIs(5)` - 最多运行 5 步
- **特定工具**：`hasToolCall('finalAnswer')` - 当调用特定工具时停止

**步骤限制**：`stepCountIs(5)` - 最多运行 5 步

**特定工具**：`hasToolCall('finalAnswer')` - 当调用特定工具时停止

```
1import { openai } from "@ai-sdk/openai";2import { generateText, stepCountIs, hasToolCall } from "ai";3const result = await generateText({4  model: openai("gpt-4o"),5  tools: {6    7  },8  9  10  stopWhen: [stepCountIs(5), hasToolCall("weather")],11});
```

### 链接到标题prepareStep

虽然 `stopWhen` 让您的智能体持续运行，但 `prepareStep` 允许您控制每个步骤的设置。

在每个步骤执行之前，您可以调整：

- 消息：压缩或筛选上下文以保持在限制范围内，或过滤掉无关的Token。
- 模型：根据任务复杂度切换不同模型。
- 系统提示词：为不同任务调整指令。
- 工具：根据需要启用/禁用工具。
- 工具选择：在需要时强制使用特定工具（或不使用任何工具）。

## 工具改进

AI SDK 5 通过全面的改进增强了工具能力，包括动态工具、提供商执行的函数、生命周期钩子，以及贯穿整个工具调用过程的类型安全。

### 参数与结果重命名

在 AI SDK 5 中，我们通过重命名关键概念，使工具定义接口更贴近模型上下文协议规范：

- parameters → inputSchema：此重命名更好地描述了该模式用于验证和定义工具输入的目的。
- result → output：类似地，工具输出现在被统一命名为output。

parameters → inputSchema：此重命名更好地描述了该模式用于验证和定义工具输入的目的。

result → output：类似地，工具输出现在被统一命名为output。

AI SDK 5 还引入了一个可选的`outputSchema`属性，这与 MCP 规范保持一致，并为客户端工具调用实现了类型安全。

这些变化使得工具定义更加直观，并与新兴的行业标准保持一致：

```
12const weatherTool = tool({3  name: 'getWeather',4  parameters: z.object({ location: z.string() }),5  execute: async ({ location }) => {6    return `Weather in ${location}: sunny, 72°F`;7  }8});9
