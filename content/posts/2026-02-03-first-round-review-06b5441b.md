Neha Narkhede 是 Confluent 的联合创始人，该公司是一家数据流软件公司，于 2021 年以 91 亿美元的估值完成融资。Neha 后来联合创立了 Oscilar，这是一个帮助公司检测和管理欺诈的无代码平台。在创立这两家公司之前，Neha 是 LinkedIn 的首席软件工程师，她在此共同创建了 Apache Kafka。Neha 在《福布斯》"2023 年美国白手起家最富有女性"榜单中排名第 50 位，估计净资产为 5.2 亿美元。

Brett：让我们开始吧。非常感谢你加入我们。

Neha：谢谢你的邀请。我对此感到非常兴奋。

Brett：也许我们可以从谈论 Kafka 的故事开始，以及它在你还在 LinkedIn 时是如何起步的。然后我们将以此为基础，来谈谈对 Confluent 的洞察。

Neha：是的，你知道，当我在 LinkedIn 时，我和我的两位联合创始人一起，LinkedIn 当时面临着一个大问题，我们正在为用户构建许多实时功能。同时，我们也在构建许多离线分析类产品，以支持其中一些功能。

所以我们有两个互不沟通的世界。一个是实时应用，另一个是 Hadoop 和后端基础设施。而且系统经常出问题。你知道，有一个支持在线应用的消息系统无法扩展。

还有一个完全不同的系统支持离线应用。所以我们真正需要的是一个可扩展的数据流平台，能够融合这两个世界，拥有一个可以将数据从一处移动到另一处的系统。并且在此基础上以非常实时的方式处理数据，同时支持离线应用。

因此，我们实际上是从研究市场开始的，看看是否有人已经构建了类似的东西。所以，构建 Apache Kafka 绝对不是首选，但当我们意识到没有其他东西存在时，我们决定迈出这一步，开始了 Apache Kafka 项目，并在大约三年后将其开源。

然后，在接下来的几年里，它就起飞了。这最终促成了 Confluent 的诞生。

Brett：是什么让你决定将其开源？

Neha：是的，这是个好问题。首先，我们将其视为一项基础技术。它是公司数据的核心和灵魂。为了让它在世界上取得成功——这也是我们的雄心之一——它需要开源，因为开发者喜欢基础技术是开源的，这样他们可以获取、调整、贡献等等。这是原因之一。

第二个原因是，开源使得产品更容易使用、更好，因为整个开发者社区都会为其做出贡献。这算是第二个原因。第三个原因是，我们相信没有其他东西像 Apache Kafka 一样。为了让其他公司从中受益，它需要开源。这些是当时想到的一些原因，我们在 Apache 软件基金会下将其开源。因此，它是独立管理的。我认为这在其整体成功中发挥了作用。

Brett：但是，你在决定围绕它建立商业公司之前就开源了它？

Neha：是的，远在那之前。Confluent 是在 Apache Kafka 开源并取得成功四年后才成立的，这很大程度上是一个自然而然、水到渠成的决定。

Brett：那么，当你们将其用作内部基础设施时，主要的好处是什么？仅仅是在内部使用。以及让其他人利用它的愿望。

Neha：有几个不同的用例，对吧？一个是让它作为一个日志系统运行，这意味着它要成为我们离线系统的数据管道，这些系统在 Apache Kafka 之前经常出问题。另一个是支持流处理能力。

即处理来自我们实时应用的实时数据。比如"你可能认识的人"或"谁查看了你的个人资料"等等。然后促成这两个世界的融合，为 LinkedIn 创建一个中央神经系统，只需一次 API 调用即可构建任何新应用。

这大致是 LinkedIn 使用它的方式。如果你将其泛化，你知道，所有数字公司都需要这个，因为它们都有面向消费者的应用，都有后端系统，并且它们确实都需要数据的单一视图，以及实时或批量使用数据的灵活性。这就是我们对 Apache Kafka 的设想。

而且，你知道，它现在仍然是这样被使用的。

Brett：在 Kafka 开源很久之前，它的第一个版本是什么样的？产品的第一次迭代是怎样的？

Neha：所以第一次迭代很大程度上受到了我们想要解决的第一个用例的影响，那就是让消息系统更具可扩展性。这是因为当时现有的消息系统根本不是为互联网规模的数据设计的。因此，我们，你知道，保持了它的简洁性。

我们有一个发布 API 和一个订阅 API，它只是更简单、更具可扩展性。你知道，Apache Kafka 最初只是一个更具可扩展性的发布-订阅系统。所有其他功能都是后来才添加的。

布雷特：开源之后，你认为它为何能取得如此巨大的成功？

妮哈：我认为第一个原因是企业正在经历数字化转型。当时有大量数据需要被挖掘、采集并围绕其构建应用程序，而现有系统都无法实现规模化扩展。第二个原因是我们极大地简化了API调用和用户界面。

我们让它变得非常非常容易上手，可以在任何应用程序中快速使用。因此用户体验也是成功的重要因素。我认为除了产品差异化以及将Kafka概念化为一种独特的类数据库系统之外，可扩展性和用户体验确实是其成功的两大关键原因。

布雷特：在开源之前，你是否预料到它会如此成功？还是仅仅觉得"发布出去或许对其他人有用"？

妮哈：我们最初只是觉得发布出去后，像领英这样的科技公司可能会使用——那些技术前沿的硅谷科技公司。但后来大型企业也开始采用，甚至发展到有大型企业主动联系我们咨询，寻求Kafka部署的帮助。

那是我第一次意识到：应该围绕Apache Kafka创立一家公司，开发企业级产品，让大小公司都能采用。

布雷特：你们是否需要做大量准备工作才能开源产品？还是产品架构本身就让开源变得相对容易？

妮哈：产品从一开始就设计得非常易于开源。我们的架构设计方式、用户体验设计以及整个模块化系统的工作方式，都让开源变得极其简单。我认为最好的开源技术都是经过生产环境验证的——被用于主要应用程序中，这才是开源系统成功的关键。我很庆幸我们在开源前等待了Kafka以稳定方式投入实际应用，因此用户最初接触产品时的体验门槛很低。

布雷特：这里的一个洞见是：你们为解决自身问题而构建的东西，其实代表了众多其他公司的需求。因为项目启动时显然没考虑过会有成千上百家机构使用它。这引发了一系列有趣的连锁反应，与传统企业软件公司的路径截然不同——后者通常先设想用例，再构建闭源产品，寻找设计合作伙伴等等。

妮哈：是的，我认为成功有两个原因。一是技术差异化，这很明显——它解决了重要问题。二是市场顺风：当时企业正在以近乎10倍的规模采集数据，他们需要这样的解决方案。所以除了技术因素，还存在时机和机遇因素。

布雷特：能否详细说说你们如何从成功的开源项目过渡到创立公司？

妮哈：Confluent的创立过程非常独特，尤其是与许多初创企业的典型历程相比。Confluent植根于Apache Kafka——一个实时数据处理系统，至今仍是非常流行的开源系统。因此Confluent从0到1的阶段在几个方面都很特别。首先是起点：与传统初创企业从想法开始再寻求验证不同，Confluent始于一项已被验证的技术——当时约50%的财富500强公司都在使用Kafka。我们拥有庞大的可商业化社区基础，这是截然不同的起点。

其次是验证：Confluent的验证并非针对核心技术（我们认为时机成熟正是原因之一），而是理解Kafka生态系统的缺口，明确Confluent如何提供解决方案使Kafka更易用、易管理和可扩展。因此我们的战略高度聚焦现有用户群。Kafka拥有广泛用户基础，所以我们初期重点是：如何通过运维工具、生产环境系统连接支持、培训服务来帮助现有Kafka用户。我们对此投入了大量思考，这真正决定了产品方向。

最后但同样重要的是社区参与。Kafka的成功也离不开开发者布道，以及我们如何鼓励开发者转变思维方式，而不仅仅是使用新系统。这对Kafka社区和Confluent的从0到1阶段都至关重要。

布雷特：当Kafka开始腾飞时，决定正式创立公司是轻松的选择，还是经历了曲折过程？

妮哈：这个决定是这样的：当时我和联合创始人正在与一家财富500强企业开会，协助解决他们在生产环境中遇到的Kafka问题。当我的联合创始人在解答问题时，我突然想到："如果未来出现一家围绕Kafka的公司（这必然会发生），而Kafka的联合创始人或共同创造者却没有参与创立，那将非常遗憾。"于是我向联合创始人提议："让我们减少遗憾，尝试做这件事吧。"这其实是个相当轻松的决定，因为我们深谙产品背景，且产品已被极其广泛地采用，我们是从非常有利的起点出发的。

布雷特：我想了解更多细节：你们如何确定产品构建的起点？针对哪些客户类型和用例场景来打造最终成为Confluent的第一个版本？

妮哈：随着时间的推移，Confluent已扩展其产品线，增加更多功能、工具和托管服务来增强Kafka生态系统（这完全是另一个话题）。但最初的最小可行产品（MVP）根本上是让企业能更便捷地使用、管理和扩展Kafka。因此Confluent的MVP本质上是围绕Apache Kafka的核心能力构建的。

我们重点投入了三个领域：首先是运维便捷性——提供工具和增强功能来简化Kafka集群的部署、管理和扩展，我认为这对许多开源公司都是正确的起点。其次是数据集成——Kafka本就用于流转数据、连接系统，因此我们提供了开箱即用的连接器，方便用户连接生产系统。第三更具战略性：我们不希望Kafka仅是数据管道，而希望它成为中枢神经系统。所以我们引入了流处理能力，以Kafka为基础支持实时多方式数据处理。我们高度聚焦现有Kafka用户群的需求，既考虑开源视角，也兼顾企业视角。

布雷特：能否详细说明你们如何确定产品初版这三个支柱方向的过程？

Neha：是的。你知道，我们非常重视来自社区的反馈。这尤其影响了我们在流处理方面的开源投资决策。我们还与许多商业用户进行了大量研究——那些潜在的大型企业客户。这些研究在很大程度上影响了我们对工具的投资方向，旨在提升操作便捷性。

以Confluent为例，你知道，在当时许多企业仍在使用本地部署。因此我们必须提供软件产品，后来我们才补充了SaaS服务。所以，这很大程度上是以用户为中心的理念驱动的。

Brett：在产品早期阶段，你们是否考虑过先针对特定客户类型开发，再逐步扩展？还是说产品路线图从一开始就旨在具备扩展性，无论客户是百人规模的硅谷公司还是纽约的银行？

Neha：是的，早期我们专注于基础性投资。无论客户是小型初创公司、中型还是大型企业，我们都相信这些是Kafka必须具备的基础能力，因为它们对应着实际使用场景。因此我们深入研究用例，提炼出共性需求，最终明确：这些是开源开发者入门必备的功能，而那些则是企业级生产部署必需的能力。

总体而言，我们始终秉持第一性原理思维，注重基础性建设。

Brett：为什么这种策略比“先为20家银行定制特定功能，再逐步拓展到其他目标客户群体”更有意义？

Neha：这确实是Kafka和Confluent面临的一个挑战，因为它是一个基础性系统。各类公司将其用于各种场景，这也是我们无法像传统企业那样 narrowly focus on an ICP（聚焦单一理想客户画像）的原因之一。

Kafka的用户基础非常广泛，对Confluent而言，快速将这些用户转化为商业客户至关重要。因此我们采用这种基础性策略来加速这一进程，坦白说，任何公司都可能进入这个领域做同样的事。

所以我们采取了“先广度，后深度”的策略。

Brett：当Confluent首个版本构建完成时，你们是直接推向市场，还是通过设计合作伙伴计划等方式进行推广？

Neha：推广方式基本上就是直接推向市场，并借助我们的布道优势来推动产品采用。这很合理，因为产品很大部分实际上是开源的，所以开源并公开发布是顺理成章的。第二个原因是，我们在社区建设和核心产品开发方面积累了多年经验，对早期产品愿景和方向有很强的信念。因此直接发布并推动采用对我们来说是非常合理的策略。

事实证明，这对我们是正确的路径。

Brett：它是否立即见效？

Neha：实际上是的，MVP（最小可行产品）立即开始发挥作用。多年来，流处理能力已成为Kafka生态系统的核心优势，尤其是Confluent的产品。而操作便捷性工具对企业采用至关重要，它们基本上开箱即用。

随着市场需求和客户群的发展，产品逐步演进。但正是基于这种理解和经验，我们才真正实现了腾飞。

Brett：在创业第一年，你们遇到的最大困难是什么？

Neha：我认为最难的是将Confluent确立为企业级公司。我们当时拥有“差异化前沿技术背后的聪明开发者”的品牌形象，但要建立“Confluent不仅是社区领导者，更是可靠的企业级公司”的认知，这需要我们在市场营销、品牌建设方面投入，并发布能在开源开发与企业级需求间取得平衡的产品。

Brett：能否详细谈谈你们如何实现这一目标？如何确立企业级公司定位？

Neha：至今仍驱动公司发展的核心原则之一，也是当时驱动我们的，就是以客户为中心。

我们高度关注开源用户群和企业用户群的反馈收集。这种以用户为中心的理念很大程度上塑造了我们的愿景。第二点是Confluent做得非常好的反馈循环机制——我们与社区及客户群始终保持紧密的反馈循环。

这始终对我们有利。第三点是开发者布道。Confluent过去和现在都在开发者布道方面做得非常出色，我认为任何开源产品都必须做好这一点。这三方面因素真正影响了早期的成功。

Brett：在开发者布道方面，前1%的公司与前20%的公司还有哪些区别？

Neha：我认为从执行层面看，有些战术性事项必须做好——比如优秀的文档、定期发布更新说明等。这些是基础工作。

但区分前20%与其余公司的关键在于思想领导力。将开发者布道视为确立领域思想领袖地位的重要工具，因为这本身就是核心差异化因素。你可以在产品上竞争，但很难与领域思想领袖竞争。

我认为这正是思想领导力让前20%公司脱颖而出的原因。

Brett：优秀的思想领导力具体是怎样的？你们在头一两年围绕思想领导力做了哪些工作？

Neha：思想领导力的本质是教会终端用户新的方法论。沟通重点应聚焦领域动态、为何需要新方法，而非大量谈论产品功能、试图说服人们使用。思想领导力更多是关于教导人们不同的做事方式，在提出解决方案前，先探讨“为什么”和问题本质。

Brett：那么对Confluent而言，在头一两年你们向客户和市场传递的核心叙事是什么？

Neha：在Confluent，我们想创建一个名为“数据流”的新类别。我们不愿被归入消息系统或日志系统领域，而是从根本上希望开创一个新类别——我们称之为数据流类别。这需要大量的思想领导力建设。

要知道，说服人们理解这个新类别的必要性及其含义，以及后来为什么Apache Kafka是最佳的数据流技术——这正是我们所做的最重要的工作，而开发者布道是达成这一目标的核心工具，它实质上创造了一个全新的类别。创造一个类别需要数年时间。

这至少需要3到4年，但在类别创建上进行持续且不懈的投入，确实会带来回报。

**Brett：** 当时你们觉得应该创建一个新类别是显而易见的事吗？

**Neha：** 确实如此。因为Kafka的出现，并非仅仅因为它是一个可扩展的消息系统，也并非仅仅因为它是一个可扩展的日志系统，而在于我们洞察到它兼具两者特性。当它兼具两者时，它看起来就与两者都不同。这正是我们认为"嘿，这是一种新的做事方式，这完全是一个新事物"的原因之一。新的做事方式需要一个新名称和新概念。这就是Kafka和Confluent进行类别创建的过程。

**Brett：** 数据流这个类别是自然而然产生的吗？是人们当时就在使用这种说法，还是你们聚在一起决定"我们要创建一个新类别，我们该如何阐述它？我们将使用什么术语？"

**Neha：** 是的，我们进行了很多次讨论，思考应该称它为什么。在最初的几年里我们并没有弄对。我们曾称它为某个名字，结果让人们感到困惑。最初几年我们称它为"事件流"，人们会联想到Netflix和电影流媒体。

在我们最终确定使用今天这个名称——"数据流"——之前，经历了很多次不同的迭代。而"数据流"这个说法确实流行起来了。

**Brett：** 你们是怎么想出"数据流"这个说法的？是某天一次随意的谈话吗？

**Neha：** 这实际上是许多次对话的结晶，也是真正的产品营销介入的结果。产品营销影响了创始人和公司，促使我们以客户可能理解的术语来思考这个问题。所以，内部确实进行了大量的讨论。但我认为企业产品营销确实发挥了重要作用。

**Brett：** 我们还没怎么谈过公司的联合创始人。早期的合作动态是怎样的？你们关系都很密切，所以觉得一起创业很合理吗？然后你们是如何决定谁负责什么、谁担任什么职务这类事情的？

**Neha：** 是的。要知道，对于一个开源公司的成功，所有创造者都应该参与其中。当这种情况没有发生时，就会出现两家公司，它们会在本不该竞争的方面展开竞争，这通常归结为价格战，最终只会导致恶性竞争。因此，我们三个人一起合作，共同创立Confluent是合理的。

这实际上是一个容易的决定，因为我们从一开始就在紧密合作开发这个产品。我们有很多一起工作的经验。这是创始团队做出的最重要决定之一。第二个决定则非常自然。我们每个人都有自己的优势，在LinkedIn的Apache Kafka团队中，我们每个人都扮演着特定的角色。

因此，我们在公司担任的角色，只是我们在LinkedIn所扮演角色的延伸。所以这是一个非常容易且自然的决定，但也是我们做出的一个关键决定。

**Brett：** 你们最初是如何开始在这个项目上合作的？

**Neha：** 我想我们的合作方式是：我的联合创始人Jay Kreps编写了最初的版本，因为他先加入公司，接触到了很多问题，构想了一种新的做事方式，并编写了一个原型。那时我加入了，并非因为这是一个有趣的问题。

而是因为这是一个没人愿意解决的问题。因为在当时，构建数据管道并不那么吸引人，它们总是出问题。所以我主要是为了解决公司一个非常重要的问题而加入的，而Apache Kafka正是解决这个问题的正确方案。然后我们的第三位联合创始人，饶军，作为资深的数据库专家加入了团队，这对于将Kafka概念化为类似数据库的变更日志非常重要，深受其影响。所以，我们团队拥有非常多元化的视角。

**Brett：** 在加入这个项目之前，你在LinkedIn做什么？

**Neha：** 我当时在负责LinkedIn搜索功能的后端开发。到我接手时，那部分已经相当成熟了。我总是对公司最重要的事情充满好奇。正是这种想法影响了我对解决这个问题的兴趣。

我就是这样接触到Apache Kafka，并从搜索部门转到了Apache Kafka团队。

**Brett：** 这算不算你整个职业生涯中选择尝试加入哪些团队的一个指导原则？就是弄清楚对公司最重要的是什么。

**Neha：** 绝对是。我认为，如果你将自己的方向与公司最重要的目标对齐，你最终就会致力于解决最重要的问题，并在此过程中学到新东西。所以，将我的兴趣与公司最重要的问题对齐，这一直是我职业生涯的指导原则。

**Brett：** 你提到了这种技能互补的动态关系？能多谈谈这方面吗？在早期，每个人具体负责什么？这种配合效果非常好。我认为在一个技术性极强的产品背景下，这尤其有趣，因为考虑到有如此多的技术和产品构建工作，相比于传统的SaaS公司之类的，我本以为技能重叠会更多。

那么，技能互补具体是怎样的？在最初几年里，分工合作的动态是怎样的？

**Neha：** 对我们来说，这很自然。我的联合创始人Jay Kreps是Apache Kafka的代表人物，因此他成为Confluent的代表人物是顺理成章的，这也使他能够担任CEO的角色，而他做得非常出色。我在LinkedIn管理和运营流处理和Apache Kafka团队，因此由我在早期担任CTO和工程副总裁的角色非常合理，后来出于需要演变为CPO角色。我们的第三位联合创始人饶军，则是一位非常专注于内部的架构师，他深入代码库，并与开源社区深度互动。所以他是开源社区的代表。你看，即使在我们三人都有的工程背景范围内，这些也算是互补的技能组合。我认为，对于通常聚在一起创业的工程师来说，情况并非总是如此，但这也是首次创业的创始人的美妙之处：只要你们拥有成长型思维，你们会了解彼此的职责，并有望创造出成功的事业。

**Brett：** 在你职业生涯的早期，你是否想过要创办自己的公司？还是你只是偶然遇到了这个机会，然后觉得它可能很有趣？

**Neha：** 我是偶然遇到的，并且认为它会很有趣，主要是因为我真的对这个问题的领域充满热情，对我们构想的解决方案和新的做事方式既感兴趣又充满热情，并且一直相信，如果人们热衷于一个他们真正关心并希望带来新解决方案的新领域，就应该创办公司，而不是先想着创办公司，然后再去找问题——当然，那也是创办公司的另一种方式。

但我从根本上相信，热情应该主导创业的决定。

**Brett：** 稍微转换一下话题，我们还没谈过实际的业务。那么你们是如何赚到第一笔收入，并解决这类问题的？

**Neha：** 首先，我们从Confluent创立之初就考虑了盈利模式。我们深入思考了如何帮助开源社区的开发者，以及如何服务企业客户群中的用户。

我们主要做了几件事。我们鼓励开源采用。我们认为，那些直接赋能开源产品终端用户的功能，理想情况下应该保持开源或成为开源功能。这种做法确保了产品的核心价值主张对所有人开放。

最重要的是，我们这样做也是希望Kafka能成为解决该问题的事实标准方案。关注免费用户和社区用户基础至关重要。但另一方面，为了推动企业采用以支持关键任务应用，我们开发了一系列功能。

因此，Confluent紧密跟随企业用户的需求，提供基于传统计算订阅模式的软件产品，这包括通过运营工具对安全增强和集成能力进行收费。

这样就形成了一个非常全面的、基于订阅的软件产品。随后，我们逐步发展，提供了SaaS产品，这成为了最有效的付费门槛。我认为，对开源产品进行货币化最有效的方式之一，就是围绕它开发一个完全托管的SaaS服务。

采用SaaS模式后，用户不再期望该服务是开源的。他们受益于托管服务的便利性和可扩展性，而公司则可以从SaaS平台的附加价值中获利。核心产品仍然可以保持开源，公司也能从其分发和品牌知名度中受益，但通过SaaS产品，你可以很自然地对这个开源产品进行货币化。

**Brett：** SaaS服务是什么时候推出的？在Confluent发展历程的哪个阶段？

**Neha：** 推出得相当快。大约两年后，我们意识到必须提供SaaS服务，因为当时整个行业正处于从本地环境向公有云过渡的时期。为了保持在该领域的领先技术和产品地位，我们开发的SaaS产品Confluent Cloud变得至关重要。

你知道，这很难解决，因为你本质上是在公司内建立两种业务：一种是基于软件订阅的业务模式，另一种是基于消耗的SaaS业务模式。整合这两个世界是一个挑战，但最终为Confluent带来了丰厚的回报。

**Brett：** 你能详细谈谈你们是如何做到并应对这个挑战的吗？

**Neha：** 有几个方面。从产品角度看，我们保持了大量的重叠。我们思考了那些能同时惠及开源用户和企业用户的基础能力，并确保这些能力始终由同一个团队开发。

同时，我们有一个Kafka团队，专注于社区开发和面向社区的功能开发。我们也有独立的团队来构建专有功能集，但这些团队的所有成员都充分意识到，要以云原生的方式思考这些功能。

后来，随着我们推出云服务，我们引入了相关工具和人员，鼓励整个组织以云原生的方式思考。这大概是从产品角度整合两个世界所做的努力，这可以说比进入市场（GTM）方面的问题更容易解决。因为从进入市场方面，特别是营销来看，情况就不同了。

以软件产品为例，它的动态是不同的。你需要考虑如何影响产品的购买者，这是传统营销的起点。对于开源公司，你还需要有一个开发者布道营销部门，然后需要与产品营销部门整合。

但从SaaS服务的角度来看，营销方式又变了。你需要鼓励用户轻松开始使用产品，鼓励他们从免费试用开始等等。因此，营销看起来非常不同。整合这两个世界意味着，我们需要采取一种自下而上、以用户为中心的方法，让任何人都能轻松开始使用产品，无论是SaaS服务还是软件产品。这成为了营销领域的指导原则。而销售则完全是另一回事，动态完全不同。

这是最难解决的部分，因为在早期，如果你提供软件产品，你有传统的销售人员，他们习惯于销售基于订阅的软件产品。但是，销售基于消耗业务模式的SaaS产品则完全不同。销售团队的DNA是不同的，所以这是公司仍在进行的一个转型，可能是整个方程中最困难的部分。

**Brett：** 你们最终是否将其分成了两个拥有不同技能的进入市场团队？解决这个问题的路径是什么？

**Neha：** 我们实际上决定不拆分。这是因为我们最终希望转型为云优先的业务。因此，我们投入了大量思考和投资在销售赋能上，比如内部教育，解释为什么这很重要，并逐步改变薪酬结构，以鼓励云产品的采用和扩展。

我认为这是一个非常渐进且深思熟虑的方法，但我们没有在进入市场端或产品端进行隔离。

**Brett：** 当你们回顾公司赚到的第一两百万美元时，这些都是创始团队完成的吗？还是很早就开始组建实际的进入市场团队了？

**Neha：** 对于Confluent来说，头很多年基本上完全是自然流入的，因为Kafka用户知道我们是Kafka公司，他们开始主动联系Confluent。所以最初的几笔销售是创始人主导的，相当直接。然后我们聘请了一位更全面的产品负责人，他帮助了下一阶段的销售，并且非常迅速地增加了第一批企业销售代表，他们做得非常好。

**Brett：** 他们是在公司成立多少年后加入的？

**Neha：** 在第一年年底左右，我想我们有两名企业销售代表开始积极超额完成任务。这使得我们能够在接下来的几年里相当积极地扩大销售团队。

**Brett：** 但你们是从销售代表开始的，而不是一开始就聘请进入市场负责人。

**Neha：** 是的，我们先招了一些销售代表，然后在那之后才增加了进入市场负责人。

**Brett：** 你认为这个操作顺序正确吗？

**Neha：** 这个操作顺序，总的来说我认为是正确的。因为第一阶段，创始人主导的销售对于理解客户为什么购买、他们的问题是什么、销售代表需要什么样的背景才能成功，都极其重要。然后通过招聘最初几名销售代表来试水，看到他们成功，从而确保你的假设实际上是正确的。

然后，聘请一位销售负责人来接手并扩大规模。所以，这是一个经过深思熟虑的、非常刻意的决定。

Brett：你是否经历了一个学习如何销售的过程，还是说这很自然？我认为，大多数超级技术型人才的典型情况是，销售往往不是自然而然就会的，但我很好奇你的销售之路是怎样的。

Neha：对于Confluent来说，情况有点不同。你知道，这使得我们某种程度上不需要学习一些销售技巧。对于Confluent，我们是在转化一个现有用户群，从一个产品升级销售到另一个产品。我们并不是在销售一个产品本身，他们已经对这个产品有购买意向了。

他们只是需要一些帮助，我们说服他们，这就是他们成功所需的功能集和支持。所以，实际上，有趣的是，这是一个非常简单的销售过程，因为他们已经确信自己需要它。

Brett：你们是从什么时候开始需要做更多传统的对外销售或企业销售，而不是依赖自然流入的销售？是仅仅因为产品性质使然，还是在很多很多年之后？

Neha：几年后，你知道，很明显，为了让Confluent达到下一个规模阶段，我们需要发展对外销售的能力，而这件事发生得相当自然，当时我们的ARR（年度经常性收入）接近1亿美元，这令人惊讶，但你知道，这都要归功于我们庞大的Kafka用户群。

所以几年后，我们开始，你知道，开展一些营销活动，并建立了像SDR（销售开发代表）这样的组织，可以，你知道，利用现有的Kafka用户群，但更主动地去接触，这，你知道，如今已成为Confluent一个相当核心的优势。

Brett：实际的买家是谁？在非常早期的时候，比如，最终是谁能够签署并支付产品费用？

Neha：在早期，甚至今天可能仍然如此，他们很大程度上是业务线买家，主要是因为Kafka在公司内部支持特定但多样化的用例。所以Confluent面临的挑战较少在于说服每个用例他们需要Confluent。更多的是，与不同的业务线买家接触，你知道，将他们聚集到一个中心平台上。

并且说服和，你知道，某种程度上帮助他们扩展多个用例，或者在公司内部逐步地在一段时间内扩展。所以，你知道，销售代表们花了更多时间在这种"先立足后扩展"的策略上，并在一段时间内将不同的买家聚集在一起，直到Kafka或Confluent成为公司内所有实时应用事实上的系统。

Brett：当你稍微退一步，回想Confluent创立的前五年，有哪些艰难的决定或困难的时刻最终被证明是正确的？听起来有很多事情是凭直觉就很明显的。但我确信一路上也有你们做出的、最终获得回报的赌注，而这些在当时并非共识。

我也认为Confluent的故事在事后看来似乎有些显而易见。但我确信，你知道，一周又一周，一个月又一个月，它并不总是像现在看起来那样必然。

Neha：我认为，你知道，如果回想前五年，有很多小事，但我想对我来说印象最深的两件大事是SaaS（软件即服务）产品推出的时机。我个人当时并不像我的联合创始人那样确信SaaS产品推出的时机，但结果证明他是对的，因为那个SaaS产品需要一段时间才能稳定下来，并且，你知道，本质上实现自助服务才能成功。

所以我们真的，你知道，做出了一个非常艰难的决定，在一家公司内运营两项业务，并在第二年就开始着手SaaS产品的工作。那可能是最关键的决定。而今天，我们的云业务正在增长，并且，你知道，几乎也快超过软件业务了。

第二件事是关于战略性的许可决策。我们面临着来自云提供商的竞争，就像开源领域的任何其他参与者一样。这些提供商的情况是，虽然他们没有积极投资开源社区，但他们通常通过托管原样的开源产品来提供托管服务。

所以他们可以利用产品的流行度，但并没有，你知道，为社区的发展做出太多贡献。因此，为了应对这种竞争，Confluent在许可方面做出了一些战略决策。我们引入了增值产品，如流处理、ksql，采用的许可证在精神上与开源友好的Apache 2.0非常相似，但这些许可证有特定的条款，防止其他实体仅仅围绕该产品提供托管服务。所以这一举动在当时非常有争议。当然，许多其他开源公司也采取了类似的方法，但并非完全相同。

但它实际上帮助我们保护了开源社区的利益，并且，你知道，确保那些从产品中受益的人也能为其增长和可持续性做出贡献。所以，你知道，这为我们提供了某种商业实力，以便投资于开源社区。

所以这些，我认为是我想起来的两件主要事情。

Brett：你能更详细地解释一下那个战略性许可决策的见解吗？也许它为什么有争议，以及最终结果如何？

所以这是每个开源公司实际上都会面临的问题。因此这相当普遍，但开源公司并没有采用相同的方法，你知道，他们各自使用不同的许可证来做出这种许可决策。因此，需要做大量工作来向人们解释你正在使用的具体方法。

来做出这个许可决策。我们花了大量时间做的另一件事，可以说实际上更重要，那就是说服我们的开源社区，我们并没有背离开源精神。我们仍然非常致力于此。而且这是非常开源友好的。

这基本上是Apache 2.0，加上一个事实，即没有人可以直接拿去围绕它提供托管服务。所以，我们投入了大量的思考和努力，以确保两个社区都理解我们为什么这样做。以及许可证本质上是Apache 2.0这一事实。

Neha：你知道，这是个很好的问题，因为我们遵循了相同的方法，在开源用户群和企业用户群之间取得平衡。所以，有一些产品是开源社区为了推动Kafka采用进入下一阶段所需要的。

Brett：你们是如何在如此早期阶段规划团队结构来推进云服务的？

Neha：从团队结构的角度来说，如果从产品侧开始讲，我们保持了基础团队不变。始终只有一个团队负责Apache Kafka及其基础架构的开发，也只有一个团队负责开发特定的专有功能集。我们从未将团队拆分。

这是我们在产品侧为应对这次转型所做出的一个非常有条理的决定。

Brett：你认为如果Confluent一开始就只提供云服务，而不是同时构建这两种业务模式，公司还能成功吗？还是说实际采取的方式才是全局最优的路径？

Neha：这是个很好的问题。我们事后也思考过这个问题——所有创始人都想过：如果我们一开始就只做SaaS服务是否可行？答案可能是否定的。因为开源公司需要做的最重要的事情之一，就是迅速确立自己作为市场领导者的地位，迅速让用户明白，如果你需要那个开源产品，只有这家公司是真正值得选择的。

这意味着你需要尽快聚集现有用户群并转化他们。而当时的高端用户群是在本地部署环境中的。如果我们早期就放弃这部分，很可能会错失转化这个重要用户群的机会——虽然多年后可能仍能转化他们，但这会带来被其他进入者抢占这部分市场的风险。

所以我认为，这是我们以软件产品起步的主要原因之一。这至今仍是一项重要业务。事实证明这是正确的决定，但它也带来了一系列独特的运营挑战。

Brett：我想稍微转个话题，谈谈创办另一家公司。我认为这是一个非常值得探讨的领域，因为能创建一家持续发展、取得成功的杰出公司，然后又继续创办另一家公司的创始人案例实在太少了——这背后有各种原因。我很好奇，你决定创建另一家公司的过程是怎样的？早期是如何形成坚定信念的？

Neha：对我来说，在因家庭原因短暂休息后，我感到有些无聊，于是开始深入思考：我是谁？我热爱什么？这让我得出一个至今珍视的领悟：我本质上是一个建造者。我热爱每天早晨醒来，与团队一起构建有趣的事物。

**Brett：** 我注意到的一点是，当你成功创立一家公司后，往往会形成非常固化的观点，涵盖从产品构建到定价、包装及分销等各个方面。

我认为，这就是构建Oscilar与构建Confluent的不同之处。

**Brett：** 你是有意选择了一种相当不同的公司类型，还是恰好它就不同了？

**Brett：** 能否更具体地谈谈，你是如何探索并最终确定这个新公司的方向的？这个过程有多少是刻意的规划，又有多少是直觉使然？

**Neha：** 有几个因素促成了它。首先，我通过研究了解到，如果你顺应市场趋势，在一个正在发生巨大市场转型的领域，提供最具差异化优势的产品，那么这就是一个值得投入的正确问题。这也是我选择风险和欺诈领域的一个重要原因。特别是随着疫情到来，一切都在向线上转移，但目前仍然没有一个合适的、实时的、AI驱动的解决方案来解决这个问题，无法在几毫秒内（而不是几秒或几分钟）给出风险决策。

**Neha：** 确实还考虑过另一个问题领域，同样源于我的经验，即帮助公司构建客户360度产品，拥有一个中心平台来了解客户使用模式和客户数据，这显然由像Kafka这样的流处理系统驱动。

结果发现，那个市场确实有些拥挤，时机也不对。而且差异化不够明显。所以，它并没有完全符合我作为二次创业者所学到的那些重要标准。我之所以能迅速锁定风险领域，是因为：

这是一个我真正热衷的问题，也是市场上一个重大的问题。

**Brett：** 当你听到头条风险和欺诈这类词时，感觉上这是一个非常拥挤的领域，但显然你找到了一个你认为尚未被满足需求的细分方向。能多谈谈这个吗？因为听起来好像有50家甚至150家公司在这个领域做些什么。

你是如何找到这个切入点的？

**Neha：** 是的。我们的观察和核心论点是，事实上，你说得对，这个领域确实有很多公司。但问题在于，这些都是点状解决方案，它们只专注于问题的某个特定部分。比如，一家公司可能专注于评估你的邮件风险，另一家公司可能专注于评估你的设备风险，等等。

所以现在，企业最终采用了大量这类点状解决方案，每个方案都只提供用户风险画像的碎片化视图，这实际上导致了对用户风险的决策不够优化。因此，主要的洞见是，你需要一个统一的风险平台，能够从客户旅程的每个触点收集信号，能够将所有点状解决方案的数据整合到一个平台中，从而提供用户风险画像的360度全景视图，并最终在50到100毫秒内做出关于该用户风险的最终决策。所以，正是洞察到这个领域缺乏这种整合的统一平台，

才让我高度确信存在一种非常差异化且独特的方法，而这种方法也与我客户调研中风险负责人所传达的问题和需求相匹配。

**Brett：** 我真正感兴趣的一点是，一个问题和一个背后蕴藏巨大商业机会的问题之间的区别。也就是说，人们要么愿意为此付费，要么愿意花大价钱来解决它。在你创建这两家公司的过程中，你是否发现这两者有时并不一致？

**Neha：** 是的，特别是当我为Oscilar进行客户调研时，我发现，确保达到让某人说出“我愿意为此付费，你能把它做出来吗？”这个阶段，真的非常重要。而达到这个阶段需要大量的时间和迭代。我选择这个问题的另一个重要原因是，这里存在一个清晰的“自建还是购买”的决策，
