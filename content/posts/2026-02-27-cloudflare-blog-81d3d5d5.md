# 互联网上最常见的用户界面？重新设计Turnstile与质询页面

2026-02-27

- Leo Bacevicius
- Ana Foppa
- Marina Elmore

![](/images/posts/3cf5c1369256.png)

你一定见过它。也许你并未有意识地注意到，但你肯定见过。那个请求你验证是否为人类的小组件。那个访问网站前需要经历的全页安全检查。只要你在互联网上花费过时间，就必然遇到过Cloudflare的Turnstile小组件或质询页面——次数可能多到难以计数。

Turnstile小组件——数百万网站上的熟悉景象

当我们说互联网的很大一部分流量经由Cloudflare时，我们是认真的。我们的Turnstile小组件和质询页面每天被呈现76.7亿次。这不是笔误。是数十亿次。这很可能就是互联网上曝光量最高的用户界面。

而这带来了巨大的责任。

设计一个拥有数十亿用户眼球的产品不仅充满挑战——它需要一种根本不同的方法。每个像素、每个词语、每次交互，都必须同时为日本乡村的老奶奶、圣保罗的青少年、柏林的有视力障碍的开发者、以及拉各斯忙碌的高管顺畅工作。尤其是在他们感到沮丧的时刻。

今天，我们将分享重新设计Turnstile和质询页面的故事。这个故事由我们三人分三部分讲述：影响我们决策的设计过程与研究（Leo）、在空前规模上部署变更的工程挑战（Ana）、以及对数十亿用户产生的可衡量影响（Marina）。

让我们从设计视角如何切入这个问题开始。

## 第一部分：设计过程

### 问题所在

坦率地说：没人喜欢被要求证明自己是人类。你知道自己是人类。我知道我是人类。唯一似乎不相信的，就是那个横亘在你和你试图访问的网站之间的小组件。往好了说，这只是个小麻烦。往坏了说呢？你可能曾愤怒到想把电脑扔出窗外。我们都经历过。没人会责怪你。

Turnstile集成在登录流程中

随着世界逐渐适应这场看似不可避免的AI革命，安全验证的需求只增不减。在Cloudflare，我们观察到机器人攻击显著增加——作为回应，各组织正加大安全措施的投入。这意味着更多终端用户会更频繁地面临验证质询。

数据说明了一切：

2023年：每日21.4亿次
2024年：每日30亿次
2025年：每日53.5亿次

这意味着安全检查次数年均增长58.1%。更多的安全检查意味着终端用户感到沮丧的机会更多。越多公司集成这些验证系统来保护自身和客户，某个地方的某个人遭遇糟糕体验的可能性就越高。

我们知道，是时候认真审视我们的旗舰产品并扪心自问了：我们对遭遇这些体验的数十亿用户做得对吗？我们是否在履行"构建更好的互联网"的使命——不仅是更安全的互联网，更是更具人性化的互联网？

我们发现答案是：我们可以做得更好。

### 设计审计

在重新设计任何东西之前，我们需要了解我们正在处理的对象。我们首先对Turnstile和质询页面的每一种状态、每一条错误信息、每一次交互进行了全面审计。

我们发现的情况并不理想。

Turnstile小组件中不一致的状态。多种状态缺乏统一处理方法

不一致之处非常明显。对于众多不同的错误场景，我们没有统一的方法。有些信息过于冗长和技术化（"您的设备时钟设置错误，或者此质询页面被中间节点意外缓存且不再可用"）。另一些则过于模糊，毫无帮助（"超时"）。视觉语言差异巨大——不同的布局、不同的层级结构、不同的语气。

我们还研究了在线收到的反馈。社交媒体、支持工单、社区论坛——我们阅读了所有内容。沮丧感是显而易见的，而且其中许多本可避免。

以我们的反馈机制为例。我们为用户提供了"小组件有时失败"和"小组件总是失败"等反馈选项。但这到底有什么区别？用户又该如何知道它失败的频率？我们是在用户最沮丧的时刻，要求他们解读模糊的选项。我们留给用户解读的空间越大，反馈的用处就越小——我们在社交媒体渠道上看到的沮丧情绪也就越多。

之前的反馈界面："小组件有时失败"与"小组件总是失败"——区别何在？

我们的质询页面——即当我们检测到可疑活动或网站所有者提高了安全设置时出现的全页安全拦截——也存在类似问题。有些状态令人困惑。另一些使用了过多技术术语。许多页面在用户最需要时未能提供可操作的指导。

质询页面上不一致的状态。多种状态缺乏统一处理方法

这次审计令人谦卑。但它让我们清晰地看到了需要关注的焦点。

## 绘制用户旅程图

为了设计更好的体验，我们首先需要理解用户可能采取的每一条路径。什么是顺畅路径？甚至存在顺畅路径吗？哪些是导致沮丧升级的不顺畅路径？

绘制完整的用户旅程图——从初次遇到到错误场景，并跟踪情绪变化

这是一次真正的跨职能协作。我们与像Ana这样的工程师紧密合作，她了解每个边缘案例的技术细节；也与产品端的Marina合作，她不仅理解产品如何工作，更理解用户对它的感受——我们在网上看到的喜爱与厌恶。

Cloudflare拥有一些最聪明的人在研究机器人防护。但智能与清晰不是一回事。技术复杂性与用户简洁性之间需要微妙的平衡。只有当这两者成功共舞时，我们才能以真正让人理解的方式传达信息。

关键在于：信息传达必须对所有人都有效。任何年龄的人。任何心智或身体能力的人。任何文化背景的人。任何技术水平的人。这就是大规模设计的真正含义——你不能忽视边缘案例，因为在如此规模下，它们不再是边缘案例。

## 建立统一的信息架构

UX设计领域最具影响力的书籍之一是Steve Krug的《点石成金》。其核心原则很简单：用户花费在尝试解读、理解或解码你界面上的每一刻，都是摩擦时刻。而摩擦，尤其是在沮丧时刻，会导致用户放弃。

我们的审计显示，我们让用户思考得太多了。不同的信息在不同状态下占据了UI中的相同位置。没有一致的视觉层级。用户在Turnstile中遇到错误状态时，会发现信息位置与在质询页面上完全不同。

我们做了一个根本性的决定：**一个统一的信息架构统领全局**。

展示统一信息架构的视觉图表，Turnstile小组件和质询页面采用一致的结构

现在，Turnstile和质询页面都将遵循相同的结构模式。相同的视觉层级。操作按钮、解释性文本、文档链接的相同位置。

这是否限制了我们的设计选择？毫无疑问。我们不得不放弃许多不符合框架的创意想法。但约束并非优秀设计的敌人——它们往往是其最佳盟友。通过限制选择范围，我们能够更深入地打磨真正重要的细节。

对用户而言，其益处是深远的：他们无需重新理解界面每个元素的含义。错误状态呈现一致的外观。帮助链接始终位于固定位置。一旦理解一种状态，就能理解所有状态。这将认知负荷降至最低——而这正是安全验证过程中应有的状态。

## 用户研究给我们的启示

当你重新设计一个数十亿人使用的产品时，如何保持责任感？答案是：大量测试。

我们在8个不同国家招募了8位参与者，刻意追求年龄、数字素养和文化背景的多样性。我们寻找的不是精通技术的早期使用者——我们想了解重新设计如何为所有人服务。

我们的方法严谨细致：参与者同时体验现有版本和改版方案，且不知道哪个是"旧版"或"新版"。我们采用平衡定位法消除偏见。我们不仅测试新想法，还重新审视那些我们认为需要改变的前提假设。

A/B测试中正在测试的两种不同版本的Turnstile

### 有些东西无需修正

我们曾有一个假设：是否应该与竞争对手保持一致？大多数验证码服务商在所有状态都显示"我是人类"。而我们使用差异化的内容——"验证您是人类"，然后是"验证中..."，最后是"成功！"

我们是否把事情复杂化了？我们对此进行了直接对比测试。

我们的方案取得了决定性胜利。在交互状态中，"验证您是人类"获得8分中的5分，而"我是人类"仅得3分。在验证状态中，差距更为显著——7.5分对0.5分。用户希望了解正在发生什么，而不仅仅是被告知自己的身份。

用户测试结果：用户明显更青睐我们的方案而非竞品风格设计

这项实验最终并未作为功能发布，但其价值无可估量。它让我们确信，我们的差异化设计并非为了不同而不同。有些设计原本就是正确的。

### 但这些需要改变

研究揭示了四个未能满足用户需求的领域：

**提供帮助，而非官僚流程。** 当用户遇到错误时，我们提供"发送反馈"选项。测试中用户感到困惑："我要发给谁？网站？Cloudflare？还是我的网络服务商？"更重要的是，我们发现了一个根本问题：在极度沮丧的时刻，人们不想提交报告——他们想解决问题。我们将"发送反馈"改为"故障排除"——这个词承诺的是行动而非官僚流程。

存在问题的"发送反馈"提示：用户不知道反馈对象是谁

**引起注意，而非制造恐慌。** 我们曾大量使用红色背景表示错误。测试中的反应非常强烈——参与者感到自己失败了，感到无能为力。即使是重试即可解决的简单问题，用户也会往最坏处想并选择放弃。高饱和度的红色传递的不是"这里有需要处理的问题"，而是"你已失败，且无能为力"。解决方案是：红色仅用于图标，绝不用于文字或背景。

演变过程：从使用红色但错误状态描述不清的状态，到使用中性色文字实现更清晰简洁的错误信息传达。

**简洁可读，而非冗长繁琐。** 我们曾试图详尽解释，用技术细节描述错误。结果适得其反。非技术用户感到疏远，技术用户又不需要。所有人都在小部件有限的显示区域里费力阅读。教训是：少即是多，尤其在压力时刻的有限空间内。

**为所有人而设计。** 审计发现某些状态下使用了10px字体。灰色文字虽然技术上符合AA标准（正常文本至少4.5:1对比度，大文本3:1），但实际阅读困难。当服务对象是整个互联网时，"技术合规"远远不够。

我们设定了明确目标：达到**WCAG 2.2 AAA标准**——这是最高、最严格的网络可访问性合规等级，旨在让最广泛的用户群体（包括重度残障人士）都能访问内容。在整个重新设计过程中，当视觉一致性与可读性冲突时，可读性永远优先。

这超越了视觉范畴。我们为屏幕阅读器用户、纯键盘导航者和色觉异常者进行设计——超越了自动化合规工具能检测的范围。

可访问性不仅关乎残障——还关乎语言。英语中合适的表述，在德语中可能溢出界面。西班牙语中的简洁表达，在日语中可能产生歧义。支持40多种语言迫使我们彻底简化设计。如今，"无法连接到网站/故障排除"这一模式在英语、保加利亚语、丹麦语、德语、希腊语、日语、印尼语、俄语、斯洛伐克语、斯洛文尼亚语、塞尔维亚语、菲律宾语等众多语言中都能适用。

重新设计的错误状态在12种语言中的呈现——尽管文本长度不同，布局保持统一

## 最终设计方案

那么我们实际交付了什么？

首先谈谈我们未作改变的部分。顺畅流程——"验证您是人类"→"验证中..."→"成功！"——测试结果极佳。用户能理解每个阶段的状态。我们曾担心可能过于复杂的差异化状态内容，实际上成为了我们的竞争优势。

顺畅流程：验证您是人类→验证中→成功！这些状态测试良好，基本保持不变

但对于需要改进的状态，我们根据所有研究发现进行了重大调整。

### 简化、可快速浏览的内容

我们大幅减少了错误状态的文本量。不再使用冗长的解释如"您的设备时钟设置错误，或此验证页面被中间设备意外缓存且已失效"，现在仅显示：

1. 清晰简洁的状态名称（例如"设备时间不正确"）
2. 醒目的"故障排除"链接

清晰简洁的状态名称（例如"设备时间不正确"）

醒目的"故障排除"链接

仅此而已。详细指引现在置于专用模态窗口中，在用户需要时打开——为他们提供实际阅读和遵循故障排除步骤的空间。

故障排除模态窗口：在用户需要时提供详细指引，避免小部件杂乱

故障排除模态窗口提供背景说明（"当您的设备时钟或日历不准确时会出现此错误。要完成此网站的安全验证流程，您的设备必须设置为所在时区的正确日期和时间。"）、编号的尝试步骤、文档链接，以及——仅在用户尝试解决问题后——向Cloudflare提交反馈的选项。帮助优先，反馈其次。

### AAA级可访问性合规

所有状态现在均符合WCAG 2.2 AAA标准的对比度和可读性要求。字体大小设定了最小值。交互元素可清晰聚焦，并能被屏幕阅读器正确播报。

### Turnstile与验证页面的统一体验

无论用户遇到的是紧凑的Turnstile小部件还是完整的验证页面，信息架构现在保持一致。相同的层级结构。相同的布局位置。相同的心理模型。

验证页面现在遵循清晰的结构：顶部显示网站名称和图标，中间是明确的状态信息（如"验证成功"或"浏览器版本过旧"），下方是可操作的指引。不再有满屏的橙色或红色文字。不再有没有上下文的技术术语。

重新设计的验证页面状态，配有清晰的故障排除指引。

### 跨语言验证

所有内容均在40多种支持的语言中经过测试。我们的流程包含三层验证：

1.  设计团队进行初步设计评审
2.  由我们合格的供应商进行专业翻译
3.  由母语为对应语言的 Cloudflare 员工进行最终审核

设计团队进行初步设计评审

由我们合格的供应商进行专业翻译

由母语为对应语言的 Cloudflare 员工进行最终审核

这不仅仅是翻译准确性的问题——更是要确保当不同语言的内容长度差异巨大时，视觉设计依然能够保持良好。

### 完整的图景

最终的结果是一个更清晰、更易访问、更少令人沮丧，并且——关键的是——同样安全的安全验证体验。我们并没有为了改善体验而牺牲保护。我们证明了优秀的设计和强大的安全性并不冲突。

左侧是重新设计的 Turnstile 小组件，右侧是重新设计的验证页面

但设计体验只是成功了一半。将其交付给数十亿用户？这就是 Ana 发挥作用的地方。

## 第二部分：交付给数十亿用户

#### 不仅仅是居中一个 div

有些人可能会说，前端工程师最难的部分是居中一个 div。实际上，真正的挑战往往要深入得多，尤其是在接近平台原语进行工作时。使用原生 API 构建互联网基础设施的关键部分，迫使你以不同的方式思考 UI 开发、权衡取舍和长期可维护性。

在我们的案例中，我们使用 Rust 来处理 Turnstile 小组件和验证页面的 UI。这个决定在跨平台的安全性和一致性方面带来了明显的好处，但也增加了前端的复杂性。我们中的许多人已经习惯了像 React 这样的现代框架的人体工程学设计，常见的 UI 交互几乎是免费的。使用 Rust 意味着即使是简单的交互，也需要使用像 `document.getElementById`、`createElement` 和 `appendChild` 这样的底层结构重新实现。

除此之外，与基于 JavaScript 的框架相比，编译时间和严格的检查自然减缓了快速的 UI 迭代。调试也更为复杂，因为工具生态系统仍在发展中。这些限制促使我们在 UI 开发方法上更加审慎、更加深思熟虑，并最终更加自律。

#### 小的视觉改动，巨大的全球影响

最初看起来像是内边距调整或对齐方式改变等小的视觉调整，很快揭示了一个更大的挑战：国际化。

一旦翻译就绪，我们必须确保内容在 38 种语言和 16 种不同的 UI 状态下保持可读性和可用性。仅文本长度的变化就需要仔细的设计决策。有些翻译可能比英语长 30% 到 300%。像 "Stuck?" 这样的简短英文字符串，在印尼语中变成 "Tidak bisa melanjutkan?"，在德语中变成 "Es geht nicht weiter?"，这极大地改变了布局要求。

从右到左的语言支持增加了另一层复杂性。支持阿拉伯语、波斯语或法尔西语以及希伯来语，不仅仅是翻转文本方向。整个布局，包括对齐方式、导航模式、方向性图标和动画流程，都必须镜像。这些元素中有许多是隐含地按照从左到右的假设设计的，因此我们必须重新审视这些决定，使其真正具有双向性。

有序列表也需要特别注意。并非每种文化都使用西方的 1、2、3 编号系统，硬编码数字序列可能会让界面感觉陌生或不正确。我们依赖于区域感知的编号和完全可翻译的列表格式，以确保排序在每种语言中都感觉自然且符合文化习惯。

#### 通过测试建立信心

当我们在反馈报告中开始列出行动要点时，正确性变得更加关键。每个操作都需要正确渲染、触发正确的流程，并在不同状态、语言和边缘情况下表现一致。

为了实现这一点，我们在测试上投入了大量精力。单元测试帮助我们独立验证逻辑，而端到端测试则确保新状态和语言在真实场景中按预期工作。这个测试基础给了我们安全迭代的信心，防止了回归，并确保反馈报告对用户来说保持可靠和可操作。

## 第三部分：影响

为数十亿人设计是我们认真对待的责任。在这个规模上，必须利用可衡量的数据来告诉我们设计选择的真实影响。在我们准备推出这些更改时，我们专注于**五个关键指标**，这些指标将告诉我们是否真正成功地使互联网上最常见的 UI 更具人性化。

#### 1. 验证完成率

我们的主要北极星指标是**验证解决率**：成功完成的验证占已发出验证的百分比。通过摒弃像"中间缓存"这样的技术术语，转向像"设备时间不正确"这样简单、可操作的标签，我们预计 CSR 会有显著提升。更高的 CSR 并不意味着我们对机器人更宽松；这意味着我们正在消除那些意外绊倒合法人类用户的障碍。

#### 2. 完成时间

用户在验证页面上花费的每一秒，都是他们无法获得所需信息的一秒。我们的研究表明，当用户看到一堵红色的文字墙时，常常会因为选择而不知所措。凭借我们新的可扫描、中性色调的设计，我们正在跟踪**完成时间**，以确保用户能在几秒钟内而不是几分钟内识别并解决问题。

#### 3. 放弃率变化

过去，我们大量使用"饱和红色"引起了本能的反应：用户觉得自己失败了，干脆放弃了。通过将红色仅用于图标并采用统一的架构，我们的目标是降低**放弃率**。我们希望用户感到有能力点击"故障排除"，而不是感到无能为力而点击离开。

#### 4. 支持工单量

从产品角度来看，一个较大的转变是我们新的故障排除模态框。通过在小组件内直接提供清晰的、带编号的步骤，我们将自助服务支持构建到了 UI 中。我们预计这将导致我们的客户和我们自己的内部团队的支持工单量出现可衡量的下降。

#### 5. 社交情绪

我们知道安全验证很少受人喜爱，但它们不应该因为令人困惑而被憎恨。我们正在监控社区论坛、反馈报告和社交渠道上的**社交情绪**，看看讨论是否从"这个小部件坏了"转变为"我遇到了问题，但我解决了它"。

作为一名产品经理，我的目标常常是看不见的安全——最好的验证是用户永远看不到的验证。但是，当验证**必须**被看到时，它应该是一个助手，而不是一个保镖。这次重新设计证明，**AAA 级可访问性**和**高标准安全性**并不冲突；它们是同一枚硬币的两面。通过统一 Turnstile 和验证页面的架构，我们建立了一个基础，使我们能够比以往更快地迭代，并以更人性化的方式保护互联网。
