# 推出SPEED-Bench：一个统一且多样化的推测解码基准测试

推测解码已成为加速LLM推理的关键技术。SD使用一个轻量级草稿模型来推测多个未来的Token，然后由目标模型并行验证。通过这种方式，SD可以在保持目标模型精确输出分布的同时，显著提高吞吐量。

尽管SD算法发展迅速，但其评估仍然零散，且往往不能代表真实世界的数据和服务条件。实际上，SD的推测质量和推理加速效果本质上依赖于数据、服务机制和系统。然而，现有的大多数基准测试依赖于小型提示词集、有限的语义多样性、较短的输入序列长度、批处理大小为1，或者使用不能反映生产环境的高级推理栈。

为了弥补这些不足，我们推出了**SPEED-Bench**：一个统一的基准测试，旨在使用生产级推理引擎，评估SD在不同语义领域和现实服务机制下的表现。

## 什么是SPEED-Bench？

SD必须从两个角度进行评估。

一方面，草稿质量取决于输入文本的**语义领域和熵**。另一方面，现实世界的加速效果取决于**批处理大小、输入序列长度和系统限制**，这些因素决定了推理是受内存限制还是受计算限制。

因此，SPEED-Bench为SD引入了一个基准测试生态系统。它结合了两个专门构建的数据集划分和一个统一的测量框架，每个部分都旨在捕捉SD行为的不同方面：

1.  **"定性"数据划分**：为语义多样性优化，旨在跨领域测量推测质量（草稿模型准确性）。
2.  **"吞吐量"数据划分**：构建用于评估不同输入序列长度和高并发下的系统级加速效果。
3.  **统一测量框架**：与生产推理引擎集成，标准化跨系统的评估。

这些组件共同使从业者和研究人员能够分析那些常被现有基准测试掩盖的SD行为。

## 定性划分：语义覆盖与草稿准确性

定性划分的目标是测量推测解码的质量，具体来说是跨广泛语义领域的**条件接受率和接受长度**。

SpecBench通过将广泛使用的数据集中的实例聚合到一个统一的测试环境中，首次引入了跨多样化应用场景（如多轮对话、翻译和数学推理）的统一SD基准测试。然而，尽管这是迈向标准化评估的重要一步，但它在规模和多样性方面存在关键局限。大多数类别仅包含10个样本，且平均输入长度很短（< 100个Token），这可能无法对现代草稿模型构成压力。此外，其某些类别通常缺乏结构多样性，例如多语言类别完全由德语到英语的翻译提示词组成。

虽然理论上可以在众多数据集上进行广泛评估，但这过程繁琐，不利于快速实验，并且阻碍了发布SD算法和模型的不同研究小组之间的直接比较。我们没有依赖于对不同数据集进行详尽的评估，而是策划了一个紧凑但极具代表性的子集，旨在最大化语义多样性。我们聚合了来自18个公开可用来源的数据，并将其组织成11个类别，包括编码、数学、人文、STEM、写作、摘要、角色扮演、RAG、多语言、推理和问答。

每个类别包含80个样本，总共880个提示词。与先前基准测试中常出现的类别内多样性低的问题不同，SPEED-Bench的定性划分明确优先考虑**语义多样性**。

为了实现这一点，我们使用预训练的文本嵌入模型将每个候选提示词嵌入到一个密集的向量空间中。然后，我们应用一种选择算法，最小化每个类别内样本对的平均余弦相似度。这确保了所选样本尽可能广泛地覆盖语义空间，减少了冗余并提高了评估的保真度。

**图2** 展示了这种方法的有效性，它比较了SPEED-Bench和SpecBench之间的平均语义相似度。

![图2：样本间平均语义相似度比较（越低越好）。SPEED-Bench在所有类别中实现了比随机选择和SpecBench更低的相似度。](/images/posts/6a18c5e056f7.png)

这种语义多样性对于揭示SD中依赖于领域的行为至关重要，例如低熵领域（如编码、数学）和高熵领域（如角色扮演、写作）之间的强烈对比。

## 吞吐量划分：现实的服务工作负载

虽然定性划分捕捉了草稿准确性，但它不足以评估**系统级加速效果**。

我们使用两个指标来评估系统级加速效果：**吞吐量**，即所有并发请求每秒生成的总Token数；以及**用户TPS**，即每个请求的Token生成速率。用户TPS可作为最终用户延迟的代理指标。

在生产环境中，模型在高并发和广泛的输入序列长度下提供服务，这些长度通常远长于许多SD基准测试中使用的短ISL样本。随着批处理大小的增加，推理通常会从计算受限机制转变为内存受限机制，从根本上改变了推测解码的成本效益权衡。

**吞吐量划分**正是为了捕捉这种行为而设计。

我们构建了从**1k到32k个Token**的固定ISL桶，反映了长上下文应用（如编码助手和检索增强生成）日益增长的重要性。对于每个ISL桶，提示词被聚合到三个粗略的难度类别中，分别对应低熵、混合熵和高熵领域。

每个ISL桶包含1,536个提示词（每个难度类别512个），提供了足够的数量，以便在广泛的批处理大小范围内构建稳定的吞吐量帕累托曲线（图3）。

![图3：吞吐量作为用户TPS的函数，比较了吞吐量划分（2k）上DL=1和DL=3的情况。目标是GPT-OSS 120B与EAGLE3，在vLLM上测量。点代表批处理大小从2到512。](/images/posts/29a5690c504a.png)

为了确保确定性的预填充成本，提示词以受控的方式进行截断或填充，同时保留其语义内容。

重要的是，SPEED-Bench避免使用随机Token输入进行吞吐量基准测试。正如我们稍后所示，随机Token会严重扭曲接受行为、MoE模型中的专家路由和吞吐量测量，导致过于乐观的结论。

## 统一的测量框架

跨推理引擎对SD进行基准测试带来了一个微妙但关键的挑战。

不同的引擎可能应用不同的聊天模板，以不同方式处理BOS Token，或者对输入进行不一致的分词。这些差异可能会悄无声息地改变草稿序列，使得跨引擎比较不可靠。

SPEED-Bench 引入了一个轻量级的测量框架，它在外部处理分词和提示词格式化。推理引擎接收预分词的序列，确保所有系统处理完全相同的输入。

该框架与生产级引擎集成：TensorRT-LLM、vLLM 和 SGLang。它从流式响应中捕获细粒度的时间信息，以计算接受行为、步骤延迟、用户级每秒生成 Token 数以及整体吞吐量。

### 随机 Token 会高估吞吐量

推理基准测试中的一个常见做法是使用随机 Token 来模拟提示负载。虽然这可能足以用于自回归解码，但对于推测解码算法，甚至对于没有推测的专家混合模型来说，这种方法存在根本性缺陷，如下所述。

随机 Token 会触发两种扭曲测量的失败模式：

- **简单响应**：模型识别出噪声并默认给出可预测的确认回复，人为地抬高了接受长度。

示例输出（基础模型：GPT-OSS 120b，草稿模型：EAGLE3，草稿长度：3，平均接受长度：3.44）：

看起来您粘贴了一段非常长的混合语言文本，没有形成一个清晰的问题或请求。我很乐意帮忙，但我需要更多指导。您能告诉我您想用这段文本做什么吗？例如：...

- **话题锁定**：模型锚定到噪声中的特定关键词，并产生连贯的幻觉回复，通常导致较低的接受长度。

示例输出（基础模型：GPT-OSS 120b，草稿模型：EAGLE3，草稿长度：3，平均接受长度：1.877）：

以下是一份**扩展的、可用于生产的路线图**，它将带您从第一次安装 Unity 开始，一直到完成一个**完整的、精良的 2D 平台游戏**（玩家、摄像机、敌人、收集品、UI、音频、关卡加载和最终构建）。所有内容都被分解成小块任务，每个任务都有您需要执行的确切操作以及可直接复制的 C# 代码片段....

图 5 比较了使用随机 Token 与 SPEED-Bench 工作负载测得的吞吐量。当启用推测解码时，随机 Token 将吞吐量高估了约 23%。

![Figure 5:Throughput as a function of user TPS, comparing random input tokens to the Throughput Split (8k). Target is GPT-OSS 120B with EAGLE3 drafter, measured on TensorRT-LLM. DL=3. Points represent BS from 1 to 128.](/images/posts/3826e5416414.png)

随机输入也无法在专家混合模型中触发真实的专家路由，即使在非推测设置下也会导致不准确的吞吐量测量，如图 6 所示。

## 开始使用 SPEED-Bench

发布 SPEED-Bench 是为了在研究和生产环境中建立评估推测解码的统一标准。

它使从业者能够分析不同领域的草稿准确性，测量实际服务场景下的吞吐量，并使用相同的工作负载比较推理引擎。

数据集和测量框架是公开可用的，并设计为可直接与现有的推测解码实现集成。
