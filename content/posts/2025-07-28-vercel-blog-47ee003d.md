几个月前，我们发布了**Fluid compute**，这是一种无服务器计算方案，它能更高效地利用资源、最大限度地减少冷启动并显著降低成本。最近在**Vercel Ship 2025**上，我们推出了**Active CPU 定价**，以便在 Vercel 上实现更具成本效益的计算。

采用 Active CPU 定价的 Fluid compute 每周处理超过 450 亿次请求，为客户节省高达 95% 的成本，并且空闲时间从不收取 CPU 费用。

幕后，我们花费了两年多时间来构建实现这一切所需的基础设施。

## 一切始于一个简单的想法

2020 年，React 团队开始着手开发 **React Server Components (RSC)**。RSC 允许服务器端渲染的请求在 UI 部分可用时以流式方式返回。**Next.js** 团队坚信这种模式，并开始在 RSC 之上构建 **App Router** 以实现流式功能。

Vercel Functions 运行在 AWS Lambda 上，而当时 Lambda 不支持流式响应（现在可以了，但有一些限制）。我们必须找到一种支持 HTTP 流式传输的方法。这项工作成为了后来支撑 Fluid compute 的基础。

## 构建新的传输层

AWS Lambda 有一个简单的执行模型。你发送一个请求（输入），它运行，然后返回一个响应（输出）。之后，函数实例暂停并等待下一个请求。如果在一定时间内没有其他请求进入，函数实例会自动关闭。

在底层，这些输入和输出是简单的字符串数据块，我们过去分别用它们来编码 HTTP 请求和响应。我们思考，如何才能返回一个真正的逐块流式响应，而不是单个数据块？

答案是在 Lambda 简单的事件-响应模型之外，引入**一种新的安全传输方式**，它能安全地将每个实例连接回我们的基础设施。换句话说，我们在 Vercel 和 AWS Lambda 函数之间创建了一个隧道。我们构建了一个基于 TCP 的安全协议，定义了从 Lambda 发送到我们基础设施的不同数据包类型。例如，要返回一个 HTTP 响应，我们会发送一个 `ResponseStarted` 包、一个或多个 `ResponseBody` 包，以及一个最终的 `ResponseEnd` 包。

现在，Vercel Function Router 可以通过隧道接收这些数据包，而不是仅仅一个数据块，从而重建一个正常的 HTTP 响应，并将其逐块流式传输回客户端。

为了安全地跨运行时处理这种通信，我们构建了一个**基于 Rust 的核心组件**，作为我们基础设施和用户代码之间的粘合剂。它通过 HTTP 与语言进程（Node.js 或 Python）通信，并通过我们基于 TCP 的协议与 Vercel Function Router 通信。来自语言进程的每个响应块都会被转换成相应的数据包类型，并发回我们的基础设施。

将这种协议扩展到简单的 HTTP 响应数据包之外，使我们能够快速推出诸如 `waitUntil`、`request metrics`、`session tracing`、`larger logs` 等功能。这些功能中的每一个都只是我们协议中的一个新数据包类型。

## 启示带来新的可能性

在构建过程中，我们产生了一个想法。这种传输方式允许我们将任何类型的信息从 Lambda 发送回我们的基础设施，但如果我们也能从我们的基础设施向 Lambda 发送数据呢？例如，如果我们能发送一个额外的 HTTP 请求让 Lambda 处理会怎样？

这是个很棒的想法，但最终实现起来（并且要安全地实现）比我们最初预期的更具挑战性。让我们来看看这个想法为何如此吸引人。

AWS Lambda 有一个严格的调用扩缩容模型：

- 每个函数调用（对我们来说是一个 HTTP 请求）总是由一个实例处理
- 一个实例一次总是处理一个调用
- 你需要为每个函数付费，无论它们在做什么

一旦涉及任何 I/O 操作（如查询数据库、调用 API 或 LLM），函数实例大部分时间都处于空闲状态，但传统上你仍然需要为整个挂钟时间付费。这对于需要较长时间进行推理和回复的 LLM，以及对于大多数函数实例处于空闲状态的高流量网站来说，变得尤其成问题。

复用已打开的隧道，将更多流量发送到空闲的函数实例，将使我们能够避免资源浪费、减少冷启动，并最终为任何并发请求数大于 1 的 I/O 密集型工作负载降低成本。利用相同的资源，我们可以在提升性能的同时处理更多工作，并摆脱每个实例一次调用的模型。

## 工作（和挑战）开始了

为了向给定的函数发送更多流量，我们首先需要知道哪些函数当前正在运行，以及它们在哪里运行。

Vercel 的 19 个区域中，每个区域都有许多 Vercel Proxy 和 Vercel Function Router 副本在运行，而一个给定的 Vercel Function 可能被其中任何一个 Pod 调用。

这意味着区域越大，我们运行的 Vercel Function Router Pod 数量就越多，请求进入某个 Vercel Function Router Pod 时，该 Pod 恰好已至少有一个正在运行的请求指向同一函数的可能性就越低。换句话说，我们能够复用至少一个已建立的、指向同一函数的 TCP 连接来发送更多工作请求的可能性就越低。

我们的解决方案是构建一个名为 `compute-resolver` 的新服务，它充当类似 DNS 解析器的角色，供 Vercel Proxy Pod 解析并找到 Vercel Function Router Pod。它会跟踪针对特定函数 ID 的先前请求被路由到了何处，以提高复用已建立的 TCP 连接、将更多流量发送给同一函数的几率。同时，它也负责避免 Vercel Function Router Pod 出现热点集中问题，如果客户的某个路由流量突然激增，热点集中将是灾难性的。

该服务在峰值时处理超过 10 万 RPS，p99.99 的解析延迟低于 1 毫秒。平均而言，超过 99% 的请求被路由到可能已有相同函数 ID 请求正在运行的 Vercel Function Router Pod，这显著提高了 Fluid compute 的效率。

## 保持函数健康

还有一个挑战：我们不能盲目地向一个函数实例发送更多流量。它拥有固定数量的 vCPU 和 RAM，因此向其发送过多请求会耗尽资源，导致 TTFB 增加、延迟升高，甚至更糟，出现内存不足（OOM）错误和崩溃。我们必须构建一个系统，能够在任何时间点判断给定的函数实例当前是否可以接受更多流量，这是一个很难回答的问题。例如：

*   当前的 CPU 和内存使用率是多少？CPU 是否被限制？
*   该实例的剩余生命周期是否足够处理更多请求？
*   是否接近文件描述符限制？
*   之前的请求是否出错或超时？

这些要求与典型的 Kubernetes 部署类似：您希望高效利用资源并处理突发工作负载，同时不降低性能。然而，并非所有工作负载都相同：有些会故意使用 99% 的可用内存，有些则会在事件循环延迟突发时变慢，最终您需要为每个工作负载手动调整 `HorizontalPodAutoscaler`（HPA）配置。相比之下，我们的系统每秒多次自动适应每个实例的负载状况，收集各种指标以维持最佳的资源利用率与性能比。

大多数启发式方法都在我们基于 Rust 的核心中实现，该核心包装了每个 Vercel Function，并持续通过带内方式将指标发送回 Vercel Function Router。这使得 Vercel Function Router Pod 可以尽可能乐观，当某个指标超过特定阈值时，则停止向该函数实例发送更多流量。

然而，我们不能总是信任 Vercel Function Router 内部的指标状态，在网络延迟激增或其他竞态条件下，这些状态可能不正确。为了解决这个问题，基于 Rust 的核心也维护其自身的本地指标，并且可以 `nack` 任何额外的请求，表明其当前无法处理更多流量。当 Vercel Function Router 收到 `nack` 时，它会更新该实例的内部状态，暂停向其发送更多流量，直到收到新的健康指标，然后将请求转发给另一个可用实例或启动一个新实例。

对于给定的请求，一个 Vercel Function Router Pod 可能有多个正在运行的函数实例可供选择，它会选择可用资源最多、并发请求最少的实例。我们发现这比经典的轮询算法有效得多，因为并非所有请求（即使是访问同一路由）对 CPU 和内存的影响都相同。

这些启发式方法都不是 Node.js 特有的，而是设计成完全与语言无关。Fluid compute 也支持 Python，更多语言即将推出。

## 实现更高效的定价模式

Fluid compute 使许多工作负载的成本显著降低、效率大幅提升，但仍有改进空间。如果您的请求并发量极少或没有，您的账单将完全不变，因为我们无法在同一实例中同时处理这些请求。此外，随着像模型上下文协议（MCP）这样长期运行但大部分时间空闲的 AI 工作负载持续增加，在 Vercel 上运行计算的成本可能仍然高于我们的期望。

得益于这种改进的架构，我们能够为 Fluid compute 引入 **Active CPU 定价** 来填补这一空白。您现在只需为实际使用的资源付费：

*   每个函数实例使用的活跃 CPU 时间，以毫秒计
*   每个函数实例使用的已分配内存，以 GB-小时计

对于重度 I/O 密集型工作负载，这带来了在原有 Fluid compute 节省基础上的额外 90%+ 节省。这使得 Vercel 成为部署任何类型应用程序（从前端到后端再到 AI 应用）的理想平台。

如今，超过 75% 的 Vercel Function 调用都在使用 Fluid compute，节省了高达 95% 的计算账单成本。计算资源利用更高效，并且得益于我们新的 **Active CPU 定价**，成本效益更高。Fluid compute 是 **新项目的默认选项**，如果您尚未启用，我们建议 **在您的现有项目上启用它**。
