## 链接到标题图像生成

根据文本提示生成图像是一项新颖的生成式 AI 能力，它开启了新型应用和工作流程。生态系统正在快速增长，像 Replicate 这样的提供商支持数百种不同的图像生成模型，并且每天都在增加更多模型。

通过 AI SDK 4.1，我们迈出了实现多模态输出的第一步，通过新的实验性 `generateImage` 函数引入了对图像生成的支持。

```
1import { experimental_generateImage as generateImage } from 'ai';2import { replicate } from '@ai-sdk/replicate';3

```

- 立即返回，并允许您按需流式传输数据
- 完全控制流式传输的时间和内容
- 支持向消息添加注释和元数据

```typescript

7
8  const lastMessage = messages.pop();
9
10  return createDataStreamResponse({
11    execute: async (dataStream) => {
12      const relevantContent = await getRelevantContent(lastMessage.content);
13      for (const content of relevantContent) {
14        dataStream.writeData({
15          type: "source",
16          url: content.url,
17          title: content.title,
18        });
19      }
20
21      lastMessage.content =
22        lastMessage.content +
23        "\n\n使用以下信息回答问题：" +
24        relevantContent.join("\n");
25
26      const result = streamText({
27        model: openai("gpt-4o"),
28        messages: [...messages, lastMessage],

32      });

37}
```

## 工具调用改进

工具是生产级 AI 应用的核心构建模块——它们使语言模型能够与真实世界的系统和数据进行交互。然而，让工具可靠地工作可能具有挑战性。在 AI SDK 4.1 中，我们投入了大量精力使工具调用更加稳健：

### 工具调用中的改进上下文

执行工具调用时，`execute` 函数现在可以通过第二个参数访问有用的上下文：

- `toolCallId`：用于跟踪特定执行并添加工具相关注释
- `messages`：包含完整对话历史的数组，包括之前的工具调用和结果
- `abortSignal`：用于取消长时间运行的操作并转发到 fetch 调用

以下是使用这些上下文选项的示例：

```typescript
1const result = await generateText({
2  model,
3  abortSignal,
4  tools: {
5    weather: tool({
6      parameters: z.object({ location: z.string() }),
7      execute: async ({ location }, { toolCallId, messages, abortSignal }) => {
8        // 添加工具状态注释
9        data.appendMessageAnnotation({
10          type: 'tool-status',

12          status: 'in-progress',
13        });

15        // 使用 abortSignal 进行 fetch 调用
```

### 工具调用修复

当工具调用失败时，您现在可以使用 `experimental_toToolCallRepair` 函数尝试修复：

- 使用具有结构化输出的模型生成参数。
- 将消息、系统提示词和工具架构发送到更强的模型以生成参数。
- 根据调用的工具提供更具体的修复指令。

```typescript
1import { openai } from '@ai-sdk/openai';

3
4const result = await generateText({
5  model,
6  tools,
7  prompt,
8
9  // 可选的工具调用修复函数
10  experimental_repairToolCall: async ({
11    toolCall,
12    tools,
13    parameterSchema,
14    error,
15  }) => {

20    const tool = tools[toolCall.toolName as keyof typeof tools];

22    const { object: repairedArgs } = await generateObject({
```

### 细粒度错误处理

为了帮助构建更具弹性的工具调用，AI SDK 现在提供细粒度的错误类型，使调试和错误处理更加精确。每种错误类型都公开了关于出错原因的详细信息，并包含上下文数据以帮助诊断和修复问题：

- `NoSuchToolError`：处理模型尝试调用未定义工具的情况。
- `InvalidToolArgumentsError`：捕获工具参数与预期参数不匹配时的架构验证失败。
- `ToolExecutionError`：识别工具执行期间的运行时问题。
- `ToolCallRepairError`：跟踪自动工具调用修复尝试期间的失败。

这些特定的错误类型允许您实现有针对性的错误处理策略，并在工具执行失败时向用户提供更好的反馈。了解更多，请查看[错误处理文档](https://sdk.vercel.ai/docs/ai-sdk-core/error-handling)。

## 结构化输出改进

我们扩展了结构化输出能力，以支持更动态和更具弹性的 AI 应用：

### 带工具的结构化输出

我们最常请求的功能之一现已可用：将结构化输出与工具使用相结合的能力。通过 `generateText` 和 `streamText` 中新增的 `experimental_output` 选项，您可以构建既能与外部系统交互又能返回可预测结构化数据的复杂大语言模型（LLM）调用。

以下示例展示了结构化输出和工具如何协同工作：

```typescript
1import { openai } from '@ai-sdk/openai';
2import { generateText, tool, Output } from 'ai';
3import { z } from 'zod';
4
5const result = await generateText({
6  model: openai('gpt-4o', { structuredOutputs: true }),
7  prompt: "伦敦和纽约的天气怎么样？",
8  maxSteps: 5,
9  tools: {
10    getWeather: tool({
11      parameters: z.object({
12        city: z.string(),
13        units: z.enum(['celsius', 'fahrenheit']),
14      }),
15      execute: async ({ city, units }) => {
16        // 获取天气数据
17      },
18    }),
19  },
20  experimental_output: Output.object({
21    schema: z.object({
22      cities: z.array(
23        z.object({
24          name: z.string(),
25          temperature: z.number(),
26          conditions: z.string(),
27        }),
28      ),
29    }),
30  }),
31});
```

模型无需分别调用函数来确定要查询哪些城市，再为每个城市调用天气工具，而是可以在单个函数中处理整个工作流程。这使得代码更高效且更易于维护，尤其适用于输入不可预测或存在多条潜在工具路径的复杂场景。欲了解更多信息，请查阅`generateText`和`streamText`的结构化输出文档。

目前，仅OpenAI模型支持带工具的结构化输出。
