# 弥合鸿沟：从启动到登录的持续安全执行新工具

在公司会议或安全会议小组讨论中，我们最喜欢的一个“问我任何问题”环节的经典问题是：“什么让你夜不能寐？”

对于首席信息安全官来说，这个问题本身或许就有点像是噩梦。它没有一个单一的答案；它有几十个。这始终是在赋能全球分布式员工队伍发挥最佳工作效能，与确保“最佳工作”不会无意中为灾难性安全漏洞打开大门之间，存在的持续张力。

我们经常谈论“零信任之旅”，但现实是，这条道路几乎肯定铺满了摩擦。如果安全措施过于繁琐，用户会找到有创意（且危险）的规避方法。如果它以牺牲有效性为代价实现无缝体验，那么它可能不够安全，无法阻止坚定的攻击者。

今天，我们很高兴地宣布 Cloudflare 的 SASE 工具箱中新增了两款工具，旨在通过消除网络安全的“黑暗角落”来实现远程访问的现代化，同时不给用户体验增加摩擦：强制身份验证和 Cloudflare 自家的多因素身份验证。

## 解决安装与执行之间的鸿沟

当您部署 Cloudflare One Client 时，您获得了惊人的可见性和控制力。您可以应用允许访问目的地的策略，定义通过 Cloudflare 路由的互联网流量，并在应用层和网络层设置流量检查。但是，在用户尚未实际通过身份验证时，始终存在一个可见性挑战。

这种鸿沟主要出现在两种场景中：

1.  新设备：Cloudflare One Client 通过移动设备管理安装，但用户尚未进行身份验证。
2.  重新验证的灰色地带：会话过期，用户或因遗忘或因希望绕过限制，没有重新登录。

无论是哪种情况，该设备现在都处于未知状态。这是危险的。您失去了可见性，您的安全态势会回退到本地机器允许的任何状态。

### 引入强制身份验证

为了闭合这个循环，我们引入了**强制身份验证**。当通过您的 MDM 配置启用后，从机器启动的那一刻起，Cloudflare One Client 就成为互联网访问的守门人。

如果用户未处于活跃的身份验证状态，Cloudflare One Client 将：

-   默认使用系统防火墙阻止所有互联网流量。
-   使用特定于进程的例外，允许来自设备客户端身份验证流程的流量。
-   提示用户进行身份验证，并引导他们完成流程，这样他们就不必费力寻找正确的按钮。

通过将身份验证作为连接的前提条件，您可以确保每台受管设备始终处于受控状态。

注意：强制身份验证将首先在我们的 Windows 版 Cloudflare One Client 中提供，随后将支持其他平台。

## 当单一信任源不够时

大多数组织已转向将**单点登录**作为其主要的安全锚点。如果您使用 Okta、Entra ID 或 Google，您很可能在初始登录时要求 MFA。这是一个很好的开始，但在现代威胁环境中，这不再是终点线。

一个严峻的事实是，**身份提供商**是高价值目标。如果攻击者通过复杂的会话劫持或社会工程学成功入侵用户的 SSO 会话，他们实际上就掌握了该 SSO 背后每个应用程序的钥匙。

### Cloudflare 的独立 MFA：第二信任根

这就是 Cloudflare 的 MFA 可以发挥作用的地方。可以将其视为位于网络边缘的“升级 MFA”，独立于您的 IdP。

通过保持与您的 IdP 分离，这引入了另一个权威机构，任何试图访问受保护资源的用户都必须获得其“批准”。这意味着，即使您的主要 IdP 凭据被泄露或伪造，攻击者在尝试访问诸如您的生产数据库之类的资源时也会碰壁——因为他们无法访问第二个因素。

Cloudflare Access 将提供几种不同的 MFA 方式：

-   生物识别技术（例如，Windows Hello、Apple Touch ID 和 Apple Face ID）
-   安全密钥（WebAuthn 和 FIDO2，以及用于基础设施访问 SSH 的 PIV）
-   通过验证器应用程序生成的基于时间的一次性密码

管理员将可以灵活定义用户必须如何进行身份验证以及频率。这不仅可以在全局级别配置（即为所有 Access 应用程序建立强制 MFA），还可以针对特定应用程序或策略进行更精细的控制。例如，您的组织可能决定允许对聊天应用使用较低保证级别的 MFA 方法，但要求使用安全密钥来访问源代码。

或者，您可以对承包商等第三方访问敏感资源强制执行强 MFA，否则他们可能会使用个人电子邮件或 LinkedIn 等社交身份。您还可以轻松地为原本不支持现代 MFA 方法的遗留应用程序添加这些方法，而无需修改一行代码。

最终用户将能够通过其**App Launcher**轻松注册 MFA 设备。
