# 托管式 OAuth 访问：一键让内部应用支持智能体

2026-04-14

- Eduardo Gomes

![](/images/posts/4db81d82c095.png)

Cloudflare 拥有数千个内部应用。有些是我们自己构建的，有些则是他人软件的自托管实例。它们的范围从几乎每个人都在使用的关键业务应用，到副业项目和原型。

所有这些应用都受到 Cloudflare Access 的保护。但当我们开始使用和构建智能体时——特别是用于编码之外的用途时——我们遇到了障碍。人们可以访问 Access 后面的应用，但他们的智能体却不能。

Access 位于内部应用的前端。您定义策略后，Access 会将未经身份验证的用户发送到登录页面，让他们选择身份验证方式。

Cloudflare Access 登录页面示例

这个流程对人类用户来说效果很好。但所有智能体看到的只是一个重定向到登录页面，而它们无法对此采取行动。

为智能体提供对内部应用数据的访问至关重要，以至于我们立即为自己内部使用实施了一个临时解决方案。我们修改了 OpenCode 的 `web fetch` 工具，使其针对特定域名触发 `cloudflared` CLI 来打开授权流程以获取 `JWT`（JSON Web Token）。通过在请求中附加此令牌，我们实现了对内部生态系统的安全、即时访问。

虽然这个解决方案是我们自身困境的临时答案，但今天我们正在淘汰这个变通方法，并为所有人解决这个问题。现在，每个 Access 应用都支持托管式 OAuth，并已进入公开测试阶段。只需为 Access 应用点击启用，支持 OAuth 2.0 的智能体就能轻松发现如何进行身份验证（RFC 9728），引导用户完成授权流程，并接收回授权令牌（与我们最初解决方案相同的 JWT）。

现在，这个流程对人类和智能体都能顺畅运行。Cloudflare Access 提供慷慨的免费层级。并且基于我们新推出的组织测试版，您很快将能够跨 Cloudflare 账户桥接身份提供商。

## 托管式 OAuth 的工作原理

一旦启用托管式 OAuth，Cloudflare Access 就充当授权服务器。它返回 `www-authenticate` 头部，告知未经授权的智能体去哪里查找如何获取授权令牌的信息。它们在 `https://<your-app-domain>/.well-known/oauth-authorization-server` 找到这些信息。有了这个指引，智能体只需遵循 OAuth 标准：

1.  智能体动态地将自身注册为客户端（此过程称为动态客户端注册——RFC 7591），
2.  智能体引导用户通过 PKCE（代码交换证明密钥）授权流程（RFC 7636）
3.  用户授权访问，从而授予智能体一个令牌，该令牌可用于代表用户进行身份验证的请求

如果这个授权流程看起来很熟悉，那是因为它正是模型上下文协议（MCP）所使用的。我们最初在我们的 MCP 服务器门户产品中构建了对它的支持，该产品代理和控制对许多 MCP 服务器的访问，以允许门户充当 OAuth 服务器。现在，我们将此功能引入所有 Access 应用，以便智能体不仅可以访问需要授权的 MCP 服务器，还可以访问网页、Web 应用和 REST API。

## 大规模升级您的内部应用以支持智能体

升级长尾的内部软件以支持智能体是一项艰巨的任务。原则上，为了支持智能体，每个内部和外部应用理想情况下都应具有可发现的 API、CLI、精心设计的 MCP 服务器，并采用许多新兴的智能体标准。

AI 的采用不能等待一切都被改造。大多数组织都有多年来积累的大量应用积压。许多内部“应用”在被智能体视为简单网站时运行良好。对于像内部维基这样的东西，您真正需要做的就是为智能体启用 Markdown，打开托管式 OAuth，智能体就具备了读取受保护内容所需的条件。

为了让基础功能在最广泛的内部应用集合中工作，我们使用托管式 OAuth。通过在您的遗留内部应用前放置 Access，您可以立即让它们支持智能体。无需更改代码，无需改造。取而代之的是即时兼容性。

## 这是用户的智能体。无需服务账户和令牌

智能体需要代表组织内的用户行事。我们看到的最大反模式之一是为其智能体和 MCP 服务器配置服务账户，并使用静态凭据进行身份验证。这些在简单用例和快速原型中有其用武之地，Cloudflare Access 为此目的支持服务令牌。

但是，当需要细粒度的访问控制和审计日志时，服务账户方法很快就显示出其局限性。我们认为，智能体执行的每个操作都必须能够轻松追溯到发起它的人类用户，并且智能体只能执行其人类操作员同样被授权执行的操作。服务账户和静态凭据成为追溯性丢失的节点。通过服务账户“洗白”其所有操作的智能体容易受到混淆代理人问题的影响，并导致审计日志看起来源自智能体本身。

为了安全和问责，智能体必须使用能够表达这种用户-智能体关系的安全原语。OAuth 是请求和委托第三方访问的行业标准协议。它为智能体提供了一种代表用户与您的 API 对话的方式，令牌范围限定在用户的身份上，以便访问控制正确应用，审计日志正确地将操作归因于最终用户。

## 标准制胜：智能体如何以及为何应该在其 Web 获取工具中采用 RFC 9728

RFC 9728 是使智能体能够发现在哪里以及如何进行身份验证的 OAuth 标准。它标准化了这些信息的存放位置和结构方式。该 RFC 于 2025 年 4 月正式发布，并迅速被模型上下文协议（MCP）采用，该协议现在要求 MCP 服务器和客户端都支持它。

但在 MCP 之外，智能体应该采用 RFC 9728 来实现一个更基本的用例：向受 OAuth 保护的网页发出请求，以及向普通的旧 REST API 发出请求。

大多数智能体都有一个用于向网页发出基本 HTTP 请求的工具。这通常被称为“web fetch”工具。它类似于在 JavaScript 中使用 `fetch()` API，通常对响应进行一些额外的后处理。它允许您将 URL 粘贴到您的智能体中，并让您的智能体去查找内容。

如今，大多数智能体的 web fetch 工具不会对 URL 返回的 `www-authenticate` 头部做任何处理。底层模型可能会选择内省响应头部并自行解决这个问题，但工具本身并不遵循 `www-authenticate`、查找 `/.well-known/oauth-authorization-server` 并充当 OAuth 流程中的客户端。但它可以，并且我们坚信它应该！智能体已经这样做来充当远程 MCP 客户端。

## 即将推出：跨多个Cloudflare账户共享统一身份提供商（IdP）

在Cloudflare内部，我们自己的应用部署在数十个不同的Cloudflare账户中，这些账户都属于一个组织——这是新引入的管理功能，允许管理员跨多个Cloudflare账户管理用户、配置和查看分析数据。我们与许多客户面临同样的挑战：每个Cloudflare账户必须单独配置IdP，因此Cloudflare Access需要使用我们的身份提供商。确保组织内配置的一致性至关重要——您不会希望某个Cloudflare账户意外地允许人们仅通过一次性PIN登录，而不是要求他们通过单点登录（SSO）进行身份验证。

为解决此问题，我们正在开发跨Cloudflare账户共享身份提供商的功能，让组织能够指定一个主IdP供其组织内的所有账户使用。

当组织内创建新的Cloudflare账户时，管理员只需点击一次即可配置到主IdP的桥接，从而使跨账户的Access应用程序都能受到统一身份提供商的保护。这消除了逐账户手动配置IdP的需求，对于拥有多个团队和个人各自运营账户的组织而言，这种手动流程难以扩展。
