# 推出 SWE-bench Verified

我们发布了一个经过人工验证的 SWE-bench 子集，能够更可靠地评估 AI 模型解决真实世界软件问题的能力。

更新于 2025 年 2 月 24 日

作为我们《准备框架》的一部分，OpenAI 开发了一系列指标来跟踪、评估和预测模型自主行动的能力。自主完成软件工程任务的能力是我们模型自主风险类别中“中等风险”水平的关键组成部分。由于软件工程任务的复杂性、准确评估生成代码的难度以及模拟真实开发场景的挑战，评估这些能力十分困难。因此，我们的准备方法还必须包括对评估本身进行仔细审查，以减少在重要风险类别中低估或高估性能的可能性。

最流行的软件工程评估套件之一是 SWE-bench（在新窗口中打开）¹——这是一个用于评估大语言模型（LLMs）解决来自 GitHub 的真实世界软件问题能力的基准测试。该基准测试要求向 Agent（智能体）提供一个代码仓库和问题描述，并挑战它们生成一个补丁来解决该问题所描述的问题。编码 Agent（智能体）在 SWE-bench 上取得了令人瞩目的进展，根据 SWE-bench 排行榜（在新窗口中打开）截至 2024 年 8 月 5 日的数据，得分最高的 Agent（智能体）在 SWE-bench 上获得了 20% 的分数，在 SWE-bench Lite 上获得了 43% 的分数。

我们的测试发现，一些 SWE-bench 任务可能很难甚至无法解决，导致 SWE-bench 系统性地低估了模型的自主软件工程能力。我们与 SWE-bench 的作者合作，在新发布的基准测试中解决了这些问题，该版本应能提供更准确的评估。

## SWE-bench 背景

SWE-bench 测试集中的每个样本都来自 GitHub 上 12 个开源 Python 仓库中已解决的 GitHub 问题。每个样本都有一个相关的拉取请求（PR），其中包含解决方案代码和用于验证代码正确性的单元测试。这些单元测试在 PR 中的解决方案代码添加之前会失败，但在添加之后会通过，因此被称为 FAIL_TO_PASS 测试。每个样本还有相关的 PASS_TO_PASS 测试，这些测试在 PR 合并前后都通过，用于检查 PR 是否破坏了代码库中现有的无关功能。

对于 SWE-bench 中的每个样本，Agent（智能体）会获得来自 GitHub 问题的原始文本（称为问题描述），并被授予对代码库的访问权限。基于这些信息，Agent（智能体）必须编辑代码库中的文件以解决问题。测试不会展示给 Agent（智能体）。

一个提议的编辑通过运行 FAIL_TO_PASS 和 PASS_TO_PASS 测试来评估。如果 FAIL_TO_PASS 测试通过，意味着该编辑解决了问题。如果 PASS_TO_PASS 测试通过，则说明该编辑没有意外破坏代码库中无关的部分。编辑必须通过这两组测试，才能完全解决原始的 GitHub 问题。

## 将 SWE-bench 适配为准备评估

鉴于 SWE-bench 对准备框架的潜在相关性，我们旨在寻找可以提高该基准测试稳健性和可靠性的方法。我们确定了三个主要改进领域²：

1.  用于评估解决方案正确性的单元测试通常过于具体，在某些情况下甚至与问题无关。这可能导致正确的解决方案被拒绝。
2.  许多样本的问题描述不够明确，导致对问题是什么以及如何解决存在歧义。
3.  有时很难为 Agent（智能体）可靠地设置 SWE-bench 开发环境，这无意中导致单元测试无论解决方案如何都会失败。在这种情况下，完全有效的解决方案可能被判定为不正确。

## SWE-bench Verified

为了解决这些问题，我们发起了一项人工标注活动，由专业软件开发人员对 SWE-bench 测试集中的每个样本进行筛选，以确保其具有适当范围的单元测试和明确的问题描述。

我们与 SWE-bench 的作者合作，发布了 SWE-bench Verified：这是原始 SWE-bench 测试集的一个子集，包含 500 个经我们人工标注员验证为无问题的样本。此版本取代了原始的 SWE-bench 和 SWE-bench Lite 测试集。此外，我们还发布了所有 SWE-bench 测试样本的人工标注。这些标注允许按难度对数据集进行切片。“简单”子集包含 196 个 <15 分钟的修复任务，而“困难”子集包含 45 个 >1 小时的任务。

我们还与 SWE-bench 作者合作，为 SWE-bench 开发了一个新的评估框架（在新窗口中打开），该框架使用容器化的 Docker 环境，使在 SWE-bench 上进行评估更加容易和可靠。

在 SWE-bench Verified 上，GPT‑4o 解决了 33.2% 的样本³，而性能最佳的开源框架 Agentless 将其之前在 SWE-bench 上 16% 的分数翻了一番。

## 我们的方法

我们与 93 名具有 Python 经验的软件开发人员合作，手动筛选 SWE-bench 样本的质量。我们标注了来自 SWE-bench 测试集的 1,699 个随机样本，以生成 SWE-bench Verified。以下分析基于这 1,699 个样本。

我们标注样本以捕获：

- 我们是否认为问题描述不够明确，因此在此条件下进行测试有失公平。
- 是否`FAIL_TO_PASS`单元测试过滤掉了有效的解决方案。

每个标注标准都有一个范围为[0, 1, 2, 3]的标签，严重程度递增。标签0和1为轻微问题；标签2和3为严重问题，表明样本在某些方面存在不足，应予以剔除。我们选择在四个有序类别上进行标注，而非单一的严重/非严重二元标签，以捕捉更细粒度的细节。

此外，我们通过让标注者估算开发人员在样本无问题的情况下决定并实现解决方案所需的时间，来评估每个样本的难度。最后，我们提供一个自由文本输入选项，用于标记样本中的任何其他重大问题（例如，如果`FAIL_TO_PASS`单元测试容易被绕过，可能导致无效解决方案被标记为正确）。

我们的工程师团队首先手工标注了50个样本，达到高度置信度，用于标注人员的入职测试。要参与标注工作，每位候选标注者必须通过我们的入职测试。我们在入职过程中为每位标注者提供详细反馈，以更好地培训他们完成任务。标注者不一定是与SWE-bench相关代码库的先前专家，但会给予时间熟悉他们处理的每个代码库。

为确保数据集的高质量，每个样本由不同的标注者标注三次。由于容易意外遗漏潜在问题，且问题本身可能具有歧义，我们保守地采用集成标注方法，取三位标注者中最高严重程度的标签。

### 数据集构建

为构建SWE-bench Verified，我们从原始测试集中过滤掉任何问题描述或`FAIL_TO_PASS`单元测试的集成标签严重程度为2或以上的样本。我们还过滤掉所有标记有其他重大问题的样本。根据我们的集成方法，这相当于过滤掉三位标注者中任何一位标记了问题的样本。这种方法在剔除样本时会导致更高的误报率，但有助于提高最终数据集样本质量的置信度。

### 按难度分层的性能

## 讨论与局限性

我们将SWE-bench作为我们准备框架中模型自主性风险类别中等风险级别的多项评估之一。通过评估追踪灾难性风险级别，依赖于确保我们能够信任评估结果，并校准对分数含义的理解。

我们的经验表明，我们应该：

投入精力深入理解我们的基准测试。尽管SWE-bench经过精心设计，但由于本文提到的若干问题，它仍低估了模型的能力。随着我们的系统日益接近通用人工智能（AGI），我们需要在更具挑战性的任务上对其进行评估。这也对基准测试的策划与验证提出了更高的专业水平和审慎要求，以确保其具备足够的挑战性和鲁棒性（例如CriticGPT这类探索AI如何辅助标注流程的工作，可能会对此有所助益）。

考虑生态系统中的进展。社区主导的Agent（智能体）脚手架开发进展表明，在评估风险时需要考虑模型可能获得的外部增强。通过观察SWE-bench排行榜上同一模型在最优与最差脚手架下的表现差异，我们可以看到，例如GPT-4在SWE-bench Lite上的性能，从使用早期基于RAG（检索增强生成）的脚手架时的2.7%，到使用CodeR时的28.3%，差异显著。因此，《准备框架》要求持续且按需频繁进行评估，以识别任何非平凡的能力变化；这包括训练前、训练中甚至训练后——因为模型可通过与外部系统集成得到增强。此外，策划评估是整个生态系统的工作，我们期待继续与研究人员合作，构建可信、高质量的评估。

认识到局限性。基于静态数据集的评估天生存在局限，SWE-bench也不例外。由于该基准测试由公开GitHub仓库的抓取数据构成，基于互联网文本预训练的大型基础模型很可能在任务上受到数据污染。此外，SWE-bench仅覆盖了模型自主性中等风险水平的狭窄分布，因此必须辅以其他评估。

我们信奉以实证和科学的方法来追踪和防范灾难性风险。构建并持续改进评估体系是这项工作的关键要素。仍有大量工作有待完成，我们热切期待社区贡献更多像SWE-bench这样有价值的基准测试。
