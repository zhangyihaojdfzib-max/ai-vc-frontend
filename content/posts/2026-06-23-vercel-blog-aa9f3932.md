Vercel 的 OIDC 签发方（oidc.vercel.com）现在支持自定义受众。部署可以请求带有特定受众声明的 OIDC Token，从而实现与第三方提供商的安全服务间认证。

### 链接到标题为什么需要自定义受众？

Vercel OIDC Token 签发时带有固定受众（https://vercel.com/{owner}）。虽然大多数云提供商不要求特定的受众值，但为每个提供商使用唯一受众是一项安全最佳实践。如果某个提供商被攻破，攻击者无法将 Token 重放给其他提供商——不匹配的 `aud` 声明会导致验证失败。这项新服务让您无需管理额外基础设施即可轻松生成提供商专属 Token。

### 链接到标题工作原理

当 Vercel 部署运行时，会收到一个由 Vercel 签名的 OIDC Token。新的交换服务接受此 Token，并返回一个使用相同密钥签名但更新了受众（aud）声明以指向下游服务的新 Token。

```
1import { awsCredentialsProvider } from '@vercel/oidc-aws-credentials-provider';2import { S3Client } from '@aws-sdk/s3-client';3const s3 = new S3Client({4  region: 'us-east-1', 5  credentials: awsCredentialsProvider({6    7    audience: 'sts.amazonaws.com',8    clientConfig: { region: 'us-east-1' },9  }),10});
```

交换后的 Token：

- 保留所有原始声明（项目、环境、所有者、过期时间）
- 将 `iss`（签发方）设置为 `https://oidc.vercel.com/{owner}`，范围限定为拥有该部署的团队
- 包含一个 `act`（执行者）声明，带有原始 Token 的受众和签发时间，提供可审计的委托链
- 将 `aud` 更新为请求的下游受众
- 将 `iat`（签发时间）更新为当前时间戳，反映新 Token 的创建时间

您可以选择传递 `jti`（JWT ID）来为交换后的 Token 分配唯一标识符。这对于跨服务审计和追踪 Token 使用非常有用——例如，在日志中将特定 Token 交换与下游 API 调用关联起来。

下游服务使用 `https://oidc.vercel.com/{owner}/.well-known/jwks` 提供的公钥验证交换后的 Token。

签名密钥和 Token 交换端点均复制到所有 Vercel 区域，确保无论部署在何处运行，都能实现低延迟的 Token 交换。
