# 在 Hugging Face Hub 上推出存储桶

Hugging Face 的模型和数据集仓库非常适合发布最终成果。但生产环境中的机器学习会持续生成大量中间文件（检查点、优化器状态、处理后的分片、日志、追踪记录等），这些文件经常变动、可能同时来自多个任务，并且很少需要版本控制。

**存储桶**正是为此而构建：它是一种可变的、类似 S3 的对象存储，你可以在 Hub 上浏览、通过 Python 脚本操作，或使用 `hf` CLI 进行管理。并且，由于它们由 **Xet** 提供支持，因此对于跨文件共享内容的机器学习产物尤其高效。

![](/images/posts/cd1ee3572a87.png)

## 我们为何构建存储桶

当你处理以下情况时，Git 很快就会让人觉得不是合适的抽象：

*   训练集群在整个运行过程中写入检查点和优化器状态
*   数据管道迭代处理原始数据集
*   Agent（智能体）存储追踪记录、记忆和共享知识图谱

所有这些情况下的存储需求是相同的：快速写入、需要时覆盖、同步目录、删除陈旧文件，并保持流程运转。

存储桶是 Hub 上的一个非版本化存储容器。它位于用户或组织的命名空间下，具有标准的 Hugging Face 权限，可以是私有或公开的，拥有一个可以在浏览器中打开的页面，并且可以通过类似 `hf://buckets/username/my-training-bucket` 这样的句柄以编程方式访问。

## Xet 为何重要

存储桶构建在 **Xet** 之上，这是 Hugging Face 基于分块的存储后端，这一点的重要性可能超出表面所见。

Xet 不是将文件视为单一的整体，而是将内容分解成块并进行跨文件去重。上传一个与原始数据集大部分相似的处理后数据集？许多块已经存在。存储模型大部分参数冻结的连续检查点？情况相同。存储桶会跳过已经存在的字节，这意味着更少的带宽占用、更快的传输速度和更高效的存储。

这非常适合机器学习工作负载。训练管道不断产生一系列相关的产物——原始和处理后的数据、连续的检查点、Agent（智能体）追踪记录和衍生摘要——而 Xet 的设计正是为了利用这种重叠。

对于企业客户，计费基于去重后的存储量，因此共享的块直接减少了计费占用空间。去重有助于提高速度和降低成本。

## 预热：将数据靠近计算资源

存储桶位于 Hub 上，这意味着默认是全球存储。但并非所有工作负载都能承受从数据所在位置拉取数据的开销。对于分布式训练和大规模管道，存储位置直接影响吞吐量。

预热功能让你可以将热数据移动到离你的计算资源运行的云提供商和区域更近的地方。无需每次读取时都跨区域传输数据，你只需声明需要数据的位置，存储桶会确保在你的任务开始时数据已经就位。这对于需要快速访问大型数据集或检查点的训练集群，以及管道的不同部分在不同云中运行的多区域设置尤其有用。

## 快速开始

你可以使用 `hf` CLI 在 2 分钟内创建并运行一个存储桶。首先是安装并登录：

```bash
curl -LsSf https://hf.co/cli/install.sh | bash
hf auth login

```

为你的项目创建一个存储桶：

```bash
hf buckets create my-training-bucket --private

```

假设你的训练任务正在本地将检查点写入 `./checkpoints`。将该目录同步到存储桶：

```bash
hf buckets sync ./checkpoints hf://buckets/username/my-training-bucket/checkpoints

```

对于大型传输，你可能想在移动任何内容之前查看将要发生什么。`--dry-run` 会打印计划而不执行任何操作：

```bash
hf buckets sync ./checkpoints hf://buckets/username/my-training-bucket/checkpoints --dry-run

```

你也可以将计划保存到文件中以供审查，稍后再应用：

```bash
hf buckets sync ./checkpoints hf://buckets/username/my-training-bucket/checkpoints --plan sync-plan.jsonl
hf buckets sync --apply sync-plan.jsonl

```

完成后，从 CLI 检查存储桶：

```bash
hf buckets list username/my-training-bucket -h

```

或者直接在 Hub 上浏览：`https://huggingface.co/buckets/username/my-training-bucket`。

这就是完整的流程。创建一个存储桶，将你的工作数据同步进去，需要时进行检查，并将值得发布的内容保存到版本化的仓库中。对于一次性操作，`hf buckets cp` 可以复制单个文件，`hf buckets remove` 可以清理陈旧的对象。

## 从 Python 使用存储桶

上述所有操作也可以通过 `huggingface_hub` 库（自 v1.5.0 起可用）在 Python 中完成。API 遵循相同的模式：创建、同步、检查。

```python
from huggingface_hub import create_bucket, list_bucket_tree, sync_bucket

create_bucket("my-training-bucket", private=True, exist_ok=True)

sync_bucket(
    "./checkpoints",
    "hf://buckets/username/my-training-bucket/checkpoints",
)

for item in list_bucket_tree(
    "username/my-training-bucket",
    prefix="checkpoints",
    recursive=True,
):
    print(item.path, item.size)

```

这使得将存储桶集成到训练脚本、数据管道或任何以编程方式管理产物的服务中变得简单直接。Python 客户端还支持批量上传、选择性下载、删除以及存储桶移动，以便在你需要更精细控制时使用。

存储桶支持在 JavaScript 中也可用（通过 `@huggingface/hub`，自 v2.10.5 起），因此你也可以将存储桶集成到 Node.js 服务和 Web 应用程序中。

## 文件系统集成

存储桶也可以通过 `HfFileSystem` 工作，这是 `huggingface_hub` 中兼容 `fsspec` 的文件系统。这意味着你可以使用标准的文件系统操作来列出、读取、写入和通配符匹配存储桶内容——任何支持 fsspec 的库都可以直接访问存储桶。

```python
from huggingface_hub import hffs


hffs.ls("buckets/username/my-training-bucket/checkpoints", detail=False)


hffs.glob("buckets/username/my-training-bucket/**/*.parquet")


with hffs.open("buckets/username/my-training-bucket/config.yaml", "r") as f:
    print(f.read())

```

由于 fsspec 是远程文件系统的标准 Python 接口，像 pandas、Polars 和 Dask 这样的库可以使用 `hf://` 路径读写存储桶，无需额外设置：

```python
import pandas as pd


df = pd.read_csv("hf://buckets/username/my-training-bucket/results.csv")


df.to_csv("hf://buckets/username/my-training-bucket/summary.csv")

```

这使得将存储桶插入现有的数据工作流变得容易，而无需改变代码读写文件的方式。

## 从存储桶到版本化仓库

存储桶是产物在动态变化期间存放的快速、可变的位置。一旦某些内容成为稳定的可交付成果，它通常属于一个版本化的模型或数据集仓库。

在我们的路线图中，我们计划支持存储桶和仓库之间的双向直接传输：将最终的检查点权重提升到模型仓库中，或在管道完成后将处理后的分片提交到数据集仓库。工作层和发布层保持分离，但融入一个连续的、原生于 Hub 的工作流中。

## 总结与资源

存储桶为 Hub 带来了一个缺失的存储层。它们为机器学习的可变、高吞吐量方面提供了一个原生于 Hub 的场所：检查点、处理后的数据、Agent（智能体）追踪记录、日志，以及在成为最终成果之前所有其他有用的东西。

由于构建于Xet之上，Buckets不仅比强行通过Git处理所有内容更易用，对于AI系统持续产生的各类相关产物而言也更为高效。这意味着更快的传输速度、更优的去重能力，并且在企业版方案中，计费将基于去重后的实际存储量受益。

若您已使用Hub平台，Buckets能让您将更多工作流程集中在一处管理。若您来自S3类存储体系，Buckets为您提供了熟悉的操作模式，同时更适配AI产物特性，并清晰指向最终在Hub平台发布的路径。
