作者：Robert Xu

最近在LangChain，我们一直在构建技能，以帮助Codex、Claude Code和Deep Agents CLI等编码智能体与我们的生态系统（即LangChain和LangSmith）协同工作。这并非我们独有的尝试——大多数（如果不是全部）公司都在探索如何创建技能并赋予编码智能体。构建这些技能的一个关键部分是确保它们确实有效。在这篇博客中，我们将分享在创建技能时如何评估它们的一些经验和最佳实践。

## 什么是技能？

技能是经过精心设计的指令、脚本和资源，用于提升智能体在特定领域的表现。重要的是，技能通过渐进式披露动态加载——智能体只有在任务相关时才会检索技能。这有助于智能体扩展其性能；历史上，给智能体过多的工具会导致其性能下降。

在实践中，技能可以被视为在智能体需要时动态加载的提示词。与任何提示词一样，它们可能以意想不到的方式影响智能体行为。因此，技能需要像测试LLM提示词一样进行测试。哪些技能能提升编码智能体的性能？哪些内容改动带来了最大的改进？

## 基础评估流程

我们测试技能的基本方法如下：

1.  定义你希望Claude Code成功完成的任务
2.  定义有助于完成这些任务的技能
3.  让Claude Code在没有技能的情况下执行任务
4.  让Claude Code在有技能的情况下执行任务
5.  比较性能并迭代优化你的技能

下面，我们将分享我们在创建自己的评估流程中总结出的一些最佳实践。

## 步骤1：设置一个干净的测试环境

技能通常与Claude Code等编码智能体或Deep Agents等框架一起使用。当你测试一个技能时，实际上是在测试这些强大的智能体是否能有效利用技能信息。你是在测试智能体的性能是否有所提升——因此，在实践中，你测试的是编码智能体本身。

编码智能体和框架拥有一个庞大的可操作空间。它们也对初始条件很敏感：Claude Code在开始工作前通常会探索你的目录，而它发现的内容将影响其采取的方法。这意味着在测试技能时，为使用你技能的智能体准备一个一致且干净的环境至关重要。这能确保你最大限度地提高测试的可复现性。

在我们的测试中，我们使用了一个轻量级的Docker脚手架来运行Claude Code。其他替代方案包括Harbor或你选择的沙箱环境。

```python
def run_claude_in_docker(
    test_dir: Path, prompt: str, timeout: int = 300, model: str = None
) -> subprocess.CompletedProcess:
    """Run Claude CLI in Docker. Returns CompletedProcess."""
    if not check_docker_available():
        raise RuntimeError("Docker not available")
    cmd = ["run-claude", str(test_dir), prompt, "--timeout", str(timeout)]
    if model:
        cmd.extend(["--model", model])
    try:
        return run_shell("docker.sh", *cmd, timeout=timeout + 30, check=False)
    except subprocess.TimeoutExpired:
        return subprocess.CompletedProcess(cmd, 124, "", f"Timeout after {timeout}s")

```

## 步骤2：定义任务

虽然很容易凭感觉来判断你的编码智能体是否因技能而有所提升，但性能在不同任务上确实会发生变化。明确定义的任务提供了一个系统化的基准，用以捕捉性能退化。我们从构建任务中学到的经验包括：

**创建约束性任务**

*   开放式的输出难以评分，尤其是对于编码智能体。如果我们要求Claude Code创建一个LangChain智能体来进行研究，它可能采取多种方法。评估Claude Code创建的智能体的质量很困难，而对智能体设计规定过细又会惩罚那些能产生有效研究的工作方案。
*   我们发现一个有用的策略是让Claude Code修复有错误的代码。这限制了其设计空间，并使验证正确性变得更容易：如果生成的智能体在预定义的测试中仍然产生错误行为，我们就可以放心地判定Claude的解决方案失败。此外，设计检查也不那么脆弱，因为Claude被引导使用我们现有的方法。

**为任务配备清晰的指标**

*   清晰的指标对于量化你的技能是否提升了编码智能体至关重要。我们跟踪的一些指标包括：
    *   **技能是否被调用？** 反之，在不相关时技能是否未被调用？
    *   **智能体是否完成了任务？** 一个任务可能涉及几个不同的步骤，跟踪已完成步骤的数量有助于区分“完全失败”和“几乎成功”。
    *   **Claude Code完成该任务用了多少轮对话？** 即使Claude Code没有技能也能完成任务，但包含技能可能会提高效率。
    *   **Claude Code实时完成该任务用了多长时间？** 在衡量效率时，并非每一轮对话都是等价的。
*   我们使用LangSmith评估来跟踪这些指标，以及它们每次运行的变化。这为我们提供了一个统一的视图来理解实验结果。

**不要过度追求难度**

*   当你评估一个技能时，你是通过一个编码智能体或框架来进行的。编码智能体本身已经是很好的问题解决者。如果你把任务设计得过于对抗性或复杂，你就有可能是在测试编码智能体的问题解决能力，而不是你的技能。
*   基于你在编码智能体中观察到的真实失败来设计任务是有用的。作为一个启发式方法，你应该致力于创建能导致失败的最直接的测试用例。例如，我们注意到Claude Code在将评估器有效上传到LangSmith方面存在困难。我们的测试用例要求Claude：“给定这个数据集，创建一个轨迹评估器并将其上传到LangSmith”。

**任务示例**

在我们的内部测试中，我们评估了Claude在基本LangChain和LangSmith任务上的表现。一个示例任务可能如下所示：

```json
// TASK
Create a trajectory dataset with 5 examples from the 5 most recent traces 
in our LangSmith project, plus an evaluator measuring tool call match percentage.

Output: trajectory_dataset.json and trajectory_evaluator.py 
(upload both as "bench-{run_id}" to LangSmith)

Run any code you write directly.

```

作为任务的一部分，我们期望它创建一个`trajectory_dataset.json`。我们可以使用如下指标对输出进行评分：

```python
# SAMPLE METRIC, checks the dataset Claude generates
def check_accuracy(runner: TestRunner):
    """Trajectories match ground truth.""" # Did Claude's output match ground truth
    dataset_file = runner.artifacts[0] # Claude's output: trajectory_dataset.json
    test_dir = Path(".")
    p, f = check_trajectory_accuracy(
        test_dir=test_dir,
        outputs=runner.context,
        filename=dataset_file,
        expected_filename="expected_dataset.json", # Ground Truth, expected_dataset.json
        data_dir=test_dir / "data", # folder in task where we store ground truth
    )
    for msg in p:
        runner.passed(msg)
    for msg in f:
        runner.failed(msg)

```

## 步骤3：定义技能

在创建和迭代技能时，主要考虑的是包含哪些内容，以及将这些内容放在何处。技能通常附带始终会加载到上下文中的 `AGENTS.md` 或 `CLAUDE.md` 文件。需要回答的一些问题是：

- 内容应该放在 `AGENTS.md`（预加载）中还是放在一个技能里？
- 内容应该放在一个大型技能中，还是拆分到多个较小的技能中？
- 内容能否被移除而不影响性能差异？

我们遵循的有用指导原则是：

**使技能模块化**

- 我们使用 XML 标签来划分技能内容的不同部分。这不仅为 Agent（智能体）增加了结构，还提供了一种便捷的方法来替换或移除部分内容以进行技能的 A/B 测试。
- 我们通常发现，对于较大的技能（300-500 行），小的格式或措辞更改影响有限。例如，正面（"做这个"）与负面（"不要做这个"）的指导，以及 Markdown 与 XML 标签，在记录的任务上表现相似。
- 在迭代时，我们通常对一个或多个部分进行更改以进行测试，而不是在更细粒度的级别上进行优化。

**利用 AGENTS.md 和 CLAUDE.md**

- 在实践中，技能并非总是能被可靠地调用。在一项创建 LangChain Agent（智能体）的任务中，Claude Code 从未调用过 "langchain agents" 技能。即使包含调用技能的提示词，也只能将调用率提高到 70%。
- 由于 `AGENTS.md` 和 `CLAUDE.md` 能被可靠地加载到上下文中，这是告诉 Claude 如何以及何时使用技能的好地方。这使我们能够实现一致的技能调用。
- `AGENTS.md` 和 `CLAUDE.md` 是包含关于如何一起使用多个技能的指导的好地方。在使用 LangSmith 技能的任务中，我们看到通过更少的轮次获得了更一致的通过率。

**平衡技能间的内容**

- 技能名称和描述对于 Claude Code 决定调用哪个技能至关重要。在我们运行约 20 个类似 LangGraph 技能的测试中，Claude Code 有时会调用错误的技能。当技能数量为 12 个时，Claude 能持续调用正确的技能。
- 将技能内容整合到少数几个技能中，意味着内容会更一致地进入上下文——然而，这也意味着 Claude Code 每次会加载更大的内容块，并且可能会看到它不需要的内容。找到正确的平衡点需要测试！

## 步骤 4：运行并比较性能

为了测试我们的技能，我们在任务上运行 Claude Code，使用不同的技能组合。一些测试用例包括：

- 控制案例：运行 Claude Code 时不使用任何技能
- 在任务上运行 Claude Code 并使用所有技能
- 运行 Claude Code，技能被整合到几个大型技能中
- 运行 Claude Code，技能被拆分到较小的技能中

我们使用 LangSmith 的 pytest 集成比较了每种场景下的任务完成率。包含技能几乎总是有益的，并且不同的内容拆分方式在不同的任务上表现更好。总体而言，使用技能的 Claude Code 在 82% 的情况下完成了任务，但没有技能时性能下降到 9%。然而，仅仅比较任务完成率是不够的，我们需要理解 Claude 失败的原因，以便对技能进行迭代。

理解 Claude Code 在 Docker 内部做了什么以及原因并非易事——我们需要对 Claude Code 的执行轨迹有可观测性。在我们的测试中，我们集成了 LangSmith 来捕获 Claude Code 采取的每一个动作。我们可以了解它读取了哪些文件、创建了哪些脚本，以及它调用（或未调用）了哪些技能。

![](/images/posts/3f57aad855ed.png)

关键的是，我们让 Claude Code 使用它自己的追踪技能来检查追踪记录并总结发生了什么。这使得迭代技能内容的速度快了很多。Claude Code 会将追踪记录发送到 LangSmith，拉取这些追踪记录，并总结问题供人工检查。然后，人工可以提出修复方案，重新运行测试，并在 LangSmith 的实验门户中查看性能如何变化。

![](/images/posts/e767d09bacfd.png)

在建立你自己的技能评估流程时，拥有良好的可观测性和良好的评估至关重要——特别是因为像 Claude Code 这样的 Agent（智能体）非常强大。
