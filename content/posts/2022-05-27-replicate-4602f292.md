## 回到正题

这是我第一次使用PyTorch。在此过程中，我遇到了一些问题和学习机会：看似运行了但未产生任何结果的代码（因为梯度流中断了）；学习向量化以及如何在张量上执行操作；以及重新思考如何设计代码，使其从过程式变为可微分的。

## 中断梯度流

由于我的Web工程背景，我最初解决这个问题的方法大概是：对于每个点，计算其到中心的距离，并对任何超出某个最小距离的点增加损失。在代码中，在掌握了张量的工作原理之后，可能看起来像这样：

```
for path_points in points_vars:
    loss += torch.sum(torch.tensor([torch.sqrt((p[0] - 112) ** 2 + (p[1] - 112) ** 2) for p in path_points]))
```

这段代码可以运行，但遗憾的是它对输出完全没有影响。当我们打印出添加到`loss`中的值时，看起来它应该有效。而且这是一个很大的数字，所以它应该完全主导来自CLIP的损失（用于使其看起来像提示词）。为了弄清楚发生了什么，我们尝试将其隔离，这样来自提示相似性的损失就不是一个因素了：

```
loss = 0
for path_points in points_vars:
    loss += torch.sum(torch.tensor([torch.sqrt((p[0] - 112) ** 2 + (p[1] - 112) ** 2) for p in path_points]))
```

这段代码失败并报错`element 0 of tensors does not require grad and does not have a grad_fn`。此时，Andreas耐心地向我解释说，其工作原理是PyTorch会跟踪所有参与计算`loss`的变量和操作，这样当需要最小化它时，它就能计算出如何改变变量以使总损失更小。为了实现这一点，我们需要通过我们对`loss`执行的每一个操作建立连接。通过创建一个新的张量，我们中断了这个链条，PyTorch无法回溯我们的操作来计算如何最小化损失。

调试这个问题最困难的部分之一是，失败是完全不透明的。Andreas告诉我，这是处理机器学习模型时常见的问题——你必须培养一种直觉，判断是什么导致了模型崩溃，这样你才能逐步检查正确的部分并找出修复方法。简化代码以隔离问题是一种有用的方法。

## 向量化

修复我们中断链条的一个选项是使用`torch.stack()`而不是创建一个新的张量：

```
for path_points in points_vars:
    loss += torch.sum(torch.stack([torch.sqrt((p[0] - 112) ** 2 + (p[1] - 112) ** 2) for p in path_points]))
```

这行得通，但我们实际采用的方法是移除列表推导式并使用向量化，这样我们一开始就不必创建新的张量。这种方法对于Andreas来说相当自然，他在可微分编程方面经验丰富。当处理张量时，我们可以同时对张量中的所有项应用一个操作。在我们上面的代码中，`path_points`是一个二维张量：

```
path_points
# tensor([[ 11.8556, 108.8475],
#         [ 10.2365, 107.1833],
#         [ 10.8576, 117.2298],
#         [  0.6850, 123.0041]], requires_grad=True)
```

我们可以对整个张量减去112，而不是对每个单独的元素：

```
path_points - 112
# tensor([[-100.1444,   -3.1525],
#         [-101.7635,   -4.8167],
#         [-101.1424,    5.2298],
#         [-111.3150,   11.0041]], grad_fn=<SubBackward0>)
```

像`** 2`或`torch.sqrt`这样的操作也以同样的方式工作。要沿着张量的一个维度求和（而不是对整个张量求和），我们可以使用`axis`：

```
torch.sqrt(torch.sum((path_points - 112) ** 2, axis=1))
# tensor([10038.8438,
#         10379.0088,
#         10257.1445,
#         12512.1221], grad_fn=<SumBackward1>)
```

综合起来，我们得到如下代码：

```
for path_points in points_vars:
    loss += torch.sum((torch.sqrt(torch.sum((path_points - 112) ** 2, axis=1))))
```

![路径被吸入图像中心的一个单一斑点](/images/posts/de77e86ec298.webp)

这样可行！运行后，点会迅速移动到画布中央并聚集在一起形成紧凑的团块。虽然实现了预期效果，但模型对"位于中心"的重视程度远高于"看起来像提示词"。我们来修复这个问题：

```
for path_points in points_vars:
    loss += torch.sum(torch.sqrt(torch.sum((path_points - 112) ** 2, axis=1))) * 0.001 / num_paths
```

除以 `num_paths` 意味着无论使用32条路径还是512条路径，权重都将保持一致——这使其成为相对权重，而非随着路径增加而不断增大的绝对权重。乘以0.001将总值降低几个数量级，使其与提示词相似性检查的损失值相当。我们并没有应用什么高深的科学方法，只是打印出每个来源的损失值并反复调整，直到感觉大致合适为止。

![路径在向中心移动时形成潜艇图像](/images/posts/5aa9934b022a.webp)

虽然这个方法可行，但我认为可以做得更好。模型在中心区域之外仅使用了相对较少的笔触来表现"这是一艘潜艇"，这意味着细节不够丰富。我们希望点更靠近中心而非完全位于中心，因此减去一个阈值。此外，我们并不介意点稍微偏离中心，但绝对不希望它们出现在远角，所以对距离进行平方处理。

```
for path_points in points_vars:
    loss += torch.sum((torch.sqrt(torch.sum((path_points - 112) ** 2, axis=1)) - 25) ** 2) * 0.001 / num_paths
```

这样可行！现在我们得到了精致、密集的输出：

![路径形成小潜艇图像](/images/posts/fd2e85640308.webp)

## 从程序化方法转向可微分方法

虽然输出效果不错，但图像有点太小了，让我们尝试提高惩罚开始生效的距离：

```
for path_points in points_vars:
    loss += torch.sum((torch.sqrt(torch.sum((path_points - 112) ** 2, axis=1)) - 75) ** 2) * 0.001 / num_paths
```

糟糕，我们做出了一个甜甜圈。

![路径形成环形](/images/posts/a2b572c63f66.webp)

出现甜甜圈形状是因为我们对模型将点放在中心区域的惩罚几乎与放在图像外边缘一样严重。要解决这个问题，我们需要停止对阈值内的点施加惩罚。

如果距离在圆内，则不施加惩罚。如果在圆外，则施加随距离增加而增大的惩罚。以下是相当程序化的伪代码：

```
if distance - threshold > 0:
    distance_loss = (distance - threshold) ** 2
else
    distance_loss = 0
```

这段代码很难直接向量化，因此不能直接套用。第一个问题是我们伪代码仅对单个点操作，而当前我们处理的是包含多个点的张量。第二个问题是 if/else 不可微分。为了确定实现方式，Andreas 鼓励我从数学函数的角度思考：f(x) = y。（在编程中通常有多种实现方式，使用 `torch.maximum` 的版本也能达到目的！但这样做的一部分目的是帮助我更多地以可微分编程的方式思考，所以我们没有采用那种方法。）

```
if x > 0:
    y = x ** 2
else:
    y = 0
```

我们仍然使用了 if 语句，因此尚未实现可微分，但足以勾勒出函数图形。据此，我们可以找出如何在 PyTorch 中用可微分操作来建模。

![显示上述不连续函数的图形](/images/posts/ff0fc39eb7c8.webp)

ReLU（修正线性单元函数）是一种能够模拟 x=0 处不连续性的函数。为了在 x>0 时获得二次曲线，我们可以在应用 ReLU 之前或之后对张量进行平方处理。有了这个工具，我们可以最终实现这个约束条件：

```
for path_points in points_vars:
    distances = torch.sqrt(torch.sum((path_points - 112) ** 2, axis=1))
    loss += torch.sum(torch.nn.functional.relu(distances - 75) ** 2) * 0.001 / num_paths
```
