# 优化 LoRA 推理的障碍

在部署 LoRA 时，通常需要热插拔（即动态加载和卸载不同的 LoRA）。LoRA 会改变基础模型架构。此外，不同的 LoRA 之间可能存在差异——每个 LoRA 可能具有不同的秩，并针对不同的层进行适配。为了应对 LoRA 的这些动态特性，我们必须采取必要措施，确保我们所应用的优化是稳健的。

例如，我们可以对加载了特定 LoRA 的模型应用 `torch.compile` 来获得推理延迟的加速。然而，一旦我们更换为另一个 LoRA（可能具有不同的配置），就会遇到重新编译的问题，从而导致推理速度下降。

另一种方法是将 LoRA 参数融合到基础模型参数中，运行编译，然后在加载新 LoRA 时解除融合。然而，由于可能存在架构层面的变化，这种方法在每次运行推理时同样会遇到重新编译的问题。

我们的优化方案尽可能考虑了上述实际情况。以下是优化方案的关键组成部分：

- Flash Attention 3 (FA3)
- `torch.compile`
- TorchAO 的 FP8 量化
- 支持热插拔

请注意，在上述组件中，FP8 量化是有损的，但通常能提供最显著的“速度-内存”权衡。尽管我们主要使用 NVIDIA GPU 测试了该方案，但它也应该能在 AMD GPU 上运行。

# 优化方案

在我们之前的博客文章（[文章 1](post 1) 和 [文章 2](post 2)）中，我们已经讨论了优化方案中前三个组件的好处。逐一应用它们只需几行代码：

```py
from diffusers import DiffusionPipeline, TorchAoConfig
from diffusers.quantizers import PipelineQuantizationConfig
from utils.fa3_processor import FlashFluxAttnProcessor3_0
import torch


pipe = DiffusionPipeline.from_pretrained(
    "black-forest-labs/FLUX.1-dev",
    torch_dtype=torch.bfloat16,
    quantization_config=PipelineQuantizationConfig(
        quant_mapping={"transformer": TorchAoConfig("float8dq_e4m3_row")}
    )
).to("cuda")


pipe.transformer.set_attn_processor(FlashFluxAttnProcessor3_0())


pipe.transformer.compile(fullgraph=True, mode="max-autotune")


pipe_kwargs = {
    "prompt": "A cat holding a sign that says hello world",
    "height": 1024,
    "width": 1024,
    "guidance_scale": 3.5,
    "num_inference_steps": 28,
    "max_sequence_length": 512,
}


image = pipe(**pipe_kwargs).images[0]

```

FA3 处理器来自[此处](here)。

当我们尝试在已编译的扩散 Transformer (`pipe.transformer`) 中热插拔 LoRA 而不触发重新编译时，问题就开始浮现。

通常，加载和卸载 LoRA 需要重新编译，这会抵消编译带来的任何速度优势。幸运的是，有一种方法可以避免重新编译。通过传递 `hotswap=True`，diffusers 将保持模型架构不变，仅交换 LoRA 适配器本身的权重，这就不需要重新编译。

```py
pipe.enable_lora_hotswap(target_rank=max_rank)
pipe.load_lora_weights(<lora-adapter-name1>)

pipe.transformer.compile(mode="max-autotune", fullgraph=True)
image = pipe(**pipe_kwargs).images[0]

pipe.load_lora_weights(<lora-adapter-name2>, hotswap=True)
image = pipe(**pipe_kwargs).images[0]

```

（提醒一下，第一次调用 `pipe` 会比较慢，因为 `torch.compile` 是即时编译器。但是，后续调用应该会快得多。）

这通常允许在不重新编译的情况下交换 LoRA，但也存在限制：

- 我们需要提前提供所有 LoRA 适配器中的最大秩。因此，如果我们有一个秩为 16 的适配器和另一个秩为 32 的适配器，我们需要传递 `max_rank=32`。
- 热插拔进来的 LoRA 适配器只能针对第一个 LoRA 所针对的相同层或其子集。
- 目前不支持针对文本编码器。

有关 Diffusers 中热插拔及其限制的更多信息，请访问文档的[热插拔部分](hotswapping section of the documentation)。

当我们查看不使用热插拔编译时的推理延迟时，这种工作流程的好处就变得显而易见。

关键要点：

- “常规 + 编译”选项相比“常规”选项提供了不错的加速，但它会引发重新编译问题，从而增加总执行时间。在我们的基准测试中，我们没有呈现编译时间。
- 当通过热插拔（也称为“优化”选项）消除重新编译问题时，我们获得了最高的加速。
- 在“优化”选项中，启用了 FP8 量化，这可能导致质量损失。即使不使用 FP8，我们也能获得不错的加速（“no_fp8”选项）。
- 出于演示目的，我们使用两个 LoRA 的池进行带编译的热插拔。完整代码请参考随附的[代码仓库](code repository)。

# 在消费级 GPU 上实现优化的 LoRA 推理

使用 Bfloat16 数据类型的 **Flux.1-Dev**（不带任何 LoRA）运行需要约 **33GB** 内存。根据 LoRA 模块的大小，并且在不使用任何优化的情况下，这个内存占用可能会进一步增加。许多像 RTX 4090 这样的消费级 GPU 只有 24GB 内存。在本节剩余部分，我们将以一台 RTX 4090 机器作为测试平台。

首先，为了能够端到端地执行 Flux.1-Dev，我们可以应用 CPU 卸载，将当前计算不需要的组件卸载到 CPU，以释放更多的加速器内存。这样做使我们能够在 RTX 4090 上以约 **22GB** 内存运行整个流程，耗时 **35.403 秒**。启用编译可以将延迟降低到 **31.205 秒**（加速 **1.12 倍**）。代码方面，只需几行：

```py
pipe = DiffusionPipeline.from_pretrained(
    "black-forest-labs/FLUX.1-dev", torch_dtype=torch.bfloat16,
)
pipe.enable_model_cpu_offload()




pipe.transformer.compile_repeated_blocks(fullgraph=True)
image = pipe(**pipe_kwargs).images[0]

```

请注意，这里我们没有应用 FP8 量化，因为它不支持与 CPU 卸载和编译同时使用（相关支持[议题](supportingissue thread)）。因此，仅对 Flux Transformer 应用 FP8 量化也不足以解决内存耗尽问题。在这种情况下，我们决定将其移除。

因此，为了利用FP8量化方案，我们需要找到一种无需CPU卸载的方法。对于Flux.1-Dev，如果我们额外对T5文本编码器应用量化，应该能够在24GB内存中加载并运行完整的流程。以下是对T5文本编码器进行量化（使用bitsandbytes的NF4量化）与不进行量化的结果对比。

正如我们在上图中可以注意到的，量化T5文本编码器不会造成太大的质量损失。将量化后的T5文本编码器与FP8量化的Flux Transformer结合，并使用`torch.compile`，我们得到了相对合理的结果——从32.27秒降至9.668秒（大幅提升约3.3倍），且没有明显的质量下降。

即使不量化T5文本编码器，也有可能用24GB显存生成图像，但这会使我们的生成流程稍微复杂一些。

现在，我们有一种方法可以在RTX 4090上使用FP8量化运行整个Flux.1-Dev流程。我们可以应用先前建立的优化方案，在同一硬件上优化LoRA推理。由于RTX 4090不支持FA3，我们将坚持使用以下优化方案，其中新加入了T5量化：

## 热插拔的技术细节

为了实现热插拔而不触发重新编译，必须克服两个障碍。首先，LoRA缩放因子必须从浮点数转换为torch张量，这相对容易实现。其次，LoRA权重的形状需要填充到所需的最大形状。这样，就可以替换权重中的数据，而无需重新分配整个属性。这就是为什么前面讨论的`max_rank`参数至关重要。由于我们用零填充值，结果保持不变，尽管计算速度会因填充大小而略有减慢。

由于没有添加新的LoRA属性，这也要求第一个LoRA之后的每个LoRA只能针对与第一个LoRA相同的层或其子集。因此，需要明智地选择加载顺序。如果LoRA针对的是不相交的层，则可以创建一个针对所有目标层并集的虚拟LoRA。

要查看此实现的具体细节，请访问PEFT中的`hotswap.py`文件。
