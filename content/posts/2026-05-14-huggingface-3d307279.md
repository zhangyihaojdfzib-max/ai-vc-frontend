# Granite Embedding Multilingual R2：开源 Apache 2.0 多语言嵌入模型，支持 32K 上下文——最佳 Sub-100M 检索质量

TL;DR：两个基于 ModernBERT 构建的全新 Apache 2.0 多语言嵌入模型——一个 97M 参数的紧凑型模型，在 MTEB 多语言检索（60.3）上击败了所有开源 sub-100M 多语言嵌入器；另一个 311M 全尺寸模型，在 MTEB 多语言检索上得分 65.2（500M 参数以下开源模型中排名第二），并支持 Matryoshka。两者均覆盖 200+ 语言，针对 52 种语言进行了调优，支持 32K Token 上下文（R1 的 64 倍），并增加了跨 9 种编程语言的代码检索。

![Granite Embedding Multilingual R2](/images/posts/dde2209be495.png)

本文内容：企业级设计·强大的 Sub-100M 多语言模型·R1 的改进·训练全尺寸 311M 模型·构建紧凑型 97M 多语言模型·基准测试结果·Matryoshka 嵌入·部署选项·框架集成指南·应使用哪个模型？·试用模型

多语言嵌入模型面临一个持续存在的矛盾：广泛的语言覆盖通常以模型规模为代价，而小模型通常牺牲语言覆盖。如果你需要跨语言工作——多语言语料的检索增强生成、跨语言搜索、国际团队的代码检索——你可能不得不在足够快的模型和足够好的模型之间做出选择。

Granite Embedding Multilingual R2 的发布显著缩小了这一差距。我们发布了两个新的多语言嵌入模型：

- **granite-embedding-311m-multilingual-r2**——一个 311M 参数的全尺寸模型，具有 768 维嵌入、Matryoshka 维度支持以及顶级的多语言检索质量。
- **granite-embedding-97m-multilingual-r2**——一个 97M 参数的紧凑型模型，具有 384 维嵌入，在其规模下提供强大的检索质量。

两个模型均支持 **200+ 种语言**，对 **52 种语言和编程代码** 的检索质量进行了增强，处理上下文长度高达 **32,768 个 Token**（比 R1 前代产品提升 64 倍），并在 **Apache 2.0** 许可下发布。它们可直接与 **sentence-transformers** 和 **transformers** 配合使用，无需任务特定指令，并可作为 **LangChain**、**LlamaIndex**、**Haystack** 和 **Milvus** 中的即插即用替代品，只需更改一行模型名称。对于当前使用仅英文默认设置的框架，这一行更改即可为社区中的每个用户提供 200+ 种语言的支持——无需更改 API、无需新增依赖、无需用户端修改代码。两个模型均附带 ONNX 和 OpenVINO 权重，用于 CPU 优化推理。

底层编码器在 200+ 种语言的文本上进行了预训练，为其中任何一种语言生成通用嵌入。以下 52 种语言接受了显式的检索对和跨语言训练，以获得更高质量的检索：

此外，这些模型还针对**编程代码**（Python、Go、Java、JavaScript、PHP、Ruby、SQL、C、C++）进行了训练，并支持跨语言代码检索。

## 企业级设计

两个嵌入模型均在 IBM 策划的数据集、公开可用数据以及内部生成或合成数据的混合数据上进行训练。训练中使用的公开网络衍生数据经过 IBM 开发的质量、去重和治理流程进行选择和过滤，旨在降低下游商业使用中的风险。我们有意避免使用 MS-MARCO 训练数据集以及具有明确非商业许可限制的数据集。这些模型使用 **GneissWeb** 进行预训练，这是一个源自公开网络内容并使用 IBM 数据准备和治理工具处理的数据集，同时结合了其他 IBM 策划和公开可用的来源。数据集经过 IBM 治理审查，以评估许可考量、所有权信号和个人数据风险。这些流程旨在促进负责任的使用和企业部署。

## 强大的 Sub-100M 多语言模型

本次发布的亮点是 **granite-embedding-97m-multilingual-r2**。在 9700 万参数下，它在跨 18 种语言的 **多语言 MTEB 检索** 上得分 **60.3**——这是我们在任何 1 亿参数以下的开放多语言嵌入模型中找到的最高检索分数。在该规模类别中次优的模型 multilingual-e5-small 在同一基准上得分为 50.9——在一个成熟的基准上存在 **+9.4 分的差距**。

其规模约为 311M 全尺寸模型的三分之一，但在多语言、代码和长文档基准上保留了大部分检索质量——相比其直接前代产品，在 MTEB 多语言检索上取得了 **+12.2 分的提升**，这得益于新的架构、更好的训练数据以及新颖的剪枝方法（下文详述）。全尺寸的 **granite-embedding-311m-multilingual-r2** 在同一基准上得分 **65.2**，相比其 R1 前代产品提升了 **+13.0 分**。

## R1 的改进

Granite Embedding Multilingual R1 模型基于 XLM-RoBERTa 编码器构建，上下文窗口为 512 Token。R2 代是完全重建的：

![Granite Embedding R2 架构](/images/posts/11c66b2cd22e.png)

**ModernBERT** 是一种最新的编码器架构，它利用过去五年 Transformer 研究的技术重新审视了原始的 BERT 设计。这一转变带来了几个实际好处：交替注意力长度减少了长序列的计算量（显著提高了长序列的吞吐量），旋转位置嵌入允许 32K 上下文窗口，而无需困扰旧架构的位置插值技巧，并且 Flash Attention 2.0 支持加快了现代 GPU 上的编码速度。

新的多语言分词器值得强调。我们没有重复使用 XLM-RoBERTa 的 25 万 Token 词汇表，而是采用了具有强大多语言和代码覆盖的现有分词器。311M 模型使用 Gemma 3 分词器（26.2 万 Token）；97M 模型从 GPT-OSS 分词器开始，并将其剪枝为紧凑的 18 万 Token 词汇表，在保持广泛多语言覆盖的同时减少了嵌入表的参数占用。分词器效率比人们意识到的更重要——一个 32K Token 的窗口听起来令人印象深刻，直到你的分词器用其中一半来编码一段泰语文本。

## 训练全尺寸 311M 模型

311M 模型是一个 22 层 ModernBERT 编码器，具有 26.2 万 Token 的多语言词汇表，通过多阶段流水线进行训练：

1. 知识蒸馏：模型同时从多个教师模型学习。教师模型为 Granite 3.3 Instruct 和 Mistral v0.2 Instruct 解码器模型，经过进一步微调用于文本嵌入，将检索特定知识迁移至 311M 编码器架构。
2. 对比微调：基于多语言检索对（涵盖 52 种语言和代码的查询与相关及难负例段落匹配）的标准对比训练，增强了模型区分相关与不相关结果的能力。
3. 模型合并：训练后，我们合并来自不同训练阶段和配置的检查点。这将针对不同目标（如多语言广度 vs. 英语深度）优化的模型优势整合到单一权重集合中，无需额外训练计算。
4. 套娃表示学习：模型采用套娃目标进行训练，使其 768 维嵌入可截断至 512、384、256 或 128 维，且质量损失极小（参见下方套娃嵌入）。

最终模型在 MTEB 多语言检索上得分为 65.2，整体平均得分为 56.3——较其 R1 前代版本平均提升 14.5 分。

## 构建紧凑型 97M 多语言模型

97M 模型通过词汇选择与知识蒸馏相结合的方式训练：

1. 词汇选择：将 262K Token 词汇表缩减为专门训练的 180K Token 词汇表，在保持广泛多语言覆盖的同时大幅缩小嵌入表规模。
2. 知识蒸馏：随后对精简模型进行微调，采用来自多个教师模型（包括基于 Granite 4.1 8B 和 Mistral Instruct 解码器的教师模型）的知识蒸馏及对比训练，以提升检索质量。

该方法从多个强教师模型中迁移检索特定知识，在减少模型参数的同时不牺牲语言覆盖。最终得到的高效紧凑模型在 MTEB 多语言检索上得分为 60.3（全尺寸模型为 65.2），而体积约为其三分之一。

## 基准测试结果

### 多语言检索

按模型大小排序的主要基准套件性能表现。分数为各基准内任务的平均值（越高越好）：

几个突出亮点：

- 97M R2 模型在平均分及大多数单项基准上击败了 multilingual-e5-base 和 gte-multilingual-base（约 300M 参数模型），尽管体积约小三倍。
- paraphrase-multilingual-MiniLM-L12-v2（广泛使用的框架默认模型）得分为 36.6，落后 97M R2 模型整整 23.7 分，而后者体积略小（97M vs. 110M 参数）且输出维度同为 384 维。
- LongEmbed 是 R1 到 R2 的最大提升点：97M 模型提升 31.3 分，311M 模型提升 34.0 分。这是 32K 上下文窗口的直接回报——R1 的 512 Token 限制意味着法律合同只能根据第一页进行判断。许多实际多语言工作负载涉及长文档（法律合同、技术手册、研究论文、多页报告），R1 根本无法完整处理。
- 代码检索显著提升：较 R1 提升 19.7（97M）和 15.3（311M），反映了新的代码训练集、更大的上下文窗口和更优的训练方法。
- 在更广泛的竞争领域中，harrier-oss-v1-270m 在 MTEB 多语言检索（66.4）和 RaR-b（32.9）上领先，而 jina-embeddings-v5-text-nano 在代码（71.2）和英语检索（58.8）上领先。311M Granite 模型平均分具有竞争力（56.3），并在 LongEmbed（71.7）上领先，同时提供远高于 jina-embeddings-v5-text-nano 的编码吞吐量（见下方速度表）。

### 速度与吞吐量

编码速度对生产工作负载至关重要，尤其是在索引数百万文档或需要低延迟查询编码时。我们在单个 NVIDIA H100 GPU 上使用 512 Token 块测量了延迟和吞吐量：

![速度 vs. 质量：97M 模型以 3 倍参数效率达到 300M 级质量](/images/posts/7c0e100a1dcd.png)

97M 模型每秒编码超过 2500 个文档——吞吐量与 multilingual-e5-small 相当——同时提供显著更高的检索质量。311M 模型约 1800 文档/秒，在检索质量上优于 jina-embeddings-v5-text-nano（65.2 vs. 63.3），编码速度快 5.5 倍以上（注：速度数据使用最新 Transformer 代码计算，该代码相比 4.57 版本存在速度回归——Jina 和 Granite 模型均受影响——详见技术报告）。harrier-oss-v1-270m 在此列出的竞争对手中提供了速度与检索分数的最佳组合。

## 套娃嵌入（311M）

311M 模型支持套娃表示学习，允许将嵌入从完整的 768 维优雅地截断至 512、384、256 或 128 维，质量逐步下降。这在存储、内存或相似度计算成本成为考量时非常有用——256 维嵌入的存储量仅为 768 维的三分之一，余弦相似度计算成本也相应降低。

以下是检索质量在不同嵌入维度下的表现：

![套娃嵌入：截断维度下的优雅质量衰减](/images/posts/a3c7b31461eb.png)

维度缩减带来的质量损失极小。从 768 维降至 256 维——存储和相似度计算成本降低 3 倍——MTEB 多语言检索仅下降 0.5 分（65.2 → 64.7），代码检索下降 0.5 分（63.9 → 63.4）。即使在 128 维（降低 6 倍）时，模型在 MTEB 多语言检索上仍得分为 63.7，代码检索为 62.3——保留了全维度性能的 97% 以上。实际应用中，这意味着可以大幅缩减索引大小和搜索延迟，而对结果质量影响极小。（注：上图结果在上下文长度为 1024（英语和多语言检索）和 8192（代码）时评估）。

作为对比，311M 模型截断至 384 维（与 97M 模型原生输出维度相同）后，在所有三个基准上仍优于 97M 模型。如果需要 384 维嵌入且能承受 311M 模型的编码成本，套娃截断是更优选择。

```python
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("ibm-granite/granite-embedding-311m-multilingual-r2")

full = model.encode(["示例文本"])
print(full.shape)  # 输出：(1, 768)

small = model.encode(["示例文本"], truncate_dim=384)
print(small.shape)  # 输出：(1, 384)
```

97M 模型不支持套娃——384 维已属紧凑。

### 跨语言检索

MTEB 检索中跨语言任务的平均性能。Belebele 衡量 122 种语言的跨语言段落匹配；MLQA 衡量 7 种语言的抽取式跨语言问答检索。

311M R2 模型在 Belebele 上较 R1 前代提升 4.3 分，在 MLQA 上提升 4.1 分，显示在更大规模下两个基准的跨语言迁移能力均有提升。

97M R2 模型在 Belebele 上得分较低（52.9 vs. 55.1，-2.2），而在 MLQA 上与 R1 前代持平（60.5）。Belebele 差距是剪枝和词汇缩减过程中固有的权衡——R2 模型的训练优先考虑了更广泛的 18 语言 MTEB 多语言检索集（较 R1 提升 12.2 分）和长文档检索（提升 31.3 分），而较小的词汇表（180K vs. 250K Token）和减少的层数（12 vs. 22）影响了狭窄的跨语言迁移任务。如果跨语言对之间的迁移是主要用例，全尺寸 311M 模型是更优选择。

## 部署选项

两个模型均提供多种生产部署路径。使用以下命令安装核心库：

```bash
pip install sentence-transformers
```

Sentence Transformers（推荐给大多数用户）：

```python
from sentence_transformers import SentenceTransformer, util

model = SentenceTransformer("ibm-granite/granite-embedding-97m-multilingual-r2")

queries = [
    "What is the tallest mountain in Japan?",          
    "Wer hat das Lied Achy Breaky Heart geschrieben?", 
    "ドイツの首都はどこですか？",                            
]

passages = [
    "富士山は、静岡県と山梨県にまたがる活火山で、標高3776.12 mで日本最高峰の独立峰である。",  
    "Achy Breaky Heart is a country song written by Don Von Tress.",                        
    "Berlin ist die Hauptstadt und ein Land der Bundesrepublik Deutschland.",                
]

q_emb = model.encode(queries)
p_emb = model.encode(passages)
print(util.cos_sim(q_emb, p_emb))


```

```python
from haystack.components.embedders import (
    SentenceTransformersDocumentEmbedder,
    SentenceTransformersTextEmbedder,
)
from haystack.components.retrievers.in_memory import InMemoryEmbeddingRetriever
from haystack.dataclasses import Document
from haystack.document_stores.in_memory import InMemoryDocumentStore

doc_embedder = SentenceTransformersDocumentEmbedder(
    model="ibm-granite/granite-embedding-97m-multilingual-r2"
)
query_embedder = SentenceTransformersTextEmbedder(
    model="ibm-granite/granite-embedding-97m-multilingual-r2"
)
doc_embedder.warm_up()
query_embedder.warm_up()


document_store = InMemoryDocumentStore()
result_docs = doc_embedder.run(documents=[
    Document(content="富士山は日本最高峰の独立峰です。"),
    Document(content="Mount Fuji is Japan's highest peak."),
    Document(content="Achy Breaky Heart is a country song written by Don Von Tress."),
    Document(content="Berlin ist die Hauptstadt und ein Land der Bundesrepublik Deutschland."),
])
document_store.write_documents(result_docs["documents"])


result_query = query_embedder.run(text="What is Japan's tallest mountain?")
retriever = InMemoryEmbeddingRetriever(document_store=document_store)
results = retriever.run(query_embedding=result_query["embedding"], top_k=2)
for doc in results["documents"]:
    print(f"{doc.score:.3f}  {doc.content}")



```

```python
from pymilvus import MilvusClient
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("ibm-granite/granite-embedding-97m-multilingual-r2")

)
```
