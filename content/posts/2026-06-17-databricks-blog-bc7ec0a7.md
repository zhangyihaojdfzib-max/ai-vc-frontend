- AI Agent（智能体）框架将模型推理转化为可靠行动。它提供了Agent完成现实任务所需的工具、记忆、执行环境和护栏。
- 框架设计直接影响Agent性能。强大的上下文管理、编排和验证与底层模型同等重要。
- 共享框架基础设施对于规模化部署企业级Agent至关重要。集中化治理、评估和可观测性有助于防止Agent无序扩张并保持系统可靠性。

AI Agent（智能体）框架是包裹在大语言模型（LLM）周围的软件基础设施，使其能够执行任务，而不仅仅是响应提示词。模型对问题进行推理并决定下一步行动。框架将其连接到执行这些行动所需的工具、系统、记忆和执行环境。

Agent（智能体）= 模型 + 框架

将模型视为产生推理和决策的“大脑”。框架则是围绕模型的一切，帮助Agent安全可靠地运行，包括：

- 工具：API、代码执行、搜索、数据库和业务应用
- 记忆：先前的上下文、用户偏好和工作流历史
- 工作空间：Agent可以访问的文件、数据、环境和系统
- 护栏：权限、策略、审批和监控

没有框架，模型可以回答问题，但无法可靠地运行代码、调用API、访问文件、记住先前的工作或独立完成多步骤工作流。

在本指南中，我们将介绍AI Agent框架的核心组件、框架为何影响Agent性能、生产级Agent系统如何构建，以及框架工程为何正成为一门独立学科。

## 为什么AI Agent既需要模型也需要框架

AI Agent依赖于两个互补的层次：一个负责推理的模型和一个负责行动的框架。

模型（无论是GPT-5.5、Claude、Llama还是其他LLM）读取上下文并决定下一步行动。框架通过将模型连接到工具、记忆和外部系统，将这些决策转化为行动。

现代Agent系统越来越多地围绕这种推理与执行的分离来构建。这两个层次共同使Agent能够在现实工作流中可靠地完成任务。

### 推理→行动→观察循环

许多AI Agent的核心是一个重复循环。理解这个循环有助于看清框架的作用。

1. 推理。模型读取其上下文中的所有内容，包括任务、相关记忆和先前结果，然后决定下一步采取什么行动。
2. 行动。框架通过运行工具、在沙箱中执行代码、调用API或写入存储来执行该行动。
3. 观察。框架捕获结果并将其作为新上下文反馈给模型。
4. 重复。模型利用该结果决定下一步做什么。循环持续直到任务完成。

这种模式通常被称为ReAct循环，即“推理与行动”的缩写，它构成了当今许多生产级Agent系统的基础。ReAct循环由Shunyu Yao等人在2022年的论文《ReAct: Synergizing Reasoning and Acting in Language Models》中提出。

考虑一个负责修复Bug的编码Agent。模型提出代码更改。框架在隔离的沙箱中运行代码，捕获测试结果并将其返回给模型。如果测试失败，模型会推理问题所在并再次尝试。框架管理与底层系统的交互，而模型则专注于解决任务。

## Agent、模型和框架：有什么区别？

“Agent”、“模型”和“框架”经常被互换使用，但它们指的是系统的不同部分。厘清这些区别有助于团队理解他们实际在构建、调试或改进什么。

## 每个生产级框架所需的八个构建模块

大多数可运行的框架都由相同的基础组件构建而成，每个组件旨在解决原始模型的不同局限性。

### 系统提示词

系统提示词是每次运行时给予模型的一套固定指令，告诉模型它是什么、要完成什么目标以及必须遵守哪些规则。系统提示词在任何用户输入到达之前就塑造了Agent的行为、个性和护栏。编写不当的提示词是导致行为不一致或不可预测的最常见原因之一。

### 工具与工具执行

工具是模型可以调用的预构建函数，用于与外部系统交互，例如搜索网页、查询数据库、发送电子邮件、运行代码或调用API。模型决定使用哪个工具以及何时使用。框架则实际运行该工具并将结果返回给模型。

开发者正逐渐远离大量定义狭窄的工具集合。相反，他们赋予Agent更通用的能力：编写和执行代码的能力。这使得模型能够动态构建工作流，而不是依赖一组固定的预定义操作。

### 沙箱与执行环境

沙箱是一个隔离的工作空间，Agent可以在其中运行代码或执行操作，而不会影响环境之外的任何内容。这一点很重要，因为直接在真实系统上运行Agent生成的代码存在风险。

通过隔离环境，沙箱让Agent能够安全地进行实验，并为团队提供一个可控的工作空间，当出现问题时可以干净地监控、重置或关闭。它们还使得大规模并行运行多个Agent成为可能。

### 文件系统与持久化存储

文件系统为Agent提供了读写文件的位置，例如代码、笔记、计划和会话之间持续存在的中间工作成果。

持久化存储允许Agent在长时间运行的任务中积累进展，并通过共享的文件工作空间（而不仅仅是聊天消息）与人类或其他Agent协作。

### 记忆与上下文管理

基础模型不会保留超出其当前上下文窗口的记忆。框架管理任务内部以及跨会话的记忆。随着对话变长，框架决定哪些内容保持活跃，哪些内容被总结，这个过程称为上下文压缩。

在实践中，这意味着修剪对话中较旧的部分，以免模型因上下文增长而不堪重负。跨会话时，框架存储和检索相关历史记录。这使得Agent能够在了解已完成工作的基础上恢复任务。

### 反馈循环与自我验证

好的框架不仅让模型行动——它们还会检查工作。每次行动后，框架可以运行测试、检查结果或提示模型在继续之前审查自己的输出。

这些反馈循环使得Agent能够通过反复尝试工作、检查结果、捕获错误并自动纠正方向，从而可靠地处理长时间或复杂的任务。

### 护栏与人在回路控制

护栏是内置于框架中的规则，用于阻止不安全或未经批准的操作。例如，在Agent删除文件、发送客户消息或进行购买之前需要获得人工批准。

一种常见的护栏类型是人在回路控制，即某些操作在执行前需要人工审查或批准。在企业环境中，这些审批检查点通常是强制性的。

### 可观测性与日志记录

可观测性意味着能够通过日志、追踪和仪表盘看到Agent做了什么、为什么做出每个决策以及哪里出了问题。对于开发者来说，可观测性有助于诊断和调试Agent行为。对于企业团队来说，这通常是合规要求。受监管的行业需要审计追踪，以精确显示Agent做了什么以及由谁授权。

在规模化部署中，可观测性还为评估基础设施提供支持——这些系统持续衡量Agent在数千次运行中（而不仅仅是演示中）是否表现正确。

## 同一模型，更优框架，更佳结果

随着模型在原始能力上趋于一致，框架对性能的影响日益显著。内存、工具编排、反馈循环和安全护栏共同决定了可靠性。在公开基准测试中，同一模型的表现可能因框架构建方式的不同而大幅波动。对于许多工作流密集型任务而言，围绕中等模型构建的强框架，其表现可能超越围绕更强模型构建的弱框架。

这种影响是可量化的。当 Databricks 将 GPT-5.5 与 OfficeQA Pro Agent 框架（专为复杂、多步骤的企业文档任务设计）结合使用时，其得分从 GPT-5.4 的 36.10% 提升至 52.63%，错误率几乎减半。模型有所改进，但正是框架将这种改进转化为可靠的生产性能。AI Agent 评估框架帮助团队精确衡量这一点：框架设计是否将模型能力转化为一致、可信的结果。

## 提示词工程、上下文工程与框架工程

框架工程是开发者与 AI 系统协作方式更广泛转变的最新阶段。随着模型能力增强，关注点逐渐向外转移：从编写更好的提示词，到控制模型所见信息，再到围绕模型设计整个系统。

提示词工程和上下文工程都属于框架工程的范畴。框架是围绕模型的系统；提示词和上下文是该系统的组成部分。

## 生产环境中 AI Agent 框架的常见故障模式

框架功能强大，但容易出错。大多数运行中的 Agent 故障源于框架，而非模型本身。以下是团队在实际系统中遇到的一些最常见问题：

- **上下文腐化**。随着对话历史增长，模型的推理质量下降。如果没有修剪或总结旧上下文的策略，长时间运行的任务性能往往会崩溃。
- **工具过载**。一次性给模型过多工具会增加混乱，并在任何工作开始前就减慢决策速度。
- **脆弱的工具连接**。工具描述或调用方式的微小变化可能导致模型错误使用它们，从而造成难以诊断的静默故障。
- **延迟**。涉及多次工具调用的多步骤 Agent 可能需要 10 秒或更长时间才能响应，造成令人沮丧的用户体验。
- **无关检索**。当框架从记忆或搜索系统中拉取错误信息时，模型可能自信地生成错误答案。
- **弱验证**。缺乏测试循环或自我检查，Agent 可能过早停止或在未完成工作时宣布成功。
- **缺失安全护栏**。Agent 在没有充分监督或人工批准的情况下采取不可逆的操作——发送消息、删除数据或进行购买。

## AI 框架如何融入企业 AI 战略

大多数公司并非构建单一的 AI Agent，而是在不同团队、工作流和底层模型之间构建数十个 Agent。如果没有一致的框架设计方法，很快就会导致 Agent 蔓延：互不关联的 Agent 使得任何单一团队都无法可靠地治理、评估或改进它们。

### Agent 蔓延造成企业控制难题

随着 Agent 更接近生产工作流，团队需要集中控制 Agent 可以访问的内容、可以采取的行动以及如何评估其输出。他们还需要可审计性、可观测性，以及在不重建周围系统的情况下交换底层模型的灵活性。

## 随着模型改进，框架将如何变化

随着 AI 模型在规划、多步骤推理和错误纠正方面变得更好，目前由框架处理的部分工作可能会更接近模型本身。模型将更擅长保持任务专注、验证自身工作以及从错误中恢复，而无需过多的外部协调。

框架工程不太可能消失。执行环境、工具编排、安全护栏、可观测性和反馈循环仍然决定模型能否在实际系统中可靠运行。更好的工具、更清洁的工作空间和更强的安全保障使每个模型都更有用，无论模型自身能力有多强。

两个新兴概念有助于说明该领域可能的发展方向：

- **一次性框架**。为单个工作流创建的轻量级、任务特定框架，在使用后即被丢弃，而非作为长期运行的基础设施运行。随着执行环境的配置变得更快、更便宜，这种方法正变得更加实用。
- **自然语言 Agent 框架（NLAH）**。工程师无需通过代码配置框架，而是使用自然语言指令描述 Agent 应如何行为。共享运行时解释并执行这些指令，降低了跨项目构建、修改和重用框架的门槛。

模型包含智能。框架将智能转化为可靠的工作。只要这一点不变，框架设计就至关重要。

## 常见问题

**AI Agent 和 AI 框架有什么区别？**  
AI Agent 是由模型和框架共同组成的完整工作系统。框架是提供工具、内存、安全护栏和工作流控制的执行层。你与 Agent 交互，而框架使其工作。

**框架工程和提示词工程有什么区别？**  
提示词工程专注于为模型构建更好的输入。框架工程专注于设计围绕模型的完整系统，包括工具、执行环境、安全控制和反馈循环。提示词工程是更大框架架构的一部分。

**AI Agent 框架的核心组件有哪些？**  
大多数生产框架包括系统提示词、工具、沙箱、内存管理、反馈循环、安全护栏和可观测性。每个组件都解决了原始模型的不同局限性。

**为什么框架比模型更重要？**  
随着 AI 模型能力增强，框架质量日益决定实际性能。强框架通过更好的内存管理、工具编排、验证和安全护栏提高可靠性。在许多实际系统中，如果基础设施不稳定，仅升级模型带来的收益较小。

**企业如何大规模治理 AI Agent 框架？**  
有效的企业治理需要集中控制数据访问、评估系统、可审计性、成本控制，并支持多种底层模型。像 Databricks Agent Bricks 这样的平台通过由 Unity Catalog 和 MLflow 驱动的共享治理、可观测性和评估基础设施来应对这些挑战。

## 从 AI 模型到 AI 系统

框架通过提供工具、内存、安全护栏和反馈循环，将语言模型转变为可工作的 Agent，使可靠工作成为可能。强框架让普通模型变得有用，弱框架则浪费最好的模型。随着 AI Agent 进入生产环境，框架设计正成为大部分工程工作——以及大部分价值——的所在之处。
