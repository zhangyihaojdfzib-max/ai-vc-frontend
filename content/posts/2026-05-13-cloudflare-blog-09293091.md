# Browser Run：现已运行于 Cloudflare Containers，速度更快、可扩展性更强

2026-05-13

- Ruskin Constant
- Rui Figueira
- Sofia Cardita

![](/images/posts/17608815b45d.png)

我们通过在 Cloudflare Containers 上重建 Browser Run，实现了更高的使用限制、更快的性能和更好的可靠性。

现在，您可以通过 Workers 绑定每分钟启动 60 个浏览器，并同时运行多达 120 个——是之前限制的 4 倍。此外，Quick Action 响应时间下降了超过 50%。您无需做任何更改：这些改进现已生效。在此基础上，我们比以往更快地推出修复和新功能。请继续阅读，了解我们是如何做到的，并查看相关数据。

### 提醒一下：什么是 Browser Run？

Browser Run 使开发者能够以编程方式控制并操作运行在 Cloudflare 全球网络上的无头浏览器实例。这对于 Web 应用的端到端测试、安全调查可疑 URL、利用浏览器轻松渲染 PDF 文档，以及其他快速操作（如截取屏幕截图和提取内容）非常有用。最近，它已成为 AI Agent 与 Web 交互的关键推动力。我们正在将 Browser Run 打造成负责任地大规模安全使用自动化浏览器的首选平台。

### 超越我们的双层床

在采用 Cloudflare Containers 之前，我们与 Browser Isolation (BISO) 共享基础设施。虽然技术上相似，但 BISO 更大的容器镜像拖慢了启动和开发速度。关键的是，BISO 浏览器缺乏最佳的全球分布，影响了弹性和延迟。此外，典型 BISO 用户长时间、稳定的会话与 Browser Run 短促、突发的使用模式相冲突，造成了扩展瓶颈和可用性延迟。

幸运的是，经过大量内部开发，Cloudflare 去年发布了支持 Durable Object (DO) 的 Containers 公开测试版，这意味着我们准备好进行尝试性采用，这最终使两个产品平台都受益。像大多数成功的产品平台一样，我们致力于在可行的情况下尽可能在自己的平台上构建，以便在任何外部客户遇到问题之前，我们自己就能感受并解决痛点。

### 迁移：Containers

我们开始了逐步迁移，在传入请求路径中插入一个 Worker，为少量用户提供一些由 Container 驱动的浏览器，同时保留 BISO 的浏览器。开发期间的这种双重支持至关重要：它使我们能够比较性能、隔离实现错误，并最终对 Container 驱动方法的好处建立信心。

在逐步扩大采用范围时，我们首先将 Container 浏览器用于所有 Quick Action 端点，然后用于通过免费账户的 Workers 浏览器绑定建立的连接，接着是即用即付账户，以验证稳定性，最后才推广到所有剩余的合同客户，确保过渡过程无需客户采取任何操作或重新部署现有 worker。

### 挑战：性能和规模瓶颈

然而，在我们这边，我们面临一系列全新的挑战：需要熟悉一个新颖、不稳定且处于早期阶段的 Containers 平台接口，该接口文档匮乏、可观测性不足，且处于重叠时区的同事也很少。不过，我们作为 Customer Zero 向自己团队提供的反馈，意味着我们可以形成紧密的反馈循环，从而带来实质性的升级，这也惠及我们的外部客户。尽管如此，最初需要克服的摩擦很多，其中大部分对于处于积极开发中的封闭测试版来说是可以预料的。其他需要克服的障碍则是新技术环境固有的。

例如，一旦我们的浏览器可以全球运行，我们的架构就必须适应。支持 DO 的 Containers 会尽可能在靠近传入请求的位置创建一个 Durable Object，但连接的 Container 可能会在世界另一端启动。这对于像"启动我的应用"这样的一次性消息来说没问题，但当你在它们之间建立 WebSocket 并为一次截图请求交换数十条消息时，那些跨越全球的额外毫秒数就开始累积了。

我们的解决方案？创建预热的、由 DO 支持的浏览器容器的区域池，以限制 DO 和容器之间的最大距离（从而限制最大延迟）。当请求到来时，我们在该区域内选择最接近用户的 DO-容器对。这使两跳的延迟都保持在较低水平：用户到 DO，以及 DO 到容器。这为我们的整体架构增加了一些活动部件，但我们认为只要能够观测到每个浏览器的全局状态，以便根据变化的需求分配和重新分配容量，这就是值得的。这在一定程度是 Workers KV 的完美用例。

自去年初以来，对我们无头浏览器的需求一直在增长。简而言之，AI Agent 构建者发现了 Browser Run，并迅速带来了超出我们现有容量的请求量。我们很快就达到了通过可扩展方法调整池容量以满足这种新需求的速度极限。KV 约 30 秒的最终一致性成为了我们关键请求路径上的瓶颈。您可能检查 KV，看到一个容器标记为"可用"，但当您路由到它时（30 秒后），它已经被占用了。这种延迟造成了竞态条件和浏览器的过度分配，严重限制了我们扩展以应对需求高峰的速度。

### 从 KV 迁移到 D1 + Queues

我们之前将每个容器的状态存储在 KV 中。这意味着由于缓存 TTL，我们可能一直获取到一分钟前的状态（最近 KV 将最小缓存 TTL 改为 30 秒，但即便如此，这个值仍然太高）。

我们决定将容器状态迁移到 D1 实例中。D1 的事务性特性非常适合这里。一旦我们将浏览器分配给用户，它就专属于该用户。浏览器不是共享资源。SQLite 事务确保了原子性分配，并防止了两个请求可能同时认领同一个浏览器的竞态条件。

以下是我们的浏览器获取查询的简化版本：

```SQL
WITH candidate_pool AS (
    -- 基于延迟和其他规则选择候选池的逻辑
)
UPDATE containers
SET status = 'picked'
WHERE sessionId IN (
    SELECT sessionId
    FROM candidate_pool
    ORDER BY RANDOM()
    LIMIT ?5
)
RETURNING data
```

我们在每个位置维护 D1 分片，考虑到我们可能运行数千个容器，并且每个容器需要每 5 秒更新一次状态，我们不断遇到一个问题：数据库过载。例如，如果每次写入需要 1ms，我们最多只能写入 1,000 次，按每次写入一行计算，这意味着在数据库过载之前我们只能拥有 5,000 个容器。

然而，如果我们将这些写入进行批处理，我们可以获得更高的数值，因为批量写入的时间并不比单个写入长多少，因此我们可以将吞吐量提高几个数量级。在我们的案例中，我们使用 100 行的批次，这意味着现在每个位置最多可以更新 500,000 个容器。这种余量意味着容量规划不再是瓶颈。

目前，我们批量写入的 P95 为 0.1ms！

为了实现批量写入，我们使用 Queues：每 5 秒，每个容器计算自己的状态并将其添加到其位置队列中。然后我们配置一个 worker 消费者，设置批次大小为 100，批次超时为 1 秒：

```JSON
{
    ...
    "queues": {
        "consumers": [
            {
                "queue": "production-core-containers-queue-weur",
                "max_batch_size": 100,
                "max_batch_timeout": 1,
                "max_retries": 1,
            },
            ...
        ]
        ...
    }
}
```

通过这种配置，我们实现了远低于2秒的可接受延迟时间。不过，队列积压仍可能导致状态过期。当这种情况发生时，每个区域会回退到指定的备用区域，直到主队列赶上进度。

### 快速操作的额外优势

借助专用基础设施，我们现在可以升级浏览器容器镜像，而不会对BISO等其他产品产生不良副作用或造成臃肿。这为优化截图和内容提取等快速操作打开了大门。此前，我们的工作进程需要与远程浏览器建立WebSocket连接，并逐一发送指令：打开页面、导航到URL、等待页面加载、截取屏幕截图。每个步骤必须完成后才能开始下一步。

然而，现在我们通过单个HTTP请求将所有参数直接发送到容器，整个流程在内部执行，无需工作进程与浏览器之间来回通信。

### 成果：性能大幅提升与限制突破

我们看到平均快速操作响应时间显著下降，用户能够更快地从浏览器会话中获取所需内容：等待浏览器就绪的时间更短，处理DevTools协议消息的速度也更快。

在新规模下克服实时状态管理的挑战，意味着我们可以在实验环境中投入更多时间，发现并开发新功能，例如我们最近推出的/crawl端点。

### 更好的浏览器灵活性

通过摆脱共享的浏览器隔离基础设施，我们还获得了另一个重要优势：更快的升级速度。

当我们的浏览器运行在共享产品基础设施上时，升级Chrome需要协调多个团队和产品，每个团队都有自己的路线图和优先级。但现在，由于我们运行自己的容器镜像，我们可以以更快的节奏进行升级。例如，备受期待的WebGL功能现已可用于基于浏览器的渲染，同时WebMCP（面向网络的模型上下文协议）也得以实现，支持新的Agent（智能体）交互模式。这些功能的实现都得益于我们能够控制浏览器版本和标志，而不会对Cloudflare其他产品产生不良副作用。

简而言之，我们才刚刚开始释放大规模浏览器的潜力，尤其是在Agent（智能体）开发方面。希望您也能加入其中——请查看我们的文档。
