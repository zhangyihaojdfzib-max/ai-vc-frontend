# FLUX 微调现已提速

-   bfirsh

![由 levelsio/analog-film 生成](/images/posts/1c393f7fefdb.webp)

![由 fofr/flux-mona-lisa 生成](/images/posts/da3333fe82fd.webp)

您可以在 Replicate 上使用自己的数据对 FLUX 进行微调。我们已大幅提升了在 Replicate 上运行微调任务的速度，并且这些优化是开源的。

这建立在我们上个月的工作基础之上，当时我们大幅提升了 FLUX 基础模型的速度。

现在运行微调任务的速度与基础模型相同：

-   FLUX.1 [schnell] 在 512x512 分辨率和 4 步采样下：0.6 秒 (P50)
-   FLUX.1 [dev] 在 1024x1024 分辨率和 28 步采样下：2.8 秒 (P50)

此外，首次运行微调任务时，加载模型需要一点时间，通常约为 2.5 秒。一旦模型加载完成，我们将尝试将您的请求路由到已加载该模型的实例，其运行速度将与基础模型一样快。

要启用所有优化，请在您的预测请求中传递 `go_fast=true`。如果您省略 `go_fast` 选项，速度仍将比之前快一倍，且不会影响输出质量。

所有模型（包括现有和未来的）都将自动获得这些优化。

### 从其他地方加载 LoRA

如果您不使用 Replicate 来微调模型，我们还新增了从 Hugging Face、Civitai 和任意 HTTP URL 加载 LoRA 的支持。

只需将 Hugging Face、Civitai 或 HTTP URL 传递给这些新的 FLUX LoRA 版本中的 `lora_weights` 输入即可：

-   black-forest-labs/flux-dev-lora
-   black-forest-labs/flux-schnell-lora

### 我们是如何做到的？

Replicate 上的大多数模型由我们的社区贡献，但我们与 Black Forest Labs 合作维护 FLUX 模型。

我们以 Alex Redden 的 flux-fp8-api 为起点优化了基础模型，使用 `torch.compile` 进行优化，并在 Torch 的 nightly 版本中使用了快速的 CuDNN 注意力内核。更多细节，请查看我们关于优化基础模型的博客文章。

Replicate 上的微调任务以 LoRA 形式呈现。我们将 LoRA 量化为 fp8，然后将权重合并到基础模型中。当 `go_fast=true` 时，我们还会自动将 `lora_scale` 输入增加 1.5 倍，因为我们发现这样能产生更好的输出。您可能也想尝试调整这个参数。

flux-fp8-api 中的量化会略微改变输出，但我们发现这对质量影响很小。

我们希望向您公开我们优化模型的方式。众所周知，比较不同模型和提供商之间的输出非常困难，而且通常不清楚提供商是否采取了影响模型质量的措施。我们只会告诉您我们是如何做的，并允许您禁用任何优化。

### 开源也应该很快

开源模型通常开箱即用时速度较慢。模型提供商随后会优化这些模型以提高速度，并通过专有 API 发布，却没有将改进贡献回社区。

我们希望改变这一点。我们认为开源也应该很快。

我们将对 FLUX 所做的所有改进都开源。请阅读我们关于使基础模型变快的博客文章以了解更多。

### 未来会更快

这使得运行微调模型的速度更快，但要让训练过程变快，仍有工作要做。接下来将会有一些重大的改进。

此外，不断有新技术涌现来加速模型，通过与社区合作，您可以确保这些技术能尽快在 Replicate 上实现。敬请关注。
