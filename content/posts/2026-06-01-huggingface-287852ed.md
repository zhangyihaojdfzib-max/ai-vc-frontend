# 超越LLM：为何可扩展的企业级AI应用依赖于Agent逻辑

指南在人类历史上一直发挥着辅助作用。史前文明就明白，太阳和月亮可用于在陆地和远洋上导航长途距离。随着时间的推移，各种旅程促进了地图的制作，以便更好地规划路线并缩短重复目的地的旅行时间。几个世纪后，指南针的引入使航海者能够更准确地探索未知目的地。而今天，GPS导航应用指引着我们的每一次旅程。在当今的Agent AI世界中，AI Agent（智能体）确实有潜力实现可扩展的AI应用，从而改变我们所知的各个行业。然而，需要一种智能的指南——Agent逻辑——通过提升Agent质量、成本效益以及随之而来的最终用户信任，来实现这一潜力。

## 企业工作流程与用例

众多研究都指出了AI试点项目的普遍失败，而其他研究则强调了AI需要在企业工作流程的核心运行，以实现可扩展的应用。[1][2] 为了更好地理解这一现象及其相关论断，需要对企业的这些工作流程进行一些分析。这些工作流程具有以下特点：

A. 动态且长期运行
B. 拥有大量的API、数据库和服务
C. 常常受到业务策略和/或法规的约束

要使Agent在这些特性下有效运作，自然需要扩展的模型上下文，最先进的前沿LLM当然具备这一点，但代价是什么？是幻觉增加、Token消耗增多吗？此外，能否为LLM配备一个智能指南（GPS），使其能够在工作流程的核心执行Agent AI，从而驱动更理想的结果？我们通过为IBM产品设计和构建配备了相关Agent逻辑的Agent来测试这些假设，并充分考虑了上述特性。这些产品涉及领域专家面临的一些最具挑战性的任务，这些专家负责关键任务工作负载的企业软件交付生命周期的各个阶段，包括：

1. 理解用遗留代码（Cobol / PL/1）编写的应用程序
2. 加快开发人员的测试生成
3. 主动响应事件并实现左移应用韧性
4. 实现关键环境的合规性现代化自动化

在详细审视这些领域之前，让我们先定义Agent逻辑的特征。Agent逻辑是软件原语，例如知识图谱、算法、程序分析库，它们在Agent层（在Agent框架内）运行，能够有意地引导LLM朝着企业工作流程的方向前进，从而缩减上下文空间。这样做，极有可能以更具成本效益的方式驱动更优的结果。现在，让我们看看Agent逻辑如何能在上述四个领域中实现这样的结果。

### 1. 理解用遗留代码（Cobol / PL/1）编写的应用程序 - 程序分析。[3]

IBM watsonx Code Assistant for Z (WCA4Z) 用于通过AI和自动化加速大型机应用程序开发和现代化，它配备了一个用于应用程序理解的App Insights Agent——这是在IBM大型机上运行关键任务工作负载的企业客户的主要关注领域之一。该Agent利用跨应用程序的深度静态分析，并将预索引的表示存储在一个跨越数百个具有复杂语义的相互关联表的数据库模式中，使Agent能够检索精确、结构化的已有信息；从而提高了答案的准确性，减少了Token使用量，并最小化了与语言模型（本例中为Mistral Medium 250B）的来回交互。当这种方法应用于多个关键任务遗留系统（多达100万行代码和1000个程序）时，与仅使用基础前沿LLM的方法相比，它保持了略优的应用程序理解性能，同时Token消耗降低了约30倍。

### 2. 使用Aster加快开发人员的测试生成 - 程序分析。[4], [5]

Aster是一个基于IBM专有程序分析和数据预处理/后处理的库，用于基于Agent生成单元测试、集成测试、API测试和变更测试；根据对多个开发者社区的分析，与各种开源工具或开发人员编写的测试相比，它获得了更高的开发者评分。基于后者以及相比类似开源工具（集成测试）和零样本LLM及编码Agent（单元测试）更优的行覆盖率、分支覆盖率和方法覆盖率基准（均在开源应用程序上测试），我们已在75多个Java IBM CIO应用程序（多达560多个类和67K多行代码）上，使用Devstral 24B模型以预生产模式运行Aster。迄今为止的稳态结果显示，行覆盖率、分支覆盖率和方法覆盖率提高了20% - 45%，同时在这些应用程序的子集上，与最先进的编码Agent相比，性能更优，而Token消耗却低数个数量级（高达15倍）。这些结果的原因是，程序分析输出（用于提示和“聚焦”LLM）结合用于增强覆盖率和修复运行时及编译错误的子Agent，能够以显著降低的成本实现更优的结果。

### 3. 主动响应事件并实现左移应用韧性 - 知识图谱、程序分析库和调查（可观测性）驱动的编排。[6],[7]

虽然如1和2所述的应用相关用例的LLM上下文被“限制”在应用源代码中，但对于已部署基础设施上应用的运行时管理，底层的IT全栈就发挥作用了。这里我们定义了一个知识图谱（KG），包含实体（微服务、数据库/中间件服务、MELT等）以及来自领域专家的嵌入式（“部落”）知识。利用这样的图谱，并将LLM约束在局部有界推理中以处理非确定性结果，采用可观测性驱动的方法来实现缩减涵盖IT栈和底层应用源代码（如果相关）的上下文空间，用于事件根因分析（及其他用例）。通过这种方法，利用等效的Instana数据模型，我们看到专有的Instana“I3”（智能事件调查[8]）Agent在使用ITBench[9]衡量时，比使用GPT-5.1的ReAct Agent实现了高达4.0倍的改进。使用Gemini 3 Flash时，ReAct Agent的性能提升到仅比I3 Agent低17%，同时消耗的Token多1.6倍。我们将这种方法扩展到源代码，使用用于代码分析（利用程序依赖图）和错误修复（利用推理扩展）的Agent，同样在ITBench上进行了测试，结果表明，用于源代码分析和错误修复的Agent（Gemini 2.5 Flash）在查找有责任的微服务（3.0倍）和错误修复（1.6倍）方面均优于最先进的编码Agent，同时分别消耗少3.7倍和5.9倍的Token。这个多Agent系统已在IBM Think上作为新发布的IBM Concert Platform for Shift-Left IT Operations的一部分公布，并正在IBM CIO内部进行试点。[10]

### 4. 实现关键环境的IT合规性现代化自动化 - 算法与自适应规划和编排。[11]

企业面临日益复杂且碎片化的合规要求，迫使团队花费大量时间手动创建管控措施、评估方案与修复计划。由于缺乏集中化的知识体系，修复方案均需人工编写，这引入了错误与安全漏洞的风险。合规工作具有多步骤的复杂性，需要跨专业Agent（智能体）的协调性策略驱动自动化，而非依赖人工操作或简单的AI提示词。我们的多Agent（智能体）系统通过算法将复杂任务分解为协调步骤，采用自适应规划、动态分解与工作流排序，并辅以持续反馈来迭代识别修复方案并扩展评估范围，从而实现合规自动化。相较于采用固定规划策略的先前Agent（智能体）（Claude 4 Sonnet），其性能提升1.3至2.0倍（亦经ITBench验证）。该方法将合规转化为持续引导的自我修正流程，显著改善结果，尤其在复杂场景中，成功率从个位数提升至高达+80%（Claude 4 Sonnet）。该多Agent（智能体）系统及16,000余项数字化管控映射已在IBM Think大会上作为IBM Sovereign Core的一部分发布，集成监控、漂移检测功能，提供自动化证据生成，确保审计证据始终安全地保留在客户控制范围内。[12]

领域案例研究案例1：可配置通用Agent（智能体）（CUGA）医疗基准测试——算法化策略执行。[13]

以下健康保险客户服务示例简洁说明了为何在受监管环境中，Agent（智能体）系统优于纯LLM（大语言模型）对话模型。CUGA（可配置通用Agent（智能体））的策略系统实现了用于Agent（智能体）治理的"策略即代码"，该策略在运行时独立于模型提示词执行，且无需微调。我们的实验表明，Agent（智能体）的策略系统显著缩小了任务正确性方面的差距，在所有模型家族（Claude Opus – 4.5、GPT OSS 120B及GPT – 4.1）中强制执行结构化工作流、安全意图处理、可靠工具使用及受控输出格式，准确率提升幅度达15%至26%。权限通过最小特权披露、明确合规规则及人工升级路径得以执行。智能操作被提出，而权限由策略与监督机制行使。推理是自主的；决策权受到约束。CUGA亦是IBM Think Sovereign Core发布中的关键组件。

案例2：IBM全球房地产实物资产状态维护——有向无环图。[14],[15]

企业维护系统收集了大量资产数据，但无法有效整合，需要专家手动拼接碎片化信号并做出决策，缺乏统一、基于证据的洞察。我们最新推出的Maximo Condition Insights [16] Agent（智能体）分析跨数千资产与地点的大规模资产数据（传感器、工单、故障模式与事件分析），利用结构化证据与验证循环可靠地识别问题、确定行动优先级，并以一致且可追溯的洞察支持决策。我们已在IBM全球房地产（GRE）内部试点该Agent（智能体）（使用GPT OSS 120B），将资产分析时间从15-20分钟缩短至15-30秒（提升97%），并将资产审查覆盖率从约1%提升至约30%，覆盖超过120个站点及6,000项实物资产。使用AssetOpsBench基准测试，Condition Insights Agent（智能体）将无依据声明减少了57%，冗长性降低了35%，规则合规性提升了30%，保持了近乎零矛盾，Token使用量平均降低77%，同时诊断特异性略有提升。该Agent（智能体）配备有向无环图，提供结构工程与运营背景，以减少朴素提示下的无依据推理，而约束感知提示则显著改善规则遵循度、降低冗长性并减少总体Token消耗，且不会引入不稳定性。
