# 将Kubernetes扩展到7,500个节点

![将Kubernetes扩展到7,500个节点](/images/posts/9e8c7d784d3e.webp)

我们已将Kubernetes集群扩展到7,500个节点，为GPT‑3⁠(在新窗口中打开)、CLIP⁠和DALL·E⁠等大型模型提供了可扩展的基础设施，同时也支持诸如神经语言模型缩放定律⁠(在新窗口中打开)等快速的小规模迭代研究。

将单个Kubernetes集群扩展到如此规模并不常见，需要一些特殊处理，但其优势在于能够提供一个简单的基础设施，使我们的机器学习研究团队无需修改代码即可更快推进并扩展规模。

自我们上一篇关于扩展到2,500个节点⁠的文章以来，我们持续扩展基础设施以满足研究人员需求，在此过程中积累了许多额外经验。本文总结了这些经验，以便Kubernetes社区中的其他人能够从中受益，并在最后列出了我们仍需解决的下一个问题。

## 我们的工作负载

在深入讨论之前，有必要先描述一下我们的工作负载。我们使用Kubernetes运行的应用程序和硬件与典型公司可能遇到的情况有很大不同。我们的问题及相应解决方案可能适合，也可能不适合您的实际环境！

一个大型机器学习任务跨越多个节点，当它能访问每个节点上的所有硬件资源时运行效率最高。这使得GPU能够使用NVLink⁠(在新窗口中打开)直接进行交叉通信，或使用GPUDirect⁠(在新窗口中打开)让GPU直接与NIC通信。因此，对于我们的许多工作负载，单个Pod会占用整个节点。任何NUMA、CPU或PCIE资源争用都不是调度考虑因素。装箱或碎片化也不是常见问题。我们当前的集群具有全对分带宽，因此我们也不考虑机架或网络拓扑。所有这些意味着，尽管我们拥有许多节点，但调度器的压力相对较低。

不过，kube-scheduler的压力是突发的。一个新任务可能同时创建数百个Pod，然后恢复到相对较低的变更率。

我们最大的任务运行MPI，任务中的所有Pod都参与同一个MPI通信器。如果任何一个参与Pod宕机，整个任务就会停止并需要重新启动。任务会定期检查点，重启时从最后一个检查点恢复。因此，我们将这些Pod视为半有状态——宕机的Pod可以被替换，工作可以继续，但这样做会造成中断，应尽量减少。

我们并不太依赖Kubernetes的负载均衡。我们的HTTPS流量很少，不需要A/B测试、蓝绿部署或金丝雀发布。Pod之间通过SSH使用MPI直接通过Pod IP地址通信，而不是通过服务端点。服务"发现"有限；我们只在任务启动时一次性查找哪些Pod参与了MPI。

大多数任务与某种形式的对象存储交互。它们通常直接从对象存储流式读取数据集或检查点的某些分片，或将其缓存到快速的本地临时磁盘。在POSIX语义有用的情况下，我们使用少量PersistentVolume，但对象存储的可扩展性更强，且不需要缓慢的分离/附加操作。

最后，我们的工作本质上是研究，这意味着工作负载本身在不断变化。虽然超级计算团队致力于提供我们认为的"生产"级计算基础设施，但运行在该集群上的应用程序生命周期短，其开发人员迭代迅速。新的使用模式可能随时出现，挑战我们对趋势和适当权衡的假设。我们需要一个可持续的系统，同时能够在情况变化时快速响应。

## 网络

随着集群中节点和Pod数量的增加，我们发现Flannel难以扩展所需的吞吐量。我们转而使用Azure VMSS的原生Pod网络技术及其相关CNI插件。这使我们能够在Pod上获得主机级别的网络吞吐量。

我们改用基于别名的IP寻址的另一个原因是，在我们最大的集群上，任何时候都可能同时使用约200,000个IP地址。当我们测试基于路由的Pod网络时，发现我们能有效使用的路由数量存在显著限制。

避免封装增加了对底层SDN或路由引擎的需求，但使我们的网络设置保持简单。VPN或隧道可以在无需额外适配器的情况下添加。我们无需担心因网络部分MTU较低而导致的数据包分片。网络策略和流量监控变得直接；数据包的源和目的地址没有歧义。

我们在主机上使用iptables标记来跟踪每个命名空间和Pod的网络资源使用情况。这让研究人员能够可视化他们的网络使用模式。特别是，由于我们的许多实验具有不同的互联网和Pod内通信模式，能够调查瓶颈可能发生的位置通常很有用。

iptables的mangle规则可用于任意标记符合特定条件的数据包。以下是我们用于检测流量是内部还是互联网流量的规则。FORWARD规则覆盖来自Pod的流量，而INPUT和OUTPUT规则覆盖来自主机的流量：

#### 纯文本

```
1iptables -t mangle -A INPUT ! -s 10.0.0.0/8 -m comment --comment "iptables-exporter openai traffic=internet-in"2iptables -t mangle -A FORWARD ! -s 10.0.0.0/8 -m comment --comment "iptables-exporter openai traffic=internet-in"3iptables -t mangle -A OUTPUT ! -d 10.0.0.0/8 -m comment --comment "iptables-exporter openai traffic=internet-out"4iptables -t mangle -A FORWARD ! -d 10.0.0.0/8 -m comment --comment "iptables-exporter openai traffic=internet-out"
```

标记后，iptables将启动计数器来跟踪匹配此规则的字节数和数据包数。您可以使用iptables本身来查看这些计数器：

```
1% iptables -t mangle -L -v2Chain FORWARD (policy ACCEPT 50M packets, 334G bytes)3 pkts bytes target     prot opt in     out     source               destination4....51253K  555M            all  --  any    any     anywhere            !10.0.0.0/8           /* iptables-exporter openai traffic=internet-out */61161K 7937M            all  --  any    any    !10.0.0.0/8           anywhere             /* iptables-exporter openai traffic=internet-in */
```

我们使用一个名为iptables-exporter⁠(在新窗口中打开)的开源Prometheus导出器，将这些数据纳入我们的监控系统。这是一种跟踪匹配各种条件的数据包的简单方法。

我们网络模型的一个独特之处在于，我们向研究人员完全公开节点、Pod和服务网络的CIDR范围。我们采用中心辐射型网络模型，并使用原生的节点和Pod CIDR范围来路由流量。研究人员连接到中心，从那里可以访问任何单个集群（辐射点）。但集群之间不能相互通信。这确保了集群保持隔离，没有可能破坏故障隔离的跨集群依赖关系。

我们使用一个"NAT"主机来转换来自集群外部的流量的服务网络CIDR范围。这种设置让研究人员在选择实验的网络配置方式和类型时拥有极大的灵活性。

## API服务器

Kubernetes API 服务器和 etcd 是集群健康运行的关键组件，因此我们特别关注这些系统的压力。我们使用 kube-prometheus⁠（在新窗口打开）提供的 Grafana 仪表板，以及额外的内部仪表板。我们发现，针对 API 服务器上 HTTP 状态码 429（请求过多）和 5xx（服务器错误）的速率设置告警，是发现问题的有效高级信号。

虽然有些人在 kube 内部运行 API 服务器，但我们一直将它们运行在集群外部。etcd 和 API 服务器都运行在各自的专用节点上。我们最大的集群运行 5 个 API 服务器和 5 个 etcd 节点，以分散负载并最小化单个节点宕机的影响。自从我们在上一篇博客文章中将 Kubernetes 事件拆分到独立的 etcd 集群后，etcd 就没有再出现明显问题。API 服务器是无状态的，通常很容易在自愈实例组或规模集中运行。我们尚未尝试构建 etcd 集群的自愈自动化，因为相关事件极为罕见。

API 服务器会占用相当多的内存，并且这通常与集群中的节点数量呈线性增长。对于拥有 7500 个节点的集群，我们观察到每个 API 服务器使用的堆内存高达 70GB，因此幸运的是，这应该在未来仍能很好地保持在硬件能力范围内。

API 服务器的一个重大压力来自对 Endpoints 的 WATCH。有一些服务，例如 'kubelet' 和 'node-exporter'，集群中的每个节点都是其成员。当节点被添加或移除时，这个 WATCH 会触发。并且由于每个节点通常通过 kube-proxy 监视 kubelet 服务，这些响应所需的 # 和带宽将达到 N²N²N² 级别且非常巨大，有时会达到 1GB/s 或更多。在 Kubernetes 1.17 中推出的 EndpointSlices⁠（在新窗口打开）带来了巨大好处，将此负载降低了 1000 倍。

总的来说，我们非常关注任何随集群规模增长的 API 服务器请求。我们尽量避免让任何 DaemonSet 与 API 服务器交互。在确实需要每个节点监视变化的情况下，引入一个中间缓存服务，例如 Datadog Cluster Agent⁠（在新窗口打开），似乎是避免集群范围瓶颈的良好模式。

随着集群的增长，我们实际进行的集群自动扩缩容操作减少了。但有时一次性自动扩缩容过多时，我们也会遇到问题。当新节点加入集群时会产生大量请求，一次性添加数百个节点可能会使 API 服务器容量过载。平滑处理这个过程，即使只是延迟几秒钟，也有助于避免中断。

## 使用 Prometheus 和 Grafana 进行时间序列指标监控

我们使用 Prometheus 收集时间序列指标，并使用 Grafana 进行图表、仪表板和告警。我们最初部署了 kube-prometheus⁠（在新窗口打开），它收集了广泛的指标并提供了良好的可视化仪表板。随着时间的推移，我们添加了许多自己的仪表板、指标和告警。

随着我们添加越来越多的节点，我们在处理 Prometheus 收集的海量指标时遇到了困难。虽然 kube-prometheus 暴露了许多有用的数据，但其中一些我们实际上从未查看过，而另一些则过于细粒度，无法有效收集、存储和查询。我们使用 Prometheus 规则⁠（在新窗口打开）来“丢弃”其中一些指标，使其不被摄入。

有一段时间，我们遇到了一个问题：Prometheus 会消耗越来越多的内存，最终导致容器因内存不足（OOM）错误而崩溃。即使在为应用程序分配了巨大内存容量后，这种情况似乎仍然会发生。更糟糕的是，当它崩溃时，启动时需要花费数小时重放预写日志文件，然后才能再次使用。

最终，我们追踪到这些 OOM 的根源⁠（在新窗口打开）是 Grafana 和 Prometheus 之间的交互，Grafana 会使用查询 {le!=""} 调用 Prometheus 上的 /api/v1/series API（基本上就是“给我所有直方图指标”）。/api/v1/series 的实现无论在时间还是空间上都是无界的——对于结果很多的查询，它会持续消耗越来越多的内存和时间。即使在请求方放弃并关闭连接后，它也会继续增长。对我们来说，内存永远不够用，Prometheus 最终会崩溃。我们修补了⁠（在新窗口打开）Prometheus，将此 API 限制在 Context 内以强制执行超时，从而彻底解决了这个问题。

虽然 Prometheus 崩溃的频率大大降低，但在确实需要重启的时候，WAL 重放仍然是一个问题。在 Prometheus 启动并开始收集新指标和响应查询之前，通常需要花费数小时重放所有 WAL 日志。在 Robust Perception⁠（在新窗口打开）的帮助下，我们发现应用 GOMAXPROCS=24 带来了巨大的改进。Prometheus 在 WAL 重放期间会尝试使用所有核心，而对于拥有大量核心的服务器，争用会扼杀所有性能。

我们正在探索新的方案来增加监控能力，详情请参见下面的“未解决的问题⁠”部分。

## 健康检查

对于如此大的集群，我们当然依赖自动化来检测并将行为异常的节点从集群中移除。随着时间的推移，我们构建了多个健康检查系统。

### 被动健康检查

一些健康检查是被动的，始终在所有节点上运行。这些检查监控基本的系统资源，例如网络可达性、故障或满的磁盘，或 GPU 错误。GPU 会以多种不同方式出现问题，但一个常见的简单问题是“不可纠正的 ECC 错误”。Nvidia 的数据中心 GPU 管理器（DCGM）工具可以轻松查询此错误以及许多其他“Xid”错误。我们追踪这些错误的一种方式是通过 dcgm-exporter⁠（在新窗口打开）将指标摄入到我们的监控系统 Prometheus 中。这将显示为 DCGM_FI_DEV_XID_ERRORS 指标，并设置为最近发生的错误代码。此外，NVML 设备查询 API⁠（在新窗口打开）暴露了关于 GPU 健康和运行的更详细信息。

一旦检测到错误，通常可以通过重置 GPU 或系统来修复，尽管在某些情况下，这会导致需要物理更换底层的 GPU。

另一种形式的健康检查会跟踪上游云提供商的维护事件。每个主要的云提供商都提供了一种方式来了解当前 VM 是否即将发生会导致中断的维护事件。VM 可能需要重启以应用底层虚拟机监控程序补丁，或者将物理节点更换为其他硬件。

这些被动健康检查在所有节点上持续在后台运行。如果健康检查开始失败，节点会被自动封锁，以便没有新的 Pod 被调度到该节点上。对于更严重的健康检查失败，我们还会尝试 Pod 驱逐，以请求所有正在运行的 Pod 立即退出。是否允许此驱逐发生，仍然取决于 Pod 本身，可通过 Pod 中断预算进行配置。最终，要么在所有 Pod 终止后，要么在 7 天（我们 SLA 的一部分）后，我们将强制终止该 VM。

### 主动 GPU 测试

不幸的是，并非所有 GPU 问题都会表现为通过 DCGM 可见的错误代码。我们构建了自己的测试库，通过运行 GPU 来捕获其他问题，并确保硬件和驱动程序按预期运行。这些测试无法在后台运行——它们需要独占使用 GPU 几秒或几分钟才能执行。

我们首先在节点启动时运行这些测试，这一系统被称为“preflight”。所有节点在加入集群时都会带有“preflight”污点和标签。该污点阻止普通 Pod 被调度到该节点上。我们配置了一个 DaemonSet，用于在所有带有此标签的节点上运行 preflight 测试 Pod。测试成功完成后，测试本身会移除污点和标签，节点随后即可供常规使用。

此外，我们还会在节点生命周期内定期运行这些测试。我们将其作为 CronJob 运行，使其能够落在集群中任何可用的节点上。诚然，这种方式在哪些节点被测试上有些随机且不受控制，但我们发现，随着时间的推移，它能够以最小的协调或干扰提供足够的覆盖率。

## 配额与资源使用

随着集群规模的扩大，研究人员开始发现自己难以获得分配的全部容量。传统的作业调度系统拥有许多可用于在竞争团队之间公平运行工作的不同功能，而 Kubernetes 并不具备这些功能。随着时间的推移，我们从这些作业调度系统中汲取灵感，并以 Kubernetes 原生方式构建了若干能力。

### 团队污点

我们在每个集群中都有一个服务，名为“team-resource-manager”，它具有多种功能。其数据源是一个 ConfigMap，该 ConfigMap 为在给定集群中拥有容量的所有研究团队指定了（节点选择器、要应用的团队标签、分配量）元组。它会与集群中的当前节点进行协调，为适当数量的节点打上 `openai.com/team=teamname:NoSchedule` 污点。

team-resource-manager 还有一个准入 Webhook 服务，这样，当每个作业被提交时，会根据提交者所属的团队应用相应的容忍度。使用污点使我们能够灵活地约束 Kubernetes Pod 调度器，例如允许低优先级 Pod 具有“任意”容忍度，这使得团队之间可以借用彼此的容量，而无需进行重量级的协调。

### CPU 与 GPU 占位容器

除了使用 cluster-autoscaler 来动态扩展我们的虚拟机支持集群外，我们还使用它来修复（移除并重新添加）集群内的不健康成员。我们通过将集群的“最小规模”设置为零，并将“最大规模”设置为可用容量来实现这一点。然而，如果 cluster-autoscaler 发现空闲节点，它会尝试缩减到仅需要的容量。由于多种原因（虚拟机启动延迟、预分配成本、上述 API 服务器影响），这种空闲缩减并不理想。

因此，我们为纯 CPU 和 GPU 主机引入了一个占位容器 Deployment。该 Deployment 包含一个 ReplicaSet，其副本数为“最大规模”的低优先级 Pod。这些 Pod 占用节点内的资源，因此自动扩缩器不会将它们视为空闲。但由于它们是低优先级的，调度器可以立即驱逐它们，为实际工作腾出空间。（我们选择使用 Deployment 而不是 DaemonSet，是为了避免 DaemonSet 被视为节点上的空闲工作负载。）

需要注意的一点是，我们使用 Pod 反亲和性来确保 Pod 均匀分布在各个节点上。早期版本的 Kubernetes 调度器在处理 Pod 反亲和性时存在 `O(N^2)` 的性能问题。这个问题自 Kubernetes 1.18 起已得到修正。

## 组调度

我们的实验通常涉及一个或多个 StatefulSet，每个 StatefulSet 负责训练工作的不同部分。对于优化器，研究人员需要 StatefulSet 的所有成员都被调度后，才能开始任何训练（因为我们经常使用 MPI 来协调优化器成员，而 MPI 对组成员变更很敏感）。

然而，Kubernetes 默认情况下不一定会优先满足一个 StatefulSet 的所有请求而不是另一个。例如，如果两个实验各自请求了集群 100% 的容量，Kubernetes 可能不会调度完一个实验或另一个实验的全部 Pod，而是只调度每个实验的一半 Pod，从而导致死锁，两个实验都无法取得进展。

我们尝试了一些需要自定义调度器的方法，但遇到了边缘情况，导致与普通 Pod 的调度方式发生冲突。Kubernetes 1.18 为核心 Kubernetes 调度器引入了一种插件架构，使得以原生方式添加此类功能变得更加容易。我们最近采用了 `Coscheduling` 插件作为解决此问题的好方法。

## 未解决的问题

随着我们扩展 Kubernetes 集群，仍有许多问题需要解决。其中一些包括：

### 指标
