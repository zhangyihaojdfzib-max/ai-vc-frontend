# 我们如何在hyper HTTP库中发现了一个bug

2026-06-22

- Deanna Lam
- Diretnan Domnan
- Matt Lewis

![](/images/posts/7dc4c0017b05.png)

TheImages服务基于Rust构建，运行在Workers上，部署于Cloudflare边缘网络的每台机器中。为了处理客户端连接，我们使用了hyper——一个开源的Rust HTTP库。

去年，我们引入了Images绑定，使得在Workers中处理远程图像时能够实现自定义的程序化工作流。2025年底，我们对绑定进行了重新架构，以在Workers运行时与Images服务之间提供更直接的本地连接。

上线后不久，我们收到报告称绑定中的转换请求会失败——但只是间歇性发生，且仅针对较大的图像。更奇怪的是，这些请求的响应返回了200状态码，且没有任何错误日志。图像数据被截断了：本应是两兆字节的响应，有时只到达了几百千字节。

我们花了六周时间追踪hyper库中一个几乎不可见的bug——一个仅在特定条件下出现的竞态条件——它影响了Images绑定将处理后的图像数据返回给客户端的方式。最终，修复它只用了四行代码。

### 跳转、交接与hyper

当开发者在Cloudflare上构建应用时，他们会通过一组平台服务组合出全栈应用，这些服务通过绑定对Workers开放。绑定提供了开发者平台上资源的直接API，例如计算、存储、AI推理和媒体处理。

Images绑定将图像优化与交付解耦；你可以对图像进行转码、合成或处理，而无需将输出作为HTTP响应返回。它还允许你以任意顺序应用优化参数，而不必遵循URL接口所强制的固定顺序。在这里，Worker可以将图像数据直接传递给Images API，将操作串联起来，并将处理结果以流的形式返回：

```JavaScript
const result = await env.IMAGES
  .input(image)
  .transform({ width: 800, rotate: 90 })
  .output({ format: "image/avif" });
return result.response();
```

从高层来看，图像数据通过我们各个服务的流动方式如下：

管道表示中间服务与Images之间的套接字连接，数据通过内核缓冲区从一个进程传递给下一个进程。

绑定通过Workers运行时管理的套接字连接与Images通信。套接字连接是两个进程之间的通信通道。套接字的每一端都有由操作系统内核管理的缓冲区；这些缓冲区是临时存储区域，数据在一侧写入后、另一侧读取前暂存于此。

Hyper在Images服务端管理连接，从套接字读取传入请求，并将响应写回套接字。

当请求使用Images绑定时，Images服务读取输入，执行请求的优化操作，并对结果进行编码。然后它将整个编码后的图像作为一个内存块传递给hyper。

Hyper将这段响应数据写入自己的内部缓冲区。此时，hyper认为编码工作已完成，因为它已拥有发送所需的所有字节。下一步是将内部缓冲区刷新到套接字的出站缓冲区，将数据从Images服务移动到另一端的中间服务。

如果另一端的读取器速度很快，那么hyper可以一次性刷新所有数据——出站缓冲区会有空间，因为读取器在数据到达时立即消费。一旦所有数据发送完毕，hyper会在套接字上发起shutdown，表示连接已完成，不再写入更多数据。但如果读取器较慢（即使只慢几毫秒），出站缓冲区就会填满，hyper需要等待有空间时才能继续写入。

### 走向本地化

Cloudflare网络上的所有传入流量都会经过FL——一个内部中间服务，它运行安全和性能功能，并将请求路由到相应的后端。当我们首次推出绑定时，图像数据从Workers运行时流经FL，再到Images服务。

这条路径对我们最初的发布来说是自然的选择，并且遵循与URL接口相同的架构。但随着时间的推移，与FL的耦合变成了一个约束：对绑定的每次更改都必须遵循FL的发布周期。

2025年12月，Images团队用一个新的中间服务替换了FL——一个运行在同一台机器上的内部Worker绑定。在原始架构中，数据通过网络套接字经过FL；这条路径带来了FL完整处理管道的开销，例如DNS查找和路由。

内部绑定用Unix套接字取代了这些，直接连接同一台机器上的服务，绕过了FL和网络栈的开销。这使得通往Images的请求路径更快，并让团队能够独立控制绑定的发布。

上线几天内，我们就收到了第一份客户报告。

### 200 OK（并不OK）

问题的第一个迹象来自一个采用非标准设置的客户：两层图像处理，其中一个管道嵌套在另一个管道内。

首先，他们的Worker使用Images绑定将来自R2的多张大尺寸源图像——一个JPEG背景加上PNG叠加层——合成为一张组合JPEG。其次，他们通过URL接口对结果进行进一步压缩、转码和调整大小。

这个bug源于内部管道的返回路径，响应在到达外部管道之前被截断了。

内部管道（转换绑定）处理合成。外部管道（转换URL）处理交付优化，如缩放和格式转换。这种分层方式意味着，当内部管道静默返回截断的响应时，唯一可见的错误出现在上一层：

```Rust
从连接读取主体时出错：在达到消息长度之前遇到文件结束
```

外部管道从内部管道收到HTTP 200，其Content-Length头部承诺了几兆字节。实际主体只有其中的一小部分：在一次请求中，预期的3.3 MB只到达了约200 KB。错误出现在外部管道，但截断可能源于绑定、中间服务、Images服务，或两者之间的某个环节。

当浏览器接收到截断的图像时，结果是可见的。根据格式不同，图像要么部分渲染（例如，下半部分缺失或显示为灰色），要么完全无法解码，显示为损坏的图像。

### 在黑暗中调试

从这里开始，我们沿着请求路径向内排查，测试每一层以定位截断发生的位置。其中一些努力走进了死胡同；另一些则留下了缩小搜索范围的线索：

- **构建复现环境。** 我们构建了一个工作进程来模拟客户的多层嵌套设置，然后逐层剥离，直到仅凭绑定就能触发该错误。一个小脚本让我们能够批量发送请求。在早期的一次运行中，25个请求里有19个失败了。实际到达的数据量——大约200 KB——与生产环境中套接字缓冲区的大小惊人地接近。这证实了问题与客户的配置无关，并为我们提供了一种按需可靠触发该错误的方法。

- **调查超时问题。** 起初，我们怀疑数据截断可能与超时行为有关（即连接在达到时间限制后被关闭）。但这个理论站不住脚，因为截断现象与请求持续时间并无关联。

- **更新 hyper 版本。** 当该错误首次被报告时，我们运行的是 0.14.x 版本，而最新的 hyper 版本大约是 1.8.x。我们在 hyper 的 0.14、1.7 和 1.8 版本上进行了测试，以防最明显的答案就是正确（且最简单）的那个。但该错误在每个版本中都出现了，这意味着上游并没有修复方案。

- **本地复现。** 我们在 macOS 和一个 Debian 虚拟机上运行了本地集成测试。即使在高负载下，我们的本地请求也从未触发任何失败。直接向绑定套接字发起 curl 请求以及重放捕获的请求似乎总是成功的。该错误只出现在完整的生产路径上，即存在真实并发且套接字另一端是真实的 Workers 运行时客户端时。这让我们开始怀疑运行时本身。

- **排除 Workers 运行时。** 我们检查了 Workers 运行时用于通过绑定套接字与 Images 服务通信的 HTTP 客户端。连接两端的跟踪信息均未显示有任何系统调用表明发生了意外关闭或提前终止。我们观察到客户端行为正确，并且多个其他服务也使用相同的客户端而未出现问题。

- **分布式追踪。** 通过端到端检查请求跟踪信息，我们确认截断的响应体在到达客户设置中的外部转换层之前就已经存在了。这便将问题范围缩小到了内部管道——即通过 Images 服务的绑定路径。

- **为中介服务添加监控。** 我们在中介服务中添加了监控，以在转发响应数据之前测量响应体大小。响应体在离开 Images 服务时就已经被截断了，因此中介服务被排除了。

- **在 Images 服务内部进行更深入的追踪。** 在服务层面，请求被处理，图像被正确编码，并且响应以 HTTP 200 状态码发送。

唯一一致的信号是，该错误与时机相关：它只出现在生产路径上，且只有在真实并发以及处理较大图像时才会出现。

### 真相内核

用于应用层调试的工具只能告诉我们系统自认为它在做什么。但根据系统的反馈，一切正常：追踪显示响应已发送；日志未报告任何错误；Images 服务对每个请求都返回了 200 状态码。

为了查看系统实际在做什么，我们将 `strace` 附加到了 Images 服务上。`strace` 会记录进程向内核发起的系统调用，这可以精确地显示哪些字节被写入、何时调用了关闭操作，以及客户端是否发送了任何终止信号。

设置追踪需要非常小心。`strace` 的工作原理是在系统调用发生时进行拦截，这会给每个系统调用增加少量的时间开销。将过滤范围限定在一小组系统调用上可以将这种开销降至最低。然而，扩大过滤范围会稍微减慢进程速度，足以改变刷新操作和关闭检查之间的时间点——从而使该错误完全消失。仅此一点就强化了我们的理论：该问题对时间非常敏感。

使用一个复现工作进程，我们触发了该错误，并比较了成功请求和失败请求的系统调用输出。

在一个成功的请求中，响应会随着套接字缓冲区的允许而分块写入，只有在所有数据都发送完毕后才会调用关闭操作。例如，可能看起来像这样：

```Shell
sendto(42, "HTTP/1.1 200 OK\r\nContent-Length: 14991808\r\n...", ...) = 219264
sendto(42, "\xff\xd8\xff\xe0...", 292352) = 292352
// ... 持续写入直到缓冲区耗尽 ...
sendto(42, "...", 292352) = 292352
shutdown(42, SHUT_WR) = 0
```

当我们复现该错误时，一个失败的请求看起来像这样：

```Shell
sendto(42, "HTTP/1.1 200 OK\r\nContent-Length: 14991808\r\n...", ...) = 219264
shutdown(42, SHUT_WR) = 0
```

这里，在立即调用关闭操作之前，只有一次写入——刚好够写入头部和一小部分响应体。在一个 14.9 MB 的响应中，只发送了大约 219 KB。剩余的约 14.8 MB 图像数据从未离开 hyper 的内部缓冲区，并且在写入和关闭操作之间也没有来自客户端的任何终止信号。相反，Images 服务自己提前关闭了连接，并且它真心相信自己已经完成了任务。

失败的请求证实，这个Bug是一个间歇性触发的竞态条件。请求成功与否取决于刷新和关闭操作是否重叠，而这种情况因请求而异。当hyper在决定连接完成的那一瞬间缓冲区仍然满时，数据就会丢失。

当读取方消费数据的速度慢于hyper写入速度时，出站缓冲区就会填满。如果hyper在缓冲区排空之前关闭连接，那么只有部分响应数据能到达中间层；这些不完整的数据会被转发回Workers运行时和客户端。

12月的重构并未引入这个Bug，它在hyper中已经存在多年，跨越了多个主要版本。但新的中间层改变了套接字响应端的读取方。我们的工作理论是，之前的中间层FL消费数据的速度足够快，使得套接字缓冲区在响应期间很少填满。新的读取方读取速度较慢，偶尔会在较大响应期间让缓冲区填满。

这短短几毫秒的反压，由一个让其他所有操作都更快的改进所引入，却足以暴露一个一直隐藏的缺陷。

### 调度循环内部

Hyper的HTTP/1连接生命周期由`dispatch.rs`文件中的一个状态机驱动。它运行一个循环，读取请求、写入响应、将写入缓冲区刷新到套接字，并决定何时关闭。简化形式如下：

```Rust
fn poll_loop(&mut self, cx: &mut Context<'_>) -> Poll<Result<(), Error>> {
    loop {
        let _ = self.poll_read(cx)?;
        let _ = self.poll_write(cx)?;
        let _ = self.poll_flush(cx)?;

        if !self.conn.wants_read_again() {
            return Poll::Ready(Ok(()));
        }
    }
}
```

更准确地说，`let _` 在 `poll_flush` 之前就是Bug所在。

在Rust中，`let _ = expr` 会丢弃表达式的结果，包括 `Poll::Pending`——这个信号表示刷新尚未完成。刷新可能还有数兆字节的数据在缓冲区中，但循环永远不会知道。

当请求失败时，事件序列如下：

1. Images服务完成图像编码，将整个响应作为单个内存块交给hyper。
2. Hyper将该块写入其内部缓冲区，并将写入状态标记为 `Writing::Closed`。从编码角度来看，工作已经完成——没有剩余内容需要编码。
3. Hyper调用 `poll_flush` 将缓冲数据移动到套接字。在我们之前的例子中，套接字接收了约219 KB。剩余的约14.8 MB留在hyper的缓冲区中。套接字已满，因此内核返回 `Poll::Pending`。
4. `poll_loop` 通过 `let _` 丢弃了 `Poll::Pending`。
5. 它检查 `wants_read_again()`。完整请求已经收到，因此返回 `false`。
6. `poll_loop` 返回 `Poll::Ready(Ok(()))`，表示循环已完成，尽管刷新尚未完成。
7. `poll_shutdown()` 触发。发出 `SHUT_WR` 系统调用。
8. 客户端收到219 KB和一个EOF（文件结束符），表示连接已关闭，尽管它期望收到14.9 MB。

Images服务完成图像编码，将整个响应作为单个内存块交给hyper。

Hyper将该块写入其内部缓冲区，并将写入状态标记为 `Writing::Closed`。从编码角度来看，工作已经完成——没有剩余内容需要编码。

Hyper调用 `poll_flush` 将缓冲数据移动到套接字。在我们之前的例子中，套接字接收了约219 KB。剩余的约14.8 MB留在hyper的缓冲区中。套接字已满，因此内核返回 `Poll::Pending`。

`poll_loop` 通过 `let _` 丢弃了 `Poll::Pending`。

它检查 `wants_read_again()`。完整请求已经收到，因此返回 `false`。

`poll_loop` 返回 `Poll::Ready(Ok(()))`，表示循环已完成，尽管刷新尚未完成。

`poll_shutdown()` 触发。发出 `SHUT_WR` 系统调用。

客户端收到219 KB和一个EOF（文件结束符），表示连接已关闭，尽管它期望收到14.9 MB。

在第二步中，hyper在响应体被缓冲（即编码完成）时就标记写入操作完成，而不是在数据实际刷新之后。大多数情况下，刷新一次就能完成，这种区别是不可见的。在极少数情况下，当套接字缓冲区满时，刷新必须等待——即使hyper不等待。字节仍然在hyper的缓冲区中，等待刷新到套接字。Hyper在数据仍在缓冲区中时继续关闭连接。

这也解释了为什么curl从未触发这个Bug。Curl读取数据的速度与数据到达速度一样快：套接字缓冲区永远不会填满，刷新总是立即完成，丢弃的返回值无害。生产路径中，读取方偶尔暂停几毫秒，是唯一让缓冲区在错误时刻填满的配置。

### 不要忘记刷新

经过数周的调查，修复方案本身在概念上很简单。Hyper需要在继续之前检查刷新是否实际完成。

我们的复现Worker确认了Bug存在，但无法告诉我们某个请求为何失败。在编写修复之前，我们需要一个能够触发hyper内部精确套接字条件的测试。

我们知道触发Bug的条件：一个套接字接收一块数据然后阻塞。为了在可控场景下测试，我们构建了一个围绕TCP流的自定义包装器，模拟满的套接字缓冲区。该包装器在第一次写入时接收8 KB，然后每次后续写入都返回 `Poll::Pending`，模拟停止排空缓冲区的读取方。

测试通过这个受限套接字发送了一个500 KB的响应，并检查hyper是否在492 KB仍在缓冲时调用了关闭。没有修复时，确实如此。有了修复，它会等待。

最初，我们在hyper的调度循环中应用了修复。我们没有丢弃 `poll_flush` 的结果，而是检查刷新是否实际完成：

```Rust
let flush_result = self.poll_flush(cx)?;

if flush_result.is_pending() {
    return Poll::Pending;
}

if !self.conn.wants_read_again() {
    return Poll::Ready(Ok(()));
}
```

如果刷新尚未完成，循环向异步运行时返回 `Poll::Pending`。运行时等待套接字变为可写，然后唤醒任务继续刷新。只有在所有数据都已发送后，连接才会关闭。

当我们部署这个修复时，我们观察到每个字节都被写入，并且只有在缓冲区实际为空时才调用关闭。第一个报告问题的客户也确认问题消失了。

虽然我们的初始方案有效，但调度循环并不是修复的正确位置。过早返回 `Poll::Pending` 可能会减少同一连接上读取的轮询频率，从而减慢其他操作，导致意外的反压。它也无法正确处理长连接，即单个连接顺序处理多个请求——这些连接即使在之前的响应仍在刷新时也应保持可重用。这两个问题都没有影响我们的特定服务（其中长连接已禁用），但如果修复被贡献到上游，都可能影响其他hyper用户。

我们追踪了hyper的连接生命周期，找到了一个更有针对性的方法。我们没有改变调度循环的行为，而是在实际调用关闭的位置应用了修复。在关闭套接字之前，hyper应该首先刷新缓冲区中任何剩余的数据：

```Rust
pub(crate) fn poll_shutdown(
    &mut self,
    cx: &mut Context<'_>,
) -> Poll<io::Result<()>> {
    ready!(self.poll_flush(cx)?);
    Pin::new(&mut self.io).poll_shutdown(cx)
}
```

这使得调度循环保持不变。它仅在数据丢失可能发生的精确位置——即关闭前的瞬间——添加了一次刷新。

### 我们从中得到的启示

应用层的所有工具均未报告任何错误、崩溃或能提供有用线索的日志条目。应用层的可观测性对于其感知范围之外的漏洞可能存在盲区。

该故障间歇性发生，随响应大小扩展，无法通过curl等简单工具复现，并且在我们更密切地观察系统时消失。这些信号指向连接层中一个与时序相关的漏洞，而非应用逻辑。

我们的突破来自于使用内核级工具`strace`——这一层记录了套接字上实际发生的情况。底层漏洞存在于部分刷新与过早关闭之间的几毫秒内——这个时间窗口仅在我们使系统运行更快后才出现。

我们将修复方案及确定性测试合并到了`hyperium/hyper`的PR #4018中。该修复将在未来的hyper版本中可用，确保任何使用hyper的HTTP/1实现的服务不会因相同的竞态条件而丢失响应数据。

与此同时，我们正在运行一个应用了该补丁的内部分支。此修复稳定了绑定的架构，为其功能扩展奠定了可靠的基础。

Images绑定最初仅覆盖远程图像的转换。本月早些时候，我们宣布Images绑定现已支持托管图像的操作，为开发者提供了一种在Cloudflare上构建富媒体应用的统一方式。
