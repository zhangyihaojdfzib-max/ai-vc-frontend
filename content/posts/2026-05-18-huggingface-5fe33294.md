# 使用 LoRA/DoRA 微调 NVIDIA Cosmos Predict 2.5 用于机器人视频生成

## 动机

NVIDIA Cosmos Predict 2.5 是一个大规模世界模型，能够根据文本、图像或视频片段生成符合物理规律的视频。为了将其适配到特定领域，例如机器人操作或特定摄像机视角，团队仍需进行针对性微调。

训练机器人策略需要演示数据，但收集真实机器人轨迹既缓慢又昂贵。使用微调后的视频世界模型生成合成轨迹提供了一种可扩展的替代方案。然而，对 2B 参数模型进行全量微调成本高昂，且存在遗忘通用知识的风险。LoRA 和 DoRA 将小型可训练适配器模块注入冻结的基础模型中，降低了内存需求，同时保持适配器文件小巧便携。这使得在单 GPU 上进行微调成为可能，并可在推理时灵活地为不同领域切换适配器。

本指南将介绍如何使用 diffusers 和 accelerate 库，通过 LoRA 和 DoRA 对 Cosmos Predict 2.5 进行参数高效微调，支持单 GPU 和多 GPU 训练。随后，我们将展示如何使用微调后的模型为下游机器人学习任务生成合成机器人轨迹。

### 初始化适配器

Cosmos Predict 2.5 包含三个子模块：

1. 一个将视频编码为潜在表示的 VAE
2. 一个将文本提示编码为提示词嵌入的文本编码器
3. 用于在潜在空间中进行扩散的 DiT

训练期间，所有 VAE、文本编码器和 DiT 的权重均被冻结。LoRA 适配器被注入到 DiT 的注意力投影层（to_q、to_k、to_v、to_out.0）和前馈层（ff.net.0.proj、ff.net.2）中。然后，可训练的 LoRA 参数被提升为 float32，以便在 bf16 混合精度下保持数值稳定性。

```py
from diffusers import Cosmos2_5_PredictBasePipeline
from peft import LoraConfig

pipe = Cosmos2_5_PredictBasePipeline.from_pretrained(
    "nvidia/Cosmos-Predict2.5-2B",
    revision="diffusers/base/post-trained",
    torch_dtype=torch.bfloat16,
)


dit = pipe.transformer
vae = pipe.vae
text_encoder = pipe.text_encoder

dit.requires_grad_(False)
vae.requires_grad_(False)
text_encoder.requires_grad_(False)

lora_config = LoraConfig(
    r=args.lora_rank,
    lora_alpha=args.lora_alpha,
    target_modules=['to_q', 'to_k', 'to_v', 'to_out.0', 'ff.net.0.proj', 'ff.net.2'],
    use_dora=args.use_dora,  
)
dit.add_adapter(lora_config)
cast_training_params(dit, dtype=torch.float32)  

```

传递 use_dora=True 可切换到 DoRA，该方法在应用低秩更新之前将每个权重分解为幅度和方向。训练循环无需其他更改。

Cosmos Predict 2.5 使用整流流：模型被训练来预测将噪声样本线性传输到原始"干净"数据的速度。具体来说，在时间步 t，在采样的噪声水平 σt 下构建一个噪声插值 xt = σt·噪声 + (1−σt)·干净，模型通过均方误差（MSE 损失）学习预测目标速度 噪声 − 干净。视频的前两帧用作条件，因此不对其潜在表示添加噪声。

训练损失遵循 Cosmos Predict 2.5 使用的整流流公式：

```py

sigma_t = sample_train_sigma_t(bsz, distribution='logitnormal', device=device)


xt = noise * sigma_t + clean_latent * (1 - sigma_t)



xt = clean_latent * cond_mask + xt * (1 - cond_mask)
in_timestep = cond_indicator * 0.0001 + (1 - cond_indicator) * sigma_t


pred_velocity = dit(
                    hidden_states=xt,
                    condition_mask=cond_mask,
                    timestep=in_timestep,
                    encoder_hidden_states=prompt_embeds,
                    padding_mask=padding_mask,
                    return_dict=False,
                )[0]


target_velocity = noise - clean_latent
pred_velocity = target_velocity * cond_mask + pred_velocity * (1 - cond_mask)
loss = F.mse_loss(pred_velocity.float(), target_velocity.float())

```

### 优化器和调度器

我们使用 torch.optim.AdamW 作为优化器，并使用来自 diffusers.optimization 的 get_linear_schedule_with_warmup 作为调度器。调度器在 scheduler_warm_up_steps 步内线性预热学习率，在 scheduler_f_max × learning_rate 处达到峰值，然后在剩余的 num_training_steps 步内线性衰减至 scheduler_f_min × learning_rate。

```py
lora_params = [p for p in dit.parameters() if p.requires_grad]

optimizer = torch.optim.AdamW(lora_params, lr=args.learning_rate, weight_decay=args.weight_decay)
lr_scheduler = get_linear_schedule_with_warmup(
    optimizer,
    num_warmup_steps=args.scheduler_warm_up_steps,
    num_training_steps=args.num_training_steps,
    f_min=args.scheduler_f_min,
    f_max=args.scheduler_f_max,
)

```

### 训练命令

使用提供的 shell 脚本作为起点：

```shell
export MODEL_NAME="nvidia/Cosmos-Predict2.5-2B"
export DATA_DIR="gr1_dataset/train"
export OUT_DIR=YOUR_OUTPUT_DIR
lora_rank=32

accelerate launch --mixed_precision="bf16" train_cosmos_predict25_lora.py \

  --revision diffusers/base/post-trained \

  --train_batch_size=1 \
  --num_train_epochs=500 \
  --checkpointing_epochs=100 \
  --seed=0 \
  --output_dir=$OUT_DIR \
  --report_to=wandb \
  --height 432 --width 768 \
  --allow_tf32 --gradient_checkpointing \
  --lora_rank $lora_rank --lora_alpha $lora_rank

```

`lora_rank` 控制低秩分解的秩。更高的秩意味着更多可训练参数和更强的表达能力，但代价是更大的内存占用和更大的适配器文件。我们使用 rank=32 作为起点，产生了约 5000 万可训练参数。

`lora_alpha` 是应用于 LoRA 更新的缩放因子：权重增量在添加到冻结的基础权重之前，会按 `lora_alpha / lora_rank` 进行缩放。将 `lora_alpha = lora_rank`（如此处所示）可将此缩放因子保持在 1.0，因此 LoRA 更新以完整强度应用，无需任何额外衰减。

要使用 DoRA 而非 LoRA，请在命令中添加 `--use_dora`。

对于多 GPU 训练，`accelerate` 会自动处理分布。根据经验，我们发现在此任务上训练 100 个 epoch 已能产生不错的结果，在单张 H100 上需要 17 小时，在 8 张 H100 GPU 上需要 2.5 小时。

## 评估指标

### Sampson 误差

Sampson 误差是一种几何误差度量，用于测量匹配的关键点到其对应极线的距离。在生成视频的上下文中，较低的 Sampson 误差意味着帧间（或相机视角间）的运动在几何上是一致的。较高的值表示抖动、幻觉运动或多视角不一致。

我们遵循 Cosmos Predict 评估指南，并使用两个指标评估生成视频的几何质量：

- 时间 Sampson 误差：在单个相机视角内的连续帧之间计算，衡量时间稳定性。
- 跨视角 Sampson 误差：在不同相机视角的同步帧之间计算，衡量多视角几何对齐。

### LLM 作为评判者

我们使用 Cosmos Reason2 作为 LLM 评判者，对每个示例从 1 到 5 进行评分。我们设计了两个评分标准：

- 物理合理性（video_physics.yaml）：评判者评估视频是否遵循物理常识，而不查看文本提示。
- 指令遵循（video_IF.yaml）：评判者将提示和视频作为输入，并评估描述的任务是否正确完成。

video_physics.yaml

```yaml
system_prompt: "你是一个有用的助手。"
user_prompt: |
  你是一个有用的视频分析器。评估视频是否遵循物理常识。

  评估标准：
  1. **物体行为：** 物体的行为是否符合其预期的物理属性（例如，刚性物体不会不自然地变形，流体自然流动）？
  2. **运动与力：** 视频中描绘的运动和力是否与现实世界物理一致（例如，重力、惯性、动量守恒）？
  3. **交互：** 物体之间以及物体与环境之间的交互是否合理（例如，没有不自然的穿透，碰撞时有适当的反应）？
  4. **时间一致性：** 视频是否在帧间保持一致性，没有物体行为或运动的突然、无法解释的变化？

  评分说明：
  - **1：** 完全不遵循物理常识。视频包含大量违反基本物理定律的情况。
  - **2：** 遵循程度差。部分元素符合物理，但存在重大违规。
  - **3：** 遵循程度中等。视频大部分遵循物理，但存在明显的不一致。
  - **4：** 遵循程度良好。视频中大多数元素遵循物理定律，仅有轻微问题。
  - **5：** 完美遵循。视频展现出对物理常识的深刻理解，没有违规。

  这个视频是否遵循物理定律？

```

## 结果

### 定性分析

我们比较了基础模型（微调前）、LoRA 和 DoRA 在测试集前两个示例上生成的视频。

提示：用左手从圆形灰色垫子上拿起深绿色黄瓜，放到米色碗上方。

提示：用右手从粉色盘子中央拿起橙汁盒，放到绿色碗中央。

在微调之前，基础模型在几个方面存在困难：机器人手部属于分布外数据，导致模型在后续帧中幻觉出人类手部；它不能可靠地使用提示中指定的正确手部；并且生成的视频表现出明显的抖动。使用 LoRA 和 DoRA 进行微调解决了所有这三个问题。

### 定量分析

我们在不同设置下微调了四个适配器：秩为8和32的LoRA与DoRA。对于每个测试样本，我们使用不同随机种子生成5个视频，并采用评估指标部分介绍的三个指标报告所有种子的平均得分。

结论：训练100个周期（在8× H100上约2.5小时）已足以显著提升所有三个指标。LoRA和DoRA均收敛到相似性能，证实DoRA中额外的幅度-方向分解在极低秩情况下不会造成损害且可能有所帮助，但在本场景中并非必要。

更大的秩（32 vs 8）提升了指令遵循能力（模型拥有更多容量来精确学习使用哪只手以及与哪些物体交互），但未改善几何一致性或物理合理性。我们推测这是因为几何与物理先验主要由世界模型的冻结权重捕获；LoRA适配器仅需将分布向域内机器人外观与任务结构偏移，这在秩为8时即可实现。

何时使用DoRA vs LoRA：若内存极为紧张或适配器文件大小至关重要，建议从LoRA r=8开始。若预算充足且观察到低秩LoRA训练不稳定，DoRA r=32是合理替代方案，因为其幅度-方向分解有助于稳定学习。
