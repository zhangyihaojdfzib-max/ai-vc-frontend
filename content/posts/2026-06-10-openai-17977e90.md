# 解析Codex Agent循环

作者：Michael Bolin，技术团队成员

Codex CLI⁠（在新窗口打开）是我们的跨平台本地软件Agent，旨在生成高质量、可靠的软件变更，同时在您的机器上安全高效地运行。自今年4月首次推出CLI以来，我们在如何构建世界级软件Agent方面学到了大量经验。为了分享这些见解，这是系列文章的第一篇，我们将探讨Codex工作原理的各个方面，以及来之不易的经验教训。（若想更细致地了解Codex CLI的构建方式，请查看我们的开源代码库https://github.com/openai/codex⁠（在新窗口打开）。如果您想了解更多，我们设计决策的许多细节都记录在GitHub议题和拉取请求中。）

首先，我们将聚焦于Agent循环，这是Codex CLI中的核心逻辑，负责协调用户、模型以及模型为执行有意义的软件工作而调用的工具之间的交互。我们希望这篇文章能让您深入了解我们的Agent（或称“框架”）在利用LLM方面所扮演的角色。

在深入探讨之前，先简要说明一下术语：在OpenAI，“Codex”包含一套软件Agent产品，包括Codex CLI、Codex Cloud和Codex VS Code扩展。本文重点介绍Codex框架，它提供了支撑所有Codex体验并通过Codex CLI呈现的核心Agent循环和执行逻辑。为方便起见，本文将交替使用“Codex”和“Codex CLI”这两个术语。

## Agent循环

每个AI Agent的核心都是所谓的“Agent循环”。Agent循环的简化示意图如下所示：

首先，Agent接收来自用户的输入，将其纳入为模型准备的文本指令集，即所谓的提示词。

下一步是通过向模型发送我们的指令并请求其生成响应来查询模型，这一过程称为推理。在推理过程中，文本提示词首先被转换为一系列输入Token⁠（在新窗口打开）——即索引到模型词汇表中的整数。然后使用这些Token对模型进行采样，生成新的输出Token序列。

输出Token被转换回文本，成为模型的响应。由于Token是逐步生成的，这种转换可以在模型运行的同时进行，这就是许多基于LLM的应用程序显示流式输出的原因。在实践中，推理通常被封装在一个对文本进行操作的API后面，从而抽象掉分词处理的细节。

作为推理步骤的结果，模型要么（1）对用户的原始输入生成最终响应，要么（2）请求Agent执行一个工具调用（例如，“运行ls并报告输出”）。在情况（2）中，Agent执行工具调用，并将其输出附加到原始提示词中。此输出用于生成新的输入，以重新查询模型；然后Agent可以考虑这些新信息并再次尝试。

此过程重复进行，直到模型停止发出工具调用，而是为用户生成一条消息（在OpenAI模型中称为助手消息）。在许多情况下，此消息直接回答用户的原始请求，但也可能是向用户提出的后续问题。

由于Agent可以执行修改本地环境的工具调用，其“输出”并不仅限于助手消息。在许多情况下，软件Agent的主要输出是它在您机器上编写或编辑的代码。尽管如此，每一轮对话总是以一条助手消息结束——例如“我已添加您要求的architecture.md”——这标志着Agent循环的终止状态。从Agent的角度来看，其工作已完成，控制权返回给用户。

图中所示的从用户输入到Agent响应的过程被称为一次对话轮次（在Codex中称为一个线程）。尽管这个对话轮次可能包含模型推理和工具调用之间的多次迭代。每次您向现有对话发送新消息时，对话历史记录都会作为新轮次提示词的一部分包含在内，其中包括之前轮次的消息和工具调用：

这意味着随着对话的增长，用于对模型进行采样的提示词长度也会增加。这个长度很重要，因为每个模型都有一个上下文窗口，即一次推理调用所能使用的最大Token数量。请注意，此窗口同时包含输入和输出Token。正如您可能想象的那样，一个Agent可能决定在单次轮次中发出数百次工具调用，这可能会耗尽上下文窗口。因此，上下文窗口管理是Agent的众多职责之一。现在，让我们深入了解Codex如何运行Agent循环。

## 模型推理

Codex CLI向Responses API⁠（在新窗口打开）发送HTTP请求以运行模型推理。我们将研究信息如何在Codex中流动，Codex使用Responses API来驱动Agent循环。

Codex CLI使用的Responses API端点是可配置的⁠（在新窗口打开），因此它可以与任何实现了Responses API⁠（在新窗口打开）的端点一起使用：

- 当使用ChatGPT登录⁠（在新窗口打开）与Codex CLI时，它使用https://chatgpt.com/backend-api/codex/responses作为端点
- 当使用API密钥认证⁠（在新窗口打开）与OpenAI托管的模型时，它使用https://api.openai.com/v1/responses作为端点
- 当使用`--oss`参数运行Codex CLI以配合`gpt-oss`与ollama 0.13.4+⁠（在新窗口打开）或LM Studio 0.3.39+⁠（在新窗口打开）时，它默认使用您计算机上本地运行的http://localhost:11434/v1/responses
- Codex CLI可以与云提供商（如Azure）托管的Responses API一起使用

让我们探讨Codex如何为对话中的第一次推理调用创建提示词。

### 构建初始提示词

作为最终用户，当您查询Responses API时，您并非逐字指定用于对模型进行采样的提示词。相反，您将各种输入类型作为查询的一部分进行指定，Responses API服务器决定如何将这些信息结构化为模型设计要消费的提示词。您可以将提示词视为一个“项目列表”；本节将解释您的查询如何转换为该列表。

在初始提示词中，列表中的每个项目都与一个角色相关联。角色指示相关内容的权重，其值为以下之一（按优先级降序排列）：system、developer、user、assistant。

Responses API⁠（在新窗口打开）接受一个包含许多参数的JSON负载。我们将重点关注以下三个：

- instructions⁠（在新窗口打开）：插入到模型上下文中的系统（或开发者）消息
- tools⁠（在新窗口打开）：模型在生成响应时可能调用的工具列表
- input⁠（在新窗口打开）：提供给模型的文本、图像或文件输入列表

在Codex中，`instructions`字段从`~/.codex/config.toml`中的`model_instructions_file`⁠（在新窗口打开）读取（如果指定）；否则，使用与模型关联的`base_instructions`⁠（在新窗口打开）。特定于模型的指令位于Codex仓库中，并捆绑到CLI中（例如，`gpt-5.2-codex_prompt.md`⁠（在新窗口打开））。

`tools`字段是一个工具定义列表，这些定义符合Responses API定义的架构。对于Codex，这包括由Codex CLI提供的工具、由Responses API提供并应提供给Codex的工具，以及由用户提供的工具（通常通过MCP服务器）：

#### JavaScript

```
1[2  3  {4    "type": "function",5    "name": "shell",6    "description": "运行一个 shell 命令并返回其输出...",7    "strict": false,8    "parameters": {9      "type": "object",10      "properties": {11        "command": {"type": "array", "description": "要执行的命令", ...},12        "workdir": {"description": "工作目录...", ...},13        "timeout_ms": {"description": "命令的超时时间...", ...},14        ...15      },16      "required": ["command"],17    }18  }1920  21  {22    "type": "function",23    "name": "update_plan",24    "description": "更新任务计划...",25    "strict": false,26    "parameters": {27      "type": "object",28      "properties": {"plan":..., "explanation":...},29      "required": ["plan"]30    }31  },3233  34  {35    "type": "web_search",36    "external_web_access": false37  },3839  40  41  {42    "type": "function",43    "name": "mcp__weather__get-forecast",44    "description": "获取美国某个州的天气警报",45    "strict": false,46    "parameters": {47      "type": "object",48      "properties": {"latitude": {...}, "longitude": {...}},49      "required": ["latitude", "longitude"]50    }51  }52]
```

最后，JSON 负载的 `input` 字段是一个列表。Codex 在将用户消息添加到 `input` 之前，会插入以下项目⁠(在新窗口中打开)：

1. 一条 `role=developer` 的消息，描述了仅适用于 `tools` 部分中定义的、由 Codex 提供的 `shell` 工具的沙箱。也就是说，其他工具（例如从 MCP 服务器提供的工具）不受 Codex 的沙箱限制，需要自行负责实施其护栏。

该消息是根据一个模板构建的，其中关键内容来自捆绑在 Codex CLI 中的 Markdown 片段，例如 `workspace_write.md`⁠(在新窗口中打开) 和 `on_request.md`⁠(在新窗口中打开)：

#### 纯文本

```
1<permissions instructions>2  - 沙箱的描述，解释文件权限和网络访问3  - 关于何时请求用户权限以运行 shell 命令的说明4  - Codex 可写入的文件夹列表（如果有）5</permissions instructions>
```

2. （可选）一条 `role=developer` 的消息，其内容是从用户的 `config.toml` 文件中读取的 `developer_instructions` 值。

3. （可选）一条 `role=user` 的消息，其内容是“用户指令”，这些指令并非来自单个文件，而是聚合自多个来源⁠(在新窗口中打开)。通常，更具体的指令出现在后面：

- `$CODEX_HOME` 中 `AGENTS.override.md` 和 `AGENTS.md` 的内容
- 受限于一个限制（默认为 32 KiB），从 `cwd` 的 Git/项目根目录（如果存在）向上查找到 `cwd` 本身：添加任何 `AGENTS.override.md`、`AGENTS.md` 或 `config.toml` 中 `project_doc_fallback_filenames` 指定的任何文件名的内容
- 如果已配置任何 `skills`⁠(在新窗口中打开)：
  - 关于技能的简短前言
  - 每个技能的 `skill metadata`⁠(在新窗口中打开)
  - 关于如何使用技能的部分⁠(在新窗口中打开)

4. 一条 `role=user` 的消息，描述了 Agent 当前操作的本地环境。这指定了当前工作目录和用户的 shell⁠(在新窗口中打开)：

```
1<environment_context>2  <cwd>/Users/mbolin/code/codex5</cwd>3  <shell>zsh</shell>4</environment_context>
```

一旦 Codex 完成了上述所有计算来初始化 `input`，它就会附加用户消息以开始对话。

前面的示例侧重于每条消息的内容，但请注意，`input` 的每个元素都是一个 JSON 对象，包含 `type`、`role`⁠(在新窗口中打开) 和 `content`，如下所示：

```
1{2  "type": "message",3  "role": "user",4  "content": [5    {6      "type": "input_text",7      "text": "向 README.md 添加一个架构图"8    }9  ]10}

当 OpenAI Responses API 服务器收到请求时，它会使用 JSON 来推导模型的提示词，如下所示（可以肯定的是，Responses API 的自定义实现可能会做出不同的选择）：

如您所见，提示词中前三项的顺序由服务器决定，而非客户端。也就是说，在这三项中，只有 `system` 消息的内容也由服务器控制，因为 `tools` 和 `instructions` 是由客户端决定的。这些之后是来自 JSON 负载的 `input`，以完成提示词。

现在我们有了提示词，就可以对模型进行采样了。

### 第一轮

这个对 Responses API 的 HTTP 请求启动了 Codex 中对话的第一“轮”。服务器以服务器发送事件（SSE⁠(在新窗口中打开)）流的形式回复。每个事件的 `data` 是一个 JSON 负载，其 `"type"` 以 `"response"` 开头，可能如下所示（完整的事件列表可以在我们的 API 文档⁠(在新窗口中打开) 中找到）：

```
1data: {"type":"response.reasoning_summary_text.delta","delta":"ah ", ...}2data: {"type":"response.reasoning_summary_text.delta","delta":"ha!", ...}3data: {"type":"response.reasoning_summary_text.done", "item_id":...}4data: {"type":"response.output_item.added", "item":{...}}5data: {"type":"response.output_text.delta", "delta":"forty-", ...}6data: {"type":"response.output_text.delta", "delta":"two!", ...}7data: {"type":"response.completed","response":{...}}
```

Codex 消费事件流⁠(在新窗口中打开)，并将其重新发布为客户端可以使用的内部事件对象。像 `response.output_text.delta` 这样的事件用于支持 UI 中的流式传输，而像 `response.output_item.added` 这样的其他事件则被转换为对象，并追加到 `input` 中，以供后续的 Responses API 调用使用。

假设对 Responses API 的第一次请求包含了两个 `response.output_item.done` 事件：一个 `type=reasoning`，一个 `type=function_call`。当我们使用对工具调用的响应再次查询模型时，这些事件必须在 JSON 的 `input` 字段中表示：

```
1[2  3  {4    "type": "reasoning",5    "summary": [6      "type": "summary_text",7      "text": "**为 README.md 添加架构图**\n\n我需要..."8    ],9    "encrypted_content": "gAAAAABpaDWNMxMeLw..."10  },11  {12    "type": "function_call",13    "name": "shell",14    "arguments": "{\"command\":\"cat README.md\",\"workdir\":\"/Users/mbolin/code/codex5\"}",15    "call_id": "call_8675309..."16  },17  {18    "type": "function_call_output",19    "call_id": "call_8675309...",20    "output": "<p align=\"center\"><code>npm i -g @openai/codex</code>..."21  }22]
```

### 性能考量

你可能会问自己：“等等，在对话过程中发送给 Responses API 的 JSON 量，Agent（智能体）循环不是呈**二次方**增长吗？” 你的想法是对的。虽然 Responses API 确实支持可选的 `previous_response_id`⁠(在新窗口中打开) 参数来缓解此问题，但 Codex 目前并未使用它，主要是为了保持请求完全**无状态**并支持零数据保留 (ZDR) 配置。

避免使用 `previous_response_id` 简化了 Responses API 提供方的工作，因为它确保了每个请求都是**无状态的**。这也使得支持选择**零数据保留 (ZDR)**⁠(在新窗口中打开) 的客户变得简单直接，因为存储支持 `previous_response_id` 所需的数据会与 ZDR 相冲突。请注意，ZDR 客户不会丧失从先前轮次的专有推理消息中获益的能力，因为相关的 `encrypted_content` 可以在服务器上解密。（OpenAI 会保留 ZDR 客户的解密密钥，但不会保留他们的数据。）请参阅 PRs #642⁠(在新窗口中打开) 和 #1641⁠(在新窗口中打开) 了解 Codex 为支持 ZDR 所做的相关更改。

通常，模型推理的成本主导了网络流量的成本，这使得推理成为我们效率优化的主要目标。这就是提示词缓存如此重要的原因，它使我们能够重用先前推理调用的计算结果。当我们获得缓存命中时，模型推理是**线性**而非二次方的。我们的**提示词缓存**⁠(在新窗口中打开) 文档对此有更详细的解释：

缓存命中仅可能发生在提示词内的精确前缀匹配上。为了实现缓存优势，请将静态内容（如指令和示例）放在提示词的开头，并将可变内容（如用户特定信息）放在末尾。这也适用于图像和工具，它们在请求之间必须完全相同。

考虑到这一点，让我们思考一下哪些类型的操作可能导致 Codex 中出现“缓存未命中”：

- 在对话中途更改模型中可用的 `tools`。
- 更改作为 Responses API 请求目标的 `model`（实际上，这会更改原始提示词中的第三项，因为它包含特定于模型的指令）。
- 更改沙箱配置、审批模式或当前工作目录。

Codex 团队在 Codex CLI 中引入可能影响提示词缓存的新功能时必须非常谨慎。例如，我们最初对 MCP 工具的支持引入了一个**错误，导致我们未能以一致的顺序枚举工具**⁠(在新窗口中打开)，从而导致了缓存未命中。请注意，MCP 工具可能特别棘手，因为 MCP 服务器可以通过 `notifications/tools/list_changed`⁠(在新窗口中打开) 通知动态更改它们提供的工具列表。在长对话中途处理此通知可能会导致代价高昂的缓存未命中。

在可能的情况下，我们通过在 `input` 中**追加一条新消息**来反映配置更改，而不是修改较早的消息，从而处理对话中途发生的配置更改：

- 如果沙箱配置或审批模式发生更改，我们会**插入**⁠(在新窗口中打开) 一条新的 `role=developer` 消息，其格式与原始的 `<permissions instructions>` 项相同。
- 如果当前工作目录发生更改，我们会**插入**⁠(在新窗口中打开) 一条新的 `role=user` 消息，其格式与原始的 `<environment_context>` 项相同。

我们竭尽全力确保缓存命中以获得良好性能。我们还需要管理另一个关键资源：上下文窗口。
