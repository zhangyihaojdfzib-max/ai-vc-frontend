# 边缘设备上的LLM推理：通过React Native在手机上运行LLM的趣味简易指南

随着LLM（大语言模型）不断演进，它们正变得越来越小巧智能，能够直接在手机上运行。以拥有15亿参数的DeepSeek R1 Distil Qwen 2.5为例，这个模型真正展示了先进AI如何能够轻松握于掌中！

在本博客中，我们将指导你创建一个移动应用，让你能够在本地与这些强大的模型进行对话。本教程的完整代码可在我们的EdgeLLM仓库中找到。如果你曾因开源项目的复杂性而感到不知所措，别担心！受Pocket Pal应用的启发，我们将帮助你构建一个简洁的React Native应用，该应用可从Hugging Face中心下载LLM（大语言模型），确保一切数据私密并在你的设备上运行。我们将利用llama.rn（一个针对llama.cpp的绑定）来高效加载GGUF文件！

## 0. 选择合适的模型

在深入构建应用之前，我们先来讨论哪些模型适合在移动设备上运行，以及选择时需要考虑的因素。

### 模型大小考量

在移动设备上运行LLM（大语言模型）时，模型大小至关重要：

- 小型模型（1-3B参数）：适合大多数移动设备，性能良好且延迟极低
- 中型模型（4-7B参数）：在较新的高端设备上运行良好，但在旧手机上可能导致卡顿
- 大型模型（8B+参数）：通常对大多数移动设备来说资源消耗过大，但如果量化为低精度格式（如Q2_K或Q4_K_M）则可以使用

### GGUF量化格式

下载GGUF模型时，你会遇到各种量化格式。了解这些格式有助于你在模型大小和性能之间做出正确选择：

#### 传统量化（Q4_0, Q4_1, Q8_0）

- 基础、直接的量化方法
- 每个块存储包含：• 量化值（压缩后的权重）• 一个（_0）或两个（_1）缩放常数
- 速度快但效率低于新方法 => 现已不广泛使用

#### K量化（Q3_K_S, Q5_K_M, ...）

- 在此PR中引入
- 相比传统量化，位分配更智能
- "K量化"中的"K"指混合量化格式，即某些层获得更多位以提高精度
- 后缀如_XS、_S或_M指特定的量化混合方式（越小=压缩越多），例如：• Q3_K_S对所有张量使用Q3_K• Q3_K_M对attention.wv、attention.wo和feed_forward.w2张量使用Q4_K，其余使用Q3_K• Q3_K_L对attention.wv、attention.wo和feed_forward.w2张量使用Q5_K，其余使用Q5_K

#### I量化（IQ2_XXS, IQ3_S, ...）

- 仍使用基于块的量化，但引入了受QuIP启发的新特性
- 文件体积更小，但在某些硬件上可能较慢
- 最适合计算能力强但内存有限的设备

### 从 Hub 获取可用的 GGUF 模型

首先，我们来定义应用将要支持的模型格式及其存储库。当然，`llama.rn` 是 `llama.cpp` 的绑定，因此我们需要加载 `GGUF` 文件。要找到我们想要支持的模型的 GGUF 存储库，可以使用 Hugging Face 上的搜索栏搜索特定模型的 GGUF 文件，或者使用[此处](https://github.com/ggml-org/llama.cpp/blob/master/scripts/quantize_gguf.py)提供的 `quantize_gguf.py` 脚本自行量化模型，并将文件上传到我们的中心存储库。

```typescript
const modelFormats = [
  {label: 'Llama-3.2-1B-Instruct'},
  {label: 'Qwen2-0.5B-Instruct'},
  {label: 'DeepSeek-R1-Distill-Qwen-1.5B'},
  {label: 'SmolLM2-1.7B-Instruct'},
];

const HF_TO_GGUF = {
    "Llama-3.2-1B-Instruct": "medmekk/Llama-3.2-1B-Instruct.GGUF",
    "DeepSeek-R1-Distill-Qwen-1.5B":
      "medmekk/DeepSeek-R1-Distill-Qwen-1.5B.GGUF",
    "Qwen2-0.5B-Instruct": "medmekk/Qwen2.5-0.5B-Instruct.GGUF",
    "SmolLM2-1.7B-Instruct": "medmekk/SmolLM2-1.7B-Instruct.GGUF",
  };

```

`HF_TO_GGUF` 对象将用户友好的模型名称映射到对应的 Hugging Face 存储库路径。例如：

- 当用户选择 'Llama-3.2-1B-Instruct' 时，它映射到 `medmekk/Llama-3.2-1B-Instruct.GGUF`，这是包含 Llama 3.2 1B Instruct 模型 GGUF 文件的存储库之一。

`modelFormats` 数组包含将在选择界面中显示给用户的模型选项列表，我们选择了 Llama 3.2 1B Instruct、DeepSeek R1 Distill Qwen 1.5B、Qwen 2 0.5B Instruct 和 SmolLM2 1.7B Instruct，因为它们是最流行的小型模型。

接下来，让我们创建一种方法，从中心获取并显示所选模型格式的可用 GGUF 模型文件。

当用户选择一种模型格式时，我们使用在 `HF_TO_GGUF` 对象中映射的存储库路径，向 Hugging Face 发起 API 调用。我们专门查找以 '.gguf' 扩展名结尾的文件，这些文件是我们的量化模型文件。

收到响应后，我们提取这些 GGUF 文件的文件名，并使用 `setAvailableGGUFs` 将它们存储在我们的 `availableGGUFs` 状态中。这使我们能够向用户显示他们可以下载的可用 GGUF 模型变体列表。

```typescript
const fetchAvailableGGUFs = async (modelFormat: string) => {
  if (!modelFormat) {
    Alert.alert('错误', '请先选择模型格式。');
    return;
  }

  try {
    const repoPath = HF_TO_GGUF[modelFormat as keyof typeof HF_TO_GGUF];
    if (!repoPath) {
      throw new Error(
        `未找到模型格式的存储库映射：${modelFormat}`,
      );
    }

    const response = await axios.get(
      `https://huggingface.co/api/models/${repoPath}`,
    );

    if (!response.data?.siblings) {
      throw new Error('API 响应格式无效');
    }

    const files = response.data.siblings.filter((file: {rfilename: string}) =>
      file.rfilename.endsWith('.gguf'),
    );

    setAvailableGGUFs(files.map((file: {rfilename: string}) => file.rfilename));
  } catch (error) {
    const errorMessage =
      error instanceof Error ? error.message : '获取 .gguf 文件失败';
    Alert.alert('错误', errorMessage);
    setAvailableGGUFs([]);
  }
};

```

注意：如果尚未导入，请确保在文件顶部导入 axios 和 Alert。

我们需要测试该函数是否正常工作，让我们在 UI 中添加一个按钮来触发该函数，我们将使用 `SafeAreaView` 组件（稍后会详细介绍）而不是 `View`，并在 `ScrollView` 组件中显示可用的 GGUF 文件。`onPress` 函数在按钮按下时触发：

```typescript
<TouchableOpacity onPress={() => fetchAvailableGGUFs('Llama-3.2-1B-Instruct')}>
  <Text>获取 GGUF 文件</Text>
</TouchableOpacity>
<ScrollView>
  {availableGGUFs.map((file) => (
    <Text key={file}>{file}</Text>
  ))}
</ScrollView>

```

### 模型下载实现

现在，让我们在 `handleDownloadModel` 函数中实现模型下载功能，该函数应在用户点击下载按钮时调用。这将从 Hugging Face 下载选定的 GGUF 文件，并将其存储在应用的 Documents 目录中：

const handleDownloadModel = async (file: string) => {

### 聊天实现

模型已加载到上下文中，我们可以继续实现对话逻辑。我们将定义一个名为 `handleSendMessage` 的函数，该函数将在用户提交输入时触发。此函数会更新对话状态，并通过 `context.completion` 将更新后的对话发送给模型。模型的响应将用于进一步更新对话，这意味着该函数中对话会被更新两次。

```typescript
const handleSendMessage = async () => {
  
  if (!context) {
    Alert.alert('Model Not Loaded', 'Please load the model first.');
    return;
  }

  if (!userInput.trim()) {
    Alert.alert('Input Error', 'Please enter a message.');
    return;
  }

  const newConversation: Message[] = [
    
    ...conversation,
    {role: 'user', content: userInput},
  ];
  setIsGenerating(true);
  
  setConversation(newConversation);
  setUserInput('');

  try {
    
    const stopWords = [
      '</s>',
      '<|end|>',
      'user:',
      'assistant:',
      '<|im_end|>',
      '<|eot_id|>',
      '<|end▁of▁sentence|>',
      '<｜end▁of▁sentence｜>',
    ];
    
    const result = await context.completion({
      messages: newConversation,
      n_predict: 10000,
      stop: stopWords,
    });

    
    if (result && result.text) {
      setConversation(prev => [
        ...prev,
        {role: 'assistant', content: result.text.trim()},
      ]);
    } else {
      throw new Error('No response from the model.');
    }
  } catch (error) {
    
    Alert.alert(
      'Error During Inference',
      error instanceof Error ? error.message : 'An unknown error occurred.',
    );
  } finally {
    setIsGenerating(false);
  }
};

```

### 其他功能

目前应用已具备完整功能：你可以下载模型、选择GGUF文件并与模型对话，但用户体验仍有提升空间。在EdgeLLMPlus代码库中，我添加了其他功能，例如实时生成、自动滚动、推理速度追踪、模型思维过程（如deepseek-qwen-1.5B）等。此处不展开细节以免博客过长，我们将介绍部分实现思路，完整代码均可在代码库中获取。

#### 实时生成

应用采用增量式响应生成方式，每次生成一个Token而非一次性输出完整回复。这种方式通过`context.completion`中的回调函数实现——每生成一个Token即触发回调，从而实时更新对话状态，让用户能边阅读边等待回复生成。

#### 自动滚动

自动滚动功能确保最新消息或Token始终可见：当新内容添加时，聊天视图自动滚动至底部。实现时需通过`ScrollView`的引用进行编程式滚动控制，使用`scrollToEnd`方法在对话状态更新时滚动至底部。同时定义`autoScrollEnabled`状态变量，当用户向上滚动超过100像素时自动禁用该功能。

#### 推理速度追踪

该功能追踪每个Token的生成时间，并在模型生成的每条消息下方显示。实现简单，因为`context.completion`返回的`CompletionResult`对象包含`timings`属性（内含推理过程的多项指标），我们使用`predicted_per_second`指标追踪模型速度。

#### 思维过程

思维过程功能可展示模型（如deepseek-qwen-1.5B）的内部推理。应用通过识别`<thought>`和`</thought>`等特殊Token处理模型推理内容：遇到`<thought>`时进入"思维块"并累积推理Token，检测到`</thought>`后提取推理内容并与消息关联，用户可切换显示模型推理。实现时需在`Message`类型中添加`thought`（存储推理内容）和`showThought`（布尔值，用户点击消息时切换显示）属性。

#### Markdown渲染

应用使用`react-native-markdown-display`包渲染对话中的Markdown内容，支持代码块等格式的优化显示。

#### 模型管理

在`App.tsx`文件中添加`checkDownloadedModels`函数，用于检查模型是否已下载至设备：若未下载则执行下载，若已下载则直接加载至上下文。同时UI中新增元素显示模型下载状态。

#### 停止/返回按钮

UI中新增两个重要按钮：停止按钮用于终止回复生成，返回按钮用于导航至模型选择界面。实现时在`App.tsx`中添加`handleStopGeneration`函数（调用`context.stop`停止生成，并将`isGenerating`状态设为false），以及`handleBack`函数（通过设置`currentPage`状态为`modelSelection`实现页面跳转）。
