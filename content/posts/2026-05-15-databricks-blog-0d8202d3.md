- Lakebase 打破了 OLTP/OLAP 的界限，让团队能够在 Databricks 内部的 Serverless Postgres 界面上运行像 Backstage 这样的生产应用，具备 1 秒数据库分支和低于 4 秒的时间点恢复功能，将有风险的 schema 迁移转变为常规、可测试的操作。
- Unity Catalog 将运营数据库纳入单一治理平面，用一个 SQL 查询取代了碎片化的 CloudTrail/pgaudit/CloudWatch 审计工作流，同时自动将行级安全性和数据脱敏应用于每个临时分支。
- 一个简单的 SQL 连接即可统一基础设施所有权和云账单数据，无需任何 ETL 管道，为 FinOps 和工程负责人提供按分支的成本归属，并将 DBA 的角色从工单队列转变为平台架构师。

在本系列的第一部分中，我们探讨了将 Backstage 的底层数据库迁移到 Databricks Lakebase 如何将有风险的 schema 迁移转变为 1 秒的分支和测试操作。但如果安全和治理团队仍然将你的运营数据库视为黑盒，那么更快的开发周期也只能帮你走到这一步。

在传统技术栈中，你的应用数据库和数据湖存在于两种完全不同的安全范式下。你的基础设施所有权图存在于 Backstage 中，由独立的 RDS 实例支持，并通过复杂的 IAM 角色和 Postgres 原生权限进行治理。与此同时，你的数仓数据由数据团队使用 Unity Catalog 进行治理。Unity Catalog 是 Databricks 创建的一个开源框架，为数据、AI 以及现在的运营数据库提供统一的治理层——一个在平台上管理所有内容的访问控制、审计追踪、数据沿袭和合规性的单一位置。

要审计 RDS 上的一个表删除操作，你需要交叉引用 CloudTrail 以获取 IAM 主体、`pg_stat_activity` 或 `pgaudit` 日志以获取 SQL 语句，以及 CloudWatch 以获取时间戳——三个服务、三种查询语言、三种访问策略。运营数据库变成了合规性的旁路通道。

## 按分支的自动成本归属

治理团队不仅想知道谁创建了分支，还想知道它的成本是多少。

在传统的 AWS 环境中，跟踪临时 RDS 实例的成本需要自定义的 CloudWatch 标记策略，而这些策略通常会遗漏短期工作负载。由于 Lakebase 与 Unity Catalog 的系统账单表原生集成，计算成本会自动按 `project_id`、`branch_id` 和 `endpoint_id` 分解。

在这个 POC 中，生产分支被计费为 31.6130 DBU，而已删除的测试分支被独立归属为 0.0107 DBU。审计追踪和成本追踪在完全相同的地方进行治理。

## 这对每天进行分支操作的团队意味着什么

我们的治理故事回答了合规性问题：我们能否证明谁做了什么、何时做的以及成本是多少？答案是肯定的——一个 SQL 查询代替了三个服务。但对于采用第一部分中分支工作流的开发团队来说，还有另一个同样重要的治理问题：当你的团队在每个冲刺中创建数十个分支时，治理会发生什么？

在第一部分中，我们描述了一个工作流，其中每个功能分支和每个拉取请求都有自己的独立数据库副本。一个由六名开发者组成的团队，运行两周的冲刺，可能在一个冲刺中创建和销毁 30-40 个分支。这意味着 30-40 份生产数据副本，每份都可能包含敏感字段——客户 PII、财务记录、健康数据。

这就是 Unity Catalog 的分支级治理变得至关重要而不仅仅是方便的地方。当创建一个 Lakebase 分支时，Unity Catalog 的属性级脱敏策略会自动传播到新分支。在其功能分支上工作的开发者永远不会看到未脱敏的生产数据——不是因为有人记得配置它，而是因为治理层在创建时就强制执行。运行 PR 测试的 CI 分支与生产环境受到相同的治理。测试人员运行破坏性场景的 QA 分支与生产环境受到相同的治理。不存在因有人忘记应用策略而导致敏感数据泄露的“非生产例外”。

这比看起来更重要。根据 Perforce 的 2025 年数据合规状况报告，60% 的组织在非生产环境中经历过敏感数据未充分匿名化导致的泄露或盗窃。传统方法——在配置开发/测试环境时手动脱敏数据——在环境在几秒钟内创建和销毁时无法扩展。治理必须是自动的，否则就无法实现。

## DBA 的新机遇

审计追踪和成本归属数据也标志着一个更微妙的变化：DBA 的角色正在从被动的工单工作演变为战略性的平台架构。

如今，DBA 的大部分时间都花在运营请求上——环境配置、schema 审查、数据刷新、访问授权。一个六人开发团队每个冲刺可以产生 30 多个工单，DBA 的日程表变成了一个队列。使 DBA 有价值的专业知识——深入理解数据完整性、性能和治理——被埋没在重复的配置工作中。

当分支是自助式的且治理是自动的时，这些重复性工作就会消失。开发者在一秒钟内自行配置环境。Schema 变更在拉取请求中异步审查——DBA 看到 CI 发布的格式化 schema 差异，按照自己的时间表进行审查，并通过正常的 PR 工作流批准或请求更改。现在有了可用的时间，这些审查可以更深入：DBA 帮助团队成员理解生产中的现有数据和结构，与他们合作得出更好的解决方案，并进行彻底的审查以维护数据完整性和治理标准。数据脱敏由策略强制执行，而非手动干预。成本归属是自动的，而非每月的对账工作。

真正发挥DBA专业能力的工作随之展开：定义分支策略、设计治理规则、架构升级工作流、调优性能，以及建立确保自助服务安全性的防护栏。DBA从亲力亲为执行操作转变为设计工作流程——每个迭代周期从处理30多张运维工单减少到不到5次高价值策略评审。上文展示的审计追踪不仅是合规性产物，更是DBA全新的战略仪表盘，能实时查看平台使用情况并确定下一步投入方向。

## 从角色转型到工具支撑

DBA从运维工单转向平台设计的前提是工具必须与角色同步转型。平台需要自主完成常规工作，而DBA则需要一个设计工作流程的空间。

两款开源工具——均以Databricks应用形式部署，并受上述同一Unity Catalog授权和审计追踪体系管控——实现了这一闭环。

**LakebaseOps**负责平台自主运行。三个Agent（智能体）——配置管理、性能优化和健康监控——替代了DBA原先需要提交工单处理的51项任务。其中7个以Databricks定时任务形式运行，取代了DBA原本需要手动维护的pg_cron定时任务。监控界面实时展示pg_stat指标、慢查询回归、分支TTL强制执行情况，以及包含9项KPI的平台采用仪表盘。迁移向导对十种源引擎（Aurora、RDS、Cloud SQL、AlloyDB、Cosmos DB等）进行Lakebase评分，并实时调用AWS和Azure API获取定价信息。

**Lakebase MCP**则是DBA在平台之上进行操作的界面。这是一个模型上下文协议服务器，向任何支持MCP的AI Agent（智能体）（Claude、Copilot、GPT）开放46个工具。DBA不再打开pgAdmin，而是直接描述意图：
