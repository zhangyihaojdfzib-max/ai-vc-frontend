# 自动回程路由如何解决IP地址重叠问题

2026-03-05

- Steve Welham
- Lauren Joplin
- Jackson Kruger
- Thea Heinen

![](/images/posts/ca1613156735.png)

公共互联网依赖于可预测路由的基本原则：单个IP地址指向逻辑上唯一的目的地。即使在像Cloudflare这样的任播架构中，一个IP地址从数百个位置宣告，该IP的每个实例都代表相同的服务。路由表总能精确知道数据包应送往何处。

这一原则之所以成立，是因为全球地址分配机构将IP空间分配给各组织以防止重复或冲突。当每个人都遵循单一权威注册机构时，路由表就能作为绝对真实性的来源。

在公共互联网上，IP地址就像全球注册的唯一国民身份证。在私有网络中，IP地址则如同"张三"这样的名字——当同一个房间里有三个"张三"试图与同一个人交谈时，问题就出现了。

随着我们将Cloudflare One扩展为企业骨干网的连接云，我们已进入私有IP地址空间的混乱现实。重复地址的出现有其合理原因，而企业需要解决方案来处理这些冲突。

今天，我们推出封闭测试版的自动回程路由（ARR）。ARR是Cloudflare One客户的可选工具，它让您能够灵活地将流量路由回其发源地，而无需在路由表中配置IP路由。这一功能允许重叠网络共存，无需任何网络地址转换（NAT）或复杂的虚拟路由和转发（VRF）配置。

### 模糊性问题

在企业网络中，IP地址重叠是常态。我们通常在三种常见场景中遇到这种情况，这些场景传统上给管理员带来大量繁琐工作：

- 并购：两家公司合并，都使用`10.0.1.0/24`作为核心服务。
- 外联网：合作伙伴、供应商或客户使用其内部IP方案安全连接到您的网络，导致不可避免的冲突。
- 标准化架构：SaaS提供商或零售品牌为每个分支机构使用相同的IP空间以简化部署和运营。

当这些站点试图通过Cloudflare与互联网或数据中心通信时，问题就出现了。如果两个不同站点从相同的源IP发送流量，返回的数据包就会遇到架构障碍。管理员必须根据模糊的目的地决定如何路由流量。如果管理员将两条路由都放入路由表，选择哪条路径（正确路径或错误路径）将变得不确定。从标准路由表的角度看，无法区分两条完全相同的路径。

下图显示两个分支机构（站点A和站点B）都使用`10.0.1.0/24`。它们向Cloudflare发送数据包。来自互联网的返回数据包到达Cloudflare边缘，由于路由表有两个相同的出口选项，这些返回流量有时会被发送到错误的站点。

### 传统解决方案为何失效

解决这种模糊性有多种方法，我们致力于以客户最容易管理的方式解决这些问题。传统的"行业标准"解决方案虽然功能有效，但引入了显著的管理开销和复杂性，而我们的目标正是消除这些：

1. 虚拟路由和转发（VRF）：这涉及创建"虚拟"路由表以保持流量隔离。虽然能有效实现分离，但增加了管理开销。管理跨VRF通信（路由泄漏）在大规模部署时既脆弱又复杂。
2. 网络地址转换（NAT）：您可以将每个重叠子网从未管理的IP空间NAT转换到您网络中唯一的受管理IP范围。这种方法效果良好，但每增加一个新站点或合作伙伴都需要进行映射配置，带来管理负担。

通常，我们从客户那里听到的用例是重叠网络需要访问互联网或私有数据中心。我们如何在不增加管理开销的情况下解决这个问题？

### 自动回程路由（ARR）简介

我们开发ARR作为这一问题的"零接触"解决方案。ARR将智能从路由表转移到状态跟踪。

那么什么是状态跟踪？

在传统网络中，路由器是"健忘的"（即"无状态"）。它将每个数据包都视为完全陌生的对象。即使它刚刚在毫秒前看到来自完全相同源地址、前往完全相同目的地的数据包，它也必须重新查看路由表来决定下一个数据包发送到哪里。

通过状态跟踪，系统具有记忆能力。它能识别一系列数据包是否都属于同一个"流"（即两个端点之间的网络对话），并记住该流的关键信息直到对话结束。使用ARR时，我们在初始化流时额外记住一条信息：发起该流的具体隧道。这使我们能够将返回流量发送回同一隧道，而无需查询路由表！

ARR不再询问网络"这个IP地址在哪里？"，而是询问"这个特定对话源自何处？"

逻辑流程：

1. 入口：数据包通过特定连接（如IPsec隧道、GRE隧道或网络互联）从站点到达Cloudflare边缘。
2. 流匹配：Cloudflare虚拟网络首先检查（通过头部检查）该数据包是否匹配现有流。
   - 代理：如果数据包匹配，太好了！关于此流量的所有决策都已制定并存储在我们的内存中。我们只需将数据包沿已建立的路径传递。
   - 流建立：如果不匹配现有流，我们决定将其传递给Cloudflare One堆栈的哪些部分（如网关、DLP、防火墙），以及其最终目的地。我们将所有这些状态存储在内存中。使用ARR时，此时我们会记录哪个隧道发起了该流。
3. 对称返回：当返回流量从目的地到达时，Cloudflare虚拟网络使用其现有的内存状态来代理流量。关键的是，它无需检查流量的目的IP地址（该地址很可能在不同站点间重复使用）就能完成此操作。这完全绕过了查询路由表的需要。我们从流状态中看到发起隧道，并将数据包直接送回该隧道。

1. 代理转发：若数据包匹配成功，则一切顺利！所有关于该流量的决策均已制定并存储于内存中。我们只需将数据包沿已建立的路径传递即可。
2. 流建立：若数据包未匹配现有流，我们将决定其需经过 Cloudflare One 技术栈的哪些组件（例如网关、数据防泄漏、防火墙）以及最终目的地。所有状态信息均存储于内存中。在 ARR（年度经常性收入）机制下，此时我们会记录由哪条隧道发起该数据流。

代理转发：若数据包匹配成功，则一切顺利！所有关于该流量的决策均已制定并存储于内存中。我们只需将数据包沿已建立的路径传递即可。

流建立：若数据包未匹配现有流，我们将决定其需经过 Cloudflare One 技术栈的哪些组件（例如网关、数据防泄漏、防火墙）以及最终目的地。所有状态信息均存储于内存中。在 ARR 机制下，此时我们会记录由哪条隧道发起该数据流。

对称回传：当返回流量从目的地抵达时，Cloudflare 虚拟网络利用其现有的内存状态代理转发流量。关键在于，此过程无需检查流量的目标 IP 地址（该地址很可能在不同站点间重复使用），从而完全绕过了查询路由表的需求。我们通过流状态识别发起隧道，并将数据包直接回传至该隧道。

示例：通过内存流状态追踪重叠源 IP，并标注源接入点以指导回传路由决策。

通过记录每个流的发起隧道，ARR 实现了零接触路由。若您的站点流量仅为客户端到互联网的访问，则完全无需配置回传路由，这大幅减少了部署新分支站点或"咖啡厅网络"时的运维负担。

### 基于统一路由构建

为使 ARR 能在 Cloudflare 的规模下实现，我们将其接入另一个持续研发的项目：统一路由。

历史上，Cloudflare Zero Trust（用户/代理层）与 Cloudflare WAN（网络层/站点层）处于系统的不同层级。Cloudflare WAN 依赖内核原语（Linux 网络命名空间、路由、eBPF 等），而 Zero Trust 则运行在用户空间，使代理能执行深度检测和应用层安全防护。这种"脑裂"架构通常需要复杂逻辑在组件服务间转移流量，部分复杂性甚至演变为客户可能察觉的产品限制。

通过全新的统一路由模式，我们将初始路由决策从网络层数据平面迁移至现有的 Zero Trust 用户空间路由逻辑——这正是 Cloudflare One 客户端和 Cloudflare Tunnel 在 Zero Trust 解决方案中使用的成熟软件。这一变革为客户通过 Cloudflare 平台产品使用私有网络带来多重益处，它修复了 Cloudflare WAN 与 Zero Trust 之间长期存在的互操作性问题。统一路由意味着您可以在同一账户中同时使用 Cloudflare Mesh、Cloudflare Tunnel 及 IPsec/GRE 接入点，且完全避免冲突。

2025 年 9 月，我们在内部为所有 Cloudflare 员工和站点部署了统一路由模式。如上图所示，Cloudflare One 客户端的性能立即提升了 3-5 倍。

在设计 ARR 时，我们深知需要摒弃基于内核的路由方式，并构建于全新的统一路由框架之上。

启用统一路由后，所有 Cloudflare WAN 流量均流经我们的 Zero Trust 枢纽 Apollo。与 Linux 内核标准路由表不同，我们的用户空间数据平面完全可编程。我们可以将元数据（如发起隧道 ID）直接附加到 Apollo 的流条目中。

每个数据包自抵达边缘节点起即被流追踪，我们不再需要为每个数据包独立做出路由决策，而是能在流的整个生命周期内做出一致且感知会话的决策。

### ARR 实践应用

对于企业架构师而言，ARR 是消除 IP 地址冲突持续摩擦的工具。无论是整合收购资产还是接入合作伙伴，目标都是让网络隐形，使您能专注于应用本身而非底层连接。

目前 ARR 处于封闭测试阶段，支持通过安全 Web 网关访问互联网时的重叠 IP 地址场景。我们正在将其扩展至支持私有数据中心访问，增加流中故障转移（将流绑定至主接入点，并在流故障切换至备份接入点时无缝检测），并持续投入架构能力建设，旨在使 IP 重叠问题即使对最复杂的全球部署也不再构成障碍。
