Vercel Workflows 是编写长时间运行、持久化、可靠且可观测的智能体（Agent）与后端服务的最简捷方式。

原型系统与生产就绪系统之间存在巨大鸿沟。在本地运行轻而易举的代码，一旦需要处理故障、重启和真实流量时便会分崩离析。

框架定义的基础设施为 Web 应用解决了这个问题。当你部署时，Vercel 会从应用本身推断出正确的配置。Workflows 将这种模式扩展到了长时间运行的系统。你无需再为编排管理单独的代码库，持久化工作流就是你应用代码的延伸。

自 2025 年 10 月推出测试版以来，Workflows 已为超过 1,500 家客户处理了超过 1 亿次运行和 5 亿个步骤，每周 npm 下载量超过 20 万次。

## 工作原理

将可靠的长时运行流程交付生产，通常意味着需要将代码拆分到队列、工作进程、状态表、重试逻辑和监控中。专用的编排服务又增加了一层，你需要运行在 Kubernetes 中的长生命周期后台进程、进行水平扩展，并投入工程时间以保持其健康运行。它们是你为核心应用计算之上额外付费的分布式系统。

Workflows 彻底消除了编排器。所有协调都在你的应用代码中运行，而非一个单独的服务。其基础设施建立在三个组件之上：

- **事件日志**：记录一次运行中每个步骤的输入、输出、流数据块、休眠、钩子和错误。它是执行状态和历史的唯一事实来源。
- **运行在 Fluid 计算上的函数**：每个步骤都在 Fluid 计算上作为其自身的函数调用运行。每个函数内部的工作流库负责出列、状态加载、加密、执行以及移交到下一步。
- **Vercel Queues**：每个函数会自动将下一步骤加入队列。队列可以在 Vercel 上、在你自己的 Postgres 中或本地内存中自动运行。

**事件日志**：记录一次运行中每个步骤的输入、输出、流数据块、休眠、钩子和错误。它是执行状态和历史的唯一事实来源。

**运行在 Fluid 计算上的函数**：每个步骤都在 Fluid 计算上作为其自身的函数调用运行。每个函数内部的工作流库负责出列、状态加载、加密、执行以及移交到下一步。

**Vercel Queues**：每个函数会自动将下一步骤加入队列。队列可以在 Vercel 上、在你自己的 Postgres 中或本地内存中自动运行。

由于没有单独的编排服务，你只需为函数运行时步骤实际使用的计算资源付费。

### 编程模型：你的代码就是编排器

Workflows 让你能够使用正常的控制流和少量的 API 接口，用 TypeScript 或 Python 编写长时间运行的函数。

在 TypeScript 中，你使用 `"use workflow"` **创建一个工作流**，并使用 `"use step"` 来隔离工作单元。Workflows 会处理底层的一切：队列、重试、步骤隔离、可观测性、持久化状态和流式处理。

```
1export async function createSite(input: { userId: string }) {2  "use workflow"3
4  const profile = await fetchUserProfile(input.userId)5  const plan = await generateSitePlan(profile)6  const site = await buildSite(plan)7
8  return site9}10
11async function fetchUserProfile(userId: string) {12  "use step"13  return db.user.findUnique({ where: { id: userId } })14}15
16async function generateSitePlan(profile: unknown) {17  "use step"18  return callModel({ prompt: `Generate a site plan for ${JSON.stringify(profile)}` })19}20
21async function buildSite(plan: unknown) {22  "use step"23  return provisionSite(plan)24}
```

一个通过三个持久化步骤创建站点的工作流。`"use workflow"` 将函数标记为工作流，`"use step"` 则通过自动重试、持久化和可观测性来隔离每个工作单元。

乍一看，这就像一个函数调用另一个函数，而这正是关键所在。每个步骤都会自动获得隔离、重试、持久化、可观测性和持久化延续能力。编排逻辑存在于应用代码中，而非一个独立的系统。

安装了 Workflow SDK 的 Next.js 应用，在本地运行的方式与在生产环境中大规模运行时完全相同，使用相同的代码，提供真实的保证，且无需配置单独的编排工具。

### Vercel Workflows 实战：Guillermo 的无限象棋游戏

来自 Beta 测试的最佳示例之一是 **Guillermo 的无限象棋游戏**。它持续让模型相互对弈，为它们提供当前棋盘状态，验证走法，渲染游戏，并无限期地运行比赛，一局接一局。

每场象棋比赛都是一次工作流运行，当一场比赛结束时，**最后一步会启动一次新的运行**。无限性被建模为跨运行的递归。

由于每个工作流运行都与特定的部署版本绑定，一场比赛可以安全地在其开始的版本上结束，而下一场比赛则可以在最新的部署版本上开始。这创建了一个清晰的升级边界，每场比赛在其版本内保持稳定，而每场新比赛都能获得最新的改进。

如果支撑象棋比赛的后端代码崩溃或遇到暂时性错误，工作流运行会自动重试，而不会导致应用失败。

## Workflows 对智能体的重要性

Workflows 是为智能体时代量身打造的。它是唯一一个安全优先、持久化的 SDK，专为构建智能体而设计，也专为智能体构建而设计。

### 默认安全

默认情况下，Vercel Workflows 会在数据离开你的部署环境之前，**加密所有数据**，包括步骤输入、输出和流数据块。在传输过程中或在你环境之外的静态存储中，任何内容都不可读，解密只发生在运行工作流的部署环境内部。

加密是内置且免费的，不是你事后配置的安全附加组件。这之所以可能，是因为 Workflows 在同一环境中同时拥有编排和执行能力，因此加密可以自动进行，无需单独的服务或你端的任何额外基础设施。

当你需要通过仪表板或 `workflow` CLI 检查加密数据时，系统支持显式解密，并提供完整的**审计追踪**。

### 持久化流

**持久化流** 可持久保存智能体输出。`getWritable()` 提供一个持久化流，允许多个客户端连接、断开连接、稍后重新连接，并能从任意点恢复。即使用户关闭浏览器，工作流也会持续运行。当用户返回时，客户端重新连接并从流中断处继续，无需 Redis 或自定义发布/订阅系统。

在此示例中，一个航班预订智能体在规划行程和搜索航班时，持续流式传输行程更新：

### 钩子与休眠

工作流可以暂停而无需任何计算开销。

**钩子** 允许工作流等待外部触发器来恢复。钩子对于构建 **人工介入审批流程** 和与第三方服务集成非常有用。

**休眠** 允许工作流暂停任意指定的时间，从几分钟到几天或几个月。休眠适用于电子邮件滴灌营销活动和日期敏感型用例。

### 为多模态智能体构建的容量限制

**工作流** 支持每个步骤负载 50 MB，整个运行过程最高 2 GB，并具有宽松的事件限制。这为智能体在长执行链中传递图像、视频和大上下文提供了充足的余量。
