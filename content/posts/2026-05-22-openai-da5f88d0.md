# 我们如何构建 OWL——ChatGPT 浏览器 Atlas 背后的新架构

揭秘我们全新的进程架构，它为您提供更快、更智能的网络使用方式。

作者：Ken Rockot，技术团队成员；Ben Goodger，ChatGPT Atlas 工程负责人

上周，我们发布了 ChatGPT Atlas，这是一种在 ChatGPT 陪伴下浏览网页的全新方式。除了作为一款功能齐全的网络浏览器外，Atlas 还让我们得以一窥未来：一个你可以带着 ChatGPT 畅游互联网，让它为你提问、提建议、完成任务的世界。在这篇文章中，我们将剖析该产品最复杂的工程方面之一：我们如何将 ChatGPT 打造成一个越用越顺手的浏览器。

让 ChatGPT 成为真正的网络副驾驶，意味着要重新构想浏览器的整个架构：将 Atlas 从 Chromium 运行时中分离出来。这需要开发一种集成 Chromium 的新方法，使我们能够实现产品目标：即时启动、即使打开更多标签页也能保持响应，并为 Agent（智能体）用例奠定坚实基础。

## 奠定基础

Chromium 是一个天然的构建基础。它提供了一个最先进的网络引擎，具有强大的安全模型、久经考验的性能表现和无与伦比的网页兼容性。此外，它由一个全球社区持续开发和改进。它是现代桌面网络浏览器的常见选择。

## 重新构想浏览器体验

我们才华横溢的设计团队为用户体验设定了雄心勃勃的目标，包括为 Agent（智能体）模式等功能提供丰富的动画和视觉效果。这要求我们的工程团队利用最现代的原生框架（SwiftUI、AppKit 和 Metal）来构建我们的 UI，而不是简单地重新包装开源的 Chromium UX。因此，Atlas 的 UI 是对整个应用 UX 的全面重建。

我们还有其他产品目标，例如快速启动时间以及支持数百个标签页而不影响性能。使用开箱即用的 Chromium 很难实现这些目标，因为它在启动顺序、线程模型和标签页模型等许多细节上都有自己固执的见解。我们考虑过在此进行重大修改，但我们希望保持针对 Chromium 的补丁集范围明确，以便能够快速集成新版本。为了确保我们的开发速度最大化，我们需要想出一种不同的方法来集成和驱动 Chromium 运行时。

我们技术投入的试金石不仅是它能让我们更快地实验、迭代和交付新功能——它还能让我们保持 OpenAI 工程文化的核心部分：入职当天即交付。每位新工程师都会在入职第一天的下午完成并合并一个小改动。我们需要确保即使 Chromium 可能需要数小时来检出和构建，这也是可行的。

## 我们的解决方案：OWL

我们对这些挑战的答案是构建一个新的架构层，我们称之为 OWL：OpenAI 的网络层。OWL 是我们对 Chromium 的集成，它涉及在主要的 Atlas 应用进程*之外*运行 Chromium 的浏览器进程。

可以这样理解：Chromium 通过将标签页移动到独立的进程中，彻底改变了浏览器。我们通过将 Chromium 本身移出主应用程序进程并移入一个隔离的服务层，将这一理念推向了更远。这一转变带来了一系列好处：

*   **更简单、更现代的应用：** Atlas 几乎完全使用 SwiftUI 和 AppKit 构建。一种语言，一个技术栈，一个干净的代码库。
*   **更快的启动：** Chromium 在后台异步启动。Atlas 无需等待——像素几乎瞬间呈现在屏幕上。
*   **与卡顿和崩溃隔离：** Chromium 是一个强大而复杂的网络引擎。如果它的主线程挂起，Atlas 不会。如果它崩溃了，Atlas 依然运行。
*   **更少的合并难题：** 因为我们没有在大量的 Chromium 开源 UI 上进行构建，我们与上游 Chromium 的差异要小得多，也更易于维护。
*   **更快的迭代：** 大多数工程师永远不需要在本地构建 Chromium。OWL 作为预构建的二进制文件在内部发布，因此 Atlas 的构建只需几分钟，而不是几小时。

由于我们团队中的大多数工程师并不经常从源代码构建 Chromium，开发速度可以大大加快——即使是新团队成员也可以在入职第一天的下午合并简单的改动。

## OWL 的工作原理

在高层面上，Atlas 浏览器是 OWL 客户端，而 Chromium 浏览器进程是 OWL 主机。它们通过 IPC 进行通信，具体来说是使用 Mojo（在新窗口中打开），即 Chromium 自身的消息传递系统。我们为 Mojo 编写了自定义的 Swift（甚至 TypeScript）绑定，因此我们的 Swift 应用可以直接调用主机端的接口。

OWL 客户端库公开了一个简单的公共 Swift API，该 API 抽象了主机服务层暴露的几个关键概念：

*   **Session（会话）：** 全局配置和控制主机
*   **Profile（配置文件）：** 管理特定用户配置文件的浏览器状态
*   **WebView（网页视图）：** 控制和嵌入单个网页内容（例如渲染、输入、导航、缩放等）
*   **WebContentRenderer（网页内容渲染器）：** 将输入事件转发到 Chromium 的渲染管道，并接收来自渲染器的反馈
*   **LayerHost/Client（图层主机/客户端）：** 在 UI 和 Chromium 之间交换合成信息

还有一系列用于管理高级功能（如书签、下载、扩展和自动填充）的服务端点。

### 渲染：跨越进程边界获取像素

WebView（网页视图）在客户端应用中共享互斥的展示空间，它们被交换进一个共享的合成容器中。例如，一个浏览器窗口通常有一个可见的共享容器，在标签页栏中选择一个标签页会将该标签页的 WebView（网页视图）交换到该容器中。在 Chromium 端，这个容器对应一个 `gfx::AcceleratedWidget`，它最终由一个 `CALayer` 支持。我们将该图层的上下文 ID 暴露给客户端，客户端中的一个 `NSView` 使用私有的 `CALayerHost` API 将其嵌入。

像 `<select>` 下拉菜单或颜色选择器这样的特殊情况，Chromium 会在单独的弹出窗口小部件中渲染，它们使用相同的方法。它们没有 `content::WebContents`，但*确实*有带有自己 `gfx::AcceleratedWidget` 的 `content::RenderWidgetHostView`，因此相同的委托渲染模型适用。

OWL 在内部保持视图几何形状与 Chromium 端同步，以便 GPU 合成器可以相应地进行更新，并始终能够生成正确大小和设备比例的图层内容。

我们还重用此技术，有选择地将 Chromium 自身原生视图 UI 的元素投射到 Atlas 中（这对于快速引导权限提示等功能非常有用，而无需在 SwiftUI 中从头开始构建替代品）。此技术大量借鉴了 Chromium 在 macOS 上用于可安装 Web 应用的现有基础设施。

### 输入事件：解析和转发

Chromium UI 在将平台事件（如 macOS NSEvents）转发给渲染器之前，会将其转换为 Blink 的 WebInputEvent 模型。但由于 OWL 在隐藏进程中运行 Chromium，我们在 Swift 客户端库中自行完成该转换，并将已转换的事件转发给 Chromium。

从那里开始，它们遵循与真实输入事件在网页内容中通常会经历的相同生命周期。这包括每当页面指示它未处理该事件时，将事件*返回*给客户端。当这种情况发生时，我们重新合成一个 NSEvent，并让应用的其余部分有机会处理该输入。

### Agent（智能体）模式：特殊情况

Atlas的代理浏览功能对我们的渲染、输入事件转发和数据存储方法提出了一些独特的挑战。

我们的计算机使用模型期望将单个屏幕图像作为输入。但某些UI元素（如<select>下拉菜单）会在标签页边界之外的单独窗口中渲染。在代理模式下，我们将这些弹出窗口以正确的坐标合成回主页面图像中，使模型能够在一个画面中看到完整的上下文。

对于输入，我们应用相同的原则：代理生成的事件直接路由到渲染器，绝不通过特权浏览器层。即使在自动化控制下，这也保持了沙箱边界。例如，我们不希望这类事件合成键盘快捷键，使浏览器执行与所显示网页内容无关的操作。

代理浏览还可以在临时的“已退出登录”上下文中运行。我们不共享用户现有的隐身配置文件（这可能会泄漏状态），而是使用Chromium的StoragePartition基础设施来启动隔离的内存存储。每个代理会话都从全新状态开始，当会话结束时，所有Cookie和站点数据都会被丢弃。您可以运行多个“已退出登录”的代理会话，每个会话都在自己的浏览器标签页中，并且彼此完全隔离。

## 使用网络的新方式

如果没有全球Chromium社区以及他们为现代网络构建基础的卓越工作，这一切都不可能实现。OWL以一种新的方式建立在这个基础之上：将引擎与应用程序解耦，将世界级的网络平台与现代原生框架融合，并解锁更快、更灵活的架构。

通过重新思考浏览器如何承载Chromium，我们为新型体验创造了空间：更快的启动、更丰富的UI、与操作系统其他部分的更紧密集成，以及以创意速度运行的开发循环。如果这听起来像是您感兴趣的挑战，请查看我们的职位空缺，加入Atlas团队，担任软件工程师、iOS软件工程师等职位。
