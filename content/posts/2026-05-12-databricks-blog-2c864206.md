## Delta Lake 和 Unity Catalog 的演进

当 Delta Lake 被创建时，湖仓一体首先需要在开放的云存储上实现可靠的事务。当时，目录并非为协调现代数据工作负载而设计，因此 Delta 做出了一个革命性的架构选择：它将 ACID 保证直接引入数据湖文件系统。这一基础使湖仓一体成为可能。

随着湖仓一体成为更多团队、引擎和 AI 工作负载的记录系统，跨这些不同资产进行统一治理的需求变得至关重要。Unity Catalog 提供了缺失的治理层：一个跨云、格式和引擎发现、保护、审计和协调数据及 AI 资产访问的单一位置。

Delta Lake 和 Unity Catalog 共同构成了现代湖仓一体的基础。然而，它们是并行运作的——Delta 在存储层管理事务状态，而 Unity Catalog 在目录层管理访问控制。这种架构在早期是足够的，但随着组织在更多引擎和工作负载上扩展规模，这种设计带来了新的协调挑战。

### 当前跨表和跨目录协调的挑战

Delta 最初面向文件系统的架构对于将事务引入数据湖是强大的，但它并非为一个目录必须跨多个引擎一致地协调表标识、访问和状态的世界而设计。随着组织对其数据提出更高要求，缺乏目录协调暴露了三个持续的挑战：

1.  “脑裂”问题：外部引擎直接在对象存储中写入 Delta 表，导致目录元数据（如模式）与实际表状态悄然偏离。
2.  多引擎、多 Agent（智能体）访问蔓延：每个引擎、工具和 Agent（智能体）可以不同方式访问表，导致表发现碎片化、审计不一致，并且跨系统缺乏标准化的行级或列级控制执行。
3.  多表事务协调：开放的湖仓一体架构历来不支持跨多个表的原子写入，因此组织被迫为事务性工作负载维护传统数据仓库。

#### 挑战 #1：“脑裂”问题——保持治理目录和表同步

如今，目录不在 Delta 引擎的读取或写入路径上。因此，如果像 Apache Flink 这样的引擎想要通过直接写入存储层来对表进行模式更改，目录将无法感知这些更改，从而产生“脑裂”状态，即目录元数据和实际表状态发生偏离。这可能导致静默的元数据漂移和下游管道故障。

![挑战 #1：“脑裂”问题——保持治理目录和表同步](/images/posts/ac667e7bfcfe.png)

#### 挑战 #2：多引擎、多 Agent（智能体）访问蔓延

现代组织使用许多引擎和工具来分析数据、构建管道和驱动 AI。历史上，这些系统使用静态路径直接从对象存储访问数据。这使工作负载与物理存储紧密耦合，使得表难以被发现。此外，由于每个引擎直接从通常只支持粗粒度权限的存储层读取 Delta 表，因此跨所有引擎强制执行一致的行/列级治理非常具有挑战性。同样，审计数据访问仍然是碎片化的，因为没有一致的访问层来捕获跨引擎的活动，因此管理员可能对数据实际使用方式有不一致的看法。

组织需要一个中心位置来发现、治理和审计其数据。随着 AI Agent（智能体）成为企业数据的主要消费者，这种需求变得更加迫切。

#### 挑战 #3：跨多个表协调事务

数据仓库工作负载通常需要多表事务，例如原子性地更新 `sales` 和 `inventory` 表，以便下游读取器始终看到一致的视图。然而，Delta Lake 历史上面向文件系统的设计将事务限制在单个表内。因此，即使许多组织希望整合到湖仓一体架构上，他们也不得不专门为这些工作负载维护传统数据仓库。

## Catalog Commits 是开放湖仓一体的下一个演进

Catalog Commits 是 Delta 表与目录集成的开放标准，使目录负责协调表访问和跟踪最新的表状态。现在，Delta 和 Iceberg 都是目录导向的，客户可以依赖其表拥有标准化的表发现和治理模型。要了解有关 Catalog Commits 规范的更多信息，请阅读 Delta 协议，并查看 Unity Catalog 对 Catalog Commits 的参考实现。

在 Databricks 上，可以在 UC 托管的 Delta 表上启用 Catalog Commits。启用后，Unity Catalog 代理所有表访问，为任何引擎创建一致的发现和授权模型。这使得组织能够真正集中治理其数据资产。

![Catalog Commits](/images/posts/5de40550dff5.png)

Catalog Commits 解决了长期存在的脑裂、多引擎蔓延和多表协调挑战。

1.  消除脑裂问题：表状态和目录保持同步，因为所有引擎通过相同的 API 访问表，消除了任何静默元数据漂移的风险。

    解锁外部引擎写入 Unity Catalog 托管 Delta 表的能力

2.  解决多引擎访问蔓延：通过每个引擎和 Agent（智能体）通过标准化的目录 API 解析表，组织不再需要硬编码存储路径或管理粗粒度的文件系统级权限。

    解锁跨所有引擎的一致且增强的治理能力

3.  在湖仓一体上启用传统仓库工作负载：Databricks 引擎和 Unity Catalog 可以协调跨多个表的原子写入。这将多表 ACID 语义引入湖仓一体，解锁了传统数据仓库工作负载。

    解锁在 Databricks 上执行多表事务的能力

除此之外，在 UC 托管表上启用 Catalog Commits 还解锁了：

-   整体可审计性：Unity Catalog 集中了表元数据和访问策略，允许团队通过一致的目录界面检查权限和表所有权，而不仅仅依赖底层存储日志。
-   自动化表优化：Unity Catalog 利用其对所有表访问的可见性，通过 Liquid Clustering 和 Predictive Optimization，针对组织的特定查询模式优化布局其数据。
-   更好性能的基础：Unity Catalog 可以直接将表级元数据告知引擎，而无需引擎从云存储获取元数据，从而消除了元数据延迟的一个主要来源。

总之，这些能力使启用了 Catalog Commits 的 UC 托管表成为现代湖仓一体最开放、治理最完善且性能最优的基础。
