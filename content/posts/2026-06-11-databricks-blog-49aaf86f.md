- ERGO Hestia 借助 Databricks Lakebase 和 Mosaic AI Model Serving 对其实时定价引擎进行了现代化改造，将数据、特征和决策整合到一个湖仓一体原生平台中，实现了毫秒级定价。
- 随着规模扩大，采用外部 PostgreSQL 数据库和自定义适配器层的多跳架构造成了数据提取开销以及跨系统的碎片化治理，在受监管环境中拖慢了创新速度。
- ERGO Hestia 现在能够更快地将新定价模型投入生产，使定价团队能够即时响应市场状况，并通过 Unity Catalog 端到端地证明每一项决策——将定价转变为战略增长引擎，而非 IT 瓶颈。

## 构建下一代实时定价

ERGO Hestia 是波兰领先的保险公司之一，运营着一个支持超过 100 个模型和 1000 个变量的大规模定价平台。作为 Databricks 在波兰最大的用户之一，ERGO Hestia 在先进的毫秒级定价和行业领先的执行速度方面积累了雄厚实力。

然而，由于团队不断追求保险技术的下一次重大创新，他们意识到通过引入实时 B2C 能力可以进一步最大化收入。虽然现有架构功能强大，但向持续模型更新和即时客户响应的转变揭示了一个新挑战：随着复杂性的增加，如何维持创新速度。该团队已经掌握了竞争性定价的艺术，现在正准备开创下一代实时定价交付。

基于紧密且富有成效的合作关系，ERGO Hestia 和 Databricks 共同探讨了如何增强之前的架构以支持下一代实时定价。这一出色的合作促成了平台的演进，利用 Lakebase 提供在线特征存储，并结合 Mosaic AI Model Serving 端点，将所有数据和逻辑保留在 Databricks 生态系统内。该架构将数据和模型服务都保留在湖仓一体内，消除了外部系统并缩短了模型部署时间。通过 Unity Catalog 统一治理，团队整合了数据和模型管理，确保了历史训练集和模型版本的完全可追溯性和长期保留。该架构为定价专家提供了可靠的审计追踪，确保每一项决策在保持模型最佳性能的同时，完全可追溯且可验证。最终，这一转型使团队能够加速创新速度，快速响应市场状况，同时持续改进其最先进的定价模型。

## 挑战：在扩大规模的同时避免摩擦增加

之前的架构遵循一种逻辑模式：Databricks 通过其 medallion 架构摄取和转换定价数据，然后将处理后的数据集导出到外部的 Azure PostgreSQL 数据库。一个中间适配器层负责处理缓存并将数据暴露给定价引擎。当吞吐量适中时，这运行良好。然而，随着数据量增长和模型迭代加速，这种将数据移出湖仓一体、经过外部数据库、再经过缓存层到达应用程序的多跳模式开始限制性能和敏捷性。

![image3.png](/images/posts/2cbccedcc517.png)

1. 运营复杂性：维护外部数据库和自定义适配器层造成了显著的运营开销，因为数据提取作业的巨大维护负担成为了此类关键用例的主要挑战。此外，处理跨系统的碎片化数据治理（包括用于数据处理的 Databricks、用于生产服务的 PostgreSQL，以及用于处理请求逻辑和集成的自定义应用程序代码）意味着血缘追踪变得困难，从而导致合规性和可审计性受损。
2. 模型部署速度：之前的多跳架构要求模型逻辑与底层服务基础设施之间进行仔细同步，这造成了需要协调部署窗口的技术依赖。虽然团队已经具备快速迭代的定价专业知识，但外部数据库和适配器层的复杂性意味着更新通常被安排在非高峰时段以确保系统稳定性。这种专门的编排限制了工作日内更新的频率，以避免外部适配器层的性能风险。
3. 数据新鲜度瓶颈：大量数据摄取造成了性能限制，需要仔细编排以避免影响运营时间。具体来说，这些更新通常会在外部服务层触发 10 到 20 倍的延迟峰值，这实际上将数据刷新限制在定时的批处理窗口内。为了支持向实时 B2C 定价的战略转变，团队需要一个能够全天提供持续数据可用性且无需这些运营权衡的架构。

对于一个在受监管行业中管理着超过 100 个模型和 1000 个变量的组织来说，这种碎片化既造成了运营摩擦，也带来了治理风险。

## 解决方案：在湖仓一体内整合

ERGO Hestia 的转型建立在三个核心技术支柱之上，这些支柱将所有运营统一在湖仓一体内。

用于统一数据服务的 Lakebase。Databricks Lakebase 直接在 Delta 表之上提供了一个关系型事务层。通过使用同步表，团队实现了处理后的数据与服务层之间的持续自动同步。这消除了手动编排和外部数据提取作业的需要。结果是定价数据在湖仓一体内拥有了单一事实来源。

用于直接 API 访问的模型服务端点。模型服务端点不再通过带有单独缓存层和外部数据库的中间应用程序进行路由，而是直接将数据暴露给定价引擎应用程序。这完全消除了适配器层，并确保请求逻辑原生保留在 Databricks 内。请求从定价引擎流向模型服务端点并在毫秒内返回，通过将查询执行和数据服务整合到一个托管层中简化了架构。

用于高速生态系统集成的模型服务。模型首先在 MLflow 中记录，然后获胜版本注册到 Unity Catalog，再通过专用的模型服务端点暴露。该架构扩展了现有的部署能力，提供了一个统一的治理平面，定价专家可以在其中实时针对 Databricks 生态系统中的实时数据验证模型。他们可以同时运行多个模型版本，通过 A/B 测试和回归测试比较性能，同时整个模型生命周期与底层数据源原生连接。这种方法通过内置的血缘追踪和治理控制（涵盖 Unity Catalog 中的数据和模型服务层）确保了完全的可见性。

![image5.png](/images/posts/95b97b1e26b7.png)

Databricks 中现有的 ETL 管道无需更改，数据只需同步到 Lakebase，而无需提取到 PostgreSQL。模型服务可以使用已在 MLflow 中注册、发布到 Unity Catalog 并通过模型服务端点暴露的现有 GLM 和 ML 模型。在 Azure DevOps 中使用了 CI/CD 管道来编排部署。

## 通过增量迁移取得成功：降低转型风险

ERGO Hestia 并未采用统一切换的方式，而是采取了一种分阶段的方法，优先考虑信心和业务连续性。团队从关键性最低的端点开始，在验证性能和稳定性后，再扩展到关键任务系统。在整个过程中，定价运营始终保持不间断。

**第一阶段：概念验证（第1-3周）**

团队使用一个范围较小、定义明确的数据端点进行验证，该端点服务于一个有限的产品组合板块。他们将现有的 Delta Table 同步到 Lakebase，并通过模型服务端点将其暴露出来。Lakebase 的一个决定性优势是其独特的计算与存储分离架构，这使得系统能够轻松扩展以满足峰值请求量，而无需对计算基础设施进行任何手动调整。Databricks 提供了对延迟、CPU 利用率、吞吐量和内存消耗的内置可观测性。性能测试表明，该架构远超预期，即使在每秒40个请求的高负载下，也能实现20毫秒的延迟和低于5%的 CPU 利用率。这一早期的成功在项目扩展到生产环境之前就消除了架构风险。

**第二阶段：生产部署（第3-6周）**

概念验证通过后，团队开始进行生产迁移，从低关键性端点入手。随着端点逐一迁移，每个端点都带来了一致的结果：Delta Lake 和 Lakebase 之间的数据完全一致，性能稳定且可预测，并且所需的定制组件少于传统架构。每次成功的迁移都增强了利益相关者的信心，使团队能够加速向更高风险的端点扩展。

**第三阶段：流量拆分与真实世界验证**

ERGO Hestia 没有接受单一的切换日期，而是在生产规模上实施了流量拆分：一半客户群通过传统系统路由，另一半通过新系统路由。这种方法使得团队能够真实观察实际用户行为和负载模式。新的定价模型生成了稳定的报价，并满足了所有延迟预期。关键的是，团队能够验证新系统完全按照设计处理生产负载。

**这种方法成功的原因：** 增量策略在每一步都消除了风险。低关键性验证防止了灾难性的全系统故障。成功的里程碑建立了组织信心，并实现了更快的扩展。大规模的真实世界生产验证确认了架构按设计运行。在所有阶段中，定价运营始终保持不间断，对客户零影响。

## 业务成果：实现创新速度

**加速模型上市时间：** 定价专家现在可以直接在 Databricks 生态系统中部署模型，这些模型会自动与最新的生产数据同步。通过将模型与 Lakebase 在线特征商店一起通过 Mosaic AI 提供服务，团队消除了将外部模型输出与实时数据流同步时通常出现的延迟。这种集成确保 ERGO Hestia 现有的敏捷性现在与实时市场数据和定价引擎之间的直接高速连接相匹配。运行这些同步模型的能力使得能够快速响应市场状况和竞争性定价压力，最终为组织带来更高的收入。

**运营简化：** 运营复杂性已被架构简洁性所取代。通过移除外部 PostgreSQL 数据库和适配器层，ERGO Hestia 将其技术栈统一在 Unity Catalog 的单一治理模型下。此举消除了补丁和容量规划的手动开销，同时将定价请求逻辑集中到 Databricks 生态系统内。随着逻辑和数据现在共存于一个高性能环境中，工程能力从维护定制中间件转向推动模型创新。

**统一治理与合规：** 在受监管的行业中，能够证明定价决策所依据的具体信息至关重要。通过将所有定价数据和模型统一在 Unity Catalog 下，ERGO Hestia 获得了自动审计追踪、版本跟踪和访问控制。数据一致性现在在整个管道中自动验证，确保测试中使用的逻辑与生产中的逻辑完全匹配。借助内置的平台血缘关系，关于哪个模型在特定时期为哪些客户提供了服务的问题可以立即得到回答，并且完全可审计。这消除了手动合规工作。合规团队现在可以准确证明每个定价决策所依据的数据、谁有权访问以及何时发生了更改。所有内容都被自动捕获，消除了手动重建的需要，同时提供了一个信任基础，使组织能够充满信心地扩展其收入最大化策略。

## 未来之路：在企业内扩展创新

定价部门的成功为跨定价领域的扩展建立了蓝图。ERGO Hestia 现在正在其更广泛的定价组织内扩展 Lakebase 和模型服务架构。每个领域都将在湖仓平台内整合其决策引擎，将模型和数据作为具有统一治理的一等公民服务来提供。

最初使用一个小的、低关键性端点进行的验证证明了该方法的有效性。现在，定价办公室正在使用相同的增量策略扩展到更大、更关键的工作负载和更大的数据库。更高风险的数据集和更复杂的模型需要更严格的验证，但增量迁移模式保持不变。

目标很明确：一旦所有定价领域成功迁移到 Lakebase，就停用外部的 Azure PostgreSQL 数据库。每个领域的迁移都为下一个领域建立了信心，加速了迈向统一的、基于 Databricks 的定价基础设施的整体时间表。

这种经过验证的模式现在为其他业务功能提供了一个模板，以消除外部依赖并加速其自身的决策速度。
