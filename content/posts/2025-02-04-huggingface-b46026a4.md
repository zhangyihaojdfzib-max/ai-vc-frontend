## 引言

罗伯特·海因莱因认为，一个全面发展的人应该能够处理广泛的任务——无论是智力上的还是体力上的——而不是狭隘地专精于某一领域。将全面发展的人与机器智能进行类比：AI系统千差万别，但人类智能在适应性方面表现出色——能够适应各种任务、环境和突发情况。虽然大语言模型和视觉-语言模型（LLM、VLM）展现出潜力，但它们缺乏与物理世界的交互。为弥补这一差距，我们需要在机器人数据上训练的模型。通用机器人模型可以增强适应性，利用多样化数据提升泛化能力和鲁棒性。与在孤立任务上训练不同，在多样化机器人数据上进行预训练——类似于LLM——能够提升效率和性能。

开发通用机器人策略（即机器人基础模型）面临三个关键挑战：

1. 需要大规模研究以充分利用预训练的优势。
2. 设计能够整合多样化数据源同时捕捉复杂物理交互的模型架构。这方面的一个关键挑战是跨本体训练，即模型必须从具有不同配置、控制空间和动作表示的多种机器人类型中学习。现有方法通过以下方式解决这一问题：
   - 结合来自不同机器人平台的多模态数据集以增强泛化能力。
   - 使用共享表示来弥合不同机器人形态（如单臂、双臂和移动操作器）之间的差距。
3. 制定有效的训练方案，因为NLP和视觉领域的最新进展高度依赖于精心设计的预训练和后训练策略。

在本文中，我们介绍由Physical Intelligence开发的π0和π0-FAST原型模型及学习框架，它们旨在克服这些挑战。

## 🔍 什么是π0？

论文 | Jax代码

π0（Pi-Zero）是一种视觉-语言-动作（VLA）模型，由Physical Intelligence团队开发，专为通用机器人控制而设计。它基于大规模预训练和基于流匹配的动作生成，使机器人能够跨不同本体执行灵巧操作任务。

π0在来自7个机器人平台和68个独特任务的数据上进行了训练，在复杂的现实世界任务（如叠衣服、收拾餐桌、装袋杂货、组装盒子和物品取回）中展现出强大的零样本和微调性能。

与标准机器人策略不同，π0采用流匹配来生成50Hz的平滑实时动作轨迹，使其在现实部署中具有高效、精确和适应性强的特点。流匹配曾用于连续归一化流，并提升了扩散模型的生成质量。π0使用的去噪过程以相同方式工作，从随机噪声开始，逐步收敛到有意义的电机动作序列。

## VLM和VLA有什么区别？

视觉-语言模型（VLM）和视觉-语言-动作模型（VLA）共享一个共同基础：Transformer。然而，关键区别在于动作表示。VLM处理并生成多模态表示（图像和文本），而VLA通过引入动作和观测状态Token扩展了这一能力。有了这些额外的Token，下一个挑战是理解注意力是如何计算的。

## 机器人策略中的注意力机制

让我们扩展词汇表并介绍关键术语：

#### 状态Token

- 它是一个单一Token，表示机器人的当前环境状态（例如，关节角度、传感器值或其他相关观测）。
- 掩码规则允许该Token关注前缀中的图像和文本，这意味着状态Token可以“看到”决策所需的任何视觉或文本线索。
- 它还以三角形方式关注先前状态。如果使用多个状态Token，每个新状态Token可以看到旧状态，但反之则不行。

#### 动作Token

- 表示电机命令序列。
- 对除填充区域外的所有内容具有完全可见性。这意味着每个动作Token可以关注：
  - 所有非填充图像Token（整个场景），
  - 所有非填充文本Token（指令或描述），
  - 状态Token（当前和先前），
  - 其他动作Token。

#### 前缀Token

- 表示完整场景，并完全相互关注，类似于PaliGemma。

### 核心思想

这些Token封装了：

- 机器人对环境的内部表示（状态），
- 机器人发出的命令或控制（动作），
- 时间或步骤索引的编码（时间嵌入）。

它们被附加在前缀部分（图像+文本）之后，因此前缀作为上下文（例如，场景图像、语言指令如"做个好机器人"或"转移方块"），而后缀则捕捉策略特定的特征。

## ⚡ 迈向π0中的更快注意力

然而，在π0中高效处理注意力机制也伴随着一系列挑战。其注意力掩码的独特形状会影响注意力的计算方式——让我们深入探讨细节！

### 处理二维注意力掩码

生成的二维因果掩码表现出显著的块稀疏性，但定义每个块的边界——尤其是在一批样本中——有点棘手。我们习惯于自回归建模中具有三角形结构的因果掩码，但这里并非如此。

如下例所示：图像（第一个元素）包含一些填充Token，代表空的摄像头。然后，添加文本Token和状态Token。这个“前缀”部分形成完全非因果注意力，如同PaliGemma。接着，“后缀”（状态+动作/时间Token）具有因果块结构。朴素的急切实现会执行矩阵乘法并对整个输入应用Softmax，导致效率极低。

### 我们可以使用FlashAttention2吗？

- FlashAttention2提供了varlen接口，但cu_seqlens（累积前缀长度）必须手动计算。它专为具有统一查询和键长度的连续（或严格因果）注意力模式而设计。
- 它不能自然处理不规则的块掩码或每个Token的任意“允许”位置，而这正是我们所需要的。
- 因此，虽然以一定的实现成本使用它是可能的，但我们决定转向……

### 在PyTorch中使用FlexAttention

这看起来是FlexAttention的任务！它有一个纯PyTorch接口，我们探索了两种选项：

1. 在注意力被屏蔽的位置向我们的因果掩码添加一个score_mod。然而，即使是标量加法也会显著降低FlexAttention的性能。这是因为在我们的情况下，score_mod是在优化的CUDA内核之外添加的。
2. 正确的选项是对我们的因果掩码进行索引，并将生成的签名传递给create_block_mask。这个块掩码有效地指示了哪些位置需要计算注意力，哪些位置可以完全跳过。

```python

causal_mask = generate_causal_mask(your_samples) 

def precomputed_mask_factory(precomputed_mask: torch.Tensor) -> _mask_mod_signature:
    def mask_mod(b, h, q_idx, kv_idx):
        return precomputed_mask[b][h][q_idx][kv_idx]
    return mask_mod
flex_attention_output = flex_attention(query, key, value, mask_mod=mask_mod)

mask_mod = precomputed_mask_factory(causal_mask)

block_mask = create_block_mask(
    mask_mod=mask_mod,
    
)


attn_output, attention_weights = flex_attention(
    query,
    key,
    value,
    block_mask=block_mask,
)

```

## 如何有效表示动作？

既然我们知道动作只是可以Token化的n维向量，我们就可以探索视觉-语言-动作（VLA）模型中动作表示的挑战。动作的表示方式直接影响效率、泛化能力和执行保真度。

一种方法是语义动作表示，其中动作被描述为高级概念，如子任务或关键点。虽然这允许少样本和零样本学习，但它通常依赖于手工设计的低级控制器，限制了跨不同机器人的灵活性。相比之下，低级控制表示将动作直接映射到电机指令，实现精确运动，但使训练更不稳定且难以扩展。

大多数现有的VLA使用离散动作Token化，将连续动作转换为自回归生成的离散Token。最常见的方法——每个维度、每个时间步的分箱——在高频控制任务中表现不佳，导致有损表示和低效训练。向量量化（VQ）和时间序列压缩等替代方案有所帮助，但VQ对超参数敏感，使其对多样化的机器人设计不太可靠。

为了解决这些问题，频域动作序列Token化（FAST）引入了一种使用离散余弦变换（DCT）的新型时间序列压缩方法。FAST减少了冗余，提高了效率，并增强了动作保真度。

由此，我们推出了π0-FAST，这是π0的一个更快、自回归的版本，也可在Lerobot仓库中找到，它是π0的扩展，利用这种新的Token化器实现更好的动作表示。

## FAST是如何工作的？

FAST使用离散余弦变换（DCT）将连续动作序列压缩为离散Token。如图2所示，该过程首先对原始机器人动作进行归一化，将每个动作维度的第1和第99分位数映射到[-1,1]范围。这种归一化用于确保跨不同机器人系统的一致性，并提高对异常值的鲁棒性。

然后，每个动作维度使用DCT独立变换，将时域信号转换为频域。为了减少冗余，通过缩放和取整操作去除不重要的系数，其中超参数平衡压缩率和重建精度。生成的DCT系数矩阵（通常是稀疏的）被展平为一维整数序列，首先跨维度交错低频分量以保留关键信息。

为了进一步压缩序列，应用了字节对编码（BPE）。像往常一样，BPE合并跨维度的频繁出现模式，同时保持固定大小的词汇表。

![image/png](/images/posts/72f60bfb35ac.png)

图2：FAST动作Token化流程

由于所有操作都是可逆的，动作可以从Token高效且无损地重建。Token化流程只有两个超参数：取整前应用的缩放系数和BPE词汇表大小。这两个参数在不同数据集上保持鲁棒。

此外，FAST的通用版本，称为FAST+，已在来自单臂、双臂和移动操作机器人的一百万个动作序列上训练，使其适用于各种机器人设置。FAST+可作为Hugging Face AutoProcessor使用，允许用户仅用几行代码即可Token化动作序列。

为了获得最佳压缩，输入动作应在Token化之前进行分位数归一化到[-1,1]。使用AutoProcessor模块，用户可以在自己的数据集上训练自定义的FAST Token化器。
