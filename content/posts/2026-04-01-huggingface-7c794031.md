# gradio.Server：使用 Gradio 后端构建任意自定义前端

几周前，我们介绍了如何使用 `gr.HTML` 一次性构建完整的 Web 应用：完全在 Gradio 内部使用自定义的 HTML、CSS 和 JavaScript 构建丰富、交互式的前端。这解锁了许多可能性。但如果这还不够呢？

如果你想**完全使用自己的前端框架（如 React、Svelte，甚至是纯 HTML/JS）进行构建**，同时仍然受益于 Gradio 的队列系统、API 基础设施、MCP 支持以及 Spaces 上的 ZeroGPU，该怎么办？

这正是 `gradio.Server` 要解决的问题。它改变了 Gradio 和 Hugging Face Spaces 的可能性边界。

## 我们想要构建什么

"图后文字"：一个编辑器，你上传一张照片，使用 ML 模型移除背景，然后将样式化的文本放置在前景主体**和**背景**之间**。文本看起来像是位于图像中的人或物体**后面**。

这需要：

*   一个支持**拖放的画布**，具有分层渲染功能（背景 → 文本 → 前景）
*   一个**丰富的控制面板**，带有用于调整字体大小、字重、字间距、颜色、不透明度、描边、阴影、3D 挤出、透视变换等参数的滑块
*   一个**后端 ML 端点**，运行背景移除模型并返回透明的 PNG
*   在客户端**导出为 PNG**

无法用 Gradio 组件来表达这种 UI。这是一个完整的 Web 应用程序。但我们仍然希望拥有 Gradio 的后端能力：队列、并发管理、ZeroGPU 支持，以及在 HF Spaces 上托管而无需担心基础设施问题。

## 引入 `gradio.Server`

`gradio.Server` 扩展了 FastAPI。它赋予你 FastAPI 的全部能力（自定义路由、中间件、文件上传、任何响应类型），同时在其之上添加了 Gradio 的 API 引擎：队列、SSE 流式传输、并发控制和 `gradio_client` 兼容性。

以下是 "图后文字" 的完整后端代码：

```python
import os
import torch
from PIL import Image
from torchvision import transforms
from transformers import AutoModelForImageSegmentation
from gradio import Server
from gradio.data_classes import FileData
from fastapi.responses import HTMLResponse
import spaces

torch.set_float32_matmul_precision("high")

birefnet = AutoModelForImageSegmentation.from_pretrained(
    "ZhengPeng7/BiRefNet", trust_remote_code=True
)
birefnet.to("cuda")
birefnet.float()

transform_image = transforms.Compose([
    transforms.Resize((1024, 1024)),
    transforms.ToTensor(),
    transforms.Normalize([0.485, 0.456, 0.406], [0.229, 0.224, 0.225]),
])

app = Server()


@spaces.GPU
def segment(image: Image.Image) -> Image.Image:
    """Run BiRefNet segmentation to produce a transparency mask."""
    image_size = image.size
    input_images = transform_image(image).unsqueeze(0).to("cuda")
    with torch.no_grad():
        preds = birefnet(input_images)[-1].sigmoid().cpu()
    pred = preds[0].squeeze()
    mask = transforms.ToPILImage()(pred).resize(image_size)
    image.putalpha(mask)
    return image


@app.api(name="remove_background")
def remove_background(image_path: FileData) -> FileData:
    """Remove background from an image. Returns transparent PNG."""
    im = Image.open(image_path["path"]).convert("RGB")
    result = segment(im)
    out_path = image_path["path"].rsplit(".", 1)[0] + ".png"
    result.save(out_path)
    return FileData(path=out_path)


@app.get("/", response_class=HTMLResponse)
async def homepage():
    html_path = os.path.join(os.path.dirname(os.path.abspath(__file__)), "index.html")
    with open(html_path, "r", encoding="utf-8") as f:
        return f.read()


app.launch(show_error=True)

```

就这样。大约 50 行 Python 代码。模型在启动时加载，`@spaces.GPU` 处理 ZeroGPU 分配，`gradio.Server` 管理队列和并发。让我们分解一下发生了什么。

### 为什么用 `@app.api()` 而不是普通的 FastAPI 路由？

如果这是一个普通的 FastAPI 应用，你会为背景移除定义一个 `@app.post()` 路由。这可以工作，直到两个用户同时访问它。没有并发管理，两个请求会争夺 GPU，导致应用崩溃或返回垃圾结果。

`@app.api()` 解决了这个问题。它用 Gradio 的队列引擎包装你的函数：请求被序列化，并发得到控制，并且在 ZeroGPU Spaces 上，GPU 分配通过 `@spaces.GPU` 自动处理。此外，任何 `@app.api()` 端点也可以通过 `gradio_client` 调用，因此其他应用或脚本可以以编程方式使用你的 Space：

```python
from gradio_client import Client, handle_file
client = Client("ysharma/text-behind-image")
result = client.predict(
      image_path=handle_file("photo.jpg"),
      api_name="/remove_background"
  )

```

同时，`@app.get("/")` 是一个标准的 FastAPI 路由，用于提供 HTML 页面。两者自然共存，因为 `Server` **就是**一个 FastAPI 应用。

### 前端：纯 HTML/CSS/JS

本例中的 `index.html` 是一个独立的约 1300 行的 Web 应用程序。没有 React，没有构建步骤，没有打包器。只是纯原生的 HTML，包含：

*   一个**三层画布**：背景图片 → 文本层 → 前景（透明 PNG），使用 CSS `z-index` 堆叠
*   使用指针事件进行**拖放文本定位**
*   一个包含 **20 多个参数的控制面板**：字体系列（25 种以上字体）、大小、字重、间距、颜色、不透明度、背景填充、描边、阴影、3D 挤出深度和角度、旋转、倾斜以及完整的 CSS 透视变换
*   使用 `<canvas>` 合成进行**客户端 PNG 导出**

前端使用 **Gradio JS 客户端** 与后端通信：

```javascript
import { Client, handle_file } from "https://cdn.jsdelivr.net/npm/@gradio/client/dist/index.min.js";

const client = await Client.connect(window.location.origin);
const result = await client.predict("/remove_background", {
    image_path: handle_file(file),
});
foregroundLayer.src = result.data[0].url;  

```

这是关键部分：通过使用 Gradio JS 客户端而不是原始的 `fetch()` 调用，前端会经过 Gradio 的队列。这意味着并发得到管理，GPU 请求不会冲突，你甚至可以向用户显示队列位置或进度。其他所有事情，如文本渲染、图层合成、导出，都在浏览器中完成。

## 这解锁了什么

以下是 **在 `gradio.Server` 之前不可能** 的事情：

有了 `gradio.Server`，Gradio 同时也是一个后端框架，当你需要时使用它的 UI 系统，当你不需要时自带前端。

如果你想要 Gradio 的 UI，你可以使用 `gr.Blocks`、`gr.Interface`、`gr.ChatInterface`。如果你想要自己的 UI，使用 `gradio.Server` 并带上你喜欢的任何前端。无论哪种方式，你都能获得 Spaces 托管、API 队列、`gradio_client` 访问、完整的 HF 生态系统等等。
