# 如何在 Chrome 扩展中使用 Transformers.js

我们最近发布了一个由 Gemma 4 E2B 驱动的 Transformers.js 演示浏览器扩展，帮助用户浏览网页。

在构建过程中，我们遇到了几个关于 Manifest V3 运行时、模型加载和消息传递的实际观察，值得分享。

## 适用人群

本指南面向希望在 Manifest V3 约束下，在 Chrome 扩展中使用 Transformers.js 运行本地 AI 功能的开发者。

阅读完成后，你将掌握该项目中使用的相同架构：一个托管模型的后台 Service Worker、一个侧边栏聊天 UI，以及一个用于页面级操作的 Content Script。

## 我们将构建什么

在本指南中，我们将以已发布的扩展为参考，以开源代码库为实现蓝图，重现 Transformers.js Gemma 4 浏览器助手的核心架构。

- 在线扩展：Chrome 网上应用店

## 1) Chrome 扩展架构（MV3）

在深入之前，先做一个简要说明：我不会深入讲解 React UI 层或 Vite 构建配置。重点在于高层架构决策：每个 Chrome 运行时中运行什么，以及这些部分如何协同工作。

如果你对 Manifest V3 不熟悉，请先阅读这篇简短概述：什么是 Manifest V3？。

### 1.1 运行时上下文和入口点

在 MV3 中，你的架构始于 `public/manifest.json`。该项目定义了三个入口点：

- `background.service_worker` = `background.js`，由 `src/background/background.ts` 构建。
- `side_panel.default_path` = `sidebar.html`，由 `src/sidebar/index.html` 构建。
- `content_scripts[].js` = `content.js`，带有 `matches: http(s)://*/*` 和 `run_at: document_idle`，由 `src/content/content.ts` 构建。

后台 Service Worker 还处理 `chrome.action.onClicked`，以打开当前标签页的侧边栏。
需要了解的相关入口点：可以使用 `action.default_popup` 定义弹出窗口，适用于快速操作。本项目使用侧边栏进行持久聊天，但编排模式是相同的。

### 1.2 各部分运行位置

关键设计决策是将繁重的编排工作放在后台，并保持 UI/页面逻辑精简。

- **后台**（`src/background/background.ts`）是控制平面：Agent 生命周期、模型初始化、工具执行以及共享服务（如特征提取）。
- **侧边栏**（`src/sidebar/*`）是交互层：聊天输入/输出、流式更新和设置控制。
- **Content Script**（`src/content/content.ts`）是页面桥梁：DOM 提取和高亮操作。

这种划分的一个实际结果是，对话历史也保存在后台（`Agent.chatMessages`）：UI 发送 `AGENT_GENERATE_TEXT` 等事件，后台追加消息、运行推理，然后向侧边栏发出 `MESSAGES_UPDATE` 更新。

这种分离避免了重复加载模型，保持了 UI 的响应性，并尊重了 Chrome 围绕 DOM 访问的安全边界。

### 1.3 消息传递契约

一旦运行时分离，消息传递就成为核心。在该项目中，所有消息都通过 `src/shared/types.ts` 中的枚举进行类型化。

- **侧边栏 -> 后台**（BackgroundTasks）：`CHECK_MODELS`、`INITIALIZE_MODELS`、`AGENT_INITIALIZE`、`AGENT_GENERATE_TEXT`、`AGENT_GET_MESSAGES`、`AGENT_CLEAR`、`EXTRACT_FEATURES`
- **后台 -> 侧边栏**（BackgroundMessages）：`DOWNLOAD_PROGRESS`、`MESSAGES_UPDATE`
- **后台 -> Content**（ContentTasks）：`EXTRACT_PAGE_DATA`、`HIGHLIGHT_ELEMENTS`、`CLEAR_HIGHLIGHTS`

编排规则很简单：后台是唯一的协调者；侧边栏和 Content Script 是专门的执行者，负责请求操作并渲染结果。

典型的请求流程：

1. 侧边栏发送 `AGENT_GENERATE_TEXT`。
2. 后台追加到 `Agent.chatMessages` 并运行模型/工具步骤。
3. 后台发出 `MESSAGES_UPDATE`。
4. 侧边栏根据更新后的消息列表重新渲染。

## 2) Transformers.js 集成细节

### 2.1 模型与职责

在 `src/shared/constants.ts` 中，该扩展使用了两个模型角色：

- **文本生成 / LLM**：`onnx-community/gemma-4-E2B-it-ONNX`（文本生成，q4f16）
- **向量嵌入**：`onnx-community/all-MiniLM-L6-v2-ONNX`（特征提取，fp32）

这种分离是有意为之：Gemma 4 负责推理/工具决策，而 MiniLM 为 `ask_website` 和 `find_history` 中的语义相似度搜索生成向量嵌入。

### 2.2 推理运行位置

所有推理都在后台运行（`src/background/background.ts`）：

- 通过 `pipeline("text-generation", ...)` 进行文本生成，并启用我们新的 `DynamicCache` 类实现的一致性 KV 缓存
- 通过 `pipeline("feature-extraction", ...)` 进行嵌入，并进行向量归一化

这为所有标签页/会话提供了单一的模型宿主，避免了重复的内存使用，并保持了侧边栏 UI 的响应性。由于模型是从后台 Service Worker 加载的，工件缓存在扩展源（`chrome-extension://<extension-id>`）下，而不是按网站源缓存，从而为整个扩展安装提供了一个共享缓存。

MV3 生命周期说明：Service Worker 可能被挂起和重启，因此模型运行时状态应被视为可恢复的，并在需要时重新初始化。

### 2.3 下载与缓存生命周期

模型生命周期是明确的：

- `CHECK_MODELS` 检查已缓存的内容并估算剩余下载大小。
- `INITIALIZE_MODELS` 下载/初始化模型，并向 UI 发出 `DOWNLOAD_PROGRESS` 更新。
- 设置完成后，长期存在的实例被重用：`src/background/agent/Agent.ts` 中的生成管道，`src/background/utils/FeatureExtractor.ts` 中的嵌入管道。

权限和隐私是架构的一部分，而不是最后勾选的复选框。在该项目中，`public/manifest.json` 请求了 `sidePanel`、`storage`、`scripting` 和 `tabs`，以及 `http(s)://*/*` 的 `host_permissions`：

- `sidePanel`：需要用于打开和控制侧边栏 UX。
- `storage`：需要用于跨会话持久化工具/设置状态。
- `tabs` + `scripting`：需要用于标签感知工具和页面级操作。
- `http(s)://*/*` 上的 `host_permissions`：需要，因为内容提取/高亮设计用于任意网站。

为什么要保持狭窄：权限定义了用户信任和 Chrome 网上应用店审核风险。只请求你的功能实际需要的权限，并明确说明推理在扩展运行时本地运行，以便用户了解其数据的处理位置。

## 3) Agent 与工具执行循环

### 3.1 工具调用基础（为什么需要这一层）

在执行循环之前，了解模型工具调用如何工作（任何 Agent 工作流的基础）会有所帮助。你传递消息加上工具模式（名称、描述和参数），Transformers.js 使用模型的聊天模板从这些输入中格式化实际的提示词。由于聊天模板是模型特定的，确切的工具调用格式取决于你使用的模型。使用 Gemma-4 风格的模板，当模型决定调用某个工具时，它会发出一个特殊的工具调用 Token 块。

```ts
import { pipeline } from "@huggingface/transformers";

const generator = await pipeline(
  "text-generation",
  "onnx-community/gemma-4-E2B-it-ONNX",
  {
    dtype: "q4f16",
    device: "webgpu",
  },
);

const messages = [{ role: "user", content: "伯尔尼的天气怎么样？" }];
```

```javascript
const output = await generator(messages, {
  max_new_tokens: 128,
  do_sample: false,
  tools: [
    {
      type: "function",
      function: {
        name: "getWeather",
        description: "获取某个位置的天气",
        parameters: {
          type: "object",
          properties: {
            location: {
              type: "string",
              description: "要获取天气的位置",
            },
          },
          required: ["location"],
        },
      },
    },
  ],
});
```

在生成时，模型可能会输出类似以下内容：

```
<|tool_call|>call:getWeather{location:<|"|>Bern<|"|>}<tool_call|>
```

这正是本项目设置归一化层（webMcp）和解析器（extractToolCalls）的原因：模型输出必须转换为确定性的工具执行。

### 3.2 本项目的工具接口

src/background/agent/webMcp.tsx 将扩展工具归一化为模型友好的格式：

- name、description、inputSchema、execute

示例工具包括 get_open_tabs、go_to_tab、open_url、close_tab、find_history、ask_website 和 highlight_website_element。

### 3.3 循环设计（Agent.runAgent）

这里的核心设计选择是将内部模型消息与面向 UI 的聊天消息分离：

- 内部模型记录（messages）：用于 generator(...) 中 system/user/tool/assistant 轮次的消息。
- UI 记录（chatMessages）：用户看到的内容，包括流式输出的助手文本以及工具执行元数据（tools）和性能指标。

执行流程：

1. 将用户输入添加到 chatMessages，创建占位符助手消息，并流式输出 Token。
2. 使用 extractToolCalls.ts 将流式/最终模型输出解析为 { message, toolCalls }。
3. 将用户可见的助手消息保留为纯文本，同时工具调用在后台执行。
4. 将工具结果附加到助手工具元数据中，并将结果作为下一轮提示词输入反馈给模型。
5. 重复执行，直到没有工具调用为止，然后最终确定助手内容及指标。

这保持了用户通信的简洁性，同时在后台维护了确定的工具循环。

## 4) 数据边界与持久化

状态放置是 MV3 中另一个重要的架构决策。在本实现中，状态根据生命周期和访问模式进行拆分：

- 对话状态：后台内存（Agent.chatMessages），用于快速的逐轮编排。
- 工具偏好设置：chrome.storage.local，使设置跨会话持久化。
- 语义历史向量：IndexedDB（VectorHistoryDB），用于更大的本地检索数据。
- 提取的页面内容：后台缓存（WebsiteContentManager），以当前活动 URL 为键。

如第 1.2 节所述，将对话历史保留在后台可在 UI 更新时提供单一规范状态。这使得短期状态驻留在内存中，持久设置存储在扩展存储中，而大量检索数据则存储在本地数据库中。

## 5) 构建与打包说明

您不需要复杂的构建配置，但 MV3 确实要求每个运行时都有可预测的输出。

- 多入口构建在 vite.config.ts 中：src/sidebar/index.html、src/background/background.ts、src/content/content.ts
- 确保输出名称/路径与清单文件对齐（sidebar.html、background.js、content.js）。
- 将内容脚本保持为自包含输出，以避免运行时块加载问题。

- src/sidebar/index.html
- src/background/background.ts
- src/content/content.ts

目标很简单：每个 Chrome 入口点一个产物，且位于 public/manifest.json 所期望的确切位置。

## 最终总结

实现整个项目的架构选择是清晰的关注点分离：后台负责编排和模型执行，UI 层保持精简，内容脚本处理页面访问。

本项目使用侧边面板，但同样的方法也适用于其他设置：

- 弹窗优先的助手：使用 action.default_popup 进行快速交互，后台拥有对话状态和模型执行。
- 侧边面板副驾驶：在持久面板中保持长时间运行的对话，同时后台处理工具循环和缓存。
- 每个标签页的 Agent：当每个标签页应有自己的上下文时，在后台为每个 tabId 维护一个 Agent 状态。
- 混合 UI（弹窗 + 侧边面板 + 选项页面）：所有 UI 入口点与同一个后台协调器通信，并复用相同的消息契约。

实用规则很简单：决定状态存放的位置（全局、tabId 或站点范围），将该状态和模型推理保留在后台（基本上作为后台服务），并让 UI/内容运行时充当专注的客户端。
