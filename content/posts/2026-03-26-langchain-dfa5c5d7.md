## 评估塑造智能体行为

我们一直在精心设计评估方案，以衡量和改进 Deep Agents。Deep Agents 是一个开源、模型无关的智能体框架，为 Fleet 和 Open SWE 等产品提供支持。评估定义并塑造着智能体的行为，这就是为什么深思熟虑地设计它们如此重要。

每一次评估都是一个向量，会改变你智能体系统的行为。例如，如果一个关于高效文件读取的评估失败了，你很可能会调整系统提示词或 `read_file` 工具的描述，以微调行为直到它通过。随着时间的推移，你保留的每一个评估都会对整个系统施加压力。

在添加评估时，深思熟虑至关重要。盲目添加成百上千个测试可能很诱人。这会导致一种“改进智能体”的幻觉，因为在一个可能无法准确反映你在生产环境中关心的行为的评估套件上取得了好成绩。

更多的评估 ≠ 更好的智能体。相反，应该构建有针对性的评估，以反映生产环境中期望的行为。

在构建 Deep Agents 时，我们记录了生产环境中重要的行为，例如在文件系统中跨多个文件检索内容，或准确按顺序组合 5 个以上的工具调用。我们没有使用汇总的基准测试任务，而是采用以下方法来进行评估管理：

1.  决定我们希望智能体遵循哪些行为。然后研究并精心设计有针对性的评估，以可验证的方式衡量这些行为。
2.  对于每个评估，添加一个文档字符串，解释它**如何**衡量智能体的能力。这确保了**每个评估都是自文档化的**。我们还为每个评估打上诸如 `tool_use` 等类别标签，以便进行分组运行。
3.  审查输出轨迹以了解失败模式，并更新评估覆盖范围。

因为我们将每次评估运行都追踪到一个共享的 LangSmith 项目中，所以团队中的任何人都可以介入分析问题、进行修复并重新评估给定评估的价值。这为添加和维护良好的评估创造了共同责任。跨多个评估运行多个模型也可能成本高昂，因此有针对性的评估可以在改进智能体的同时节省资金。

在本博客中，我们将介绍：

*   我们如何管理数据
*   我们如何定义指标
*   我们如何运行评估

## 我们如何管理数据

我们获取评估数据有几种方式：

1.  利用内部使用我们智能体时获得的反馈
2.  从外部基准测试（如 Terminal Bench 2.0 或 BFCL）中选取评估，并经常针对特定智能体进行调整
3.  对于我们认为是重要的行为，手动编写我们自己的（手工制作的）评估和单元测试

### 内部使用智能体与阅读轨迹是评估的绝佳来源

这使得发现错误成为可能。**轨迹**为我们提供了理解智能体行为的数据。因为轨迹通常很大，我们使用内置的智能体（如 Polly 或 Insights）来大规模分析它们。你也可以使用其他智能体（如 Claude Code 或 Deep Agents CLI）加上一种拉取轨迹的方式（如 LangSmith CLI）来做同样的事情。我们的目标是理解每个失败模式，提出修复方案，重新运行智能体，并跟踪随时间的进展和回归。

例如，现在很大一部分错误修复 PR 是通过 Open SWE（我们的开源后台编码智能体）驱动的。使用它的团队会接触到许多具有不同上下文、惯例和目标的代码库。这自然会导致错误。Open SWE 的每一次交互都被追踪，因此这些可以很容易地变成评估，以确保错误不再发生。

其他评估是从现有基准测试（如用于函数调用的 BFCL）中提取并调整的。对于编码任务，我们与 Harbor 集成，在沙盒环境中运行从数据集（如 Terminal Bench 2.0 任务）中选取的任务。许多评估是从头编写的，作为观察孤立行为的针对性测试，例如测试 `read_file` 工具。

## 我们如何定义指标

在为我们的智能体选择模型时，我们首先关注正确性。如果一个模型不能可靠地完成我们关心的任务，其他一切都无关紧要。我们在评估上运行多个模型，并随着时间的推移改进框架以解决我们发现的问题。

衡量正确性取决于测试的内容。大多数内部评估使用自定义断言，例如“智能体是否并行化了工具调用？”。像 BFCL 这样的外部基准测试使用与数据集中真实答案的精确匹配。对于那些正确性是语义性的评估（例如智能体是否在内存中持久化了正确的内容），我们使用 LLM 作为评判者。

一旦有几个模型达到了这个标准，我们就转向效率。在实践中，两个解决相同任务的模型可能表现非常不同。一个可能会进行额外的轮次、进行不必要的工具调用，或者因为模型大小而更慢地完成任务。在生产环境中，这些差异表现为更高的延迟、更高的成本和更差的整体用户体验。

总的来说，我们为每次评估运行测量的指标是：

解决率衡量智能体完成任务的速度，通过预期步骤数进行归一化。与延迟比类似，它捕获了解决问题的端到端时间，包括模型往返、提供商延迟、错误转向和工具执行时间。对于可以定义理想轨迹的简单任务，解决率可能比延迟比更容易处理，因为它只需要测量给定智能体的任务持续时间。

这为我们提供了一种简单的方法，通过有针对性的评估集来选择模型：

1.  首先检查正确性：哪些模型在你实际关心的任务上足够准确？
2.  然后，比较效率：在足够好的模型中，哪一个在正确性、延迟和成本之间提供了最佳权衡？

### 围绕评估的有用指标示例

为了使模型比较具有可操作性，我们检查模型**如何**成功和失败。这需要一个具体的参考点，来定义超出准确率的“良好”执行是什么样子。我们使用的一个基本概念是**理想轨迹**。这是一系列步骤，能产生正确的结果，且没有“不必要”的操作。

对于简单、范围明确的任务，变量定义得足够紧密，最优路径通常是显而易见的。对于更开放式的任务，我们使用迄今为止看到的性能最佳的模型来近似一个轨迹，然后随着模型和框架的改进重新审视基线。通过这种方式，观察智能体行为有助于我们完善关于理想轨迹的先验知识。

考虑一个简单的请求：

一个智能体的理想轨迹可能如下所示：

*   它进行最少数量的必要工具调用（例如，解析用户 → 解析位置 → 获取时间和天气）
*   它在可能的情况下并行化独立的工具调用
*   它生成最终答案，无需不必要的中间轮次

理想轨迹：4 步，4 次工具调用，约 8 秒

低效轨迹：6 步，5 次工具调用，约 14 秒。

正确但低效的轨迹：6 个智能体步骤，5 次工具调用，包含一次不必要的工具调用，并且没有并行化工具调用。

以上示例是说明性的：一个 REPL 可能更快地解决这个特定任务，但更简单的工具调用版本使这个想法更容易解释。

两次运行都是正确的，但第二次运行会增加延迟和成本，并带来更多失败的可能性。

## 我们的评估运行方式

我们使用 pytest 和 GitHub Actions 在 CI 中运行评估，确保变更在干净、可复现的环境中进行。每次评估都会创建一个使用指定模型的 Deep Agent 实例，为其提供任务，并计算正确性和效率指标。

我们也可以通过标签运行评估的子集，以节省成本并衡量针对性实验。例如，如果要构建一个需要大量本地文件处理和合成的 Agent，我们可以专注于标记为`file_operations`和`tool_use`的子集。

```bash
export LANGSMITH_API_KEY="lsv2_..."

uv run pytest tests/evals --eval-category file_operations --eval-category tool_use --model baseten:nvidia/zai-org/GLM-5
```

我们的评估架构和实现已在 **Deep Agents 代码库**中开源。
