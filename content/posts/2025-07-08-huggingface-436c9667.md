## [阶段 4]：使用背包问题更智能地打包

现在，我们准备完全重新思考批处理。填充是敌人，我们需要一种策略来最小化填充，同时最大化每个批次能容纳的数据量。这就是**背包问题**，一个来自计算机科学的经典问题，非常适合这种情况。

想象一下，你正在为徒步旅行打包背包。它只能承受一定的重量，而你想塞进尽可能多的有用物品。在我们的案例中：

-   **背包**是一个具有最大Token限制（`max_length`）的训练批次。
-   每个**物品**是一个序列（一个Token化的提示词-响应对），其**重量**就是Token的数量。
-   我们的目标是在不超过Token限制的情况下，将尽可能多的序列打包到批次中，从而最小化浪费的空间。

为了测试这个想法，我们从一个玩具数据集开始：只是一个从1到25的数字列表，每个数字代表一个序列长度。这让我们可以在没有图像和文本复杂性的情况下进行实验。

### 切换到可迭代数据集

大多数PyTorch数据集是**映射式**的（你可以用`dataset[i]`访问它们）。但对于动态批处理，我们需要更灵活的东西。因此，我们通过子类化`torch.utils.data.IterableDataset`构建了一个**可迭代式**数据集。这让我们可以动态生成批次，并处理诸如在多个工作进程间分片数据等技巧：

```python
def _get_data_range(self):
    worker_info = get_worker_info()
    if worker_info is None:  
        return self.start, self.end
    else:  
        per_worker = int(
            math.ceil((self.end - self.start) / worker_info.num_workers)
        )
        worker_id = worker_info.id
        iter_start = self.start + worker_id * per_worker
        iter_end = min(iter_start + per_worker, self.end)
        return iter_start, iter_end

```

### 生产者-消费者魔法

打包序列可能很慢，特别是如果我们正在排序或打乱顺序。为了保持流程顺畅，我们使用**生产者-消费者**模式，利用Python队列：

```python
def _producer(self, data_iter, queue, stop_signal):
    if self.strategy == "greedy":
        for pack in self._greedy_packing(data_iter):
            queue.put(pack)
    elif self.strategy == "binpack":
        while True:
            buffer = list(itertools.islice(data_iter, self.buffer_size))
            if not buffer:
                break
            knapsacks = self._bin_packing(buffer)
            for pack in knapsacks:
                queue.put(pack)
    queue.put(stop_signal)

```

生产者线程打包批次并将其放入队列，而主线程则在需要时取出它们。这种重叠使流水线顺畅流动。

### 贪心打包

首先，我们尝试一个简单的**贪心打包**策略：

```python
def _greedy_packing(self, iterator):
    pack, pack_sum = [], 0
    for item in iterator:
        if item > self.max_length:
            continue
        if pack_sum + item <= self.max_length:
            pack.append(item)
            pack_sum += item
        else:
            yield pack
            pack = [item]
            pack_sum = item
    if pack:
        yield pack

```

这按顺序遍历数据，将物品添加到包中直到装满，然后开始一个新的包。它很快，但并不完美。以下是批次的样子：

### 装箱打包以获得更紧密的填充

让我们尝试一种更聪明的方法：**装箱打包**（具体来说，是首次适应递减法）：

```python
def _bin_packing(self, buffer: List[int]):
    buffer = sorted(buffer, reverse=True)
    knapsacks = []
    for item in buffer:
        for pack in knapsacks:
            if sum(pack) + item <= self.max_length:
                pack.append(item)
                break
        else:
            knapsacks.append([item])

```

这按长度（最长优先）对序列进行排序，并尝试将每个序列放入第一个有空间的包中。如果没有合适的，就启动一个新包。结果呢？

```bash

[tensor([20]), tensor([19]), tensor([18]), tensor([17]), tensor([16]), tensor([9]), tensor([1])]
[tensor([15]), tensor([14]), tensor([13]), tensor([12]), tensor([11]), tensor([8]), tensor([7]), tensor([6]), tensor([5]), tensor([4]), tensor([3]), tensor([2])]

```

## [阶段5] 多模态数据的背包打包

现在进入正题，将背包打包应用于我们的多模态数据集。

我们回到了图像、提示词和响应，需要在遵守Token限制和图像预算的同时高效地打包它们。图像预算是为了平衡每个样本的图像数量。我们希望避免一个GPU需要处理的图像数量远多于另一个GPU的情况。

我们新的`ConstantLengthDataset`类负责处理繁重的工作。以下是其工作原理，与阶段4相比：

`ConstantLengthDataset`完成所有工作：
- 读取样本（图像和文本）。
- 过滤掉过长或图像过多的样本。
- 使用贪心背包策略将样本打包成批次，平衡Token数量和图像数量。
- 将最终批次填充到固定长度，但填充量比以前少得多。
