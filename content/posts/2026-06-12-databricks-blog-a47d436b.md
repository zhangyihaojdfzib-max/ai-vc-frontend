《进化式数据库设计》中描述并在《重构数据库：进化式数据库设计》中付诸实践的方法论，二十年来一直清晰明确。七项实践、70多个已命名的重构目录、转换机制——所有这些都有文档记录、经过同行评审并得以传授。

该方法论于2010年通过《持续交付》（第12章：管理数据）进入CI/CD领域。迁移成为部署流水线中的一等公民。数据库即代码的规范进入了更广泛的CI/CD运动。但持续交付未能解决的是每条流水线的隔离问题：流水线可以执行迁移，但它们仍然需要一个目标数据库，而这个目标是共享的。实践#4——每个人都拥有自己的数据库实例——在大多数团队中仍停留在理想层面，因为真正为每个开发者提供生产级数据库需要时间、金钱和DBA的工作周期。为弥补这一差距而出现的补偿层（模拟对象、共享预发布环境、内存数据库替代品、DBA工单队列）并非有意设计，而是默认成为了基础方法论。

到2026年，写时复制数据库分支技术将登陆Databricks Lakebase。对TB级生产数据库进行一秒创建、零存储消耗的分支操作，现在是一个O(1)操作。曾经让实践#4停留在理想层面的约束已被解除。

本系列将描述当约束解除后会发生什么变化：不是方法论本身——它依然成立——而是那些首次出现的实践、自动实现的团队级治理、DBA角色的演变，以及Agent（智能体）与其人类同事共享的新基础。

## 认识Jen

Jen是《进化式数据库设计》中的开发者角色。在那篇文章中，她将数据库重构——把`inventory_code`字段拆分为`location_code`、`batch_number`和`serial_number`——作为常规用户故事来实现，展示了DBA和开发者可以协作、模式可以小步演进、迁移可以安全地推动变更。

本系列从二十年后的Jen开始。她遵循的方法论与2003年时相同。新的变化是她工作流程下的基础：写时复制数据库分支，这让她一直阅读的实践在生产规模上成为现实。在本系列的三部分中，她是同一个Jen，但处于三个不同的范围——她的一天（第一部分）、她的新剧本（第二部分）和她的团队（第三部分）。

## 第三部分：Jen的团队规模化

第一部分带领Jen完成了一个功能。第二部分列出了她在2026年遵循的十一项实践剧本。第三部分将同样的剧本应用于一个由五十名开发者组成的团队，Agent（智能体）与人类一起创建分支，并提出问题：在这种规模下，什么变得具有结构性？

有三件事成为关键支撑。

首先，层级拓扑结构，即代表晋升路径中每个环境的长期运行分支。对于一个开发者，你有一个功能分支和生产环境。对于五十个开发者，你有一个结构化的层级体系，包含稳定的通道和在其上叠加的临时通道。

其次，权限模型，即规定谁可以对哪个分支做什么的框架。对于一个开发者，你可以信任约定。对于五十个开发者，并且有Agent（智能体）参与其中，这个框架必须一次性设计好并自动执行。

第三，DBA的角色。对于一个开发者，DBA是Jen在PR上的设计伙伴。对于五十个开发者，DBA是平台工程师，他设计了Jen和她的队友们在其内部工作的框架。

本文涵盖了以上每一点，然后转向Agent（智能体）。基于相同能力的Agent（智能体）是实践#11。Agent（智能体）就像初级开发者：他们能生成可运行的代码、通过的测试、可应用的迁移，但如果没有指导，也会产生不可维护的系统。测试是团队让他们保持诚实的手段。接下来介绍的TDD剧本是团队如何让测试先行的方法。

## 层级作为长期运行的分支，而非独立实例

在分支技术出现之前，一个环境就是一个实例：为预发布环境部署一个专用的Postgres，为UAT部署另一个，为性能测试再部署一个，每个都需要单独配置、打补丁、脱敏和同步。第二部分提到的补偿层也存在于这里。环境之间的漂移是结构性的。

在团队规模下，层级模型坍缩为从同一个Lakebase父节点分出的长期运行分支。

一个分支是以下两者之一：**层级**（长期存在，晋升层级中的父节点）或**功能**（临时存在，从某个层级派生出来，最终被清理）。层级有一个父节点。父节点链就是晋升层级。

![图1：主线及其分支的简单布局](/images/posts/c548243a93cb.png)

在图1中：我们看到一个简单的层级结构，主线是生产环境，功能分支在需要时创建。这种设置通常适用于早期原型开发或非常小团队的早期阶段工作。在拥有更多开发者和/或需要大量环境的成熟团队中，需要如下所示的设置。

![图2：包含最新模式和参考数据的主线及其所有分支的布局](/images/posts/37cb2ea58755.png)

在一些企业中，需要有一个发布候选版本（RC），这个发布候选版本会开发一段时间，在成功测试后晋升到生产环境。图3展示了一种允许开发发布候选版本并随后晋升到生产环境的布局，之后发布候选分支可以被清理。

![图3：使用发布候选版本进行开发和测试的布局](/images/posts/514454115754.png)

分支的名称是任意的，重要的是关于父节点约定如何设置的惯例。可以实现一个策略，不允许违反父节点链层级的转换，以防止直接的功能合并。

策略定义为流水线管理带来了许多好处：

- **一个流水线定义，分支感知**。第二部分引入的`pr.yml`针对每个PR运行；`merge.yml`针对每次晋升运行。同一个工作流覆盖功能、层级以及它们之间的转换。
- **晋升即合并，而非重新部署**。从预发布环境到生产环境的发布是一个git合并操作，其下游效果是Lakebase分支晋升。迁移在每个层级应用一次，并像代码在早期阶段被验证一样，先在上一层级进行验证。
- **“测试环境”与生产环境之间无漂移**。每个层级都从同一个父节点派生。任意两个层级之间的模式差异是真实可计算的：模式是一系列带有分歧标记的页面链，而不是两个独立的数据库软件安装。使团队无需处理需要打补丁和升级的数据库服务器集群。
- **通过重定向实现回滚**。有问题的晋升可以通过将应用程序指向该层级晋升前的快照来恢复。快照本身是另一个分支，使团队能够从错误的部署中恢复。
- **按project_id、branch_id、endpoint_id进行成本归因**。Unity Catalog自动捕获元数据。针对审计和计费表的SQL查询可以回答谁创建了哪个分支、分支何时创建以及保持该分支运行的成本。

大量数据库实例的数量也急剧下降。一个拥有六个环境实例的世界（生产、预发布、UAT、QA、性能、演示）坍缩为一个Lakebase父节点，带有一个由长期运行分支组成的父节点关联层级。曾经需要配置、监控和打补丁的实例，现在变成了一个逻辑分支，具有与生产环境相同的数据形态，受与生产环境相同的策略管理，并且可以在需要时在一秒内重置为生产环境状态。

不同的约定允许你创建许多不同类型的分支作为父节点。一个常见的约定是维护一个包含数据库模式和任何参考数据的分支，这样任何人都可以从中分支，并用测试数据填充它，或运行创建真实数据的自动化测试，从而避免与预发布环境或其他分支发生冲突。

## 现在该做什么：权限模型

实践#10 在第二部分的操作手册中，我们讨论了"一次设计治理，各分支继承"的原则。现在来看看它是如何实现的。

设计工作并非运行时门控，而是一种结构性设计，使通用任务能够据此强制执行。

在团队扩张或引入Agent之前，现在需要做出的决策包括：

- 从各层级创建分支。从生产环境创建分支与从预发布环境或功能分支创建分支的权限不同。默认规则应为：功能分支从入口（底层）层级创建，绝不能从生产环境创建。生产分支仅保留用于热修复和恢复流程。
- 层级间的晋升。"功能到预发布"的晋升属于代码审查范畴；"预发布到生产"的晋升则属于发布协调范畴。两者是独立的门控，由不同的审查人员负责，使业务团队和开发团队能够保持独立性。
- 读取与写入。对分支上生产环境数据的读取权限，与该分支的写入权限是不同的权限。许多工程角色需要读取权限，但只有少数需要写入权限。
- Unity Catalog策略。Unity Catalog策略（如脱敏、行过滤和列级权限）在生产环境中生效。这些策略默认被所有下游分支继承；特定层级的例外情况（例如，用于负载测试的QA层级使用合成PII数据）只需声明一次。
- 审计追踪捕获。每次分支创建、每次晋升、每次迁移应用、每种访问模式，都记录在一个可查询的单一位置。

使这一机制在团队规模下有效运作的原则是：角色声明，策略执行。平台工程师声明层级结构、权限模型、晋升路径和Unity Catalog策略姿态。策略会拒绝任何与声明相矛盾的变更。没有任何地方允许人类或Agent通过以不同形式重试操作来覆盖已声明的边界。

这是现在就需要完成的工作，在团队发展到五十名开发者、Agent创建分支的速度超过任何人审查能力之前。这个框架是将团队凝聚在一起的共同约定和护栏。其他一切都在其内部运作。

## 新角色：从DBA到平台工程师

二十年前，2003年《进化式数据库设计》一文的结尾这样写道：

"使用我们在此描述的技术听起来工作量很大，但实际上并不需要大量人员。在许多项目中，我们有三十多名开发人员，团队规模（包括QA、分析师和管理人员）接近一百人。在任何一天，我们都有大约一百份不同模式的副本分布在员工的工作站上。然而，所有这些活动只需要一名全职DBA和几名了解流程和工作流运作的开发人员。"

这一论点延续到2026年，并得到了五个方面的强化。

1. 比例保持不变，每位DBA有更多空间。每约100人配备一名全职DBA，同时处理一百多个并发分支，每个分支的成本更低，因为分支创建现在是一秒级的元数据操作。关键不在于比例本身，而在于DBA如何利用这些时间。

2. 工作内容向上层栈迁移。2003年用于基础设施配置、模式配置、访问控制和偶尔手动干预的时间，现在转向了分支策略设计、脱敏策略、晋升工作流和审计追踪可观测性。具体的产出物包括：在每个PR上发布模式差异的机器人、每晚重置开发分支的定时任务、追踪分支生命周期和TTL合规性的可观测性仪表盘、基于模式验证来门控合并的CI定义。这是平台设计工作，比以往的工作层次高得多。

3. Agent进入等式。2003年的文章无需处理的是Agent编写代码。Neon报告每天约有五十万个分支，其中超过80%由Agent创建。一个DBA无法通过工单来门控如此大的数量。角色向平台架构师的演变，是唯一能在Agent规模下运作的角色。

4. 数据变得具体。一个六人开发团队在旧模式下每个迭代通常会产生30多个运维工单（配置、模式审查、数据刷新、访问授权）。在分支原生模式下：每个迭代只有不到5个高价值的策略审查。DBA的苦力劳动从每周20多小时降至不到5小时，平均修复时间（MTTR）从4小时以上降至30分钟以内。这种苦力劳动的减少有助于DBA与开发人员协作，为正在开发的功能找到最优解决方案。

5. 审计追踪成为战略仪表盘。过去需要交叉引用三个服务和三种查询语言的工作，现在只需对平台的系统表执行一次SQL查询。每个分支、每个操作、每项成本、每个治理事件都在一个地方。DBA不再需要手动构建这个视图；这是平台提供的。

在《重构数据库》（2006年）的前言中，Martin Fowler希望这本书能代表"仅仅是第一步"，迈向能够像IDE自动化代码重构那样自动化数据库重构的工具。分支化就是那下一步。Fowler在2006年所期望的——以代码的速度进行严谨的数据库变更——正是平台现在所提供的。DBA设计严谨性；平台应用它。

新角色的头衔各不相同（平台工程师、数据库平台负责人、仍称DBA）。实质是一样的：设计框架的人，其他人都在该框架内运作。

## 具备相同能力的Agent

实践#11 在第二部分中，我们描述了"具备相同分支能力的编码Agent即实践者"。现在来讨论这一点。

Agent可以访问分支，但不能访问生产环境。适用于Jen的相同工作流规则也适用于Agent。测试在分支上的真实数据库上运行，而不是在Agent可能修改或删除的模拟对象上运行。无论迁移的作者是谁，模式差异都会出现在每个PR上。保护Jen的策略同样保护Agent。

但仅有策略是不够的。不加引导的Agent就像初级开发人员。

一个初级开发人员，拿到一个功能工单而没有进一步指导，可以产出能编译的代码、能通过的测试和一个能顺利应用的迁移脚本。但代码可能重复了代码库中其他地方已有的逻辑，引入重复。迁移可能添加了一个名称正确但类型错误的列。测试可能通过，因为它只测试了快乐路径。这些失败都不会在绿色的CI运行中显现；它们会在六周后，当其他人需要扩展该工作时才暴露出来。

Agent做同样的事情，但速度更快，数量更大。

没有明确指导的情况下，Agent会：

- 重新发明代码库中已有的模式。
- 编写看起来正确但跳过了命名重构转换机制的Schema变更（例如，没有先移动数据就删除列，或者添加NOT NULL列却没有更新现有行）。
- 编写针对它想象的数据形状（而非生产环境中存在的数据形状）能通过的测试。
- 编写能应用但在回滚时会产生不一致状态的迁移。
- 为了满足一个小改动而层层叠加抽象。

基础架构使绿色构建变得诚实（没有模拟对象；分支上的真实数据库）。但它并不能使代码可维护。

团队通过四个强化措施使代码可维护：

- **护栏**：权限模型。Agent（智能体）无法在生产分支上创建分支，无法在不同层级之间提升，也无法对不属于自己的层级应用迁移。底层系统会拒绝这些操作。
- **模式**：命名重构。2006 年 databaserefactoring.com 的目录列出了 70 多种重构模式，并附有明确的转换机制。被引导执行“应用拆分列重构”的 Agent（智能体），与引导执行“拆分此列”的 Agent（智能体）会产生不同的迁移结果。
- **工作流**：源代码管理（SCM）状态机。Agent（智能体）遵循一系列状态，状态之间设有阻塞门。底层系统会拒绝不满足声明契约的状态转换。
- **审查**：PR 流程中的人工审查。每个 PR 上都会显示 Schema 差异，DBA 参与审查路径。第 2 部分中重新定义的第 1 条实践使其变为异步；在团队规模下，当 Agent（智能体）参与其中时，这种异步审查能够捕捉到测试未能发现的疏漏。

SCM 工作流是核心支撑。在 Lakebase 应用开发工具包中，源代码管理不仅涵盖代码分支：它还包括配对分支（如第 1 部分所介绍的，代码分支和 Lakebase 分支作为一个单元进行管理），这种配对分支作为 Lakebase 应用开发工具包公共底层平台的一项功能提供，强制执行通用的护栏，例如禁止违反层级结构的合并、随分支一起迁移的迁移操作、CI 门控，以及将迁移应用到父层级的合并。该开发工具包将这种 SCM 工作流作为一个可执行状态机提供。

![图 4：SCM 工作流中的各种状态。](/images/posts/daae70bab2d9.png)

图 4：上图描述了开发过程中的五个状态：scaffold-complete（脚手架完成）、feature-claimed（功能认领）、pr-ready（PR 就绪）、ci-green（CI 通过）、merged（已合并）。不同状态之间的每次转换都由 CLI 命令驱动（lakebase-scm-claim-feature-branch、lakebase-scm-prepare-pr、lakebase-scm-wait-ci、lakebase-scm-merge）。每个 CLI 命令在执行工作前会验证前置条件，执行状态转换，并将新状态写入 .lakebase/workflow-state.json（一个经过 Schema 验证的门控接口）。失败的门控会使状态机保持在之前的状态并可恢复。这些门控是阻塞性的，而非建议性的。

Agent（智能体）调用这些 CLI，它们不能发明并行路径。如果前置条件失败，底层系统会拒绝推进状态机：父分支位于错误层级的功能分支会被拒绝；在 CI 通过之前尝试合并会被拒绝；不一致的状态文件会阻塞下一个门控。交接契约由 Scrum Master 角色负责；底层系统强制执行。结构决策（层级结构、功能的源层级、提升路径）属于架构师或 Scrum Master，会被记录下来，然后由底层系统遵守。底层系统从不发明层级或父分支；它遵守已声明的内容，并拒绝与之矛盾的状态转换。

这个框架打破了团队目前使用 Agent（智能体）的方式。天真的集成将 Agent（智能体）视为聊天窗口中的高级工程师：倾倒上下文，要求输出，迭代。这在单个开发者规模下有效，但在团队规模下会失效，因为 Agent（智能体）的“上下文”无法被审查、管理或重放。相反，将 Agent（智能体）视为初级开发者：在可执行状态机内给它一个狭窄、有文档记录的任务，根据 Schema 验证其输出，推进门控，重复。底层系统强制执行；workflow-state 文件就是 API。

## 团队规模实践的剧本条目

### 实践 #10：治理一次设计，按分支继承

规则。权限模型、用于管理访问控制和审计追踪的 Unity Catalog 策略在主分支上设计一次，并由每个后代分支自动继承。

为什么这现在成为一种持久习惯？在团队规模下，治理必须是 DBA 或平台工程师的职能，而不是开发者需要记住的纪律。分支在几秒钟内创建和销毁；每个分支的手动治理配置会消耗掉分支化带来的时间节省。

机制：

-   声明层级结构：存在哪些长期运行的分支，它们的父链接是什么，每个层级承载什么治理姿态。
-   声明权限边界：谁可以在每个层级上创建分支，谁可以在层级之间提升，谁可以读取与写入。
-   声明 Unity Catalog 策略继承：默认情况下，掩码、行过滤器以及哪些列级权限从父级继承；层级特定的异常声明一次。所有 Unity Catalog 策略类型的自动传播即将完成；为目标状态设计框架。
-   声明审计追踪捕获：每次分支创建、每次提升、每次迁移应用、每次访问模式都会自动记录到可查询的系统表中。
-   DBA 或平台工程师通过策略强制执行。与声明模型相矛盾的状态转换会被拒绝。

反模式。在运行时为每个分支配置治理。一次性声明的意义在于，即使分支创建速度快到人类无法审查，框架仍然有效。手动为每个分支配置会重新造成分支化刚刚消除的瓶颈。

Jen 的团队如何扩展。Jen 的平台工程师或 DBA 在项目创建时声明了层级结构。Jen、她的队友或团队 Agent（智能体）创建的每个分支都继承声明的掩码、权限和审计配置。当团队添加新层级时，框架会记录新的父链接；进行中的功能保留其原始父级（不进行追溯性的重新父级化）；新功能从新的入口层级分叉。

### 实践 #11：Agent（智能体）作为实践者，具备相同的分支能力

规则。Agent（智能体）在 SCM 工作流的可执行状态机内运行：五个状态，状态之间设有阻塞门，经过 Schema 验证的状态文件。相同的工作流规则约束 Jen 和 Agent（智能体）；一个公共底层平台强制执行，无论谁在操作。

为什么这现在成为一种持久习惯？分支创建是一种元数据操作，因此 Agent（智能体）驱动的数量是可行的。为 Agent（智能体）开发的底层平台可以拒绝与声明的层级结构或记录的门控状态相矛盾的状态转换。没有聊天窗口上下文可以依赖；只有磁盘上的工件（workflow-state.json）跨越门控转换之间的边界。

-   SCM 状态机有五个状态：scaffold-complete（脚手架完成）、feature-claimed（功能认领）、pr-ready（PR 就绪）、ci-green（CI 通过）、merged（已合并）。每个转换由 CLI 驱动；每个 CLI 在执行工作前验证前置条件。
-   门控接口是 .lakebase/workflow-state.json，根据 scm-workflow-state.schema.json 进行验证。每次转换都会写入新状态以及下一个门控所需的不变量。
-   结构决策（层级结构、每个功能的源层级、提升路径、交接契约）属于特定角色（架构师或 Scrum Master），会被记录下来，然后由底层系统强制执行。
-   Agent（智能体）调用 CLI。底层系统强制执行；Agent（智能体）无法绕过。失败的门控使状态机保持在之前的状态并可恢复；Agent（智能体）不会“以不同形式重试”。
-   在 pr-ready 到 ci-green 阶段运行的 CI 门控会在分支上对真实的 Postgres 进行测试，并在 PR 上发布 Schema 差异。真实的数据库状态是衡量 Agent（智能体）工作的标准。

反模式。将 Agent（智能体）视为聊天窗口中的高级工程师，使用“倾倒上下文并要求输出”的方式，在单个开发者规模下有效，但在团队规模下会失效，因为“上下文”无法被审查、管理或重放。改用工件即 API 模型：Agent（智能体）读取 workflow-state.json 及其阶段的有文档记录的输入；它们写入有文档记录的输出；验证器进行检查；只有当契约成立时，下一个门控才会触发。

Jen 团队的扩展。Jen 团队中的每个 Agent 都在与 Jen 及其队友相同的五状态机内运行。Scrum Master 角色拥有交接合约；底层架构会拒绝不满足这些合约的状态转换。Agent 无法交付从错误层级分叉的功能；Agent 无法在 CI 通过前合并；Agent 无法绕过 Schema 差异产物。无论操作由谁或什么发起，该框架都保持不变。

## TDD 作为可选的叠加层

实践 #11 将 SCM 工作流确立为基线：每个工具包使用者（无论是 Agent 还是人类）都遵循它。TDD 是一个独立的考量，在基线之上叠加，适用于希望采用测试优先纪律的团队。它是可选的；而 SCM 关卡无论路径如何都是强制性的。

为什么测试很重要，甚至在 TDD 之前：当 Agent 编写代码时，测试是唯一可扩展的强制手段。Kent Beck 在 2026 年《Pragmatic Engineer》采访中公开指出了失败模式：他难以阻止 AI Agent 删除测试以使其通过。当循环中没有机制迫使 Agent 面对系统的真实形态时，绿条很容易满足。Mock 让这变得微不足道。内存替代方案也是如此。

分支让绿条在数据层变得诚实。Agent 针对的是真实分支上的真实数据库进行测试；Schema 约束会拒绝不匹配的行插入，外键会拒绝孤立数据，真实数据形态会暴露 Mock 会默默吸收的假设，Agent 无法删除表。伪造合规的成本随着这些护栏而增加。

但底层架构是必要的，而非充分的。测试必须有来源。如果 Agent 编写测试，Agent 也可以删除它们。这就是 TDD 填补的空白。

TDD 工作流叠加在 SCM 工作流之上。它在 SCM 状态 `feature-claimed` 和 `pr-ready` 之间触发；它向下调用 SCM 进行分支操作（循环实验分支在底层使用 SCM 原语）；它不向上调用 SCM。依赖关系是单向的。不希望使用 TDD 层的团队可以通过在功能分支上直接编辑来交付功能，并且仍然满足所有 SCM 关卡。

Lakebase App Dev Kit 将 TDD 工作流作为第二个状态机提供，包含其各自的角色 Agent 和关卡验证器：

- **Spec-author**：将请求者的叙述转化为结构化的功能产物，经过 Schema 验证。
- **Architect-reviewer**：将功能的非功能性需求和架构原则映射到架构决策，输出为 `architecture.json` 及散文描述。
- **Test-strategist**：生成测试列表及每个验收标准的场景，输出为 `test-list.json` 及渲染后的 Markdown。每个 NFR 至少有一个验收标准 (AC)；每个 AC 有一个场景。
- **Scrum-master**：编排构建周期。每个周期分叉一个实验分支（在底层使用 SCM 底层架构），运行一个 Driver Agent 来实现下一个 AC，运行一个 Navigator Agent 来审查并验证结果。
- **Driver 和 Navigator**：是内循环的测试编写者和代码编写者，配对工作，遵循 RED-GREEN-REFACTOR 模式。

每个角色都有文档化的输入和输出，并针对 Schema 进行验证。每个 Agent 只能获取其文档化的输入；输出在下一个角色运行前进行验证。产物是角色之间的 API；Schema 是类型检查。缺失产物被视为关卡失败。格式错误的产物被视为缺失。TDD 层借用了实践 #11 为 SCM 建立的相同“产物即 API”模型。

TDD 层的完整手册位于 Companion: Lakebase App Dev Kit（开源，并附有面向人类实践者的配套电子书）。SCM 和 TDD 状态机、每个角色的 Agent 合约、产物一致性检查以及关卡验证器都以 CLI 形式提供，任何编排器（工具包、IDE 扩展、CI 任务、人类 Shell 会话）都可以调用。

简而言之：SCM 是基线（实践 #11）。TDD 是叠加层。分支让测试变得诚实；TDD 让测试先行；工具包让两种工作流都可执行。
