# 超越LoRA：你能击败最流行的微调技术吗？

![LoRA是最佳的PEFT技术吗？](/images/posts/cdc5d8805e13.png)

# 当你计划以参数高效的方式微调模型时，请将目光投向LoRA之外

如果你想用自己的数据微调一个开源模型，你可能会对所谓的参数高效微调（简称PEFT）感兴趣。这个术语描述了能够显著降低微调模型所需内存的技术。尽管有数十种这样的技术，但几乎所有人都选择了一种叫做"LoRA"的方法。在这篇博客文章中，我们将探讨LoRA是否真的是最佳选择，有哪些工具可以帮助你做出明智的决策，以及如何通过将视野扩展到LoRA之外而受益。

# 什么是PEFT以及何时需要它

市面上有无数开源模型可用，但它们通常不足以满足你的使用场景。提示词可能会有所帮助，但通常还不够。与其从头开始训练一个新模型，你应该考虑微调一个现有模型。

然而，微调非常消耗内存：你通常需要足够的内存来容纳整个模型多次。量化可以减少模型的内存占用，但量化后的模型无法直接微调。因此，出现了一系列用于削减微调所需内存的技术，称为"参数高效微调"（PEFT）。

使用PEFT，你只需使用一小部分内存即可微调模型，甚至可以微调量化后的模型。它还提供了其他优势，例如检查点体积小、更强的抗灾难性遗忘能力，以及能够从同一个基础模型提供多个微调版本。

在Hugging Face，我们开发了PEFT库，它在统一的API背后实现了许多PEFT技术，并与生态系统（例如Transformers和Diffusers）良好集成。它还支持多种量化方法，进一步提高了参数高效微调的可及性。无论你是想用自己的数据微调模型，还是研究新的PEFT方法，PEFT都提供了一个良好的起点。

# LoRA：微调技术中的女王 👑

一种早期出现并被证明相当有效的参数高效微调技术叫做"低秩适应"，简称"LoRA"。它的工作原理是在基础模型之上添加少量参数，冻结基础模型的权重，只训练这些少量参数。

在所有PEFT技术中，LoRA是目前最流行的。以下是一些估算数据：

- 在Hugging Face Hub上提及恰好一种PEFT技术的20,834个模型卡片样本中，有20,509个提到了LoRA（占98.4%）。
- 我们还在一个外部网站上检查了图像生成领域流行的PEFT技术。使用10,000个检查点的样本，我们发现其中7,111个是LoRA。其他被识别的PEFT技术包括LoCon（363个）和DoRA（11个，可视为LoRA的变体）。这意味着95.0%的PEFT检查点是LoRA。
- 在GitHub上搜索代码片段`from peft import <PEFT CONFIG>`（示例GH查询），71.3%的结果是LoRA。紧随其后的是LoHa（3.7%）和AdaLoRA（3.5%）。

尽管这些估算并不完美，但结论仍然是LoRA几乎毫无疑问是最常见的PEFT技术。

这可能仅仅意味着LoRA对所有人都效果最好，这一事实反映在其使用统计数据中。然而，还有另一种可能性：LoRA是较早流行的PEFT技术之一。因此，它的使用可能形成了自我强化：LoRA拥有最高的可见度、最多的教程/示例，并且在下游包中获得最佳支持。因此，LoRA的流行度在自我循环中不断增长。

这一切引出了一个问题：我们是否因为回避更好的技术而都在性能上有所损失？毕竟，有无数研究人员在论文中声称他们的技术击败了LoRA。这难道不足以证明我们应该超越LoRA，转向更新的技术吗？

# 基于论文结果选择正确的PEFT技术存在问题

有数十篇论文研究了除LoRA之外的微调技术。仅在PEFT库中，截至撰写本文时就有超过40种不同的PEFT技术（如果算上PEFT技术的变体则更多）。对于其中几乎所有的技术，你都会发现研究人员声称根据他们的基准测试，他们的技术击败了LoRA。

这些说法的问题在于，研究人员面临着提供能击败现有基准测试结果的压力。即使没有恶意意图，这也可能使结果产生偏差，例如，与研究人员提出的技术相比，在调整替代技术参数上花费的时间更少。例如，一项研究发现，通过调整学习率，LoRA可以匹配据称更好的PEFT技术（https://arxiv.org/abs/2602.04998）。

另一个复杂因素是，每篇论文都选择了不同的PEFT技术集进行比较，以及不同的基准测试集来运行。即使在同一基准测试上比较相同的技术，代码通常也不可用或不易自行运行，这使得结果难以复现。

总的来说，仅通过查看论文结果很难找出最适合你的PEFT技术。因此，你可能会倾向于直接使用默认的LoRA。

# 我们在PEFT中进行基准测试的方法

在Hugging Face，我们思考了如何帮助用户就使用哪种PEFT技术做出明智的决策。通过PEFT库，我们已经提供了一个实现了许多PEFT技术并通过相同API暴露它们的包。下一步是提供能够更清晰地阐明所讨论问题的基准测试。

我们已经有了一段时间的基准测试，用于检查LLM在数学数据集上的微调。该基准测试选取一个LLM，在思维链推理上进行微调，使其能够使用未经过指令微调的基础模型生成数学问题的答案。因此，该基准测试检查模型是否能够学会执行数学推理，并将生成的输出调整为预期的格式。

为了将我们的发现扩展到另一种模态，我们还添加了一个图像生成基准测试。该基准测试测试模型是否能够被微调以学习一个新概念（一个猫玩偶），并在不遗忘现有概念的情况下在新的上下文中生成它。

所有PEFT技术都在完全相同的条件下进行评估：相同的基础模型、相同的数据集、相同的训练和评估代码、相同的硬件。由于不同用户有不同的需求，我们追踪的不仅仅是测试性能。除了VRAM使用量，我们还追踪遗忘/漂移、运行时间和检查点大小等指标。这些结果设计为在消费级硬件上运行，添加新实验只需要添加一个新的PEFT配置并运行一个脚本。

由于我们在同等条件下比较所有PEFT技术，并且没有偏袒任何一方，我们相信这些基准测试能够客观地描绘不同PEFT技术的效果。我们认为，如果你有自己的数据集，你可以采取类似的方法，并利用PEFT库来评估多种PEFT技术。

# 我们的发现：LoRA效果不错，但不一定是最佳选择

完成基准测试运行后，我们发现虽然LoRA效果不错，但其他PEFT方法可以在一个或多个维度上击败它，因此应该予以考虑。查看下图，该图比较了LoRA和其他五种PEFT技术的性能。

![](/images/posts/57ea16cfdc78.png)

对上述结果的一种解读方式是权衡取舍，例如：模型在测试集上的表现与其训练所需的内存之间如何平衡？如果某种PEFT技术在这两项指标上都无法被其他技术同时超越，那么它就处于**帕累托前沿**。换句话说：如果你想要更好的测试准确率，就需要更多内存；如果你想要更高的内存效率，就必须牺牲准确率。

让我们仔细看看LLM数学数据集基准测试的结果。在测试准确率与内存的权衡中，我们发现LoRA确实处于帕累托前沿。它实现了53.2%的测试准确率，峰值需要22.6 GB的显存。然而，还有其他PEFT技术也处于帕累托前沿。例如，**BEFT**实现了32.9%的测试准确率，最大仅需20.2 GB内存。另一端是**Lily**，它实现了54.9%的测试准确率，但需要25.6 GB内存。根据对你而言更重要的因素，你可能会认为LoRA并未提供最佳权衡。

![](/images/posts/76dd6b8b7dae.png)

同样值得注意的是，尽管LoRA在此任务上表现良好，但我们讨论的并非标准LoRA。一方面，我们有**秩稳定初始化**的LoRA，这是一种以不同于默认初始化的方式缩放LoRA贡献的技术，提供了非常好的测试准确率（53.2%）。另一方面，我们有**LoRA-FA**，它使用专门针对LoRA的优化器，冻结了部分LoRA权重，因此内存效率更高（20.2 GB）。普通LoRA在22.5 GB内存下仅达到48.1%的准确率，因此应避免使用，转而选择替代方案。

对于此任务，主要指标是“dino相似度”，它衡量生成图像与保留测试数据集中图像的相似程度，数值越高越好。一如既往，我们还需要关注内存使用情况。在绘制这两项指标的帕累托前沿时，我们发现LoRA位于该前沿之下。让我们看具体数字：LoRA的相似度得分为0.697，而**OFT**得分为0.708；在内存方面，LoRA需要9.97 GB，OFT需要9.01 GB。因此，OFT在这些指标上严格优于LoRA。

当然，你还应检查其他接近帕累托前沿的PEFT方法，因为指标可能因随机性而产生微小变化。此外，你还应探索其他指标：运行时性能对你是否重要？或者你更关心检查点的大小？从下拉菜单中选择相关指标，图表可能会发生显著变化。对于图像生成基准测试，请务必检查生成的样本图像，以感受微调模型的能力。

# 局限性

反对意见：但基准测试偏向于某种方法！

对**PEFT**基准测试可能提出的一个批评是，超参数的选择可能偏向于某种技术。这确实如此，用这么多技术进行详尽且公平的超参数搜索是困难的。然而，每个人都可以轻松地向**PEFT**贡献自己的实验：如果你认为通过选择不同的超参数可以改进某种特定的PEFT技术，请创建一个PR！我们添加了相关说明。同样，如果你想贡献一个全新的基准测试，请联系我们讨论你的想法。

基准测试的另一个问题是，它们可能无法完全反映特定PEFT技术的能力。我们提供了沿多个维度比较这些技术的可能性，并根据这些权衡发现最佳技术。但不可能以这种方式捕捉所有方面。例如，一种名为Cartridges（https://huggingface.co/docs/peft/package_reference/cartridges）的PEFT技术旨在压缩长提示词，这在基准测试中并未衡量。其他因素也可能影响选择，例如：

- 根据PEFT技术的不同，只能修改某些层类型。
- 并非所有PEFT技术都支持量化基础模型（但我们在**PEFT**中积极扩展支持）。
- 某些PEFT技术允许**合并适配器**以减少运行时开销，但其他技术则不允许。

基准测试无法完全替代你自行研究，但它们可以作为合理的参考。

![](/images/posts/13feed7a48f7.png)

反对意见：但 llama.cpp/vLLM/... 只支持 LoRA

使用LoRA以外的PEFT技术的一个局限性是，它们无法获得LoRA在下游包中的广泛支持。例如，如果你想使用vLLM部署模型，只能加载LoRA检查点。幸运的是，**PEFT**现在支持将其他适配器转换为LoRA。这样，你可以将非LoRA检查点转换为LoRA，并在vLLM或其他下游包中使用。

为了测试这一点，我们使用GraLoRA技术将一个图像适配器转换为LoRA检查点。转换后的测试分数几乎相同（相似度0.702 → 0.694，0.260 → 0.269）。以下是提示词“sks cat at the beach”的测试图像：

# 结论与你可以做什么

在开发**PEFT**包的过程中，我们注意到LoRA拥有很大的发展势头，尽管其他PEFT技术可能更优。因此，我们着手在PEFT中添加基准测试，以便更客观地展示不同PEFT技术在不同指标上的表现。

根据我们得到的结果，我们可以自信地得出结论：LoRA绝非一个糟糕的选择，但可能存在更优的选择。特别是在检查图像生成基准测试时，LoRA被其他技术击败。我们讨论了在选择合适的PEFT技术时，除了指标外，还必须考虑其他因素。然而，即便如此，我们仍在推动**PEFT**进一步发展，以实现LoRA与这些其他技术的功能对等。

我们的旅程远未结束，我们希望扩展和改进现有的基准测试，并计划在未来添加更多基准测试。我们确保社区能够轻松贡献，因此如果你有兴趣，请在**PEFT**仓库中提交一个issue，告诉我们你希望如何贡献。

如果你从本文中只记住一件事，那就是在为你的用例选择PEFT技术时，不应自动默认选择LoRA。鉴于**PEFT**提供的统一API，从一种PEFT技术切换到另一种就像在代码中切换一个配置一样简单。即使你坚持使用LoRA，也请查看**PEFT**中支持的所有变体：DoRA、rs-LoRA、LoRA-FA等。尝试这些其他技术，你可能会感到惊喜。

示例：使用 `PEFT` 从 LoRA 切换到 OFT

```diff

-from peft import LoraConfig, get_peft_model
+from peft import OFTConfig, get_peft_model

base_model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3.2-3B", dtype="bfloat16")
-config = LoraConfig(target_modules=["q_proj", "v_proj"])
+config = OFTConfig(target_modules=["q_proj", "v_proj"])
model = get_peft_model(base_model, config)
```
