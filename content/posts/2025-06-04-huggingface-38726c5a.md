## 引言

自回归语言模型通过**每次采样一个Token**的方式生成文本。在推理过程中，模型处理给定的输入序列，预测下一个Token，将其追加到序列中，并重复此过程直至满足停止条件：

![自回归示意图](/images/posts/a528df945dc8.png)

这种逐步生成过程本质上是顺序进行的：

- 为生成Token \( t_{i+1} \)，模型必须考虑从 \( t_0 \) 到 \( t_i \) 的整个序列。以上图为例，\( t_{i+1} \) 将是 `the`，而所有先前Token \( t_0 \) 到 \( t_i \) 则是 `[What, is, in]`。
- 尽管Transformer内部是并行的，但每个新预测都需要通过所有Transformer层进行一次完整的前向传播，这会带来随序列长度呈二次增长的内存/计算开销。

这种重复还导致了**计算冗余**。本文将探讨**KV缓存**——一种优化技术，旨在缓解这种低效问题。

## 重温Transformer架构

在深入缓存机制前，让我们先回顾注意力机制在Transformer模型中的运作方式。Transformer语言模型由堆叠的层组成，每层包含：

- 多头自注意力
- 前馈网络（MLP）
- 残差连接与层归一化

为理解**KV缓存的作用点**，我们聚焦于**自注意力机制**，特别是单个注意力头内的计算。

让我们通过一个简单的PyTorch实现来可视化关键计算。

```python
import torch

input_seq_length = 5
dim_model = 10

input_ids_emb = torch.randn(input_seq_length, dim_model)
W_q = torch.randn(dim_model, dim_model)
W_k = torch.randn(dim_model, dim_model)
W_v = torch.randn(dim_model, dim_model)

Q = input_ids_emb @ W_q
K = input_ids_emb @ W_k
V = input_ids_emb @ W_v

```

### 自注意力计算

对于表示为 \( X \in \mathbb{R}^{T \times D} \) 的 \( T \) 个输入嵌入序列，自注意力计算如下：

- \( Q = XW_Q \)，其中 \( W_Q \in \mathbb{R}^{D \times D_q} \)
- \( K = XW_K \)，其中 \( W_K \in \mathbb{R}^{D \times D_k} \)
- \( V = XW_V \)，其中 \( W_V \in \mathbb{R}^{D \times D_v} \)
- 因果掩码 \( M \) 以防止访问未来Token

最终输出为：

\[
\text{Attention}(X; Q, K, V) = \text{softmax}\left( \frac{QK^\top \cdot M}{\sqrt{d_k}} \right)V
\]

以下是使用因果掩码的最小PyTorch等效实现：

```python
import torch.nn.functional as F
import math

d_k = K.shape[-1]
attention_scores = (Q @ K.T) / math.sqrt(d_k)


causal_mask = torch.tril(torch.ones(input_seq_length, input_seq_length))
masked_scores = attention_scores.masked_fill(causal_mask == 0, float('-inf'))

attention_weights = F.softmax(masked_scores, dim=-1)
output = attention_weights @ V

```

## 冗余从何而来

在自回归生成中，模型每次生成一个Token。每一步中，它都会为**整个序列**重新计算 \( Q \)、\( K \) 和 \( V \)，即使先前的Token并未改变。

```python
new_token_emb = torch.randn(1, dim_model)
extended_input = torch.cat([input_ids_emb, new_token_emb], dim=0)

Q_ext = extended_input @ W_q
K_ext = extended_input @ W_k
V_ext = extended_input @ W_v



```

为确认冗余：

```python
torch.testing.assert_close(K, K_ext[:input_seq_length]) 
torch.testing.assert_close(V, V_ext[:input_seq_length]) 

```

这些检查表明，除最新Token外，\( K \) 和 \( V \) 与先前计算的值完全相同。

```
原始（5×5）：         扩展后（6×6）：
■ ■ ■ ■ ■              ■ ■ ■ ■ ■ □
■ ■ ■ ■ ■              ■ ■ ■ ■ ■ □
■ ■ ■ ■ ■    →         ■ ■ ■ ■ ■ □
■ ■ ■ ■ ■              ■ ■ ■ ■ ■ □
■ ■ ■ ■ ■              ■ ■ ■ ■ ■ □
                       □ □ □ □ □ □

```

- ■ = 已计算并可复用
- □ = 不必要的重新计算

大部分注意力计算被不必要地重复。随着序列增长，开销会越来越大。

## KV缓存如何解决问题

为消除这种低效，我们使用**KV缓存**：

- 处理完初始提示后，我们**缓存**每层计算出的键（\( K \)）和值（\( V \)）。
- 在生成过程中，我们**仅计算**新Token的 \( K \) 和 \( V \)，并将其**追加**到缓存中。
- 我们计算当前Token的 \( Q \)，并将其与**缓存的** \( K \) 和 \( V \) 结合以获得输出。

这将生成过程从全序列重新计算转变为轻量级的增量更新。

✅ 实践中，此缓存是每层的字典，包含键"key"和"value"，每个的形状为（batch_size, num_heads, seq_len_cached, head_dim）。

这是现代LLM能够高效生成长输出的基础。

### 1. 在注意力块中更新KV缓存

在`LanguageModelGroupedAttention`类中，我们修改`forward`函数以接受和更新键值缓存（`block_kv_cache`）。

此前，模型在每个生成步骤都重新计算 \( K \) 和 \( V \)。现在，我们仅计算当前Token的 \( K_{\text{new}} \) 和 \( V_{\text{new}} \)，并将其追加到缓存值中。

```python
def forward(self, x, cos, sin, attention_mask=None, block_kv_cache=None):
    is_prefill = block_kv_cache is None
    B, T_curr, C = x.size()

    
    q_curr, k_curr, v_curr = project_current_tokens(x)
    q, k_rotated = apply_rotary_pos_embd(q_curr, k_curr, cos, sin)

    if not is_prefill and block_kv_cache['key'] is not None:
        
        k = torch.cat([block_kv_cache['key'], k_rotated], dim=2)
        v = torch.cat([block_kv_cache['value'], v_curr], dim=2)
    else:
        
        k, v = k_rotated, v_curr

    block_kv_cache = {'key': k, 'value': v}
    return attention_output, block_kv_cache

```

### 2. 跨层跟踪缓存

在`LanguageModel`类中，我们引入了**逐层缓存跟踪**。`start_pos`参数帮助模型为新生成的Token计算正确的**旋转位置编码**。

```python
def forward(self, x, kv_cache=None, start_pos=0):
    T_curr = x.size(1)
    position_ids = torch.arange(start_pos, start_pos + T_curr, device=x.device)
    cos, sin = self.rotary_embd(position_ids)

    for i, block in enumerate(self.blocks):
        
        x, kv_cache[i] = block(x, cos, sin, attention_mask, kv_cache[i])

    return x, kv_cache

```

- `kv_cache`：字典列表，每个Transformer层一个，保存先前的键和值。
- `start_pos`：确保旋转嵌入与当前生成索引对齐。

### 3. 生成循环中的预填充与解码

最大的架构变更在于`VisionLanguageModel`的`generate()`方法。

我们将生成过程**拆分为两个阶段**：

- **预填充阶段**：编码完整提示词并构建初始缓存。
- **解码阶段**：利用缓存的键/值逐Token生成。

```
预填充阶段（缓存构建）
[提示词："What is"] → [Transformer] → [缓存：所有层的K、V]

解码阶段（逐Token生成）
[Token："the"] → [Q("the") + 缓存的K/V] → [下一个Token："?"] → ...
```

通过分离这两个阶段，我们避免了冗余计算，显著加快了推理速度，尤其对于长提示词场景。
