作者：Vishnu Suresh，LangChain 软件工程师

这篇博客最初发表在 X 上。

我为我们的 GTM Agent（智能体）构建了一个自愈部署流水线。每次部署后，它会检测回归问题，判断是否是本次变更导致的，然后启动一个 Agent（智能体）来提交包含修复方案的 PR，直到需要人工审核前，整个过程无需任何手动干预。

发布代码的难点不在于把代码推出去。而在于之后的一切：弄清楚你上次的部署是否破坏了某些功能，这是否真的是你的错，并在用户注意到之前修复它。我希望部署完成后就能继续前进，并相信如果出现任何回归问题，系统会捕获它并自行完成闭环。

## 自愈流程如何运作

GTM Agent（智能体）运行在 Deep Agents 上，并通过 LangSmith Deployments 进行部署。我们内部已经有一个名为 Open SWE 的编码 Agent（智能体），这是一个开源的异步编码 Agent（智能体），能够研究代码库、编写修复方案并提交 PR。缺失的一环是自动化的回归检测和问题分类，以便将生产环境中的错误反馈给 Open SWE。

![](/images/posts/ee4a979c93f5.png)

在向主分支部署后，一个自愈 GitHub Action 会立即触发，捕获构建和服务器日志。该流程有两条路径：立即捕获构建失败，以及在监控窗口期内检测服务器端回归问题。如果任何一条路径发现真正的问题，就会启动 Open SWE 来修复它并提交 PR。

### 捕获 Docker 构建失败

首先，我们检查构建日志以确保 Docker 镜像构建正确。如果镜像构建失败，流水线会自动从 CLI 获取错误日志，获取最后一次提交到主分支的 git diff，并将其交给 Open SWE——全程无需人工参与。构建失败几乎总是由最近的变更引起的，因此一个范围较窄的 diff 为 Open SWE 提供了足够的上下文来采取行动。

### 监控部署后错误

服务器端问题比构建失败更棘手。任何生产系统都存在背景错误率、网络超时、第三方 API 问题、瞬时故障。在理想情况下，你会追踪并修复每一个问题，但当你试图回答“我上次的部署是否破坏了某些功能”时，你需要将你的变更引起的错误与已经存在的噪音区分开来。这就是这一步要做的事情。

首先，我收集过去 7 天所有错误日志的基线。这些日志被规范化为错误签名，使用正则表达式替换 UUID、时间戳和长数字字符串，然后截断到 200 个字符，这样即使具体细节不同，逻辑上相同的错误也会被归为一类。

![](/images/posts/323613001e48.png)

接下来，我在部署后的 60 分钟窗口期内轮询当前修订版本的错误，并以同样的方式进行规范化。一旦该窗口期结束，我就得到了两个时间尺度非常不同的错误计数——一周的基线数据和一小时的部署后数据。虽然我可以简单地比较这两个数字来检测我们最近的变更是否导致了错误，但我希望采用一种更有原则的方法（并复习一下我的概率分布知识 🙃）。

### 使用泊松检验进行门控

泊松分布模拟了在给定已知平均速率（λ）且事件相互独立的假设下，事件在固定间隔内发生的次数：

![](/images/posts/f01a4f9f32d4.png)

我的观点是，任何生产系统总是存在背景错误率、网络超时、第三方 API 问题等。这些基线错误相当符合泊松模型。

使用 7 天的基线数据，我估算出每个错误签名每小时预期的错误率，然后将其缩放到 60 分钟的部署后窗口期。如果观察到的计数显著超过分布预测的值（p < 0.05），我就将其标记为潜在的回归问题。对于完全新的错误签名（基线中完全不存在），如果它们在监控窗口期内重复出现，我也会将其标记出来。

![](/images/posts/cb8bc1b5921b.png)

但服务器错误并不总是独立的。由流量激增或 API 中断引起的相关故障可能违反独立性假设，而仅靠统计检验无法区分“这个错误激增是因为我们的代码变更”和“这个错误激增是因为第三方 API 宕机”。这就是问题分类 Agent（智能体）的作用所在。

### 问题分类 Agent（智能体）

![](/images/posts/9e4e1a8b6623.png)

与其直接将错误输入 Open SWE（它倾向于进行修改），我增加了另一个门控机制。最后一次提交的 diff 和具体错误会被传递给一个基于 Deep Agents 构建的问题分类 Agent（智能体）。

首先，问题分类 Agent（智能体）将每个更改的文件分类为运行时、提示词/配置、测试、文档或 CI。如果一次部署只触及非运行时文件，那么部署导致错误的可能性极低。这可以防止出现误报，即 Agent（智能体）可能从测试文件到生产环境错误虚构出一条因果链。

对于运行时变更，Agent（智能体）必须在 diff 中的特定行与观察到的错误之间建立具体的因果联系。

Agent（智能体）会返回一个结构化的裁决，包含其决定、置信度、推理过程以及它归因于该变更的错误签名。这种聚焦意味着 Open SWE 收到的是一个有针对性的调查提示词，而不是所有激增错误的转储。

### 通过 Open SWE 完成闭环

一旦问题分类 Agent（智能体）批准进行调查，Open SWE 就会接手，处理该错误并提交 PR。当它准备好接受审核时，我会收到通知，因此从错误检测到提出修复方案的整个流程无需任何手动干预。

到目前为止，它在捕获那些不会导致严重崩溃的 bug 方面最为有用：例如返回错误默认值的静默故障、代码与部署之间的配置不匹配，以及连锁回归问题——修复一个 bug 会在后续部署中暴露出下一个 bug。

## 未来的改进

### 更宽的回溯窗口

目前，问题分类 Agent（智能体）只查看当前部署版本与上一个部署版本之间的 diff。在更早的部署中引入但直到后来才显现的 bug 将不会被自动归因。扩大回溯范围是显而易见的解决方案，但提供给问题分类的 diff 越多，信号就越嘈杂，也就越难精确定位因果联系。我还没有找到合适的平衡点。

### 更智能的错误分组

当前的方法通过对错误消息中的 ID 和时间戳进行清理来实现模糊匹配。这花了一些时间才调整好，并且由于清理逻辑的限制，可能仍然存在相关错误没有被归为一类的情况。

我一直在考虑的一个想法是将错误消息嵌入（向量）到向量空间中并进行聚类，而不是依赖正则表达式规范化。无论表面差异如何，含义相同的错误自然会聚集在一起，我们可以通过监控部署后新集群的形成或现有集群的增长来检测回归问题。挑战在于调整距离阈值，以区分有意义的集群变化与正常方差。

另一个选择是使用一个更小、更便宜的模型来对错误进行分类和分组，然后将这些结构化的集群作为调查提示词的一部分直接传递给 Open SWE——让它更全面地了解哪些地方失败了以及完整的错误情况。

Ramp 采用了一种有趣的方法使其 Sheets 产品能够自我维护。在每次 PR 合并时，一个 LLM（大语言模型）会读取 diff 并生成针对变更代码的监控器，每个监控器都有明确的错误率激增、延迟回归等阈值。当监控器触发时，一个 webhook 会将警报上下文直接传递给一个 Agent（智能体）进行分类。预先定义一个有针对性的监控器会产生更清晰的信号，使下游 Agent（智能体）更容易诊断问题。

### 向前修复 vs 向后查看

目前系统总是采用前向修复，Open SWE在修复PR的同时，故障部署仍保持在线。更智能的方法是根据严重程度、错误率和排查置信度在两种方案间做出选择。对于严重程度高但因果链置信度低的突发问题，可能需要立即回滚；而对于原因明确、修复路径清晰的已定位缺陷，则更适合通过推送补丁进行前向修复。

## 结论

我认为这种模式将在部署服务中越来越普遍。系统越能自主检测、排查并修复自身的回归问题，产品交付速度就越快，工程师也能将更多时间从监控仪表盘转向实际构建工作。
