# OpenAI内部数据Agent探秘

作者：Bonnie Xu、Aravind Suresh、Emma Tang

数据驱动着系统的学习方式、产品的演进方向以及企业的决策过程。但快速、准确且带着恰当上下文获取答案，往往比想象中困难得多。为了在OpenAI规模化发展的过程中简化这一过程，我们构建了一个专属的内部AI数据Agent，用于在我们自己的平台上进行探索和推理。

我们的Agent是一个定制化的内部工具（非对外产品），专门围绕OpenAI的数据、权限和工作流程构建。我们将展示我们如何构建并使用它，以帮助呈现AI能够以真实、 impactful 的方式支持我们团队日常工作的实例。我们用于构建和运行它的OpenAI工具（Codex、我们的GPT‑5旗舰模型、Evals API⁠(在新窗口中打开)以及Embeddings API⁠(在新窗口中打开)）与我们在全球范围内提供给开发者的工具相同。

我们的数据Agent让员工能够在几分钟内（而非几天）从问题获得洞察。这降低了跨所有职能（而不仅仅是数据团队）获取数据和进行细致分析的门槛。如今，OpenAI的工程、数据科学、市场推广、财务和研究等团队都依赖该Agent来回答高影响力的数据问题。例如，它可以通过直观的自然语言格式，帮助回答如何评估产品发布以及了解业务健康状况等问题。该Agent将Codex驱动的表级知识与产品和组织上下文相结合。其持续学习的记忆系统意味着它会在每次交互中不断改进。

![截图显示用户询问2025年10月6日与2023年DevDay相比的ChatGPT WAU。Agent报告2025年约为8亿WAU，2023年约为1亿WAU，并附有注释显示+7亿的变化和约8倍的增长，随后是解释性上下文。](/images/posts/5411a61a9f5f.png)

在这篇文章中，我们将剖析为什么我们需要一个定制的AI数据Agent，是什么使其代码增强的数据上下文和自学习功能如此有用，以及我们在此过程中学到的经验教训。

## 为何需要定制工具

OpenAI的数据平台服务于超过3500名内部用户，涵盖工程、产品和研究领域，跨越超过600 PB的数据，分布在7万个数据集中。在如此规模下，仅仅找到正确的表格就可能成为分析中最耗时的环节之一。

正如一位内部用户所言：

“我们有很多非常相似的表格，我花了大量时间试图弄清楚它们之间的区别以及该用哪一个。有些包含已注销用户，有些则没有。有些字段重叠，很难分辨哪个是哪个。”

即使选择了正确的表格，产生正确的结果也可能具有挑战性。分析师必须推理表数据和表关系，以确保正确应用转换和过滤器。常见的失败模式——多对多连接、过滤器下推错误以及未处理的空值——可能会在不知不觉中使结果无效。在OpenAI的规模下，分析师不应将时间浪费在调试SQL语义或查询性能上：他们的重点应放在定义指标、验证假设和做出数据驱动决策上。

## 工作原理

让我们来了解我们的Agent是什么，它如何整理上下文，以及如何持续自我改进。

我们的Agent由GPT‑5.2驱动，旨在对OpenAI的数据平台进行推理。它在员工已有的工作环境中均可使用：作为Slack Agent、通过Web界面、在IDE内、通过MCP在Codex CLI中⁠(在新窗口中打开)，以及通过MCP连接器直接在OpenAI的内部ChatGPT应用中⁠(在新窗口中打开)。

![图表标题为“数据Agent工作原理”。入口点——Agent-UI、本地Agent-MCP、远程Agent-MCP和Slack Agent——汇入一个Agent-API。该API连接到内部数据知识和公司上下文，与数据仓库和平台源同步，并通过Agent-MCP与GPT-5.2模型交换请求。](/images/posts/e1d604183e4e.jpg)

用户可以提出复杂的、开放性的问题，这些问题通常需要多轮手动探索。以这个使用测试数据集的提示词为例：“对于纽约市出租车行程，哪些上车点至下车点的邮政编码对最不可靠，即典型行程时间与最差行程时间之间的差距最大，并且这种变异性发生在何时？”

Agent端到端地处理分析，从理解问题到探索数据、运行查询以及综合发现。

![Agent对问题的回应。](/images/posts/82c3a610424c.png)

Agent对问题的回应。

Agent的超能力之一是其推理问题的方式。Agent并非遵循固定脚本，而是评估自身的进展。如果中间结果看起来有误（例如，由于连接或过滤器错误导致零行），Agent会调查出错原因，调整方法，然后重试。在整个过程中，它会保留完整的上下文，并在步骤之间传递学到的知识。这种闭环、自学习的过程将迭代从用户转移到Agent本身，从而比手动工作流程实现更快的结果和始终更高质量的分析。

![Agent识别最不可靠的纽约市出租车上下车点对的推理过程。](/images/posts/c85804ed6fa5.png)

该Agent涵盖了完整的分析工作流程：发现数据、运行SQL以及发布笔记本和报告。它理解内部公司知识，可以搜索网络获取外部信息，并通过学习到的用法和记忆随着时间的推移不断改进。

## 上下文即一切

高质量的答案依赖于丰富、准确的上下文。没有上下文，即使是强大的模型也可能产生错误的结果，例如严重错误估计用户数量或误解内部术语。

![截图显示用户询问“过去30天ChatGPT图像生成功能的登录DAU是多少？”下方状态行显示Agent已“工作22分41秒”，表示正在执行一个长时间运行的查询。](/images/posts/03645f05530a.png)

没有记忆的Agent，无法有效查询。

为了避免这些失败模式，Agent围绕多层上下文构建，这些上下文将其锚定在OpenAI的数据和机构知识中。

![图表标题为“数据Agent的上下文层次”，显示六个堆叠层级：1) 表使用情况，2) 人工注释，3) Codex增强，4) 机构知识，5) 记忆，6) 运行时上下文。每一层在金字塔形状中显示为水平条。](/images/posts/9d9c3e8a9808.jpg)

### 第一层：表使用情况

- 元数据锚定：Agent依赖模式元数据（列名和数据类型）来指导SQL编写，并使用表血缘关系（例如，上游和下游表关系）来提供不同表如何关联的上下文。
- 查询推断：摄取历史查询有助于Agent理解如何编写自己的查询，以及哪些表通常被连接在一起。

### 第二层：人工注释

- 由领域专家提供的表和列的精心描述，捕捉意图、语义、业务含义以及从模式或历史查询中不易推断出的已知注意事项。

仅有元数据是不够的。要真正区分表格，你需要了解它们是如何创建的以及源自何处。

### 第三层：Codex 知识增强

- 通过推导表的代码级定义，Agent（智能体）能够更深入地理解数据实际包含的内容。表中存储内容的细微差别以及如何从分析事件中推导出这些内容，提供了额外信息。例如，它可以提供关于值唯一性、表数据更新频率、数据范围（例如，如果表排除了某些字段，则具有此粒度级别）等方面的上下文。
- 这通过展示该表在 SQL 之外，如何在 Spark、Python 和其他数据系统中被使用，提供了增强的使用上下文。
- 这意味着 Agent（智能体）能够区分那些看起来相似但在关键方面存在差异的表。例如，它可以判断一个表是否只包含第一方 ChatGPT 流量。此上下文也会自动刷新，因此无需手动维护即可保持最新。

![标题为“Codex 增强的知识管道”的图表。热门表输入到多个 Codex 任务中，这些任务从 OpenAI 代码库中提取详细信息，包括表的用途、粒度和主键、下游使用模式、备选表选项以及数据新鲜度。](/images/posts/03497a311ffd.jpg)

### 第四层：机构知识

- Agent（智能体）可以访问 Slack、Google Docs 和 Notion，这些工具捕获了关键的公司上下文，例如产品发布、可靠性事件、内部代号和工具，以及关键指标的规范定义和计算逻辑。
- 这些文档被摄取、嵌入，并与元数据和权限一起存储。一个检索服务在运行时处理访问控制和缓存，使 Agent（智能体）能够高效且安全地提取这些信息。

![用户询问为什么连接器使用量在 12 月下降的截图。Agent（智能体）解释下降是由于从 2025 年 11 月 13 日开始的一个日志记录问题，导致在 ChatGPT 5.1 发布后使用量被低估。旧版遥测数据变为空，直到一个新的事件成为数据真相来源。](/images/posts/1fbb7a566ddd.png)

### 第五层：记忆

- 当 Agent（智能体）收到修正或发现关于某些数据问题的细微差别时，它能够保存这些学习成果以备下次使用，从而使其能够与用户一起不断改进。因此，未来的回答从一个更准确的基线开始，而不是反复遇到相同的问题。记忆的目标是保留和重用那些对于数据正确性至关重要但难以仅从其他层推断出的非显而易见的修正、过滤器和约束。例如，在一个案例中，Agent（智能体）不知道如何过滤特定的分析实验（它依赖于匹配实验门控中定义的特定字符串）。记忆在这里至关重要，确保它能够正确过滤，而不是模糊地尝试字符串匹配。
- 当你给 Agent（智能体）一个修正，或者它从你的对话中发现一个学习点时，它会提示你保存该记忆以备下次使用。用户也可以手动创建和编辑记忆。记忆的范围分为全局和个人级别，Agent（智能体）的工具使其易于编辑。

- 因此，未来的回答从一个更准确的基线开始，而不是反复遇到相同的问题。

- 例如，在一个案例中，Agent（智能体）不知道如何过滤特定的分析实验（它依赖于匹配实验门控中定义的特定字符串）。记忆在这里至关重要，确保它能够正确过滤，而不是模糊地尝试字符串匹配。

- 用户也可以手动创建和编辑记忆。

### 第六层：运行时上下文

- 当表不存在先前的上下文或现有信息过时时，Agent（智能体）可以向数据仓库发出实时查询，以直接检查和查询该表。这使其能够验证模式、实时理解数据并做出相应响应。
- Agent（智能体）还能够根据需要与其他数据平台系统（元数据服务、Airflow、Spark）通信，以获取仓库外存在的更广泛的数据上下文。

我们运行一个每日离线管道，将表使用情况、人工标注和 Codex 衍生的知识增强聚合到一个单一的、标准化的表示中。然后，使用 OpenAI 嵌入 API⁠(在新窗口中打开)将此增强上下文转换为嵌入向量，并存储以供检索。在查询时，Agent（智能体）通过检索增强生成⁠(在新窗口中打开)（RAG）仅拉取最相关的嵌入上下文，而不是扫描原始元数据或日志。这使得表理解快速且可扩展，即使在数万个表中也能保持运行时延迟可预测且较低。根据需要，实时查询会实时发送到我们的数据仓库。

![标题为“数据 Agent（智能体）中的上下文检索”的图表。离线预处理层——表使用情况、人工标注、Codex 知识增强、机构知识和记忆——输入到 RAG 嵌入向量中。实时检索显示 Agent（智能体）通过语义搜索或精确文本检索查询数据库，以产生运行时上下文。](/images/posts/7ce8be06045a.jpg)

这些层共同确保 Agent（智能体）的推理基于 OpenAI 的数据、代码和机构知识，从而显著减少错误并提高答案质量。

## 旨在像队友一样思考和协作

当问题明确时，一次性答案很有效，但大多数问题并非如此。更常见的是，要获得正确的结果需要反复的细化和一些方向修正。

Agent（智能体）被设计成行为像你可以与之推理的队友。它是一个对话式的、始终在线的系统，既能处理快速回答，也能处理迭代探索。

它在轮次之间传递完整的上下文，因此用户可以提出后续问题、调整意图或改变方向，而无需重复陈述所有内容。如果 Agent（智能体）开始走向错误的方向，用户可以在分析过程中打断并重新引导它，就像与一个会倾听而不是一味向前冲的人类协作者一起工作一样。

当指令不明确或不完整时，Agent（智能体）会主动提出澄清性问题。如果没有提供响应，它会应用合理的默认值来推进工作。例如，如果用户询问业务增长但没有指定日期范围，它可能会假设是最近 7 天或 30 天。这些先验知识使其能够保持响应性和非阻塞性，同时仍然收敛于正确的结果。

最终的结果是一个 Agent（智能体），无论你是确切知道你想要什么（例如，“告诉我关于这个表的信息”），还是在探索时（例如，“我看到这里有一个下降，我们可以按客户类型和时间段来分析吗？”），它都能很好地工作。

上线后，我们观察到用户经常为常规重复性工作运行相同的分析。为了加快这一过程，Agent（智能体）的工作流将重复性分析打包成可重用的指令集。示例包括用于每周业务报告和表验证的工作流。通过一次编码上下文和最佳实践，工作流简化了重复分析，并确保跨用户的结果一致。

## 快速前进而不破坏信任

构建一个始终在线、不断演进的 Agent（智能体）意味着质量可能像提升一样容易下降。没有紧密的反馈循环，回归是不可避免且不可见的。在不破坏信任的情况下扩展能力的唯一方法是通过系统性的评估。

在本节中，我们将讨论如何利用 OpenAI 的 Evals API⁠（在新窗口中打开）来衡量和保护 Agent（智能体）的响应质量。

其评估系统基于精心策划的问答对构建。每个问题都针对我们极为关注的重要指标或分析模式，并配有人工编写的“黄金”SQL 查询，该查询可生成预期结果。对于每次评估，我们将自然语言问题发送至其查询生成端点，执行生成的 SQL，并将输出与预期 SQL 的结果进行比较。

![标题为“数据 Agent（智能体）的评估流程”的图表。带有预期 SQL 的问答评估对输入到生成步骤中，该步骤生成 SQL 和结果。OpenAI Evals 使用数据框和 SQL 比较来比较生成的结果与预期的结果，并输出评分和推理。](/images/posts/c0143769cc63.jpg)

评估不依赖于简单的字符串匹配。生成的 SQL 可能在语法上不同，但仍然是正确的，并且结果集可能包含对答案没有实质性影响的额外列。为了解决这个问题，我们比较了 SQL 和结果数据，并将这些信号输入到 OpenAI 的 Evals 评分器中。评分器会生成最终评分以及解释，同时捕捉正确性和可接受的差异。

这些评估就像在开发过程中持续运行的单元测试，用于识别回归问题，如同生产环境中的金丝雀；这使我们能够及早发现问题，并在 Agent（智能体）能力扩展时自信地进行迭代。

## Agent（智能体）安全

我们的 Agent（智能体）直接接入 OpenAI 现有的安全与访问控制模型。它纯粹作为一个接口层运行，继承并强制执行管理 OpenAI 数据的相同权限和护栏。

Agent（智能体）的所有访问都是严格的直通式，这意味着用户只能查询他们已有权限访问的表。当访问权限缺失时，它会标记此问题，或回退到用户有权使用的替代数据集。

最后，它专为透明性而构建。与任何系统一样，它可能会犯错。它通过在每个答案旁边总结假设和执行步骤来暴露其推理过程。当查询被执行时，它会直接链接到底层结果，允许用户检查原始数据并验证分析的每一步。

## 经验教训

从头构建我们的 Agent（智能体）揭示了关于 Agent（智能体）行为方式、其困难所在以及什么才能真正使其在大规模下保持可靠的实践经验。

#### 教训 #1：少即是多

早期，我们向 Agent（智能体）暴露了全部工具集，并很快遇到了功能重叠的问题。虽然这种冗余对于特定的定制案例可能有帮助，并且在手动调用时对人类来说更明显，但它会让 Agent（智能体）感到困惑。为了减少歧义并提高可靠性，我们限制并合并了某些工具调用。

#### 教训 #2：引导目标，而非路径

我们还发现，高度规定性的提示词会降低结果质量。虽然许多问题共享一个通用的分析框架，但细节差异很大，以至于僵硬的指令常常将 Agent（智能体）推向错误的路径。通过转向更高层次的指导，并依赖 GPT‑5 的推理来选择适当的执行路径，Agent（智能体）变得更加稳健，并产生了更好的结果。

#### 教训 #3：意义存在于代码中
