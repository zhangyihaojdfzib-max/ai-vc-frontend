当MCP标准首次推出时，许多团队急于推出产品。许多服务器最终只是对现有API进行了极简改动的薄封装层。这是一种快速宣称"我们支持MCP"的方式。

在当时，这种做法有其道理。MCP是新生事物，团队希望快速推出产品，而最直接的方法就是照搬现有的API结构。既然可以重新包装，何必重新发明？

但这种方法的问题在于，LLM（大语言模型）的工作方式与开发者不同。它们不会复用过去的代码，也不会保持长期状态。每次对话都是全新的开始。LLM必须重新发现存在哪些工具、如何使用它们以及使用顺序。使用低层级的API封装器会导致重复编排、行为不一致，以及LLM反复解决相同难题造成的精力浪费。

当工具处理完整的用户意图，而非暴露单个API操作时，MCP才能发挥最佳效果。一个能端到端部署项目的工具，比四个各自处理部署流程一部分的工具效果更好。

## LLM使用API的关键差异

关键区别在于上下文和状态管理。当你编写代码时，你会在API调用之间跟踪信息。你存储来自创建调用的项目ID，在添加域名前检查部署状态，并在每个步骤周围包裹错误处理，以便故障不会破坏整个流程。

LLM的工作方式不同。每次对话都是全新的开始，没有对先前对话的记忆。虽然它们能在当前对话中看到工具结果，但它们必须根据可用工具来找出正确的使用顺序。当这些工具是低层级的API封装器时，LLM每次都必须编排多个调用，并管理将它们链接在一起的复杂性。

例如，以使用Vercel API部署项目为例。开发者可能会编写如下代码：

```
12const project = await client.projects.create({3  name: domain.replace(/\./g, '-'),4  gitRepository: { repo: repoUrl }5});6
7await client.projects.createProjectEnv({8  idOrName: project.id,9  upsert: 'true',10  requestBody: Object.entries(env).map(([key, value]) => ({11    key, value,12    target: ['production', 'preview', 'development'],13    type: 'encrypted'14  }))15});16
17const deployment = await client.deployments.createDeployment({18  requestBody: {19    name: project.name,20    target: 'production',21    gitSource: { type: 'github', repo: repo.replace('.git', ''), ref: 'main' }22  }23});24
25await client.projects.addProjectDomain({26  idOrName: project.id,27  requestBody: { domain: domain }28});
```

这种方法涉及嵌套配置、跨调用的ID管理以及解析仓库URL。开发者会一次性解决这个问题并复用。但LLM每次都要面对这个难题，常常搞错嵌套结构或忘记必填字段。

## 单一工作流工具 vs 多个端点

解决方案是围绕完整的用户目标而非API能力来构建工具。与其创建四个独立的工具，不如创建一个`deploy_project`工具，在内部处理整个工作流。

这彻底改变了工具设计的方方面面：

**API形态的工具**

**基于意图的工具**

`create_project`, `add_env`, `deploy`, `add_domain`

`deploy_project`

需要状态管理的多次调用

单一原子操作

返回技术状态码

返回对话式更新

LLM负责组装工作流

工具拥有完整流程

考虑一下实践中的差异。一个API形态的MCP服务器可能会暴露这些工具：

```
1create_project(name, repo)2add_environment_variables(project_id, variables)3create_deployment(project_id, branch)4add_domain(project_id, domain)
```

LLM必须按顺序调用每个工具，在调用之间传递ID，并在每个步骤处理可能的失败。

而一个基于意图的工具则看起来不同：

```
1deploy_project(repo_url, domain, environment_variables, branch="main")
```

这个单一工具在内部处理完整的工作流，并返回一个对话式响应。LLM无需处理`{ status: 200, data: { id: "proj_123" } }`，而是可以回应"项目已部署在example.com。构建在45秒内完成。所有系统运行正常。"

## 设计基于工作流的MCP工具

在编写任何代码之前，先从手动测试工作流开始。以一个真实的用户请求（如"为我的项目设置身份验证和数据库"）为例，使用你现有的API逐步完成它。那些感觉繁琐或重复的部分，正是适合整合为单一MCP工具的候选对象。

将MCP工具视为帮助AI完成特定任务的定制工具包，而非API镜像。一个MCP工具背后可能涉及多个API和业务逻辑。如果用户将某件事视为一个工作流，就将其设计为一个工具。

以下是构建一个完整工作流工具的结构示例：

```
1server.tool(2  "deploy_project",3  "Deploy a project with environment variables and custom domain",4  {5    repo_url: z.string(),6    domain: z.string(),7    environment_variables: z.record(z.string()),8    branch: z.string().default("main")9  },10  async ({ repo_url, domain, environment_variables, branch }) => {11    12    const project = await createProject(repo_url, branch);13    await addEnvironmentVariables(project.id, environment_variables);14    const deployment = await deployProject(project.id);15    await addCustomDomain(project.id, domain);16    17    return {18      content: [{19        type: "text",20        text: `Project deployed successfully at ${domain}. Build completed in ${deployment.duration}s.`21      }]22    };23  }24);
```

对于确定性的部分使用普通代码。像API顺序、错误恢复和状态管理这类事情，更适合用常规编程来处理。只在真正需要推理或自然语言处理的部分才涉及LLM。

用真实场景进行测试。通过你的工具运行实际的用户工作流。当你看到LLM进行多次尝试或要求澄清时，那就是对你工具设计的反馈。目标是让复杂的工作流能在第一次尝试时就成功。

## 工作流工具带来的性能提升

从API形态工具转向工作流形态工具的团队，在可靠性和效率方面都看到了显著的改进。

这些工具设计的共同主线是：

- 它们专注于用户意图而非API覆盖范围
- 它们处理完整的工作流而非暴露单一操作
- 它们以对话方式响应而非返回技术代码

当工具反映完整的用户目标时，MCP效果最佳。LLM不像开发者那样管理状态，因此围绕工作流构建工具能产生更好的结果。
