# 将Kubernetes扩展到2500个节点

![将Kubernetes扩展到2500个节点](/images/posts/07a073ce1624.webp)

我们已经在深度学习研究中运行Kubernetes超过两年。虽然我们最大规模的工作负载直接管理裸云虚拟机，但Kubernetes提供了快速的迭代周期、合理的可扩展性以及无需样板代码的特点，使其成为我们大多数实验的理想选择。我们现在运营着多个Kubernetes集群（部分在云端，部分在物理硬件上），其中最大的集群已扩展到超过2500个节点。该集群在Azure上运行，采用D15v2和NC24虚拟机的组合。

在达到这一规模的过程中，许多系统组件出现了故障，包括etcd、Kube主节点、Docker镜像拉取、网络、KubeDNS，甚至我们机器的ARP缓存。我们认为分享我们遇到的具体问题以及如何解决这些问题会很有帮助。

在集群超过500个节点后，我们的研究人员开始报告`kubectl`命令行工具出现定期超时。我们尝试增加更多的Kube主节点（运行`kube-apiserver`的虚拟机）。这似乎暂时解决了问题，但一旦我们超过10个副本，我们意识到我们只是在治标而非治本（相比之下，GKE对500个节点只使用一个32核的虚拟机）。

这让我们强烈怀疑我们的etcd集群，它是Kube主节点的中央状态存储。查看Datadog时，我们发现运行etcd副本的DS15v2机器上的写入延迟飙升到数百毫秒，尽管每台机器都使用能够提供5000 IOPS的P30 SSD。

![磁盘延迟随时间变化的图表](/images/posts/1f5a07bc442d.png)

使用`fio`进行性能基准测试时，我们发现etcd只能使用约10%的可用IOPS，因为写入延迟为2毫秒，而etcd执行顺序I/O，因此受延迟限制。

随后，我们将每个节点的etcd目录迁移到本地临时磁盘，这是一个直接连接到实例的SSD，而非网络连接的SSD。切换到本地磁盘后，写入延迟降至200微秒，etcd恢复了健康！

我们的集群运行良好，直到超过约1000个节点，此时我们再次看到etcd出现高提交延迟。这一次，我们注意到kube-apiserver从etcd读取数据的速度超过500MB/s。我们设置了Prometheus来监控apiserver，并设置了`--audit-log-path`和`--audit-log-maxbackup`标志以在apiserver上启用更多日志记录。这暴露了一些慢查询以及对Events的LIST API的过度调用。

根本原因：Fluentd和Datadog监控进程的默认设置是从集群中的每个节点查询apiserver（例如，这个现已修复的问题）。我们只是将这些进程的轮询频率降低，apiserver上的负载便再次稳定下来：

![网络流量随时间变化的图表（字节/秒）](/images/posts/ff1d4b394a1a.png)

另一个有用的调整是将Kubernetes Events存储在一个独立的etcd集群中，这样Event创建的峰值就不会影响主etcd实例的性能。为此，我们只需将`--etcd-servers-overrides`标志设置为类似以下内容：`--etcd-servers-overrides=/events#https://0.example.com:2381;­https://1.example.com:2381;­https://2.example.com:2381`

超过1000个节点后的另一个故障是达到了etcd的硬存储限制（默认2GB），这导致它停止接受写入。这引发了一连串的故障：我们所有的Kube节点都未能通过健康检查，我们的自动扩缩器因此决定终止所有工作节点。我们通过`--quota-backend-bytes`标志增加了etcd的最大大小，并且自动扩缩器现在有一个健全性检查，如果它将终止超过50%的集群，则不会采取行动。

## Kube主节点

我们将kube-apiserver、kube-controller-manager和kube-scheduler进程部署在同一台机器上。为了实现高可用性，我们始终至少有两个主节点，并将`--apiserver-count`标志设置为我们运行的apiserver数量（否则Prometheus监控可能会在实例之间混淆）。

我们主要将Kubernetes用作批处理调度系统，并依赖我们的自动扩缩器来动态扩展和缩减集群——这使我们能够显著降低空闲节点的成本，同时在快速迭代时仍提供低延迟。默认的kube-scheduler策略是在节点之间均匀分布负载，但我们希望相反，这样未使用的节点可以被终止，并且大型Pod可以快速调度。因此，我们切换到了以下策略：

#### 纯文本

```
1{2"kind" : "Policy",3"apiVersion" : "v1",4"predicates" : [5  {"name" : "GeneralPredicates"},6  {"name" : "MatchInterPodAffinity"},7  {"name" : "NoDiskConflict"},8  {"name" : "NoVolumeZoneConflict"},9  {"name" : "PodToleratesNodeTaints"}10  ],11"priorities" : [12  {"name" : "MostRequestedPriority", "weight" : 1},13  {"name" : "InterPodAffinityPriority", "weight" : 2}14  ]15}
```

我们广泛使用KubeDNS进行服务发现，但在推出新的调度策略后不久，它开始出现可靠性问题。我们发现故障只发生在KubeDNS的某些Pod上。使用新的调度策略，一些机器最终运行了10个以上的KubeDNS副本，形成了热点，并且我们超过了每个Azure VM对外部域名查询允许的约200 QPS。

我们通过为KubeDNS Pod添加反亲和性规则解决了这个问题：

```
1affinity:2 podAntiAffinity:3   requiredDuringSchedulingIgnoredDuringExecution:4   - weight: 1005     labelSelector:6       matchExpressions:7       - key: k8s-app8         operator: In9         values:10         - kube-dns11     topologyKey: kubernetes.io/hostname
```

## Docker镜像拉取

我们的Dota项目最初是在Kubernetes上开始的，随着它的扩展，我们注意到新的Kubernetes节点上的Pod经常长时间处于Pending状态。游戏镜像大约17GB，在全新的集群节点上通常需要30分钟才能拉取完成，所以我们理解为什么Dota容器会Pending一段时间——但其他容器也是如此。深入调查后，我们发现kubelet有一个`--serialize-image-pulls`标志，默认值为`true`，这意味着Dota镜像的拉取会阻塞所有其他镜像。将其改为`false`需要将Docker切换到overlay2而不是AUFS。为了进一步加快拉取速度，我们还像对etcd机器一样，将Docker根目录移到了实例挂载的SSD上。

即使在优化了拉取速度之后，我们仍然看到Pod启动失败，并出现一条晦涩的错误信息：`rpc error: code = 2 desc = net/http: request canceled`。kubelet和Docker日志中也包含指示镜像拉取因进度不足而被取消的消息。我们追踪到根本原因是大型镜像拉取/解压耗时过长，或者有时我们有大量镜像需要拉取。为了解决这个问题，我们将kubelet的`--image-pull-progress-deadline`标志设置为30分钟，并将Docker守护进程的`max-concurrent-downloads`选项设置为10。（第二个选项并没有加快大型镜像的解压速度，但允许镜像拉取队列并行进行。）

我们上次的 Docker 拉取问题是由 Google 容器注册表导致的。默认情况下，kubelet 会从 `gcr.io`（由 `--pod-infra-container-image` 标志控制）拉取一个特殊镜像，该镜像在启动任何新容器时使用。如果该拉取因任何原因失败，例如超出配额⁠（在新窗口中打开），该节点将无法启动任何容器。由于我们的节点通过 NAT 访问 `gcr.io` 而非拥有自己的公网 IP，我们很可能会触及这个按 IP 计算的配额限制。为了解决这个问题，我们只需在 Kubernetes 工作节点的机器镜像中预加载该 Docker 镜像，具体方法是使用 `docker image save -o /opt/preloaded_docker_images.tar` 和 `docker image load -i /opt/preloaded_docker_images.tar`。为了提升性能，我们对一个包含常见 OpenAI 内部镜像（如 Dota 镜像）的白名单也做了同样的处理。

## 网络

随着我们的实验规模越来越大，它们也变成了日益复杂的分布式系统，其运行高度依赖网络。当我们首次开始运行分布式实验时，很快就发现我们的网络配置不佳。机器之间的直接吞吐量能达到 10-15Gbit/s，但使用 Flannel 的 Kube Pod 吞吐量最高只有约 2Gbit/s。Machine Zone 的公开基准测试⁠（在新窗口中打开）显示了类似的数字，这意味着问题可能不仅仅是配置不当，而是我们环境中的某些固有因素。（相比之下，Flannel 在我们的物理机器上并不会增加这种开销。）

为了解决这个问题，用户可以通过添加两个不同的设置来为其 Pod 禁用 Flannel：`hostNetwork: true` 和 `dnsPolicy: ClusterFirstWithHostNet`。（不过在这样做之前，请阅读 Kubernetes 文档中的警告⁠（在新窗口中打开）。）

## ARP 缓存

尽管我们调整了 DNS，但仍然看到间歇性的 DNS 解析问题。有一天，一位工程师报告说，`nc -v` 连接到他们的 Redis 服务器需要超过 30 秒才能打印出连接已建立的信息。我们将问题追踪到了内核的 ARP 栈。对 Redis Pod 所在主机的初步调查显示网络存在严重问题：任何端口上的通信都会挂起数秒，并且无法通过本地的 `dnsmasq`⁠（在新窗口中打开）守护进程解析任何 DNS 名称，`dig`⁠（在新窗口中打开）只打印了一条晦涩难懂的失败消息：`socket.c:1915: internal_send: 127.0.0.1#53: Invalid argument`。`dmesg`⁠（在新窗口中打开）日志提供了更多信息：`neighbor table overflow!`，这意味着 ARP 缓存已耗尽空间。ARP 用于将网络地址（如 IPv4 地址）映射到物理地址（如 MAC 地址）。幸运的是，通过在 `/etc/sysctl.conf` 中设置几个选项可以轻松解决这个问题：

```
1net.ipv4.neigh.default.gc_thresh1 = 800002net.ipv4.neigh.default.gc_thresh2 = 900003net.ipv4.neigh.default.gc_thresh3 = 100000
```

在 HPC 集群中调整此设置很常见，并且在 Kubernetes 集群中尤其相关，因为每个 Pod 都有自己的 IP 地址，这会消耗 ARP 缓存的空间。
