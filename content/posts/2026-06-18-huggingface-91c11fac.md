# 它是否足够具备智能体能力？使用自有工具对开源模型进行基准测试

![在不同指标上对Transformer版本进行基准测试](/images/posts/26d36f1d39c2.png)

在不同指标上对Transformer版本进行基准测试

这是一篇由人类撰写的、聚焦智能体的博客文章。

编程智能体越来越多地代替我们与软件打交道：描述一个任务，智能体就会选择库、编写调用、运行代码并调试自己的错误。当库成为障碍时，它会愉快地绕过它，从头重写逻辑。这为库开发引入了一个新概念：代码不仅应该正确且快速，还应该设计成让智能体能够有效驱动它。笨拙的API或过时的文档会困扰我们开发者，现在也会让智能体走上一条更长、更昂贵的路径。

大多数基准测试只关注最终答案。而我们想要的是整个过程：不仅看智能体是否做对了，还要看它花了多少功夫才达到目标，以及这些指标在不同模型、库版本和任务之间如何变化。我们精确测量了这些，并以transformers作为案例研究。

在这里，我们将介绍一个专注于答案发现过程的工具特定基准测试，并提供一种简单实现方案，完全运行在由picoding智能体驱动的开源模型上，将模型×版本×任务的完整扫描分布在Hugging Face Jobs上，使每次运行都使用相同的硬件。

但是，**如何为智能体优化软件？**

我们坚信以下两个软件原则：

- 未经测试，则无法运行
- 未经文档化，则不存在

在面向智能体优化的工具领域，这一点同样适用，而且这一次，这两者直接相互关联。

你希望你的工具对智能体而言是存在的：它需要是可发现的。API需要清晰，文档需要详尽。它们需要以智能体能够快速访问有用文件和示例的方式进行结构化。如果你希望你的工具能为智能体工作，那么你应该针对智能体使用场景进行测试。

## 为智能体使用场景测试软件

在整篇博文中，我们将以transformers为例：智能体使用它来解决机器学习任务（文本分类、图像描述、音频转录），而不是为其贡献代码；不过该测试框架设计为可与任何能从命令行操作的工具配合使用。

我们对transformers的直觉是，通过一些改动可以大幅简化其使用方式：一个CLI、一个Skill以及自包含的、针对特定任务的示例。这与最近应用于hfCLI的方案相同，该CLI被重新设计为面向智能体优化，智能体使用的Token减少了1.3–1.8倍（最高可达6倍）。我们想知道这种优势是否具有普适性，以及是否也能对transformers有所帮助。

直觉是强大的工具，但在我们向transformers这样广泛使用的代码库提交增加数千行代码的PR之前，我们希望获得更多证据。我们开始衡量成功是什么样的。

### 并非所有成功都相同

两个智能体都可以为情感分类任务生成正确的标签，但其中一个：

- 编写一个40行的Python脚本，导入transformers，调试形状错误，重新运行两次，最后打印答案；

而另一个：

- 输入`transformers classify --model ... --text "..."`，一次调用就完成。

两者都得到`POSITIVE (0.9999)`，以下是两个智能体在此确切任务上实际采取的路径：

```diff
# 任务：对"I absolutely loved the movie, it was fantastic!"进行情感分类

- # 一个智能体：将脚本通过管道输入python并解析输出
- python - <<'PY'
- from transformers import AutoTokenizer, AutoModelForSequenceClassification
- import torch
- import torch.nn.functional as F
-
- model = AutoModelForSequenceClassification.from_pretrained("distilbert/distilbert-base-uncased-finetuned-sst-2-english")
- tokenizer = AutoTokenizer.from_pretrained("distilbert/distilbert-base-uncased-finetuned-sst-2-english")
- inputs = tokenizer("I absolutely loved the movie, it was fantastic!", return_tensors="pt")
- with torch.no_grad():
-     logits = model(**inputs).logits
- probs = F.softmax(logits, dim=1)
- idx = torch.argmax(probs, dim=1).item()
- print(model.config.id2label[idx], probs[0][idx].item())
- PY

+ # 另一个智能体：一条命令
+ transformers classify \
+   --model distilbert/distilbert-base-uncased-finetuned-sst-2-english \
+   --text "I absolutely loved the movie, it was fantastic!"

```

两种方法都得到相同的结果。但它们在成本、延迟、Token使用量和失败率方面有着截然不同的特征。

如果你的评估只检查最终字符串，你就无法看到这些差异，也无法知道你推送到库的改动（CLI改进、更好的错误消息、一个Skill）是否真正帮助了智能体。

我们使用这个测试框架的目标是评估智能体完成给定任务需要付出多少努力，以及库的改动是否提升了性能。

### 我们如何运行评估？

这里简单说明我们将如何评估智能体。

我们在三种变体（或"层级"）下运行每个任务；智能体可以通过三种不同方式接触transformers：

```text
bare     仅pip install transformers，无其他操作
clone    完整的transformers源代码，检出在工作目录中
skill    一个打包好的Skill：CLI文档+任务示例，加载到上下文中

```

这些不是嵌套关系：skill不包含clone（它提供精选文档，而非源代码树），两者也不严格包含对方，每种方式都给智能体提供不同类型的帮助。正如我们将看到的，模型有时在clone上的表现优于skill。

还有几个选择：

- 目前我们只关注确定性任务，这些任务可以提供精确匹配，为实验提供了很好的基础。模型作为评判者和其他方案显然是其他任务的下一步。
- 每次运行都是独立的Hugging Face Job：每个（模型×版本×任务）对应一个Job，因此整个扫描在相同硬件上并行运行，保证了大规模比较的公平性。
- 结果和轨迹存储在Hugging Face Bucket中：速度快，无需版本控制，并能处理非常高的写入并发。

### 应该对哪些模型进行基准测试？

并非所有驱动智能体的模型都相同，它们的差异会改变你在运行时应关注的内容。

**大型开源模型**

一方面，你有最大、能力最强的开源模型。在相当常见的任务上，这些模型最终应该能得到正确答案。对它们来说，任务完成率接近100%，不再能告诉你太多关于工具的信息；更相关的基准测试是智能体达到目标所付出的努力：花费了多少轮次、Token和秒数，以及它们走的是清晰路径还是使用了已弃用的API。

**本地模型**

本地模型在规模上差异很大，它们的能力也是如此。像"匹配率"这样的指标比大型模型更相关，因为你可以看到模型大小/能力如何影响你在特定工具上的结果。

这个测试框架不仅为库维护者提供关于如何改进仓库以利于智能体交互的指导，还帮助评估不同智能体和模型在用户关心的任务上的表现。

该测试框架在多个维度上对每次运行进行评分，以便你可以针对每类模型提出真正重要的问题：

- 匹配率：最终答案是否包含预期结果（按任务区分，不区分大小写的子串/正则表达式/精确匹配，均在报告中明确说明）；
- 中位时间 和 中位Token（新生成的 vs. 缓存的 vs. 生成的）；
- 运行错误率：包括一个保护机制，标记产生"无结果"（0个输出Token，无工具调用，无答案）的运行，使静默失败不会伪装成"0"；
- 标记采用率：工具定义的行为标记；见下方解释。

所有内容都汇总到一份你可以直接查看的报告中：

实时报告：概览、覆盖范围和结果，全部在客户端侧。

由于它捕获了每次运行的本地Agent（智能体）轨迹，数字仅仅是个开始：你可以逐条指令精确读取Agent（智能体）执行的操作。这些轨迹可通过Hub的agent-traces查看器共享：

![在Hub的agent-traces查看器中渲染的一次运行：MiniMax-M2.7 在问答任务上的表现](/images/posts/60bfcfff7918.png)

在展示结果之前，先快速回顾一下实验设置。每次运行会变化四个要素：驱动Agent（智能体）的模型、它所运行的transformers版本、任务以及层级（bare/clone/skill）。如前所述，我们针对两类不同的模型类别考察不同的指标。

### 大型开源模型：固定模型，变化版本

由于大型开源模型通常能得出正确结果，你真正衡量的是达成结果所付出的努力。它用了十轮还是一轮？它是否因为信任过时的文档而遵循了你已弃用的API路径？它是否遇到了你未曾预见的错误？

自然的实验方法是固定一个强大的模型，然后变化工具的版本：我们测试的transformers的连续git版本，从v5.8.0和v5.9.0等发布标签，到引入CLI和Skill的特定提交。我们想观察它给Agent（智能体）带来的负载是上升还是下降。我们使用该测试框架在transformers上检查添加专用CLI和Skill是否确实减轻了Agent（智能体）的工作量。

对于我们测试中使用的三个大型模型，所有任务的平均耗时表明，Skill提交使得完成任务所需的时间更少：

按层级划分的每个版本的中位时间：skill提交（绿点）速度最快。

![按层级划分的每个版本的中位时间](/images/posts/10f31d70a09c.png)

另一方面，在克隆仓库的实验中，我们可以看到由于引入CLI和示例的提交，Token消耗显著增加，我们稍后会看到这一点。

按层级划分的每个版本的新Token中位数：一旦CLI进入仓库，clone变体的数值就会跃升。

![按层级划分的每个版本的新Token中位数](/images/posts/9bfd2199cec5.png)

阅读clone变体的轨迹可以解释原因。该提交添加了一个命令，但它也将CLI的实现和一组cli/agentic/*.py使用示例直接放入了仓库。

在clone变体上，Agent（智能体）面前有一个完整的transformers代码库，大约三分之一的运行会在调用新接口之前先去读取新的表面（/cli/tree和示例脚本）以学习接口。这使得输入Token中位数从约4k上升到约6.4k。

因此，这两个图表是同一权衡的两个方面：该提交为大型模型节省了时间（它们直接使用CLI，而不是调试Python），代价是消耗了更多Token（它们读取了教会它们使用CLI的代码）。这是在合并PR之前值得了解的权衡。

不过，有一个有利于CLI的注意事项尚未经过基准测试：读取它的成本会随着连续运行而摊销。我们的设置是为一次性实验构建的。每次运行都是一个全新的Agent（智能体），从头开始重新发现CLI，因此它每次都要支付发现成本。在实际使用中，Agent（智能体）学习一次接口，然后在同一会话中一个接一个地解决问题，将该成本分摊到多个请求中。我们这里测量到的Token增加更接近最坏情况，而非用户日常会看到的情况。

### 小型模型：固定版本，变化模型

开源模型让我们能对这里最重要的变量进行精细控制：大小、配置、量化、提供商、训练，以及任何模型间可能不同的因素。同时，良好的工具表面对于小型模型也最为重要：一个被要求“在bare环境中使用transformers做X”的小型模型，可能会猜测一个已在几个版本前更改过的API，可能会进行不必要的工具调用，并且可能得出错误答案。

因此，这里的实验与上述相反：固定版本，遍历模型。这有助于观察哪些模型真正能完成任务，不仅仅是根据Token数量和时间，而是深入到哪些模型无法可靠地处理工具调用。我们的直觉是，模型越小，工具使用和任务难度都越大；我们在一系列模型大小上运行了测试框架，正是为了验证这一点：

![按层级划分的各模型匹配率](/images/posts/a15031b1d99b.png)

## 调整工具：标记与结果

这里涉及两件事：如何超越Agent（智能体）是否成功，去观察它做了什么以及如何做的；以及我们从测试框架中提取出的首批结果。

### 什么是标记？

匹配率、Token数量和时间能告诉你一次运行的成本，但不能告诉你太多关于底层发生了什么的信息。

这就是我们引入标记概念的原因。标记是一个命名的模式，配置文件（教测试框架如何构建和驱动给定库的每个小工具插件）会针对一次运行进行匹配。

它是针对你关心的行为的一行标签，根据Agent（智能体）运行的shell命令、编写的代码、读取的文件或其最终答案进行检查。一次运行可以触发多个标记，也可以一个都不触发；报告会显示每个标记在每个模型和每个版本下的触发频率。

对于transformers，我们声明了几个标记，但我们只关注最相关的两个：

- cli：Agent（智能体）调用了transformers命令行工具（例如transformers classify …），而不是编写Python代码。
- pipeline：它使用了高级的pipeline(...) Python API。

我们通过观察这些标记来判断一个更改是否确实改变了Agent（智能体）的行为。有趣的是，模型越大，它越倾向于利用新的上下文而不是使用其记忆；因此也更倾向于利用新引入的CLI。

跨模型的按层级划分的CLI采用率：只有skill层级会使用它，并且随着模型增大，采用率更高。

![跨模型的按层级划分的CLI采用率](/images/posts/b82077616de4.png)

CLI采用是新的：CLI在单个提交中引入，不在任何模型的训练数据中，并且文档很少。效果很明显：正是Skill变体（即附带CLI文档的变体）实际使用了它，采用率为55.3%。

### CLI + Skill提交有帮助吗？

比较不同模型大小的提交，CLI + Skill帮助了较大的模型：在skill层级上，Kimi和其他大型Agent（智能体）会使用CLI并在更少的轮次内完成。（在clone层级上，它们首先花费更多输入Token来读取新的CLI代码，正如我们上面看到的，因此优势体现在时间和轮次上，而不是原始Token上。）

跨版本的Kimi-K2.6、GLM-5.1和MiniMax-M2.7

![跨版本的Kimi-K2.6、GLM-5.1和MiniMax-M2.7](/images/posts/4f1dd4b6fc77.png)

但在某些较小模型的设置中，它似乎损害了性能。一个合理的解释是，小型模型依赖记忆的API模式，复现它们在训练数据中见过的pipeline(...)代码片段。新的概念对它们来说是一个更容易出错的大表面。你可以直接在测试框架上观察到这一点：匹配率更低，重试次数更多，cli标记几乎不触发。这在Qwen3-4B模型上尤为明显：Skill几乎不改变其匹配率，但其成本分布却受到了显著影响。

几乎全部来自 `theclonetier`。结账流程现在包含 CLI 的实现和 `cli/agentic/*.py` 示例，而 4B Agent 会批量读取它们：其中位数新 Token 从约 2.4k 跃升至约 23k，时间和输出量也随之飙升，但准确率毫无提升。

Qwen3-4B 在各版本间的表现。CLI + Skill 提交将成本分布大幅拉宽，在 `theclonetier` 上，Agent 批量读取了新发布的 CLI 源码（新 Token 量约为原来的 10 倍），但匹配率毫无提升。（重复 Token 保持平稳：此设置未使用提示词缓存。）

![Qwen3-4B 在各版本间的成本分布：耗时、新 Token、重复 Token、输出 Token](/images/posts/1cb3ee533e8f.png)

不过，有时 Skill 会直接破坏正确性。查看追踪记录可以发现，例如对于 Qwen3-14B：添加 Skill 使其整体匹配率从 67%（裸版）降至 43%，而在最简单的任务上，这种崩溃非常明显：`classify-sentiment` 在 `clone` 变体上从 100% 降至使用 Skill 时的 0%。

Qwen3-14B 在 `classify-sentiment` 任务上按层级划分：`clone`（蓝色）在各版本中保持 100%，但 Skill 变体（绿色）在 CLI + Skill 版本中崩溃至 0%。

![Qwen3-14B classify-sentiment 在各版本间的匹配率按层级划分](/images/posts/8d5a1eb13bd0.png)

查看追踪记录，模型将 CLI 误认为是一个可以直接调用的工具（类似于 agentic-harness 中的工具，如 web-search）。Skill **并非**可执行工具：它是加载到 Agent 上下文中的文档，而 `transformers` CLI 始终只能通过 shell（通过 `bash`）运行；因此这行不通。

Qwen3-14B 读取了 Skill，在其 56 次 Skill 运行中，有 39 次要么发出了 `transformers(command="classify", ...)` 工具调用（一个从未注册过的工具），要么在其 `read`/`bash`/`edit`/`write` 工具中找不到类似工具，从而得出结论：无法运行模型并放弃。无论哪种情况，它都没有回退到在 `clone` 结账流程中取得 100% 得分的一行 `pipeline(...)` 代码，而是声明任务不可行。

Qwen3-14B 在 classify-sentiment 任务上（Skill 变体）：它推断 `read`/`bash`/`edit`/`write` 无法运行模型，并放弃。

![Qwen3-14B 在 Skill 变体下放弃 classify-sentiment 任务](/images/posts/48535a0a651b.png)
