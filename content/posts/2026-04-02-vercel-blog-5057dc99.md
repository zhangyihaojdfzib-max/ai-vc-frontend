最近，我们在 Vercel Sandbox 中推出了文件系统快照功能，让团队能够捕获并恢复沙箱的整个文件系统状态。最初，我们的工程重点完全放在可靠性上，确保系统永远不会出现快照失败或数据丢失的情况。

当这一基础稳定后，我们的注意力转向了性能。p75 快照恢复耗时超过 40 秒，通过并行化和本地缓存，我们将其降低到了 1 秒以下。

## 快照在磁盘上的样子

Vercel Sandbox 运行在与我们内部构建产品 Hive 相同的基础设施上。每个沙箱都是 Firecracker 微虚拟机内的一个隔离容器。

快照是沙箱磁盘的压缩副本。我们处理两种不同的文件：

- 原始磁盘镜像（.img），可能有数 GB
- 我们自定义 VHS 格式（Vercel Hive Snapshot）的压缩版本，该版本会上传到 S3 并从 S3 下载

当你调用 sandbox.snapshot() 时，我们会将 .img 压缩成 .vhs 并上传到 S3。当你使用快照调用 Sandbox.create() 时，我们会下载 .vhs 并将其解压还原。如果没有压缩，每次快照操作都会通过网络传输数百 MB 到数 GB 的数据，每次恢复都会增加数秒到数十秒的时间。

## 必须并行化

在可靠性就绪后，我们转向了恢复路径，这一过程原本是顺序执行的，非常耗时。我们通过单个请求从 S3 下载整个 .vhs 文件，等待下载完成，然后通过单个线程进行解压。

快照大小从 200MB 到数 GB 不等，因此仅 S3 下载一项就可能耗时数秒到数十秒。我们改用 Range HTTP 标头并行下载分块，由 AWS Go SDK 的 transfermanager API 负责编排。在对不同的并发级别和分块大小进行基准测试后，我们将下载速度提升了 2-5 倍。

我们将同样的思路应用于解压。我们的 .vhs 格式为磁盘镜像的每个已分配区域存储一个头部和一个帧，因此我们不再逐个解码和解压帧，而是改用单个解码器向 N 个解压协程分发数据。这使得 .vhs 到 .img 的恢复速度提升了 2-4 倍，具体取决于快照大小。

即使下载和解压都已并行化，流程仍然需要先将下载的数据写入磁盘，然后才能开始解压。将 S3 范围请求流直接通过管道送入解压过程，消除了这一中间步骤，将端到端恢复时间又缩短了 2 倍。

## 我们……没做缓存？

你可能已经注意到，到目前为止我们只讨论了如何改进慢路径，即缓存未命中时需要从 S3 检索快照的情况。实际上，我们根本没有快路径，所以所有情况都是缓存未命中。是的，我们最初确实没有关注性能。

我们的沙箱运行在配备 NVMe 磁盘的裸金属实例上，拥有数 TB 的快速本地存储，大部分处于闲置状态。我们利用 LRU（最近最少使用）淘汰策略，以总磁盘空间而非条目数量为基准，将其用于本地磁盘缓存。我们直接缓存解压后的 .img 文件，而不是压缩后的 .vhs 文件，因此缓存命中可以跳过下载和解压两个步骤。当缓存填满时，最近最少使用的快照会被淘汰以腾出空间。

大多数客户会在多个沙箱中重复使用一个“基础”快照，这使我们的缓存命中率达到 95%。在这些命中情况下，启动时间仅受限于启动微虚拟机和容器。

## 从 40 秒到亚秒级

p75 从 40 秒降至亚秒级，p95 从 50 秒降至 5 秒。凭借我们的缓存命中率，大多数沙箱启动完全跳过了下载和解压流程。

我们还可以做更多。例如，缓存亲和性可以将沙箱路由到已缓存所请求快照的裸金属实例，从而可能消除热门快照的冷路径。但这有风险，可能导致惊群效应并使某些机器过热，因此我们正在慎重考虑。

长期来看，我们希望冷路径足够快，使缓存成为锦上添花，而非必需。

这些优化已经为自动持久化功能提供了支持，该功能目前处于测试阶段，当你停止命名沙箱时，它会自动快照其文件系统，并在恢复时还原所有内容。凭借亚秒级的恢复，这种停止和恢复的循环感觉是即时的。

文件系统快照现已适用于所有 Vercel Sandbox。请查看 Sandbox 文档开始使用。
