# 2026年2月20日Cloudflare服务中断事件

2026-02-21

- David Tuber
- Dzevad Trumic

![](/images/posts/900da7832108.png)

2026年2月20日UTC时间17:48，Cloudflare发生服务中断，部分使用Cloudflare"自带IP"服务的客户，其通往互联网的路由通过边界网关协议被撤销。

该问题并非直接或间接由网络攻击或任何类型的恶意活动引起。此次事件是由Cloudflare对我们网络管理通过BYOIP流程接入的IP地址方式所做的一项变更导致的。这项变更导致Cloudflare无意中撤销了客户的前缀。

对于一些BYOIP客户，这导致他们的服务和应用程序无法从互联网访问，造成其使用BYOIP的Cloudflare部署出现连接超时和失败。Cloudflare递归DNS解析器的网站也出现了403错误。事件总持续时间为6小时7分钟，其中大部分时间用于将前缀配置恢复到变更前的状态。

Cloudflare工程师在开始观察到故障时已回滚了变更，前缀停止被撤销。然而，在工程师能够回滚变更之前，约有1,100个BYOIP前缀已从Cloudflare网络中被撤销。部分客户能够通过使用Cloudflare控制台重新宣告其IP地址来恢复自身服务。我们在恢复所有前缀配置后解决了此次事件。

我们对给客户造成的影响深表歉意。今天我们让您失望了。本文将深入详述事件发生的具体经过，以及哪些系统和流程出现了故障。我们还将概述我们正在采取的步骤，以防止此类中断再次发生。

## Cloudflare的Addressing API

Addressing API是Cloudflare网络上存在的地址的权威数据集。对该数据集的任何更改都会立即反映在Cloudflare的全球网络中。虽然我们正在改进这些系统推出变更的方式，作为"Code Orange: Fail Small"计划的一部分，但目前客户可以通过与面向公众的API交互来配置其IP地址，这些API配置一组数据库，从而触发将变更传播到Cloudflare边缘的操作工作流。这意味着对Addressing API的更改会立即传播到Cloudflare边缘。

在Cloudflare上宣告和配置IP地址涉及几个步骤：

- 客户通过Addressing API或BGP控制向Cloudflare发出宣告/撤销IP地址的信号
- Addressing API指示机器更改前缀宣告
- 一旦足够多的机器收到更新前缀的通知，路由器上的BGP将被更新
- 最后，客户可以通过服务绑定配置Cloudflare产品以使用BYOIP地址，这将把产品分配给这些地址范围

Addressing API使我们能够自动化围绕宣告或撤销地址的大部分流程，但某些流程仍需要手动操作。这些手动流程因其与生产环境的紧密关联而具有风险。作为"Code Orange: Fail Small"的一部分，修复目标之一就是移除Addressing API中执行的手动操作，并用安全的工作流取而代之。

## 事件是如何发生的？

导致故障的具体配置项是一项修改，该修改试图自动化客户从Cloudflare BYOIP服务中移除前缀的操作——这是当前需要手动处理的常规客户请求。移除这个手动流程是我们"Code Orange: Fail Small"工作的一部分，旨在将所有变更推向安全、自动化、健康检查中介的部署。由于BYOIP前缀的相关对象列表可能很大，因此该功能被实现为定期运行的子任务的一部分，该子任务检查应被移除的BYOIP前缀，然后将其移除。不幸的是，这个定期清理子任务在查询API时存在缺陷。

以下是清理子任务的API查询：

```javascript
 resp, err := d.doRequest(ctx, http.MethodGet, `/v1/prefixes?pending_delete`, nil)

```

以下是API实现的相关部分：

```javascript
	if v := req.URL.Query().Get("pending_delete"); v != "" {
		// 忽略其他行为，从 ip_prefixes_deleted 表中获取待删除对象
		prefixes, err := c.RO().IPPrefixes().FetchPrefixesPendingDeletion(ctx)
		if err != nil {
			api.RenderError(ctx, w, ErrInternalError)
			return
		}

		api.Render(ctx, w, http.StatusOK, renderIPPrefixAPIResponse(prefixes, nil))
		return
	}

```

由于客户端传递的 `pending_delete` 参数没有值，`Query().Get("pending_delete")` 的结果将是一个空字符串（""），因此 API 服务器将其解释为请求所有 BYOIP 前缀，而不仅仅是那些本应被删除的前缀。系统将此解释为所有返回的前缀都已排队等待删除。随后，新的子任务开始系统地删除所有 BYOIP 前缀及其所有相关的依赖对象（包括服务绑定），直到影响被注意到，并由工程师识别出该子任务并将其关闭。

### 为什么恢复不是立即的？

受影响的 BYOIP 前缀并非都以相同方式受到影响，因此需要更复杂的数据恢复步骤。作为"Code Orange: Fail Small"计划的一部分，我们正在构建一个系统，可以通过健康状态介导的部署安全地推出操作状态快照。如果确实推出了导致意外行为的内容，可以非常快速地回滚到已知的良好状态。然而，该系统目前尚未在生产环境中运行。

在此次事件中，BYOIP 前缀处于不同的影响状态，每种不同的状态都需要不同的操作：

*   大多数受影响的客户只是其前缀被撤销。处于此配置的客户可以进入仪表板并切换其广告，从而恢复服务。
*   一些客户的前缀被撤销，并且部分绑定被移除。这些客户处于部分恢复状态，他们可以切换某些前缀，但无法切换其他前缀。
*   一些客户的前缀被撤销，并且所有服务绑定都被移除。他们无法在仪表板中切换其前缀，因为没有服务（Magic Transit、Spectrum、CDN）绑定到这些前缀。这些客户的缓解时间最长，因为必须启动全局配置更新，为所有这些客户重新应用服务绑定到 Cloudflare 边缘网络的每一台机器。

## 补救和后续步骤

### API 模式标准化

此次事件中的一个问题是，`pending_delete` 标志被解释为字符串，这使得客户端和服务器都难以合理化该标志的值。我们将改进 API 模式以确保更好的标准化，这将使测试和系统更容易验证 API 调用格式是否正确。这项工作是第三个"Code Orange"工作流的一部分，旨在在所有条件下创建定义明确的行为。

### 操作状态和配置状态之间更好的分离

目前，客户对寻址模式所做的更改会持久保存在一个权威数据库中，而该数据库也用于操作行动。这使得手动回滚流程更具挑战性，因为工程师需要利用数据库快照，而不是在期望状态和实际状态之间进行合理化。我们将重新设计回滚机制和数据库配置，以确保我们有一种简单的方法来快速回滚更改，并在客户配置和生产环境之间引入隔离层。

我们将对从数据库读取并应用于生产环境的数据进行快照，并以部署所有其他生产变更相同的方式应用这些快照，整个过程由健康指标监控，若出现问题可自动停止部署。这意味着，下次若数据库再次出现异常状态，我们能够近乎即时地将个别客户（或所有客户）回滚至正常运行的版本。

尽管在故障期间，这将暂时阻止客户通过我们的API直接进行更新，但这意味着我们可以在修复数据库期间继续处理他们的流量，而非完全中断服务。此项工作与第一、第二项"橙色代码"工作流目标一致，即实现快速回滚以及通过健康监控的安全配置部署。

### 优化大规模撤回操作的仲裁机制

我们将改进监控系统，以检测变更是否发生得过于迅速或广泛（例如快速撤回或删除BGP前缀），并在检测到此类情况时禁用快照部署。这将形成一种"熔断机制"，阻止任何失控的数据库操作流程造成大规模影响，正如本次事件中所见。

我们还在持续推进另一项工作：直接监控客户运行的服务是否表现正常，这些监控信号同样可用于触发熔断机制，在完成调查前阻止潜在的危险变更被应用。此项工作与第一项"橙色代码"工作流目标一致，即实现安全变更部署。
