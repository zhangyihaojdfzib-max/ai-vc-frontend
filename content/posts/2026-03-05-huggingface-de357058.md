# 介绍 Modular Diffusers —— 用于扩散流程的可组合构建模块

Modular Diffusers 引入了一种通过组合可复用模块来构建扩散流程的新方法。您无需从头编写整个流程，而是可以通过混合搭配模块来创建满足您需求的定制工作流！这为现有的 `DiffusionPipeline` 类提供了一个更灵活、可组合的替代方案。

在本文中，我们将详细介绍 Modular Diffusers 的工作原理——从运行模块化流程的熟悉 API，到构建完全自定义的模块并将其组合到您自己的工作流中。我们还将展示它如何与 Mellon（一个基于节点的可视化工作流界面）集成，您可以使用 Mellon 将 Modular Diffusers 模块连接在一起。

## 快速开始

以下是一个使用预构建模块运行 `FLUX.2 Klein 4B` 推理的简单示例：

```python
import torch
from diffusers import ModularPipeline


pipe = ModularPipeline.from_pretrained(
    "black-forest-labs/FLUX.2-klein-4B"
)

pipe.load_components(torch_dtype=torch.bfloat16)
pipe.to("cuda")


image = pipe(
    prompt="a serene landscape at sunset",
    num_inference_steps=4,
).images[0]

image.save("output.png")

```

您会得到与标准 `DiffusionPipeline` 相同的结果，但流程的内部结构却大不相同：它由灵活的**模块**组成——文本编码、图像编码、去噪和解码——您可以直接检查这些模块：

```python
print(pipe.blocks)

```

```bash
Flux2KleinAutoBlocks(
  ...
  Sub-Blocks:
    [0] text_encoder (Flux2KleinTextEncoderStep)
    [1] vae_encoder (Flux2KleinAutoVaeEncoderStep)
    [2] denoise (Flux2KleinCoreDenoiseStep)
    [3] decode (Flux2DecodeStep)
)

```

每个模块都是独立的，拥有自己的输入和输出。您可以独立运行任何模块作为其自己的流程，也可以自由地添加、移除和交换模块——它们会动态重组以与剩余的模块协同工作。使用 `.init_pipeline()` 将模块转换为可运行的流程，使用 `.load_components()` 加载模型权重。

```python

blocks = pipe.blocks


text_blocks = blocks.sub_blocks.pop("text_encoder")


text_pipe = text_blocks.init_pipeline("black-forest-labs/FLUX.2-klein-4B")


text_pipe.load_components(torch_dtype=torch.bfloat16)
text_pipe.to("cuda")
prompt_embeds = text_pipe(prompt="a serene landscape at sunset").prompt_embeds



remaining_pipe = blocks.init_pipeline("black-forest-labs/FLUX.2-klein-4B")
remaining_pipe.load_components(torch_dtype=torch.bfloat16)
remaining_pipe.to("cuda")
image = remaining_pipe(prompt_embeds=prompt_embeds, num_inference_steps=4).images[0]

```

有关模块类型、组合模式、延迟加载以及使用 `ComponentsManager` 进行内存管理的更多信息，请查阅 **Modular Diffusers 文档**。

## 自定义模块

Modular Diffusers 在创建您自己的模块时真正大放异彩。自定义模块是一个 Python 类，它定义了其组件、输入、输出和计算逻辑——一旦定义，您就可以将其插入任何工作流。

### 编写自定义模块

以下是一个使用 `Depth Anything V2` 从图像中提取深度图的示例模块。

```python
class DepthProcessorBlock(ModularPipelineBlocks):
    @property
    def expected_components(self):
        return [
            ComponentSpec("depth_processor", DepthPreprocessor,
                          pretrained_model_name_or_path="depth-anything/Depth-Anything-V2-Large-hf")
        ]

    @property
    def inputs(self):
        return [
            InputParam("image", required=True,
                       description="Image(s) to extract depth maps from"),
        ]

    @property
    def intermediate_outputs(self):
        return [
            OutputParam("control_image", type_hint=torch.Tensor,
                        description="Depth map(s) of input image(s)"),
        ]

    @torch.no_grad()
    def __call__(self, components, state):
        block_state = self.get_block_state(state)
        depth_map = components.depth_processor(block_state.image)
        block_state.control_image = depth_map.to(block_state.device)
        self.set_block_state(state, block_state)
        return components, state

```

- `expected_components` 定义了模块需要的模型——在本例中是一个深度估计模型。`pretrained_model_name_or_path` 参数设置了默认加载的 Hub 仓库，因此除非您在 `modular_model_index.json` 中覆盖，否则 `load_components` 会自动获取深度模型。
- `inputs` 和 `intermediate_outputs` 定义了输入和输出内容。
- `__call__` 是计算逻辑所在之处。

### 将模块组合成工作流

让我们将这个模块用于 Qwen 的 ControlNet 工作流。提取 ControlNet 工作流并在开头插入深度模块：

```python

pipe = ModularPipeline.from_pretrained("Qwen/Qwen-Image")

print(pipe.blocks.available_workflows)








blocks = pipe.blocks.get_workflow("controlnet_text2image")

print(blocks)



blocks.sub_blocks.insert("depth", DepthProcessorBlock(), 0)


blocks.sub_blocks['depth'].doc

```

序列中的模块会自动共享数据：深度模块的 `control_image` 输出会流向需要它的下游模块，而其 `image` 输入由于没有更早的模块提供，因此成为流程的输入。

![blocks_composed](/images/posts/684dbd076894.png)

```python
from diffusers import ComponentsManager, AutoModel
from diffusers.utils import load_image



manager = ComponentsManager()

pipeline = blocks.init_pipeline("Qwen/Qwen-Image", components_manager=manager)
pipeline.load_components(torch_dtype=torch.bfloat16)




controlnet = AutoModel.from_pretrained("InstantX/Qwen-Image-ControlNet-Union", torch_dtype=torch.bfloat16)
pipeline.update_components(controlnet=controlnet)


image = load_image("https://huggingface.co/datasets/huggingface/documentation-images/resolve/main/diffusers/astronaut.jpg")
output = pipeline(
    prompt="an astronaut hatching from an egg, detailed, fantasy, Pixar, Disney",
    image=image,
).images[0]

```

## 模块化仓库

`ModularPipeline.from_pretrained` 开箱即用地适用于任何现有的 Diffusers 仓库，但 Modular Diffusers 还引入了一种新型仓库：**模块化仓库**。

模块化仓库能够从其原始模型仓库引用组件。例如，`diffusers/flux2-bnb-4bit-modular` 包含一个量化后的 transformer，并从原始仓库加载其余组件。

```json

{
    "transformer": [
        "diffusers", 
        "Flux2Transformer2DModel", 
        {
            "pretrained_model_name_or_path": "diffusers/flux2-bnb-4bit-modular",
            "subfolder": "transformer",
            "type_hint": ["diffusers", "Flux2Transformer2DModel"]
        }
    ],
    "vae": [
        "diffusers", 
        "AutoencoderKLFlux2", 
        {
            "pretrained_model_name_or_path": "black-forest-labs/FLUX.2-dev",
            "subfolder": "vae",
            "type_hint": ["diffusers", "AutoencoderKLFlux2"]
        }
    ],
    ...
}

```

模块化仓库还可以托管自定义流程模块的 Python 代码以及用于 **Mellon** 等工具的可视化 UI 配置——所有这些都集中在一处。

## 与 Mellon 集成

💡 Mellon 处于早期开发阶段，尚未准备好用于生产环境。请将此视为集成工作原理的抢先预览！

**Mellon** 是一个与 Modular Diffusers 集成的可视化工作流界面。如果你熟悉像 ComfyUI 这样的基于节点的工具，你会感到非常熟悉——但有一些关键区别：

- **动态节点** — 我们不是提供数十个特定于模型的节点，而是拥有一小组节点，它们会根据你选择的模型自动调整其界面。学习一次，即可用于任何模型。
- **单节点工作流** — 得益于 Modular Diffusers 的可组合块系统，你可以将整个流水线折叠成一个节点。在同一画布上运行多个工作流，而不会显得杂乱。
- **开箱即用的 Hub 集成** — 发布到 Hugging Face Hub 的自定义块可以在 Mellon 中立即使用。我们提供了一个实用函数，可以根据你的块定义自动生成节点界面——无需编写 UI 代码。

这种集成之所以可能，是因为每个块都暴露了相同的属性（`inputs`、`intermediate_outputs`、`expected_components`）。这种一致的 API 意味着 Mellon 可以从任何块定义自动生成节点的 UI，并将块组合成更高级别的节点。

例如，`diffusers/FLUX.2-klein-4B-modular` 包含一个流水线定义、组件引用和一个 `mellon_pipeline_config.json` 文件——全部在一个仓库中。在 Python 中使用 `ModularPipeline.from_pretrained("diffusers/FLUX.2-klein-4B-modular")` 加载它，或在 Mellon 中加载以创建单节点或多节点工作流。

这里有一个简单的例子。我们向一个现有的文生图工作流添加一个 Gemini 提示词扩展节点——该节点作为一个模块化仓库托管在 `diffusers/gemini-prompt-expander-mellon`：

1.  拖入一个 **Dynamic Block** 节点并输入 `repo_id`（即 `diffusers/gemini-prompt-expander-mellon`）
2.  点击 **LOAD CUSTOM BLOCK** — 节点会自动生成一个用于输入提示词的文本框和一个名为 "prompt" 的输出接口，所有这些都根据仓库配置完成
3.  输入一个简短的提示词，将输出连接到 **Encode Prompt** 节点，然后运行

在生成图像之前，Gemini 会将你的简短提示词扩展为详细描述。无需代码，无需配置——只需一个 Hub 仓库 id。

这只是一个例子。如需详细教程，请查看 **Mellon x Modular Diffusers 指南**。
