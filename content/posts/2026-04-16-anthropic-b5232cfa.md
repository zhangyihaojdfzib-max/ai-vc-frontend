# 推出 Claude Opus 4.7

![](/images/posts/7ebb38ccb652.jpg)

我们的最新模型 Claude Opus 4.7 现已全面上市。

Opus 4.7 在高级软件工程方面相比 Opus 4.6 有显著提升，尤其是在最困难的任务上取得了特别的进展。用户反馈称，他们现在可以放心地将最棘手的编码工作——那种以前需要密切监督的工作——交给 Opus 4.7。Opus 4.7 能够严谨、一致地处理复杂、长期运行的任务，精确关注指令，并在反馈前设计方法来验证自己的输出。

该模型的视觉能力也大幅提升：它可以以更高的分辨率查看图像。在完成专业任务时，它更具品味和创造力，能产出更高质量的界面、幻灯片和文档。并且——尽管其整体能力不如我们最强大的模型 Claude Mythos Preview——但在一系列基准测试中，它都取得了比 Opus 4.6 更好的结果：

![](/images/posts/bfbaacb7fad1.jpg)

上周我们宣布了“玻璃翼项目”，强调了 AI 模型在网络安全方面的风险和益处。我们表示将限制 Claude Mythos Preview 的发布范围，并首先在能力较弱的模型上测试新的网络安全防护措施。Opus 4.7 就是第一个这样的模型：其网络安全能力不如 Mythos Preview 先进（事实上，在训练过程中，我们尝试了差异化降低这些能力的方法）。我们发布 Opus 4.7 时配备了防护措施，能自动检测并阻止表明是禁止或高风险网络安全用途的请求。我们从这些防护措施的实际部署中学到的经验，将帮助我们朝着最终广泛发布 Mythos 级别模型的目标迈进。

希望将 Opus 4.7 用于合法网络安全目的（如漏洞研究、渗透测试和红队演练）的安全专业人员，欢迎加入我们的新“网络安全验证计划”。

Opus 4.7 现已通过所有 Claude 产品、我们的 API、Amazon Bedrock、Google Cloud 的 Vertex AI 和 Microsoft Foundry 提供。定价与 Opus 4.6 保持一致：输入 Token 每百万个 5 美元，输出 Token 每百万个 25 美元。开发者可以通过 Claude API 使用 `claude-opus-4-7`。

## 安全性与对齐

总体而言，Opus 4.7 显示出与 Opus 4.6 相似的安全特性：我们的评估显示，在欺骗、谄媚、配合滥用等令人担忧的行为方面，其发生率较低。在某些指标上，例如诚实度和抵抗恶意“提示词注入”攻击的能力，Opus 4.7 比 Opus 4.6 有所改进；在其他方面（例如其对受管制物质提供过于详细的减害建议的倾向），Opus 4.7 略有不足。我们的对齐评估结论是，该模型“大体上对齐良好且值得信赖，但其行为并非完全理想”。请注意，根据我们的评估，Mythos Preview 仍然是我们训练过的对齐度最好的模型。我们的安全评估在《Claude Opus 4.7 系统卡》中有完整讨论。

## 今日同步发布

除了 Claude Opus 4.7 本身，我们还发布了以下更新：

- **更多努力程度控制**：Opus 4.7 在 `high` 和 `max` 之间引入了一个新的 `xhigh`（“特高”）努力级别，让用户能更精细地控制在难题上的推理与延迟之间的权衡。在 Claude Code 中，我们已将所有计划的默认努力级别提升至 `xhigh`。在测试 Opus 4.7 用于编码和 Agent 用例时，我们建议从 `high` 或 `xhigh` 努力级别开始。
- **在 Claude 平台（API）上**：除了支持更高分辨率的图像，我们还推出了任务预算功能的公开测试版，为开发者提供一种引导 Claude Token 消耗的方式，使其能在较长的运行中优先处理工作。
- **在 Claude Code 中**：新的 `/ultrareview` 斜杠命令会产生一个专门的审查会话，通读更改并标记出细心的审查者会发现的错误和设计问题。我们为 Pro 和 Max 级别的 Claude Code 用户提供三次免费的 ultrareview 试用机会。此外，我们已将自动模式扩展至 Max 用户。自动模式是一种新的权限选项，Claude 可代表您做出决定，这意味着您可以运行更长时间的任务而减少中断，并且比选择跳过所有权限的风险更低。

## 从 Opus 4.6 迁移到 Opus 4.7

Opus 4.7 是 Opus 4.6 的直接升级版，但有两项变化值得规划，因为它们会影响 Token 使用量。首先，Opus 4.7 使用了更新的分词器，改进了模型处理文本的方式。代价是相同的输入可能映射到更多的 Token——根据内容类型，大约为 1.0–1.35 倍。其次，Opus 4.7 在更高的努力级别下会进行更多思考，尤其是在 Agent 设置中的后续轮次。这提高了其在难题上的可靠性，但也意味着它会产生更多的输出 Token。

用户可以通过多种方式控制 Token 使用量：使用努力程度参数、调整任务预算，或提示模型更加简洁。在我们自己的测试中，净效果是积极的——如下图所示，在内部编码评估中，所有努力级别的 Token 使用量都有所改善——但我们建议在实际流量上测量差异。我们编写了一份迁移指南，提供了从 Opus 4.6 升级到 Opus 4.7 的进一步建议。
