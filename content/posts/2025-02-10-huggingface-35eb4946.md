# Open R1：第二次更新

![image/png](/images/posts/679ef9a13148.png)

Open R1 项目现已启动两周，该项目旨在重建 DeepSeek R1 缺失的部分——具体来说，是训练流程和合成数据。

在这篇文章中，我们很高兴地分享 OpenR1-Math-220k 的构建成果：这是我们在数学推理领域的第一个大规模数据集！

我们还将关注社区在整理用于微调的小型高质量数据集方面取得的一些令人兴奋的进展，以及关于如何在训练时和推理时控制推理模型思维链长度的见解。

让我们开始吧！

## OpenR1-Math-220k 数据集

DeepSeek R1 的关键优势之一在于它能够通过蒸馏将高级推理能力迁移到较小的模型上。DeepSeek 团队通过生成 60 万条推理轨迹并对一系列 Qwen 和 Llama 模型进行微调，证明了这一点，表明直接从 R1 进行蒸馏可以在无需强化学习的情况下实现具有竞争力的推理性能。值得注意的是，DeepSeek-R1-Distill-Qwen-7B 在 AIME 2024 上达到了 55.5% 的准确率，超越了 QwQ-32B-Preview 等更大的模型。

然而，用于蒸馏的推理轨迹并未公开发布，这促使社区独立重建类似的数据集。到目前为止，社区已经发布了多个开源数据集，包括 OpenThoughts-114k、Bespoke-Stratos-17k、Dolphin-R1 和 LIMO。

🐳 隆重推出 OpenR1-Math-220k，这是一个大规模数学推理数据集，在 512 块 H100 上本地生成，每个问题包含多个答案。为了创建 OpenR1-Math-220k，我们与 Numina 合作，他们开发了其广受欢迎的 NuminaMath-CoT 数据集的全新版本。

OpenR1 数据集与现有数据集相比的新特性：

- **80 万条 R1 推理轨迹**：我们使用 DeepSeek R1 为 40 万个问题生成了两个答案。经过筛选的数据集包含 **22 万个问题**及其正确的推理轨迹。
- **512 块 H100 本地运行**：我们不再依赖 API，而是利用 vLLM 和 SGLang 在我们的科学集群上本地生成数据，每天生成 **18 万条推理轨迹**。
- **基于 NuminaMath 1.5**：我们专注于数学推理轨迹，为 NuminaMath 1.5（NuminaMath-CoT 数据集的改进版本）中的问题生成答案。
- **自动筛选**：我们应用 Math Verify 仅保留至少有一个正确答案的问题。我们还利用 Llama3.3-70B-Instruct 作为评判者来检索更多正确的示例（例如，对于答案格式错误、无法通过基于规则的解析器验证的情况）。
- **性能匹配**：通过在我们的数据集上微调 Qwen-7B-Math-Instruct，我们达到了与 DeepSeek-Distill-Qwen-7B 相当的性能。

通过展示可扩展的高质量推理数据生成，我们希望这一流程能够扩展到数学以外的领域，例如代码生成。

### 数据生成

为了构建 OpenR1-220k，我们提示 DeepSeek R1 为 NuminaMath 1.5 中的 40 万个问题生成解答。我们遵循模型卡片推荐的参数，并在用户提示前添加以下指令：

“请逐步推理，并将最终答案放在 \boxed{} 中。”

我们将每次生成的 Token 限制设为 1.6 万，因为我们的分析显示，只有 75% 的问题能在 8000 Token 以内解决，而其余大部分问题需要完整的 1.6 万 Token。最初，我们使用 vLLM 进行推理，每块 H100 每小时可生成 15 个解答，并在之前的更新中和 Open R1 仓库中分享了我们的生成脚本。最近，我们开始尝试使用 SGLang，每块 H100 每小时能够生成 25 个解答（速度提升近 2 倍！），这使得我们能够在 512 块 H100 上每天生成 30 万个问题解答。这让我们在短短几天内就生成了 80 万条推理轨迹。

我们为每个问题生成两个解答——在某些情况下是四个——以便在筛选和训练中提供灵活性。这种方法允许进行拒绝采样，类似于 DeepSeek R1 的方法论，同时也使数据集适用于 DPO 等偏好优化方法。

未筛选的数据集可在此处获取：https://huggingface.co/datasets/open-r1/OpenR1-Math-Raw

### 数据筛选

为了仅保留高质量、正确的推理轨迹，我们利用 Math Verify，这是一个稳健的数学表达式评估系统，旨在评估 LLM 生成的答案。我们从模型生成的内容中提取最终答案，并将其与数据集中的标准答案进行比较。

我们发现 55% 的问题至少有一个正确答案。然而，NuminaMath 1.5 中的一些标准答案为空或格式无法验证，这使得自动验证具有挑战性。虽然我们已经改进了 Math-Verify 以更准确地处理这些不太常见的输出格式（见下文 Math-Verify 的改进），但我们还探索了另一种方法，从被拒绝的样本中恢复有效的解答：使用 Llama-3.3-70B-Instruct 作为评判者对一部分被拒绝的问题进行评估。在执行此验证步骤之前，我们筛选掉不完整或标准答案为空的样本，确保只考虑格式正确且最终答案明确框出的回答。此过程成功恢复了 2.8 万个之前被拒绝的问题。

我们按如下方式提示 Llama3.3-70B-Instruct：

```
你是一个数学答案验证器。你将获得一个数学问题，你需要比较参考答案中的答案和模型解答中的最终答案，以确定它们是否等价，即使格式不同。

问题：

{problem}

参考答案：

{answer}

模型解答：

{generation}

只关注比较模型提供的最终数学答案，忽略以下方面的差异：

- 格式（例如，\\boxed{{}} 与纯文本）
- 选择题格式（例如，“A”与完整解答）
- 坐标对或解答的顺序
- 等价的数学表达式或符号变体
- 如果模型的答案毫无意义，则返回“判定：模糊”

首先简要解释你的比较（2-3 句话）。然后以以下格式之一输出你的最终答案：

- “判定：等价”
- “判定：不同”
- “判定：模糊”
```

通过将基于规则的验证（Math Verify）与基于 LLM 的评估相结合，我们在保持规模的同时提高了数据集质量。最终数据集包含 22 万个带有已验证推理轨迹的问题，使其成为训练推理模型的宝贵资源。为每个问题提供多个解答，使社区能够灵活地筛选出更好的生成结果，并根据 NuminaMath 数据源和问题类型进行更有针对性的优化。

![image/png](/images/posts/b6694316b649.png)

该数据集分为两个子集：

- **默认子集**（9.4 万个问题），在 SFT 后达到最佳性能。
- **扩展子集**（13.1 万个问题），包含来自 cn_k12 等额外 NuminaMath 1.5 数据源的更多推理轨迹。然而，我们观察到，在此子集上进行 SFT 后的性能低于默认子集，这可能是由于 cn_k12 包含的问题比其他数据源更简单。

对于包含多个正确答案的行，我们还尝试使用奖励模型（RM）作为最终筛选器来选择最佳回答。对于每一行中由R1生成的多个正确答案，我们通过移除思考Token（<think>…</think>）提取最终答案，然后将问题与提取的答案输入到通过vLLM部署的Qwen/Qwen2.5-Math-RM-72B模型中以获取评分。利用这些评分，我们对每个包含多个正确答案的行进行排序，选择排名第一的正确生成结果纳入训练数据集。但遗憾的是，训练消融实验表明，与随机选择一个正确生成结果相比，这种方法并未提升模型性能。可能的改进方向是在使用RM评分时纳入推理轨迹，而非仅依赖最终答案。

### Math-Verify的改进

在检查验证结果时，我们发现了Math-Verify中的若干失败案例。为解决这些问题，我们实施了重大改进与修复。强烈建议更新至最新版本（0.5.2）以享受这些增强功能：

```python
pip install math-verify==0.5.2

```

以下是最重要改进的摘要：

- 改进了纯文本答案的解析与验证（例如 $\text{E}$ 等价于 $E$）
- 改进了答案列表的解析（例如 $1$ 和 $2$ 和 $3$ 等价于 $1,2,3$）
- 修复了单个LaTeX环境中多个框选答案的解析（例如 $\boxed{1},\boxed{2}$ 等价于 {1,2}）
- 引入有序元组。推断列表是元组还是集合非常困难，因此我们使用标准答案作为指导：(1,2,3) ≠ {3,2,1}; 1,2,3 == {3,2,1}; {3,2,1} == {1,2,3}
- 支持标准答案中的关系表达式（例如小于）和预测中的区间表示（例如 $1 < x < 2$ 等价于 $(1,2)$）

- (1,2,3) ≠ {3,2,1}; 1,2,3 == {3,2,1}; {3,2,1} == {1,2,3}

## 社区亮点

本周社区从多个角度探索了GRPO，同时多个研究实验室表明，仅需约1000个高质量训练样本即可激发现有开源模型的推理能力。

### GRPO的实际应用

- nrehiew展示了将GRPO直接应用于Qwen2.5-0.5B基础模型，在GSM8k基准上达到约51%的准确率，比Qwen2.5-0.5B-Instruct模型提升10个百分点。此类令人瞩目的结果引发了关于指令数据在预训练中作用的广泛讨论，因为人们尚未能在Llama 3等其他基础模型上通过GRPO获得类似增益。特别是，Sea AI Lab（SAIL）的研究人员表明，基础模型可被轻松提示以产生自我反思，而DeepSeek-R1论文中的“顿悟”时刻可能更多是基础模型的特性，而非RL优化过程的结果。
- Unsloth运用其优化魔法，使参数量高达15B的模型仅需15GB显存即可通过GRPO进行训练🤯。这意味着你现在可以在Google Colab中免费使用GRPO！
- Axolotl的Wing Lian展示了DoRA的收敛速度优于LoRA和全参数微调。
- Alexander Doria找到了一种为诗歌设计奖励函数的方法。这令人兴奋，因为它提供了GRPO应用于非传统“可验证”领域的首批公开案例之一。

### 评估

AIME 2025的第一部分于本周发布，包含15道用于训练高中生参加国际数学奥林匹克竞赛的难题。过去一年中，AIME 2024一直是检验LLM数学能力的主要基准，社区热切关注模型在新题集上的表现：

- ETH Zurich的研究人员评估了一系列闭源和开源模型，发现性能漂移远低于预期，通常在10-20个百分点范围内。
- 然而，Dimitris Papailiopoulos发现AIME 2025中的多道题目已存在于互联网论坛！这可能构成意外的训练-测试泄漏，凸显了为LLM创建新颖问题的难度。

### LLM是否需要以自然语言进行推理？

![image/png](/images/posts/f9408323c8cd.png)

一篇有趣的新研究论文表明，通过使用循环语言模型，可以在潜在空间中进行隐式推理，从而扩展测试时计算。这类似于Meta的Coconut工作——在潜在空间中训练语言模型，但现已适配至推理任务。这些方法的优势在于计算效率更高：通过探索潜在空间，无需生成大量“思考”Token即可获得高性能。

### 转向更小规模、高质量的推理数据？

尽管DeepSeek R1利用60万条推理轨迹进行蒸馏，但近期研究表明，复杂推理能力并非必须通过大规模训练，而是可以通过少量精心筛选的样本在语言模型中涌现。

此类方法的一个例子是s1K数据集。它包含1000道精心挑选的数学题，并附带从Gemini Flash中蒸馏的推理轨迹。筛选方法聚焦于难度、多样性和质量。作者在s1K上微调Qwen2.5-32B-Instruct，在竞赛数学基准上比OpenAI的o1-preview高出最多27%。

另一个数据集LIMO进一步推进了这一思路，仅使用817个训练样本便在AIME和MATH基准上取得强劲表现。作者假设，当模型在预训练阶段已获取广泛领域知识时，仅需少量结构良好的示例即可解锁高级推理能力。

### CoT长度：预算强制与奖励塑造

s1K中微调后的Qwen2.5-32B-Instruct模型能达到如此强劲性能的关键因素之一是预算强制——一种测试时计算技术，通过向模型生成结果追加“Wait”或思考结束Token分隔符来分别延长或截断推理过程。该工具使作者能够调整思考时间，并得出结论：模型展现出测试时扩展性——随着思考时间增加，不同数学基准上的准确率也随之提升。

![image/png](/images/posts/9d1f2e60cd62.png)

类似地，《Demystifying Long Chain-of-Thought Reasoning in LLMs》（Yeo等人）也研究了思维链（CoT）长度对模型性能的影响。他们引入了余弦奖励——一种新颖的奖励函数，用于激励正确生成时使用较短CoT、错误生成时使用较长CoT——从而稳定RL训练，尤其在模型最大上下文长度相对有限且平均响应长度可能爆炸时。当模型在难题上开始出现通过重复而非尝试求解来人为增加CoT长度的奖励黑客行为时，还会采用重复惩罚。
