# 解锁连续批处理中的异步性

摘要：我们将解释如何分离CPU和GPU工作负载，从而大幅提升推理性能。

这是关于高效LLM推理系列的第二篇文章。第一篇文章从基本原理出发介绍了连续批处理。它引入了一些我们将在此基础上构建的概念：KV缓存、FlashAttention、注意力掩码等。

在Inference Endpoints上，一块H200每小时大约花费5美元。按小时算很便宜，但用上一天就要支付120美元。如果是这样，你肯定希望GPU得到充分利用。我们已经看到，连续批处理通过调度紧密打包的批次来提高GPU利用率，这样就不会在填充上浪费算力。但还有第二个浪费来源是连续批处理没有解决的：默认情况下，它是同步的。这意味着CPU和GPU轮流工作：当GPU计算时，CPU等待；而当CPU准备下一个批次时，GPU等待。在以每秒数百步运行的循环中，这些空闲间隙会累积起来，正如我们将要展示的，它们可能占到总运行时间的近四分之一。为了确保GPU100%的时间都在忙碌计算，我们需要消除这些间隙。

为此，我们可以使用异步批处理：我们将把CPU批次准备与GPU批次计算分离开来，这样两者可以并行运行，我们就能始终拥有一个高效工作的GPU 🔥

## 同步批处理

这就是朴素同步批处理的工作方式：

![同步批处理](/images/posts/eadb6482a4d7.png)

当CPU准备新批次时，它会选择包含哪些请求，更新KV缓存表，驱逐在前一轮运行中完成的请求，并接纳新请求来填充释放的空间。完成后，它将准备好的输入传输到GPU。GPU执行前向传播并为每个请求采样（即选择）一个新Token。结果返回给CPU，这样CPU就知道每个请求刚刚生成了什么Token，然后整个循环再次重复。

注意右侧的红色标注：GPU完成计算后，它就进入空闲状态。下一个批次必须等到CPU完成其更新步骤（对输出Token进行采样、更新请求状态、重新调度批次）后才能开始。

这就是同步批处理的核心低效之处：CPU和GPU轮流工作。当GPU计算时，CPU空闲；当CPU更新时，GPU空闲。在任何情况下，它们都不会同时做有用的事情。对于单次前向传播来说，这似乎是一个很小的代价，但在每秒运行数百步的连续批处理循环中，这些空闲间隙会累积成实际的吞吐量损失。

为了说明这一点，我们分析了在使用8B模型、批次大小为32、生成8K Token时，CPU和GPU各自花费的时间：

![CPU和GPU活动时间线](/images/posts/b7165b08b17e.png)

如果你想生成类似的图表，可以对连续批处理代码进行插桩，以转储CPU和GPU活动区间，并使用这个脚本。

时间线在绿色（GPU活跃，CPU空闲）和红色（CPU活跃，GPU空闲）之间交替：两者从未重叠。总生成时间为300.6秒，其中24.0%的时间GPU处于空闲状态，等待CPU完成。从GPU的角度来看，近四分之一的总生成时间被浪费了。这是悲观的观点。

乐观的观点是，如果我们能完全消除CPU开销，生成时间将从300秒降至228秒（免费获得24%的加速！）。这不需要新的内核或模型更改，只需要对硬件进行精心的协调。

从根本上说，这个想法很简单：我们需要弄清楚如何在批次N计算的同时，为批次N+1运行批次准备。但这个简单的想法隐藏着一些技术难点：

- 我们如何在GPU上启动某些操作，同时将控制权交还给CPU？
- 我们如何确保在CPU或GPU任务启动时，数据已经准备就绪？
- 如果批次N+1基于批次N的预测结果，我们如何准备它？

通过回答这些问题，我们将从头构建异步批处理。我们遵循相同的步骤，在transformers库中将其作为连续批处理的一部分实现。欢迎查看代码并进行比较！

## 创建并发

我们的最终目标是实现CPU和GPU操作的并发执行。我们需要一种对操作进行分类的方法，以便让机器知道哪些操作可以并发运行。我们可以使用CUDA流来实现这一点。

### 什么是CUDA流？

要理解CUDA如何对其操作进行排序，我们需要谈谈CUDA流。流是一个有序的GPU操作队列（内核启动、内存拷贝、同步屏障），这些操作按照提交的顺序执行。每个GPU操作总是在某个流内调度。同一流内的操作是顺序执行的：GPU在前一个操作完成之前不会启动下一个操作。不同流中的操作彼此独立，可以并发运行。为了说明这一点，如果你在3个不同的流中启动3个操作，执行过程如下所示：

![CUDA流并发](/images/posts/00966ef83a5a.png)

所有三个操作同时开始。这稍微简化了一些：每个GPU操作最终都是由CPU发起的，而发起过程需要少量时间：找到合适的内核、发出调用、将命令从CPU传输到GPU等。这被称为CPU启动开销，更现实的图表如下所示：

![现实的CUDA流并发](/images/posts/1bb18e8ac7ac.png)

操作仍然是并发的，但它们的开始时间因每次CPU启动的开销而错开。我们将继续在整篇文章中展示这些CPU启动事件，因为它们会占用实际时间，并且在我们转向异步工作流时，它们将帮助我们追踪"何时启动了哪些操作"。例如，我们经常会检查一个流是否已刷新：这意味着该流中的所有操作都已被执行。

### 默认流和非默认流

如果你从未在PyTorch中显式使用过CUDA流，你可能会惊讶于它们的存在。典型的PyTorch脚本从不提及它们，而且感觉上GPU操作并不是异步的：CPU似乎在继续执行之前会等待GPU完成。这种感觉是准确的，这源于默认流。

当你调用PyTorch操作而未指定流时，它会落在默认流上。默认流有一个特殊属性：它是同步的。如果某个操作被调度到默认流上，它会等待所有其他流被刷新，即GPU上的所有工作必须在默认流上的单个操作开始之前完成。反之亦然：任何操作，无论其属于哪个流，都会在启动前等待默认流被刷新。

因此，如果你将默认流操作的结果传输到CPU，即使使用的是本应对CPU非阻塞的传输，你的CPU仍然会阻塞，直到所有GPU操作完成，因为这些操作是在默认流上调度的。这实际上摧毁了任何构建并发的努力。

这就是为什么我们需要使用非默认流。将内核启动或非阻塞内存拷贝加入队列会立即将控制权返回给CPU。GPU会在后台运行该操作，但CPU不会等待。这回答了我们第一个问题：要在启动GPU工作后取回CPU控制权，我们使用非默认流。

![阻塞与非阻塞传输](/images/posts/166dd9be9e8d.png)

在本文的其余部分，我们将假设所有从一个设备到另一个设备的内存传输都是非阻塞的。因此，我们必须自己同步它们。

### 回到连续批处理

我们已确定，任何GPU操作都不应落在默认流上。但问题依然存在：如果不使用默认流，我们应该使用哪些流？让我们回到同步批处理的示意图：

我们可以识别出三种不同的GPU操作：

1. 将输入从CPU传输到GPU
2. 在GPU上进行计算
3. 将输出从GPU传输到CPU

这意味着我们需要三个流：一个用于计算，一个用于CPU到GPU的传输，一个用于GPU到CPU的传输。传输操作是相互独立的，因此没有理由将它们串行化，每个传输操作都有自己的流。

关于术语的说明：在讨论CPU和GPU时，CUDA文档中使用的惯例是将CPU称为主机（host），将GPU称为设备（device）。从现在起我们将采用这一惯例。CPU到GPU的传输称为主机到设备（H2D）传输，GPU到CPU的传输称为设备到主机（D2H）传输。因此，这三个流分别是H2D流、计算流和D2H流。

现在让我们尝试使用流来异步地在GPU上启动一个批次并收回CPU控制权。从CPU的角度，我们执行以下操作：

1. 在CPU上准备批次输入数据（无流，仅CPU操作）
2. 将其传输到GPU（使用H2D流）
3. 在GPU上运行计算（使用计算流）
4. 取回批次输出（使用D2H流）
5. 查看结果（无流）

如果我们仅使用CUDA流来执行此操作，结果几乎会立即返回，但它们是错误的。为了理解原因，让我们看看发生了什么：

![失败的异步批处理](/images/posts/88ba186850bf.png)

由于流之间相互独立，所有三个GPU操作几乎同时启动。计算流没有等待H2D传输完成，因此前向传播运行在GPU内存中已有的任何数据上。D2H流没有等待计算完成，因此它传输了尚未计算出的结果。第5步立即返回，因为没有任何东西阻塞CPU：没有默认流需要同步。

这些操作在孤立情况下都正确运行。问题在于我们从未告诉流要相互等待。我们知道计算必须在H2D完成后开始，D2H必须在计算完成后开始，但我们没有强制执行这种顺序。我们需要一种机制来跨流边界声明“在此操作完成之前，不要启动该操作”。

## 强制执行同步

为了在流之间强制执行同步，我们将使用CUDA事件。

### 什么是CUDA事件？

CUDA事件是一个可以记录到流中的标记。当GPU在执行过程中到达该标记时，它会将该事件标记为已完成。然后可以告诉任何其他流在开始其下一个操作之前等待该事件。具体来说，有两个操作：`stream.record(event)`，将标记插入到流中的当前位置；以及`stream.wait(event)`，阻止流继续执行，直到该事件被标记为完成。重要的是，`wait`阻塞的是流，而不是CPU或其他并行运行的流：CPU调用会立即返回，只有等待的流被阻止。

![CUDA事件](/images/posts/ebe0c2b5151c.png)

上图展示了一个事件同步两个流的过程。CPU快速连续发出三个操作（三个小方块）：在流1上启动输入准备，在流1上记录事件，然后告诉流2等待该事件。然后CPU立即继续执行。流1运行其操作，当操作完成时，事件被设置。流2在此期间一直被阻塞在等待标记处，只有在事件被标记为完成后才开始计算。CPU没有参与其中任何环节：顺序完全在GPU端强制执行。

### 在连续批处理中使用事件

应用到我们的场景中，修复方法很简单。在将H2D传输加入队列后，我们调用`h2d_stream.record(h2d_done)`：只有当传输完成时，该事件才会被标记为已完成。在将前向传播加入队列之前，我们调用`compute_stream.wait(h2d_done)`，这样计算流在`h2d_done`被设置之前不会启动。我们在计算和D2H之间也做同样的事情：在通过`model.forward`启动前向传播后，我们调用`compute_stream.record(compute_done)`，然后在将输出传输加入队列之前调用`d2h_stream.wait(compute_done)`。结果是一个具有显式顺序的流水线：

1. H2D传输在`h2d_stream`上运行
2. `compute_stream`等待`h2d_done`，然后运行前向传播
3. `d2h_stream`等待`compute_done`，然后将输出传输回来

CPU按顺序将所有操作加入队列，然后继续执行。CPU在任何时候都不会阻塞。GPU通过事件强制执行顺序，并且所有三个流在其依赖条件满足后立即变为活动状态。

![成功的异步批处理](/images/posts/cf173bb697b7.png)

上图展示了这是如何展开的。CPU准备批次，然后快速将所有GPU工作加入队列：H2D传输、前向传播、D2H传输，并在每个阶段之间插入`record`和`wait`调用。之后，CPU就空闲了。GPU接管，按照依赖事件被设置的顺序依次执行每个流。注意右侧的绿色注释：一旦D2H传输完成，CPU返回并读取结果。这个最终的同步是整个步骤中CPU唯一阻塞的点。为了实现这一点，我们在输出传输之后在D2H流上记录第三个事件，然后在CPU端调用`d2h_done_event.synchronize()`。`synchronize`会阻塞CPU，直到D2H流到达该标记。

这是与同步批处理的关键区别：之前，CPU在每个操作之后都会阻塞。现在，在GPU工作时，CPU可以自由地做“某件事”。我们需要弄清楚那件“某件事”是什么，因为从GPU利用率的角度来看，目前没有任何变化。

## 填补空白

CPU可用的时间窗口位于将批次N分派给GPU和将批次N+1分派给GPU之间。其自然用途是准备批次N+1的输入，这样我们就可以将它们分派给GPU，并在批次N计算结束时让它们准备就绪。让我们看看如何做到这一点。

为了准备批次N+1，我们可以重用准备批次N时使用的相同CPU端对象：当前请求列表、缓存状态、主机端张量缓冲区等。但是，我们需要注意两件事：

- 数据损坏：批次N+1的设备端输入缓冲区不能与批次N的相同：我们会破坏GPU仍在读取的数据
- 数据传输：如果一个请求同时出现在批次N和批次N+1中，并且它在批次N的输出中生成了一个新的Token，那么该Token在批次N+1的输入中是必需的

我们将在接下来的两节中解决这些问题，即数据损坏和数据传输。

### 竞态条件

首先，我们将解决潜在的数据损坏问题。假设批次N和批次N+1共享相同的设备端输入缓冲区，并且批次N+1输入的H2D传输在批次N仍在计算时开始。CPU可能会写入批次N+1的输入，而GPU仍在从同一内存中读取批次N的输入。因此，GPU可能会读取到部分被覆盖的数据，结果就是数据损坏。这是一种竞态条件。主机端也存在同样的风险：在批次N的H2D复制仍在进行时重用相同的源进行复制，会破坏传输。

解决方法是使用两组张量并在它们之间交替使用。当GPU从插槽A处理批次N时，CPU使用批次N-1的结果更新请求的状态。然后CPU在输入插槽B中准备批次N+1。下一步，它们交换。如下图所示：

![输入/输出插槽](/images/posts/d56d23f3e2ef.png)

当然，这需要付出代价：它会使用于存储输入和输出张量的 RAM 和 VRAM 量翻倍。这是一个可接受的权衡，尤其是在使用 FlashAttention 时，因为它不需要注意力掩码，而注意力掩码是迄今为止最大的输入张量。

但拥有两个槽位会带来另一个问题。在推理中，我们通常使用 CUDA graphs 来降低延迟。简而言之，CUDA graph 是一个预先记录的 CUDA 操作序列。它是针对特定内存地址记录的：为槽位 A 捕获的图无法针对槽位 B 的缓冲区重放。所以我们需要两个图。如果每个图都有自己的内存缓冲区，那又会占用双倍的 VRAM。

解决方案是一个内存池：一个两个图都可以从中分配的共享内存缓冲区。唯一的限制是同一个池中的两个图绝不能同时执行。由于批次 N 必须在批次 N+1 开始之前完成，情况总是如此。实际上，两个图一起使用的 VRAM 量几乎与一个图相同。我们只在初始化时支付两次捕获的成本。我们可以在同一个池中创建任意数量的 CUDA graph，总内存使用量仍然受限于所有图中的最大值。如下所示。

![CUDA graph 内存池](/images/posts/8577ee59c5de.png)

现在我们知道了如何防止数据损坏，我们可以解决第二个问题：将批次 N 的输出 Token 送入批次 N+1 的输入。

### 延续

考虑一个同时出现在批次 N 和批次 N+1 中的请求。在批次 N 中，它产生了一个新 Token。这个 Token 是它在批次 N+1 中的输入。问题在于，当我们准备批次 N+1 的输入缓冲区时，我们还没有那个 Token：批次 N 仍在运行。

为了解决这个问题，我们在构建批次 N+1 时使用一个占位符 Token。我们将使用 0 作为占位符，原因稍后会变得清晰。我们在批次 N 计算完成之后、批次 N+1 开始前向传播之前替换那个占位符。我们将这一步称为延续，因为我们正在将新 Token 从批次 N 延续到批次 N+1。延续背后的思想如下所示：

![延续原理](/images/posts/59c013da70c6.png)

要执行延续，我们只需要三样东西：批次 N 的输出 Token ID、批次 N+1 的输入 Token ID，以及一个包含如何执行延续指令的张量。我们将这个张量称为延续掩码。它包含需要延续的 Token 的目标位置，对于不需要延续的位置则为 -1。我们在下面表示一个：

![延续掩码](/images/posts/976e4bb01f4c.png)

延续本身由四个操作组成：

- 我们从批次 N 的输出中选择要延续的 Token 到一个新的张量 T 中
- 我们将 T 中不想延续的 Token 置零
- 我们截断 T 以匹配批次 N+1 的输入长度
- 我们将 T 加到批次 N+1 的输入 ID 上（这就是为什么占位符输入 ID 的值为零）

由于这四个操作非常廉价，我们在每个新批次开始时执行它们，并将延续操作捕获到 CUDA graph 中。如果延续掩码只包含 -1（值为 -1 表示：不要延续此位置），那么最后一步就是与零张量相加。这种情况不常发生，因为跨越多个批次的解码请求通常会被调度到连续的批次中。

## 完整的异步循环

让我们把所有内容整合起来，并追踪前两步。

步骤 0 是冷启动：没有前一个批次在运行，因此 CPU 在槽位 A 中准备批次 0，并像同步批处理一样将其调度出去。还没有重叠。

![异步回顾](/images/posts/872f5b3929a8.png)

步骤 1 是异步循环开始的地方。GPU 现在在槽位 A 上运行批次 0，CPU 空闲。它立即开始在槽位 B 中准备批次 1：驱逐已完成的请求、接纳新请求、更新 KV 缓存路由表、构建延续掩码。所有这些都与 GPU 完全重叠运行。一旦批次 1 的输入准备就绪，CPU 按顺序将工作排入队列：它为槽位 B 启动 H2D 传输，记录并等待计算和 D2H 流的事件，然后继续。

![异步回顾](/images/posts/d8c06bca860b.png)

现在，GPU 上并行发生两件事。在槽位 A 上，GPU 完成计算并设置 compute_done，这会释放批次 0 输出的 D2H 传输。在槽位 B 上，批次 1 输入的 H2D 传输正在运行。一旦完成，h2d_done 事件被设置，批次 1 的计算开始。从批次 0 到批次 1 的延续是计算的一部分：它发生在常规前向传播之前。由于槽位 A 和槽位 B 是独立的，所有这些都可以自由重叠。

![异步回顾](/images/posts/d5cf2eaf13b3.png)

与此同时，CPU 阻塞在 d2h_done_event.synchronize() 上，直到批次 0 的输出到达。然后它处理输出，更新批次 0 中所有请求的状态，并开始调度批次 2。循环现在正在运行，后续的每一步都遵循完全相同的模式。我们在下面展示了完整的工作负载。每个槽位都有专用的颜色来表示 CPU 和 GPU 操作以及事件（事件也是特定于槽位的）。为了可读性，我们没有显示 CPU 启动 GPU 操作（如计算或数据移动），但它们仍然会发生。这是合理的，因为与显示的操作相比，启动 GPU 操作的延迟可以忽略不计。

![异步回顾](/images/posts/f071f85a3ab9.png)

只要批次 N 完成时批次 N+1 的输入已在 GPU 上准备就绪，GPU 就永远不会在批次之间空闲。唯一的问题是 CPU 是否在 GPU 完成计算之前完成其工作。情况通常如此：模型不断增长，而批次调度仍然相对廉价，因此 GPU 计算是瓶颈，而不是 CPU。

## 它真的有效吗？

为了找出答案，我们运行了与之前相同的实验：8K Token，批次大小 32，8B 模型。

![CPU 和 GPU 活动时间线](/images/posts/641392ac5cfb.png)

时间线几乎完全是深绿色：CPU 和 GPU 同时运行。偶尔出现的浅绿色细条是 GPU 活跃但 CPU 已完成准备工作并处于等待状态的时刻。几乎不可见的红色标记是批次之间的同步点，CPU 在此处阻塞以采样批次 N 的输出。GPU 在总运行时间中的活跃时间从 76.0% 上升到 99.4%。总生成时间从 300.6 秒下降到 234.5 秒，加速了 22%。我们预测如果完全消除 CPU 开销，加速比可达 24%。剩余的小差距是那个不可避免的同步点。没有新的内核，没有模型更改：只是让 CPU 和 GPU 同时工作。

## 结论

我们从 CPU 和 GPU 依次工作的同步工作负载开始，导致两者都未充分利用。通过从基于调度的依赖关系转向基于数据的依赖关系并优化同步点，我们成功地将 CPU 和 GPU 的工作负载解耦，使得两者的并行执行成为可能。因此，我们能够填满 GPU 工作队列，确保它始终在运行。这最终在保持模型准确性的同时，大幅提升了生成速度。这几乎是一个稳赢的局面。
