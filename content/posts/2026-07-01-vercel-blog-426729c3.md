服务绑定使得一个 Vercel 服务能够安全地在同一部署中调用另一个服务。

当一个服务声明对另一个服务的绑定时，Vercel 会自动注入配置的环境变量。这意味着用户代码可以正常获取该 URL，而 Vercel 在后台处理内部重写、路由、身份验证和 TLS。

这使得在 Vercel 上可以实现多服务应用，例如一个 Next.js 前端调用一个 FastAPI 后端，同时保持服务隔离并简化路由配置：

```
1{2  "services": {3    "my_frontend": {4      "root": "frontend/",5      "framework": "nextjs",6      "bindings": [7        { 8          "type": "service", 9          "service": "my_backend", 10          "format": "url", 11          "env": "BACKEND_INTERNAL_URL" 12        }13      ]      14    },15    "my_backend": {16      "root": "backend/",17      "entrypoint": "main:app"18    }19  }20}
```

在上面的例子中，`my_frontend` 可以通过向 `BACKEND_INTERNAL_URL` 发起 `fetch()` 请求来访问 `my_backend` 服务。

```
1export async function GET() {2  const url = new URL("/users", process.env.BACKEND_INTERNAL_URL);3  const res = await fetch(url);4  const users = await res.json();5  return Response.json(users);6}
```

前端使用注入的内部 URL 调用后端。流量永远不会离开 Vercel 的网络。

在 FastAPI 端，`my_backend` 暴露了一个常规路由：

```
1from fastapi import FastAPI2
3app = FastAPI()4
5@app.get("/users")6def get_users():7    return [ ... ]
```

## 复制链接到标题底层原理

该 URL 指向 Vercel 内部，而非公共地址。当你的代码获取该 URL 时，请求会通过 Vercel 的内部网络到达与处理你部署中所有其他请求相同的路由层，然后由该层将请求发送到目标服务。它永远不会触及公共路由表，并且 Vercel 会为你设置 TLS 信任，因此普通的 HTTPS `fetch` 无需证书配置即可正常工作。

只有当你暴露服务时，它才可被访问——要么通过重写路由公开暴露，要么通过绑定私有暴露。
