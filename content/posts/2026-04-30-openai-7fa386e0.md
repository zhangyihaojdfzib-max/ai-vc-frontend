# 扩展PostgreSQL以支撑8亿ChatGPT用户

作者：Bohan Zhang，技术团队成员

多年来，PostgreSQL一直是支撑ChatGPT和OpenAI API等核心产品最关键、最底层的数据库系统之一。随着用户群的快速增长，我们对数据库的需求也呈指数级增长。过去一年中，我们的PostgreSQL负载增长了10倍以上，并且仍在快速上升。

我们为维持这一增长而推进生产基础设施的努力揭示了一个新的见解：PostgreSQL可以扩展到可靠地支持比许多人先前认为可能的大得多的读密集型工作负载。这个最初由加州大学伯克利分校的一个科学家团队创建的系统，使我们能够通过单个主Azure PostgreSQL灵活服务器实例（在新窗口打开）和分布在全球多个区域的近50个只读副本，来支持庞大的全球流量。这就是我们如何在OpenAI扩展PostgreSQL，通过严格的优化和扎实的工程，为8亿用户每秒支持数百万次查询的故事；我们还将介绍在此过程中学到的关键经验。

## 初始设计中的裂痕

ChatGPT发布后，流量以前所未有的速度增长。为了支撑这一增长，我们在应用层和PostgreSQL数据库层迅速实施了广泛的优化，通过增加实例大小进行垂直扩展，并通过添加更多只读副本进行水平扩展。这种架构在很长一段时间内为我们提供了良好的服务。随着持续的改进，它继续为未来的增长提供充足的空间。

听起来可能令人惊讶，一个单主架构能够满足OpenAI规模的业务需求；然而，在实践中实现这一点并不简单。我们曾多次遇到由Postgres过载引起的SEV（严重事件），它们通常遵循相同的模式：上游问题导致数据库负载突然飙升，例如缓存层故障导致的大范围缓存未命中、大量昂贵的多表连接耗尽CPU，或新功能发布引发的写入风暴。随着资源利用率攀升，查询延迟增加，请求开始超时。重试进一步放大了负载，引发了一个可能使整个ChatGPT和API服务降级的恶性循环。

![扩展负载图](/images/posts/decb0c8fffb3.jpg)

尽管PostgreSQL在我们的读密集型工作负载下扩展良好，但在高写入流量期间我们仍然遇到挑战。这主要是由于PostgreSQL的多版本并发控制（MVCC）实现，它使得写密集型工作负载的效率较低。例如，当一个查询更新一个元组甚至单个字段时，整个行都会被复制以创建一个新版本。在重写入负载下，这会导致显著的写入放大。它还会增加读取放大，因为查询必须扫描多个元组版本（死元组）才能检索到最新版本。MVCC还带来了额外的挑战，如表和索引膨胀、索引维护开销增加以及复杂的自动清理调优。（你可以在我与卡内基梅隆大学的Andy Pavlo教授合写的一篇名为《我们最讨厌的PostgreSQL部分》（在新窗口打开）的博客中找到对这些问题的深入探讨，该博客被引用（在新窗口打开）在PostgreSQL维基百科页面上。）

## 将PostgreSQL扩展到每秒数百万次查询

为了缓解这些限制并减少写入压力，我们已经并将继续将可分片（即可以水平分区的工作负载）、写密集型工作负载迁移到分片系统，如Azure Cosmos DB，并优化应用逻辑以最小化不必要的写入。我们也不再允许向当前的PostgreSQL部署添加新表。新的工作负载默认使用分片系统。

即使我们的基础设施已经发展，PostgreSQL仍然保持不分片，由一个主实例处理所有写入。主要原因是分片现有的应用工作负载将非常复杂且耗时，需要更改数百个应用端点，可能需要数月甚至数年。由于我们的工作负载主要是读密集型，并且我们已经实施了广泛的优化，当前的架构仍然提供了充足的空间来支持持续的流量增长。虽然我们不排除将来对PostgreSQL进行分片，但鉴于我们为当前和未来增长拥有足够的空间，这并非近期优先事项。

在以下部分中，我们将深入探讨我们面临的挑战以及我们为应对这些挑战和防止未来中断而实施的广泛优化，将PostgreSQL推向极限并将其扩展到每秒数百万次查询（QPS）。

### 减少主实例的负载

挑战：只有一个写入器，单主设置无法扩展写入。严重的写入峰值可能迅速使主实例过载，并影响ChatGPT和API等服务。

解决方案：我们尽可能减少主实例的负载——包括读取和写入——以确保它有足够的容量来处理写入峰值。读取流量尽可能卸载到副本上。然而，某些读取查询必须保留在主实例上，因为它们是写入事务的一部分。对于这些查询，我们专注于确保它们高效并避免慢查询。对于写入流量，我们已经将可分片、写密集型的工作负载迁移到分片系统，如Azure Cosmos DB。难以分片但仍产生高写入量的工作负载需要更长时间迁移，这一过程仍在进行中。我们还积极优化了应用程序以减少写入负载；例如，我们修复了导致冗余写入的应用错误，并在适当的地方引入了延迟写入，以平滑流量峰值。此外，在回填表字段时，我们强制执行严格的速率限制以防止过度的写入压力。

### 查询优化

挑战：我们在PostgreSQL中识别出几个昂贵的查询。过去，这些查询的突然数量激增会消耗大量CPU，拖慢ChatGPT和API请求。

解决方案：少数昂贵的查询，例如连接多个表的查询，可能会显著降低甚至拖垮整个服务。我们需要持续优化PostgreSQL查询，以确保它们高效并避免常见的在线事务处理（OLTP）反模式。例如，我们曾识别出一个极其昂贵的查询，它连接了12个表，该查询的激增是过去高严重性SEV的原因。我们应尽可能避免复杂的多表连接。如果必须进行连接，我们学会了考虑分解查询并将复杂的连接逻辑移到应用层。许多此类有问题的查询是由对象关系映射（ORM）框架生成的，因此仔细审查它们生成的SQL并确保其行为符合预期非常重要。在PostgreSQL中，长时间空闲的查询也很常见。配置诸如idle_in_transaction_session_timeout之类的超时设置对于防止它们阻塞自动清理至关重要。

### 单点故障缓解

挑战：如果一个只读副本宕机，流量仍然可以路由到其他副本。然而，依赖单个写入器意味着存在单点故障——如果它宕机，整个服务都会受到影响。

解决方案：大多数关键请求只涉及读取查询。为了缓解主实例的单点故障，我们将这些读取从写入器卸载到副本，确保即使主实例宕机，这些请求也能继续服务。虽然写入操作仍然会失败，但影响已经减小；由于读取仍然可用，这不再是SEV0级别的事件。

为了减轻主库故障，我们以高可用（HA）模式运行主库，并配备热备节点——一个持续同步的副本，随时准备接管服务流量。如果主库宕机或因维护需要下线，我们可以快速提升备库以最大限度地减少停机时间。Azure PostgreSQL 团队做了大量工作，确保即使在极高负载下，这些故障转移也能保持安全可靠。为了处理只读副本故障，我们在每个区域部署了多个副本，并留有足够的容量余量，确保单个副本故障不会导致区域级服务中断。

### 工作负载隔离

挑战：我们经常遇到某些请求在 PostgreSQL 实例上消耗不成比例资源的情况。这可能导致同一实例上运行的其他工作负载性能下降。例如，新功能上线可能引入低效查询，大量消耗 PostgreSQL CPU，从而拖慢其他关键功能的请求响应。

解决方案：为了缓解“吵闹邻居”问题，我们将工作负载隔离到专用实例上，确保资源密集型请求的突发峰值不会影响其他流量。具体来说，我们将请求分为低优先级和高优先级两类，并路由到不同的实例。这样，即使低优先级工作负载变得资源密集，也不会降低高优先级请求的性能。我们在不同产品和服务之间也采用了相同的策略，确保一个产品的活动不会影响另一个产品的性能或可靠性。

### 连接池

挑战：每个实例都有最大连接数限制（Azure PostgreSQL 中为 5,000）。很容易耗尽连接或积累过多空闲连接。我们之前曾因连接风暴导致所有可用连接耗尽而引发事故。

解决方案：我们部署了 PgBouncer 作为代理层来池化数据库连接。在语句或事务池模式下运行它，使我们能够高效地重用连接，大大减少了活跃客户端连接的数量。这也降低了连接建立延迟：在我们的基准测试中，平均连接时间从 50 毫秒下降到 5 毫秒。跨区域连接和请求成本较高，因此我们将代理、客户端和副本部署在同一区域，以最大限度地减少网络开销和连接使用时间。此外，PgBouncer 必须仔细配置。空闲超时等设置对于防止连接耗尽至关重要。

### 缓存

挑战：缓存未命中的突然激增可能会触发对 PostgreSQL 数据库的大量读取，使 CPU 饱和并拖慢用户请求。

解决方案：为了减轻 PostgreSQL 的读取压力，我们使用缓存层来服务大部分读取流量。然而，当缓存命中率意外下降时，突发的缓存未命中会将大量请求直接推送到 PostgreSQL。这种数据库读取的突然增加会消耗大量资源，从而拖慢服务。为了防止缓存未命中风暴期间的过载，我们实现了缓存锁定（和租约）机制，使得只有对特定键未命中的单个读取者才会从 PostgreSQL 获取数据。当多个请求对同一个缓存键未命中时，只有一个请求获取锁并继续检索数据和重新填充缓存。所有其他请求等待缓存更新，而不是同时全部命中 PostgreSQL。这显著减少了冗余的数据库读取，并保护系统免受级联负载峰值的影响。

### 扩展只读副本

挑战：主库将预写日志（WAL）数据流式传输到每个只读副本。随着副本数量的增加，主库必须将 WAL 发送到更多实例，从而增加网络带宽和 CPU 的压力。这会导致更高且更不稳定的副本延迟，使系统更难可靠地扩展。

解决方案：我们在多个地理区域运行了近 50 个只读副本，以最大限度地减少延迟。然而，在当前架构下，主库必须将 WAL 流式传输到每个副本。尽管目前使用非常大的实例类型和高网络带宽可以很好地扩展，但我们不能无限期地添加副本而不最终使主库过载。为了解决这个问题，我们正在与 Azure PostgreSQL 团队合作开发级联复制⁠(在新窗口中打开)，其中中间副本将 WAL 中继到下游副本。这种方法使我们能够扩展到可能超过一百个副本，而不会使主库不堪重负。然而，这也引入了额外的操作复杂性，特别是在故障转移管理方面。该功能仍在测试中；我们将在将其部署到生产环境之前，确保其健壮性并能够安全地进行故障转移。

### 速率限制

挑战：特定端点的流量突发峰值、昂贵查询的激增或重试风暴会迅速耗尽 CPU、I/O 和连接等关键资源，从而导致广泛的服务降级。

解决方案：我们在多个层面（应用程序、连接池、代理和查询）实施了速率限制，以防止突发的流量峰值压垮数据库实例并引发级联故障。避免过短的重试间隔也至关重要，这可能会触发重试风暴。我们还增强了 ORM 层以支持速率限制，并在必要时完全阻止特定的查询摘要。这种有针对性的负载削减方式能够从昂贵的查询突发激增中快速恢复。

### 模式管理

挑战：即使是微小的模式更改，例如更改列类型，也可能触发全表重写⁠(在新窗口中打开)。因此，我们谨慎地应用模式更改——将其限制为轻量级操作，并避免任何重写整个表的操作。

解决方案：只允许轻量级的模式更改，例如添加或删除某些不会触发全表重写的列。我们对模式更改强制执行严格的 5 秒超时。允许并发创建和删除索引。模式更改仅限于现有表。如果新功能需要额外的表，它们必须位于其他分片系统（如 Azure CosmosDB）中，而不是 PostgreSQL。在回填表字段时，我们应用严格的速率限制以防止写入峰值。尽管此过程有时可能需要一周以上，但它确保了稳定性并避免了对生产环境的影响。

## 成果与未来展望

这项工作表明，通过正确的设计和优化，Azure PostgreSQL 可以扩展以处理最大的生产工作负载。PostgreSQL 处理了数百万 QPS 的读取密集型工作负载，为 OpenAI 最关键的产品（如 ChatGPT 和 API 平台）提供支持。我们增加了近 50 个只读副本，同时将复制延迟保持在接近零的水平，在跨地理分布的区域维持了低延迟读取，并构建了足够的容量余量以支持未来的增长。

这种扩展方式在最小化延迟的同时，也提升了可靠性。我们持续实现了个位数毫秒级的p99客户端延迟，并在生产环境中达到了99.999%的可用性。过去12个月里，我们仅发生过一次SEV-0级别的PostgreSQL事故（发生在ChatGPT ImageGen病毒式传播期间，当时一周内新增用户超过1亿，写入流量突然激增10倍以上）。

尽管我们对PostgreSQL目前的表现感到满意，但仍在不断突破其极限，以确保为未来增长留出足够空间。我们已将可分片的写入密集型工作负载迁移至CosmosDB等分片系统。剩余难以分片的写入密集型工作负载——我们也在积极迁移，以进一步减轻PostgreSQL主库的写入压力。同时，我们正与Azure合作实现级联复制，以便安全扩展至更多只读副本。

展望未来，随着基础设施需求持续增长，我们将继续探索更多扩展方案，包括分片PostgreSQL或其他分布式系统。
