# 海底光缆中断、风暴与DNS：2025年第四季度互联网中断事件回顾

2026-01-26

- David Belson

![](/images/posts/cc43e22fc07a.png)

2025年，我们观测到超过180起由各类原因引发的互联网中断事件——其中部分短暂且影响有限，另一些则是持续数天的完全瘫痪。在第四季度，我们仅追踪到一例政府主导的互联网关闭事件，但多起海底光缆中断对多个国家的网络连接造成了严重破坏。电力中断与极端天气在多地影响了互联网服务，而持续的乌克兰冲突也对该国网络连接造成冲击。与往常一样，我们观测到的多起中断事件源于技术问题——部分问题已获相关服务商确认，另一些则原因不明。此外，多家超大规模云平台及Cloudflare的事故也影响了网站与应用的可用性。

本文旨在总结已观测并确认的中断事件，并非本季度所发生问题的完整清单。这些异常情况是通过我们网络中观测到的流量模式与预期值的显著偏差所识别的。完整核验的异常事件与确认中断清单请查阅Cloudflare Radar中断中心。

## 政府主导事件

### 坦桑尼亚

10月29日坦桑尼亚总统选举期间爆发暴力抗议，该国互联网随即被关闭。当地时间12:30（世界标准时间09:30）流量开始下降，较前一周同期暴跌超90%。中断持续约26小时，直至10月30日当地时间14:30（世界标准时间11:30）流量开始恢复。但恢复过程极为短暂，流量在恢复约两小时后（当地时间16:15/世界标准时间13:15）再次大幅下降。第二次近乎完全中断的状态持续至11月3日，当日当地时间17:00（世界标准时间14:00）后流量才强势回升。中断期间亦观测到公布的IPv4与IPv6地址空间出现名义性减少，但从未完全停止地址宣告——若完全停止则意味着该国与互联网彻底断开连接。（自治系统通过向其他互联网服务商宣告IP地址空间，告知其负责的IP地址区块范围。）

坦桑尼亚总统事后就互联网关闭造成的影响，向外交团体成员及旅居该国的外籍人士表示歉意。该国在2020年大选前也曾限制互联网及社交媒体服务。

## 海底光缆中断

### Digicel海地公司

Digicel海地公司对光缆中断导致的网络问题并不陌生，第四季度其网络又遭遇两起类似事件。10月16日，来自Digicel海地公司（AS27653）的流量于当地时间14:30（世界标准时间18:30）开始下降，至当地时间16:00（世界标准时间20:00）趋近于零。该公司总经理在X平台发布的翻译帖文中说明：“特此告知客户，@DigicelHT的国际光纤基础设施正遭遇两处中断。”流量于当地时间17:00（世界标准时间21:00）后开始恢复，并在一小时内达到预期水平。当地时间17:33（世界标准时间21:34），总经理再次发帖称“国际基础设施的首条光纤已修复”，服务得以恢复。

11月25日，该运营商总经理在X平台发布的另一篇翻译帖文称，其“位于国家1号公路的国际光纤基础设施”遭切断。我们观测到Digicel网络流量约提前一小时开始下降，当地时间02:00至08:00（世界标准时间07:00至13:00）期间出现完全中断。当地时间08:22（世界标准时间13:22）的后续X帖文宣布所有服务均已恢复。

### Cybernet/StormFiber（巴基斯坦）

10月20日当地时间17:30（世界标准时间12:30），Cybernet/StormFiber（AS9541）的互联网流量急剧下降，降至一周前同期水平的约50%。同时，该网络公布的IPv4地址空间减少超三分之一。此波动源于PEACE海底光缆受损——该光缆在苏丹附近红海区域发生中断。

PEACE是承载巴基斯坦运营商国际互联网流量的多条海底光缆系统（包括IMEWE与SEA-ME-WE-4）之一。运营商承诺在10月27日前全面恢复服务，但流量与公布的IPv4地址空间已于10月21日当地时间02:00左右（10月20日世界标准时间21:00）恢复至接近预期水平。

喀麦隆电信、MTN喀麦隆、Orange喀麦隆
据报道，10月23日在喀麦隆多家互联网服务商观测到的不寻常流量模式，是由连接非洲西海岸国家与葡萄牙的**WACS（西非电缆系统）**海底光缆故障引起的。一份（翻译后的）公开报告称，MTN告知用户"由于WACS光纤光缆发生事故，互联网服务暂时中断"，Orange喀麦隆也通知用户"因国际接入光纤发生事故，互联网服务中断"。喀麦隆电信（CAMTEL）在X平台发文称："喀麦隆电信（CAMTEL）谨告知公众，2025年10月23日凌晨在巴托克（林贝）发生的涉及WACS光缆设备的技术事故，导致全国范围内的互联网连接中断。"

受影响服务商的流量最初在当地时间约05:00（UTC时间04:00）开始下降，至当地时间约22:00（UTC时间21:00）恢复至预期水平。当日这些网络的流量波动剧烈，有时下降90-99%。目前尚不清楚流量模式中明显峰值的原因——可能是试图将互联网流量切换至连接喀麦隆的其他海底光缆系统。在此期间，**MTN喀麦隆**和**Orange喀麦隆**公布的IP地址空间也出现下降，但**喀麦隆电信**公布的IP地址空间未发生变化。据报道，**中非共和国**和**刚果共和国**的连接也受到WACS问题的影响。

**多米尼加Claro**
12月9日，我们看到多米尼加共和国互联网服务商**Claro Dominicana（AS6400）**的流量在当地时间约12:15（UTC时间16:15）急剧下降。流量在当地时间约14:15（UTC时间18:15）再次下降，较前一周暴跌77%后迅速恢复至预期水平。连接中断可能由两起光纤中断引起，该服务商在中断期间于X平台发文称，它们"导致部分服务出现间歇性和缓慢"。Claro随后在X平台发文称，技术人员已通过修复中断的光纤光缆，在全国范围内恢复了互联网服务。

**停电**
**多米尼加共和国**
根据（翻译后的）多米尼加输电公司（ETED）在X平台的发文，11月11日一条输电线路中断导致**多米尼加共和国**电力服务中断。此次停电影响了该国的互联网流量，导致流量较前一周**下降近50%**，始于当地时间13:15（UTC时间17:15）。流量水平持续较低，直至当地时间12月12日约02:00（UTC时间06:00），ETED后来（翻译后）在X平台发文称"凌晨2点20分，我们已完成国家电力系统的恢复，满足了96%的需求……"随后的**技术报告发现**，"停电始于138 kV圣佩德罗德马科里斯一号变电站，一条带电线路被手动断开，引发了高强度短路。保护系统立即响应，但故障导致附近多条线路断开，使东部地区575兆瓦的发电量与电网其余部分分离。这种不平衡导致主要发电厂作为其内置安全机制的一部分自动跳闸。"

**肯尼亚**
12月12月9日，一次**重大停电**影响了**肯尼亚**多个地区。肯尼亚电力公司解释称，停电"由肯尼亚-乌干达区域互联电网上的事故引发，导致系统肯尼亚侧出现扰动"，并声称"电力在大约30分钟内恢复到大部分受影响地区。"然而，对互联网连接的影响持续了近四个小时，介于当地时间19:15 - 23:00（UTC时间16:15 - 20:00）之间。停电导致全国流量下降高达18%，流量变化在**纳库鲁郡**和**卡扬布郡**最为明显。

**军事行动**
**乌克兰敖德萨**
12月12日，俄罗斯对**乌克兰敖德萨地区**的**无人机袭击**损坏了仓库和能源基础设施，后者导致该地区部分地方停电。这些停电中断了互联网连接，导致流量较前一周**下降高达57%**。在12月13日午夜（UTC时间12月12日22:00）首次下降后，流量在随后几天逐渐恢复，于当地时间12月16日约14:30（UTC时间12:30）回到预期水平。

**天气**
**牙买加**
飓风梅丽莎于10月28日在**牙买加**登陆，沿途造成一系列破坏。相关的**停电**和基础设施损坏影响了互联网连接，导致流量最初**下降约一半**，始于当地时间约06:15（UTC时间11:15），最终较前一周**低70%**。牙买加的互联网流量在几天内持续远低于飓风前水平，最终在11月4日上午开始更明显地朝着预期水平恢复。在造成大规模广泛破坏的风暴过后，一个国家的互联网流量通常需要数周或数月才能恢复到"正常"水平——虽然电力可能在几天内基本恢复，但对物理基础设施的损坏则需要更长时间来解决。

**斯里兰卡和印度尼西亚**
11月26日，**气旋塞尼亚**在**斯里兰卡**和**印度尼西亚**引发灾难性洪水和山体滑坡，造成超过1000人死亡，并损坏了这些国家的电信和电力基础设施。基础设施损坏导致多个地区**互联网连接中断，流量水平降低**。在斯里兰卡，主要西部省以外的地区受影响最严重，多个省的流量较前一周**下降80%至95%**，包括**西北省、南部省、乌瓦省、东部省、北部省、中北省和萨伯勒格穆沃省**。在印度尼西亚，**亚齐**和苏门答腊地区的互联网中断最为严重。在亚齐，流量最初较前一周下降超过75%。在苏门答腊，**北苏门答腊**受影响最大，较前一周初期下降30%，随后一周才开始更积极地恢复。

**已知或未明确的技术问题**
**Smartfren（印度尼西亚）**
10月3日，印度尼西亚互联网服务商**Smartfren（AS18004）**的用户遭遇服务中断。该服务商在X平台发文承认了问题，称（翻译后）"目前，电话、短信和数据服务在多个地区出现问题。"该服务商的流量下降高达84%，始于当地时间约09:00（UTC时间02:00）。中断持续约八小时，流量于当地时间约17:00（UTC时间10:00）恢复至预期水平。Smartfren未就服务问题的原因提供任何额外信息。

**英国沃达丰**
英国主要互联网服务商沃达丰英国（**AS5378**和**AS25135**）于10月23日经历短暂服务中断。当地时间15:00（UTC时间14:00），两个沃达丰**ASN**的流量降至零。**AS5378**公布的IPv4地址空间下降75%，而**AS25135**公布的IPv4地址空间完全消失。互联网流量和地址空间均在两小时后恢复，于当地时间约17:00（UTC时间16:00）回到预期水平。沃达丰未在其社交媒体渠道提供任何关于中断原因的信息，其**网络状态检查页面**在中断期间也无法访问。

**Fastweb（意大利）**
根据一份**公开报告**，10月22日意大利服务商**Fastweb（AS12874）**的客户因DNS解析问题遭遇互联网服务中断，导致观测到的流量下降超过75%。

Fastwe已确认该问题，该问题影响了当地时间09:30至13:00（UTC时间08:30至12:00）的有线互联网客户。虽然这不是由连接故障导致的互联网中断，但DNS解析问题对互联网流量的影响非常相似。当提供商的DNS解析器出现问题时，切换到像Cloudflare的1.1.1.1公共DNS解析器这样的服务通常可以恢复连接。

**SBIN、MTN Benin、Etisalat Benin**
12月7日，观察到SBIN (AS28683)、MTN Benin (AS37424)和Etisalat Benin (AS37136)的流量同时下降。当地时间18:30至19:30（UTC时间17:30至18:30）期间，与国家层面上一周相比，流量下降了高达80%，Etisalat和MTN的流量下降近100%，SBIN的流量下降超过80%。尽管当天早些时候发生了一次未遂政变，但尚不清楚观察到的互联网中断是否与此有任何关联。从路由角度来看，所有三个受影响的网络都共享Cogent (AS174)作为上游提供商，因此Cogent的局部问题可能导致了这次短暂的中断。

**Cellcom Israel**
根据以色列提供商Cellcom (AS1680)的一份报告公告，12月18日，"发生了一起影响互联网连接的故障，影响了我们部分客户"。与上一周相比，此故障导致流量下降近70%，发生在当地时间09:30至11:00（UTC时间07:30至09:00）。根据一份已发布的报告，该"故障"可能是一次DNS故障。

**Partner Communications (Israel)**
在2025年即将结束时，12月30日，以色列提供商Partner Communications (AS12400)发生重大技术故障，中断了全国范围内的移动、电视和互联网服务。当地时间14:00至15:00（UTC时间12:00至13:00）期间，来自Partner的互联网流量与上一周相比下降了三分之二。中断期间，对Cloudflare的1.1.1.1公共DNS解析器的查询激增，表明问题可能与Partner的DNS基础设施有关。然而，该提供商未公开确认中断原因。

**云平台**
在第四季度，我们在Radar上推出了新的Cloud Observatory页面，该页面在区域层面跟踪超大规模云平台的可用性和性能问题，包括Amazon Web Services、Microsoft Azure、Google Cloud Platform和Oracle Cloud Infrastructure。

**Amazon Web Services**
10月20日，位于弗吉尼亚州北部的Amazon Web Services us-east-1区域经历了"错误率和延迟增加"，影响了该区域内的多项服务。该问题不仅影响了依赖该区域基础设施的面向公众的网站和应用程序的客户，也影响了在us-east-1托管源站资源的Cloudflare客户。
我们在UTC时间06:30左右开始看到问题的影响，错误（5xx类）响应的比例开始攀升，在UTC时间08:00左右达到高达17%。尝试连接到us-east-1源站时遇到的故障数量也有所增加，在UTC时间12:00左右达到峰值。
影响也清晰地体现在关键网络性能指标上，这些指标在整个事件期间保持高位，直到事件结束前（UTC时间23:00左右）才恢复正常水平。在整个事件期间，TCP和TLS握手持续时间逐渐恶化——这些指标衡量了Cloudflare分别与us-east-1客户源服务器建立TCP和TLS连接所需的时间。此外，在事件最初的几个小时内，Cloudflare从源站收到响应头所经过的时间显著增加，之后才逐渐恢复到预期水平。

**Microsoft Azure**
10月29日，Microsoft Azure发生了一起影响其内容分发网络服务Azure Front Door的事件。根据Azure关于该事件的报告，"在两个不同的控制平面构建版本上执行的一系列特定客户配置更改，导致了不兼容的客户配置元数据的生成。这些客户配置更改本身是有效且非恶意的——然而它们产生的元数据，在部署到边缘站点服务器时，暴露了数据平面中的一个潜在错误。这种不兼容性触发了数据平面服务内异步处理过程中的崩溃。"
事件报告标记的开始时间为UTC时间15:41，尽管我们观察到连接到Azure托管源站的失败尝试量在大约45分钟前就开始攀升。TCP和TLS握手指标在事件期间也变得更加不稳定，TCP握手有时耗时延长超过50%，TLS握手在峰值时耗时延长近200%。受影响的指标在UTC时间20:00后开始改善，根据微软的说法，事件于10月30日UTC时间00:05结束。

**Cloudflare**
除了上述讨论的中断外，Cloudflare在第四季度也经历了两次中断。虽然这些并非传统意义上的互联网中断，但它们确实在发生时阻止了用户访问由Cloudflare交付和保护的网站和应用程序。
第一次事件发生在11月18日，是由对我们一个数据库系统权限的更改触发的软件故障引起的，该故障导致数据库向我们的Bot Management系统使用的"功能文件"中输出了多个条目。包括根本原因分析和时间线在内的更多细节，可在相关的博客文章中找到。
第二次事件发生在12月5日，影响了一部分客户，约占Cloudflare服务的所有HTTP流量的28%。它是由我们在尝试检测和缓解一个新披露的全行业React Server Components漏洞时，对我们请求体解析逻辑所做的更改触发的。一份事后分析博客文章包含了更多细节，包括根本原因分析和时间线。
