想到董事会会议开始时仪表盘上满是错误，这足以让分析团队彻夜难眠。同样令人不安的是，事后才发现招聘计划、产品发布或收入预测是基于错误的指标。或者，由于仪表盘错误地呈现了客户的购买历史，导致支持团队发放了过多的退款。

这些失败很少是由糟糕的分析造成的。与任何生产系统一样，它们通常源于随着数据模型和需求的变化，仪表盘需要手动更新——却没有版本控制，没有可靠的审查流程，也没有跨环境推广变更的可重复方法。

这篇博文提出了一个简单的观点：驱动业务的生产级仪表盘必须像生产代码一样，用相同的规范进行管理。因为 Databricks AI/BI 运行在与您的数据管道和治理层相同的 **Data Intelligence Platform** 上，团队可以将那些相同的生产实践——版本控制、环境特定配置和受控部署——也应用于仪表盘。

为了具体说明，我们将展示分析师如何在无需改变日常编写仪表盘方式的情况下，使用生产级的 Databricks 功能。

具体来说，我们将展示此流程如何让您：

*   审查并批准对仪表盘的每次更改
*   跟踪仪表盘的历史，并将代码更改与业务需求关联起来
*   将仪表盘回滚到早期版本

### 前提条件

此工作流需要一些一次性基础设施设置，大多数组织已经具备。如果您还没有这些，请让您内部的 DevOps 或 IT 团队帮助您设置：

*   **至少两个 Databricks 工作区**（例如，一个开发工作区和一个生产工作区），用于编写、测试和部署仪表盘

*   为项目配置的 **Databricks Asset Bundles (DABs)** ([AWS](https://docs.databricks.com/en/dev-tools/bundles/index.html) | [Azure](https://learn.microsoft.com/en-us/azure/databricks/dev-tools/bundles/) | [GCP](https://docs.gcp.databricks.com/en/dev-tools/bundles/index.html))

## 引言：用于安全发布仪表盘变更的结构化工作流

我们将通过一个真实场景来逐步说明：您负责一个**销售绩效**仪表盘，财务和销售领导层每周使用。它最初是实习生直接在工作区中构建的项目，但随着时间的推移不断演变，现在用于多个高管评审会。

董事会会议带来的优先级转变提出了新要求：财务现在需要跟踪**已承诺**和**未承诺的销售额**，以取代单一的汇总销售指标，并且仪表盘必须在下次预测评审会之前反映新的定义。

这些数值直接影响到真实的业务决策，包括薪酬和奖金计算，因此，让我们首次将这个仪表盘置于规范的部署路径上。

### 步骤 1：将仪表盘添加到 Databricks Asset Bundle 中

在开始流程之前，请与您的 IT 团队合作，设置一些基本的代码工具：一个包含空 'Databricks Asset Bundle' 的 Git 仓库，以及一些用于自动部署该 bundle 的 ci/cd 脚本。

Git 仓库是跟踪文件变更的工具——要开始使用，我们需要将其连接到 Databricks，以便跟踪仪表盘配置的更改。从 Databricks 工作区，**创建一个 Git 文件夹**，并将仓库 URL 粘贴到设置对话框中。这使 Databricks 能够识别该仓库，并允许我们在下一步将仪表盘添加到其中。

Databricks Asset Bundle 是一种将代码文件（在本例中是仪表盘）分组在一起的方式。如果仓库已包含一个 bundle，它会自动被检测到，并可以使用箭头图标打开。否则，可以在 Git 文件夹的 **创建** 菜单中创建一个新的 bundle。

在 Asset Bundle 编辑器中，您可以将新的和现有的组件添加到当前为空的 bundle 中。要包含仪表盘，请打开 **添加** 菜单并选择 **添加现有仪表盘**。**添加** 后，您将看到仪表盘作为 bundle 的一部分出现在 `src` 文件夹内。

从此刻起，仪表盘将作为可部署资产进行管理，从而可以轻松地在开发、测试和生产工作区之间推广同一个仪表盘。

最后，将仪表盘**提交**到仓库。这将捕获仪表盘的当前状态作为基线，并为跟踪和审查未来的变更建立一个清晰的起点。

您会看到仪表盘已添加到仓库中，同时还有几个自动生成的配置文件（以 `.yml` 结尾）。这些文件描述了仪表盘在不同环境中应如何部署——您无需编辑它们。

在 **提交信息** 字段中添加一个简短的说明，描述您所做的操作，然后选择 **提交并推送**。这为仪表盘创建了一个检查点——一个您可以稍后返回的已知良好状态——以便未来的变更可以被比较、审查和安全部署。

### 步骤 2：更新仪表盘

现在，现有的仪表盘已提交，您可以开始对其进行更改，而不会影响生产环境中的现有内容，Git 将跟踪您所做的具体更改。

通常的做法是创建一个 Git 分支——这是仪表盘的一个版本，用于在不影响他人的情况下进行工作。您可以通过 **创建分支** 按钮来完成此操作，然后给它一个描述性的名称，例如您的姓名、功能名称或与变更相关的工单号。**可以将其视为您更新的私有版本**：您可以自由地编辑、测试和完善仪表盘，然后单独决定何时您的更改已准备好接受审查和部署。

现在，您可以对仪表盘进行更改了！在本例中，您将修改左上角的销售数字，添加未承诺和已承诺的销售计数器（为清晰起见，选择了粗体的蓝色和红色）。

您会注意到，编写体验没有任何变化——像往常一样使用仪表盘 UI 编辑器进行这些更改。

一旦仪表盘在开发环境中看起来正确，您就可以准备将更改推送到生产环境。使用顶部与之前相同的 **Git** 按钮，用一个简短的提交信息来签入这些更改。

### 步骤 3：审查变更

接下来，您解锁了此工作流的另一个关键优势：一个供他人在变更到达生产环境之前审查更改并提供反馈的地方。需要第二个人审查是普遍的最佳实践，但同样重要的是，它创建了一个低风险的讨论空间，可以在变更影响报告之前讨论想法、验证假设和完善变更。

要开始审查，请在您的 Git 提供商中创建一个**拉取请求 (PR)**，这本质上是仪表盘更新的审查页面。审查者可以确切地看到更改了什么，留下需要您处理的评论，并在一切看起来正确后批准更新。

在审查期间，生产环境中的仪表盘保持不变。只有在反馈得到处理且变更获得批准后，它才会继续推进。

虽然仪表盘的变更在后台作为配置文件存储和跟踪，但通常很难理解实际更改了什么。因此，大多数团队会使用一个小型自动化流程，在每次打开 PR 时自动部署一个临时的测试版仪表盘以供审查。这样，审查者可以在任何内容到达生产环境之前，在上下文中查看提议的指标、计算和布局，并发现数据逻辑或 UI 问题。让开发者或审查者直接在 PR 中包含测试仪表盘的截图或链接，也能使反馈更快、更有把握。

审查者可以添加评论并批准，这些都会被记录下来，以便日后更容易理解变更。

### 步骤 4：使用 bundle 将仪表盘部署到生产环境

变更获得批准后，您就可以将仪表盘部署到生产环境了。

仪表盘在生产环境中通常需要与开发环境不同的设置——例如，指向生产目录或模式而不是开发数据集，或者使用不同的 SQL 仓库。

好消息是，这些差异是预期内的，并且会作为部署流程的一部分得到处理。

当您将仪表板添加到 Asset Bundle 时，Databricks 会生成一个小的 `.yml` 配置文件，用于捕获这些特定于环境的设置。该文件允许您在不更改仪表板逻辑本身的情况下，按环境覆盖值。在我们的案例中，我们通过为目录名称使用 `${variable}` 值，指定了仪表板在生产环境中使用的目录应与测试环境不同。

最后，`databricks.yml` 文件将所有捆绑资源联系在一起，并定义了每个环境中使用的目录，从而便于跨开发、测试和生产工作区管理一致的部署。

一旦拉取请求被批准并合并到主分支，您的部署自动化流程就会运行，并使用 `databricks.yml` 中定义的特定于环境的值。相同的仪表板代码在工作区之间被重用，而诸如目录、模式和仓库等设置则根据目标环境应用。这消除了为每个工作区维护单独仪表板副本的需要，并确保更改在所有地方都能按预期运行。

对于大多数 Git 提供商，您将能够在拉取请求上看到部署自动化过程，从而可以监控部署并确认其何时完成（或是否遇到问题）。如果出现问题，部署会停止，而不会影响现有的生产仪表板，以便您进行故障排除。一旦部署成功完成，更新后的仪表板就在生产环境中上线，可供利益相关者使用！

## 额外收获 1：如果您想查看历史记录怎么办？

仪表板更新上线后，您可能需要了解更改了什么、何时更改以及为何更改的历史记录。此流程的一个额外好处是，更改现在变得**可追溯**。它不再是在工作区中直接进行的一次性编辑，而是显示为一系列已保存的版本。

每个条目代表一次仪表板更新，以及作者和时间戳。您可以打开任何条目来查看更改，并在必要时回滚。

## 额外收获 2：如果需要回滚更改怎么办？

即使经过仔细的审查和测试，问题仍可能出现——例如仪表板无法加载，或者某个指标定义后来被发现不正确。

由于仪表板是通过此工作流程管理的，您可以使用与部署更新相同的受控流程，回滚到已知良好的版本。

首先，在代码仓库中打开仪表板的更改历史记录，并找到您想要撤销的更新。然后，您可以查看修改内容，以确认在继续操作之前回滚的是正确的更改。

从更改详情中，点击链接返回审查页面。要回滚更新，请选择 **Revert**。这将创建一个新的“撤销”更改，该更改**仅**反转那个特定的更新，将仪表板恢复到之前的逻辑，同时保持仪表板历史的其余部分不变。

一旦更改被合并到主分支，将仪表板部署到生产环境的同一自动化流程将执行回滚。这意味着您可以在几分钟内响应中断或高影响的计算问题，而无需绕过您已有的控制措施。

## 额外收获 3：如果您的数据源有更新怎么办？

大多数仪表板与其数据源紧密相连，这意味着仪表板的更新通常与管道的更新密切相关。好消息是，Asset Bundle 的设计旨在将相关组件分组到一个包中。

这确保了上游数据模型的更改永远不会让您措手不及，并且当可视化更改需要数据模型更新时，您可以在一次部署中同时推出这两项更改。

## 结论

将 AI/BI 仪表板视为生产级数据产品，对于可靠的业务决策和风险缓解至关重要。在此工作流程中，只需增加一小套额外的步骤，就能使仪表板更改变得可见、可审查和可逆转——而无需改变您日常构建仪表板的方式。

通过使用 Git 和 Databricks Asset Bundle 管理仪表板，团队为更新建立了一个常规、可预测的工作流程：进行更改、审查、测试和部署。无论更新是小的视觉调整还是对业务逻辑的重大更改，都适用相同的过程。

有了适当的部署规范，仪表板更改就不再是风险的来源，而是一个值得信赖的洞察来源，即使在董事会会议等高风险情况下，也能随着业务发展而演进。
