我们致力于帮助构建更好的互联网，其关键部分是为客户提供所需的工具，使其无论面临何种合规要求，都能安全高效地运营。我们的区域服务产品正是为此而生，它利用 Cloudflare 全球网络的力量，帮助客户履行数据主权法律义务。

今天，我们向前迈出了两大步：首先，我们将区域服务的预定义区域扩展至土耳其、阿拉伯联合酋长国（UAE）、IRAP（澳大利亚合规）和 ISMAP（日本合规）。其次，我们正在推出我们平台的下一阶段演进：自定义区域。

## 全球安全，本地合规：区域服务的优势

在深入了解新功能之前，让我们回顾一下区域服务如何实现本地合规与全球规模安全性的双赢。我们的方法与许多主权云提供商有根本不同。我们并非将您的流量隔离在单一地理区域（这会降低攻击缓解能力），而是利用我们全球网络的全部规模进行防护，并且只在您指定的位置检查您的数据。

以下是其工作原理概述：

1.  **全球接入与 L3/L4 DDoS 防御**：流量在最近的 Cloudflare 数据中心接入，无论该数据中心位于世界何处。在这个初始入口点，我们应用大规模 DDoS 缓解措施，在网络层和传输层阻止海量攻击。这发生在您指定区域之外，确保只有干净的流量被转发。
2.  **智能区域内路由**：在进行任何解密之前，我们会检查请求的元数据。如果请求到达了您指定区域之外的数据中心，我们会通过我们安全、私有的骨干网，将其路由到您边界内的数据中心，并选择性能最佳的路径。
3.  **区域内 TLS 终止与 L7 处理**：只有在确认流量位于您选择的区域内后，我们才会解密请求。也只有在此时，我们才会应用应用层安全服务，如 Web 应用程序防火墙（WAF）或机器人管理，并执行任何 Cloudflare Workers 逻辑。
4.  **安全传输至源站**：处理完成后，请求会被重新加密并安全地发送到您的源服务器。

这种独特的架构意味着，您可以根据需要将数据检查本地化以满足法律义务，同时又不牺牲只有大规模全球网络才能提供的强大 DDoS 保护。

## Cloudflare 托管区域内的新选项

当我们在 2020 年推出区域服务时，我们只从三个区域开始：欧盟、英国和美国。随着时间的推移，我们增加了在所有账户间共享的区域——我们称之为 Cloudflare 托管区域。

现在又有几个新区域可用：土耳其、阿拉伯联合酋长国（UAE）和 IRAP（澳大利亚合规），使我们的区域总数达到 **35 个**。

此外，我们现在还为客户提供了申请满足其账户需求的自定义区域的能力。这就是我们今天推出的**自定义区域**。

## 超越预定义边界：推出自定义区域

虽然我们的 35 个预定义区域满足了许多客户的需求，但数字世界并非一刀切。我们清楚地听到了您的声音：您要求特定的国家、独特的国家组合，以及能够从区域中排除一组国家的能力。

这就是为什么我们很高兴地宣布区域服务的下一阶段演进：**自定义区域**。

简而言之，自定义区域赋予您为流量处理定义自己地理边界的能力。您无需从我们定义的区域列表中选择，而是可以精确地告诉我们哪些位置构成您的区域。

这种灵活性开启了一个新的控制级别。我们的早期访问客户已经使用自定义区域来：

-   **AI 推理区域化**：将 LLM 提示词和响应保持在特定的一组国家内，以优化性能并满足数据本地化法律义务。
-   **启动超定向推广**：为独特的国家组合提供优化的营销活动和内容。
-   **扩展政府业务**：构建符合与政府实体合同承诺的区域。
-   **映射您的公司结构**：构建与您内部业务部门（如 EMEA、MENA 或 APAC）相匹配的区域，以实现完美契合的治理。

核心机制是相同的；唯一改变的是边界。定义区域的不再是 Cloudflare，而是您自己。

可能性是无限的。例如，您的区域可以是：

-   **北美**：加拿大、美国、墨西哥
-   **北美以外地区**：非加拿大、非美国、非墨西哥
-   **使用华氏温度的国家**：美国、巴哈马、开曼群岛、马绍尔群岛、利比里亚

### 区域服务如何工作

区域服务的核心是执行一个简单的规则：TLS 终止和第七层处理只发生在您选择的区域内。自定义区域通过允许您选择自己的区域定义来扩展此功能。

Cloudflare 托管区域和自定义区域依赖于三个构建模块：定义区域成员资格、选择区域内目的地以及在边缘强制执行边界。

#### 定义区域成员资格

一个区域本质上是一组 Cloudflare 数据中心。

-   **Cloudflare 托管区域**使用预定义的成员资格集合。
-   **自定义区域**通过表达式定义成员资格。最常见的字段是 `country_code`：即每个数据中心所在国家的 ISO 代码：

| 用例 | 表达式 | 定义 |
| :--- | :--- | :--- |
| 单一国家 | `country_code == "TR"` | 土耳其 |
| 多个国家 | `country_code in ["DE", "FR", "NL"]` | 德国、法国和荷兰 |
| 排除国家 | `!(country_code in ["US", "CA", "MX"])` | 除美国、加拿大和墨西哥之外的所有地区 |

该表达式会根据数据中心的元数据进行评估。匹配的数据中心将成为您所在区域的成员集，并在全球范围内分发，因此每个数据中心都能快速回答："我是否属于这个区域？"

随着Cloudflare基础设施的发展，成员资格会更新，新的匹配数据中心可以自动加入。您无需担心数据中心的添加或移除定义；Cloudflare会为您处理这一切。

#### 计算最优区域内路由

如果请求在您所在区域之外进入Cloudflare，下一步就是为该入口位置选择最佳的区域内目的地。

Cloudflare的选择过程分为两步：

1.  允许的目的地：区域的成员集（哪些数据中心在区域内）
2.  此入口的最佳目的地：针对请求进入我们网络的数据中心定制的性能排名列表

允许的目的地：区域的成员集（哪些数据中心在区域内）

此入口的最佳目的地：针对请求进入我们网络的数据中心定制的性能排名列表

这些按入口的排名是集中计算并通过Quicksilver分发到边缘的。它们基于我们网络测量的路径质量（不仅仅是物理距离）构建，使用的信号包括：

-   网络性能：延迟和可靠性指标（例如，丢包和超时）
-   容量和负载：可用资源和当前利用率
-   运行状态：健康度和可用性

网络性能：延迟和可靠性指标（例如，丢包和超时）

容量和负载：可用资源和当前利用率

运行状态：健康度和可用性

在路由时，我们将排名列表与区域成员集取交集，并从排名靠前的候选者中选择。最终选择会根据实时可用性进行验证：被禁用或无法访问的目的地会被跳过，因此流量可以故障转移到次优的区域内选项。

#### 执行边界

这是请求到达Cloudflare时的处理过程：

1.  入口。请求到达最近的数据中心。立即应用第3/4层DDoS缓解。
2.  配置查找。是否为该区域配置了区域？
3.  成员资格检查。此数据中心是否在配置的区域中？
4.  路由决策。
    -   在区域内：本地处理。TLS终止和所有第7层服务在此运行。
    -   在区域外：选择一个区域内数据中心，请求通过Cloudflare的私有骨干网转发。
5.  区域内处理。首次进行TLS终止。第7层服务在此运行。
6.  源站连接。处理后的请求发送到您的源站。

#### 我们如何处理错误

弹性设计内置于多个层面：

-   多个候选者：路由会考虑多个区域内选项，并实时选择一个可用的目的地。
-   健康感知路由：不健康或禁用的数据中心会被排除。
-   数据质量门控：只有在获得足够的监控数据时，才会发布新的路由输入。
-   故障关闭设计：如果不存在有效的区域内目的地，连接将失败，而不是在您区域外进行处理。

### 如何开始使用

新的Cloudflare托管区域现已面向使用区域服务的客户开放。如果您想使用这些功能，只需按照标准流程通过Cloudflare仪表板或Cloudflare API启用即可。自定义区域是全新的功能，遵循不同的流程。

为确保完美满足您的需求，自定义区域的初始设置是一个协作过程。要开始使用，只需联系您的客户团队。他们将与您合作定义您的区域并完成部署。虽然该服务目前尚不能自助使用，但我们正在持续开发该技术，并将在功能成熟时重新审视这一点。请注意，可能存在一些技术限制，您的解决方案工程师是讨论细节的最佳人选。
