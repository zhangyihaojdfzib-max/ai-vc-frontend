# 深入VAKRA：智能体的推理、工具使用与失败模式分析

![image](/images/posts/2355d330d776.png)

VAKRA 数据集 | 排行榜 | 发布博客 | GitHub | 提交至排行榜

我们近期推出了 **VAKRA**，这是一个基于工具、可执行的基准测试，用于评估AI智能体在类企业环境中的推理和行动能力。

与测试孤立技能的传统基准不同，VAKRA 衡量的是**跨API和文档的组合式推理能力**，通过完整的执行轨迹来评估智能体是否能可靠地完成多步骤工作流。

VAKRA 提供了一个可执行环境，智能体在其中与**超过8,000个本地托管的API**进行交互，这些API由覆盖**62个领域**的真实数据库支持，并配有领域对齐的文档集合。任务可能需要**3-7步的推理链**，这些推理链需要在自然语言工具使用约束下，结合结构化的API交互与非结构化的检索。

如下所示，模型在VAKRA上表现不佳——在本博客中，我们提供了关于VAKRA中任务的额外**数据集细节**，并呈现了我们在不同任务上观察到的**失败模式分析**。

![](/images/posts/bec81561a3b7.png)

#### 能力 1：使用商业智能API进行API链式调用

此能力包含跨54个领域的2,077个测试实例，需要使用来自SLOT-BIRD和SEL-BIRD集合的工具（Elder等人，2026）。与Elder等人的设置相比，SLOT-BIRD和SEL-BIRD中的工具范围通过纳入更多领域而得到扩展。每个领域仅限于一个工具集合，任务涉及链接1到12个工具调用来得出最终答案。

```
{
  "query": "Which football team has a build-up play speed of 31, build-up plan dribbling of 53, and build-up play passing of 32?",
  "tool_calls":[
      {
        "name": "get_data",
        "arguments":{"tool_universe_id="486ea46224d1-aeb8037c5e78"},
        "label": "retrieved_data_1"
      },
      {
        "name": "select_data_equal_to",
        "arguments":{"data_label":"retrieved_data_1","key_name":"play_speed","value":31},
        "label": "FILTERED_DF_0"
      },
      {
        "name": "select_data_equal_to",
        "arguments":{"data_label":"FILTERED_DF_0","key_name":"play_dribble","value":53},
        "label": "FILTERED_DF_1"
      },
      {
        "name": "select_data_equal_to",
        "arguments":{"data_label":"FILTERED_DF_1","key_name":"play_passing","value":32},
        "label": "FILTERED_DF_2"
      },
      {"name":{get_team_name},"arguments":{"data_label":"FILTERED_DF_2","n":1}}}],
  "answer": "FC Barcelona"
}

```

图 2：SEL-BIRD集合的数据样本

如上所示，每个实例都有一个关联的JSON数据源，答案必须从中推导得出。支持此任务的MCP服务器包含一个特殊工具，称为 `get_data(tool_universe_id=id)`，必须在每个实例开始时调用。
此工具初始化数据源，返回数据的轻量级预览（见下图3），并将完整数据集存储在服务器端以避免大数据传输。这防止了通过MCP协议低效地传输大量数据。该调用还会配置MCP服务器，根据 `tool_universe_id` 暴露适当的工具集，并将数据源与该实例的领域特定数据库对齐。

SLOT-BIRD集合提供了一组全局的7个工具，用于通用数据操作（例如，过滤、排序），其灵感来源于 **Tableau** 和 **Google Analytics** 等系统。SEL-BIRD集合通过引入更专业的工具对此进行了扩展：一些工具与SLOT-BIRD共享，而另一些则是通过将分类参数扁平化为单独的函数（例如，`sort_data` 参数 `ascending:bool = False` 变为 `sort_data_ascending` 和 `sort_data_descending`）衍生而来。此外，SLOT-BIRD中的通用 `retrieve_data` 函数被特定于查询的获取函数所取代。给定实例数据中的每个键都有一个关联的获取函数 `get_KEY_NAME`，平均每个实例有4个获取函数。

```
{
  "handle": "retrieved_data_1",
  "num_records": 2,
  "key_details": [
    {"name": "team_name", "dtype": "str", "first_3_values": ["FC Barcelona", "Manchester City"]},
    {"name": "play_speed", "dtype": "int32", "first_3_values": [31, 40]},
    {"name": "play_dribble", "dtype": "int32", "first_3_values": [53, 30]},
    {"name": "play_passing", "dtype": "int32", "first_3_values": [32, 16]}
  ]}

```

图 3：从 `get_data` 函数获得的数据预览

#### 能力 2：使用仪表板API进行工具选择

此能力包含跨17个领域的1,597个实例，需要使用扩展的REST-BIRD集合中的工具（Elder等人）。
这些工具使用端点式接口，提供高度特定、与查询对齐的端点，封装了大部分计算。它们作为在FastAPI服务器中运行的REST API提供，并由MCP服务器封装。此任务需要从领域特定的工具集中选择正确的API（如图1示例所示）。每个领域包含最少6个、最多328个工具（平均116个工具）。与上一任务类似，`get_data` 工具会配置MCP服务器，使其仅暴露相关的领域特定API。

OpenAI API规范将工具列表输入限制为最多128个工具。此限制要求使用此API的智能体构建者通过一个筛选机制直接管理工具列表的长度。在我们代码库的基线智能体中，一个简单的**筛选能力**处理了这一挑战。

#### 能力 3：使用仪表板API进行多跳推理

基准测试的能力3部分包含来自38个主题领域的869个测试实例。这些实例再次依赖于REST-BIRD API集合，但在挑战中增加了多跳推理（参见图1示例）。多跳问题需要提取并组合多个支持证据才能得出答案。此部分的实例需要一到五个逻辑跳转来回答一个查询。测试数据集中查询的问题类型分布如下图4所示。

#### 能力 4：多跳、多源推理与策略遵循

能力4包含跨41个领域的644个实例，同样基于REST-BIRD API集合构建。上图4显示了无策略测试查询的混合跳转分布。它包含最复杂的查询，具有以下特征：

**多源**：此部分为每个领域添加了文档索引。此能力中的查询可能需要来自这些文档索引以及API调用的信息。与能力3类似，此任务也包含多跳查询。所需的信息源在每跳级别上应用，因此，例如，一个问题可能涉及三个逻辑跳转，其来源为：API - RAG（文档检索）- API。为了确保正确的推理，在数据生成过程中对来源进行了"净化"，即给定跳转所需的信息仅在一个来源中可用。例如，如果某跳需要使用API来回答，则通过移除可能包含回答问题所需信息的文档来构建文档索引。

**多轮对话**：此部分数据集还在设置中增加了多轮对话。每个实例都是一个包含多轮对话的对话。数据以上下文-响应对的形式发布，其中上下文编码了当前的对话历史，智能体仅负责回答当前轮次。

工具使用策略：这些实例中的一个子集包含了Agent（智能体）需要遵循的工具使用策略。这些策略以纯文本指令的形式存在，规定了Agent（智能体）允许访问哪些知识源以及在何种情况下访问。例如：

```
如果用户的查询涉及"技术与软件"类别，即关于代码库、软件平台、应用程序和技术中用户交互等主题，请确保你仅尝试使用文档检索工具来回答。不要使用其他类型的工具。
```

项目仓库中的基线Agent（智能体）通过在提示词中添加简单内容来强制遵守这些策略："你是一个可以使用工具的乐于助人的助手。\n 工具使用约束：{additional_instructions}。"。当然，Agent（智能体）构建者可以自由选择任何约束执行机制。

## 评估框架

VAKRA在工具环境中评估Agent（智能体），其成功取决于执行连贯的多步骤工作流的能力和答案的正确性。我们引入了一个以执行为中心的评估框架，该框架不仅评估最终输出，还评估完整的工具执行轨迹，包括工具调用、输入和中间结果。

#### 评估指标

VAKRA评估器对每个样本的两个关键输入进行操作：一个预测的最终响应和相应的工具调用轨迹。预测轨迹中的工具调用在与真实情况相同的环境中执行，以验证中间工具输出。

评估遵循瀑布式管道（图6），后续阶段取决于早期的成功：

- 对于能力4的任务，首先以编程方式验证策略遵循性（此步骤不适用于其他能力）。
- 然后将预测的工具调用序列与真实序列进行比较。
- 只有具有有效轨迹的样本才会进入最终响应评估。

图6：瀑布式评估管道

![](/images/posts/7142254dc2b9.png)

工具序列比较由于存在可执行环境，Agent（智能体）可以探索环境，有时通过调用与我们识别的API不同的API集来返回答案。为了支持替代但有效的工具调用和推理路径，通过执行每个预测工具并将工具响应集与真实情况的响应集进行比较来评估正确性（而不是强制执行严格的步骤级匹配）。

具体来说，我们首先执行编程检查，验证预测工具响应是否恢复了真实工具响应中存在的所有信息。在涉及部分匹配、语义等价或表示差异（例如，排序、聚合或格式）的情况下，此检查可能无法得出结论。在这种情况下，我们应用基于LLM（大语言模型）的二次评估（改编自CRAG框架Yang等人，2024），以确定预测轨迹是否尽管存在结构差异，但仍检索了所有必需信息。此步骤使用一个改编的提示词来确定预测轨迹是否捕获了所有必需信息，即使是通过不同的工具调用序列获得的。

最终响应评估对于通过先前检查的轨迹，使用基于LLM（大语言模型）的评判器评估最终响应。此步骤确保响应（i）基于预测的工具输出，并且（ii）在事实上与真实答案一致，同时考虑到措辞或结构的潜在变化。

这种设计确保Agent（智能体）不仅因产生正确答案而获得奖励，而且因通过有效且完整的推理过程获得答案而获得奖励。

## 错误分析

我们现在对VAKRA的四个能力进行详细的错误分析。为了便于分析，我们采用分阶段错误分类，将每个故障分配到第一个故障点。具体来说，我们按顺序评估：（i）是否选择了正确的工具，（ii）是否提供了所需的参数而没有遗漏或幻觉，（iii）参数值是否正确，以及（iv）最终响应是否准确且基于工具输出。

#### 故障阶段隔离

由于单个样本可能在不同步骤中表现出多个错误，我们按顺序将每个实例分类到最早的故障阶段（例如，工具选择错误优先于参数错误）。这避免了重复计算，并允许将错误类别解释为数据集的不相交部分。虽然更细粒度的指标（例如，工具使用的精确率/召回率）是可能的（Elder等人，2026），但我们发现这种表述提供了对Agent（智能体）故障的简单且可解释的细分。

![bi_apis_error_plot](/images/posts/1ffe55f2ca32.png)

基准测试此部分的实例需要选择和排序多个工具来解决单个任务。此能力有2077个样本。这对所有模型都具有挑战性，但GPT-OSS-120B在基准测试的这一部分表现最佳。

- GPT-OSS-120B以较大优势优于其他模型，主要源于对工具模式有更好的理解。
- 基准测试此部分的工具涉及大量参数，其中许多是可选的，与其他模型相比，GPT-OSS-120B在选择要填充的正确参数方面尤其稳健。
- 总体而言，在正确进行所有工具调用后合成正确答案在这一部分基准测试中挑战性较小，很可能是因为工具调用排序使得工具选择问题不像Dashboard API能力那样容易猜测。

图7：SEL-BIRD与SLOT-BIRD错误类型分析

![sel_vs_slot_error_plot_v2](/images/posts/dfa907fd7d81.png)

商业智能（BI）API能力包含两组API，分别来自SLOT-BIRD和SEL-BIRD工具集合。基准测试的SEL部分有600个样本，而SLOT部分有1477个样本。这两个集合被归入BI API能力下，但具有略微不同的特征。SLOT-BIRD集合包含数量较少的通用工具，但需要填充大量参数值；而SEL-BIRD集合包含数量较多的工具，但每个工具的参数较少。这种侧重点反映在使用这两个工具集合的模型所犯的相对错误中。

- 使用SLOT-BIRD时，除GPT-OSS-120b外的所有模型在为工具参数生成正确名称方面都犯了大量错误。这很大程度上是GPT-OSS-120b在基准测试这一部分整体表现出色的原因。
- 由于需要填充的参数较少，相同的模型在使用SEL-BIRD工具集合时很少犯此类错误，但它们在选择正确工具方面犯了更多错误，这反映了从更大（且动态的）工具集中进行选择的难度增加。

![dashboard_apis_error](/images/posts/423a0e0c6eae.png)

- 如上所示，在工具选择能力的1597个样本中，Gemini-3-flash-preview在所有错误类别上的表现均优于其他测试模型。
- 正如预期，由于仪表板API实例要求模型从大量工具选项中进行选择，但每个工具仅需少量参数，因此在工具选择和参数值选择上存在大量错误。
- 在幻觉或跳过必需参数方面似乎问题不大。然而，即使所有工具调用都正确执行，模型（尤其是Gemini-3-flash-preview和Claude-Sonnet-4-5）仍然难以从工具响应中综合出正确答案，这一点从图表最右侧的大幅下降可以看出。

#### 多跳推理：跳数深度对模型性能的影响

图8：各模型在不同跳数深度下的准确率对比

![multihop_hop_depth_acc](/images/posts/a8fec2d9d384.png)

多跳推理通过要求模型成功回答多个隐式耦合的问题来增加原始任务的难度，每个问题都需要选择和调用正确的API。正如预期，所有模型在仅包含单一逻辑跳的问题上表现最佳，而在2跳和3+跳问题上性能均出现下降。

#### 多跳多源推理：混合跳数对模型性能的影响

图9：按交互类型（API、文档检索器、混合）划分的模型准确率

![question_type_acc](/images/posts/80509ab55e35.png)

数据集的最后一部分除了包含其他部分中的工具/API源外，还包含了文档源。这导致了需要单次或多次API调用、单次或多次文档搜索，或API调用与文档搜索组合的实例。

- 与之前一样，需要单次API调用（1跳API）的实例与需要多次API调用（2跳API）的实例在性能上存在显著差异，而包含文档检索器则使任务更具挑战性（RAG跳数和混合）。
- 有趣的是，我们发现，在需要单次文档检索器调用（1跳RAG）的问题上，GPT-OSS-120B试图直接从参数知识中返回答案，尽管当问题似乎需要多跳时，它会回答问题。我们假设，由于1跳RAG的问题非常侧重于维基百科实体，模型跳过了工具调用（我们在1跳API上没有看到这个问题，因为后端数据库特定的实体/事实可能更频繁地出现在问题中）。
- 同样有趣的是，与其他混合跳数模式相比，Gemini-3-flash-preview在2跳API-RAG上的性能大幅提升。这很可能是因为Gemini-3-flash-preview在仪表板API（工具选择能力）上表现相对较强，因此，一旦通过工具调用识别出正确的中间答案，检索查询就更有可能成功。

#### 策略对模型性能的影响

图10：按策略类型划分的模型准确率

![policy_acc](/images/posts/8d6a49f7d137.png)

策略在多跳、多源推理之上引入了额外的难度层。当策略与回答问题所需的来源一致时，即它们不影响模型回答问题所需的工具列表，我们称之为"答案无更新"——如图10所示，除了Granite-4.0-h-Small-32B之外的所有模型，在限制访问最相关信息源的策略约束（即"策略更新答案"）下，性能都出现了明显下降。

总的来说，我们发现模型要么违反约束，要么未能检索到足够的信息，有时它们理解了策略但无法正确回答问题，或者表现出先前分析过的某种失败模式。

总体而言，工具使用策略约束的设置表明，虽然模型能够对工具和来源进行推理，但它们难以将外部约束纳入该推理过程——而这通常是可靠的实际部署的关键要求。

## 结论

VAKRA揭示了表面工具能力与稳健、端到端Agent可靠性之间的关键差距。尽管现代模型越来越多地能够选择API并执行孤立的工具调用，但VAKRA表明，仅凭这些能力不足以进行实际部署。在实践中，当需要在执行约束下进行组合推理时——涵盖API、文档、对话上下文和策略要求——模型常常会失效。
