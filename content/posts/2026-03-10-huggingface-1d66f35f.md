# 保持Token流动：16个开源RL库的经验总结

TL;DR——对于那些没时间阅读5000字关于异步RL系统设计的读者（我们理解，你们还要训练模型）：

- 问题所在：在同步RL（强化学习）训练中，数据生成（模型推理以创建数据样本）占据了绝大部分实际耗时——在320亿参数模型上生成单批32K Token的轨迹可能需要数小时，而用于训练的GPU在此期间处于闲置状态。
- 普遍采用的解决方案：将推理和训练解耦（分离）到不同的GPU资源池，通过轨迹缓冲区（模型输出的临时存储）连接它们，并异步（无需等待）传输权重，使双方互不等待。
- 我们调研了16个实现此模式的开源库，并从7个维度进行比较：编排原语、缓冲区设计、权重同步协议、陈旧性管理、部分轨迹处理、LoRA支持以及分布式训练后端。
- 关键发现：Ray在编排方面占主导地位（调研的16个分布式计算库中有8个使用）。NCCL（NVIDIA集体通信库）广播是传输模型权重的默认方法。陈旧性管理指如何处理过时的数据样本，范围从简单丢弃旧样本到使用高级重要性采样校正。LoRA（低秩适应）训练支持较少。分布式MoE（专家混合）支持是新兴的差异化特性。

## 1. 动机：从同步RL训练到异步架构

异步RL训练已成为大规模后训练的主导范式。现代后训练中的几个趋势使得同步训练循环几乎无法扩展：

- 推理模型生成长轨迹。思维链训练会产生非常长的轨迹，单个同步生成批次在单个GPU上可能需要数小时才能完成。在此期间，训练GPU完全闲置。
- 像GRPO这样的无价值函数训练器使用组相对优势。这意味着每个提示需要生成多达G倍的轨迹，并且整个批次受组内最慢完成项的制约。
- 智能体RL训练的兴起。当模型在多轮轨迹中与工具、沙盒和外部环境交互时，轨迹长度和延迟变得高度可变。一个简单的API调用可能几秒内返回，而一个使用工具的复杂推理链可能运行数分钟或数小时。用于训练MiniMax-M2.5的MiniMax的Forge框架，说明了实践中达到的规模：上下文长度高达200K Token，超过十万个不同的智能体框架和环境，每日吞吐量达数百万样本级别。在这种规模下，生成和训练之间的任何同步屏障都会成为严重的瓶颈。仅拖尾问题（少数慢速轨迹阻塞整个批次）就可能导致数百个GPU闲置。

开源生态系统已汇聚于一个共同的架构响应：将推理与训练解耦到独立的GPU资源池，通过轨迹缓冲区连接它们，并让双方并发运行。

我们正在为TRL（最广泛使用的模型后训练库之一）开发一个新的异步训练器。为了指导我们的设计，我们调研了十六个从头开始围绕异步训练构建的开源库，并从七个维度对它们进行了比较：编排原语、缓冲区设计、权重同步协议、陈旧性管理、部分轨迹处理、LoRA支持以及分布式训练后端。本文提炼了我们从该调研中提取的设计原则。

除了RL之外，对异步基础设施的需求日益明显。例如，策略蒸馏（学生生成序列，教师为其评分）与GRPO类似，只是将奖励函数替换为教师前向传播。认识到这种结构相似性，本调研中的所有内容同样适用于异步蒸馏。我们将在第5节回到这个更广泛的点。

### 1.1 TRL当前如何进行RL训练

TRL当前的GRPOTrainer在单个同步的`training_step()`调用中实现了完整的GRPO循环（提示采样、生成、奖励评分、优势计算、梯度更新和权重同步）。这种设计简单且正确，但它无法将生成与训练重叠进行，导致显著的GPU利用率损失。

观察GRPOTrainer，每个训练步骤内依次包含以下阶段：

1. 提示采样：从数据集中抽取一批提示。这里没什么特别的，我们继续。
2. 生成：调用`model.generate()`（或向前端vLLM服务器发送请求）为每个提示生成G个补全。这是自回归的，并占据实际耗时的主导地位。
3. 奖励评分：根据一个或多个奖励函数评估每个补全。
4. 优势计算
5. 前向和后向传播：计算裁剪后的策略梯度损失并进行反向传播。
6. 优化器步骤，更新模型权重。
7. 权重同步：将更新后的权重推送到推理引擎（vLLM），以便下一次生成使用新策略。

每个阶段在完成后才会阻塞进入下一阶段。时间线如下所示：

![同步TRL训练时间线](/images/posts/d966e2aa5e7a.png)

TRL 提供了 `steps_per_generation` 配置选项，以便在多个梯度步中复用同一组 rollout（时间复用），从而分摊生成成本。但生成调用本身仍然是完全同步和阻塞的；训练器必须等到批次中的所有补全都完成后，才能开始梯度计算。

该库也支持以 `server` 模式将 vLLM 作为独立进程运行。这能在生成过程中释放训练 GPU，但仍存在两个硬同步屏障：一是需要等待所有补全返回的 HTTP 调用，二是在权重同步传输期间，训练器和 vLLM 都会被阻塞。

### 1.2 共置式与分离式训练

在讨论异步训练之前，有必要先了解使用独立推理引擎进行 RL 训练的两种部署拓扑：

*   **共置模式**将推理和训练放在**同一组 GPU** 上。单个 GPU（或 TP 组）同时持有训练模型（在 FSDP 或 ZeRO 下）和推理引擎（vLLM 或 SGLang）。同一时间只有一个角色是活跃的：在生成期间，训练模型的参数可能会被卸载或重新分片为适合推理的布局（例如，从 FSDP 分片转换为 vLLM 的张量并行布局）；在训练期间，推理引擎则被暂停或休眠。权重“同步”基本上是免费的；最多只是在同一 GPU 上进行原地重新分片，而非网络传输。共置模式的优点是简单和成本低；你需要的 GPU 总数更少。其根本限制在于**推理和训练无法重叠**。例如，以下是 Trl 与 vllm 在 `colocate_mode` 下的情况：

![TRL with vLLM in colocate mode](/images/posts/fe40cd027e75.png)

*   **分离模式**将推理和训练放在**独立的 GPU 池**上。推理池持续运行 vLLM 或 SGLang；训练池持续运行优化器。两个池通过权重同步协议（NCCL 广播、文件系统检查点、HTTP 等）和数据传输机制（Ray 对象存储、Redis 流、共享内存等）进行通信。

分离模式的最大优势是**推理和训练可以并发运行**。当训练器在计算批次 N 的梯度时，推理池已经在为批次 N+K 生成 rollout，从而实现异步训练。然而，这种优势是有代价的：需要额外的 GPU。

并发性、异步性和并行性是三个不同的概念，经常被混淆。在本文中，当我们说“异步训练”时，我们特指：**生成和训练并行运行，并有效重叠**；推理池正在生成下一批 rollout，而训练池正在计算当前批次的梯度。这本质上是分离模式的能力。共置模式可以通过睡眠/唤醒内存管理或快速原地重新分片等优化来加速推理，但它无法实现真正的同步重叠；推理和训练仍然在同一组 GPU 上轮流进行。本调查中所有实现了有意义异步重叠的库，都以分离模式为基础。

### 1.3 生成瓶颈

在推理模型的 RL 训练中，**自回归生成占据了绝大部分的挂钟时间**。对于一个数学或编码任务，单个 rollout 可能产生 8K–64K 个 Token 的思维链推理（参见 QED-Nano rollout 长度）。

为了具体说明这一点，考虑**在单个 H100 80GB GPU 上的 vLLM 基准测试**（bf16，无量化，离线吞吐量模式）。一个 **7B 模型**（DeepSeek-R1-Distill-Qwen-7B）实现了约 6,300 个输出 Token/秒的聚合吞吐量；一个 **32B 模型**（DeepSeek-R1-Distill-Qwen-32B）则降至约 1,200 个输出 Token/秒。这些是**所有并发请求的总吞吐量**，即推理引擎每秒能处理的总 Token 数，无论有多少个序列共享该 GPU。

现在考虑一个典型的 GRPO 训练步骤：`G=8` 个补全/提示词 × 64 个提示词/批次 = 512 个 rollout。生成需要多长时间？

即使在较短的生成长度下（7B 模型生成 2K Token），仅生成阶段每个训练步骤就要消耗几分钟。在较长的生成长度下（前沿推理模型越来越多地在此操作），单个生成阶段在一个 GPU 上可能需要**数小时**。扩展到 8 个推理 GPU 可以将这些时间大致除以 8 倍（假设吞吐量线性扩展），但即便如此，在 32B 模型上生成 32K Token 的 rollout，每个步骤仍然需要约 28 分钟。

**掉队者问题**进一步加剧了这种情况。在像 GRPO 这样的基于组的算法中，每个提示词采样 G 个补全。批次必须等到**最慢的**补全完成后才能继续。思维链输出的长度变化很大；单个提示词可能产生从 1K 到 32K Token 不等的补全。批次受限于最长的补全，而连续批处理只能部分缓解这个问题：较短的序列会为新工作释放槽位，但 GRPO 组中的**最后一个**序列仍然会阻塞该组的奖励计算和训练步骤。

### 1.4 核心洞见

本调查中的每个库都独立地收敛于同一个架构原则：**在物理上将推理 GPU 与训练 GPU 分离，并异步推送权重，从而使生成永不停止，训练永不等待。**

推理池持续运行，将完成的 rollout 送入缓冲区。训练池从缓冲区拉取数据，计算梯度更新，并定期将新权重推送回推理池以保持同步。两个循环以各自的节奏运行，通过缓冲区解耦。

这种设置具有高度可扩展性，但也引入了一系列新问题：陈旧性（在旧策略下生成的 rollout）、权重同步开销、部分 rollout 处理等。本文的其余部分将详细剖析当前的开源库如何解决这些问题。

## 3. 比较框架：七个维度

为了理解快速扩张的异步 RL 库生态系统，我们提出了七个正交的比较维度。每个维度都捕捉了一个影响库性能、复杂性和权衡取舍的基本设计决策。

*   **维度 1 – 编排与并发原语**：分布式组件如何协调（Ray actors、asyncio、发布/订阅、HTTP）。
*   **维度 2 – Rollout 缓冲区设计**：rollout 如何从推理流向训练。
*   **维度 3 – 权重同步协议**：更新后的权重如何到达推理服务器，以及系统是否必须暂停以接收它们或继续生成。
*   **维度 4 – 陈旧性管理**：如何处理离策略的 rollout：版本拒绝、深度限制，或重要性采样校正。
*   **维度 5 – 部分 Rollout 处理**：当权重更新在序列生成中途到达时，如何处理正在进行的生成。
*   **维度 6 – LoRA 训练支持**：通用的 LoRA 支持，以及是否仅能训练和同步适配器参数，从而实现亚毫秒级的权重传输。
*   **维度 7 – 分布式训练后端与并行性**：训练使用何种并行策略，这限制了最大模型大小。

### 维度 1：编排与并发原语

系统如何协调其分布式组件？

编排框架的选择决定了编程模型、故障语义和可扩展性上限。与其列出每个库的实现细节，不如将整个领域清晰地分解为四种**编排类型**，这些基本的协调范式在抽象级别、故障模型和部署要求上有所不同：

**关于 Tunix 的说明**：Tunix（Google）使用原生的 JAX 网格模型，通过 `ThreadPoolExecutor` 实现异步重叠，并通过 `jax.device_put` 进行跨网格权重传输。它在架构上与 PyTorch 生态系统有足够大的区别，因此在编排方面进行直接比较意义不大；它存在于 XLA/TPU 世界中，拥有自己的协调原语。

上表揭示了一个显著规律：在接受调研的十六个库中，有八个使用 Ray 作为其编排核心。这并非巧合；它反映了 Actor 模型与 RL 训练结构之间深刻的架构契合。Ray 背后的公司 Anyscale 对面向 LLM 的开源 RL 库进行的一项调查证实了这种趋同性。大规模 RL 训练本质上涉及异构组件（推理引擎、训练引擎、环境、奖励模型），这些组件必须在集群中进行编排，通常运行在不同的硬件类型上，具有不同的扩展需求和故障模式。Ray 的 Actor 模型直接映射了这一点：

1.  **Actor 隔离与异构资源。** 每个 RL 组件（vLLM 推理服务器、FSDP 训练器、奖励模型、环境池）都成为一个具有自身资源需求（`num_gpus`、`num_cpus`、`memory`）的 Ray Actor。Placement groups 提供了对 GPU 亲和性的细粒度控制，无需手动进行 SSH/`torchrun` 编排。
2.  **调度与自动扩缩容。** Ray 的调度器处理在集群中放置异构 Actor 的组合问题。当生成所需的 GPU 时数比训练多 8 倍时，你可以直接告诉 Ray 独立扩缩你的推理 Actor。
3.  **容错性。** 长时间的 RL 训练运行（数天到数周）容易受到 GPU 故障、OOM 终止和网络分区的影响。Ray 的 Actor 重启策略和对象存储复制提供了弹性，若使用原始的 `asyncio` 和 `multiprocessing` 则需要大量定制基础设施才能实现。容错性的具体例子：例如，`open-instruct` 依赖 Ray 的 Actor 监督机制，以从 vLLM 引擎在 rollout 过程中崩溃的状态中恢复。
4.  **用于零拷贝数据传输的对象存储。** Rollout 数据可能很大，对于超长上下文推理，每批次可达数十 GB。Ray 的共享内存对象存储支持同一节点上 Actor 之间的零拷贝传输，避免了通常伴随 `multiprocessing.Queue` 方法而来的序列化开销。
5.  **生态系统成熟度。** Ray 自 2017 年以来已在大规模场景下经过实战检验，拥有在数千个 GPU 上的生产部署。调试开销确实存在（Ray Dashboard、分布式堆栈跟踪、placement group 故障），但另一种选择——从零开始构建等效的协调机制——在多节点规模下会更糟。话虽如此，Ray 是一个重量级依赖：它引入了自己的调度器、对象存储和仪表板，增加了并非每个团队都需要的运维复杂性。这正是为什么像 PRIME-RL、PipelineRL 和 AReaL 这样的库选择了轻量级的原生 Python 协调机制（`asyncio`、线程、Redis 流）——当你控制整个技术栈且部署拓扑固定时，原生 Python 的简单性和可调试性往往胜过 Ray 提供的便利。

**Actor 隔离与异构资源。** 每个 RL 组件（vLLM 推理服务器、FSDP 训练器、奖励模型、环境池）都成为一个具有自身资源需求（`num_gpus`、`num_cpus`、`memory`）的 Ray Actor。Placement groups 提供了对 GPU 亲和性的细粒度控制，无需手动进行 SSH/`torchrun` 编排。

**调度与自动扩缩容。** Ray 的调度器处理在集群中放置异构 Actor 的组合问题。当生成所需的 GPU 时数比训练多 8 倍时，你可以直接告诉 Ray 独立扩缩你的推理 Actor。

**容错性。** 长时间的 RL 训练运行（数天到数周）容易受到 GPU 故障、OOM 终止和网络分区的影响。Ray 的 Actor 重启策略和对象存储复制提供了弹性，若使用原始的 `asyncio` 和 `multiprocessing` 则需要大量定制基础设施才能实现。容错性的具体例子：例如，`open-instruct` 依赖 Ray 的 Actor 监督机制，以从 vLLM 引擎在 rollout 过程中崩溃的状态中恢复。

**用于零拷贝数据传输的对象存储。** Rollout 数据可能很大，对于超长上下文推理，每批次可达数十 GB。Ray 的共享内存对象存储支持同一节点上 Actor 之间的零拷贝传输，避免了通常伴随 `multiprocessing.Queue` 方法而来的序列化开销。

**生态系统成熟度。** Ray 自 2017 年以来已在大规模场景下经过实战检验，拥有在数千个 GPU 上的生产部署。调试开销确实存在（Ray Dashboard、分布式堆栈跟踪、placement group 故障），但另一种选择——从零开始构建等效的协调机制——在多节点规模下会更糟。话虽如此，Ray 是一个重量级依赖：它引入了自己的调度器、对象存储和仪表板，增加了并非每个团队都需要的运维复杂性。这正是为什么像 PRIME-RL、PipelineRL 和 AReaL 这样的库选择了轻量级的原生 Python 协调机制（`asyncio`、线程、Redis 流）——当你控制整个技术栈且部署拓扑固定时，原生 Python 的简单性和可调试性往往胜过 Ray 提供的便利。

代价是对一个非轻量级运行时的硬依赖。这种权衡可能是值得的，特别是对于生产规模的训练（64+ GPU、多天运行、复杂的奖励计算）。

虽然 Ray 的 Actor 模型是当前领域的主要参与者，但 **Monarch** 作为 Meta 推出的一个全新的 PyTorch 原生分布式 Actor 框架崭露头角，专为 GPU 工作负载打造。与 Ray 类似，Monarch 基于 Actor 模型；组件是独立的 Actor，通过邮箱和消息进行通信，但它是为 PyTorch/CUDA 生态系统从头设计的，而非一个通用的分布式运行时。

Monarch 提供了几个与异步 RL 特别相关的能力。一个使用 Monarch 实现异步 RL 的示例（来自 GPU Mode 讲座系列）展示了其架构：生成器、回放缓冲区和训练器被建模为 Monarch Actor，回放缓冲区吸收掉队 rollout 的延迟差异，而 RDMA 权重同步将更新后的参数推送给生成器，且不阻塞训练。该模式在结构上与基于 Ray 的设计（verl、SkyRL、open-instruct）相同，但使用纯 PyTorch 原生原语实现。

### 维度 2：Rollout 缓冲区设计

生成的 rollout 如何从推理流向训练，以及流水线有多深？

缓冲区是位于生成和训练之间的数据结构。其深度控制着异步的最大程度，从而也控制着最大陈旧度。

**双缓冲区模式** 是从同步训练升级到异步训练的最简单方式：它恰好将一个生成步骤与一个训练步骤重叠，并最多引入一个策略步的滞后！

另一方面，更深的队列提高了吞吐量，但需要管理陈旧度。

缓冲区控制着有多少数据在传输中。但数据只是等式的一半。另一半是在这些 rollout 变得陈旧之前，将更新后的权重**回传**给推理服务器。这就是权重同步发挥作用的地方！

### 维度 3：权重同步协议

梯度更新后，新的模型权重如何到达推理服务器？

**范围说明：** 本维度聚焦于**解耦模式**，即推理和训练运行在独立的 GPU 池上，因为这是异步重叠（以及因此权重同步设计）真正重要的部署拓扑。同址部署（同一 GPU 承担两种角色）本质上是同步的，不面临下面讨论的传输/中断权衡。

这是架构上影响最大的维度。协议决定了同步延迟、中断粒度以及是否可能进行部分 rollout。

这里需要做一个关键区分：**传输机制**和**中断模型**。大多数库在启动权重传输之前，会在一个粗粒度边界（一个 HTTP 请求、一个完整批次，甚至一个完整的训练步骤）处暂停生成。PipelineRL 是个例外：它完全不会停止生成。

**传输机制：**

在中断模型中，生成何时暂停以接受新权重？

这正是PipelineRL与其他所有库的根本区别所在。与其逐一列举每个库，不如将整个领域归纳为五个概念层级，按中断粒度从最精细到最粗略排序：

"永不停机"层级在性质上与其他所有层级都不同：例如，PipelineRL直接挂钩到推理引擎中，使得锁的获取和释放是**按每个Transformer前向传播**（一个序列的一个Token步骤）进行的。一次权重更新最多只需等待一个前向传播（约几毫秒），即可交换所有参数，并立即恢复生成。其他所有库都在更粗略的边界处停止生成，从一个HTTP请求（约数百毫秒）到完整的批次边界（约数秒）不等。

权重同步控制着**何时**获得新权重。但异步训练意味着生成过程总是在**某个**策略版本下进行，而这个**生成中**的策略可能落后于训练器好几个梯度步。各库如何处理这种策略滞后，就是**陈旧性管理**。

### 轴4：陈旧性管理

系统如何处理生成的轨迹数据可能来自比当前训练策略更旧的策略这一事实？

一旦生成和训练重叠，样本就变成了离策略的。目前出现了三种**正交**的策略来管理这种陈旧性，大多数生产系统会结合使用不止一种：

**策略1：按样本版本拒绝。** 每个样本都带有生成它的整数策略版本标签。在训练时，版本落后于当前策略超过某个阈值的样本，在进入损失计算前会被直接丢弃。简单且正确，但浪费了生成这些被丢弃样本所耗费的宝贵计算资源。

**策略2：深度限制。** 生成和训练之间的队列或缓冲区具有有限的容量（或明确的陈旧性门限），这在架构上限制了任何样本可能落后的最大程度。这范围从深度=1（超前一步的双缓冲，从构造上就不可能产生陈旧性）到与版本差距绑定的显式容量公式。不需要按样本进行版本追踪；限制由系统的流水线深度强制执行。

**策略3：IS加权损失校正。** 到达训练器的陈旧样本通过重要性采样比率 $\frac{\pi_{\theta}(a \mid s)}{\pi_{\text{old}}(a \mid s)}$ 重新加权，通常会被裁剪（截断IS）。一些库也应用OPSM（对具有负优势的离策略样本将损失置零）。这保持了吞吐量；没有样本被丢弃，但IS比率会带来梯度方差增加的代价。

这些策略是正交的：一个系统可以单独使用版本拒绝、单独使用深度限制、单独使用IS校正，或者它们的任意组合。同步系统通过从不重叠生成和训练，完全避免了这个问题。

✅ = 是，❌ = 否，⚠️ = 可选/可配置，— = 不适用（同步）

- **版本拒绝**简单且正确，但当许多样本被丢弃时会浪费计算。
- **IS校正**以梯度方差为代价保持了吞吐量。
- **深度限制**是最粗略的机制，但它完全避免了按样本记账。

生产系统（PRIME-RL、AReaL、open-instruct）的趋势是采用**混合方法**，将深度限制与可选的IS校正相结合，既获得了有界队列的架构简单性，又获得了重要性加权在损失层面的安全网，以实现稳定训练。

陈旧性管理处理的是在旧策略下生成的数据。但是，当权重更新到达时，**仍在生成中**的数据怎么办？

### 轴5：部分轨迹处理

当权重更新到达时，正在进行的生成过程会发生什么？

这对于长上下文任务至关重要，因为单个轨迹生成可能需要数分钟。有四种策略：

到目前为止，每个轴都假设是全参数训练。但在LoRA训练中，你只训练几百万个适配器参数，而不是数十亿个，权重同步问题几乎消失了。让我们看看这些库如何支持LoRA训练。

### 轴6：LoRA训练支持

该库是否支持通过LoRA适配器进行参数高效训练，支持哪些模式，以及是否利用了仅适配器权重同步？

对于GPU预算有限的团队来说，LoRA可以说是最具实际意义的轴。它将可训练参数数量减少了99%以上，将峰值激活内存减半，并且当推理服务器支持LoRA时，可以实现**仅适配器权重同步**：无需广播一个7B+模型的每个参数（约100-500毫秒的NCCL通信），只需将适配器增量推送到vLLM，在秩为32时大约为50 MB，传输时间不到一毫秒。

* NeMo-RL：仅在DTensor后端支持GRPO和DPO的LoRA；Megatron Core后端仅支持SFT（RL LoRA列为"即将推出"）。使用自定义的与DTensor兼容的LoRA模块（非`peft`），可选择使用Triton内核。
† ROLL：仅在使用DeepSpeed训练后端时官方支持LoRA。Megatron后端的LoRA出现在2026年2月的更新日志中，但仍处于实验阶段。
‡ open-instruct：模型配置暴露了LoRA相关字段（`use_peft`、`lora_r`、`lora_alpha`），适配器保存由检查点逻辑处理。然而，在RL训练路径中从未初始化`peft`模型；截至2026年3月，LoRA对于RL训练器来说仍然是仅SFT/DPO的功能。

三种LoRA实现家族：

1. **HuggingFace `peft`**（PipelineRL、SkyRL/FSDP、verifiers-rl、ROLL、OAT、Atropos）：最常见的选择。标准检查点格式（`adapter_model.safetensors`），兼容任何HF Transformers训练循环。与ZeRO-3的交互需要注意：例如，OAT需要禁用融合的LM头；ROLL必须完全禁用梯度检查点。
2. **Megatron-Bridge**（verl/Megatron、SkyRL/Megatron、MILES）：3D并行训练（TP × PP × DP）所必需。支持多种LoRA类型：`lora`、`canonical_lora`（将合并的QKV拆分为独立的Q/K/V适配器）、`vlm_lora`和`dora`。`canonical_lora`变体避免了QKV合并，从而提高了训练稳定性。MILES以HF `peft`格式和Megatron原生的每秩格式保存检查点。
3. **自定义实现**（NeMo-RL、PRIME-RL、Tunix/qwix）：库特定的LoRA模块，与`peft`检查点不互操作。PRIME-RL独特地支持在单次运行中同时使用多个适配器，以实现多实验并行。Tunix使用Google的`qwix` JAX库，该库增加了内置的QLoRA（NF4量化）和TPU原生的梯度路由。NeMo-RL使用自定义的与DTensor兼容的模块，并带有可选的Triton融合内核。

**仅适配器权重同步的机会（与轴3的交互）：**

在十三家库中，有八家支持仅将LoRA适配器增量推送到推理服务器。这彻底改变了权重同步问题（维度3）的性质。在使用全参数训练时，中断模型（每次前向传播锁定 vs. 每次请求中止 vs. 每批次暂停）决定了在NCCL广播期间有多少生成过程被浪费。当使用仅同步适配器的LoRA时，传输的数据量非常小，以至于几乎任何中断模型都能提供相当的吞吐量！即使是Atropos那种粗暴的HTTP热交换也变得可行。

### 维度7：分布式训练后端与并行策略

库使用何种并行策略进行训练？这如何约束或赋能异步架构？

这个维度贯穿所有其他维度。训练后端的选择决定了每个GPU能容纳多大的模型、在向推理服务器广播之前需要多少次集合操作来收集权重，以及哪些模型架构根本可以被训练。对于团队将规模扩展到超过300亿参数或从稠密模型转向混合专家模型而言，这是最具决定性意义的决策。

训练后端对异步强化学习库的设计有直接影响：

权重同步速度直接取决于训练后端，更快的同步意味着更低的陈旧度。

在解耦的异步设置中，权重同步不一定会阻塞推理。关键的设计决策在于权重更新如何与正在进行的生成过程交互；存在四种策略，按破坏性从低到高排列：

- **原子交换，无中断。** 完整的权重更新作为一个阻塞式RPC发送到推理引擎。每次前向传播要么看到全部旧权重，要么看到全部新权重，绝不会混合。生成过程最多暂停一个前向传播的间隔（约几毫秒）。(PipelineRL)
- **逐参数流式传输，无中断。** 每个参数作为单独的RPC + NCCL广播发送。前向传播穿插在单个参数更新之间进行，因此正在处理的序列确实会在不同层看到新旧权重的混合。重叠度最高，但一致性最弱。(open-instruct, inflight模式)
- **调度门控，排空进行中请求，然后同步。** 新请求被暂缓，直到进行中的序列自然完成；权重仅在流水线排空后才广播。没有Token浪费，但同步气泡与最长的进行中序列成正比。(PRIME-RL, AReaL, open-instruct默认模式, verl完全异步模式)
- **硬暂停或中止。** 推理被暂停，或者在权重传输开始前中止进行中的请求。一致性最干净，计算浪费最高。(verl, SkyRL)

但即使在推理继续进行的库中，**同步越慢，推理在陈旧权重上运行的时间就越长**。训练器和推理池之间的策略版本差距会随着同步持续时间而增长。这是需要考虑的因素。

**随着领域向稀疏模型发展，对MoE的支持正成为一个日益重要的差异化因素。** 趋势很明显：前沿模型是稀疏的（DeepSeek-V3、Qwen3-MoE、Mixtral、DBRX），而开放权重的MoE模型正成为训练后微调的默认起点。训练这些模型需要专家并行，将不同的专家分配到不同的计算秩上，而大多数异步强化学习库并不支持这一点。只有基于Megatron的库（verl、SLIME、MILES、ROLL、NeMo-RL）和PRIME-RL的FSDP2+EP路径能正确处理EP。基于ZeRO的库（PipelineRL、verifiers-rl、OAT、open-instruct）可以加载MoE的HuggingFace模型类，但在没有EP的情况下，每个专家会在所有ZeRO-3秩上进行分片，而不是被放置在专用的秩上；每次前向传播都会AllGather每个专家，完全抵消了稀疏性的优势。EP也使权重同步复杂化：在向vLLM/SGLang（通常从单个TP组服务所有专家）广播之前，训练器必须从每个EP秩上AllGather专家参数，这是一个O(N∗experts×E∗size)O(N*{\text{experts}} \times E*{\text{size}})O(N∗experts×E∗size)的通信（其中E_sizeE\_{\text{size}}E_size是每个专家的参数量），这在稠密模型中是不存在的。对于一个拥有256个专家的235B MoE模型，这是一个相当大的同步开销。希望在未来一代开放MoE模型的训练后微调中保持相关性的库，需要具备EP感知的训练和EP感知的权重同步能力。

**MoE LoRA是一个新兴需求，也是一个棘手的问题。** 稠密模型上的LoRA（维度6）已经很好理解：将适配器附加到注意力投影层，训练它们，仅同步适配器增量。MoE LoRA更难，因为自然的目标是专家前馈网络层，这意味着每个专家都有自己的适配器。对于一个拥有64个专家、每个专家的门控/上/下投影层使用秩为32的LoRA的模型，适配器数量会从约20个（稠密）激增到200多个（MoE），并且这些适配器分布在EP各秩上。权重同步必须在将适配器推送到推理服务器之前，从每个EP秩收集适配器，这是一个在稠密LoRA中不存在的协调问题。在调查的库中，只有ART明确实现了MoE专家LoRA层（使用每个专家LoRA和手动Allreduce的Megatron EP路径），MILES通过Megatron-Bridge支持LoRA，可以定位专家层。verl的Megatron-Bridge路径支持包括vlm_lora在内的LoRA类型，但未记录针对MoE的专家LoRA。vLLM的LoRA服务本身不支持每个专家的适配器；它加载一个统一应用的单一适配器，因此目前MoE LoRA的仅适配器同步需要自定义推理端逻辑。随着MoE模型成为训练后微调的默认选择，具备高效仅适配器同步的MoE LoRA将是一个需要填补的关键能力缺口。

以上涵盖了七个维度，每个维度都捕捉了同一根本问题的不同侧面。它们共同为我们提供了一个比较不同库的完整视角。是时候把所有信息汇总到一页了。

## 5. 下一波浪潮：设计影响

以下趋势并非新技术的目录；每一个都对当前的基础设施和算法选择施加了具体的压力。问题不在于"前沿是什么？"，而在于"如果这个趋势胜出，我当前的堆栈中什么会出问题？"

### 5.1 无评论家算法：内存释放，但权重同步压力增加

PPO的价值网络使任何训练节点的内存占用翻倍。该领域正汇聚于无评论家变体（GRPO、REINFORCE++、在线DPO），正是因为长思维链推理在8K–64K上下文长度下使得这种开销变得难以承受。

**这解锁了什么：** 消除评论家释放了约50%的训练GPU内存。这些空闲资源可以重新分配到：(a) 更大的轨迹批次，直接减少掉队者方差问题，或 (b) 在同一GPU上共置推理和训练，这完全消除了对单独的NCCL权重同步进程组的需求。

**它没有解决什么：** 无评论家方法仍然需要频繁地将权重推送到推理服务器。实际上，它们可能**增加**同步压力：没有价值网络提供稳定的基线，GRPO风格的算法需要更大的组大小（G=8–32）来获得低方差的优势估计，这意味着每步有更多的轨迹和更快的策略漂移。仅在粗粒度边界（每个训练步或每K步）进行同步的库，在无评论家训练下将看到陈旧度增长得更快。

非对称轨迹过滤（GRPO-RoC：对轨迹进行过采样，严格过滤正样本，均匀下采样负样本；DeepSeek-V3.2 和 MiniMax-M1 中采用的 CISPO/DAPO 风格非对称裁剪）对陈旧性的影响更为微妙。问题不在于批次本身缩小，而在于**幸存批次的构成**。正样本轨迹（简单提示词的正确解决方案）收敛更快，被优先保留；较难的提示词则产生大量被丢弃的负样本轨迹。结果是：通过过滤幸存下来的样本，在系统上比缓冲区中的平均轨迹**更陈旧**，因为它们所解决的简单提示词是在训练早期发出的。一个充满名义上"新鲜"轨迹的缓冲区，其幸存的正样本可能跨越了多个策略版本。在批次层面跟踪陈旧性的准入控制（例如，SkyRL 的 `max_staleness_steps` 容量门，Atropos 的 `max_batches_offpolicy`）无法检测到这种批次内的版本分布。在这种机制下，**按样本进行版本标记（轴 4）** 不是可选项；训练器必须能够拒绝或进行重要性采样校正那些策略版本偏离过远的单个样本，即使它们所属的批次是最近才被准入的。

无批评器方法简化了训练侧。但**评分侧**即将变得更加昂贵：过程奖励模型对中间推理步骤进行评分，而不仅仅是最终答案，这引入了一个全新的同步瓶颈。

### 5.2 过程奖励：一个新的同步障碍

结果奖励是标量且廉价，只需在轨迹结束时调用一次验证器。过程奖励模型对中间步骤进行评分，这需要 (a) 对整个推理轨迹进行一次单独的 PRM 前向传播，或者 (b) 在生成过程中逐 Token 计算的在线效用函数。

**PRPO**（基于熵峰值的分段，并对每段进行 PRM 评分）和 **DEEP-GRPO**（通过在线效用函数识别关键点）都会在**生成和训练之间**产生计算开销。在当前的开源库生态中，这个阶段难以映射到预处理池（PipelineRL），或者需要一个额外的 Ray actor（verl, NeMo-RL）。两者都不是为此设计的。

关键影响是：**基于 PRM 的信用分配打破了"奖励计算廉价"的假设**。对一个来自 70 亿参数模型的 32K Token 推理轨迹进行 PRM 前向传播，成本可能非常高。在每个提示词生成 G=8 个补全的情况下，奖励计算可能消耗相对于生成本身不可忽视的挂钟时间。这导致两个后果：

1.  **异步奖励流水线变得必要**。PRIME-RL 将奖励评分与训练并发运行，作为其完全异步的 Orchestrator-Trainer 流水线的一部分；Orchestrator 处理评分，而 Trainer 独立执行反向传播和优化器步骤。对于基于 PRM 的方法，这种流水线化的奖励计算不是可选的；同步的奖励评分将主导训练挂钟时间。
2.  **独立的预处理池变得必要**。例如，在专用 GPU 层级上运行参考对数概率计算和 PRM 评分，并在生成和训练之间进行流水线化，这是密集信用分配的正确架构。

**DEEP-GRPO 的关键点重采样**引入了除标准轨迹和部分轨迹恢复之外的第三种生成模式：**从序列中间状态进行局部重采样**。这需要在关键点保存 KV 缓存状态，而**目前没有任何异步库开箱即用地支持此功能**。关键点边界的权重同步可能成为一个新的正确性要求：如果在关键点生成和局部重采样之间权重发生变化，优势估计就会被破坏。当然，我们可以通过一次前向预填充重新计算 KV 缓存，但这可能会浪费我们训练中宝贵的计算资源。

### 5.3 多智能体协同进化：掉队者问题加剧

单智能体 GRPO 训练一个策略，每个提示词生成 G 个补全。新兴的多智能体自我对弈意味着有效的"组"包含多个顺序链接的模型调用。只有在链中所有模型都完成后才能获得奖励。

**掉队者动态发生了质变**。在单智能体 GRPO 中，掉队者是一个组中最长的补全，是单峰长度分布中的一个尾部事件。在多智能体流水线中，掉队者是**两个或更多长度分布的乘积**。在提议者/求解者多智能体架构中，如果每个智能体的第 90 百分位完成时间（中位数的 5 倍），那么联合的第 90 百分位时间大约是**中位数的 25 倍**。

对智能体群进行 RL 意味着一个新的工作单元。目前，每个库中的原子单元是单个（提示词，补全，奖励）三元组。在多智能体训练中，原子单元变成了一个**回合**，一个由回合、工具调用和智能体间消息组成的有向图。缓冲区设计、陈旧性跟踪和优势计算都需要在回合层面进行操作。重放或分叉回合也可能变得必要。

当模型至少内部一致时，跨智能体的掉队者问题已经够糟糕了。对于 MoE 架构，即使是单个模型在其推理和训练框架之间也可能自相矛盾，这在 RL 训练中引发了一系列新的问题。

### 5.4 训练-推理不匹配：Deepseek v3.2 MoE 案例研究

训练-推理不匹配问题在异步 RL 中普遍存在；只要轨迹数据是在策略 π*old 下生成，而梯度更新是在 π*θ 下计算，两个策略就会产生分歧。大多数库通过重要性采样校正或硬版本拒绝来解决这个问题。但 DeepSeek-V3.2 的生产经验揭示了两个**结构性**的不匹配来源，这是重要性采样校正无法解决的。

**来源 1：MoE 专家路由不一致**。混合专家模型为每个 Token 激活一个稀疏的专家子集。推理框架（vLLM, SGLang）和训练框架（Megatron, FSDP）独立实现路由器，门控函数中浮点数舍入的差异可能导致**对相同输入选择不同的专家**。当专家路由出现分歧时，活跃的参数子空间会发生不连续的变化；假设专家 A 活跃时计算的梯度步长，被应用到了在专家 B 下活跃的权重上。DeepSeek-V3.2 发现这"导致活跃参数子空间发生突变，从而破坏优化稳定性并加剧离策略问题。"

他们的解决方案 **Keep Routing**，保留了采样（推理）期间使用的确切专家路由路径，并在训练前向传播期间强制执行这些路径。这要求推理框架在记录并返回 Token 对数概率的同时，记录并返回路由决策，并且训练框架接受并强制执行它们。目前没有任何开源异步 RL 库实现此功能。对于任何训练 MoE 模型的团队（DeepSeek-V3 级别、Mixtral、未来的开源 MoE），这是一个正确性问题，而非性能问题。

**来源 2：采样截断掩码不匹配**。Top-p 和 top-k 采样在生成时截断词汇表，将低概率 Token 排除在采样分布之外。在训练期间，完整词汇表对 π*θ 是可见的。这违反了重要性采样的恒等式：π*old（截断）和 π*θ（完整）的动作空间不同，因此对于在采样期间被掩码的 Token，重要性采样比率 π*θ(o*t) / π*old(o_t) 是未定义的。

DeepSeek-V3.2 的 **Keep Sampling Mask** 解决方案：在采样期间记录截断掩码，并在训练前向传播期间将其应用于 π_θ，从而使两个策略在相同的词汇表子集上操作。这需要将掩码从推理服务器传递回训练器，这同样是当前库基础设施不支持的功能。

对库设计的影响：保持路由和保持采样掩码都要求推理服务器在返回Token概率对数、路由决策和采样掩码的同时，返回额外的元数据。当前推理服务器（vLLM、SGLang）与训练器之间的API约定是`(token_ids, logprobs, finish_reason)`。将其扩展为`(token_ids, logprobs, finish_reason, expert_routing, sampling_mask)`，这对每个库的数据流而言都是一项破坏性变更。

### 5.5 蒸馏：同一异步问题，不同名称

同策略蒸馏，即学生模型生成序列，教师模型用Token级别的概率对数进行评分，在结构上与GRPO中的异步协调问题完全相同。

本综述中的每一个设计维度——经验池缓冲区、权重同步协议、陈旧性管理以及部分经验处理——都完全适用于蒸馏。生成池产生学生经验，教师对其进行评分（替代验证器），然后训练器使用经过优势调整的GRPO损失函数或独立的KL目标函数进行反向传播。自蒸馏则增加了一项协调要求：教师是学生在第N-k步的冻结快照，因此系统必须定期对策略进行快照，并在不中断流水线的情况下热切换教师服务器，这是一个目前没有任何库实现完全自动化的基础功能。

这对库设计的实际意义在于，异步强化学习基础设施不应构建为GRPO专用的系统。生成-评分-训练的流水线是一个通用模式，涵盖了带结果奖励的强化学习、带过程奖励的强化学习、同策略蒸馏和自蒸馏。像SLIME、MILES、PRIME-RL、AReaL和NeMo-RL这样的库已经同时支持GRPO和同策略蒸馏，正是因为它们的异步框架将奖励/评分阶段视为可插拔的组件，而非硬编码的验证器调用。任何旨在通用性的异步训练器都应如此：将评分阶段定义为一个接口（一个HTTP端点、一个Ray Actor或一个同址前向传播），并让缓冲区、陈旧性和权重同步机制以相同的方式运行，无论其内容是什么。

## 6. TRL异步训练器的设计选择

在全面审视了编排模型、缓冲区设计、权重同步协议、陈旧性策略和部分经验处理后，我们现在可以为TRL中的异步训练器制定具体的设计选择，以及我们计划探索的未来发展方向。

### 设计原则：保持编排轻量化

当前TRL实现的一个优势在于，它不依赖繁重的编排器系统来管理训练生命周期。库内部的数据保持为原生Python对象，不受外部库的影响。我们希望保持这一点：编排应尽可能简单，不依赖重量级的外部框架。

### 1. 带每Token模型版本的有界队列（无双缓冲）

与其从双缓冲开始再逐步过渡到更细粒度的方案，我们直接采用一个有界队列，其中每个Token都带有生成它的模型版本标签。这是从一开始就实现的最低可能粒度；它支持Token级别的重要性采样校正，支持简单的准入门控（丢弃或降低权重超过陈旧性阈值的Token），并避免了后期为批次级缓冲区添加Token级溯源功能所带来的架构债务。

### 2. 使用打包传输的NCCL权重同步

NCCL进程组是必需的，并且我们已经在使用它们。下一步应添加分桶功能，因为vLLM的`NCCLWeightTransferEngine`在`packed=True`时直接支持分桶广播：它将参数打包成可配置大小的`uint8`缓冲区（默认为1 GB，在CUDA流之间双缓冲），并通过一个独立于训练进程组的专用NCCL通信器进行广播。这消除了在简单广播中占主导地位的每个参数调用开销，从而大幅提升了同步速度。

除了vLLM内置的引擎，我们还将探索用于更苛刻场景的高性能权重打包库：

- **Awex (inclusionAI)**，一个专为强化学习训练设计的权重同步框架，它解决了跨引擎传输的难题：训练引擎（Megatron、DeepSpeed）和推理引擎（SGLang、vLLM）使用完全不同的并行策略和张量布局。Awex通过统一的转换层和确定性的P2P传输计划将其抽象化。它同时支持分离GPU和同址（CUDA IPC）模式。
- **Mooncake传输引擎**，SGLang已开始集成Mooncake传输引擎作为其高性能传输层，集成范围涵盖PD解耦、分层KV缓存和弹性专家并行。具体到权重同步，配套的`checkpoint-engine`项目利用Mooncake基于RDMA的P2P传输，可在约16-17秒内更新万亿参数模型（Kimi-K2，256×H20 GPU）。Mooncake现已成为PyTorch生态系统的一部分，并作为`NVIDIA's NIXL`传输库的后端插件。

### 3. 为智能体工作负载提供部分经验支持

复杂环境中的多轮工具使用任务，每次经验生成可能需要数分钟。如果没有在权重更新期间处理进行中经验的机制，同步窗口将成为流水线的瓶颈。我们可能会实验性地探索两种策略：

- **前缀恢复**：当权重在经验生成中途更新时，保存KV缓存前缀，并在新策略下从检查点恢复生成。这保留了部分工作成果，但需要推理引擎支持序列中途的权重交换。
- **中止并重试**：丢弃超过陈旧性阈值的进行中经验，并将提示词重新加入队列。实现更简单，但会浪费与中止时平均经验长度成比例的计算资源。
