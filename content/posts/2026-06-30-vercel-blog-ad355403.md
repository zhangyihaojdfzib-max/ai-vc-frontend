你的服务器运行在容器中。它可能是一个 Go 服务、一个 Rails 应用、一个 Spring Boot API，或者一个运行在 nginx 背后的 Web 服务器。它使用 HTTP 协议通信，监听某个端口。它只需要一个运行环境。

在你的项目中添加一个 `Dockerfile.vercel` 文件，Vercel 就会在 Fluid Compute 上构建、存储、部署并自动缩放该镜像，你只需为代码使用的 CPU 付费。无需在本地运行守护进程、设置镜像仓库或照看集群。

## 工作原理

这是一个用 Go 编写的简单 HTTP 服务器，监听 `$PORT` 端口：

```
1package main
2
3import (
4	"fmt"
5	"net/http"
6	"os"
7)
8
9func main() {
10	port := os.Getenv("PORT")
11	if port == "" {
12		port = "80"
13	}
14
15	http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
16		fmt.Fprintln(w, "Hello from a container on Vercel 👋")
17	})
18
19	http.ListenAndServe(":"+port, nil)
20}
```

一个极简的 HTTP 服务器，从 `$PORT` 环境变量读取端口，并响应所有请求。

添加一个 `Dockerfile.vercel` 文件，将其构建成一个小型镜像并运行：

```
1FROM golang:1.24-alpine AS build
2WORKDIR /src
3COPY . .
4RUN go build -o /server main.go
5
6FROM alpine:3.20
7COPY --from=build /server /server
8CMD ["/server"]
```

一个两阶段构建：先编译二进制文件，然后将其复制到一个启动时运行的最小化 Alpine 镜像中。

然后部署：

```
▲ vercel deploy
Vercel CLI
✓ Building image from Dockerfile.vercel
✓ Stored image in your project's registry
✓ Deployed to Fluid compute
Production: https://my-server.vercel.app
```

一条命令即可构建镜像、存储并发送到 Fluid Compute，然后输出生产环境 URL。

仅此而已。两个文件，你的服务就上线了。每次 `git push` 都会重新构建镜像，并为你提供一个全新的预览 URL。或者直接运行 `vercel` 进行部署，无需提交代码。

本例中我们使用了 Go，但任何技术栈都可以。Rails、Spring Boot、Express、Laravel、ASP.NET、FastAPI，以及运行在 nginx 背后的 Web 服务器，都以相同的方式部署。唯一的规则是你的服务器需要监听 `$PORT` 端口，默认值为 `80`。只要它使用 HTTP 协议通信，就能部署。是的，Java 也可以。是的，PHP 也可以。

## 你能获得什么

Vercel 上的容器是一等公民。它与你 Vercel 上的前端和其他服务运行在同一个平台和计算资源上。

- **每次推送都有预览部署**：每次提交都会获得一个不可变的 URL，你可以打开、分享或回滚。
- **双向自动缩放**：流量到来时自动扩容，流量停止时实例自动缩减。你无需手动调整集群规模或猜测并发数。
- **活跃 CPU 计费**：Fluid Compute 仅按代码实际运行时间计费。空闲的服务器，即使因慢查询或上游 API 而挂起，也不会在等待时消耗 CPU。你只为执行时间付费，而非挂墙时间。
- **内置可观测性**：容器的日志、链路追踪和指标与你部署的其他所有内容共享同一个仪表盘。
- **一个项目，一个域名**：你的容器与前端及其他服务并排放置，并通过 Vercel 网络进行私有通信。你的全栈应用作为一个整体部署。

## 专为快速启动而构建

容器的价值取决于它响应第一个请求的速度。

当 Vercel 构建你的镜像时，它会将其存储为**优化的启动镜像**，这是一个针对快速启动而优化的容器磁盘压缩快照。

当容器启动时，我们流式传输该快照并按需解压，而不是在运行任何内容之前下载整个镜像。你的服务器可以在完整镜像就位之前开始处理请求，因此较大的镜像无需等待下载完成。

一旦实例运行起来，Fluid Compute 会保持其热状态，并用它处理多个请求，而不是为每个请求启动一个新副本。你既能获得热服务器的响应速度，又能享受空闲时休眠的计费方式。

每个容器都是一个无状态进程：它接收请求，返回响应，不保留任何中间状态。持久化状态存储在你附加的后端服务中，例如来自 Vercel Marketplace 的数据库或缓存。由于实例不保留任何需要持久化的数据，Vercel 可以在流量到来时添加实例，并在流量停止时回收它们。我们也在努力尽快推出容器附带的持久化存储功能。

## 为什么是现在？

我们的第一个平台允许你用一条命令部署 Dockerfile。那是十年前的事了，想法是对的，但当时还没有足够好的基础设施来支撑它。

此后的时间里，我们一直在构建能够良好处理它的基础组件。它们为你运行在 Vercel 上的一切提供动力：构建、函数、沙箱，以及现在的容器。所有这些都随流量自动缩放，你只需为使用的 CPU 付费。容器现在是一等公民，与所有其他内容运行在同一个系统上。

框架检测是我们的入口。当我们识别出你的框架时，我们会读取你的代码并推导出应用所需的基础设施，因为代码本身已经描述了它应该做什么。对于大多数应用来说，这是最快的部署方式。Dockerfile 则用于其他所有情况：需要系统库（如 FFmpeg 或 Chromium）的服务、我们尚未自动检测到的框架，或者你想原样部署的应用。它是一种通用的方式，用于描述程序应如何构建，因此当没有框架可读取时，我们直接通过它来对接。

你的 Dockerfile 周围的一切都是零配置。你只需指向镜像，构建、镜像仓库、部署、缩放和 URL 都会自动完成。
