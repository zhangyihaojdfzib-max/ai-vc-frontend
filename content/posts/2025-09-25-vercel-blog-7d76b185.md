当你使用**增量静态再生（ISR）**部署 Next.js 应用时，页面会在其缓存过期后按需重新生成。ISR 让你既能享受静态生成的性能优势，又能保持内容的新鲜度。

但这里存在一个问题。当许多用户同时请求同一个 ISR 路由且缓存已过期时，每个请求都可能触发其自身的函数调用。这被称为“缓存踩踏”。它会浪费计算资源，使你的后端过载，并可能导致服务中断。

Vercel CDN 现在通过**请求合并**来防止这种情况。当多个请求命中同一个未缓存路径时，每个区域只有一个请求会调用函数。其余请求则等待并获取缓存后的响应。

Vercel 通过框架定义的基础设施自动推断每个请求的可缓存性，从而配置我们全球分布的路由器。无需手动配置。

## 缓存通常如何工作

要理解请求合并为何重要，我们先来看看在一切运行顺畅时，Vercel CDN 是如何处理请求的。

Vercel CDN 是全球分布的，首先是 **ISR 缓存**，它与你的函数共存，并存储静态再生的结果。它是将内容**复制**到每个区域的 **Vercel 缓存**的真相源。当用户发出请求时，最近的 CDN 区域会为其提供服务（更具体地说，是每个区域内的一个节点）。

每个区域都有多个节点（服务器实例），它们根据流量进行扩展。每个节点运行多个工作线程来处理并发请求，并为频繁请求的内容维护一个小的内存缓存。

## 何时可以应用合并

请求合并仅在已知请求会产生可缓存响应时才有效。可能返回不同结果或根本不应缓存的请求无法合并。

考虑以下场景：

- 为所有用户重新生成相同内容的 ISR 页面可以安全地合并
- 返回用户特定数据的动态 API 路由无法合并
- 包含随机内容或时间戳的页面不应合并

为所有用户重新生成相同内容的 ISR 页面可以安全地合并

返回用户特定数据的动态 API 路由无法合并

包含随机内容或时间戳的页面不应合并

挑战在于如何自动确定这一点。Vercel 通过框架集成解决了这个问题。当你部署应用时，Vercel 会分析你的路由，并了解哪些路由使用 ISR、静态生成或动态渲染。这些元数据会被分发到每个 CDN 区域。

当请求到达时，CDN 已经知道该特定路由是否可以安全地缓存和合并。这一切都无需你进行任何配置。框架的代码告诉 Vercel 每个路由的行为方式，Vercel 会自动应用正确的缓存策略，包括请求合并等功能。

## 分布式锁

请求合并通过一个两级分布式锁系统工作。

在**节点**级别，每个 CDN 节点维护一个内存锁。当多个针对同一未缓存路径的请求到达该节点时，锁确保只有一个请求继续执行。其他请求等待直到缓存被填充。这防止单个节点为同一路径触发多个函数调用。

在**区域**级别，每个区域在所有节点之间强制执行自己的锁。在获取节点锁之后，请求会尝试获取区域锁。每个区域只有一个请求可以同时持有两个锁，这意味着每个区域只有一个函数调用可以重新生成页面。

这种设计使合并具有可扩展性。如果没有节点级别的分组，数百个并发请求可能会同时竞争区域锁。这将造成惊群效应，锁协调本身会成为瓶颈。

相反，节点锁首先在本地对请求进行分组。区域级别的等待者数量与区域内的节点数成正比，而不是与总请求数成正比。在节点内部，等待者数量与命中该特定节点的并发请求数成比例。

通过限制每个级别的等待者数量，CDN 防止锁协调机制不堪重负。每个请求要么立即从缓存中获取服务，要么短暂等待锁持有者完成。

该机制可以用如下伪代码表示：

```
1function createDistributedLock(cacheKey) {2  const nodeLock = createNodeLock(cacheKey);3  const regionalLock = createRegionalLock(cacheKey);4  return combineLocks([nodeLock, regionalLock]);5}6
7async function respond(request) {8  const cacheKey = getCacheKey(request);9  const cachedResponse = await cache.get(cacheKey);10  if (cachedResponse) return cachedResponse;11
12  const lock = createDistributedLock(cacheKey);13  await lock.lock();14  const response = await invokeFunction(request);15  lock.unlock();16
17  return response;18}
```

基于锁的流程：检查缓存，获取锁，如有需要则重新生成。

每个请求首先检查缓存，然后在重新生成前获取两个锁。

## 双重检查锁定

仅靠锁定并不能合并请求。如果每个等待者在获取锁后都调用函数，工作仍然会被重复执行。

为解决此问题，CDN 使用了著名的双重检查锁定模式。其思路简单但强大：对缓存进行两次检查。

1.  首次检查：当请求到达时，检查缓存。如果内容存在，立即返回，无需获取锁。
2.  获取锁：如果缓存为空，则获取节点锁和区域锁。这确保只有一个请求可以进行重新生成。
3.  二次检查：获取锁后，再次检查缓存。在等待锁的过程中，可能已有另一个请求完成并填充了缓存。如果是这种情况，当前请求可以跳过重新生成，直接返回缓存值。
4.  重新生成：只有当缓存仍然为空时，请求才会调用函数，并在释放锁之前设置缓存。

首次检查：当请求到达时，检查缓存。如果内容存在，立即返回，无需获取锁。

获取锁：如果缓存为空，则获取节点锁和区域锁。这确保只有一个请求可以进行重新生成。

二次检查：获取锁后，再次检查缓存。在等待锁的过程中，可能已有另一个请求完成并填充了缓存。如果是这种情况，当前请求可以跳过重新生成，直接返回缓存值。

重新生成：只有当缓存仍然为空时，请求才会调用函数，并在释放锁之前设置缓存。

这确保了并发请求不会重复工作。第一个请求重新生成页面。其他请求要么立即命中缓存，要么在等待后读取新值。

以下是逻辑代码示例：

```
1async function respond(request) {2  const cacheKey = getCacheKey(request);3  const cachedResponse = await cache.get(cacheKey);4  if (cachedResponse) return cachedResponse;5
6  const lock = createDistributedLock(cacheKey);7  await lock.lock();8  let cachedResponse = await cache.get(cacheKey);9  if (cachedResponse) return cachedResponse;10
11  const functionResponse = await invokeFunction(request);12  13  (async () => {14	  await cache.set(cacheKey, functionResponse);15	  lock.unlock();16  })()17
18  return functionResponse;19}
```

双重检查锁定：仅在获取锁后缓存仍为空时才重新生成。

请注意，缓存是在函数返回后异步写入的。这使得响应可以在不等待缓存设置操作完成的情况下发送回用户，从而减少了首字节时间。同时，锁在缓存填充后立即释放，因此等待者可以快速继续。

## 处理故障

到目前为止，我们假设函数总是成功。但真实系统会失败，请求合并需要优雅地处理这些情况。有两种故障模式尤其重要：

-   函数错误：如果函数调用抛出错误，结果将无法被缓存。这意味着第二次缓存查找仍然返回空值。下一个获取锁的请求必须再次尝试重新生成。在这种情况下，合并没有帮助，因为没有有效的响应可以共享，但系统仍然确保错误不会污染缓存。
-   函数超时：超时更危险。如果锁持有者花费很长时间（或永不完成），其他等待锁的请求可能会无限期地阻塞。在流量高峰期间，这可能意味着数十或数百个请求堆积在一个缓慢的调用后面。

函数错误：如果函数调用抛出错误，结果将无法被缓存。这意味着第二次缓存查找仍然返回空值。下一个获取锁的请求必须再次尝试重新生成。在这种情况下，合并没有帮助，因为没有有效的响应可以共享，但系统仍然确保错误不会污染缓存。

函数超时：超时更危险。如果锁持有者花费很长时间（或永不完成），其他等待锁的请求可能会无限期地阻塞。在流量高峰期间，这可能意味着数十或数百个请求堆积在一个缓慢的调用后面。

为防止这种情况，锁创建时带有明确的超时设置。如果请求在固定窗口内（例如，几秒钟）无法获取锁，它将放弃等待并继续自行调用函数。这种“对冲”机制确保了缓慢的重新生成不会阻塞该路由的所有请求。即使在最坏的情况下，用户也能继续获得响应，尽管代价是多次调用。

以下是代码示例：

```
1function createDistributedLock(cacheKey) {2  const nodeLock = createNodeLock(cacheKey, { timeout: 3000 });3  const regionalLock = createRegionalLock(cacheKey, { timeout: 3000 });4  return combineLocks([nodeLock, regionalLock]);5}
```

创建带超时的锁，以防止请求无限期等待。

在此配置下，每个请求最多等待节点级锁和区域锁各三秒钟。如果锁未能及时获取，请求将独立向前执行。

这种超时策略取得了平衡。它在正常情况下保留了合并的好处，但也避免了在函数运行缓慢或不稳定时发生级联故障。实际上，系统优化了常见情况，同时仍然对错误和长尾延迟保持弹性。
