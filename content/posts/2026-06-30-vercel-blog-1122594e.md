Hydrogen 让无头商店前端易于部署，但不可移植。在纽约举办的 Vercel Ship 26 大会上，我们宣布正在与 Shopify 合作，从头重建它，这是我们对更开放网络的一次共同押注。

![Shopify 产品副总裁 Vanessa Lee 与 Vercel 首席产品官 Tom Occhino 在 Ship NYC 舞台上](/images/posts/08d4ca2d1978.jpg)

新版本是开源的，且与运行时无关，这意味着它可以在任何支持 JavaScript 的环境中运行。你可以使用 Svelte、Nuxt、Next.js 进行构建，甚至可以自带自定义框架。

我们的策略包括三个层面：核心层、客户端层和服务端层。

## 链接到标题 核心层

核心层是我们过去都用来编写 Shopify API 的 JavaScript 代码，但从未共享过。现在它集中在一个地方。

以 `formatMoney` 为例。开放网络已经通过 `Intl.NumberFormat` 解决了大部分问题。

```
1const price = new Intl.NumberFormat("en-US", {2  style: "currency",3  currency: "USD",4}).format(19.99); 
```

但 Shopify API 返回的不是一个 `number`。它返回的是一个自定义类型 `MoneyV2`，其中金额是一个序列化为字符串的有符号十进制数。

```
1import { formatMoney, type MoneyV2 } from "@shopify/hydrogen";2
3export function formatPrice(money: MoneyV2, locale = "en-US") {4  return formatMoney(money, { locale }).toString(); 5}
```

结果是一样的，但你不再需要编写或维护这些胶水代码。当 API 发生变化时，升级也变得微不足道。

将核心层集中化，你只需修复每个 bug 一次，就能将改进推送给所有人，然后回到构建本身。

## 链接到标题 服务端层

开发者需要全栈访问权限来构建可扩展且不牺牲性能的商店前端。静态内容应从 CDN 即时提供，而库存等动态数据则应流式传输。

开源社区已经用 Next.js、Nuxt 和 SvelteKit 等框架解决了这个问题：提供全栈能力，且不锁定于专有运行时。

假设你的商店前端使用按需重新验证来缓存产品查询。你编写 GraphQL 查询。Hydrogen 为你提供一个类型安全的客户端。Next.js 处理缓存，这样你就得到了全栈框架加上无头 Shopify API，而无需任何胶水代码。

```
1import { PRODUCT_QUERY } from "@/lib/gql";2import { storefrontConfig } from "@/lib/config";3import { cacheTag } from 'next/cache'4import { createStorefrontClient } from "@shopify/hydrogen";5
67export async function getProductData({ handle }) {8  "use cache";9  cacheTag(handle);10
11  const client = createStorefrontClient({12    type: "private_shared_rate_limit",13    config: storefrontConfig,14  });15
16  const { data } = await client.graphql(PRODUCT_QUERY, {17    variables: { handle },18  });19
20  return data21}
```

Shopify 已经通过其无头销售渠道支持这些框架，但在此之前，我们每个人都在为同一个 API 契约编写自己的绑定。在这个层面，解决方案是提供指导，而不是编写更多代码。人类和 Agent（智能体）都需要知道如何使用这些框架已有的功能，而不是为 Shopify 重新发明轮子。

这些指导将以文档、模板和技能的形式提供。

42## 缓存组件兼容性 — 请先阅读此内容43
44模板以 `cacheComponents: true`（Next.js 16）运行。这会改变一些此技能需要正确处理的事项。跳过任何一项都会导致看似无关的构建错误：45
46### A. `app/[locale]/` 之上不能存在 `app/layout.tsx`47
48为了使 `[locale]` 被识别为根参数，动态段必须是根布局。在步骤 2 之后，`app/layout.tsx` 文件应被移除（移至 `app/[locale]/layout.tsx`）。如果两者都存在，`rootParams.locale()` 将返回 `undefined`。49
50### B. 不使用 `setRequestLocale`51
52next-intl 文档有时会在布局/页面中展示 `setRequestLocale(locale)` 调用。**在 cacheComponents 下不要添加它们。** 该辅助函数会写入请求作用域的存储并强制动态渲染 — 这会破坏缓存。下面的 rootParams + 请求配置模式使其变得不必要，因为解析出的语言已经是缓存键。53
54### C. 不要将 `next/link` 替换为 next-intl 的 `<Link>`55
56最直接的想法是将每个 `import Link from "next/link"` 替换为 `import { Link } from "@/lib/i18n/navigation"`。**不要这样做。** next-intl 的 Link 会在渲染时读取请求上下文（语言）；在 cacheComponents 下的服务器组件树中，这会触发：57
58```59Error: Route "/[locale]/..." accessed [...] which is not defined in the `unstable_samples` of `instant`.60```61
62或一个通用的"阻塞路由"预渲染失败。63
64**请这样做：** 保留 `next/link`，让 `proxy.ts` 中间件重定向无前缀的路径（`/products/foo` → `/en-US/products/foo`）。内部链接正常工作；点击无前缀的 href 时会触发一次性的中间件重定向。用少量重定向换取干净的预渲染。65
66如果您必须为程序化 URL（服务器操作、`redirect()`、`permanentRedirect()`）添加语言前缀，请自行构建路径：`` `/${await getLocale()}/account/login` ``。67
68### D. `instant` 样本需要在 `params` 中包含 `locale`69
70任何导出 `instant` 的路由（当前：产品 `[handle]`、集合 `[handle]`、搜索）都需要在每个样本中添加 `locale`，否则构建会失败：71
72```73Error: Route "/[locale]/products/[handle]" accessed root param "locale"74       which is not defined in the `unstable_samples` of `instant`.75```76
77修复方法：78
79```ts80export const instant = {81  unstable_samples: [82    {83      params: { locale: "en-US", handle: "__placeholder__" }, // ← 添加 locale84      searchParams: { variant: "1" },85      cookies: [{ name: "shopify_cartId", value: null }],86    },87  ],88};89```90
91### E. 如果任何布局级别的服务器组件读取 `headers()`，`instant` 样本需要声明 `headers`92
93这很容易忘记。如果您（或下游技能）在布局中添加了一个调用 `headers()` 的服务器组件 — 例如一个读取 `x-vercel-ip-postal-code` 的"Shipping to {postal}"栏 — 应用程序中的每个 `instant` 样本都必须声明它可能访问的 headers：94
95```ts96unstable_samples: [97  {98    params: { locale: "en-US", handle: "__placeholder__" },99    searchParams: { variant: "1" },100    cookies: [{ name: "shopify_cartId", value: null }],101    headers: [["x-vercel-ip-postal-code", null]], // ← 添加此项102  },103],104```105
106`null` 表示"header 可能不存在"。如果您忘记了，构建错误会明确指出：107
108```109Error: Route "..." accessed header "x-vercel-ip-postal-code" which is not110       defined in the `unstable_samples` of `instant`. Add it to the111       sample's `headers` array, or `["...", null]` if it should be absent.112```113
114### F. next-intl 的 `redirect()` 不返回 `never`115
116```ts117// 错误：TS 在 redirect 后不会缩小 session 类型118import { redirect } from "@/lib/i18n/navigation";119if (!session) redirect({ href: "/account/login", locale });120return session; // 类型错误：session 是 CustomerSession | null121```122
123next-intl 的 `redirect` 类型返回 `void`，因此 TypeScript 不会将其视为控制流结束。请使用 `next/navigation` 的 `redirect`（返回 `never`）并自行添加语言前缀：124
125```ts126import { redirect } from "next/navigation";127import { getLocale } from "@/lib/params";128
129if (!session) redirect(`/${await getLocale()}/account/login`);130return session; // 正确，类型已缩小131```132

217### 步骤 5：`proxy.ts` 中间件  
218  
219```ts  
220import createMiddleware from "next-intl/middleware";  
221import { type NextRequest, NextResponse } from "next/server";  
222import { routing } from "@/lib/i18n/routing";  
223  
224const handlei18n = createMiddleware(routing);  
225  
226export default function middleware(request: NextRequest): NextResponse {  
227  const response = handlei18n(request);  
228  if (!response.ok) return response;  
229  
230  const rewriteHeader = response.headers.get("x-middleware-rewrite");  
231  if (!rewriteHeader) return response;  
232  
233  const rewriteTarget = new URL(rewriteHeader, request.url);  
234  const [, ...segments] = rewriteTarget.pathname.split("/");  
235  const normalized = new URL(`/${segments.filter(Boolean).join("/")}`, request.url);  
236  normalized.search = rewriteTarget.search;  
237  return NextResponse.rewrite(normalized, { headers: response.headers });  
238}  
239  
240export const config = {  
241  matcher: ["/((?!api|_next|_vercel|.*\\..*).*)"],  
242};  
243```
