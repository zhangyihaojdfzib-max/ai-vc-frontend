# olmo-eval：面向模型开发循环的评估工作台

💻 代码：https://github.com/allenai/olmo-eval

![Ai2 Olmo-Eval 图形开发 v3](/images/posts/01c8aab0c563.png)

在构建LLM的过程中，你会在多次干预中反复对其进行评估。对数据、架构或超参数的每一次调整——以及每一次规模上的提升——都会让你回到同一个循环：添加或重新配置基准测试，在每个新的模型检查点上重新运行它们，记录结果，并检查在小实验中有效的方法是否在完整训练运行中仍然成立。

大多数评估工具并非为此而设计——它们要么是为了在已完成的模型上运行既定基准测试而构建，要么是在沙盒中通过多步骤、使用工具的问题来运行模型。它们无法跟上不断变化的模型，也无法反映模型在特定真实世界条件下的行为方式。

我们上一个应对这一评估挑战的项目是OLMES，即开放语言模型评估标准。它于2024年推出，旨在使LLM基准测试分数在不同版本之间更易于比较。相同的模型在相同的基准测试上以不同的方式被评分——提示词格式和任务制定等方面常常因论文而异——因此关于哪些模型表现最佳的声明往往无法复现。OLMES将基准测试选择固定在一个开放、有文档记录的标准中，并成为我们从Olmo到Tulu评估开放模型的基础。

但模型的最终分数只是评估过程的一部分——这就是为什么我们发布olmo-eval，一个基于OLMES构建并将其扩展到LLM开发其余部分的新工作台。与OLMES相比，olmo-eval减少了实施新评估的工作量，在定义评估运行位置和方式方面提供了更多灵活性，并使将单个组件组合成更大工作流程变得更加容易。Agent（智能体）和多轮评估作为一级用例得到支持，更强大的分析工具帮助你判断某项干预是否确实改善了基线，或者差异是否只是噪声。

## olmo-eval与现有工具的区别

![olmo-eval 博客 Kyles 草稿 - Google 文档-图片-1 (1)](/images/posts/01c2865ea658.png)

2.4个百分点的性能变化足以做出判断吗？

olmo-eval在某些方面与Harbor重叠，Harbor是一个用于在容器化沙盒环境中评估AI Agent（智能体）的开放框架。但这两个工具的范围不同。Harbor主要旨在运行和发布Agent（智能体）基准测试；olmo-eval则是为开发模型的日常工作而构建——添加和配置基准测试，在检查点上运行它们，并逐个提示词分析结果，而不是作为一个单一的总体分数。

Harbor以相同的方式运行所有内容——在密封、可复现的容器内。由于容器可能消耗大量资源，olmo-eval让你可以选择每个基准测试的运行方式。只需要模型回答问题的基准测试可以直接运行，这样更快、成本更低；需要锁定环境——例如，运行模型编写的代码——的基准测试则会获得隔离的容器设置。轻量级路径是默认选项，只有当基准测试确实需要时，olmo-eval才会选择重型设置。

Harbor添加基准测试的流程是为你计划发布和公开分享的评估而构建的，带有额外的验证步骤。olmo-eval是为在开发过程中快速推进而构建的，添加基准测试的方式取决于基准测试的需求：对于基本评估，只需简短定义，并提供选项让模型在完成基准测试时使用工具；或者——对于已有自己代码和流程的基准测试——只需一个薄包装器，让olmo-eval可以按原样运行，并以相同格式与其他基准测试分数一起报告结果。

Harbor和olmo-eval都将基准测试与运行时策略（模型如何运行以产生答案）分离，这样你可以在不重写另一个的情况下更改其中一个，但olmo-eval设计为具有更高的模块化程度。在olmo-eval中，被评估的模型、它可以使用的工具、容器化环境以及任何辅助模型——如LLM作为评判者——都是可替换的组件。你可以在多个测试框架中重用工具，或者将评分模型插入一个基准测试而不影响其他测试，并且可以轻松调整小设置（例如，提示词的确切措辞），而无需大量工作。

Harbor为每个模型报告一个总体分数。olmo-eval也报告这些分数，每个分数都带有标准误差和最小可检测效应（可以可靠地与噪声区分的最小差异）。但更有用的视图是将相同的问题在两个模型检查点上对齐，并在保持其他所有条件不变的情况下逐一比较。这有助于你看到总体平均值中的微小变化是否可能表明真正的改进，或者仅仅是噪声。

## 集成评估栈

olmo-eval由四个组件组成，这些组件各自有用，但设计为协同工作以收紧实验性LLM开发循环：

1. **任务/套件/测试框架抽象，将基准测试逻辑与运行时策略解耦。** 任务是你在olmo-eval中定义基准测试的方式——即被评估的内容。套件将任务分组为一组一起运行，测试框架控制每个任务的运行方式。这种分离允许同一任务作为标准基线运行，或使用工具和脚手架运行，而不改变其衡量的内容。
2. **沙盒和能力路由层，包括异步沙盒规划器。** 这支持评估模型响应依赖于其使用工具所采取行动的情况，例如编写和运行代码或浏览网页。关键在于评估模型的真实工具使用：当基准测试需要工具时，olmo-eval运行这些工具并将结果反馈给模型。
3. **标准化实验模式，以相同的结构化格式记录每次运行、其配置和结果。** 这使得可以分组相关实验，随时间比较检查点，并避免在长期运行的模型开发工作流程中经常积累的不一致性。
4. **用于成对模型比较的结果查看器：** 将两个模型或检查点逐问题对齐，揭示总体平均值可能隐藏的微小但真实的性能变化。

**任务/套件/测试框架抽象，将基准测试逻辑与运行时策略解耦。** 任务是你在olmo-eval中定义基准测试的方式——即被评估的内容。套件将任务分组为一组一起运行，测试框架控制每个任务的运行方式。这种分离允许同一任务作为标准基线运行，或使用工具和脚手架运行，而不改变其衡量的内容。

在大多数模型评估设置中，添加基准测试是一个相当大的集成项目。在olmo-eval中，所需要的只是一个任务——任务定义基准测试数据集、如何构建评估请求以及如何对模型答案进行评分（所有代码均为Python）：

```python

```python
from olmo_eval.common.formatters import ChatFormatter
from olmo_eval.common.metrics import AccuracyMetric
from olmo_eval.common.scorers import ExactMatchScorer
from olmo_eval.common.types import Instance, SamplingParams
from olmo_eval.data import DataLoader, DataSource
from olmo_eval.evals.tasks.common import Task, register, register_variant

@register("internal_freshqa")
class InternalFreshQA(Task):
    data_source = DataSource(path="s3://evals/internal/freshqa.jsonl", split="test")
    formatter = ChatFormatter()
    sampling_params = SamplingParams(temperature=0.0)
    metrics = (AccuracyMetric(scorer=ExactMatchScorer),)

    @property
    def instances(self):
        loader = DataLoader()
        for idx, doc in enumerate(loader.load(self.config.get_data_source())):
            yield Instance(
                question=doc["question"],
                gold_answer=doc["answer"],
                metadata={"id": doc.get("id", f"freshqa_{idx}")},
            )
```

变体用于表达评估策略的变化，而无需重复基准测试：

```python
register_variant("internal_freshqa", "3shot", num_fewshot=3, fewshot_seed=1234)
register_variant("internal_freshqa", "zero", num_fewshot=0)
```

测试套件将基准测试分组为可一起运行的标准集合：

```python
from olmo_eval.evals.suites import Suite, register

register(Suite(
    name="base_qa_few_shot",
    tasks=(
        "sciq:mc:3shot",
        "arc_challenge:mc:3shot",
        "internal_freshqa:mc:3shot",
    ),
))
```

由于运行时策略存在于测试框架而非任务定义中，因此同一基准测试可以轻松地在不同执行条件下重新运行，而无需依赖生成的分数点是否看起来合理。

```shell
# 基线
olmo-eval run -m my-instruct-checkpoint -t internal_freshqa:zero

# 相同任务，相同评分，启用搜索/工具运行时
olmo-eval run -m my-instruct-checkpoint -t internal_freshqa:zero --harness search_agent
```

## 开源的可复现评估

当评估是持续模型开发的一部分，而非一次性运行时——当你需要在可复现条件下跨多个检查点重复运行相同的基准测试，并在总体和逐问题层面比较干预效果时——请使用 `olmo-eval`。

如果你反复提出的问题是：“这个检查点与上一个有何不同？它在哪些方面有所改进或退步？”，这正是 olmo-eval 所设计的工作流程。

可复现的评估应与模型的构建方式保持同步——而不仅仅是模型完成后如何评分。olmo-eval 将 OLMES 标准带入活跃的模型开发中，我们将其开源发布，以便社区能够在此基础上继续发展。
