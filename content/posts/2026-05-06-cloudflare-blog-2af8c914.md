# 当 DNSSEC 出问题时：我们如何应对 .de 顶级域中断事件

2026年5月5日，大约世界协调时19:30，.de 国家代码顶级域（TLD）的注册管理机构 DENIC 开始为 .de 区域发布错误的 DNSSEC 签名。任何收到这些签名的验证型 DNS 解析器都必须按照 DNSSEC 规范拒绝它们，并向客户端返回 SERVFAIL，包括 Cloudflare 运营的公共 DNS 解析器 1.1.1.1。

德国国家代码顶级域 .de 是互联网上最大的顶级域之一。在 Cloudflare Radar 上，它始终位列全球查询最广泛的 TLD 之一。DNS 层级中这一层面的中断有可能导致数百万个域名无法访问。

在这篇文章中，我们将介绍我们所观察到的情况、这些事件的影响，以及我们在 DENIC 解决问题期间采取的临时缓解措施。

## DNSSEC 的工作原理

DNSSEC（域名系统安全扩展）为 DNS 增加了加密认证。当一个区域使用 DNSSEC 签名时，每组记录都附带一个称为 RRSIG 记录的数字签名，让解析器能够验证这些记录未被篡改。与加密 DNS 协议（如 DNS over TLS (DoT) 和 DNS over HTTPs (DoH)）不同，DNSSEC 关注的是完整性，而非隐私性。记录是可见的，但其真实性可以得到证明。

DNSSEC 的独特之处在于签名与其保护的记录一起传输。这意味着无论响应经过了多少缓存或跳数，完整性都可以得到验证。缓存的记录与新鲜记录一样可验证。

DNSSEC 建立在信任链之上。从根区域开始（其信任锚硬编码在解析器中），每个区域通过委托签名者（DS）记录将信任委托给子区域。父区域中的 DS 记录包含子区域中公钥的加密哈希值。当解析器验证 example.de 时，它会验证信任链：根信任 .de，.de 信任 example.de。该链中任何一环的中断都会导致其下所有内容的验证失败，这就是为什么像 .de 这样的 TLD 配置错误会影响其下的每个域名。

区域通常使用两种类型的密钥：区域签名密钥（ZSK），用于签署区域的记录；以及密钥签名密钥（KSK），用于签署 ZSK 本身。KSK 的公钥是父区域 DS 记录指向的对象，从而锚定信任链。轮换 ZSK 相对简单：生成一个新密钥，重新签署区域记录，然后等待缓存过期。轮换 KSK 则更为复杂，因为父区域的 DS 记录也必须更新，通常需要与注册商或注册管理机构协调。

在密钥轮换期间，存在一个关键窗口期，旧密钥正在被淘汰，新密钥正在被引入。如果区域中发布的签名使用的是解析器无法根据区域发布的 DNSKEY 记录验证的密钥——无论是因为签名步骤失败、时间错误，还是新密钥尚未完全分发——解析器别无选择，只能拒绝响应并返回 SERVFAIL。

还可以看到查询量大幅增加。这在 DNS 事件期间很典型，因为客户端会重试失败的查询（通常三次或更多次），从而增加了原始数字。SERVFAIL 比率看起来比实际用户影响更令人担忧，因为其中许多查询代表同一用户重试同一域名。

可能令人惊讶的是，在整个事件期间，NOERROR 率保持相对稳定。这是“提供过期缓存”在起作用，我们将在下一节中介绍。

## 提供过期缓存

递归解析器会在每条记录的 TTL（生存时间）期间缓存从权威名称服务器收到的记录。当记录被缓存时，解析器会直接提供该记录，而无需返回权威名称服务器。当 TTL 过期时，解析器会获取新副本并重新缓存。

在中断期间，新请求的记录最终解析为 SERVFAIL。DNSSEC 签名已损坏，解析器正确地拒绝了它们。但许多 .de 记录在事件开始前仍存在于缓存中。1.1.1.1 没有立即丢弃这些记录并向用户返回 SERVFAIL，而是在它们 TTL 过期后继续提供这些记录。这被称为“提供过期缓存”。

1.1.1.1 实现了 RFC 8767，该规范正式定义了这种行为。当上游解析失败时，解析器可以继续提供过期的缓存记录，而不是返回错误。这大大缓冲了上游中断的影响，为运营商争取了响应时间。

结果如下图所示，该图显示了事件期间 .de 查询的响应码，排除了过期缓存提供的响应。如果没有过期缓存提供的响应，NOERROR 率从世界协调时19:30开始稳步下降。这些代表用户之所以获得良好答案，仅仅是因为他们的记录仍在缓存中。

## 我们的缓解措施

虽然这个问题在很大程度上超出了我们自身的控制范围，并且提供过期缓存正在发挥作用，但对许多用户来说仍然存在实际影响。幸运的是，我们能够采取一些措施来改善情况。

### 负信任锚

RFC 7646 定义了负信任锚（NTA）的概念。在正常的 DNSSEC 操作中，验证型解析器维护一组信任锚：信任链根部的公钥。每个使用 DNSSEC 签名的 DNS 区域都有一个信任锚，每个子区域都在此基础上构建自己的信任锚。当连接信任链的加密签名损坏时，响应将被拒绝并导致 SERVFAIL。NTA 是一种显式例外。它告诉解析器将特定区域视为未签名，从而绕过该区域下名称的验证。

NTA 正是为此类事件而设计的。当 TLD 运营商发布损坏的签名时，每个 DNSSEC 验证型解析器都不得不为该 TLD 下的每个域名返回 SERVFAIL。这不是因为这些域名本身有任何问题，而是因为它们的父区域配置错误。在这种情况下继续返回 SERVFAIL 不提供任何安全价值：失败已经是已知的、公开的，并且正在修复中。RFC 7646 明确将 TLD 配置错误列为 NTA 的主要用例。

### 我们实际部署的措施

对于 1.1.1.1，我们拥有自己的解析器，称为 Big Pineapple，它也为 1.1.1.1 for Families、Gateway DNS、DNS Firewall 等提供支持。目前，我们尚未实现原生的 NTA 机制。相反，我们使用现有的覆盖规则机制将 .de 标记为不安全区域，这会导致所有 .de 查询被解析为好像没有启用 DNSSEC 一样。这在功能上等同于 NTA，尽管它未在任何 RFC 中正式定义。

绕过DNSSEC是一个刻意的权衡。在没有DNSSEC验证的情况下，.de域名在事件期间容易遭受真正的攻击。在此类事件中，我们认为这是可以接受的，因为签名失败是广泛存在的、已公开确认的，并且平等地影响了互联网上所有执行验证的解析器。正如我们在内部事件响应室中所说：“当前，任何使用1.1.1.1解析.de域名的用户，都不会更愿意收到SERVFAIL响应，而不是一个未经验证的响应。”

我们在UTC时间22:17部署了缓解措施，这标志着1.1.1.1所受影响的结束。我们通过DNS-OARC Mattermost与其他DNS运营商进行了沟通。

### 源站解析缓解措施

虽然所有互联网用户都可以访问我们的1.1.1.1解析器，但我们对于使用我们CDN平台服务的客户负有特殊责任。那些使用.de源站域名的客户也受到了此次故障的影响。

Cloudflare运营着一个独立的内部解析器用于源站解析，与我们公开的1.1.1.1服务不同。为了减轻影响，我们在内部解析器服务上对.de应用了类似的NTA（负面信任锚），恢复了受影响客户的源站连接。

### 扩展DNS错误

在我们的缓解措施实施之前，无法从缓存提供服务的查询会从1.1.1.1收到SERVFAIL响应。每个SERVFAIL都包含一个扩展DNS错误（EDE）代码，定义于RFC 8914，该代码向客户端提供有关错误原因的更多详细信息。

一些解析器返回了EDE 6（DNSSEC无效），并附带一条直接指向损坏签名的描述性消息。这是正确的行为：

```unset
EDE: 6 (DNSSEC Bogus): RRSIG with malformed signature found for example.de/nsec3 (keytag=33834)

```

另一方面，1.1.1.1返回了EDE 22（无可达权威服务器），这表面上表明上游名称服务器存在连接问题，而非DNSSEC验证失败。

原因在于我们如何将DNSSEC EDE代码从信任链验证器向上传播的过程中存在一个错误。当验证器检测到无效签名时，它会创建DNSSEC无效EDE代码，但该代码从未被插入到响应中。相反，解析器的外层发现递归解析存在问题且没有错误代码，于是回退报告“无可达权威服务器”。这掩盖了底层的DNSSEC原因。

我们意识到这对1.1.1.1用户没有帮助，并将修复我们的响应，以暴露DNSSEC错误。

## 这是否意味着DNSSEC作为一项技术失败了？

DNS是大多数互联网通信请求链中的关键部分。人们很容易得出结论，认为此次故障及其采取的缓解措施意味着DNSSEC作为一项技术已经失败。然而，任何配置错误的技术都可能给依赖它的用户带来风险。将关键光纤电缆暴露在海床上供鲨鱼啃咬，并不会否定海底电缆在当今互联网通信中的重要作用。它只是凸显了我们有时未能准确地保护它。这里也是如此。DNSSEC在确保我们能够信赖未经恶意行为者篡改的DNS应答方面发挥着关键作用。

## #HugOps

没有人喜欢发生严重事件。不幸的是，这些事情会发生在每一个大规模运营关键基础设施的人身上。当它们发生时，DNS社区往往会互相支持。

此类事件也凸显了运营商之间关系的重要性。DNS是一个去中心化的系统，没有任何单一组织能够完全控制它，而保持其可靠运行依赖于注册机构、解析器运营商和更广泛社区之间的相互信任和开放的沟通渠道。像DNS-OARC这样的论坛正是提供了这一点：共享的邮件列表和聊天室，当出现问题时，运营商可以跨组织边界快速协调。

DENIC已经发布了一篇关于此事件的简短博客文章，其中指出：“此次故障与一次例行的、计划中的密钥轮换有关。在此过程中，生成了无法验证的签名并进行了分发。作为预防措施，在确切的技术原因被查明之前，未来的密钥轮换已被暂停。”

我们相信，当他们自己的分析完成时，我们会听到更多消息。

## 从此次事件中吸取的教训

此次事件凸显了DNS层级结构的一个结构性现实：当TLD级别的注册机构出现故障时，该TLD下的每个域名都会同时受到影响，无论其托管在何处或使用哪个解析器。这并非DNSSEC独有；如果TLD的名称服务器变得不可达，情况也是如此。使全球DNS得以运行的层级结构，也正是导致顶层故障向下传播的原因。

对此没有简单的解决方案。行业能做的就是当这种情况发生时，快速且一致地做出响应。在此次事件中，互联网上的解析器运营商在一小时内独立应用了负面信任锚，恢复了域名解析，同时DENIC致力于修复该区域。运营实践、像DNS-OARC这样的行业沟通渠道，以及“提供过期数据”等功能，都减轻了影响，即使它们无法消除底层的依赖关系。

我们还总结了一些自身需要改进的地方。我们将致力于改进我们的EDE错误，以更好地暴露DNSSEC错误。

我们期待DENIC的事件后报告，并感谢他们自始至终展现的透明度。
