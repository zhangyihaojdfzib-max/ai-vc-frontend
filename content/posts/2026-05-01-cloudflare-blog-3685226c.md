# 介绍动态工作流：跟随租户的持久化执行

2026-05-01

- Dan Lapid
- Luís Duarte

![](/images/posts/30e914362556.png)

八年前我们首次推出 Workers 时，它是一个面向开发者的直接平台。多年来，我们扩展和规模化了这个生态系统，使得平台不仅可以直接在 Workers 上构建，还能通过众多多租户应用让他们的客户向我们提交代码。现在我们看到 Workers 上运行着：用户描述需求、AI 编写实现的应用；每个客户的业务逻辑在运行时都是平台从未见过的 TypeScript 代码的多租户 SaaS；编写并运行自己工具的 Agent（智能体）；每个仓库定义自己管道的 CI/CD 产品。

上个月，当我们发布 Dynamic Workers 公开测试版时，我们为这些平台提供了一个干净的计算原语：在运行时向 Workers 运行时提交一些代码，在个位数毫秒内，在同一台机器上获得一个隔离的、沙箱化的 Worker。Durable Object Facets 将同样的理念扩展到存储——每个动态加载的应用可以拥有自己的 SQLite 数据库，按需启动，平台作为监管者位于前端。Artifacts 对源代码控制做了同样的事情：一个 Git 原生、版本化的文件系统，可以创建数千万个，每个 Agent（智能体）一个、每个会话一个、每个租户一个。至此，我们有了存储和源代码控制的动态部署。接下来是什么？

今天，我们通过动态工作流将持久化执行与动态部署连接起来。

## 持久化执行与动态执行之间的差距

Cloudflare Workflows 是我们的持久化执行引擎。它将 `run(event, step)` 函数转变为一个程序，其中每个步骤都能在故障中存活，可以休眠数小时或数天，可以等待外部事件，并在隔离区被回收时从上次中断的地方精确恢复。它是任何需要"持续进行"超越单个请求的场景的正确原语：入职流程、视频转码管道、多阶段计费、长时间运行的 Agent（智能体）循环，以及——自 Workflows V2 起——每个账户最多 50,000 个并发实例和每秒 300 个新实例，专为 Agent（智能体）时代重新设计。

但 Workflows 一直有一个内置假设：工作流代码是你的部署的一部分。你的 `wrangler.jsonc` 中有一个块写着"当引擎调用 WORKFLOWS 时，运行名为 MyWorkflow 的类。"一个绑定，一个类。每次部署如此。

如果你拥有所有代码，这没问题。如果你运行的是传统应用，这也没问题。

当你希望让客户提交他们自己的工作流时，这就不再适用了。

假设你正在构建一个应用平台，AI 为每个租户编写 TypeScript。假设你正在运行一个 CI/CD 产品，每个仓库都有自己的管道。假设你正在使用一个 Agent（智能体）SDK，每个 Agent（智能体）编写自己的持久化计划。在所有这些情况下，每个租户、每个 Agent（智能体）、每个请求的工作流都不同。没有单一的类可以绑定。

这与 Dynamic Workers 为计算解决的问题、Durable Object Facets 为存储解决的问题是同一类问题。我们只是还没有为持久化执行解决它。

## 动态工作流

`@cloudflare/dynamic-workflows` 是一个小型库。大约 300 行 TypeScript。它让单个 Worker——Worker Loader——将每个 `create()` 调用路由到不同租户的代码，并且关键的是，让 Workflows 引擎在工作流实际执行时——几秒、几小时或几天后——将 `run(event, step)` 调度回同一段代码。

以下是整个模式。一个 Worker Loader：

```TypeScript
import {
  createDynamicWorkflowEntrypoint,
  DynamicWorkflowBinding,
  wrapWorkflowBinding,
} from '@cloudflare/dynamic-workflows';

// 库在 cloudflare:workers 导出中查找此类。
export { DynamicWorkflowBinding };

function loadTenant(env, tenantId) {
  return env.LOADER.get(tenantId, async () => ({
    compatibilityDate: '2026-01-01',
    mainModule: 'index.js',
    modules: { 'index.js': await fetchTenantCode(tenantId) },
    // 租户将此视为普通的 Workflow 绑定。
    env: { WORKFLOWS: wrapWorkflowBinding({ tenantId }) },
  }));
}

// 在 wrangler.jsonc 中注册为 class_name。
export const DynamicWorkflow = createDynamicWorkflowEntrypoint<Env>(
  async ({ env, metadata }) => {
    const stub = loadTenant(env, metadata.tenantId);
    return stub.getEntrypoint('TenantWorkflow');
  }
);

export default {
  fetch(request, env) {
    const tenantId = request.headers.get('x-tenant-id');
    return loadTenant(env, tenantId).getEntrypoint().fetch(request);
  },
};
```

添加到你的 `wrangler.jsonc`：

```JSON
"workflows": [
		{
			"name": "dynamic-workflow",
			"binding": "WORKFLOW",
			"class_name": "DynamicWorkflow"
		}
	]
```

租户编写普通的、惯用的 Workflows 代码。他们不知道自己在被调度：

```TypeScript
import { WorkflowEntrypoint } from 'cloudflare:workers';

export class TenantWorkflow extends WorkflowEntrypoint {
  async run(event, step) {
    return step.do('greet', async () => `Hello, ${event.payload.name}!`);
  }
}

export default {
  async fetch(request, env) {
    const instance = await env.WORKFLOWS.create({ params: await request.json() });
    return Response.json({ id: await instance.id });
  },
};
```

就是这样。租户调用 `env.WORKFLOWS.create(...)`，看起来就像一个完全正常的 Workflow 绑定。Workflow ID、`.status()`、`.pause()`、重试、休眠、持久化步骤、`step.sleep('24 hours')`、`step.waitForEvent()`——一切照常工作。

这个库处理一件事：确保当 Workflows 引擎最终唤醒并调用 `run(event, step)` 时，它最终进入正确租户的代码中。

## 工作原理

三层结构：顶部的 Workflows 引擎（平台），中间的 Worker Loader，底部的租户代码（一个 Dynamic Worker）。

当请求到达 Worker Loader 时，它动态地将执行路由到正确的动态代码。其余的执行是这三层之间的交接，按时间从左到右：请求进入，向上传递到引擎，被持久化，稍后再次向下传递。

流程详解：

① → ② 进入租户代码。Worker Loader 收到一个 HTTP 请求，确定它属于哪个租户，通过 Worker Loader 加载该租户的代码，并将请求转发到其 `default.fetch`。它交给租户的 `env` 包含 `WORKFLOWS: wrapWorkflowBinding({ tenantId })`。对租户而言，这看起来和表现得就像一个真正的 Workflow 绑定。

③ 向上到 Worker Loader。当租户调用 `env.WORKFLOWS.create({ params })` 时，它实际上是在向 Worker Loader 发起远程过程调用（RPC）——被包装的绑定是一个 `WorkerEntrypoint` 子类（`DynamicWorkflowBinding`），运行时在加载时用租户的元数据对其进行了特化。这就是为什么你必须从 Worker Loader 中 `export { DynamicWorkflowBinding }`：运行时通过在 `cloudflare:workers` 导出中查找该类来构建每个租户的存根。跨越 Dynamic Worker 边界的绑定必须是 RPC 存根——一个普通的 `{ create, get }` 对象无法进行结构化克隆，原始的 Workflow 绑定也无法序列化。

在 Worker Loader 内部，被包装的绑定透明地重写负载：

```Rust
租户调用：  create({ params: { name: 'Alice' } })
                            │
                            ▼
引擎看到：   create({ params: {
                  __workerLoaderMetadata: { tenantId: 't-42' },
                  params: { name: 'Alice' }
               }})

```

③ 交给引擎。Worker Loader 随后在真实的 `WORKFLOWS` 绑定上调用 `.create()`，并将信封作为参数传入。至此，Workflows 引擎接管。它持久化 `event.payload`（现在包含信封），并调度运行。每当引擎后续唤醒工作流时（无论是经过 24 小时休眠、崩溃还是部署后），元数据都会随 payload 一起传递，等待路由本次运行。

一个推论：将元数据视为路由提示，而非授权信息。租户可以通过 `instance.status()` 读取它。不要在元数据中存放机密信息。

④ → ⑤ 引擎返回。当引擎准备运行一个步骤时，它会在你在 `wrangler.jsonc` 中注册的类上调用 `.run(event, step)`——也就是 `createDynamicWorkflowEntrypoint` 返回的那个类。该类解包信封，将元数据交给你编写的 `loadRunner` 回调，并将解包后的事件转发给该回调返回的任何 runner。

回调是发生所有有趣事情的地方，并且完全由你掌控。从 R2 获取租户的最新源码。检查他们的套餐等级并选择一个区域。附加一个 tail Worker 用于每个租户的日志记录。使用 `@cloudflare/worker-bundler` 即时打包 TypeScript。在常见情况下，你只需将任务交给 Worker Loader：

```TypeScript
const stub = env.LOADER.get(tenantId, () => loadTenantCode(tenantId));
return stub.getEntrypoint('TenantWorkflow');
```

Worker Loader 按 ID 缓存，因此一个运行多个步骤、持续数小时的工作流会在这些步骤之间复用同一个动态 Worker。当隔离区最终被驱逐时，下一个 `step.do()` 会再次拉取代码并继续运行——租户的工作流对此毫无感知。一个动态 Worker 在个位数毫秒内启动，仅消耗几兆字节内存，因此调度开销基本为零。你可以拥有百万级租户，每个租户都有自己独特的工作流代码，按需在步骤边界上懒加载，空闲时零成本。

### 逃生舱

如果你想自己继承 `WorkflowEntrypoint`——例如在 `run()` 周围添加日志记录、接入每个租户的可观测性，或通过线程传递自定义状态——该库暴露了 `createDynamicWorkflowEntrypoint` 所基于的低层级 `dispatchWorkflow` 原语：

```TypeScript
import { dispatchWorkflow } from '@cloudflare/dynamic-workflows';

export class MyDynamicWorkflow extends WorkflowEntrypoint {
  async run(event, step) {
    return dispatchWorkflow(
      { env: this.env, ctx: this.ctx },
      event,
      step,
      ({ metadata, env }) => loadRunnerForTenant(env, metadata),
    );
  }
}
```

其他所有功能——ID、暂停/恢复、`sendEvent`、重试——都原封不动地交由真实的 Workflows 引擎处理。

## 动态 Worker 是原语

暂且退一步看。这个库中每一行有趣的代码，要么是对出站侧 `.create()` 的封装，要么是对入站侧 `WorkflowEntrypoint` 的封装。实际的工作——启动租户代码、沙箱化、跨边界路由 RPC、缓存隔离区、在步骤之间休眠——全部由底层的动态 Worker 完成。

这才是真正的故事，而且它远比 Workflows 宏大。

动态 Worker 是吞噬一切的原语。`Durable Object Facets` 是将同一模式应用于 Durable Objects。动态 Workflows 是将同一模式应用于 `WorkflowEntrypoint`。每一个都是在你一直拥有的静态绑定和你现在可以交给客户的动态版本之间，少量“信封与解包”的粘合剂。

我们不会止步于 Workflows。Workers 当前暴露的每一个绑定都将拥有动态对应物——队列（每个生产者自带处理器）、缓存、数据库、对象存储、AI 绑定，以及 MCP 服务器（每个租户自带工具）。你今天绑定到 Worker 的任何东西，很快都将能够动态绑定：按租户、按 Agent、按请求分发，空闲时零成本。

坦率地说，运行这样一个平台的单位经济模型是荒谬的。交付多租户产品过去意味着给每个客户提供自己的容器、自己的数据库、自己的磁盘、自己的调度器，然后用编排胶水、服务网格和令人抓狂的计费数学将它们拼接在一起。许多这类应用至少需要支持数千个客户；最多则达数百万。而在动态 Worker 及其之上构建的一切之上，空闲租户几乎不花任何成本，活跃租户通过隔离级多租户共享同一硬件。成本底线下降了数个数量级。一个过去最多只能容纳数千付费客户的平台，现在可以合理地服务数千万客户。

## 这解锁了什么

### 像工程师一样规划的 Agent 平台

编码 Agent——OpenCode、Claude Code、Codex、Pi——在过去一年中已经证明，LLM 在编写代码方面远优于进行顺序工具调用。Cloudflare Agents SDK 和 Project Think 将该洞察扩展到持久化执行：借助 fibers 和子 Agent 等原语，Agent 的长期规划可以在崩溃、休眠和重新部署后持续存在，而用户毫无察觉。

动态 Workflows 是让该规划成为一等公民 Cloudflare Workflow 的拼图——Agent 实际编写、平台实际运行，并拥有完整的持久化机制。一个模型一分钟前编写的 `run(event, step)` 函数，其中每个 `step.do(...)` 可独立重试，每个 `step.sleep('24 hours')` 免费休眠，每个 `step.waitForEvent(...)` 无限等待人类批准下一步操作。Agent 编写工作流；平台运行它；两者都不需要预先知道规划是什么样子。

### 用户自带逻辑的 SDK 和框架

如果你正在交付一个框架，让客户编写 `run(event, step)` 函数——工作流构建器 UI、可视化自动化工具、每租户扩展系统、面向非开发者的低代码工具——动态 Workflows 现在就是那个让它无妥协运行的原语。你调用一次 `wrapWorkflowBinding({ tenantId })`，将结果作为 `WORKFLOWS` 交给他们的代码，他们创建的每个工作流实例都会自动打标签、路由回并在他们的沙箱中执行。框架拥有 Worker Loader；用户拥有工作流；两者互不关心。

### 原语速度的 CI/CD

以下是我们最兴奋的用例。

每个现有的 CI/CD 平台，本质上都是按仓库配置文件的调度器：“按此顺序运行这些步骤，使用这些密钥，缓存这些目录，上传这些工件。”每个仓库都有自己的流水线。每个分支可能有自己的变体。每个拉取请求都会生成该流水线的一个实例，该实例必须运行完成、在机器崩溃后存活、重试不稳定的步骤、流式传输日志、暂停等待审批，并持久化结果。

这正是持久化工作流的形态。CI 之前没有以这种方式构建的原因是，没有人拥有一个云原语，让工作流本身对每个仓库都不同，在运行时调度，且零配置成本。现在你有了。

以下是 CI 流水线看起来的样子——它只是客户随仓库提供的代码——例如，在 `.cloudflare/ci.ts` 中。工作流本身是真实的；下面的 `runInSandbox()` / `summarise()` / GitHub 绑定辅助函数是平台提供的粘合剂，是你会在调度器中一次性交付的那种东西：

```TypeScript
import { WorkflowEntrypoint } from 'cloudflare:workers';

export class CIPipeline extends WorkflowEntrypoint {
  async run(event, step) {
    const { repo, sha, branch, pr } = event.payload;

    // 在此提交处 fork 仓库的隔离副本。秒级，而非分钟级。
    const workspace = await step.do('checkout', () =>
      this.env.ARTIFACTS.fork(repo, { sha })
    );
```

```javascript
await step.do('install', () => runInSandbox(workspace, ['pnpm', 'install']));

    // 每个并行步骤均可独立重试。
    const [lint, test, build] = await Promise.all([
      step.do('lint',  () => runInSandbox(workspace, ['pnpm', 'lint'])),
      step.do('test',  () => runInSandbox(workspace, ['pnpm', 'test'])),
      step.do('build', () => runInSandbox(workspace, ['pnpm', 'build'])),
    ]);

    if (pr) {
      await step.do('comment', () =>
        this.env.GITHUB.commentOnPR(repo, pr, summarise({ lint, test, build }))
      );
    }

    // 工作流将休眠，直到收到审批。不会占用虚拟机资源。
    if (branch === 'main') {
      await step.waitForEvent('approval', { type: 'deploy-approval', timeout: '24 hours' });
      await step.do('deploy', () => runInSandbox(workspace, ['pnpm', 'deploy']));
    }
  }
}
```

平台拥有调度器。它接收一个 webhook，识别其来源仓库，将该仓库的 `CIPipeline` 类作为动态 Worker 加载，并将执行过程交给动态工作流。平台无需了解管道中的具体内容，也不需要了解。它只是运行一个恰好位于客户仓库中的持久化函数。

现在，我们来梳理每个步骤的实际作用：

- **Artifacts** 为每个仓库提供一个基于 Git、版本化的文件系统，该系统位于 Cloudflare 的全球分布式网络上。**ArtifactFS** 惰性地加载文件树，因此即使是数 GB 的仓库也能在几秒内准备就绪——而 `fork()` 为每次 CI 运行提供独立的隔离副本，无需 `git clone` 的开销。
- **动态 Worker** 在沙箱隔离环境中运行每个轻量级步骤（lint、格式化、类型检查、打包），该环境在毫秒内启动，并与仓库数据位于同一台机器上。无需配置虚拟机、拉取镜像或冷启动。
- **动态工作流** 将整个运行过程整合在一起。步骤可重试且持久化。运行在等待审批时免费休眠。状态和进度在部署、驱逐和崩溃后依然保留。
- **沙箱** 处理繁重的任务——需要 `docker build` 的步骤、需要运行 Postgres 的集成测试套件、需要 8 核的 Rust 编译。快照存储到 R2 意味着即使是这些任务也能在几秒内热启动。

**Artifacts** 为每个仓库提供一个基于 Git、版本化的文件系统，该系统位于 Cloudflare 的全球分布式网络上。**ArtifactFS** 惰性地加载文件树，因此即使是数 GB 的仓库也能在几秒内准备就绪——而 `fork()` 为每次 CI 运行提供独立的隔离副本，无需 `git clone` 的开销。

**动态 Worker** 在沙箱隔离环境中运行每个轻量级步骤（lint、格式化、类型检查、打包），该环境在毫秒内启动，并与仓库数据位于同一台机器上。无需配置虚拟机、拉取镜像或冷启动。

**动态工作流** 将整个运行过程整合在一起。步骤可重试且持久化。运行在等待审批时免费休眠。状态和进度在部署、驱逐和崩溃后依然保留。

**沙箱** 处理繁重的任务——需要 `docker build` 的步骤、需要运行 Postgres 的集成测试套件、需要 8 核的 Rust 编译。快照存储到 R2 意味着即使是这些任务也能在几秒内热启动。

对于一个中等规模的 JS 仓库，传统的 CI 运行流程大致如下：分配虚拟机（15-30 秒）→ 拉取基础镜像（10 秒）→ `git clone`（10 秒）→ `npm ci`（30-60 秒）→ 运行测试（实际工作）→ 销毁。在第一个测试运行之前需要几分钟的准备时间，并且整个过程都需要为整个虚拟机付费。

同样的管道在此技术栈上的运行流程如下：边缘 fork 仓库（秒级）→ 每个步骤在毫秒内启动一个全新的隔离环境或从快照恢复的沙箱 → 运行实际工作 → 休眠。无需冷启动，无需提前预置资源，无需保持热状态。仓库无需移动——计算资源会主动靠近它。

CI 从未如此之快，原因在于这些基础组件从未在一个地方同时存在过。而现在，它们做到了。
