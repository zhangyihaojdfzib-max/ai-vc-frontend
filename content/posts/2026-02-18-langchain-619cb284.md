许多团队将评估视为最后一步的检查，但我们将其作为第0天的必备要求。

monday Service是一个AI原生的企业服务管理平台，旨在自动化处理并解决所有服务部门的咨询。在构建我们全新的**AI服务团队**（一个由可定制、基于角色的AI Agent组成的团队，旨在减轻人工客服的工单负担）时，我们从一开始就将评估嵌入到开发周期中，而不是等待Alpha用户来发现问题。

本文将向您展示我们如何构建一个**评估驱动的开发框架**，以便在我们的用户之前发现AI质量问题。

我们取得的成果：

*   **速度**：评估反馈循环速度提升**8.7倍**（从162秒缩短至18秒）。
*   **覆盖率**：在几分钟内完成数百个示例的全面测试，而非数小时。
*   **Agent可观测性**：使用多轮评估器，对生产环境中的追踪数据进行实时、端到端的质量监控。
*   **评估即代码**：评估逻辑作为受版本控制的生产代码进行管理，采用GitOps风格的CI/CD部署。

AI服务团队是一个可定制的、基于**LangGraph**的**ReAct Agent**，旨在自动化处理并解决任何企业服务管理用例中的咨询。

无论是应用于**IT、HR还是法律**等领域，monday Service的客户都可以利用他们自己的**知识库文章**和**工具**，定制Agent以驱动任何**服务**部门内的执行。

![](/images/posts/8acd7a4c1ae6.png)

然而，正是这种赋予ReAct Agent强大能力的**自主性**，也带来了一个独特的挑战：因为推理链的每一步都依赖于上一步，提示词或工具调用结果中的**微小偏差**可能会**级联**放大，导致显著不同——甚至可能是错误的——结果。

### 评估的两大支柱

通过对Agent评估最佳实践的研究，我们很快意识到需要采用双层方法：

**离线评估 —— "安全网"**：其作用类似于单元测试层，针对精心整理的"黄金数据集"运行Agent。测试内容包括**核心逻辑**（例如，事实依据、检索准确性、工具调用）和**特定的边缘情况**（例如，知识库文章冲突或优先级解决）。这一层有助于确保简单的提示词调整不会**无意中破坏**Agent处理**其他**任务的能力。

**在线评估 —— "监控器"（持续质量监控）**：这一层从端到端的**业务**角度，处理对Agent**性能**的持续收集、分析和提升。通过利用**在线**评估流水线，我们**追踪**并**优化业务**指标（例如，自动解决率和问题控制率），**实时**确保Agent在真实环境中的表现。

### 支柱A：离线评估 —— "安全网"

#### 设计我们的评估覆盖策略

在编写任何评估之前，我们需要回答一个基本问题：我们究竟应该评估什么？挑战**并非**设计一个完美的覆盖策略，而仅仅是选择一个**实用的起点**。

我们构建了一个包含约30个真实（已脱敏）已解决IT工单的小型数据集，这些工单选自我们内部的IT服务台，涵盖了常见的请求类别，例如：

*   访问与身份（例如，身份提供商、单点登录、软件访问）
*   VPN和连接问题
*   设备/操作系统支持（更新、性能、硬件问题）

在第一个测试套件中，我们的检查有意设计得**很简单**：

*   **确定性"冒烟"测试**：
    *   **运行时健康度**：Agent运行无崩溃/超时，请求端到端成功。
    *   **输出格式**：响应符合预期的模式/格式（甚至在判断内容之前）。
    *   **状态与持久化**：线程/会话已创建，对话已正确持久化到我们的应用数据库中。
    *   **基础工具健全性检查**：所有必要的工具都以适当的输入被正确调用，并完成执行且无错误。
*   **LLM即评判员**：我们从一个现成的评估器开始，使用**OpenEvals（正确性）** 来比较Agent的响应与同一已解决工单数据集中的参考输出。

一旦这个基线建立起来，我们就用更小的、**特定于用例的**数据集进行扩展，以探测**特定的行为**——包括会话记忆、知识库检索、事实依据与冲突解决，以及防护栏。随着这些行为变得更加微妙，我们从**一个正确性分数**转向一组更全面的检查：

*   **知识库事实依据 / 引用**："每个事实性陈述是否都能追溯到提供的知识库内容？"（我们使用LangSmith预建的**幻觉/答案相关性**检查来验证这一点）。
*   **冲突处理**："当政策因地区/时间而异时，Agent是否要求澄清或选择了最新的适用政策？"（或使用预建的**正确性**检查）。
*   **防护栏**："Agent在需要时是否拒绝？" / "它是否避免了透露内部工具名称或提示词内容？"（或使用预建的**毒性/简洁性**检查）。
*   **知识库使用时机**：知识库应在合理的时间点获取（不要太早，也不要在答案已经形成之后），使用**AgentEvals的轨迹LLM即评判员**进行检查。
*   **防护栏执行顺序**：安全/政策防护栏应在正确的阶段运行（在生成最终答案之前）。这是另一个**轨迹**检查。

#### 框架：langsmith/vitest

为了实现这一层，我们利用了**LangSmith Vitest**集成。这种方法提供了一个久经考验的测试框架（Vitest）的强大功能，同时与**LangSmith生态系统**保持无缝集成。

通过这种设置，每次CI运行都会**自动**作为一次独立的实验记录在**LangSmith平台**中，每个测试套件都充当一个**数据集**。这使我们能够深入了解特定的**运行**，并确切地看到Agent在何处偏离了**事实依据**，从而在代码变更**到达生产环境之前**，轻松验证其影响。

![](/images/posts/de0d6a9da865.png)

#### 深刻的教训：不要在开发者体验上妥协

起初，我们的离线评估是串行运行的。标准的开发循环——评估（失败）→ 修复 → 重新评估（通过）——成为了一个主要的瓶颈。

我们发现，缓慢的反馈循环不可避免地会损害我们的测试深度或开发速度。为了在不出现回归的情况下维持高速交付，我们意识到评估过程必须足够快，以确保一个顺畅的迭代循环。

#### 解决方案：使用 Vitest + ls.describe.concurrent 进行并行化

![](/images/posts/ef38df48a220.png)

通过优化我们的Vitest和LangSmith集成，我们通过将负载分配到**本地工作进程和远程API调用**上，实现了速度的大幅提升。关键在于采用混合方法：**并行化测试文件**以最大化CPU使用率，并**并发运行LLM评估**以处理I/O密集型延迟。

- **并行处理（CPU密集型）**：我们利用 Vitest 的 `pool: 'forks'` 将工作负载分配到多个核心。通过将每个数据集分片分配给独立的测试文件，允许多个工作进程并行运行，而无需竞争 CPU 资源。这种设置确保即使数据集增长，我们也能通过将分片分配到可用核心上来快速处理它们。
- **并发处理（I/O密集型）**：在每个测试文件内，我们使用 `ls.describe.concurrent` 来最大化吞吐量。由于 **LLM（大语言模型）** 评估具有高延迟性，并发处理允许我们通过同时发起数十个评估来**重叠延迟时间**，确保运行器永远不会闲置。
- **评估函数**：这是负责评估每个示例的核心逻辑。我们用它来单次运行双层验证：
    - **确定性基线**：通过硬性断言确保 **Agent（智能体）** 遵守**响应模式**并保持**状态持久性**（通过检查点/存储）。
    - **LLM 作为评判者**：针对“黄金数据集”进行语义评分。我们利用 **OpenEvals** 和 **AgentEvals** 等开源库来对**正确性**和**事实依据性**等维度进行评分。

**结果**：在**数分钟**内完成对**数百个**示例的全面反馈！

使用 MacBook Pro 16 英寸（2023 年 11 月，Apple M3 Pro，36 GB RAM，macOS Tahoe 26.2）对 20 个脱敏 IT 工单的**基准测试**结果：

| 执行模式 | 总耗时 | 相对于顺序执行的加速比 |
| :--- | :--- | :--- |
| 并行 + 并发 | 18.60 秒 | 快 8.7 倍 |
| 仅并发 | 39.30 秒 | 快 4.1 倍 |
| 顺序执行 | 162.35 秒 | 基线 |

### 支柱 B：在线评估 —— “监控器”

#### 在线、多轮次评估

虽然离线评估常用于在受控沙箱中捕捉回归问题，但它们本质上是**合成或脱敏示例**的静态快照。为了捕捉生产环境的**不可预测性**，我们需要**在线评估**——在真实的生产环境追踪数据上**实时**运行。

由于我们的 Agent 处理复杂的**多轮次**对话，成功通常不由单次响应定义，而是由**整个对话**轨迹决定。这需要一种评估策略，能够考量 Agent 如何在**多个轮次**中引导用户解决问题。

我们发现 **LangSmith 的多轮次评估器**非常契合，它利用 **LLM 作为评判者**来对端到端的对话线程进行评分。我们现在可以使用自定义**提示词**来评估整个对话轨迹，衡量诸如**用户满意度**、**语气**和**目标达成**等高层次结果，而不再是孤立地评估单个运行。

![](/images/posts/eb6ab5228565.png)

最令人印象深刻的是我们能够**快速**上线。**LangSmith 平台**使得多轮次设置**极其直观**：我们可以定义**自定义非活动窗口**来精确确定会话何时应被视为“完成”并准备好进行评估，并且可以轻松应用**采样率**来平衡数据量与 LLM 成本。

![](/images/posts/4f19876814f2.png)

#### 评估即代码

随着我们从原型阶段进入生产阶段，我们希望以管理其他生产代码相同的标准来管理我们的“评判者”：**版本控制**、**同行评审**和**自动化 CI/CD 流水线**。

为了实现这一点，我们将事实来源移入我们的代码仓库，将“评判者”定义为结构化的 TypeScript 对象。

```typescript
// conversation-analysis.ts
export const conversationAnalysis = new MultiSignalEvaluationPrompt({
  name: 'conversation-analysis',
  variables: ['all_messages'],
  modelConfig: { model: 'gpt-5.2-pro', reasoning: { effort: 'high' } },
  extractionFields: [
    new ExtractionField({ key: 'human_handoff', type: 'boolean', includeComment: true }),
    new ExtractionField({ key: 'meaningful_interaction', type: 'boolean', includeComment: true }),
    new ExtractionField({ key: 'is_automated_resolution', type: 'boolean', includeComment: true }),
    // ... additional atomic signals
  ],
  systemPrompt: `You are an expert conversation analyst...`,
  humanPrompt: `Analyze the following conversation:
<conversation>
{{{all_messages}}}
</conversation>`,
});
```

将我们的评判者移入代码解锁了两项关键能力：

1.  我们可以利用 **Cursor** 和 **Claude Code** 等 AI IDE 在我们的主要工作区内直接优化复杂的提示词。
2.  为我们的评判者编写离线评估以确保其在接触生产流量之前的准确性变得非常自然。

得益于 LangChain 的 IDE 集成，迁移相对容易。我们使用 **Documentation MCP** 将库上下文拉入编辑器，使用 **LangSmith MCP** 直接获取运行和反馈。**LangChain Chat** 对于澄清具体的实现细节也是一个有用的参考。

#### 部署

为了形成闭环，我们构建了一个自定义 CLI 命令 `yarn eval deploy`，它在我们的 CI/CD 流水线中运行。这确保了我们的代码仓库始终是我们评估基础设施的绝对**事实来源**。

![](/images/posts/29aea917236f.png)

当我们合并一个 PR 时，我们的同步引擎会执行一个三步**协调循环**：

1.  **同步提示词**：将 TypeScript 定义推送到 LangSmith 提示词注册表。
2.  **协调规则**：将本地评估（规则）定义与活跃的生产定义进行比较，**自动**更新或创建它们。
3.  **清理**：识别并删除 LS 平台上不再存在于代码库中的“僵尸”评估/提示词。

### 技术栈的演进：展望未来

随着我们的评估逻辑日趋成熟，我们希望以与生产代码相同的严谨性来管理它——版本控制、PR 评审、CI/CD。LangSmith 的 **API 优先架构**使这得以自然实现，允许我们的自定义部署流水线将 TypeScript 定义直接同步到 LangSmith 平台。

这让我们两全其美：既拥有 LangSmith 强大的评估基础设施，又保持我们团队的 GitOps 工作流。我们预计随着生态系统的成熟，这种模式将变得更加普遍，并可能演变为类似于 Terraform 模块的标准化“评估即基础设施”工具。
