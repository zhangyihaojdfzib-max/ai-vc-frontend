# 将Cloudflare威胁指标转化为实时WAF规则

2026-06-08

- Alexandra Moraru
- Harsh Saxena
- Georgie Yoxall
- Brian Seel

![](/images/posts/28aac4fc6183.png)

Cloudflare的威胁事件为安全分析师提供了了解全球威胁态势的窗口。该平台让您一窥Cloudflare每天处理的庞大流量，从而实时查看哪些IP正在攻击特定行业，或者哪些威胁行为者在全球范围内活跃。然而，将这种可见性转化为主动防御通常是一个手动、被动的过程。

安全团队一直面临一个反复出现的困扰：他们知道某些IP地址与特定威胁行为者（如Tycoon 2FA或RaccoonO365）有关联，或者这些IP曾在其他地区针对其所在行业进行攻击，但除非手动配置规则，否则他们无法轻松地在其WAF中自动阻止这些高风险IP。

我们很高兴地宣布一项新的集成功能，将Cloudflare庞大的威胁情报直接引入您的WAF引擎：您现在可以使用实时情报数据编写主动规则。这意味着您可以添加更多情报上下文，在已知恶意行为者甚至尝试接触您的基础设施之前，就保护您的应用程序免受其攻击。

通过在请求的早期阶段填充专用字段，WAF现在可以基于以下条件筛选流量：

- **谁在攻击**：通过匹配特定威胁行为者名称
- **他们针对谁**：通过行业或国家过滤器查看该IP过去曾针对哪些目标
- **攻击类型**：利用丰富的威胁上下文，按攻击类型（DDoS、WAF、网络犯罪等）和最近出现的时间范围进行过滤

## 始终在线的检测

这项新功能基于我们最近为攻击签名检测引入的相同**始终在线检测框架**构建，该系统无需预配置规则即可实时识别常见攻击模式。通过将检测与防御分离，我们确保威胁情报在后台持续运行，在您决定采取行动之前，就用富有洞察力的威胁元数据丰富您的HTTP请求分析。

"始终在线"模式的主要优势在于消除了传统的"日志与阻断"之间的权衡：日志模式下获得可见性，或阻断模式下获得保护。这是因为当规则阻断一个请求时，您将失去对其他签名如何评估该请求的可见性——而这些洞察本可以帮助您加强防御。

如果您拥有**Cloudforce One**订阅，这些洞察将自动出现在您的分析中。您可以查看哪些威胁行为者正在攻击您的站点，以及这些IP通常针对哪些行业，从而在"打开开关"进行阻断之前验证流量模式。

这些检测以可忽略的延迟执行，确保您的性能保持闪电般快速，同时提供构建稳健安全策略所需的高置信度数据。虽然本次初始发布侧重于基于IP的匹配，但我们已经在考虑将这些能力扩展到**JA3指纹**和基于域名的匹配。这将使您能够通过识别攻击者在有效载荷中使用的独特软件签名或恶意目标链接，在攻击者轮换IP时也能阻断恶意流量。

### 新的WAF字段

为实现这一目标，我们已将以下特定信号直接暴露给WAF引擎：

| 字段 | 描述 |
|------|------|
| cf.intel.ip.attacker_names | 已知威胁组织的名称（例如，CRAVENFLEA） |
| cf.intel.ip.target_industries | 该IP针对的行业（例如，加密货币、汽车） |
| cf.intel.ip.attacker_countries | 威胁事件的来源国家 |
| cf.intel.ip.target_countries | 威胁事件针对的国家 |
| cf.intel.ip.datasets | 提供数据的来源（例如，ddos、waf） |

### 规则表达式示例

由于单个IP地址可能同时与多个威胁行为者或目标行业相关联，这些字段以数组形式表示。我们使用`any()`函数和`[*]`通配符来检查该威胁画像中的任何值是否与您的条件匹配：

- **阻止针对您所在区域的已知DDoS参与者**：`any(cf.intel.ip.target_countries[*] == "FR") and any(cf.intel.ip.datasets[*] == "ddos")`
- **防范针对金融行业的特定威胁行为者**：`any(cf.intel.ip.target_industries[*] == "Banking & Financial Services") and any(cf.intel.ip.attacker_names[*] == "BLACKBASTA")`
- **针对特定高风险来源国家的广泛防护**：`any(cf.intel.ip.attacker_countries[*] == "IR")`

## 跨网络的全球情报

可见性和易用性只有在底层引擎足够快速时才能实现。我们如何在不让流量变慢的情况下处理数百万个威胁指标？

这些威胁情报数据集被压缩成高性能格式，并分发到全球每一个Cloudflare数据中心。当请求到达我们的网络时，Cloudflare WAF会针对这些本地数据集执行**O(1)常量时间**查找。这确保无论我们检查十个指标还是一千万个指标，延迟开销都几乎为零（以微秒计）。

由于一个IP地址可能与多种威胁向量相关联，我们的引擎不会在首次匹配时停止。它会同时评估与该IP相关的所有信号集。这确保了查找"攻击者 = 俄罗斯"且"目标行业 = 银行业"的规则能够通过单次评估这些属性的交集而正确触发，从而在不增加计算复杂度的前提下，为多向量攻击者提供最大覆盖范围。
