# 构建 Hugging Face MCP 服务器

TL;DR：Hugging Face 官方 MCP 服务器为访问 Hub 的 AI 助手提供了独特的自定义选项，并通过一个简单的 URL 即可访问数千个 AI 应用。我们使用 MCP 的“可流式 HTTP”传输进行部署，并详细探讨了服务器开发者面临的权衡取舍。

在过去一个月里，我们学到了很多关于构建实用 MCP 服务器的知识——我们将在此描述我们的历程。

## 引言

模型上下文协议（MCP）正在兑现其承诺，成为连接 AI 助手与外部世界的标准。

在 Hugging Face，通过 MCP 提供对 Hub 的访问是一个显而易见的选择，本文分享了我们在开发 `hf.co/mcp` MCP 服务器方面的经验。

## 远程服务器

在构建远程 MCP 服务器时，第一个决定是确定客户端将如何连接到它。MCP 提供了多种传输选项，各有不同的权衡取舍。TL;DR：我们的开源代码支持所有变体，但在生产环境中，我们选择了最现代的一种。本节将详细探讨不同的选项。

自 2024 年 11 月推出以来，MCP 经历了快速演变，在 9 个月内进行了 3 次协议修订。这见证了 SSE 传输被可流式 HTTP 取代，以及授权机制的引入和重构。

这些快速变化意味着客户端应用程序对不同 MCP 功能和修订版本的支持各不相同，这给我们的设计选择带来了额外的挑战。

默认情况下，`STDIO` 和 `HTTP with SSE` 都是完全双向的——这意味着客户端和服务器保持开放的连接，并且可以随时相互发送消息。

SSE 指的是“服务器发送事件”——这是 HTTP 服务器维护开放连接并发送事件以响应请求的一种方式。

#### 理解可流式 HTTP

MCP 服务器开发者在设置可流式 HTTP 传输时面临许多选择。

主要有 3 种通信模式可供选择：

- **直接响应** - 简单的请求/响应（类似于标准的 REST API）。这非常适合简单、无状态的操作，例如简单搜索。
- **请求作用域流** - 与单个请求关联的临时 SSE 流。如果工具调用需要很长时间（例如视频生成），这对于发送**进度更新**非常有用。此外，服务器可能需要通过**询问**向用户请求信息，或进行**采样**请求。
- **服务器推送流** - 支持服务器发起消息的长连接 SSE 连接。这使得资源、工具和提示词列表变更通知或临时的采样和询问成为可能。这些连接需要额外的管理，例如**保活**和重连时的恢复机制。

当使用官方 SDK 的请求作用域流时，使用 `RequestHandlerExtra` 参数（TypeScript）中提供的 `sendNotification()` 和 `sendRequest()` 方法，或设置 `related_request_id`（Python）来将消息发送到正确的流。

### 生产部署

对于生产环境，我们决定以**无状态、直接响应**配置启动我们的 MCP 服务器（使用可流式 HTTP），原因如下：

**无状态**：对于匿名用户，我们提供一套标准工具集用于使用 Hub 以及一个图像生成器。对于已认证用户，我们的状态包括他们**选择的工具**和选定的 Gradio 应用。我们还确保用户的 ZeroGPU 配额正确应用于其账户。这是使用提供的 `HF_TOKEN` 或 OAuth 凭据进行管理的，我们在请求时查找这些凭据。我们现有的工具都不需要我们在请求之间维护任何其他状态。

您可以通过在 MCP 服务器 URL 后添加 `?login` 来使用 OAuth 登录——例如 `https://huggingface.co/mcp?login`。一旦 `claude.ai` 远程集成支持最新的 OAuth 规范，我们可能会将此设为默认选项。

**直接响应**提供了最低的部署资源开销——并且我们目前没有任何工具需要在执行过程中进行采样或询问。

**未来支持**：在发布时，“HTTP with SSE”传输在许多 MCP 客户端中仍然是远程默认选项。然而，由于其即将被弃用，我们不想在管理它上投入大量精力。幸运的是，流行的客户端已经开始进行切换（VSCode 和 Cursor），并且在发布后一周内 `claude.ai` 也添加了支持。如果您需要使用 SSE 连接，欢迎在 **FreeCPU Hugging Face Space** 上部署我们服务器的副本。

#### 工具列表变更通知

未来，我们希望在用户在 Hub 上更新其设置时支持实时工具列表变更通知。然而，这引发了一些实际问题：

首先，用户倾向于在客户端中配置他们最喜欢的 MCP 服务器并保持启用状态。这意味着在应用程序打开时，客户端保持连接。发送通知将意味着需要维护与当前活跃客户端数量一样多的开放连接——无论是否处于活跃使用状态——以防用户更新其工具配置。

其次，大多数 MCP 服务器和客户端在一段时间不活动后会断开连接，并在必要时恢复。这不可避免地意味着即时推送通知会被错过——因为通知通道已经关闭。实际上，让客户端根据需要刷新连接和工具列表要简单得多。

除非您对客户端/服务器对拥有合理的控制权，否则在公共部署中使用**服务器推送流**会带来很多复杂性，而刷新工具列表存在资源消耗更低的解决方案。

#### URL 用户体验

就在发布之前，`@julien-c` 提交了一个 PR，为访问 `hf.co/mcp` 的用户添加友好的说明。这极大地改善了用户体验——否则默认响应是一段不友好的 JSON。

最初，我们发现这产生了巨大的流量。经过一番调查，我们发现当返回一个网页而不是 HTTP 405 错误时，VSCode 会每秒轮询端点多次！

`@coyotte508` 建议的修复方法是正确检测浏览器，并仅在该情况下返回页面。同时感谢 VSCode 团队迅速**修复了它**。

虽然未明确说明——但以这种方式返回页面在MCP规范中似乎确实是可以接受的。

#### MCP客户端行为

MCP协议在初始化期间会发送多个请求。典型的连接顺序是：`Initialize`、`Notifications/Initialize`、`tools/list`，然后是`prompts/list`。

考虑到MCP客户端在打开时会连接和重连，且用户会定期进行调用，我们发现每100条MCP控制消息大约对应一次工具调用。

一些客户端还会发送对我们的无状态、直接响应配置无意义的请求——例如Ping、取消请求或尝试列出资源（这不是我们当前宣传的功能）。

2025年7月的第一周，有惊人的164个不同的客户端访问了我们的服务器。有趣的是，最受欢迎的工具之一是`mcp-remote`。大约一半的客户端使用它作为连接到我们远程服务器的桥梁。
