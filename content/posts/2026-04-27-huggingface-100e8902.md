# 如何使用OpenAI的隐私过滤器构建可扩展的Web应用

OpenAI本周在Hub上发布了隐私过滤器：一个开源的个人身份信息(PII)检测器，可在128k上下文的单次前向传播中标记八类文本。模型卡片。我们花了几个小时用它构建应用，最终确定了三个应用，每个都展示了其不同方面的能力。

- 文档隐私浏览器：导入PDF或DOCX文件，文档中所有PII片段会高亮显示。
- 图像匿名化器：上传图片，返回的图片中姓名、邮箱和账号会被黑色遮条覆盖。图片也可在画布上编辑，方便下载前自行添加标注。
- 智能脱敏粘贴：粘贴敏感文本，分享一个提供脱敏版本的公开URL，同时为自己保留一个私密查看链接。

这三个应用都基于`gradio.Server`构建，它允许你将自定义HTML/JS前端与Gradio的队列、ZeroGPU分配和`gradio_client`SDK配合使用。在这些应用中，`gradio.Server`扮演着相同的后端角色，而这种一致性正是其强大之处。

## 模型

隐私过滤器是一个15亿参数的模型，其中5000万参数为活跃参数，采用Apache 2.0许可协议。PII类别包括：private_person、private_address、private_email、private_phone、private_url、private_date、account_number、secret。上下文窗口为128,000个Token。在PII-Masking-300k基准测试中达到了最先进水平。完整数据和方法论请参见官方发布博客。

## 1. 文档隐私浏览器

试用地址：ysharma/OPF-Document-PII-Explorer

用户问题。你想阅读一份包含大量PII的文档（合同、简历、导出的聊天记录），希望每个检测到的片段按类别高亮显示，侧边栏有筛选功能，顶部有汇总仪表板。阅读体验应像普通文档一样，而非表单形式。

隐私过滤器的作用。整个文件通过单次128k上下文的前向传播处理，因此无需分块、拼接，片段偏移量直接与渲染文本对齐。BIOES解码可在长段模糊文本中保持片段边界清晰。

gr.Server的作用。你可以用Blocks配合gr.HighlightedText和侧边栏来实现，也能正常工作。但我们想要的阅读体验（衬线字体正文、在客户端切换CSS类别而非重新运行模型的筛选器、无需页面重新渲染的汇总仪表板）手动编写比组合实现更简单。gr.Server允许我们将阅读器视图作为单个HTML文件提供，并通过一个排队端点暴露模型：

```python
import gradio as gr
from fastapi.responses import HTMLResponse
from gradio.data_classes import FileData

server = gr.Server()

@server.get("/", response_class=HTMLResponse)
async def homepage():
    return FRONTEND_HTML                           

@server.api(name="analyze_document")
def analyze_document(file: FileData) -> dict:
    text = extract_text(file["path"])              
    source_text, spans = run_privacy_filter(text)  
    return {
        "text":  source_text,
        "spans": spans,                            
        "stats": compute_stats(source_text, spans),
    }

```

注意装饰器：`@server.api(name="analyze_document")`，而非普通的`@server.post`。正是这个组件将处理器接入Gradio的队列，使并发上传得以序列化，`@spaces.GPU`在ZeroGPU上正确组合，且同一端点可从浏览器和`gradio_client`访问，无需重复代码。浏览器通过Gradio JS客户端调用它：

```html
<script type="module">
import { Client, handle_file } from "https://cdn.jsdelivr.net/npm/@gradio/client/dist/index.min.js";
const client = await Client.connect(window.location.origin);

async function uploadFile(file) {
  const result = await client.predict("/analyze_document", { file: handle_file(file) });
  renderResults(result.data[0]);                   
}
</script>

```

## 2. 图像匿名化器

试用地址：ysharma/OPF-Image-Anonymizer

用户问题。你想分享一张图片或任何截图（Slack对话、收据、Stripe仪表板），用黑色遮条覆盖PII。你想开关遮条、拖拽重新定位，或手动绘制模型遗漏的部分，然后导出结果。

隐私过滤器的作用。Tesseract运行OCR并返回每个词的边界框。后端通过字符偏移到框的映射重建完整文本，然后对整个文本运行一次隐私过滤器。检测到的字符片段通过词映射查找，合并为每行的像素矩形。

gr.Server的作用。gr.ImageEditor支持分层标注，是图像脱敏的合理起点。但我们想要的工作流程（每个遮条的类别元数据、一键切换某类所有遮条、客户端以自然分辨率导出PNG且无需服务器往返）在自定义`<canvas>`前端上构建更简洁。gr.Server从一个排队端点返回像素矩形，让画布处理其余所有内容：

```python
@server.api(name="anonymize_screenshot")
def anonymize_screenshot(image: FileData) -> dict:
    img = Image.open(image["path"]).convert("RGB")
    full_text, char_to_box = ocr_image(img)        
    spans = run_privacy_filter(full_text)
    boxes = spans_to_pixel_boxes(spans, char_to_box)
    return {
        "image_data_url": pil_to_base64(img),
        "width":  img.width,
        "height": img.height,
        "boxes":  boxes,                           
    }

```

前端通过`client.predict("/anonymize_screenshot", { image: handle_file(file) })`调用它，与上述模式相同。开关、拖拽、绘制新遮条和PNG导出均在浏览器中完成；编辑操作无需往返服务器。

## 3. 智能脱敏粘贴

试用地址：ysharma/OPF-SmartRedact-Paste

用户问题。你想要一个在分享前进行脱敏处理的粘贴板。你粘贴一行日志、一封邮件、一个支持工单。你会得到两个URL。公开URL提供脱敏版本，使用`<PRIVATE_PERSON>`、`<PRIVATE_EMAIL>`、`<ACCOUNT_NUMBER>`占位符，遵循官方博客示例中的脱敏规范。私有URL通过你持有的Token进行访问控制，显示原始文本并高亮显示片段。

隐私过滤器的作用。将存储的粘贴内容中每个检测到的片段替换为`<CATEGORY>`占位符。这就是整个脱敏步骤。多语言文本（模型卡片示例中的西班牙语、法语、中文、印地语等）通过同一调用处理，无需更改。

gr.Server的作用。此应用需要为同一粘贴ID提供两个不同的GET路由，一个公开，一个受Token保护，且URL格式很重要，因为查看URL是你需要保留的内容。gr.Server在此适用，因为它底层是一个FastAPI应用——这也是`@server.api`和普通`@server.get`能在同一进程中并存的原因。注意：这也可以通过gr.Blocks()挂载FastAPI自定义路由来构建：

```python


@server.api(name="create_paste")
def create_paste(text: str, ttl: str = "never") -> dict:
    source_text, spans = run_privacy_filter(text)
    redacted = redact(source_text, spans)          
    pid, reveal_token = secrets.token_urlsafe(6), secrets.token_urlsafe(22)
    PASTES[pid] = Paste(pid, reveal_token, source_text, redacted, spans,
                        expires_at=_ttl(ttl))      
    return {
        "view_path":   f"/view/{pid}",
        "reveal_path": f"/view/{pid}?token={reveal_token}",
    }




@server.get("/view/{pid}", response_class=HTMLResponse)
async def view_paste(pid: str, token: str | None = None):
    p = _store_get(pid)                            
    if p is None:
        return HTMLResponse(_not_found(), status_code=404)
    revealed = bool(token) and secrets.compare_digest(token, p.reveal_token)
    return HTMLResponse(_render_view(p, revealed))

```

一个守护线程每30秒清理一次过期的粘贴内容。整个服务（包括存储）大约只有200行应用代码，因为所有内容都运行在同一个进程中。

## `gradio.Server` 提供的能力

三个应用的拆分逻辑相同——任何涉及模型的操作都通过 `@server.api` 处理，其余内容则使用普通的 FastAPI 路由：

`@server.api` 提供了 Gradio 的队列（序列化请求、ZeroGPU 上正确的 `@spaces.GPU` 组合、进度事件），同时也是浏览器通过 `@gradio/client` 访问的接口。同样的端点也被 `gradio_client` 用户从 Python 调用——一个函数、两个 SDK，无需重复代码。普通的 `@server.get`/`@server.post` 则保留给静态表层：HTML 页面、文件查找、简单的字典读取。这是 `gradio.Server` 介绍文章中的经验法则，也是让这三个应用尽管界面差异很大却仍能保持一致性的关键。
