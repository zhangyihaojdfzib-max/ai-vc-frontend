# Foundry 托管计算上的 Hugging Face 模型

在 Microsoft Build 2026 大会上，我们发布了 Foundry 托管计算 和 Foundry 上的 Hugging Face 模型——一个来自 Hugging Face 生态系统的精选开放权重模型目录，每周更新，可一键部署到 Foundry 托管计算上。权重预置于 Azure 中，运行时由 Microsoft 构建和扫描，且集合中的每个模型都享有与 Foundry 上其他模型相同的企业级安全性、治理、可观测性和计费功能。

## 平台：Microsoft Foundry 与托管计算

Microsoft Foundry 是一个用于构建和运行 Agent（智能体）AI 应用的平台。Foundry 从任何云平台上最广泛的模型选择开始——来自 Microsoft、OpenAI、Anthropic、Meta、Mistral、DeepSeek、Hugging Face 等厂商的模型，涵盖前沿、开源和自定义权重——全部通过单一端点和一套 Python、C#、JavaScript 和 Java SDK 访问。

在这些模型之上是 Foundry Agent 服务：具备内置记忆的多 Agent（智能体）编排、通过 Foundry IQ 实现的知识基础，以及通过 Agent（智能体）协议连接的工具目录，使 Agent（智能体）能够与企业数据协同工作。Agent（智能体）运行后，Foundry 提供端到端追踪、实时监控、持续评估，以及一个基于评估结果改进 Agent（智能体）行为的提示词优化器——这些可观测性和质量闭环是平台的一部分。

此外，开发者还可获得：

- 内容安全过滤器
- 任务合规护栏
- 用于对抗性测试的 AI 红队 Agent（智能体）
- 统一 RBAC
- 私有网络
- 平台内直接集成的 Azure Policy

除了按 Token 付费（最低门槛的入门方式）和预配吞吐量（前沿模型上可预测的高性能生产工作负载）之外，Foundry 托管计算 是 Foundry 中的第三种部署选项：一个面向开源和自定义模型的托管 GPU 平台即服务。

您部署一个由工作负载关键要素描述的模型实例——参数数量、上下文窗口长度，以及您希望优化延迟还是吞吐量——Foundry 负责底层的 GPU 拓扑，无论实例落在单个还是多个加速器上，让您以模型术语进行思考和规划。

Microsoft 负责机器管理：容器更新、运行时升级和安全补丁在支持的运行时（vLLM、SGLang、TensorRT-LLM、NIM、TEI、llama.cpp）上自动执行，无需重新部署模型，而模型配置、部署行为和路由仍由您掌控。

这种一致性贯穿整个开发者界面——按 Token 付费、预配吞吐量和托管计算共享：

- 单一端点
- 相同的 SDK
- 相同的身份验证
- 相同的可观测性
- 单一账单

开源模型与前沿模型以相同方式集成到 Foundry Agents（智能体）中，因此您可以在单个 Agent（智能体）中混合使用模型类型，无需单独的集成路径。

托管计算提供：

- 全球部署——最广泛的容量和最优定价
- 数据区域部署——数据驻留和主权

相同的代码，相同的工作流程。配额与加速器系列对齐，因此基于 H100 系列构建的计划可在新硬件世代上线时延续使用。

## 为什么选择 Hugging Face

Hugging Face 是开放 AI 的公共广场：1500 万构建者、40 万个组织以及超过 300 万个已发布的开源模型，每周都有新的前沿能力落地——Agent（智能体）编码、视频分割、语音、嵌入/向量。它是开源模型的 GitHub，社区在此发布权重、编写模型卡片、比较评估结果并拉取模型进行实验。

开源模型在各项基准测试中已缩小了与专有模型的差距，并解锁了专有端点无法实现的功能：

- 前沿技术现已开放。领先的开放权重模型在最广泛使用的基准测试中与顶级封闭前沿模型具有竞争力。
- 深度定制。完整权重使得通过 LoRA 进行微调、蒸馏、量化和适配成为可能——根据您的领域、数据以及延迟和成本目标定制模型。
- 您的模型，您的托管。权重在您的租户中运行，位于您控制的基础设施上，在您的推理端点之后，遵循您的身份和网络边界。
- 成本塑造。按小时为加速器付费，空闲时缩至零，并为特定模型选择合适的 GPU 规模——适用于按 Token 定价难以预测的稳定、高容量或延迟敏感型工作负载。
- 版本控制。锁定特定模型版本，进行评估、部署，并按照自己的发布节奏推进或回滚。

操作层面一直是痛点：发现、许可证审查、安全筛选、运行时选择、GPU 规模确定、镜像构建、CVE 修补，以及将模型部署到企业级端点之后。Hugging Face 本身并非企业级服务平台。Foundry 上的 Hugging Face 模型 就是由 Microsoft 运营的这个操作层。

## Foundry 上的 Hugging Face 模型

Hugging Face 集合将精选的模型子集直接引入 Foundry 模型目录：

- 每周更新——随着社区发布，来自 Hugging Face 生态系统的热门模型持续添加。
- 涵盖所有模态——文本、视觉、音频和多模态：用于聊天和 Agent（智能体）的 LLM（大语言模型）和 VLM、ASR 和语音翻译、嵌入/向量、分割、图像生成。
- 仅限 SafeTensors，无不可信代码——集合中的每个模型都经过安全筛选，以 SafeTensors 权重格式发布，除非经过严格审查，否则不启用 trust_remote_code 执行路径。
- 为模型匹配正确的运行时——LLM（大语言模型）使用 vLLM 和 SGLang，适用时使用 TensorRT-LLM 和 NIM，嵌入/向量使用 TEI，CPU 使用 llama.cpp——Foundry 选择与模型匹配的引擎。

从您的角度来看，Hugging Face 集合中的开放权重模型与 Foundry 模型目录中的任何其他模型外观和行为一致，并且集合中的每个模型在出现在目录之前都经过了多阶段发布流程。

### 策展流程

Hugging Face 和 Microsoft 通过系统化的策展流程，将 Hugging Face 生态系统中流行的开放权重模型引入 Microsoft Foundry——为企业环境做好生产准备：

1. 识别 Hugging Face 生态系统中的热门模型——基于社区信号、合作伙伴请求和客户需求——并筛选出适合企业就绪的候选模型。
2. 合规性和安全性筛选——根据 Microsoft 的企业分发策略审查模型许可证（许可证元数据捕获并保留在目录模型卡片上），并检查仓库中的 trust_remote_code 模式和自定义可执行代码；任何需要在加载时执行第三方 Python 的模型将被修复或排除。
3. 构建、扫描和发布运行时——Microsoft 在支持的运行时（vLLM、SGLang、TensorRT-LLM、NIM、TEI、llama.cpp）上构建推理容器镜像，扫描 CVE，签名并发布到 Microsoft 管理的容器注册表。
4. 上传权重到安全的 Azure 存储——模型权重从 Hugging Face 拉取一次，根据已发布的模型卡片进行验证，并存储在模型服务区域的 Microsoft 管理 Azure 存储中。
5. 验证并发布到目录——每个模型 + 运行时 + 加速器组合都经过 API 合规性（聊天补全、嵌入/向量、重排序等）和性能（延迟、吞吐量、首 Token 时间、Token 间解码时间）测试，然后经过验证的模型——连同其模板、运行时镜像和权重——发布到 Foundry 模型目录，并可通过一键部署到托管计算上。

由于权重文件预置于 Azure 存储中，运行时镜像则存放在微软托管的注册表内，您的部署无需出站网络访问 Hugging Face Hub——您可以在私有网络内部署至生产环境。

## 模型运行时

Foundry 上的 Hugging Face 模型由一系列社区构建的开源推理运行时驱动——每个运行时都针对 Foundry 托管计算进行了精选和调优，并匹配其最适合的模型架构。在所有运行时中，系统化的筛选流程意味着新版本和补丁能快速落地 Foundry，现有模型部署会自动升级——无需您重新部署。

- **vLLM**——面向开源大语言模型的默认高吞吐量服务引擎，针对生产级 GPU 工作负载进行了调优。由于 Hugging Face 是 vLLM 的直接贡献者，Transformers 库中的任何模型均可开箱即用地在 vLLM 上运行——因此当新模型登陆 Hugging Face 时，当天即可在 Foundry 上提供服务，无需等待自定义集成。
- **SGLang**——面向语言和多模态模型的服务引擎，对 Agent（智能体）和工具调用工作负载所依赖的结构化输出（JSON、正则表达式、语法约束生成）提供强支持。Hugging Face 与 SGLang 团队已为 SGLang 构建了 Transformers 后端集成，因此 Transformers 库中的任何模型均可开箱即用地在 SGLang 上运行——并在登陆 Hugging Face 当天即可在 Foundry 上使用。
- **Text Embeddings Inference (TEI)**——面向嵌入、重排序和序列分类模型的运行时。加速器专属镜像附带针对 Foundry 支持的每种 GPU 和 CPU 系列编译的内核，使嵌入热路径在 RAG（检索增强生成）和语义搜索工作负载中保持精简。
- **llama.cpp**——面向 GGUF 量化模型的 CPU 和小型 GPU 路径。适用于成本优化部署、小型模型及纯 CPU 区域，提供与 vLLM 和 SGLang 相同的 OpenAI 兼容 API。
- **TensorRT-LLM 和 NIM**——用于 NVIDIA 硬件，NVIDIA 的优化内核和基于 Triton 的服务可为特定模型系列带来显著的延迟或吞吐量提升。
- **hf-serve**——Hugging Face 自家的多模型推理服务器，用于 LLM（大语言模型）和嵌入快速路径之外的模型架构（视觉、音频、分割及其他 Transformers 原生流水线），使模型集合能够通过一致的服务层覆盖所有模态。

vLLM——面向开源大语言模型的默认高吞吐量服务引擎，针对生产级 GPU 工作负载进行了调优。由于 Hugging Face 是 vLLM 的直接贡献者，Transformers 库中的任何模型均可开箱即用地在 vLLM 上运行——因此当新模型登陆 Hugging Face 时，当天即可在 Foundry 上提供服务，无需等待自定义集成。

## 部署与评分开放权重模型

Foundry 模型目录中的 Hugging Face 模型集合是您的起点，部署只需五个步骤：

1. **浏览目录并选择模型**——部署向导还会显示模型 ID、部署模板 ID 和 acceleratorType，如果您通过 SDK 或 REST 编写部署脚本，这些信息是必需的。
2. **选择部署模板**——延迟优化与吞吐量优化、加速器系列、上下文长度、量化。
3. **配置实例数量**——通过增加模型实例来扩展吞吐量。
4. **部署**——通过门户、CLI、SDK 或 REST 进行。
5. **评分**——通过统一的 Foundry 端点，使用您已有的 SDK 进行评分。

### 部署模板

部署模板是步骤 2 中的选择单元：一个命名的、带版本号的资产，它固定了运行时、加速器系列和数量、上下文长度以及为良好服务模型所需的运行时特定调优——因此选择模板是您控制“我希望模型如何运行”的唯一旋钮。

例如，qwen3-32b 附带了四个模板，部署向导会并排展示：

每个模板都针对模型进行了预调优——运行时设置、工具调用和推理解析器、评分路径、健康探测、请求并发以及任何模型特定的上下文扩展设置均由微软设置，任何权衡都在模板描述中内联说明。当您编写部署脚本时，您引用模板，Foundry 会处理其余部分。

### 部署——Python SDK

```python
from azure.identity import DefaultAzureCredential
from azure.mgmt.cognitiveservices import CognitiveServicesManagementClient

client = CognitiveServicesManagementClient(DefaultAzureCredential(), SUBSCRIPTION_ID)

deployment = client.managed_compute_deployments.begin_create_or_update(
    resource_group_name=RESOURCE_GROUP,
    account_name=ACCOUNT_NAME,
    deployment_name="qwen3-32b",
    resource={
        "sku": {"name": "GlobalManagedCompute", "capacity": 1},
        "properties": {
            "model": "azureml://registries/azure-huggingface/models/qwen--qwen3-32b/versions/1",
            "deploymentTemplate": "azureml://registries/azure-huggingface/deploymenttemplates/qwen--qwen3-32b--40k-nvidia-h100/labels/latest",
            "acceleratorType": "H100_80GB",
        },
    },
).result()

```

### 评分——OpenAI SDK

部署可通过统一的 Foundry 端点使用 OpenAI SDK 访问——`model` 字段使用您刚刚创建的部署名称：

```python
from openai import OpenAI

api_key  = client.accounts.list_keys(RESOURCE_GROUP, ACCOUNT_NAME).key1
endpoint = f"https://{ACCOUNT_NAME}.services.ai.azure.com/openai/v1"

openai_client = OpenAI(base_url=endpoint, api_key=api_key)

completion = openai_client.chat.completions.create(
    model=deployment.name,
    messages=[{"role": "user", "content": "What is the capital of France?"}],

print(completion.choices[0].message)

```
