# Codex编排的开源规范：Symphony

作者：Alex Kotliarskyi、Victor Zhu 和 Zach Brock

六个月前，在开发一个内部生产力工具时，我们的团队做出了一个（在当时）颇具争议的决定：我们将构建一个没有任何人工编写代码的代码库。我们项目仓库中的每一行代码都必须由 Codex 生成。

为了实现这一目标，我们从头开始重新设计了工程工作流程。我们构建了一个对 Agent（智能体）友好的仓库，大力投资于自动化测试和安全护栏，并将 Codex 视为一个成熟的团队成员。我们在之前的关于驾驭工程的博文中记录了这段历程。

这个方法奏效了，但随后我们遇到了下一个瓶颈：上下文切换。

为了解决这个新问题，我们构建了一个名为 Symphony 的系统。Symphony（在新窗口中打开）是一个 Agent（智能体）编排器，它将 Linear 这样的项目管理看板转变为编码 Agent（智能体）的控制平面。每个开放的任务都会获得一个 Agent（智能体），Agent（智能体）持续运行，并由人类审查结果。

本文解释了我们是怎样创建 Symphony 的——这使得某些团队的合并拉取请求数量增加了 500%——以及如何使用它来将你自己的问题跟踪器转变为一个始终在线的 Agent（智能体）编排器。

### 交互式编码 Agent（智能体）的天花板

即使编码 Agent（智能体）——无论是通过 Web 应用还是 CLI 访问——变得更容易使用，它们仍然是交互式工具。

随着 OpenAI 内部基于 Agent（智能体）的工作规模扩大，我们发现了一种新的负担。每个工程师会打开几个 Codex 会话，分配任务，审查输出，引导 Agent（智能体），然后重复。在实践中，在上下文切换变得痛苦之前，大多数人可以轻松同时管理三到五个会话。超过这个数量，生产力就会下降。我们会忘记哪个会话在做什么，在终端之间跳转以引导 Agent（智能体）回到正轨，并调试那些中途停滞的长时间运行的任务。

Agent（智能体）很快，但我们遇到了一个系统瓶颈：人类的注意力。我们实际上建立了一个由极其能干的初级工程师组成的团队，然后指派我们的人类工程师去微观管理他们。这无法规模化。

### 视角的转变

我们意识到我们优化错了方向。我们围绕编码会话和合并的 PR 来构建系统，而 PR 和会话实际上只是达到目的的手段。软件工作流程主要围绕可交付成果组织：问题、任务、工单、里程碑。

因此，我们问自己，如果我们不再直接监督 Agent（智能体），而是让它们从我们的任务跟踪器中拉取工作，会发生什么。

这个想法演变成了 Symphony，一个作为监督者来编排基于 Agent（智能体）工作的书面规范。

### 将我们的问题跟踪器转变为 Agent（智能体）编排器

Symphony 从一个简单的概念开始：任何开放的任务都应该被 Agent（智能体）拾取并完成。我们不再在多个标签页中管理 Codex 会话，而是让我们的问题跟踪器成为控制平面。

在这种设置中，每个开放的 Linear 问题都映射到一个专用的 Agent（智能体）工作空间。Symphony 持续监控任务看板，并确保每个活跃的任务都有一个 Agent（智能体）在循环中运行，直到完成。如果 Agent（智能体）崩溃或停滞，Symphony 会重启它。如果出现新工作，Symphony 会拾取它并开始组织工作。

我们基于工单状态构建了工作流程，将任务管理器 Linear 用作状态机。

![编码 Agent（智能体）使用 Linear 作为状态机与我们协同工作。](/images/posts/82fb1868dd5f.jpg)

在实践中，Symphony 将工作与会话和拉取请求解耦。有些问题会在多个仓库中产生多个 PR；其他问题则是纯粹的调查或分析，从不触及代码库。

我们经常使用 Symphony 来编排复杂的功能和基础设施迁移。例如，我们可能会提交一个任务，要求 Agent（智能体）分析代码库、Slack 或 Notion，并生成一个实施计划。一旦我们对计划满意，Agent（智能体）就会生成一个任务树，将工作分解为多个阶段，并定义任务之间的依赖关系。

Agent（智能体）只开始处理未被阻塞的任务，因此对于这个 DAG（一系列执行步骤）来说，执行会自然且最优地并行展开。例如，我们将 React 升级标记为依赖于迁移到 Vite。正如预期的那样，Agent（智能体）只有在迁移到 Vite 完成后才开始升级 React。Agent（智能体）也可以自己创建工作。在实施或审查过程中，它们经常注意到当前任务范围之外的改进：性能问题、重构机会或更好的架构。当这种情况发生时，它们只需提交一个新问题，我们可以稍后评估和安排——这些后续任务中的许多也会被 Agent（智能体）拾取。在我们监督这个过程的同时，Agent（智能体）保持组织有序，并推动工作向前发展。

这种工作方式极大地降低了启动模糊工作的认知成本。如果 Agent（智能体）做错了什么，那仍然是有效信息，而且对我们的成本几乎为零。我们可以非常廉价地提交工单让 Agent（智能体）去制作原型和探索，并丢弃任何我们不喜欢的探索结果。

由于编排器在开发盒上运行且从不休眠，我们可以从任何地方添加任务，并知道会有 Agent（智能体）拾取它。例如，我们团队的一名工程师在信号不佳的舒适小屋里，通过手机上的 Linear 应用完成了三项重大更改。

### 以这种方式工作带来的探索增加

在观察使用 Symphony 工作的效果时，最明显的变化是产出。在 OpenAI 的一些团队中，我们看到合并的 PR 数量在前三周内增加了 500%。在 OpenAI 之外，Linear 创始人 Karri Saarinen 指出，随着我们发布 Symphony，创建工作空间的数量出现了激增（在新窗口中打开）。然而，更深层次的转变是团队思考工作的方式。

当我们的工程师不再花费时间监督 Codex 会话时，代码变更的经济性完全改变了。每次变更的感知成本下降了，因为我们不再投入人力来驱动实施本身。

这改变了我们的行为。在 Symphony 中启动探索性任务变得微不足道。尝试一个想法，探索一次重构，测试一个假设，只保留看起来有希望的结果。

这也拓宽了可以发起工作的人员范围。我们的产品经理和设计师现在可以直接将功能请求提交到 Symphony。他们不需要检出仓库或管理 Codex 会话。他们描述功能，然后收到一个审查包，其中包含该功能在实际产品中运行的视频演示。

Symphony 在大型单体仓库（比如我们在 OpenAI 拥有的那种）中也表现出色，在这些仓库中，合并 PR 的最后一步既缓慢又脆弱。该系统监控 CI，在需要时变基，解决冲突，重试不稳定的检查，并通常引导变更通过流水线。当一个工单到达“合并”状态时，我们有很高的信心，该变更无需人工看护就能进入主分支。

![Symphony 的前后对比网格图](/images/posts/4752a1ee00b4.jpg)

实施 Symphony 后，我们将更多工作委托给 Agent（智能体），并专注于更困难、更具探索性的任务。

### 进步伴随着新的、不同的问题

在这种水平上运作需要权衡取舍。当我们从交互式引导 Agent（智能体）转变为在工单级别分配工作时，我们失去了在任务执行过程中不断引导并在需要时纠正方向的能力。有时 Agent（智能体）产生的结果完全偏离了目标。这很有用——这些失败揭示了系统中的差距，并帮助我们使其更加健壮。

我们没有手动修补结果，而是添加了安全护栏和技能，以便 Agent（智能体）下次能够成功。随着时间的推移，这促使我们为驾驭系统增加了新能力，例如运行端到端测试、通过 Chrome DevTools 驱动应用以及管理 QA 冒烟测试。我们显著改进了文档，并明确了什么是好的标准。

并非所有任务都适合Symphony的工作方式。有些问题仍需要工程师直接通过交互式Codex会话解决，尤其是那些模糊不清或需要较强判断力和专业知识的任务。在实践中，这些通常是我们工程师最感兴趣、也最乐于投入时间的任务。

区别在于，Symphony能够处理大部分常规实现工作。这让工程师可以一次专注于一个难题，而不是在多个小任务之间频繁切换上下文。

我们还认识到，将Agent（智能体）视为状态机中的刚性节点效果并不理想。模型变得越来越智能，能够解决比我们试图限制它们的框架更大的问题。我们早期版本的Agent（智能体）工作方式只是要求Codex实现任务，这种方法被证明过于局限。Codex完全有能力创建多个PR，同时还能阅读评审反馈并加以处理。因此，我们为它提供了工具——ghCLI、读取CI日志的技能等——现在我们可以要求Codex做更多事情，比如关闭旧的PR，或者拉取已完成与已放弃工作的对比报告。这些类型的任务远远超出了最初功能实现的框架。

因此，我们最终转向为Agent（智能体）设定目标，而非严格的流程转换，这就像一位优秀的管理者会为团队中的直接下属分配目标一样。模型的力量来自于它们的推理能力，所以给它们工具和上下文，让它们自由发挥。

### 用Symphony构建Symphony

当你打开Symphony仓库⁠（在新窗口中打开）时，首先会注意到，Symphony在技术上只是一个SPEC.md文件——即对问题及预期解决方案的定义。我们没有构建一个复杂的监督系统，而是定义了问题和预期解决方案，为Agent（智能体）提供高层级的指导。

6## 1. 问题陈述  
7  
8Symphony 是一个长期运行的自动化服务，它持续从问题跟踪器（本规范版本中为 Linear）读取工作，为每个问题创建隔离的工作空间，并在该工作空间内为该问题运行编码 Agent（智能体）会话。  
9  
10该服务解决四个操作性问题：  
11  
12- 将问题执行转变为可重复的守护进程工作流，而非手动脚本。  
13- 将 Agent（智能体）执行隔离在每个问题的工作空间中，使 Agent（智能体）命令仅在各问题的工作空间目录内运行。  
14- 将工作流策略保留在仓库内（`WORKFLOW.md`），使团队能够将 Agent（智能体）提示词和运行时设置与其代码一起进行版本管理。  
15- 提供足够的可观测性，以操作和调试多个并发运行的 Agent（智能体）。  
16  
17实现应明确记录其信任与安全策略。本规范不要求单一的审批、沙箱或操作员确认策略；某些实现可能针对高信任配置的受信任环境，而其他实现则可能要求更严格的审批或沙箱机制。  
18  
19重要边界：  
20  
21- Symphony 是一个调度器/运行器和跟踪器读取器。  
22- 工单写入（状态转换、评论、PR 链接）通常由编码 Agent（智能体）使用工作流/运行时环境中可用的工具执行。  
23- 成功的运行可能结束于工作流定义的交接状态（例如“人工审核”），而不一定是“已完成”。

25## 2. 目标与非目标  
26  
27### 2.1 目标  
28  
29- 按固定周期轮询问题跟踪器，并以有界并发度分派工作。  
30- 维护一个单一的权威编排器状态，用于分派、重试和协调。  
31- 为每个问题创建确定性的工作空间，并在多次运行之间保留它们。  
32- 当问题状态变更使其不符合条件时，停止正在进行的运行。  
33- 通过指数退避从瞬时故障中恢复。  
34- 从仓库拥有的 `WORKFLOW.md` 契约中加载运行时行为。  
35- 暴露操作员可见的可观测性（至少包括结构化日志）。  
36- 支持无需持久化数据库的重新启动恢复。  
37  
38### 2.2 非目标  
39  
40- 丰富的 Web UI 或多租户控制平面。  
41- 规定特定的仪表盘或终端 UI 实现。  
42- 通用工作流引擎或分布式作业调度器。  
43- 内置的编辑工单、PR 或评论的业务逻辑。（该逻辑存在于工作流提示词和 Agent（智能体）工具中。）  
44- 强制要求超出编码 Agent（智能体）和主机操作系统所提供的强沙箱控制。  
45- 为所有实现强制规定单一的默认审批、沙箱或操作员确认策略。

50  
511. `工作流加载器`  
52   - 读取 `WORKFLOW.md`。  
53   - 解析 YAML 前置元数据和提示词主体。  
54   - 返回 `{config, prompt_template}`。

63   - 获取特定问题 ID 的当前状态（协调）。  
64   - 在启动清理期间获取终止状态的问题。

68   - 拥有轮询周期。  
69   - 拥有内存中的运行时状态。

71   - 跟踪会话指标和重试队列状态。  
72

74   - 将问题标识符映射到工作空间路径。  
75   - 确保每个问题的工作空间目录存在。  
76   - 运行工作空间生命周期钩子。  
77   - 清理终止状态问题的工作空间。  
78  
796. `Agent（智能体）运行器`  
80   - 创建工作空间。  
81   - 从问题和工作流模板构建提示词。

83   - 将 Agent（智能体）更新流式传输回编排器。  
84  
857. `状态界面`（可选）  
86   - 呈现人类可读的运行时状态（例如终端输出、仪表盘或其他面向操作员的视图）。  
87  
888. `日志记录`  
89   - 向一个或多个配置的接收端发送结构化运行时日志。

110   - 跟踪器数据的 API 调用和标准化。  
111  
1126. `可观测性层`（日志 + 可选状态界面）  
113   - 操作员对编排器和 Agent（智能体）行为的可见性。

## 7. 编排状态机

编排器是唯一变更调度状态的组件。所有工作进程的结果都回传给编排器，并转换为显式的状态转换。

### 7.1 问题编排状态

这与跟踪器状态（`Todo`、`In Progress` 等）不同。这是服务内部的声明状态。

1. `Unclaimed`（未声明）
   - 问题未在运行，且没有安排重试。

2. `Claimed`（已声明）
   - 编排器已预留该问题以防止重复调度。
   - 实际上，已声明的问题要么是 `Running`（运行中），要么是 `RetryQueued`（重试排队中）。

3. `Running`（运行中）
   - 工作进程任务存在，且该问题在 `running` 映射中被跟踪。

4. `RetryQueued`（重试排队中）
   - 工作进程未运行，但 `retry_attempts` 中存在重试计时器。

5. `Released`（已释放）
   - 声明被移除，因为问题已终止、非活跃、缺失，或重试路径已完成但未重新调度。

重要细节：

- 工作进程成功退出并不意味着问题永远结束。
- 工作进程可能在退出前连续执行多个编码代理轮次。
- 每次正常轮次完成后，工作进程会重新检查跟踪器问题状态。
- 如果问题仍处于活跃状态，工作进程应在同一工作区的同一活跃编码代理线程上开始另一轮次，最多达到 `agent.max_turns`。
- 第一轮应使用完整的渲染任务提示词。
- 后续轮次应仅向现有线程发送继续指导，而非重新发送已在线程历史中的原始任务提示词。
- 一旦工作进程正常退出，编排器仍会安排一个短暂的继续重试（约1秒），以便重新检查问题是否仍处于活跃状态并需要另一个工作进程会话。

### 7.2 运行尝试生命周期

一次运行尝试经历以下阶段：

1. `PreparingWorkspace`（准备工作区）
2. `BuildingPrompt`（构建提示词）
3. `LaunchingAgentProcess`（启动 Agent 进程）
4. `InitializingSession`（初始化会话）
5. `StreamingTurn`（流式轮次）
6. `Finishing`（完成中）
7. `Succeeded`（成功）
8. `Failed`（失败）
9. `TimedOut`（超时）
10. `Stalled`（停滞）
11. `CanceledByReconciliation`（被协调取消）

不同的终止原因很重要，因为重试逻辑和日志会有所不同。

### 7.3 转换触发器

- `Poll Tick`（轮询周期）
  - 协调活跃运行。
  - 验证配置。
  - 获取候选问题。
  - 调度直到槽位耗尽。

- `Worker Exit (normal)`（工作进程正常退出）
  - 移除运行条目。
  - 更新总运行时间。
  - 在工作进程耗尽或完成其进程内轮次循环后，安排继续重试（尝试 `1`）。

- `Worker Exit (abnormal)`（工作进程异常退出）
  - 移除运行条目。
  - 更新总运行时间。
  - 安排指数退避重试。

- `Codex Update Event`（Codex 更新事件）
  - 更新实时会话字段、Token 计数器和速率限制。

- `Retry Timer Fired`（重试计时器触发）
  - 重新获取活跃候选并尝试重新调度，若不再符合条件则释放声明。

- `Reconciliation State Refresh`（协调状态刷新）
  - 停止问题状态已终止或不再活跃的运行。

- `Stall Timeout`（停滞超时）
  - 终止工作进程并安排重试。

### 7.4 幂等性与恢复规则

- 编排器通过单一权威序列化状态变更，以避免重复调度。
- 启动任何工作进程前必须检查 `claimed` 和 `running` 状态。
- 每次周期中，协调在调度之前执行。
- 重启恢复由跟踪器和文件系统驱动（无需持久化编排器数据库）。
- 启动时的终止清理会移除已处于终止状态问题的陈旧工作区。

## 8. 轮询、调度与协调

### 8.1 轮询循环

启动时，服务验证配置、执行启动清理、安排立即执行一次周期，然后每 `polling.interval_ms` 重复一次。

当工作流配置重新应用时，有效轮询间隔应随之更新。

周期序列：

1. 协调运行中的问题。
2. 执行调度预检验证。
3. 使用活跃状态从跟踪器获取候选问题。
4. 按调度优先级对问题排序。
5. 在槽位剩余时调度符合条件的问题。
6. 通知可观测性/状态消费者状态变更。
