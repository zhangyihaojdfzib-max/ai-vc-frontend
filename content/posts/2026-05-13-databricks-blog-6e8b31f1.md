- **它是什么：** 站点可行性工作台是一个开源的 Databricks 应用，完全在 Databricks 工作区内运行临床试验站点选择——结合了机器学习驱动的站点评分、用于操作状态的 Lakebase，以及用于自然语言数据访问的 AI/BI Genie，无需外部 API 调用或同步管道。
- **它解决的挑战：** 37% 的研究者站点未能达到入组目标，其根本原因在于架构问题——临床操作数据及其使用的应用程序存在于相互隔离的系统中，迫使决策在电子表格中进行，并造成了集成开销、凭证蔓延和同步延迟，从而削弱了对数据的信任。
- **结果与成效：** 基于您自己的 CTMS、EDC 和 IRT 历史数据（而非行业平均值）训练的、按治疗领域分组的 LightGBM 模型，会随着您的项目组合增长而生成更优的评分，并附带以受治理、版本化的 Delta 表形式存储的 SHAP 驱动解释。每个预测都带有以受治理的 Delta 表形式存储的 SHAP 归因，使得模型原理与评分本身一样可审计且可版本化。

临床数据问题并非存储问题。大多数组织已经拥有数据仓库、CTMS、EDC，以及下游的某个 BI 层。问题在于，这些系统之间无法以支持临床团队所需实际决策的方式进行有效沟通——因此决策最终只能在电子表格中完成。

今天，我们发布了站点可行性工作台，作为一个完全开源的 Databricks 应用——旨在展示当应用程序、模型和数据位于同一平台时，临床运营智能会是什么样子。塔夫茨药物开发研究中心已记录到，37% 已激活的研究者站点入组患者数低于目标，另有 11% 的站点完全没有入组患者——综合效应是，53% 的试验超出了计划的入组时间线，其中六分之一的试验耗时超过计划的两倍（Lamberti 等人；后续的 CSDD 影响报告继续追踪到类似水平的低效表现）。每天高达 50 万美元的未实现药品销售额和 4 万美元的直接试验成本，使得站点长期低效成为药物开发中后果最严重的成本驱动因素之一。这种综合低效表现率在过去至少二十年里基本保持不变。问题不在于工具，而在于架构。

临床运营团队不需要更多连接到现有系统的仪表板。他们需要的是决策支持应用程序与其数据和模型位于同一位置——这样，预测与验证该预测的操作结果之间的反馈循环才能真正闭合。

## 架构论点

传统的临床决策支持方法如下：分析数据存在于数据仓库或 Lakehouse 中。一个单独的应用程序数据库保存操作状态。一个管道使它们保持松散同步。一个 Web 应用程序位于两者之前，在 Silver 层添加语义协调。每一层都会引入集成开销、凭证暴露面，以及削弱应用程序所展示数据信任度的同步延迟。

Databricks Apps、Lakebase 和 AI/BI Genie 消除了这些层中的每一个——不是通过抽象化，而是使其变得不再必要。

Databricks Apps 在工作区内运行 Web 应用程序。该应用作为一等工作区服务主体进行身份验证，通过 SQL Statement API 查询 Unity Catalog，并通过工作区 REST API 调用 AI/BI Genie——全部通过内部连接。临床操作数据永远不会跨越工作区边界。该应用无需任何额外配置即可继承 Unity Catalog 的访问控制。

Lakebase 是操作数据库层——托管的 PostgreSQL，空闲时可缩减至零，完全在工作区身份系统内进行配置和凭证管理。传统应用程序需要一个单独管理的 RDS 实例，伴随其自身的模式漂移、同步任务和凭证轮换，而 Lakebase 则与数据和模型位于同一平台。

AI/BI Genie 填补了最后一项空白：对受治理数据的自然语言访问，直接嵌入到应用程序工作流中。研究经理可以用通俗英语针对 ML 模型训练所依据的同一 Unity Catalog 表提问，并应用相同的访问控制。

其结果是一个无需外部 API 调用、无需维护单独的操作数据库基础设施、也无需在分析和操作层之间建立同步管道的临床运营应用程序。

## 可审计性论点

行业标准的站点可行性方法依赖于供应商的商业评分产品或 CRO 提供的分析平台。这些工具基于聚合的行业数据构建——作为基线有用，但对您项目组合的具体情况视而不见。拥有十年 CTMS、EDC 和 IRT 历史数据的申办方，携带着关于其站点在其方案上表现如何的重要信号。

当机器学习技术栈位于 Databricks 上时，这种机构知识就成为了训练数据。本工作台中的模型是基于您自己的历史入组率、站点资质历史、筛选失败模式和方案执行记录进行训练的——而非行业平均值。CMS 开放支付数据增加了一个公共信号层，在适当使用时，该信号层与研究参与度和基础设施相关，并且是免费提供的。随着试验项目组合的增长，模型在同一基础设施上会不断改进。这就是单一平台架构所能实现的、而许可评分产品无法做到的复利回报：每项新研究都会使预测更准确，每个新的站点关系都会反映在下一轮训练中。MLflow 跟踪每一次模型训练运行、参数、指标和工件——支持跨模型版本比较、按需重现，以及从原始 CTMS 和 EDC 记录到已部署预测的完整审计追踪。

监管层面在此也很重要。21 CFR Part 11、ICH E6(R3) 和 FDA 的良好机器学习实践（GMLP）指南，以及 FDA 对算法决策支持透明度日益增长的重视，使得模型可解释性和数据治理成为实质性考量，而非可选功能。由于每个预测都带有以受治理的 Unity Catalog Delta 表形式存储的 SHAP 归因——在 MLflow 中版本化，通过 Unity Catalog 进行血缘追踪，可查询——因此站点选择背后的理由与评分本身一样可审计。临床事务团队可以通过 SQL 查询来回答数据监查委员会的问题，而非依赖黑盒供应商报告。

## 我们构建了什么

站点可行性工作台是一个用于临床试验站点选择的六步引导式工作流：方案选择、评分约束、地理概览、站点排名、SHAP 驱动的站点深度分析，以及最终候选名单。多样性考量是一个一等评分维度，符合 FDORA 2022 下 FDA 的多样性行动计划期望。

综合可行性评分结合了真实世界证据、患者可及性数据、历史站点表现、站点资质历史、开放支付数据的关键意见领袖信号以及方案执行因素——全部由基于组织自身 CTMS、EDC 和 IRT 历史数据训练的、按治疗领域分组的 LightGBM 模型驱动。

值得强调的并非工作流步骤或模型特性。患者级数据继承Unity Catalog的访问控制，且受保护健康信息（PHI）的处理遵循申办方在目录或架构层面配置的HIPAA安全港/专家判定策略。

真正重要的是该架构所实现的能力：每次预测都附带一个SHAP解释，并以受治理的Delta表形式与预测结果一同存储，使模型推理过程与它所解释的评分一样具备可审计性和版本控制。由于每次预测都被分解为受治理的SHAP归因，申办方可以审计推荐结果是否存在对社区站点、少数族裔服务机构或首次研究者的系统性权重不足——从而将可解释性转化为公平性控制。

保存的候选名单持久化至Lakebase供团队共享。AI/BI Genie助手能够以自然语言针对同一Unity Catalog表回答跨领域问题。所有这些均无需工作区之外的基础设施。

这是一个决策支持层，而非记录源系统。CTMS/EDC/IRT仍保持权威地位。工作台生成的预测结果，其数据血缘在Unity Catalog和MLflow中受治理。

## 更大平台中的一个模块

站点可行性工作台是更广泛架构——Databricks临床运营智能中枢——的首个公开发布模块，覆盖完整试验生命周期：

- 站点可行性与选择——本仓库所涵盖的内容
- 患者队列与招募——基于Lakehouse规模，从EHR和真实世界证据中构建符合方案要求的队列
- 入组速度优化器——以1-3个月前瞻周期，按站点每月进行机器学习停滞预测
- 基于风险的监测与合规——持续监测入组异常、数据延迟和方案偏离

所有四个模块均以Databricks应用形式部署。所有模块均直接查询Unity Catalog。均不进行外部API调用。当临床应用与数据和模型位于同一环境时，反馈闭环得以形成。站点选择模型从入组结果中学习。风险评分随修订历史增长而更新。每项AI驱动的推荐都带有可追溯至生成它的CTMS、EDC和IRT记录的血缘轨迹。
