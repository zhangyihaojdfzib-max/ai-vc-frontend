# 从文件到分块：提升 Hugging Face 存储效率

Hugging Face 在 Git LFS 仓库中存储了超过 30 PB 的模型、数据集和空间。由于 Git 在文件级别进行存储和版本管理，任何对文件的修改都需要重新上传完整资产——当 Hub 上的平均 Parquet 和 CSV 文件大小在 200-300 MB 之间、平均 Safetensor 文件约 1 GB、而 GGUF 文件可能超过 8 GB 时，这种操作成本高昂。想象一下，仅修改 GGUF 文件中的一行元数据，就需要等待数 GB 的文件上传；除了用户时间和传输成本外，Git LFS 还需要保存两个文件的完整版本，从而增加存储成本。

Hugging Face 的 Xet 团队采用了一种不同的存储方法，将文件以分块形式存储。通过仅传输修改过的分块，我们可以显著提升存储效率和迭代速度，同时确保对不断变化的数据集和模型的可靠访问。以下是其工作原理。

## 内容定义分块基础

我们用于对文件进行分块的方法称为内容定义分块（CDC）。CDC 不是将文件视为不可分割的单元，而是将文件分解为可变大小的分块，利用数据来定义边界。为了计算分块，我们应用一种滚动哈希算法来扫描文件的字节序列。

考虑一个包含以下内容的文件：

```bash
transformerstransformerstransformers
```

我们使用文本进行说明，但这可以是任何字节序列。

滚动哈希算法在数据的滑动窗口上计算哈希值。在这种情况下，窗口长度为 4，哈希值将首先在 `tran` 上计算，然后是 `rans`，接着是 `ansf`，依此类推，直到文件末尾。

当哈希值满足预定义条件时，分块边界被确定，例如：

```
hash(data) % 2^12 == 0
```

如果序列 `mers` 产生的哈希值满足此条件，文件将被分割为三个分块：

```bash
transformers | transformers | transformers
```

这些分块的内容被哈希处理，以创建分块哈希与字节之间的映射，并最终存储在内容寻址存储（CAS）中。由于所有三个分块完全相同，我们只需在 CAS 中存储一个分块，从而实现内置去重。🪄

## 插入与删除

当文件内容发生变化时，CDC 允许进行细粒度的更新，使其能够稳健地处理插入和删除。让我们通过插入 `super` 来修改文件，使新文件内容变为：

```bash
transformerstransformerssupertransformers
```

再次应用相同的边界条件进行滚动哈希后，新的分块如下所示：

```bash
transformers | transformers | supertransformers
```

我们不需要保存之前见过的分块；它们已经存储好了。然而，`supertransformers` 是一个新分块。因此，保存此文件更新版本的成本仅为上传和存储一个新分块。

CDC 将如何应用于 Hugging Face Hub 上存储的文件类型？我们快速构建了一个简单的去重估算工具，以可视化将 CDC 应用于文件集合时潜在的存储节省。在 `openai-community/gpt2` 仓库的提交历史中，对 `model.safetensors` 文件的两个版本运行此工具，得到以下结果：

绿色部分反映了两个版本之间的显著重叠，因此有机会在文件内部以及版本之间进行去重。

在这种情况下，使用基于 Xet 的存储后端将显著节省第二个版本的上传/下载时间，并将总存储占用减少 53%。通过压缩，我们估计还能额外节省 10%。

我们对 Hub 上仓库的初步研究显示，一些微调模型和许多模型检查点取得了积极结果。微调模型仅修改部分参数，因此模型的大部分内容在版本之间保持不变，使其成为去重的理想候选。模型检查点捕获增量训练状态，也是很好的目标，因为检查点之间的变化通常很小。两者的去重比率在 30-85% 之间。PyTorch 模型检查点约占 Hub 总存储的 200 TB。以 50% 的去重率计算，我们将立即节省多达 100 TB 的存储空间，并且未来每月可节省约 7-8 TB。

除了降低存储成本，分块级去重还提升了上传/下载速度，因为仅传输修改过的分块。这对处理多个版本模型或数据集的团队来说是一个巨大优势，因为它最大限度地减少了用户和机器的等待时间。
