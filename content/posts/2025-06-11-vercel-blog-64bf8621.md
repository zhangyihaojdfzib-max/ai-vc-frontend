我们的两场大会（Vercel Ship 和 Next.js Conf）是我们展示最新成果、分享思考方向、并展望未来愿景的契机。

这也是一个推动自我、打造令人兴奋的体验、并体现我们在产品中追求品质的机会。对于 Vercel Ship 2025，我们希望这种体验能够**流畅且快速**。

本文介绍了我们如何打造大会平台和视觉设计，从磁流体启发的 3D 视觉效果和生成式 AI 工作流，到模块化组件系统等等。

## 概念演变

Vercel Ship 2024 的视觉方向是磁性粒子。随后，我们推出了**Fluid compute**，让粒子动了起来。对于 Ship 2025，我们希望更进一步。

我们探索了**磁流体**，一种由 NASA 开发的黑色磁性液体。它完美融合了运动、磁性，以及表面之下复杂（但之上简洁）的特质。

## 借助 AI 快速迭代

生成式 AI 帮助我们快速制作原型并测试视觉方向。

我们在 Cinema4D、Figma 和 Flux 之间来回切换，以启动视觉形象的创作，快速生成塑造今年品牌基础的草稿。

当我们最初探索磁流体时，其结构化的优雅和响应性感觉非常契合。但在内部评审中，我们发现了一个重大问题。这些视觉效果会引发密集恐惧症的症状，即对密集的凸起或孔洞产生厌恶。因此我们决定转向。

我们探索了全光谱色彩、油污般的质感，甚至回顾了之前 Next.js Conf 的棱镜元素。但视觉效果开始偏离方向，过于花哨、闪亮和喧闹。

于是我们收敛了。我们已经训练了一个模型来输出一致的照明、形态和材质。这为我们提供了一个稳定的基础，可以在不从头开始的情况下进行演变。我们柔化了形状，去掉了尖刺。最终的效果更暗、更流畅、也更合适。一个近乎金属的液体系统。反射性强且沉稳。

## 使用 v0 进行原型设计

对于网站和注册流程，我们使用了 **v0**，它可以通过简单的文本提示帮助你快速创建想法。一旦在 Figma 中规划好核心结构和功能，直接进入代码阶段就顺理成章了。

我们直接在 **v0** 中提示了注册、日程和画廊交互的关键部分，并与我们的设计系统保持一致。由于我们在第一天就发布了实时原型，我们可以快速分享、收集反馈并进行迭代。

在设计团队内部拥有这种原型设计能力是一个巨大的突破。它节省了时间，并使交接更高效。设计工程团队从真实的组件开始，而不是从设计稿开始。当他们介入时，结构、布局和交互已经设定好，只等他们进行最后的魔法润色。

## 简化技术栈

大会网站可能出奇地复杂。内容需要能够按需编辑，无需重新部署。注册数据需要可查询。一切都需要感觉快速。

### 从过去学习

在过去几年中，我们通过混合使用外部工具和内部缓存层来支持这些需求。这完成了任务，但维持这种平衡变得越来越困难。

为了减少延迟，我们首先尝试引入 **Redis** 作为快速的辅助数据层。它在关键领域提升了性能，但也带来了新的挑战：数据重复、额外的更新步骤、有限的查询能力，以及对高优先级信息（如促销代码或工作坊容量）的编辑依赖于工程团队。

### 面向未来的新基础

今年，我们退后一步，进行了简化。我们将网站构建为一个 **Next.js** 应用，由 **Postgres** 数据库支持，并使用 **Payload** 作为 CMS。由于完全掌控了 API 和数据库，我们可以根据实际工作方式来塑造系统。

以前，内容在一个地方，注册者数据在另一个地方，工程团队有自己的模型。Payload 让我们将这些关注点统一到一个单一事实来源中：

-   一个用于编辑落地页内容的 CMS
-   一个可搜索、实时的注册者数据库
-   一个工程团队可以直接构建的后端，无需同步逻辑或重复模型

结果是一个团队可以独立工作的系统。活动团队可以在没有工程参与的情况下管理容量、内容和促销代码。开发者可以在不等待上游 API 的情况下优化性能。

### 渲染策略与缓存

这个技术栈给了我们更多控制权，但我们仍然需要刻意追求速度。我们对 CMS 驱动的页面使用了**增量静态再生 (ISR)**，并通过按需缓存失效与更新关联。**部分预渲染 (PPR)** 让我们可以在构建时生成大部分首页，同时保持 CTA 和登录状态等元素为动态。

我们使用了`use cache`指令和`cacheLife`来根据内容类型调整新鲜度。例如，我们通过较短的缓存生命周期保持票务可用性实时更新，同时允许变动较少的内容缓存更长时间。`after()` 处理了结账后的逻辑，如发送确认邮件和记录注册信息，而不会阻塞用户流程。

### Link to heading在Vercel平台上自研自用

Payload管理面板与Vercel和Ship网站位于同一域名下，这使得调试和日志记录更加高效。通过Vercel登录功能，我们无需额外设置即可获得基于账户的安全访问。我们的后端位于单体仓库中，因此可以直接在CMS中使用共享UI资源，如标志或设计令牌。

这保持了体验的一致性，简化了设置，并促进了跨团队协作。

### Link to heading模块化、可交付的组件

为了支持每年多个活动，我们使用`shadcn`、`Geist`和`cva`构建了一个可复用、可定制主题的组件库。这让我们能够快速迭代，同时保持各属性间的一致性。

对于事务性邮件，我们使用了`React Email`并将其直接集成到CMS中。市场团队可以实时预览和编辑内容，无需工程支持。

这些共享系统使品牌、工程和市场团队能够更轻松地在不同场景下协作，帮助我们加快速度，同时保持所有内容符合品牌规范。

### Link to heading挑战与未来改进

管理自己的数据库为我们提供了灵活性，但也暴露了本地工具的一些局限性。在本地搭建Postgres、运行迁移和填充数据增加了处理UI更改或原型设计时的开销。种子脚本有所帮助，但本地开发者体验尚未达到理想状态。

我们现在正专注于自动化更多设置流程。包括在本地环境中预览真实数据、简化模式更新，以及改进动态内容（如Open Graph生成和活动预览）的工作流。

在前端，我们继续探索如何在不牺牲速度或性能的情况下呈现更多视觉细节。像`Motion`和`Three.js`这样的工具帮助我们推动了今年视觉效果的发展。

## Link to heading滚动动画与固定头部

我们希望网站的动效和外观一样流畅。其中一个亮点是“你可以期待什么”部分，它使用了来自Motion for React的`useTransform`构建。固定容器和全屏部分让我们能够控制滚动行为，包括一个需要一些有趣数学计算才能实现的定制缩放效果。`v0`帮助我们快速原型化逻辑并调整感觉。

另一个细微之处是固定头部。为了确保它在亮色和暗色部分都能适应，我们测试了多种方案，包括`clip-paths`。最终，我们采用了一个更简单的解决方案，使用`mix-blend-mode`。
