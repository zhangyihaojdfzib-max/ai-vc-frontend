# Skipper：构建Airbnb的嵌入式工作流引擎

收听

Airbnb如何构建轻量级工作流引擎以解决持久化执行问题。

作者：Ricardo Gamba，Andriy Sergiyenko

## 引言：持久化执行问题

设想这样一个假设流程：房东向Airbnb提交关于其房源的保险索赔。系统需要验证索赔、运行信任与安全检查、评估估算、处理付款并发送通知。流程进行到一半——验证通过后，但付款前——服务器崩溃了。

接下来会发生什么？

在传统架构中，答案往往是"视情况而定"。可能操作超时，客人重试，导致重复处理。可能部分状态破坏了后续流程。而对于跨越数分钟、数小时甚至数天的工作流，例如我们的保险索赔示例，中断是极有可能发生的。

业界已为此开发了解决方案，例如专用编排集群和云管理工作流服务。然而，这些解决方案本身也有代价：运维复杂性、基础设施依赖以及架构约束。

我们需要持久化执行，但无需额外开销。本文介绍我们如何构建Skipper，这是一个嵌入式工作流引擎，目前为Airbnb的保险、支付、媒体处理和基础设施自动化等关键工作流提供支持。

## 为什么现有解决方案存在不足

在构建Skipper之前，我们广泛评估了现有工作流解决方案。每种方案都有优点，但没有一种符合我们的特定约束。

## 外部编排问题

外部编排引擎是持久化工作流执行的行业黄金标准，提供精确一次语义和经过实战检验的可靠性。然而，它们需要专用基础设施——一组服务器和持久化层，以及运维专业知识——来维护。对于我们最高关键性的"Tier 0"服务（直接影响用户交易的服务），增加新的关键依赖是有问题的。编排集群宕机意味着每个依赖服务都将失去启动或推进工作流的能力。

云管理工作流服务消除了运维开销，但引入了供应商锁定、超出我们自身基础设施的监管和数据处理要求、执行限制，以及同样根本性的问题：关键外部依赖。

与此同时，自研的基于队列的系统避免了外部依赖，但换来了定制化的复杂性：每个团队都要实现和维护自己的重试逻辑、状态管理和补偿流程。

## 领域逻辑问题

除了基础设施问题，我们还注意到一个更微妙的问题；这并非Airbnb独有，而是行业普遍模式。当团队使用队列或临时异步管道来编排多步骤流程时，领域逻辑最终变得支离破碎。单个业务工作流，例如处理保险索赔，被分散到队列消费者、定时任务、回调端点和对账脚本中。代码中没有一个地方能让你读到业务流程实际做了什么。更糟糕的是，这些碎片中的每一个都将领域规则与基础设施关注点纠缠在一起：重试退避、去重检查、超时处理、异步协调。

## 为什么我们构建了Skipper

在Airbnb，各个团队在尝试解决持久化执行问题时都独立遇到了同样的问题。在每种情况下，工程师都构建了定制化的解决方案。这些方案虽然有效，但存在相同的问题：构建成本高、测试困难，并且每个方案都重新发现了关于幂等性、部分失败和崩溃恢复的相同边界情况。我们反复为解决持久化执行问题付出代价，而每个新实现都带有自己的一套微妙缺陷。

我们退后一步，从整体角度审视这个模式。我们决定构建一个共享库，而不是让每个服务都重新发明轮子。Skipper是一个工作流引擎，可以直接嵌入到任何服务中，具有简洁的人体工程学设计、无外部运行时依赖，并允许开发者专注于编写领域逻辑而非管道代码。

Skipper不强制架构纯洁性，但其编程模型促进了这一点。通过将工作流和操作暴露为普通的Java/Kotlin类，并采用基于注解的最小化契约，Skipper使开发者能够编写看起来像业务逻辑的代码，而不是框架样板代码。

诸如条件等待、持久化重试和信号等原语都是可用的，但它们通过同样直接的类结构呈现，而不需要开发者学习单独的执行模型或配置基础设施抽象。契约不会碍事：工作流方法读起来就像它所代表的过程，操作方法看起来就像它所包装的服务调用。而且，由于领域规则凝聚在单个类中，而不是分散在基础设施组件中，端到端测试工作流变得简单直接。

以下是实际效果：一个持久化的多步骤业务流程表示为单个工作流类，副作用隔离在操作后面，只需一个注解：

```
// 1) 像普通类型化调用一样调用它（无需代码生成客户端）
val out = workflow<ChargeAndAccept>("reservation:${req.id}").execute(req)

// 2) 将工作流逻辑定义为普通的Kotlin代码
class ChargeAndAccept : Workflow() {
  private val billing = actions<BillingActions>()
  private val reservations = actions<ReservationActions>()
  @StateParam var paymentCaptured = false

  @WorkflowMethod
  suspend fun execute(r0: Reservation): Reservation {
    val r1 = billing.charge(r0)                // 持久化副作用边界
    waitUntil { paymentCaptured }              // 持久化等待（重启后恢复）
    return reservations.markAccepted(r1)
  }
}

// 3) 副作用存在于Actions中；一个注解使其可检查点
class BillingActions : Actions() {
  @Execute(checkpoint = true)
  suspend fun charge(r: Reservation): Reservation =
    billingApi.chargeAsync(r.id, r.amount).await()
}
```

Skipper并非第一个提供"编写普通代码，获得持久化执行"的系统——其他工作流引擎也能做到——但Skipper的重点是消除采用摩擦：更少的必要构造和更少的设置，使得使用Java/Kotlin的团队能够以最少的仪式获得第一个持久化工作流。

## 我们的需求

通过这些评估，我们明确了需求：

1. 无新的关键依赖：Tier 0服务不能增加单点故障
2. 利用现有基础设施：使用服务已依赖的数据库
3. 自助集成：使团队无需专门支持即可采用Skipper
4. 简单的编程模型：支持快速开发和易于维护
5. 性能中立：工作流引擎不应限制宿主服务的可扩展性

这些需求指向了嵌入式架构：一个在每个服务内部运行的库，而不是中央编排系统。

## 设计理念

五项原则指导了Skipper的设计。

简洁的人体工程学。工作流代码应该读起来像它所代表的业务逻辑。一个"等待审批，然后处理付款，然后发送确认"的工作流在代码中应该看起来就是那样。

无单点故障。Skipper嵌入在每个宿主服务中运行。如果一个服务的工作流处理失败，其他服务独立继续运行。没有中央协调器会导致整个系统崩溃。

利用现有依赖。Skipper将其状态存储在宿主服务已使用的同一数据库中：MySQL或Airbnb内部的统一数据存储。无需管理单独的持久化层。

## 自助就绪
Skipper 是一个库依赖：将其添加到你的构建中，提供一些配置，然后开始定义工作流。无需复杂设置，无需依赖中心团队。

## 性能中性
Skipper 使用独立的线程池、可配置的并发限制以及高效的休眠模式，与对延迟敏感的处理请求和平共存。

## Skipper 的工作原理
Skipper 的编程模型围绕两个抽象概念：**工作流**和**操作**。工作流定义编排逻辑：按什么顺序发生什么，以及在什么条件下发生。操作封装单个操作，例如 API 调用、数据库更新、通知。每个操作都会自动设置检查点，因此操作的结果能在崩溃和重启后存活。

## 一个实际的工作流
在 Airbnb 平台上，房东提交其房源照片，这些照片在上线前需要经过审核流程。在这个虚构的例子中，房东提交照片以进行质量和准确性审核，然后房源被激活；接着，如果一切正常，房东会收到通知。这个简化的虚构工作流展示了 Skipper 的核心概念：

```kotlin
class ListingPublicationWorkflow : Workflow() {
    private val actions = actions<ListingActions>();
    @StateField val photosApproved: Boolean? = false;
    @WorkflowMethod
    suspend fun publishListing(submission: ListingSubmission): PublicationResult {
        // 提交照片进行审核
        val reviewId = actions.submitPhotosForReview(submission.getListingId());
        // 等待照片审核完成（手动或自动）
        val reviewTimedOut = waitUntil(() -> photosApproved != null, Duration.ofHours(24));
        if (reviewTimedOut || !photosApproved) {
            actions.notifyHost(submission.getHostId(), "照片需要更新");
            return PublicationResult.rejected("照片审核失败");
        }
        // 发布房源
        actions.activateListing(submission.getListingId());
        actions.notifyHost(submission.getHostId(), "您的房源现已上线！");
        return PublicationResult.success(submission.getListingId());
    }
    @SignalMethod
    fun completePhotoReview(approved: Boolean) {
        photosApproved = approved;
    }
}
```

这段代码读起来很自然：提交照片，等待照片审核，发布。工作流本身没有可见的重试逻辑、队列管理或异步协调。信号（`@SignalMethod`）允许外部事件将数据推入正在运行的工作流，更新 `@StateField` 字段，工作流的 `waitUntil` 条件会针对这些字段进行评估。

## 重放：持久性如何工作
Skipper 通过其带有检查点操作的重放机制实现持久性。当工作流启动时，Skipper 执行工作流方法，并将每个操作的结果检查点到数据库。如果工作流需要等待（通过 `waitUntil`），Skipper 会持久化当前状态，工作流进入休眠状态，不消耗计算资源。

当条件发生变化——信号到达、计时器到期或服务重启——Skipper 从头开始重放工作流方法。先前执行的操作不会重新执行；它们会立即返回其检查点的结果。工作流从它离开的地方继续执行。

与通过重放整个事件历史来重建状态的事件溯源编排系统不同，Skipper 直接持久化状态字段。没有需要重放的事件日志，只有当前状态和检查点的操作结果。这使得执行更精简，特别是对于具有许多信号或长历史的工作流，尽管它为了效率而牺牲了一些可审计性。下一节将解释这在最常见的情况下如何转化为最小的运行时开销。

## 快乐路径：不碍事
大多数工作流引擎在每次执行时都会强加开销，即使一切正常。外部编排引擎需要为每次活动调用向中心集群进行网络往返——工作线程执行活动，然后回调集群以持久化结果，然后工作流才能继续。这对其架构至关重要；集群是协调者。

Skipper 采取了不同的方法。当工作流启动时，在数据库层面会发生两件事：创建工作流实例，并安排一个延迟超时任务作为持久性保证。然后工作流完全在进程内执行。操作作为正常的方法调用在专用线程池上的内存执行队列上运行，检查点被批处理，工作流可以在没有任何进一步协调的情况下运行完成。

延迟任务充当安全网：如果进程在执行过程中崩溃，持久化调度器会在租约期到期后接管工作流并重放它。如果工作流正常完成，超时任务会无害地触发并被丢弃。

结果是，在快乐路径下：工作流运行且所有操作成功，没有崩溃；Skipper 增加的开销非常小（仅几次数据库写入）。工作流的执行几乎就像没有工作流引擎一样。引擎仅在出现问题时才被调用：崩溃触发重放，`waitUntil` 使工作流休眠，或错误触发补偿。

这就是使 Skipper 适用于对延迟敏感、高吞吐量服务的原因——持久性得到保证，但你只在需要时才为其付费。

## 确定性要求
重放施加了一个关键约束：**工作流方法必须是确定性的**。给定相同的输入、检查点的操作结果和状态字段，工作流必须做出相同的决策并按相同顺序调用操作。所有副作用，例如 API 调用、时间相关逻辑和随机性，都属于操作，绝不能直接放在工作流方法中。

## 错误处理与补偿
Skipper 区分可重试错误（临时故障，例如网络超时，会自动重试，并具有可配置的回退）和不可重试错误（永久故障，例如卡被拒绝，会停止工作流的正常流程）。

当工作流中途失败时，你会陷入尴尬状态：某些操作已成功完成，但整个工作流并未完成。例如，一个房源可能通过了内容验证和质量检查，但在照片审核提交时失败。

内容验证结果现在悬而未决；为发布所做的工作现在不会发生。在传统架构中，团队使用临时清理逻辑来处理这种情况：扫描孤立记录的定时任务、每晚运行的核对脚本或手动干预。这些方法很脆弱，通常有延迟，并且随着工作流的演变很容易被遗忘。

我们将补偿机制作为一等公民，防止工作流代码被掩盖业务逻辑的错误处理代码所污染。@Compensate 注解让开发者能够为每个操作配对相应的撤销方法。如果某个操作在之前操作成功后失败，Skipper 会自动按逆序执行补偿方法（释放已占用的库存、退款、恢复状态变更），将系统回滚至一致状态。开发者只需定义每个操作的“撤销”逻辑；Skipper 负责编排撤销操作的执行时机与顺序。最终实现无需分布式事务的最终一致性，且工作流代码始终聚焦于业务流程而非清理编排。

## 关键权衡

## 我们的收获

无需管理基础设施：Skipper 运行在您的服务内部。无需部署、监控或值守单独的集群。

复用现有依赖：如果您的服务依赖 MySQL，Skipper 即使用该 MySQL 实例；如果使用 Airbnb 的统一数据存储，Skipper 也直接使用。无需引入新的数据存储或故障模式。

简洁的编程模型：工作流即 Java/Kotlin 类，操作即方法调用。开发者使用熟悉的工具和调试流程。

独立扩缩容：每个服务独立管理自身的工作流处理。某个服务的高负载不会影响其他服务。

## 我们付出的代价

确定性要求：重放模型要求工作流方法具有确定性，这对不熟悉该模式的开发者可能不够直观。

至少一次执行：在边缘情况下（执行后但检查点前崩溃），操作可能执行多次。操作应具备幂等性。

演进复杂性：修改工作流结构可能破坏正在执行的工作流。团队需要为工作流演进制定版本管理策略。

这些权衡是嵌入式模型的固有特性；需要跨语言支持或跨服务编排的团队，可能更适合采用专用编排系统。

## 生产环境效果

Skipper 已在生产环境运行超过一年，支撑保险、支付、媒体、基础设施、激励和钱包团队的 15 多个用例。用例涵盖多步骤理赔处理、保单生命周期管理、弹性事务编排，以及跨数天或数周的定时金融操作。媒体基础团队使用 Skipper 协调视频处理流水线——验证、转码、缩略图生成——在跨数小时的任务中应对 Pod 重启。基础设施团队依赖它实现持久的 Flink 作业生命周期管理和可靠的数据管道 CRUD 操作。在所有场景中，Skipper 确保每个工作流都能达到终态，即使遭遇基础设施故障、部署和中断。在峰值时，Skipper 凭借其精简的执行模型，在 Amazon DynamoDB 上实现了每秒 10,000 个工作流的处理能力。

## 经验教训

行之有效的做法。嵌入式模型显著降低了运维负担。团队无需新增基础设施、部署流程或值班轮换即可采用 Skipper。支持多种存储后端（MySQL 和内部 UDS）意味着没有团队会因数据库不兼容而受阻。简单的 API（“操作已检查点，工作流必须具有确定性”）加速了学习过程，工作流代码读起来就像直白的业务逻辑。

值得反思之处。工作流演进仍是最大的痛点。虽然我们有版本管理模式（创建新方法版本、迁移流量、废弃旧版本），但更好的工具——自动化兼容性检查、迁移助手、运行时版本支持——将优化体验。调试重放的工作流也需要调整思维模式：工程师必须理解日志时间戳和调用序列反映的是重放过程，而非原始执行。更好的可观测性工具，尤其是重放可视化，将大有裨益。

## 结论

持久化工作流执行是可靠分布式系统的基础能力。Skipper 代表了设计空间中的一个特定点：一种嵌入式引擎，用集中式编排换取运维简洁性，在服务内部而非外部运行，使用现有数据库，并提供直观的 Java/Kotlin 编程模型。

这种方法并非适用于所有场景。但对于寻求持久化执行且无需基础设施开销的服务，尤其是那些将最小化依赖视为首要目标的场景，嵌入式模型提供了显著优势。其核心洞察——基于重放的执行与检查点操作相结合，无需协调服务即可实现持久化——不仅适用于 Airbnb 的实现，也可推广至任何构建长时间运行、易出错工作流的场景。
