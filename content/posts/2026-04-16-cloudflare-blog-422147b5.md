# AI Search：为你的智能体打造的搜索原语

2026-04-16

- Gabriel Massadas
- Miguel Cardoso
- Anni Wang

![](/images/posts/8a966e8c714f.png)

每个智能体都需要搜索：编码智能体需要跨代码仓库搜索数百万个文件，客服智能体需要搜索客户工单和内部文档。虽然用例不同，但底层问题是相同的：在正确的时间为模型提供正确的信息。

如果你自己构建搜索功能，你需要一个向量索引、一个用于解析和分块文档的索引管道，以及一套在数据变化时保持索引更新的机制。如果你还需要关键词搜索，那又是另一个独立的索引和顶层的融合逻辑。如果你的每个智能体都需要自己可搜索的上下文，你就需要为每个智能体单独搭建这一切。

AI Search（原 AutoRAG）正是你所需要的即插即用搜索原语。你可以动态创建实例，提供数据，并进行搜索——无论是从 Worker、Agents SDK 还是 Wrangler CLI。以下是我们发布的功能：

- **混合搜索**。在同一查询中同时启用语义匹配和关键词匹配。向量搜索和 BM25 并行运行，结果融合。（我们博客的搜索现已由 AI Search 提供支持。试试右上角的放大镜图标。）
- **内置存储和索引**。新实例自带存储和向量索引。通过 API 直接将文件上传到实例，它们就会被索引。无需设置 R2 存储桶，也无需先连接外部数据源。新的 `ai_search_namespaces` 绑定允许你在运行时从 Worker 创建和删除实例，因此你可以为每个智能体、每个客户或每种语言启动一个实例，而无需重新部署。

现在，你还可以为文档附加元数据，并在查询时利用它来提升排名，并且可以在单次调用中跨多个实例进行查询。

现在，让我们看看这在实践中意味着什么。

## 实践：客户支持智能体

让我们来看一个支持智能体，它搜索两种知识：共享的产品文档，以及每个客户的历史记录（如过去的解决方案）。产品文档太大，无法放入上下文窗口，并且每个客户的历史记录会随着每个已解决的问题而增长，因此智能体需要通过检索来找到相关信息。

以下是使用 AI Search 和 Agents SDK 的实现方式。首先搭建一个项目：

```Shell
npm create cloudflare@latest -- --template cloudflare/agents-starter
```

首先，将一个 AI Search 命名空间绑定到你的 Worker：

```Typescript
// wrangler.jsonc
{
  "ai_search_namespaces": [
    { "binding": "SUPPORT_KB", "namespace": "support" }
  ],
  "ai": { "binding": "AI" },
  "durable_objects": {
    "bindings": [
      { "name": "SupportAgent", "class_name": "SupportAgent" }
    ]
  }
}
```

假设你的共享产品文档存放在一个名为 `product-doc` 的 R2 存储桶中。你可以在 Cloudflare 仪表板中，在 `support` 命名空间下，创建一个由该存储桶支持的一次性 AI Search 实例（命名为 `product-knowledge`）：

这就是你的共享知识库，每个智能体都可以引用的文档。

当客户带着新问题回来时，了解已经尝试过的方法可以节省每个人的时间。你可以通过为每个客户创建一个 AI Search 实例来跟踪这一点。在每个问题解决后，智能体会保存一份关于问题所在及如何修复的摘要。随着时间的推移，这会建立起一个可搜索的过往解决方案日志。你可以使用命名空间绑定动态创建实例：

```Typescript
// 当客户首次出现时，为其创建一个专属实例
await env.SUPPORT_KB.create({
  id: `customer-${customerId}`,
  index_method:{ keyword: true, vector: true }
});
```

每个实例都拥有自己内置的存储和向量索引——由 R2 和 Vectorize 提供支持。实例最初为空，并随着时间的推移积累上下文。下次客户回来时，所有这些内容都是可搜索的。

以下是经过几个客户后，命名空间的样子：

```unset
namespace: "support"
├── product-knowledge     (R2 作为源，所有智能体共享)
├── customer-abc123       (托管存储，按客户)
├── customer-def456       (托管存储，按客户)
└── customer-ghi789       (托管存储，按客户)
```

现在来看智能体本身。它扩展了 Agents SDK 中的 `AIChatAgent`，并定义了两个工具。我们通过 Workers AI 使用 Kimi K2.5 作为 LLM。模型根据对话决定何时调用工具：

```Typescript
import { AIChatAgent, type OnChatMessageOptions } from "@cloudflare/ai-chat";
import { createWorkersAI } from "workers-ai-provider";
import { streamText, convertToModelMessages, tool, stepCountIs } from "ai";
import { routeAgentRequest } from "agents";
import { z } from "zod";

export class SupportAgent extends AIChatAgent<Env> {
  async onChatMessage(_onFinish: unknown, options?: OnChatMessageOptions) {
    // 客户端通过 Agent SDK 的 sendMessage({ body: { customerId } }) 在请求体中传递 customerId
    const customerId = options?.body?.customerId;

    // 当客户首次出现时，为其创建一个专属实例。
    // 每个实例都拥有自己的存储和向量索引。
    if (customerId) {
      try {
        await this.env.SUPPORT_KB.create({
          id: `customer-${customerId}`,
          index_method: { keyword: true, vector: true }
        });
      } catch {
        // 实例已存在
      }
    }

    const workersai = createWorkersAI({ binding: this.env.AI });

    const result = streamText({
      model: workersai("@cf/moonshotai/kimi-k2.5"),
      system: `你是一个支持智能体。在回答之前，使用 search_knowledge_base
        来查找相关文档。搜索结果
        包括产品文档和该客户过去的
        解决方案——利用它们来避免重复失败的修复
        并识别重复出现的问题。当问题
        解决后，在回复前调用 save_resolution。`,
      // this.messages 是完整的对话历史，由 AIChatAgent 在重连时自动持久化
      messages: await convertToModelMessages(this.messages),
      tools: {
        // 工具 1：在一次调用中搜索共享产品文档和该客户的过往解决方案
        search_knowledge_base: tool({
          description: "搜索产品文档和客户历史记录",
          inputSchema: z.object({
            query: z.string().describe("搜索查询"),
          }),
          execute: async ({ query }) => {
            // 始终搜索产品文档；
            // 如果可用，则包含客户历史记录
            const instances = ["product-knowledge"];
            if (customerId) {
              instances.push(`customer-${customerId}`);
            }
            return await this.env.SUPPORT_KB.search({
              query: query,
              ai_search_options: {
                // 让较新的文档比旧的文档排名更高
                boost_by: [
                  { field: "timestamp", direction: "desc" }
                ],
                // 同时跨两个实例搜索
                instance_ids: instances
              }
            });
          }
        }),
```

```javascript
// 工具2：解决问题后，Agent保存摘要，
        // 以便后续Agent拥有完整上下文
        save_resolution: tool({
          description:
            "解决客户问题后保存解决方案摘要",
          inputSchema: z.object({
            filename: z.string().describe(
              "简短的描述性文件名，例如 'billing-fix.md'"
            ),
            content: z.string().describe(
              "问题是什么、原因是什么以及如何解决的"
            ),
          }),
          execute: async ({ filename, content }) => {
            if (!customerId) return { error: "No customer ID" };
            const instance = this.env.SUPPORT_KB.get(
              `customer-${customerId}`
            );
            // uploadAndPoll会等待索引完成，
            // 因此在下次查询前解决方案已可搜索
            const item = await instance.items.uploadAndPoll(
              filename, content
            );
            return { saved: true, filename, status: item.status };
          }
        }),
      },
      // 将Agent工具使用循环限制在10步以内
      stopWhen: stepCountIs(10),
      abortSignal: options?.abortSignal,
    });

    return result.toUIMessageStreamResponse();
  }
}

// 将请求路由到SupportAgent持久对象
export default {
  async fetch(request: Request, env: Env) {
    return (
      (await routeAgentRequest(request, env)) ||
      new Response("Not found", { status: 404 })
    );
  }
} satisfies ExportedHandler<Env>;

```

通过这种方式，模型自行决定何时搜索、何时保存。当它搜索时，会同时查询产品知识和该客户过往的解决方案记录。当问题解决后，它会保存一份摘要，这份摘要在未来的对话中可以立即被搜索到。

## AI Search如何找到您所需的内容

在底层，AI Search运行一个多步骤的检索流程，其中每一步都是可配置的。

### 混合搜索：理解意图并匹配术语的搜索

在此之前，AI Search仅提供向量搜索。向量搜索擅长理解意图，但可能会丢失具体细节。在查询"ERR_CONNECTION_REFUSED timeout"时，嵌入向量捕捉到了连接失败的宽泛概念。但用户寻找的并非通用的网络文档。他们寻找的是提及"ERR_CONNECTION_REFUSED"的具体文档。向量搜索可能会返回关于故障排除的结果，却从未展示包含该确切错误字符串的页面。

关键词搜索填补了这一空白。AI Search现在支持BM25，这是应用最广泛的检索评分函数之一。BM25根据查询词在文档中出现的频率、这些词在整个语料库中的稀有程度以及文档长度来给文档评分。它奖励特定术语的匹配，惩罚常见的填充词，并根据文档长度进行归一化处理。当您搜索"ERR_CONNECTION_REFUSED timeout"时，BM25会找到实际包含"ERR_CONNECTION_REFUSED"这个术语的文档。然而，BM25可能会错过关于"troubleshooting network connections"的页面，即使该页面描述的是同一个问题。这正是向量搜索的优势所在，也是您需要两者结合的原因。

当您启用混合搜索时，它会并行运行向量搜索和BM25，融合结果，并可选择性地进行重排序：

让我们看看BM25的新配置项，以及它们如何协同工作。

1.  **分词器** 控制在索引时如何将文档拆分为可匹配的术语。波特词干提取器（选项：`porter`）对单词进行词干提取，使"running"能匹配"run"。三元组分词器（选项：`trigram`）匹配字符子串，使"conf"能匹配"configuration"。您可以将波特分词器用于文档等自然语言内容，将三元组分词器用于需要部分匹配的代码场景。
2.  **关键词匹配模式** 控制在查询时哪些文档有资格进行BM25评分。`AND`要求所有查询词都出现在文档中，`OR`则包含至少有一个匹配项的文档。
3.  **融合** 控制在查询时如何将向量和关键词结果合并到最终的结果列表中。倒数排名融合（选项：`rrf`）根据排名位置而非分数进行合并，这避免了比较两个不兼容的评分尺度；而最大融合（选项：`max`）则取较高的分数。
4.  **（可选）重排序** 增加一个交叉编码器步骤，通过将查询和文档作为一对进行评估来重新评分结果。它有助于捕捉那些包含正确术语但并未回答问题的结果。

每个选项在省略时都有合理的默认值。您可以在创建新实例时灵活配置所需的部分：

```javascript
const instance = await env.AI_SEARCH.create({
  id: "my-instance",

  },
  retrieval_options: {
    keyword_match_mode: "or"

```

### 提升相关性：突出重要内容

检索能为您提供相关结果，但仅有相关性有时并不足够。例如，在新闻搜索中，一篇上周的文章和三年前的文章在语义上都可能与"选举结果"相关，但大多数用户可能想要最近的那一篇。提升功能允许您通过基于文档元数据微调排名，在检索之上叠加业务逻辑。

您可以基于时间戳（每个项目内置）或您定义的任何自定义元数据字段进行提升。

```javascript
// 提升高优先级文档
const results = await instance.search({
  query: "deployment guide",
  ai_search_options: {
    boost_by: [
      { field: "timestamp", direction: "desc" }
    ]
  }
});

```

### 跨实例搜索：跨越边界查询

在支持Agent的示例中，产品文档和客户解决历史记录是设计上存放在不同实例中的。但当Agent回答问题时，它需要同时从这两个地方获取上下文。如果没有跨实例搜索，您就需要进行两次单独的调用并自行合并结果。

命名空间绑定暴露了一个`search()`方法为您处理此事。传入一个实例名称数组，即可获得一个统一的排名列表：

```javascript
const results = await env.SUPPORT_KB.search({
  query: "billing error",
  ai_search_options: {
    instance_ids: ["product-knowledge", "customer-abc123"]
  }
});

```

结果会在各个实例间合并和排名。Agent无需知道或关心共享文档和客户解决历史记录存放在不同的地方。

## AI Search实例如何工作

到目前为止，我们已经介绍了AI Search如何找到正确的结果。现在让我们看看如何创建和管理您的搜索实例。

如果您在此版本发布前使用过 AI Search，应该熟悉其设置流程：创建一个 R2 存储桶，将其链接到 AI Search 实例，AI Search 会为您生成一个服务 API Token，然后您需要管理在您账户上配置的 Vectorize 索引。上传对象需要您先写入 R2，然后等待同步作业运行以完成对象索引。

现在创建的新实例工作方式有所不同。当您调用 `create()` 时，实例会自带内置的存储和向量索引。您可以上传文件，文件会立即发送进行索引，并且您可以通过一个 `uploadAndPoll()` API 来轮询索引状态。完成后，您可以立即搜索该实例，无需连接任何外部依赖项。

```javascript
const instance = env.AI_SEARCH.get("my-instance");

// 上传并等待索引完成
const item = await instance.items.uploadAndPoll("faq.md", content, {
  metadata: { category: "onboarding" }
});
console.log(item.status); // "completed"

// 索引完成后立即搜索
const results = await instance.search({
  // 除了使用参数 query 外，另一种传入用户查询的方式
  messages: [{ role: "user", content: "onboarding guide" }],
});

```

每个实例还可以连接一个外部数据源（一个 R2 存储桶或一个网站）并按同步计划运行。它可以与提供的内置存储并存。在客服 Agent 示例中，`product-knowledge` 由 R2 存储桶支持，用于存储共享文档，而每个客户的实例则使用内置存储来存放实时上传的上下文。

### 命名空间：在运行时创建搜索实例

`ai_search_namespaces` 是一个新的绑定，您可以用它在运行时动态创建搜索实例。它取代了之前通过 `AI` 绑定访问 AI Search 的 `env.AI.autorag()` API。旧的绑定将通过 Workers 兼容性日期继续工作。

```javascript
// wrangler.jsonc 
{
  "ai_search_namespaces": [
    { "binding": "AI_SEARCH", "namespace": "example" },
  ]
}

```

命名空间绑定为您提供了命名空间级别的 API，例如 `create()`、`delete()`、`list()` 和 `search()`。如果您需要动态创建实例（例如，每个 Agent、每个客户、每个租户），这就是要使用的绑定。

```javascript
// 创建一个实例
const instance = await env.AI_SEARCH.create({
  id: "my-instance"
});

// 删除一个实例及其所有索引数据
await env.AI_SEARCH.delete("old-instance");

```
