# 通过OAuth解锁Cloudflare应用生态系统，面向所有用户

![](/images/posts/954117940365.png)

Cloudflare提供的服务支撑着20%的互联网运行，但我们并非独自完成这一切。我们平台上的开发者也在使用来自其他公司的众多工具和服务。Cloudflare为我们的平台提供了丰富的API，使开发者能够创建自动化、CI/CD以及集成方案，将基础设施的各个部分连接起来。本月早些时候，我们宣布了自助管理OAuth，让客户能够更轻松地创建和管理自己的OAuth客户端，以实现对Cloudflare API的委托访问。

Cloudflare对OAuth并不陌生。如果你使用过Wrangler，或使用过PlanetScale等合作伙伴的集成功能，那么你已经体验过它。然而，在此之前，第三方OAuth仅通过少量手动接入的集成方式提供，并未广泛开放给开发者。这意味着构建自有集成的开发者不得不依赖API令牌，而API令牌更难管理，且不适合许多委托应用流程。

在过去一年中，我们逐步接入了越来越多的早期合作伙伴，同时改进了Cloudflare OAuth背后的授权、撤销和安全模型。但随着我们的开发者平台不断壮大，以及Agent（智能体）工具对委托访问的需求日益增长，显然向所有客户开放OAuth对我们平台的成功至关重要。

借助自助管理OAuth，开发者现在可以提供标准的OAuth流程，让客户直接授予限定范围的访问权限，从而更轻松地构建SaaS集成、内部开发者平台和Agent（智能体）工具，同时让用户获得更清晰的授权、更便捷的撤销方式，以及对应用权限的更强控制。

## 安全地扩展生态系统

虽然我们早期的OAuth解决方案足以应对少量精心管理的合作伙伴，但我们意识到，我们的权限模型、授权体验以及缓解潜在滥用途径的方式还不够成熟。

今年早些时候，我们更新了授权体验，使请求访问的应用及其将获得的权限更加清晰。我们还在仪表盘中增加了撤销功能，让开发者能够轻松控制哪些应用可以访问其数据，并提高了应用所有权的可见性，以防止OAuth钓鱼攻击。

向所有客户开放自助管理OAuth还需要对我们底层的OAuth引擎进行重大升级。这一过程需要大量规划，以尽量减少对用户的干扰，同时确保数据的稳定性和安全性。

## 规划OAuth引擎升级

多年前，我们部署了Hydra（一个开源OAuth引擎），为Cloudflare OAuth提供底层支持。在用量有限时，该部署运行良好，但随着开发者平台的发展以及Agent（智能体）工作流变得越来越普遍，显然我们需要一次重大升级，以解锁新功能并提升性能。

在规划升级时，我们决定分两步进行较小的顺序升级，而不是一次性完成大规模升级。首先，我们将迁移到最新的1.X版本，评估任何行为或性能变化，然后再进行2.X升级。

在升级规划过程中，我们发现即使是1.X升级也会对客户产生影响，因为Hydra数据库需要进行大量的模式迁移，具体包括：

1. 创建索引的方式会对关键表施加排他锁，阻止活跃用户执行重要的OAuth操作
2. 向关键表添加列，并将其他列移至新表

此外，我们使用的Hydra版本存在一个特性，即SDK会执行SELECT *操作，导致模式变更时出现反序列化问题。

为避免对用户造成影响，我们重写了SQL迁移，使用CREATE INDEX CONCURRENTLY等功能，并构建了一个自定义版本的Hydra，选择显式列而非SELECT *。

在规划好最新的1.X升级后，我们现在需要为更大的2.X升级制定计划。我们确定了三个潜在方案，并权衡了各自的利弊。由于主版本升级带来了大量模式变更，原地升级对我们来说行不通。我们决定采用蓝绿部署策略，但需要做的工作远不止简单地切换开关以开始使用新版本。升级和迁移过程需要数小时，我们需要系统在此期间继续正常运行。

第一个蓝绿方案涉及禁用数据库写入，阻止任何新的授权操作。这意味着这些操作不会在过渡中丢失，但也意味着除非用户已有有效凭证，否则任何人都无法使用现有的OAuth应用。这还带来了另一个大问题：如果用户因任何原因需要撤销对某个应用的访问权限，在升级期间将无法实现。

为解决这些问题，我们想出了一个办法，即保持数据库写入功能开启，代价是在切换到绿色版本时可能会丢失部分写入。首先要解决的是尽量减少新Token的写入数量。我们采取了一个操作手段：将Token的过期时间延长至数小时。这样，在升级前获得新Token的应用可以继续使用它们，而无需刷新。

在解决了减少写入的问题后，我们需要想出一个办法，确保用户在升级窗口期间执行的撤销操作不会丢失。为此，我们创建了一个队列系统（使用Cloudflare Queues！），在撤销事件发生后，将包含该撤销信息的记录写入队列。这样，在数据库切换到绿色版本后，我们可以清空队列，重放在可能丢失的时间窗口内发生的所有撤销事件。这一点至关重要，否则用户已撤销的应用可能会意外恢复访问权限。

## 执行升级

### 升级到1.X

从操作角度来看，我们首次升级到最新的1.X版本进行得非常顺利，没有出现任何问题。我们自定义的数据库迁移运行速度比预期更快，且未对用户造成影响。我们不得不进行硬切换至新版本，因为旧版本无法检查由新版本创建的Token。

切换后，我们遇到了之前未出现过的刷新Token错误增加的情况。这最终是由于新版本中更严格的刷新失效行为所致：如果刷新Token被重复使用，Hydra会使整个访问和刷新Token链失效。这对Wrangler和MCP客户端来说是个问题。这些客户端都有很高的请求量，单个重复使用的刷新Token会导致整个会话失效。

我们通过在负责将OAuth流量路由到正确目的地的Worker中添加刷新Token合并行为来缓解这一问题。这使我们能够在请求到达Hydra之前短暂缓存刷新Token请求，这样如果检测到重试，我们可以短路该请求并直接响应，而不会使Token失效。幸运的是，Hydra的2.X版本具有可配置的“刷新Token宽限期”，通过允许刷新Token在一段时间内被重试而不使整个链失效，解决了这个问题。

### 升级到2.X

由于长时间对用户产生高影响是不可接受的，我们制定了蓝绿升级策略。从高层来看，这听起来很简单：迁移将在生产数据库的副本上运行，然后在完成后与新版本的 Hydra 一起切换。实际上，涉及更多动态环节：

- 启用撤销重放捕获队列
- 复制并恢复数据库到新目标
- 定向数据清理——现有数据违反了新版本中引入的一些新约束，可能导致迁移失败
- 同时执行 Hydra 服务以及另外两个关键内部系统的切换，以防止任何错误
- 切换后的监控与验证

我们选择了一个 Hydra 每秒请求量最低的升级窗口，以尽量减少丢失的 Token 写入。除了一些超时调优外，我们的生产迁移在新数据库上运行良好：生产环境中的净运行时间约为三小时。迁移完成后，我们谨慎地推出了新版本的 Hydra 服务，并同时调整了两个额外的系统配置，以使我们的系统使用新的 SDK 版本。

在切换流量后不久，我们观察到授权服务（依赖于 Hydra 同意会话 API）中的一项数据清理任务在清除 OAuth 策略数据时过于激进。经过调查，我们发现其中一个 Hydra 迁移存在一个问题，导致某些有效 OAuth 会话的状态被破坏，从而使迁移将其标记为无效。有效会话被破坏导致 Hydra 与我们的授权服务之间出现不一致，表现为 403 错误增加。为了缓解这一问题，我们进行了数据恢复，并开始改进 OAuth 授权行为，以消除对静态策略数据的依赖。

除了数据清理问题外，还有一些由特定客户端行为驱动的小修复，我们迅速完成了这些修复。

随着 Hydra 版本升级完成，OAuth 流量保持稳定，系统性能和可靠性得到了提升，为客户带来了更好的体验。同时，这也使生产环境与我们较新的 OAuth API 在预发布环境中已验证的基础保持一致，为我们在 6 月 3 日发布自管理 OAuth 扫清了道路。

## 面向所有用户的自管理 OAuth

向所有客户开放 OAuth 是迈向更广泛的 Cloudflare 应用生态系统的重要一步。如今，任何 Cloudflare 客户都可以创建自己的 OAuth 应用程序，并在 Cloudflare 之上构建集成。我们非常兴奋地推出面向所有用户的 Cloudflare 自管理 OAuth。
