Apache Spark Structured Streaming 长期以来一直为关键任务的数据管道提供大规模支持，从流式ETL到分析和机器学习。但随着运营用例的发展，团队开始要求更多：为欺诈检测、个性化推荐、异常检测、实时告警和报告等应用提供亚秒级延迟。

历史上，满足这些超低延迟需求意味着需要在 Spark 之外引入专门的系统。随着 Spark Structured Streaming 中实时模式的引入，这种权衡已不再必要。在本博客中，我们将探讨 Spark 如何为特征工程等常见用例简化实时流处理架构，消除长期存在的运营复杂性，并提供行业领先的性能。

## 实时流处理不再需要运行多个不同的系统

实时处理和响应数据的能力现已成为核心需求。现代应用，特别是 AI Agent（智能体），依赖于持续的新鲜上下文流来运作。如果底层数据不完整或滞后，用户体验就会受到影响。实时性能不仅对欺诈检测等传统用例是必需的，对于用户期望获得精确、最新响应的每一次常见交互也是如此。在这种环境下，延迟直接影响收入、客户信任和竞争优势。

构建实时流处理应用的数据团队历来必须管理两个不同的数据处理栈：用于大规模分析的 Apache Spark™，以及用于亚秒级延迟敏感型应用（如 Apache Flink® 或 Kafka Streams）的专门系统。这种割裂要求团队维护重复的代码库、管理独立的治理模型，并聘请专门人才来调优和维护特定于引擎的基础设施。

于 2025 年 8 月公开发布的 Apache Spark Structured Streaming 实时模式旨在消除这种摩擦。通过对 Spark 执行引擎的根本性演进，我们消除了对第二个系统的需求。这一转变使工程师能够使用他们已熟知的同一套 Spark API 来处理从高吞吐量 ETL 到低延迟实时应用的全部用例。这意味着管理基础设施的时间更少，而专注于业务用例的时间更多。

## Spark 现能以毫秒级处理事件；比 Flink 快高达 92%

实时模式引入了一个新的优化执行引擎，使 Spark 能够提供稳定的亚秒级延迟。为了评估性能，我们对 Spark RTM 和 Apache Flink 进行了并行比较。测试基于我们在生产环境中常见的实时特征计算工作负载。这些特征计算模式代表了大多数低延迟 ETL 用例，如欺诈检测、个性化推荐和运营分析。

我们评估了三种常见的特征模式：

-   特征编码（无状态转换）：截断输入行并进行编码
-   特征丰富（通过连接）：将流与静态表连接
-   特征计算（通过聚合）：GroupBy + Count 聚合

结果表明，Spark 演进后的架构提供了与专门流处理框架相当的延迟表现。

![Apache Spark Real-Time Mode vs. Apache Flink](/images/posts/35ce9cb73a78.png)

这一性能得益于 RTM 中的三项关键技术革新：

-   连续数据流：数据在到达时即被处理，而非离散化、周期性的数据块。
-   流水线调度：各阶段同时运行而不阻塞，允许下游任务立即处理数据，无需等待上游阶段完成。
-   流式 Shuffle：数据在任务间即时传递，绕过了传统基于磁盘的 Shuffle 的延迟瓶颈。

这些革新共同将 Spark 转变为一个高性能、低延迟的引擎，能够处理要求最苛刻的运营用例。

## 团队运营更少的基础设施，并通过 Spark 更快地行动

虽然原始速度至关重要，但实时模式的真正价值在于其能够消除通常阻碍构建超低延迟管道的运营复杂性。Spark RTM 通过三个核心优势使您的架构显著简化。为了具体说明，我们在实时机器学习应用的背景下描述这些优势。

最小化训练与推理之间的"逻辑漂移"：实时机器学习（如欺诈检测）需要在高吞吐量批处理（用于模型训练）和低延迟流处理（用于实时推理）之间无缝衔接。Spark 是数据科学家进行模型训练的首选，而强制从 Spark 切换到 Flink 进行推理会产生业务逻辑鸿沟。最终，您会得到用于训练的 Spark 逻辑版本和用于生产的 Flink 完全不同的代码库。这种业务逻辑的复制容易出错并导致逻辑漂移，即您的模型基于一种现实进行训练，却对另一种现实进行评分。使用 Spark RTM，您的转换代码保持相同，使您能够更快、更准确地实现特征生产化。

通过单行代码更改实现按需新鲜度：业务需求很少是静态的。今天以 1 分钟 SLA 开始的特征管道，随着模型对新鲜度的需求演变，明天可能需要亚秒级延迟。相反，对于许多用例，当不需要即时新鲜度时，"降低速度"（例如，每日或每小时批处理）的成本效益要高得多。Spark 提供了与您的产品一同成长和扩展的空间。它使您能够通过单行代码更改轻松调整特征工程策略。例如，您可以将触发器设置为 `AvailableNow` 以按每日或每小时计划运行管道。当业务需求发生变化时，您只需切换到实时模式即可过渡到连续、超低延迟的流处理：`.trigger(RealTimeTrigger.apply())`。相比之下，在 Flink 中实现这一点是一个手动过程。通常需要您调整并行度，并协调计算资源的关闭和重启，仅仅是为了匹配新的处理频率。

加速开发：RTM 建立在您的团队已经熟悉的同一套 Spark API 之上。这消除了维护多个系统的摩擦，使您能够在单一、一致的环境中更快地构建和扩展实时应用。
