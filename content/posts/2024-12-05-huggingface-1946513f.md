## 1. 引言

我并不指望 LLM（大语言模型）能解决重大难题，恰恰相反。我希望它们能处理繁琐事务，即便第一次尝试未能正确完成，只需一句简短的英文提示就能修正。简而言之，我想要一个助手，就像老科幻电影里的计算机那样，只是不要那句“抱歉戴夫，我恐怕做不到”😅。

这篇论文探讨了这样一种用于编程的工具。暂且抛开其夸张的标题不谈（不，AI 尚未击败 Kaggle 大师），论文作者所做的，是将各类 Kaggle 问题手动拆解为微任务，让 LLM（大语言模型）为其生成代码，并不断迭代直至单元测试通过。例如，对于一个图像分类问题，一个微任务可能是“弄清输入数据的格式，并将其重新格式化为包含‘id’、‘image_filename’和‘class’列的 CSV 文件”。

我喜欢这种方法，因为这也是我未来希望借助 AI 开展项目的方式。让 AI 生成那些枯燥的代码片段，比如数据重新格式化，这样我就能专注于有趣的部分：正确界定问题并设计出通往解决方案的步骤。

但这种交互式编程助手必须能够理解用简单英语表达的反馈，并修正其代码中的错误。凭借 LLM（大语言模型）从知识和上下文中推断信息的能力，这可能会成为一种非常高效的计算机交互界面。但如果 LLM（大语言模型）的怪癖，如幻觉或缺乏形式逻辑，成为障碍，我们最终可能得到的不是 AI，而是“人工愚蠢”。

因此，我决定用当今的 LLM（大语言模型）做一个小测试。一个极其简化的测试，看看当你指出错误时，LLM（大语言模型）修正自身错误的效果如何。

## 2. 实验

在回到与 LLM（大语言模型）闲聊的正事之前，先简单介绍一下它的构建方式。

### 3.1 为何选择 TPU？

因为其快速的推理速度和巨大的内存。一个 TPU v5e 2x4 拥有 8 个核心，每个核心 16GB 内存，总计 128GB 内存。有了这么大的内存，只要我们将模型分片到所有核心上，就可以同时加载多个 LLM（大语言模型），并在 UI 中随意切换。在这个实验中，我成功同时加载了五个约 8B 参数的模型（再多一个就会内存不足）和三个约 2B 参数的模型，总共 7 个 LLM（大语言模型）同时驻留在内存中，格式为 bfloat16。

### 3.2 为何选择 JAX 和 Keras？

JAX 是 TPU 上首选的机器学习环境，这得益于其强大的 XLA 编译器。Keras 现在原生运行在 JAX 之上（以及 PyTorch 和 TensorFlow），是我最喜欢的建模环境，并且在其姊妹库 KerasHub 中拥有大量预训练的 LLM（大语言模型）。它甚至可以加载来自 Hugging Face 的特定非 Keras 检查点，这对于比较非常有用。我之前在《Keras 中的 Llama 3.2》一文中写过相关内容。

### 3.3 模型分片？

我使用 Keras 的另一个原因是，它拥有迄今为止最用户友好的模型并行 API。在这里，我想在 TPU 内存中一次性加载尽可能多的模型。为此，模型必须分片到所有 8 个 TPU 核心的内存中。幸运的是，大多数模型都带有默认的布局映射，正是用于此目的。例如：

```
layout_map = keras_hub.models.Llama3Backbone.get_layout_map(device_mesh)

```

关于完整的加载代码以及模型并行的更多背景信息，请参见我之前的文章。在那篇文章中，你还会找到一个代码片段，用于可视化模型加载后实际应用的分片情况。这对调试非常有用。是的，调试和一些布局映射调整是必要的。

### 3.4 选用哪些模型？

在这个实验中，我选择了参数低于 10B 的 LLM（大语言模型），主要是出于实用性考虑，因为可以同时加载多个。同时，实验测试的内容相当简单，这些较小的模型应该能够胜任。所有模型都经过指令微调，以便能够进行对话。你可以在演示的实现中看到它们的对话模板。欢迎复制粘贴代码，用于你自己的 Keras 聊天机器人需求。这些模型来自 Gemma、Llama3、Mistral 和 Vicuna 系列。完整列表请参见下面的结果表格。

## 4. 结果

### 4.1 可靠性

首先，让我们看看我们的 LLM（大语言模型）能否可靠地回答第一个问题。系统提示词和第一个问题“添加一个与 Fred 在 11 月 11 日下午 5 点的会议”重复了五次。

颜色代码：

- A✓ 勾号表示模型产生了预期的输出，即 API 调用 `action.add_calendar_entry("Meeting with Fred", date="2023-11-11", time="17:00", duration=30)`
- A🍄 红色毒蘑菇表示答案基本正确但包含一个错误（例如：日期错误）
- A🔥 垃圾堆表示响应是垃圾信息，没有可识别的 API 调用。

好消息是，有些模型每次都正确完成了，并且所有模型都至少成功用 API 调用（或多或少正确）回答了一次。然而，较小的 1-2B 参数模型和较旧的模型（如 Vicuna）表现挣扎。它们大多数时候响应不佳。

### 4.2 完整对话——修正错误

现在，让我们每次两个模型地运行完整对话。如果模型犯了错误，我会尝试将其引导回正轨。让我们看看是否有效。

- A✔︎ 勾号表示生成了有效的 API 调用
- A🍄 红色毒蘑菇表示模型犯了一个错误
- A🥦 绿色西兰花表示模型在被要求时能够成功修正错误

表格中使用了简化的提示词以节省屏幕空间。第一个问题故意不精确：给出了会议的月份、日期和时间，但没有年份。这是为了确保所有模型至少犯一个错误，并测试它们修正错误的能力。

Gemma2 9B 和 Llama 3.1 8B 都成功了。Llama 需要多一个“修正它”的提示词，但最终获得了它的绿色西兰花🥦。

第三列给出了与 Google 的 Gemini（在线版）的一次运行结果作为对比。这是一个比前两个模型大得多的模型，但令人惊讶的是，它并不是最好的。它需要略微不同的提示词，因为 Gemini 实际上可以将条目添加到你的 Google 日历中，所以每次都必须提醒它“使用提供的 API 中的 API 调用来回答”。即便如此，它还是犯了几个错误，甚至在最后一个提示词中弄错了日期。这表明，对于这个任务，一个巨大的模型并不一定更好。

让我们转向小模型：Llama 3.2 3B、Llama 3.2 1B 和 Gemma 2B。这个练习对这些模型来说似乎极其困难。这里需要新的符号：

- 🔥🔥当回答出现3个及以上错误时，就会变成灾难现场。逐个修正这些错误毫无意义。
- (🍄) 括号中的红色蘑菇表示反复出现的错误，每一行都存在相同问题。

请记住，这些已经是模型的最佳运行结果。正如上方"可靠性"部分所示，有些模型在五次尝试中仅有一次能通过前几个问题。

在小模型中，只有Gemma 2B能完成对话，尽管存在反复出现的错误(🍄)：它无法克制闲聊的倾向，总会在请求的API调用之外添加额外内容，比如"当然，这是更新后的代码……"。它还经常混淆日期和时间。不过，当被要求修正时，它能够改正错误🥦。

最后，让我们尝试一些较旧的模型，如Vicuna 1.5 7B和Mistral 7B。它们将与Codegemma 7B进行对比——后者本应是此任务的理想模型，但如你所见，三个模型都表现挣扎。

Codegemma受到一个顽固的反复错误影响(🍄)：它开始将年份拼写为"20 24"（中间带空格），且无法修正。
Vicuna 1.5 7B可能过于陈旧。在某个时刻它开始重复输出🔥🔥，生成多个重复的API调用和其他垃圾内容。虽然能在一定程度上回归正轨，但仍存在剩余错误。最后，Mistral在各方面都出错，但也能修正这些错误。虽然需要大量交互，但凭借修正的错误获得了6个西兰花🥦。

### 4.3 更多错误修正

部分模型在测试中几乎未出错，因此修正错误和获得西兰花🥦的机会很少。通过使用Keras聊天机器人竞技场界面，我们可以让它们处理其他LLM犯下的错误🍄，观察它们能否修正。

颜色编码与之前相同：绿色西兰花🥦表示正确修正错误，红色毒蘑菇🍄表示错误依然存在，🔥表示多处错误。完整对话记录在此处。

(1) 反复错误：在正确的API调用旁输出道歉内容。对于Gemma，只需要求"仅返回API调用"即可可靠修正。这对Llama也有效，但不够稳定。

这对模型来说是一次很好的现实检验。Gemma 2 9B和Codegemma 7B几乎完全正确，但会为某些错误道歉，而非输出干净的API调用。
Llama 3.1 8B紧随其后，但在可靠修正错误API调用方面存在困难。其余所有模型都是🔥灾难现场。
