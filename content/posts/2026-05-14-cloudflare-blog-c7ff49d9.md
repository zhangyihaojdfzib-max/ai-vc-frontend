# 我们的计费管道突然变慢了。罪魁祸首是 ClickHouse 中一个隐藏的瓶颈

2026-05-14

- James Morrison
- Christian Endres

![](/images/posts/3fedf0174507.png)

在 Cloudflare，我们是 ClickHouse（一个开源分析型数据库管理系统）的重度用户。我们重新设计了最大的 ClickHouse 表之一，在分区键中增加了一个列。这一变更使得服务于数百个内部团队的表能够实现按租户的数据保留策略。该设计经过多轮修订和跨团队工程师的评审后，我们才最终确定了方案。但上线几周后，负责生成 Cloudflare 大部分账单的任务开始逼近其严格的每日截止时间。

所有常见的嫌疑点看起来都正常：I/O、内存、扫描行数、读取的分区数。我们在 ClickHouse 查询变慢时通常会检查的所有指标都显示正常。结果问题出在查询规划中的锁竞争上——这是我们之前从未有理由去查找的问题。

这就是此次迁移如何暴露 ClickHouse 内部一个隐藏瓶颈，以及我们为修复它而编写的补丁的故事。

## 背景：一个 PB 级分析平台

我们使用 ClickHouse 在几十个集群中存储了超过一百 PB 的数据。为了简化众多内部团队的上线流程，我们在 2022 年初构建了一个名为“Ready-Analytics”的系统。

其理念很简单：团队无需设计新表，只需将数据流式传输到一个单一的大表中。数据集通过 `namespace` 进行区分，每条记录使用标准模式（例如，20 个浮点字段、20 个字符串字段、一个时间戳和一个 `indexID`）。

在 ClickHouse 中，数据的排序方式对查询性能至关重要。这就是 `indexID` 发挥作用的地方。它是一个字符串字段，构成主键的一部分，这意味着每个单独的 namespace 都可以按照其所有者预期运行的查询的最优方式来排序数据。总的来说，我们最终得到的主键如下所示：`(namespace, indexID, timestamp)`。

这个系统很受欢迎，有数百个应用程序在使用它。到 2024 年 12 月，它已经增长到超过 2 PiB 的数据，并且每秒摄入数百万行数据。但它有一个关键缺陷：其数据保留策略。

## 问题：统一的数据保留策略

Cloudflare 使用 ClickHouse 已有多年，早在它原生支持 TTL（生存时间）功能之前。因此，我们基于分区构建了自己的数据保留系统。Ready-Analytics 表按 `day` 分区，我们的数据保留任务只是删除超过 31 天的分区。

这种“一刀切”的 31 天保留策略是一个重大限制。一些团队由于法律或合同义务需要将数据存储数年，而另一些团队只需要几天。这一限制意味着这些用例无法使用 Ready-Analytics，而必须选择传统的设置方式，后者的上线流程要复杂得多。

我们需要一个支持**按 namespace 保留数据**的新系统。

## 解决方案：新的分区方案

我们考虑了两种主要方法：

1. **每个 namespace 一个表**：这自然能解决数据保留问题，但需要大量新的自动化工作来按需管理数千个表。
2. **新的分区键**：我们可以将分区键从仅 `(day)` 改为 `(namespace, day)`。

**每个 namespace 一个表**：这自然能解决数据保留问题，但需要大量新的自动化工作来按需管理数千个表。

**新的分区键**：我们可以将分区键从仅 `(day)` 改为 `(namespace, day)`。

我们选择了第二种方案。这将使我们现有的数据保留系统能够继续管理分区，但现在具有按 namespace 的粒度。

我们知道这会增加表中的数据部分总数，但我们做了一个关键假设：**由于每个查询都按特定 namespace 进行过滤，任何单个查询读取的分区数量不应改变**。我们相信这意味着性能不会受到影响。

这展示了我们如何更改分区，从而能够以低成本删除单个 namespace 的数据

这个新系统还使我们能够构建一个复杂的存储管理层。使用 **max-min 公平性算法**，我们可以设定目标磁盘利用率（例如 90%），并自动“共享”可用空间。使用量低于其公平份额的 namespace 会将其未使用的容量让给那些需要更多空间的 namespace。这使我们能够自信地将集群运行在 90% 的利用率上。

我们于 2025 年 1 月开始迁移。利用 ClickHouse 的 `Merge` 表功能，我们合并了旧表和新表，将所有新数据写入新分区的表，同时旧数据逐渐过期。

## 谜团：当计费开始出问题

两个月后，也就是 2025 年 3 月底，我们的计费团队报告他们的每日聚合任务变慢了。这些任务对时间要求严格；如果它们不能完成，账单就无法发出。任务变得越来越慢，我们正逼近截止时间。

我们进行了调查，但常见的嫌疑点都不是原因。I/O 正常。内存正常。单个查询的指标显示它们**并没有**比之前读取更多的数据或分区。我们最初的假设似乎正确，但系统却逐渐陷入停滞。

我们花了几天时间才形成一个理论。最后，我们绘制了查询持续时间与集群中**总分区数**的关系图。相关性是毋庸置疑的。

Ready Analytics ClickHouse 集群上的平均 SELECT 查询持续时间，显示性能逐渐下降。

每个表副本的总数据分区数线性增长，遵循新的 `(namespace, day)` 分区方案。

但是**为什么**？如果我们**没有读取**额外的分区，为什么它们的存在会拖慢我们的速度？

## 调查：使用火焰图寻找瓶颈

我们转向 ClickHouse 内置的 `trace_log` 来生成火焰图。这是一个内置表，用于记录运行中 ClickHouse 服务器的跟踪信息。它不仅包含正在执行的代码的跟踪信息，还将这些信息与特定用户、查询 ID 和其他元数据关联起来，这意味着如果需要，你可以过滤到相当精确的事件集。在我们的案例中，我们想专门查看**叶子 SELECT 查询**。由于该表中提供了可用的元数据，这很容易做到。

第一个基于 CPU 的火焰图迅速证实了我们的怀疑：大量时间花费在**查询规划**上。这是**执行之前**的阶段，ClickHouse 在此阶段决定读取哪些分区。

火焰图显示，叶子查询 CPU 时间的 45% 用于根据分区 ID 过滤分区向量

火焰图很清楚：采样 CPU 时间的 45% 花费在一个名为 `filterPartsByPartition` 的函数中。

我们第一次尝试修复是对这段代码路径做了一个小补丁。规划器会评估启发式方法来修剪分区，我们相信对于我们的表，这些方法没有按最优顺序进行评估。我们的补丁改变了顺序，带来了 5% 的小幅改进。我们走在正确的道路上，但错过了真正的问题。

我们之前一直在生成“CPU”跟踪信息，它只采样活动线程。我们切换到“Real”跟踪信息，它采样**所有**线程，包括那些不活动或等待中的线程。新的火焰图令人震惊。

火焰图显示，超过一半的叶子查询持续时间花费在等待保护活动分区列表的互斥锁上

问题不在于 CPU 密集型工作，而在于**严重的锁竞争**。超过一半的查询持续时间花费在**等待**获取一个保护表分区列表的单一互斥锁（`MergeTreeData`）上。为了规划一个查询，每个线程都必须：

1. 获取这个互斥锁的**排他锁**。
2. 制作表中**所有**分区列表的完整副本。
3. 释放锁。
4. 将该列表过滤到相关的分区。

完整复制表中所有部件的列表。

释放锁。

将该列表筛选至相关部件。

面对数万个部件和数百个并发查询，它们全都排成单列等待。

## 修复方案：三项补丁

这一洞察帮助我们规划了一系列优化措施来缓解这些热点问题。与我们为 ClickHouse 所做的所有补丁一样，我们力求使其具有通用性，并最终将其贡献到上游代码库中。这使我们更容易维护自己的分支，也意味着社区能受益于我们所做的改动！

### 优化 1：使用共享锁

查询规划器不会修改部件列表；它只是读取该列表。它本不该使用排他锁。

修复方案：我们修改了代码，改用共享锁（std::shared_lock）。这使得所有查询规划器能够同时进入临界区。

结果：查询时长立即大幅下降。锁竞争消失了。

共享锁优化（优化 1）对平均 SELECT 查询时长的直接影响，展示了锁竞争的消除。

### 优化 2：停止复制向量

性能显著提升，但仍未恢复到基准水平。我们重新查看了跟踪日志，并制作了另一个“真实”火焰图。

火焰图显示，我们花费了四分之一的时间在复制所有部件的向量上，另外四分之一的时间在筛选该向量（再次复制）。

新的火焰图显示瓶颈只是转移了。现在，即使使用了共享锁，时间也花在了复制巨大的部件向量上。直观上，复制一个向量听起来开销不大，但当它包含数万个元素，并且每秒执行数百次时，开销就会累积。

修复方案：我们完全推迟了复制操作。我们创建了部件列表的“共享副本”。只读操作（如查询规划）只需读取此副本。任何修改部件集合的操作（如新数据插入）都会重新生成缓存。现在，规划器只复制它们实际需要的筛选后的部件列表。

结果：性能再次显著提升。

推出向量复制优化（优化 2）后的进一步性能提升。

在内部看到这些巨大的节省后，我们决定将这些改动带给社区。在与 ClickHouse Inc. 的维护者进行了一些小的设计迭代后，我们通过 PR #85535 合并了这些改动。自 ClickHouse 25.11 版本起，这些改动已可用。

### 优化 3：对部件进行二分查找

我们仍未完成。随着部件数量的增长，性能仍然会下降，只是速度慢得多。与部件数量的相关性依然存在。几个月后重新审视这个问题，一个新的火焰图（与图 3 类似）显示时间花在了筛选代码路径上（我们最初试图修复的那个）。这段代码对所有部件执行线性扫描，对每个部件评估谓词。几个月后，我们的查询时长又回到了优化前的水平。

但我们知道这个部件列表是按分区键排序的。请记住，分区键的第一列是命名空间，绝大多数查询都会对其进行筛选，因为它标识了“租户”。我们如何利用这一点？

修复方案：我们基于分区 ID 的命名空间部分实现了二分查找。这之所以有效，是因为向量是排序的，因此无需实际查看就能过滤掉大量条目。由于命名空间是排序键的第一部分，这种方法尤其有效。经过这第一轮二分查找后，我们需要检查的部件范围就小得多了，对于这些部件，我们仍然逐个检查，应用与之前相同的逻辑，根据其他条件排除部件。

结果：在 2026 年 3 月部署此补丁后，查询时长下降了 50%（见图 8）。更重要的是，这最终打破了查询时长与部件数量之间的相关性。不幸的是，此解决方案对于任意查询条件（例如，namespace in (5,10) 这样的条件）的通用性并不好。我们正在研究更通用的方法，例如扩展查询条件缓存以覆盖部件筛选。

实施二分查找进行部件剪枝（优化 3）后的持续延迟降低。

## 脆弱的平衡

这些优化措施解决了计费系统的燃眉之急。但这一过程暴露了我们分区选择所带来的深层且不明显的代价。

其他问题依然存在。在这篇博文中，我们仅描述了部件数量增加对查询时长造成的问题，但它也给 ZooKeeper 带来了问题，ZooKeeper 会跟踪 ClickHouse 中所有部件的元数据。也许有一天我们会讲述那个 100GB ZooKeeper 集群的故事。

我们为自己赢得了显著的喘息空间，但根本问题依然存在：这种分区方案是长期正确的选择吗？还是我们最终需要咬紧牙关，转向不同的架构？目前，我们的补丁还能维持，但这次经历清楚地表明，即使是计划周密的变更也可能因错误的假设而失败。

当计费团队首次报告此问题时，每个副本有 30,000 个部件。部件数量从未停止增长，一年后，每个副本达到了 16 万个部件，但由于我们在此所做的优化，查询时长保持稳定。
