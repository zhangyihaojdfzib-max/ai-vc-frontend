在最新一期的《Executive Function》节目中，Brett 邀请到了安全与合规平台 Vanta 的首席产品官 Jeremy Epling。Jeremy 详细讲述了他的职业历程，剖析了从资深的微软高管转型为初创公司 CPO 所需的条件。他还分享了一些来之不易的洞见：如何在员工数量激增时保持产品交付速度，如何在没有大公司流程安全网的情况下进行绩效管理，以及真正由你负全责地运营一个产品组织意味着什么。

在本期节目中，我们讨论了：

- 让 Jeremy 成功转型为初创公司 CPO 的思维转变
- 为什么 CPO 必须与细节保持联系至关重要
- 确保团队在快速增长的同时也能快速交付的规则
- 为什么僵化的层级制度会破坏高质量的决策
- Jeremy 如何为整个公司设立开放的办公时间

Brett: 好的，我们开始吧。感谢你的加入。
Jeremy: 是的，谢谢邀请。
Brett: 通常，当你看到像你这样背景的人——基本上在微软担任各种职务20年——他们之后加入的并非一个小型初创公司，而是一个处于成长期的初创公司，当然相对于微软来说，是一个极其微小的公司。
Jeremy: 是的。
Brett: 90天内，情况完全变成一场灾难，然后这个人离开，回到那个美好的"母舰"。你认为你自身的原因、你所领悟到的东西，或者也许你在微软所做的具体工作，是什么让你在如此长时间后，在如此剧烈地改变环境的情况下，依然能够高效工作？
Jeremy: 我想对我来说，大学一毕业，我就创办过一家初创公司。所以我想我骨子里有那么一点创业的基因。那家公司持续了九个月，最终为我带来了一堆工作机会，但并没有真正发展起来。那是在互联网公司第一波浪潮泡沫刚刚破裂的时候。我是2002年加入微软的，差不多就在那个时候。所以我认为，对我来说，很大一部分学习在于我总是想换团队，总是想学习新东西。我想这已经成了我基因的一部分。即使在微软的时候，我也试图每三四年就换到一个全新的产品领域。所以刚开始时，我在 Windows 部门从事安全和深度技术方面的工作。我在 Internet Explorer 待过一段时间。然后我加入了 OneDrive，那是一个从零到一的过程，我非常享受那个过程。不过，一个重大的转变是去了 GitHub。当微软收购 GitHub 后，我在收购后不久就过去负责那里所有的新产品开发。对我来说，Nat 非常像一位创始人 CEO。显然，他成功退出了很多公司，做过 VC，完成了很多了不起的事情。而且那家公司运营得非常独立。我实际上是离开了微软。你的合同结束了。我们当时全部使用谷歌工具，一切都构建在 AWS 上，都用 Zoom。当 Satya 给公司发邮件时，这些邮件不会发给任何在 GitHub 工作的人。徽章不能互通。人力资源、福利、职级，一切都完全不同。所以我想这真的让我尝到了那种感觉会是什么样。我认为我喜欢的事情之一就是整个业务紧密相连的感觉。我觉得微软有很多优点，但根据我在那里的经验，你有时会感觉与市场推广活动脱节。感觉像是在另一个星球上。你只是努力打造最好的产品，然后在某个遥远的地方，赚了很多钱，感觉相当脱节。而在 GitHub，我看到了所有环节如何结合在一起。从我们的产品，到信息传递，到市场营销，到营收，与销售工程师深入合作，思考我们的定位，所有环节。这让我非常兴奋。我当时就想，我想要更多这样的体验。然后随着 GitHub 不断成长壮大——这很棒——它一直在增长。我当时想，其实最有意思的阶段是早期阶段。
Brett: 在 GitHub 的经历中，当你从非常以产品为中心，转变为更全面，至少负责或参与了许多商业环节时，有哪些有趣的收获？

杰里米：我认为其中一些重要的收获包括：首先，我与我们的售前技术负责人建立了非常紧密的关系，我认为这至关重要。在从事像GitHub这样的技术销售时，售前团队扮演着巨大角色。客户经理对于建立关系、梳理组织架构、处理所有相关环节都极为重要，但更重要的是构建一个紧密的反馈闭环——在每次销售周期中切实理解哪些方法有效、哪些无效，以及客户如何实施产品，并将这些反馈反哺到产品中。我在微软时有过类似闭环，但远不如在GitHub时那样紧密，那时我几乎每天都与售前负责人沟通。当时我正在开发新产品，处于那种必须摸索出解决方案的阶段。

纳特则真正推动了一种理解市场定位与品牌叙事的文化。因此我投入了大量时间去学习我们如何讲述品牌故事、如何讲得更好，并在定价与产品包装方面花费了比在微软时多得多的时间。以GitHub Actions为例，那是我当时负责的产品之一。我们采用了基于使用量的定价模式，这是GitHub首次在任何产品上尝试这种定价方式。我们该如何思考这种转型？幸运的是，我仍可以咨询我在Azure认识的同事，了解他们如何完成类似转型，但更要思考这对GitHub意味着什么——毕竟我们每天都要销售产品。用户习惯为团队每个开发者购买许可证，现在却要让他们思考计算和存储资源，如何估算用量？如何让开源开发者感到友好？我们应该为开源提供多少免费额度？这段经历让我对整个产品通过定价与包装对业务产生的全局影响有了全新的认识。

我还花了很多时间思考销货成本。当大规模运行像CI/CD这样的大型基础设施时，成本非常高昂。我们甚至在自己的数据中心里架设了Mac服务器。这让我开始思考硬件供应链的周期——比如预计新款Mac Mini即将发布时，我们需要如何规划网络配置等各个环节？这段经历让我接触到更广泛的业务维度，使我更深刻地理解到，要交付卓越的产品体验，远不止关注单个功能或局部细节（至少我在微软时是如此）。我相信微软的情况也已发生变化，不再局限于那种工作模式。

布雷特：基于以上背景，当你思考处于规模化阶段的初创公司中首席产品官的角色时，如何定义卓越？

杰里米：我认为必须深入细节。最近一年有很多关于"创始人模式"的讨论，我坚定支持这种模式——你必须深入细节。对我而言，需要理解我们交付给客户的体验及其卓越之处。一旦与这些脱节，比如变成纯粹的职业经理人，只负责指导团队、制定战略，就很容易偏离为客户打造正确产品的方向。如果不参与具体会议、不销售产品、不每天演示和使用产品，就会产生这种脱节。我常在大型企业中看到这种情况：最终形成职业管理阶层与实际执行者的分离，这时产品就容易迷失方向，因为并非所有人都在使用它，也不清楚其发展路径与背后的清晰逻辑。

布雷特：这是其中一个方面。首席产品官的工作职责还包括哪些？

杰里米：我认为首先是以合理节奏交付产品的能力，这是最重要的之一。需要持续向客户推出新功能，保持高迭代速度，同时确保高胜率。胜率是我经常思考的指标——尤其是失败时，要分析原因：是定价包装问题？产品可以介入改进。还是产品效能问题？我们能如何优化？其次，制定愿景的能力取决于CEO的风格：有些CEO希望全权掌控，有些则愿意授权部分给首席产品官。因此需要明确自己在愿景构建和叙事传达中的角色。即使市场团队擅长讲述精彩故事，产品团队也必须具备解释能力，因为你们才是直接与客户沟通、构建产品的人，需要清楚"我们为何以这种方式为这类用户构建"，并传递这种认知。

就我而言，我还领导工程团队，因此工作包含技术层面：理解架构、SLA达成情况、扩展能力，在满足短期交付目标的同时为未来构建，这在规模化阶段尤其关键。如果是加入B轮或C轮公司，我与CEO沟通时常被问到："请说明你的推进速度与工作方式，以及如何帮助我们构建多产品战略。"这就需要思考产品协同性：可以拓展哪些相邻领域？如何决策选择方向？如何验证决策正确性？毕竟并非每个决策都正确。

在这个过程中，我也坦诚说明自身特质以确认匹配度。我会表示："我非常重视产品质量，致力于打造精良、美观、易用且客户喜爱的体验。"但并非所有优秀公司都注重这点。我会补充："如果公司不认同这种价值观，我可能会感到不适。"我记得与克里斯蒂娜等创始人沟通时说过："让我兴奋的是能发挥最大价值的领域"，而不是勉强适配某个岗位。

布雷特：基于如此细致的描述，按照你的定义，首席产品官与高级产品副总裁、产品副总裁或总监（按你的职级体系低一级的职位）之间存在怎样的期望差距？

杰里米：我认为在这些职位上，最大的变化在于，你需要将高管团队（C-team）作为你的首要团队来关注。我记得当我首次担任首席产品官时，我咨询了许多创始人朋友，我问他们：“在这个职位上，我最可能犯的最大错误是什么？”他们一致认为：“如果你只专注于工程、产品和设计，而没有真正与高管团队协作，那就是最大的问题。”事实证明，这个建议百分之百正确。对我而言，这其实很容易理解，甚至可以说是我的默认倾向，因为我大部分时间都花在工程和产品设计上——那是我感到最自在的领域。我很想深入团队，思考如何优化产品，但如果我们不将产品与公司其他业务联系起来，就无法产生所需的影响力。所以我认为这是最重要的转变。

另外一点是，许多责任最终都落在你身上，尤其是在Vanta的组织架构中，我负责产品、设计和工程。克里斯蒂娜（Christina）的理念是，产品开发相关的一切都是我的职责。因此，设定节奏、质量标准，明确愿景和方向，创造清晰度——这些在以前担任副总裁或高级副总裁时，总有上级可以商讨想法，或知道他们会设定高层级的指导原则。但对克里斯蒂娜来说，她需要关注整个公司和业务，她的职责范围非常广。所以对我来说，我必须百分之百地承担起这些责任：设定执行节奏、明确方向、深入处理客户的具体问题、权衡重大战略决策，比如如何分配对下沉市场和高端市场的投入。我觉得在这个职位上，肩上的担子很重，你必须习惯于独立做出大量决策，因为没有人可以真正替你承担——至少Vanta的架构是如此。

布雷特：除了建立关系、保持沟通和提供帮助外，在高管层面，如何做到10分水准的团队成员？

杰里米：对我们来说，关键在于大家都能提出业务中看到的问题或机遇，在一个安全、互信的环境中分享，并作为团队共同解决，然后以统一口径对外呈现。例如，如果首席营收官（CRO）提出：“我担心这里的成交率在下降，我们该怎么办？”她可以放心地与我们讨论。实际上，我更喜欢……我的意思是，当我和史蒂维（Stevie）一对一讨论时固然不错，但更理想的是团队一起讨论，这样我们能达成共识——也许市场部门可以提供帮助，也许财务部门需要知情，因为涉及定价或其他数据分析。所以，能够把问题带回团队，花时间共同探讨并做出重大战略决策，这很重要。

我认为作为首席产品官（CPO），另一项关键能力是把握沟通的“高度”。当我在工程、产品和设计团队内部工作时，我们可以深入细节，讨论产品的每个功能，进行技术性很强的交流。但当我与首席营收官或人力资源负责人协作时，他们对这些细节并不那么熟悉。因此，我需要调整对话的层面，基于他们对产品的了解程度进行沟通，从而推动整个业务前进。

布雷特：当你在高管层面处理问题，比如首席营收官或首席人力官提出的议题，整个高管团队共同协作时，你是否认为自己的职责是带入产品和工程视角？还是说，无论你负责什么职能，大家只是以不同视角共同解决问题——并不一定非要由你来带入产品或工程观点？

杰里米：很多时候，我确实更多地带入产品和工程视角，但也有一些讨论是相对客观的，比如公司的核心价值观、团队协作方式，或者是否采用远程办公。这类议题可能因团队而异，我们可以允许不同做法，也可以不——但这更像是一般性讨论。多数时候，我试图以易于理解的方式传递产品洞察，因为我认为CPO的角色与其他团队成员的不同之处在于，你可能比其他人花更多时间与客户交流。例如，GitHub有出色的市场推广负责人，但他们在高管层面更多是构建体系和机制，可能不像我那样参与大量客户会议。因此，我会带入客户视角，同时结合市场团队从客户经理（AE）和解决方案工程师（SE）那里听到的具体反馈，然后共同探讨解决方案。

对我们来说，关键问题往往是：反馈循环如何运作？竞争对手调整销售策略、降价、发布新功能……我认为关键在于召集团队，共同分析：我们是否看到主要竞争对手有所动作？他们发布了有趣的功能。我们作为团队该如何应对？是否需要调整产品路线图？是否需要回应？还是说我们的产品明显更优，现有市场信息仍有效？如果需要调整信息，如何快速传达给一线？这需要整个高管团队达成一致，因为可能涉及定价调整、市场策略变化，或新的营销活动。有时可能是：“我们被超越了，需要在产品上做出改变”，或者对手暴露了我们未曾想到的短板，我们需要快速修复——抑或意识到这需要较长时间，那么如何改变客户的购买标准？这就变成了信息传递的问题。例如，如果确实存在短板，工程产品设计（EPD）团队可能需要一两个月来构建。在此期间，我们该如何与客户沟通？如何强调我们的优势？我认为这类对话最令人兴奋且富有成效。我可以带入产品视角，分析如何调整路线图、哪些不可行，以及我从不同客户会议中观察到的信息，同时学习市场团队和营销团队的洞察。

布雷特：如何避免形成一种人人相互指责的局面？我认为这很自然，尤其是对于雄心勃勃的高管而言。如果赢单率出现问题，通常情况会是首席营收官指责首席产品官，认为他们没有打造出正确的产品，或者没有——

杰里米：是的，没错。

布雷特：……具备应有的特性和功能。除了我们正在建设的文化之外，你是否对如何实际创建一种文化有所思考？在这种文化中，面对问题，我们所有人会共同致力于解决，而不是基本上归咎于他人？

杰里米：坦率地说，我认为这非常困难。我认为营收团队和产品团队之间存在着一种难以调和的动态，在某种程度上存在根本性的对立。产品团队总是考虑得更长远。产品团队的薪酬模式就是基于此建立的，主要是薪水，可能股权占比也更高。而营收团队则不同，尤其是在面向中低端市场的业务中，他们关注的是月度配额。随着业务规模扩大，可能会变成季度配额等，在向上销售市场时，他们反应极其迅速。我认为企业要成功，这两部分都不可或缺。你需要快速响应客户，不能让工程、产品、设计部门只沉浸在对未来一切都会更好的空想中，只专注于战略。所以，对我来说，最重要的事情是尝试在这两者之间建立共同的认知基础。我通常会回到客户反馈循环这个话题，我觉得这很有意思。根据过去的经验，当产品团队感觉他们只是营收团队的“接单员”时，产品通常会陷入糟糕的境地。你明白我的意思吗？就像“客户抱怨这五个问题，去解决这五个问题”，这变得非常被动。然后，你就没有太多关于未来方向的战略了。就像你只是疲于应付——

布雷特：但这其中也有值得关注的地方，因为其中也有很多积极因素。

杰里米：没错，而且那些（响应客户需求）确实能促成交易。所以你需要能够在一定程度上做到这一点。你只是不希望完全变成那样。因此，我们一直在做的一件事是，努力做好分享工作，比如：“嘿，产品团队从我们参与的所有客户通话中，看到了客户反馈的这些情况。这与营收团队看到的情况是相似还是不同？”

布雷特：考虑到我们讨论的这种产品与营收之间的根本性张力，例如，解决这个问题有多大程度上仅仅依赖于首席产品官和首席营收官之间建立真正出色的关系？这是两个人之间的事，如果他们总体上彼此欣赏和尊重，是否就能解决70%的问题？或者，即使关系良好，仍然存在一个核心的、类似物理定律般的固有张力，你必须去管理它？

杰里米：是的。我的意思是，我认为这是成功的关键。我认为这正是史蒂维和我非常重视的一点，就是去理解：“嘿，我们俩如何能快速、轻松地与各自的团队协同处理事情？”我确实认为系统中总是存在一些固有的张力，实际上我并不认为这是坏事。我能坦然面对冲突，并且认为那些更能接受冲突、不将升级问题视为坏事的公司会更成功、运营更顺畅。我认为很大程度上这都归结于那种关系，比如，你能否分享你真正担心的事情？让对方倾听并接受，然后反馈回来，因为也会有反向的反馈，对吧？产品团队可以旁听几次销售电话，然后可能会想：“天哪，我们刚刚发布了这个很棒的新功能，却没人提起它。为什么我们不谈论它？”你明白我的意思吗？这可能是市场营销的问题，或者是销售培训的问题，也可能只是那个销售不知道，或者我们的销售渠道还没铺开。所以我认为开放的沟通渠道非常重要。我认为这正是我们之前提到的那种高管团队层面的事情，能够分享：“嘿，我们看到了这个情况。”而且我认为，少带着解决方案去沟通通常效果更好，可能对双方都是如此，但我至少可以从工程、产品、设计部门的角度说，当有人来找我说：“嘿，我们遇到了这个问题。竞争态势在这里发生了变化。我们该怎么办？”我总是觉得这让我更容易接受。而不是像“请你明天就发布这三个功能好吗？”那样，那样我只会想：“客户的实际问题到底是什么？”也许那三个功能是正确解决方案，但也许不是。所以我认为，越多的人以这种方式互动——“我们处理的问题是什么？让我们一起来审视，看看每个人能做些什么来改善它。”——这才是让事情顺利运转的最佳方式。

布雷特：那么，在高管团队的背景下，你如何看待良性冲突与恶性冲突？

杰里米：我喜欢直截了当地讨论事情。我觉得人们……如果有人表示“嘿，我觉得这个行不通”，我会希望直接把它摆到台面上说：“很好，我们来谈谈这个问题。”我认为当我们最终回避问题，或者因为大家都感到疲惫而进行太多零散的单独沟通时，情况就会变得像是每个人都在谈论这件事，却没有一起面对面交流，然后你会感觉自己被排除在信息圈外，或是从第三方那里听到消息等等。我认为克里斯蒂娜做得很好的一点是，我们每周都为高管团队预留了大量时间聚在一起，我们会逐一过议题：“嘿，这些是我们需要讨论的重点话题，我们所有人都可以把事情列到清单上，内容可以是关于我们团队的，也可以不是。”我们尝试以开放和尊重的方式处理所有议题，同时也会直接指出：“嘿，这里有个问题。我不太满意，或者我担心这里的平均合同金额正在下降；又或者我对这个产品表现优异、销售火爆感到超级兴奋。我在考虑将更多工程、产品和设计资源投入到这里，但这意味着我需要在其他领域做出牺牲。市场团队对此怎么看？”所以我认为，讨论的内容不一定总是问题，也可以是令人兴奋的新事物，但我总是努力保持对我们所做事情的前瞻性。我认为充分沟通是关键。我仍在努力提升的一点是：在合适的层面进行更多沟通，让每个人都知道我的团队内部发生了什么，因为产品的许多动作都会对其他人产生连锁影响。比如我们说：“嘿，我们今年在联邦政府业务上下了重注，已经完成了FedRAMP 20X认证。我们获得了低级授权，并且深入参与了因行政和国会等方面带来的所有安全变更。”这意味着其他领域需要做出权衡，就像在说：“嘿，这是我们正在下的一场赌注。”所以我把这件事带到了高管团队，我说：“嘿，以下是我要下这个赌注的原因和投入强度。这会对其他领域产生什么影响。大家感觉如何？我们还能完成营收目标吗？我们是否认同这个增长机会的价值？具体会是什么样子？”我很喜欢文档，可以说我们公司有浓厚的文档撰写文化，但我们尽量保持简洁。我不知道，我有很多口头禅，其中一句是“简洁即清晰”。所以我常说：“直接告诉我我们想讨论什么。我们能不能列几个要点然后做决定，而不是搞出一份80页的产品需求文档——我觉得这方式现在已经有点过时了。”在那种情况下，我宁愿直接拿到一个v0版本或Figma原型，或者直接看图片来了解客户体验。但在高管团队层面，是的，我认为就是讨论难题。这样就不会有“房间里的大象”。努力围绕问题本身提出建设性意见，并尽可能提供大量数据支持。我花了很多时间试图理解我们的产品营收数据，观察销售端的进展，同时也分析产品使用数据，以便将这些不同的部分整合起来。我们的财务团队做得非常出色，他们深入整合产品营收数据，并将其与销售环节挂钩后呈现出来。所以在做决策时，我喜欢做的是：我们需要什么？我们需要什么数据才能 confidently 做出这个决定？因为我最讨厌的情况是——不知道你是否遇到过——微软有个术语叫“捡石头”，就是有人说：“嘿，你能去捡块石头吗？”你说：“什么石头？”对方说：“我不知道，但捡一块来。”你捡来后他们说：“其实不是这块。那你想要哪块？我不知道。再去捡一块吧。”我认为这是最糟糕的情况。所以我们尝试界定事务的边界并明确：“嘿，这是要做的决定。我们需要什么数据才能达成共识……”有时可能靠直觉，有时需要真实数据。我们可以做出这个决定，然后作为团队共同代表这个决定，因为我认为“不同意但依然执行”也是工作的重要部分。肯定有些时候我不确定自己是否相信某个观点而我错了，也有些时候我是对的但我们最终没有执行，但我们都将其作为团队决策来代表：“嘿，这是我们想要的方向和原因，我们共同为此负责。”

布雷特：你认为高管团队在成员个性各异时运作效率最高，还是在所有人思维模式相同时更高效？比如整个团队都是一群喜欢辩论和争论的“难相处”的人，或者你想要一个非常有主见、爱唱反调的人，也想要更多调解者性格的人，诸如此类。

杰里米：我不确定。就我们团队而言，我认为如果所有人总是在反对，对每个参与者来说都会感到非常痛苦和煎熬。我觉得当有人只是为了扮演“魔鬼代言人”而反对时，从我的角度看就变得消极了，因为这就像是在所有事情、任何计划中刻意挑毛病，你明白我的意思吗？所以我认为需要将其限制在：哪些是我们真正担心的、有影响力的事情？我们年度规划流程的一部分是，我们会逐一询问：“每位高管，你最重要的五个问题是什么？”为了敲定明年的年度计划，首席财务官需要知道的五件事是什么？你懂吧？他可能会说：“关于产品或营收等方面，我有这五个问题。”然后我们可以逐一讨论：“很好，我们来回答这些问题。现在大家都觉得没问题了吗？”因为问题总是层出不穷，但尝试聚焦其中一些关键点。我确实认为，如果完全没有挑战，你或公司就无法突破。所以你必须能够提出尖锐的问题。我认为这是Vanta企业文化的一部分，克里斯蒂娜 definitely 鼓励的是：让我们提出尖锐的问题并进行讨论。只是你不想把它推向“魔鬼代言人”的极端，比如让我总是对每件事都持反对意见。

布雷特：如果有高管表现未达到质量标准呢？这只是CEO的管理职责，还是其他每位高管也有责任？如果是后者，这如何以一种非背后中伤的方式运作……

Jeremy：我认为在这种情况下，我感觉职位越高——不一定非得是高管团队，即使级别稍低也是如此——我尝试鼓励我的团队，比如我的直接下属……我负责工程、产品和设计，如果他们意见不合或有人表现不佳，他们会首先尝试以同事身份相互协作。我认为这是最好的方式。如果我说：“嘿，我需要财务、市场或营收部门的支持”，或者如果他们觉得“嘿，我某件事没做好”，我认为他们能够来找我，直接说：“嘿，我需要这个，但我觉得我没得到它来完成工作，或者我觉得这些是当前领域的不足。具体是这些。”作为产品人，我就是靠细节驱动的。所以我觉得泛泛而谈对我效果不好。我会说：“嘿，在我们写的这份文档里，你能告诉我具体是数学计算、你引入的先验假设中的哪一点，导致这个方案无法让你认同吗？”然后可能发现：“哦，太好了，实际上我们在这点上存在分歧，所以后面所有内容都说不通，因为你觉得我们应该往A方向，而我认为应该往B方向，那我们就来专门讨论这一点。”但我确实认为，一对一地解决问题是关键。当然，最终的决定，比如“多少指导算过度？”当你已经过了收益递减点，需要请那个人离开或做出调整时，我认为这显然得由CEO来定夺，因为他是管理者。但我认为，团队成员之间沟通越多越好。然后，如果你发现某个领域持续存在差距，我认为CEO有责任去弄清楚：“嘿，是系统导致的吗？是个人绩效问题吗？我能介入做些什么来让事情运转起来？”尤其是当整体业务进展顺利时。某种程度上，我觉得我们在董事会会议上开过一个玩笑：当我们做得好的时候，董事会反而更苛刻，因为没有真正的大问题可讨论。你明白我的意思吗？所以他们会揪住所有这些有点重要但又不那么重要的小事不放。但如果事情进展不顺，每个人都知道问题在哪，然后大家就会一起努力解决。所以审查会严格得多。

Brett：你在对话开始时提到，在Vanta的组织架构中，工程部门向你这位首席产品官汇报。这种组织设计有什么优缺点？

Jeremy：我认为，理解你的CEO希望把时间花在哪里、擅长什么，会真正改变这个角色的形态。Christina在产品方面真的很在行，但她可能不想去思考和处理产品与工程之间的分歧。所以她希望有一个人能统管，就像“我希望有一个明确的问责人”。你明白吗？涵盖所有执行层面：我们执行得快吗？服务质量高吗？能扩展吗？我们是否在各个部分都打造了用户想要的产品？所以产品、工程和设计都归在一起。我认为这让她能专注于很多跨公司的问题，她可以和我讨论产品方向和愿景、她感兴趣的东西，她可以把更多时间花在营收、市场等方面，而不必深入管理另一个技术负责人或设计负责人，也不必参与所有那些更细节的讨论和解决其中的冲突。

Brett：但考虑到这种好处，一方面是围绕CEO打造一个形态合适的组织，这显然非常重要。另一方面是，当你从最宏观的角度思考时，从组织或整个公司的角度来看，好处是什么？当一位首席产品官也掌管工程时，你会说你获得了X，但必须权衡Y。X和Y分别是什么？

Jeremy：我认为你获得的是更快的决策流程。所以当你想决定“嘿，我们是否需要做这次重大的技术架构重构，而这会牺牲一些产品功能？”时，你需要推进并做出这些决策。你有一个可以直接拍板的人，我能非常轻松地快速解决任何此类分歧。比如设计和产品通常都归首席产品官管，所以我觉得这对大家来说更熟悉，但关键还是更快的决策。我认为缺点可能是会给人一种工程部门不那么重要的感觉。我不认为Vanta存在这个问题。我认为工程超级重要，我确实充分授权我们的工程负责人去推进和负责。比如我们的工程团队外出活动，他们是公司里最大的团队。你明白吗？如果他们不向我汇报，他们的团队规模会比销售团队和其他所有团队都大，他们负责那个领域的纪律、招聘、预算等所有不同方面。所以我尝试把权力下放。因此，我管理的方式是：是的，我是管理者，但同时我也把它当作一种同级关系来对待，我会说：“嘿，我们俩是一起思考如何构建这个产品的。”对设计负责人也是如此。但同时，我也是管理者，所以我明白这永远不会感觉像纯粹的同级关系，但我认为这就是权衡之一。如果我所在的公司需要超级、超级深度的专业技术知识，我想那种情况下我会对让一个人统管所有事务更加怀疑，我会希望有一位独立的技术负责人。我认为Vanta确实有非常难的技术问题要解决，但这是不同类型的问题，不同于你要做底层3D图形、量子计算，或者像Figma第一次研究WebGL以及如何在网页上做设计软件时面临的问题，我认为那些是需要解决的不同类型的问题。

Brett：你提到的另一点是，当你考虑规模化初创公司背景下的首席产品官时，一个更有趣的问题是产品与CEO角色之间的关系。我认为这通常最有趣的原因是，大多数技术导向的公司都是由创始人创立的，他们的专长领域或独特之处之一就是他们的产品品味、判断力等等。这通常是公司最初能成功的原因。所以你经常会发现产品负责人和CEO之间的关系非常微妙。正如你开头提到的，这种关系可以有各种不同的形态。一种是首席产品官真的只是在执行CEO的愿景。也可能是CEO意识到他们需要把大部分时间花在其他事情上，以及介于两者之间的各种形态。对于首席产品官和产品导向的CEO如何合作并取得成功，你有什么具体的思考？

杰里米：我认为需要花大量时间在核心理念上达成共识，比如他们对优秀产品的愿景是什么，并努力理解决策过程，这样才不至于总是意见相左或产生冲突——你们总是一个向左一个向右。长期来看这对谁都没有好处。所以我花了很多时间，至少回想早期和克里斯蒂娜合作时，我会主动去理解产品：你认为产品成功的关键是什么？未来怎样才能持续成功？需要哪些防护机制？你希望把时间花在哪些产品决策上？哪些事情让你兴奋，哪些让你觉得乏味？你擅长什么？不擅长什么？这样我就能规划自己该专注的方向，并观察我们的能力是否互补。和克里斯蒂娜合作时，有个很有利的因素——她对某些领域充满热情。我不知道大多数首席产品官是否都这样，但我对产品出身的创始人持开放态度：欢迎深入参与我的工作，和我一起投入。我最近和一家公司建立了合作，克里斯蒂娜对此非常兴奋。她撰写了最初的产品需求文档，还做了她设想中的v0版本雏形。我问她：“你想主导这个项目吗？”她说：“当然。”我说：“太好了。”随即告诉我的团队：“直接和克里斯蒂娜协作就好。”明白我的意思吗？我完全不觉得被削弱或有什么不妥。她想做这件事，思考这些能给她生活带来快乐——这是能激发她热情的事，而她还要管理一家大公司。CEO的工作有令人兴奋的部分，肯定也有不少枯燥之处。这一点火花般的快乐，我觉得“尽管去做就好”。因此我尽量不在团队里设限。这是我在GitHub学到的：和产品出身的创始人纳特共事时，感觉也很相似。只要团队里有人比我更懂某个领域，我就会说：“纳特，这件事直接和克里斯里斯或纳哈协作，不用经过我，我甚至不需要参与。”那种军事化的命令与控制结构，凡事都要层层上报审批，正是拖垮大公司的原因。对我来说，我希望每个人都觉得可以和CEO无障碍协作，不会因此感到压力。我知道克里斯蒂娜接手某个项目后，就可以放心去处理其他几件事。所以明确这些边界对我来说是最重要的。

布雷特：当CEO想参与某个新产品或产品细节时，你是否会用同样的标准要求他们像负责该产品的敬业PM那样行事？还是说如果CEO要主导这件事，即使他们不像专职PM那样深入，你也会给予弹性？或者说……既然你要主导，就必须执行到位？

杰里米：是的。我从不让他们完全独立负责——这种情况下我总会指派一名PM协同。通常PM会很乐意，因为能和CEO及创始人协作是件很棒的事。所以一般不难安排，他们也不会觉得被架空，反而觉得“这很有意思，我能和克里斯蒂娜一起工作”。这样她也能保持在概念层面参与。如果突然出现重大事项，比如“天啊，我们马上要融资了，这会占据她大量时间，她就没法每天和工程师跟进具体工作了”，所以我总会确保有PM和设计师参与。通常也会选择更资深的人员，最好是达到专家级别的，这样他们能理解高管的意图并转化为具体执行，避免克里斯蒂娜陷入细节决策。

布雷特：你认为这种让CEO可以跨层级、跨职能参与的模式是普遍适用的，还是只在你工作过的某些公司环境下有效？

杰里米：我觉得肯定存在规模限制。桑达尔或萨提亚那样管理二十万人的CEO恐怕很难这样操作。公司达到那种规模后性质就完全不同了。但在数千人规模时，我认为仍然应该这样做。这也是我们的理念之一。在Vanta，我们确实要求员工深入细节，这是我深信不疑的。如果我不了解工程师、设计师或PM的实际工作内容和方法，不在某种程度上参与细节，就很难指导他们提升。我告诉新入职的设计师：“我会在你的Figma文件里留言评论。”这不该让他们害怕，也不代表他们做得不好，而是要让大家习惯这种协作方式。我从设计总监布雷登那里学到个好方法：他在所有反馈前都会标注层级——这是想法、建议，或是需要执行的指令。随着团队规模和职级提升，我越来越意识到高管话语权会产生意想不到的负面影响。所以我尽量更谨慎地表达，并反复说明：发给团队信息时会强调“这需要立即执行”或“这只是脑力激荡，请不要调整路线图”，甚至用全大写标注。但即使这样，有时他们还是会说“我们做了三处调整”，而我只能回应“不”。团队规模扩大后，确实需要不断明确期望：什么时候需要行动，什么时候不需要——尤其当允许高层直接介入时。比如CEO来和团队讨论某事，或者像我这样……现在公司大概350人了吧？我负责的EPD组织已经相当庞大。对新员工来说，突然看到我在他们的PRD上留言可能会不知所措。所以我们尽可能让这种互动常态化。

布雷特：你刚才略有提及——工程师的产出是软件，最终是客户愿意付费的优质产品；设计师产出原型、效果图或终稿，负责的可能是产品质量或影响力等。你如何定义自己作为首席产品官的角色产出？

Jeremy：我认为对我来说，在最高层面上，我想关键问题是：客户是否喜爱这个产品？以及你打算如何衡量这一点？我认为其中一部分可以通过考察赢单率、我们的表现情况，以及有多少是源于产品效能提升而非其他因素来衡量。特定领域的客户满意度对我而言确实影响重大。我不是NPS的忠实信徒。我知道有些人喜欢它，但我不认为它有那么实用。我更倾向于针对核心工作流程进行客户满意度调查，比如：“这个功能让你的工作更轻松了吗？还是没起到作用？存在哪些不足？体验如何？”这些都是需要关注的其他重要方面。

我认为另一个关键是速度——我们是否在持续推出新功能？我们是否不仅拥有让客户兴奋的产品，更拥有让客户兴奋且感觉与众不同的愿景？所以，我认为这些是我们关注的一些重要产出指标。

我的意思是，我经常关注**单产品收入**。我认为销售方面的考量，至少在我们公司和大多数公司我看到的方式，更多是区分新客户与老客户扩展，并按不同细分市场、区域等维度进行拆分。而我发现，我让财务团队为我构建的仪表板，完全是基于产品客户来划分的。所以我会看：很好，我们在这个产品上投入了大量资源。作为产出指标，平均合同金额（ACV）是否在上升？当然有很多输入指标，比如用户使用情况等等，但产出指标就是：它增长了吗？我们卖得更多了吗？附加率看起来怎么样？是否销售给了正确的用户角色和细分市场？

在我们的市场数据中，当我们进行外拓或客户主动咨询时，我们会将他们分类到不同的用户角色中，因为安全团队差异很大。对于初创公司，可能根本没有专职的安全人员，可能是由创始人或技术创始人兼任。而对于大公司，他们有首席信息安全官（CISO）和多层级的、高度专业化的团队架构，比如有专门负责内部风险的团队，这和处理审计或内部审计的团队完全不同。所以专业化程度很高。我们会对这些进行分类，然后在构建产品时思考：“嘿，我正在为‘初级安全专员苏茜’构建一个产品。”对她而言，我们的用户角色基本上是公司招聘的第一位安全人员。我们会评估：针对她的情况，我们做得怎么样？她是否购买了我们的目标产品？她是否按我们预期的方式使用？增长情况如何？

另外，我们的产品组合广度是否在增长？我认为Christina（Vanta联合创始人）采用的方法可能很多其他创始人也用，她的第一段经历在Dropbox，当时她在做Paper产品，那是一个他们试图扩展、希望变得比现在更多元化的产品。我认为Dropbox在某些方面成功了，某些方面没有。我觉得她将这种理念完全带入了Vanta：我们必须是多产品公司，我们如何实现这一点？如何尽早布局并将其融入我们的DNA，始终保持构建新产品的能力？因此，我认为这个视角给了我一个非常有用的收入产出指标：太好了，我们两年前推出了客户信任产品，它现在表现如何？好的，我们为它增加了一个新附加功能，卖得好吗？客户采用情况如何？我们做得怎么样？

然后，再回溯到专项销售团队：哦，如果卖得不好，是产品本身不行吗？那我们可以改进产品；还是说，哦，是因为我们没有专项销售人员？这就像我们今年所做的一个调整：当我们和通用销售人员沟通时，发现他们根本不知道如何就新产品进行深入对话。他们有点害怕推销新产品，因为现有产品已经能完成他们的全部配额了，为什么要冒险尝试新东西呢？于是我们做出了调整，这释放了更多的增长潜力。当然，我们也发现了产品本身可以改进的地方。但我觉得，结合考察单产品收入、用户如何使用产品、他们是否满意，以及我们是否对高端市场客户拥有引人注目、令人兴奋的愿景，这些都很重要。这些责任可能落在产品部门，也可能落在CEO身上，或者介于两者之间。

Brett：除了你在特定时间点针对特定产品所做的类似独立贡献者（IC）性质的产品经理工作之外，你认为你自己必须产出的、真正重要的工作成果具体有哪些？

Jeremy：对我来说，我最终会负责几份文档并与团队协作。其中一份现在是这种“元文档”——由于Google Docs有了标签页功能，它基本上就是一个包含不同标签页的文档，我们称之为产品规划文档。这份文档会逐一审视我们独立销售的每个核心产品，并定义：这是用户画像、这是目标细分市场、这是核心客户问题、我们试图交付的价值、我们的差异化优势、这些功能的发布时间，以及我们如何开拓新类型的客户。比如，可能是更大规模的企业层级，或是我们正在开拓的一种特殊新客户类型。所以我试图把这看作EPD和产品团队需要对齐的“原材料”，即：我们为谁构建产品？我们将如何取胜？这其中还有一个时间要素，它会指导营销和销售工作，对吧？比如我们会说：“太好了。我们认为这类形态的客户，目前我们还无法赢得，但我们预计明年第二季度能赢得他们。”然后我就可以去告诉销售和营销团队：“我们应该为此策划一个营销活动。我们应该在这里重点推广。我们应该在这个节点上做一个专题。”之后我们就能看到这个赌注是否奏效，你明白我的意思。然后，他们在进行收入预测时就可以据此判断：“好的，在第二季度或第三季度，这个细分市场预计会改善，改善方式是这样的……”我们会与财务团队基于一个模型来协作。这份文档对我非常重要。关于这份文档，我学到的是：它对产品市场经理（PMM）和一些一线人员来说是极好的原材料，但很多时候它过于详细，对他们来说反而有些模糊不清。我想说，我们目前产品迭代速度极快，每个季度我们发布超过一百项功能更新。我从客户经理（AE）那里得到的反馈是：“天啊，我们发布的东西太多了。我根本跟不上。我不太确定……”所以他们有点在噪音中迷失了信号。因此，我认为这份文档是赋能团队、PMM等所有相关方的原材料。我们用它来梳理，比如：我们的下沉市场演示稿应该这样设计，或上行市场演示稿应该那样设计。这里是我们预期会出现拐点和赢得率提升的地方，或者是我们即将进入一个新市场以及我们预计的表现如何。所以这份大文档我投入了大量时间。我还会花很多时间与财务团队一起分析产品仪表板，每月分析产品收入来自哪里、哪里没有收入？我们是否感到意外？用户画像或企业规模是否与我们预期不同？另一件事是，我们还有所谓的“产品管线”。这是在Figma中整理的，我会思考未来三到四年，我们何时、何地、向哪些客户推出新产品。我会用一些小表情符号来标记，比如这是一个新的理想客户画像（ICP），或者我们认为这个产品可能成为价值超过一亿美元的业务，而另一个感觉更像是一个附加功能。这帮助我与高管团队（C-team）进行战略层面的沟通，比如：很好，今年我们计划在这个细分市场开始孵化两个产品。我们认为明年它们能产生X收入，之后……我们可以开始规划招聘计划。我们是想进行并购（M&A）？还是想自建、购买或合作？并开始进行这些不同的讨论。另一件事就是为我们一些核心的、令人兴奋的场景制作高层次的“愿景点击演示”。也就是说，确定三四件我们能向客户展示的、关于产品体验的非常好的事情，比如：“太好了。这里有一个很酷的场景：Vanta Agent将完成这件惊人的、我们独有的事情，因为我们拥有所有这些数据，它能做到。让我们完整地演示一遍这是什么，并维护好几个这样的重要导览。”这对我来说是另一件大事。这些可以说是我作为独立贡献者（IC）主要负责的事情。此外，我还有很多频繁的会议。我会进行所谓的“深度探讨”会议。每周我们都会有一大块时间，任何产品团队都可以进来，详细讲解不同领域。有时我会要求他们，比如：“很好，我想了解我们审计师体验的更新。谁能带我完整过一遍我们正在做的事情及其背后的愿景？我们来讨论一下。”这对我来说是另一个很好的检查点机制，有时是我要求了解某些事情，有时是团队主动来说：“嘿，我们想给你展示这个。关于如何以这种令人兴奋的新方式思考第三方风险，我们有了新想法。我们带来一起过一遍吧。”在这些会议中，我尽量保持高度视觉化导向。基本上就是：给我看客户体验，我们多快能拿出视觉稿或原型？

Brett：也许在此基础上延伸一下，如果有人观察你某一周的工作，包括所有会议、独立贡献者工作、你撰写文档时、你参加高管团队会议时，他们会注意到什么？一周中工作发生的主要大块时间是如何分配的？

Jeremy：因为我人在东海岸，早上我有大量的独立贡献者时间，我真的很努力保护这段时间。所以我想说，我试着以两到三个小时开始我的一天，要么是处理文档或Slack消息，要么是在写东西、设计东西、留下反馈，或者就是“巡视店铺”。我喜欢花很多时间直接进入产品并使用它。我有一套固定的场景流程，我会去运行，比如：“好的，今天我要专注于我们的风险管理产品，或者审计产品，或者客户信任。”我会逐一操作，发现并记录缺陷，同时思考：“这感觉对吗？我们方向正确吗？我们做到位了吗？”我想这可能会让人有些惊讶。我的日历时间被切分得很细。我想说这是这个角色的另一个难点，也是我一直努力改进的地方。但我通常每周都有几次客户会议，无论是路线图讨论、升级问题沟通，还是其他人只是想见面聊聊某事。所以总有一大块时间花在这上面。而且我相信一对一会议的价值，所以我也有相当数量的一对一会议。

Brett：过去25年里，如果你没有一对一会议，你就是个糟糕的管理者或领导者。而现在Jensen（可能指英伟达CEO黄仁勋）没有一对一会议，也没有人有任何一对一会议。这里的教训是，对于“我们做所有这些工作到底是为了什么？”没有人有任何真正原创性的思考。你认为这对你有效的原因是什么？你有什么样的理论？

杰里米：是的。我的意思是，我觉得人们开发产品时需要与人建立关系，即便只是紧凑的30分钟，这样的时间投入也真的很重要。我尽量让一对一会议成为有效的工作交流。所以我通常总是尝试让所有状态更新异步进行。你明白我的意思吗？直接私信我，我们找个文档协作，或者用任何方式，这样我们就能在会议中真正一起解决问题，或者进行诸如“我们应该这样做还是那样做”的深入讨论，真正把问题琢磨透。我发现这些会议非常重要。我每周都会与我的直接下属进行一对一交流，然后每月与组织内的许多其他人开会。我也会留意组织内那些有影响力的人，我可能会花更多时间与他们相处，这效果可能更好，也可能更糟。我在一些公司见过这种情况产生负面影响，可能会形成“圈内人”和“圈外人”的感觉。我认为我们在控制这方面做得很好。我也会非常审慎地对待这件事，但确实有些人对组织有着超乎寻常的影响力，我会仔细思考：“好的，他们现在关注什么？他们在想什么？我应该花多少时间与他们交流？”我花在团队首席工程师身上的时间比花在其他大多数工程师身上的都多，只是因为我真的认为他们拥有巨大的机会，能在技术层面领导整个公司甚至更广的范围。我想确保我们在产品目标以及如何与工程结合方面保持高度一致。我发现一对一会议很有帮助。我仍然设有办公室时间，任何人都可以预约。我想大概是每周一个半小时。有时没人预约，很多时候会有人预约，我们就进来聊聊各种事情。

布雷特：这是向整个公司开放的吗？还是你……

布雷特：是的。我的意思是，我认为公司和团队是复杂的有机体。所以我的猜测是，如果黄仁勋几十年来一直这样运营公司，并且每个人都接受了这种文化，或者存在积极的选择性偏差——即他们觉得这样有趣且愉快——那么这种方式可以运行得非常好。我认为危险的是，假设在那个有机体中成功的方法，你可以直接生搬硬套到你的公司，并指望它也能成功。

杰里米：是的，我同意。我觉得这就像一整套……是的，当你已经有一个系统以某种方式运行时，有一群人选择加入那个系统；而如果你说：“哦，我决定要这么做。”可能会有一群人说：“我当初可不是冲着这个来的。这感觉不好。”也许这是正确的决定，你也想这么做，但确实，情况大不相同。

Brett：回顾你在Vanta担任首席产品官的角色效能，并追溯过去20到25年的产品构建与工程经验，有哪些故事对你影响深远，塑造了你在当前角色中的特质？

Jeremy：我记得我确实经历过一些那种严厉的反馈谈话，你懂的，我当时觉得那些反馈很有帮助。比如有一次我的团队发布了一个产品，我知道那大概只是个B类水准的东西，但就是想赶紧把它推出去，因为我有点厌倦了和他们反复迭代。后来我记得自己因此被狠狠批评了一顿，因为我没有坚持更高的标准。我觉得那件事对我真的很有帮助。嗯，有两点：第一，当时的感觉确实不好受。第二，我当时的经理做得特别好的一点是，他在那次会议前就提醒了我，他说：“做好心理准备，这次谈话不会很愉快。你会得到一些非常直接的反馈……你懂的，来自CEO的反馈，关于这个产品的问题以及为什么它不达标。”这至少让我有了正确的心理预期。我想如果我毫无准备地走进那个会议，我的反应可能会更糟。所以我觉得这真的帮助我理解了。

我觉得我早期的一些经理非常注重设计技艺，他们帮助我理解了创造产品体验意味着什么，以及为什么拥有一个用户喜爱而不仅仅是功能可用的产品体验是有益的。我认为没有这些也能做成成功的生意，但对我来说那就不够激动人心了，这些经历帮助我建立了那些技能。回想我在微软的头几年，甚至在我去那里之前……我是说，我在高中自学了编程，在90年代末为别人建网站，那是我的高中兼职。所以我想到，我长期以来同时从事工程、产品和设计工作，这让我真正理解了这三者的价值。我认为这对我现在所做的工作起到了非常重要的塑造作用，因为我记得我曾有过一个想法，然后自己写所有代码、做设计、实现它、展示给客户看，再从头开始迭代。这让我对打造一个端到端产品所需的一切有了深刻体会，从而能做出更好的决策。

我觉得这些是其中一些重要的经历。在GitHub，对我来说最重要的事情是真正理解了高质量体验的标准是什么。我觉得GitHub过去在拉取请求（pull requests）等各个方面一直做得很好，我认为他们在这方面做得很出色。Max，我在那里时领导设计团队的负责人，真的让我大开眼界，让我明白了那个标准是什么，如何坚持那个标准，以及一个设计团队可以多么技术化。我们努力推动设计师去深入参与、负责前端、构建这些体验，并真正理解如何让一个东西落地的所有细节。根据我在其他公司的经验，我甚至不知道这可以成为一种选择。我根本不知道什么是可能的。现在我觉得，哇，太棒了。现在有了AI，希望情况会变得……对于没有那么多技术专长的人来说，做这些事情正变得越来越容易。但我觉得这些是我职业生涯中的几个重要时刻。

Brett：考虑到你现在已经在不同规模的公司工作过，而且现在在一家处于快速成长期的公司，似乎总会发生的一件事是，随着公司成长，速度会变慢。正如你描述的，作为同时掌管工程部的首席产品官，你经常思考的一件事就是如何加快节奏？在这个职位上，你对于如何加快速度、而不是随着公司每增加10个人就变得越来越慢，有什么心得？

Jeremy：是的，这非常难。

Brett：为什么这么难？

Jeremy：我认为对我们来说如此困难的部分原因是我们有太多新员工。我考虑的几件事是：我们一直在大量招聘。我想在工程、产品和设计团队里，我们每个季度招聘50人，这已经持续了一年多，而且我们将在未来12个月继续这样做。所以团队规模大幅增长。从Vanta整体来看，我想如果你在这里工作了三个月，你的资历就已经超过了公司里20%左右的人。所以新员工非常多。

我努力去做的一件大事，这个教训我是从负面经历中学到的——我觉得在GitHub的最后一年，或者可能是一年半的时间里——就是我们招聘得太快了。然后，就在所有人都刚刚上手的时候，我们解雇了一大批人。我当时觉得，这简直糟透了。首先，从人性角度，解雇人本身就让我感觉不好。有时你不得不这样做，但你会想，我们花了那么多时间招聘他们。这简直是雪上加霜，你懂我的意思吗？好不容易让他们上手，结果一切都没了。

Brett：每个团队都不同吗？

Brett：因为大多数人认为质量和速度是——

Jeremy：它们无法兼得。
