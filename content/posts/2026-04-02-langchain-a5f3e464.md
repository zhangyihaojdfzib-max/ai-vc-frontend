在过去的几周里，我们一直在通过 Deep Agent 测试框架对开源权重的大语言模型进行评估，初步结果显示，它们是一个可行的选择，可以替代或与闭源前沿模型一起使用。GLM-5（z.ai）和 MiniMax M2.7 在文件操作、工具使用和指令遵循等核心智能体任务上的得分与闭源前沿模型相当。

如果你一直通过大量开源基准测试（如 SWE-Rebench 和 Terminal Bench 2.0）关注开源模型的进展，这并不令人意外。工具调用是可靠的，指令遵循也是一致的。对于在生产环境中部署智能体的开发者来说，开源模型现在提供了一致性和可预测性，使得现实世界的工作流程更加可行。

## 为何选择开源模型

在探索开源模型时，构建者和客户往往会关注几个关键因素：**成本**、**延迟**和**任务性能**。

理想情况下，我们希望为每项任务都使用最智能、推理能力最强的前沿模型。但实际上，两个限制使其难以实现：成本和延迟。对于高吞吐量工作负载，闭源前沿模型的成本可能高出 8-10 倍，并且对于交互式产品中用户期望的响应时间来说，它们通常太慢。

为了更直观地理解定价：一个每天输出 1000 万 Token 的应用程序，在 Opus 4.6 上每天大约花费 250 美元，而在 MiniMax M2.7 上每天仅需约 12 美元。这大约是每年 8.7 万美元的差额。

开源模型通常比闭源前沿模型更小，并且可以在专门的推理基础设施上加速——像 Groq、Fireworks 和 Baseten 这样的提供商针对延迟和吞吐量进行了优化，其性能远超大多数团队自己所能达到的水平。OpenRouter 的数据显示，在 Baseten 上运行的 GLM-5 平均延迟为 0.65 秒，速度为每秒 70 个 Token，而 Claude Opus 4.6 的延迟为 2.56 秒，速度为每秒 34 个 Token。对于延迟敏感的产品来说，这个差距很难通过工程手段弥补。

## 我们的评估方法

我们已在《[How we build evals for Deep Agents](https://www.langchain.com/blog/how-we-build-evals-for-deep-agents)》一文中深入介绍了我们的评估方法。我们使用托管推理提供商运行评估，但 Deep Agents 也可以通过 Ollama、vLLM 等工具使用完全本地和私有的模型运行。

对于开源模型，我们运行了七个评估类别：文件操作、工具使用、检索、对话、记忆、摘要和“单元测试”。这些涵盖了测试基础能力的任务：模型能否可靠地调用工具、遵循结构化指令以及对文件进行操作？这些能力决定了模型是否能在智能体测试框架中使用。

每个评估案例都定义了成功断言（决定正确性的硬性失败检查）和效率断言（衡量模型如何达成目标的软性检查）。我们报告四个指标：

- **正确性** — 模型解决的测试比例：`通过数 / 总数`。得分 0.68 意味着 68% 的测试用例被正确解决。这是主要的质量信号。
- **解决率** — 准确性和速度的综合衡量。对于每个测试，我们计算 `预期步数 / 实际耗时秒数`；失败的测试贡献为零。最终得分是所有测试的平均值。越高越好——既能正确又能快速解决任务的模型得分最高。
- **步数比** — 模型实际采取的智能体步数与预期步数之比，在所有测试中汇总：`总实际步数 / 总预期步数`。值为 1.0 表示模型使用了恰好预期的步数。高于 1.0 意味着它需要更多步数（效率较低）；低于 1.0 意味着它需要的步数比最初预期的少。
- **工具调用比** — 与步数比概念相同，但计算的是单个工具调用次数而不是步数。1.0 表示符合预算，高于表示超出预算，低于表示低于预算。

步数比和工具调用比是**效率**指标。它们不影响测试是否通过，但揭示了模型达到答案的经济性。一个用 2 步而不是预期 5 步解决任务的模型，既正确又高效。

#### 自己动手：本地运行 Deep Agent 评估

我们的 CI 在 52 个模型上运行相同的评估套件，这些模型被分成若干组——包括一个 `open` 组（baseten:zai-org/GLM-5, ollama:minimax-m2.7:cloud, ollama:nemotron-3-super），它在每个评估工作流中运行。你可以针对任何模型组运行：

对所有开源模型运行评估：`pytest tests/evals --model-group open`
针对特定模型运行：`pytest tests/evals --model baseten:zai-org/GLM-5`

这使得在相同任务、相同评分标准下，直接比较开源模型之间以及开源模型与闭源前沿模型变得简单明了。

## 在 Deep Agents SDK 中使用开源模型

切换到开源模型只需更改一行代码：

GLM-5：

```python
# pip install langchain-baseten
from deepagents import create_deep_agent

agent = create_deep_agent(model="baseten:zai-org/GLM-5")

```

MiniMax M2.7：

```python
# pip install langchain-openrouter
from deepagents import create_deep_agent

agent = create_deep_agent(model="openrouter:minimax/minimax-m2.7")

```

就这样。测试框架会处理其余的事情——它会检测模型的上下文窗口大小，禁用不支持的模态，并将正确的身份信息注入系统提示词，以便智能体知道它正在处理什么。

同一个开源模型通常可以通过多个提供商获得。选择符合你限制条件的那个。例如，GLM-5 可以作为 `baseten:zai-org/GLM-5`、`fireworks:fireworks/glm-5` 或用于自托管的 `ollama:glm-5` 使用。相同的模型，相同的测试框架，不同的基础设施。

LangChain 为最流行的开源模型提供商提供支持。我们为此版本测试的提供商有：Baseten、Fireworks、Groq、OpenRouter 和 Ollama（云）。

### 为你的模型调整测试框架级别

开源模型与闭源前沿模型具有不同的上下文窗口、不同的工具调用格式和不同的失败模式。Deep Agents 测试框架吸收了这些差异，因此你无需操心：

- **模型身份注入** — 系统提示词在运行时被修补，包含模型的名称、提供商、上下文限制和支持的模态。智能体知道它是什么以及它能做什么。
- **上下文管理** — 压缩、卸载和摘要阈值会根据模型的实际上下文窗口进行调整，而不是硬编码的默认值。一个拥有 4K 上下文的模型会比拥有 1M 上下文的 Opus 模型得到更积极的压缩处理。

### Deep Agents CLI

每个模型也可以在 Deep Agents CLI 中使用。**Deep Agents CLI** 是我们的开源编码智能体，是 Claude Code 的替代品。

除了 Deep Agents SDK 中的所有功能外，CLI 还支持**运行时模型切换**。我们引入了一个新的中间件（ConfigurableModelMiddleware），以实无需重启智能体即可在会话中途切换模型。这使得诸如使用前沿模型进行规划、使用开源模型进行执行的模式成为可能。

你可以使用 `/model` 斜杠命令在会话中途切换模型。这使得诸如使用前沿模型开始任务进行规划，然后切换到更便宜的开源模型进行执行的模式成为可能。
