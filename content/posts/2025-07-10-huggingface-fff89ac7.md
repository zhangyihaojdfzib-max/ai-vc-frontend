# 异步机器人推理：解耦动作预测与执行

TL;DR 机器人策略日益庞大，它们预测的是未来一系列动作而非单个下一步动作。这导致机器人在等待新动作执行时处于闲置状态，造成明显的执行延迟并缺乏响应性。异步推理通过解耦动作预测与动作执行，收紧控制循环，消除运行时延迟，从而实现更具适应性的控制。本篇博文将介绍异步推理的基本原理，以及如何利用它提升机器人策略在现实世界中的性能。

## 异步推理：深入解析

通过异步推理，我们将动作执行与动作预测解耦。考虑到当前流行模型如 [ACT]、[OpenVLA]、[PI0] 和 [SmolVLA] 倾向于输出动作块 \(a_{t:t+H}\) 而非给定观测 \(o_t\) 下的单个动作 \(a_t\)，这一点尤为重要。
您可以通过使用 LeRobot 运行所有这些模型来自行验证。

顺序使用动作块会导致（1）运行时延迟，影响任务执行时间，以及（2）由于以广泛开环方式执行动作而缺乏响应性。
异步推理通过解耦动作预测与动作执行，缓解了这两个限制。
我们在 SmolVLA 中引入了异步推理，并发现它在保持可比任务成功率的同时，使任务完成时间缩短了约2倍。

具体而言，我们设计了一个包含两个组件的系统，策略推理和动作执行在两个不同的进程中执行，可能运行在通过网络连接的两台不同机器上：

- 一个**策略服务器**，部署在加速硬件上，能够使用比现实世界机器人分配的计算资源更多的资源来运行推理。
- 一个**机器人客户端**，将接收到的动作加入队列并在计算下一个动作块时执行它们。

策略服务器和机器人客户端之间的通信依赖于 gRPC，它保证了比同类 REST API 快约5倍的性能。这一切的结果是一个**从不**等待推理的机器人。

异步推理示意图，突出显示：（1）客户端发送第一个观测进行推理，不久后收到第一个动作块；（2）客户端在当前动作块尚未耗尽时发送另一个观测进行处理；（3）客户端收到一个更新的动作块，并将其与之前正在执行的动作块的剩余部分进行聚合。

## 1. 顺序推理为何不足

假设一个策略 \(\pi\) 将当前观测 \(o_t\) 映射到未来 \(H\) 个动作序列。
形式上，\(\pi : \mathcal{O} \;\mapsto\; \mathcal{A}, \mathbf{A}_t = \begin{pmatrix} a_{t}, & a_{t+1}, & \dots & a_{t+H} \end{pmatrix} = \pi(o_t)\)。

因此，传统的控制循环将包含以下步骤：

1. 捕获 \(o_t\)。
2. 运行 \(\pi(o_t)\) 以获得 \(\mathbf{A}_t = \pi(o_t)\)。
3. 将 \(\mathbf{A_t}\) 加入队列，并开始从队列中取出动作执行。
4. 如果队列为空，则等待 \(\mathbf{A}_{t+H}\)，否则重复步骤3。

在步骤2期间，机器人处于**闲置**状态。延迟随着模型大小而增长（并且模型往往随着时间的推移越来越庞大），并且可能迅速主导交互时间（通常在 1/fps 左右），如下方视频所示（来自我们的 Discord 社区 🤗）：

这直接导致（1）任务完成时间方面的性能下降——机器人需要等待下一个动作块被计算出来——以及（2）响应性降低，原因是（2.1）在有可用动作时以广泛开环方式执行，以及（2.2）在等待下一个动作块时完全闲置。

## 2. 异步推理，简述

我们的系统通过重叠计算与执行来消除闲置期：

1. **机器人客户端**将最新观测流式传输给**策略服务器**。
2. 当服务器执行推理时，客户端执行**当前队列**中的动作。
3. 新动作到达，被合并到队列中，循环继续。

关键思想是，机器人已经知道接下来几个时间步要做什么，因此它可以在服务器上计算新动作时继续移动。

**异步推理**通过解耦这两个过程（可能让它们在通过网络连接的完全不同的机器上运行），在时间上重叠执行当前动作块与计算下一个动作块。

这形成了一个更紧密的控制循环，以及一个从不等待推理的机器人。反过来，这带来了任务完成时间约2倍的加速（同时保持可比的任务成功率），以及来自更紧密循环的更具适应性的控制（见下方视频）。

## 3. 系统架构

由于 gRPC 基于 HTTP/2 并使用协议缓冲区，它实现了低延迟的二进制消息传递和开箱即用的双向流，这反过来帮助我们维持更紧密的控制循环和低于100毫秒的往返延迟（在我们的本地网络上，并在 NVIDIA RTX 4090 上托管 SmolVLA）。

**机器人客户端**在机载设备上运行，并通过 gRPC 将观测流式传输给**策略服务器**。**策略服务器**为推理准备接收到的观测，并将一个动作块发送回**机器人客户端**。

### 机器人客户端

![从客户端视角](/images/posts/429dee10cd6d.png)

从客户端的角度来看，观测根据本地队列状态流式传输到服务器。传入的动作块会与当前可用的动作队列在重叠部分进行聚合。

**机器人客户端**维护一个本地动作队列，并遵循一个简单而有效的策略：**当队列长度低于可配置阈值时发送新的观测**（在 SmolVLA 论文中为 \(g\)，在代码中为 `chunk_size_threshold`）。
该阈值表示为最大块大小的一部分，作为一个触发条件，用于平衡计算负载与响应性。

![客户端到服务器](/images/posts/9fd67b61581f.png)

客户端根据本地队列状态，将观测流式传输到服务器。

从客户端的角度来看，过程展开如下：

1.  队列监控：客户端持续监控其动作队列长度，并与一个**分块大小阈值**参数进行比较。当队列长度低于此阈值时，表示应发送新的观测数据进行处理。
2.  观测流式传输：一旦满足阈值条件，客户端会捕获当前观测数据，并通过 gRPC 将其流式传输到 **PolicyServer**。关键在于，**观测数据采用流式传输而非通过一元 RPC 发送**，因为它们通常超过 4MB 的最大消息大小限制（高分辨率的多摄像头捕获会导致这种情况）。
3.  动作分块聚合：当从服务器接收到新的动作分块时，客户端会将其与当前队列中剩余的动作在重叠部分进行合并。此时，**自定义聚合器**开始发挥作用，以不同方式处理当前分块与传入分块之间的重叠部分。目前，我们支持通过指定一个自定义的 `aggregate_fn(chunk1: torch.Tensor, chunk2: torch.Tensor) -> torch.Tensor` 函数来灵活地聚合分块，该函数会为每个重叠的时间步调用，并且可以由用户提供。
    重叠部分（图中以浅蓝色显示）需要仔细处理。我们可以设计不同的聚合策略：
    *   **替换**：直接用较新的预测动作替换重叠的动作。
    *   **加权混合**：使用时间权重组合重叠的动作（时间上更接近的动作获得更高权重）。

该系统高度可配置，因为可以根据网络延迟、模型推理时间和期望的响应速度来调整分块大小阈值。
较低的阈值意味着更频繁的更新（以及更高的计算成本），而较高的阈值则以可能导致的队列饥饿为代价来减少通信开销。
最后，我们通常在一个线程中从 **PolicyServer** 接收动作，并在另一个线程中执行它们。这使得客户端可以在单独的线程中监听传入的分块，而不会阻塞执行，并且始终消耗当前分块，直到新的分块完全可用。

### 策略服务器

从 **RobotClient** 接收到观测数据后，**PolicyServer** 会执行必要的观测数据清理工作，以使接收到的观测数据准备好进行推理。此过程如下图所示：

![服务器流水线](/images/posts/b4a5ded27c3e.png)

在服务器上运行的观测数据清理流水线，突出了与（1）键匹配、（2）预处理和（3）推理准备相关的三个主要步骤。

一旦观测数据准备就绪，会将其与上一次用于推理的观测数据进行比较。
这避免了陷入循环处理非常相似的观测数据，从而触发不必要的推理和执行相似动作（这反过来又导致再次处理非常相似的观测数据）。
我们根据观测数据的关节空间相似性进行比较，这为我们提供了一种近似且快速测量机器人变化的方法。显然，此指标无法适应环境的动态变化（例如物体位置改变或施加扰动），但我们发现对于大多数情况来说这是一个很好的折衷方案，并且能非常有效地避免不必要的推理和状态崩溃。
关键的是，**RobotClient** 保留了对给定观测数据是否必须被处理的控制权，以避免死锁。
客户端发送并标记为 `must_go=True` 的观测数据，无论相似性指标如何，都会被处理。

![策略工作流](/images/posts/27d4c2785d05.png)

策略工作流，其中传入的观测数据会与上一次用于推理的观测数据进行比较，只有在足够不同或标记为 `must_go` 时才会被处理。

最后，为确保 **PolicyServer** 始终处理最新的可用观测数据，我们会阻塞传入的观测数据，直到前一个观测数据被成功处理。为此，我们利用 **PolicyServer** 上的队列来确保传入的观测数据在服务器准备好处理它们之前不会被加入队列（见下文）。

![客户端轮询服务器](/images/posts/4cf83ef48f42.png)

客户端每 1/fps 秒轮询一次服务器，但观测数据在前一个观测数据被成功处理之前不会被加入队列进行处理。

## 4. 分析异步推理

出于所有实际目的，在异步推理中有两个重要的时间尺度：

*   **环境步长** `environment_dt = 1/fps`，描述机器人执行动作的速度。
*   **推理延迟** `inference_time`：前向传播 + 网络往返时间。我们可以假设网络往返时间相对于策略推理时间可以忽略不计，尽管在某些设置中可能并非如此。

重要的是，比率 `c = environment_dt / inference_time` 会导致不同的行为：

*   `c << 1`：环境演化速度快于推理速度。在这种情况下，队列迅速清空，我们退化为顺序控制。
*   `c >= 1`：服务器能够跟上。队列始终（几乎）是满的。

关键的是，`c` 会影响在任何给定时间队列中可用动作的数量。为避免上述的顺序限制控制，可以：

1.  为策略服务器使用更多计算资源，将服务器托管在 GPU 上，通过分配更多计算资源来减少 `inference_time`。
2.  更频繁地向服务器发送观测数据，当队列长度 `k` 下降到其最大容量 `H` 的一个分数 `g = k/H` 以下时，发送新的观测数据。`g = 0` 重现顺序推理（空队列，等待）。`g = 1` 每个时间步都发送观测数据（最大计算量，最小延迟）。

    *   `g = 0` 重现顺序推理（空队列，等待）。
    *   `g = 1` 每个时间步都发送观测数据（最大计算量，最小延迟）。

实验（见下图）表明，当发送的观测数据未被过滤掉（它们都是 must-go）时，`g ≈ 0.7` 提供了一个良好的折衷方案。我们建议将 `g` 设置为 0.5，并遵循 **我们的文档** 根据您的需求调整此参数。

![队列](/images/posts/97907b2cebfe.png)

队列中任意时刻可用动作的数量，作为g的函数。g值越大，更新越频繁，计算成本也越高。g值越接近0，则越接近顺序推理（队列为空，等待）。在我们的实验中，发现g~0.7是一个良好的折中点。
