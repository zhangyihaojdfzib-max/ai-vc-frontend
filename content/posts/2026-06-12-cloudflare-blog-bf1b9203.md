# 扩展安全洞察：如何实现全球扫描能力提升10倍

2026-06-12

- Dave Baxter

![](/images/posts/9b0cdce0124a.png)

安全洞察为每个Cloudflare账户提供可操作的安全建议。为了发现这些洞察，我们定期对所有账户、区域和DNS记录执行扫描，寻找潜在的安全风险和配置错误。

然而，出现了两个关键问题。首先，我们的扫描频率过低。扫描每两周才执行一次，因此新引入的安全风险可能长达两周未被发现。其次，自动扫描对许多免费计划账户是可选加入的——这意味着大量账户根本没有被扫描。

扫描频率过低或缺失的风险正在上升：随着自动化攻击加速，发现安全配置错误的时间窗口正在缩小。确保我们为所有客户发现这些问题，对于构建更美好互联网的目标至关重要。

我们计算得出，要提高扫描频率并为所有账户启用自动扫描，需要将扫描吞吐量平均提升约10倍——从每秒10次扫描提升到每秒100次。但我们的系统已经在负载中挣扎：数百万事件填满了待处理的后备队列；API频繁超时；进程不断崩溃。我们需要修复系统，并且需要让它能够扩展。

这就是我们如何将安全洞察的扫描吞吐量提升超过10倍，为数百万客户启用安全洞察，并将所有客户的扫描频率提高一倍的故事。请继续阅读，了解我们如何实现这些改进。

## 我们如何扫描安全洞察

从高层来看，我们的自动安全扫描由调度器触发。当账户或区域需要扫描时，调度器向Apache Kafka（一个开源分布式事件流平台）发布一条或多条消息。这些消息分发到多个检查器：专门扫描特定资产或配置的Go微服务。

对于每条消息，每个检查器将其结果（发现的安全洞察）发送到我们的内部API，然后API将这些结果持久化到Postgres数据库中。

## 实现扩展

### 扩展Kafka

Apache Kafka严格来说不是一个队列：它是一个分区事件流（尽管最近增加了队列语义）。在分区内，消息必须按顺序消费和处理。这与典型队列不同，典型队列中消息可以按顺序消费但无序处理。因此，在消费者组内，每个分区只能有一个活跃消费者。

这对我们有两个影响：

- 处理缓慢的消息会阻塞消费者处理下一条消息
- 对于每个检查器，我们只能拥有与分区数量相同的消费者（每个检查器有自己的消费者组）

处理缓慢的消息会阻塞消费者处理下一条消息

我们本可以尝试通过增加分区来扩展。但这会增加Kafka代理本身的资源使用，而该代理被许多其他服务共享。我们将其作为最后手段，首先致力于改进代码和架构。

### 引入并行处理

虽然我们只能按顺序消费消息，但没有任何限制阻止我们一次消费多条消息。

我们修改了检查器，使其批量消费消息，并在单独的goroutine中处理每条消息。权衡之处在于：如果进程在批处理中途崩溃，我们需要重做更多工作，并且内存使用会略有增加。在我们的案例中，这两点都是可以接受的。

### 避免队头阻塞

某些检查器处理的某些消息比其他消息耗时更长。例如，一个账户/区域可能拥有比另一个多得多资产。在最坏情况下，这些消息可能需要几分钟或几小时处理，而平均情况只需几秒或几毫秒。

我们选择了一个非常简单的方法：将消费者组和检查器分为两类——"慢车道"和"快车道"。我们可以快速判断消息处理是慢还是快。如果"快车道"检查器遇到慢消息，它会跳过该消息。

这解决了问题：慢消息获得专用资源和时间，以最小延迟处理；快消息能够以常规快速速度继续处理。

## 优化数据库查询

我们发现的每个洞察都会被写入Postgres数据库。这由单个API端点处理，检查器通过该端点提交洞察列表。实现如下：

```Go
for _, issue := range issues {
	_, err = tx.Exec(ctx, `INSERT INTO table ... VALUES ($1, $2, ...) ON CONFLICT DO UPDATE ...`, ...)
	if err != nil {
		return err
	}
}
```

敏锐的读者会注意到，对于大型洞察集，这段代码每条洞察都会进行一次数据库往返。最大观测规模为500,000条，这意味着单次API调用中进行了50万次往返、查询和事务。

我们最初尝试了Postgres中批量插入的黄金标准：COPY到临时表。然而，我们发现这种方法导致Postgres系统表膨胀。

我们最终采用了混合方法：

- 当问题数量低于阈值时使用UNNEST
- 当问题数量超过阈值时使用COPY

这提供了两全其美的效果：对于大型洞察集实现相当快的插入（秒级），对于小型洞察集实现更快的插入（毫秒级）。

## 调查API超时

在尝试扩展时，我们注意到内部API出现几个奇怪行为：

- 大量请求触发客户端超时
- 许多检查器将20-90%的处理时间花费在单个API调用上
- 触发大量扫描时，吞吐量开始很高但随后恶化

所有这些问题的根本原因相同：延迟。

我们的主数据库位于俄勒冈州波特兰。然而，我们的API在波特兰和阿姆斯特丹以双活模式运行。即使以光速传输，波特兰和阿姆斯特丹之间的往返延迟也达到50毫秒。

由于这种延迟，来自阿姆斯特丹API实例的数据库查询耗时更长，占用了客户端连接池的连接。由于我们向API发送大量请求，连接池迅速耗尽，导致等待空闲连接时超时。我们的平均API调用在波特兰只需10毫秒完成，但在阿姆斯特丹几乎需要3秒！

但消息吞吐量为何下降？每个检查器进程被分配消费Kafka流的一组分区。我们的API是负载均衡的。由于我们在进程生命周期内保持连接打开，某些进程连接到阿姆斯特丹API，其他进程连接到波特兰API。连接到波特兰的分区处理迅速，但由阿姆斯特丹连接的进程消费的分区则滞后：

根据我们的一个检查器，按分区显示的Kafka滞后（单个消费者组内等待处理的消息数量）。注意，本例中有30个分区。恰好有15个分区出现滞后（在03/10 03:00之后才达到或接近零的线条）。这是因为负载均衡器在我们的API端点之间均匀分配流量。

这是一个简单的修复：我们将API切换为主动-被动模式，确保主动API跟随我们的主数据库。我们的延迟问题一夜之间消失了。

## 重新思考调度器

我们已经扩展了Kafka。我们已经优化了数据库查询。我们已经修复了API。然而，我们仍然有一个问题：我们需要确保扫描在时间上大致均匀分布。将所有扫描同时排队是不可行的，因为我们的Kafka主题使用基于时间的保留策略：扫描会在Kafka中堆积，最终在处理之前被删除。

我们的调度器不擅长均匀分布扫描。在给定时间触发的扫描数量是突增且不可预测的。在一周中的某些时间点，数十万次扫描会在几分钟内相继触发。这是怎么回事？

调度器在固定的重复周期触发扫描。用伪代码表示，调度器看起来像这样：

```Rust
无限循环：
    查找 last_scheduled_at + 扫描频率 <= 当前时间的账户
    对于每个账户：
        触发该账户的扫描
        触发该账户下所有区域的扫描
        更新 last_scheduled_at = 当前时间
```

我们很快注意到，数据库中大量账户的 last_scheduled_at 相似，这导致了部分不均匀性。

然而，即使分布完全均匀，提高扫描频率也会加剧这个问题。例如，将扫描频率从每15天改为每7天，意味着53%的账户会突然需要进行扫描。

这个逻辑还存在另一个问题。有些账户拥有非常多的区域。当这些账户被调度时，会引发其所有区域的级联扫描。这使我们的Kafka分区饱和，并导致规模小得多的账户的扫描出现延迟。

为了解决这些问题，我们做了三项关键更改：

- 独立于账户调度区域：每个区域拥有自己的 last_scheduled_at 字段。
- 随机化现有账户和区域的 last_scheduled_at 时间。
- 为扫描调度引入自适应速率限制。

独立调度区域是解决大账户问题的明显方法。随机化 last_scheduled_at 时间（并确保在此过程中没有扫描被延迟）使我们能够修复数据库中现有的不均匀性。

自适应速率限制稍微更有趣一些。速率限制使我们能够解决更改扫描频率时扫描激增的问题。例如，如果我们想将扫描频率提高到每7天一次，并且我们有5000万个账户，那么大约83次扫描/秒的速率限制将确保它们在7天内均匀分布。

但是，如果我们再增加1000万个账户呢？那么，这个速率限制将迫使我们花费8天时间来扫描所有这些账户。这就是“自适应”部分的用武之地：速率限制每半小时根据我们拥有的账户和区域总数以及我们的扫描频率异步重新计算。这确保即使我们新增数千或数百万个账户和区域，也能继续按时扫描。

```Go
func computeRate(free, pro, biz, ent int64) rate.Limit {
   r := float64(free)/freeScanInterval.Seconds() +
      float64(pro)/proScanInterval.Seconds() +
      float64(biz)/bizScanInterval.Seconds() +
      float64(ent)/entScanInterval.Seconds()


   // 防止零计数。我们始终希望每秒至少调度一次扫描。
   if r < 1 {
      r = 1
   }


   // 将速率限制提高到“完美”值以上，以便在出现任何停机或负载峰值时留有缓冲。
   r *= rateLimitBufferFactor


   return rate.Limit(r)
}
```
