在 Vercel，我们与开源的关系是根基性的。我们开发开源软件并非为了盈利，而是旨在建立一项持久的事业，使我们能够持续开发卓越的开源软件。我们坚信，无论是否为 Vercel 客户，都应提升全球各地软件的默认质量。水涨则船高。

### 反对锁定

我们在 Vercel 构建了 Next.js，并在此过程中收获良多。

我们见证了将开源与托管基础设施同步建设，如何能创造出卓越的开发者体验并产生巨大影响。但若将项目与单一平台绑定过紧，即使能提升质量和采用率，也可能引发对开放性和控制权的质疑。

即使锁定并非本意，情况也是如此。

### 树立旗帜

我们信奉开源、社区、可移植性和选择自由，因此我们希望借此机会，将这一承诺阐述为一项新的“开放 SDK”战略。我们承诺如下：

1.  默认开放：我们承诺尽可能公开构建我们的框架、SDK 和工具，并在所有情况下默认采用最宽松的开源许可证。
2.  松耦合，高内聚：我们希望所构建的产品在 Vercel 上运行极佳，但不会以锁定为代价。我们承诺构建开放、可移植且可在任何平台上使用的框架、SDK 和工具。
3.  先创新，后泛化：我们将首先在 Vercel 上构建，以便能以最快速度迭代，确保最佳的端到端开发者和用户体验。当我们确信项目成熟时，我们将投入资源确保我们的 SDK 和工具可部署到任何平台。

这是我们希望您能监督我们履行的承诺。长远来看，我们相信这种开放性意味着您将更有可能选择 Vercel。不是因为您不得不如此，而是因为您主动选择如此。

### 战略缘起

此战略源于一系列现实观察和产品决策：

1.  React 的普及性：许多开发者选择 React 的一个有力原因是，它对部署方式和地点没有限制。
2.  Next.js 作为独立选择：开发者希望在其他托管平台上充分发挥 Next.js 的潜力。这一愿望已成为我们开放战略的北极星。
3.  AI SDK：当 Vercel 收购 `ai` npm 包名称时，我们深知有责任使其与平台和框架无关。这一点不容妥协。
4.  Nuxt 与 NuxtLabs：随着 NuxtLabs 加入 Vercel，以及 Nuxt 和 Nitro 团队的核心成员现在与我们并肩工作，我们必须明确表态。我们希望 Vercel 成为开源创作者在为开放网络构建时心驰神往之地。

### 战略应用

这不仅关乎 Next.js 或 AI SDK 等单一框架，也未必适用于 Vercel 的商业产品，但我们正积极将这一战略应用于不断增长的库和框架家族：

-   Nuxt：将同样的设计原则引入 Vue 生态系统。
-   Svelte：支持具有灵活部署选项的高性能应用。
-   Flags：一个无需特定功能标记提供商的、可移植的功能标记系统。
