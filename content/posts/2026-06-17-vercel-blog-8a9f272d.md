让您的Agent（智能体）能够访问您的工具、数据和各项服务，正是其价值所在。随着Agent在多个系统中执行更深入的工作，对这种访问进行身份验证和授权，便成为了您应用架构的核心。

如今，Agent的访问权限通常通过存储在环境变量中的长期有效提供商Token来授予，这些Token为Agent可能需要的一切而预置。这些Token在所有用户间共享，永不过期，并且赋予Agent执行任何任务（无论任务大小）的完全权限。

一个保险库能让Token更难被盗取，但并不能降低其危险性。问题在于Token泄露时会发生什么：它能触及的一切都将暴露。

我们构建了Vercel Connect来解决这个问题。Vercel Connect现已进入公开测试阶段，它用运行时凭证交换取代了存储的Token。您只需注册一次连接器。当您的Agent有任务要执行时，您的应用向Vercel Connect证明其身份，并获取一个针对该任务范围、短期有效的凭证。您之前使用Token完成的一切功能依然有效。区别在于，Agent现在是每次请求访问权限，而不是持有固定权限。

## 链接到标题一次注册连接器，即可在项目和环境中重复使用

连接器是您的Vercel团队与Slack或GitHub等提供商之间的可复用连接。您可以通过仪表盘或CLI创建一次，然后将其附加到需要它的项目和环境中，并设置项目级别的访问控制。

```
vercel connect create slack --name mybot
```

创建一个Slack连接器

与提供商的关系变成了一个您可以查看和管理的单一实体，而不是分散在十几个环境变量面板中，以至于轮换密钥时需要找到每一个副本。

您的编码Agent也可以运行此设置。使用 `npx skills add vercel/vercel-plugin --skill vercel-connect` 安装vercel-connect技能，它就能为您创建和附加连接器。

## 链接到标题在运行时请求限定范围的Token

连接器就位后，Agent仅在需要执行任务时才请求凭证。`@vercel/connect` SDK会返回一个您可以立即用于提供商API的Token，您的应用中不会存储任何提供商密钥。

```
1import { getToken } from '@vercel/connect';2
3const token = await getToken('slack/mybot', {4  subject: { type: 'app' },5});
```

在运行时请求一个Token

Token是短期有效的，其生命周期取决于提供商。SDK会自动刷新它们，因此您无需手动轮换密钥。这就引出了一个问题：如果您的应用不持有任何密钥，那凭什么证明它有权请求呢？

## 链接到标题应用通过OIDC证明其身份

这个凭证是您的应用已经拥有的身份。Vercel上的每个部署都会获得一个OIDC身份，当您的应用或Agent请求Token时，SDK会将该身份提交给Vercel Connect。Vercel Connect验证该身份，检查项目和环境是否被允许使用该连接器，然后返回提供商凭证。这个往返过程就是运行时凭证交换。

在本地开发期间，通过 `vercel link` 和 `vercel env pull` 也可以使用相同的身份。在Vercel之外，SDK接受Vercel访问令牌。无论哪种方式，您的应用中都没有提供商密钥可以泄露、提交或在环境之间复制。

## 链接到标题将每个Token的范围精确限定到任务所需

并非所有任务都需要相同的权限范围，即使是在同一个Agent内部。一个步骤可能只需读取仓库，而下一步可能需要创建一个Issue。每个步骤都精确请求它所需的访问权限，并且请求本身会设定限制。一个请求可以包含：

- 提供商范围
- 安装ID
- 资源限制
- 提供商特定的授权详情

GitHub是最典型的例子，因为它可以将Token限制在特定的仓库和权限内。

```
1import { getToken } from '@vercel/connect';2
3const token = await getToken('github/mybot', {4  subject: { type: 'app' },5  authorizationDetails: [6    {7      type: 'github_app_installation',8      repositories: ['myorg/repo1'], 9      permissions: ['contents:read'], 10    },11  ],12});
```

将Token范围限定到一个仓库，只读

部署Agent只能读取那个仓库，不能做其他任何事情。一个细粒度的GitHub App安装也可以做到范围狭窄，但安装是一种常设授权，一次性设置后便一直信任。而这个限制仅针对一个请求、一个任务。最小权限原则成为了请求的形态。

## 链接到标题代表特定用户操作，支持按用户进行Token范围限定

一个共享的机器人Token会让每个用户的请求都具有相同的身份和权限范围。Vercel Connect允许您设置该身份。将 `subject` 从应用切换到指定用户，Token就会代表该用户行事，其范围限定在该用户已授权的范围内。

```
1import { getToken } from '@vercel/connect';2
3const token = await getToken('linear/mybot', {4  subject: { type: 'user', id: 'user_123' },5});
```

为特定用户请求一个Token

当用户首次授予访问权限时，`startAuthorization` 会通过回调URL、Webhook或设备代码运行授权流程。之后，Agent就可以作为该用户请求Token了。

## 链接到标题按环境限制访问，并在需要时撤销

连接器附加到您选择的项目和环境中，因此您可以为开发、预览和生产环境分别运行独立的连接器，而不是用一个连接器指向所有三个环境。当每个环境都有自己的连接器，并带有独立的授权许可和范围时，在开发环境中被泄露的凭证就无法在生产环境中重放。

独立的连接器限制了凭证的工作范围，但它们无法收回已经发出的访问权限。这通常是令人头疼的部分。对于存储的Token，这意味着需要轮换。您需要生成一个新密钥，更新所有旧密钥存在的地方，然后重新部署所有依赖它的东西。使用Vercel Connect，您可以撤销连接器的Token，无论是您自己的Token还是所有Token。

```
vercel connect revoke-tokens slack/mybot --my-tokens
vercel connect revoke-tokens slack/mybot --all-tokens
```

撤销连接器的Token

撤销操作的具体效果取决于提供商。如果提供商支持撤销，Vercel Connect会在提供商处撤销该Token。如果提供商不支持，Vercel Connect会停止为该授权发放新Token，而已发放的Token在提供商处将保持有效直至过期。对于任何没有提供撤销API的提供商来说，这是一个实际的限制，而提供商保持Token有效的时间越短，这个风险窗口就越小。

## 链接到标题通过经过验证的Slack触发器驱动事件驱动的Agent

到目前为止，都是您的Agent主动发起连接。它请求Token，并在有任务时调用服务。触发器则反向工作。一个已连接的服务向您的应用发送事件，然后您的Agent做出响应。

Vercel Connect接收提供商的Webhook，进行验证，然后将其转发到您的项目。触发器转发功能目前处于测试阶段，现已支持Slack、GitHub和Linear。一个Slack连接器可以将其经过验证的Webhook转发到最多三个您的项目，因此Slack中的一条消息可以唤醒一个Agent来执行操作。

整个流程端到端运行，您的应用中无需存储提供商密钥：

- 用户在Slack中发布一条消息。
- Slack将事件发送给Vercel Connect。

- 您的应用验证该认证，然后请求一个限定范围的运行时Token。
- Agent执行操作并做出响应。

## Link to headingVercel Connect 与您已有的代码无缝衔接

一切的核心是一次调用。无论您的 Agent（智能体）是构建在 AI SDK 之上、作为后台任务运行，还是您自己编写的循环，它都通过 `getToken` 以相同的方式请求一个 Token。围绕这次调用，有适用于您现有技术栈的适配器。`Better Auth`（`@vercel/connect/betterauth`）和 `Auth.js`（`@vercel/connect/authjs`）以它们期望的格式提供提供商配置，而 `@vercel/connect/ai-sdk` 和 `@vercel/connect/mcp` 则为 AI SDK 工具和 MCP 客户端提供相同的功能。`Nuxt starter` 为您提供了一个可直接构建的工作应用，已连接 GitHub 和 Linear，无需提供商密钥，其数据库中也未存储 OAuth 刷新 Token。

框架可以更进一步，使连接本身变为声明式。在 Vercel 的开源 Agent（智能体）框架 `eve` 中，一个连接就是一个文件，而 `@vercel/connect/eve` 适配器提供该连接的凭证。

```
1import { defineMcpClientConnection } from "eve/connections";2import { connect } from "@vercel/connect/eve";3
4export default defineMcpClientConnection({5  url: "https://mcp.linear.app/sse",6  auth: connect("linear/mybot"),7});
```

一个通过 MCP 连接到 Linear 的 eve 连接

Agent（智能体）的代码中没有 Token 处理逻辑，因为 `connect` 将授权流程、刷新和错误情况映射到了 eve 上。任何支持 OAuth 的 MCP 服务器都可以通过其 URL 成为一个连接器，这就是 `mcp.linear.app` 最终采用与 Slack 或 GitHub 相同的限定作用域 Token 模型的方式。

同一个适配器也连接了 Slack 频道。一次 `connectSlackCredentials` 调用涵盖了双向操作：用于发送消息的机器人凭证和用于接收消息的 Webhook 验证。

```
1import { slackRoute } from "eve/channels/slack";2import { connectSlackCredentials } from "@vercel/connect/eve";3
4export default slackRoute({5  credentials: connectSlackCredentials("slack/mybot"),6});
```

一个 eve Slack 频道

Slack 集成通常保存在您环境中的两个密钥，`SLACK_BOT_TOKEN` 和 `SLACK_SIGNING_SECRET`，已从您的应用中消失。无需再配置、存储或轮换任何东西。

## Link to heading访问权限成为您按需请求、限定于任务范围的东西

Agent（智能体）能访问的范围越广，其作用就越大，这正是为什么访问权限是需要正确处理的关键。Agent（智能体）能触及的每个系统，都是有人可能通过泄露的 Token 触及到的系统。通过运行时凭证交换，无需为所有事情预配置任何东西。没有什么是所有用户共享的。没有什么是永久有效的。没有任何东西能超出其当前任务的范围。

凭证管理曾经是架构的一部分。它涉及轮换脚本、在环境之间复制的密钥，以及范围宽泛到您希望没人会泄露它们的机器人 Token。现在，您无需存储任何这些东西。您在 Agent（智能体）需要的那一刻请求访问权限，且权限限定于任务范围。

## Link to heading常见问题

什么是 Vercel Connect？

Vercel Connect 允许您的 Agent（智能体）和服务代表您的用户和团队访问外部系统。您无需将提供商凭证存储在长期有效的环境变量中，而是在运行时请求用户授权的 Token，并应用项目级别的访问控制。

Vercel Connect 解决了什么问题？

它消除了运行时中长期有效的第三方密钥，同时仍然允许 Agent（智能体）在外部 API 上执行操作。您为某个提供商注册一个连接器，将其链接到项目和环境，并在运行时请求提供商 Token。

我应该在什么时候使用 Vercel Connect 而不是 Integrations？

对于 Vercel Marketplace 中由市场管理的安装和提供商管理的产品，请使用 `Vercel Integrations`。当您需要为 Agent（智能体）工作流（例如，需要项目级作用域访问 Slack 工作区的 Agent（智能体））提供委托的运行时凭证和用户授权时，请使用 Vercel Connect。

有哪些可用的连接器？

Vercel Connect 支持通用的 OAuth 和 API 密钥连接器，以及针对 Slack、GitHub、Linear、Discord、Notion、Salesforce、Figma 和 Snowflake 的专用连接器。Resend、Workday、Microsoft Teams 等更多连接器即将推出。

定价是如何运作的？

定价基于 Token 请求次数。Hobby 计划每月包含 5K 次 Token 请求，无需额外付费。在 Pro 和 Enterprise 计划中，Token 请求按每 10K 次 3 美元计费。

当前的 Beta 版限制有哪些？

触发器转发仅限于 Slack、GitHub 和 Linear；连接器品牌字段在设置后无法完全清除；Token 撤销、Token 生命周期和作用域粒度取决于提供商的支持情况。
