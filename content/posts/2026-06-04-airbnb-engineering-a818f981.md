# Sitar-agent：大规模构建可靠的动态配置边车容器

## Airbnb 如何构建一个 Kubernetes 边车容器，以在大规模场景下可靠地交付动态配置

收听

作者：Bo Teng、Cosmo Qiu、Siyuan Zhou、Ankur Soni、Xin Huang、Willis Harvey

## 引言

在之前的文章中，我们探讨了 Airbnb 的动态配置系统 Sitar，重点介绍了服务架构和配置变更安全性。现在面临一个更棘手的问题：一旦配置变更被提交（这种情况每分钟发生数次），它如何在不重新部署服务的情况下，可靠、快速地实际到达数千个 Airbnb 服务实例？

本文介绍 sitar agent：一个轻量级的 Kubernetes 边车容器，它与每个订阅的服务 Pod 一同运行，持续从服务后端同步最新配置，并将其提供到本地文件系统以供读取。在本文中，我们将首先介绍配置交付的生命周期，然后讨论 sitar-agent 边车容器的一些关键设计选择。

## 配置交付生命周期

下图展示了配置变更从面向开发者的层到生产服务集群的端到端旅程。

步骤 1 — 配置创建/更新

开发者通过 Git 流程或 Web UI 创建或更新配置值。这些变更被提交到 Sitar Service，并以完整的版本控制、变更日志和 ACL 强制执行进行存储。

步骤 2 — 每小时快照上传

Snapshot Service 定期打包所有配置组的完整状态，并将压缩后的快照上传到 AWS S3。

步骤 3.1 — 从 S3 预加载快照（Pod 启动时）

当生产服务 Pod 启动时，sitar-agent 边车容器首先运行。它从 S3 下载每个订阅租户配置的最新快照到挂载磁盘（sitar-agent 与主容器共享）。这使得代理能够从已知的良好状态启动，而无需在每次重启时从 Sitar Service 重新获取所有配置。从 S3 预加载快照可以加快重启速度，使服务能够应对 Sitar Service 的临时不可用，并避免部署期间的负载峰值。

步骤 3.2 — 从 Sitar Service 预加载最新配置（Pod 启动时）

加载 S3 快照后，代理与 Sitar Service 执行初始同步，以获取自上次快照以来发布的所有变更。一旦此步骤成功，代理即发出就绪信号，允许应用程序主容器开始启动。

步骤 4 — 定期更新

启动后，代理进入一个持续轮询循环（秒级，带抖动）。在每个周期中，sitar agent 向 Sitar Service 查询所有订阅组的变更。

步骤 5 — 读取配置

应用程序主容器通过 Sitar 客户端库从挂载磁盘读取配置，该库维护一个内存缓存。客户端透明地检测文件变更并刷新其缓存。

在了解了交付生命周期后，以下部分将介绍塑造边车容器设计的主要架构选择。

## 关键设计决策

2024 年，sitar-agent 从 Ruby 完全重写为 Airbnb 主流的 JVM 语言 Java，这使团队有机会在语言迁移的同时实现架构现代化。上一节介绍的基于快照的 S3 预加载就是这项工作的成果之一：它显著减少了 Pod 的冷启动时间，并将启动可靠性与 Sitar Service 的可用性解耦。这次重写还带来了其他几个关于可靠性、性能和操作安全性的深思熟虑的设计决策。以下部分将逐一介绍这些选择。

## Sitar 系统的需求

在深入探讨具体设计选择之前，了解影响每个决策的约束条件会有所帮助。在 Airbnb，动态配置交付不仅仅是一种便利：它控制着数千个服务的关键功能。这意味着配置必须始终可用，即使 Sitar Service 本身宕机；稍微过时的值是可以容忍的，但不可读的配置则不行。同时，当工程师推送变更时，它需要在几十秒内（而非几分钟）到达每个订阅的服务。在大规模场景下实现这一点并非易事：数万个 Pod 同时获取更新，系统必须能够吸收这些负载而不降级。此外，由于 Airbnb 的服务集群涵盖 Java、Python、Go、Typescript 和 Ruby，解决方案需要服务于所有这些语言，理想情况下尽量减少维护单独语言实现的工作量。

上述关于可靠性、性能、可扩展性和多语言支持的需求并非相互独立。正如您将看到的，我们下面描述的大多数设计决策都归结为在这些需求之间进行权衡。

## 主容器 vs 边车容器

在 Java 重写过程中，sitar-agent 应该作为边车容器运行还是作为主容器中的一个进程运行，成为了一个关键的架构决策。我们评估了每个选项的优缺点如下：

迁移到主容器的优点：

- 降低成本。这是迁移到主容器的主要驱动力：将代理作为库运行消除了每个 Pod 的 JVM 开销，允许与主容器共享内存和 CPU。
- 减少运维面。少一个容器意味着服务所有者需要配置和调优的组件少一个。然而，考虑到 Airbnb 的多语言服务集群，这一优势有所减弱。

迁移到主容器的缺点：

- 多语言复杂性。Airbnb 服务语言涵盖 Java、Python、Go、Typescript 和 Ruby。库方法需要将现有的边车逻辑在所有语言中实现，这显著增加了开发和维护工作量。
- 无隔离性。sitar 逻辑中的错误或资源峰值可能导致主容器崩溃或资源匮乏，反之亦然。这种耦合增加了事件的爆炸半径，并使调试期间的资源归属复杂化。
- 运维噪音。将 Sitar 的日志及其 CPU/内存使用情况与主进程的日志及其指标混合在一起，使得调试 sitar 和主进程问题都更加困难。
- 可优化性。拥有一个单独的容器可以针对其用途进行优化，并简化测试和调试。

决策：

尽管将 sitar-agent 逻辑迁移到主容器可以节省成本并减少运维面，但预计的节省不足以证明在可靠性和运维开销方面的权衡，以及在多种语言中支持边车逻辑的开发开销是合理的。因此，我们决定将 sitar-agent 保持为一个独立的边车容器。

## 拉取模型与服务器端优化

Sitar-agent 通过每 10 秒轮询 Sitar service 来获取配置更新。这是一种拉取模型：代理通过定期向服务器请求变更来驱动更新周期。这种基于拉取的架构虽然简单且易于维护，但在不需要更新时会对服务器产生不必要的负载。

基于推送的架构变更可以大大减少服务器端负载和变更传播时间，但代价是架构更加复杂。为了在保持当前简单架构的同时减少服务端负载，sitar 系统实现了以下优化：

1. 由于 Sitar 配置主要通过手动更改，耗时超过数秒，因此配置更新交付中的轻微延迟是可以接受的。因此，采用短 TTL（10秒）的服务端缓存是减少 Sitar 服务端处理的好方法。大多数 Sitar-Agent 对服务的调用会命中缓存层，而不会触发繁重的服务端计算或数据库访问，从而大幅降低处理请求的资源消耗。
2. 当发生缓存未命中且请求实际触发了数据库访问时，会传递一个 Token（最后扫描的数据库行），这告知服务跳过对上次获取之前变更的扫描，从而大幅减少每次定期拉取过程中的服务端处理和数据库访问时间。

基于上述优化，Sitar 服务在应对 Airbnb 所有服务 Pod 的拉取请求时，能够良好地扩展和运行，并且我们可以保留简单、无状态的服务端加拉取架构。

对于 Sitar 的用例，秒级的轮询延迟是可以接受的；动态配置并非实时信令机制，且大多数配置变更是手动的，因此几秒的传播延迟无关紧要。拉取模型的无状态简单性在 Airbnb 的规模下是一个强大的运营优势。团队选择保留拉取模型，并转而投资于降低每次轮询的成本。

## 本地数据存储选择

Sitar-Agent 维护一个本地磁盘键值存储，主容器从中读取数据。旧的数据存储是一个基于 Sparkey 的内部实现，并在 Sparkey 数据存储之上封装了一层用于并发协调的薄层。随着 Sitar 使用量的持续增长和演变，基于 Sparkey 的数据存储与 Sitar 需求之间的不匹配变得明显：

- Sparkey 专为一次写入、多次读取的工作负载而设计，不支持多线程读写协调。这需要在 Sparkey 数据存储之外封装一层用于并发协调，以支持 Sitar 对数据存储的频繁写入，增加了复杂性，并可能成为潜在 Bug 的来源。
- Sparkey 在设计上不包含原生并发支持，我们需要一个外部锁机制，在写入时锁定整个数据存储文件。随着数据存储更新频率的增加，这种锁竞争开始限制并发读/写性能。
- 由于 Sparkey 的设计要求在每次写入时对整个数据存储重新建立索引，频繁写入基于 Sparkey 的数据存储成本越来越高。然而，随着 Sitar 在 Airbnb 几乎所有服务中广泛使用，对数据存储的写入非常频繁；我们几乎在每个轮询周期（约每 10 秒）都会看到配置更新。
- Sparkey 的多语言支持有限：它没有提供 Airbnb 服务所需的所有语言的实现，而在 Airbnb 支持所有语言需要复杂的互操作。

团队评估并基准测试了两个候选方案来替换旧的基于 Sparkey 的数据存储：SQLite 和 RocksDB。我们运行了一系列实验矩阵，涵盖不同的数据集大小、读取 QPS 和内存分配，每次运行固定三个维度中的两个，改变第三个维度。我们还研究了两个方案的社区支持、开源活动、支持的语言以及采用广度。以下是我们发现的总结：

SQLite：

优点：
- 成熟、广泛采用的库，为 Java、TypeScript/Node.js、Python、Go 和 Ruby 提供官方维护的绑定：这些都是 Sitar 服务消费者使用的语言。
- 内置预写日志（WAL）模式支持写入期间的并发读取，消除了对自定义并发封装层的需求。
- 简单的操作模型：单个文件，无需后台压缩或调优。
- 读写性能显著优于基于 Sparkey 的数据存储，足以满足 Sitar 的工作负载。

缺点：
- 读取延迟比 RocksDB 慢 2-3 倍，并随数据大小线性增加。
- 写入延迟也随数据大小增加而增加。

RocksDB：
- 在所有测试维度中具有最佳的原始读/写性能。
- 一致的高 QPS 读取性能；测试高达 1500 ops/sec，性能下降极小。
- 操作模型更复杂；需要调优压缩、块缓存、列族和内存设置。
- 多语言库生态系统不如 SQLite 成熟且维护一致性较差。
- 对于没有深厚 RocksDB 专业知识的团队来说，运营负担更高。

在我们的测试中，RocksDB 和 SQLite 在所有三个测试维度（数据大小、内存分配和读取 QPS）上均显著优于基于 Sparkey 的数据存储。虽然 RocksDB 提供更好的原始性能，但 Sitar-Agent 的工作负载在 SQLite 的能力范围内运行良好。SQLite 一流的多语言库支持、基于 WAL 的原生并发访问模型以及更简单的操作足迹，使其成为支持多种语言运行时的团队的更佳整体选择。团队选择 SQLite 作为基于 Sparkey 的数据存储的替代方案。

从 Sparkey 到 SQLite 的安全迁移

操作安全是首要任务。除了广泛的测试外，我们还依赖两种机制来确保安全推出：

1. 影子读取：在迁移每个服务之前，我们运行了一个影子读取和比较阶段；服务继续从 Sparkey 读取，同时并行获取 SQLite 结果进行验证。
2. 功能标志控制的逐步推出：我们增量迁移，从最不关键的服务开始，逐步推进到最关键的服务。一些关键的 Tier 0 服务最后接入，每一步都有专门的协调。

## 结论

Sitar-Agent 位于 Airbnb 动态配置交付系统的核心。本文介绍了其工作原理以及我们在 Java 重写过程中权衡的关键点：成本与隔离、简单性与基于推送的效率、原始性能与操作实用性。每个决策都回归到相同的约束：配置必须始终可用，变更必须在数万个 Pod 的集群中快速传播，并且解决方案必须能在 Airbnb 的多语言服务栈中工作，而不会增加维护负担。
