## Agent（智能体）技能

Agent（智能体）缺乏您和您的团队所拥有的相同上下文。即使当Agent（智能体）扫描您的代码库或连接到您的文档管理系统时，它也不知道您团队的流程、质量标准或目标。为了改进响应并纠正错误，人们会使用重复的提示词来提供正确的上下文。

Agent（智能体）技能解决了这个问题。它们是一种简单、开放的格式，打包了Agent（智能体）能够自动发现和使用的指令、脚本和资源，从而提高了输出准确性。

可以将技能视为集中化的、按需提供的专业知识。编写一次，Agent（智能体）就能在需要时立即获取关键信息。技能为您提供了一条从“Agent（智能体）勉强能用”到“Agent（智能体）真正了解我们这里的做事方式”的路径。

## 什么是技能？

技能是为AI Agent（智能体）打包的、可重复使用的指令。技能建立在许多顶级供应商和Agent（智能体）平台采用的开放标准之上。

您只需安装一次技能包，您的Agent（智能体）就可以在任务匹配时加载它们。技能适用于任何类型的Agent（智能体），无论是编写代码、分析数据、管理工作流还是处理客户支持。

虽然编码Agent（智能体）推动了当前的大部分采用，但技能适用于任何Agent（智能体）用例。相同的打包格式和生态系统适用于各个领域。

## 技能与MCP服务器、工具、规则和系统提示词相比如何？

技能、MCP服务器、工具、规则和系统提示词各自解决了AI Agent（智能体）配置的不同方面。

技能擅长打包包含上下文和防护措施的完整工作流。MCP服务器提供标准化的工具访问。工具提供特定的功能。规则和系统提示词提供基础的行为控制。

这些方法可以协同工作。技能可以引用MCP服务器，建立在系统提示词之上，并整合基于规则的逻辑。

### 技能

技能打包了结合指令、上下文和决策逻辑的完整工作流。它们不仅告诉Agent（智能体）有哪些工具可用，还告诉它们何时使用这些工具、如何安排操作顺序以及成功的标准是什么。

它们最适合处理复杂的、多步骤的、包含重复性、领域特定任务的过程。这些场景需要对工具使用和输出处理进行上下文决策，而技能提供了做出最佳选择所需的特定任务专业知识。

### MCP服务器

模型上下文协议（MCP）服务器为AI Agent（智能体）访问外部工具和服务提供了标准化接口。它们处理AI系统与第三方API之间的技术集成。

它们最适合工具集成、API访问，以及为Agent（智能体）提供访问数据库、文件系统或Web服务等外部服务的可靠接口。

### 工具

工具提供Agent（智能体）可以调用来执行特定操作的独立功能。每个工具处理一个离散的操作，例如进行API调用、执行网络搜索、读取文件或处理数据。

它们最适合单一用途的操作、API集成，以及为Agent（智能体）提供可用作更大工作流构建模块的特定能力。

### 规则

规则定义了AI行为的具体约束和逻辑。团队和个人都使用规则来一致地应用安全策略、数据处理标准和操作约束，无论任务或Agent（智能体）角色如何。

它们最适合强制执行合规要求、设置访问控制，以及定义需要一致应用的特定行为边界。

### 系统提示词

系统提示词确立了AI的基础行为和个性。它们为Agent（智能体）应如何响应、使用何种语气以及强调哪些基本能力设定了初始上下文。

它们最适合定义核心行为、设定沟通风格，以及建立适用于所有交互的基线操作参数（而规则则作为更窄的约束，在该基线内限制或要求特定的行动）。

## 技能解决了哪些问题？

技能解决了团队依赖Agent（智能体）处理复杂工作流时出现的问题。

**提示词漂移**。两个人可能要求同一件事却得到不同的结果，因为他们使用了不同的措辞，或者因为Agent（智能体）选择关注不同的上下文。没有一致的指令，即使执行相同的任务，Agent（智能体）的行为也会变化。

**工作流惯例丢失**。每个流程都有其自己的质量检查、审批流程、数据格式和决策标准模式。没有指导，Agent（智能体）无法正确推断这些，并且不可能为每个大规模任务重新陈述它们。

**指令蔓延导致上下文膨胀**。将详细的执行手册复制到提示词中，会与Agent（智能体）需要推理的其他所有内容竞争，关键的工作流上下文会被埋没。

技能将这些指令从临时提示词中提取出来，并将其置于一个集中的位置和格式中，您可以在您的Agent（智能体）工作流中对其进行版本控制、审查和重用。

## 什么是技能包？

技能包是一个可共享的、包含一个或多个技能的集合，团队可以在需要的地方采用和安装。

每个包包含使技能工作的组件。唯一必需的组件是SKILL.md文件。其他所有内容都是可选的。

SKILL.md文件包含告诉Agent（智能体）该技能做什么以及如何使用的指令。scripts目录存放可执行的辅助程序和自动化脚本。references目录包含支持文档、示例和上下文文件。配置文件定义设置和依赖项要求。

您无需为每个代码库准备一套庞大的指令集。您可以在合适的地方安装技能包，同时保持智能体（Agent）的其余配置不变。这种模块化方法让团队能够采用特定功能，而无需彻底改造整个工作流程。

一个技能包可能包含单一专注技能，也可能包含多个协同工作的相关技能，您还可以根据需要添加支持文件，使其在您的开发环境中发挥作用。

## 如何安装技能包？

要安装技能包，您可以将技能文件放置在项目的 `skills/` 目录下，或全局用户作用域内。您也可以使用 `skills` 命令行工具。

**通过命令行安装技能**

使用 `skills` CLI 直接从命令行添加技能：

```
npx skills add <owner/repo>
```

## 如何找到技能包？

技能拥有自己的生态系统，具有一致的格式，并可通过公共目录 [skills.sh](skills.sh) 进行发现。

## 安装后会发生什么？

安装后，技能会出现在您的智能体（Agent）的可用技能列表中。智能体（Agent）会根据技能元数据和当前上下文，在需要时加载它们。

## 我可以创建自己的技能吗？

是的，您可以创建自定义技能。技能是包含一个 `SKILL.md` 文件（以及根据需要添加的其他文件）的文件夹，这些文件定义了技能的目的和实现。

您可以在本地创建技能，无需外部托管。创建一个文件夹，添加一个包含必需 YAML 前置元数据和技能定义的 `SKILL.md` 文件，智能体（Agent）即可立即使用它。

要了解结构和实现模式，可以**探索现有技能作为示例**。这有助于您构建能与智能体（Agent）系统有效集成的技能。

## 智能体（Agent）如何使用技能？

大多数支持技能的智能体（Agent）都遵循相同的模式。

启动时，智能体（Agent）会加载一个轻量级的可用技能索引。它看到的是名称和描述，而不是完整的指令正文。

当任务看起来匹配时，智能体（Agent）会加载相关技能的全部内容。这使得默认上下文保持较小，同时在需要时仍能提供详细的指导。

一些智能体（Agent）平台也支持显式调用。当您想强制某个工作流程，或者在调试某个技能为何适用或不适用时，这很有用。

## SKILL.md 里有什么？

`SKILL.md` 包含两部分：YAML 前置元数据和 Markdown 内容。

### YAML 前置元数据

前置元数据用于智能体（Agent）发现和元数据。它必须包含 `name` 和 `description` 字段。

`name` 字段要求包括 1-64 个字符，小写字母和数字，可包含单个连字符，必须与目录名匹配，并且必须符合 `^[a-z0-9]+(-[a-z0-9]+)*$` 正则表达式。

`description` 字段要求为 1-1024 个字符。

未知的前置元数据字段会被忽略，这使得格式具有向前兼容性。

### Markdown 内容

Markdown 内容部分包含给智能体（Agent）的实际指令。在这里，您定义智能体（Agent）应该做什么、应该如何行为，以及在使用此技能时应遵循的任何具体准则。

## 哪些 SKILL.md 前置元数据字段是可选的？

可选的前置元数据字段包括 `license`、`compatibility`（最多 500 个字符）、`metadata`（任意的字符串到字符串键值映射）和 `allowed-tools`。

这些字段让团队能够传达约束和环境要求，而无需将它们硬编码到叙述中。

## 技能还可以包含什么？

一个技能可以不仅仅是单个指令文件。

除了 `SKILL.md`，技能目录还可以包含一个 `scripts/` 目录，用于存放可执行的辅助脚本。当您希望某个步骤每次都以相同方式运行时，这很有用。`references/` 目录存放较长的支持文档，智能体（Agent）仅在需要时加载它们。`assets/` 目录包含支持技能输出的文件，如模板、示例或其他工件。

这些都是可选的，但它们增加了高级功能，简化了 `SKILL.md`，并将冗长的参考资料排除在默认上下文之外。

## scripts 目录是做什么用的？

`scripts/` 可以包含可执行的辅助脚本。

这可以使技能更节省 Token 且更具确定性，因为智能体（Agent）可以运行脚本，而不是用自然语言重新推导一个多步骤过程。

脚本也使审查更容易。如果有一个步骤必须每次都正确无误，脚本通常比一段文字段落更容易审计。

## references 目录是做什么用的？

`references/` 用于存放设计为按需加载的支持文档。

这符合渐进式披露模型。智能体（Agent）可以先看到摘要，只有在需要细节时才加载完整内容。

一个实用的最佳实践是避免在 `SKILL.md` 和 `references/` 中重复相同的信息。

如果参考文件很大，请在 `SKILL.md` 中包含 grep 搜索模式，以便智能体（Agent）能快速定位到正确的部分。

## 关于技能的其他常见问题

**技能安全吗？**

一个有用的心智模型是：技能改变的是智能体（Agent）的行为方式。它们不会神奇地让智能体（Agent）变得可信。如果一个技能包包含脚本，请像对待您运行的其他任何代码一样对待它们。审查它的功能，在可能的情况下固定版本，并优先选择设计为可审计的包。

**技能的替代方案是什么？**

您也可以使用提示词（Prompt）库、仓库级别的指令文件（如 `AGENTS.md`）或自定义智能体（Agent）包装器。行业正趋向于将技能作为一个共享标准，因此对于大多数团队来说，它们是推荐的方法。

## 为什么技能很重要

团队将花费更多时间来监督智能体（Agent）。

如果您的上下文和规范只存在于某人的聊天记录中，您就无法审查它们、系统地更新它们，或在出现问题时进行调试。

技能使智能体（Agent）行为更容易标准化。

它们也使得随着时间的推移进行改进变得更加容易。您可以通过更新技能包来交付更好的工作流程，而不是作为一个需要某人记住的新提示词（Prompt）。
