# MachinaCheck：在AMD MI300X上构建多Agent CNC可制造性系统

在lablab.ai举办的AMD开发者黑客松上构建 — 2026年5月

## 我们解决的问题

走进任何一家小型CNC机加工车间，问经理他们如何决定是否接受客户订单。

答案几乎总是相同的：他们打印图纸，手动读取每个尺寸，在车间里走动检查哪些工具可用，估算他们的机器能否达到所需公差，并在记事本上做笔记。整个过程每张图纸需要30到60分钟。对于一个每周收到10到20份询价的繁忙车间来说，仅可行性分析一项就需要花费熟练经理5到20小时的时间。

有时他们会判断失误。他们接受了一个订单，开始生产，中途发现没有合适的丝锥，或者他们的铣床无法达到关键特征的公差。零件报废。客户不满。机器时间被浪费。

我们构建了MachinaCheck来彻底消除这个问题。

## MachinaCheck的功能

MachinaCheck是一个多Agent AI系统。您上传一个STEP文件——客户发送给机加工车间的标准CAD格式——以及三个简单输入：材料类型、所需公差和任何螺纹规格。30秒后，您将获得一份完整的可制造性报告，准确告诉您是否能制造该零件、需要什么工具、缺少什么以及在生产开始前需要采取什么行动。

无需手动读取图纸。无需在车间走动。无需猜测。

## 为什么我们选择在AMD MI300X上构建

在解释架构之前，这一点值得单独说明，因为这不仅是技术选择——更是业务需求。

制造客户签署保密协议。他们的STEP文件包含专有几何形状，代表着多年的工程工作和数百万美元的研发投入。医疗设备上的孔阵列或航空航天部件上的凹槽几何形状都是机密知识产权。

将这些数据发送给OpenAI、Anthropic或任何商业API端点都是违反保密规定的。毋庸置疑。

AMD Instinct MI300X完全改变了这一局面。凭借192GB的HBM3 VRAM和5.3 TB/s的内存带宽，我们完全在本地运行Qwen 2.5 7B Instruct。没有数据离开车间的基础设施。没有STEP几何形状传输到第三方服务器。客户的IP始终留在它应该在的地方。

这就是"设计即隐私"在制造环境中的真正含义——不是一个复选框，而是一个根本性的架构决策，使产品对真正的企业客户具有可行性。

## Agent架构

MachinaCheck使用基于LangChain构建并通过FastAPI编排的五组件流水线。

### 组件1 — STEP文件解析器（纯Python，无需LLM）

我们使用cadquery（一个基于OpenCASCADE构建的Python库）直接解析STEP文件。这使我们能够获得数学上精确的特征提取：

- 所有圆柱孔及其直径和深度
- 平面及其面积
- 倒角和圆角
- 边界框尺寸
- 总体积和表面积

这种提取是100%准确的，因为它直接读取数学几何形状——无需视觉模型、无需OCR、无需近似。一个Ø6.0mm的孔在输出中就是精确的Ø6.0mm。

```python
def extract_features(step_file_path: str) -> dict:
    model = cq.importers.importStep(step_file_path)
    shape = model.val()
    bb = shape.BoundingBox()
    
    holes = {}
    for face in model.faces().vals():
        adaptor = BRepAdaptor_Surface(face.wrapped)
        if adaptor.GetType() == GeomAbs_Cylinder:
            radius = adaptor.Cylinder().Radius()
            diameter = round(radius * 2, 3)
            holes[diameter] = holes.get(diameter, 0) + 1
    
    return {
        "bounding_box_mm": {"length": round(bb.xlen, 3), ...},
        "holes": [...],
        "flat_surfaces_count": len(flat_surfaces),
    }

```

### Agent 1 — 操作分类器（Qwen 2.5 7B）

提取的几何形状加上用户输入——材料、公差、螺纹——通过vLLM传递给在AMD MI300X上运行的Qwen 2.5 7B。

该Agent回答："制造该零件需要哪些CNC操作和工具？"

它应用制造领域知识：304不锈钢需要硬质合金刀具。圆柱孔需要钻头，而不是立铣刀。±0.005mm的公差需要精密机床，而不是标准铣床。

### Agent 2 — 工具匹配器（纯Python）

该Agent不使用LLM。它查询车间的工具库存数据库，并将每个所需工具与可用工具进行核对。纯确定性逻辑——数据库查找、比较、结果。数据库查询不需要LLM，在这里使用LLM会增加不必要的延迟和幻觉风险。

### Agent 3 — 可行性决策Agent（Qwen 2.5 7B）

匹配结果返回给Qwen。该Agent对整体情况进行推理，并生成结构化决策：

```json
{
  "decision": "CONDITIONAL",
  "confidence": "HIGH",
  "reason": "除M10x1.5丝锥外所有工具均可用",
  "action_items": ["购买M10x1.5丝锥（$15）"],
  "risk_flags": ["验证304不锈钢的主轴转速"],
  "estimated_setup_hours": 2.5
}

```

### Agent 4 — 报告生成器（Qwen 2.5 7B）

最后一个Agent将所有内容综合成一份专业的可制造性报告，包含总体状态、执行摘要、零件分析、工具状态、机器状态和最终建议。

## AMD技术栈

通过ROCm和vLLM在AMD MI300X上运行Qwen 2.5 7B非常简单。AMD开发者云上的vLLM快速启动镜像已预配置好一切。

```bash
python -m vllm.entrypoints.openai.api_server \
  --model Qwen/Qwen2.5-7B-Instruct \
  --host 0.0.0.0 \
  --port 8000 \
  --dtype float16 \
  --gpu-memory-utilization 0.5

```

使用gpu-memory-utilization 0.5，我们使用了可用192GB中的约96GB，留有充足余量。我们的Agent调用推理延迟平均低于3秒。

LangChain通过兼容OpenAI的端点连接到vLLM：

```python
from langchain_community.llms import VLLMOpenAI

llm = VLLMOpenAI(
    openai_api_base="http://localhost:8000/v1",
    openai_api_key="EMPTY",
    model_name="Qwen/Qwen2.5-7B-Instruct",
    temperature=0.1,
    max_tokens=1000
)

```

## 我们学到的经验

仅在需要推理的地方使用LLM。Agent 2（工具匹配）是纯Python。在那里使用LLM会更慢、更昂贵且更不可靠。数据库查找的正确工具是数据库查询。

结构化输出的提示词工程很重要。让Qwen可靠地输出有效JSON需要在提示词中设置仔细的规则——明确说明圆柱孔需要钻头而不是立铣刀，直径必须精确匹配，丝锥仅在指定螺纹时出现。

AMD MI300X对此用例确实令人印象深刻。192GB的VRAM意味着如果需要，我们可以运行更大的模型。对于生产部署，Qwen 2.5 72B可以轻松适配，并提供显著更好的推理质量。
