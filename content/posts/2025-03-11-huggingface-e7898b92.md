# Open R1：更新 #3

![image/png](/images/posts/d9374fcfe102.png)

在过去几周里，我们集中精力复现 DeepSeek-R1 方案中竞技编程（代码推理）方面的内容。

在这篇文章中，我们很高兴分享：

- CodeForces-CoTs 数据集的构建：一个包含近 10 万条高质量样本的数据集，这些样本从 R1 中蒸馏得到，用于生成 C++ 和 Python 的解决方案。
- IOI 基准测试：一个来自 2024 年国际信息学奥林匹克竞赛（IOI）的挑战性难题的新基准。
- OlympicCoder：两个经过微调的 7B 和 32B 代码模型，在 IOI 问题上表现优于 Claude 3.7 Sonnet 等闭源前沿模型。

以下是 OlympicCoder 模型与各种指令微调和推理模型的对比概览。我们发现，在 CodeForces-CoTs 上训练模型能够产生顶级性能，其中 OlympicCoder-32B 的表现优于我们测试的所有开源权重模型，包括一些规模大 100 倍以上的模型 🤯。

## CodeForces-CoTs 数据集

CodeForces 是竞技程序员中最受欢迎的网站之一，定期举办比赛，参赛者必须解决具有挑战性的算法优化问题。这些问题的挑战性使其成为改进和测试模型代码推理能力的有趣数据集。

虽然之前的工作（如 DeepMind 的 CodeContests 数据集）已经收集了大量 CodeForces 问题，但今天我们发布了自己的 open-r1/codeforces 数据集，包含超过 1 万个问题，覆盖从最早的比赛一直到 2025 年，其中约 3000 个问题未包含在 DeepMind 的数据集中。此外，对于约 60% 的问题，我们包含了题解（editorial），即由比赛组织者编写的解释正确解决方案的说明。您还可以找到从官方网站提取的每个问题的 3 个正确解决方案。

此外，我们还发布了 open-r1/codeforces-cots，其中包含 DeepSeek-R1 在这些问题上生成的思维链，我们要求模型用 C++（竞技编程中使用的主要语言）和 Python 生成解决方案，总计接近 10 万个样本。

我们在这个数据集上对 Qwen2.5 Coder Instruct 7B 和 32B 进行了微调，得到了我们的 OlympicCoder-7B 和 OlympicCoder-32B 模型。您将在博客文章的后面部分找到关于这些模型的更多细节。

## 代码可验证性危机

虽然像 DeepMind 的竞赛数据集和其他包含竞技编程问题的数据集都包含测试用例并声称是可验证的，但这些测试用例通常只是竞赛网站上使用的完整测试套件的一小部分。特别是 CodeForces，将显示的测试用例限制在约 500 个字符以内，这意味着这些数据集只包含符合此限制的较短、较容易的测试用例。

举个例子，我们选取了 7 个 R1 生成的解决方案通过了所有公开测试用例的问题，并尝试将它们提交到 CodeForces 平台：

![image/png](/images/posts/57ed1cd0225f.png)

尽管它们通过了较短的测试，但这些解决方案每一个都在完整的测试集上失败了。这凸显了对新的完全可验证的竞技编程数据集的需求。虽然我们计划尝试基于模型的解决方案来生成和验证额外的具有挑战性的测试，以便将来可能添加到我们的 CodeForces 数据集中，但目前我们转向其他地方寻找完全可用的问题数据。

国际信息学奥林匹克竞赛（IOI）是五大国际科学奥林匹克竞赛之一（如果您熟悉 AIME，IOI 相当于编程界的 IMO，参加 AIME 的最优秀学生会被邀请参加），它测试经过严格选拔的高中生（每个国家 4 名）解决复杂算法问题的能力。

这些问题极具挑战性，完整的测试集可用，并在宽松的（CC-BY）许可下发布。这意味着 IOI 是测试模型代码推理能力的完美数据集。

在 IOI 中，每个问题都有几个子任务，每个子任务有不同的输入约束。要解决一个子任务，提交的解决方案需要在（严格的）时间限制内通过其所有测试用例。虽然最后一个子任务通常是"完整问题"，但有些子任务实际上描述了一个更简单（约束更多）的问题，参赛者通常针对特定子任务以获得部分分数，而不是仅仅尝试解决完整问题（满分相对罕见）。

根据最近一篇 OpenAI 论文（其中 o1 在 IOI'2024（上一届）上实时参赛），我们类似地处理了 IOI'2024（以及直到 2020 年的之前 IOI）的所有问题，并将它们拆分为子任务，这样每个提示词要求模型解决一个特定的子任务。我们在 open-r1/ioi 和 open-r1/ioi-test-cases 中发布了处理过的问题陈述，以及运行它们所需的所有评分/检查器文件和测试用例。

我们创建了自定义代码来运行解决方案（许多问题有复杂的设置，需要一个"管理器"进程与多个运行用户提交的进程通信，以及特殊的检查器来验证解决方案）并根据 IOI 规则进行评分，该代码可在 https://github.com/huggingface/ioi 获取，并在 IOI'2024 上评估了 40 多个领先的推理模型。

在 IOI 中，参赛者每个问题有 50 次提交限制。我们为每个子任务生成了 50 个提交，然后应用了类似于 OpenAI 使用的选择策略，以获得每个问题在竞赛条件下的分数。结果如下所示，其中水平线代表来自真实竞赛数据的铜牌、银牌和金牌门槛。虽然 o1 非常接近铜牌，但最终没有模型能达到奖牌门槛（参赛者的第 50 百分位）。

![image/png](/images/posts/211d2ab0b49e.png)

我们的 OlympicCoder 模型（红色）与其他前沿模型相比表现相当不错，甚至超过了一些闭源模型（金色），如 claude-3.7-sonnet-thinking，并且在 50 次提交限制设置下，OlympicCoder-32B 甚至优于 o1-mini 和我们从中蒸馏的 DeepSeek-R1 模型。

### 提交策略

一个重要说明是，我们的提交策略可能会对非推理模型（如 OlympicCoder-32B 的基础模型 Qwen-2.5-Coder-32B-Instruct）不利。为了模拟真实的竞赛条件（其中提交的分数只有在实际提交后才知道），我们采用了类似于 OpenAI 为 o1-ioi 采用的轮询提交策略：我们首先提交一个针对问题最后一个子任务的解决方案，然后是一个针对倒数第二个子任务的解决方案，依此类推，只有在解决方案被选中提交时才进行评估。我们跳过针对已被先前选中的提交解决的子任务的提交，并且在每个目标子任务中，我们优先选择来自较长生成结果的提交——这对于推理模型来说是一个合理的标准，但对于其他模型则不太适用。

## 从 R1 轨迹训练代码模型的经验教训

在创建 OlympicCoder 模型的过程中，我们进行了大量 SFT 实验，以理解应用于 CodeForces 数据集的各种过滤器的作用。我们发现以下来自 `open-r1/codeforces-cots` 的子集在整体性能上表现最佳：

- `solutions`：R1 根据问题描述生成的解决方案。
- `solutions_w_editorials`：R1 根据问题描述以及解释正确解法的说明生成的解决方案。

请注意，我们只关注了 C++ 解决方案，但混合使用 Python 解决方案可能会进一步提升性能。

我们使用 `LiveCodeBench` 作为模型的测试平台，然后通过难度更高的 IOI 基准测试对表现最佳的检查点进行了评估。我们测试了多种超参数配置来训练模型，并最终确定了以下设置：

- 模型：Qwen2.5 Coder Instruct 7B 和 32B
- 训练轮数：10
- 有效批量大小：128
- 学习率：4e-5
- 调度器：余弦退火，衰减至峰值学习率的 10%
- 上下文大小：7B 模型为 32,768 个 Token，32B 模型为 22,528 个 Token

以下是我们从使用 R1 推理轨迹微调 Qwen2.5 Coder 模型中学到的一些经验。

### 经验一：样本打包会损害推理性能

样本打包是一种广泛使用的方法，用于高效处理可变长度序列并加速训练。如下图所示，其工作原理是将训练样本（彩色部分）拼接成大小相等的块，从而无需在批次间使用填充 Token（灰色部分）：

![image/png](/images/posts/e83d0547386d.png)

使用打包时，样本可能会跨越每个块的边界，但在实践中，如果大多数样本远小于块大小，这通常影响不大。

然而，对于我们从 R1 蒸馏得到的推理轨迹，我们担心打包可能会损害性能，因为许多轨迹很长，且被截断的答案比例可能很高。这意味着模型可能难以学习关注长上下文信息，尤其是当问题和答案被打包到不同的块中时。

如下方引人注目的图表所示，我们发现打包严重损害了模型性能：使用打包时，模型几乎无法解决 LiveCodeBench 中的问题；而不使用打包时，性能在多个训练轮次中持续提升，直至趋于平稳：

![image/png](/images/posts/0b562a649b5e.png)

我们推测，这种显著差异因以下事实而加剧：我们的训练集仅包含 C++ 解决方案，而 LiveCodeBench 仅评估 Python 性能。尽管如此，我们发现，在我们消融的所有数据集中，打包的效果始终更差。

### 经验二：使用较大的学习率以获得最佳性能

在我们使用 Qwen 模型进行的大多数 SFT 实验中，2e-5 的学习率通常足以获得强劲的性能。然而，当对现有的指令模型应用包含推理数据的 SFT 时，我们发现使用更大的学习率（4e-5）可以带来显著的性能提升。如下图所示，学习率每翻一倍，在 LiveCodeBench 上的性能提升几乎达到 10 个点！

![image/png](/images/posts/b2517810a564.png)

### 经验三：包含题解并不会提升性能

在创建 `open-r1/codeforces-cots` 的 `solutions_w_editorials` 子集时，我们的直觉是，通过向 R1 提供问题和解决方案，我们将获得更好的推理轨迹。令人惊讶的是，这种直觉在评估中并未得到证实，而仅从问题描述进行简单采样在训练过程中带来了适度但持续的性能提升：

![image/png](/images/posts/8a21fed581ba.png)

### 经验四：使用 `<think>` 进行预填充以持续启用长思维链

在对我们的一些模型进行“氛围测试”时，我们遇到了一个奇特的现象：如果使用领域内查询（例如竞赛编程）进行提示，模型会在 `<think>` 和 `</think>` Token 之间输出预期的长思维链，随后给出最终答案。然而，对于领域外查询，例如“瑞士的首都是什么”，模型会恢复到原始指令模型的行为！

以下是一个简单查询未能返回长思维链的示例：

用户：1+1 等于多少？

助手：2。这是最简单的数学加法问题之一。1 + 1 等于 2。这个问题经常被用作谜语或脑筋急转弯，特别是在求职面试或在线测试中，用于考察候选人快速解决问题的能力。虽然对大多数人来说这似乎显而易见，但对于雇主来说，这可能是评估批判性思维和解决问题能力的一个有用问题。此外，它是数学中的一个重要概念，并为更复杂的算术运算奠定了基础。所以下次有人问你 1+1 等于多少时，你可以自信地回答是 2！

```python

def add_numbers(a, b):
    return a + b


result = add_numbers(1, 1)
print("1 + 1 的和是：", result)

```

在这个解决方案中，我们定义了一个函数 `add_numbers`，它接受两个参数 `a` 和 `b`，并返回它们的和。然后我们用 1 和 1 调用这个函数，将结果存储在变量 `result` 中，然后打印出来。这演示了一种计算和显示两个整数之和的简单方法。这个例子简单而有效地说明了 Python 中的基本算术运算。

然而，如果我们用 `<think>` 预填充助手的响应，期望的行为就会出现：

助手：<think>
好的，我需要回答“1+1 等于多少？”这个问题。让我想想。哦，对了，1 加 1 等于 2。但是等等，也许还有更多含义。比如，如果有人用不同的方式提问呢？或者考虑其他进制？比如在十进制中是 2，但在二进制中是 10。嗯，但这个问题很直接。用户可能期望一个简单的答案。所以正确答案是 2。我应该直接说出来吗？
</think>
是的，1 + 1 等于 2。关于这个计算，您还有什么想了解的吗？

这表明，当将推理轨迹蒸馏到现有的后训练模型中时，应该在最终模型的聊天模板中默认强制进行预填充。巧合的是，DeepSeek 在其蒸馏模型中也采用了这种做法，上述行为可能就是原因所在。（有趣的是，可以推测在流行的聊天界面上点击“思考”按钮是否只是在预填充助手的响应 🙂）。

综合所有这些经验，我们得到了 OlympicCoder-7B，其性能与 DeepSeek 的蒸馏模型相当，并显著优于基础的 Qwen2.5-Coder 模型：

![image/png](/images/posts/eede1befc21a.png)

### 经验五：使用 8 位优化器来扩展具有长上下文的大模型

在训练 OlympicCoder-7B 的整个过程中，我们发现 DeepSpeed ZeRO-3 足以在单个 8 x H100 节点上以 32k 上下文训练每个模型。然而，当我们尝试将方案扩展到 32B 模型时，我们遇到了一系列内存问题。特别是，一旦上下文扩展到超过 20k Token，即使使用 16 个节点，我们的运行也会出现内存不足的情况 😢。这并不理想，因为 20% 的 CodeForces-CoTs 轨迹长度超过 20k Token，这意味着它们在训练过程中会被截断。

问题的根源在于 `transformers` 和 `trl` 尚不支持上下文并行，尽管可以关注此 issue 以了解进展。

与此同时，我们探索了多种内存节省技术，发现将 FSDP 与 `paged_adamw_8bit` 优化器结合使用，使我们能够将上下文扩展到 22,528 个 Token：虽然仍不理想，但只有 9% 的数据被截断。

## 更新

### GRPO 更新

近期在改进 TRL 中 GRPO 的实现方面取得了进展，带来了效率、可扩展性和资源利用率的进一步提升。以下是自上次更新以来最重要的变更摘要：

生成复用

GRPO 的主要瓶颈之一与任何在线方法相同：生成需要时间。提高 GRPO 样本效率的一个关键方法是在优化过程中多次复用生成的样本，而不是在单次使用后就丢弃它们。实际上，这一技术早在 PPO 中就已引入。

对于 GRPO，样本被复用的次数记为 μ。

![image/png](/images/posts/f1b399b5e16d.png)

现在可以多次复用生成的样本，从而显著加快处理速度。

```python
from trl import GRPOConfig

training_args = GRPOConfig(..., num_iterations=...)

```

![image/png](/images/posts/304d7ea5e53e.png)

然而，需要谨慎——如果 μ 过大，可能会对学习产生负面影响。根据我们的经验，一个良好的平衡范围是 2 到 4。

奖励加权

在训练模型时，并非所有奖励都同等重要。例如，我们可能希望模型优先考虑正确性而非格式，而不是将两者同等对待。

为了解决这个问题，现在可以为不同的奖励分配不同的权重，从而对优化过程进行更精细的控制。通过调整这些权重，我们可以引导模型更关注特定任务中最重要的方面。

```python
from trl import GRPOConfig, GRPOTrainer

def very_important_reward(completions, **kwargs):
    ...

def less_important_reward(completions, **kwargs):
    ...

training_args = GRPOConfig(
    ...,
    reward_weights=[0.9, 0.1],
)
trainer = GRPOTrainer(
    ...,
    reward_funcs=[very_important_reward, less_important_reward],
    args=training_args,
)

```

其他改进

GRPO 还进行了几项较小但影响深远的改进：

- PEFT + vLLM 集成——现在可以将 PEFT（参数高效微调）和 vLLM 结合使用，将高效微调与优化推理相结合，以获得更好的可扩展性。
- 梯度检查点——新增了此功能，通过重新计算某些激活值而非存储它们来减少训练期间的内存消耗，从而能够训练更大的模型。
- 优化的选择性 Log Softmax 计算——引入了一种计算 log softmax 的新方法，减少了训练期间的内存峰值。

后续步骤与进行中的工作

当前的重点集中在两个关键领域：

1. 提升生成速度——正在探索进一步的优化（如静态缓存），以使生成过程更快。
2. 将 GRPO 扩展到多节点设置——正在努力使 GRPO 能够跨多个节点扩展，从而能够训练更大的模型。

### Open R1 Math-Dataset 更新

我们进一步丰富了之前发布的 OpenR1-Math-Raw 数据集，添加了新的元数据，以便在过滤和验证过程中做出更明智的决策。具体来说，我们新增了以下列：

- reparsed_answers：我们观察到 answer 列中的许多条目要么 LaTeX 格式不正确，要么只包含部分答案。此外，由于某些问题是多项选择题，正确答案本身和对应的字母都应被视为有效回答。为了解决这个问题，我们使用 Llama-3.3-70B-Instruct 模型从 solution 列中重新提取了所有答案，以确保 reparsed_answers 包含正确答案，并且在多项选择题的情况下也包含对应的字母。我们相信这一新增内容将对社区非常有价值，既能提高列评分准确性，也能改进 GRPO 期间的验证过程。
- correctness：当依赖基于模型的验证时，运行答案验证可能消耗大量资源。因此，我们使用 Llama-3.3-70B-Instruct 作为评判模型，并结合 math_verify 对 answer 和 reparsed_answers 两列进行了所有解决方案的评估。

实验细节

为了帮助社区理解基于验证的过滤对用于 SFT 蒸馏的数学数据集的影响，我们进行了几项消融实验。从一个随机选择的 20 万样本池开始，我们根据以下过滤规则创建了六个不同的 SFT 数据集：

1. no_restrictions（20万）——未应用任何过滤。
2. llama_verification（12.4万）——被 LLAMA-70B 验证评为正确的样本。
3. math_verify_answer（8.87万）——被 math_verify 在 answer 列上验证为正确的样本。
4. math_verify_reparsed（10.1万）——被 math_verify 在 reparsed_answers 列上验证为正确的样本。
5. llama_verification_or_math_verify_reparsed（LorMV）（15.4万）——数据集 2 和 4 的并集。
6. llama_verification_and_math_verify_reparsed（LandMV）（7.12万）——数据集 2 和 4 的交集。

训练与评估

在数据受限的环境中进行过滤时，精确率和召回率都是重要的考量因素。因此，我们没有在相同的 Token 预算下运行每个实验，而是在所有数据上训练一个 epoch。对于模型，我们选择了 Qwen7B-Instruct，并使用 RoPE 扩展将其微调到 32k 上下文长度，并采用余弦调度。为了跟踪性能进展，我们使用 lighteval 在 AIME-24、AIME-25 和 MATH-500 上每 40 步评估一次模型。结果总结如下：

关键观察

- 验证显著影响早期阶段的性能。过滤在前 40 步中尤其重要。在 MATH-500 数据集上，更严格的验证方法在早期阶段带来了显著的性能提升（例如，no_restrictions 得分为 0.61，而 LandMV 为 0.72）。然而，随着训练的进行，这种性能差距逐渐缩小，拥有更多样本被证明是有益的——即使其中一些包含错误。
- 不同数据集之间的训练损失差异显著。使用 math_verify 过滤的数据集表现出持续更低的训练损失。我们假设 math_verify 有效地识别了特定的任务子集（主要是数值型任务），而基于 Llama 的验证或未过滤的数据集则保持了更广泛的数据多样性。
- 令人惊讶的是，未过滤的数据集并未遭受严重退化。尽管包含错误样本，no_restrictions 数据集在长时间的训练运行中仍保持了有竞争力的性能。

建议

根据我们的发现，最佳的过滤方法在很大程度上取决于训练 Token 预算。对于较短的运行，更严格的验证方法具有显著优势。作为一般性建议，我们建议将 Llama 验证与 math_verify 结合使用——就像在 LorMV 数据集中所做的那样。
