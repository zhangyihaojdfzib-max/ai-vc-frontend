# 欢迎来到智能体周

2026-04-12

- Rita Kozlov
- Dane Knecht

![](/images/posts/9019404287c9.png)

Cloudflare 的使命始终是帮助构建更好的互联网。有时这意味着为现有的互联网而构建。有时则意味着为即将到来的互联网而构建。

今天，我们正式启动智能体周，致力于为未来的互联网而构建。

## 互联网并非为人工智能时代而建。云也是如此。

我们所熟知的云，是上一次重大技术范式转变的产物：智能手机。

当智能手机将互联网装进每个人的口袋时，它们不仅仅是增加了用户——它们改变了“在线”的本质。永远连接，永远期待即时响应。应用程序必须处理数量级更多的用户，支撑它们的基础设施也必须随之演进。

行业最终采用的方法是直截了当的：用户越多，应用程序的副本就越多。随着应用程序变得越来越复杂，团队将它们拆分成更小的部分——微服务——这样每个团队都能掌控自己的命运。但核心原则保持不变：有限数量的应用程序，每个服务于众多用户。扩展意味着更多的副本。

Kubernetes 和容器成为了默认选择。它们使得启动实例、负载均衡以及销毁不需要的资源变得容易。在这种一对多模式下，单个实例可以服务许多用户，即使用户数量增长到数十亿，你需要管理的东西的数量仍然是有限的。

智能体打破了这一点。

## 一个用户，一个智能体，一项任务

与之前的所有应用程序不同，智能体是一对一的。每个智能体都是一个独特的实例。服务一个用户，执行一项任务。传统的应用程序无论谁在使用都遵循相同的执行路径，而智能体则需要自己的执行环境：一个由 LLM（大语言模型）决定代码路径、动态调用工具、调整其方法、并持续运行直到任务完成的环境。

可以将其想象成餐厅与私人厨师之间的区别。餐厅有菜单——一套固定的选项——以及一个为批量生产这些菜品而优化的厨房。这就是今天大多数应用程序的模式。智能体则更像是一位私人厨师，他会问：你想吃什么？他们每次可能需要完全不同的食材、器具或技术。你无法用经营餐厅的厨房配置来运营私人厨师服务。

在过去的一年里，我们看到智能体开始兴起，其中编码智能体一马当先——这并不奇怪，因为开发者往往是早期采用者。如今大多数编码智能体的工作方式是通过启动一个容器，为 LLM 提供所需的一切：文件系统、git、bash 以及运行任意二进制文件的能力。

但编码智能体仅仅是个开始。像 Claude Cowork 这样的工具已经让技术背景较弱的用户也能使用智能体。一旦智能体超越开发者，进入每个人——行政助理、研究分析师、客服代表、个人规划师——的手中，规模计算的数字很快就会变得令人警醒。

## 将智能体扩展到大众的数学计算

如果美国超过 1 亿的知识工作者每人使用一个智能体助手，并发率约为 15%，那么你将需要支持大约 2400 万个同时会话的能力。按每个 CPU 支持 25-50 个用户计算，这大约需要 50 万到 100 万个服务器 CPU——这还只是美国，且每人一个智能体。

现在想象一下每个人并行运行多个智能体。再想象一下世界上其他地区还有超过 10 亿的知识工作者。我们不是稍微缺乏计算资源。我们相差了好几个数量级。

那么，我们如何弥合这个差距呢？

## 为智能体构建的基础设施

八年前，我们推出了 Workers——这是我们开发者平台的开端，也是对无容器、无服务器计算的一次押注。当时的动机很实际：我们需要为依赖 Cloudflare 速度的客户提供无需冷启动的轻量级计算。基于 V8 隔离环境而非容器构建的 Workers，最终被证明效率高出一个数量级——启动更快、运行成本更低，并且天生适合“启动、执行、销毁”的模式。

我们当时没有预料到的是，这种模式会如此完美地映射到智能体时代。

容器为每个智能体提供了一个完整的商业厨房：固定的设备、步入式冰箱等等，无论智能体是否需要；而隔离环境则恰恰为私人厨师提供了制作这顿特定餐点所需的操作台空间、炉灶和刀具。在几毫秒内完成配置。菜品上桌后即刻清理。

在一个我们需要支持的不是数千个长期运行的应用程序，而是数十亿个短暂、单一用途的执行环境的世界里——隔离环境是正确的底层原语。

每个隔离环境在几毫秒内启动。每个都安全地沙盒化。并且，与容器相比，你可以在相同的硬件上运行数量级更多的隔离环境。

就在几周前，我们通过 Dynamic Workers 公开测试版进一步推进了这一点：在运行时按需启动的执行环境。一个隔离环境只需几毫秒启动，占用几兆字节内存。这大约比容器快 100 倍，内存效率最高可达容器的 100 倍。

你可以为每一个请求启动一个新的隔离环境，运行一段代码，然后将其丢弃——规模可达每秒数百万次。

为了让智能体超越早期采用者，进入每个人的手中，它们还必须负担得起。让每个智能体在自己的容器中运行成本高昂，以至于今天的智能体工具大多局限于为能够证明其成本合理的工程师服务的编码助手。而隔离环境通过实现数量级更高的运行效率，使得在智能体所需的规模下，单位经济效益成为可能。

## 无马马车阶段

虽然为未来构建正确的基础至关重要，但我们尚未到达那里。每一次范式转变都有一段时期，我们试图让新事物在旧模型中运作。第一辆汽车被称为“无马马车”。第一批网站是数字宣传册。第一批移动应用是缩小的桌面用户界面。我们现在正处于智能体的这个阶段。

你随处可见这种现象。

我们给智能体提供无头浏览器来浏览为人类眼睛设计的网站，而它们真正需要的是像 MCP 这样的结构化协议来直接发现和调用服务。

许多早期的 MCP 服务器只是现有 REST API 的薄包装层——相同的 CRUD 操作，新的协议——而实际上 LLM 在编写代码方面远比进行顺序工具调用更擅长。

我们使用验证码和行为指纹来验证请求的另一端是什么，而越来越多的情况下，另一端是代表某人行事的智能体——正确的问题不是“你是人类吗？”，而是“你是哪个智能体，谁授权了你，以及你被允许做什么？”

我们为那些只需要进行几次 API 调用并返回结果的智能体启动完整的容器。

这些只是几个例子，但都不足为奇。这就是转型期的样子。

## 为两者而构建

互联网总是处于两个时代之间。IPv6 客观上优于 IPv4，但放弃 IPv4 支持会破坏一半的互联网。HTTP/2 和 HTTP/3 共存。TLS 1.2 仍未完全让位于 1.3。更好的技术已经存在，旧的技术仍在持续，而基础设施的工作就是连接两者。

Cloudflare 一直致力于连接这些转型。向智能体的转变也不例外。

编程智能体确实需要容器——文件系统、Git、Bash、任意二进制执行能力。这是不可或缺的。本周，我们基于容器的沙箱环境将正式发布，因为我们致力于将其做到最好。我们正在深入探索智能体的浏览器渲染能力，因为未来仍会有大量服务尚未支持MCP协议，而智能体仍需与它们交互。这些并非权宜之计——它们是一个完整平台的重要组成部分。

但我们也在构建未来所需：智能体真正需要的隔离环境、协议和身份模型。我们的职责是确保您无需在当下可行方案与未来正确方案之间做出取舍。

## 安全内置于模型，而非外围附加

如果智能体将要处理我们的工作和个人任务——阅读邮件、操作代码、与金融服务交互——那么安全必须内建于执行模型中，而非事后叠加。

首席信息安全官们最先直面这一问题。将智能体交到每个人手中带来的生产力提升是真实的，但如今大多数智能体部署都充满风险：提示词注入、数据外泄、未授权API访问、不透明的工具使用。

开发者的灵感编程智能体需要访问代码库和部署流水线。企业的客服智能体需要访问内部API和用户数据。在这两种情况下，当前保障环境安全意味着要拼凑起那些本非为自主软件设计的凭证、网络策略和访问控制。

Cloudflare一直在并行构建两个平台：面向应用开发者的开发者平台，以及面向需要安全访问权限组织的零信任平台。曾有一段时间，这两个平台服务于不同的受众。

但"如何构建这个智能体？"与"如何确保其安全性？"正日益成为同一个问题。我们将这些平台整合，使所有功能都原生融入智能体的运行方式，而非额外附加的独立层。

## 遵守规则的智能体

智能体时代还有一个超越计算和安全的维度：经济与治理。

当智能体代表我们与互联网交互——阅读文章、调用API、访问服务——必须有一种方式，让创建这些内容和运营这些服务的个人及组织能够设定条款并获得报酬。如今，网络的经济模式围绕人类注意力构建：广告、付费墙、订阅。

智能体没有注意力（至少不是那种注意力）。它们不看广告，不点击cookie横幅。

如果我们想要一个智能体既能自由运作，又能让出版商、内容创作者和服务提供商获得公平报酬的互联网，就需要为此构建新的基础设施。我们正在开发工具，让出版商和内容所有者能够轻松设定和执行智能体与其内容交互的策略。

构建更好的互联网始终意味着确保它服务于所有人——不仅是构建技术的人，还包括那些以其工作和创造力让互联网值得使用的人。这在智能体时代不会改变，反而变得更加重要。

## 面向开发者和智能体的平台

我们对开发者平台的愿景始终是提供一个开箱即用的综合平台：从实验、到最小可行产品、再到扩展至数百万用户。但提供基础组件只是等式的一部分。一个优秀的平台还必须思考所有组件如何协同工作，以及如何融入您的开发流程。

这项工作正在演变。过去它纯粹关乎开发者体验，让人类能够轻松构建、测试和发布。如今，它也日益关乎帮助智能体协助人类，让平台不仅服务于构建智能体的人，也服务于智能体本身。智能体能否找到最新、最前沿的最佳实践？它能多容易地发现并调用所需的工具和CLI？从编写代码到部署代码的过渡能有多无缝？

本周，我们将在两个维度上发布改进——让Cloudflare对在其上构建的人类和在其上运行的智能体都更加友好。

## 构建未来需要团队协作

构建未来不是我们能独自完成的事。从HTTP/1.1到HTTP/2和HTTP/3，从TLS 1.2到1.3，每一次重大的互联网转型都需要行业在共同标准上达成一致。向智能体的转变也将如此。

Cloudflare长期以来一直致力于贡献并推动互联网运行标准的发展。我们深入参与IETF已超过十年，帮助开发和部署了QUIC、TLS 1.3和加密客户端问候等协议。我们是WinterTC（ECMA JavaScript运行时互操作性技术委员会）的创始成员。我们开源了Workers运行时本身，因为我们相信基础应该开放。

我们将以同样的方式迎接智能体时代。我们很高兴能成为Linux基金会和AAIF的一部分，帮助支持和推动像MCP这样的标准，这些标准将成为智能体未来的基石。自Anthropic推出MCP以来，我们与他们紧密合作，构建了远程MCP服务器的基础设施，开源了我们自己的实现，并投入资源使该协议能够大规模实用化。

去年，我们与Coinbase共同创立了x402基金会，这是一个开放、中性的标准，它复兴了长期沉寂的HTTP 402状态码，为智能体提供了一种原生方式来支付其消费的服务和内容。

智能体身份、授权、支付、安全：这些都需要开放标准，没有哪家公司能独自定义。

## 敬请关注

本周，我们将在智能体堆栈的每个维度发布公告：计算、连接、安全、身份、经济和开发者体验。

互联网并非为AI而建。云也非为智能体而建。但Cloudflare的使命始终是帮助构建更好的互联网——而"更好"的含义随着每个时代而变化。如今是智能体的时代。本周，请持续关注，我们将向您展示我们为此构建的一切。
