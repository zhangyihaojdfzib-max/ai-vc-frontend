# 与Mozilla合作提升Firefox安全性

AI模型现已能独立识别复杂软件中的高危漏洞。正如我们近期记录所示，Claude在久经测试的开源软件中发现了500多个零日漏洞（软件维护者未知的安全缺陷）。

本文我们将分享与Mozilla研究人员的合作细节：Claude Opus 4.6在两周内发现了22个漏洞。其中Mozilla将14个评定为高危漏洞——这相当于2025年Firefox修复的所有高危漏洞的近五分之一。换言之：AI正以前所未有的速度实现严重安全漏洞的检测。

![各来源报告的Firefox安全漏洞月度统计。Claude Opus 4.6在2026年2月发现22个漏洞，超过2025年任何单月报告数量。](/images/posts/60cce16c3d08.jpg)

在此次合作中，Mozilla处理了我们提交的大量报告，帮助我们理解哪些发现值得提交漏洞报告，并通过Firefox 148.0向数亿用户推送了修复补丁。他们的合作经验以及我们获得的技术认知，为AI赋能的安研人员与维护者如何协同应对当前挑战提供了范本。

## 从模型评估到安全合作

2025年末，我们注意到Opus 4.5已接近完成CyberGym基准测试的所有任务——该基准用于检验LLM能否复现已知安全漏洞。我们希望构建更困难且更贴近现实的评估，包含更高浓度的技术复杂漏洞（如现代网页浏览器中的漏洞）。因此我们建立了历史Firefox通用漏洞披露（CVE）数据集，以测试Claude能否复现这些漏洞。

选择Firefox是因为它既是复杂代码库，也是全球测试最充分、最安全的开源项目之一。这使其成为检验AI发现新型安全漏洞能力的更高难度测试——相比我们此前用于测试模型的开源软件。数亿用户每日依赖它，而浏览器漏洞尤其危险，因为用户常接触非可信内容并依赖浏览器提供保护。

我们的第一步是使用Claude在旧版Firefox代码库中定位已知CVE。令人惊讶的是，Opus 4.6能复现大部分历史CVE——尽管每个漏洞当初都需要大量人力才能发现。但我们仍不确定该结果的可靠程度，因为至少部分历史CVE可能已存在于Claude的训练数据中。

因此我们让Claude在Firefox当前版本中寻找新型漏洞——这些漏洞按定义此前不可能被报告过。我们首先聚焦Firefox的JavaScript引擎，随后扩展到浏览器其他模块。JavaScript引擎是理想的起点：它是Firefox代码库中可独立分析的模块，且因其广泛的攻击面（处理用户浏览网页时的非可信外部代码）而具有特殊的安全重要性。

仅探索二十分钟后，Claude Opus 4.6报告在JavaScript引擎中发现一处释放后使用漏洞（一种允许攻击者用任意恶意内容覆盖数据的内存漏洞）。我们的研究员在独立虚拟机中使用最新Firefox版本验证了该漏洞，随后转交另两位Anthropic研究员进行复核。接着我们在Mozilla的问题追踪系统Bugzilla提交了漏洞报告，附上漏洞描述及修复方案（由Claude编写并经报告团队验证）以辅助根因分析。

在我们验证并提交首个漏洞给Firefox期间，Claude已发现另外50个独特的崩溃输入。当我们正在分析这些崩溃时，一位Mozilla研究员主动联系我们。经过双方技术流程讨论并分享更多我们人工验证的漏洞后，他们鼓励我们批量提交所有发现——即使我们不确定所有崩溃测试用例都涉及安全问题。在此次工作结束时，我们扫描了近6000个C++文件，累计提交112份独立报告，包含前文提及的高危与中危漏洞。大部分问题已在Firefox 148中修复，其余将在后续版本解决。

在外部软件进行此类漏洞挖掘时，我们始终意识到可能遗漏代码库的关键信息而导致误报。我们尽力通过验证履行尽职调查，但误差难免存在。我们衷心感谢Mozilla对其分析流程的透明态度，并帮助我们调整方法以确保只提交他们关注的测试用例（即使并非所有最终都与安全相关）。Mozilla研究人员现已开始在内部尝试将Claude用于安全研究。

## 从识别漏洞到编写初级攻击程序

为测量Claude网络安全能力的上限，我们还开发了新评估方案以检验Claude能否利用所发现的漏洞。换言之，我们希望了解Claude是否能开发黑客用于利用这些漏洞执行恶意代码的工具。

为此，我们向Claude开放提交给Mozilla的漏洞信息，要求其针对每个漏洞创建攻击程序。为证明成功利用漏洞，我们要求Claude演示真实攻击——具体而言，需像攻击者那样在目标系统中读写本地文件。

我们以不同起点进行了数百次测试，消耗约4000美元API额度。尽管如此，Opus 4.6仅在两例中成功将漏洞转化为攻击程序。这说明两点：其一，Claude发现漏洞的能力远强于利用漏洞；其二，识别漏洞的成本比创建攻击程序低一个数量级。然而，Claude能自动开发原始浏览器攻击程序（即使仅少数案例）的事实令人担忧。

“原始”是此处的重要限定词。Claude编写的攻击程序仅在我们测试环境中生效——该环境刻意移除了现代浏览器的部分安全功能，最重要的是沙箱机制（其作用是降低此类漏洞的影响）。因此Firefox的“纵深防御”本可有效缓解这些特定攻击。但突破沙箱的漏洞并非前所未闻，而Claude的攻击正是端到端攻击程序的必要组成部分。您可在我们的Frontier Red Team博客阅读Claude如何开发其中一款Firefox攻击程序的更多细节。

## AI赋能网络安全的未来展望

AI参与攻击程序开发的早期迹象，凸显了加速防御方“发现-修复”流程的重要性。为此，我们希望分享在此次分析中发现的技术与流程最佳实践。

首先，在研究使用LLM开发和验证补丁的“补丁Agent”时，我们开发了若干方法，希望能帮助维护者使用Claude等LLM更快地分析与处理安全报告。

根据我们的经验，当Claude能够借助其他工具检查自身工作时，其表现最佳。我们将这类工具称为“任务验证器”：一种确认AI智能体（Agent）输出是否真正达成其目标的可靠方法。任务验证器能在智能体探索代码库时提供实时反馈，使其能够深入迭代直至成功。

任务验证器帮助我们发现了上文所述的Firefox漏洞，并且在独立研究中，我们发现它们对于修复漏洞也很有用。一个优秀的补丁生成智能体至少需要验证两件事：漏洞是否确实被移除，以及程序的预期功能是否得以保留。在我们的工作中，我们构建了能自动测试原始漏洞在应用建议修复后是否仍能被触发的工具，并分别运行测试套件以捕捉回归问题（即意外破坏其他功能的更改）。我们预计维护者最清楚如何为自己的代码库构建这些验证器；关键在于，为智能体提供可靠的方法来检查这两项属性，能显著提升其输出质量。

我们不能保证所有通过这些测试的智能体生成补丁都足够完善，可以立即合并。但任务验证器让我们更有信心，确信生成的补丁既能修复特定漏洞，又能保持程序功能——从而满足一个可信补丁的最低要求。当然，在审查AI生成的补丁时，我们建议维护者采用与审查外部作者提交的任何其他补丁相同的严格标准。

从提交漏洞和补丁的流程来看：我们知道维护者工作繁重。因此，我们的方法是向维护者提供他们所需的信息，以便信任和验证报告。Firefox团队强调了我们的提交中三个对其信任结果至关重要的组成部分：

1.  附带的最小化测试用例
2.  详细的概念验证
3.  候选补丁

我们强烈建议使用基于LLM（大语言模型）的漏洞研究工具的研究人员，在提交基于此类工具输出的报告时，包含类似的验证和可复现性证据。

我们还发布了我们的《协调漏洞披露操作原则》，其中描述了我们在与维护者合作时将遵循的程序。目前，我们的流程遵循标准的行业规范，但随着模型的改进，我们可能需要调整流程以跟上能力的发展。

## 当下的紧迫性

前沿语言模型如今已是世界级的漏洞研究员。除了我们在Firefox中发现的22个CVE之外，我们还使用Claude Opus 4.6发现了其他重要软件项目（如Linux内核）中的漏洞。在接下来的几周和几个月里，我们将继续报告我们如何使用我们的模型，并与开源社区合作以提升安全性。

目前，Opus 4.6在识别和修复漏洞方面的能力远强于利用漏洞。这给了防御者优势。随着最近Claude Code Security以有限研究预览版发布，我们正将漏洞发现（和修复）能力直接带给客户和开源维护者。

但考虑到进展速度，前沿模型在漏洞发现和利用能力之间的差距不太可能持续很长时间。如果未来的语言模型突破了这一利用障碍，我们将需要考虑额外的防护措施或其他行动，以防止我们的模型被恶意行为者滥用。

我们敦促开发者利用这个窗口期，加倍努力使其软件更加安全。就我们而言，我们计划显著扩大网络安全工作，包括与开发者合作寻找漏洞（遵循上述CVD流程）、开发工具帮助维护者分类处理漏洞报告，以及直接提出补丁。
