# Fastweb + Vodafone：使用 LangGraph 和 LangSmith 通过 AI Agent（智能体）变革客户体验

了解 Fastweb + Vodafone 如何通过其智能体 Super TOBi 和 Super Agent 彻底革新客户服务和呼叫中心运营。

Fastweb + Vodafone 是瑞士电信集团的一部分，是欧洲领先的电信提供商之一，为数百万意大利客户提供服务。认识到传统的客户服务方法有待改进以满足其不断增长的需求，Fastweb + Vodafone 开启了一场 AI 转型，以彻底改变其提供客户支持的方式。

电信行业的客户服务规模庞大，挑战复杂且相互关联。客户需要立即获得关于账单查询、服务激活、漫游问题和技术支持的帮助，通常期望在一次互动内解决问题。传统的 TOBi 难以处理那些需要上下文理解、访问多个系统并进行端到端解决的复杂请求。

对于 Fastweb + Vodafone 的呼叫中心顾问而言，挑战同样复杂。客服人员需要快速访问客户数据、了解服务历史，并在保持高标准服务质量的同时确定合适的解决方案。虽然有效，但手动咨询多个系统和知识库的过程可以从智能支持中受益，以进一步提升客户体验的速度和一致性。因此，Fastweb + Vodafone 需要一个能够同时处理面向客户的互动，并赋能呼叫中心客服人员始终提供卓越服务的 AI 解决方案。

## 使用 LangChain 和 LangGraph 构建智能客户互动

Fastweb + Vodafone 选择 LangChain 和 LangGraph 作为其 AI 转型的基础，因为他们的客户服务流程自然地映射到了基于图的决策流程。他们的实施围绕两个旗舰项目展开：Super TOBi 和 Super Agent。

### Super TOBi：企业级 AI 驱动的客户服务

Super TOBi 在 Fastweb + Vodafone 现有的聊天机器人（TOBi）基础上构建，将其转变为一个能够跨多个渠道处理复杂客户互动的智能体系统。该系统遵循结构化方法，模仿经验丰富的客户服务代表处理查询的方式。

该系统的架构围绕两个主要智能体组织，两者都实现为 LangGraph 图：**Supervisor** 和 **Use Cases**。

**Supervisor** 是所有用户查询的中央入口点。它的首要职责是应用护栏、过滤和塑造输入，以确保其有效且安全。除此之外，它还管理特殊场景，例如对话结束、转接人工客服、简单互动（如问候）等。一旦输入被处理，Supervisor 要么将查询直接路由到最合适的 Use Case，要么在不确定的情况下**提出澄清问题**以确定正确的前进路径。通过这种方式，它编排整个对话流程，确保每个查询都能得到一致且高效的处理。

**Use Cases** 代表专门的智能体，每个都设计用于解决特定类别的客户需求。它们可以访问明确定义的客户 API 子集，并遵循 **LLM Compiler 模式**。这种方法使它们能够推理应该调用哪些 API，在需要时协调多个步骤，并生成针对客户上下文定制的最终响应。重要的是，一些 Use Cases 不仅限于自然语言响应：它们还可以发出**结构化操作标签**。这些标签使系统能够在对话中直接执行事务流程。例如，一个操作标签可能启动优惠激活、禁用正在进行的服务或触发支付方式更新。

当返回此类操作标签时，**ChatBot** 会在对话界面中自动执行它们。这种集成创造了对话与行动的无缝融合，使客户能够在同一次对话体验中，轻松地从提问过渡到完成交易。

Fastweb + Vodafone 在 LangGraph 中实现的 LLM Compiler 模式使系统能够为每个客户请求生成全面的计划，无缝执行 API 调用、数据检索和多步骤问题解决。

目前，Super TOBi 通过客户伴侣应用和语音渠道为近 **950 万客户**提供服务，处理包括成本控制、有效优惠、漫游、销售和账单在内的用例，并取得了令人印象深刻的业务指标：**90% 的正确率，82% 的解决率，以及 5.2 分（满分 7 分）的客户费力指数**，这转化为更快的响应时间、更少的人工客服转接和更高的客户满意度。

### Super Agent：使用 LangGraph 赋能卓越呼叫中心

Super Agent 是 Fastweb + Vodafone 面向内部的 AI 项目，通过即时诊断、合规指导和有来源支持的说明来增强每位顾问的能力。与面向消费者的机器人 Super TOBi 不同，Super Agent 从不直接与客户对话；相反，它为人工顾问提供确切的下一步指导。得益于这种方法，生成式 AI 的使用已将**一次性解决率推高至 86% 以上**。

该解决方案融合了 LangChain 的可组合工具与 LangGraph 的可控编排，并将所有运营知识作为动态图存储在 Neo4j 中。

#### 从业务模板到图：一个自动化的 ETL 流程

业务专家使用具有定义步骤、条件和操作的结构化模板来编写故障排除和信息程序。提交后，一个由 LangGraph 和特定任务 LLM Agent（包括 ReAct Agent）驱动的自动化 ETL 管道会将文档解析为 JSON，提取验证 API，执行一致性检查，并完善步骤定义。

内容被分解为节点和关系，并作为知识图谱的一部分存储在 Neo4j 中：步骤链接到条件、操作和相关的 API 节点。然后，一个 CI/CD 管道自动化构建、验证和部署，在数小时内将更新后的图谱推送到生产环境，且无需停机。

#### 意图路由器和执行流程

来自顾问的传入请求，无论是关于用户问题的故障排除支持请求，还是一般知识库问题，首先由 LangGraph Supervisor 处理，该 Supervisor 确定请求是匹配**基于图的程序**还是**开放式问题**。对于故障排除请求，会识别出由业务部门编写并作为知识图谱存储在 Neo4j 中的相应程序。在此阶段会自动注入 CRM 数据，以可靠地识别用户，确保每个请求都链接到正确的用户，并且响应是针对适当的客户上下文定制的。

#### 基于图的程序执行

针对结构化的故障排查与隔离场景，Supervisor会启动一个流程子图执行器。该系统利用LangChain和LangGraph，从Neo4j中检索流程的**首个步骤节点**及其关联的**条件节点**、**操作节点**和**API节点**。对于每个步骤，系统会执行所需的API来验证步骤条件。若某个条件得到满足，流程即告结束：此时用户问题已被识别，系统会将对应的**操作**与从API收集的上下文信息相结合，生成响应。如果所有条件均未满足，则流程将转向该过程中的**下一个步骤节点**，并迭代重复此过程，直至找到问题及其解决方案。

#### 用于开放式问题的图RAG

对于公司知识库中通用或非结构化的问题，系统会将其路由至一个混合**检索增强生成（RAG）**链，该链结合了**向量存储**与**Neo4j知识图谱**。向量存储负责查找一组广泛的潜在相关段落，随后知识图谱用正确的上下文锚定答案，添加来源引用，并确保其遵循公司政策。

### 技术亮点

• **LangSmith**将链追踪、延迟指标和评估分数流式传输至Fastweb + Vodafone的内部仪表板，以实现持续优化。

• **设计即治理**：每项建议都需根据编码了Fastweb + Vodafone政策的**规则节点**进行验证。

**每日评估流程：**
