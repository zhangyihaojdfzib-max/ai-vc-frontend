作者：Rahul Verma，LangChain 部署工程师

当 AI Agent（智能体）能够体现您的团队长期积累的知识与判断力时，其表现最佳。其中一部分是已形成文档的制度性知识，Agent 可以直接使用。但大多数优秀的组织也依赖于存在于员工头脑中的隐性知识。团队往往在尝试构建 AI Agent 以实现工作流程自动化时，才意识到这些信息对于完成有意义的工作有多么关键。要确保这种智慧能够融入 Agent，就需要建立一个包含领域专家意见的改进循环。

在本指南中，我们将涵盖：

*   您的 Agent 中哪些组件将从吸收人类判断中受益
*   如何在开发生命周期的每个步骤中将人类判断融入您的 Agent

## 现实启发示例：交易员助手

设想一家金融服务公司，其交易员需要最新的市场数据。目前，他们会将问题发送给数据科学团队。数据科学家编写 SQL 查询，检索相关数据，然后将结果发回。由于 LLM（大语言模型）擅长生成 SQL，这个工作流程自然成为用 AI Agent 实现自动化的理想候选：交易员能获得更快的响应，而数据科学家则可以腾出时间从事更有趣的项目。

为了使这个系统可靠工作，Agent 需要在金融服务领域层面和技术数据库层面都具备上下文知识。前者包括决定如何解释诸如"今日风险敞口"或"近期波动性"等请求的、未成文的交易惯例。后者包括关于数据库的实用知识，例如哪些表格是权威的、哪些已过时，或者哪些查询模式往往不正确或效率低下。我们需要与相应的领域专家合作，以纳入 Agent 所需的所有未成文的上下文。

我们将在整个指南中使用这个示例来提供具体的实现案例。这是一个很好的例子，因为其架构简单，并且它展示了在 Agent 设计中融入人类判断的关键原则。

让我们回顾一下可以通过人类判断来改进的 Agent 的各个组成部分。

## 人类输入如何改进 AI Agent 的每个组件

构建一个 Agent 意味着决定何时调用 LLM，并管理每次调用时提供的上下文（例如，文档、记忆、对话历史、工具）以实现期望的结果。这些设计选择中的每一项都能从相关利益相关者的输入中受益。

### 工作流设计

如今的 LLM 非常擅长规划自身的行动顺序。只需给它们一些工具和自然语言指令，它们就能弄清楚按什么顺序调用哪些工具。然而，使用确定性代码来定义工作流的某些部分是有好处的：延迟更低、Token 使用更少，并且能确保关键步骤确实执行。在一些受监管或高风险的环境中，您需要代码来严格控制行动序列。在我们的交易员助手中，我们将让 LLM 自主生成并执行 SQL 查询，但会添加代码，要求其在将最终答案返回给交易员之前，验证该答案是否符合公司的风险与合规要求。我们需要风险与合规专家的输入来创建自动检查，以强制执行公司的标准。我们还可以将这些信息包含在 Agent 的预加载上下文中，以提高其首次尝试就生成有效答案的几率。

### 工具设计

开发人员必须实现 Agent 可以使用的工具，并配置 LLM 赖以决定何时调用这些工具的名称、参数和描述。在工作流的不同阶段改变可用工具集通常很有用。限制单次 LLM 调用的工具集可以引导模型产生预期行为，并降低其选择不相关选项的可能性。

我们助手的工具可能包括数据库模式检查、查询执行和数据库文档检索。一个关键的权衡是 LLM 生成查询的灵活性与控制力：通用的 `execute_sql` 步骤允许灵活的查询但增加了风险；参数化的查询工具更安全但能力较弱。仔细审视您的业务约束可能会让您感觉哪种选择适合您，但要确切知道，您需要进行评估，以确定您工具设计的性能和风险特征，并且只有在所有利益相关者对结果都满意时才能发布。

### Agent 上下文

早期的 Agent 只是给模型一个单一的系统提示词和一组工具定义。随着时间的推移，行业已转向在 Agent 开始执行时为其提供更丰富的上下文。Anthropic 的 Skills（自 10 月推出后迅速流行的标准）就是这一趋势的一个突出例子。

与其将所有内容塞进一个系统提示词，您的团队可以预先整理文档、示例和领域规则，然后让 Agent 在运行时获取所需内容。这使得 Agent 能够使用远更多的知识，而不会使系统提示词变得臃肿。有效的 Agent 设计涉及决定 Agent 应访问哪些知识，并对其进行组织，以便 Agent 能在正确时刻检索到正确的信息。

至少，我们的交易员助手需要知道如何使用数据库并理解其模式。根据我们的助手所需来自团队的额外知识的性质和数量，我们不仅需要花时间收集这些知识，还需要确定如何构建这些知识并逐步向我们的 Agent 披露。

在 Agent 启动时，选择并构建其可用的信息，是上下文工程这门学科的一部分。上下文工程还涵盖随着 Agent 执行任务，您在每次 LLM 调用中提供的信息如何演变。您的人类利益相关者在审查 Agent 的输出和评估分数时提供的反馈，可能会影响您为 Agent 进行端到端上下文工程的方法。

既然我们已经概述了 Agent 中能从人类判断中受益的部分，接下来我们将介绍如何收集这些人类输入。

## 将人类判断融入 Agent 改进循环

在 LangChain，我们与数百家部署 AI Agent 的组织合作过。最成功的团队遵循一个紧密的迭代循环：他们快速构建一个 Agent，将其部署到生产或类生产环境，并在每个步骤收集数据以指导改进。我们称之为"Agent 改进循环"，因为我们观察到大多数成功的 Agent 都多次经历这个循环，并且之前在此处更详细地介绍过。

快速且频繁地迭代至关重要，因为决定 Agent 行为的是 LLM 的实时推理，而不是代码。在 AI Agent 运行之前，不可能知道它会做什么。AI Agent 的界面通常是自由形式的，例如用户可以输入任何内容的文本框，这使得预测用户与您的 Agent 之间的交互会是什么样子更加困难。将您的 Agent 呈现在用户面前，是收集使其最终成功所需数据的唯一途径。

### 开发阶段：构建测试集与评估器

在开发开始之前，工程师应至少拥有一小部分用例场景和预期行为作为项目需求的一部分。这些初始测试有助于确认Agent能正确执行核心任务。当Agent接近生产就绪状态时，工程师应与产品经理和领域专家合作，构建更全面的测试集，以评估整体行为和关键子组件。

对于我们的助手，我们将使用LangSmith的"数据集"功能手动创建一些包含自然语言问题及其正确答案的基准数据集。我们还将创建包含在我们的数据库上下文中表现良好、高效的SQL示例的数据集。随着开发人员构建Agent，我们将使用LangSmith的"评估"功能针对这些数据集运行测试。LangSmith界面让技术和非技术团队成员都能审查评估结果并添加注释，从而使每个人都能就开发人员的后续步骤达成一致。

在此阶段，我们可以通过手动测试中遇到的、有启发性的案例来扩充初始数据集，从而创建一个微型飞轮效应。这有助于逐步自动化我们的反馈循环，并确保在准备发布Agent的v1版本时，我们拥有一个全面的测试集。

### 部署后：使用自动化评估与监控，将人力引导至最需要的地方

一旦您的Agent上线，您需要确保其可靠性，并快速识别问题或改进机会。验证用户体验的传统方法是满意度调查和用户访谈，但该方法的缺陷在于它测量的是用户告诉您的内容，而非他们的实际行为。"LLM即评委"评估器为我们提供了一种更稳健的方法。在生产数据上运行的自动化评估有助于监控Agent，并发现值得人工关注的情况。例如，LLM评委可以自动检测用户何时表达不满，并标记这些交互以供审查。团队成员随后可以调查追踪记录，并判断问题反映的是错误、Agent知识缺口还是工作流程中的弱点。

对于我们的交易助手，我们可能会运行一份洞察报告，自动识别相似的对话并将其聚类到用例类别中。了解交易员向助手提问背后的潜在主题，有助于我们识别那些需要特别确保良好支持的用例，甚至可能发现未来可以更好地服务这些用例的产品新增功能。

随着自动化评估、人工注释和聚合层面洞察的积累，它们清晰地描绘了Agent在现实世界中的表现。这些学习成果将反馈到循环的最后一步：通过构建Agent的下一个版本来重启迭代循环。

## 持续优化：将今日的生产数据转化为明日的测试集

当您构建Agent的第一个版本时，您的评估集充其量只是对需要哪些测试来验证其工作的有根据的猜测。上线后，您将获得一个更好的测试用例来源：真实的生产数据。

您需要将这些数据整理成全面但不过于庞大的测试集。自动化系统可以帮助生成候选数据集，例如，根据评估器结果筛选生产追踪记录。但我们通常需要人类判断来整理平衡、有代表性的评估集。如果精心挑选，仅用几百个示例运行的评估也可能很有用，因此值得让专家参与决定哪些示例应构成测试集。

一旦我们的交易公司让助手在生产环境中运行一段时间，团队将能够访问真实的SQL查询和聊天机器人对话，以及通过监控和精心管理的注释队列收集的在线评估器结果和人工意见。

我们的团队可以从已审查的追踪记录中创建数据集，以运行更稳健的评估套件，从而显著提升我们智能体在v2及后续版本中的性能。我们能整理的最有用的数据集之一是“黄金数据集”，它包含迄今为止助手最佳工作的示例，因此我们可以将其作为基准，确保未来版本至少表现相当。LangSmith使得这一过程易于实现。使用在线评估器分数来识别候选追踪记录，然后将它们放入标注队列，以便我们的领域专家可以决定哪些记录真正属于黄金数据集。

## 结论

有效的智能体开发将人类判断与自动化评估的可扩展性相结合。人类专业知识通过塑造工作流程、工具、上下文和评估标准，帮助定义什么是“好”。自动化评估器大规模应用这种判断，帮助团队快速测试智能体、监控其在生产环境中的行为，并将人类的注意力引导到最重要的案例上。

随着时间的推移，这会形成一个飞轮效应。人类反馈改进评估器、测试套件和智能体本身，我们部署的改进后智能体为我们提供更多数据，这些数据告诉我们如何进一步改进它，而这些洞察力驱动着下一个开发迭代。

我们使用一个简单的用例来说明这个过程，但同样的原则适用于构建任何智能体：建立紧密的迭代循环，在可扩展的评估中捕捉专家判断，并持续将生产数据转化为更好的测试。这是创建能为您的业务创造有意义价值的AI智能体的关键。
