# 修复 Pingora 开源部署中的请求走私漏洞

2026-03-09

- Edward Wang
- Fei Deng
- Andrew Hauck

![](/images/posts/af156cdc2a28.png)

2025年12月，Cloudflare 收到报告，称当 Pingora 开源框架被用于构建入口代理时，存在 HTTP/1.x 请求走私漏洞。今天，我们将讨论这些漏洞的工作原理以及我们如何在 Pingora 0.8.0 版本中修复了它们。

这些漏洞编号为 CVE-2026-2833、CVE-2026-2835 和 CVE-2026-2836。这些问题由 Rajat Raghav (xclow3n) 通过我们的漏洞赏金计划负责任地报告给我们。

我们的调查发现，Cloudflare 的 CDN 和客户流量未受影响。Cloudflare 客户无需采取任何行动，也未检测到任何影响。

由于 Cloudflare 网络的架构，这些漏洞无法被利用：Pingora 在 Cloudflare 的 CDN 中并未用作入口代理。

然而，这些问题会影响暴露在互联网上的独立 Pingora 部署，并可能使攻击者能够：

*   绕过 Pingora 代理层安全控制
*   使 HTTP 请求/响应与后端不同步，以进行跨用户劫持攻击（会话或凭据窃取）
*   污染从共享后端检索内容的 Pingora 代理层缓存

绕过 Pingora 代理层安全控制

使 HTTP 请求/响应与后端不同步，以进行跨用户劫持攻击（会话或凭据窃取）

污染从共享后端检索内容的 Pingora 代理层缓存

我们已经发布了包含修复和加固措施的 Pingora 0.8.0。虽然 Cloudflare 客户未受影响，但我们强烈建议 Pingora 框架的用户尽快升级。

## 漏洞是什么？

报告描述了几种可能导致去同步攻击的不同 HTTP/1 攻击载荷。此类请求可能导致代理和后端对请求体结束位置产生分歧，从而允许第二个请求"走私"通过代理层检查。研究人员提供了一个概念验证，以验证基本的 Pingora 反向代理如何错误解析请求体长度并将这些请求转发给 Node/Express 或 uvicorn 等服务器后端。

收到报告后，我们的工程团队立即展开调查并确认，正如报告者所证实的那样，Cloudflare CDN 本身不易受攻击。然而，团队也确认，当 Pingora 作为共享后端的入口代理时，确实存在漏洞。

根据设计，Pingora 框架确实允许处理不完全符合 RFC 规范的边缘情况 HTTP 请求或响应，因为我们必须为使用遗留 HTTP 堆栈的客户接受此类流量。但这种宽松性是有限度的，以避免 Cloudflare 自身暴露于漏洞之下。

在这种情况下，Pingora 在其 HTTP/1 堆栈中对请求体的解析存在不符合 RFC 规范之处，从而允许这些去同步攻击存在。Cloudflare 内部的 Pingora 部署并未直接暴露于入口流量，我们发现到达 Pingora 服务的生产流量并未受到这些错误解析的影响。因此，与 2025 年 5 月披露的先前 Pingora 走私漏洞不同，这些攻击无法在 Cloudflare 自身的流量上利用。

我们将逐一解释这些攻击载荷的工作原理。

### 1. 未完成 101 握手即提前升级

第一份报告显示，带有 `Upgrade` 头值的请求会导致 Pingora 在后端尚未接受升级（通过返回 `101 Switching Protocols`）之前，立即通过 HTTP 连接传递后续字节。因此，攻击者可以在同一连接上的升级请求之后，流水线发送第二个 HTTP 请求：

```Rust
GET / HTTP/1.1
Host: example.com
Upgrade: foo


GET /admin HTTP/1.1
Host: example.com
```

Pingora 将仅解析初始请求，然后将剩余的缓冲字节视为"已升级"的流，并由于 `Upgrade` 头（直到收到响应）而直接以"直通"模式转发给后端。

这完全不符合 RFC 9110 规定的 HTTP/1.1 升级流程的预期工作方式。后续字节只有在收到 `101 Switching Protocols` 头时才应被解释为升级流的一部分；如果收到的是 `200 OK` 响应，则后续字节应继续被解释为 HTTP。

发送升级请求然后流水线发送部分 HTTP 请求的攻击者可能导致去同步攻击。即使后端服务器以 200 拒绝升级，Pingora 也会错误地将两者解释为同一个升级请求。

通过这种不当的直通方式，收到非 101 响应的 Pingora 部署仍会将第二个部分 HTTP 请求按原样转发给上游，绕过任何 Pingora 用户定义的 ACL 处理或 WAF 逻辑，并污染到上游的连接，使得来自不同用户的后继请求可能不当收到 `/admin` 响应。

在攻击载荷之后，Pingora 和后端服务器现在处于"不同步"状态。后端服务器将等待，直到它认为 Pingora 转发的部分 `/attack` 请求头的其余部分完成。当 Pingora 转发另一个用户的请求时，从后端服务器的角度看，这两个请求头被组合在一起，攻击者现在已经污染了其他用户的响应。

我们随后修补了 Pingora，使其仅在上游响应 `101 Switching Protocols` 后才切换对后续字节的解析。

我们确认 Cloudflare 未受影响，原因有二：

1.  入口 CDN 代理没有这种不当行为。
2.  我们内部 Pingora 服务的客户端不会尝试流水线发送 HTTP/1 请求。此外，这些客户端直接与之通信的 Pingora 服务会通过注入 `Connection: close` 头来禁用这些 `Upgrade` 请求的保持连接；这防止了在同一连接上发送（并随后被走私）额外的请求。

入口 CDN 代理没有这种不当行为。

我们内部 Pingora 服务的客户端不会尝试流水线发送 HTTP/1 请求。此外，这些客户端直接与之通信的 Pingora 服务会通过注入 `Connection: close` 头来禁用这些 `Upgrade` 请求的保持连接；这防止了在同一连接上发送（并随后被走私）额外的请求。

### 2. HTTP/1.0、关闭定界和传输编码

报告者还演示了看似更经典的"CL.TE"去同步类型攻击，即 Pingora 代理使用 Content-Length 作为帧界定，而后端使用 Transfer-Encoding 作为帧界定：

```Rust
GET / HTTP/1.0
Host: example.com
Connection: keep-alive
Transfer-Encoding: identity, chunked
Content-Length: 29

0

GET /admin HTTP/1.1
X:

```

在报告者的示例中，Pingora 会将第一个 `GET /` 请求头之后的所有后续字节视为该请求体的一部分，但 node.js 后端服务器会将请求体解释为分块编码并在零长度块处结束。这里实际上涉及几个方面：

1. Pingora的分块编码识别机制相当简陋（仅检查`Transfer-Encoding`是否为"chunked"），并且假设只能存在一个编码或`Transfer-Encoding`头部。但RFC仅**要求**最终的编码必须是**chunked**才能应用分块帧结构。因此根据RFC，此请求本应具有分块消息体（如果它不是HTTP/1.0版本——下文将详述）。
2. Pingora**实际上也未使用**`Content-Length`（因为根据RFC，`Transfer-Encoding`会覆盖`Content-Length`）。由于存在无法识别的`Transfer-Encoding`和HTTP/1.0版本，请求体**反而被当作关闭分隔式**处理（这意味着响应体的结束由底层传输连接的关闭来标记）。在HTTP/1.0中，缺少帧结构头部也会触发相同的误解。虽然响应体允许采用关闭分隔式，但请求体**绝不**允许采用关闭分隔式。事实上，这一说明现在已作为单独注释在**RFC 9112**中明确列出。
3. 这是一个**未定义**`Transfer-Encoding`的HTTP/1.0请求。RFC**要求**，对于包含`Transfer-Encoding`的HTTP/1.0请求，必须"将消息视为帧结构错误"并关闭连接。像nginx和hyper中的解析器会直接拒绝此类请求，以避免帧结构歧义。

当攻击者在HTTP/1.0 + `Transfer-Encoding`请求之后流水线式发送部分HTTP请求头时，Pingora会错误地将该部分头部解释为同一请求的一部分，而非独立请求。这使得攻击者能够实施与前述过早升级示例中描述的相同类型的去同步攻击。

这暴露了对RFC更根本性的误读，特别是在响应与请求消息帧结构方面。我们随后修复了**多重`Transfer-Encoding`解析**不当的问题，严格遵守请求长度规范以确保HTTP请求体**绝不被视为关闭分隔式**，并拒绝**无效的`Content-Length`**和**HTTP/1.0 + `Transfer-Encoding`**请求消息。我们增加的进一步防护措施包括**默认拒绝**`CONNECT`请求，因为HTTP代理逻辑目前未将CONNECT视为特殊的CONNECT升级代理处理，而这些请求具有特殊的**消息帧结构规则**。（请注意，传入的CONNECT请求会被Cloudflare CDN**拒绝**。）

当我们内部调查并检测服务时，未发现任何到达Pingora服务且可能被误解的请求。我们发现CDN中的下游代理层仅会以HTTP/1.1版本转发请求，拒绝存在歧义的帧结构（如无效的`Content-Length`），并且对于分块请求仅转发单个`Transfer-Encoding: chunked`头部。

### 3. 缓存键构建

研究人员还报告了另一个关于默认`CacheKey`构建的缓存投毒漏洞。**朴素的默认实现**仅考虑URI路径（未包含其他因素如主机头或上游服务器HTTP方案），这意味着使用相同HTTP路径的不同主机可能发生冲突并相互投毒缓存。

这将影响选择使用默认`CacheKey`实现的alpha版代理缓存功能用户。我们随后**移除了该默认实现**，因为虽然类似HTTP方案 + 主机 + URI的构建方式对许多应用是合理的，但我们希望用户在为自身构建缓存键时保持谨慎。例如，如果其代理逻辑会条件性地调整上游请求的URI或方法，那么该逻辑很可能也必须纳入缓存键方案以避免投毒。

在Cloudflare内部，**默认缓存键**使用多种因素来防止缓存键投毒，且从未使用先前提供的默认实现。

## 建议

如果您将Pingora用作代理，请尽快升级至**Pingora 0.8.0**。

我们对此漏洞可能给Pingora用户造成的影响深表歉意。随着Pingora在Cloudflare之外成为关键互联网基础设施，我们认为该框架默认促进严格RFC合规使用至关重要，并将持续推动这项工作。极少数框架用户需要处理与Cloudflare相同的"互联网复杂环境"。我们的初衷是，默认更严格地遵循最新RFC标准将增强Pingora用户的安全性，并推动整个互联网迈向最佳实践。
