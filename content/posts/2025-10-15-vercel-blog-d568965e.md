ChatGPT应用是在ChatGPT对话中运行的交互式应用程序。当您要求ChatGPT“为我找一家巴黎的酒店”时，Booking.com的小部件会直接出现在聊天界面中，并显示您可以与之交互的真实搜索结果。这些应用覆盖超过8亿ChatGPT用户，并在对话相关时自然出现。

ChatGPT应用基于**模型上下文协议（MCP）**构建，这是一个连接AI模型与外部工具和数据的开放标准。MCP的工作原理类似于REST或GraphQL，但专为AI智能体设计。您无需硬编码API调用，而是以结构化格式定义模型可以发现和使用的工具与资源。当ChatGPT需要显示您的应用时，它会从您的MCP服务器获取HTML并在iframe中渲染。

对于Next.js开发者而言，这意味着您现有的应用程序通过适当的补丁即可在ChatGPT内部运行，无需从头重建即可获得即时分发渠道。我们的**入门模板**为您提供了一个可立即**部署到Vercel**的可用ChatGPT应用。

## ChatGPT的嵌套iframe架构如何破坏现代框架

ChatGPT采用三层iframe架构渲染应用，以保护自身免受潜在恶意应用的侵害。iframe是一种HTML元素，可将一个网页嵌入另一个网页，创建隔离的沙箱环境。

```
1chatgpt.com2  └── web-sandbox.oaiusercontent.com (沙箱iframe)3     └── web-sandbox.oaiusercontent.com (内部iframe)4        └── 您的应用HTML 
```

沙箱域名（web-sandbox.oaiusercontent.com）充当ChatGPT与您的应用之间的安全缓冲区。这能将应用与ChatGPT主界面隔离，并保护用户免受潜在有害代码的影响。

但这种安全模型带来了若干技术挑战。当Next.js运行在最内层iframe时，它会误以为自己的源是`web-sandbox.oaiusercontent.com`而非您的实际域名。

此行为破坏了多项核心功能：

1. **静态资源加载失效**：Next.js会向沙箱域名而非您的实际域名请求`/_next/static/chunks/app.js`，导致返回404错误
2. **相对URL解析错误**：使用相对路径（如`/logo.png`或`/api/data`）的图片、字体和API调用会针对沙箱域名而非您的应用域名进行解析
3. **浏览器历史记录存储完整URL**：`history.pushState`保存的是`https://your-app.vercel.app/about`，这会暴露您的真实域名并破坏沙箱安全模型
4. **客户端导航失败**：点击Next.js Link时，fetch请求会发送到沙箱域名而非您的服务器
5. **CORS阻止React服务器组件**：从iframe到您服务器的跨域请求因缺少适当头部而失败。浏览器还会发送Next.js默认不处理的`OPTIONS`预检请求
6. **父框架修改您的HTML**：ChatGPT偶尔会向您的根`<html>`元素添加属性，导致React水合不匹配
7. **外部链接在iframe内导航**：指向外部站点的链接会尝试在受限的iframe内加载，而非在用户浏览器中打开

## 使静态资源从您的实际域名加载

Next.js生成的资源路径如`/_next/static/chunks/app-pages-browser.js`。在ChatGPT iframe中运行时，这些请求会解析到`https://web-sandbox.oaiusercontent.com/_next/static/chunks/app-pages-browser.js`。但该URL并不存在。资源实际位于您的真实域名上。

Next.js为此场景提供了`assetPrefix`配置。将`assetPrefix`设置为您的应用URL可强制所有`/_next/`请求使用正确的源：

```
1import type { NextConfig } from "next";2import { baseURL } from "./baseUrl";3
4const nextConfig: NextConfig = {5  assetPrefix: baseURL, 6};7
8export default nextConfig;
```

`baseURL`根据环境动态确定：

```
1export const baseURL =2  process.env.NODE_ENV == "development"3    ? "<http://localhost:3000>"4    : "https://" +5      (process.env.VERCEL_ENV === "production"6        ? process.env.VERCEL_PROJECT_PRODUCTION_URL7        : process.env.VERCEL_BRANCH_URL || process.env.VERCEL_URL);
```

这确保了资源在开发、预览和生产环境中都能正确加载。

## 为所有相对路径设置基础URL

即使配置了`assetPrefix`，其他相对URL仍会失效。使用相对路径（如`/logo.png`、`/fonts/inter.woff2`或`/api/data`）的图片、字体和API调用都会解析到`https://web-sandbox.oaiusercontent.com`而非您的域名。Next.js没有内置方法重写应用中的所有相对URL。

HTML的`<base>`元素解决了这个问题。它为文档中的所有相对URL设置基础URL：

```
1function NextChatSDKBootstrap({ baseUrl }: { baseUrl: string }) {2  return (3    <>4      <base href={baseUrl}></base>5      {}6    </>7  );8}
```

通过这个单一元素，每个相对URL都会自动针对您的实际域名进行解析。这包括图片（`<img src="/logo.png">`）、样式表（`<link href="/styles.css">`）和API调用（`fetch('/api/data')`）。

## 修补浏览器历史记录以防止URL泄露

Next.js使用History API（`history.pushState`和`history.replaceState`）在客户端导航期间更新地址栏。然而，当在显示不同源的iframe内运行时，这些API会存储完整URL如`https://your-app.vercel.app/about`。这会暴露您的真实域名并破坏沙箱安全模型。

我们的解决方案是拦截对`history.pushState`和`history.replaceState`的调用，将其精简为仅包含路径、查询参数和哈希部分：

```
1const originalReplaceState = history.replaceState;2history.replaceState = (state, unused, url) => {3  const u = new URL(url ?? "", window.location.href);4  const href = u.pathname + u.search + u.hash;5  originalReplaceState.call(history, state, unused, href);6};7
8const originalPushState = history.pushState;9history.pushState = (state, unused, url) => {10  const u = new URL(url ?? "", window.location.href);11  const href = u.pathname + u.search + u.hash;12  originalPushState.call(history, state, unused, href);13};
```

这确保导航历史条目仅包含`/about`之类的相对路径而非完整URL，在保持浏览器历史记录和前进/后退按钮功能的同时，维护了iframe的安全边界。

## 重写客户端导航的fetch请求

Next.js 使用由 React Server Components（RSC）驱动的复杂客户端导航系统。当你点击一个 Link 时，Next.js 会发起一个 `fetch` 请求来加载新页面的 RSC 负载（一种包含组件树和数据的特殊格式）。

但请记住，你的应用运行在 `web-sandbox.oaiusercontent.com` 的 iframe 中，而服务器位于 `your-app.vercel.app`。如果不加干预，这些 `fetch` 请求会因为目标域名错误而失败。

我们的解决方案是修补 `window.fetch`，以重写同源请求，使其使用正确的基础 URL：

```

4if (isInIframe && window.location.origin !== appOrigin) {5  const originalFetch = window.fetch;6
7  window.fetch = (input: URL | RequestInfo, init?: RequestInit) => {8    9    let url = ;10

24    return originalFetch.call(window, input, init);25  };26}
```

此补丁仅在以下情况下运行：

1.  应用位于 iframe 内部（`window.self !== window.top`）
2.  iframe 的来源与应用的实际来源不同

## 为跨域请求添加 CORS 头

通过实施 fetch 修补，客户端导航现在会从 iframe 向你的服务器发起跨域请求。但是，如果没有正确的 CORS 头，这些请求将会失败。

现代浏览器在发送跨域 `POST` 请求（Next.js 用于 RSC 负载）之前，还会发送 `OPTIONS` 预检请求。如果不处理这些 `OPTIONS` 请求，导航就会挂起。

Next.js 中间件在每个请求被处理之前运行，这使其成为添加 CORS 头并解决此问题的理想位置：

```
1import { NextResponse } from "next/server";2import type { NextRequest } from "next/server";3
4export function middleware(request: NextRequest) {5  6  if (request.method === "OPTIONS") {7    const response = new NextResponse(null, { status: 204 });8    response.headers.set("Access-Control-Allow-Origin", "*");9    response.headers.set(10      "Access-Control-Allow-Methods",11      "GET,POST,PUT,DELETE,OPTIONS"12    );13    response.headers.set("Access-Control-Allow-Headers", "*");14    return response;15  }16
17  18  return NextResponse.next({19    headers: {20      "Access-Control-Allow-Origin": "*",21      "Access-Control-Allow-Methods": "GET,POST,PUT,DELETE,OPTIONS",22      "Access-Control-Allow-Headers": "*",23    },24  });25}26
27export const config = {28  matcher: "/:path*", 29};
```

此中间件：

1.  响应 OPTIONS 请求，返回 204 状态码和适当的 CORS 头
2.  向所有响应添加 CORS 头，使跨域 fetch 请求成功

响应 OPTIONS 请求，返回 204 状态码和适当的 CORS 头

向所有响应添加 CORS 头，使跨域 fetch 请求成功

现在，当 Next.js 在导航期间发起 RSC 请求时，浏览器的预检检查会通过，实际请求会收到所需的数据。

## 防止父框架干扰 DOM 变更

ChatGPT 的父框架偶尔会通过添加属性来修改根 `<html>` 元素。当 React 尝试进行水合时，服务器渲染的 HTML 与 DOM 不匹配，导致水合不匹配错误。

我们的解决方案是使用 `MutationObserver` 来监视根元素上的属性变化，并立即移除未经授权的修改：

```
1const htmlElement = document.documentElement;2const observer = new MutationObserver((mutations) => {3  mutations.forEach((mutation) => {4    if (5      mutation.type === "attributes" &&6      mutation.target === htmlElement7    ) {8      const attrName = mutation.attributeName;9      if (attrName && attrName !== "suppresshydrationwarning") {10        htmlElement.removeAttribute(attrName);11      }12    }13  });14});15
16observer.observe(htmlElement, {17  attributes: true,18  attributeOldValue: true,19});
```

我们还在 `<html>` 标签中添加了 `suppressHydrationWarning`，以防止 React 记录警告：

```
1<html lang="en" suppressHydrationWarning>
```

虽然这不是一个完美的解决方案（父框架不应该修改我们的 HTML），但它可以防止控制台被水合警告淹没。

## 在用户浏览器中打开外部链接

如果不加干预，指向外部站点的链接会尝试在受限的 iframe 内导航。当用户点击文档、博客文章或其他外部资源时，这会带来糟糕的用户体验。

ChatGPT 提供了一个 `openai.openExternal()` API 专门处理这种情况。我们的解决方案是拦截外部链接的点击，并使用此 API 在用户的浏览器中打开它们：

```
1window.addEventListener(2  "click",3  (e) => {4    const a = (e?.target as HTMLElement)?.closest("a");5    if (!a || !a.href) return;6
7    const url = new URL(a.href, window.location.href);8    if (9      url.origin !== window.location.origin &&10      url.origin !== appOrigin11    ) {12      try {13        if (window.openai) {14          window.openai.openExternal({ href: a.href });15          e.preventDefault();16        }17      } catch {18        console.warn("openExternal failed, likely not in OpenAI client");19      }20    }21  },22  true 23);
```

这确保了外部链接在用户的浏览器中打开，而不是在受限的 iframe 内导航。

## 使用 MCP 将你的 Next.js 应用连接到 ChatGPT

在完成所有浏览器 API 修补后，最后一步是实现将 ChatGPT 连接到你的 Next.js 应用的 MCP 服务器。MCP 服务器公开两个主要原语：资源（可以显示的内容）和工具（模型可以调用的操作）。

### MCP 资源如何向 ChatGPT 提供 HTML

资源提供 ChatGPT 在 iframe 中渲染的 HTML 内容。我们获取主页 HTML 并将其注册为资源：

```

```

`mimeType` 为 `text/html+skybridge` 告诉 ChatGPT 将其渲染为交互式小组件。`_meta` 对象包含 OpenAI 特定的配置，如小组件描述和是否显示边框。

### MCP 工具如何触发应用显示

工具是 ChatGPT 可以调用的操作。通过 `openai/outputTemplate` 将工具链接到资源，ChatGPT 就知道在调用工具后渲染小组件：

## 您的应用如何从 ChatGPT 接收数据

应用可以通过 `window.openai.toolOutput` 属性接收来自工具调用的数据。ChatGPT 会自动用工具响应中的 `structuredContent` 填充此对象：

```
1const [name, setName] = useState<string | null>(null);2
3useEffect(() => {4  if (typeof window === "undefined") return;5
6  if (!window.openai) {7    (window as any).openai = {};8  }9

12  Object.defineProperty((window as any).openai, "toolOutput", {13    get() {14      return currentValue;15    },16    set(newValue: any) {17      currentValue = newValue;18      if (newValue?.name) {19        setName(newValue.name);20      }21    },22    configurable: true,23    enumerable: true,24  });25
26  if (currentValue?.name) {27    setName(currentValue.name);28  }29}, []);
```

这种响应式模式允许应用在 ChatGPT 调用工具时更新其 UI。属性设置器会拦截对 `toolOutput` 的更改，并相应地更新 React 状态。

## 此方法带来的优势

与简单的 iframe 嵌入相比，此方法提供了几个显著优势。

如果没有这些补丁，您将仅限于提供单个静态页面或添加另一个 iframe 层，这会引入更多的复杂性。

- 用于表单的服务器操作

- API 路由
