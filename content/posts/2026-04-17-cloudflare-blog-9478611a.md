# 推出Flagship：为AI时代打造的功能开关

2026-04-17

- Rohan Mukherjee
- Abhishek Kankani

![](/images/posts/c200b107df4d.png)

AI正在编写比以往任何时候都多的代码。如今，AI辅助贡献在平台新代码中所占的份额正在快速增长。像OpenCode和Claude Code这样的Agent（智能体）式编码工具，可以在几分钟内交付完整的功能。

AI生成的代码进入生产环境只会加速。但更大的转变不仅仅是速度——而是自主性。

今天，一个AI Agent（智能体）编写代码，然后由人类审查、合并和部署。明天，Agent（智能体）将自行完成所有这些工作。问题变成了：如何在保留所有安全网的情况下，让Agent（智能体）向生产环境发布代码？

功能开关就是答案。Agent（智能体）在开关后编写一条新的代码路径并部署它——开关是关闭的，因此对用户来说没有任何变化。然后，Agent（智能体）为自己或一个小的测试群组启用开关，在生产环境中运行该功能，并观察结果。如果指标看起来不错，它就扩大推广范围。如果出现问题，它就禁用开关。人类不需要参与每一步——他们设定边界，而开关控制着影响范围。

这就是功能开关一直致力于实现的工作流程：不仅将部署与发布解耦，而且将人类的注意力与发布过程的每个阶段解耦。Agent（智能体）可以快速行动，因为开关使其快速行动变得安全。

今天，我们宣布推出Flagship——Cloudflare的原生功能开关服务，它基于OpenFeature构建，这是CNCF用于功能开关评估的开放标准。它可以在任何地方工作——Workers、Node.js、Bun、Deno和浏览器——但在Workers上速度最快，因为开关是在Cloudflare网络内评估的。使用Flagship绑定和OpenFeature，集成看起来像这样：

```TypeScript
await OpenFeature.setProviderAndWait(
    new FlagshipServerProvider({ binding: env.FLAGS })
);
```

Flagship现已进入封闭测试阶段。

### Workers上功能开关的问题

许多Cloudflare开发者采用了务实的变通方法：直接将开关逻辑硬编码到他们的Workers中。老实说，这在初期效果足够好。Workers可以在几秒钟内部署，因此在代码中切换一个布尔值并将其推送到生产环境，对于大多数情况来说已经足够快了。

但这并不会一直保持简单。一个硬编码的开关会变成十个。十个会变成五十个，由不同的团队拥有，没有中心视图来查看哪些是开启或关闭的。没有审计追踪——当出现问题时，你只能搜索`git blame`来弄清楚谁切换了什么。

#### 对外部服务的网络调用

在Workers上使用的另一种常见模式是向外部服务发出HTTP请求，如下所示：

```TypeScript
const response = await fetch("https://flags.example-service.com/v1/evaluate", {
      ...
      body: JSON.stringify({
        flagKey: "new-checkout-flow",
        context: {
          ...
        },
      }),
    });
const { value } = await response.json();
if (value === true) {
    return handleNewCheckout(request);
}
return handleLegacyCheckout(request);
```

那个出站请求位于每个用户请求的关键路径上。根据用户距离开关服务区域的远近，它可能会增加相当大的延迟。

这是一个奇怪的情况。你的应用程序在边缘运行，距离用户只有几毫秒。但功能开关检查迫使它在决定呈现什么之前，必须通过互联网访问另一个API。

#### 为什么本地评估不能解决问题

一些功能开关服务提供“本地评估”SDK。SDK不是在每个请求上都调用远程API，而是将完整的开关规则集下载到内存中并在本地进行评估。每次评估没有出站请求，开关决策在进程内完成。

在Workers上，这些假设都不成立。没有长生命周期的进程：一个Worker isolate可以被创建、服务一个请求，并在一个请求和下一个请求之间被驱逐。一次新的调用可能意味着需要从头重新初始化SDK。

在无服务器平台上，你需要一个已经位于边缘的分发原语，一个为你管理缓存、读取是本地化的、并且你不需要持久连接来保持更新的原语。

Cloudflare KV正是这样一个绝佳的原语！

### Flagship的工作原理

Flagship完全构建在Cloudflare的基础设施之上——Workers、Durable Objects和KV。在评估路径中没有外部数据库、没有第三方服务、也没有集中化的源服务器。

当你创建或更新一个开关时，控制平面会原子性地将更改写入一个Durable Object——这是一个基于SQLite的、全局唯一的实例，作为该应用开关配置和变更日志的单一事实来源。在几秒钟内，更新后的开关配置会同步到Workers KV，这是Cloudflare的全局分布式键值存储，它在Cloudflare网络中复制。

当请求评估一个开关时，Flagship直接从边缘的KV读取开关配置——也就是已经在处理该请求的同一个Cloudflare位置。然后，评估引擎就在一个isolate中运行：它将请求上下文与开关的目标规则进行匹配，解析推广百分比，并返回一个变体。数据和逻辑都位于边缘——没有任何内容被发送到其他地方进行评估。

### 使用Flagship：Worker绑定

对于运行Cloudflare Workers的团队，Flagship提供了一个直接绑定，可以在Workers运行时内部评估开关——无需HTTP往返，没有SDK开销。将绑定添加到你的`wrangler.jsonc`中，你的Worker就连接上了：

```JSON
{
  "flagship": [
    {
      "binding": "FLAGS",
      "app_id": "<APP_ID>"
    }
  ]
}
```

就这样。你的账户ID是从你的Cloudflare账户推断出来的，而`app_id`将绑定与特定的Flagship应用关联起来。在你的Worker中，你只需要请求一个开关值：

```TypeScript
export default {
  async fetch(request: Request, env: Env) {
    // 简单的布尔值检查
    const showNewUI = await env.FLAGS.getBooleanValue('new-ui', false, {
      userId: 'user-42',
      plan: 'enterprise',
    });
    // 需要时获取完整的评估详情
    const details = await env.FLAGS.getStringDetails('checkout-flow', 'v1', {
      userId: 'user-42',
    });
    // details.value = "v2", details.variant = "new", details.reason = "TARGETING_MATCH"
  },
};
```

该绑定支持每种变体类型的类型化访问器——`getBooleanValue()`、`getStringValue()`、`getNumberValue()`、`getObjectValue()`——以及返回解析值、匹配的变体和选择原因的`*Details()`变体。在评估错误时，会优雅地返回默认值。在类型不匹配时，绑定会抛出异常——这是你代码中的错误，而不是暂时性故障。

### SDK：原生支持OpenFeature

大多数功能开关SDK都带有自己的接口和评估模式。随着时间的推移，这些会深深嵌入到你的代码库中——更换提供商意味着重写每个调用点。

我们不想再构建一个这样的SDK。Flagship基于OpenFeature构建，这是CNCF用于功能开关评估的开放标准。OpenFeature为跨语言和提供商的功能开关评估定义了一个通用接口——它与OpenTelemetry在可观测性方面的关系相同。你只需针对该标准编写一次评估代码，并通过更改一行配置来交换提供商。

```TypeScript
import { OpenFeature } from '@openfeature/server-sdk';
import { FlagshipServerProvider } from '@cloudflare/flagship/server';
await OpenFeature.setProviderAndWait(
  new FlagshipServerProvider({
    appId: 'your-app-id',
    accountId: 'your-account-id',
    authToken: 'your-cloudflare-api-token',
  })
);
const client = OpenFeature.getClient();
const showNewCheckout = await client.getBooleanValue(
  'new-checkout-flow',
  false,
  {
    targetingKey: 'user-42',
    plan: 'enterprise',
    country: 'US',
  }
);
```

如果您在带有 Flagship 绑定的 Workers 上运行，可以直接将其传递给 OpenFeature 提供者。该绑定已包含您的账户上下文，因此无需配置任何内容——身份验证是隐式的。

```TypeScript
import { OpenFeature } from '@openfeature/server-sdk';
import { FlagshipProvider } from '@cloudflare/flagship/server';
let initialized = false;
export default {
  async fetch(request: Request, env: Env) {
    if (!initialized) {
      await OpenFeature.setProviderAndWait(
        new FlagshipServerProvider({ binding: env.FLAGS })
      );
      initialized = true;
    }
    const client = OpenFeature.getClient();
    const showNewCheckout = await client.getBooleanValue('new-checkout-flow', false, {
      targetingKey: 'user-42',
      plan: 'enterprise',
    });
  },
};
```

您的评估代码无需更改——OpenFeature 接口是相同的。但在底层，Flagship 通过绑定而非 HTTP 来评估功能标志。您既获得了标准的可移植性，又拥有了绑定的高性能。

我们还提供了适用于浏览器的客户端提供者。它会预取您指定的标志，以可配置的 TTL 进行缓存，并同步地从该缓存提供评估结果。

### 您可以使用 Flagship 做什么

Flagship 支持您期望从功能标志服务中获得的所有模式，以及当 AI 生成的代码每日投入生产时变得至关重要的那些模式。

标志值可以是布尔值、字符串、数字或完整的 JSON 对象——这对于配置块、UI 主题定义或在不维护单独代码路径的情况下将用户路由到不同的 API 版本非常有用。

#### 定向规则

每个标志可以包含多个规则，按优先级顺序评估。第一个匹配的规则生效。

一个规则包含：

- 用于确定规则是否适用于给定上下文的条件
- 规则匹配时要提供的标志变体
- 用于基于百分比交付的可选渐进式发布
- 当存在多个规则时，决定评估顺序的优先级（数字越小，优先级越高）

规则匹配时要提供的标志变体

#### 嵌套逻辑条件

条件可以使用 AND/OR 逻辑组合，最多可嵌套五层。单个规则可以表达如下内容：

```Rust
(plan == “enterprise” AND region == “us” ) OR (user.email.endsWith(“@cloudflare.com”))
= serve (“premium”)
```

在规则的顶层，多个条件通过隐式的 AND 组合，所有条件都必须通过，规则才能匹配。在每个条件内部，您可以嵌套 AND/OR 组以实现更复杂的逻辑。

#### 按百分比进行标志发布

与在不同上传版本的 Worker 之间分流流量的渐进式部署不同，功能标志允许您在服务于 100% 流量的单一版本内，按百分比逐步发布新行为。

任何规则都可以包含百分比发布。您不是向所有匹配条件的用户提供变体，而是只向其中一定百分比的用户提供。

发布使用指定上下文属性的一致性哈希。相同的属性值（例如 userId）总是哈希到同一个桶，因此它们在请求之间不会在不同变体之间切换。您可以从 5% 到 10% 到 50% 再到 100% 的用户逐步扩大发布范围，这样已经处于发布中的用户会继续保持在其中。

### 为未来而构建

AI 生成的代码进入生产环境只会加速。Agent（智能体）驱动的工作流将更进一步——那些能在生产中自主部署、测试和迭代代码的 Agent（智能体）。在这个世界中蓬勃发展的团队，将不是那些发布最快的团队，而是那些能够快速发布、同时仍能控制用户所见内容、在出现问题时几秒钟内回滚、并有信心逐步开放新代码路径的团队。

这就是 Flagship 的构建目标：

- 跨地球区域进行评估，使用 K/V 进行全局缓存。
- 完整的审计追踪。每次标志更改都会记录字段级别的差异，因此您知道谁在何时更改了什么。
- 仪表板集成。团队中的任何人都可以切换标志或调整发布，而无需接触代码。
- OpenFeature 兼容性。无需重写评估代码即可采用 Flagship。离开时也无需重写。
