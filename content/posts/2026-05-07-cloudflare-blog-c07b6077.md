2026年4月29日，一个名为“Copy Fail”（CVE-2026-31431）的Linux内核本地权限提升漏洞被公开披露。Cloudflare的安全与工程团队在漏洞披露后立即开始评估。我们审查了利用技术，评估了基础设施中的暴露面，并验证了现有的行为检测机制能够在数分钟内识别出该利用模式。

Cloudflare环境未受影响，客户数据未面临风险，任何服务均未中断。请继续阅读，了解我们的准备工作如何发挥作用。

## 背景

#### 我们的Linux内核发布流程

Cloudflare以巨大规模运营全球Linux服务器基础设施，数据中心遍布330个城市。我们基于社区长期支持（LTS）版本维护自定义Linux内核构建，以便有效管理如此规模下的更新。在任何给定时间，我们可能使用来自不同系列的多个LTS版本（如6.12或6.18），这些版本受益于更长的更新周期。

社区定期合并并发布安全与稳定性更新，这大约每周触发一次自动化任务，生成新的内部内核构建。这些构建会在我们的预发布数据中心进行稳定性测试，然后进行全球部署。成功发布后，边缘重启发布（ERR）管道会按照四周周期对边缘基础设施进行系统化更新和重启。我们的控制平面基础设施通常采用最新内核，并根据具体工作负载需求安排重启。

当CVE公开时，相关修复通常已集成到稳定的Linux LTS版本中数周。我们的既定流程确保已部署这些补丁。

在“Copy Fail”披露时，我们大部分基础设施运行6.12 LTS版本，部分机器已开始过渡到更新的6.18 LTS版本。

### 关于Copy Fail漏洞

在了解响应过程之前，先理解该漏洞会有所帮助。完整的分析可在原始Xint Code披露文章中找到。

#### AF_ALG与内核加密API

Linux内核内部加密API管理kTLS和IPsec等功能。用户空间程序通过AF_ALG套接字家族访问该API，允许非特权进程请求加密或解密。algif_aead模块为带关联数据的认证加密（AEAD）密码提供支持。

非特权程序遵循以下步骤：

1. 打开AF_ALG套接字并绑定到AEAD模板。
2. 设置密钥并接受请求套接字。
3. 通过sendmsg()或splice()提交输入。
4. 使用recvmsg()执行操作。

splice()系统调用在此至关重要，因为它通过传递页缓存引用来移动数据。

#### 内存机制：页缓存与原地加密

页缓存是文件内容的共享系统缓存。修改属于setuid二进制文件的页面实际上会为所有用户编辑该程序，直到该页面被驱逐。

加密API使用散列表，这些是链接各种内存页面的结构。2017年，algif_aead针对原地操作进行了优化，将目标页和引用页链接在一起。这种设计缺乏防止算法写入超出预期边界的机制。

#### 漏洞：越界写入

当用户执行recvmsg()时，内核中的authencesn包装器会在合法输出区域之后执行4字节写入：

```c++
scatterwalk_map_and_copy(tmp + 1, dst, assoclen + cryptlen, 4, 1);

```

通过使用splice()，攻击者可以将目标文件的页缓存页面链接到散列表。越界写入随后污染缓存文件，使攻击者能够控制修改哪个文件、偏移量以及写入的具体4字节。这意味着攻击者可以利用该漏洞操纵以下内容：

- 文件：任何可读文件。
- 偏移量：可通过assoclen和splice参数调整。
- 值：通过sendmsg()中的AAD字节4-7控制。

#### 利用步骤详解

默认利用目标为/usr/bin/su，这是一个几乎存在于所有发行版上的setuid-root二进制文件。

1. 缓存引用：以O_RDONLY打开/usr/bin/su并执行read()以填充页缓存。使用splice()在文件描述符上将这些页缓存引用传递到加密散列表。
2. 设置：创建AF_ALG套接字，绑定到authencesn(hmac(sha256),cbc(aes))，设置密钥，并接受请求套接字，无需特权。
3. 写入构造：对于每个4字节shellcode块：使用包含shellcode的AAD字节4-7调用sendmsg()。将二进制文件splice()到管道，然后到AF_ALG套接字，使assoclen + cryptlen指向所需的.text偏移量。
4. 触发：recvmsg()启动解密。authencesn将其临时数据写入页缓存中/usr/bin/su的目标偏移量。尽管函数返回-EBADMSG，但4字节写入已存在于全局页缓存中。
5. 执行：运行execve("/usr/bin/su")加载被污染的页缓存。由于该二进制文件是setuid-root，注入的shellcode以root权限执行。

### 我们如何缓解？

由于部署修补后的Linux内核需要较长时间，我们还探索了无需重启即可缓解此漏洞的方法。

#### 移除模块

漏洞位于`algif_aead`内核模块中。因此，简单的修复方法是直接移除该模块并禁止其重新加载。

这一缓解措施正是发现该漏洞的安全研究人员在Copy Fail报告中推荐的方案。

```shell
echo "install algif_aead /bin/false" > /etc/modprobe.d/disable-algif.conf
rmmod algif_aead 2>/dev/null || true
```

不幸的是，移除模块会影响依赖内核加密API的软件。这意味着我们需要设计更精准的缓解方案。

#### Bpf-lsm

我们已为此类场景开发并部署了工具：`bpf-lsm`。该工具不移除模块，而是将其保留给合法用户使用，并通过BPF Linux安全模块程序拒绝其他所有用户的`socket_bind`LSM钩子。这从根本上阻断了任何利用的入口。

eBPF程序的草稿在一夜之间完成。团队成员次日接手，进行验证并使其达到生产就绪状态。该程序相当直接。每次`socket_bind`调用时：

1. 如果套接字族不是`AF_ALG`，则允许调用通过。
2. 如果族是`AF_ALG`，则检查调用二进制文件的路径是否在已知合法用户的白名单中。
3. 如果二进制文件在白名单中，允许绑定；否则拒绝。

如果套接字族不是`AF_ALG`，则允许调用通过。

如果族是`AF_ALG`，则检查调用二进制文件的路径是否在已知合法用户的白名单中。

如果二进制文件在白名单中，允许绑定；否则拒绝。

为验证某台机器上的缓解措施而不实际利用漏洞，Copy Fail报告提供了一行命令：

```shell
python3 -c 'import socket; s = socket.socket(socket.AF_ALG, socket.SOCK_SEQPACKET, 0); s.bind(("aead","authencesn(hmac(sha256),cbc(aes))"));'
```

在已缓解的机器上，你会收到PermissionError: [Errno 1] Operation not permitted（或FileNotFoundError，取决于激活的缓解措施），而非成功绑定。

#### 逐步部署

在启用强制策略前，我们验证了已知内部服务是唯一的合法`AF_ALG`用户，以避免意外中断。我们使用`prometheus-ebpf-exporter`钩住`socket()`系统调用，追踪全平台每个二进制文件的`AF_ALG`使用情况。这无需内核更改，并在数小时内提供了来自数十万台服务器的聚合数据。结果确认，识别出的服务确实是唯一的合法用户。

1. **先获取可见性**。通过salt推送ebpf-exporter配置。在指标层确认已知服务实际上是唯一创建`AF_ALG`套接字的实体。

## 结论

"Copy Fail" 漏洞给我们带来了独特的挑战。尽管我们每两周部署一次 Linux 补丁更新，但由于一个月前的主线修复尚未反向移植到我们的主要内核版本，我们仍然存在漏洞。尽管如此，我们仍能在反向移植发布后的数小时内推出修补后的内核。在此期间，bpf-lsm 提供了一种无需重启的精准缓解措施，保护了我们的服务器集群。虽然我们最初禁用问题模块的尝试失败了，但这一失败安全地发生在我们的内部预发布环境中，而非生产环境，从而让我们发现了这一依赖关系。

到部署完成时，我们集群中的每台机器都受到修补后的内核或 bpf-lsm 程序的保护，后者拒绝非白名单二进制文件访问易受攻击的代码路径。在此事件过程中，客户未受到任何影响，并且我们已承诺开展上述后续工作，以便在未来类似情况发生时，能够更快响应并提高可见性。负责任的披露行之有效，内核内可见性工具在此类时刻发挥了巨大价值，而 bpf-lsm 仍然是我们拥有的用于运行时内核缓解的最有用的工具之一。
