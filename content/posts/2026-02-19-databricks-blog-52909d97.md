## 引言

数据民主化的第一步是让获取洞察变得简单且安全。借助 Databricks Genie，用户现在可以直接从他们日常使用的工具（如 Teams、Slack、Confluence 或自定义 Web 应用）中与数据对话。无论您使用我们原生的 Copilot Studio/Foundry 集成，还是通过 Genie Conversation API/SDK 进行构建，Genie 现在都能将自然语言分析带入日常工作流。在幕后，可以应用 OAuth 来安全地验证每个用户的身份并强制执行数据访问权限。

此前，我们看到像 The AA 和 Casas Bahia 这样的客户独立地将他们自己的 Genie 集成构建到 Microsoft Teams 和内部应用中。我们强大的可扩展性套件现在使这种体验更简单、更快速且更具可扩展性。

在本博客中，我们将介绍两种在组织内通过企业 OAuth 部署 Genie 的常见方法：

*   通过我们的 Copilot Studio 集成将 Genie 引入 Microsoft Teams
*   使用 Genie Conversation API 将 Genie 嵌入到您的自定义 Web 应用中

## 将 Genie 引入 Microsoft Teams

在群组对话中，临时性的数据问题时常出现。借助 Databricks Genie 的原生 Copilot Studio 集成，您的用户现在可以在 Microsoft Teams 中直接提问时即刻获得答案。要利用此集成，请按照以下步骤操作：

### 先决条件

*   确保您有一个目标 Genie 空间，该空间已根据我们的最佳实践进行整理，以提供最高质量的结果。
*   最终用户/服务主体必须拥有对目标 Genie 空间的访问权限（至少为“可以查看”）、对该空间中 Unity Catalog 数据的 SELECT 权限，以及对该空间中 SQL 计算的“可以使用”权限。可选地为最终用户分配“消费者访问”权利，以获得简化的“只读”体验。

### 步骤 1：将 Azure Databricks 连接到 Power Platform

在 Microsoft Teams 中启用 Genie 的第一步是将 Azure Databricks 连接到 Power Platform（文档）。在您的 Microsoft Power Apps 中，点击“连接”，然后选择 Azure Databricks（如果您使用 AWS/GCP，则选择 Databricks）。配置以下字段：

*   为确保每个最终用户使用自己的身份验证登录 Databricks，请选择 OAuth 作为“身份验证类型”。
*   对于“服务器主机名”和“HTTP 路径”，请转到包含您目标 Genie 空间的工作区。选择一个 SQL 仓库并打开“连接详细信息”以获取此信息（不需要与连接到您的 Genie 空间的 SQL 仓库是同一个）。

### 步骤 2：将 Genie 空间连接到您的 Copilot Studio Agent

接下来，您需要将您的 Genie 空间连接到 Copilot Studio（文档）。我们的集成处理了所有 API 和 MCP 逻辑，因此只需点击几下即可完成连接。

在 Copilot Studio 中，点击“Agents”。选择“Create blank agent”为 Genie 空间创建一个新的独立 Agent。如果您想将 Genie 引入现有的 Agent 框架，也可以选择一个现有的 Copilot Studio Agent 并将您的 Genie 空间添加到其中。

在您的新 Agent 中，点击“Tools”，然后点击“Add a tool”。在 MCP 部分下选择 Azure Databricks Genie（对于 AWS/GCP，选择 Databricks Genie）。

现在，您可以选择所需的 Genie 空间并配置连接详细信息：

*   **要使用的凭据**：选择“最终用户凭据”以确保每个应用程序用户都将使用自己的身份和数据访问权限登录。这确保了如果应用程序用户无权访问 Genie 空间或其中的表，他们将无法从 Genie 检索数据洞察。
*   如果您希望最终用户使用单个共享身份（服务主体——推荐——或您自己的身份）进行身份验证，请选择“Maker provided credentials”。
*   **重要提示**：确保您的目标 Genie 空间具有清晰的标题和描述，概述其上下文、关键概念和限制。这将帮助您的 Copilot Studio Agent 有效地编排请求。

### 步骤 3：启用连接参数共享

当您选择“最终用户凭据”时，每个人都必须使用自己的帐户登录 Databricks。为了简化此过程，我们建议共享连接参数（如 Microsoft 文档中所述），这样用户就不必自己提供这些信息。实际上，这仅意味着提供服务器主机名和 HTTP 路径，以确保他们验证到与 Copilot Studio Agent 中连接的 Genie 空间相关联的确切 Databricks 工作区。

*   打开您的 Copilot Studio Agent 的“Settings”页面。
*   打开“Connection Settings”并确保 Azure Databricks 显示为“Connected”状态。
*   接下来点击“See Details”，并在“Connection parameters”选项卡中允许共享参数的权限。

### 步骤 4：将您的 Agent 引入 Teams

现在您已经拥有了一个连接到 Genie 空间的 Copilot Studio Agent，您可以将其发布到 Teams。

*   确保您的 Agent 具有清晰的“名称”和“描述”。
*   我们还建议：
    *   选择一个推理模型（例如 GPT-5 Reasoning, Claude Sonnet 4.5）以有效轮询和使用 Genie。
    *   添加自定义 Agent 指令以定制体验（例如，答案格式和延迟偏好）。
*   查看您的 Copilot Studio Agent 后，点击“Publish”。然后在“Channels”中，选择 Teams 作为您所需的渠道。

一切就绪！Genie 现已上线 Microsoft Teams，可在问题出现时即时提供受管控的数据洞察。

要了解最终用户如何在 Microsoft Teams 中利用 Genie，请参阅我们的客户案例。

## 将 Genie 引入自定义 Web 应用

许多组织还希望将 Genie 直接嵌入到他们的自定义 Web 应用中，以便用户可以在他们已使用的工具中提问——例如，门店经理可以直接在他们现有的销售终端中询问有关库存的临时问题。借助 Genie Conversation API 和 Databricks OAuth，这现在成为可能。

在构建 Web 应用与 Genie 之间的集成之前，重要的是决定您将使用哪种 OAuth 模式：用户到机器（U2M）、机器到机器（M2M）或代理（OBO）模型。每种方法对应不同类型的应用程序用例：

*   **用户到机器（U2M）** - 当每个最终用户都需要受管控的、个性化的数据访问时最佳。在此模型中，用户使用其企业身份（例如 SSO）登录，Genie 收到特定于用户的 OAuth 令牌，查询使用该用户的权限运行。
    *   **示例用例**：一个销售 Copilot，销售代表与一个底层 Genie 空间聊天，并且应该只看到来自他们自己交易的数据洞察。
*   **机器到机器（M2M）** - 适用于希望所有用户拥有相同数据访问权限和更简单治理的用例。此模型允许服务主体进行身份验证并向 Genie 颁发关联的 OAuth 令牌，然后使用该令牌在服务主体的权限下运行查询。
    *   **示例用例**：一个“公司 KPI”聊天机器人，任何员工都可以询问公司范围的 KPI 指标并获得相同的共享洞察。
*   **代理（OBO）** - 适用于需要按用户进行数据治理但位于集中后端之后的应用程序。在此模型中，您的应用程序将首先验证到 Databricks，然后“代表”最终用户调用 Genie API，并应用其数据权限。
    *   **示例用例**：一个财务分析门户，用户与一个利用 Genie 的统一聊天机器人对话，每个用户只能看到他们有权访问的数据。

在本博客的其余部分，我们将重点介绍与 Genie 集成的第一种模式：使用 Databricks 内置 OAuth 支持的 **OAuth U2M 流程**。

**注意**：Databricks 还支持 OAuth 令牌联合，您可以使用它来引入由您自己的身份提供商颁发的令牌，并将其与上述任何用于访问 Genie 的方法结合使用。

- 确保您拥有一个按照**最佳实践**精心配置的目标Genie空间，以提供最高质量的服务。
- 最终用户/服务主体必须拥有对目标Genie空间的访问权限（至少为**可查看**），对该空间Unity Catalog数据的**SELECT**权限，以及对该空间SQL计算的**可使用**权限。可选地，可以为最终用户分配**消费者访问**权限，以获得简化的“只读”体验。

### 步骤 1：注册 OAuth 应用程序

要将您的自定义 Web 应用程序安全地连接到 Genie，首先需要在您的 Databricks 账户中注册它。此步骤允许 Databricks 在后续步骤中为您的应用程序安全地颁发用户范围的令牌。请查阅**产品文档**以了解更多信息。

在 Databricks 账户控制台中，添加一个新的 OAuth 连接并配置以下信息：

- **应用程序名称**：在用户登录时显示给用户的、易于理解的名称。
- **重定向 URL**：一个或多个 Databricks 在认证后允许将用户发送到的 URL。这些 URL 必须与您的应用程序在后续步骤中使用的 URL 完全匹配。
- **访问范围**：授予**所有 API**的访问权限，以便您的应用程序可以代表用户调用 Genie 对话 API。

保存此连接后，Databricks 将生成以下信息：

- **客户端 ID**：您应用程序的公共标识符。
- **客户端密钥**：用于后端的私有凭证。

请在后端安全地存储这些**凭证**——它们将用于将授权码交换为访问令牌，并对调用 Genie 对话 API 的请求进行身份验证。

### 步骤 2：引导用户到 Databricks 进行身份验证并授予访问权限

下一步是确保您的应用程序引导最终用户前往 Databricks，以便他们能够**登录并批准您的应用程序**代表他们与 Genie 对话。成功登录并批准后，Databricks 将把用户重定向回您的应用程序，并附带一个短期有效的授权码。

此授权码证明了用户已成功通过 Databricks 身份验证，并且用户已批准您的应用程序所请求的访问权限。您的应用程序后端将在下一步中使用此授权码来获取访问令牌。

首先，为每次登录生成 **PKCE** 和状态值，以保护您的 Web 应用程序：

- 根据 **OAuth PKCE 标准**，使用 SHA-256 和 Base64 URL 编码生成一个 `code_verifier` 和一个匹配的 `code_challenge`。此步骤可防止授权码被盗和重复使用（请参阅**文档**中的代码示例）。
- 创建一个随机的 `state` 字符串，并确保将其存储在 Cookie 或会话中。这确保了授权码是为真实的最终用户会话生成的。

接下来，您的前端应使用 **Databricks OAuth 端点**构建授权 URL：

包含以下表单参数，以便向用户标识您的应用程序：

- `<databricks-instance>`：包含工作区**实例名称**的您的 Databricks 实例（例如 dbc-a1b2345c-d6e7.cloud.databricks.com）
- `<client_id>`：上一步中注册的 OAuth 应用程序的客户端 ID
- `<redirect-url>`：与上一步中指定的重定向 URL 相同
- `<state>`：用于验证响应的任何纯文本字符串
- `<code-challenge>`：从 `code_verifier` 派生的 PKCE 代码质询

用户登录其 Databricks 账户后，将被重定向到带有查询参数的 `redirect_url`：`https://<redirect_url>/oauth/callback?code=<authorization_code>&state=<state>`

您的回调处理程序应从查询字符串中读取 `authorization_code` 和 `state`。验证 `state` 值是否与存储在 Cookie 或 Web 会话中的值匹配。如果不匹配，则丢弃 `authorization_code`。获得返回的 `authorization_code` 后，您的应用程序现在可以将其交换为访问令牌。
