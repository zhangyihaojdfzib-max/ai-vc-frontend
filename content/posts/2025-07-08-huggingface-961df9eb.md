# 支撑Hugging Face生产基础设施的三大关键告警

Hugging Face基础设施团队很高兴能分享我们生产系统内部运作的幕后故事，我们有幸参与其构建与维护工作。我们团队致力于设计和实施稳健的监控告警系统，这对保障平台稳定性与可扩展性至关重要。我们时刻感受到这些告警在潜在问题演变成重大事故前，帮助我们识别和响应的关键作用。

本文将深入探讨三大关键告警的细节，它们在生产基础设施中发挥着独特作用，并阐述其如何帮助我们维持社区所依赖的高性能与高可用性。

### NAT网关高吞吐量告警

在云计算架构中，当数据在私有网络与公共网络间流动时，部署NAT（网络地址转换）网关是一项稳固的最佳实践。该网关作为战略守门员，监控并引导所有流向公共互联网的出站流量。通过集中出口流量，NAT网关为全面可视化提供了战略制高点。我们的团队可以轻松查询分析此流量，使其成为处理安全、成本优化或各类调查场景时的宝贵资产。

成本优化是云基础设施管理的关键环节，理解定价机制至关重要。数据中心定价结构通常区分东西向流量（通常指同一机架或建筑内的通信）和南北向流量（较远私有网络间或与互联网的通信）。通过监控网络流量，Hugging Face能深入洞察这些流量模式。这种认知使我们能就基础设施配置与架构做出明智决策，确保避免不必要的成本支出。

我们的核心告警之一，是在网络流量超过预设阈值时触发通知。该告警具有多重作用：首先，它作为早期预警系统，提醒我们任何可能预示潜在问题或异常行为的流量突增；其次，它促使我们定期审视流量趋势，确保紧跟基础设施的增长与演变需求。此告警采用静态阈值，并经过持续微调以保证其持续有效。触发时，常与基础设施重构阶段重合。

例如，在集成第三方安全与自动扩缩工具时，我们观察到节点遥测数据出口增加，触发告警并促使我们优化配置。

另一次，基础设施调整导致我们意外避开了产品特定基础设施间的私有低成本路径（例如，从Space发往Hub以操作仓库数据的流量）。进一步说明，我们发现最具成本节约潜力的工作负载是访问对象存储的请求。对于我们的LFS仓库存储，直接获取对象比通过CDN托管资源更便宜，且相较于到达我们前端的公共请求，无需Web应用防火墙提供的同等安全措施。利用DNS覆盖在私有网络路径和公共网络路径间切换流量，已成为我们由CDKTF AWS驱动的重要技术手段。

### Hub请求日志归档成功率告警

Hugging Face的日志基础设施是一个复杂系统，旨在收集、处理和存储应用程序与服务产生的大量日志数据。该系统的核心是Hub应用日志管道，这是一个架构精良的解决方案，确保Hub模型使用数据能被高效捕获、丰富并存储，用于报告和归档。管道始于Filebeat——一个作为守护进程集运行在每个Kubernetes集群应用Pod旁的轻量级日志采集器。Filebeat负责从包括应用容器在内的多种源收集日志，并将其转发至管道下一阶段。

日志被Filebeat收集后，会发送至强大的日志处理工具Logstash。Logstash作为数据处理主力，对传入日志应用一系列变更与转换操作，包括：添加GeoIP数据以获取地理位置洞察、根据预设条件将日志路由至特定Elasticsearch索引、通过增删或重格式化日志字段确保一致性与易分析性。Logstash处理完成后，日志被转发至Elasticsearch集群。

分布式搜索分析引擎Elasticsearch构成我们日志存储与分析平台的核心。它接收来自Logstash的日志，并通过Elasticsearch管道应用自有的处理规则。这些管道执行最小化处理任务，例如添加时间戳字段以记录处理时间——这对日志分析与关联至关重要。Elasticsearch提供可扩展的灵活存储方案，使我们能够缓冲日志供运营使用和实时分析。

为管理Elasticsearch内日志的生命周期，我们采用稳健的存储与生命周期管理策略。这确保日志在Elasticsearch中保留指定时长，为运营和故障排查提供快速访问。保留期结束后，日志将卸载至长期归档存储。归档过程由自动化工具执行，该工具从Elasticsearch索引读取日志，将其格式化为高效的列式存储格式Parquet文件，并写入对象存储系统。

日志管道的最终阶段利用AWS数据仓库服务。在此，AWS Glue爬虫程序用于发现和分类对象存储中的数据，自动生成提供统一元数据存储库的Glue数据目录。Glue表模式会定期更新，确保其与日志数据结构的演进保持同步。与AWS Glue的集成使我们能使用无服务器交互式查询服务Amazon Athena查询归档日志。Athena允许我们直接对对象存储中的数据运行SQL查询，为日志分析和历史数据探索提供经济高效的可扩展解决方案。

日志处理流水线虽然经过精心设计，但仍面临挑战和潜在的故障点。其中最关键的脆弱性之一在于系统的弹性，尤其是Elasticsearch集群。Elasticsearch作为分布式系统，在高入口流量、密集查询或分片重定位等内部操作场景下都可能遇到背压。一旦发生背压，可能引发整个流水线的连锁问题。例如，当Elasticsearch集群不堪重负时，可能开始拒绝或延迟日志摄入，导致Logstash甚至Filebeat出现积压，进而造成日志丢失或处理延迟。

另一个脆弱点是Elasticsearch中的自动模式检测机制。虽然该机制旨在适应不断变化的日志结构，但当应用程序日志发生重大字段类型变更时可能失效。如果模式检测未能识别新的字段类型，可能导致从Logstash到Elasticsearch的写入失败，引发日志处理瓶颈和潜在的数据不一致问题。这凸显了主动式日志模式管理的重要性，以及需要健全的监控机制来及时检测和处理此类问题。

内存管理同样是流水线稳定性的关键环节。包括Logstash和Filebeat在内的日志处理层在有限内存资源下运行以控制成本。当背压发生时，这些组件可能遭遇内存不足问题，尤其在系统降速期间。随着日志累积和背压增加，这些进程的内存占用会持续增长，逐渐逼近其极限。若未及时处理，可能导致进程崩溃或进一步加剧背压问题。

负责将日志从Elasticsearch传输到对象存储的归档作业也面临挑战。这些作业有时会消耗大量资源，其性能对节点规格和内存可用性非常敏感。当流水线中出现垃圾数据或异常庞大的日志条目时，可能对归档过程造成压力，导致因内存耗尽或节点容量限制而失败。这强调了在流水线早期进行数据验证和过滤的重要性，以防止此类问题影响归档阶段。

为缓解这些潜在故障，我们实施了具有独特设计目标的强大告警系统：验证端到端日志流。该告警通过对比应用负载均衡器接收的请求数量与成功归档的日志数量，提供贯穿整个流水线的日志数据流全景视图。这种方法使我们能快速识别可能预示日志丢失或处理问题的异常差异。

告警机制基于一个简单而有效的对比：抵达ALB的请求数（代表进入系统的总日志量）与成功归档到长期存储的日志数。通过监控这个比率，我们可以确保"有进必有出"，为日志基础设施的健康状况提供可靠验证。当告警触发时，表明可能存在数据不匹配，需要立即调查和修复。

实践中，该告警已被证明是极具价值的工具，特别是在基础设施重构期间。例如，当我们将ALB迁移至VPC源站时，该告警有效识别并解决了由此产生的日志流差异。同时，它也在更隐蔽的场景中发挥作用。例如，当归档作业因意外问题未能运行时，告警会标记缺失的归档日志，使我们能在影响日志分析和保留流程前及时调查解决问题。

尽管这个告警功能强大，但它只是我们综合监控策略的一部分。我们持续优化和调整日志基础设施，以应对不断增长且日益复杂的日志数据。通过结合主动监控、高效资源管理和对系统行为的深入理解，Hugging Face确保日志流水线保持弹性、可靠，并能支持平台的发展和不断变化的需求。这个告警系统体现了我们对维护健壮透明日志系统的承诺，为团队提供维持Hugging Face平稳运行所需的洞察力。

### Kubernetes API请求错误与速率限制

在运营云原生应用和基于Kubernetes的基础设施时，即使是看似微小的问题，若未及时处理也可能升级为重大停机事件。这对于作为Kubernetes集群中枢神经系统的Kubernetes API尤为关键，它负责协调容器的创建、管理和网络连接。Hugging Face的经验表明，监控Kubernetes API错误率和速率限制指标是至关重要的实践，能够预防潜在灾难。

Hugging Face的基础设施与Kubernetes深度集成，其中`kube-rs`库在高效构建和管理这个生态系统方面发挥了关键作用。`kube-rs`为Kubernetes应用开发提供了以Rust为核心的方法论，为开发者提供了熟悉而强大的工具集。其核心引入了三个关键概念：反射器、控制器和自定义资源接口。反射器确保Kubernetes资源的实时同步，使应用能快速响应变更。控制器作为决策者持续协调资源的期望状态与实际状态，实现Kubernetes的自愈能力。自定义资源接口则扩展了Kubernetes，允许开发者定义应用特定资源以实现更好的抽象。

此外，`kube-rs`引入了监视器和终结器。监视器追踪特定资源的变更，触发对应事件的响应动作。终结器则通过定义自定义逻辑确保资源的正确清理和终止。通过为这些Kubernetes概念提供基于Rust的抽象，`kube-rs`使开发者能够构建健壮高效的应用，在保持Rust核心开发方式的同时充分利用Kubernetes平台的强大功能和灵活性。这种集成简化了复杂Kubernetes应用的构建和管理流程，使其成为Hugging Face基础设施中的重要工具。

Hugging Face与Kubernetes的集成是我们基础设施的基石，而`kube-rs`库在管理这个生态系统中起着关键作用。`kube::api`模块在自动化各种任务方面至关重要，例如管理支持Spaces产品的自定义域名的HTTPS证书。通过编程方式处理证书生命周期，我们确保服务的安全性和可访问性，为用户提供无缝体验。此外，在常规维护期间，我们也在非用户侧功能中使用该模块来促进节点排空和终止，确保基础设施更新期间的集群稳定性。

`kube::runtime`模块对我们同样至关重要，它支持开发和部署自定义控制器，从而增强基础设施的自动化和弹性。例如，我们在托管服务中实施了用于计费管理的控制器，通过对客户Pod的监视器和终结器确保准确的资源追踪和计费。这种定制化水平使我们能够根据特定需求调整Kubernetes。

通过kube-rs，Hugging Face 在云原生应用方面实现了高效率、高可靠性和高控制力。该库以 Rust 为核心的设计理念与我们的工程哲学相契合，使我们能够利用 Rust 的优势来管理 Kubernetes 资源。通过自动化关键任务和构建自定义控制器，我们创建了一个可扩展、具备自愈能力的基础设施，以满足我们用户和企业客户多样且不断变化的需求。这种集成体现了我们致力于充分发挥 Kubernetes 潜力，同时保持适合我们独特需求的开发方法。

虽然我们的基础设施很少遇到与 Kubernetes API 相关的问题，但我们始终保持警惕，尤其是在部署期间和部署之后。Kubernetes API 是我们使用`kube::runtime::`管理客户 Pod 和云网络资源的关键组件。API 通信中的任何中断或效率低下都可能对我们的服务产生连锁影响，可能导致停机或性能下降。

其他 Kubernetes 用户的经验凸显了监控这些 API 指标的重要性。例如，OpenAI 分享了一份**状态更新**，详细说明了 DNS 可用性问题如何导致重大停机。虽然这与 Kubernetes API 没有直接关系，但他们的经验突显了各种基础设施组件之间的相互关联性以及发生连锁故障的可能性。正如 DNS 可用性对应用程序的可访问性至关重要一样，一个健康且响应迅速的 Kubernetes API 对于管理和编排我们的容器化工作负载也至关重要。

作为一项最佳实践，我们已将这些 API 指标集成到我们的监控和告警系统中，确保任何异常或趋势都能及时引起我们的注意。这使我们能够采取主动措施，在问题影响客户之前进行调查和解决。例如，有一次，一个集群开始对 Kubernetes API 的请求进行速率限制。我们能够追溯到是我们的一个第三方工具遇到了一个错误，并反复请求排空一个节点，即使它已经这样做了。作为回应，我们能够在任何明显的性能下降影响用户之前，从系统中清除这个故障作业。这是一个很好的例子，说明告警场景并不总是在部署新版本的自定义控制器后立即发生——错误可能需要时间才会在生产环境中显现为问题。

总之，虽然我们的基础设施稳健且架构良好，但我们认识到，保持警惕和主动监控对于维持其健康和稳定至关重要。通过密切关注 Kubernetes API 错误率和速率限制指标，我们保护了我们的托管服务，确保了顺畅的客户体验，并坚守我们对可靠性和性能的承诺。这证明了我们的信念：在云原生技术领域，每一个组件，无论多么微小，都在我们平台的整体韧性和成功中扮演着重要角色。

### 额外告警：新集群未发送任何指标

感谢您阅读至此，这里有一个额外的告警彩蛋！

在 Hugging Face，我们的实验环境不断变化，经常会有为特定目的创建的集群随着功能和产品的迭代而启动和关闭。此外，我们的增长也是一个重要因素，集群会扩展到其极限，并触发类似细胞分裂的过程以保持平衡。为了在这种动态环境中导航，同时避免硬编码或引入额外的集群发现层，我们设计了一个巧妙的告警来适应这些变化。

```bash
(
  (sum by (cluster) (rate(container_network_transmit_packets_total{pod="prometheus"}[1h] ) ) > 0)
or
  (-1 * (sum by (cluster) (rate(container_network_transmit_packets_total{pod="prometheus"}[1h] offset 48h) ) > 0))
) < 0

```

此查询中使用的指标是`container_network_transmit_packets_total`，它表示容器传输的数据包总数。该查询筛选出来自集群本地 Prometheus 实例的指标，该实例负责指标收集以及远程写入我们的中央指标存储——Grafana Mimir。数据包的传输近似于健康的远程写入，这正是我们希望在所有活动集群中确保的。

查询的第一部分执行当前速率检查。查询的第二部分通过使用与当前速率检查相同的计算加上`offset 48h`子句来执行历史速率检查。`-1 *`乘法用于反转结果，因此如果历史速率大于 0，结果将小于 0。

`or`运算符组合了查询的两个部分。如果满足以下任一条件，查询将返回`true`：

*   集群的当前数据包传输速率大于 0。
*   集群的历史数据包传输速率（48 小时前）大于 0，但当前速率不大于 0。

外层的`< 0`条件检查`or`运算的结果是否小于 0。这意味着，只有当两个条件都不满足时，即如果一个集群从未发送过任何指标（当前和历史速率均为 0），查询才会触发。

查询将在两种情况下触发：

1.  **没有指标的新集群**：添加了一个新集群，但它尚未发送任何指标。在这种情况下，当前和历史速率都将为 0，查询将触发。
2.  **从未发送过指标的集群**：一个集群已存在超过 48 小时，但从未发送过任何指标。在这种情况下，历史速率将为 0，当前速率也将为 0，从而触发查询。

在这两种情况下，查询都会检测到集群未发送任何指标并触发告警。

这个简单而有效的解决方案会在我们的指标基础设施崩溃、集群设置期间以及集群被拆除时触发，为我们提供关于基础设施健康状况的及时洞察。虽然它可能不是我们告警库中最关键的警报，但它具有特殊的意义，因为它诞生于协作。它证明了通过严格的代码审查进行团队合作的力量，这得益于 Hugging Face 基础设施团队同事们的专业知识和乐于助人的精神 🤗
