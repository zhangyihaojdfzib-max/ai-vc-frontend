在 Vercel，功能开关是我们交付产品的方式。从新功能到 v0 中的模型更新，甚至像生产数据库迁移这样的基础设施变更，都会使用功能开关作为切换点。仅 v0 团队在任何时刻就运行着数百个功能开关。

合并代码会将构建发送到生产环境，但功能开关控制着用户能否看到变更的内容。功能开关让你可以按照自己的节奏发布，在准备好时向特定用户群体发布，并通过切换开关立即回滚，而无需修改源文件或重新部署。

Vercel Flags 是平台原生的：默认在服务端运行，对页面性能零影响，并直接与你正在使用的框架集成。

## 链接到标题什么是 Vercel Flags

Vercel Flags 让你可以创建功能开关，根据用户属性、用户群体或环境定义目标规则，执行渐进式发布，并在生产环境出现问题时一键关闭。

在你的代码中，你可以通过 Flags SDK 读取开关状态。Flags SDK 是我们维护的一个开源、与提供商无关的库，并为 Next.js 和 SvelteKit 提供了一流的适配器。如果你使用其他框架，可以通过内置的 OpenFeature 提供商来使用 Vercel Flags。

Vercel Flags 仪表盘与你的项目和部署环境并列，你可以在其中创建和管理功能开关。

## 链接到标题为什么框架原生如此重要

其他功能开关提供商提供通用的 SDK，让你自己与框架集成，以及一个单独的仪表盘来管理开关。Vercel Flags 内置于 Vercel 平台中，因此你可以在与部署环境相同的仪表盘中管理开关，并且你的代码通过框架原生的 Flags SDK 读取开关状态。

### 链接到标题服务端评估

当在客户端评估功能开关时，用户会看到加载器、闪烁或布局偏移。浏览器在获取到开关值之前无法渲染正确的视图。Flags SDK 在服务端进行评估。使用 Next.js React 服务端组件，你可以在渲染期间使用 `await` 读取开关状态。正确的视图由服务端确定，浏览器直接渲染，无需单独的开关请求。该值来自 Vercel Flags，配置变更会在毫秒内传播到所有区域。

```
1import { showNewFeature } from "@/flags"2
3export default async function Page() {4  const isEnabled = await showNewFeature()5  return isEnabled ? <NewDashboard /> : <OldDashboard />6}
```

在路由中服务端读取开关状态。用户在页面渲染前就能收到正确的变体。

对于高级用例，你可以将开关作为一个 Promise 传递给客户端组件，而不是等待它。这允许页面在开关值到达之前开始渲染，同时组件显示一个回退内容。开关仍然在服务端解析，因此不会有浏览器端的请求。

### 链接到标题自动开关注册

Vercel Flags 会自动注册开关。在代码中定义一个开关，部署后，它就会作为草稿出现在仪表盘中。当你准备好配置目标规则并发布时，提升该草稿。从代码中移除开关后，仪表盘会将其标记为未引用，这样你始终知道哪些可以安全归档。你编写的开关就是你管理的开关，无需手动维护单独的列表。

```
1import { flag } from "flags/next"2import { vercelAdapter } from "@flags-sdk/vercel"3
4export const showNewFeature = flag({5  key: "show-new-feature",6  adapter: vercelAdapter()7})
```

使用 Flags SDK 和 Vercel 适配器定义一个开关。

### 链接到标题预计算

静态页面快速且一致，因为它们从离你和你的用户最近的 CDN 区域提供服务。但添加一个开关会使页面变为动态。要么你在服务端渲染并失去 CDN 分发，要么你在客户端获取开关并重新出现布局偏移。然而，Flags SDK 提供了一个可选的、高级的模式来解决这个问题。预计算让你在构建时构建所有变体，通过 CDN 分发它们，并让路由中间件（Next.js 中的 `proxy.ts` 文件）将每个用户路由到正确的变体。每个页面保持静态，加载时没有布局偏移。

注意：预计算是一个高级但强大的模式。阅读文档以了解更多。

## 链接到标题 Vercel 在 Vercel Flags 上交付产品

虽然我们在 2026 年 4 月将 Vercel Flags 全面开放，但我们已经内部使用了一年多。v0 团队是一个很好的例子，展示了其在大规模场景下的应用，在任何时刻都有数百个开关处于活动状态。

以下是一些团队放在开关后面的示例：

- 正在开发的新功能
- 按用户或用户群体进行的 AI 模型路由
- 运营紧急开关
- 数据库迁移和提供商切换
- 面向早期客户或内部团队的 Beta 访问

对于像 v0 这样的产品，任何增加的延迟都会降低最终用户体验。由于 Vercel Flags 在服务端评估，团队可以在不增加客户端评估带来的额外往返延迟的情况下使用功能开关。

因为每个新功能都构建在开关后面，开发者可以持续合并到主分支而无需发布未完成的工作。没有长期存在的分支，也没有痛苦的合并冲突需要解决。部署代码和发布功能变成了两个独立的决策。

发布遵循一个受控的流程。构建该功能的开发者首先看到它，然后是内部团队。之后，开关逐步向 5%、10%、25% 和 50% 的用户开放，每个阶段持续六小时，最后才向所有用户开放。如果在任何阶段出现问题，团队可以在不修改代码或重新部署的情况下关闭该功能。

开关还控制着 v0 的 AI 模型流量，在新模型发布时逐步切换，而不是一次性全部切换。

v0 甚至使用一个开关来运行生产数据库迁移。我们保持新旧数据库同步，开关控制着使用哪个数据库。切换开关本身就是切换操作。我们在预演环境中反复演练，然后在生产环境中运行，没有影响流量。这个开关将一项高风险的基础设施变更变成了团队可以演练、安排并自信交付的事情。
