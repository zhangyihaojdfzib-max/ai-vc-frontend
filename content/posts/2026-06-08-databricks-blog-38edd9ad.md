## 从维护 PDF 到 AI Agent（智能体）驱动的可操作洞察

太阳能和风能电站的运维供应商通常以 PDF 形式交付报告，关键信息分散在自由文本、表格和图像中。这种格式易于访问但不可扩展：团队必须手动阅读每份文档以了解故障、趋势或重复出现的问题，随着资产数量的增长，跨电站比较变得缓慢且不一致。

Plenitude 和 Databricks 构建了一个基于 Agent（智能体）的系统，可将这些 PDF 维护报告转换为结构化数据。核心思路很简单：将文档转换为数据，然后使用 AI Agent（智能体）从这些数据中获取可操作的洞察。用户现在可以用自然语言提问、分析随时间变化的趋势、比较电站并导出结构化输出，而无需逐一浏览报告。

## 基于 Agent（智能体）的 PDF 数据分析架构

该解决方案始于电站级 PDF 报告的事件驱动摄取。每份新报告都会触发一个 Databricks 作业，该作业解析文档并应用基于 LLM（大语言模型）的提取。提取的元素被序列化为 JSON 并存储在 Delta Lake 中，Delta Lake 会保留完整的版本历史以用于审计和回放。

![图 1：用于多电站分析的自动化文档智能架构](/images/posts/8e0cb712b581.png)

为了解决维护信息几乎完全存在于非结构化 PDF 中的根本问题，Plenitude 使用 Databricks Document Intelligence AI Functions——特别是 `ai_parse_document`——从每页中提取多种类型的元素，包括文本块、表格、图形和元数据。每个元素都附有电站、报告周期、页码和内容类型等属性，并且每条记录都保留与原始报告的直接链接以实现可追溯性。

这种结构解锁了强大的功能：

- 按时间、类别和地理位置进行筛选。
- 识别内容类型并使用空间坐标。
- 将每个洞察追溯到原始 PDF。
- 与 BI 工具和数字 Agent（智能体）集成，而无需更改底层文档。

维护报告不再是静态文件，而是成为可用于高级分析和 Agent（智能体）推理的持久数据层。

## Databricks 上的数据处理：从 PDF 到 Delta Lake

该架构主要分为三层：摄取与解析、数据结构化以及基于 Agent（智能体）的交互。

![图 2：提取 - 查询 - 推理](/images/posts/1a2324329fc2.png)

步骤 1：解析

使用 `ai_parse_document`，管道从每页中提取文本、表格和元数据，并将其序列化为结构化的 JSON 对象。即使是复杂的表格也能捕获完整的上下文，包括其在页面上的位置和 HTML 表示形式。

步骤 2：规范化与存储

对于每个页面（page_id）和对象（id），系统在 Delta Lake 表中创建一行。每行包含：

- 提取的 JSON 内容。
- 页面和对象标识符。
- 表示页面上边界框的坐标（coords）。
- 内容类型（例如，文本或表格）。
- 高价值元数据，如月份、年份、文件名、类别和国家。

这种规范化模型将 PDF 转换为统一、可查询的数据集，该数据集透明且易于与其他数据源连接，同时保留与原始文档的完全可追溯性。

步骤 3：Genie 空间与 Agent（智能体）模式

在此精心策划的数据层之上，Plenitude 构建了一个专用的 Genie 空间，然后利用 Genie 的 Agent（智能体）模式对数据执行深度研究。Genie 使用结构化的 Delta Lake 表作为其主要上下文，并允许用户使用自然语言与维护数据进行交互。

当用户提问时，Genie：

- 使用 Unity Catalog 中的语义元数据来识别可用的表和列。
- 利用详细的列描述、精心策划的知识库和 SQL 示例来指导查询生成。
- 针对结构化层生成并执行 SQL。
- 返回答案、可视化结果以及可选的导出结果。

这种设计使 Genie 能够理解维护数据的业务语义及其底层结构，从而产生准确、上下文感知的答案。

## 为什么元数据和指令对 Genie 至关重要

要从复杂的 PDF 派生数据集中获得可靠的结果，仅靠上下文是不够的。Plenitude 发现两种设计模式至关重要：丰富的元数据和针对 Genie 空间的显式指令。

## 元数据作为与 Agent（智能体）的契约

定义良好的表和列描述告诉 Genie 每个字段的含义以及应如何使用。例如，page_id 标识原始报告中的源页面，type 指示元素是文本还是表格，coords 编码空间位置，content 包含提取的文本或表格表示形式。这种元数据将原始 JSON 转换为 Genie 可以推理的可理解知识。

## 通用指令作为操作基础

当数据碎片化或跨多个页面时，添加到 Genie 空间本地知识库中的领域特定指令变得至关重要。Plenitude 编码了处理多页表格、忽略 HTML 伪影、排除标题行以及应用特定电站过滤器的规则。

一个实际例子：即使拥有完整的元数据，如果 Genie 对 YTD 列求和或忽略缺失的月份，它也可能计算出不正确的季度总计。通过添加诸如“仅使用月份级别列，绝不使用 YTD 字段”和“在求和之前验证所有必需月份都存在”之类的清晰指令，团队为 Genie 提供了操作护栏，确保结果一致。

这些特定于 Genie 空间的指令，结合来自 Unity Catalog 的元数据，帮助 Genie 应用正确的逻辑来正确解释数据。

## 使用 Genie 和 Agent Bricks 实现可扩展的 Agent（智能体）工作流

虽然 Genie 在结构化的维护层之上提供了强大的研究 Agent（智能体）体验，但 Plenitude 还需要可重复的工作流和编排来支持不断增长的用例集。Agent Bricks 是这一演进过程中的下一步。

借助 Agent Bricks，Plenitude 可以从“LLM（大语言模型）加提示词”模式转变为代表维护分析师和工程师执行一系列操作的 Agent（智能体）工作流。为 Genie 提供支持的同一套精心策划的 Delta Lake 表、元数据和指令，可以被使用 Agent Bricks 构建的 Supervisor 风格 Agent（智能体）重用，以：

- 将复杂问题分解为更小的分析任务。
- 调用 Genie 工具流程来生成和执行 SQL。
- 触发下游操作，例如生成报告或创建警报。

以前需要手动连接提示词、工具和验证逻辑的工作，现在可以集中到 Agent Bricks 中，在管理数据的同一 Databricks 平台上进行。

## 使用自动 Liquid Clustering 优化性能

由于 Agent（智能体）驱动的查询是探索性和动态的，传统的基于 Z-ORDER 的调优并不总是理想的。Plenitude 观察到，随着新报告、用户和问题的出现，访问模式会发生变化，这使得手动聚类难以维护。

相比之下，自动聚类能够学习表的实际使用方式并相应调整布局。这减少了对前期索引设计和持续调优的需求，在概念验证和早期上线阶段尤为重要。在此背景下，自动聚类成为Delta表上Agent（智能体）和LLM驱动工作负载的首选方案。

## 保护Genie Rooms的数据访问安全

运维数据通常具有国家或地区特定的访问要求。为一致地执行这些规则，Plenitude结合使用了Unity Catalog和表级行级安全策略。

一个Unity Catalog函数用于判断当前用户可访问哪些国家，并返回国家列表；若用户拥有完全可见权限，则返回关键字ALL。随后，一个表基于该函数过滤行，确保每位用户仅能看到其授权国家的数据。

当用户通过Genie Room进行交互时，所有查询均在过滤后的表上运行，因此行级安全策略会自动生效。这意味着用户可以用自然语言提问，但只能从被允许查看的数据中获取结果。同一数据集同时支撑Genie、Agent（智能体）和BI工具，而数据可见性则根据用户动态调整。

## 未来改进：迈向预测性维护

由于运维报告包含未解决事件和故障详情，该结构化数据模型为预测性维护奠定了坚实基础。逆变器便是一个典型例子：单台设备故障可能导致数兆瓦时的发电损失，而重复性问题往往首先出现在运维记录中。

通过分析故障模式随时间的变化，Plenitude能够：
- 识别潜在记录问题。
- 检测早期预警信号。
- 优先排查需要深入调查的电站。
- 为预测模型提供更高质量的事件历史数据。

基于Agent（智能体）的系统将这些信号转化为可访问的分析、趋势和可视化结果，使团队能够预见问题，而非被动应对。
