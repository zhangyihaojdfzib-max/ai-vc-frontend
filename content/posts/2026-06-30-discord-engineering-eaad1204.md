Discord 的 API 由一个统一的 Python 代码库驱动，包含超过 1700 个 API 端点以及约 700 个后台任务。工程师们每天都在对这个共享代码进行修改，并通过分阶段发布流程将其持续部署到数百个独立的 Kubernetes 部署中。

这涉及大量的代码、工程师、端点和部署！要追踪每天所做的所有更改可能颇具挑战性，但我们拥有良好的监控手段，能够让我们密切关注延迟、吞吐量和错误率，从而帮助检测可能对用户或系统产生负面影响的性能回退。

去年我们希望改进的一个可观测性缺口，是了解托管成本如何在产品功能之间分配的。例如，运营用于发送和接收消息的 API 部分需要多少成本？启动一个直播流呢？给朋友发送一份 Nitro 礼物呢？这些数值随时间如何变化？上周某人提交的更改是否显著影响了某个团队的托管支出？我们既想知道单个端点（例如在文本频道中发送消息）的答案，也想知道整个功能（例如聊天——稍后会详细介绍）的答案。

大多数云服务商很乐意按 Kubernetes 部署来拆分成本，这很有帮助，但由于我们部署 API 的方式，这只是第一步。我们在所有 Kubernetes 部署中运行相同的代码库，每个部署处理特定子集的 HTTP 流量或后台任务。由于我们已经拥有如此多的部署，进一步拆分它们以方便成本追踪是不可行的。我们需要找到一种方法，在不改变部署拓扑结构的情况下，为现有系统增加更好的追踪能力。

另一个挑战是，每个 API 工作进程会同时处理多个任务。在任何时刻，它都需要处理与任意数量功能相关的工作（我们确实将某些流量隔离到特定部署，但这在此处并无帮助）。最终，为了理解服务于某个特定功能的 API 流量的成本，我们需要能够根据部署在该功能相关代码上花费的时间来分配该部署的成本。通过扩展我们应用程序的性能分析工具，我们恰好实现了这一点。
