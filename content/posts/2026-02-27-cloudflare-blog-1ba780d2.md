# 毒性组合：当微小信号累积成安全事件

2026-02-27

- Bashyam Anant
- Himanshu Anand

![](/images/posts/960483bce912.png)

凌晨3点，一个IP地址请求了登录页面。这看似无害。但随后，在多个主机和路径上，同一来源开始附加 `?debug=true`——这是攻击者探测环境以评估技术栈并计划入侵的迹象。

微小的配置错误、被忽略的防火墙事件或请求异常，单独看似乎都无害。但当这些微小信号汇聚时，它们可能爆发成被称为"毒性组合"的安全事件。这是指攻击者发现并组合多个微小问题——例如Web应用程序上遗留的调试标志或未经身份验证的应用程序路径——以入侵系统或窃取数据。

Cloudflare的网络观察着对您技术栈的请求，因此拥有数据来识别这些正在形成的毒性组合。在本文中，我们将展示如何从我们的应用程序安全数据中浮现这些信号。我们将介绍最常见的毒性组合类型及其带来的危险漏洞。我们还将提供详细信息，说明您如何利用这些情报来识别和解决技术栈中的弱点。

## 我们如何定义毒性组合

您可以用几种不同的方式定义"毒性组合"，但这里有一个基于我们如何看待自身数据集的实用定义。大多数Web攻击最终通过自动化扩大规模；一旦攻击者找到可行的利用方法，他们通常会将其编写成机器人程序来完成工作。通过观察机器人流量、特定应用程序路径、请求异常和配置错误的交集，我们可以发现潜在的入侵。我们使用这个框架来每秒推理数百万个请求。

虽然Web应用程序防火墙（WAF）、机器人检测和API保护等点状防御已经发展到包含行为模式和信誉信号，但它们仍然主要侧重于评估单个请求的风险。相比之下，Cloudflare对"毒性组合"的检测将视角转向更广泛的意图，分析围绕多个信号的上下文汇合，以识别正在酝酿的事件。

**毒性组合即情境化检测**

这种视角的转变很重要，因为许多真实事件没有明显的利用载荷、没有清晰的签名，也没有单一事件能直接表明"攻击"。因此，在接下来的内容中，我们结合以下上下文来构建几种毒性组合：

- **机器人信号**
- **应用程序路径**，尤其是敏感路径：管理、调试、指标、搜索、支付流程
- **异常**，包括：意外的HTTP状态码、地理位置跳跃、身份不匹配、高ID流失率、绕过速率限制（分布式IP执行相同操作）、请求或成功率激增
- **漏洞或配置错误**：缺少会话Cookie或身份验证头、可预测的标识符

## 流行应用程序栈上的毒性组合示例

我们查看了Cloudflare 24小时内的数据，以了解这些模式在流行应用程序栈中实际出现的频率。如下表所示，我们分析的主机中约有11%易受这些组合的影响，其中易受攻击的WordPress网站占比较大。排除WordPress站点后，只有0.25%的主机显示出可利用的毒性组合迹象。虽然罕见，但它们代表了易受攻击的主机。

为了理解这些数据，我们将其分解为攻击的三个阶段：

- **探测到的预估主机数**：这是"广撒网"。它统计了我们看到针对特定敏感路径（如`/wp-admin`）发起HTTP请求的独立主机数量。
- **按毒性组合筛选的预估主机数**：在这里，我们将列表缩小到实际符合我们毒性组合标准的特定主机。
- **可达的预估主机数**：对利用尝试成功响应的独立主机——攻击的"确凿证据"。一个简单的`200 OK`响应（例如由附加`?debug=true`触发的响应）可能是误报。我们验证了路径，以过滤掉由以下情况引起的噪音：尽管状态码为200但仍需要凭据的已认证路径、掩盖真实利用路径的重定向，以及为不可达路径返回成功代码的源站配置错误。

在接下来的部分中，我们将深入探讨具体发现以及驱动这些组合的逻辑。提供的检测查询是必要的，但如果不测试可达性，则可能不充分；这些发现可能是误报。在某些情况下，**Cloudflare Log Explorer**允许在未采样的Cloudflare日志上执行这些查询。

## 跨多个应用程序主机探测敏感管理端点

### 我们检测到了什么？

我们观察到自动化工具扫描常见的管理登录页面——例如WordPress管理面板（`/wp-admin`）、数据库管理器和服务器仪表板。可在**Cloudflare Log Explorer**中执行的查询模板版本如下：

```Rust
SELECT
  clientRequestHTTPHost,
  COUNT(*) AS request_count
FROM
  http_requests 
WHERE
  timestamp >= '{{START_DATE}}'
  AND timestamp <= '{{END_DATE}}'
  AND edgeResponseStatus = 200
  AND clientRequestPath LIKE '{{PATH_PATTERN}}' //e.g. '%/wp-admin/%'
  AND NOT match( extract(clientRequestHTTPHost, '^[^:/]+'), '^\\d{1,3}(\\.\\d{1,3}){3}(:\\d+)?$') // comment this line for Cloudflare Log Explorer
  AND botScore < {{BOT_THRESHOLD}} // we used botScore < 30
GROUP BY
  clientRequestHTTPHost
ORDER BY
  request_count DESC;
```

### 为什么这很严重？

公开可访问的管理面板可能助长**暴力破解攻击**。如果成功，攻击者可以通过将主机添加到僵尸网络来进一步危害该主机，该僵尸网络会探测其他网站是否存在类似漏洞。此外，这种毒性组合可能导致：

- **漏洞扫描**：攻击者识别您正在运行的具体软件版本（如Tomcat或WordPress），并针对已知漏洞（CVE）发起定向利用。
- **用户枚举**：许多管理面板意外泄露有效的用户名，这有助于攻击者策划更具说服力的钓鱼或登录攻击。

### 支持证据是什么？

机器人自动化与暴露的管理接口（如：`/wp-admin/`、`/admin/`、`/administrator/`、`/actuator/*`、`/_search/`、`/phpmyadmin/`、`/manager/html/` 和 `/app/kibana/`）的毒性组合。

### 如何缓解此发现？

1.  实施零信任访问。
2.  如果出于任何原因端点必须保持公开，请实施挑战平台以增加机器人的访问难度。
3.  实施IP白名单：使用您的WAF或服务器配置，确保管理路径仅能从您的企业VPN或特定办公室IP地址访问。
4.  隐藏管理路径：如果您的平台允许，请重命名默认的管理URL（例如，将`/wp-admin`更改为唯一的、不可猜测的字符串）。
5.  部署地理封锁：如果您的管理员仅从特定国家/地区操作，请阻止所有来自这些区域之外的、指向这些敏感路径的流量。
6.  强制实施多因素认证：确保每个管理入口点都需要第二重验证；仅凭密码不足以阻止专门的爬虫。

## 未经认证的公共API端点允许通过可预测标识符导致大规模数据暴露

我们发现了一些API端点，互联网上的任何人无需密码或登录即可访问（参见OWASP: API2:2023 – 身份验证失效）。更糟糕的是，其识别记录的方式（使用简单、可预测的ID号，参见OWASP: API1:2023- 失效的对象级授权）允许任何人简单地“计数”遍历您的数据库——这使得攻击者无需直接访问您的网站就能更轻松地枚举和“抓取”您的业务记录。

```Rust
SELECT
  uniqExact(clientRequestHTTPHost) AS unique_host_count
FROM  http_requests
WHERE timestamp >= '2026-02-13'
  AND timestamp <= '2026-02-14'
  AND edgeResponseStatus = 200
  AND bmScore < 30
  AND (
       match(extract(clientRequestQuery, '(?i)(?:^|[&?])uid=([^&]+)'),  '^[0-9]{3,10}$')
    OR match(extract(clientRequestQuery, '(?i)(?:^|[&?])user=([^&]+)'), '^[0-9]{3,10}$')
    OR length(extract(clientRequestQuery, '(?i)(?:^|[&?])uid=([^&]+)'))  BETWEEN 3 AND 8
    OR length(extract(clientRequestQuery, '(?i)(?:^|[&?])user=([^&]+)')) BETWEEN 3 AND 8
  )
```

这是一个“零利用”漏洞，意味着攻击者无需是黑客即可窃取您的数据；他们只需要更改网页链接中的一个数字。这会导致：

-   大规模数据暴露：您的整个客户数据集被大规模抓取。
-   二次攻击：被盗数据被用于定向钓鱼或账户接管。
-   监管风险：由于暴露敏感的个人身份信息，导致严重的隐私违规（GDPR/CCPA）。
-   欺诈：竞争对手或恶意行为者得以洞察您的业务量和客户群。

## 调试参数探测暴露系统详情

我们发现存在将`debug=true`附加到Web路径以暴露系统详情的证据。以下是一个可在Cloudflare日志浏览器中执行的查询模板化版本：

```Rust
SELECT
  clientRequestHTTPHost,
  COUNT(rayId) AS request_count
FROM
  http_requests
WHERE
  timestamp >= '{{START_TIMESTAMP}}'
  AND timestamp < '{{END_TIMESTAMP}}'
  AND edgeResponseStatus = 200
  AND clientRequestQuery LIKE '%debug=false%'
  AND botScore < {{BOT_THRESHOLD}}
GROUP BY
  clientRequestHTTPHost
ORDER BY
  request_count DESC;
```

虽然这不会立即窃取数据，但它为攻击者提供了您内部基础设施的高清地图。这种“侦察”使他们下一次攻击成功的可能性大大增加，因为他们可以看到：

-   隐藏的数据字段：不应向用户显示的敏感内部信息。
-   技术栈详情：特定的软件版本和服务器类型，使他们能够查找这些确切版本的已知漏洞。
-   逻辑提示：错误消息或堆栈跟踪，准确解释您的代码如何工作，帮助他们找到破坏它的方法。

## 公开暴露的监控端点，提供内部基础设施可见性

我们发现"健康检查"和监控仪表板对整个互联网可见。具体来说，像 `/actuator/metrics` 这样的路径会对任何请求者做出响应。一个可在 Cloudflare Log Explorer 中执行的模板化查询版本如下：

```Rust
SELECT
  clientRequestHTTPHost,
  count() AS request_count
FROM http_requests
WHERE timestamp >= toDateTime('{{START_DATE}}')
  AND timestamp <  toDateTime('{{END_DATE}}')
  AND botScore < 30
  AND edgeResponseStatus = 200
  AND clientRequestPath LIKE '%/actuator/metrics%' // an example
GROUP BY
  clientRequestHTTPHost
ORDER BY request_count DESC
```

虽然这些端点通常不会直接泄露客户密码，但它们为复杂的攻击提供了"蓝图"。暴露会导致：

- **战略时机选择**：攻击者可以实时监控你的 CPU 和内存使用情况，以便在你的系统已经处于压力之下时，精确地发起**拒绝服务 (DoS) 攻击**。
- **基础设施映射**：这些日志通常会揭示内部服务、依赖项和版本号的名称，帮助攻击者找到已知漏洞加以利用。
- **利用链构建**：关于线程计数和环境提示的信息可用于绕过安全层或在你的网络内提升权限。
