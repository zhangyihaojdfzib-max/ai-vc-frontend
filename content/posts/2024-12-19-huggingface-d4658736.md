# 终于，BERT 的替代品来了

## TL;DR

这篇博客介绍了 ModernBERT，一系列最先进的仅编码器模型，在各个方面都优于上一代编码器，拥有 8192 序列长度、更好的下游性能和更快的处理速度。

ModernBERT 可作为任何 BERT 类模型的即插即用替代品，提供 base（1.49 亿参数）和 large（3.95 亿参数）两种模型规模。

ModernBERT 将包含在 transformers 的 v4.48.0 版本中。在此之前，需要从主分支安装 transformers：

```sh
pip install git+https://github.com/huggingface/transformers.git
```

由于 ModernBERT 是一个掩码语言模型（MLM），你可以使用 fill-mask 管道或通过 AutoModelForMaskedLM 加载它。要将 ModernBERT 用于分类、检索或问答等下游任务，请按照标准的 BERT 微调方法进行微调。

⚠️ 如果你的 GPU 支持，我们建议将 ModernBERT 与 Flash Attention 2 一起使用以达到最高效率。为此，请按如下方式安装 Flash Attention，然后正常使用模型：

```bash
pip install flash-attn
```

使用 AutoModelForMaskedLM：

```python
from transformers import AutoTokenizer, AutoModelForMaskedLM

model_id = "answerdotai/ModernBERT-base"
tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForMaskedLM.from_pretrained(model_id)

text = "The capital of France is [MASK]."
inputs = tokenizer(text, return_tensors="pt")
outputs = model(**inputs)

masked_index = inputs["input_ids"][0].tolist().index(tokenizer.mask_token_id)
predicted_token_id = outputs.logits[0, masked_index].argmax(axis=-1)
predicted_token = tokenizer.decode(predicted_token_id)
print("Predicted token:", predicted_token)
```

使用管道：

```python
import torch
from transformers import pipeline
from pprint import pprint

pipe = pipeline(
    "fill-mask",
    model="answerdotai/ModernBERT-base",
    torch_dtype=torch.bfloat16,
)

input_text = "He walked to the [MASK]."
results = pipe(input_text)
pprint(results)
```

注意：与一些早期的 BERT 模型不同，ModernBERT 不使用 Token 类型 ID。在 Hugging Face Hub 上，大多数下游用法与标准 BERT 模型相同，只是可以省略 token_type_ids 参数。

## 引言

BERT 于 2018 年发布（在 AI 领域已是千年之前！），但至今仍被广泛使用：事实上，它目前是 HuggingFace Hub 上下载量第二大的模型，月下载量超过 6800 万次，仅次于另一个为检索而微调的编码器模型。这是因为其仅编码器架构非常适合日常出现的各种实际问题，如检索（例如用于 RAG）、分类（例如内容审核）和实体提取（例如隐私和合规监管）。

终于，6 年之后，我们有了替代品！今天，我们 Answer.AI 和 LightOn（以及合作伙伴！）发布了 ModernBERT。ModernBERT 是一个新的模型系列，在速度和准确性上都是对 BERT 及其后续模型的帕累托改进。该模型借鉴了近年来大语言模型（LLM）研究中的数十项进展，并将其应用于 BERT 风格的模型，包括架构和训练过程的更新。

![](/images/posts/ad93a5bc5524.png)

我们期待 ModernBERT 成为众多仅编码器模型应用场景的新标准，例如 RAG 流水线（检索增强生成）和推荐系统。

除了更快、更准确之外，ModernBERT 还将上下文长度提升至 8k Token（相比之下，大多数编码器仅为 512），并且是首个在训练数据中包含大量代码的仅编码器模型。这些特性开辟了以往开源模型无法触及的新应用领域，例如大规模代码搜索、新的 IDE 功能，以及基于全文检索而非小片段的新型检索流水线。

但为了解释我们具体做了什么，让我们先退一步，看看我们走过的历程。

## 仅解码器模型

近年来 LLM 领域备受瞩目的进展体现在 GPT、Llama 和 Claude 等模型上。这些是仅解码器模型，或称生成式模型。它们生成类人内容的能力催生了令人惊叹的新 GenAI 应用领域，如生成式艺术和交互式聊天。这些引人注目的应用吸引了大量投资，推动了蓬勃发展的研究，并带来了快速的技术进步。而我们所做的，本质上是将这些进展移植回仅编码器模型。

为什么？因为许多实际应用需要一个精简高效的模型！而且它不需要是生成式模型。

更直白地说，仅解码器模型对于许多任务来说太大、太慢、不够私密且成本过高。想想最初的 GPT-1 是一个拥有 1.17 亿参数的模型。相比之下，Llama 3.1 模型拥有 4050 亿参数，其技术报告描述的数据合成和筛选方法对于大多数公司来说过于复杂和昂贵，难以复现。因此，要使用像 ChatGPT 这样的模型，你需要按分计费，并等待数秒才能从你无法控制的重型服务器获得 API 回复。

当然，这些巨型生成式模型的开放式能力意味着，在紧急情况下，你可以将它们用于非生成式或判别式任务，例如分类。这是因为你可以用简单的英语描述一个分类任务，然后……直接让模型进行分类。虽然这种工作流程非常适合原型开发，但一旦进入大规模生产，你就不想再支付原型开发的价格了。

GenAI 的热潮掩盖了仅编码器模型的作用。它们是实际语言处理中的主力军，是当前许多科学和商业应用中真正用于此类工作负载的模型。

## 仅编码器模型

仅编码器模型的输出是一个数值列表（嵌入向量）。可以说，编码器模型不是用文本回答，而是将其"答案"字面编码为这种压缩的数值形式。该向量是模型输入的压缩表示，这就是为什么仅编码器模型有时被称为表示模型。

虽然仅解码器模型（如 GPT）可以完成仅编码器模型（如 BERT）的工作，但它们受到一个关键限制的制约：由于它们是生成式模型，数学上"不允许""偷看"后面的 Token。它们只能向后看。这与仅编码器模型形成对比，后者经过训练，每个 Token 既可以向前看也可以向后看（双向）。它们为此而生，这使得它们在所做的事情上非常高效。

基本上，像 OpenAI 的 O1 这样的前沿模型就像一辆法拉利 SF-23。它是一项显而易见的工程壮举，专为赢得比赛而设计，这就是我们谈论它的原因。但仅仅换轮胎就需要一个专门的维修团队，而且你无法为自己买一辆。相比之下，BERT 模型就像一辆本田思域。它同样是一项工程壮举，但更为精妙，因为它被设计得价格实惠、省油、可靠且极其实用。这就是为什么它们无处不在。

你可以从多个角度来看待这一点。

支持生成式模型：理解表示模型（仅编码器）普遍存在的一种方式是，注意到它们经常与仅解码器模型协同使用，以构建安全高效的系统。

最明显的例子是RAG（检索增强生成）。系统不再依赖LLM（大语言模型）参数中训练的知识，而是使用文档存储来为LLM提供与查询相关的信息。但这当然只是推迟了问题。如果LLM不知道哪些文档与查询相关，那么系统就需要其他流程来选择这些文档？它需要一个足够快速且廉价的模型，用于编码大量信息以使LLM发挥作用。这个模型通常是类似BERT的仅编码器模型。

另一个例子是监督架构，其中可能使用廉价的分类器来确保生成的文本不违反内容安全要求。

简而言之，每当你在部署中看到仅解码器模型时，很有可能系统中也包含一个仅编码器模型。但反之则不成立。

基于编码器的系统：在GPT出现之前，社交媒体和Netflix等平台已有内容推荐系统。这些平台以及搜索引擎和其他领域存在广告定向投放。还有用于垃圾邮件检测、滥用检测等内容分类系统。这些系统并非基于生成式模型，而是基于表示模型（如仅编码器模型）。所有这些系统至今仍在运行，并且规模巨大。想象一下全球每秒有多少广告被定向投放！

下载量：在HuggingFace上，领先的BERT模型之一RoBERTa的下载量超过了HuggingFace上10个最受欢迎的LLM的总和。事实上，目前仅编码器模型每月下载量超过10亿次，几乎是仅解码器模型（每月3.97亿次下载）的三倍。实际上，由ModernBERT等"基础模型"组成的`fill-mask`模型类别（可微调用于其他下游应用）是整体下载量最高的模型类别。

推理成本：上述情况表明，按每次推理计算，仅编码器模型每年执行的推理次数远多于仅解码器或生成式模型。一个有趣的例子是FineWeb-Edu，其中基于模型的质量过滤需要对超过15万亿个Token进行处理。FineWeb-Edu团队选择使用仅解码器模型Llama-3-70b-Instruct生成标注，并使用微调后的BERT模型进行大部分过滤工作。这次过滤耗时6,000个H100小时，按HuggingFace推理端点每小时10美元的定价计算，总成本为60,000美元。另一方面，将15万亿个Token输入流行的仅解码器模型，即使选择成本最低的Google Gemini Flash（推理成本为每百万Token 0.075美元），成本也将超过一百万美元！

## 性能

### 概述

以下是ModernBERT和其他模型在各项任务中的准确率快照，通过标准学术基准衡量——如你所见，ModernBERT是唯一在每个类别中都名列前茅的模型，这使得它成为你所有基于编码器任务的通用选择：

![](/images/posts/3f83e4ab737c.png)

如果你曾在Kaggle上参加过NLP竞赛，就会知道DeBERTaV3多年来一直是冠军的选择。但这种情况已不复存在：ModernBERT不仅是首个在GLUE上击败DeBERTaV3的基础规模模型，而且其内存使用量不到DeBERTa的五分之一。

当然，ModernBERT速度很快。它的速度是DeBERTa的两倍——实际上，在输入长度混合的常见情况下，速度可提升至四倍。其长上下文推理速度几乎是NomicBERT和GTE-en-MLM等其他高质量模型的三倍。

ModernBERT的上下文长度为8,192个Token，是大多数现有编码器的16倍以上。这在RAG管道中至关重要，因为较小的上下文通常会导致分块过小，难以进行语义理解。ModernBERT也是使用ColBERT的最先进长上下文检索器，比其他长上下文模型高出9个百分点。更令人印象深刻的是：这个训练速度极快的模型，仅通过简单调优与其他骨干模型进行比较，就在长上下文任务上超越了广泛使用的检索模型！

在代码检索方面，ModernBERT独一无二。几乎没有可比较的对象，因为此前从未有过在大量代码数据上训练的类似编码器模型。例如，在混合了代码和自然语言的StackOverflow-QA数据集上，ModernBERT专门的代码理解能力和长上下文能力使其成为唯一在此任务上得分超过80的骨干模型。

这意味着全新的应用可能基于这一能力构建。例如，想象一个AI连接的IDE，整个企业代码库都使用ModernBERT嵌入进行索引，从而在所有仓库中快速检索相关代码的长上下文。或者一个代码聊天服务，描述一个集成了数十个独立项目的应用功能如何运作。

与主流模型相比，ModernBERT在检索、自然语言理解和代码检索三大任务类别中几乎都表现更优。虽然在一个领域（自然语言理解）略逊于DeBERTaV3，但速度快了数倍。请注意，ModernBERT与任何其他基础模型一样，开箱即用只能进行掩码词预测。要执行其他任务，应按照这些模板对基础模型进行微调。

与专业模型相比，ModernBERT在大多数任务中表现相当或更优。此外，ModernBERT在大多数任务中比大多数模型更快，并且可以处理长达8,192个Token的输入，是主流模型的16倍。

### 效率

以下是ModernBERT和其他解码器模型在NVIDIA RTX 4090上的内存（最大批处理大小）和推理（每秒千Token数）效率结果：

![](/images/posts/b2d568c384a7.png)

你可能首先注意到，我们是在一款价格实惠的消费级GPU上分析效率，而不是最新难以获取的热门硬件。首先，ModernBERT注重实用性，而非炒作。

作为这一重点的一部分，我们还确保ModernBERT在实际应用中表现良好，而不仅仅是基准测试。这类模型通常只在其最擅长的单一精确尺寸（最大上下文长度）上进行测试。这就是表中"固定"列显示的内容。但实际世界中输入大小各不相同，因此我们努力优化的正是这种性能——"可变"列。如你所见，对于可变长度输入，ModernBERT比所有其他模型快得多。

对于我们认为将成为未来最有价值和最重要应用基础的长上下文输入，ModernBERT比次快模型快2-3倍。同样在"实用性"方面：ModernBERT不需要额外的重型"xformers"依赖，而只需要现在常见的Flash Attention作为依赖。

此外，得益于ModernBERT的效率，它可以比几乎所有其他模型使用更大的批处理大小，并且可以在更小、更便宜的GPU上有效使用。特别是基础规模的效率，可能使直接在浏览器、手机等设备上运行的新应用成为可能。

## 为什么ModernBERT如此"现代"？

现在，我们已经论证了为什么应该给予编码器模型更多关注。作为值得信赖但未被充分重视的"老黄牛"，自2018年的BERT以来，它们得到的更新少得惊人！

更令人惊讶的是：自RoBERTa以来，尚未出现任何无需权衡就能全面改进的编码器（即所谓的"帕累托改进"）：DeBERTaV3在GLUE和分类性能上表现更优，但牺牲了效率和检索能力。其他模型，如AlBERT，或更新的GTE-en-MLM，都在某些方面超越了原始BERT和RoBERTa，但在其他方面有所退步。

然而，自这对组合最初发布以来，我们在如何构建更好的语言模型方面积累了海量知识。如果你使用过LLM，对此一定深有体会：虽然编码器领域罕见帕累托改进，但在解码器领域，这种改进却屡见不鲜——模型在各方面持续进步。而正如我们现已了解的：模型改进只有小部分靠魔法，大部分靠工程。

因此，（希望名副其实的）ModernBERT项目的目标相当简单：将这种现代工程方法引入编码器模型。我们通过三个核心途径实现这一目标：

1. 现代化的Transformer架构
2. 对效率的特别关注
3. 现代数据规模与来源

### 新Transformer，老Transformer

Transformer架构已成为主流，如今绝大多数模型都在使用。但重要的是要记住，Transformer并非只有一种，而是有许多种。它们的共同核心是坚信"注意力就是一切"，并围绕注意力机制构建了各种改进。

ModernBERT深受Transformer++（由Mamba提出）的启发，该架构最早被Llama2系列模型采用。具体来说，我们用改进后的等效模块替换了旧式BERT类构建块，即：

- 将旧的位置编码替换为"旋转位置嵌入"（RoPE）：这使模型能更好地理解词语之间的位置关系，并允许我们扩展到更长的序列长度。
- 将旧的MLP层替换为GeGLU层，改进了原始BERT的GeLU激活函数。
- 通过移除不必要的偏置项来精简架构，使参数预算得到更有效的利用。
- 在嵌入层后添加额外的归一化层，有助于稳定训练。

### 将本田思域升级为赛道赛车

我们已经提到过：编码器不是法拉利，ModernBERT也不例外。但这并不意味着它不能很快。当你驶上高速公路时，通常不会把车换成赛车，而是希望日常代步的可靠座驾能舒适地达到限速。

事实上，对于我们上面提到的所有应用场景，速度都至关重要。编码器在需要处理海量数据的场景中非常流行——即使微小的速度提升也会迅速累积——或者在延迟至关重要的场景中，如RAG。在许多情况下，编码器甚至在CPU上运行，此时效率对于在合理时间内获得结果更为关键。

与研究中的大多数事物一样，我们是站在巨人的肩膀上构建的，并大量利用了Flash Attention 2的速度改进。我们的效率提升依赖于三个关键组件：交替注意力（提高处理效率）、去填充与序列打包（减少计算浪费）以及硬件感知模型设计（最大化硬件利用率）。

#### 全局与局部注意力

ModernBERT最具影响力的特性之一是交替注意力，而非完全的全局注意力。从技术上讲，这意味着我们的注意力机制每3层才关注一次完整输入（全局注意力），而所有其他层则使用滑动窗口，每个Token只关注其最近的128个Token（局部注意力）。由于注意力的计算复杂度会随着每个额外Token的增加而急剧膨胀，这意味着ModernBERT处理长输入序列的速度远快于其他任何模型。

在实践中，它看起来是这样的：

![](/images/posts/8efccdc8b0c5.png)

从概念上讲，其工作原理相当简单：想象你在读一本书。对于读到的每个句子，你需要完全了解整个情节才能理解大部分内容吗（完全全局注意力）？还是说，只要偶尔回顾一下当前章节对主要情节的意义（全局注意力），了解当前章节就足够了（局部注意力）？在绝大多数情况下，答案是后者。

#### 去填充与序列打包

ModernBERT效率的另一个核心机制是其对去填充和序列打包的使用。

为了能够在同一批次中处理多个序列，编码器模型要求它们具有相同的长度，以便进行并行计算。传统上，我们依赖填充来实现这一点：找出最长的句子，然后添加无意义的Token（填充Token）来填充所有其他序列。

虽然填充解决了问题，但方式并不优雅：大量计算最终被浪费在填充Token上，而这些Token不贡献任何语义信息。

![比较填充与序列打包。序列打包（'去填充'）避免了在填充Token上浪费计算资源，并且每批次的非填充Token数量更一致。通过仔细的掩码处理，样本仍然被单独处理。](/images/posts/df61e25565be.png)

去填充解决了这个问题：我们不保留这些填充Token，而是将它们全部移除，并连接成批次大小为1的小批量，从而避免所有不必要的计算。如果你使用Flash Attention，我们的去填充实现甚至比之前的方法更快——之前的方法严重依赖在模型处理过程中对序列进行去填充和重新填充。我们更进一步，引入了自己的去填充实现，并大量利用了Flash Attention在RoPE支持方面的最新进展。这使得ModernBERT只需去填充一次，并可在处理后选择性地重新填充序列，从而比之前的方法提速10-20%。

为了进一步加速预训练，去填充在我们的模型中并非孤军奋战，我们将其与序列打包结合使用。这里的序列打包是合乎逻辑的下一步：由于我们将输入连接成单个序列，而GPU非常擅长并行化，我们希望最大化单次前向模型传递所能榨取的计算效率。为此，我们使用一种贪心算法，将单个序列分组为尽可能接近模型最大输入长度的连接序列。

#### 关注硬件

最后，ModernBERT效率的第三个方面是硬件设计。

我们试图平衡先前研究强调的两个见解：

1. 深而窄 vs 宽而浅：研究表明，具有较窄层的更深模型通常比具有较少、较宽层的浅模型表现更好。然而，这是一把双刃剑：模型越深，其可并行化程度越低，因此在相同参数数量下运行速度越慢。
2. 硬件效率：模型维度需要与GPU硬件良好对齐才能实现最大性能，不同的目标GPU会导致不同的约束。

遗憾的是，没有神奇的配方能让模型在广泛的GPU上同样良好地运行，但有一本优秀的"食谱"：《论模型架构与硬件的协同设计》。其中详细阐述了如何针对特定GPU优化模型架构。我们提出了一种启发式方法，将其方法扩展到一组GPU，同时遵守一组给定的约束。逻辑上，第一步是定义这些约束，在我们的案例中：

- 设定一组在所选GPU上普遍适用的性能约束条件。

#### 过程

我们沿用了原始BERT的训练方案，并借鉴后续工作进行了少量升级：我们移除了下一句预测目标，因为后续研究表明它只会增加开销而没有明显收益，并将掩码率从15%提高到30%。

两个模型都采用三阶段流程进行训练。首先，我们在序列长度为1024的情况下训练1.7万亿个Token。然后，我们采用长上下文适应阶段，在序列长度为8192的情况下训练2500亿个Token，同时通过降低批次大小来保持每批次看到的Token总数大致一致。最后，我们在500亿个Token上进行退火训练，这些Token的采样方式不同，遵循了ProLong强调的长上下文扩展理想混合方案。

采用三阶段训练是我们确保模型全面优秀的方式，这体现在其结果上：它在长上下文任务上具有竞争力，同时不影响处理短上下文的能力……

……但它还有另一个好处：在前两个阶段，一旦预热阶段完成，我们使用恒定的学习率进行训练，仅在最后500亿个Token上进行学习率衰减，遵循梯形（或预热-稳定-衰减）学习率方案。更重要的是：受Pythia启发，我们将发布这些稳定阶段的每一个中间检查点。我们这样做的主要原因是支持未来的研究和应用：任何人都可以从我们任何衰减前的检查点重新开始训练，并根据其预期用途在领域适配数据上进行退火训练！

#### 技巧，关键在于技巧！

如果你已经读到这里，你可能已经习惯了：当然，我们在这里也使用技巧来加快速度。准确地说，我们有两个主要技巧。

先看第一个，这很常见：由于初始训练步骤是在更新随机权重，我们采用了批次大小预热：我们从较小的批次大小开始，这样相同数量的Token可以更频繁地更新模型权重，然后逐渐将批次大小增加到最终的训练大小。这显著加快了模型训练的初始阶段，模型在此阶段学习最基本的语言理解。

第二个技巧则远不常见：受微软Phi系列模型的启发，我们通过拼接方式对较大模型进行权重初始化。这基于以下认识：当我们已经有了一套完美的（如果我们敢这么说的话）ModernBERT-Base权重时，为什么还要用随机数来初始化ModernBERT-Large的初始权重呢？

事实证明，将ModernBERT-Base的权重拼接到ModernBERT-Large中，效果比从随机权重初始化更好。它还有一个额外的好处，即与批次大小预热很好地叠加，从而实现更快的初始训练。
