## 大规模运行LLM推理的挑战

一个可靠的推理平台意味着什么？契约看似简单。可用性是指请求能否被处理。但实际上，不同用例对延迟的要求差异巨大，而这会影响到可用性。最先进的Agent（智能体）无法容忍p95首次Token生成时间（TTFT）和每秒输出Token数（OPTS）的下降。

在多租户LLM服务系统中，同时实现可靠性和低延迟极具挑战性。

可靠性

前沿性能需要最新的GPU以及用于KV缓存传输的高带宽互连。这些计算配置本质上比传统CPU系统更不可靠，而且成本高昂。由于需要全互联通信，在分离式预填充/解码架构中，单个节点的宕机需要对多个其他节点进行重新配置。最高带宽的网络需要单个物理机架内的单脊连接（例如NVL72系统）。这意味着单个数据中心机架内特定系统的故障可能导致大范围的中断。分布式系统中的标准技巧，如多可用区或利用备用实例类型，意味着让昂贵的备用GPU闲置，这是一个成本高昂的选择。过度配置是另一个经典技巧，但由于计算资源供应极度紧张，这极其昂贵且不切实际。因此，系统必须在高负载下保持运行。

在这些限制下，交付速度也必须保持高速——我们的推理需求每年增长数个数量级，在推动增长的同时交付创新功能极具挑战性。图像、视频和安全分类等功能各自需要不同的预处理系统，这些系统都必须独立扩展。

最后，实现一流性能并支持新的模型架构需要涵盖从自定义内核到专有推理引擎的全方位优化。随着架构的细微变化，新的底层软件常常被引入，这些软件在大规模运行时可能以不透明的方式失败，导致从服务器挂起到GPU崩溃等难以调试的场景。

延迟

在多样化的负载模式下控制延迟极具挑战性。这是因为服务请求的成本高度可变且难以预先估计。即使是健康的服务器，在更高负载下处理所有请求也会变慢，这暴露了吞吐量（以及成本效率）与产品所需的最快延迟之间的权衡。这同样可能表现为可靠性问题，因为服务器可能会根据分配给它们的请求组合，非常迅速地进入不健康状态。

![图2：基于某大客户客服Agent（智能体）工作负载的实际并发量与延迟基准测试。](/images/posts/bab4e00487eb.png)

此外，延迟主要由输出Token生成决定，但预先估算成本很困难，因为很难预测模型会生成多长的输出。因此，低延迟服务需要复杂的容量管理、负载均衡和请求优先级排序系统。

## 整体架构

在深入探讨如何解决这些问题的具体细节之前，我们先概述一下我们的服务基础设施。

在数据平面，

- 推理运行时（开源和专有内部引擎）部署在前沿GPU上
- 为了处理跨模型部署的流量，数据平面运行一个路由器（我们称之为Axon），用于在同一模型的副本之间平衡负载，以及一个自动扩缩器，用于调整副本数量。

在控制平面，

- 请求在到达数据平面之前会经过速率限制。
- 基于请求指标，容量管理算法决定每个工作负载获得多少GPU容量，然后由自动扩缩器执行。

### 掌控容量

我们需要能够大致规划容量——我们拥有多少、已售出多少、以及客户正在使用多少。为此，我们引入了一种称为"模型单元"的抽象。如果我们预测一个副本每分钟可以处理固定数量的模型单元（例如100个），我们可以做出以下假设：

- 输入或输出较长的请求会消耗更多模型单元，因为在相同时间窗口内能完成的请求更少。
- 预填充和解码具有不同的吞吐量特性，因此输出较长的请求比输入较长的请求成本更高。

### 基于成本的负载均衡和自动扩缩容

由于请求对服务器的影响高度可变，因此做出近乎最优的路由决策至关重要。通常，负载均衡倾向于依赖统计方法，如P2C（二选一），它基于队列大小估计负载，并利用采样来减少了解所有可能目标的内存和延迟开销。然而，LLM的延迟通常较高，服务器数量少于扩展后的CPU系统，而且错误路由的代价非常严重。因此，LLM服务需要一种不同的方法。

今天，我们使用 Databricks 的自动分片工具 Dicer 来动态路由跨服务器的工作负载。如果没有负载感知路由，长上下文请求会导致个别服务器成为热点，而其他服务器则利用率不足。我们将模型单元与 Dicer 集成，使得路由决策基于模型单元中的服务器负载，而非传统的基于请求的启发式方法。Dicer 还提供有状态会话，使请求路由具有粘性。一个工作负载的请求仅发送到部分服务器，这提高了缓存命中率（对编码 Agent 等延迟敏感的工作负载至关重要），并限制了爆炸半径。

我们还可以调整负载指标，甚至在未来基于更高保真度的成本指标使用更优的路由系统，随着我们不断学习优化。

![图 4：路由器和自动缩放器都消耗服务器负载，因此少量昂贵的长上下文请求可能触发与大量廉价的短请求不同的路由和缩放决策。](/images/posts/3ac54336c2b3.png)

自动缩放也存在类似问题。仅凭待处理请求数量并不能反映真实负载。长上下文请求的激增与短请求的激增看起来相同，而 CPU 和内存指标同样与实际 GPU 利用率不相关。

使用模型单元，我们的自动缩放器可以根据模型单元利用率比率决定是扩容还是缩容。当推理引擎运行接近其最大模型单元（由硬件类型和工作负载形状决定）的某个百分比时，它接近峰值吞吐量，从而触发扩容。反之则触发缩容。这种方法无需为每个模型手动调整自动缩放规则，而是实现了与模型无关的缩放基础设施。

基于 LLM 推理模式构建自动缩放，使我们不必始终扩展到最大副本数。对于流量突发的模型，自动缩放使副本数接近实际需求，与静态峰值配置相比，实现了超过 80% 的 GPU 节省。

## 运行时可靠性

智能路由和缩放提供了良好的基础，但它们无法防止引擎级别的故障。无论我们部署哪种推理引擎（自研引擎或流行的开源选项），在生产规模下都会出现边缘情况和资源争用。我们需要自动检测和恢复故障的机制。

### 检测和恢复静默故障

我们遇到的一种故障模式是静默挂起。涉及边缘情况（结构化输出、多模态输入）的请求可能会触发推理引擎多进程架构中未处理的错误，导致服务器停止响应而不报错。

我们通过定期进行黑盒健康检查来检测此问题：当最近没有实际请求完成时，发送最小的端到端请求。如果健康检查失败，Kubernetes 存活探针会重启服务器。这种方法适用于所有引擎，无论其内部实现如何。

然而，在高负载下，健康检查本身可能超时，导致存活探针杀死实际上健康的服务器。这有引发级联故障的风险。为了解决这个问题，我们为健康检查请求分配最高调度优先级，确保它们即使在重负载下也能完成。通过优先健康检查，从检测到挂起、杀死不健康服务器到恢复的完整周期不到 5 分钟。虚假的存活探针失败从每周几次降为零。

### 处理多模态请求带来的意外负载

当大批量多模态请求到达时，我们看到了来自完全不同来源的错误率和超时激增。

调查发现，请求甚至没有到达推理引擎的核心进程。处理图像请求比纯文本请求消耗更多资源，这不仅是因为 GPU 上运行的额外视觉编码器，还因为 CPU 密集型的图像处理。对于某些模型，图像处理极其缓慢，完全阻塞了事件循环。

将阻塞操作移至单独的线程和进程并不能解决问题；在高图像负载下，请求仍然堆积。因此，我们对 Python 进程进行了性能分析，并有了几个发现：

- 在图像的所有 CPU 操作中，图像处理（调整大小和归一化）比其他操作（如 base64 解码）慢 10 倍。
- 一些 Hugging Face 模型默认使用基于 PIL 的图像处理器，而其他模型则使用更快的基于 Torchvision 的处理器。
- 在容器化环境中，`OMP_NUM_THREADS`（控制 Torch 用于 CPU 操作的 OpenMP 线程数）默认为主机上的 vCPU 数量。在多租户设置中，这是一个糟糕的默认值：主机可能有 192 个 vCPU，但容器只能访问 12 个。结果导致运行的线程数远超可用核心数。这使 CPU 使用率超过容器限制并触发节流。

通过切换到基于 Torchvision 的图像处理器并正确配置 `OMP_NUM_THREADS`，我们维持了更高的 QPS 并充分利用了 GPU。修复上线后，在相同副本数和负载下，每秒完成的请求数跃升了 3 倍以上。CPU 节流消失，服务器运行在更健康的状态。
