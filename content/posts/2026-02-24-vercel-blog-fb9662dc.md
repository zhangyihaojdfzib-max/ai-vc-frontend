如今大多数Agent（智能体）在运行时都能完全访问您的密钥。

随着越来越多的Agent采用编码Agent模式——它们读取文件系统、执行Shell命令并生成代码——它们正演变为由多个组件构成的系统，每个组件需要不同级别的信任。

虽然大多数团队会在单一安全上下文中运行所有这些组件（因为默认工具就是这样工作的），但我们建议以不同的方式思考这些安全边界。

## 所有Agent都开始呈现编码Agent的特征

越来越多的Agent采用编码Agent架构。这些Agent会对文件系统进行读写操作。它们运行bash、Python或类似程序来探索环境。并且，Agent越来越多地通过生成代码来解决特定问题。

即使未被宣传为“编码Agent”的Agent，也将代码生成作为其最灵活的工具。一个生成并运行SQL来查询账户数据的客服Agent，使用的正是相同模式，只是其目标指向数据库而非文件系统。相比仅限于固定工具调用的Agent，能够编写和执行脚本的Agent可以解决更广泛的问题类别。

## 无边界会导致什么问题

设想一个正在调试生产问题的Agent。该Agent读取了一个包含精心构造的提示词注入的日志文件。

隐藏在日志文件中的提示词注入

该注入指示Agent编写一个脚本，将`~/.ssh`和`~/.aws/credentials`的内容发送到外部服务器。Agent生成脚本并执行它，凭证随即泄露。

这就是编码Agent模式的核心风险。提示词注入使攻击者能够影响Agent，而代码执行则将这种影响转化为对您基础设施的任意操作。Agent可能被诱骗从其自身上下文中窃取数据、生成恶意软件，或两者兼而有之。该恶意软件可以窃取凭证、删除数据，或破坏Agent运行机器可访问的任何服务。

攻击之所以成功，是因为Agent、Agent生成的代码以及基础设施都享有相同的访问级别。要在正确的位置划定边界，您需要理解这些组件是什么，以及每个组件应获得何种程度的信任。

## Agent系统中的四个参与者

一个Agent系统包含四个不同的参与者，每个参与者具有不同的信任级别。

### Agent

Agent是由其上下文、工具和模型定义的LLM（大语言模型）驱动的运行时。Agent运行在一个Agent框架内，该框架是您通过标准软件开发生命周期构建和部署的编排软件、工具以及连接到外部服务的集合。您可以像信任任何后端服务一样信任这个框架，但Agent本身容易受到提示词注入和不可预测行为的影响。信息应根据最小知情原则披露，例如，Agent不需要看到数据库凭证就能使用执行SQL的工具。

### Agent密钥

Agent密钥是系统运行所需的凭证，包括API令牌、数据库凭证和SSH密钥。框架会负责任地管理这些密钥，但当其他组件能够直接访问它们时，就会变得危险。下文讨论的整个架构设计，归根结底是哪些组件有途径接触到这些密钥。

### 生成代码执行

Agent创建并执行的程序是最大的变数。生成代码可以做语言运行时允许的任何事情，这使其成为最难推理的参与者。这些程序可能需要凭证来与外部服务通信，但赋予生成代码直接访问密钥的权限，意味着任何提示词注入或模型错误都可能导致凭证被盗。

### 文件系统

文件系统及更广泛的环境是指系统运行的基础，无论是笔记本电脑、虚拟机还是Kubernetes集群。环境可以信任框架，但不能信任Agent在无安全边界的情况下拥有完全访问权限或运行任意程序。

这四个参与者存在于每个Agent系统中。问题在于您是否在它们之间划定安全边界，还是让它们都在同一个信任域中运行。

根据这些信任级别，可以得出以下设计原则：

- 框架绝不应直接向Agent暴露其自身的凭证
- Agent应通过限定范围的工具调用来访问能力，并且这些工具应尽可能狭窄。为特定客户执行支持职责的Agent，应获得一个限定于该客户数据的工具，而不是一个接受客户ID参数的工具，因为该参数容易受到提示词注入的影响。
- 需要自身凭证的生成程序是一个独立的问题，下文架构将对此进行探讨

考虑到这些参与者和原则，以下是我们实践中看到的架构，按安全性从低到高排列。

## 零边界：当前的默认状态

像Claude Code和Cursor这样的编码Agent虽然配备了沙箱，但这些沙箱通常默认关闭。实际上，许多开发者在运行Agent时没有任何安全边界。

在这种架构中，四个参与者之间没有任何边界。Agent、Agent的密钥、文件系统和生成代码执行都共享单一的安全上下文。在开发者的笔记本电脑上，这意味着Agent可以读取`.env`文件和SSH密钥。在服务器上，则意味着可以访问环境变量、数据库凭证和API令牌。生成代码可以窃取其中任何信息、删除数据，并访问环境能够到达的任何服务。框架可能会在执行某些操作前提示用户确认，但一旦工具运行，就没有强制执行的边界。

## 无沙箱的密钥注入

一个**密钥注入代理**位于主安全边界之外，拦截出站网络流量，仅在请求到达其预期端点时注入凭证。框架使用凭证和域名规则配置该代理，但生成代码永远看不到原始的密钥值。

该代理可以防止数据泄露。密钥无法从执行上下文中复制出来并在其他地方重用。但是，代理无法防止在活动运行时期间的滥用。生成的软件在系统运行时，仍然可以使用注入的凭证进行意外的API调用。

秘密注入是一种从零边界架构向后兼容的路径。您可以在不重构组件运行方式的情况下添加代理。其代价在于，除了秘密本身之外，Agent（智能体）和生成的代码在所有其他方面仍共享相同的安全上下文。

### 为何将所有内容一起沙盒化仍不足够

一种自然的直觉是将Agent运行框架和生成的代码包裹在共享的虚拟机或沙盒中。共享沙盒能将两者与更广泛的环境隔离，这确实有用。生成的程序无法渗透到更广泛的基础设施中。

但在共享沙盒中，Agent和生成的程序仍然共享相同的安全上下文。生成的代码仍然可以窃取运行框架的凭据，或者，如果存在秘密注入代理，则通过代理滥用凭据。沙盒保护环境免受Agent的影响，但并不能保护Agent免受其自身生成的代码的影响。

缺失的一环是在独立的计算资源上运行Agent运行框架以及Agent生成的程序，即在具有不同安全上下文的独立虚拟机或沙盒中运行。运行框架及其秘密存在于一个上下文中。文件系统和生成的代码执行则存在于另一个上下文中，且无法访问Agent的秘密。

目前，Claude Code 和 Cursor 都提供了沙盒执行模式，但在桌面环境中的采用率一直很低，因为沙盒化可能导致兼容性问题。在云端，这种分离更为可行。您可以为生成的代码提供一个专门针对Agent需要运行的软件类型定制的虚拟机，这实际上可以提高兼容性。

在实践中，这种分离是一个直接的改变。Agent通过抽象层执行工具调用，这种抽象使得将代码执行路由到独立环境变得自然而然，而无需重写Agent本身。

这两种工作负载具有非常不同的计算特征，这意味着将它们分离可以让您独立优化每一项。Agent运行框架大部分时间都在等待LLM API响应。在 Vercel 上，**Fluid compute** 非常适合这种工作负载，因为计费在 I/O 期间暂停，只计算活跃的 CPU 时间，这使得成本与实际工作成正比，而不是对空闲时间计费。

生成的代码则具有相反的特征。Agent创建的程序是短命的、不可预测且不受信任的。每次执行都需要一个干净、隔离的环境，以便一个程序无法访问另一个程序留下的秘密或状态。像 **Vercel Sandbox** 这样的沙盒产品通过临时 Linux 虚拟机提供这种环境，这些虚拟机每次执行时启动，之后销毁。虚拟机边界正是强制执行安全上下文分离的机制。沙盒内的生成代码没有访问运行框架秘密的网络路径，也无法访问主机环境。

沙盒是双向起作用的。它既保护Agent的秘密不被生成的代码访问，也保护更广泛的环境免受生成的代码所做的任何事情的影响。

最强大的架构是将应用沙盒与秘密注入相结合。这种组合提供了两者单独都无法实现的两个特性：

- Agent运行框架与生成的程序之间的完全隔离，各自运行在自己的安全上下文中
- 生成的代码无法直接访问凭据，它可以在运行时通过注入代理使用秘密，但无法读取或窃取它们。注入的标头会覆盖沙盒代码设置的任何同名标头，从而防止凭据替换攻击。

对于生产级的Agent系统，我们建议将两者结合使用。Agent运行框架作为受信任的软件在标准计算资源上运行。生成的代码在隔离的沙盒中运行。秘密在网络层面注入，永远不会暴露在生成的代码可以直接访问的地方。

这种将Agent计算与沙盒计算分离的架构将成为Agent系统的标准架构。大多数团队尚未进行这种转变，因为默认的工具链并未强制执行它。随着Agent承担更敏感的工作负载，现在就开始划定这些边界的团队将获得显著的安全优势。
