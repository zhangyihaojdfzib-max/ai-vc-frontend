## 概览

- 我们证明其代码安全且正确地实现了标准算法，尤其在后量子密码学领域。
- 我们首先针对 SHA-3 和 ML-KEM 发布了经过验证的代码、规范、属性及证明。

- Agent 通过编写可独立验证的证明，实现了自动化能力的规模化扩展。

## 形式化验证的介绍与动机

加密代码是现代计算的基石。它保护着操作系统、云服务、固件、消息系统以及连接它们的各类协议。微小的错误可能带来巨大的后果：一次算术失误、缺少边界检查或错误的状态转换，都可能破坏原本安全设计的防护能力。

测试和审计仍然至关重要，但仅靠它们并不足够。加密实现通常经过优化、采用恒定时间逻辑、针对特定架构设计，并且有意保持底层特性。实际发布的代码很少像标准中那样简洁清晰：它包含约简运算、位操作、SIMD 内建函数、精心设计的循环以及面向多种环境的可移植层。

形式化验证通过部署机器检查的证明来弥补这一差距，而非仅仅依赖测试。验证并非简单检查代码在通常情况下行为正确，而是针对所有满足既定前置条件的输入，实现一个精确的数学规范。

去年六月，微软宣布将对 SymCrypt（Windows 和 Azure 等产品与服务所使用的加密提供程序）中用 Rust 编写的新算法进行形式化验证。新的加密实现将使用安全的 Rust 编写，然后通过 Aeneas 工具链在 Lean 形式化证明框架中进行验证。这尤其适用于后量子密码学，这类算法需要快速且安全地实现复杂算法。这种组合为我们提供了两层保障：Rust 排除了大部分内存安全漏洞，而 Lean 证明则依据标准推导出的形式化规范，确立了功能正确性。

其成果是为生产级加密算法建立了一套新的验证方法论：在开发者编写代码的同时进行验证，保留面向性能的实现选择，并使证明过程具备足够的可扩展性，以跟上不断演进的代码库。

## SymCrypt 中的验证现状

我们已开源了一个 SymCrypt 分支，其中包含形式化规范和证明。该公共分支将证明产物与其所验证的 Rust 算法实现一同提供，展示了该方法论如何应用于生产级加密代码。SymCrypt 并非一个独立的研究原型；它是微软的开源加密库，被包括 Windows 和 Azure Linux 在内的产品与服务所使用。

此次首次发布包含了 Rust 版 ML-KEM 和 SHA3 代码的完整证明，这些代码目前已用于 Windows 的 Insider 版本中。SymCrypt 正在将相同的基于 Rust、Lean 和 Aeneas 的工作流程扩展到更多 Rust 原生算法，并将其集成到 Windows 和 Linux 的生产版本中，例如包括针对 AES-GCM、FrodoKEM 和 ML-DSA 的经过验证的 Rust 代码。本文的其余部分将以 SymCrypt 的这项工作为具体示例，首先介绍公开标准如何转化为可执行的 Lean 规范。

## 将标准转化为形式化的 Lean 规范

第一步是将算法应实现的功能形式化。对于密码学原语，事实来源通常是公开标准：一份 NIST 规范、一份 IETF RFC，或其它经过仔细审查的算法描述。

在我们的方法中，Lean 规范被设计为尽可能贴近标准。当标准描述一个循环、一次数组更新或一个数学运算时，Lean 模型会尽可能遵循相同的结构。这种语法上的接近性很重要：它使形式化规范更易于审计，因为审查者可以将标准与 Lean 代码并排比较。

Lean 还允许我们编写可执行的规范。这意味着我们可以将形式化模型与官方测试向量进行比对，以捕获转录错误、差一错误或对标准的误解。对于像 ML-KEM 这样的算法，我们可以更进一步，证明高级数学属性，例如展示数论变换的形式化模型对应于相关多项式环上的预期运算。

一个典型的例子是 ML-KEM 中的数论变换（NTT）。标准将该算法描述为对模 q 的 256 个系数进行原地变换，包含三个嵌套循环，使用常数 ζ（= 17）的连续幂次更新成对的系数。

## 将形式化规范与代码连接起来

一旦规范被形式化，下一个挑战就是将其与实现连接起来。我们并不要求开发者用面向验证的语言重写生产级加密代码，也不生成需要产品团队维护的代码。相反，我们验证工程师编写的 Rust 代码，完全按照他们编写的样子进行验证。

Aeneas 通过将 Rust 的中间表示转换为纯 Lean 模型来实现这一点。Rust 的所有权和借用机制在此至关重要。它们使 Aeneas 能够安全地消除大量关于指针别名、活跃性和可变性的推理，这些推理使得对 C 风格代码的验证成本高昂。

例如，一个原地更新数组的 Rust 函数，在 Lean 中会变成一个显式接收并返回函数式数组的函数。可变借用被转换为值变换。这在保留相关行为的同时，为证明工程师提供了一个更易于推理的函数式模型。

一旦进入 Lean，该函数就可以配备一个定理，声明它精炼了形式化规范。换句话说，对于每个满足所需边界和良好形式条件的输入，实现函数返回与源自标准的 Lean 规范相同的数学结果。

这种风格清晰地划分了职责。软件工程师继续编写地道、高性能的 Rust 代码。验证工程师则针对生成的 Lean 模型工作，并证明关于这些模型的定理。Rust 代码和证明并存，但证明的负担并不会迫使代码变得不自然。

回到 NTT 的例子，其 Rust 实现是一个函数 `fn ntt(&mut [u16; 256])`，它使用可变引用来原地更新数组。Lean 的翻译将其纯化为一个函数 `ntt : Array U16 256#usize → Result (Array U16 256#usize)`，该函数直接输出更新后的数组，同时将其包装在 `Result` 类型中，以显式捕获 Rust 函数可能 panic 的事实。

在这种情况下，定理声明：如果数组满足一个良构不变量（确保它表示一个有效的多项式），那么运行 Rust 模型 `ntt` 将返回数学规范 `Spec.ntt` 结果的良构表示，并经过从底层数组到高层多项式的转换。

将这种方法扩展到实际密码学代码中的每个函数需要大量的自动化。Lean 的可扩展性使我们能够构建一个自动化的梯度，使用用于符号执行、算术、数组和位向量推理的策略。体验变得更接近于调试：自动化处理常规的证明义务，而工程师可以在目标未自动关闭时检查和改进证明。

## 支持内联函数和多架构

生产级密码学不能忽视硬件。SymCrypt 必须在从嵌入式、内核环境到云服务的各种环境中运行。它还需要在可用时利用平台特定的指令，包括 SIMD 内联函数和架构特定的优化路径。

因此，一个仅适用于可移植参考实现的验证故事是不完整的。我们需要验证实际发布的代码：包括分发逻辑、优化例程和特定于目标的变体。

内联函数需要稍微不同的处理。一些底层包装器，特别是那些操作原始指针或暴露平台指令的包装器，由经过仔细审查的小型 Lean 规范建模。其他内联函数可以使用 Rust 代码建模，这些代码可以针对硬件参考文档进行测试，然后进行翻译和验证。周围的 safe Rust 代码则针对这些模型进行验证。这保持了较小的可信计算基，同时保留了硬件加速的性能优势。

重要的一点是，验证并不要求放弃优化。该方法旨在保留生产代码的复杂性——包括内联函数、分发和平台特定实现——同时仍然证明一个单一的、可审计的正确性声明。

## 向代码开发者反映形式化保证

只有在开发者能够理解已证明的内容时，形式化验证才能在一个工程组织中规模化。仅仅在仓库中存在一个证明是不够的；保证必须是可见的、可审查的，并且与工程师维护的代码同步。

为了支持这一点，我们通过自动生成的仪表板展示验证结果。这些仪表板以开发者易于理解的术语总结定理：前置条件、后置条件、覆盖的函数、可信模型以及剩余的假设。工程师无需打开 Lean 就能看到哪些内容已被验证。例如，下面是我们 `ntt` 函数的仪表板显示的页面。

规范清晰地展示了 Lean 形式化开发中包含的定理陈述：它将函数输入和前置条件与后置条件分开，放在一条水平线上方，并使用带有链接的完全限定名称来导航到 Rust 和 Lean 定义。

这个反馈循环对于审查关于内联函数、特定目标代码和边界条件的假设特别有用。例如，密码学开发者可以检查定理是否完全捕获了他们期望代码保证的内容，并注意到形式化陈述可能太弱，或者前置条件有误。

仪表板还将验证与持续开发对齐。随着 Rust 代码的变化，Lean 模型和证明可以被重新生成和重放。当证明失败时，这个失败就成为一个信号：要么是实现发生了变化，需要更新证明；要么是更改暴露了与规范之间的真实差异。

这将形式化验证从一次性的研究成果转变为工程工作流程的一部分。

## 智能体证明

最后的要素是超越传统策略的自动化：AI 智能体。Lean 非常适合这一点，因为证明是由一个小的可信内核进行机器检查的。智能体可以提出一个证明脚本，但 Lean 会独立验证该证明是否有效。

我们在两个地方使用智能体。首先，它们帮助将标准翻译成 Lean 规范。因为生成的规范是可执行的，与原始标准一致，针对官方测试向量进行了测试，有数学定理支持，并且比实现简单得多，所以即使智能体帮助起草了它，也可以对其进行彻底审计。

其次，智能体帮助编写和维护证明。有了合适的库、策略、示例和文档，智能体可以处理大量的证明工作：展开生成的模型、应用辅助函数的规范、处理算术义务、以及在重构后修复证明。

这一点尤其强大，因为 Rust 代码和 Lean 证明是分离的。智能体无需注释或修改生产级的 Rust 实现就能使证明通过。它们在证明端操作，并且只有当 Lean 验证了结果，并且最终定理陈述了所需的保证而没有引入未经审查的假设时，结果才会被接受。

在实践中，这改变了验证的经济性。以前需要专家花费数月时间的工作现在可以大大加速。证明工程师的角色从手动编写每个证明转变为设计规范、管理自动化、审查定理陈述以及引导智能体完成其证明。

## 结论

经过验证的密码学常常面临一个艰难的权衡：最强的保证来自专门的工具链、生成的代码以及产品团队难以采用的工作流程。Rust、Lean、Aeneas 和智能体证明自动化让我们得以重新审视这种权衡。

通过按原样验证 Rust 代码、从标准中推导出可审计的规范、支持优化的多架构实现，并将证明结果反馈给开发者，形式化验证可以成为常规密码学工程的一部分，而不是事后的研究练习。

这就是长期愿景：密码学代码既要保持快速、可移植、可维护且由开发者自主掌控，同时还要携带机器可验证的证据，证明其实现了预期要遵循的标准。
