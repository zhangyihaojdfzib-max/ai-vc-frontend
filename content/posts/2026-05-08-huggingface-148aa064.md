# EMO：面向涌现模块化的专家混合预训练

🧠模型：https://huggingface.co/collections/allenai/emo | 📄技术报告：https://allenai.org/papers/emo | 💻代码：https://github.com/allenai/EMO | 📊可视化：https://emovisualization.netlify.app/

![EMO博客文章草稿ryan - Google文档-图片-1 (1)](/images/posts/b1b3ce6bfbca.png)

今天，我们发布了EMO，一种新的专家混合（MoE）模型，采用端到端预训练方式，使得模块化结构直接从数据中涌现，无需依赖人工定义的先验知识。EMO允许你在给定任务中仅使用其专家的一小部分——仅占总数的12.5%——同时保持接近完整模型的性能，并且当所有专家一起使用时，它仍然是一个强大的通用模型。

大语言模型通常作为整体系统进行训练和部署：单个模型被初始化、预训练、微调，并作为一个统一实体提供服务。但应用程序通常只需要部分能力，例如代码生成、数学推理或领域特定知识。随着前沿语言模型常规性地达到数万亿参数，使用和适配完整模型对大多数用户来说变得不切实际，并且会带来不必要的计算成本和内存开销来托管可能根本不需要的参数。

专家混合（MoE）模型似乎是缓解这一约束的自然方式。MoE不是在每一层使用一个大型前馈网络，而是包含许多较小的网络，称为专家，并且仅为每个输入Token激活其中一小部分。原则上，只需要一种能力的任务可以只加载相关专家。

然而，在实践中，现有的MoE仍然需要完整模型才能良好运行。即使在单个输入中，不同的Token通常也会激活不同的专家，因此一个任务在其生成过程中最终可能会使用所有专家。正如我们在论文中所展示的，这部分是因为标准MoE中的专家通常专门处理低层次的词汇模式，如介词或标点符号，而不是高层次领域或能力。因此，小规模的专家子集无法可靠地独立使用。

相反，我们希望MoE模型的专家能够组织成连贯的组，这些组可以被选择性地使用和组合。

在预训练期间鼓励这一点的一种方法是基于预定义的语义领域（如数学、生物学或代码）将Token路由到专家。先前的工作如BTX和我们的FlexOlmo项目已经尝试过这种方法。然而，预定义领域存在重要局限性。它们需要跨预训练语料库的领域标签，这可能存在歧义且获取成本高昂，并且它们可能将过多的人类偏见注入模型允许自我组织的方式中。更重要的是，预先固定领域也固定了模型的模块化结构：如果在推理时出现新的领域或能力，则不清楚应该使用哪些专家。

这就是EMO的用武之地。

我们证明，EMO——一个1B活跃参数、14B总参数（8个活跃专家，共128个专家）的MoE，在1万亿Token上训练——支持选择性专家使用：对于给定任务或领域，我们可以仅使用一小部分专家（仅占总专家的12.5%），同时保持接近完整模型的性能。同时，当所有专家一起使用时，EMO仍然是一个强大的通用模型。相比之下，在相同数据上训练的同等架构的标准MoE，在选择性地使用其专家子集时表现出严重的性能下降。

![EMO博客文章草稿ryan - Google文档-图片-2 (1)](/images/posts/0cb7af12750a.png)

EMO是一种以模块化作为首要目标进行训练的MoE。对于给定领域（例如数学、代码、生物医学），用户可以选择任意大小的一小部分专家，并保持接近完整模型的性能。这将单个模型转变为可组合的架构，使得对于大型稀疏MoE，能够以改进的内存-精度权衡实现灵活的部署。

## 我们如何让模块化涌现？

在MoE中，一个称为路由器的微型网络决定每个Token激活哪些专家。我们希望路由器学习到，来自相似领域的Token应该激活相似的专家子集。我们的关键观察是，来自同一文档的Token通常来自同一领域。因此，我们使用文档边界作为弱监督信号：在训练期间，文档中的所有Token被限制从共享的专家池中选择它们的活跃专家。

![EMO博客文章草稿ryan - Google文档-图片-3 (1)](/images/posts/490038368520.png)

标准MoE和EMO的训练比较（k = 2，n = 10，为简化起见省略共享专家）。（左）在标准MoE中，每个Token独立选择其top-k专家。跨Token，所有专家都被使用。（右）在EMO中，路由器首先为每个文档选择一个专家子集，并且所有Token被限制在此子集内进行路由。这强制了文档内专家使用的一致性，鼓励专家群组形成领域专业化。

例如，在一个总共有10个专家、每个Token有2个活跃专家的MoE中，文档中的所有Token被限制在同一个4个专家的池内进行路由，如上图所示。这个池由路由器自行选择：我们对文档中所有Token的路由器专家偏好进行平均，然后选择最常用的专家作为文档的共享池。不同的文档可以使用不同的池，允许重复出现的专家群组直接从训练数据中涌现。

在实现该系统时，有几个考虑因素：

**负载均衡。** 一个技术挑战是负载均衡。在标准MoE训练中，负载均衡目标用于防止模型崩溃到仅使用少量专家。乍一看，这似乎与EMO的训练目标相冲突：我们明确限制每个文档仅使用一部分专家。

这种冲突源于负载均衡通常应用的尺度。在许多MoE实现中，负载均衡是在局部计算的，通常在一个仅包含少量文档的微批次内。这个局部目标可能会推动同一文档内的Token分散到许多专家，直接与EMO保持文档内专家使用一致性的目标相悖。

为了解决这个问题，我们在跨多个文档的全局范围内应用负载均衡。在这个更大的尺度上，这两个目标变得互补：EMO鼓励同一文档内的Token使用一致的专家池，而全局负载均衡鼓励不同文档共同覆盖所有专家。在实践中，我们发现全局负载均衡对于稳定训练很重要。

**文档池大小。** 文档池大小控制模块化约束的限制程度。较小的池迫使同一文档中的Token共享更紧密的专家集，鼓励更强的模块化；较大的池给予模型更多灵活性，但削弱了约束。

我们不是固定一个池大小，而是在训练期间随机采样它。这防止了EMO过拟合到单一子集大小，并使其在推理时能够支持不同的专家子集大小。

## 基准测试结果

在通用基准测试上，EMO与标准MoE模型的性能相匹配，表明模块化目标并不以牺牲完整模型性能为代价。然而，更重要的问题是，当我们只保留一部分专家时，模型是否仍然能够工作。在这种设置下，我们通过根据少量任务验证数据上的路由使用情况对专家进行排序，保留最常用的专家并丢弃其余部分，来构建任务特定的专家子集。

下图显示，EMO 在选择性使用专家的情况下依然保持稳健。当我们只保留 25% 的专家（32 个专家子集）时，EMO 在所有基准测试中仅损失约 1% 的绝对性能；即使只保留 12.5% 的专家（16 个专家子集），整体下降也仅为约 3%。这一结论在微调前后均成立。相比之下，匹配的标准 MoE 随着专家子集缩小而急剧退化，在最小的专家子集设置中，其性能往往接近或低于随机水平。

## 专家子集专门化于什么？

为了观察 EMO 在训练后实际学到了什么，我们对 12K 预训练文档中前 100 个 Token 的路由器激活进行了聚类。与标准 MoE 的差异非常显著。

EMO 的 Token 聚类对应诸如“健康、医疗与保健”、“新闻报道”、“美国政治与选举”以及“电影与音乐”等类别。标准 MoE 产生的聚类则是“介词”、“专有名词”、“系动词”或“定冠词”等。在 EMO 中，来自同一文档的 Token 大多落在同一个聚类中；而在标准 MoE 中，它们则分散在多个聚类中。

这种对比在单个示例上最容易看出。以一篇健康文章为例：在 EMO 中，几乎每个 Token 都会路由到“健康、医疗与保健”聚类。在标准 MoE 中，最大的聚类是“所有格与定冠词”；模型会将这篇文章与所有恰好使用了“the”或“your”这两个词的文本归为一类，而不管这些文本的内容是什么。

![EMO blog post draft ryan - Google Docs-image-6 (1)](/images/posts/7b3ad14be33b.png)

在 1T Token 上训练的 MoE 的预训练数据 Token 聚类。EMO 聚类对应语义上有意义的领域，来自同一文档的 Token 大致被归为一组。标准 MoE 训练产生的是表层或句法特征的聚类，文档 Token 分散在多个聚类中。

由于 EMO 形成的模块映射到语义领域而非表层特征，你可以选择一个小型专家子集，仍然拥有一个功能正常的模型：该子集对应一种真实的能力。

## 我们发布的内容

我们正在发布完整的 EMO 训练模型、一个在相同数据上训练的匹配标准 MoE 基线，以及训练代码。我们希望这些成果能对其他研究 MoE 中涌现模块化特性的团队有所帮助。
