# 一天内构建领域专用嵌入模型

如果你正在构建RAG（检索增强生成）系统，很可能遇到过这堵墙：一切运行正常……直到出现问题。通用嵌入模型训练的目标是理解互联网内容，而非你的合同、制造日志、专有化学配方或内部分类体系。它们能捕捉广泛的语义相似性，但无法理解你所在领域中至关重要的细粒度区别。当现成模型无法有效捕捉领域特定细微差别时，对嵌入模型进行微调可以提升检索流程的性能。尽管嵌入对RAG性能至关重要，但相关流程仍然出奇地零散，所需技能专业，时间投入也令人望而却步。

仅需一块GPU和不到一天的训练时间，你就能将通用嵌入模型转变为真正理解你领域的模型，且无需手动标注。为了帮助你快速上手，我们还发布了一个**可直接使用的合成训练数据集**，该数据集使用本文所述流程，基于NVIDIA公开文档生成。使用这些数据和方案，我们在Recall@10和NDCG@10指标上均观察到超过10%的提升。**Atlassian**将此方案应用于其**JIRA数据集**进行微调，在单块GPU上将Recall@60从0.751提升至0.951，改进了26%。

## 📚 步骤 1：从文档生成训练数据

微调嵌入模型需要成千上万的（查询，相关文档）对。大多数用例没有现成的此类数据。手动创建成本高昂、速度缓慢，并且常常因标注者对“相关性”的个人理解而产生偏差。与其手动标注数据，你可以使用LLM（nvidia/nemotron-3-nano-30b-a3b）来阅读你的文档，并自动生成高质量的合成问答对。

```shell
nemotron embed sdg -c default corpus_dir=./data/my_domain_docs

```

```json
}

```

注意区别：第一个问题是简单的事实查找。第二个问题需要多跳因果推理。该流程会生成两种类型的问题，并具有可配置的复杂度级别（2-5）和跳数（1-3）。每个问答对随后都会经过质量评估，获得相关性、准确性、上下文支持和清晰度的子分数以及一个总体分数。只有达到阈值的配对才会被纳入训练。

## ⛏️ 步骤 2：挖掘难负例（及其重要性）

如果你只用正例对（查询 + 正确文档）训练嵌入模型，它只能学会区分明显不同的文档，但在困难案例上会失败——那些**看起来**相关但并非正确答案的段落。在实际的检索系统中，这些接近的误判正是导致错误答案的文档。难负例挖掘能找到这些容易混淆的段落，从而使模型学会区分它们。

```shell
nemotron embed prep -c default

```

上述命令自动运行三个子步骤：

### 2a. 训练/验证/测试集划分

生成的问答对被划分为训练集（80%）和测试集（20%）。测试集被格式化为兼容 **BEIR** 的基准，用于步骤5中的标准化评估。

### 2b. 难负例挖掘

使用基础嵌入模型，该流程：

1.  嵌入语料库中的每个查询和每个段落。
2.  计算每个查询与所有段落之间的相似度。
3.  屏蔽掉每个查询已标注的正例文档。
4.  应用边界过滤器：任何得分**高于**最小正例得分95%的非正例文档将被排除。这个排除区域是为了防止假阴性——那些未标注但非常接近正例、可能实际上相关的段落。
5.  从剩余的候选文档中，选择得分最高的前k个文档作为难负例（默认每个查询5个）。

结果：难负例是那些最相似的非正例段落，且其得分安全地低于正例得分上限。它们是当前模型认为高度相关但并非标注答案的段落。

为何有效：用简单负例（完全不相关的段落）训练无法教会模型新知识。用难负例训练则迫使模型学习你领域中至关重要的细微区别。例如，在医学语料库中，一个关于“二甲双胍治疗2型糖尿病的剂量”的问题，其难负例可能是关于“二甲双胍副作用”或“胰岛素治疗1型糖尿病的剂量”——内容接近但关键不同。95%的边界上限防止挖掘器选择过于接近正例的段落，这些段落可能实际上是正确答案，只是在SDG过程中未被标注。

### 2c. 多跳展开

多跳问题引用了多个正例文档。例如，像“第3.2节中的热管理系统如何与第5.1节中描述的功率限制相关联？”这样的问题就有两个正例段落。

展开操作会为每个（查询，正例文档）对创建一个训练样本，因此对比损失会独立处理每个正例。一个带有2个正例文档的问题会变成2个训练样本，每个样本具有相同的困难负例，但正例不同。

最终的输出是一个可用于训练的JSON文件：

```json
{
  "question_id": "q42_0",
  "question": "H100 SXM 的 700W TDP 如何限制多 GPU 节点中的冷却方案选择？",
  "pos_doc": [{"id": "d_a1b2c3"}],
  "neg_doc": [{"id": "d_x7y8z9"}, {"id": "d_m4n5o6"}, {"id": "d_p1q2r3"}, {"id": "d_s4t5u6"}, {"id": "d_v7w8x9"}]
}

```

## 🔍 步骤 3：理解多跳问题及其为何能改进检索

标准的嵌入微调会为每个段落生成一个问题，并训练模型进行匹配。这对于简单的事实查找是有效的，但真实用户会提出跨越多个文档或部分的复杂问题。如果模型只见过单跳训练数据，它将难以检索出这些复杂查询的所有相关段落。

SDG 管道默认生成 1 到 3 跳的问题：

- **1 跳**："H100 SXM 的 TDP 是多少？" —— 由单个段落回答。
- **2 跳**："H100 的 TDP 如何与密集部署中的冷却要求相关联？" —— 需要连接两个段落的信息。
- **3 跳**："考虑到 TDP、冷却限制和机架密度限制，在标准数据中心行中最多可部署多少个 H100 GPU？" —— 综合了三个段落。

每一跳都通过其自身的上下文摘要和段落 ID 进行跟踪，因此训练数据保留了完整的推理链。在展开操作（步骤 2c）之后，每个（问题，相关段落）对都成为一个独立的训练信号，教导模型所有这些段落都与多跳查询相关。

经过微调的模型学会检索上下文相关的文档，而不仅仅是词汇相似的文档。

## 📈 步骤 5：衡量改进效果

微调真的有效吗？让我们通过运行一个标准化评估来找出答案，该评估在留出的测试集上比较基础模型与微调后的检查点：

```shell
nemotron embed eval -c default

```

评估使用 **BEIR 框架**，并在 k = 1, 5, 10 和 100 时计算四个标准信息检索指标：

1.  **nDCG@k**：排序质量 —— 最佳文档是否排在最高位？
2.  **Recall@k**：覆盖率 —— 相关文档中有多少比例出现在前 k 个结果中？
3.  **Precision@k**：准确率 —— 前 k 个结果中实际相关的比例是多少？
4.  **MAP@k**：所有查询的平均精度

一次成功的微调通常能在 <1 天内使 nDCG@10 和 Recall@10 提升 15%。

### 如果指标没有提升怎么办？

该管道便于迭代：

- **SDG 分数质量低？** 检查你的文档质量 —— 干净、格式良好的文本能产生更好的合成数据。尝试使用更大、更强大的 LLM。
- **训练数据不足？** 向你的语料库添加更多文档并重新运行阶段 0。
- **过拟合？** 减少训练轮数或提高质量阈值，只保留最好的训练样本。
- **学习率不合适？** 对于较大的数据集尝试 5e-6，对于非常小的数据集尝试 2e-5。

## 🚀 步骤 6：导出和部署

PyTorch 检查点非常适合评估，但对于生产环境来说太慢。最后两个阶段将转换模型并通过 API 提供服务。

### 导出到 ONNX / TensorRT

```shell
nemotron embed export -c default

```

这将微调后的检查点导出为 ONNX 格式（opset 17）。可选地，它会编译一个 TensorRT 引擎以获得最大推理吞吐量，并具有可配置的优化配置文件，用于批次大小（1–64）和序列长度（3–256）：

```shell
# 仅 ONNX（可在任何地方运行）
nemotron embed export -c default export_to_trt=false

# FP8 量化以进一步加速
nemotron embed export -c default quant_cfg=fp8

```

### 使用 NVIDIA NIM 部署

导出的模型部署在 **NVIDIA NIM** 容器内 —— 这是一个生产就绪的推理微服务，暴露一个 OpenAI 兼容的 `/v1/embeddings` 端点：

```shell
nemotron embed deploy -c default

```

运行后，任何客户端都可以调用它：

```shell
curl -X POST http://localhost:8000/v1/embeddings \
  -H "Content-Type: application/json" \
  -d '{"input": ["What cooling is needed for 8 H100 GPUs in a 2U chassis?"],
       "model": "custom",
       "input_type": "query"}'

```

由于 NIM 提供 OpenAI 兼容的 API，你可以将其直接插入任何使用 embeddings API 格式的现有 RAG 管道中 —— 无需更改代码。

### 验证部署准确性

该管道包含一个 NIM 准确性验证步骤，针对已部署的端点运行相同的 BEIR 评估：

```shell
nemotron embed eval -c default eval_nim=true eval_base=false

```

这可以捕捉到 ONNX/TensorRT 转换可能导致的任何准确性损失。在容差范围内匹配的指标（@1 为 0.03，@5+ 为 0.01）会标记为勾选；超出转换噪声的偏差会被标记出来。

## 整体流程概览

完整的嵌入微调管道可以通过六个命令运行，从原始文档到已部署的模型。

```shell
# 1. 从你的文档生成合成训练数据
nemotron embed sdg -c default corpus_dir=./data/my_docs

# 2. 准备训练数据（拆分数据、挖掘困难负例、展开）
nemotron embed prep -c default

# 3. 微调嵌入模型
nemotron embed finetune -c default

# 4. 评估基础模型与微调后模型
nemotron embed eval -c default

# 5. 导出优化后的模型
nemotron embed export -c default

# 6. 部署模型
nemotron embed deploy -c default

```
