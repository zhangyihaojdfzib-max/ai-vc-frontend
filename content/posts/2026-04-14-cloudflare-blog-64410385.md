# 规模化采用MCP：我们为企业实现更简单、更安全、更经济的MCP部署的参考架构

2026-04-14

- Sharon Goldberg
- Matt Carey
- Ivan Anguiano

![](/images/posts/0f02ba5a456a.png)

我们Cloudflare已将**模型上下文协议（MCP）** 积极采纳为我们AI战略的核心部分。这一转变已远远超出我们的工程组织范围，产品、销售、市场和财务团队的员工现在都在使用智能体工作流来提高日常任务的效率。但采用基于MCP的智能体工作流并非没有安全风险。这些风险包括授权泛滥、**提示词**注入和供应链风险。为了确保这种全公司范围的广泛采用安全无虞，我们整合了来自**Cloudflare One（SASE）平台**和**Cloudflare开发者平台**的一系列安全控制措施，使我们能够在管理MCP的AI使用的同时，不影响员工的工作效率。

在这篇博客中，我们将介绍我们保护MCP工作流的最佳实践，通过整合我们平台的不同部分，为自主AI时代创建一个统一的安全架构。我们还将分享两个支持企业MCP部署的新概念：

*   我们推出**带有MCP服务器门户的代码模式**，以大幅降低与MCP使用相关的**Token**成本；
*   我们描述如何使用**Cloudflare Gateway**进行影子MCP检测，以发现未经授权的远程MCP服务器的使用。

我们还将讨论我们的组织如何部署MCP，以及我们如何使用Cloudflare产品（包括**远程MCP服务器**、**Cloudflare Access**、**MCP服务器门户**和**AI Gateway**）构建我们的MCP安全架构。

## 远程MCP服务器提供更好的可见性和控制

**MCP**是一个开放标准，使开发者能够在AI应用程序与其需要访问的数据源之间建立双向连接。在此架构中，MCP客户端是与**LLM（大语言模型）** 或其他**AI Agent（智能体）** 的集成点，而MCP服务器则位于**MCP客户端**和企业资源之间。

MCP客户端与MCP服务器的分离使得智能体能够自主追求目标并采取行动，同时在AI（集成在MCP客户端）与企业资源的凭证和API（集成在MCP服务器）之间保持清晰的边界。

我们在Cloudflare的员工经常使用MCP服务器来访问各种内部资源中的信息，包括我们的项目管理平台、内部维基、文档和代码管理平台等。

很早我们就意识到，本地托管的MCP服务器是一个安全隐患。本地MCP服务器部署可能依赖未经审查的软件源和版本，这增加了**供应链攻击**或**工具注入攻击**的风险。它们使得IT和安全管理员无法管理这些服务器，而由个别员工和开发者自行决定他们想要运行哪些MCP服务器以及如何保持其更新。这是一场必输的游戏。

相反，我们在Cloudflare有一个集中团队来管理整个企业的MCP服务器部署。该团队在我们的单体代码库内构建了一个共享的MCP平台，提供开箱即用的受治理基础设施。当员工希望通过MCP公开内部资源时，他们首先需要获得我们的AI治理团队的批准，然后复制一个模板，编写他们的工具定义并进行部署，同时免费继承默认拒绝的写入控制、审计日志记录、自动生成的**CI/CD**流水线和**密钥管理**。这意味着搭建一个新的受治理的MCP服务器只需几分钟的脚手架工作。治理已内置于平台本身，这正是采用得以迅速普及的原因。

我们的CI/CD流水线将它们作为**远程MCP服务器**部署在**Cloudflare开发者平台**的自定义域名上。这使我们能够了解员工正在使用哪些MCP服务器，同时保持对软件源的控制。作为一个额外的好处，Cloudflare开发者平台上的每个远程MCP服务器都会自动部署在我们的全球数据中心网络中，因此无论我们的员工身处世界何处，都可以低延迟地访问MCP服务器。

### Cloudflare Access提供身份验证

我们的一些MCP服务器位于公共资源之前，例如我们的**Cloudflare文档MCP服务器**或**Cloudflare Radar MCP服务器**，因此我们希望任何人都可以访问它们。但我们员工使用的许多MCP服务器位于我们的私有企业资源之前。这些MCP服务器需要用户身份验证，以确保除了授权的Cloudflare员工外，其他人无法访问。为了实现这一点，我们为MCP服务器设计的单体代码库模板集成了**Cloudflare Access**作为OAuth提供商。Cloudflare Access保护登录流程并向资源颁发访问令牌，同时充当身份聚合器，验证终端用户的**单点登录（SSO）**、**多因素身份验证（MFA）** 以及各种上下文属性，如IP地址、位置或设备证书。

## MCP服务器门户集中化发现与治理

MCP服务器门户统一了所有AI活动的治理和控制。

随着我们的远程MCP服务器数量增长，我们遇到了一个新的障碍：发现。我们希望让每位员工（尤其是那些MCP新手）能够轻松找到并使用所有可用的MCP服务器。我们的MCP服务器门户产品提供了一个便捷的解决方案。员工只需将其MCP客户端连接到MCP服务器门户，门户便会立即显示他们有权使用的所有内部和第三方MCP服务器。

除此之外，我们的MCP服务器门户还提供集中日志记录、一致的政策执行和**数据丢失防护（DLP护栏）**。我们的管理员可以查看谁登录了哪个MCP门户，并创建DLP规则，防止某些数据（如个人身份信息（PII））与特定的MCP服务器共享。

我们还可以创建策略来控制谁有权访问门户本身，以及每个MCP服务器应公开哪些工具。例如，我们可以设置一个MCP服务器门户，仅允许**财务**部门的员工访问，该门户仅公开位于我们内部代码库前的MCP服务器的只读工具。同时，另一个仅允许使用公司笔记本电脑的**工程**团队员工访问的MCP服务器门户，可以向我们的代码库MCP服务器公开更强大的读写工具。

我们的MCP服务器门户架构概览如上图所示。该门户支持托管在Cloudflare上的远程MCP服务器，以及托管在其他任何地方的第三方MCP服务器。使此架构具有独特高性能的原因是，所有这些安全和网络组件都在我们全球网络内的同一台物理机器上运行。当员工的请求通过MCP服务器门户、Cloudflare托管的远程MCP服务器和Cloudflare Access时，其流量无需离开同一台物理机器。

## 带有MCP服务器门户的代码模式降低成本

经过数月的高容量MCP部署，我们已经支付了相当可观的Token费用。我们也开始认为大多数人在使用MCP时方法有误。

标准的MCP方法要求为通过MCP服务器公开的每个API操作定义一个单独的工具。但这种静态且详尽的方法会迅速耗尽智能体的**上下文窗口**，特别是对于拥有数千个端点的大型平台。

我们此前曾撰文介绍如何利用服务器端代码模式为Cloudflare的MCP服务器提供支持，使我们能够开放Cloudflare API中的数千个端点，同时将Token使用量降低99.9%。Cloudflare MCP服务器仅开放两个工具：一个搜索工具让模型编写JavaScript来探索可用资源，一个执行工具让它编写JavaScript来调用找到的工具。模型按需发现所需内容，而非一次性接收所有信息。

我们非常青睐这种模式，因此决定将其开放给所有人使用。现在，我们已推出在MCP服务器门户中使用"代码模式"的功能。您可以通过一个集中式门户来管理所有MCP服务器，执行审计控制和渐进式工具披露，从而降低Token成本。

其工作原理如下：您无需向客户端开放每个工具定义，所有底层MCP服务器将合并为仅两个MCP门户工具：`portal_codemode_search`和`portal_codemode_execute`。搜索工具让模型能够访问`codemode.tools()`函数，该函数返回每个连接的上游MCP服务器的所有工具定义。模型随后编写JavaScript来筛选和探索这些定义，精确找到所需工具，而无需将所有模式加载到上下文中。执行工具提供一个`codemode`代理对象，每个上游工具都作为可调用函数提供。模型编写JavaScript直接调用这些工具，串联多个操作、筛选结果并在代码中处理错误。所有这些都在MCP服务器门户上由动态工作者驱动的沙盒环境中运行。

以下是一个需要查找Jira工单并使用Google Drive信息更新该工单的Agent示例。它首先搜索合适的工具：

```javascript
// portal_codemode_search
async () => {
 const tools = await codemode.tools();
 return tools
  .filter(t => t.name.includes("jira") || t.name.includes("drive"))
  .map(t => ({ name: t.name, params: Object.keys(t.inputSchema.properties || {}) }));
}
```

模型现在知道了所需的确切工具名称和参数，而无需将完整的工具模式载入其上下文。随后，它编写一个单独的`execute`调用来串联操作：

```javascript
// portal_codemode_execute
async () => {
 const tickets = await codemode.jira_search_jira_with_jql({
  jql: 'project = BLOG AND status = "In Progress"',
  fields: ["summary", "description"]
 });
 const doc = await codemode.google_workspace_drive_get_content({
  fileId: "1aBcDeFgHiJk"
 });
 await codemode.jira_update_jira_ticket({
  issueKey: tickets[0].key,
  fields: { description: tickets[0].description + "\n\n" + doc.content }
 });
 return { updated: tickets[0].key };
}
```

这仅需两次工具调用。第一次发现可用资源，第二次执行工作。若不使用代码模式，相同的工作流程将要求模型预先接收来自两个MCP服务器的每个工具的完整模式，然后进行三次单独的工具调用。

让我们从节省成本的角度来看：当我们的内部MCP服务器门户仅连接到四个内部MCP服务器时，它开放了52个工具，仅其定义就消耗约9,400个Token的上下文。启用代码模式后，这52个工具合并为2个门户工具，消耗约600个Token，减少了94%。关键是，这一成本保持固定。随着我们将更多MCP服务器连接到门户，代码模式的Token成本不会增长。

可以通过在URL中添加查询参数来激活MCP服务器门户上的代码模式。您无需通过常规URL（例如`https://myportal.example.com/mcp`）连接到门户，而是在URL后附加`?codemode=search_and_execute`（例如`https://myportal.example.com/mcp?codemode=search_and_execute`）。

## Cloudflare网关发现并阻止影子MCP

既然我们已经提供了对授权MCP服务器的受控访问，接下来让我们探讨如何处理未经授权的MCP服务器。我们可以使用Cloudflare网关执行影子MCP发现。Cloudflare网关是我们全面的安全Web网关，为企业安全团队提供对其员工互联网流量的可见性和控制。

我们可以使用Cloudflare网关API执行多层扫描，以查找未通过MCP服务器门户访问的远程MCP服务器。这可以通过多种现有的网关和数据防泄漏（DLP）选择器实现，包括：

- 使用网关的`httpHost`选择器扫描已知的MCP服务器主机名（如`mcp.stripe.com`）或使用通配符主机名模式扫描`mcp.*`子域名
- 使用网关的`httpRequestURI`选择器扫描MCP特定的URL路径，如`/mcp`和`/mcp/sse`
- 使用基于DLP的正文检查来发现MCP流量，即使该流量使用的URI不包含`mcp`或`sse`等明显标识。具体来说，我们利用MCP通过HTTP使用JSON-RPC这一事实，这意味着每个请求都包含一个"method"字段，其值为"tools/call"、"prompts/get"或"initialize"等。以下是一些可用于检测HTTP正文中MCP流量的正则表达式规则：

## 面向公众的MCP服务器受到AI应用安全保护

截至目前，我们一直专注于保护员工对我们内部MCP服务器的访问权限。但与其他许多组织一样，我们也设有面向公众的MCP服务器，供客户以Agent（智能体）方式管理和操作Cloudflare产品。这些MCP服务器托管在Cloudflare开发者平台上。（您可以在[此处](https://developers.cloudflare.com/mcp/)查看特定产品的独立MCP列表，或回顾我们通过[代码模式](https://developers.cloudflare.com/mcp/code-mode/)为整个Cloudflare API提供更高效访问的新方案。）

我们认为，每个组织都应为自身产品发布官方的、第一方MCP服务器。否则，您的客户可能会从公共仓库获取未经审核的服务器，这些软件包可能包含**危险的信任假设、未披露的数据收集以及各种未经授权的行为**。通过发布自有MCP服务器，您可以掌控客户所用工具的代码、更新节奏和安全态势。

由于每个远程MCP服务器都是HTTP端点，我们可以将其置于[Cloudflare Web应用程序防火墙（WAF）](https://www.cloudflare.com/learning/ddos/glossary/web-application-firewall-waf/)之后。客户可在WAF中启用[AI应用安全功能](https://developers.cloudflare.com/ai-security/)，自动检测入站MCP流量中的提示词注入尝试、敏感数据泄露和主题分类。面向公众的MCP服务器将像其他Web API一样受到保护。

## 企业环境中MCP的未来

我们希望自身的经验、产品和参考架构能对其他组织有所助益，助力他们在企业范围内广泛采用MCP的进程中稳步前行。

我们通过以下方式保障了自身MCP工作流的安全：

- 为开发者提供模板化框架，使其能在我们的开发者平台上使用Cloudflare Access进行身份验证，从而构建和部署远程MCP服务器
- 通过将全体员工连接至MCP服务器门户，确保基于身份的安全访问机制覆盖所有授权MCP服务器
- 利用AI Gateway调控对支撑员工MCP客户端的LLM（大语言模型）的访问以控制成本，同时在MCP服务器门户中使用代码模式减少Token消耗和上下文膨胀
- 通过Cloudflare Gateway发现影子MCP使用情况

对于正在推进企业MCP部署的组织，我们建议首先将现有远程及第三方MCP服务器置于[Cloudflare MCP服务器门户](https://developers.cloudflare.com/mcp/)之后，并启用代码模式，从而以更低成本、更高安全性和更简化的方式开启企业级MCP部署。
