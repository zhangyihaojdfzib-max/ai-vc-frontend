# Falcon Perception

TL;DR — Falcon Perception 是一个 0.6B 参数的早期融合 Transformer 模型，用于根据自然语言提示词进行开放词汇的定位与分割。该模型使用混合注意力掩码，在一个序列中处理**图像块 + 文本**，并通过一个轻量级、结构化的 Token 接口和轻量级输出头，生成数量可变的实例。在 SA-Co 基准测试中，Falcon Perception 达到了 **68.0 的宏观 F1 分数**（对比 SAM 3 的 62.3），主要差距在于存在性校准（MCC 0.64 对比 0.82）。我们还引入了 **PBench**，这是一个诊断性基准测试，它按能力（属性、OCR 引导的消歧、空间约束、关系）和密集长上下文拥挤场景来分解性能表现。

我们还发布了 **Falcon OCR**，这是一个 0.3B 参数的模型，在 olmOCR 基准测试和 OmniDocBench 上分别获得了 80.3 和 88.6 的分数，同时拥有所有开源 OCR 模型中最高的吞吐量。

这篇文章是对我们所构建内容、为何如此构建以及沿途所学的一个简要、实用的总结。

## 问题：为何感知系统最终都变成了流水线？

许多开放词汇感知系统被构建为模块化流水线：一个（通常是冻结的）视觉主干网络提取特征，一个独立的融合/解码器阶段将其与语言结合，额外的组件处理匹配和后处理。这类设计在许多场景下效果良好，但也伴随着权衡：难以清晰地扩展、难以将改进归因于正确的组件，并且容易在针对每种失败模式添加新修复时积累复杂性。

我们提出了一个更简单的问题：**如果我们选择了正确的注意力模式、输出接口和训练信号，一个单一的早期融合 Transformer 主干网络能否同时处理感知和语言建模？**

在我们的实验中，答案基本是肯定的。本文其余部分将描述主要的设计选择及其背后的依据。

## 架构：早期融合、混合注意力与高效的密集接口

![falcon_inference](/images/posts/fe05cf3e602d.gif)

一个单一的自回归 Transformer 处理图像块、文本和任务 Token 的统一序列。
模型按固定顺序预测对象属性：`<coord>` → `<size>` → `<seg>`。
边界框坐标和尺寸通过专用头解码，并作为傅里叶特征重新注入序列。
高分辨率分割掩码通过 `<seg>` Token 与上采样图像特征的点积生成。

### 一个主干，两种行为

Falcon Perception 的核心是一个密集的 Transformer，它从第一层开始就在**共享的参数空间**中处理图像块和文本 Token。我们摒弃了先有独立视觉主干网络、后有后期融合解码器的做法，而是保留单一主干网络，并依靠掩码和轻量级输出接口来使密集预测问题易于处理。

图像和文本具有不同的结构：像素是二维的，受益于双向上下文，而预测接口本质上是顺序的。我们通过**混合注意力掩码**来解决这个问题：

*   **图像 Token** **双向地**关注所有其他图像 Token，构建全局视觉上下文（类似于视觉编码器）。
*   **文本和任务 Token** **因果地**关注它们之前的所有内容——完整的视觉前缀加上先前的文本。

这使得同一个主干网络在图像 Token 上表现得像一个双向视觉编码器，同时仍然支持在任务 Token 上进行自回归预测。

### 感知链：针对密集输出的由粗到细监督

密集感知不是一个固定大小的预测问题：一张图像可能包含零个实例，也可能包含数百个。自回归生成提供了一个清晰的变长接口，但完全自回归的密集生成（例如，逐个 Token 生成多边形或高分辨率掩码）很快就会变得非常昂贵。

我们使用一个小的结构化接口——**感知链**，它将每个实例分解为三个步骤：

```
<coord> → <size> → <seg>

```

1.  **坐标 Token**：模型首先预测实例的中心——解决**是哪个**对象的问题。
2.  **尺寸 Token**：然后预测空间范围——解决**有多大**的问题。
3.  **分割 Token**：最后，生成一个单一的嵌入，当它与上采样的图像特征进行点积时，会产生一个全分辨率的二值掩码。

这个顺序是经过深思熟虑的。首先确定几何形状可以减少歧义（“哪个实例？”），并使掩码预测步骤更接近于在已确定对象基础上的像素细化。

### 专用头，最小开销

主干网络是共享的，而解码则使用针对输出类型定制的轻量级头：

*   **坐标与尺寸头**使用**傅里叶特征编码**：通过随机高斯投影将连续坐标映射到高维正弦空间。这克服了神经网络的频谱偏差，比单纯的离散分箱能实现更精确的定位。解码后的坐标作为条件重新注入序列，供后续 Token 使用。
*   **分割头**计算 `<seg>` Token 的隐藏状态与内容感知的上采样图像特征之间的**点积**。因为 `<seg>` Token 是在几何信息之后产生的，并且可以访问早期融合的视觉上下文，所以我们可以避免在基于解码器的实例分割训练中常见的独立掩码查询机制和匈牙利匹配。

## PBench：一个旨在分离缺失能力的基准测试

现有的指代表达式基准测试（如 RefCOCO）已经饱和——模型通常能达到 90% 以上——并且它们混淆了**出错的原因**。模型失败是因为无法读取文本？无法理解空间关系？还是无法处理拥挤场景？

我们引入了 **PBench**，这是一个诊断性基准测试，它根据所需的主要能力来分离样本：

每个样本都针对**一种**主要能力：OCR 提示词避免使用空间限定词，空间提示词避免使用图像内文本消歧器。这产生的是一个能力**剖析图**，而不是一个单一的不透明分数，并且更容易决定下一步投资于何处（数据、训练课程或训练后处理）。

## 训练：知识蒸馏、大规模数据与三阶段方案

### 多教师知识蒸馏

Falcon Perception 并非从随机权重开始训练（在我们的消融实验中，这对分割任务来说不稳定），而是通过**多教师知识蒸馏**进行初始化。两位强大的视觉教师贡献了互补的信号：

*   **DINOv3 (ViT-H)**：对分割至关重要的强大局部特征
*   **SigLIP2**：用于开放词汇理解的语言对齐特征

经过蒸馏的初始化在 ImageNet-1k 上实现了 74.25% 的零样本准确率，在 Pascal VOC 上实现了 85.11% 的线性探测 mIoU，为后续的感知特定训练提供了强大的视觉基础。

### 数据：5400 万张图像，1.95 亿个正例表达，4.88 亿个困难负例

我们通过一个多阶段的流水线构建训练集：

1. 通过DINOv3嵌入对网络爬取的图像进行层次聚类，确保概念覆盖的均匀性。
2. VLM驱动的列表生成功能为每张图像生成密集的对象描述，并按PBench复杂度等级分类（60%基础，40%高级）。
3. 负样本挖掘生成语义、视觉和细粒度难负样本，以对抗幻觉。
4. 集成共识——SAM 3、Qwen3-VL-30B和Moondream3必须达成一致（IoU > 0.8）才能自动接受。
5. 人工验证——存在分歧的样本交由标注员处理，从而回收那些令自动化系统困惑的难样本。

我们严格保持正负样本1:1的比例。这使得存在性校准成为首要目标：模型应能可靠地说出“不存在”，而不仅仅是在有把握时绘制掩码。

### 三个阶段（总计700 GT）

**阶段 1 — 上下文内列举（450 GT）：** 模型学习自回归地列举场景清单——预测文本表达式及其位置。查询之间的完全因果注意力机制使得模型能够学习对象的共现关系（“叉子，然后是刀子，然后是盘子”）。这构建了广泛的场景理解能力。

**阶段 2 — 任务对齐（225 GT）：** 修改注意力掩码，使查询之间无法相互看到，模拟推理时的独立查询。文本Token的损失被掩蔽，将梯度信号完全集中在存在性分类和定位上。此阶段从“场景理解”过渡到“回答这个具体问题”。

**阶段 3 — 长上下文微调（10 GT）：** 一个短暂的阶段，将每个表达式的掩码限制提高到600，并使用极小的恒定学习率。这使模型适应极端拥挤的场景，同时不遗忘先前的能力。

通过消融实验验证的关键设计选择：

- **Muon优化器**用于专用头部（对比AdamW）——在SA-Co检测上带来+4.8分的提升
- **实例的光栅排序**（对比随机/大小排序）——在SA-Co上比随机排序高出+10分
- **Gram特征正则化**——防止蒸馏特征漂移，将分割性能提升+1.5分
- **跨等级的全局损失归一化**——纠正了FSDP中可变长度打包序列带来的偏差

## 结果

### SA-Co：一流的掩码质量

在SA-Co开放词汇分割基准测试中，Falcon Perception（0.6B参数）实现了**68.0的宏观F1分数**，而SAM 3为**62.3分**，在属性密集（+8.2）、食品饮料（+12.2）和运动器材（+4.0）等子集上均有大幅提升。同时，Falcon Perception在存在性校准方面落后于SAM 3（MCC：0.64 vs 0.82），这是最明确的待改进方向。

### PBench：随提示词复杂度扩展

这是早期融合设计显示出最大差异的地方：

对于简单对象，差距不大。随着提示词变得更加组合化——需要OCR引导的消歧、空间约束或关系绑定——差距会扩大。

在我们的PBench密集子集中，Falcon Perception（0.6B）显著优于通用VLM基线（例如，在我们的评估设置中，对比Qwen3-VL-30B的8.9分，我们得到72.6分），并且在空间和关系层级上匹配或超越了8B模型。

#### 定性结果：OCR、空间、关系和密集

随着提示词变得更加组合化——需要OCR引导的消歧、空间约束、关系绑定或扩展到数百个实例——早期融合的优势在视觉上变得清晰：

- **OCR引导的定位（第2级）：** 当区分信号是物体上写的文本时，Falcon Perception能正确读取，而SAM 3无法区分。
- **空间理解（第3级）：** 当提示词指定空间关系时，Falcon Perception会形成一个连贯的2D场景地图。
- **关系推理（第4级）：** 当目标通过交互而非外观定义时，Falcon Perception能够理解场景图。
- **密集场景：扩展到数百个实例：** 当场景极其拥挤时，自回归接口特别有用，而固定查询的解码器可能会遇到实际限制。

## Falcon OCR：将早期融合扩展到文档理解

现代OCR已经远远超越了从清晰扫描件中提取文本。今天的系统必须能够一次性处理多栏布局、数学公式、表格、图表和多语言内容。大多数有竞争力的OCR VLM采用熟悉的配方来解决这个问题：一个视觉编码器馈送给一个独立的文本解码器，再加上特定任务的粘合层。这些系统有效，但它们往往体积庞大（1B–3B+参数）。

我们选择了不同的路径：**重用Falcon Perception中相同的早期融合密集Transformer**，但专门为OCR从头训练一个更小的**0.3B参数**变体。结果是**Falcon OCR**——一个单一的主干网络，在共享参数空间中处理图像块和文本Token，使用相同的混合注意力掩码（图像Token为双向，文本Token为因果），并通过提示词而非额外模块来切换任务。

我们从头开始训练（没有多教师蒸馏），因为OCR所需的视觉特征——细粒度的字形识别、笔画级别的区分——与分割有用的对象级特征有本质不同。从头开始让主干网络能够从零开始发展出针对文本优化的表征。

### 训练

我们在一个精心策划的英语混合数据集上进行训练，涵盖三个核心任务：通用文档文本解析（数字PDF、旧扫描件、打字机文档）、数学和科学公式识别以及表格结构识别。该混合数据集还包括手写体、真实世界场景文本以及从渲染的LaTeX和HTML源生成的合成样本。训练目标是对结构化文本输出进行纯下一个Token预测。

训练分两个阶段进行：一个长期的**预训练**阶段，使用恒定学习率，模型学习所有元素类型的核心OCR能力；随后是一个短暂的**余弦衰减微调**阶段，学习率衰减到接近零。

### 基准测试结果

我们在olmOCR（针对多样化输入的二进制正确性检查）和OmniDocBench（全页解析的连续性指标）上进行了评估。所有对比模型的参数量都显著更大和/或使用了专有基础设施。Falcon OCR仅以0.3B参数就在olmOCR上达到了80.3%的准确率，与顶级系统的差距在1.7个百分点以内，并且在**多栏**（87.1%）和**表格**（90.3%）任务上领先所有模型。在OmniDocBench上，其总体得分为88.64，超过了DeepSeek OCR v2、GPT 5.2和Mistral OCR 3。

### 服务吞吐量

Falcon OCR的参数量为0.3B，比0.9B级别的OCR视觉语言模型**大约小3倍**，这直接转化为更高的服务吞吐量。在单个A100-80GB GPU上使用vLLM在高并发场景下测得：

其紧凑的模型体积和vLLM集成（连续批处理、分页注意力、优化的CUDA内核）使其适用于需要处理数百万页文档的大规模文档数字化场景。

### 我们从结果中看到的

更广泛地说，这些结果表明，对于OCR任务而言，早期融合的单栈Transformer架构是"视觉编码器加文本解码器"方案的一个可行替代方案。一个主干网络、共享的参数空间、一个解码接口，以及更好的数据和训练信号，而非日益复杂的处理流程。我们希望这能鼓励更多朝这个方向的研究工作。

## 推理：快速、实用且开源

本次发布包含一个基于**PyTorch FlexAttention**构建的推理栈，这使得表达自定义注意力模式并高效服务打包的变长序列变得切实可行。

### 分页推理引擎

- **分页KV缓存**，带有虚拟页表（无填充导致的内存浪费）
- **连续批处理**：新序列在生成过程中加入，已完成的序列立即释放页面
- **CUDA图捕获**用于解码循环
- **后台分词**与GPU计算重叠
- **高分辨率特征缓存**：使用固定内存缓冲区的LRU缓存，用于上采样图像特征的异步GPU-CPU传输——对同一图像的后续查询可跳过昂贵的上采样步骤

在我们的H100配置中，典型延迟约为：预填充~100ms，上采样~200ms（若缓存则为0ms），以及少量实例的解码~50ms。（这些数字取决于分辨率、序列长度和预测实例的数量。）
