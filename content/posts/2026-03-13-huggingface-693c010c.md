# 超越语义相似性：NVIDIA NeMo Retriever 推出通用性智能体检索流程

我们激动地宣布，NVIDIA NeMo Retriever 团队开发了一种新的智能体检索流程，该流程已在 ViDoRe v3 流程排行榜上正式获得第一名。此外，完全相同的流程架构在要求极高、注重推理的 BRIGHT 排行榜上也取得了第二名的成绩。

在快速发展的 AI 检索领域，许多解决方案都是高度专业化的，旨在特定的、狭窄的任务上表现异常出色。然而，现实世界的企业应用很少能拥有完美整理的单领域数据。它们需要的系统能够无缝适应各种挑战——从解析复杂的视觉布局到执行深度的逻辑推理。

这就是为什么我们在设计中优先考虑**通用性**。我们没有依赖特定数据集的启发式方法，而是构建了一个智能体流程，能够动态地根据手头的数据调整其搜索和推理策略。这使得我们能够在截然不同的基准测试中提供最先进的性能，而无需对底层架构进行任何更改。

以下是我们的构建过程。

## 动机：为何语义相似性不足

多年来，基于语义相似性的密集检索一直是查找信息的标准方法。然而，随着检索应用范围的扩大，查找相关文档已不仅仅依赖于语义相似性。复杂的文档搜索需要推理技能、对现实世界系统的理解以及迭代探索。

存在一个根本性的差距：LLM（大语言模型）擅长思考和推理，但无法一次性处理数百万份文档。相反，检索器可以轻松筛选数百万份文档，但推理能力有限。智能体检索通过在 LLM 和检索器之间创建一个主动的、迭代的循环来弥合这一差距。

## 工作原理：智能体循环

![智能体检索流程概览](/images/posts/17eba3f439a9.png)

智能体检索流程概览

我们的智能体检索流程依赖于 ReACT 架构。智能体不是执行单一的"一次性"查询，而是迭代地搜索、评估和改进其方法。

智能体利用内置工具，如 `think` 来规划其方法，`final_results` 来输出给定查询所需的确切文档，以及 `retrieve (query, top_k)` 工具来探索语料库。通过这个循环，我们观察到成功的搜索模式自然出现：

- **生成更好的查询**：智能体根据新发现的信息动态调整其搜索查询。
- **持续改写**：它不断改写查询，直到找到有用的信息。
- **分解复杂性**：它将复杂的、多部分的查询分解为多个目标明确的简单查询。

最后，为了综合迭代发现的结果，智能体调用 `final_results` 工具输出最相关的文档，并根据它们与给定查询的相关性进行排序。作为安全网——例如，当智能体达到最大步数或上下文长度限制时——流程会回退到 Reciprocal Rank Fusion（RRF），该算法根据文档在智能体轨迹中所有检索尝试中的排名进行评分。

## 为速度和规模而设计

众所周知，智能体工作流程速度慢且资源密集。为了使该流程能够在排行榜规模的评估中可行，我们必须重新思考 LLM 智能体和检索器之间的通信方式。

最初，检索器通过 Model Context Protocol（MCP）服务器暴露给智能体——这是一个自然的选择，因为 MCP 正是为让 LLM 访问外部工具而设计的。但在实践中，这种架构对实验速度造成了叠加负担。每次运行都需要启动一个单独的 MCP 服务器，将正确的数据集语料库加载到 GPU 内存中，并协调客户端和服务器的生命周期。这些步骤中的每一步都可能导致静默配置错误，或在大量请求下服务器冻结。网络往返增加了每次检索调用的延迟，并且管理双进程设置的整体认知负担使得其他团队难以采用和迭代该流程。

为了解决这个问题，我们用进程内的线程安全单例检索器取代了 MCP 服务器。该单例一次性加载模型和语料库嵌入，使用可重入锁保护所有访问，并向任意多个并发智能体任务暴露相同的 `retrieve()` 接口——实现了 MCP 服务器的关键优势（从多个线程安全、共享地访问驻留在 GPU 上的检索器），而不会产生网络序列化开销或需要单独的服务器进程。这一架构更改消除了整个类别的部署错误，并显著提高了 GPU 利用率和实验吞吐量。

## 跨基准测试的通用性与专业性

现代检索评估中的一个常见观察是，针对某一特定类型任务高度优化的解决方案，在应用于完全不同的领域时，往往会出现性能差距。

我们在注重推理的 BRIGHT 排行榜上以 NDCG@10 为 50.90 的成绩排名第二。该排行榜上的第一名解决方案 INF-X-Retriever 取得了令人印象深刻的 63.40。然而，为了测试跨领域适应性，我们在 ViDoRe v3（一个专注于视觉丰富且多样化的企业文档的数据集）上对 INF-X 流程（结合了与我们的智能体流程中使用的相同 `nemotron-colembed-vl-8b-v2` 嵌入模型）进行了基准测试。在这个不同的任务上，其性能为 NDCG@10 62.31，低于密集检索分数 64.36。换句话说，INF-Query-Aligner 在 ViDoRe v3 上并未超越密集检索基线。

相比之下，我们相同的智能体流程（将 Opus 4.5 与 `nemotron-colembed-vl-8b-v2` 配对）在 ViDoRe v3 上以 69.22 的分数获得了第一名。

这凸显了我们方法的一个核心优势：**通用性**。我们的智能体循环不依赖于特定数据集的启发式方法或查询重写器/对齐器，而是自然地使其策略适应手头的数据集，无论该数据集需要多步逻辑推理还是解析复杂的视觉布局。

## 消融研究：闭源模型与开源模型

我们进行了广泛的消融研究，以理解前沿闭源模型与开源替代方案之间的权衡：

- **模型选择**：在 ViDoRe v3 上，将 Opus 4.5 替换为 `gpt-oss-120b` 导致准确率小幅下降（NDCG@10 从 69.22 降至 66.38），并且检索调用次数大大减少。在 BRIGHT 上，差距更大，表明深度推理任务仍然极大地受益于像 Opus 4.5 这样的前沿模型。
- **嵌入模型**：将智能体与专用嵌入模型配对（ViDoRe 使用 `nemotron-colembed-vl-8b-v2`，BRIGHT 使用 `llama-embed-nemotron-reasoning-3b`）产生了最佳结果，证明强大的基线检索器为智能体提供了更高的性能上限。
- 同样有趣的是，智能体可以缩小强嵌入模型和弱嵌入模型之间的差距。例如，在 ViDoRe 上，较强的 `nemotron-colembed-vl-8b-v2` 与较弱的 `llama-nemotron-embed-vl-1b-v2` 在密集检索中的差距约为 8.5，但当与 `gpt-oss-120b` 智能体结合时，差距缩小到约 4。类似地，在 BRIGHT 上，`llama-embed-nemotron-reasoning-3b` 比 `llama-nemotron-embed-vl-1b-v2` 好约 19 分，但当与 `gpt-oss-120b` 智能体结合时，领先优势缩小到约 7.5。

## 自主性的代价与未来展望

天下没有免费的午餐。智能体检索比标准的密集检索成本更高、速度更慢。观察我们的ViDoRe v3结果，智能体平均每个查询耗时136秒，每个查询大约消耗76万个输入Token和6300个输出Token。（注：顺序延迟是在单张A100 GPU上通过单次并发Claude API调用测量的——即不同时处理多个查询——以反映真实使用场景中的实际搜索时间）。

然而，我们相信对于高风险、复杂的查询任务，智能体检索是一种极具可行性的方法。我们接下来的工作重点将聚焦于降低成本：我们正积极研究如何将这些智能体推理模式提炼到更小型的、专门化的开源权重智能体中。通过微调更小的模型来原生地编排“思考-检索”循环，我们的目标是以极低的延迟和成本实现Opus级别的准确性。
