今天，我们宣布对 AI Gateway 进行重大升级。作为本次发布的一部分，AI Gateway 现已作为 Unity AI Gateway 成为 Unity Catalog 的一部分。这将 Unity Catalog 的治理模型扩展到了智能体 AI 领域，因此您可以对智能体如何访问 LLM（大语言模型）以及与 MCP 服务器和 API 等工具交互，应用相同的权限、审计和策略控制。

当一个 AI 智能体回答客户问题时，会发生以下情况：它调用一个 LLM 来解释查询，通过 MCP 服务器从 Salesforce 拉取订单历史记录，通过内部 API 检查实时运输数据，然后再次调用 LLM 来起草回复。总耗时：不到一秒。而对于谁访问了什么数据、调用了哪些系统、是否遵循了策略的全面可见性：几乎为零。

改变的不仅仅是工具——更是架构。AI 智能体现在编排跨模型和系统的多步骤工作流，通常在每个步骤都会触及敏感数据。这可能意味着查询数据库、调用外部 API，或者使用像 Cursor、Codex 或 Claude Code 这样的编码智能体来生成或修改代码。

这就引发了新的问题：谁授权了每个操作？哪些数据与哪个模型共享了？策略是否得到了一致的执行？如果出现问题，您能追踪完整的链条吗？

传统的治理工具并非为此而生。它们各自为政，无法提供贯穿智能体操作全生命周期的统一视图。

通过本次发布，我们正在扩展 Unity Catalog 的治理能力，以覆盖 AI 智能体。Unity AI Gateway 让您可以控制 LLM 访问，治理智能体如何使用 MCP 服务器和 API，并在模型和工具间应用一致的策略。这包括对 MCP 治理的新支持，因此您可以控制哪些智能体可以访问哪些外部系统，并跟踪这些数据的使用方式。如需深入了解，请阅读我们关于如何安全地将智能体连接到外部 MCP 的[操作指南博客](https://www.databricks.com/blog/connect-agents-external-mcp-servers-securely)。

您还可以获得跨 LLM 和 MCP 调用的详细可观测性，以及跨模型、团队和工作流的精细成本跟踪。此外，Unity AI Gateway 提供了一种跨模型工作的统一方式，内置了故障转移、速率限制和防护栏，帮助您在生产环境中可靠地运行智能体。

### 细粒度权限与防护栏

细粒度权限和防护栏可以首先防止不该发生的事情发生。

工具的细粒度访问控制

当智能体调用 MCP 服务器访问内部系统时，Unity AI Gateway 支持**代表用户**执行。MCP 以请求用户的精确权限执行，而不是共享服务账户的权限。如果用户无法访问 Salesforce 记录，那么智能体也无法访问——即使拥有提升的权限。

由 LLM 评判器驱动的灵活防护栏（Beta）

Unity AI Gateway 的防护栏采用提示词 + 模型的方法——配置它们在请求、响应或两者上运行：

- **PII 检测与编辑**：在数据到达外部模型之前检测并屏蔽电子邮件、SSN、电话号码
- **内容安全**：使用可定制的过滤器阻止有毒、有害或不适当的内容
- **提示词注入检测**：捕获试图覆盖系统指令的越狱尝试
- **数据外泄防护**：防止训练数据或专有内容暴露
- **幻觉防护**：根据基础来源验证响应
- **自定义防护栏**：使用自定义提示词和模型定义您自己的规则

每个防护栏都由一个可编辑的提示词和可配置的模型支持——而非僵化的预构建逻辑。当违反规则时，Unity AI Gateway 可以拒绝请求或屏蔽敏感数据。所有操作都会被记录以供审计。此功能目前正在推出，并将在下周内在所有支持的区域可用。

### 端到端可观测性

当 AI 智能体投入生产时，三个团队需要答案：FinOps 想知道什么在消耗成本，工程团队需要调试故障，安全团队需要审计追踪。Unity AI Gateway 通过同一套统一的日志记录基础设施，为每个团队提供他们所需的信息。

![AI Gateway 使用分析仪表板](/images/posts/b79019aaa40b.gif)

**对于 FinOps：按对您重要的维度跟踪成本**

每个请求都会被记录到 **Unity Catalog 系统表**中，并包含实际美元成本——不仅仅是 Token 数量。预置吞吐量运行时间、按 Token 付费的使用量以及外部模型定价都会自动计算。按照您组织的预算方式进行成本细分：

- **端点标签**：按团队、环境或成本中心分组
- **请求标签**：为代理到最终客户的 SaaS 平台提供动态归因
- **身份**：按用户或服务主体聚合——将支出映射到预算负责人
- **模型和供应商**：跟踪哪些模型（Opus 与 Sonnet）和供应商（Anthropic 与 OpenAI）驱动了成本

**对于工程团队：用于调试的完整载荷**

启用 **推理表**，将完整的请求/响应载荷、延迟、状态码和错误捕获到 Delta 表中。当智能体失败时，精确追踪发送了什么提示词、模型返回了什么、以及在哪里出错——并使用 Genie Code 和 MLflow 等工具快速调试和解决问题。

![AI Gateway 推理表](/images/posts/d5a4d3400a48.gif)

**对于安全团队：完整的审计追踪**

每个请求都会记录请求者身份、时间戳，对于 MCP 调用，还会记录连接名称、HTTP 方法以及调用是否是代表用户执行的。Unity Catalog 权限控制谁可以看到什么。

单一的日志记录基础设施支撑了三个关键用例——建立在您拥有和控制的 Delta 表之上。

## 面向生产的可靠性与灵活性

Unity AI Gateway 根据您的应用程序需求，为您提供调用模型的灵活性。

**用于无缝切换供应商的统一 API（Beta）**

如果可移植性很重要——也确实应该重要——请使用 Unity AI Gateway 的 OpenAI 兼容 API。您的代码在每个供应商间保持不变。编写一次应用程序，然后通过更新端点配置在任何模型之间切换。无需更改代码，无需重新部署。

**自动故障转移保持系统运行（Beta）**

配置**备用模型**，Unity AI Gateway 会自动处理故障。如果您的主模型达到速率限制或返回错误，请求会按顺序路由到您的备用模型，直到一个成功为止。Opus 配额耗尽？流量会回退到 Sonnet。供应商遇到中断？您的应用程序会路由到替代方案。无需手动干预，无停机时间。

最后，Unity AI Gateway 使您能够在端点、用户或组级别设置速率限制，以防止成本失控，并在问题发生前保护您的 SLA。
