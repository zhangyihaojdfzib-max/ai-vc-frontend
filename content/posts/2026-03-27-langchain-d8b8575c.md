作者：Victor Moreira，LangChain 部署工程师

本清单是《Agent 可观测性赋能 Agent 评估》一文的实用伴侣。该文阐述了为何 Agent 评估不同于传统软件测试，介绍了核心的可观测性原语（运行、追踪、线程），并解释了它们如何映射到评估层级。如果你是 Agent 评估的新手，请先阅读那篇文章。

本文聚焦于“如何做”，提供了一份构建、运行和交付 Agent 评估的逐步清单。

从能给你提供信号的最简单评估开始。即使你的架构仍在变化，几个测试 Agent 是否完成其核心任务的端到端评估也能立即为你提供一个基线。只有当你有证据表明更简单的方法遗漏了真正的失败时，才增加复杂性。

### 深入探讨

#### 在构建任何评估基础设施之前，手动审查 20-50 条真实的 Agent 追踪记录

使用 LangSmith 从追踪到标注队列，再到数据集与实验。

在构建任何基础设施之前，花 30 分钟仔细阅读真实的 Agent 追踪记录。你从中了解到的失败模式将比任何自动化系统都多。LangSmith 的追踪和标注队列非常适合此目的。

#### 为单个任务定义明确无误的成功标准

如果两位专家无法就通过/失败达成一致，那么任务需要改进：

*   不明确的成功："很好地总结这份文档。"
*   明确的成功："从这份会议记录中提取 3 个主要行动项。每个应少于 20 个词，如果提到负责人则需包含。"

#### 将能力评估与回归评估分开

两者你都需要，因为它们目的不同。能力评估通过衡量困难任务的进展来推动你的 Agent 向前发展，而回归评估则保护已正常工作的部分。如果不加以区分，你将要么因为只守护现有行为而停止改进，要么因为只追求新能力而引入回归。

*   **能力评估**回答"它能做什么？" 从低通过率开始，给你一个需要攀登的山峰。
*   **回归评估**回答"它还能正常工作吗？" 应具有约 100% 的通过率，并能捕捉到倒退。

#### 确保你能识别并阐明每次失败发生的原因

如果你无法阐明某次失败的原因，那么在构建自动化评估之前，你需要进行更多的错误分析。这应该是你投入**60-80% 评估精力**的地方。遵循以下流程：

1.  收集追踪记录：从生产环境或测试中收集有代表性的失败案例。
2.  开放式编码：与领域专家一起审查追踪记录，记下你看到的每个问题，无需预先分类（或使用我们的标注队列让主题专家自行审查追踪记录）。
3.  分类：将问题分组到失败分类法中（提示词问题、工具设计问题、模型限制、工具故障、数据缺口等）。
4.  迭代：持续审查，直到你不再发现新的失败类别。

一旦完成分类，修复措施取决于根本原因：

*   **提示词问题**：Agent 误解是因为你的指令不清晰 → 修复提示词。
*   **工具设计问题**：工具接口导致 Agent 容易犯错 → 重新设计参数、添加示例、明确边界。
*   **模型限制**：指令清晰，但 LLM 无法泛化到边缘情况 → 添加示例、尝试不同架构或使用不同模型。
*   **尚不清楚**：你还没有查看足够多的失败案例来发现模式 → 先进行更多错误分析。

#### 将评估所有权分配给单一领域专家

需要有人负责评估流程：维护数据集、重新校准评判标准、对新出现的失败模式进行分类，并决定"足够好"意味着什么。理想情况下，应由一位领域专家担任模糊案例的质量仲裁者，而不是通过委员会来设计。

#### 在归咎于 Agent 之前，先排除基础设施和数据管道问题

Witan Labs 团队发现，一个单一的提取错误将其基准测试结果从 50% 提升到了 73%。基础设施问题（超时、格式错误的 API 响应、缓存陈旧）经常伪装成推理失败。先检查数据管道。

## 选择你的评估层级

并非所有评估都测试相同的内容。将你的评估与 Agent 行为的正确层级相匹配。关于每个层级的深入探讨，请参见《Agent 可观测性赋能 Agent 评估》。

### 单步评估 vs. 完整轮次评估 vs. 多轮次评估

☑️ 理解三个评估层级：单步（运行）、完整轮次（追踪）和多轮次（线程）

☑️ 从追踪层级（完整轮次）评估开始，然后根据需要加入运行层级和线程层级评估

#### 单步评估

这些评估回答："Agent 是否选择了正确的工具？" "它是否生成了有效的 API 调用？" 它们最容易自动化，但需要稳定的 Agent 架构；如果你仍在更改工具定义，运行层级的评估可能会失效。

#### 完整轮次评估

这是大多数团队应该开始的地方。从三个维度对一个完整的追踪记录进行评分：

*   **最终响应**：输出是否正确且有用？
*   **轨迹**：Agent 是否采取了合理的路径？（不一定是你预期的*确切*路径，只要是有效的即可）
*   **状态变更**：Agent 是否创建了正确的产物？（写入的文件、更新的数据库、安排的会议等）

状态变更评估经常被忽视，但对于那些*执行*操作而不仅仅是*说*的 Agent 至关重要。例如，如果你的 Agent 安排会议，不要只检查它是否说了"会议已安排！"。要验证日历事件确实存在，且时间、参与者和描述都正确。如果它编写代码，就运行代码。如果它更新数据库，就查询相关行。最终响应可能说"完成！"，而实际状态却是错误的。

#### 多轮次评估

这是最难实现的层级，在你的追踪层级评估稳固之后，再逐步加入。

#### 从追踪层级（完整轮次）评估开始，然后根据需要加入运行层级和线程层级评估

追踪层级每次评估能提供最多的信号。运行层级对于调试特定步骤很有用。当你的 Agent 涉及多轮对话时，线程层级就很重要。

## 数据集构建

![](/images/posts/6301b4d08c62.png)

☑️ 确保每个任务都是明确的，并有一个参考解决方案证明其可解性

☑️ 测试正面案例（应发生的行为）和负面案例（不应发生的行为）

☑️ 确保数据集结构与你选择的评估层级相匹配

☑️ 根据你的 Agent 类型（编码、对话、研究）定制数据集

☑️ 如果缺乏生产数据，则生成种子示例

☑️ 数据来源包括内部试用错误、改编的外部基准测试和手写的行为测试

☑️ 建立从追踪到数据集的飞轮效应以实现持续改进

#### 确保每个任务都是明确的，并有一个参考解决方案证明其可解性

*   不明确："帮我找去纽约的好航班。"
*   明确："查找从 SFO 到 JFK 的往返航班，出发日期 12 月 15-17 日，返回日期 12 月 22 日，经济舱，价格低于 400 美元。"

如果 Agent 不可能成功（信息缺失、约束条件不可能），那是任务本身有问题，而不是 Agent。为每个任务包含一个参考解决方案，这样你可以证明它是可解的，并有一个用于评分的基线。

#### 测试正面案例（应发生的行为）和负面案例（不应发生的行为）

如果只测试“它是否在应该搜索时进行了搜索？”，你会优化出一个对所有内容都进行搜索的Agent（智能体）。同时也要测试负面情况。包含那些旨在证伪你假设的示例，而不仅仅是确认预期行为。

#### 确保数据集结构与你选择的评估级别相匹配

-   运行级别（单步）评估需要参考工具调用或决策
-   轨迹级别（完整回合）评估需要预期的最终输出和/或状态变化
-   线程级别（多轮）评估需要包含预期上下文保持的多轮对话序列

#### 根据你的Agent（智能体）类型定制数据集（编码、对话、研究）

-   编码Agent（智能体）：包含确定性测试套件（通过/失败的单元测试）以及质量评估标准
-   对话Agent（智能体）：包含多维标准、任务完成度和交互质量（同理心、清晰度）
-   研究Agent（智能体）：包含事实依据检查（主张是否有来源支持？）和覆盖度检查（是否包含关键事实？）

#### 如果缺乏生产数据，生成种子示例

定义任务的关键变化维度（查询复杂度、主题、边界情况类型）。手动创建约20个覆盖这些维度的示例输入，通过你现有的Agent（智能体）运行它们，审查并修改后存储为可靠的基准事实。

#### 从内部使用错误、改编的外部基准测试和手写的行为测试中获取数据

一旦度过冷启动阶段，你需要一个持续的流程来发现新的评估。以下三种策略结合使用效果良好：

1.  每天内部使用你的Agent（智能体），并将每个错误转化为一个评估。这与生产监控不同；这是你的团队有意在真实工作流程中对Agent（智能体）进行压力测试。
2.  从外部基准测试（如Terminal Bench或BFCL）中提取并改编任务。不要整体运行完整的基准测试；挑选那些测试你关心能力的任务，并为你的Agent（智能体）进行改编。
3.  针对你认为重要的特定行为手动编写聚焦测试，例如“Agent（智能体）是否并行化工具调用？”或“对于模糊请求，它是否会询问澄清问题？”

关于此方法的具体示例，请参见“How we build evals for Deep Agents”。

## 评分器设计

![](/images/posts/bdd02145de69.png)

☑️ 根据评估维度选择专门的评分器：客观检查默认使用基于代码的评分器，主观评估使用LLM-as-judge，模糊情况使用人工，版本比较使用成对比较

☑️ 区分防护栏（内联、运行时）和评估器（异步、质量评估）

☑️ 优先使用二进制通过/失败，而非数字量表

☑️ 将LLM-as-a-Judge评分器校准至人类偏好

☑️ 对结果评分，而非精确路径，并为渐进式进展设置部分得分

☑️ 使用从错误分析中得出的自定义评估器，而非通用的现成指标

#### 根据评估维度选择专门的评分器

当存在客观正确答案时，默认使用基于代码的评估器。对于客观任务，LLM-as-judge评分可能不可靠，不一致的判断可能掩盖真正的回归。切换到确定性比较通常可以消除不一致性并提供更好的信号。将LLM-as-judge保留用于真正主观的评估。

#### 区分防护栏和评估器

安全检查与格式验证是防护栏，应内联运行。质量评估与回归测试是评估器，应异步运行。不要混淆两者。

#### 优先使用二进制通过/失败，而非数字量表

1-5分的量表会引入相邻分数之间的主观差异，并且需要更大的样本量才能达到统计显著性。二进制迫使思考更清晰：Agent（智能体）要么成功，要么失败。你总是可以将复杂任务分解为多个二进制检查。

注意：近期研究表明，当专门使用LLM-as-judge时，短量表（0-5）可能产生更强的人类-LLM对齐，但对于人工评审员和快速迭代而言，二进制仍然更简单。

#### 将LLM-as-a-Judge评分器校准至人类偏好

-   从使用LangSmith的Align Evaluator功能标记的20多个示例开始，然后逐步增加到约100个以获得生产级置信度
-   在评分器的输出中包含推理过程；这能提高准确性，并让你可以审核其评分的*原因*（Anthropic的Demystifying Evals也强调了这一点）
-   定期重新校准；评分器会随时间漂移，且*没有一个评分器在所有基准测试中都是始终可靠的*
-   使用*少量示例*来提高评估器的一致性；在LangSmith中，修正可以自动填充为少量示例

#### 对结果评分，而非精确路径，并为渐进式进展设置部分得分

Agent（智能体）会找到创造性的解决方案。正如Anthropic在Demystifying Evals中所言：“不要对Agent（智能体）采取的路径评分，要对它产生的结果评分。”如果你要求“必须按顺序调用工具A → B → C”，那么你会让那些找到更聪明路径的Agent（智能体）失败。*更好的做法是*：“会议是否被正确安排？”而不是“它是否在`create_event`之前调用了`check_availability`？”

一个能正确识别问题但在最后一步失败的Agent（智能体），比一个立即失败的Agent（智能体）要好。设置部分得分，以便你的指标能反映渐进式进展。

#### 使用从错误分析中得出的自定义评估器，而非通用的现成指标

现成的指标如“帮助性”或“连贯性”会产生虚假的信心。真正重要的是那些能捕捉*你*特定失败模式的评估器，这些模式是通过上述错误分析过程发现的。

## 运行与迭代

![](/images/posts/be2c211c94c4.png)

☑️ 区分离线、在线和临时评估，并同时使用三者

☑️ 每项任务运行多次试验以考虑非确定性

☑️ 手动审查失败评估的轨迹以验证评分器的公平性

☑️ 确保每次试验在干净、隔离的环境中运行，没有共享状态

☑️ 按能力类别标记评估，记录每个评估的衡量内容，并跟踪效率指标（步骤数、工具调用次数、延迟）以及质量

☑️ 识别通过率何时趋于稳定，并相应调整你的测试套件

☑️ 只保留那些直接衡量你关心的生产行为的评估

☑️ 投资于工具接口设计和测试，而不仅仅是提示词优化

☑️ 区分任务失败（Agent（智能体）出错）和评估失败（评分器出错）

#### 区分离线、在线和临时评估，并同时使用三者

本清单的大部分内容都聚焦于离线评估，这是有意为之。离线评估是你通过以下方式进行改进的地方：精心策划的数据集、受控实验、发布前的迭代。一旦你的Agent（智能体）进入生产环境，你还需要在线和临时评估。

下面的“生产就绪度”部分详细介绍了如何设置在线评估和安排临时轨迹探索。

#### 每项任务运行多次试验以考虑非确定性

模型输出在不同运行间会变化。如果成本允许，使用*多次重复*。当运行多次试验时，在宣布改进之前计算置信区间——单次运行的基准测试是有噪声的。对于非确定性Agent（智能体），根据你的产品需求，考虑使用pass@k（k次尝试中至少一次成功）或pass^k（所有k次尝试都成功）指标。

在质量指标之外跟踪运营指标：所用轮次、Token使用量、延迟、每任务成本。一个准确率95%但速度慢10倍的Agent（智能体）可能算不上改进。

#### 按能力类别标记评估，记录每个评估的衡量内容，并跟踪效率指标以及质量

按测试内容而非来源对评估进行分组。诸如`file_operations`、`retrieval`、`tool_use`、`memory`和`conversation`等类别，为你提供了介于单一总分和单个测试结果之间的性能“中间视图”。为每个评估添加文档字符串，解释其如何衡量Agent（智能体）能力。随着套件增长，这能保持意图清晰，并让你可以运行有针对性的子集（例如，在更改工具定义后只运行`tool_use`评估）。

为每次实验附加元数据，以便您能按重要维度筛选、分组和比较运行记录。这样就能轻松回答诸如"从GPT-4.1切换到Claude Sonnet是否提升了准确率？"或"哪个提示词版本在该数据集上出现倒退？"等问题，而无需翻阅日志。LangSmith会在可用时自动捕获git信息，但随着实验量增长，显式标注模型和提示词元数据将很快显现价值。

质量确认后，需比较模型效率。一个准确率达95%但速度慢10倍的Agent（智能体）可能并非改进。追踪以下比率：实际步骤数/理想步骤数、实际工具调用数/理想工具调用数、实际延迟/理想延迟。这与"评估结果而非具体路径"的原则并不冲突：理想轨迹衡量的是效率，而非正确性。对于找到创新路径的Agent，您仍会判定通过，但可以观察其是否耗时更长。具体示例可参阅《我们如何为深度智能体构建评估体系》中的指标框架。

#### 手动审查失败评估的追踪记录以验证评估器公正性

"失败"的任务实际上可能是评估器未预料到的创新有效解决方案。阅读追踪记录是判断评估器是否公正的唯一途径。

#### 确保每次试验在无共享状态的清洁隔离环境中运行

若试验2能看到试验1的残留信息，则结果不具备独立性。具体实践要求：

- 代码Agent：每次试验使用全新容器或虚拟机
- API调用Agent：使用预发环境或模拟服务
- 数据库Agent：试验间进行快照与恢复

#### 识别通过率平台期并相应升级测试套件

当通过率进入平台期，且增加同类任务不再揭示新失败模式时，就该升级测试：增加高难度任务、测试新能力或转向不同维度。在饱和的评估集上持续投入只会浪费精力。

#### 仅保留直接衡量生产关键行为的评估项

每个评估项都会持续对系统产生压力。盲目添加数百个测试虽具诱惑力，但这只会制造进步假象。最终您优化的将是无法反映生产核心需求的评估套件。更多评估不等于更好的Agent。构建针对性评估，并定期剔除不再提供有效信号的评估项。具体案例可参考《我们如何为深度智能体构建评估体系》。

#### 投入工具接口设计与测试，而非仅优化提示词

工具设计能消除整类Agent错误。Anthropic团队在构建SWE-bench智能体时指出，他们优化工具的时间远超提示词。测试模型实际使用工具的方式：尝试不同参数格式（差异对比与完整重写、JSON与markdown），重新设计接口以降低错误概率，并投入清晰示例文档。目标是实现结构上的防错机制，而不仅是降低错误可能性。例如，要求绝对文件路径可彻底消除导航错误类别。

#### 区分任务失败（Agent出错）与评估失败（评估器出错）

明确记录运行状态（完成、错误、超时）。将超时标记为"推理错误"的评估器会污染信号质量。区分任务失败与评估失败以保持指标纯净。

攻克难关后，需巩固成果。曾经测试"能否实现？"的任务应转变为"是否仍能实现？"

#### 通过自动化质量门禁将回归评估集成至CI/CD流水线

典型流程：

1. 代码或提示词变更触发流水线（通过git推送、PromptHub更新或手动触发）
2. 离线评估运行单元测试、集成测试，并使用廉价快速评估器对精选数据集进行评估
3. 离线评估通过后部署预览环境
4. 使用LLM即评估器对预览环境进行实时数据在线评估
5. 仅当所有质量门禁通过后方可上线生产，否则将失败追踪路由至标注队列并通知团队

在CI中对每次提交使用基于代码的廉价评估器。保留昂贵的LLM即评估器用于预览/生产评估。完整实现示例可参阅LangSmith的《CI/CD流水线指南》（含GitHub Actions配置）。

#### 为生产流量设置在线评估

安全检查、格式验证、质量启发式检测。您将在生产环境中发现从未预料到的失败模式（参见《智能体上线前你永远不知道它会做什么》）

#### 收集用户反馈

Agent上线后，用户反馈将成为最宝贵的信号源。自动化评估只能捕获已知失败模式，而用户会揭示未知问题：数据集未覆盖的边缘案例、技术正确但无实际帮助的输出、以及以意外方式中断的工作流。

结构化收集这些反馈，可将其反哺至数据集，根据实际期望校准评估器，并优先改进对用户真正重要的方面。

#### 定期人工探查生产追踪记录（超越自动化检查）

切勿仅依赖自动化通过/失败判定。定期探查生产追踪记录，发现评估器未覆盖的意外模式或失败案例、令人惊讶的用户行为，以及改进机会。我们的Insights Agent是实现此目标的绝佳工具！

#### 对提示词和工具定义进行版本管理

LangSmith可轻松实现提示词版本管理。缺乏版本控制将无法关联评估结果与特定变更，也无从知晓哪个修改导致了回归。

#### 确保生产故障反馈至数据集、错误分析和评估改进

![](/images/posts/62dfd36841f7.png)

生产环境的成功与失败都应反馈至数据集、错误分析和评估改进。这正是驱动Agent持续优化的飞轮效应！

无需首日实现所有事项。选择符合当前阶段的章节，落实对应事项，再逐步扩展。能交付可靠Agent的团队并非拥有最复杂评估基础设施的团队，而是那些尽早开始评估并持续迭代的团队。
