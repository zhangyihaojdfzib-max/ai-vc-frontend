当开发者加入新项目或需要在不熟悉的代码库中工作时，像 Databricks Knowledge Assistant 这样的知识助手能通过回答关于代码的自然语言问题，帮助他们快速上手。但回答质量在很大程度上取决于源代码及周边上下文是如何准备和添加的。其中一个关键因素是**分块**：即如何将源文件分割成片段以进行索引和检索。代码使得这一点变得棘手。如果你在函数体中间截断或剥离其类上下文，即使是一个能力很强的助手，也难以回答相关问题。

我们在 **Casper's Kitchens** 演示 GitHub 仓库上构建了三个知识助手，每个都使用了不同的分块策略，从简单的固定大小基线方法，到将代码解析为其语法组件的结构感知方法。该仓库模拟了 Databricks 上的一个幽灵厨房业务，使用了包括 Lakeflow 管道、DSPy Agent 和 Databricks Asset Bundles (DABs) 在内的广泛功能，文档则存放在 Markdown 文件和 Notebook 单元格中。跨文件依赖、混合文件格式和特定领域模式使得这类项目非常需要一个强大的知识助手来提供巨大帮助。

本文详细阐述了处理代码与处理典型业务文档有何不同，我们如何将每种分块策略部署为 **Databricks Knowledge Assistant**，以及我们如何使用 **MLflow 的评估框架** 来比较它们。你可以在 **这里** 找到所有代码。

## 知识助手的工作原理（以及代码为何不同）

在底层，知识助手使用各种形式的检索增强生成（RAG）。它们从源数据（通常来自**向量搜索索引**）中检索相关的数据块，并将其作为上下文传递给大语言模型，以生成对用户查询的答案。

Databricks Knowledge Assistant 在此基础上构建，采用了复杂的检索技术，包括 **Instructed Retriever**，该技术融合了查询分解、上下文感知的重新排序以及对文档元数据的推理。这些能力在处理现实世界代码库的复杂性方面大有帮助，并且当底层数据块保留了有意义的语义边界时，它们的效果最佳。

知识助手最常见的是在业务文档集合上构建和评估，这些文档往往是线性流动的，包含段落和章节。代码则具有嵌套层次结构：文件包含类，类包含方法，方法包含逻辑块。代码中的语义单元通常是一个完整的函数，而不是一个段落。

这带来了特定的挑战，包括：

*   **语义边界**：在函数体中间分割会丢失理解其功能所需的上下文。一个包含 `deletion_order = ['experiments', 'jobs'...` 的数据块，如果没有显示这个变量是在 `UCState.clear_all()` 内部，那么它的用处就不大。
*   **跨文件依赖**：代码会引用其他代码。理解一个函数通常需要其类、导入或相关函数的上下文。
*   **混合文件类型**：我们的代码库有 `.py` 文件、`.ipynb` Notebook（包含代码/Markdown 单元格的 JSON）、`.md` 文档和 `.yaml` 配置文件，每种类型都需要不同的解析方法。

由于 Databricks Knowledge Assistant 允许你使用自己的向量索引，你可以按任意方式准备数据块，只需将知识助手指向结果即可。这使我们能够比较为 RAG 准备代码库的不同方法，并选择最佳方案。

## 分块策略

要了解分块策略在实践中如何不同，请考虑当你提问时会发生什么：“资源清理按什么顺序进行？”答案位于一个跟踪实验、作业和管道的工具类中。其逻辑跨越初始化、删除顺序列表和清理方法。以下是每种方法的工作原理，以及它如何影响检索到的关于资源清理类 `UCState` 的上下文。

### 简单基线：固定大小的字符分块

最简单的方法是以固定的字符间隔（带重叠）分割源文件，将代码视为纯文本。这不是你今天为生产就绪的 RAG 系统会选择的方案。它忽略了语法和语义边界，因此恰恰在代码查询关心的方面会失败。但它也极其容易实现，对于快速实验或文档繁重的仓库来说通常“足够好”，并且常作为第一步尝试，因此它是一个有用的基线。

### 语言感知：LangChain 启发式分割器

LangChain 的 `RecursiveCharacterTextSplitter.from_language()` 使用特定于语言的分隔符（例如 Python 的 `\nclass` 和 `\ndef`）来优先在逻辑边界处分割。它试图保持函数完整，但仍强制执行严格的大小限制。从概念上讲，这通过优先在可能的语义边界（如 `def` 和 `class`）处分割，而不是任意的字符计数，从而改进了简单分块，因此数据块更可能包含完整的逻辑单元。

### 基于 AST：带有元数据头的 Tree-sitter

基于抽象语法树的分块使用像 **Tree-sitter** 这样的解析器来理解实际的代码结构。AST 是代码的树形表示，它捕获了代码的句法结构——即代码如何根据语言的语法规则组织。基于 AST 的分块策略不是在字符边界处分割或使用启发式模式，而是将代码解析成语法树，并在语义边界（如函数、类或语句块）处进行分块。必要时，它可以超出大小限制以保持一个完整单元在一起，而不是在函数中间分割。

我们使用 **ASTChunk** Python 库来处理基于 AST 的分割。该库包含一个 **chunk expansion** 选项，该选项会在每个数据块前添加一个元数据头，显示文件路径和类/函数层次结构。这个上下文成为嵌入的一部分，有助于检索将查询与相关代码匹配，即使查询词没有出现在数据块主体中。

至此，我们对哪些方法可能在我们的知识助手中效果最好有了一些直觉。但为了确定，我们需要进行系统性的评估。

## 使用 MLflow 的评估设置

MLflow 的 **GenAI 评估框架** 提供了一个完整的工具包，用于比较 LLM、Agent 和检索系统。你给它一个评估数据集、一个预测函数和 LLM 评判器，它就会通过你的流水线运行每个问题并对结果进行评分。以下是我们如何使用它来比较三种分块方法。

### 评估数据集

我们创建了涵盖不同类别的 46 个问题，范围从广泛的概念性主题到关于代码的详细查询。

我们有意将数据集权重向消歧类问题倾斜，这类问题中代码库在不同上下文中存在结构相似的代码，例如两个应用具有重叠的函数名、并行的数据库模式，或在细微处存在差异的配置文件。正是这类查询最能清晰地暴露分块策略的弱点。如果您的数据块缺乏关于代码位置的元数据，检索系统将难以区分存在于不同上下文中的相似类和函数。

### LLM评估器

我们使用了三个主要的LLM评估器，每个评估器捕捉质量的不同方面：

- **检索充分性（内置）**：检索到的数据块是否包含足够信息来回答问题？这是比较分块策略的关键指标，因为它独立于生成过程来衡量检索质量。
- **检索真实性（内置）**：回答是否基于检索到的上下文，还是引入了数据块中不存在的信息？
- **回答正确性（自定义）**：这个自定义评分器将每个答案评为正确、部分正确或不正确，使其比严格的二元正确性评估更具细微差别。考虑到上下文可能碎片化或不完整，我们希望关注那些可能遗漏细节或存在微小不准确的答案。

### 运行评估

为确保比较公平，所有策略均使用相同的目标分块大小（1,000字符）、重叠量（200字符）和嵌入模型（databricks-gte-large-en）。实际上，最终的分块大小仍会存在差异（例如，基于AST的分块可能会扩展以保留完整的语义单元，而非常小的文件自然会产生小数据块）。

对于每种分块策略，我们将数据块写入Delta表，使用托管嵌入创建向量搜索索引（按要求使用databricks-gte-large-en嵌入模型），并将索引附加到Knowledge Assistant端点。完整设置详见文档。

## 结果与经验教训

我们通过每个Knowledge Assistant运行全部46个问题，并用三个评估器对结果进行评分。以下是我们的发现：

所有三种策略均实现了85%以上的检索充分性，这意味着无论代码如何分块，Knowledge Assistant的检索技术都能找到相关上下文。检索层面的差异并不显著。

自定义正确性结果则揭示了更有趣的情况。基于AST的分块在70%的情况下产生完全正确的答案，而朴素分块为59%，语言感知分块为61%。几乎所有情况下，三种策略都至少能产生部分正确的答案。更好的数据块有助于知识助手更完整地回答问题。

优势集中在特定问题类型上。基于AST的分块在消歧问题上表现出色，这类问题中模块间存在结构相似的代码，这得益于其添加的元数据（文件路径、类、函数名）提供了必要的上下文。对于值查找和完整定义检索，三种策略表现相当。

Databricks Knowledge Assistant开箱即用，能力强大。三种策略的检索充分性都很高，几乎每个问题都至少得到了部分正确的答案。

数据准备仍然重要。在此次评估中，基于AST的分块提高了真实性和正确性，特别是对于涉及消歧相似代码的问题。检索和回答质量的微小改进，在开发团队每天提出数十个问题的场景下，会产生复合效应。

自定义LLM评估器有助于衡量我们真正关心的方面。MLflow的make_judge() API使得构建针对特定用例的LLM评估器变得容易。我们的自定义answer_correctness评估器能够提供比简单的通过/失败正确性评估更细致的正确性视图。
