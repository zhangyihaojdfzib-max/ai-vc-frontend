## 优化物理层

物理层是决定BI性能好坏的关键。如果这一层做对了，在你触及语义层之前，每一次查询都会受益。

### 从维度建模开始

星型模式仍然是BI查询性能的黄金标准。通过代理键将宽表、反规范化的维度表与事实表连接起来，为查询优化器提供了清晰、可预测的连接路径。

Databricks完全支持你所需的关系建模结构：主键和外键约束（使用`RELY`作为优化器提示）、用于代理键的标识列，以及`CHECK`和`NOT NULL`约束。如果你遵循奖牌架构，请在Silver层保留规范化或Data Vault模型，并在Gold层为BI消费构建反规范化的星型模式。

有关详细的实现模式——SCD类型1和类型2处理、使用`MERGE`的事实表ETL、延迟到达维度——请参阅《在Databricks SQL中实现维度数据仓库》博客系列。

### 使用托管表

Unity Catalog的**托管表**是本技术栈中其他所有内容的基础。Unity Catalog管理托管表的所有读取、写入、存储和优化职责。这解锁了外部表无法提供的自动功能：**预测优化**（下文将介绍）默认启用。**自动液态聚类**会选择随查询模式变化而调整的聚类键。**元数据缓存**始终开启，可减少云存储请求并加速查询规划。

在整个平台中使用托管表——不仅用于BI服务，也适用于Bronze、Silver和Gold层。它们是Unity Catalog中的默认表类型，其性能和治理优势会随着本技术栈中的其他每项优化而叠加。

### 应用液态聚类

液态聚类取代了静态分区和手动的`Z-ORDER`——与这些方法不同，你可以在不重写现有数据的情况下重新定义聚类键。在创建表时添加`CLUSTER BY (col1, col2)`，或对现有表使用`ALTER TABLE`。如果不确定选择哪些列，`CLUSTER BY AUTO`让预测优化根据观察到的查询模式来选择键。

对于BI工作负载，请在最常用的过滤和连接列上进行聚类——日期键、区域、产品类别。最多可以选择四列，如果两列高度相关，则只包含其中一列。当仪表盘在聚类列上进行过滤时，液态聚类通过数据跳过功能提高了查询性能。

### 让预测优化处理其余部分

预测优化会自动对受益于这些操作的表运行`OPTIMIZE`、`VACUUM`和统计信息收集——因此你无需自己调度这些作业。它在Photon写入期间收集Delta数据跳过统计信息和查询优化器统计信息，并为现有表回填统计信息。在观察到的负载中，这带来了**平均22%的性能提升**。对于具有重复过滤模式的BI工作负载，其影响尤其显著——更好的统计信息意味着更好的数据跳过和更高效的查询计划。

在目录级别启用预测优化，然后让它运行。使用预测优化是你能够进行的高回报、低投入的优化之一。

结果是：BI查询扫描的数据更少，连接效率更高，运行成本更低——而且你还没有触及语义层。

## 指标视图：一次性定义你的指标

这里开始变得有趣。大多数组织在多个地方定义了相同的业务指标——一个BI工具中的收入计算，另一个工具中略有不同的版本，还有某个同事上个季度写的SQL笔记本中的第三种变体。每个定义都独立地发生偏差。没人确定哪个是正确的。

Unity Catalog中的**指标视图**通过提供一个无头BI层来解决这个问题——一个单一的、受治理的语义层，你可以在其中一次性定义数据模型和KPI，独立于任何特定的BI工具。你可以在Unity Catalog Explorer中使用SQL或点击式界面集中定义它们。AI/BI仪表盘、Genie、SQL笔记本和第三方BI工具都从相同的定义中解析指标。一次性定义指标，每个消费者——无论是人类还是AI——都会得到相同的答案。

指标视图不仅仅是集中化的指标定义——语义元数据才是其与众不同之处。像`display_name`、`comment`和`synonyms`这样的字段为AI系统提供了正确解释业务问题所需的上下文。当用户问Genie“我们上周的收入是多少？”时，这些注释就是Genie将自然语言映射到正确度量和维度的方式。无需自定义提示词，无需单独的词汇表。同样的原理也适用于基于Databricks构建的AI智能体——任何有权访问Unity Catalog的智能体都可以通过语义层发现和查询受治理的指标，而不是使用硬编码的SQL。你的元数据越丰富，AI提供的正确答案就越准确。

## 指标视图物化：无额外开销的 OLAP 性能

传统上，当 BI 仪表盘速度过慢时，解决方案是构建聚合表。您会在星型模式之上创建物化视图或自定义预聚合表，设置刷新管道，并将 BI 工具重新指向新表。这种方法有效，但增加了需要维护的整个对象和管道层——而且每次聚合逻辑变更时，您都必须更新 BI 工具查询以保持匹配。

指标视图物化提供了一种更简单的替代方案。当您在指标视图上启用物化时，平台会自动在您的 BI 工具已查询的相同指标定义背后维护预聚合结果——无需构建单独的聚合表，也无需重构 BI 工具查询。以下是底层工作原理：

- 自动预聚合：指标结果被预先计算并存储
- 增量刷新：指标保持最新状态，无需完全重新计算
- 智能查询重写：引擎将查询路由到最佳可用物化结果
- 透明路由：用户以相同方式查询指标——系统提供最快路径

以前扫描完整事实表的仪表盘查询现在命中预聚合的物化结果——延迟更低，计算成本也更低。上述仪表盘和 Genie 示例都查询了同一个指标视图，并且它们的查询都被透明地路由到了物化结果。下面来自 Genie 的查询计划展示了这一过程。

## 实用 TCO 要点

更快的查询和更低的成本并非相互竞争的目标——每一项减少数据扫描的优化也会降低您需要支付的计算成本。并且堆栈中的每一项优化都会产生叠加效应。液态聚类和更好的统计信息可改善数据跳过和查询计划。物化结果可以增量刷新，从而减少 SQL 仓库为仪表盘服务所需的计算量。以下是更多降低成本的方法：

- 合理调整 SQL 仓库规模。使用具有自动扩展功能的无服务器 SQL 仓库来应对 BI 并发高峰。您按实际使用量付费，而非按峰值容量付费。
- 充分利用 DBSQL 的缓存层级。磁盘缓存将热数据保留在仓库本地，查询结果缓存（QRC）无需重新执行即可服务重复查询。对于查询模式一致的仪表盘，缓存可将许多请求转化为毫秒级延迟的响应，且计算成本近乎为零。
- 消除冗余数据移动。通过 DirectQuery 或实时连接直接从湖仓一体服务 BI，而不是使用数据提取或导入。
- 使用系统表进行监控。系统表如 `system.billing.usage` 和 `system.query.history` 可用于按仪表盘、用户和仓库跟踪 BI 使用情况。在系统表上构建指标视图和 AI/BI 仪表盘，以获得对 BI 使用情况的可见性。

## 开始使用

您无需一次性实施整个堆栈。从您能看到最大影响的地方开始：

1. 使用托管表、主键/外键和液态聚类构建（或验证）您的 Gold 层星型模式
2. 在您的目录上启用预测优化，以自动管理 `OPTIMIZE`、`VACUUM` 和统计信息收集
3. 为您的核心业务 KPI 定义指标视图——从 SQL 或 UC Explorer UI 开始
4. 为您流量最高的指标启用指标视图物化
5. 监控结果——将仪表盘指向指标视图，并通过系统表跟踪查询性能

Databricks 在 BI 服务堆栈的每一层都提供了优化。托管表、液态聚类和预测优化最大限度地减少了扫描的数据量和计算开销。指标视图将您的业务逻辑集中在一个受治理的语义层中，该层一致地为仪表盘、Genie 和 AI Agent（智能体）提供服务。物化功能无需手动预聚合管道即可实现亚秒级查询性能。这些层共同作用，叠加效应显著——既降低了查询延迟，也降低了总体拥有成本。
