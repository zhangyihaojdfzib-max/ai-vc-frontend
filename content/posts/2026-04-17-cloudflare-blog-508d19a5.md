# 具备记忆能力的智能体：Agent Memory 正式发布

2026-04-17

- Tyson Trautmann
- Rob Sutter

![](/images/posts/e402d337e232.png)

随着开发者在 Cloudflare 上构建日益复杂的智能体，他们面临的最大挑战之一是如何在正确的时间将准确信息置入上下文。模型输出结果的质量直接取决于其运作时所处上下文的质量，但即便上下文窗口大小已突破百万 Token，**上下文衰减**问题仍未得到解决。开发者陷入两难境地：要么将所有信息保留在上下文中却面临质量下降，要么激进地修剪上下文但可能丢失智能体后续所需的关键信息。

今天我们宣布推出 **Agent Memory** 的封闭测试版——这是一项托管服务，能够从智能体对话中提取信息并在需要时提供调用，同时避免占用上下文窗口空间。

该服务赋予 AI 智能体持久化记忆能力，使其能够记住重要信息、遗忘无关内容，并随时间推移变得更智能。本文将详细解析其工作原理及潜在应用场景。

## 智能体记忆技术现状

智能体记忆是 AI 基础设施领域发展最迅猛的方向之一，几乎每周都有新的开源库、托管服务和研究原型发布。这些方案在存储内容、检索方式及适用智能体类型方面差异显著。虽然 LongMemEval、LoCoMo 和 BEAM 等基准测试提供了有效的横向对比，但也容易导致系统为特定评估指标**过度优化**，在实际生产中表现不佳。

现有解决方案在架构设计上也存在分歧：有些是后台处理提取检索的托管服务，有些则是需要自行部署记忆管道的自托管框架；部分方案提供受限的专用 API 将记忆逻辑隔离在智能体主上下文之外，另一些则允许模型直接访问数据库或文件系统并自行设计查询，将 Token 消耗在存储检索策略而非实际任务上；还有方案试图将所有信息塞入上下文窗口（必要时通过多智能体分区处理），而其他方案则通过检索仅呈现相关信息。

Agent Memory 是一款采用预设 API 设计与基于检索架构的托管服务。经过审慎评估，我们认为这种组合是大多数生产工作负载的理想默认方案。相比赋予智能体原始文件系统访问权限，更紧密的摄取检索管道具有明显优势——除了提升成本效益与性能表现，它们还为生产环境所需的复杂推理任务（如时序逻辑、信息更迭与指令遵循）提供了更坚实的基础。未来我们可能会开放程序化查询接口，但这预计主要服务于边缘场景而非通用需求。

我们开发 Agent Memory 是因为平台上的实际工作负载暴露出现有方案未能完全解决的缺口。那些针对真实代码库与生产系统运行数周甚至数月的智能体，需要的是随规模增长仍保持效用的记忆系统——而非仅在能完全装入新型号上下文窗口的洁净基准数据集上表现优异的记忆功能。

它们需要高速的摄取能力，需要不阻塞对话的检索机制，还需要在保持单次查询成本合理的模型上运行。

## 使用方式

Agent Memory 将记忆存储于按名称寻址的配置文件中。每个配置文件支持多种操作：摄取对话内容、即时记忆特定信息、检索所需记忆、列出记忆条目或删除特定记忆。**摄取**是主要路径，通常在框架压缩上下文时调用；**记忆**供模型即时存储重要信息；**检索**则运行完整检索管道并返回综合答案。

```TypeScript
export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    // 获取配置文件——跨会话、智能体和用户共享的隔离记忆存储
    const profile = await env.MEMORY.getProfile("my-project");
    // 摄取——从对话中提取记忆（通常在压缩时调用）
    await profile.ingest([
      { role: "user", content: "使用 React 和 TypeScript 设置项目。" },
      { role: "assistant", content: "已完成。搭建了面向 Workers 的 React + TS 项目脚手架。" },
      { role: "user", content: "使用 pnpm 而非 npm。并默认启用深色模式。" },
      { role: "assistant", content: "已更新——默认使用 pnpm 和深色模式。" },
    ], { sessionId: "session-001" });
    // 记忆——显式存储单条记忆（模型直接调用工具）
    const memory = await profile.remember({
      content: "4月10日事件后，API 速率限制已提升至每个区域 10,000 次请求/秒。",
      sessionId: "session-001",
    });
    // 检索——获取记忆并得到综合答案
    const results = await profile.recall("用户偏好哪种包管理器？");
    console.log(results.result); // "用户更倾向于使用 pnpm 而非 npm。"
    return Response.json({ ok: true });
  },
};
```

Agent Memory 可通过绑定从任何 Cloudflare Worker 访问，也可通过 REST API 供 Workers 外部运行的智能体调用（遵循与其他 Cloudflare 开发者平台 API 相同的模式）。若使用 Cloudflare Agents SDK 进行开发，Agent Memory 服务可作为参考实现，无缝集成到 Sessions API 的**记忆模块**中，处理记忆的压缩、存储与搜索。

## 应用场景

Agent Memory 设计用于支持多种智能体架构：

**个体智能体记忆**。无论您使用 Claude Code 或 OpenCode 等编码智能体构建人机协同系统，采用 OpenClaw 或 Hermes 等自托管框架构建代理智能体，还是集成 Anthropic 托管智能体等托管服务，Agent Memory 均可作为持久化记忆层，无需修改智能体核心循环。

**自定义智能体框架记忆**。许多团队正在构建自主智能体基础设施，包括无人参与的全自动后台智能体。Ramp Inspect 是公开案例之一；Stripe 和 Spotify 也描述过类似系统。这些框架同样可通过为智能体提供跨会话持久化、重启后仍保留的记忆功能而受益。

**跨智能体、人员与工具的共享记忆**。记忆配置文件不必专属于单个智能体。工程师团队可共享记忆配置，使个人编码智能体习得的知识（如编码规范、架构决策、当前仅存于人脑或上下文修剪时丢失的团队隐性知识）可供全员使用。代码审查机器人与编码智能体可共享记忆，让审查反馈影响后续代码生成。智能体积累的知识将不再是短暂存在，而转变为持久的团队资产。

虽然搜索是记忆的组成部分，但智能体搜索与智能体记忆解决的是不同问题。**AI Search** 是我们用于在非结构化与结构化文件中查找结果的底层能力；Agent Memory 则专注于上下文回忆。Agent Memory 中的数据不以文件形式存在，而是从会话中衍生。智能体可同时使用两者，它们的设计本就相辅相成。

## 您的记忆数据归属您所有

随着智能体能力不断增强并更深融入业务流程，它们积累的记忆正显现出真实价值——不仅是运行状态记录，更是耗费实际工作构建的制度性知识。我们注意到客户对将此类资产绑定单一供应商的担忧日益增长，这是合理的。智能体学习得越多，若记忆无法随之迁移，切换成本就越高。

Agent Memory 是一项托管服务，但您的数据完全属于您。所有记忆均可导出，我们致力于确保，即使您的需求发生变化，您的 Agent（智能体）在 Cloudflare 上积累的知识也能随您一同迁移。我们认为，赢得长期信任的正确方式是让离开变得容易，并持续构建足够优秀的产品，让您不愿离开。

## Agent Memory 的工作原理

要理解上述 API 背后的运作机制，有助于拆解 Agent（智能体）管理上下文的流程。一个 Agent（智能体）包含三个组成部分：

1.  **一个驱动框架**，负责驱动对模型的重复调用、协调工具调用并管理状态。
2.  **一个模型**，接收上下文并返回补全结果。
3.  **状态**，既包括当前的上下文窗口，也包括上下文之外的其他信息：对话历史、文件、数据库、记忆。

**一个驱动框架**，负责驱动对模型的重复调用、协调工具调用并管理状态。

**一个模型**，接收上下文并返回补全结果。

**状态**，既包括当前的上下文窗口，也包括上下文之外的其他信息：对话历史、文件、数据库、记忆。

Agent（智能体）上下文生命周期中的关键时刻是**压缩**，即驱动框架决定缩短上下文以保持在模型限制内或避免上下文腐化。目前，大多数 Agent（智能体）会永久丢弃信息。而 Agent Memory 在压缩时不是丢失知识，而是将其保存下来。

Agent Memory 通过两种方式集成到这个生命周期中：

1.  **压缩时的批量摄取**。当驱动框架压缩上下文时，它会将对话发送给 Agent Memory 进行摄取。摄取过程会从消息历史中提取事实、事件、指令和任务，与现有记忆进行去重，并将其存储为记忆以供未来检索。
2.  **模型直接使用工具**。模型获得与记忆直接交互的工具，包括**回忆**（在记忆中搜索特定信息）的能力。模型还可以**记住**（基于重要内容显式存储记忆）、**忘记**（将记忆标记为不再重要或不真实）和**列出**（查看存储了哪些记忆）。这些都是轻量级操作，不需要模型设计查询或管理存储。主 Agent（智能体）永远不应在存储策略上消耗上下文。它看到的工具接口被刻意约束，以确保记忆不会干扰实际任务的执行。

**压缩时的批量摄取**。当驱动框架压缩上下文时，它会将对话发送给 Agent Memory 进行摄取。摄取过程会从消息历史中提取事实、事件、指令和任务，与现有记忆进行去重，并将其存储为记忆以供未来检索。

**模型直接使用工具**。模型获得与记忆直接交互的工具，包括**回忆**（在记忆中搜索特定信息）的能力。模型还可以**记住**（基于重要内容显式存储记忆）、**忘记**（将记忆标记为不再重要或不真实）和**列出**（查看存储了哪些记忆）。这些都是轻量级操作，不需要模型设计查询或管理存储。主 Agent（智能体）永远不应在存储策略上消耗上下文。它看到的工具接口被刻意约束，以确保记忆不会干扰实际任务的执行。

### 摄取管道

当对话到达进行摄取时，它会经过一个多阶段的管道，对记忆进行提取、验证、分类和存储。

第一步是确定性 ID 生成。每条消息都会获得一个内容寻址 ID —— 即会话 ID、角色和内容的 SHA-256 哈希值，截断至 128 位。如果同一对话被摄取两次，每条消息都会解析为相同的 ID，从而实现重新摄取的幂等性。

接下来，提取器并行运行两个通道。一个**完整通道**将消息分块，每块大约 10K 字符，并有两消息的重叠，最多同时处理四个块。每个块都会获得一个结构化转录本，包含角色标签、相对日期解析为绝对日期（例如"昨天"变为"2026-04-14"），以及用于溯源的行索引。对于较长的对话（9 条以上消息），一个**细节通道**会与完整通道同时运行，使用重叠窗口，专门聚焦于提取具体值，如姓名、价格、版本号和实体属性，这些往往是广泛提取容易遗漏的。然后将两个结果集合并。

下一步是根据源转录本验证每个提取的记忆。验证器运行八项检查，涵盖实体身份、对象身份、位置上下文、时间准确性、组织上下文、完整性、关系上下文，以及推断的事实是否确实得到对话支持。每个项目会相应地被通过、修正或丢弃。

然后，管道将每个已验证的记忆分类为四种类型之一。

*   **事实**代表当前为真的、原子化的、稳定的知识，例如"项目使用 GraphQL"或"用户偏好深色模式"。
*   **事件**捕捉在特定时间发生的事情，例如一次部署或一个决定。
*   **指令**描述如何做某事，例如程序、工作流、操作手册。
*   **任务**跟踪当前正在处理的工作，设计上是临时的。

**事实**代表当前为真的、原子化的、稳定的知识，例如"项目使用 GraphQL"或"用户偏好深色模式"。

**事件**捕捉在特定时间发生的事情，例如一次部署或一个决定。

**指令**描述如何做某事，例如程序、工作流、操作手册。

**任务**跟踪当前正在处理的工作，设计上是临时的。

事实和指令是带键的。每个都会获得一个规范化的主题键，当新记忆与现有记忆具有相同的键时，旧记忆会被取代而非删除。这会创建一个版本链，其中包含一个从旧记忆指向新记忆的前向指针。任务则完全排除在向量索引之外，以保持索引的精简，但仍可通过全文搜索发现。

最后，所有内容都使用 `INSERT OR IGNORE` 写入存储，以便内容寻址的重复项被静默跳过。在向驱动框架返回响应后，后台向量化会异步运行。嵌入文本将分类过程中生成的 3-5 个搜索查询前缀到记忆内容本身，这弥合了记忆的写入方式（陈述式："用户偏好深色模式"）与搜索方式（疑问式："用户想要什么主题？"）之间的差距。被取代记忆的向量会与新的更新插入操作并行删除。

### 检索管道

当 Agent（智能体）搜索记忆时，查询会经过一个独立的检索管道。在开发过程中，我们发现没有单一的检索方法对所有查询都最优，因此我们并行运行多种方法并融合结果。

第一阶段并行运行查询分析和嵌入。查询分析器生成排名的主题键、带同义词的全文搜索词，以及一个 HyDE（假设文档嵌入），这是一个表述为问题答案的陈述性语句。此阶段直接嵌入原始查询，两种嵌入都用于下游处理。

在下一阶段，五个检索通道并行运行。采用波特词干提取的全文搜索处理关键词精确查询，适用于你确切知道术语但不知其上下文的情况。精确事实键查找则直接返回查询映射到已知主题键的结果。原始消息搜索通过全文搜索直接查询存储的对话消息，作为安全网捕获未分类的对话片段，找回提取流程可能已泛化的逐字细节。直接向量搜索使用嵌入的查询查找语义相似的记忆。而HyDE向量搜索则查找与答案可能形态相似的记忆，这通常能发现直接嵌入搜索遗漏的结果——特别是对于抽象或多跳查询，这类问题与答案使用的词汇不同。

在第三也是最后阶段，来自所有五个检索通道的结果使用倒数排名融合进行合并，每个结果根据其在给定通道内的排名获得加权分数。事实键匹配获得最高权重，因为精确的主题匹配是最强的信号。全文搜索、HyDE向量和直接向量各自根据信号强度加权。最后，原始消息匹配也以较低权重被包含在内，作为安全网来识别提取流程可能遗漏的候选结果。平局情况通过时效性打破，较新的结果排名更高。

随后，流程将排名靠前的候选结果传递给合成模型，该模型针对原始搜索查询生成自然语言答案。某些特定查询类型会得到特殊处理。例如，时间计算通过正则表达式和算术确定性地处理，而非由LLM完成。结果作为预计算的事实被注入到合成提示词中。模型在日期计算等任务上不可靠，因此我们不要求它们执行此类操作。

## 我们的构建方式

我们最初的原型是轻量级的，包含基本的提取流程、向量存储和简单检索。它足以演示概念，但尚不足以交付使用。

因此，我们将其置于一个由Agent驱动的循环中进行迭代。循环过程如下：运行基准测试，分析存在的差距，提出解决方案，由人工审查提案以选择能够泛化而非过拟合的策略，让Agent实施更改，然后重复。

这种方法效果很好，但也带来了一个特定的挑战。LLM具有随机性，即使温度设置为零也是如此。这导致不同运行的结果存在差异，意味着我们必须对多次运行取平均值（对于大型基准测试来说耗时），并依赖趋势分析和原始分数来理解真正有效的方法。在此过程中，我们必须小心防范以无法真正改善产品通用性的方式对基准测试进行过拟合。

随着时间的推移，这使我们达到了一个状态：每次迭代基准测试分数都持续提升，并且我们拥有一个能在现实世界中工作的通用架构。我们有意针对多个基准测试（包括LoCoMo、LongMemEval和BEAM）进行测试，以从不同角度推动系统。

## 为何选择Cloudflare

我们在Cloudflare上构建Cloudflare，Agent Memory也不例外。现有强大且易于组合的基础组件使我们能够在一个周末内交付第一个原型，并在不到一个月内交付一个功能齐全、可用于生产的Agent Memory内部版本。除了交付速度之外，Cloudflare被证明是构建此类服务的理想平台，还有其他几个原因。

在底层，Agent Memory是一个协调多个系统的Cloudflare Worker：

- Durable Object：存储原始消息和分类记忆
- Vectorize：提供对嵌入记忆的向量搜索
- Workers AI：运行LLM和嵌入模型

Durable Object：存储原始消息和分类记忆

Vectorize：提供对嵌入记忆的向量搜索

每个记忆上下文映射到其自己的Durable Object实例和Vectorize索引，确保数据在不同上下文之间完全隔离。这也使我们能够轻松应对更高的需求。

通过Durable Objects实现计算隔离。每个记忆配置文件都拥有自己的Durable Object，该对象带有SQLite支持的存储，在租户之间提供强隔离，而无需任何基础设施开销。DO处理全文搜索索引、替代链和事务性写入。DO的`getByName()`寻址意味着来自任何地方的任何请求都可以通过名称访问正确的记忆配置文件，并确保敏感记忆与其他租户强隔离。

跨栈存储。记忆内容存储在SQLite支持的DO中。向量存储在Vectorize中。未来，快照和导出将存储到R2，以实现经济高效的长期存储。每个基础组件都为其工作负载量身定制，我们不需要将所有内容都塞进单一形态或数据库中。

通过Workers AI进行本地模型推理。整个提取、分类和合成流程都在部署于Cloudflare网络上的Workers AI模型上运行。所有AI调用都传递一个会话亲和性头部，该头部路由到记忆配置文件名称，因此重复的请求会命中相同的后端，从而获得提示词缓存的好处。

我们在模型选择中的一个有趣发现是：更大、更强大的模型并不总是更好。我们目前默认使用Llama 4 Scout（17B，16专家MoE）进行提取、验证、分类和查询分析，使用Nemotron 3（120B MoE，120亿活跃参数）进行合成。Scout能高效处理结构化分类任务，而Nemotron更强的推理能力则提高了自然语言答案的质量。合成器是唯一一个通过增加参数持续带来帮助的阶段。对于其他所有任务，较小的模型在成本、质量和延迟之间达到了更好的平衡点。
