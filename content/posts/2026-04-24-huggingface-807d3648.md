# DeepSeek-V4：Agent 真正可用的百万 Token 上下文

DeepSeek 今日发布 V4。两个 MoE 检查点已上传至 Hub：DeepSeek-V4-Pro 总参数量 1.6T，激活参数 49B；DeepSeek-V4-Flash 总参数量 284B，激活参数 13B。两者均支持 1M Token 的上下文窗口。基准测试成绩具有竞争力，但并非 SOTA。这并不重要。真正的创新在于 DeepSeek V4 如何针对高效的大上下文长度支持进行设计，从而成为 Agent 任务的最佳候选模型之一。

聚焦于长时间运行的 Agent 工作负载。当前将前沿开源模型作为 Agent 运行时，会出现可预见的故障模式。模型停止运行。你需要重新输入提示词。追踪记录超出上下文预算，或 KV 缓存占满 GPU，或工具调用往返在长时间任务中途降级。V4 旨在修复这些已知故障，并为社区指明方向。

本文涵盖三方面内容：架构如何实现低成本长上下文推理、在此基础上叠加的 Agent 专用后训练决策，以及论文中有助于理解这些变化的一些要点。

## Agent 的 KV 缓存问题

1M 上下文窗口只是容量，而非性能。能否使用它取决于该深度下每次前向传播的成本。对于运行长时间工具使用轨迹的 Agent（SWE-bench 任务、多步骤浏览会话、包含数百条命令的终端会话），每个工具结果都会追加到上下文中，后续每个 Token 都需要对之前所有内容支付完整的注意力成本。

两个数字至关重要：单 Token 推理 FLOPs 和 KV 缓存大小。两者均随序列长度增长。在 1M Token 下，DeepSeek-V4-Pro 的单 Token 推理 FLOPs 仅为 DeepSeek-V3.2 的 27%，因此在相同硬件上运行更快。其 KV 缓存内存使用量仅为后者的 10%。V4-Flash 进一步降低了这些数字：FLOPs 为 10%，KV 缓存为 7%。

如果我们将 KV 缓存内存与分组查询注意力（8 头，以标准 bfloat16 格式存储）等成熟架构进行比较，DeepSeek V4 所需的缓存大小约为后者的 2%。这使得处理极大上下文时的部署变得更加容易。

## 混合注意力：CSA 和 HCA

效率提升源于将注意力机制拆分为两种，并在各层之间交错使用。

压缩稀疏注意力（CSA）使用带有学习位置偏置的 softmax 门控池化，沿序列维度将 KV 条目压缩 4 倍。一个闪电索引器（FP4，ReLU 评分多头点积）为每个查询选取前 k 个压缩块。它继承了 V3.2 中 DeepSeek 稀疏注意力的稀疏选择思想，但在已比原始序列短 4 倍的块上运行。索引器的搜索空间随之缩小。

图 3：CSA。压缩器将每 4 个 Token 合并为一个压缩 KV 条目。闪电索引器为每个查询选取前 k 个压缩块。滑动窗口分支处理最近的未压缩 Token。

![图 3：压缩稀疏注意力，显示压缩器、作用于压缩块的闪电索引器以及滑动窗口分支](/images/posts/f20322086cba.png)

重度压缩注意力（HCA）将 KV 条目压缩 128 倍，并放弃稀疏选择。每个查询密集地关注每个压缩块。压缩后的序列足够短，使得密集注意力成本低廉。

图 4：HCA。更重的压缩器（128 倍 vs. 4 倍），随后对压缩流进行密集注意力，并带有相同的滑动窗口分支以处理近期内容。

![图 4：重度压缩注意力，128 倍压缩，对压缩块进行密集 MQA](/images/posts/e07364d42d43.png)

各层在 CSA 和 HCA 之间交替。不同层承载不同的注意力模式，强制所有层使用单一机制会浪费容量。在 V4-Pro 的 61 层堆叠中，第 0-1 层为 HCA，第 2-60 层交替使用 CSA 和 HCA，末尾的 MTP 块仅运行滑动窗口。

两条路径均对大部分 KV 条目使用 FP8 存储，仅对 RoPE 维度使用 BF16。CSA 内部的闪电索引器以 FP4 运行。这些存储选择与压缩比相结合，实现了 2% 的 KV 缓存占用。

图 2：整体架构。注意力层在 CSA 和 HCA 之间交替。前馈层使用 DeepSeekMoE。残差连接被流形约束超连接（mHC）取代。

## Agent 的变化

高效的长上下文注意力对于 Agent 工作流是必要的，但并非充分条件。论文描述了直接针对 Agent 用例的三项后训练和基础设施选择。

### 跨工具调用的交错思考

V3.2 在工具结果轮次间保留推理轨迹，但在每次新用户消息到达时丢弃它们。对于处理单次用户轮次的 Agent，这没问题。但对于多轮 Agent 工作流，当用户在 Agent 已链式调用多个工具后发送后续消息时，模型会丢失累积的推理内容，必须重建状态。

当对话包含工具调用时，V4 会跨用户消息边界保留推理内容。模型保留所有轮次的完整推理历史，包括跨用户轮次。这使得能够在长时间跨度的 Agent 任务上实现连贯、累积的思维链。对于无工具的对话场景，旧行为得以保留：每轮推理被刷新以保持上下文简洁。

### 带专用 Token 的工具调用模式

V4 引入了 |DSML| 特殊 Token 和基于 XML 的工具调用格式。与 JSON 字符串形式的工具调用相比，XML 格式减少了转义失败——这是模型输出嵌套引用内容时的常见故障模式。

该模式将字符串参数（按原样传递，string="true"）与结构化参数（以 JSON 传递，string="false"）分开。这消除了一类围绕数字和布尔值的解析错误，而 JSON 工具调用格式经常遇到这类错误。

### DSec：为 RL  rollout 构建的沙箱

Agent 行为是通过针对真实工具环境的 RL 训练得到的。论文描述了为此目的构建的沙箱基础设施。DeepSeek 弹性计算（DSec）是一个 Rust 平台，通过一个 Python SDK 暴露四种执行底层：函数调用、容器、微虚拟机（Firecracker）和完整虚拟机（QEMU）。单个集群可运行数十万个并发沙箱。

DSec 的三个特性对 Agent 训练至关重要：通过分层 3FS 存储实现快速镜像加载（使 RL rollout 无需等待容器启动）、支持抢占的轨迹回放（使中断的训练步骤无需重新运行工具调用即可恢复）、以及跨底层统一 API（使训练框架无需重写即可针对函数调用或完整虚拟机）。这些基础设施决策支撑了 Agent 基准测试成绩。

## 使用模型

Hub 上有四个检查点。指令模型对 MoE 专家权重使用 FP4，其他部分使用 FP8。基础模型全程使用 FP8。

- deepseek-ai/DeepSeek-V4-Pro（1.6T / 49B 激活，指令）
- deepseek-ai/DeepSeek-V4-Flash（284B / 13B 激活，指令）
- deepseek-ai/DeepSeek-V4-Pro-Base（1.6T / 49B 激活，基础）
- deepseek-ai/DeepSeek-V4-Flash-Base（284B / 13B 激活，基础）

两个指令模型支持三种推理模式：非思考（快速，无思维链）、高思考（在 `<think>` 块中显式推理）和最大思考（使用专用系统提示词的最大推理努力）。最大思考模式需要至少 384K Token 的上下文窗口。所有模式下的推荐采样参数为 `temperature=1.0`、`top_p=1.0`。

V4-Pro 在 SWE Verified、MCPAtlas 和内部研发基准测试上的表现使其在 Agent（智能体）任务上与前沿闭源模型持平。悬而未决的问题是社区的工具框架如何适应 `|DSML|` 模式，以及交错思考的收益是否能迁移到领域外的 Agent（智能体）框架。
