行级安全（RLS）是一种数据库访问控制机制，它根据用户的身份、角色或会话上下文，限制用户能够读取或更改表中的哪些行。

RLS 不是限制对整个表或特定列的访问，而是逐行过滤数据。数据库引擎在查询时自动应用该过滤器，因此无论用户使用何种工具访问数据，相同的规则都适用。

RLS 是细粒度访问控制的一部分，与之并列的还有：
- 列级安全
- 数据脱敏
- 表级授权

例如，销售人员可以查询公司的订单表，但只能看到其负责区域的订单，即使该表包含所有区域的数据。用户编写一个标准的 `SELECT` 语句，引擎只返回他们被允许看到的行。

RLS 现已成为多租户 SaaS、区域数据隔离和合规用例的核心构建模块。本文涵盖 RLS 的工作原理、适用场景、局限性以及它在 Databricks 平台上的运作方式。

## 行级安全是如何工作的？

行级安全通过向表应用一个过滤规则（通常称为策略或谓词）来工作。当用户运行查询时，数据库引擎自动应用该过滤器，并只返回用户被允许看到的行。

在实践中，RLS 通常分三步工作：
1.  **用户运行查询：** 用户编写标准查询，无需自行添加任何安全过滤器。
2.  **数据库检查用户身份：** 引擎通过内置函数（如 `CURRENT_USER`）、应用程序设置的会话变量或连接用户/组与允许数据的映射表来评估用户身份。
3.  **引擎过滤结果：** RLS 谓词对用户能看到的行返回 `TRUE`，对其他所有行返回 `FALSE`。只有通过谓词判断的行才会被返回。

由于强制检查发生在数据库层，相同的规则会一致地应用于所有访问路径，包括 BI 仪表板、笔记本、临时 SQL 查询、API 和第三方工具。这种一致性正是 RLS 的强大之处：一个规则，处处适用，由引擎强制执行。

大多数引擎还区分了读取侧和写入侧的强制检查。读取谓词控制 `SELECT` 查询返回的内容。写入谓词（通常使用 `WITH CHECK` 子句单独定义）控制用户可以插入、更新或删除哪些行。

这两个谓词可以相同，但并非必须。例如，用户可能被允许读取所有区域的行，但只能插入其所属区域的行。当表接受写入操作时，定义两侧的谓词至关重要，因为跳过写入检查是团队在生产环境中错误配置 RLS 的最常见原因之一。

## 行级安全 vs. 列级安全及其他访问控制

RLS 是几种细粒度访问控制之一，在生产环境中几乎总是与其他控制措施配合使用。下表展示了每种控制措施的定位。

| 控制措施 | 作用范围 | 典型用例 |
| :--- | :--- | :--- |
| **表级授权** | 整个表 | 授予或撤销对表的 `SELECT`、`INSERT`、`UPDATE`、`DELETE` 权限 |
| **行级安全** | 行 | 根据用户身份或角色过滤可见行 |
| **列级安全** | 列 | 隐藏或限制对敏感列的访问（如工资、社保号） |
| **数据脱敏** | 列内的值 | 在运行时动态混淆数据（如显示 `XXX-XX-1234` 而非完整社保号） |

这些控制措施旨在分层使用。一个典型的设置是：使用表级授权控制谁能访问表，使用 RLS 限定哪些行可见，再使用列级安全或数据脱敏来保护这些行中的敏感字段。将它们视为一个堆栈而非一组可选项，能使治理既可审计又具有弹性。某一层的配置错误不会危及其他层。

## 行级安全的常见用例

RLS 是在共享表内强制执行谁能看到什么内容的标准方法，它根据用户属性与关键列（如区域、租户或分类）进行行过滤。当单个数据集必须服务于具有不同可见性规则的多个受众时，大多数团队会采用 RLS。

- **多租户 SaaS：** 使用 `tenant_id` 列和会话上下文，在共享表内隔离每个客户的数据。这避免了为每个租户维护一个模式或一个数据库的操作成本，同时在查询时保持每个客户数据的完全隔离。
- **区域隔离：** 限制销售、人力资源或订单数据，使用户只能看到其国家或地区的记录，而无需按地理区域拆分底层表。
- **部门访问：** 让财务、市场和运营团队访问同一张表，但根据部门或成本中心列映射到不同的行。
- **法规合规：** 强制执行数据驻留规则，例如根据 GDPR 仅允许欧盟员工查看欧盟记录，或根据 HIPAA、CCPA 及行业特定法规限制受保护类别。
- **医疗和临床数据：** 允许临床医生共享患者表，但只能看到自己的患者，在满足 HIPAA 最小必要访问原则的同时，无需跨孤岛复制记录。
- **合作伙伴和供应商门户：** 跨外部合作伙伴共享单个数据集，同时将每个合作伙伴过滤到其自己的记录，这样一张单一事实来源表就能支撑数十个面向合作伙伴的视图。

## 如何实现行级安全：4 个步骤

通用模式在各平台间是一致的，只需在需要时填入特定于供应商的语法。

1.  **确定过滤逻辑：** 决定访问权限的依据：用户 ID、组成员身份、区域、租户 ID 或映射表。过滤逻辑应能从会话上下文或稳定的查找表中推导出来，而不能来自用户在查询时控制的值。
2.  **添加或确认关键列：** 确保表有一个可供过滤器使用的列，例如 `tenant_id`、`region` 或 `owner_id`。如果尚不存在此类列，请在策略生效前计划进行数据回填，并考虑为该列建立索引以保持谓词的高效性。
3.  **定义策略或行过滤器：** 编写谓词，对用户能看到的行返回 `TRUE`，如果表接受写入操作，则单独编写一个写入检查。尽可能将逻辑保留在 SQL 中。大多数引擎优化 SQL 谓词的性能优于调用其他语言的函数。
4.  **使用多个用户身份进行测试：** 以不同角色运行查询，确认正确的行出现，并且没有数据跨租户泄露。包含一个负面测试：没有匹配行的用户应看到空结果，而不是错误；特权用户应单独测试，以确认所有者绕过行为。

## 行级安全的优势

将访问逻辑移入数据层在多个实际方面带来好处。简而言之，数据库成为了访问权限的单一事实来源，而非每个接触数据的应用程序。

- **集中式逻辑：** 访问规则与数据共存，而非分散在应用程序代码或 BI 工具中。
- **一致的执行：** 无论用户是从笔记本、仪表板还是 API 查询，相同的规则都适用。
- **纵深防御：** 如果应用层检查被绕过或存在缺陷，RLS 提供第二层保护。
- **简化应用程序代码：** 开发人员无需在每个查询中手动添加 WHERE 子句。
- **简化审计：** 合规团队可以审查一个策略，而无需跨系统追溯访问逻辑。
- **加速新工具接入：** 新的 BI 工具或笔记本环境无需自定义集成工作即可继承现有的行级规则。

## 行级安全的局限性与风险

RLS 功能强大，但也有一些团队应提前规划的众所周知的陷阱。其中大多数问题只会在生产环境或审计期间暴露，因此值得提前了解。

### 管理员和所有者绕过

在许多数据库中，表所有者和高权限管理员默认会绕过行级安全策略。例如，PostgreSQL 要求设置 `FORCE ROW LEVEL SECURITY` 才能将策略应用于表所有者，其他引擎也存在类似设置。这是一个常见的审计发现：除非你的配置明确强制策略适用于特权用户，否则默认假设他们能看到所有行。在签署策略前，务必从特权会话（而不仅仅是普通会话）测试该策略。

### 无法隐藏列或汇总结果

行级安全策略仅过滤行，但不会隐藏列或阻止聚合结果。如果未将行级安全策略与列或聚合限制结合使用，被禁止查看单个欧盟记录的分析师仍然可以对未过滤的表执行 `SELECT COUNT(*)`。请将行级安全策略与列级安全或数据脱敏结合使用以弥补这一缺口，并考虑是否需要对最敏感的表中的聚合查询本身进行管控。

### 性能开销

每个查询都会应用行级安全策略的谓词，如果过滤逻辑复杂或键列未建立索引，可能会降低性能。请为策略引用的列建立索引，并尽可能保持谓词简单。优先使用直接的 `CASE` 表达式，而不是在过滤器中使用子查询或映射表查找。如果引擎支持，请将用户到行的映射物化为一个小型且索引良好的表，而不是在运行时动态计算。

### 调试复杂性

由行级安全策略导致的空结果集与“无匹配数据”看起来完全相同。开发人员在追踪缺失的行时，常常花费数小时才发现是策略过滤掉了这些行。在开发过程中记录有效的用户身份和策略版本，为工程师提供一种在结果看起来异常时确认行级安全策略是否生效的方法，并将策略文档与表结构放在同一位置以便查找。

### 写入规则配置错误

行级安全策略通常包含两个部分：一个 `USING` 子句用于过滤用户可读取的行，以及一个 `WITH CHECK` 子句用于控制用户可插入或更新的行。只定义其中一个而忽略另一个是经典错误：仅设置读取过滤而无写入检查，会允许用户插入或更新他们本不应拥有的行。当表允许写入时，务必同时定义这两个子句，并在策略审查中作为写入侧测试的一部分。

## Databricks 平台上的行级安全

在 Databricks 平台上，行级安全通过 Unity Catalog（Databricks 统一的数据和 AI 治理层）中的行过滤器进行处理。模式很简单：定义一个 SQL 用户定义函数，该函数对给定用户允许查看的行返回 true，然后将其附加到目标表。该过滤器在查询时自动运行，使用当前用户的身份或会话上下文来决定返回哪些行。

行过滤器在 Databricks SQL、笔记本、作业和连接的 BI 工具中一致地强制执行，无需为每个界面编写自定义逻辑。它们与列掩码协同工作，实现完整的细粒度访问控制，并且每个涉及已过滤表的查询都会被捕获到 Unity Catalog 的血缘和审计日志中，因此治理和安全团队可以准确查看哪些策略应用于哪些表，以及哪些用户查询了什么。

## 常见问题

什么是动态行级安全？动态行级安全在查询时使用当前用户的身份或会话上下文来评估访问规则，因此同一策略对不同用户返回不同结果。所有现代行级安全实现都采用这种方式，包括 Databricks 的 ABAC 策略、行过滤器和动态视图。

行级安全和列级安全有什么区别？行级安全限制用户可以看到哪些行；列级安全限制用户可以看到哪些列，通常用于隐藏敏感字段，如薪资或社会安全号码。大多数生产部署会同时使用两者。

仅靠行级安全是否足以保护敏感数据？不够。行级安全处理行的可见性，但不会掩盖列值、阻止聚合查询或替代身份和访问管理。请将其与列级安全、表级授权和审计日志结合使用，作为纵深防御策略的一部分。

Databricks 如何实现行级安全？通过 Unity Catalog，提供三种选项：ABAC 策略、表级行过滤器和动态视图。ABAC 推荐用于大规模治理；行过滤器和动态视图适用于更定制化的需求。

行级安全是否会影响查询性能？是的，但影响通常是可控的。保持策略逻辑简单，为策略引用的列建立索引，并优先使用 SQL UDF 而非 Python UDF。在策略变更前后对查询进行性能分析，以便及早发现性能回退。
