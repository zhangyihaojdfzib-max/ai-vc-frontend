# Agent Engineering：一门新学科

如果你构建过 Agent（智能体），你就会知道“在我的机器上能运行”和“在生产环境中能运行”之间的差距可能非常巨大。传统软件假设你大致了解输入并能定义输出。而 Agent（智能体）两者都不提供：用户可以说任何话，可能的行为空间是完全开放的。这就是它们强大的原因——也是它们可能以你意想不到的方式偏离轨道的原因。

过去三年，我们目睹了成千上万的团队在这一现实面前挣扎。那些成功将可靠产品部署到生产环境的团队——例如 Clay、Vanta、LinkedIn 和 Cloudflare 等公司——并没有遵循传统的软件开发手册。他们正在开创一种新事物：**Agent Engineering（智能体工程）**。

### 什么是 Agent Engineering？

Agent Engineering（智能体工程）是将非确定性的大语言模型系统迭代优化为可靠生产体验的过程。这是一个循环往复的过程：**构建、测试、部署、观察、优化、重复**。

![](/images/posts/53586c966014.png)

![](/images/posts/53586c966014.png)

这里的关键在于，部署并非最终目标。它只是你持续前进以获得新见解并改进你的 Agent（智能体）的方式。为了做出有意义的改进，你需要了解生产环境中正在发生什么。这个循环进行得越快，你的 Agent（智能体）就越可靠。

我们将 Agent Engineering（智能体工程）视为一门结合了三种技能组合协同工作的新学科：

*   **产品思维**定义范围并塑造 Agent（智能体）行为。这包括：
    *   编写驱动 Agent（智能体）行为的提示词（通常长达数百或数千行）。良好的沟通和写作能力在此至关重要。
    *   深刻理解 Agent（智能体）所要复现的“待完成工作”。
    *   定义评估标准，以测试 Agent（智能体）是否按照“待完成工作”的意图执行。
*   **工程**构建使 Agent（智能体）具备生产就绪性的基础设施。这包括：
    *   编写供 Agent（智能体）使用的工具。
    *   为 Agent（智能体）交互开发 UI/UX（包括流式传输、中断处理等）。
    *   创建健壮的运行时，以处理持久执行、人工介入暂停和内存管理。
*   **数据科学**衡量并随时间推移改进 Agent（智能体）性能。这包括：
    *   构建系统（评估、A/B 测试、监控等）来衡量 Agent（智能体）的性能和可靠性。
    *   分析使用模式和错误（因为与传统软件相比，用户使用 Agent（智能体）的方式范围更广）。

### Agent Engineering 体现在何处

Agent Engineering（智能体工程）并非一个新的职位头衔。相反，它是现有团队在构建能够推理、适应且行为不可预测的系统时所承担的一系列职责。如今，那些部署可靠 Agent（智能体）的组织正在扩展工程、产品和数据团队的技能，以满足非确定性系统的需求。

以下是该实践通常出现的领域：

*   **软件工程师和机器学习工程师**：编写提示词并为 Agent（智能体）构建使用工具，追踪 Agent（智能体）做出特定工具调用的原因，并优化底层模型。
*   **平台工程师**：构建处理持久执行和人工介入工作流的 Agent（智能体）基础设施。
*   **产品经理**：编写提示词，定义 Agent（智能体）范围，并确保 Agent（智能体）解决正确的问题。
*   **数据科学家**：衡量 Agent（智能体）可靠性并识别改进机会。

这些团队拥抱快速迭代，你经常会看到软件工程师追踪错误，并将洞察结果交给产品经理来调整提示词；或者产品经理识别出范围问题，需要工程师提供新工具。每个人都认识到，强化 Agent（智能体）的真正工作是通过观察生产行为并根据所学知识进行系统性优化的循环来完成的。

### 为什么需要 Agent Engineering，为什么是现在？

两个根本性的转变使得 Agent Engineering（智能体工程）成为必要。

首先，大语言模型已经足够强大，能够处理复杂的多步骤工作流。我们已经看到 Agent（智能体）承担整个工作，而不仅仅是任务。Clay 使用 Agent（智能体）处理从潜在客户研究到个性化外联和 CRM 更新的所有事情。LinkedIn 使用 Agent（智能体）扫描庞大的人才库进行招聘，对候选人进行排名并即时呈现最匹配的人选。我们正开始跨越一个门槛，即 Agent（智能体）在生产环境中开始交付有意义的商业价值。

其次，这种强大能力伴随着真正的不确定性。简单的大语言模型应用虽然是非确定性的，但其行为往往更受限制。Agent（智能体）则不同。它们跨多个步骤进行推理，调用工具，并根据上下文进行调整。正是这些使 Agent（智能体）有用的特性，也让它们的行为与传统软件不同。这通常意味着：

*   **每个输入都是边缘情况**。当用户可以用自然语言提出任何问题时，就不存在所谓的“正常”输入。当你输入“让它出彩”或“做你上次做的事，但要不同”时，Agent（智能体）（就像人一样）可能会以不同的方式解读提示词。
*   **你无法用旧方式调试**。因为大量逻辑存在于模型内部，你必须检查每个决策和工具调用。微小的提示词或配置调整可能导致行为发生巨大变化。
*   **“正常工作”不是二元的**。一个 Agent（智能体）可以拥有 99.99% 的正常运行时间，同时仍然可能偏离轨道或出错。对于重要问题，并不总是有简单的“是/否”答案，例如：Agent（智能体）是否做出了正确的调用？是否正确使用了工具？是否遵循了你指令背后的意图？

当你把所有这一切结合起来——Agent（智能体）运行着真实、高影响力的工作流，但其行为方式却是传统软件无法解决的——这就催生了一门新学科的机遇和需求。Agent Engineering（智能体工程）让你能够驾驭大语言模型的力量，同时构建你真正可以在生产环境中信任的系统。

### Agent Engineering 在实践中是怎样的？

Agent Engineering（智能体工程）遵循与传统软件开发不同的原则。为了实现可靠的 Agent（智能体）系统，部署是你学习的方式，而不是你在学习之后才做的事情。

我们看到成功的工程团队遵循类似以下节奏进行 Agent（智能体）开发：

- **构建智能体的基础。** 从设计智能体的基础开始，无论是简单的带工具的LLM调用，还是复杂的多智能体系统。您的架构取决于您需要多少工作流（确定性的分步流程）与自主性（LLM驱动的决策）。
- **基于可设想的场景进行测试。** 针对示例场景测试您的智能体，以发现提示词、工具定义和工作流中的明显问题。与传统软件可以规划用户流程不同，您无法预知用户与自然语言输入交互的每一种方式。将思维模式从“详尽测试，然后发布”转变为“合理测试，发布以了解真正重要的方面”。
- **发布以观察真实世界行为。** 一旦发布，您将立即开始看到未曾考虑过的输入，并且每个生产环境追踪都会显示您的智能体实际需要处理什么。
- **观察。** 追踪每一次交互，查看完整的对话、调用的每个工具，以及影响智能体每个决策的确切上下文。对生产数据运行评估，以衡量智能体质量，无论您关注的是准确性、延迟、用户满意度还是其他标准。
- **优化。** 一旦识别出故障模式，就通过编辑提示词和修改工具定义来进行优化。这是一个持续的过程，您可以将有问题的案例添加回示例场景集，以进行回归测试。
- **重复。** 发布您的改进，并观察生产环境中的变化。每个周期都会让您对用户如何与您的智能体交互，以及在您的上下文中可靠性实际意味着什么，有新的认识。

### 工程实践的新标准

当今发布可靠智能体的团队都有一个共同点：他们已不再试图在发布前将智能体做到完美，而是开始将生产环境视为他们的主要老师。换句话说，他们追踪每个决策，大规模进行评估，并在几天内而非几个季度内发布改进。

智能体工程正在兴起，因为机遇需要它。智能体现在可以处理以前需要人类判断的工作流，但前提是您能让它们足够可靠以赢得信任。没有捷径，只有系统性的迭代工作。问题不在于智能体工程是否会成为标准实践，而在于您的团队能以多快的速度采用它，以释放智能体的潜力。
