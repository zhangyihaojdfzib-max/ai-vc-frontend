# 持续检测：消除WAF“记录与拦截”的权衡

2026-03-04

- Daniele Molteni

![](/images/posts/38b16e600478.png)

传统的Web应用防火墙通常需要对其规则进行大量手动调整，才能安全地拦截恶意流量。当部署新应用时，安全团队通常从仅记录模式开始，通过筛选日志来逐步评估哪些规则可以安全地进入拦截模式。这个过程旨在最大限度地减少误报，同时不影响合法流量。它是手动的、缓慢的且容易出错。

团队被迫做出权衡：要么在记录模式下获得可见性，要么在拦截模式下获得保护。当一条规则拦截了一个请求时，评估就会停止，您将无法了解其他签名会如何评估该请求——这些宝贵的洞察本可以帮助您调整和加强防御。

今天，我们通过推出托管规则的下一个演进版本来解决这个问题：攻击签名检测。

启用后，此检测会在采取任何行动之前检查每个请求是否存在恶意负载，并附加丰富的检测元数据。您可以在不牺牲保护或性能的情况下，获得每个签名匹配的完整可见性。上线过程变得简单：流量被分析，数据不断积累，您可以准确看到哪些签名被触发以及原因。然后，您可以基于过去的流量构建精确的缓解策略，从而降低误报风险。

但我们更进一步。我们正在超越仅请求分析，迈向更强大的功能：全事务检测。

这种新的检测不是仅仅查看传入的请求，而是关联整个HTTP事务：请求和响应。通过分析完整的上下文，与传统的仅请求签名引擎相比，我们显著减少了误报。更重要的是，我们发现了其他方案可能遗漏的威胁，例如反射型SQL注入、微妙的数据外泄模式，以及仅在响应中暴露的危险错误配置。

攻击签名检测现已提供早期访问——[点击此处注册](https://www.cloudflare.com/lp/attack-signature-detection/)以表达兴趣。全事务检测正在开发中；[点击此处注册](https://www.cloudflare.com/lp/full-transaction-detection/)，以便在其准备就绪时成为首批试用者。

## 持续检测框架

为了在不拖慢互联网速度的前提下提供对您流量的完全可见性，我们必须改变对请求生命周期的思考方式。对于选择启用的客户，攻击签名检测现在是“持续开启”的。这意味着一旦流量被代理，所有检测签名都会在每个请求上执行，结果会立即在安全分析中可见。

这个“持续检测”框架将检测与缓解分离开来。检测持续运行，用关于触发检测的元数据来丰富分析。这些元数据也会作为一个新字段添加到请求中，客户可以在安全规则中使用它来创建自定义策略。

将恶意负载的检测与安全规则采取的行动分离开来，是持续检测框架的核心。这种方法增强了分析体验，并在部署新防护措施时增加了信心。

我们现有的机器人评分和攻击评分检测已经遵循这种方法。攻击签名检测提供与我们托管规则产品相同的覆盖范围，但在这个新框架内运行。

这会给请求带来额外的延迟吗？不会——这个模型是为效率而设计的。如果客户没有基于某个检测创建拦截规则，那么该检测可以在请求发送到源服务器**之后**执行，确保检测本身不会给流量带来额外的延迟。因此，在上线时，检测默认启用，但不会影响流量性能。当创建规则时，检测会与可能经历额外延迟的请求内联执行。具体值取决于应用的流量特征。

## 攻击签名检测

与传统的基于规则的系统（如Cloudflare托管规则集）相比，这种新的检测在Web应用安全方面提供了实质性的进步。这种方法使得识别恶意Web负载和部署安全规则对用户更加友好。

Cloudflare托管规则集是我们的分析师团队为常见攻击向量开发检测的地方，包括SQL注入、跨站脚本、远程代码执行和特定的常见漏洞与暴露。分析师通常每周发布新规则，并为高关注度的漏洞（例如最近的React2Shell发布）进行紧急发布。目前，我们的托管规则集中有超过700条活跃的托管规则。新的检测也被称为签名规则或简称签名。它们采用与托管规则相同的启发式方法，但不会直接对流量应用操作。

每个签名都由一个Ref ID（类似于托管规则集的规则ID）唯一标识，并标有类别和置信度。类别指定了签名针对的攻击向量，而置信度级别则表明误报（在合法流量上触发）的可能性。一条规则只能有一个置信度级别，但可以有多个类别。

类别表示规则所指的攻击向量。类别列表很长，但包括诸如SQLi、XSS、RCE或带有编号的特定CVE等标签。

置信度字段根据是否有至少一个来自相应组的签名与流量匹配，分为两个值。

| 置信度 | 描述 |
| :--- | :--- |
| **高** | 这些签名旨在实现高真阳性率和低误报率，典型适用于那些负载可识别且不会拦截合法流量的CVE。它们的功能类似于托管规则集的默认配置。 |
| **中** | 这些签名在托管规则集中默认关闭，可能会根据您的流量导致误报。在拦截匹配这些规则的流量之前，请评估其可能对应用产生的影响。 |

检测对请求的分析会填充三个字段。这些字段可在安全分析和我们用于安全规则的核心引擎——边缘规则引擎中访问。

| 字段 | 使用位置 |
| :--- | :--- |
| `cf.waf.signature.request.confidence` | 数组。汇总与匹配签名关联的置信度分数。 | 分析和安全规则 |
| `cf.waf.signature.request.categories` | 数组。汇总与匹配签名关联的类别。 |
| `cf.waf.signature.request.ref` | 数组。汇总匹配签名的Ref ID，最多10个。 |

### 在安全分析中分析您的数据

安全分析是Cloudflare应用安全工具箱的核心，它提供了一个全面的、数据驱动的视图，展示签名如何与您的Web流量交互。它为您提供了理解、衡量和优化Web保护所需的工具。将分析与签名结合使用的常见用例包括：在上线过程中设计安全态势、验证最常见的攻击尝试以及创建例外来处理误报。

一旦新应用通过Cloudflare代理，攻击签名检测就会开始用数据填充您的仪表板。第一步是检查按类型和签名分类的聚合匹配项，以确认所有潜在攻击都被拦截。分析师可以通过查看签名的顶级统计数据、过滤数据以显示请求是被拦截、从缓存提供还是被允许到达源服务器来完成此操作。如果发现任何恶意请求到达了源服务器，分析师可以快速实施安全规则。

按相应类别或签名分类的、匹配攻击签名的总请求量细分。

分析功能可提供对攻击模式的洞察，例如基于流量随时间变化的最常见CVE。此功能旨在快速识别针对应用程序的主要攻击载荷，并验证当前针对相关CVE的防护措施是否有效。例如，分析师可以监控针对应用程序特定部分（如`/api/`）的攻击频率，或确认已知恶意载荷（如React2Shell）是否到达特定端点（如`POST /_next/Node.js`路径）。分析筛选器和攻击分析工具均可用于执行此类调查。

安全分析中的可视化功能提供了针对`/api/`端点的恶意载荷时间序列视图。该视图对数据进行分组，以突出显示按流量排名的前五大CVE。

分析功能还有助于创建例外规则和识别误报。例如，特定规则匹配次数的增加可能表明是误报，而非实际攻击。例如，允许用户提交富HTML内容的应用程序（如内容管理系统或支持工单系统）可能合法包含与较通用XSS特征匹配的标记。在这些情况下，可以对受影响的端点应用限定范围的例外规则，同时保持应用程序其余部分的防护功能处于启用状态。

这种方法对于评估中等置信度的特征尤其有用，这类特征在积极拦截和误报风险之间取得了平衡。该工具支持针对历史流量进行"假设"场景测试，以实证确定生产环境性能。此过程有助于判断中等置信度特征是否适用于整体流量模式，或者是否需要因高误报率而将其部署限制在特定URL或请求类型。

通常，对历史流量匹配率极低的特征可以更安全地以拦截模式部署，而不会对合法流量造成显著干扰。为实现这种置信度水平，安全分析提供了用于深入取证调查的工具。

除了即时检测，防御管理的一个关键方面是自定义安全态势的能力。用户界面提供了所有安全特征的可搜索目录，允许您浏览完整列表并了解每个特征旨在应对的具体威胁。

提供可搜索的特征目录，提供关键检测的更多详细信息，以帮助客户了解威胁和补救措施。

### 创建安全规则

在分析数据并建立对特征在过去流量中表现的信心后，您可以轻松创建自定义规则，根据检测结果处理流量。例如，如果您想创建一个拦截匹配高置信度特征的请求的策略，可以创建以下规则：

创建规则以拦截匹配高置信度特征的请求。

这相当于Cloudflare托管规则集的默认部署。

如果您想拦截匹配至少一条规则的所有请求，将添加中等置信度标签。这相当于启用Cloudflare托管规则集的所有规则。或者，您可以配置多条规则，对高置信度检测应用更严格的操作（如"拦截"），对中等置信度检测应用较宽松的操作（如"质询"）。

通过同时选择高和中等置信度，可以在任何特征匹配时触发规则。

要创建拦截特定CVE或攻击向量的规则，您将使用类别。规则构建器允许您将攻击向量类别标签与所有现有HTTP请求数据结合使用。这使您能够创建精细的规则（或例外），并根据应用程序的不同部分调整安全态势。

客户可以创建规则来拦截（或允许）匹配特定CVE或攻击类别的请求。

要基于特定特征创建规则，您可以使用参考ID。您可以通过在规则构建器中探索可用的攻击特征规则来找到正确的参考ID。如果您想创建例外规则以管理误报，这尤其有用。

客户可以直接从规则构建器浏览特征规则。

所有客户将继续访问我们经典的托管规则集。当攻击特征检测广泛可用时，客户将能够选择最适合其需求的部署模式，无论是攻击特征检测还是托管规则。我们的分析团队确保新检测同时在托管规则集和攻击特征检测中发布。

## 全事务检测

传统的网络攻击检测主要关注"问"：HTTP请求。然而，请求只揭示了故事的一半。要知道攻击是否真正成功，必须查看"答"：HTTP响应。

通过将请求和响应元数据合并到单个检测事件中，我们可以显著减少误报，并识别仅检测请求的系统会遗漏的成功利用。

例如，考虑一个在查询参数中包含常见SQL注入字符串的请求。

```
GET /user?id=1' UNION SELECT username, password FROM users--
```

传统WAF会看到`UNION SELECT`模式并拦截它。但是，如果应用程序实际上不易受攻击，这可能是一个误报——例如安全研究人员测试自己的网站。

通过全事务检测，系统会记录请求中的SQLi特征，但会等待响应。如果源站响应`500 Internal Server Error`或标准`404`，则"成功利用"的置信度较低。如果源站响应`200 OK`且响应体包含匹配"敏感数据"特征（如用户名列表）的字符串，系统将标记为成功利用确认。

首先，我们正在推出几个检测类别，并计划随时间推移扩展此列表。以下是我们目前关注的三个领域，以及您将看到的一些标记：

- **利用尝试**。该检测通过检查整个HTTP请求-响应周期来提供网络攻击检测。它专注于三个关键领域：通过恶意特征识别XSS和SQLi等输入利用；阻止漏洞探测等自动化滥用；通过将可疑请求与异常服务器响应相关联来确认成功利用。

- **数据暴露和窃取信号**。此框架还使我们能够捕获在传入时看似合法流量的数据窃取。对`/api/v1/export`的请求是标准管理操作。但如果该特定请求触发的响应包含5,000个信用卡号（例如通过Luhn算法特征识别），则该事务将被标记为数据暴露。

- **配置错误**。暴露的管理界面通常是攻击向量。传统安全检查会遗漏此类配置错误，因为流量本身看起来是有效的（真实的端点或管理页面）。问题不在于流量，而在于其公开可访问性。我们根据客户数据中常见的现实配置错误来优先检测，例如公开的未认证Elasticsearch集群、互联网可达的管理面板，以及暴露的Apache敏感端点。

检测机制与攻击特征类似，会将结果存储在两个特定字段中。这些字段可在我们的仪表板中访问，并记录在安全分析模块中。

`cf.waf.signature.response.categories`

`cf.waf.signature.response.ref`

最初，我们专注于通过分析功能提供匹配请求的可视性。通过呈现潜在漏洞利用事件，我们为客户提供可用于事件响应的信息，以便在其基础设施和软件栈中进行针对性修复。我们未来的计划包括将安全规则扩展到响应阶段，届时客户将能够通过创建策略，基于这些检测结果来阻止响应。
