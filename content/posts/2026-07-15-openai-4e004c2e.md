# 面向开发者推出 GPT‑5.1

今天，我们在 API 平台发布 GPT‑5.1，这是 GPT‑5 系列的下一个模型，它在智能与速度之间取得平衡，适用于广泛的 Agent（智能体）和编码任务。GPT‑5.1 会根据任务的复杂程度动态调整思考时间，从而在更简单的日常任务上显著提升速度并提高 Token 效率。该模型还具备“无推理”模式，可在不需要深度思考的任务上更快响应，同时保持 GPT‑5.1 的前沿智能水平。

为了让 GPT‑5.1 更加高效，我们推出了扩展提示缓存功能，缓存保留时间最长可达 24 小时，从而以更低成本为后续问题提供更快的响应。我们的Priority Processing⁠（在新窗口中打开）客户在使用 GPT‑5.1 时，还将体验到比 GPT‑5 更明显的性能提升。

在编码方面，我们与 Cursor、Cognition、Augment Code、Factory 和 Warp 等初创公司密切合作，改进了 GPT‑5.1 的编码特性、可操控性和代码质量。总体而言，GPT‑5.1 在编码时感觉更直观，并且在完成任务时能通过面向用户的更新进行更有效的沟通。

最后，我们为 GPT‑5.1 引入了两个新工具：一个`apply_patch`工具，用于更可靠地编辑代码；以及一个 shell 工具，让模型能够运行 shell 命令。

GPT‑5.1 是 GPT‑5 系列的下一步进展，我们计划继续投资于更智能、能力更强的模型，以帮助开发者构建可靠的 Agent（智能体）工作流。

### 新的“无推理”模式

开发者现在可以通过将 reasoning_effort 设置为 'none' 来使用 GPT‑5.1 而无需推理。这使得模型在延迟敏感的使用场景中表现得像非推理模型，同时具备 GPT‑5.1 的高智能和高效工具调用的额外优势。与使用“最低”推理的 GPT‑5 相比，无推理的 GPT‑5.1 在并行工具调用（这本身提高了端到端任务完成速度）、编码任务、遵循指令和使用搜索工具方面表现更好——并在我们的 API 平台支持web search⁠（在新窗口中打开）。Sierra⁠（在新窗口中打开）分享说，GPT‑5.1 在“无推理”模式下，在其实际评估中“与 GPT‑5 最低推理相比，低延迟工具调用性能提升了 20%”。

通过引入 'none' 作为 reasoning_effort 的一个值，开发者现在对其使用场景中速度、成本和智能之间的平衡有了更大的灵活性和控制力。GPT‑5.1 默认使用 'none'，这非常适合延迟敏感的工作负载。我们建议开发者在复杂度较高的任务中选择“低”或“中等”，在智能和可靠性比速度更重要时选择“高”。

### 扩展提示缓存

扩展缓存通过允许提示在缓存中保持活跃长达 24 小时（而非目前支持的几分钟）来提高推理效率。更长的保留窗口使得更多后续请求可以利用缓存的上下文——从而降低延迟、减少成本，并为多轮对话、编码会话或知识检索工作流等长时间运行的交互提供更流畅的性能。

提示缓存定价保持不变，缓存的输入 Token 比未缓存的 Token 便宜 90%，并且缓存写入或存储不收取额外费用。要使用 GPT‑5.1 的扩展缓存，请在 Responses 或 Chat Completions API 中添加参数`“prompt_cache_retention='24h'”`。有关更多详细信息，请参阅prompt caching 文档⁠（在新窗口中打开）。

## GPT‑5.1 中的新工具

我们为 GPT‑5.1 引入了两个新工具，以帮助开发者在 Responses API 中充分利用该模型：一个自由格式的`apply_patch`工具，使代码编辑更加可靠，无需 JSON 转义；以及一个`shell`工具，让模型能够编写命令在您的本地机器上运行。

### Apply_patch 工具

自由格式的`apply_patch`工具允许 GPT‑5.1 使用结构化差异在代码库中创建、更新和删除文件。模型不是仅仅建议编辑，而是发出补丁操作，应用程序应用这些操作并报告结果，从而实现迭代的多步骤代码编辑工作流。

要在 Responses API 中使用 `apply_patch` 工具，请将其包含在工具数组中，格式为 `"tools": [{"type": "apply_patch"}]`，并在输入中包含文件内容，或为模型提供与文件系统交互的工具。模型将生成 `apply_patch_call` 项，用于创建、更新或删除包含差异（diffs）的文件，您可将这些差异应用到文件系统中。有关如何集成 `apply_patch` 工具的更多信息，请查阅我们的[开发者文档](opens in a new window)。

### Shell 工具

Shell 工具允许模型通过受控的命令行界面与本地计算机交互。模型会提出 shell 命令；开发者的集成环境执行这些命令并返回输出结果。这形成了一个简单的计划-执行循环，使模型能够检查系统、运行实用程序并收集数据，直至完成任务。

要在 Responses API 中使用 Shell 工具，开发者可以将其包含在工具数组中，格式为 `"tools": [{"type": "shell"}]`。API 将生成 `"shell_call"` 项，其中包含要执行的 shell 命令。开发者在本地环境中执行这些命令，并在下一次 API 请求中通过 `"shell_call_output"` 项返回执行结果。更多信息请查阅我们的[开发者文档](opens in a new window)。

## 定价与可用性

GPT‑5.1 和 gpt-5.1-chat-latest 现已面向 API 中所有付费层级的开发者开放。[定价和速率限制](opens in a new window)与 GPT‑5 相同。我们还在 API 中发布了 `gpt-5.1-codex` 和 `gpt-5.1-codex-mini`。虽然 GPT‑5.1 在大多数编码任务中表现出色，但 gpt-5.1-codex 模型针对 Codex 或类似 Codex 的框架中长期运行的 Agent（智能体）编码任务进行了优化。

开发者可以开始使用我们的 GPT‑5.1 [开发者文档](opens in a new window)和[模型提示词指南](opens in a new window)进行构建。我们目前不计划在 API 中弃用 GPT‑5，如果决定这样做，我们会提前通知开发者。

## 下一步计划

我们致力于迭代部署最强大、最可靠的模型，用于真实的 Agent（智能体）和编码工作——这些模型能够高效思考、快速迭代、处理复杂任务，同时让开发者保持流畅的工作状态。凭借自适应推理、更强的编码性能、更清晰的面向用户更新，以及 `apply_patch` 和 `shell` 等新工具，GPT‑5.1 旨在帮助您更顺畅地进行构建。我们将继续在此领域大力投入：未来几周和几个月内，您将看到更强大的 Agent（智能体）和编码模型。
