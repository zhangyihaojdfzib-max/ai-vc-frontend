# 介绍 Meerkat：一项全球共识实验

2026-07-08

- James Larisch
- Bob Halley
- João Pedro Leite

![](/images/posts/bae093525941.png)

Cloudflare 的许多内部服务需要从我们全球 330 多个数据中心读取和修改相同的控制平面状态。它们需要保证不同的读取者永远不会看到不一致的状态，并且即使某些数据中心或链路发生故障，系统仍能保持写入可用性。

但 Cloudflare 的网络运行在整个互联网上，而互联网是一个不可预测的地方。服务器和数据中心会宕机。队列会填满。链路和电缆会被切断。这些条件使得运行一个保证强一致性（例如，保证所有读取者都能读取到所有先前的写入）的全球可用数据系统变得困难，因为恶劣的条件阻碍了分布式系统副本之间可靠同步数据的能力。

在不利的网络条件下安全同步数据的一种方法是通过共识算法，该算法允许一组机器就相同的值序列达成一致，例如键值存储的 put 和 get 操作，只要多数机器保持存活并能通信即可。

不幸的是，像 Raft 这样普遍部署的共识算法在像 Cloudflare 这样的广域网中表现不佳，因为它们依赖于领导者和超时。领导者是唯一允许进行写入的副本，如果它因崩溃或网络降级而失效，系统将变得不可用，直到其他副本超时并选举出新的领导者。而在延迟不可预测的网络中，这些超时值很难配置。

我们曾经历过多次由共识驱动系统中领导者不可用导致的事件。

因此，在过去的一年里，Cloudflare 的研究团队一直在构建一个名为 Meerkat 的新型分布式共识服务，该服务由一种名为 QuePaxa 的共识算法驱动，该算法由 Tennage & Băsescu 等人于 2023 年发表。QuePaxa 与 Raft 的不同之处在于，所有副本都可以随时执行写入，并且进度永远不会因超时而停止，这使得它非常适合 Cloudflare 的网络。我们在 Meerkat 的共识日志之上构建应用程序，例如事务性键值存储和租约系统。据我们所知，这将是 QuePaxa 在全球范围内的首次工业部署。

Meerkat 是一项仍在开发中的实验性共识服务。它最初被设计用于管理小块的控制平面状态（例如，复制数据库的领导权），因此在短期内将仅限内部使用。本文介绍了 Meerkat，并为后续与 Meerkat 相关的博客文章奠定了基础。

## 我们对全球控制平面数据系统的需求

许多 Cloudflare 服务从分布在全球各地的多台机器上读取和写入控制平面数据，这些数据有助于这些服务正确运行。控制平面数据的一个例子是放置信息：某些资源（如 AI 模型实例）存储在哪里。另一个例子是领导权信息：当前允许哪台机器对数据库执行写入。

控制平面数据必须在特定类型的故障情况下既保持强一致性又保持可访问性。

在本节中，我们将精确描述我们对 Cloudflare 共识服务的一致性和容错性要求。我们使用键值存储作为运行在共识服务之上的应用程序的示例，尽管其他应用程序（例如，分布式租约/锁）也是可能的。

### 强一致性

分布式数据系统的一致性级别描述了系统在接收并发读取和写入时允许表现出的异常行为类型。考虑一个跨多个节点存储单个数值 `x = 6` 的分布式键值存储。同时考虑以下写入序列。这些写入以尽力而为的方式提交给不同的节点，并且可能以任何顺序到达：

1. x = x + 1
2. x = x / 2

x = x + 1

x = x / 2

在弱一致性级别下，写入可以重新排序。在更强的一致性模型下，写入不能重新排序，但读取可以。在最强的一致性级别下，操作严格按照实时发生的顺序排序。这个属性被称为线性一致性。

在 Cloudflare，许多服务需要线性一致性。与较弱的一致性形式不同，线性一致性使程序员无需考虑数据系统可能表现出的所有异常行为。相反，他们可以像在单线程机器上推理本地内存一样推理分布式系统：写入之后的所有读取都将看到该写入。关于弱一致性危险的更多阅读材料，请查看 Marc Brooker 的这篇文章。

（如果你想知道，Meerkat 的键值存储也提供了可串行化性，我们将在未来的文章中介绍。）

### 容错性

系统的容错级别描述了系统在发生灾难之前能够处理的故障类型。灾难通常是对系统旨在维护的属性的违反，例如，对同一键进行两次连续读取（中间没有写入）永远不会看到不同的值，或者系统保持写入可用性。这些故障包括网络故障或延迟、机器崩溃和机器重启。系统通常会明确处理某些故障，但不会处理其他故障（你无法处理所有故障，因为宇宙总是可能达到热寂）。例如，某些键值存储可能保证只要系统中三分之二的机器能够通信且不崩溃，就保持写入可用性，但如果机器被攻破并开始发送恶意消息，则不做出任何承诺。

首先，只要满足以下条件，数据系统应保持对位于我们任何数据中心的客户端的写入和读取可用性：

1.  我们系统中的多数机器处于存活状态并且能够相互通信。（形式上，我们在一个由 `2f + 1` 台机器组成的系统中容忍 `f` 个故障。）
2.  客户端可以联系系统中任何连接到多数存活机器的机器。

我们系统中的多数机器处于存活状态并且能够相互通信。（形式上，我们在一个由 `2f + 1` 台机器组成的系统中容忍 `f` 个故障。）

客户端可以联系系统中任何连接到多数存活机器的机器。

这意味着单台机器故障或单条链路的网络降级不会影响系统的可用性。正如我们稍后将看到的，基于 Raft 的系统不提供此属性。

其次，只要系统中的任何参与者没有主动恶意行为（当然，也没有错误），数据系统就保持正确。我们稍后将从共识安全性的角度定义正确性，但粗略地说，这意味着没有两台最新的机器会对世界状态产生分歧（例如，一台认为 `key1=1`，而另一台认为 `key1=2`）。

总之，即使机器崩溃、机器重启、网络故障或降级、数据中心宕机等，系统也必须保持正确（尽管与基于 Raft 的系统一样，我们不处理拜占庭故障）。

## 介绍 Meerkat

Meerkat 是一项共识服务，我们可以在其上构建具有上述属性（强一致性和容错性）的应用程序，例如键值（KV）存储。为了理解 Meerkat 的工作原理，我们首先概述 Meerkat 的总体架构，然后描述 Meerkat 选择的共识算法如何帮助提供强一致性和容错性。

使用Meerkat服务的开发者需要请求一组Meerkat副本集群。每个副本与其他所有副本相连。每个副本都参与共识算法，并可以处理读写请求。开发者可以指定允许托管其副本的数据中心，Meerkat会自动进行部署。

为了与集群交互，开发者的客户端会向集群中的任意副本发送特定于应用的请求。单个副本可以托管多种类型的应用，但最简单的应用是键值存储，因此最简单的应用请求类型是KV的get或put操作。副本会返回特定于应用的响应（例如，get请求所查询的记录）。需要注意的是，KV读取（get）保证能读取到最新信息。

### Meerkat的日志

在底层，副本将应用请求（如get和put）转换为日志事件。该副本使用共识算法将每个日志事件分发给所有其他副本，从而确保所有副本维护完全相同的日志事件序列（实际上，某个副本可能滞后，但绝不会记录不同的条目）。这些事件是任意的——Meerkat的核心并不关心其具体内容。Meerkat应用关心的是日志事件的内容。每个Meerkat副本会"托管"多个Meerkat应用（例如键值存储），这些应用读取日志事件并构建状态。（注意，每个副本只属于一个集群。）

例如，KV Meerkat应用会根据日志事件构建一个内存中的键值存储。因此，当客户端发送类似put k1 v1的写入请求时，接收该请求的副本会将此写入操作放入一个日志事件中，并分发给所有副本。如果随后有人向另一个副本发送put k1 v11，该事件也会被分发给所有副本。由于所有正常运行的副本拥有相同的日志，这些副本可以按顺序应用日志中的操作，从而构建完全相同的状态。需要注意的是，get请求也会创建分布式日志事件（这是为了实现线性一致性，将在下一节解释）。

### Meerkat的日志如何实现强一致性

Meerkat保证：如果客户端A执行put k1 v1，随后客户端B执行put k1 v11，再随后客户端C执行get k1（使用一致性读取），那么客户端C将始终读取到v11。即使每个请求提交给不同的副本，且这些副本随机分布在全球各地，这一保证依然成立。这就是线性一致性。要理解Meerkat如何实现这一点，我们需要更详细地审视Meerkat的日志。

Meerkat日志是一个由槽位组成的序列。每个槽位是一个可以包含或不包含事件的容器。包含事件的槽位称为已决槽位。日志中除最后一个正在决策的槽位外，所有槽位都是已决的。Meerkat的一个不变性是：如果任意两个副本对某个槽位的值做出了决策，那么这些值必定相同。换句话说，对于已决槽位的值，任何两个副本都不会产生分歧（尽管一个副本可能认为最后一个槽位为空，而另一个副本不这么认为）。这一特性有助于保证我们在上一节中描述的期望属性。

为了对日志中最后一个（空）槽位的值进行决策，Meerkat副本运行一个分布式共识算法。共识算法允许一组通过网络通信的机器就某个已决槽位的值达成一致。只要超过半数（多数）的副本处于活跃状态，我们的共识算法就能正常工作。

因此，如果当前日志包含两个条目，且某个客户端向一个副本提交了put k1 v11，该副本会为槽位3触发共识算法。但另一个客户端可能向另一个副本为槽位3提交了put k1 v111。共识算法确保槽位3只有一个这样的提案胜出。具体来说，它确保至少多数副本就同一个提案达成一致，从而将其决策为槽位3的值。非多数副本永远无法决策出不同的提案，但可能完全不知道槽位3已被决策。

为了理解这如何为我们的键值存储提供线性一致性，考虑一个写入操作后跟一个读取操作的场景。副本Z提出了put k1 v11，该提案被多数副本决策为槽位3的值，但副本Y不在其中。随后，一个读取操作在副本Y上执行get k1。副本Y认为槽位3为空，因此为槽位3提出get k1。关键的是，多数副本不会同意将该事件放入槽位3，因为该槽位已被决策。它们会强制副本Y（通过接收旧决策）将put k1 v11放入槽位3，并为槽位4提出get k1，从而在日志中将读取操作线性化到写入操作之后。（如果该副本无法联系到多数副本，它将无法完成读取操作。）

### Meerkat的共识算法如何比Raft提供更高的可用性

对日志条目进行决策需要分布式共识算法。但应该选择哪种算法呢？所有有效的共识算法都能提供所需的一致性和正确性保证，但并非所有算法都能提供相同的可用性保证。

具体来说，许多依赖权威领导者的算法无法提供我们期望的可用性保证，因为当单个机器出现问题时，它们可能变得不可用。以Raft为例，它是最著名且可能是实现最多的共识算法之一。Raft依赖一个权威领导者：集群中唯一能够驱动共识的副本。因此，所有写入请求都会被转发给领导者。这种设计选择有助于使Raft"易于理解"，并且结合租约机制，可以使领导者服务的读取操作自动实现线性一致性（因为它们保证是最新的）。但这也引入了一个单点（临时）故障。

总的来说，权威领导者存在两个问题。首先，如果领导者宕机，系统将变得不可用（所有写入操作被阻塞），直到选出新的领导者。这对Meerkat来说是不可接受的。其次，如果领导者保持在线但速度变慢，无论是由于过载还是网络延迟，性能都会下降。领导者成为瓶颈，因为没有其他方式可以执行写入操作。

第一个问题在广域网环境中尤为严重。考虑一下，当领导者宕机时，大多数算法使用超时机制来选择新领导者：如果非领导者副本在一段时间内未收到领导者的消息，它们会自荐为领导者。此时，旧领导者已被罢免，系统在选出新领导者之前无法接受写入操作。问题在于，当超时时间小于原始领导者与该副本之间的网络延迟时，副本会不断超时，从而阻塞写入操作。而当超时时间过长时，系统对领导者故障的反应迟缓，在此期间写入操作同样被阻塞。此外，如果多个副本同时自荐为领导者，它们的"竞选"可能会相互干扰，导致它们不断重新自荐——而在此期间写入操作一直被阻塞。我们在Cloudflare使用Raft的系统中就遇到过这些问题，因为我们的广域网延迟可能且确实会剧烈波动，使得调整超时时间尤为困难。

我们为Meerkat选择了一种不同的共识算法，称为QuePaxa，旨在避免Raft等协议强加的“超时暴政”。QuePaxa是一种精妙的协议，以下是其要点。客户端可以联系任意副本，该副本可以为最新槽位驱动共识。存在一个领导者，但并非必需——其唯一优势在于能以比其他副本（3次以上）更少的往返次数（1次）驱动共识。关键在于，客户端可以针对同一提案同时联系多个副本，以增加提案成功的几率。并发提案不会产生破坏性干扰：副本会协同工作，以决定其中一个提案值。

简而言之，QuePaxa在我们的应用场景中相比Raft具有三个优势：

1. 由于没有强制要求的领导者，系统永远不会因单个副本（领导者）宕机、不可用或性能下降而变得不可用或降级。只要客户端能联系到某个健康副本（全球任意位置），即可执行写入操作。
2. 因为没有领导者，也就不存在会降低系统性能的领导者选举。与Raft的领导者选举不同，不同副本提出的并发提案会产生建设性干扰。这对于Cloudflare的网络而言非常理想，因为其延迟可能变化极大。
3. QuePaxa专为可靠性较低的网络环境（“异步”）以及存在假想敌手可对副本连接发起针对性攻击的网络而设计。作者发现，在此类条件下，其吞吐量比Raft和Multi-Paxos高出约10倍。与我们网络的实际状况相比，这些条件比其他算法假设的条件更为接近。

由于没有强制要求的领导者，系统永远不会因单个副本（领导者）宕机、不可用或性能下降而变得不可用或降级。只要客户端能联系到某个健康副本（全球任意位置），即可执行写入操作。

因为没有领导者，也就不存在会降低系统性能的领导者选举。与Raft的领导者选举不同，不同副本提出的并发提案会产生建设性干扰。这对于Cloudflare的网络而言非常理想，因为其延迟可能变化极大。

## 评估Meerkat的性能

Meerkat存在局限性。它并非为创建数据库等通用数据系统而设计。

所有共识算法都有代价：大量的往返通信。QuePaxa在初始提案者与多数副本之间通常需要一到三次往返（尽管可能更多），才能决定一个提案并将事件添加到日志中。区别在于领导者：如果由领导者提案，需要一次往返（外加一次广播通知副本决策）；如果由非领导者提案，则需要三次往返（外加一次广播）。如果多个副本同时提案，可能需要更多次数。这些通信成本揭示了共识算法普遍存在的性能限制：提案决策延迟与多数副本之间的延迟成正比。因此，如果副本之间距离较远，延迟就会增加——这是无法回避的。

乍看之下，Meerkat的写入和读取延迟似乎会很差。尤其是如果所有写入和读取（为保持一致性）都必须经过日志，从而需要如此多的往返通信。

但有一些方法可以提升Meerkat的性能：

1. 由于开发者可以控制副本的部署位置，他们可以选择将副本移得更近，从而减少往返延迟（仅适用于不需要真正全球分布的服务）。
2. 写入可以批量处理。因此，如果某个副本在10毫秒内收到10次写入，它可以将所有这些写入放入单个提案中，从而提高吞吐量。
3. 并非所有读取都必须触发一轮共识。如果开发者可以接受读取过期（但绝不不一致）的数据，他们可以从任意副本的本地数据中读取。
