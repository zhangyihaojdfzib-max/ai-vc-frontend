本系列在二十年后重新审视了演进式数据库设计的方法论。将数据库变更视为代码的一个关键约束始终围绕共享数据库资源展开。借助 Databricks Lakebase 中的写时复制分支技术，对一个 TB 级生产数据库创建分支仅需一秒且不占用存储空间，这已成为 O(1) 操作，而此前使实践 #4（每个人都拥有自己的数据库实例）停留在理想层面的约束已被解除。在本系列中，作者将描述当约束解除后会发生哪些变化：方法论本身保持不变，但首次涌现出新的实践、自动化的团队级治理、DBA 角色的演变，以及 Agent（智能体）与人类同事共享的新能力。

Jen 是《演进式数据库设计》一文中的开发者角色。在那篇文章中，她通过一个常规用户故事实现了数据库重构——将 `inventory_code` 字段拆分为 `location_code`、`batch_number` 和 `serial_number`，以此说明 DBA 与开发者可以协作、模式可以小步演进、迁移能够安全地承载变更。

本系列的故事从二十年后 Jen 的视角展开。她遵循的方法论与 2003 年时相同。不同的是，其工作流底层的技术能力已由 lakebase 架构赋能：写时复制数据库分支技术，使她在书中读到的实践能够在生产规模下真正落地。在本系列的三部分中，Jen 以三种不同的范围呈现——她的一天（第一部分）、她的新工作手册（第二部分）以及她的团队（第三部分）。

第一部分带领 Jen 完成了一个功能特性。她遵循的实践在 2003 年的《演进式数据库设计》一文中已有描述，在 2006 年的《重构数据库》一书中得到扩展，并在 2010 年的《持续交付》一书（第 12 章）中被纳入 CI/CD 流水线。

## 最初的七项实践

2003 年的文章提出了七项实践。其中五项在 2026 年之前一直存在应用上的局限性。

1. DBA 与开发者紧密协作。
2. 所有数据库制品与应用程序代码一同进行版本控制。
3. 所有数据库变更均为迁移操作。
4. 每个人都拥有自己的数据库实例。
5. 开发者持续集成数据库变更。
6. 所有数据库变更均为数据库重构。
7. 开发者可按需更新自己的数据库。

## 应用中的局限性

- **实践 #1（DBA 协作）**：任何模式变更一旦失控都会产生生产级影响，因此 DBA 审查始终是同步且把关的。协作受限于 DBA 的日程安排。
- **实践 #4（每个人都拥有自己的数据库实例）**：许可成本、基础设施成本、DBA 时间。在大多数团队中只是理想。多数团队退而求其次，使用共享开发数据库并接受资源争用。
- **实践 #5（持续集成数据库变更）**：2010 年《持续交付》浪潮将迁移引入了流水线，但流水线是对共享目标数据库执行迁移。缺少按流水线隔离的能力。
- **实践 #6（所有数据库变更为重构）**：应用每次重构都需要实践空间（测试数据库），而大多数团队在 PR 粒度上并不具备这样的条件。
- **实践 #7（开发者按需更新）**：开发者可以按需对共享环境执行迁移，但无法安全地进行实验，因为实验会影响他人。

## 变化之处

Databricks Lakebase 引入的技术消除了上述实践实施过程中的障碍。Databricks Lakebase 是一种托管型 Postgres 数据库，使用与 Databricks lakehouse 其他组件相同的对象存储层（数据湖）。数据库的数据驻留在共享的持久化存储中——本质上是 S3 存储桶；Postgres 引擎作为独立的计算层运行于其上。计算与存储可独立扩展。引擎可在负载增加时扩容，流量下降时缩容，空闲时归零。Lakebase 与 Unity Catalog 集成，支持跨多个环境的统一治理。

**写时复制数据库分支技术**正是这种解耦架构在大规模场景下变得实用的关键。分支会创建一个指向同一共享存储的新指针，并附带一个差异标记。在分支写入之前，它与父分支共享所有页面。当分支写入时，仅修改的页面发生分歧；父分支保持不变。分支操作是一种元数据操作，无需复制数据，无论父分支大小如何，均可在约一秒内完成。分支仅维护已变更页面中的数据。

当分支的技术成本与其内部数据大小解耦后，实践 #4（每个人都拥有自己的数据库实例）背后的约束便被解除。每个开发者、每个 PR、每个实验的分支都成为常规操作。上层的补偿层可以移除。模拟（Mock）从测试循环中退出，被每个测试分支上的真实 Postgres 所取代。共享的预发布环境不再是测试模式变更的唯一场所。内存数据库替代方案（H2、SQLite）从单元测试层退出。创建基于 Docker 容器来运行本地数据库的 DevOps 开销不再必要，DBA 用于资源供应的工单队列大幅缩减，因为分支是自助式的。

技术使方法论优化成为可能，并完成了 2003 年原始文章背后实践的目标。

## 2026 年的新兴实践

Lakebase 的写时复制数据库分支技术解除了原始实践之前的局限性，并催生了四项需要详细阐述的新实践。

1. **DBA 与开发者紧密协作。** 每个 PR 上都会发布模式差异（schema diff），DBA 像其他代码审查者一样进行异步审查。由于资源供应成本可忽略不计，DBA 现在有带宽进行审查，甚至可以在实施前与开发者共同创建解决方案，而非仅在实施后审查。
2. **所有数据库制品与应用程序代码一同进行版本控制。** 模式差异、数据库迁移以及迁移测试结果现在都加入了制品集合。
3. **所有数据库变更均为迁移。** 外加一条新的编写规则：幂等性。允许所有已合并的迁移自动部署到下游环境，如 QA、预发布和生产环境。
4. **每个人都拥有自己的数据库实例。** 在开发者粒度、PR 粒度和实验粒度上均可操作。允许开发者尝试多种解决方案，而不仅仅是接受第一个可行的方案。这种自由将催生出针对生产规模的最优解决方案。
5. **开发者持续集成数据库变更。** 在 PR 粒度上可操作。每个 PR 都在其自己的分支上通过 CI 运行。
6. **所有数据库变更为数据库重构。** 2006 年的目录仍然适用；分支为你提供了一个低成本的预演空间，可以在不同规模的环境上尝试重构。在开发数据库中运行良好的迁移在生产环境中几乎总是表现不佳，现在你可以在更大规模的数据库上尝试迁移，以确保其性能。
7. **开发者可按需更新自己的数据库。** “按需”现在意味着在一秒内，隔离地，针对生产规模的数据进行操作。
8. **破坏性测试作为默认选项。** 影响范围为零；重置成本为一秒。无需担心“我的测试是否会污染他人的数据？”，因为现在每次测试运行后我都可以获得一个新的分支。
9. **在数据库层面进行 A/B 变体原型设计。** 在并行分支上构建两种设计方案；与更大的团队和 DBA 一起讨论解决方案并进行演示，然后保留胜出方案。
10. **借助集成的 Unity Catalog，治理只需设计一次，所有分支自动继承。** 策略自动跟随每个分支，我们将在第三部分详细讨论这一点。
11. **Agent（智能体）作为实践者，拥有相同的分支能力。** Agent（智能体）获得的是分支，而非生产环境，我们将在第三部分详细讨论这一点。

这些工作流共同强制要求每个 PR 拥有自己的数据库，并且分支是临时的，这是管道的属性，而非开发者的纪律要求。

## 演化式数据库开发的新手册

### 实践 #1：DBA 与开发者紧密协作

规则。DBA 在整个功能开发过程中与开发者协作，而不仅仅是在关卡审查时。这种协作是异步的，内嵌于 PR 中，与其他代码审查者的参与方式相同。

为什么这现在成为一种持久的习惯？模式差异和迁移测试结果会自动出现在每个 PR 上（参见上文“CI 中的工作流运行方式”）。DBA 可以按照自己的时间表审查具体的产物。迁移已经针对包含真实数据的 CI 分支运行过，因此 DBA 无需在脑中模拟变更。

机制：

- DBA 是影响模式的路径（`migrations/`、`db/`、模式测试目录）的 CODEOWNER。
- DBA 像其他审查者一样在 PR 上进行异步审查。
- 审查重点从“这会不会破坏数据库？”转变为“这是正确的设计吗？实现方式是否正确？”主题包括：数据完整性规则、索引策略、设计一致性、未来可扩展性、长期可维护性。

反模式。在 PR 流程中不包含 DBA，尽管有合理的产物需要审查。

DBA 角色在团队规模的政策管理和治理方面也获得了新的职责。第 3 部分将涵盖这些内容。

### 实践 #2：所有数据库产物都与应用程序代码一起进行版本控制

规则。每个 SQL 文件、迁移脚本和模式测试都与应用程序代码位于同一个仓库中。模式差异和迁移测试结果作为 PR 时输出加入产物集。

为什么这现在成为一种持久的习惯？分支为产物集增加了两个新产物：每个 PR 的模式差异和每个 PR 的模式迁移测试结果。两者都由 CI 根据实际分支状态生成。两者都以具体证据的形式出现在 PR 中，说明更改了什么以及更改迁移脚本的执行情况。

- 迁移文件位于版本化目录中（`migrations/`、`db/migration/`、`alembic/versions/`，取决于框架）。
- 影响模式的测试位于测试树中，与应用程序测试并列。
- 每个 PR 的模式差异由 CI 生成，作为 PR 评论发布。
- 迁移测试结果位于 CI 运行摘要和 PR 评论中。

反模式。在 PR 流程之外生成模式差异（审查者需要打开单独的仪表板）。产物必须存在于审查发生的地方。由于模式变更与代码变更相关联，打破这种依赖关系会为部署带来下游问题。

### 实践 #3：所有数据库变更都是迁移

规则。不允许在任何环境中手动执行 `ALTER TABLE`。每个模式变更都是一个版本化、已检入的迁移脚本。迁移是幂等的。

为什么这现在成为一种持久的习惯？迁移即产物的规则自 2003 年以来未变。新增的是幂等性的编写纪律。同一迁移在过渡期间会针对多个分支运行，因此每次都必须表现相同。重新应用时失败的迁移是一个缺陷。

- 使用 `flyway`、`liquibase`、`Knex`、`Alembic` 或其他迁移框架，这些框架会跟踪哪些迁移已运行、哪些尚未运行，从而允许团队应用诸如 `flyway migrate` 之类的命令，该命令仅应用尚未应用的变更（通过在元数据表中跟踪变更）。
- 最好将不可逆的工作拆分为多个迁移。例如，一个迁移同时添加新列并删除源列，会使回滚变得不必要地困难。因此，使用“先扩展后收缩”策略，在部署周期确认没有活跃读取者引用后，提供更多选择。
- 2006 年的《数据库重构目录》列出了哪些重构是可逆的，哪些是不可逆的。请参考它。

反模式。一个依赖于模式处于特定中间状态的迁移，这是由于分支中的本地更改造成的。迁移必须能够正确应用于任何包含先前迁移的父分支。

### 实践 #4：每个人都拥有自己的数据库实例

规则。每个开发者、每个 PR、每个实验、每次测试运行都拥有自己的 Lakebase 分支。

为什么这现在成为一种持久的习惯？不再需要额外创建 Docker 容器、安装本地数据库服务器、获取许可证、用现有模式和测试数据填充空数据库。只需一个简单的 `create-branch Lakebase` 命令，就能在同样的 1 秒内分支一个 1TB 的数据库，就像分支一个 1MB 的数据库一样。创建时不会复制任何数据；只有修改过的页面才会消耗存储。每个开发者、每个 PR 和每个实验的实例都是常规操作。

- 每个开发者的分支：通过 `databricks postgres create-branch` 或 SCM 扩展的分支创建流程按需创建。
- 每个 PR 的分支：在 PR 打开时由 CI 自动创建（`pr.yml`），在合并或关闭时销毁（`merge.yml`）。有关 PR 和合并的代码片段，请参见上文“CI 中的工作流运行方式”。
- 每个实验的分支：从预发布或生产环境分支出来用于设计探索；实验结束后丢弃。

反模式。为了“方便”而在团队中共享一个开发数据库。第 1 部分中提到的由争用导致的串行化问题，在分支合并时就会再次出现。

Jen 示例的扩展。她的每个开发者分支在功能开始时从预发布环境分支出来。CI 分支在 PR 打开时从预发布环境分支出来。她的 A/B 探索分支（实践 #9）并行地从预发布环境分支出来。一个功能跨四个分支，全部在几秒内完成，全部隔离。

### 实践 #5：开发者持续集成数据库变更

规则。每个 PR 都针对一个新的 Lakebase 分支通过 CI 运行，应用迁移并针对真实的 Postgres 运行测试。

为什么这现在成为一种持久的习惯？自2010年起，CI流水线就具备了迁移规范。新的变化在于每个流水线的隔离性：每个PR（拉取请求）都有自己的分支，因此集成测试可以在真实数据上运行，而不会产生争用。

- CI在PR打开时创建分支。参见上文“CI中的工作流如何运行”中的PR片段。
- 通过`lakebase-migrate apply`将迁移应用到分支。
- 应用程序测试通过ORM在迁移后的分支上运行，无需模拟。
- Schema差异发布在PR上。
- 分支在合并或关闭时销毁。

反模式。在共享的预发布环境上运行PR验证。串行化问题会卷土重来；每个PR的隔离性将丢失。

### 实践 #6：所有数据库变更都是数据库重构

规则。Schema变更遵循命名的重构模式：拆分列、重命名列、移动列、替换类型等。每种模式都有明确的转换机制（同时保留新旧列、从旧列填充数据、切换读取方、删除旧列）。

为什么这现在成为一种持久的习惯？2006年databaserefactoring.com上的目录列出了70多种重构模式并附有实例。2026年的新变化在于有了一个低成本的环境来演练转换机制：开发者分支用于演练；CI分支用于验证；生产环境只接收验证后的结果。

- 通过名称识别重构模式。在目录中查找。
- 在每个开发者分支上应用命名的转换机制。使用生产环境形状的数据进行验证。
- 打开PR。CI在其自己的分支上运行迁移并发布差异。
- 在差异和测试结果获得批准后合并。

反模式。进行一次性的、不映射到任何命名重构模式的Schema变更。70多种模式覆盖了常见情况；如果你发现自己的变更不在其中，很可能是在一次迁移中组合了多个重构，应该拆分。

Jen的示例扩展。她的V87迁移是拆分列重构：将`inventory_code`拆分为`location_code`、`batch_number`和`serial_number`。databaserefactoring.com/SplitColumns.html上的目录页面列出了转换机制。她的分支是演练空间；PR的CI运行是验证。

### 实践 #7：开发者可以按需更新他们的数据库

规则。开发者可以按需刷新其分支的数据库状态：重置为父分支的当前状态、从生产环境派生一个新分支、丢弃一个实验分支、与队友共享一个分支。所有这些操作都在几秒内完成。

为什么这现在成为一种持久的习惯？2026年的“按需”意味着在一秒内，隔离地，使用生产环境形状的数据。这些操作都不需要咨询运维日历或DBA队列。

- 重置：从其父分支删除并重新创建分支。
- 从生产环境派生：`databricks postgres create-branch --source production`。
- 丢弃：`databricks postgres delete-branch`。
- 共享：将分支端点交给队友进行配对编程。

反模式。将任何分支视为超出其用途的持久化资源。迁移是持久的产物；分支是工作空间。

### 实践 #8：将破坏性测试作为默认选项

规则。当破坏性操作的爆炸半径为零时，破坏性测试成为日常选项，而非季度性工作。

为什么这现在成为一种持久的习惯？分支在一秒内重置。你对分支所做的任何操作都可以通过从同一父分支创建一个新分支来撤销。破坏性测试不再需要运维日历和审批流程。

现在可以纳入正常功能开发周期的事项：

- “如果我的迁移在执行UPDATE语句中途失败会发生什么？”运行它。在50%处终止进程。验证回滚是否有效。重置。
- “如果在备份恢复过程中触发了故障转移会发生什么？”模拟部分状态。验证应用程序行为。重置。
- “我们灾难恢复手册的实际恢复时间是多少？”运行手册。测量。重置。
- “这个迁移能否在生产环境形状的数据或规模上成功运行？”在合并前验证。

文化影响。当重置成本为零时，团队不再将测试数据库视为宝贵资源。测试可以更激进。清理可以跳过，因为下一个分支会从全新状态开始。

Jen的示例扩展。在打开PR之前，她获取了分支上生产环境形状的数据，并故意损坏了大约百分之一的`inventory_code`值，使其看起来像边缘情况：缺失数字、嵌入空格、尾随空白。这些是历史数据积累的典型产物。她运行了迁移。有两行数据导致她的子字符串计算失败。她修复了脚本并重新运行。分支吸收了破坏性测试。生产环境从未看到这些。

### 实践 #9：在数据库层面进行A/B变体原型设计

规则。当两个设计方案存在争议时，在并行分支上构建它们，与生产环境形状的数据进行比较，并保留更好的方案。

为什么这现在成为一种持久的习惯？每个分支的成本几乎为零。探索两个Schema设计不再需要预先选择一个，也不再需要大多数团队不会为探索性问题而进行的资源调配流程。

- 从同一父分支创建两个分支。
- 将设计A的迁移应用到一个分支，将设计B的迁移应用到另一个分支。
- 针对每个分支运行应用程序。衡量重要指标：公共读取路径上的查询延迟、生产规模下的迁移时间、索引占用、模拟负载下的锁争用。
- 选择更好的方案。丢弃次优方案。在PR描述中记录决策，以便后续需要扩展Schema的人知道考虑了哪些方案以及为什么选择这个设计。

反模式。运行A/B原型而不记录决策和理由。分支瞬间消失；设计决策应该是永久的。

Jen的示例扩展。她为location/batch/serial功能考虑了两种设计：在现有`inventory`表上增加三个新列，或者创建一个由`inventory_id`关联的独立`inventory_attributes`查找表，预期未来会添加更多属性。她在从预发布环境派生的并行分支上构建了这两种设计。她针对每个分支运行了应用程序的读取路径，使用生产环境形状的数据测量了查询性能，并观察了每种迁移如何扩展到生产规模。查找表版本在公共读取路径上表现更差，因为每次库存显示都需要一次连接。她采用了增加列的版本，丢弃了查找表分支，并在PR描述中留下了一条注释：*考虑过查找表版本；因公共读取路径需要连接而被拒绝。如果属性超过五个，请重新考虑。*
