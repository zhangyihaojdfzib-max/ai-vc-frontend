# 为 Reachy Mini 添加 MCP 工具

![Reachy Mini 不再需要望向窗外就能告诉你天气](/images/posts/8830270af27e.jpg)

Reachy Mini 对话应用现在可以使用托管在公共 Hugging Face Spaces 中的工具，通过 MCP 进行调用。你可以为机器人赋予新能力，例如查询天气或搜索网络，只需从 Hub 添加一个 Space，而无需编辑应用。该工具在 Space 自身中持续运行，因此不会将任何代码下载到你的机器上。你也可以发布自己的工具供他人使用。

添加一个工具只需一条命令：

```
reachy-mini-conversation-app tool-spaces add pollen-robotics/reachy-mini-weather-tool

```

然后像往常一样启动应用：

```
reachy-mini-conversation-app

```

现在你只需提问：

```
What's the weather in Paris today?

```

下面，我们将探讨什么是工具、配置文件如何控制机器人可以使用哪些工具，以及远程路径当前的局限性。

## 内置工具

当你与机器人对话时，你得到的不仅仅是语音，而是一个能够对对话做出反应的系统：机器人在适当的情况下可以移动并做出非语言回应。我们这里要关注的是实现这一功能的工具。工具是模型在对话过程中可以执行的操作：播放情绪、移动头部、通过摄像头观察。每个工具都有一个名称和简短描述。模型读取这些信息，判断何时有用，调用它，并使用返回的结果。

目前所有工具都是本地的，随应用一起发布，其中大部分与机器人的身体相关：

## 配置文件如何控制工具

代码中的工具只有在配置文件中启用后才可使用。配置文件是一个包含两个关键文件的文件夹：`instructions.txt`（提示词）和 `tools.txt`（已启用的工具）。

`default` 配置文件启用了全套工具：

```
# profiles/default/tools.txt
dance
stop_dance
play_emotion
stop_emotion
camera
idle_do_nothing
head_tracking
move_head

```

如果某个名称不在 `tools.txt` 中，模型就无法调用它。

你也可以编写自己的工具：向配置文件（或 `external_tools/`）添加一个 Python 文件，为其指定名称和描述，然后将该名称列入 `tools.txt`。

目前有内置工具和自定义本地工具，`tools.txt` 决定哪些工具处于激活状态。这对于机器人的身体控制来说效果良好，并且保持了可信核心的简洁性。

## 本地工具的局限性

这里的限制在于每个工具都必须是本地 Python 代码。对于 `move_head` 或 `play_emotion` 来说，这没问题：它们与硬件交互，属于应用的一部分。但许多有用的功能与身体无关，例如网络搜索、天气查询或信息检索。对于这些功能，将所有内容保持本地化会带来诸多不便：

- 分享工具意味着将你的 Python 文件交给他人
- 更新工具意味着再次发送这些文件
- 修改工具意味着编辑应用，尽管该功能实际上与应用本身是分离的

## 从 Spaces 调用工具

远程工具增加了第三种类型，与你已有的内置工具和自定义本地工具并存，适用于那些更容易独立发布、分享和更新的功能：

- 内置机器人工具保持本地化和可信赖
- 可分享的远程工具可以托管在公共 Hugging Face Spaces 中
- 你仍然可以使用 `external_tools/` 中的自定义一次性工具

这非常适合搜索、天气和查询等无状态功能：任何你想迭代但又不想触及应用本身的功能。而且由于任何人都可以发布兼容的 Space，分享工具并在彼此的工作基础上进行构建变得非常容易。

我们从两个金丝雀工具开始，即用于测试新流程的小型测试工具：

- pollen-robotics/reachy-mini-search-tool
- pollen-robotics/reachy-mini-weather-tool

它们足以测试整个功能：从 Hub 安装、发现远程工具、按配置文件启用，并让实时后端像调用内置工具一样调用它们。

要同时使用两者，请在同一个配置文件中添加每个 Space 及其工具栈：

```
reachy-mini-conversation-app tool-spaces add pollen-robotics/reachy-mini-search-tool
reachy-mini-conversation-app tool-spaces add pollen-robotics/reachy-mini-weather-tool

```

现在机器人可以在同一对话中搜索网络和查询天气，这正是下面 `canary_web_search_weather` 配置文件所做的。

## 安装、列出、移除

```
# 安装并在当前激活的配置文件中启用
reachy-mini-conversation-app tool-spaces add <owner/space-name>
 
# 在特定配置文件中启用
reachy-mini-conversation-app tool-spaces add <owner/space-name> --profile <NAME>
 
# 仅安装，不启用
reachy-mini-conversation-app tool-spaces add <owner/space-name> --install-only
 
# 列出已安装的 Spaces
reachy-mini-conversation-app tool-spaces list
 
# 移除已安装的 Space
reachy-mini-conversation-app tool-spaces remove <owner/space-name>

```

`add` 命令会验证 Hub 上的 Space，探测 MCP 端点，发现其工具，并默认将工具 ID 追加到当前激活配置文件的 `tools.txt` 中。当前激活配置文件是 `default`，除非你设置了 `REACHY_MINI_CUSTOM_PROFILE`。使用 `--install-only` 可跳过此步骤。

`tools.txt` 是守门员：远程工具只有在其 ID 出现在配置文件的 `tools.txt` 中时才会被激活，同时你还可以在其中保留所需的任何内置工具。

### 清单存储位置

已安装的源持久化存储在：

- `installed_tool_spaces.json`（托管应用模式）
- `external_content/installed_tool_spaces.json`（终端模式）

## 工具命名

每个已安装的 Space 都会获得一个基于其 slug 的本地别名，连字符、点和斜杠会转换为下划线：

```
pollen-robotics/reachy-mini-search-tool → pollen_robotics_reachy_mini_search_tool

```

远程工具随后使用双下划线进行命名空间限定：

```
pollen_robotics_reachy_mini_search_tool__search_web
pollen_robotics_reachy_mini_weather_tool__get_day_brief

```

这可以防止远程工具名称与内置工具冲突，并允许多个 Space 在同一配置文件中共存。

实现还会在可能的情况下去除冗余的 Space 名称前缀，因此冗长的远程工具名称会变成更简洁的本地 ID。如果去除前缀会导致同一 Space 中的两个工具发生冲突，代码会回退到完整的命名空间限定名称。

在注册层面还有重复安全检查：`Tool.name` 值在整个合并工具集中必须唯一。如果两个源声明了相同的名称，应用会快速失败。

## 提示词为何重要

远程工具的管道将工具引入模型。而提示词则决定模型如何使用它们。

这在搜索加天气的金丝雀配置中尤为明显。像这样的组合问题：

```
Should I bring a jacket in Bordeaux today, and is there anything major happening downtown tonight?

```

至少可以通过三种方式处理：先查天气再搜索、先搜索再查天气、或者在同一轮中同时进行。如果提示词模糊不清，模型会串行调用，造成不必要的延迟。因此，金丝雀提示词成为了该功能的一部分，而不仅仅是附带配置。

```
[default_prompt]

你有一个用于获取当前网络信息的远程工具。
当用户询问最新的事实、新闻、实时可用性或任何可能近期发生变化的内容时，请使用它。

当搜索结果已直接回答问题时，请用平实的语言直接给出答案。  
以答案开头，不要先描述工具操作。  
对于可能需要等待的远程查询，你可以用一句简短的英文确认，例如“Let me check that and I'll be right back”，然后继续。  
除非用户明确要求其他语言，否则用英文回答。  
如果搜索结果片段不完整或模糊，简要说明不确定性。
```  

## 今天有效的方法，以及无效的方法  

有两件事值得指出。首先，Space 必须实际表现得像一个 MCP 服务器；如果工具发现失败，安装就会失败。其次，提示词指令可以鼓励并行调用，但无法保证一定实现。如果确定性编排对某个用例至关重要，那么该逻辑应从提示词转移到代码中。

## 结论  

该应用现在有三种工具共享一个注册表：内置工具、本地自定义工具和远程 MCP 工具，而配置文件仍然决定特定助手可以访问哪些工具。一个小而可信的核心保持在中心位置，而围绕它的可选能力可以在不触及应用本身的情况下添加、测试和替换。
