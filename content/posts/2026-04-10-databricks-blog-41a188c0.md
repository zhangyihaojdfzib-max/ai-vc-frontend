## 数据库是您开发流程中的最后瓶颈

数据库分支是现代开发流程中缺失的基础构件。技术栈的其他部分都已演进到支持快速迭代。代码有 Git。基础设施有 Terraform。部署有能在几分钟内运行的 CI/CD 流水线。但关系型数据库的工作方式仍与十年前无异。

大多数团队共享一个单一的测试数据库。在建立后的几天内，该数据库就会与生产环境不同步。随着开发人员以不同顺序应用迁移，模式发生分歧。序列值不再匹配。测试数据不断累积并污染结果。最终有人会重新初始化整个数据库，然后循环重新开始。

建立新环境的情况更糟。标准做法是针对生产环境运行 pg_dump，等待其完成（时间从几分钟到几小时不等，取决于数据库大小），将其加载到新实例中，配置访问权限，并希望结果能真实反映生产环境的运行状态。对于一个 500GB 的数据库，这意味着一次 500GB 的复制操作，外加维持其运行所需的计算和存储资源。

结果是可预见的。团队避免创建新环境，因为成本太高、速度太慢。开发人员共享一个单一、可变的测试数据库。迁移针对过时的数据进行测试，或者根本不测试。预览部署针对空的数据装置运行，而非真实的模式。CI 测试共享状态并产生不稳定的结果。

数据库成为开发人员不敢触碰的技术栈部分。

Databricks Lakebase Postgres 通过数据库分支改变了这一切。

## 数据库分支究竟是什么

数据库分支不是数据库副本。这个区别很重要，因为它彻底改变了隔离环境的经济性。

当您复制数据库时，您会将其所有数据和模式复制到一个新的、独立的实例中。所需时间和成本与数据库大小成线性比例。每个副本都是一个完整的克隆，并且每个克隆在创建的那一刻就开始变得过时。

分支的工作方式不同。当您在 Lakebase 中创建分支时，您会获得一个全新的、完全隔离的 Postgres 环境，它：

*   起始于其父分支在特定时间点的确切模式和数据
*   共享相同的底层存储，而不是复制它
*   仅在您实际进行更改时才写入新数据

这被称为写时复制。只要两个分支没有发生分歧，它们就引用相同的存储数据。当您在分支上运行迁移、插入行或修改表时，只有这些更改会被单独写入。其他所有内容都与父分支共享。

### 数据库副本 vs. 数据库分支

| 特性 | 数据库副本 (pg_dump, RDS 快照) | 数据库分支 (Lakebase) |
| :--- | :--- | :--- |
| **创建时间** | 几分钟到几小时，与数据库大小成比例 | 几秒钟，与数据库大小无关 |
| **存储成本** | 所有数据的完整副本 | 仅与更改成比例 (写时复制) |
| **隔离性** | 完全隔离，但维护成本高 | 完全隔离，拥有独立的计算资源和连接字符串 |
| **新鲜度** | 从创建那一刻起就已过时 | 起始于分支创建时父分支的确切状态 |
| **清理** | 手动拆除实例和存储 | 删除分支；计算和存储资源自动回收 |

实际上，这意味着：

*   分支创建只需几秒钟，无论数据库大小如何。一个 10GB 的数据库和一个 2TB 的数据库分支所需时间相同。
*   存储成本与更改量成比例，而非总数据大小。在一个 500GB 的数据库中修改 50MB 数据的分支，大约只使用 50MB 的额外存储。
*   每个分支都有自己的 Postgres 连接字符串和计算端点。分支之间以及分支与其父分支之间完全隔离。
*   空闲分支的计算资源会自动缩放到零。您只需为分支实际被使用时产生的活跃计算付费。

分支的设计初衷是让开发人员、CI 流水线、AI Agent 或自动化工具可以自由地创建、使用和丢弃。它们不是需要维护的珍贵环境。它们就像 Git 分支一样，是一次性的。

## 实现数据库分支的架构

传统的托管 Postgres（RDS, Azure Database for PostgreSQL）将计算和存储捆绑在一起。数据库进程及其数据位于同一实例上，数据以单一可变文件系统的形式存储。这就是为什么创建第二个环境时复制是唯一选择：您必须复制文件系统。

但 Lakebase 的构建方式不同。它将计算与存储完全分离。所有数据都写入一个分布式的、版本化的存储引擎，该引擎将每次更改记录为新版本，而不是覆盖现有数据。正是这种日志结构的架构，使得数据库分支能够作为一种基础构件实现，而不是作为附加在顶层的功能。

因为存储是版本化的，多个分支可以引用相同的底层数据而不会有冲突风险。因为计算是独立的，每个分支运行自己的 Postgres 进程并独立扩展。处于空闲状态的非生产分支会自动缩放到零，并在有连接进入时在几毫秒内重新启动。

并非所有的数据库分支实现都是相同的。有些平台创建完整的实例副本并称之为分支。另一些则只分支模式，不包括数据。Lakebase 分支同时包含模式和**数据**，在存储层使用写时复制来避免重复，并为每个分支提供独立的、自动扩展的计算资源。这使得自由、大规模地创建分支变得切实可行，而无需配置额外的基础设施。

这种架构还支持时间旅行。因为数据的每个版本都在可配置的恢复窗口内保留，您可以从过去的任何时间点创建分支，而不仅仅是当前状态。这就是实现即时时间点恢复的原因：无需重放 WAL 日志或恢复备份，您只需在所需的时间戳创建一个分支，并直接从中读取。

## 数据库分支为您的团队解锁了什么

一旦数据库分支成为一种快速、廉价的基础构件，而不是昂贵的复制操作，新的工作流程就变得可行。以下是最常见模式的总结。（我们将在本系列的下一篇文章中详细介绍每一种模式。）

**每位开发人员一个分支。** 每位工程师都获得一个拥有类生产数据的独立环境。不再需要在共享的开发数据库中互相干扰彼此的更改。当分支与生产环境偏离太远时，只需一个命令即可重置，以拉取最新的模式和数据。由于分支在空闲时会缩放到零，即使在大团队中，这种模式也能保持经济性。

**每个拉取请求一个分支。** 在 PR 打开时自动创建分支，在合并或关闭时自动删除。Vercel 或 Netlify 上的每个预览部署都获得自己的数据库分支，因此您的前端预览由真实、隔离的数据支持。迁移针对真实的数据形状和约束运行，而不是空的测试装置。这是团队最先采用的工作流程，也往往是说服他们全面采用数据库分支的工作流程。

**每次测试运行一个分支。** CI 流水线每次运行都获得一个全新的、隔离的数据库。没有先前测试留下的状态。无需等待空的容器镜像启动并用虚假数据填充。没有因共享数据或测试顺序依赖导致的不稳定结果。每次运行都从相同的基线开始。对于需要确定性数据的测试，您可以从固定的时间点或特定的日志序列号创建分支。

**即时恢复。** 在恢复窗口内的任何时间点创建分支。检查已删除的表、调试失败的迁移或审计历史数据，所有这些都无需触及生产环境。使用模式差异来比较变更前后的状态。从恢复分支导出您需要的内容，然后将其删除。整个过程只需几秒钟，而不是传统 PITR 所需的几小时或几天。

AI Agent的临时环境。AI Agent可以通过Lakebase API以编程方式配置数据库，在任务执行期间使用它们，并在完成后关闭。平台可以在快照基础上构建版本控制：每个Agent操作都会创建一个检查点，用户可以即时在不同版本间跳转。如果Agent执行了错误的数据迁移或损坏了数据，仅需一次API调用即可回滚。

## 快速开始

Databricks Lakebase的数据库分支功能，可将Postgres数据库从开发工作流中最缓慢的环节转变为最快速的环节。

您可以通过控制台、CLI或API在一分钟内创建第一个分支。以下是使用CLI的示例：

```
lakebase branch create my-branch --from production
```

就是这样。您现在拥有一个独立的Postgres环境，其中包含来自生产环境的完整模式和数据，随时可以使用。

如果您基于Postgres构建应用，并且厌倦了管理数据库环境的开销，可以从一个开发分支开始。然后尝试为每个PR创建一个分支。大多数从单一数据库分支工作流开始的团队，都会迅速采用其余功能。

Databricks Lakebase是为Agent和应用构建的无服务器Postgres。了解更多信息，请访问 databricks.com/product/lakebase。
