# PyTorch 性能分析（第二部分）：从 nn.Linear 到融合 MLP

在本系列的第一部分"PyTorch 性能分析"中，我们使用 `torch.add(torch.matmul(x, w), b)` 学习了如何阅读 PyTorch 性能分析器追踪。我们还讨论了其他几个相关主题——CPU 调度链、启动开销、开销受限与计算受限模式的区别，以及 `torch.compile` 的一些内部机制。

在第二部分（本篇博客）中，我们向上攀登一级。我们将手写的 matmul-add 对替换为 `nn.Linear`（`bias=True`）。这是每个深度学习模型都使用的基本构建块。然后我们堆叠三个这样的模块（针对我们的示例），中间加入激活函数，形成一个多层感知机（MLP）块。

本篇博客的脚本位于：`02_linear.py`、`03_simple_mlp.py` 和 `03_kernels_mlp.py`。和之前一样，建议在单独的标签页中打开它们，并在阅读时同步查看代码。我们使用 NVIDIA A100-SXM4-80GB GPU 来运行脚本。在 Hugging Face 基础设施上设置 GPU 并使用 Spaces 的开发者模式来实验这些脚本非常容易。也可以使用 Hugging Face Jobs 管道来运行这些脚本。

在开始之前，快速回顾两个我们将反复使用的概念：

1. GPU 内核是一个在 GPU 的多个线程上并行运行的程序。
2. CPU 调度和启动这些内核。你在性能分析器追踪中看到的大部分 PyTorch 开销就是这种调度工作。

## 从 matmul-add 到 Linear

`nn.Linear` 是一个模块封装，包裹了我们在第一部分已经分析过的相同矩阵乘法和加法。唯一的区别是它拥有自己的权重和偏置作为参数，并暴露了一个 PyTorch 用户已经熟悉的 `forward` 方法。

```py
linear_layer = nn.Linear(in_dim, out_dim, bias=True)
y = linear_layer(x)
```

当前的操作可以写成：

```
y = x @ w.T + b
```

其中 `x` 是输入，`w` 是权重，`b` 是偏置。让我们运行 `02_linear.py` 并查看性能分析结果。

```bash
uv run 02_linear.py --batch 1024 --in_dim 32 --out_dim 64
uvx trace-util traces -b traces
```

`trace-util` 是一个工具，它会将你的追踪同步到 Hugging Face 存储桶，然后在你的终端上提供 Preffeto URL。

图 1 显示了线性层前向调用的性能分析器追踪。我们使用与之前追踪类似的 `schedule` 设置来追踪线性层的 `forward` 调用，其中 `wait=1`、`warmup=1` 和 `active=3`。这就是为什么我们在 CPU 和 GPU 通道上看到三个 Profile Step。

### 转置操作在做什么？

![放大的 CPU 调度链，显示在 aten::linear 内部的 aten::addmm 之前嵌套了 aten::t 转置操作，GPU 通道上没有匹配的活动](/images/posts/c70bef3e9bd5.png)

如果我们放大性能分析器追踪，如图 2 所示，我们注意到在 `aten::addmm`（乘法和加法）操作之前有一个 `aten::t`（转置）操作。我们已经可以推断出 `nn.Linear` 会转置权重参数，然后将其与输入相乘。这就是我们看到 `aten::t` 操作的原因。

需要注意的一个重要点是，`aten::t` 实际上并不复制或重新组织数据：它只在 CPU 上重写张量元数据（形状和步长）来表示转置后的矩阵。它不会在 GPU 上启动内核。可以通过两种方式验证这一点：查看追踪中的 GPU 通道，或者检查性能分析器表中的 `aten::t` 行及其在 CUDA 上花费的时间。

### 为什么没有单独的 mul 和 add 内核？

如图 3 所示，线性层的调度链中没有 `aten::add`（偏置加法）。这是因为偏置加法已经被折叠到矩阵乘法内核中，使用了所谓的"尾声"（epilogue）。

尾声是 GEMM（通用矩阵乘法）内核在最后、即将将结果写回 HBM（高带宽内存，GPU 的主内存）之前执行的一个小型计算。添加偏置、应用激活函数或乘以常数都是经典的尾声操作。使用尾声的目的是避免第二次加载或写入 HBM，因为内存流量会使操作变得昂贵。

`nn.Linear` 调用 `torch.nn.functional.linear`，而后者又调用 `aten::linear`。`aten::linear` 查看输入，注意到传入了偏置，于是调度 `aten::addmm(bias, x, weight)`，而不是分别执行 matmul 和 add。`addmm` 计算：

```
out = x @ weight.T + bias
```

在 GPU 上运行的 cuBLAS GEMM 内核内置了偏置加法变体，而 `aten::addmm` 选择的就是这个内核。加法从未作为单独的内核出现，因为它是 matmul 内核写回过程的一部分，这正是尾声的作用。

此时需要注意一个微妙之处。你在第一部分 `--compile` 下看到的内核（addmm）正是 eager 模式下的 `nn.Linear` 已经使用的内核。这里没有剩下任何可供 `torch.compile` 融合的内容，接下来我们将验证这一点。

### --compile 能帮助单个 Linear 吗？

让我们编译前向调用并查看性能分析器追踪。（性能分析器追踪在下一节中可视化）

```bash
uv run 02_linear.py --batch 1024 --in_dim 32 --out_dim 64 --compile
uvx trace-util traces -b traces
```

如果你比较单个 `nn.Linear` 的 `forward` 的 eager 和编译追踪，你会发现：

- GPU 上相同的 cuBLAS GEMM 内核。
- CPU 上相同的 `aten::addmm` 操作。
- CPU 通道上多了几行编译特有的内容。

这一点值得内化。一个常见的反应是，当模型感觉变慢时，就使用 `torch.compile`。对于单个带偏置的 GEMM，编译几乎无事可做。这不是一个 bug，只是编译需要多个操作才能进行可能的融合。让我们通过观察 MLP 来证明这一点。

### 转置操作去哪了？内核布局和预处理操作

仔细阅读两个追踪（eager vs compile）的读者会注意到，eager 模式的 CPU 调度链比编译后的更复杂。

![Eager 模式 CPU 调度链，显示在 aten::linear 下分别框出的 aten::t 转置和 aten::addmm](/images/posts/8aad7ab1c220.png)

![编译后 CPU 调度链，显示一个 Torch-Compiled Region 和一个单独的 aten::addmm 调用，没有转置操作](/images/posts/1f218fa49c12.png)

`aten::linear` 内部的 eager 模式 CPU 调度链是 `aten::t` 后跟 `aten::addmm`（图 4）。要理解 `aten::t` 实际做了什么，我们需要快速了解一下步长和视图。

张量将其数据存储为内存中一个扁平的、连续的数字序列。`shape` 和 `stride` 是位于该序列之上的元数据，告诉 PyTorch 如何遍历它：步长 `(s0, s1)` 表示"移动一行需要走 `s0` 个元素，移动一列需要走 `s1` 个元素"。更改元数据，你就得到了同一原始数据的不同视图，无需复制：

```py
>>> M = torch.tensor([[0, 1],
...                   [2, 3],
...                   [4, 5]])
>>> M.shape, M.stride()
(torch.Size([3, 2]), (2, 1))   

>>> T = M.t()                  
>>> T.shape, T.stride()
(torch.Size([2, 3]), (1, 2))   
>>> T
tensor([[0, 2, 4],
        [1, 3, 5]])
>>> T.flatten()                
tensor([0, 2, 4, 1, 3, 5])
```

`M.t()` 没有移动任何一个数字。它返回了一个新的视图，其步长被交换了，因此逐行读取现在会以转置后的顺序遍历原始缓冲区 `0, 1, 2, 3, 4, 5`。底层数据完全相同；只有元数据不同。

这正是 `aten::t` 在线性层内部所做的：它不分配新的张量或复制任何数据，而是生成一个步长被重写的权重视图。

如图 5 所示，编译并没有移除 GPU 内核：它移除了调度该视图的 CPU 开销。Inductor 在编译时追踪了视图链，一次性计算了最终的步长，并直接发出一个带有硬编码步长的 `aten::addmm` 调用。几微秒的 CPU 工作消失了，而 GPU 执行了完全相同的数学运算。

正如预期的那样，当输入数据违反了编译器预计算的步长时，它会抛出一个错误。

如果你查看两个追踪中的 GPU 通道，每次前向调用恰好有一个内核，而且两次都是同一个内核：

```
cutlass_80_wmma_tensorop_bf16_s161616gemm_bf16_32x32_32x1_tn_align8

```

如果没有运行转置核函数，是谁让GEMM以转置顺序读取权重矩阵的？答案就在核函数的名称中。请看后缀：

```
cutlass_80_wmma_tensorop_bf16_s161616gemm_bf16_32x32_32x1_tn_align8
                                                          ^^

```

这个`tn`就是布局描述符。cuBLAS和CUTLASS会为输入布局的每种组合预编译一个独立的核函数二进制文件。

`n`（非转置）和`t`（转置）描述了核函数在内层循环中如何遍历其输入。调度器的工作是查看输入步长，判断匹配哪个后缀组合，然后选择正确的预编译核函数。

性能分析器追踪中的核函数名称是该核函数身份的哈希转储。如果两次运行显示相同的核函数名称，则GPU执行的是相同的工作。如果它们不同（例如`_tn`与`_nn`、`bf16`与`fp16`、或`s16816gemm`与`s161616gemm`），则GPU执行的是不同的工作，调度器走了一条不同的分支。学会解读这个名称是比较追踪结果时最有用的习惯之一。

## 堆叠三个线性层：MLP

在本节中，我们将对多层感知机（MLP）进行性能分析。为了更有趣，我们将使用GeGLU激活变体（在实际中相当常用）对一个前馈网络进行性能分析。这也是我们向深度学习研究史上最伟大的句子之一致敬的方式（图6）。

![GLU变体改进Transformer论文的结论部分，其中结尾句将架构的成功归因于神圣的恩赐，已被高亮显示](/images/posts/371047a67503.png)

```py
class SimpleGeGLUMLP(nn.Module):
    def __init__(self, dim, hidden):
        super().__init__()
        self.gate_proj = nn.Linear(dim, hidden, bias=False)
        self.up_proj = nn.Linear(dim, hidden, bias=False)
        self.down_proj = nn.Linear(hidden, dim, bias=False)

    def forward(self, x):
        g = self.gate_proj(x)
        u = self.up_proj(x)
        h = F.gelu(g, approximate="tanh")
        m = h * u
        y = self.down_proj(m)
        return y

```

你可以在以下位置找到完整的脚本：`03_simple_mlp.py`。按如下方式执行它：

```bash
uv run 03_simple_mlp.py --batch 64 --seq 128 --dim 768 --hidden 3072
uvx trace-util traces -b traces

```

在打开追踪结果之前，让我们先一起思考一下应该看到什么。`forward`函数执行了相当多的计算，但其中大部分我们已经熟悉了。

我们应该期望看到三个`aten::linear`调度，每个对应一个`nn.Linear`层。我们还应该期望看到两个逐点核函数启动，一个用于GeLU，一个用于乘法。在查看之前形成这个预期是性能分析过程中最有用的习惯：你阅读追踪结果是为了**确认或推翻**一个猜测，而不是从头开始形成一个猜测。

![线性投影追踪结果中高亮显示的占用率查询](/images/posts/8fc257244bd0.png)

从图7中我们可以自我肯定，因为我们的直觉是正确的。每次前向传播（一个`mlp_fwd`），GPU恰好运行5个核函数。图8高亮显示了线性投影层在CPU通道中看到的“占用率查询”。

三个GEMM在启动前各自执行了一次额外的`cudaOccupancyMaxActiveBlocksPerMultiprocessor`调用。我们在第1部分中有一个单独的章节讨论这个问题，你可以在这里找到。那是cuBLAS在确定网格大小。逐点操作（GeLU和mul）直接启动，没有占用率查询。所以“一个线性层”实际上是查询+启动，而“一个逐点操作”只是启动。

`aten::t`、`aten::transpose`、`aten::reshape`、`aten::view`、`aten::as_strided`和`aten::_unsafe_view`操作启动零个核函数。它们在表中显示`0.000us`的CUDA时间（图9），因为它们只在CPU上重写张量元数据（形状和步长）。扫描表格的读者会看到每个线性层大约有六个操作名称，但其中只有一个（`mm`）会真正到达GPU。

### 为什么有两种类型的GEMM核函数？

MLP将`[batch, seq, dim]`展平为`[batch * seq, dim]`以进行矩阵乘法。在我们的命令行调用中，我们使用了`64`作为`batch`，`128`作为`seq`，所以这就是下面`8192`（`batch * seq = 64 * 128`）的来源。

从追踪结果来看：

所有三个GEMM具有相同的FLOP计数，每个约`2·8192·768·3072 ≈ 38.7 GFLOP`，然而`down_proj`快了大约`10%`。相同的工作，不同的形状（`N=768`而不是`3072`），所以cuBLAS选择了不同的分块（`128×256`，具有更深的`stages_64x3`流水线），该分块对该形状有更好的重用。

如果你想更深入地了解分块，这里有一个很好的入门资源。

这正是表中出现两行GEMM的原因（图9）：`128x128`行对应gate+up，`128x256`行对应down。

### `torch.compile`做了什么？

在编译`forward`方法并可视化之前，让我们再次进行脑力练习，问问自己期望在追踪结果中看到什么。这是一个有趣的实验，也是每次你自己进行性能分析时都需要重复的重要实验。始终建立在你的直觉之上，一旦出现不匹配的情况，停下来弄清楚原因。

```bash
uv run 03_simple_mlp.py --batch 64 --seq 128 --dim 768 --hidden 3072 --compile
uvx trace-util traces -b traces

```

在即时模式下，每个`nn.Linear`被展开成一个调度操作链（`aten::linear` → `aten::t` → `aten::transpose` → `aten::matmul` → `aten::reshape` → `aten::mm`）。这些是ATen在到达真正的GEMM之前遍历的高级包装器。`torch.compile`移除了这个链。

当编译后的图运行时，不再有linear、matmul、transpose或reshape，这些元数据操作被折叠进了`mm`的调用方式中。我们可以看到三个裸的`aten::mm`外部调用（图10）。证明它是同一个GEMM的证据是核函数名称与即时模式逐字节相同：gate和up是`...128x128...stages_32x5_tn`，down是`...128x256...stages_64x3_tn`。

### 融合的Triton核函数

![编译后的MLP追踪结果，CPU通道中框出了triton_poi_fused__unsafe_view_gelu_mul_0核函数，替换了即时模式中单独的gelu和mul核函数](/images/posts/9da66c6b785b.png)

这是整个编译课程的重点。两个即时模式的逐点核函数（GeLU和mul）加上一个reshape合并成了一个核函数：`triton_poi_fused__unsafe_view_gelu_mul_0`（图11）。让我们解码这个名称：

- `triton`：由Inductor的Triton后端生成（不是cuBLAS，不是ATen）。
- `poi`：逐点（Inductor将逐点核函数标记为`poi`，归约标记为`red`，持久归约标记为`per`）。
- `fused__unsafe_view_gelu_mul`：它合并的操作：`_unsafe_view`（reshape）、GeLU和`mul`。
- `0`：在图中的唯一ID。

为什么这是一个胜利？在即时模式下，中间结果`h = gelu(g)`是一个完整的`[8192, 3072]` bf16张量（约50 MB），GeLU核函数将其写入HBM，而mul核函数立即将其读回。融合将其保留在寄存器中（位于芯片内部且比HBM更近的内存）。Triton核函数读取`g`和`u`一次，计算`gelu(g) * u`，并写入结果一次。中间结果通过全局内存的一次完整往返被消除了。

## 让我们使用手工调优的核函数

到目前为止，我们让PyTorch（即时模式）和编译器（`torch.compile`）来选择我们的核函数。现在我们接入一个由人类专家编写并手工调优的核函数。我们使用`LigerGEGLUMLP`层，可以轻松地从Hugging Face Hub通过`kernels`库获取。

```python
from kernels import get_kernel

kernels_layers = get_kernel("kernels-community/liger-kernels", version=1).layers
kernels_geglu_mlp = kernels_layers.LigerGEGLUMLP(Config()).to(device, dtype=torch.bfloat16).eval()

```

完整的脚本在这里：`03_kernels_mlp.py`。

```bash
uv run 03_kernels_mlp.py --batch 64 --seq 128 --dim 768 --hidden 3072
uvx trace-util traces -b traces

```

### 为什么要使用kernels库

用Triton或CUDA编写内核是一回事，将其交付使用又是另一回事。内核必须针对你的GPU架构、CUDA版本和PyTorch版本的精确组合进行编译。这一步通常会出现问题（"在我机器上能跑"，缺少`nvcc`，Triton版本错误）。

`kernels`库将构建步骤从你的机器上移走。`get_kernel("kernels-community/liger-kernels", version=1)` 从Hugging Face Hub下载一个预构建的、版本固定的内核包，并将其缓存到本地（此处位于`~/.cache/...kernels-community--liger-kernels`）。其优势在于：

- 内核在CI中针对多种架构和版本组合一次性编译完成。你下载的是正确的二进制文件，无需自行编译。
- `version=1`固定了精确的构建版本，因此运行你脚本的每个人都会获得相同的内核。不会出现"更新某个包后速度变慢"的情况。
- 该包暴露了一个`.layers`属性，其中包含可直接替换的`nn.Modules`（例如`LigerGEGLUMLP`）。你只需将你的模块替换为它们的模块，模型中的其他部分无需改动。

### 为何调优后的内核更优

当我们说"调优"时，我们指的是两个具体方面，这两点都可以在追踪中看到。

![编译后的MLP追踪，在编译图运行之前，TorchDynamo、前导和守卫预处理操作框在CPU通道上](/images/posts/472dce4c9f82.png)
