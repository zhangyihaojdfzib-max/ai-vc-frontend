# Agentic Resource Discovery：让Agent（智能体）自主搜索工具、技能与其他Agent（智能体）

如果你今天使用Agent（智能体）进行开发，你可能了解三种协议。MCP为Agent（智能体）提供了调用工具的标准方式。技能为Agent（智能体）提供了使用指令的方式。A2A为Agent（智能体）提供了调用其他Agent（智能体）的方式。这三种协议都假设用户已经知道自己需要哪种工具、指令或Agent（智能体）。用户仍需负责发现、集成和维护这些能力。

Agentic Resource Discovery（ARD）规范是位于这些协议之上的发现层。这是一份由微软、谷歌、GoDaddy、Hugging Face等公司的贡献者共同开发的开放规范草案，得到了业界的广泛参与。它定义了Agent（智能体）和工具如何在联邦注册表中被编目、索引和搜索，从而使Agent（智能体）能够在运行时发现能力，而无需预先安装。这不是一个产品或市场，而是一个共享标准，任何公司都可以独立实现，任何Agent（智能体）或工具都可以参与其中。

在这篇文章中，我们将探讨该规范、Hugging Face如何实现它，以及你如何开始基于ARD进行开发。

## 发现问题

当前Agent（智能体）能力的模式是"先安装，后使用"。开发者将MCP服务器URL硬编码到配置文件中。用户通过插件将服务连接到其AI应用并重复使用。这对于Agent（智能体）每天使用的少数工具有效，但无法扩展到成千上万的临时交互界面。

退而求其次的做法是将所有可用工具描述塞入LLM（大语言模型）的上下文窗口，让模型自行选择。这受到上下文预算的限制。虽然也有一些基于搜索的策略，但描述往往过于单薄，难以有效区分。

ARD将选择过程移出LLM（大语言模型）。注册表使用更丰富的信号（如发布者身份、代表性查询、合规证明和标签）对能力进行索引。它暴露一个REST端点。客户端以自然语言进行搜索，模型调用搜索返回的任何内容。这一转变是从手动安装的静态目录转向基于意图的搜索，让Agent（智能体）能够动态找到合适的能力，并接入一个不断增长的MCP工具、A2A Agent（智能体）和其他服务生态系统，而无需预先配置每一个。

该规范定义了两项内容：

- 一种名为`ai-catalog.json`的静态清单格式，让发布者能够在已知URL上托管其能力。
- 一个位于`POST /search`的动态注册表API，提供实时的、排序后的发现功能。

## Hugging Face Hub上的ARD

Hugging Face的Discover Tool是我们对ARD的参考实现。它提供对数千个技能、ML应用和MCP服务器的搜索访问——无论是在Hugging Face上还是跨其他ARD发现服务。

它的工作原理是将Hub现有的基于Spaces的语义搜索与我们的Agent（智能体）技能相结合，并将结果作为ARD目录条目提供。Hub已经托管了一个包含运行Gradio应用、MCP服务器和演示的Spaces目录。其语义搜索支持一个`agents=true`标志，返回按Agent（智能体）导向元数据排序的Spaces，而Discover将该搜索转换为ARD规范。

该适配器应用了两个过滤器。首先，响应仅包含运行时阶段为`RUNNING`的Spaces。其次，响应媒体类型由请求驱动。支持三种媒体类型：

- `application/ai-skill`：默认类型。一个生成的`SKILL.md`，包装了Space的`agents.md`。
- `application/mcp-server+json`：针对标记为`mcp-server`的Space的MCP服务器目录条目。
- `application/vnd.huggingface.space+json`：供希望自行处理的客户端使用的原始Space元数据。

技能类型涉及额外的转换。许多Space附带一个`agents.md`文件，描述Agent（智能体）应如何与之交互。Discover读取该文件，并用技能消费者期望的前置元数据进行包装：`name`、`description`，以及涵盖Space ID、Hub URL、应用URL和原始`agents.md` URL的源元数据。结果是任何支持技能的客户端都可以通过其正常的技能流程安装或加载的技能。

对于标记为MCP的Space，适配器生成一个目录条目，指向该Space通过HTTP传输的Gradio MCP端点。当Hub提供运行时域名时，URL使用该域名；否则使用标准的`.hf.spaceslug`约定。

## 使用方法

`discover`已内置于Hugging Face CLI（`hf`）中。要开始使用并为你或你的Agent（智能体）提供访问权限：

```bash

uv tool install huggingface_hub


hf discover search "Fine tune a language model"


hf discover search "Generate an image" --json --kind mcp


hf discover search "Purchase aeroplane tickets" --registry-url <catalog-url>

```

### REST API和MCP工具

你也可以直接使用REST API或MCP服务器搜索目录。

Hugging Face目录发布在其已知URL上：

```
https://huggingface.co/.well-known/ai-catalog.json

```

要直接调用搜索：

```
POST https://huggingface-hf-discover.hf.space/search

```

```bash
curl -s https://huggingface-hf-discover.hf.space/search \
  -H "Content-Type: application/json" \
  -d '{
    "query": {
      "text": "fine tune a sentence transformer",
      "filter": {
        "type": ["application/ai-skill"]
      }
    },
    "pageSize": 5
  }'

```

搜索MCP服务器

```bash
curl -s https://huggingface-hf-discover.hf.space/search \
  -H "Content-Type: application/json" \
  -d '{
    "query": {
      "text": "transcribe some audio",
      "filter": {
        "type": ["application/mcp-server-card+json"]
      }
    },
    "pageSize": 5
  }'

```

或者，连接任何MCP客户端，通过MCP端点使用`https://huggingface-hf-discover.hf.space/mcp`来搜索目录。

## 这对规范意味着什么

ARD将发现与执行分离。静态清单格式由媒体类型驱动，因此任何工件协议都可以使用相同的封装，而无需更改规范级别。注册表API是普通的HTTP REST，因此任何客户端都可以对其进行联邦化。Discover是生态系统中该规范的几个参考实现之一，并且由于联邦化内置于协议中，通过一个服务进行的搜索可以展示由另一个服务托管的能力。

Discover Tool是对该设计的工作测试。它没有发明新的工件格式。它将现有的搜索后端（Hub）包装在规范的封装中，并让相同的Spaces根据客户端的请求以技能或MCP服务器的形式呈现。

下一步是与规范的联邦化模式（`auto`、`referrals`、`none`）更紧密地集成，以及在Hub端支持用户和组织配置文件上的静态`ai-catalog.json`清单。一旦实现，任何Space发布者都将能够通过标准的已知URI机制宣传其能力。
