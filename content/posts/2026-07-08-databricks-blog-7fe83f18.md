在Databricks，随着我们积极采用AI进行工程开发，构建软件的方式正在迅速变化。过去一年中，用于代码编写的模型和工具框架迅速扩展，为开发者提供了比以往更多的选择。随着选项增多，了解哪些编码Agent（智能体）在实际编码任务中表现最佳，以及任务性能如何随价格变化，变得愈发重要。

本文分享了我们在Databricks内部构建的编码基准测试的结果和方法，该测试基于我们的工程师在Databricks代码库中执行的实际编码任务来评估工具。这些任务涉及对包含数百万行代码的代码库进行编辑，涵盖多种流行语言（Python、Go、TypeScript、Scala等），并且任务和解决方案都经过仔细审核以确保准确性。这并非旨在全面覆盖，但该实践揭示了洞察，已使我们的工程团队在使用编码Agent（智能体）时显著提高了效率。以下展示了模型和工具框架在整体基准测试中的得分情况：

![成本与性能对比图](/images/posts/1d0dacee9f18.png)

图1：我们基准测试中的成本与性能对比

我们分析的主要结论如下：

1. 编码任务的帕累托前沿（即在给定成本下获得最佳质量）包括来自OpenAI、Anthropic和开源社区的模型。这意味着目前只有混合使用多种工具才能提供前沿性能。
2. 开源模型，尤其是GLM 5.2，现在能够处理最高难度的任务。
3. 模型的Token价格并不能很好地反映端到端任务的实际成本。更大的模型可能在Token效率上高得多，且总体成本更低。
4. 调用模型的工具框架对成本和质量有显著影响。在许多情况下，像Pi这样简单的工具框架在我们的工作负载中表现最佳。

让我们对每一点进行更深入的探讨。

### 模型大致分为“能力层级”

具体结果相差几个百分点在实际任务中往往可以忽略不计。我们更关注那些能帮助我们判断针对不同任务应使用哪些模型的模式化规律。事实上，结果显示模型和工具框架清晰地聚类为三个能力层级。

![模型能力层级图](/images/posts/ff162b496644.png)

图2：在我们的整体结果中出现了三个不同的能力层级，每个层级中哪些模型有效存在细微差别

在性能的高端，我们看到最智能的模型在解决各类问题方面非常有效，但成本极高。中低智能模型在处理常见任务时仍然非常高效，并且在许多情况下，它们的成本也显著更低。

在日常工作中，工程师会处理大量复杂度差异显著的任务：像切换标志或更新配置这类常见操作任务并不需要极其智能的模型，但更深入的设计探索则需要。然而，过去我们的默认模型总是最昂贵的。基于此分析，我们确定应将更多工作推给Haiku和GPT 5.4 Mini这类模型。

### 开源模型已可用于编码

关于GLM 5.2的讨论一直很热烈，我们的结果也证明GLM可以成为我们许多开发者的日常驱动模型。它进入了顶级能力层级，在质量上与Opus 4.8在统计上不相上下，但每任务成本为1.28美元，而Opus为1.94美元。

GLM的质量得分与我们内部试用GLM进行日常开发的开发者给出的定性反馈一致。由于其在日常编码任务中的出色表现，我们一直专注于以最佳性能提供GLM服务，证据表明是时候开始将其作为编码的日常驱动模型进行部署了。

### 每任务成本 vs 每Token成本

开发者通常通过观察Token成本来估算完成编码任务时模型的费用。然而，我们发现，由于模型间推理效率的差异，Token成本往往不能很好地反映总体任务成本。这凸显了进行任务级基准测试的必要性，因为任务形态和复杂度在不同上下文中可能有所不同。

例如，Sonnet 5每Token成本比Opus 4.8便宜约1.7倍，但在我们的任务中，我们发现Sonnet每任务成本为2.09美元，而Opus为1.94美元，同时任务完成得分低6个百分点（81%对87%）。这主要是因为Sonnet 5需要更长的处理时间和读取更多内容才能完成任务，消耗的Token量是Opus的1.9倍。

### 工具框架对效率有重大影响

当我们通过两种不同的工具框架（Claude Code/Codex vs Pi）运行同一个模型并采用相同的思考努力时，我们观察到每任务成本存在显著差异（某些情况下超过2倍），而质量保持不变。主要区别在于每个工具框架在每次交互中向模型提供的上下文量。

![工具框架对效率的影响](/images/posts/3a7408ddfa4d.png)

Pi每次交互发送的上下文量大约少3倍。它更好地管理了上下文，保持了更紧凑的工作集，并以更少的运行次数完成了任务。

![每任务重新提供给模型的上下文总量](/images/posts/e84d977e894d.png)

这里的教训并非某个工具框架总是更便宜，或者原生工具框架更差。相反，模型选择只是拼图的一部分。正是为了建立这种灵活性，我们投资了Omnigent，以实现模型和工具框架的无缝切换。

## 为什么要构建自己的基准测试？

像SWE-Bench和TerminalBench这样的公开基准测试很有用，但它们无法回答我们的问题。原因如下：

- 这些任务是公开的，因此解决方案会随着时间的推移泄露到训练数据中。
- 我们发现这些结果对我们的代码库不具有代表性，我们的代码库涵盖10多种语言以及用Scala、Go、Rust、Java和Python、Bazel、Protobuf等编写的众多服务。

通过基于我们自己的Pull Request构建基准测试，我们可以更有信心地做出决策，确保不会因推出优化而阻碍我们的开发者。

### 任务构建

我们的工程师每天合并数千个代码变更，因此我们已经拥有一个很好的数据集来构建基准测试。一个好的Pull Request是一个丰富的产物，包含展示开发者迭代的提交、人工审查以及有助于验证代码变更符合其意图的测试。然而，我们需要多项质量检查和筛选，才能从中构建出高质量的基准测试：

- **时效性**：我们从近期历史中提取，以便任务反映我们当前的构建方式，包括当前使用的框架、模式和约定。
- **人工编写**：过滤掉机器人提交、服务账户、完全AI生成的变更以及自动生成的变更。
- **关联的高质量测试套件**：筛选包含用于验证代码变更的高质量测试的Pull Request。
- **自包含**：变更局限于少数几个模块。
- **代表典型任务**：我们从整个技术栈的任务分布中选择Pull Request：Scala后端服务、Rust系统代码、React和TypeScript前端、Protobuf和gRPC合约以及Bazel配置。

![任务构建分步计划](/images/posts/4dc09bffded7.png)

一旦我们有了候选的Pull Request，我们专注于通过以下方式构建明确指定的任务：

1. **理解意图并将其总结为提示词。** 我们阅读 PR 以了解其实际目的，然后描述我们想要的结果。通常，这意味着通过陈述问题或目标、注明任何约束条件，并移除对解决方案的描述，来重写 PR 描述。例如，移除关于为什么某个错误修复是正确的解释非常重要，因为这会使任务过于简单。

2. **拆分出相关的测试。** 非测试文件是模型必须自行复现的变更部分，因此我们将测试文件放在一边，并确保能够编译这些非测试文件。我们的构建系统已经能够确定哪些测试依赖于原始 PR 中修改过的文件，因此我们完整地运行了所有这些测试目标。

### 额外防护措施

![额外防护措施](/images/posts/85569d10ad10.png)

在早期实验中，一些模型的得分好得令人难以置信，因此我们手动检查了轨迹以了解这些 Agent（智能体）运行过程中发生了什么。我们发现，由于我们最初的设置，“正确”的实现仍然可以在工作树的 Git 历史中恢复！每个任务都源自一个已合并的提交，因此没有什么能阻止一个拥有 shell 的 Agent（智能体）在 Git 历史中向前查找。为了解决这个问题，我们封存了 Git 历史：在每次运行期间，我们将工作副本与仓库完全隔离。
