Fizz.hu 是一家由 OTP 集团支持的快速增长的电商市场平台。作为 OTP "超越银行业务"战略的一部分，该平台于两年前推出，现已汇聚了超过 500 家商户，提供超过 150 万件活跃商品，涵盖电子产品、家居用品等多个类别。

从一开始，数据就被置于优先地位。但公司最初的基础架构较为简单：使用 Microsoft SQL Server 和 Power BI，通过每日批量加载进行报表生成。随着产品目录的扩展和新用例的出现，这套架构开始显现出其局限性。

Fizz 需要的不仅仅是一个传统的数据仓库。它需要一个一体化平台，能够支持 SQL、Python 以及未来的 AI 计划，同时不增加运营复杂性。团队在 Databricks SQL 中找到了答案，并决定迁移到能够随业务扩展的湖仓一体架构。

## 一次务实的迁移，三个月内完成

当 Tamas Bacskai 作为数据主管加入时，他的任务很明确：建立一个数据导向的团队，并规划一条可扩展的前进道路。现有的 SQL Server 环境充当着基础仓库的角色，但 Python 工作负载在单独的虚拟机上运行，治理能力有限，扩展意味着增加基础设施支出。

团队评估了三个选项：继续只专注于数据仓库；将高级工作负载拆分给另一个开发团队；或者采用能够统一 SQL 和 Python 的湖仓一体架构。Bacskai 表示，湖仓一体模型"满足了所有要求"，包括未来扩展到机器学习和 AI 领域。

Fizz 没有追求完美的重新设计，而是采取了 MVP 优先的方法。在外部合作伙伴的支持下，他们迁移了大约 50 张表和几个存储过程，并在 Databricks SQL 中重建了核心视图。目标很简单：保持报表正常运行，但将其指向新的引擎。

"这有点非传统，" Bacskai 说。"我们不想要一个重写一切的完美迁移。我们希望尽可能快地迁移，之后再优化和现代化。一旦数据进入 Databricks，做这些就容易多了。"

三个月内，旧的 SQL Server 被完全关闭。Power BI 报表无缝地继续运行，现在由 Databricks 提供支持。"这并非不可能，只是很有雄心，" Bacskai 说，"但它是可预测且可实现的。"

## 更快的报表生成与更好的服务水平

立竿见影的影响体现在性能上。此前，每日的 ETL 周期可能需要三到四个小时，报表在早上 7 点或 8 点之前无法可靠地提供。这给更早开始工作的业务用户带来了不便。

借助 Databricks SQL，Fizz 将其端到端的夜间处理窗口缩短至大约 90 分钟。现在，即使在周末和节假日，报表也能在凌晨 4:30 前准备就绪。Power BI 的刷新周期缩短了约 50%，千兆字节级别的导出现在只需几分钟即可完成。

这些性能提升并非源于过度配置的基础设施。Fizz 运行的工作负载相对适中——青铜层和白银层总计约 10 TB——但新的 SQL 引擎和自动优化功能带来了可衡量的改进，而无需持续调优。

"并不是我们只是投入了更多资金或使用了更大的集群，" Bacskai 澄清道。"SQL 执行引擎本身就更快。它会自动优化，一切功能都为我们准备好了。"

同样重要的是，Databricks 消除了运行 Python 需要单独环境的需求。现在所有作业都在平台内原生运行，简化了运营，并为未来的机器学习计划奠定了更清晰的基础。

## 利用 AI 和自助服务扩展能力

从一开始，Fizz 就希望选择一个不会限制其 AI 雄心的平台。即使在迁移期间，团队也预见到了对机器学习、生成式 AI 和更高级数据治理日益增长的需求。

如今，Databricks 可以在单一环境中支持 SQL、Python 和机器学习工作负载。团队正在探索掩码策略和治理控制，以加强 GDPR 和欧盟 AI 法案的合规准备。AI 驱动的 SQL 函数将有助于清理和标准化产品名称，减少对复杂正则表达式的依赖，并加速数据准备工作。

通过 Databricks Genie，自助分析服务也在扩展。业务用户可以用匈牙利语提出自然语言问题，而无需编写 SQL。目前约有 20 名活跃用户依赖 Genie，这大约节省了分析师先前用于回答临时请求的 20% 的时间，使团队能够腾出精力进行更具附加值的努力。

"我们的 Genie 设置尚未完成，" Bacskai 指出，"但这意味着我们不必为了提问而学习 SQL。你可以直接与你的数据对话。"

对于一家不断增长的电商公司而言，其价值超越了速度。Databricks 提供了一个统一的、支持 AI 的基础，能够随着从营销数据集成到模型服务端点等新用例而扩展，而无需更大的团队来管理。

"Databricks SQL 比我们预期的要好得多，" Bacskai 说。"这是我们乐于使用的工具。它可以完成我们想要的一切，因此我们可以直接构建和创造我们想要的东西。"
