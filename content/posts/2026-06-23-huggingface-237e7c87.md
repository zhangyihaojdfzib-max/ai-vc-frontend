# 使用 CUGA 构建真正的 Agent 应用：轻量级框架上的二十四个工作示例

TL;DR——构建一个 Agent 主要涉及管道工程：工具、状态、护栏、从单个 Agent 扩展到多个 Agent。CUGA（pip install cuga），即可配置通用 Agent（Configurable Generalist Agent），IBM 面向企业的 Agent 框架，负责处理这些工作，你只需编写工具列表和提示词。我们构建了二十四个单文件应用来证明这一点。在此完整阅读一个示例，然后了解同一个 Agent 如何无需重写即可在生产环境中以自主可控的方式运行。

大多数 Agent 应用在 Agent 做任何有用的事情之前，都要先花一周时间搭建管道。你选择一个框架，连接模型客户端，编写工具适配器，构建某种将状态流式传输到 UI 的方式，然后还要决定 Agent 的实际用途。有趣的部分总是最后才到来。

CUGA 颠覆了这一点。它是 IBM 的开源 Agent 框架，为你处理规划、执行循环、工具调用和状态管道。剩下的才是真正属于你的部分：Agent 可以访问哪些工具，以及你告诉它做什么。为了展示这在实践中的效果，我们构建了 cuga-apps：二十四个小型工作应用，每个都是一个包含单个 CugaAgent 的 FastAPI 文件，从电影推荐器到 IBM Cloud 架构顾问。它们的存在是为了被阅读和复制。你可以点击浏览实时图库。

本文详细讲解其中一个应用，说明该框架为你省去了哪些工作，并展示当你需要为生产环境进行治理时，相同的代码会走向何方。无需先学习新框架。如果你写过 FastAPI 路由，你就能读懂每一行代码。

## 为什么是框架，而不是另一个框架

对于这个领域的任何东西，一个合理的问题是它能让你少写什么。CUGA 的答案是：围绕模型的编排工作，否则你每次都要重新构建。

它在行动之前先规划，然后通过工具调用和生成代码（CodeAct）混合执行。在一个运行二十步的长任务中，大多数 Agent 失败的原因是丢失中间结果并在下一步中（通常是错误地）重新推导它们；CUGA 持有该状态并运行一个反思步骤，可以捕获错误的调用并重新规划，而不是盲目地继续执行。正是这种机制使其在 AppWorld 和 WebArena 等 Agent 基准测试中名列前茅，而不是需要你手动调优的东西。

你还可以通过配置而非代码来设置成本/延迟权衡：快速、平衡和精确推理模式，以及在你信任的任何沙箱（本地、Docker/Podman 或 E2B 云）中执行代码。相同的 Agent 定义，不同的拨盘。这个拨盘的重要性远超其表面。大多数框架假设底层有一个前沿模型，并在计划出错时依赖它来恢复；CUGA 自己完成这项工作。规划、反思步骤、保持长程运行不偏离轨道的变量追踪——这些都是框架承担的负载，否则模型必须承担，这使得较小的开源权重模型能够在通常无法胜任的地方坚持下来。这就是托管应用在 gpt-oss-120b 而非前沿 API 上运行的原因。运行你能调用的最大模型是通常的做法；CUGA 的做法是相信一个较小的开源模型就足够了。

没有哪个单独的组件是 CUGA 独有的。不同之处在于它们已经预先组装好，所以你只需配置它们，而不是将它们连接在一起。你接触的 API 很小——用工具列表和提示词构建一个 CugaAgent，然后 await agent.invoke(...)。这行以下的所有内容都是框架。

具体来说，这包括可互换的工具（OpenAPI、MCP 和 LangChain 函数都以相同方式绑定）、带有变量管理和自我修正的长期规划（2025年7月25日至2026年2月26日在 AppWorld 以及2025年2月25日至2025年9月25日在 WebArena 上排名第一背后的机制）、声明式护栏、通过 A2A 进行的多 Agent 委派、基于 Docling 的 RAG，以及一个环境变量即可切换的提供商（pip install cuga，然后 OpenAI、watsonx、Ollama 等）——每一样都是你原本需要自己构建的。名称的第一个词说明了作用：可配置；困难的部分已经处理好了，你的工作只是完成任务。

## 一个应用，从开始到结束

这是 IBM Cloud 顾问——一个为架构推荐真实 IBM Cloud 服务的 Agent。整个应用放在一个文件中：一个包含 Agent 工厂、工具和提示词的 main.py，外加一个小型 UI。

![ibm_cloud_advisor cuga-app 的结构：main.py 文件布局，一个内联 @tool (search_ibm_catalog) 调用 IBM Cloud Global Catalog API，同时在一个工具列表中包含一个 MCP 网络搜索工具，以及一个强制"先查目录再推荐"的系统提示词。](/images/posts/87d152f429b7.png)

整个 Agent 就是这样的：

```python
def make_agent():
    from cuga import CugaAgent
    from _llm import create_llm

    return CugaAgent(
        model=create_llm(
            provider=os.getenv("LLM_PROVIDER"),
            model=os.getenv("LLM_MODEL"),
        ),
        tools=_make_tools(),
        special_instructions=_SYSTEM,
        cuga_folder=str(_DIR / ".cuga"),
    )

```

四个参数。模型来自一个小型工厂（create_llm），它根据环境变量与 OpenAI、Anthropic、watsonx、LiteLLM 或 Ollama 通信。应用代码中没有任何内容知道背后是哪个模型。cuga_folder 是这个应用保存其状态和任何策略的地方。承载应用的两个参数是 tools 和 special_instructions。

这些工具将本地函数与托管函数混合在一起：

```python
def _make_tools():
    from langchain_core.tools import tool

    @tool
    def search_ibm_catalog(query: str) -> str:
        """搜索 IBM Cloud Global Catalog 以查找真实的 IBM Cloud 服务。
        在推荐服务之前始终调用此函数以验证它们是否存在。"""
        ...  

    from _mcp_bridge import load_tools
    web_tools = load_tools(["web"])

    return [search_ibm_catalog, *web_tools]

```

这里有一个适用于每个应用的模式：MCP 工具和内联工具之间的分工。通用的、无状态的能力来自共享的 MCP 服务器；load_tools(["web"]) 引入网络搜索，无需你托管任何东西。特定于此应用的任何内容都作为普通 Python 函数内联定义，例如 search_ibm_catalog，其文档字符串是 Agent 用来决定何时调用它的依据。你编写属于你的那个工具，然后借用其余的。

Cloud 顾问的提示词告诉 Agent 在命名任何服务之前先搜索目录，推荐三到七个服务并说明每个服务在设计中的角色，并且永远不要编造服务名称。最后这条规则值得保留：推荐不存在的 IBM Cloud 服务的 Agent 比没有 Agent 更糟糕，因此提示词强制每次推荐都先进行目录查找。以有序步骤和明确的"不要编造"规则编写的提示词表现良好；以角色人设编写的提示词则会偏离方向。

这就是这个应用。一个工具，一个流程，四行构造函数。它周围的 FastAPI 路由是普通的 Web 代码：浏览器将问题发送到 /ask，实时面板轮询 /session/{thread_id} 端点以获取状态。没有数据库；状态是每个 thread_id 的 Python 字典，只有 Agent 通过其工具写入。当 Agent 在运行中途调用工具时，面板会重新绘制。UI 不是逻辑的第二个副本；它是 Agent 修改过的状态的视图。

## 承担重任的约定

有一个细节很容易被忽略，但它实际上是关键：每个内联工具都返回相同的小型信封。成功看起来像 {"ok": true, "data": {...}}；失败看起来像 {"ok": false, "code": "...", "error": "..."}。

这看起来像样板代码。其实不是。CUGA 的规划器能优雅地处理声明的失败（"地理编码没有返回任何内容，跳过该部分并继续"），但会在未声明的失败上卡住，此时原始堆栈跟踪在规划中途冒泡，导致运行脱轨。在所有应用中，那些可靠运行的应用，其工具从未向 Agent 抛出裸异常。一个乏味的约定，但这是 Agent 能够恢复还是直接失败的区别所在。

上述拆分之所以有效，是因为通用部分已经在某处运行。应用程序反复调用的能力——网页搜索、维基百科/arXiv、地理编码与天气、金融报价等——存在于托管在 IBM Code Engine 上的 7 个公共 MCP 服务器（36 个工具）中，无需认证。一个小型桥接器会自动解析它们的 URL，而在线画廊提供了一个 MCP 工具浏览器，让你在将其接入 Agent（智能体）之前，就能通过表单调用其中任何一个工具。

## 一个库，而非演示

这里有二十多个精良的应用，其意义远超任何一个单独的应用：一旦你读懂了云顾问，你就读懂了所有应用。它们共享一个骨架——电影推荐器将 IBM 目录工具替换为知识 MCP 服务器，网页研究者几乎完全依赖网页工具——因此 cuga-apps 实际上是一个起点目录。你克隆仓库，找到最接近你想法的应用，然后编辑其工具列表和提示词（`HOW_TO_BUILD_AN_APP_FAST.md` 和 `ADDING_AN_APP.md` 详细说明了这一过程）。有些应用甚至是通过将一份规范文件和一行的简要说明交给编码助手生成的——模型能够规律地复现，意味着你也能规律地学习。在克隆任何东西之前，你可以在在线画廊中逐个点击查看。

它们还按类别分布，因此无论你在构建什么，总有一个应用已经涵盖了所需的部分。有一个研究集群（Paper Scout 按引用次数对 arXiv 论文进行排序；Wiki Dive 和 Web Researcher 进行引用综合），一个日常生产力集合（城市简报、旅行、食谱、路线），一个文档与媒体组，对 PDF、音频和视频进行 RAG（检索增强生成），一个监控实时指标的运维角落，以及一个基于真实 IBM 产品文档的企业示例。Ouroboros 是一个七 Agent（智能体）的线索生成系统；打开它了解多 Agent（智能体）形态。而 Meetup Finder 通过 Playwright 驱动无头 Chromium，从 Meetup、Luma 和 Eventbrite（这些平台都关闭了其公开搜索 API）中提取结构化活动；打开它了解浏览器自动化，这也是 CUGA 的起点及其强大 WebArena 成果背后的驱动力。

克隆前有两个注意事项。真正的目录位于内部的 `cuga-apps/cuga-apps/apps/` 目录，而非外部的那个。并非所有应用都同样精良，因此 UI 将它们标记为“展示”或“附加应用”，并默认显示“展示”；从云顾问或电影推荐器开始，以获得一个可用的基线。

## 让你的 Agent（智能体）保持在边界内

一个搜索目录的演示 Agent（智能体）风险很低。将同样的模式应用于写入文件、运行 shell 命令或接触生产环境的场景，问题就变了：你如何阻止它做出让你后悔的事情？

CUGA 在运行时中回答了这个问题，而不是在你事后添加的包装器中。这个开源 Agent（智能体）附带了一个策略系统，你将策略附加到同一个 Agent（智能体）对象上：

```python
await agent.policies.add_intent_guard(
    name="阻止强制推送",
    keywords=["--force", "--no-verify"],
    response="已阻止：不允许使用破坏性的 git 标志。",
)

```

这是一个意图守卫，是六种策略类型之一，每种策略都回答了团队在放开 Agent（智能体）之前会提出的问题：

- **意图守卫**——它能直接拒绝一个请求吗？
- **工具审批**——它能在运行风险工具前暂停等待人工确认吗？
- **工具指南**——我能否在不重写工具的情况下，引导其具体使用方式？
- **剧本**——我能否为重复性任务固定一个已知良好的流程？
- **输出格式化器**——我能否强制最终响应符合要求的格式？

第六种类型是 `CustomPolicy`，当以上都不适用时，它作为备用方案。时机把握很重要，因为它并非都在同一阶段：意图守卫在 Agent（智能体）选择工具之前检查请求，工具审批在 Agent（智能体）生成代码之后运行，并检查该代码使用了哪些工具，而输出格式化器仅在最终消息生成后触发。触发条件也不仅限于关键词匹配：它们存储在一个 `sqlite-vec` 存储中，并进行语义匹配，因此策略会根据用户的意图触发，而不仅仅是精确的关键词。可以基于语义相似度、Agent（智能体）状态或特定工具的触发进行匹配。策略本身位于构造函数中的那个 `.cuga` 文件夹中，与代码一起进行版本管理，而不是在单独的配置中漂移。

如需查看工作示例，请打开 Ouroboros——一个七 Agent（智能体）的线索生成应用，它为其监管者附加了三个策略（一个意图守卫、一个工具指南和一个输出格式化器），因此它是同一个文件中同时演示治理和多 Agent（智能体）形态的应用。

## 超越单个 Agent（智能体）的扩展

当一个应用超出单个聊天循环的范畴时，有两种扩展方式很重要。当一个 Agent（智能体）会在其自身的上下文中不堪重负（工具太多，需要理清的证据太多）时，你可以拆分工作。一个 `CugaSupervisor` 将任务委派给专门的 `CugaAgent`，每个 Agent（智能体）都有自己的工具、提示词和隔离的上下文，监管者只负责决定将子任务交给哪个专家。无论底层有多少工具，其规划面始终保持精简，一个不稳定的工具只会导致一次委派失败，而不是整个运行失败。专家甚至不必是本地的；它可以是通过 A2A 协议访问的外部 Agent（智能体），以同样的方式进行委派。增加一项能力意味着增加一个专家，而不是重写协调器。

另一种扩展方式封装的是知识而非工具：Agent 技能，一个包含 `SKILL.md` 剧本的文件夹，Agent（智能体）仅在任务需要时才将其拉入上下文，这样单个提示词就不必承载 Agent（智能体）可能需要知道的所有信息。两者都使用相同的构建块（工具、提示词、状态、策略），只是组合层级更高了。

之前的线索生成应用 Ouroboros 使这种模式具体化。它有一个监管者，管理着七个专家（侦察员、网站审计员、客户之声、人员查找器、技术栈扫描器、收入估算器，以及一个进行综合的推销邮件撰写员）。每个专家都是一个加载到 `CugaAgent` 中的技能，监管者通过自动生成的 `delegate_to_<name>` 工具来调用它。增加第八个专家只需一行工厂代码，无需重写协调器。如果你想了解端到端的多 Agent（智能体）形态，请阅读其 `main.py` 和 `ARCHITECTURE.md`。

还有第三种扩展方式，它又回到了技能本身。通过 ALTK-Evolve（CUGA 的在岗学习框架），Agent（智能体）可以从自身的运行中优化一项技能，从而使今天完成的任务能让明天的任务更快、更准确。专家加载的 `SKILL.md` 最终会包含 Agent（智能体）在你编写内容基础上学习到的东西。相同的构建块，只是现在使用一个技能会教会下一个技能。你不再需要为你上周已经解决的问题重新编写提示词。

## 通过构建进行治理

治理在技术栈中的位置决定了生产环境的故事走向。一个极简的 Agent（智能体）库会提供良好的原语，而将治理（策略、审批、审计、身份）留给你去组装。CUGA 选择了另一条路：策略、人工参与审批、`.cuga` 状态文件夹和自托管，从第一行代码起就是框架的一部分，而不是你后来添加的层。

这改变了你将 Agent（智能体）投入生产时的工作方向。你不是在为为开放访问而构建的东西加装控制措施；控制平面已经存在。受治理的路径是默认路径，不受治理的捷径才是你需要主动选择的。因此，剩下的工作范围很窄：收紧围绕少数几个真正接触外部世界的工具的沙箱，而不是围绕它们发明治理机制。

## 同一个 Agent（智能体）的最终归宿

这就是回报，也是这一切如此构建的原因。因为框架小巧、开源、与模型无关，并且已经实现了自我治理，你在笔记本电脑上编写的 Agent（智能体）与在锁定部署环境中运行的 Agent（智能体）是同一个。你不需要移植它。你只需重新部署它。

这就是IBM Sovereign Core所构建的基础，也是我们在此之上推进CUGA的方向。我们已在别处详细阐述过，简而言之：Sovereign Core在我们所谓的“边界隔离”机制下运行CUGA Agent（智能体）——数据、控制平面和执行引擎均位于同一逻辑边界内，Agent（智能体）在租户自有工作空间的临时隔离容器中运行。模型也在其中运行。部署默认采用完全气隙隔离的`gpt-oss-120b`，运行于你的基础设施内部，且工具仅能通过逐工具授权访问私有VNET。每个推理步骤都会向租户内部的Grafana Tempo后端发出OpenTelemetry追踪信息，无任何遥测数据外传。没有任何信息离开该边界。

Agent（智能体）的定义无需为此改变；改变的是其周围的部署方式。而实现这一点的可能性，源于上述所有要素——能力、策略和模型选择都存在于一个可读的运行时中。这正是我们构建时所押注的方向：当Agent（智能体）的运行时是一个黑盒时，主权只是一纸承诺；但当它是开放代码时，主权便是可验证的事实。你克隆的应用和你编写的Agent（智能体）都基于同一个开放运行时，这一主张正是由此而来。

不过，对开发者而言，其意义不言自明。一个Agent（智能体）应用可以只是你脑中记住的一个文件。你真正需要编写的只有工具和提示词。这些应用是一个可供学习的代码库，而非封闭的演示。当风险升级时，治理机制已内置于运行时中——你无需为了安全而重建Agent（智能体）。
