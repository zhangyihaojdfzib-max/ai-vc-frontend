Grep 是一款极速的代码搜索工具。你可以在超过一百万个代码仓库中搜索特定的代码片段、文件或路径。搜索结果需要即时显示，无需等待加载转圈。

最初，Grep 使用 Create React App (CRA) 构建，是一个完全客户端渲染的单页应用 (SPA)，速度很快——但随着 CRA 现已弃用，我们希望更新代码库，使其在未来更快、更易于维护。

以下是我们将 Grep 迁移到 Next.js 的方式——保留了 SPA 的交互性，同时通过 React 服务端组件获得了性能提升。

## 链接到标题为什么选择 Next.js 和 React 服务端组件？

传统的 SPA 可以在页面之间快速导航。然而，这是以需要在页面可交互之前加载并解析更大的客户端 JavaScript 包为代价的。

我们希望实现尽可能快的初始页面加载，同时保持后续页面之间的快速导航。Next.js 可以预渲染页面的大部分内容，以实现可预测且快速的初始加载。然后，它可以智能地预取其他页面，保持 SPA 的快速导航体验。

让我们谈谈我们是如何迁移到 Next.js 的，同时让搜索输入交互更快，并在此过程中消除不必要的布局偏移。

## 链接到标题保留即时搜索体验

Grep 需要尽可能快的搜索。在首页开始输入，过渡到 `/search`，并继续输入——全程不丢失焦点。请注意视频中搜索框如何从首页中央移动到 `/search` 页面的导航栏。

使用传统的 SPA，搜索状态保留在客户端，UI 保持不变。但是，当将状态迁移到服务端时，我们需要在页面导航期间保持搜索输入状态的一致性。

现在，我们需要即时显示搜索结果。挑战在于服务端和客户端之间的交接。每次按键都回到服务端可能会很慢，而所有数据获取都在客户端进行则意味着初始加载体验较慢。

### 防止过期或乱序的结果

快速输入带来了另一个挑战：较旧的网络响应有时会在较新的响应之后到达，短暂显示过时的结果。例如，输入"foo"，然后删除回"f"，可能导致"foo"的结果在"f"已经渲染后重新出现。

1. 使用`useOptimistic`进行乐观UI更新：React的`useOptimistic`钩子处理跨异步更新的乐观状态，确保UI始终反映最新的用户输入——无论网络延迟如何
