## 🤔 什么是 Agent？

任何使用 AI 的高效系统都需要为 LLM 提供某种访问现实世界的途径：例如，调用搜索工具获取外部信息，或对某些程序执行操作以完成任务。换句话说，LLM 应该具备**自主性**。Agent 程序是 LLM 通往外部世界的门户。

AI Agent 是**LLM 输出控制工作流**的程序。

任何利用 LLM 的系统都会将 LLM 输出集成到代码中。LLM 输入对代码工作流的影响程度，就是 LLM 在该系统中的自主性水平。

需要注意的是，根据这一定义，"Agent"并非一个离散的、0 或 1 的概念：相反，"自主性"在一个连续的频谱上演变，取决于你在工作流中赋予 LLM 多少权力。

下表展示了不同系统中自主性的变化：

多步骤 Agent 的代码结构如下：

```python
memory = [user_defined_task]
while llm_should_continue(memory): 
    action = llm_get_next_action(memory) 
    observations = execute_action(action)
    memory += [action, observations]

```

因此，该系统在一个循环中运行，每一步执行一个新动作（该动作可以涉及调用一些预定义的**工具**，这些工具只是函数），直到其观察结果表明已达到解决给定任务的满意状态。以下是一个多步骤 Agent 如何解决简单数学问题的示例：

## ✅ 何时使用 Agent / ⛔ 何时避免使用

当你需要 LLM 确定应用的工作流时，Agent 非常有用。但它们往往被过度使用。问题是：我是否真的需要工作流中的灵活性来高效解决当前任务？
如果预定义的工作流经常无法满足需求，那就意味着你需要更多的灵活性。
举个例子：假设你正在开发一个处理冲浪旅行网站客户请求的应用。

你可以提前知道请求将属于以下两个类别之一（基于用户选择），并且为这两种情况分别设定了预定义的工作流。

1. 想了解旅行信息？⇒ 让他们使用搜索栏搜索你的知识库
2. 想与销售沟通？⇒ 让他们填写联系表单。

如果这种确定性的工作流能够处理所有查询，那么完全可以直接编写所有代码！这将为你提供一个 100% 可靠的系统，不会因为让不可预测的 LLM 介入工作流而引入错误风险。为了简单和稳健，建议尽量不使用任何 Agent 行为。

但如果工作流无法提前很好地确定呢？

例如，用户想问："我周一可以来，但我忘了带护照，所以可能会推迟到周三。是否可以在周二早上带我和我的东西去冲浪，并附带取消保险？"这个问题涉及许多因素，上述预定义的标准可能都不足以处理这个请求。

如果预定义的工作流经常无法满足需求，那就意味着你需要更多的灵活性。

这正是 Agent 设置发挥作用的地方。

在上述例子中，你可以创建一个多步骤 Agent，它可以访问天气 API 获取天气预报、Google Maps API 计算旅行距离、员工可用性仪表板以及基于知识库的 RAG 系统。

直到最近，计算机程序仍局限于预定义的工作流，试图通过堆叠 if/else 分支来处理复杂性。它们专注于极其狭窄的任务，比如"计算这些数字的总和"或"找到这个图中的最短路径"。但实际上，大多数现实任务，比如我们上面的旅行示例，并不适合预定义的工作流。Agent 系统为程序打开了广阔的现实世界任务的大门！

## 代码 Agent

在多步骤 Agent 中，每一步 LLM 都可以编写一个动作，形式为对外部工具的一些调用。编写这些动作的常见格式（由 Anthropic、OpenAI 等公司使用）通常是"将动作编写为工具名称和参数的 JSON 格式，然后解析以知道要执行哪个工具以及使用哪些参数"。

多篇**研究论文**表明，让 LLM 用代码调用工具效果要好得多。

原因很简单：**我们设计代码语言的目的，正是为了使其成为表达计算机执行动作的最佳方式。** 如果 JSON 片段是更好的表达方式，那么 JSON 将成为顶级编程语言，而编程将变成人间地狱。

下图来自《可执行代码动作能更好地引导 LLM Agent》，展示了用代码编写动作的一些优势：

![](/images/posts/a26b42d73f6d.png)

用代码而非 JSON 类片段编写动作提供了更好的：

- **组合性**：你能像定义 Python 函数一样，将 JSON 动作嵌套在一起，或定义一组 JSON 动作供以后复用吗？
- **对象管理**：如何存储像 `generate_image` 这样的动作的输出（以 JSON 格式）？
- **通用性**：代码的设计目标就是简单地表达任何你能让计算机做的事情。
- **在 LLM 训练数据中的表示**：大量高质量的代码动作已经包含在 LLM 的训练数据中，这意味着它们已经为此进行了训练！

## 介绍 smolagents：让 Agent 变得简单 🥳

我们构建 `smolagents` 的目标如下：

✨ **简洁性**：Agent 的逻辑仅包含约数千行代码（参见[此文件]）。我们将抽象层保持在原始代码之上的最小形式！

🧑‍💻 **对代码 Agent 的一流支持**，即用代码编写动作的 Agent（与"用于编写代码的 Agent"相对）。为了确保安全，我们支持通过 E2B 在沙盒环境中执行。

- 除了 `CodeAgent` 类之外，我们仍然支持标准的 `ToolCallingAgent`，它将动作编写为 JSON/文本块。

🤗 **Hub 集成**：你可以从 Hub 共享和加载工具，未来还会有更多功能！

🌐 **支持任何 LLM**：它支持加载在 Hub 上托管的模型（使用其 `transformers` 版本或通过我们的推理 API），也支持通过我们的 LiteLLM 集成使用 OpenAI、Anthropic 等公司的模型。

`smolagents` 是 `transformers.agents` 的继任者，并将在未来 `transformers.agents` 被弃用时取代它。

### 构建一个 Agent

要构建一个 Agent，你至少需要两个元素：

- **工具**：Agent 可以访问的工具列表
- **模型**：作为 Agent 引擎的 LLM。

对于**模型**，你可以使用任何 LLM，既可以使用我们的 `HfApiModel` 类（利用 Hugging Face 的免费推理 API，如上文猎豹示例所示）来使用开源模型，也可以使用 `LiteLLMModel` 来利用 `litellm` 并从 100 多个不同的云端 LLM 中进行选择。

对于**工具**，你只需创建一个带有输入和输出类型提示以及输入描述文档字符串的函数，并使用 `@tool` 装饰器将其转换为工具。

以下是如何创建一个获取 Google Maps 旅行时间的自定义工具，以及如何在旅行规划 Agent 中使用它：

```python
from typing import Optional
from smolagents import CodeAgent, HfApiModel, tool

```

@tool
def get_travel_duration(start_location: str, destination_location: str, transportation_mode: Optional[str] = None) -> str:
    """获取两地之间的旅行时间。

    参数：
        start_location: 出发地点
        destination_location: 到达地点
        transportation_mode: 交通方式，可选值为 'driving'（驾车）、'walking'（步行）、'bicycling'（骑行）或 'transit'（公共交通）。默认为 'driving'。
    """
    import os   
    import googlemaps
    from datetime import datetime

    gmaps = googlemaps.Client(os.getenv("GMAPS_API_KEY"))

    if transportation_mode is None:
        transportation_mode = "driving"
    try:
        directions_result = gmaps.directions(
            start_location,
            destination_location,
            mode=transportation_mode,
            departure_time=datetime(2025, 6, 6, 11, 0), 
        )
        if len(directions_result) == 0:
            return "未找到符合所需交通方式的路线。"
        return directions_result[0]["legs"][0]["duration"]["text"]
    except Exception as e:
        print(e)
        return e

agent = CodeAgent(tools=[get_travel_duration], model=HfApiModel(), additional_authorized_imports=["datetime"])

agent.run("你能为我推荐一个巴黎周边的一日游行程吗？包含几个地点和时间安排，可以在市内或市郊，但需在一天内完成。我只有一辆租来的自行车作为交通工具。")

```

经过几步收集旅行时间和运行计算后，Agent 返回了以下最终方案：

```
巴黎一日骑行行程：
1. 上午 9:00 从埃菲尔铁塔出发。
2. 在埃菲尔铁塔观光至上午 10:30。
3. 上午 10:46 前往巴黎圣母院。
4. 在巴黎圣母院观光至中午 12:16。
5. 中午 12:41 前往蒙马特。
6. 在蒙马特观光至下午 2:11。
7. 下午 2:33 前往卢森堡公园。
8. 在卢森堡公园观光至下午 4:03。
9. 下午 4:12 前往卢浮宫。
10. 在卢浮宫观光至下午 5:42。
11. 午餐休息至下午 6:12。
12. 计划结束时间：下午 6:12。

```

构建工具后，将其分享到 Hub 只需：

```python
get_travel_duration.push_to_hub("{your_username}/get-travel-duration-tool")

```

你可以在该空间下查看结果。
你可以在空间的 `tool.py` 文件中查看工具的逻辑。如你所见，该工具实际上被导出为一个继承自 `class Tool` 的类，这是我们所有工具的基础结构。
