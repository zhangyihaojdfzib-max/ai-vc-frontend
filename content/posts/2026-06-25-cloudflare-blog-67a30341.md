# 我们如何为 Cloudflare Workflows 构建 Saga 回滚

2026-06-25

- Vaishnav Kavitha
- Mia Malden
- AndrÃ© Venceslau

![](/images/posts/d09db4f92396.png)

Cloudflare Workflows 允许你构建持久化、多步骤的应用程序，内置重试机制和跨长时间运行进程的状态持久化。当 Workflow 执行时，每个步骤可以调用外部系统、重试失败操作，并在重启后保持状态。但如果某个步骤失败，它可能会使之前已完成步骤的工作处于不一致或不完整的状态。

今天我们为 Workflows 推出了 saga 回滚功能，允许你在步骤内部声明回滚逻辑，以应对失败情况。

例如，考虑一个在两个不同银行账户之间转账的工作流：

1. 从银行 A 的账户扣款
2. 向银行 B 的账户入账
3. 向两个账户所有者发送邮件确认

从银行 A 的账户扣款

向银行 B 的账户入账

向两个账户所有者发送邮件确认

如果步骤 2（向银行 B 的账户入账）失败会发生什么？一旦步骤 1 在银行 A 扣款成功，该交易即被提交，资金已离开其系统。作为交易的编排者，你不能简单地在银行 A 的系统中"撤销"该操作。相反，必须通过一个新的操作将资金退回银行 A 的账户，该操作在语义上逆转了第一个操作。

这种操作与其补偿逻辑的配对被称为 saga 模式。

在此之前，开发者必须在步骤的直接定义之外，自行实现补偿逻辑来追踪哪些操作成功、哪些失败，以及失败时应采取什么行动。现在，你可以在每个 `step.do()` 中将补偿逻辑作为参数定义在步骤内部，同时保持工作流在回滚时的持久性。

```TypeScript
// 追踪已完成的操作，以便知道要撤销什么
let debitA;
let creditB;
try {
  debitA = await step.do("debit-bank-a", () => bankA.debit(from, amount));
  creditB = await step.do("credit-bank-b", () => bankB.credit(to, amount));
  await step.do("notify", () => notifyBoth(from, to, amount));
} catch (error) {
  // 按逆序展开。每个撤销操作都是独立的持久化步骤，
  // 必须是幂等的，并且如果某个失败要继续执行。
  if (creditB) {
    try {
      await step.do("reverse-credit-b", () => bankB.debit(to, amount, creditB.id));
    } catch (e) {
      await alertOnCall("reverse-credit-b failed", e);
    }
  }
  if (debitA) {
    try {
      await step.do("refund-debit-a", () => bankA.credit(from, amount, debitA.id));
    } catch (e) {
      await alertOnCall("refund-debit-a failed", e);
    }
  }
  throw error;
}
```

无回滚

```TypeScript
// 每个步骤自带撤销逻辑。添加一个步骤，
// 就在这里添加它的回滚。无需不断增长的 catch
// 块，无需手动排序，无需重放逻辑。
await step.do("debit-bank-a", () => bankA.debit(from, amount), {
  rollback: async ({ output }) => bankA.credit(from, amount, output.id),
});
await step.do("credit-bank-b", () => bankB.credit(to, amount), {
  rollback: async ({ output }) => bankB.debit(to, amount, output.id),
});
await step.do("notify", () => notifyBoth(from, to, amount));
```

带回滚

## 试试看

要使用回滚，只需将一个包含 `rollback` 函数的选项对象作为最后一个参数传递给 `step.do()`。

```TypeScript
const debit = await step.do(
  "debit-account-a",
  async () => {
    return await bankA.debit({
      accountId: fromAccountId,
      amount,
      idempotencyKey: `${transferId}:debit-account-a`,
    });
  },
  {
    rollback: async () => {
      await bankA.credit({
        accountId: fromAccountId,
        amount,
        idempotencyKey: `${transferId}:rollback-debit-account-a`,
      });
    },
  }
);

// 幂等键使得正向操作和回滚操作都可以安全重试，而不会重复转账

const credit = await step.do(
  "credit-account-b",
  async () => {
    return await bankB.credit({
      accountId: toAccountId,
      amount,
      idempotencyKey: `${transferId}:credit-account-b`,
    });
  },
  {
    rollback: async ({ output }) => {
      if (output === undefined) {
        return;
      }

      await bankB.debit({
        accountId: toAccountId,
        amount,
        idempotencyKey: `${transferId}:rollback-credit-account-b`,
      });
    },
  }
);


// 如果在这里失败，我们可能需要撤销所有之前的支付。用户不应该为了撤销两笔小额支付而将代码包裹在复杂的 try-catch 逻辑中（见下文）

await step.do("send-confirmation", async () => {
  await sendTransferConfirmation({ ... });
});
```

回滚函数应该是幂等的，就像常规的 Workflow 步骤一样。如果你要退款，请使用支付提供商的幂等键。如果你要释放库存，请确保释放操作可以安全地多次调用。

如果任何步骤失败，回滚处理器将按步骤启动的逆序执行。听起来很简单：当某些操作失败时执行撤销步骤。实际上，有一些细节使得 API 和执行模型变得重要。

1. **失败的步骤可能仍然需要回滚。** 如果失败的 `step.do()` 注册了回滚处理器，它仍然可以参与回滚。

如果用户代码捕获了错误并且 Workflow 继续执行，则回滚不会启动；但如果步骤错误被捕获而 Workflow 后来因其他原因失败，则之前注册的处理器的回滚仍然可以运行，并按步骤启动的逆序执行。

为什么？步骤可能在失败前已与外部系统部分交互。例如，支付提供商可能已捕获一笔费用，但步骤在将 `chargeId` 返回给 Workflows 之前就失败了。这就是为什么回滚处理器会接收 `output`，但必须处理 `output === undefined` 的情况。

2. **回滚仅在 Workflow 失败时启动。** 添加回滚处理器并不意味着每个步骤错误都会触发回滚。如果用户代码捕获错误并继续执行，Workflow 会继续运行。回滚在 Workflow 即将最终失败时启动。

当回滚启动时，Workflows 会找到符合条件的 `step.do()` 调用，运行它们的回滚处理器，然后记录最终的 Workflow 失败。

3. **顺序必须是可预测的。** 对于顺序 Workflow，回滚顺序看起来很明显：

1. 预留库存。
2. 扣款。
3. 创建发货单。
4. 如果发货失败，退款并释放库存。

预留库存。

扣款。

创建发货单。

如果发货失败，退款并释放库存。

并行步骤使这一点更加微妙。完成顺序可能与启动顺序不同，因此 Workflows 使用步骤启动的逆序而不是完成顺序的逆序。

实际规则是：

1. 任何已启动或已完成且带有回滚处理器的步骤都符合条件。
2. 失败的 `step.do()` 如果注册了回滚处理器，也符合条件。
3. 处理器按步骤启动的逆序运行，而不是完成顺序。

任何已启动或已完成且带有回滚处理器的步骤都符合条件。

失败的 `step.do()` 如果注册了回滚处理器，也符合条件。

处理器按步骤启动的逆序运行，而不是完成顺序。

## 我们如何设计 API

一旦我们明确了预期的行为，就需要将这个新模式添加到 Workflows API 中。回滚功能在最终确定为 `rollback` 选项之前经历了多次迭代。

### 为什么不用流畅接口或构建器 API？

第一种方法是流畅形式：`step.do(...).rollback(...)`。它读起来很好。正向操作和补偿逻辑放在一起，调用点看起来像普通的 JavaScript 链式调用。

问题在于 `step.do()` 已经有一个重要的含义：它启动一个持久化步骤并返回一个 Promise 作为步骤输出。在 Workers 中，类似 Promise 的值尤其重要，因为 Workers RPC 支持 Promise 管道化，这是从 Cap'n Proto 等系统继承而来的模式。

Promise 管道化允许代码在某个未来值完全返回给调用者之前，就对其调用方法。例如：

```JavaScript
const session = api.authenticate(apiKey);
const name = await session.whoami();
```

这里，`session` 还不是真正的 session 对象。它更像是一个即将存在的 session 的句柄。当你调用 `session.whoami()` 时，Workers 可以提前将该调用发送到远端，并说：“一旦身份验证创建了 session，就对其调用 `whoami()`。”

这节省了一次往返。调用者无需等待 `authenticate()` 完全完成，就可以请求 `whoami()`。

我们考虑过一种流畅 API：

```TypeScript
step.do("charge-card", chargeCard).rollback(refundCharge);
```

对于读者来说，这看起来像是“对 `charge-card` 的结果调用 `.rollback()`”。但 rollback 并不是该步骤输出的一部分。它是 `step.do()` 选项的一部分，在步骤开始前注册，以便 Workflows 知道如果后续步骤失败，如何补偿该步骤。

流畅 API 也让步骤时序更难理解。目前，`step.do()` 在被调用时即启动步骤，因此开发者可以启动一个步骤，执行其他工作，然后稍后等待第一个步骤：

```JavaScript
const first = step.do("first", () => serviceA.call());

await step.do("second", () => serviceB.call());

await first;
```

在当前执行模型下，`first` 立即启动，在 `second` 之前。流畅 API 会使这复杂化。Workflows 需要等待并查看 `.rollback()` 是否被附加，然后才能知道完整的步骤定义。这可能会延迟步骤发送到引擎的时间。

在前面的例子中，`first` 可能从 `await first` 处开始，而不是从 `step.do("first", ...)` 处，即在 `second` 已经完成后。

这使得并发 Workflows 更难理解：步骤时序将取决于返回的 `Promise` 何时被消费，而不仅仅是 `step.do()` 在哪里被调用。

我们还考虑过一种构建器风格的 API：

```TypeScript
const charge = await step
	.saga("charge")
	.do(() => chargeCard())
	.rollback(() => refundCharge())
	.run();
```

构建器 API 避免了 `Promise` 的歧义。它也为未来的步骤级选项提供了一个明显的位置，并明确了正向操作和回滚操作属于同一个 saga 步骤。

但它增加了仪式感。每个步骤都需要一个最终的 `.run()`，忘记 `.run()` 很容易且在没有工具的情况下难以发现，并且简单的一步案例开始看起来像配置链。它还引入了一个新的 `step.saga()` 构建器，打破了现有的 `step.<action>` 模式。最重要的是，它让 `step.do()` 感觉像是一个较旧的 API，而不是 Workflows 的主要原语。回滚的目标是扩展 `step.do()`，而不是取代它。

### 回滚作为步骤元数据

```TypeScript
step.do(..., { rollback })
```

最终，我们选择了显式形式，其中回滚是步骤上的元数据。

这样，每个回滚都在正向步骤本身内部定义。每个处理程序接收导致回滚开始的错误、步骤上下文以及输出，输出要么是正向步骤返回的持久化值（可能为 `undefined`），要么如果步骤在持久化值之前失败则为 `undefined`。

回滚会触发生命周期事件，因此你可以判断补偿是否已开始、哪个回滚处理程序失败以及回滚是否成功完成。

至关重要的是，原始 Workflow 失败仍然保持独立：回滚是 Workflows 在失败后执行的操作，而不是 Workflow 失败的原因。

正如你可以通过 `WorkflowStepConfig` 在 `step` 配置中定义自定义重试和超时行为一样，你也可以在 `rollbackConfig` 中添加回滚特定的值。

```TypeScript
{
  rollback: async ({ output }) => {
    await bankA.credit({ accountId: fromAccountId, amount, transferId: `${transferId}-reversal` });
  },
  rollbackConfig: {
    retries: { limit: 10, delay: '30 seconds', backoff: 'exponential' },
    timeout: '2 minutes',
  },
}
```

这符合我们想要的生命周期事件心智模型。一个 `step.do()` 已经描述了一个持久化的工作单元，Workflows 会记录、重试并在日志中显示。回滚是该工作单元的另一个生命周期行为。它应该随步骤定义一起存在，而不是存在于单独的包装器或构建器中。

- 步骤仍然在 `step.do()` 正常启动时启动。
- 返回的 promise 仍然代表步骤输出。
- 并发 Workflow 代码保持相同的执行模型。
- 回滚的重试和超时选项位于回滚处理程序旁边。
- 现有的 `step.do()` 调用保持与今天完全相同的运行方式。

步骤仍然在 `step.do()` 正常启动时启动。

返回的 promise 仍然代表步骤输出。

并发 Workflow 代码保持相同的执行模型。

回滚的重试和超时选项位于回滚处理程序旁边。

现有的 `step.do()` 调用保持与今天完全相同的运行方式。

这种形式比流畅 API 稍微更显式，但这种显式性是有用的。操作及其补偿仍然在一个地方，并且 API 没有引入新的步骤构建器或新的 promise 类型。已经理解 `step.do()` 的开发者只需要学习一个额外的 `options` 对象。

这虽然不那么神奇，但更容易采用，也更容易理解。

## 底层工作原理

回滚感觉像是一个小的 API 添加，但它改变了 Workflows 需要为每个步骤记录的内容。

一个常规的 `step.do()` 已经有一个持久化记录。Workflows 记录步骤是否已启动、是否完成、返回了什么，以及如果 Workflow 稍后恢复，是否应跳过而不是重复执行。

回滚在该记录中增加了一项：步骤是否注册了补偿逻辑。

这意味着如果 Workflow 失败，Workflows 需要将两条信息整合在一起。

第一条是**持久化步骤历史**。Workflow 引擎存储数据以了解运行了什么、完成了什么、保存了什么输出以及是否注册了回滚。

第二条是**回滚处理程序**本身，即编写用于补偿该步骤的函数。Workflows 不会将该函数的文本作为数据保存。相反，它在 Workflow 运行时保留一个对该处理程序的可调用引用。

在 Workers RPC 中，这种可调用引用被称为 **stub**。Stub 允许系统的一部分调用在别处运行的代码。Stub 也有生命周期，当调用或执行上下文结束时，它们可以被释放。如果你需要在那个时间点之后保留一个 stub，Workers RPC 提供了一个 `dup()` 方法，该方法会创建另一个指向同一目标的句柄。

对于回滚，该模型很有用。持久化步骤历史记录了需要补偿的内容。回滚 stub 为 Workflows 提供了一种调用补偿代码的方式。并且由于回滚处理程序可能需要比注册它们的即时 `step.do()` 调用存活更久，Workflows 为回滚阶段保留了自己的可调用引用。

在常见情况下，当 Workflow 在同一引擎生命周期内进入回滚时，Workflows 已经拥有所需的回滚 stub。它可以使用持久化步骤历史来找到符合条件的步骤，然后调用在正向执行期间注册的回滚 stub。

当 Workflows 需要在重启后**恢复**时，情况变得更加微妙。

如果引擎在需要回滚时被驱逐、崩溃或重启，Workflows 仍然拥有持久化步骤历史，但可能不再拥有内存中的回滚 stub。为了恢复，Workflows 使用**重放**：一种恢复模式，它可以重新运行 Workflow 代码，而无需重新执行已完成的正向步骤主体。

当重放到达一个已完成的 `step.do()` 时，Workflows 读取持久化的结果，而不是再次运行步骤主体。对于回滚恢复，Workflows 只需要为那些附加了回滚且符合回滚条件的步骤重建处理程序。当遇到这些 `step.do()` 调用时，它们的回滚选项可以再次注册可调用的 stub。

这让 Workflows 能够恢复所需的回滚处理程序，而无需复制原始的外部副作用。

在具备这些条件后，无论处理程序仍在内存中可用，还是需要在恢复过程中重建，回滚都能正常工作。

当工作流即将失败时，Workflows 不会要求您的应用程序重建已发生的事件。它已经拥有步骤历史记录。它可以查看持久化的记录并回答以下重要问题：

- 哪些步骤已启动？
- 哪些步骤已完成？
- 哪些失败的步骤可能仍需清理？
- 哪些步骤注册了回滚处理程序？
- 每个回滚处理程序应接收什么输出？
- 补偿操作应按什么顺序执行？

哪些步骤已启动？

哪些步骤已完成？

哪些失败的步骤可能仍需清理？

哪些步骤注册了回滚处理程序？

每个回滚处理程序应接收什么输出？

补偿操作应按什么顺序执行？

然后，Workflows 会使用回滚上下文调用每个回滚存根：原始错误、步骤上下文，以及（如果已持久化）步骤输出。

执行顺序的细节很重要。在普通的 JavaScript 中，尤其是使用 `Promise.all()` 时，完成顺序并不总是与启动顺序一致。如果步骤 A 先启动，步骤 B 后启动，步骤 B 可能先完成。对于回滚，Workflows 使用持久化的启动顺序作为稳定的真实来源，然后按相反顺序展开。

回滚处理程序也通过 Workflows 的正常步骤机制运行。这意味着补偿操作具有您期望从 Workflows 获得的相同操作属性：重试、超时、生命周期事件、日志以及最终记录的结果。如果回滚处理程序在配置的重试次数后仍然失败，Workflows 会将回滚结果记录为失败，停止运行剩余的回滚处理程序，工作流实例最终进入 `Errored` 状态。

这是 Saga 回滚与 `catch` 块之间的主要区别。`catch` 块只知道在 JavaScript 执行过程中其确切位置仍在内存中的内容。Workflows 回滚使用持久化的步骤历史记录来确定已发生的事件，在常见情况下调用已有的存根，并在需要时在恢复过程中安全地重建缺失的存根。

这也是为什么 API 将回滚放在 `step.do()` 本身上的原因。回滚不是一个独立的全局错误处理程序——它是附加到 Workflows 已经理解的持久化工作单元上的元数据。
