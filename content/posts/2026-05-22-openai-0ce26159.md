# 我们如何借助Codex在28天内为Android打造Sora

作者：Patrick Hum 和 RJ Marsan，技术团队成员

自2026年4月26日起，Sora产品已不再可用。

11月，我们向全球发布了Sora Android应用，让任何拥有Android设备的人都能将一段简短的提示词转化为生动的视频。发布当天，该应用在Play商店中排名第一。Android用户在最初24小时内生成了超过一百万条视频。

这次发布的背后有一个故事：Sora生产级Android应用的初始版本在28天内构建完成，这要归功于同样可供任何团队或开发者使用的Agent（智能体）：Codex。

从2025年10月8日到11月5日，一个精简的工程团队与Codex协作，消耗了约50亿Token，将Sora for Android从原型带到了全球发布。尽管规模庞大，该应用的崩溃率仅为0.1%，并且拥有我们引以为傲的架构。如果你想知道我们是否使用了秘密模型，我们使用的是GPT‑5.1‑Codex模型的早期版本——任何开发者或企业今天都可以通过CLI、IDE扩展或Web应用使用相同的版本。

提示词：花样滑冰运动员在头上顶着一只猫完成三周跳

#### 拥抱布鲁克斯法则：保持敏捷以快速行动

当Sora在iOS上发布时，使用量激增。人们立即开始生成源源不断的视频。相比之下，在Android上，我们只有一个小的内部原型和Google Play上不断增长的预注册用户数量。

面对高风险、时间紧迫的发布，常见的反应是投入更多资源并增加流程。这种规模和质量的正式应用通常需要许多工程师花费数月时间，并因协调而减慢速度。

美国计算机架构师弗雷德·布鲁克斯曾著名地警告说：“向一个已经延迟的软件项目增加人手只会让它更延迟。”换句话说，当试图快速交付一个复杂项目时，增加更多工程师往往会因增加沟通开销、任务碎片化和集成成本而降低效率。我们没有忽视这一见解，而是深入贯彻了它；我们组建了一个由四名工程师组成的强大团队——所有人都配备了Codex，以大幅提升每位工程师的影响力。

通过这种方式工作，我们在18天内向员工交付了Sora for Android的内部构建版本，并在10天后公开发布。我们在Android工程实践上保持了高标准，投资于可维护性，并使应用达到了我们期望传统项目所具有的相同可靠性标准。（如今，我们仍然广泛使用Codex来演进应用并为其带来新功能。）

#### 入职一位新的高级工程师

要理解我们如何与Codex协作，了解它的优势所在以及需要指导的地方会有所帮助。将其视为一位新入职的高级工程师是一个不错的方法。Codex的能力意味着我们可以花更多时间指导和审查代码，而不是自己编写代码。

Codex需要指导的地方

1. Codex尚不擅长推断未被明确告知的内容（例如，你偏好的架构模式、产品策略、真实用户行为以及内部规范或捷径）。
2. 同样，Codex无法实际看到应用的运行：它无法在设备上打开Sora，注意到滚动感觉不对，或察觉到某个流程令人困惑。只有我们的团队能够处理这些体验性任务。
3. 每个实例都需要入职引导。分享带有清晰目标、约束以及“我们做事方式”指导的上下文，对于让Codex良好执行至关重要。
4. 同理，Codex在深层架构判断上存在困难：如果放任不管，它可能会在我们本意是扩展现有视图模型或本应属于仓库的逻辑推入UI层时，引入一个额外的视图模型。它的本能是让东西跑起来，而不是优先考虑长期的整洁性。

我们发现让Codex在整个代码库中创建并维护大量的AGENT.md文件非常有用。这使得在不同会话中应用相同的指导和最佳实践变得容易。例如，为了确保Codex按照我们的风格指南编写代码，我们在顶层的AGENTS.md中添加了以下内容：

#### 纯文本

```
1## 格式化和静态检查2- **在提交之前**始终运行 `./gradlew detektFix`（或针对受影响的模块）。如果存在格式或detekt问题，CI将失败。
```

Codex擅长的地方

1. 快速阅读和理解大型代码库：Codex基本上了解所有主流编程语言，这使得无需复杂的抽象就能更容易地在多个平台上利用相同的概念。
2. 测试覆盖率：Codex（独特地）热衷于编写单元测试以覆盖各种情况。并非每个测试都很深入，但拥有广泛的覆盖率有助于防止回归。
3. 应用反馈：类似地，Codex善于对反馈做出反应。当CI失败时，我们可以将日志输出粘贴到提示词中，并要求Codex提出修复方案。
4. 大规模并行、可丢弃的执行：大多数人不会挑战他们实际上可以同时运行的会话数量上限。并行测试多个想法并将代码视为可丢弃的是非常可行的。
5. 提供新视角：在设计讨论中，我们将Codex用作生成式工具，以探索潜在的失败点并发现解决问题的新方法。例如，在我们设计视频播放器内存优化时，Codex筛选了多个SDK，提出了我们本没有时间解析的方法。Codex研究的见解在最小化最终应用的内存占用方面被证明是无价的。
6. 实现更高杠杆的工作：实际上，我们最终花在审查和指导代码上的时间比我们自己编写代码的时间更多。也就是说，Codex在代码审查方面也非常出色，经常在代码合并前捕获错误，提高了可靠性。

一旦我们认识到这些特性，我们的工作模式就变得更加直接。我们依靠Codex在理解良好的模式和边界清晰的范围内承担大量繁重工作，而我们的团队则专注于架构、用户体验、系统性变更和最终质量。

#### 亲手打好基础

即使是最好的新高级员工，也没有合适的视角来立即做出长期的权衡。为了充分利用Codex并确保其工作稳健且可维护，关键是我们自己监督应用的系统设计和关键权衡。这些包括塑造应用的架构、模块化、依赖注入和导航；我们还实现了身份验证和基础网络流程。

在这个基础上，我们端到端地编写了几个代表性功能。我们使用了希望整个代码库遵循的规则，并在过程中记录了项目范围的模式。通过将Codex指向代表性功能，它能够在我们标准内更独立地工作。对于一个我们估计85%由Codex编写的项目，精心规划的基础避免了代价高昂的返工和重构。这是我们做出的最重要的决定之一。

这个想法不是尽可能快地做出“能用的东西”，而是做出“理解我们想要如何工作的东西”。编写代码有许多“正确”的方式。我们不需要确切告诉Codex做什么；我们需要向Codex展示在我们团队中什么是“正确的”。一旦我们确立了起点和我们喜欢的构建方式，Codex就准备好开始工作了。

为了看看会发生什么，我们确实尝试了提示词：“基于iOS代码构建Sora Android应用。开始。”但很快就放弃了这条路。虽然Codex创建的东西在技术上可行，但产品体验欠佳。而且，在没有清晰理解端点、数据和用户流程的情况下，Codex的一次性代码是不可靠的（即使不使用Agent，合并数千行代码也是有风险的。）

我们曾假设 Codex 能在精心编写的示例沙盒中表现出色；事实证明我们是对的。让 Codex 在几乎没有上下文的情况下“构建这个设置界面”并不可靠。而让 Codex“使用你刚才看到的另一个界面的相同架构和模式来构建这个设置界面”则效果要好得多。人类做出结构决策并设定不变项；Codex 随后在该结构内填充大量代码。

#### 编码前用 Codex 进行规划

我们最大化 Codex 潜力的下一步，是弄清楚如何让 Codex 能够长时间（最近超过 24 小时）无人监督地工作。

在使用 Codex 的早期，我们直接使用这样的提示词：“这是功能。这是些文件。请构建它。”这有时能奏效，但产生的代码大多在技术上能编译，却偏离了我们的架构和目标。

于是我们改变了工作流程。对于任何非微小的改动，我们首先让 Codex 帮助我们理解系统和代码的工作原理。例如，我们会让它读取一组相关文件，并总结该功能如何工作；比如数据如何从 API 流经仓库层、视图模型，最终进入 UI。然后我们会纠正或完善它的理解。（例如，我们会指出某个特定的抽象实际上属于不同的层，或者某个类仅用于离线模式，不应被扩展。）

类似于你如何与一位能力很强的新队友合作，我们与 Codex 一起制定一个可靠的实施计划。该计划通常看起来像一份微型设计文档，指明哪些文件需要更改、应引入哪些新状态、以及逻辑应如何流转。只有在那之后，我们才让 Codex 开始逐步应用该计划。一个有用的小技巧是：对于非常长的任务，当我们触及上下文窗口的限制时，我们会让 Codex 将其计划保存到一个文件中，这样我们就可以跨实例应用相同的方向。

这个额外的规划循环被证明是值得花时间的。它使我们能够让 Codex 长时间“无人监督”地运行，因为我们知道它的计划。它使代码审查变得更容易，因为我们可以对照计划检查实现，而不是在没有上下文的情况下阅读差异。而当出现问题时，我们可以先调试计划，再调试代码。

这种动态感觉类似于一份好的设计文档如何让技术负责人对项目充满信心。我们不仅仅是在生成代码：我们是在产出支持共享路线图的代码。

#### 分布式工程

在项目高峰期，我们经常并行运行多个 Codex 会话。一个在处理播放功能，另一个在处理搜索，还有一个在处理错误处理，有时还有一个在处理测试或重构。这感觉不像在使用工具，更像是在管理一个团队。

每个会话都会定期向我们汇报进度。一个可能会说：“我已经完成了这个模块的规划；这是我的提议，”而另一个则会为一个新功能提供一个大的差异对比。每个都需要关注、反馈和审查。这不可思议地类似于担任技术负责人，带领几位新工程师，大家都在取得进展，都需要指导。

结果是形成了一种协作流程。Codex 原始的编码能力使我们从大量的手动输入中解放出来。我们有更多时间思考架构、仔细阅读拉取请求以及测试应用。

与此同时，这种额外的速度意味着我们的审查队列中总是有东西在等待。Codex 不会被上下文切换所阻塞，但我们会的。我们的开发瓶颈从编写代码转移到了做决策、提供反馈和整合更改上。

这就是布鲁克斯的见解以新方式体现的地方。你不能简单地增加 Codex 会话并期望获得线性加速，就像你不能不断地向项目增加工程师并期望工期线性缩短一样。每增加一双“手”，即使是虚拟的，都会增加协调开销。我们变成了乐团的指挥，而不仅仅是更快的独奏者。

#### Codex 作为跨平台超能力

我们项目的起点有一个巨大的垫脚石：Sora 已经在 iOS 上发布了。我们经常让 Codex 查看 iOS 和后端代码库，以帮助它理解关键需求和约束。在整个项目中，我们开玩笑说我们重新发明了跨平台框架的概念。忘掉 React Native 或 Flutter；跨平台的未来就是 Codex。

这句玩笑背后有两个原则：

1. 逻辑是可移植的。无论代码是用 Swift 还是 Kotlin 编写，底层的应用逻辑——数据模型、网络调用、验证规则、业务逻辑——都是相同的。Codex 非常擅长阅读 Swift 实现并生成保留语义的等效 Kotlin 代码。
2. 具体示例提供了强大的上下文。一个全新的 Codex 会话，如果能看到“这是它在 iOS 上的确切工作方式”和“这是 Android 架构”，其效果远胜于仅凭自然语言描述工作的会话。

运用这些原则，我们将 iOS、后端和 Android 仓库放在同一个环境中。我们给 Codex 的提示词如下：

“阅读 iOS 代码中的这些模型和端点，然后提出一个计划，使用我们现有的 API 客户端和模型类在 Android 上实现等效行为。”

一个微小但有用的技巧是在 `~/.codex/AGENTS.md` 中详细说明本地仓库的位置及其内容。这使得 Codex 更容易发现和导航相关代码。

我们实际上是通过翻译而非共享抽象来进行跨平台开发。由于 Codex 处理了大部分翻译工作，我们避免了双倍的实现成本。

更广泛的教训是，对于 Codex 来说，上下文就是一切。当 Codex 理解了该功能在 iOS 上如何工作，同时理解了我们的 Android 应用是如何构建的，它就能发挥最佳效果。当 Codex 缺乏这种上下文时，它并不是“拒绝合作”；它是在猜测。我们越是把它当作新队友，并投入精力给予它正确的输入，它的表现就越好。

#### 未来的软件工程，今日实现

到我们为期四周的冲刺结束时，使用 Codex 不再感觉像实验，而成了我们的默认开发循环。我们用他来理解现有代码、规划更改和实现功能。我们审查它的输出，就像审查队友的输出一样。这就是我们交付软件的方式。

很明显，AI 辅助开发并没有降低对严谨性的要求；反而提高了要求。尽管 Codex 能力很强，但它的目标是从 A 点立刻到达 B 点。这就是为什么没有人类，AI 辅助编码就无法工作。软件工程师能够理解并应用系统的现实世界约束、构建软件的最佳架构方式，以及如何着眼于未来的开发和产品计划进行构建。未来软件工程师的超能力将是深刻的系统理解力，以及长期与 AI 协作的能力。

软件工程中最有趣的部分是构建引人注目的产品、设计可扩展的系统、编写复杂的算法，以及用数据、模式和代码进行实验。然而，过去和现在软件工程的现实往往更偏向于平凡：居中按钮、连接端点、编写样板代码。现在，Codex 使得专注于软件工程中最有意义的部分以及我们热爱这门手艺的原因成为可能。

一旦 Codex 被设置在一个上下文丰富的环境中，让它理解你的目标和你喜欢的构建方式，任何团队都可以倍增其能力。我们的发布回顾并非放之四海而皆准的配方，我们也没有声称已经解决了 AI 辅助开发的问题。但我们希望我们的经验能让您更容易找到赋能 Codex 的最佳方式，从而让 Codex 赋能于您。

七个月前，当Codex以研究预览版形式发布时，软件工程的面貌与现在截然不同。通过Sora，我们得以探索工程领域的下一个篇章。随着我们的模型和工具不断改进，AI将成为开发过程中愈发不可或缺的一部分。
