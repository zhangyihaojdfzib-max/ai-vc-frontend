# 使用 Claude Code 插件构建 Spotify Ads API 的自然语言接口

我们构建了一套技能、Agent（智能体）和工具，打包成一个插件，使 AI 智能体工具能够轻松访问 Spotify 广告平台。spotify-ads-api 插件已在 GitHub 和 Anthropic 的 Claude 插件市场上架。立即下载，只需向 Claude 描述需求即可创建你自己的广告活动。

广告 API 功能强大——但要求也高。Spotify Ads API v3 拥有超过 30 种资源类型、嵌套的定向结构以及多步骤实体层级关系，其中广告系列包含广告组，广告组又包含广告。对于只想针对康涅狄格州听众投放音频广告的人来说，意图与执行之间的认知差距相当大。

我们希望缩小这一差距，让人们用自然语言描述需求，由系统完成其余工作。一个简单的自然语言请求就能转化为完整、正确的广告系列结构。

最终成果是一个 Claude Code 插件，它能够将自然语言翻译为 Spotify Ads API 调用，该插件由 Markdown 文件、一个 bash 脚本和两个小型 Python 辅助工具构建而成。它是开源的，专为希望完全控制 LLM（大语言模型）与 Ads API 交互方式的开发者和广告主设计。

本文涵盖架构设计、我们做出的技术选择及其考量，以及我们在基于 LLM（大语言模型）驱动的 Agent（智能体）构建开发者工具方面的经验总结。

## 技术选择的原因

这个插件是有明确技术倾向的。为了快速构建，我们专注于 Claude Code、macOS，采用基于 CLI 优先的方法，并以 OpenAPI 规范为支撑，同时有意暂时避免使用 MCP。

### 为什么选择 Claude Code

Claude Code 的插件系统有一个特性，对这个项目至关重要：插件是用 Markdown 编写的。技能、Agent（智能体）以及为 LLM（大语言模型）提供 API 知识基础的参考文档都是 .md 文件。这个插件中没有编译代码，没有构建步骤，没有包管理器，没有打包工具，也没有转译器。

该插件架构包含四种组件类型，分别对应不同的关注点：

- **技能**定义斜杠命令。每个技能都是一个独立的 Markdown 文件，描述命令的行为、API 端点、请求格式和输出处理。
- **Agent（智能体）**处理自由形式的自然语言。当用户以对话方式描述广告任务时，我们的请求构建 Agent（智能体）会自动触发，并将其分解为 API 调用。
- **钩子**在执行前拦截工具调用。我们使用 PreToolUse 钩子来透明地刷新 OAuth 令牌并注入 HTTP 标头。
- **设置**将每个用户的配置（凭证、广告账户、环境偏好）存储在已加入 gitignore 的本地文件中。

结果是插件的每个组件都是人类可读、可比较差异、可进行版本控制的。当测试中出现新的 API 异常时，修复方法通常是在 Markdown 文件中添加一句话。这种设计使得系统易于定制、扩展和适配，无需维护传统 SDK 或编译集成的开销。

### 为什么选择 CLI + OpenAPI 规范（而非 MCP）

模型上下文协议正在成为连接 LLM（大语言模型）与外部工具的默认标准。它似乎是这里很自然的选择：我们定义一组用于广告活动管理的工具，通过 MCP 服务器暴露它们，让任何兼容 MCP 的客户端都能使用。我们考虑过这种方法，但出于几个原因选择了放弃。

API 表面对于静态工具定义来说过于庞大。Spotify Ads API 有数十个端点，带有嵌套的请求模式。将每个端点定义为带有完整参数模式的 MCP 工具，会产生一个庞大的工具注册表，在每次交互中都会消耗大量上下文窗口空间，无论用户是否需要这些端点。而采用 Claude Code 插件方法，Agent（智能体）只会按需加载给定请求所需的参考文档。

Curl 命令透明且可调试。插件发出的每个 API 调用都是一个 curl 命令，并向用户展示。用户可以确切看到发送的内容，可以复制、修改、独立运行。这种透明性在广告系统中提供了可审计性和用户控制力，因为每个请求都可能影响实际预算。用户可以验证、重现和调整执行过程，而不是依赖不透明的工具行为。

OpenAPI 规范是事实标准。Spotify Ads API 发布了一份全面的 OpenAPI v3 规范（约 8,600 行）。我们无需将其翻译为 MCP 工具模式（并随着 API 演变维护该翻译），而是直接将规范文件包含在插件中。当 Agent（智能体）需要了解某个端点的参数或响应格式时，它会读取该规范。当 API 发生变化时，我们只需更新一个文件。

## 将OpenAPI链接作为导航图谱

我们OpenAPI规范中一个被证明很有价值的现有特性是使用了**OpenAPI链接**。链接是OpenAPI 3.x规范中相对未被充分利用的部分，它定义了操作之间的关系——具体来说，就是一个操作的响应如何被用作另一个操作的输入。

对于阅读API文档的人类开发者来说，这些关系通常是隐性的；你需要通过端点命名惯例和文档描述来拼凑出实体层级结构。但对于LLM（大语言模型）Agent（智能体）来说，以机器可读的格式显式声明这些关系是一个显著的优势。运行时表达式（`$response.body#/id`、`$request.path.ad_account_id`）精确地告诉Agent（智能体）需要从哪个响应中提取哪些字段，并注入到下一个请求中。`operationId`值则让Agent（智能体）能够查找目标端点的完整模式。

这是我们选择将原始OpenAPI规范与精心整理的文档一起发布的原因之一。整理后的文档描述了每个端点的**功能**。规范中的链接则描述了使多步骤工作流成为可能的操作图谱，实际上编码了Agent（智能体）在运行时需要遵循的相同工作流图谱。当Agent（智能体）将"创建一个完整的广告活动"分解为三个连续的API调用时，它遵循的正是OpenAPI链接定义的遍历路径：广告活动 → 广告组 → 广告，每一步都通过`$response.body#/id`向前传递响应ID。

广告组响应还链接到定向查找端点，这就是Agent（智能体）知道在构建地理定向载荷之前需要调用`/targets/geos`的原因。该规范不仅记录了单个端点，还记录了端点之间的工作流。

## OpenAPI规范的作用

该插件附带完整的Spotify广告API v3 OpenAPI规范（`external-v3.yaml`，约8,600行）。Agent（智能体）通过`api-reference`技能查阅此规范，该技能提供了从规范中提取的精选端点文档、模式定义和枚举值。

该规范既作为文档也作为验证工具。当Agent（智能体）不确定字段名称或类型时，它会读取相关的模式定义。当用户询问定向选项时，Agent（智能体）可以查找有效的枚举值。将原始规范与精选文档一起发布的决定反映了一种务实的权衡。`api-reference/references/`中的精选文档涵盖了最常见的操作，并附有清晰的示例。原始规范则涵盖了一切，包括较少使用的端点、边缘情况参数和错误响应。Agent（智能体）首先使用精选文档，在需要时再回退到原始规范。

## 下一步计划？

**观察和优化用户流程。** 当前的Agent（智能体）逻辑基于对相对简单的广告流程的理解。我们计划观察常见模式和"分解"瓶颈，以进一步优化我们基于Markdown的技能。通过识别Agent（智能体）最需要澄清的地方，我们可以优化系统提示词，以预先处理这些边缘情况。

**支持幂等键。** 为了防止在网络重试或Agent（智能体）重新运行时意外创建重复的广告活动或广告，我们计划引入对幂等键的支持。通过为每个意图生成唯一的客户端密钥，我们可以确保Spotify广告API将重试的请求视为单个操作，为自动化执行提供安全网。

**支持其他平台。** 我们从Claude开始，因为它对工具调用有强大的支持，但我们也在探索其他平台，包括Codex和Gemini CLI。我们特别对GPT5.5和Images 2.0的图像生成能力在创意生成方面的应用感到兴奋。

## 总结

这个插件押注于一个特定的论点：复杂API的最佳界面是自然语言，辅以详细的文档，并具有透明的执行过程。用户说出他们想要什么。Agent（智能体）找出API调用。钩子层静默处理认证。每个curl命令都是可见的、可复制的和可调试的。

没有框架，没有运行时，除了`curl`、`jq`和`python3`之外没有其他依赖。事实来源是OpenAPI规范。业务逻辑就是文字描述。

这种方法是否能扩展到最复杂的API集成还是一个悬而未决的问题。但对于Spotify广告API来说，在"我想在美国投放一个音频广告活动"与正确的POST请求序列之间存在巨大差距的情况下，它的效果令人印象深刻。
