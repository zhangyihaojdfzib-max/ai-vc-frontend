Vercel Connect 让你的应用和 Agent（智能体）能够访问 Slack、GitHub、Salesforce 等外部服务以及你自己的自定义 API，而无需在环境中存储长期有效的提供商密钥。你只需注册一次连接器，你的代码便会在运行时按需请求具有作用域且短期有效的 Token。

你可以从控制台或 Vercel CLI 创建连接器：

```
vercel connect create slack --name mybot
```

创建一个 Slack 连接器

然后，你的 Agent（智能体）可以在运行时按需从该连接器请求 Token。SDK 会自动获取并刷新该 Token，因此你无需手动存储或轮换密钥：

```
1import { getToken } from '@vercel/connect';2const token = await getToken('slack/mybot', {3  subject: { type: 'app' },4});
```

在运行时请求一个 Token

## 链接到标题作用域 Token

每个 Token 都针对其面临的任务具有作用域，其粒度取决于提供商。一个 GitHub Token 可以被限制为仅对单个仓库具有只读权限，而不是你的整个组织。一个 Agent（智能体）恰好获得任务所需的访问权限，而绝不会获得对所有内容的广泛常驻访问权限。

```
1const token = await getToken('github/mybot', {2  subject: { type: 'app' },3  authorizationDetails: [{4    type: 'github_app_installation',5    repositories: ['myorg/repo1'],6    permissions: ['contents:read']7  }],8});
```

将 Token 的作用域限定为单个仓库，只读

Token 也可以绑定到特定用户而非应用。然后，它将代表该用户的身份进行操作并继承其权限，因此 Agent（智能体）只能执行该用户被允许执行的操作：

```
vercel connect token slack/mybot --subject user
```

为特定用户请求一个 Token

## 链接到标题隔离与撤销

连接器仅在你附加它的环境中工作，因此你可以为开发、预览和生产环境运行独立的连接器。这意味着如果 Token 在开发环境中泄露，它无法用于生产环境：

```
vercel connect attach slack/mybot --project my-app --environment production
```

将连接器附加到单个环境

由于访问权限是以 Token 形式颁发的，你可以通过一条命令撤销它，无论是你自己的 Token，还是连接器在所有用户和安装中颁发的每一个 Token：

```
vercel connect revoke-tokens slack/mybot --all-tokens
```

撤销连接器颁发的每一个 Token

## 链接到标题触发器

事件也可以从提供商流向你的应用。在连接器上启用触发器后，Vercel Connect 会验证提供商传入的 Webhook 并将其转发到你选择的项目，因此一条新的 Slack 消息可以到达一个对其进行操作的 Agent（智能体）。触发器支持 Slack、GitHub 和 Linear（Beta 版），并且一个连接器可以转发到最多三个项目：

```
vercel connect attach slack/mybot --triggers
```

将已验证的 Slack Webhook 转发到一个项目

## 链接到标题连接器与适配器

从控制台可以获取用于 Slack、GitHub、Linear、Discord、Notion、Salesforce、Figma 和 Snowflake 的专用连接器，以及通用的 OAuth 和 API 密钥连接器。你还可以从 CLI 为其他服务创建连接器，包括像 `mcp.linear.app` 这样的 MCP 服务器。Vercel Connect 还提供了用于 Better Auth、Auth.js、eve、AI SDK 和 MCP 客户端的适配器，因此它可以与你已使用的身份验证库和 Agent（智能体）工具协同工作。
