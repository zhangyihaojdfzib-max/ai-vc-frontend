# PyTorch 性能分析（第一部分）：torch.profiler 入门指南

你无法优化的，就无法分析。

无论你是想从大语言模型（LLM）中挤出更多每秒 Token 数，还是想缩短推理的毫秒级延迟，又或者只是想弄明白为什么你的训练循环跑得比规格表承诺的要慢，最终都绕不开性能分析。

问题在于，性能分析的门槛很高。追踪结果是一堵密密麻麻的彩色矩形墙。事件名称令人望而生畏。大多数教程都假设你已经能读懂它们。所以，即使我们知道应该做性能分析，打开一份追踪结果也常常感觉像是一件最好留到以后（或者留给别人）做的苦差事。这篇文章，以及它所开启的系列，是我们降低这个门槛的尝试。

这是《PyTorch 性能分析》系列的开篇之作，我们将在这个系列中逐步培养阅读性能分析追踪结果的能力，并利用它来驱动优化。计划如下：

1.  **第一部分（本文）**：从最简单的操作开始，即矩阵乘法后接偏置加法，并学习如何解读性能分析器返回的结果。
2.  **第二部分**：扩展到 `nn.Linear` 和一个小型 MLP，利用追踪结果推动优化，并一窥底层的**内核**。
3.  **第三部分**：结合 Transformer 在大语言模型上进行综合实践。

我们从初学者的角度记录整个过程。除了基础的 PyTorch 知识外，无需任何先决条件。请将此视为一次轻松的阅读，其中包含一些"啊哈！"时刻。本文的结构特意采用问题驱动：我们打开一份追踪结果，问"等等，为什么会发生这种情况？"，然后追寻答案，直到豁然开朗。学完本文，你应该能够了解：

-   如何设置 `torch.profiler` 以及它实际返回什么，
-   如何阅读性能分析表格和追踪结果（CPU 通道、GPU 通道，以及两者之间可疑的间隙），
-   从 Python 调用一直到 CUDA 内核的完整事件链，
-   当你加上 `torch.compile` 时，什么会改变（以及更有趣的是，什么**不会**改变）。

在开始之前，先明确两个定义，这将有助于你更好地理解下文：

1.  GPU **内核**是一个在 GPU 的多个线程上并行运行的程序。
2.  CPU **调度**并**启动**这些内核。

你通常不需要自己编写 GPU 内核；当你使用 PyTorch 操作时，它会自动被翻译成一个或多个在 GPU 上完成工作的内核。

带着这两个概念，让我们开始提问。

以下是我们在本文中使用的完整脚本：`01_matmul_add.py`。我们建议你在另一个标签页中打开这个脚本，并逐步浏览代码。我们使用 **NVIDIA A100-SXM4-80GB** GPU 来运行这些脚本。

## 矩阵乘法与加法操作

正如 Sara Hooker 博士精辟指出的那样，就像我们主要由水构成一样，深度神经网络主要由矩阵乘法构成。既然它们如此基础，那么用其他任何操作来开启我们的性能分析之旅都将是一种遗憾。

```py
def fn(x, w, b):
  return torch.add(torch.matmul(x, w), b)
```

矩阵加法与矩阵乘法一起模拟了权重和偏置在神经元中的交互方式。这个加法（双关语）将帮助我们理解它如何为本文后面的编译铺平道路。

为了进行性能分析，我们将使用 `torch.profiler` 模块。涉及的步骤如下：

1.  **准备好要分析的代码**（此处为 `def fn`，它封装了矩阵乘法和矩阵加法）
2.  **注释算法**。虽然这完全是可选的，但我们建议这样做。`record_function` 将我们的函数注释为 `matmul_add`，这将便于在追踪结果中导航（我们稍后会提到）

```py
def step():
  with torch.profiler.record_function("matmul_add"):
    return fn(x, w, b)
```

1.  **用 `torch.profiler.profile` 上下文管理器包裹代码**

```py
  with torch.profiler.profile(
    activities=[
        torch.profiler.ProfilerActivity.CPU,  
        torch.profiler.ProfilerActivity.CUDA, 
    ],
  ) as prof:
    
    for _ in range(5):
      step()
      prof.step()
```

1.  **导出分析结果**

```py

prof.key_averages().table(sort_by="cuda_time_total", row_limit=15)


prof.export_chrome_trace(trace_path)
```

性能分析器导出两种不同的产物：

1.  **性能分析表格**：提供算法的统计摘要。它回答了"什么占用了最多时间"这个问题。这对于找出热点非常有帮助。热点是指占用时间最多的事件，可能是流水线的瓶颈，也可能是被频繁触发的事件。
2.  **性能分析追踪结果**：提供时间维度的执行视图。回答了"操作在何时以及为何发生"，描绘了 CPU 和 GPU 上发生的活动。当我们想要调查启动的内核、启动它们的任何延迟、CPU 和 GPU 活动之间的任何重叠等情况时，这非常有用。

让我们在第一次执行中看看这两者的实际应用。（这里是完整的 `01_matmul_add.py` 脚本）

建议在带有 GPU 的机器上运行此脚本。

```bash
uv run 01_matmul_add.py --size 64
```

如果你（在 GPU 机器上）运行上述脚本，你会发现一个名为 `traces/01_matmul_add` 的文件夹，其中包含两个产物：

```bash
64_bf16_cold_eager.json
64_bf16_cold_eager.txt
```

`.txt` 文件保存了性能分析表格。打开该文件，如图 1 所示，你会看到一个大的表格，第一列由在分析作用域内触发的事件组成。

其他列与该事件在 CPU、GPU 或 `torch.profiler.profile` 的 `activities` 中指定的任何其他设备上花费的时间有关。查看哪些事件占用了最多时间，并尝试凭直觉理解该事件是否确实应该花费那么多时间。查看"# of Calls"列也很重要，它指明了事件被触发的次数。

说到这里，我们也来谈谈"Self CPU/CUDA"与"CPU/CUDA total"。"Self"列仅衡量事件本身内部花费的时间，不包括其子事件。"total"列则包括事件及其所有子事件的总和。因此，如果你查看 `matmul_add` 的"CPU total"，它包含了自身花费的时间加上它触发的子事件的时间。这是一个需要留意的重要细微差别。

如果你查看表格的最后两行，你会注意到分析器告诉我们：

```bash
Self CPU time total: 2.314ms
Self CUDA time total: 23.104us
```

CPU 时间以 **ms** 为单位，而 GPU 时间以 **us** 为单位。换个角度来看，GPU 上花费的时间（内核 `ampere_bf16_s16816gemm...`）不到 CPU 上花费的时间（`matmul_add` 操作）的 1%。GPU 大部分时间处于空闲状态，这是一个明显的危险信号。发生这种情况的原因是，GPU 可以非常快速地计算一个小型矩阵乘法，因此我们的代码大部分时间都花在了准备内核、在 GPU 上启动它们、发送要相乘的数据以及收集结果上。这个概念被称为**开销受限**算法。

摆脱这种状态的最简单方法是使用更大的矩阵乘法。

```bash
uv run 01_matmul_add.py --size 4096 
```

图 2 中的最后两行是：

```bash
Self CPU time total: 4.908ms
Self CUDA time total: 4.495ms
```

两个时间都以 **ms** 为单位，这意味着我们仅仅通过增加矩阵乘法的规模就实现了更多的 GPU 时间。如果你查看图 2，你还会注意到，现在大部分 CUDA 时间被 GPU 内核（`ampere_bf16_s16816gemm_..`）占用，而不是启动它的 CPU 操作（`matmul_add`）。这意味着我们确实能够从开销受限转变为计算受限。

现在，我们进入可视化调度链的环节，它存在于 `.json` 产物中。你可以将它们上传到 **Perfetto UI** 并查看追踪结果，或者你也可以使用 `uvx trace-util traces -b traces` 直接生成 Perfetto 链接。

### 为什么ProfilerStep#2花费了这么长时间？

在图5中，我们注意到ProfileStep#2相比其他步骤花费了更多时间，仔细观察还会在matmul_add注释中看到类似的模式。关键问题在于注释内部，而不是注释本身：

图6中显示的约228微秒是进入record_function("matmul_add")和PyTorch实际调度aten::matmul之间的"死窗口"。这可能是由多种原因造成的，包括工作空间分配、cuBLAS（NVIDIA专有的、GPU加速的基础线性代数运算库）启发式算法或惰性模块加载。我们可以选择忽略它，或者在性能分析前运行更多的预热步骤（这是标准做法）。

在性能分析方面，预热是指在实际进行性能分析之前，将事件运行几次。GPU完成的准备工作（包括上述要点）是一次性工作，我们不想对其进行性能分析。在我们的示例中，有两个预热阶段：一是在进入性能分析器之前实际循环函数，二是在性能分析器内部通过warmup参数实现。在本节中，我们启用了实际迭代以及调度器。

```bash
uv run 01_matmul_add.py --warmup

```

带预热的64x64 Perfetto追踪

在图7中，我们看到每个性能分析步骤花费的时间相似，但这并不意味着我们能够优化一次性开销。我们预热了运行，使得这些开销没有被性能分析到。我们认为，如果不给出解决此问题的提示就草率结束本节，对读者是不公平的，因此这里提供一个链接，用于了解进一步优化启动开销的相关内容。

### 为什么CPU和GPU通道之间存在约2.5毫秒的偏移？

在图8中，我们看到CPU和GPU通道之间存在约2.5毫秒的偏移：这是CPU提交CUDA内核与实际开始执行之间的延迟。人们可能会认为，预热阶段结合调度器的wait和warmup应该能让GPU保持忙碌，从而减小偏移。

为了揭示实际情况，让我们稍微修改一下调度器：

```diff
- schedule = torch.profiler.schedule(wait=1, warmup=1, active=3, repeat=1)
+ schedule = torch.profiler.schedule(wait=0, warmup=0, active=3, repeat=1)

```

图9显示，在任何操作之前，GPU通道中有一个Activity Buffer Request。让我们进一步放大。

放大GPU追踪后，我们注意到ProfileStep#0（其CPU追踪在图中有不可见）的矩阵乘法和加法内核一个接一个地发生，而ProfileStep#1的内核之间有一个间隔。对此最好的解释是，缓冲区发生了溢出，在内核执行期间发出了另一个缓冲区请求（在GPU显存上分配一些内存的请求）。

排除其他可能性的最佳方法是进行更多迭代的性能分析，并查看追踪的其他部分是否出现类似的间隔。为此，我们使用active=20运行。

如图11所示，我们在ProfileStep#1中看到了类似的趋势。这与我们之前的发现一致，我们可以安全地得出结论，这确实是另一个缓冲区请求。

### 事件链

在图12中，我们看到了嵌套的CPU调用。这是一个重要的可视化内容，可以让人理解调度链的真实面貌。

我们从ProfileStep#<id>开始，它封装了性能分析步骤。由于我们对步骤进行了注释，因此可以看到matmul_add行。matmul_add包含两个aten调用，一个用于矩阵乘法，一个用于矩阵加法。

aten::matmul是ATen级别的调度，用户面向的PyTorch matmul调用最终落在此处。aten::mm是二维矩阵-矩阵乘法的后端。

非常有趣的是，如果我们给矩阵添加批次轴，PyTorch会调用aten::bmm（批量矩阵乘法）。让我们绕道看看aten::bmm的实际运行情况。

```diff
- x = torch.randn(args.size, args.size, device=device, dtype=dtype)
- w = torch.randn( args.size, args.size, device=device, dtype=dtype)
- b = torch.randn(args.size, args.size, device=device, dtype=dtype)

+ # 添加批次大小为8
+ x = torch.randn(8, args.size, args.size, device=device, dtype=dtype)
+ w = torch.randn(8, args.size, args.size, device=device, dtype=dtype)
+ b = torch.randn(8, args.size, args.size, device=device, dtype=dtype)

```

在图13中，当给输入添加批次轴后，aten::matmul现在封装了一堆其他必需的CUDA运行时调用以及aten::bmm（而不是aten::mm）。这也暗示了cuBLAS为了调度程序最合适的内核而需要执行的启发式算法。

在本文的其余部分，除非另有说明，我们将使用简单的二维矩阵。

### 为什么matmul有一个额外的CUDA运行时调用？

我们注意到，对于aten::mm，有两个CUDA运行时调用，即cudaOccupancyMaxActiveBlocksPerMultiprocessor（在图14中用方框标出）和cudaLaunchKernel，而对于aten::add，只有cudaLaunchKernel。

cudaOccupancyMaxActiveBlocksPerMultiprocessor是一个规划调用，纯粹在CPU端进行。它询问："给定一个内核函数、一个选定的块大小和一个选定的动态共享内存大小，该内核有多少个块可以同时驻留在一个SM（流式多处理器）上？"

这引出了一个问题，为什么matmul需要规划而add不需要？

要理解这一点，我们必须查看内核的资源占用情况。如果点击GPU内核，就可以检查相应内核的资源占用情况。

在图15中，我们注意到对于矩阵乘法，每线程寄存器和共享内存是动态的（基于矩阵的大小）。cuBLAS提供了数百种内核变体，每种都有启发式驱动的启动路径，需要有关硬件容量的运行时信息。占用率查询是该启发式算法的一部分。从概念上讲，我们可以将GPU加速的矩阵乘法视为在独立的数据块上工作：我们使用多少个数据块以及每个数据块需要多大，取决于矩阵和硬件。现代算法比这复杂得多，但这仍然是一个很好的参考框架。

从图16中，我们看到加法的资源占用显示为32个寄存器和零共享内存。这显然很容易满足。没有什么需要查询的，因为没有硬件资源会限制占用率。该内核在设计上就是资源轻量型的。

你可以将此作为快速诊断工具，在阅读任何追踪记录时使用。扫描CPU通道中的`cudaOccupancyMaxActiveBlocksPerMultiprocessor`。每次出现都标记一个“重量级、自适应启动”的内核，通常是GEMM（通用矩阵乘法）、卷积或类似操作。没有前置占用查询的内核是PyTorch自动启动的逐元素/规约类内核。

### 为什么cudaDeviceSynchronize这么大（约1.78毫秒）？

`cudaDeviceSynchronize`会阻塞CPU，直到该设备上的所有GPU工作完成。性能分析器在活动窗口结束时发出此同步，以刷新事件。没有它，内核计时将缺失。

一个覆盖26微秒实际GPU工作的1.78毫秒同步告诉你，这次运行有98%的时间处于空闲状态。这就是典型的开销受限症状。

### 为什么相同的内核比其他内核花费更多时间？

图18揭示了类似的发现。虽然每个内核完全相同，但它们的计时不同。不同的计算时间可以归咎于多种原因：

- GPU时钟在空闲和升频状态
- GPU发热
- GPU电源管理
- 驱动端维护

只看平均值的读者会得出结论，矩阵乘法大约需要1毫秒（5次平均值=1084微秒）；而查看追踪记录的读者会看到，矩阵乘法大约需要580微秒，除非GPU出现问题。这是两种截然不同的思维模型，只有一种是正确的。

## 让我们看看torch compile的实际运行

使用`torch.compile`一直让我们感到惊叹。你编写正常的即时PyTorch代码，但PyTorch会尝试捕获张量密集区域，将其转换为计算图，进行优化，并运行生成的代码。默认后端通常是`TorchInductor`，整体流程如下：

1. TorchDynamo将Python执行捕获为FX图
2. 当涉及梯度时，AOTAutograd准备前向/反向图
3. Inductor将图降级为优化的CPU或GPU代码。

在本节中，我们讨论编译并查看性能分析器追踪记录。

```bash
uv run 01_matmul_add.py --size 4096 --warmup --compile
```

`args.compile`标志触发以下代码：

```py
def fn(x, w, b):
  return torch.add(torch.matmul(x, w), b)
```

### torch.compile的运行时架构

`TorchDynamo Cache Lookup`是Dynamo检查当前调用是否仍然与编译时相同的输入形状、数据类型、设备和张量元数据匹配的地方。如果任何内容不匹配，Dynamo将重新编译。每次调用都会付出这个代价，即使在编译之后也是如此。

### CUDA启动次数是否减少了一半？

查看图21中的追踪记录，我们非常高兴地注意到每一步只有一个`cudaLaunchKernel`。这个观察结果与我们之前在GPU追踪记录中看到的直接矛盾。每一步仍然有两个内核被启动，即`Memcpy DtoD (Device -> Device)`和GEMM。回到CPU追踪记录，我们注意到我们完全错过了`cudaMemcpyAsync`调度。

`addmm`计算`out = α·A·B + β·C`，cuBLAS的带偏置加法后处理的GEMM写入一个需要已经包含偏置的目标缓冲区。后处理可以被认为是GEMM之后发生的所有操作。在深度学习领域，我们经常遇到GEMM后处理，如激活函数、偏置加法、归一化等等。这就是为什么存在cuBLAS GEMM-with-内核变体的原因。

如果你为`torch.compile`使用不同的模式，你会注意到启动不同的内核变体。你可以自己尝试一下，并在下面的评论中添加你的观察结果！

所以Inductor生成的代码执行以下操作：

- `out = copy(C)` ← 这是设备到设备的内存拷贝（32 MB，大约需要33微秒）
- `out = α·(A·B) + β·out` ← 使用α=β=1的GEMM，将偏置加法融合到写回操作中

结果在数学上仍然是相同的。偏置加法并不是免费的，我们预先支付了一次内存拷贝，加上一个稍微更昂贵的GEMM后处理。

人们可能希望实现的融合，即`x·w + b`（这里`out = α·A·B + β·C`）折叠成一个没有额外内存流量的单个内核，并没有发生。Inductor保留了两次内存访问操作，只是将偏置拷贝重新标记为内存拷贝，将加法重新标记为GEMM后处理。

真正融合的实现会跳过内存拷贝。这就是FlashAttention风格的手写内核所做的，也是Inductor通过Triton代码生成可以做到的，但对于一个`4096×4096`的bf16矩阵乘法，Inductor显然决定“使用cuBLAS，通过后处理设置来实现偏置”是最佳路径。

### CPU开销增加了，而不是减少了

这是比较即时运行和编译运行时最容易忽略的事情：

编译后每一步的CPU开销大约是原来的2倍。这是因为每次调用都会遍历完整的Dynamo > AOTAutograd > Inductor栈，再加上我们本来就有的`aten::addmm`调度。编译管道是为包含数十个操作的机器学习模型构建的，其中每次调用的开销可以被摊销（对于单个操作来说，这是一种负担）。
