# 从终端到提示词：Cloudflare One 的统一数据安全愿景

多年来，Cloudflare One 已发展壮大。它始于保护网络流量，如今已扩展到终端和 SaaS 应用程序——因为工作正是在这些地方进行的。

但随着市场的发展，其核心使命已变得清晰：数据安全就是企业安全。

原因如下。我们实施控制措施并非仅仅为了控制本身。我们这样做是因为下游的后果代价高昂：恶意软件、凭证盗窃、会话劫持，以及最终最重要的事情：敏感数据离开组织。一个看似简单的访问策略，可能是一连串事件的第一个环节，最终导致事件响应、客户影响和声誉损害。

因此，当你退一步看时，大多数安全方案——即使是那些在纸面上看起来不同的方案——都在试图回答相同的问题：

*   敏感数据在哪里？
*   谁可以访问它？
*   存在哪些路径可以让它移动到不该去的地方？

*   传输中的保护（跨越互联网 + SaaS 访问）

请将这些视为一个相互连接的系统：可见性告诉你正在发生什么，控制措施限制数据可以移动的位置，而执行则在内容离开应用程序时弥补最后一公里的差距。这就是从终端到提示词的问题：数据的移动速度比产品边界更快，因此策略需要跟随数据，而不是工具。

### 远程访问而无数据蔓延：基于浏览器的 RDP 剪贴板控制

当你无法假设存在受管理的终端或已安装的客户端时，基于浏览器的 RDP 是一种提供远程访问的实用方式——这在承包商、合作伙伴和偶尔的访问工作流中很常见。Cloudflare One 的基于浏览器的 RDP 为此类访问增加了可见性和策略控制。但是，一旦你在浏览器中提供完整的 RDP 体验，问题就变得简单了：你对数据可以移动的位置（尤其是通过剪贴板）的控制有多精细？

今天，我们增加了一项直接保护数据的设置：基于浏览器的 RDP 剪贴板控制。借助这项新功能，安全和 IT 管理员现在将能够决定其用户是否可以在其本地设备和基于浏览器的 RDP 会话之间复制或粘贴信息。

剪贴板限制是生产力与安全性权衡的一个完美例子。如果用户在他们依赖的工作流中无法复制和粘贴，他们就会绕过控制，无论是通过截图、重新输入数据，还是将工作转移到不受管理的工具中。剪贴板控制让你可以做到精确：在安全的地方允许工作流，在不安全的地方阻止它。

借助基于浏览器的 RDP 中的剪贴板控制，管理员可以启用用户期望的复制/粘贴工作流，同时对方向和上下文实施精细控制。例如，如果用户访问包含敏感客户信息的客户支持门户，你可能允许为了生产力而复制/粘贴到会话中，但阻止从会话中复制/粘贴，以防止数据落到不受管理的终端上。

此功能现已在 Cloudflare One 中可用，并可作为基于浏览器的 RDP 应用程序的访问应用程序策略中的新设置进行配置。

### 无需猜测的可见性：日志中的操作映射

虽然远程访问控制降低了风险，但要很好地调整它们，你还需要了解用户在 SaaS 应用程序内部采取的具体操作。

我们使用一个称为**操作映射**的过程（在最近的一篇博客文章中有详细说明）来提供对这些操作的可见性，并简化客户为 SaaS 服务编写策略的方式。我们的映射过程提取 HTTP 请求的各个元素，并将其解释为单个操作，例如在 ChatGPT 的例子中是“SendPrompt”。我们将执行类似操作的多个操作收集到一个应用程序控制中，例如“Share”或“Upload”。[什么？] 在我们的 HTTP 策略构建器中可见，允许简单的策略编写。

今天，我们进一步推进了这一过程，通过将该映射扩展到日志记录中，来丰富日志并提供对 SaaS 应用程序在组织内使用方式的更大可见性。无需任何额外配置，与我们的**操作映射**匹配的流量的日志事件现在将显示操作和应用程序控制。

在日志详细信息中，你现在将看到应用程序控制组和具体操作（例如，ChatGPT 的 SendPrompt）。这使得调查和策略调整更加迅速。

增加的上下文有助于你理解使用模式，加速取证分析，并发现潜在的风险行为，从而让你在调整策略时减少猜测和对用户的干扰。

可见性是第一步。要保护使用中的数据，尤其是通过剪贴板移动的数据，你还需要在终端上进行执行。

### 更好的终端保护：Cloudflare One Client 中的设备端 DLP

在现代企业中，敏感信息经常通过剪贴板从受管理的应用程序移动到不受管理的环境中。风险不仅在于文件离开组织；也可能是一段专有代码或客户记录被粘贴到未经授权的**大语言模型（LLM）** 或个人工具中。

Cloudflare One 已经通过 Gateway 和 DLP 帮助保护传输中的数据，并通过 CASB 及其 API 集成提供静态时的可见性和控制。现在，我们通过将终端 DLP 执行引入 Cloudflare One Client，将覆盖范围扩展到使用中的数据，首先从剪贴板移动等高信号工作流开始，这样数据保护不会在内容离开浏览器标签页的那一刻就停止。

这意味着从受保护的 SaaS 应用程序复制的敏感数据，在接触到操作系统剪贴板的那一刻，不会立即变成“无策略”内容。借助终端 DLP，团队可以将数据保护扩展到用户的指尖，而无需部署第二个代理或拼凑复杂的集成。

对于已经使用 Cloudflare One 进行**数据保护**的团队，终端 DLP 通过为使用中的数据添加一致性的执行层，完善了该模型。

这就是从终端到提示词的问题：如果敏感数据可以在本地复制，那么它同样可以轻松地粘贴到 AI 助手中。一旦你保护了使用中的数据，下一个问题就变得不可避免——当同样的数据在提示词处被转换时会发生什么？

### 无盲区的 AI 可见性：通过 API CASB 进行 M365 Copilot 扫描

去年，Cloudflare One 和 API CASB 成为**首个**提供与 OpenAI ChatGPT、Anthropic Claude 和 Google Gemini 产品的 API 集成的服务——而我们尚未止步。

从今天开始，使用 Cloudflare One 的 **API 云访问安全代理（CASB）** 的客户——它通过 API 扫描 SaaS 应用程序以查找常见但有风险的安全问题——现在可以分析 **Microsoft 365 Copilot** 活动是否存在数据安全问题，包括与 DLP 检测配置文件匹配的聊天和上传。

Copilot 发现项会附带丰富的上下文信息（文件引用、配置文件匹配和交互元数据）呈现，使团队能够快速分类处理，而无需从原始审计日志开始梳理。

### 下一步：Cloudflare One中的统一数据安全

过去几年，企业安全已扩展到更多层面：SaaS、非托管终端、远程访问模式，以及现在的AI助手。但目标——保护敏感数据——始终未变。本文中的更新反映了一个方向：对传输中、静态、使用中以及提示词中的数据，提供一致的可见性和执行策略。因此，策略跟随数据，而非产品边界。

展望未来，我们的愿景比"数据安全产品中的数据安全功能"更为广阔。随着时间的推移，每个Cloudflare One产品都将更具数据安全意识，拥有更多面向数据的可配置性、可见性、控制措施和防护机制，这些功能将直接内置于团队已在使用的跨Access、Gateway、终端执行和SaaS集成的工作流中。目标很简单：无论您的用户在何处工作，数据流向何方，Cloudflare One都应能解释正在发生的情况并帮助您进行控制。

随着现代边界扩展到应用程序、浏览器、终端和AI提示词，拼凑零散的单点解决方案会变得难以操作且更易被绕过。通过将数据安全直接构建到Cloudflare One中——从访问控制到终端执行，再到AI可见性——并持续统一这些层面，我们正在帮助团队构建一个更清晰、更完整的视图，以了解从终端到提示词的整个过程中的数据风险和数据安全状况。
