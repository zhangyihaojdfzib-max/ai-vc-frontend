我们很高兴分享一项为期一年的研究工作，其目标是让 Python 的类型系统更具表现力和可组合性，使其在理念上更接近 TypeScript 的可编程类型，但专为 Python 的运行时模型精心设计。这项工作的成果是 **PEP 827：类型操作**。

Python 的运行时功能极其强大：类、方法乃至整个 API 都可以通过几行代码动态生成。元编程等概念可以转换类声明，装饰器可以为函数和方法添加额外行为，这些只是众多例子中的几个。

但 Python 的静态类型系统往往无法“跟上”这些动态特性，除非借助类型检查器插件或样板代码。PEP 827 提出了一套标准的、类型层面的构建模块，用于内省现有类型和构建新类型，旨在同时帮助类型检查器和运行时工具。

## 快速体验

一个具体的例子是大家熟悉的 TypeScript 工具类型，比如 `Pick` 和 `Omit`。以下是 `Pick` 在 TypeScript 和 Python 中的并列实现：

```typescript
// TypeScript
type Pick<T, K extends keyof T> = {
    [P in K]: T[P];
};
```

```python
# Python
def Pick[T, K: KeysOf[T]](cls: type[T]) -> type:
    def __init__(self, **kwargs):
        for key in K.__args__:
            setattr(self, key, kwargs[key])
    return type(f"Pick[{T.__name__}, {K}]", (), {"__init__": __init__})
```

我们立刻可以看出，TypeScript 专用的类型语法简短扼要，尽管与语言的其他部分差异很大。而 Python 则依赖于标准的 Python 命令式语法，结合了类型层面的 API。

现在让我们看看 `Omit` 如何实现：

```typescript
// TypeScript
type Omit<T, K extends keyof any> = Pick<T, Exclude<keyof T, K>>;
```

```python
# Python
def Omit[T, K: KeysOf[T]](cls: type[T]) -> type:
    def __init__(self, **kwargs):
        for key in KeysOf[T].__args__:
            if key not in K.__args__:
                setattr(self, key, kwargs[key])
    return type(f"Omit[{T.__name__}, {K}]", (), {"__init__": __init__})
```

有趣的是，Python 版本与 `Pick` 的实现更一致，唯一的区别是反转了条件。而 TypeScript 的组合方式则截然不同，需要进行更深层次的重写。

这说明，核心理念并非“让 Python 看起来像 TypeScript”，而是为 Python 类型系统提供一个可编程的核心，使其符合 Python 的语义，并能在运行时进行内省，这样像 Pydantic 这样的框架也能从中受益。
