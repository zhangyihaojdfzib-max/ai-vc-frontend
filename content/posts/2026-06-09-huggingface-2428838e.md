# 介绍 North Mini Code：Cohere 面向开发者的首款模型

以下为所有合著者

今天，我们发布 North Mini Code，这是一款拥有 300 亿参数的混合专家模型，其中 30 亿参数处于激活状态，具备强大的智能体编码能力，现已根据 Apache 2.0 许可证在 Hugging Face 上提供。

North Mini Code 是 Cohere 新模型系列中的首款模型，专为智能体软件工程任务而设计和训练。

![image1-benchmark-results](/images/posts/d3216573eb19.png)

图 1：North Mini Code 在智能体编码任务和复杂代码生成基准测试中的表现，与类似规模的主流开源模型对比。详见我们的基准测试方法说明。

North Mini Code 针对复杂软件工程工作流、基于终端的智能体任务以及高质量代码生成进行了优化。在 Artificial Analysis 的编码指数中，North Mini Code 获得 33.4 分，优于 Qwen3.5 (35B-A3B)、Gemma 4 (26B-A4B)、Devstral Small 2 (24B Dense)，甚至显著优于规模更大的模型，如 Nemotron 3 Super (120B-A12B)、Mistral Small 4 (119B-A6B) 和 Devstral 2 (123B)。¹ 它在同类规模模型中位列最强开源编码模型之列。

### 在 OpenCode 中试用 North Mini Code

现实世界中的代码智能体依赖于模型质量以及跨智能体框架的鲁棒性。我们使用多种框架而非针对单一框架优化来训练 North Mini Code。这种方法使 North Mini Code 能够为 OpenCode 等编码智能体提供可靠基础。

## 架构

![image2](/images/posts/b7e10f494103.png)

图 2：North Mini Code 是一种混合专家 Transformer 解码器，采用交错滑动窗口自注意力和全自注意力机制。

North Mini Code 是一种仅解码器、基于 Transformer 的稀疏混合专家模型。它使用我们高效的注意力实现，以 3:1 的比例交错使用带 RoPE 的滑动窗口注意力和无位置嵌入的全局注意力 [1]。前馈模块是一个包含 128 个专家的 MoE 模块，每个 Token 激活其中 8 个专家。每个专家模块是一个使用 SwiGLU 激活函数的 FFN 模块。路由器在 top-k 选择之前对 logits 应用 sigmoid 激活函数。我们还在稀疏层之前使用了一个单一稠密层。

## 面向卓越编码能力的后训练

![image3](/images/posts/9d162a8458cd.png)

图 3：后训练流程由两个阶段的监督微调 (SFT) 和一个针对软件工程和终端任务、使用可验证奖励的智能体强化学习 (RLVR) 阶段组成。

我们使用两阶段级联监督微调 (SFT)，随后进行可验证奖励的强化学习 (RLVR)，对 North Mini Code 进行后训练，重点关注智能体编码。我们的第一阶段 SFT 数据侧重于编码能力，这些能力被整合到一个更广泛的数据混合中，以增强鲁棒性和可用性。该数据混合包括跨多个领域的编程、推理和指令遵循，其中代码数据集占可训练 Token 的 70%，智能体工具使用数据占 43%，单轮竞赛或科学编程数据占 27%。在第二阶段 SFT 中，我们使用了一个仅包含智能体和推理驱动样本的 45 亿 Token 数据混合，其中代码数据占可训练 Token 的 61%。该混合包含我们最高质量的数据，涵盖编码和更广泛的智能体任务，其中工具调用和完成结果被验证为可执行且正确。

我们的内部数据管道严重依赖于容器化的智能体编码环境。我们维护这些环境的一个独立子集，用于合成 SFT 数据生成和 RLVR。其中大部分基于来自真实世界仓库的软件工程任务，其余则是来自开源和内部数据集的基于终端的智能体任务。总共，我们在约 5000 个独特仓库中使用了超过 7 万个可验证任务。我们对环境进行了去重处理，排除了来自 SWE-Bench [2] 和 SWE-Bench-Pro [3] 的仓库源，以避免评估期间的数据源泄漏 [4]。

我们在 SFT 的第一和第二阶段分别使用了 64K 和 128K 的上下文长度。这种“从长到更长”的级联方法（类似于 [5,6]）能够对有价值的较短数据进行双向训练，建立稳健的性能基线，随后仅针对高质量的可验证样本进行有针对性的长上下文训练。如果没有多阶段训练，初始训练阶段中的 200 亿非代码 Token 往往会主导后期训练中 15 亿高质量代码 Token，导致性能较差，并且由于不同阶段数据趋势的差异而产生更多的行为冲突。据观察，与仅在截断至 64K 的分布上进行训练相比，在近乎完整的样本长度分布上进行训练会在评估中产生更短的最终轨迹。

我们没有在 SFT 期间针对定量指标优化 North Mini Code，而是采用了一种严格将 SFT 用作 RLVR 预热的方法。数据混合优化了下游阶段的采样多样性和 pass@K（针对高 K 值）。我们使用样本级过滤来移除任何异常情况，例如无效的工具调用、错误的空白字符生成、格式错误的特殊 Token 或幻觉引用。会产生不良 RLVR 行为（例如低熵、无效的结构化生成）的产物或超参数通过消融实验被剔除。最终的 SFT 模型在 SWE-Bench Verified [2] 上达到了 80.2% 的 pass@10，在 Terminal-Bench v2 [7] 上达到了 55.1% 的 pass@10。

### 跨框架的鲁棒性

框架鲁棒性提高了模型在现实软件开发环境中的可用性，在这些环境中，智能体会遇到多样且不可预测的工具环境。这些环境不仅在提示词上有所不同，而且在基本的工具使用模式上也存在差异。例如，SWE-Agent [8] 提供了一个相对丰富的智能体-CLI 接口，包含专门的命令（bash、str_replace_editor 和 submit 工具）以及模板化的观察结果；mini-SWE-agent [9] 将其简化为单一的 bash 工具，仅将 shell 的原始标准输出作为反馈；而 OpenCode [10] 则使用细粒度的单独类型化工具（edit、grep、todo、write 和 task 等），返回结构化的 JSON 响应。

![image4](/images/posts/9c9055080050.png)

图 4：为了支持各种智能体编码框架，North Mini Code 在第二阶段 SFT 期间接触了多种编码框架。

我们通过在第二阶段 SFT 期间引入少量额外的基准测试框架数据（占 SFT 混合数据的 6%，而所选的 SWE-Agent 框架数据占 50%）来解决跨框架泛化问题。具体来说，这种数据混合在使用 OpenCode 框架的评估中带来了 10% 的提升，同时在使用 SWE-Agent 框架的 SWE-Bench Verified 上保持了性能，这表明跨框架迁移可以低成本地实现，而不会降低基准测试性能。值得注意的是，North-Code-Mini 在使用 mini-SWE-Agent 时达到了 61.0% 的 pass@1，这一改进在跨任务、跨框架的设置中自然产生，表明具有重叠工具能力的框架共享足够的表征结构以实现正向迁移。我们还观察到，在混合框架数据上进行训练时，数据冲突极小，这表明不同框架所需的技能通常是互补而非矛盾的。

同样地，官方Terminal-Bench使用其自有的Terminus 2测试框架，其中所有Agent（智能体）-CLI交互均通过纯文本对话轮次（而非原生工具调用）进行通信。为了使我们的模型在Terminus 2上获得初始训练，我们在数据混合中加入了少量（低于20%）纯文本格式的数据，这已被证明足以让模型自然地泛化。有趣的是，我们还发现，在不同的测试框架中引入足够多的变化（类似于数据增强）至关重要，这能迫使模型正确建立指令与行为之间的联系，而不是简单地重复固定模板而不理解其含义——当测试框架彼此相似时，这一点尤其重要。

## 用于编程Agent（智能体）的异步强化学习

编程Agent（智能体）的轨迹展开过程漫长且长度差异极大，最慢的轨迹通常比中位数长一个数量级。同步强化学习循环会使训练器空闲，等待为每一批次生成这些试验，因此我们将采样与学习解耦：一个训练器与一个持续提供轨迹展开的vLLM辅助进程并行运行。策略权重每隔几个学习步骤（K=4）导出到vLLM中，因此采样器在任何时刻最多只是略微偏离策略。然后，在损失层面修正残余的失配问题。

为了在避免任务间数据分布失衡的同时，解除学习进程对最长轨迹展开的等待阻塞，我们使用了窗口化的先进先出（FIFO）队列（训练器↔采样器）[11]：队列头部的一小部分按完成顺序被消耗以处理掉队者，其余部分则保持输入顺序。经验上，这恢复了完成顺序方案的大部分吞吐量，且没有明显损害训练稳定性。

我们使用CISPO [12]进行训练，这是一个带有Token级重要性采样校正的对数似然目标。CISPO与PPO和GRPO的不同之处在于，其重要性权重乘以对数似然而非概率比，并通过更强的正则化增强了RLOO [13]。我们在Token级别而非提示词级别聚合损失，因此梯度信号随轨迹长度缩放，并且较长的Agent（智能体）轨迹（其中包含大部分信用分配信号）不会相对于短轨迹被降低权重。

单次多环境强化学习训练——我们运行了一次跨越两个任务环境的单次多环境在线强化学习训练：基于终端的任务和软件工程任务。每个训练批次包含512个轨迹展开，每个提示词采样8个轨迹展开作为一组。所有轨迹展开共享一个128K Token的全局上下文窗口。为了应对不同的任务复杂度，每个任务被分配了一个不同的Agent（智能体）步骤预算。这些按任务设置的预算基于强化学习验证前进行的pass@k过滤来确定，确保预算根据每个任务分布的难度进行了适当校准。我们观察到，给予模型远大于必要的轮次预算会鼓励其在轨迹展开中产生不必要的冗长和跳跃行为。

对于基于终端的任务，我们为Agent（智能体）配置了一个简单的ReAct测试框架，该框架使用基于Harbor的Tmux会话实现[14]的单一终端使用工具；而对于软件工程任务，我们使用SWE-agent [8]测试框架。两个环境都为Agent（智能体）提供了一个预构建的、编码了环境状态的Docker镜像、一个自然语言用户提示词，以及一组用于验证的单元测试。我们在内部和开源数据集的组合上进行训练，并经过过滤，只保留具有可接受pass@k率的问题，即排除那些被轻易解决和完全无法解决的实例。我们使用基于单元测试验证器得出的二元奖励。此外，模型在生成无效工具调用或无法解析的输出时会获得0奖励，这使得在最初的训练步骤中，幻觉或格式错误的工具调用率急剧下降。

![image5](/images/posts/62e6a059f4dc.png)

图5：多环境强化学习训练运行提升了模型在SWE-Bench Verified和Terminal-Bench v2等基准测试上的性能。左侧展示了整个RLVR（强化学习验证）训练过程中的学习曲线。

在线强化学习带来的更高性能和鲁棒性——RLVR（强化学习验证）训练使最终模型在Terminal-Bench v2上的pass@1性能相比SFT（监督微调）初始模型提升了7.9%（绝对值），在SWE-Bench上提升了3.0%（绝对值）。我们观察到，在两个环境上的联合训练比单独训练每个环境能产生更强的结果，并且对分布外任务的泛化能力也更好。除了正确性得分，我们还观察到Agent（智能体）鲁棒性的显著提升，RLVR（强化学习验证）模型产生了更短的轨迹和更少的无效或失败工具调用。最终模型还表现出更少的重复工具调用循环，能够通过提交解决方案或响应用户来可靠地结束其轨迹。

### 内部人工评估基准

作为现有编程基准的补充，我们还开发了自己的内部基准套件，用于在成对评估中与人工标注员一起衡量模型在分布外问题上的性能。与其他基准设置一致，我们评估了通过Harbor在OpenCode中测试的模型迭代版本。为了理解模型性能，我们在四个不同的功能上进行基准测试：

- 代码解释：要求模型在README文件中或直接向用户解释给定代码仓库的特定技术方面。
- 代码编辑：要求模型基于现有代码库实现一个功能。
- 数据可视化：给定数据样本，要求模型使用特定框架创建某些可视化；不提供额外代码。
- 从头实现：仅给定设计规范和要使用的包，要求模型从头创建一个项目，主要侧重于前端设计。

评估人员会获得基于评分标准的评分问题，以帮助他们首先评估单个响应标准并对单个尝试进行评分，然后再在两个模型轨迹之间给出最终偏好评分。² 我们分享了North Mini Code的评估结果，比较了SFT（监督微调）检查点与最终模型发布检查点。

## 基准测试方法论

我们的核心智能体能力通过SWE-Bench Verified、SWE-Bench Pro、Terminal-Bench v2和Terminal-Bench Hard进行衡量。North-Code-Mini在评估中使用了Swe-Agent harness v1.1.0（用于SWE-Bench），以及一个基于Harbor的Tmux会话实现、仅使用单一终端工具的简单ReAct harness（用于Terminal-Bench v2）。对于Terminal Bench Hard，我们直接使用了Terminus-2，并遵循与Artificial Analysis Intelligence Index相同的方法论，将North Mini Code与其他模型进行对比。我们遵循各基准测试官方设定的超时时间和硬件资源限制（凡有明确规定的）。此外，我们还追踪了SciCode [15]中的代码生成能力（该基准衡量科学问题的编码性能），以及LiveCodeBench v6 [16]（该基准要求在不使用工具的情况下，凭借强大的算法推理能力完成编码任务）。我们使用3个不同的随机种子运行每个基准测试，并报告平均基准性能，参数设置为temperature=1.0和top_p=0.95。
