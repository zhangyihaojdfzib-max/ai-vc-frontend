### 训练流程

Open R1 发布后，GRPO（分组相对策略优化）已集成到最新版 TRL（版本 0.14）中。这一集成支持使用一个或多个奖励函数或模型训练任何模型。GRPO 实现集成了 DeepSpeed ZeRO 1/2/3，可实现扩展到多 GPU 的并行训练，并使用 vLLM 进行快速生成——这是在线训练方法的主要瓶颈。

```python
from datasets import load_dataset
from trl import GRPOConfig, GRPOTrainer

dataset = load_dataset("trl-lib/tldr", split="train")


def reward_len(completions, **kwargs):
    return [-abs(20 - len(completion)) for completion in completions]

training_args = GRPOConfig(output_dir="Qwen2-0.5B-GRPO", logging_steps=10)
trainer = GRPOTrainer(
    model="Qwen/Qwen2-0.5B-Instruct",
    reward_funcs=reward_len,
    args=training_args,
    train_dataset=dataset,
)
trainer.train()
```

目前仍存在高内存使用相关的限制，我们正在努力进行分析和优化。

### 合成数据生成

R1 报告中最令人兴奋的发现之一是，主模型可用于生成合成推理轨迹，而在此数据集上微调的小型模型也能获得与主模型类似的性能提升。因此我们自然希望重建合成推理数据集，以便社区能基于此微调其他模型。

对于像 R1 这样的大型模型，主要挑战在于高效快速地扩展生成规模。我们花了一周时间尝试各种设置和配置。

该模型可适配两个 8xH100 节点，因此我们自然从该配置开始实验，并使用 vLLM 作为推理服务器。然而我们很快发现这种配置并不理想：吞吐量欠佳，且由于 GPU KV 缓存过快填满，仅允许 8 个并行请求。当缓存填满时，占用大量缓存的请求会被抢占，如果配置使用 PreemptionMode.RECOMPUTE，这些请求会在 VRAM 可用时重新调度。

随后我们切换到 4 个 8xH100 节点（共 32 个 GPU）的配置。这提供了足够的空闲 VRAM 来并行运行 32 个请求，几乎没有请求因缓存利用率达到 100% 而被重新调度。

最初我们以批量方式向 vLLM 服务器发送请求，但很快发现批次中的掉队者会导致 GPU 利用率波动，因为新批次必须等上一批次最后一个样本完成后才能开始处理。将批量推理切换为流式处理显著稳定了 GPU 利用率：

![image/png](/images/posts/4dc5542942ed.png)

只需修改向 vLLM 服务器发送请求的代码即可。批量推理的代码：

```python
for batch in batch_generator(dataset, bs=500):
    active_tasks = []
    for row in batch:
        task = asyncio.create_task(send_requests(row))
        active_tasks.add(task)
    if active_tasks:
        await asyncio.gather(*active_tasks)
```

流式请求的新代码：

```python
active_tasks = []
for row in dataset:
    
    while len(active_tasks) >= 500:
        done, active_tasks = await asyncio.wait(
            active_tasks,
            return_when=asyncio.FIRST_COMPLETED
        )

    task = asyncio.create_task(send_requests(row))
    active_tasks.add(task)


if active_tasks:
    await asyncio.gather(*active_tasks)
```

我们正以相当稳定的速率进行生成，但可能还会进一步探索，例如在长查询被抢占时切换至 CPU 缓存是否为更优策略。
