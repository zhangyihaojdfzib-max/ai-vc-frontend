# 加速ND并行：高效多GPU训练指南

由于不同并行策略的复杂性，跨多个GPU训练大模型可能具有挑战性。在Accelerate中，我们与Axolotl合作，集成了一种快速简便的方法，可以在您的训练脚本中使用任意组合的并行策略！

以下是如何将其添加到您的训练脚本中：

```python
from transformers import AutoModelForCausalLM
from accelerate import Accelerator
from accelerate.parallelism_config import ParallelismConfig
from accelerate.utils import FullyShardedDataParallelPlugin



pc = ParallelismConfig(
    dp_shard_size=2, 
    dp_replicate_size=2, 
    cp_size=2, 
    tp_size=2, 
)

fsdp_plugin = FullyShardedDataParallelPlugin(
    fsdp_version=2,
    auto_wrap_policy="transformer_based_wrap",
    transformer_cls_names_to_wrap=["LlamaDecoderLayer"],
    state_dict_type="SHARDED_STATE_DICT",
)

accelerator = Accelerator(
    parallelism_config=pc,
    fsdp_plugin=fsdp_plugin
)

model = AutoModelForCausalLM.from_pretrained(
    "NousResearch/Hermes-3-Llama-3.1-8B", 
    device_mesh=accelerator.torch_device_mesh
)

model = accelerator.prepare(model)

```

我们还在Accelerate代码库中包含了一个更全面的端到端训练脚本，演示了如何设置数据加载器、优化器和训练循环，以及如何在训练后保存模型。

为了进一步简化大规模微调模型，并将并行策略与各种微调技术相结合，我们还将此技术集成到了Axolotl中。为了帮助您立即开始，我们测试了一些示例配置，您可以修改这些配置以满足您的需求——使用以下命令尝试一个：

```bash

axolotl train examples/distributed-parallel/llama-3_1-8b-hsdp-tp.yaml

```

您也可以查看Axolotl ND并行文档以获取更多详细信息——将ND并行技术添加到您现有的配置中非常简单，只需在您的Axolotl配置文件中添加以下一个或多个字段：

```yaml


dp_shard_size: 2

dp_replicate_size: 2

context_parallel_size: 2

tensor_parallel_size: 2

```

我们通过Accelerate中的`ParallelismConfig`类或Axolotl中的配置字段，使得配置不同并行策略的度以及它们如何组合变得容易。但是，我们如何知道哪种配置最适合我们的用例呢？随着我们扩展到训练具有数百甚至数千亿参数的模型，主要挑战来自于理解不同的并行策略以及它们如何相互作用以最小化设备间的通信开销。在本文中，我们将介绍不同并行策略的工作原理，以及何时以及如何组合它们。

## 数据并行

![分布式数据并行在每个设备上复制整个模型，并将数据均匀划分为每个设备的子批次。（来源：Martynas Šubonis）。](/images/posts/fe7c705d2216.png)

数据并行是跨多个GPU训练模型最常用的技术，它涉及在每个设备上复制模型、梯度和优化器状态，同时在GPU之间均匀分配数据批次，并在更新参数之前跨设备同步梯度。与单设备训练相比，这可以显著提高吞吐量，但要求您的模型能够适应单个设备。

我们可以通过Accelerate的`ParallelismConfig`中的`dp_replicate_size`参数或Axolotl中的配置字段来控制模型的副本数量。值得注意的是，DP是一种**最顶层**的并行策略，这意味着如果我们使用`dp_replicate_size=2`并将其与其他并行策略组合，将会有2个模型副本，每个副本也受到其他并行策略的影响。例如，如果我们使用`dp_replicate_size=2`和`tp_size=2`，我们将有2个模型副本，每个副本包含2个张量并行分片。

我们使用术语**分片**来描述单个设备上的数据，它是更大数据块的一个分区。

## 全分片数据并行

![全分片数据并行将模型的每个参数均匀地划分到每个设备上，并且像DDP一样，将数据均匀划分为每个设备的子批次。为了完成前向和后向传递，FSDP必须在每次前向/后向传递之前**收集**每个参数的权重，以便每个设备获得该参数的完整副本。（来源：Martynas Šubonis）。](/images/posts/d9679e96e119.png)

如果我们的模型太大，无法适应单个设备怎么办？全分片数据并行通过将模型的权重、梯度和优化器状态分片（均匀分布）到GPU上来解决这个问题（这灵感来源于DeepSpeed的ZeRO-3），同时每个设备仍然接收完整批次数据的一部分。正如您可能从上图中注意到的，我们不需要在每个设备上保存整个模型的完整副本，而是在前向传递之前每次只收集单个层的权重，之后权重可能会再次被分片。

通过这种方式，我们以内存使用为代价，换取每次前向和后向传递之前收集分片参数以及归约-分散局部梯度的通信开销。我们可以通过调整参数收集的粒度来控制FSDP中的这种权衡。在一个极端情况下，我们可以收集和重新分片模型的每一层，这将导致最低的峰值内存使用，但会产生最高的通信成本。在实践中，常见的方法是一次收集整个Transformer解码器块的权重。

虽然我们可以进行进一步的内存-计算权衡，并将模型参数和梯度卸载到CPU以训练更大的模型，但这可能会慢得令人无法接受。相反，让我们考虑如何有效地利用更多设备来训练更大的模型，同时保持较高的数据吞吐量。

我们使用术语**节点**来指代一台包含多个GPU（最多8个）的机器，节点内使用例如NVLink在GPU之间进行快速通信。当使用多个节点进行训练时，我们依赖于机器之间相对较慢的节点间通信通道，例如使用Infiniband。我们还将进程池中的设备总数称为**世界大小**——例如，一个包含8个GPU的节点代表世界大小为8，4个节点则代表世界大小为32。

当跨多个节点使用FSDP时，我们将跨节点的整个设备集视为在单个节点上进行训练。例如，对于4个节点，每个节点包含8个GPU，我们在32个设备上进行分片，并使用节点间和节点内通信后端执行集体全归约和归约-分散操作。通过这种方式，仅FSDP就可以扩展到大量GPU，并使用较大的全局批次大小来增加数据吞吐量。然而，当达到一定程度时，会出现一些挑战，可能需要将FSDP与其他并行技术组合使用。我们通常尽量避免在超过一个完整节点的范围内进行FSDP，因为通信开销可能变得太高，我们将在**混合分片数据并行**部分讨论如何解决这个问题。

您可以使用Accelerate的`ParallelismConfig`中的`dp_shard_size`参数以及一个准备好的`FullyShardedDataParallelPlugin`，或者在Axolotl中设置`dp_shard_size`配置字段来设置应用于模型的FSDP程度。

## 张量并行

![张量并行将大型线性层拆分到多个设备上，通常对第一层使用列向分片，对后续层使用行向分片。这种方法仅需一次AllReduce通信操作来合并分片输出，在跨节点内设备分布内存和计算的同时，最小化了通信开销。](/images/posts/0405c77f03c9.png)

张量并行（TP）是一种模型并行技术，其中模型的分片永久驻留在不同的设备上，与数据并行技术不同，每个设备接收的是相同批次的数据。TP通过将线性层的计算分布到多个设备上来工作，因此每个设备只计算矩阵乘法的一部分。当存在大型线性层（例如Transformer模型中的前馈层）可以跨设备拆分时，这种技术效果最佳。我们还可以在注意力层中的查询、键、值和输出投影上分别使用TP，几乎不会产生额外的通信成本。

为了获得最佳性能，连续层的参数可以以特定的方式分布，以最小化所需的通信。在处理成对的线性层时，我们可以对第一层进行列向拆分，对后续层进行行向拆分，这样我们只需一次All-Reduce操作来合并分片输出即可计算最终输出。

与FSDP的动态分片行为不同，TP创建的是静态内存分区，这会导致内存使用量随TP组大小成比例地恒定减少。这对于巨型模型至关重要，因为即使是单个解码器层也可能太大，无法在FSDP的All-Gather操作期间放入内存（回想一下，FSDP中的常见做法是一次性收集整个解码器层的权重）。然而，与FSDP在节点间相对线性扩展（在一定范围内——同构集群上可达约512个GPU，在低带宽连接间则显著减少）不同，TP仅在单个节点内部有效。TP在计算期间需要设备间频繁的激活同步，因为每个设备只计算输出的一部分，需要通信其他设备的输出才能继续前向传播。因此，如果我们希望在多节点设置中使用TP，必须考虑将TP与其他并行技术组合使用，同时将TP限制在单个节点内。由于其较大的通信开销，不建议在通过PCIe连接的GPU上使用TP。

在Accelerate中，TP大小通过 `ParallelismConfig` 中的 `tp_size` 配置，而在Axolotl中，你可以使用 `tensor_parallel_size` 配置字段。

## 上下文并行

最近，随着模型使用越来越多的Token来解决复杂任务，LLM的推理能力导致序列长度激增。为了通过微调实现这种行为，我们需要一种在超长序列上训练模型的方法——有时序列长度可达百万Token！

由于Transformer中的注意力操作随上下文长度呈二次方增长，这在单个GPU上变得不可能。例如，当微调一个相对较小的模型如Mistral-7B（使用32个注意力头）时，如果我们使用128k的序列长度，单个注意力矩阵将占用 128k * 128k * 2字节 * `num_heads`=32 = ~32GB * 32 = ~1TB 的激活内存！虽然在使用FlashAttention等优化注意力实现时，这个例子并不现实，但它有助于说明增加上下文长度导致的内存需求增长。

通过上下文并行（CP），我们可以沿着序列维度对输入进行分片，使得每个设备只处理完整上下文的一个块，并计算庞大得难以处理的完整注意力矩阵的一小部分。要理解其工作原理，回想一下注意力计算的公式：Attention(Q,K,V)=softmax(QKT)V\text{Attention}(Q, K, V) = \text{softmax}(QK^T)VAttention(Q,K,V)=softmax(QKT)V

其中QQQ、KKK和VVV分别是查询、键和值矩阵。QQQ中的每个查询向量（行，或输入嵌入）必须针对整个序列中KKK的*每个*键向量计算注意力分数，以正确应用softmax归一化。然后，这些注意力分数与VVV中的*所有*值向量进行加权。

这里的关键细节在于，QQQ中的每一行可以彼此独立地计算其注意力分数，但每个查询向量仍然需要完整的KKK和VVV矩阵。换句话说，给定一个序列长度为 $n$ 的输入，我们可以将上面的注意力公式展开为：

Attention(Q,K,V)1=softmax(Q1KT)VAttention(Q,K,V)2=softmax(Q2KT)V⋮Attention(Q,K,V)n=softmax(QnKT)V\begin{align}
\text{Attention}(Q, K, V)_1 &= \text{softmax}(Q_1 K^T) V \\
\text{Attention}(Q, K, V)_2 &= \text{softmax}(Q_2 K^T) V \\
&\vdots \\
\text{Attention}(Q, K, V)_n &= \text{softmax}(Q_n K^T) V
\end{align}Attention(Q,K,V)1​Attention(Q,K,V)2​Attention(Q,K,V)n​​=softmax(Q1​KT)V=softmax(Q2​KT)V⋮=softmax(Qn​KT)V​​

其中我们将查询矩阵的每一行表示为Q1,Q2,...,QnQ_1, Q_2, ..., Q_nQ1​,Q2​,...,Qn​。这可以概括为：Attention(Q,K,V)i=softmax(QiKT)V∀i∈{1,2,...,n}\text{Attention}(Q, K, V)_i = \text{softmax}(Q_i K^T) V \quad \forall i \in \{1, 2, ..., n\}Attention(Q,K,V)i​=softmax(Qi​KT)V∀i∈{1,2,...,n}

当我们跨设备对输入进行分片时，得到的QQQ、KKK和VVV矩阵（由这些输入分片计算得出）也会自动沿着序列维度分片——每个GPU只计算其负责的那部分序列的查询、键和值。例如，假设有WWW个GPU，序列长度为nnn：

- GPU 0 计算 Q1:n/WQ_{1:n/W}Q1:n/W​, K1:n/WK_{1:n/W}K1:n/W​, V1:n/WV_{1:n/W}V1:n/W​
- GPU 1 计算 Qn/W+1:2n/WQ_{n/W+1:2n/W}Qn/W+1:2n/W​, Kn/W+1:2n/WK_{n/W+1:2n/W}Kn/W+1:2n/W​, Vn/W+1:2n/WV_{n/W+1:2n/W}Vn/W+1:2n/W​
- ...
- GPU (W−1)(W-1)(W−1) 计算 Q(W−1)n/W+1:nQ_{(W-1)n/W+1:n}Q(W−1)n/W+1:n​, K(W−1)n/W+1:nK_{(W-1)n/W+1:n}K(W−1)n/W+1:n​, V(W−1)n/W+1:nV_{(W-1)n/W+1:n}V(W−1)n/W+1:n​

我们如何确保注意力计算正确呢？如上所述，每个设备只需要自己的QQQ分片，但需要完整的KKK和VVV矩阵才能正确计算注意力。我们可以通过一种称为**RingAttention**的技术来实现，其工作原理如下：

1.  初始时，每个GPU持有其QQQ、KKK、VVV的分片（例如，GPU 0持有Q1:n/WQ_{1:n/W}Q1:n/W​, K1:n/WK_{1:n/W}K1:n/W​, V1:n/WV_{1:n/W}V1:n/W​）。
2.  然后，每个GPU为其QiQ_iQi​分片及其本地的KjK_jKj​、VjV_jVj​分片计算一个部分注意力矩阵Ai,jA_{i,j}Ai,j​。
3.  每个GPU将其KKK、VVV分片发送给环中的下一个GPU。
4.  每个GPU从环中的前一个GPU接收一个不同的KKK、VVV分片。
5.  每个GPU使用接收到的KKK、VVV分片计算额外的部分注意力矩阵Ai,j+1A_{i,j+1}Ai,j+1​、Ai,j+2A_{i,j+2}Ai,j+2​等。
6.  每个GPU重复此过程，直到接收到所有KKK、VVV分片，并计算出所有部分注意力矩阵Ai,∗A_{i,*}Ai,∗​。

![上下文并行将输入序列分片到多个GPU上，每个设备持有其分配片段的查询和键值对。环注意力在GPU之间循环传递K、V分片（如箭头所示），使得每个查询能够针对整个序列的键和值计算注意力分数。最终的注意力输出结合了所有序列位置的信息，同时将内存和计算分布到各个设备上。](/images/posts/9d4b0bc0db02.png)

Accelerate通过 `accelerator.maybe_context_parallel` 装饰器启用此功能，这在Accelerate的示例脚本中也有展示。你也可以在我们的CP概念指南中了解更多关于其工作原理和局限性的信息。

与TP类似，在Accelerate中，CP大小通过 `ParallelismConfig` 中的 `cp_size` 配置，而在Axolotl中，你可以使用 `context_parallel_size` 配置字段。

## ND 并行策略

在多节点环境中，像 FSDP 这样的数据并行技术将整个网络拓扑视为仅存在于单一维度上。您可能会发现这种方法存在多种限制：

*   当扩展到更多节点时，FSDP 的集合通信操作会受到节点间延迟的瓶颈，导致训练速度慢得令人无法接受。
*   正如我们上面提到的，大型模型的解码器层可能无法放入 GPU 内存，或者即使在分片状态下也可能因体积过大而无法执行前向传播。
*   可能无法实现理想的批次大小——要么批次对于纯数据并行来说太大而无法高效处理，要么由于模型大小导致的内存限制而变得太小。

为了尝试解决其中一些问题，我们可以将多节点集群视为具有二维拓扑结构：沿一个轴是设备间快速的节点内通信，沿另一个轴是相对较慢的节点间通信。让我们考虑如何组合我们目前介绍的并行技术来利用这一点。

### 混合分片数据并行

![混合分片数据并行在每个副本组内执行 FSDP，并通过 AllReduce 跨副本组同步梯度，结合了 FSDP 的内存效率和跨节点 DP 的通信效率。](/images/posts/dbd377fcd4e9.png)

混合分片数据并行是一种二维并行策略，它在节点内执行 FSDP，在节点间执行 DP——也就是说，模型在每个节点间复制，并在每个节点内使用 FSDP 进行分片。这使得 FSDP 较大的通信开销可以利用更快的节点内链路，而 DP 则将较慢的节点间通信开销最小化为单个梯度同步步骤。如果您面临问题 1 并希望以增加内存使用为代价来加速训练，可以考虑这种方法。

需要注意的是，我们可以自由配置二维网络拓扑的形状，因为我们并不受限于维度必须与物理节点边界对齐——您可以跨 2 个节点应用 FSDP，同时在 2 个节点组成的组之间进行复制，这将导致内存使用量降低但吞吐量变慢，但仍能将节点内 FSDP 通信开销减少一半。这是一个我们鼓励您根据具体硬件设置和微调需求进行调整的旋钮。

您可以通过在 Accelerate 的 `ParallelismConfig` 中定义 `dp_shard_size` 和 `dp_replicate_size`，或通过 Axolotl 的配置字段来启用 HSDP。

### 全分片数据并行 + 张量并行

正如我们之前提到的，TP 应在节点内应用以利用高带宽的节点内通信。因此，结合 TP 和 FSDP 意味着跨节点使用 FSDP 对模型进行分片，在节点内使用 TP 进行分片。在某种程度上，这为上述所有三个问题提供了一个简洁的解决方案：FSDP 带来的延迟成本可能减少 8 倍，无法放入单个设备的大型层现在均匀分布在多个设备上，并且由于每个 TP 组接收相同批次的数据，我们也可以将全局批次大小减少 8 倍。然而，如果这仍然不够，我们无法跨节点增加 TP 大小，则必须考虑其他方法。

在 Accelerate 中，您可以通过在 `ParallelismConfig` 中同时定义 `dp_shard_size` 和 `tp_size` 来结合 TP 和 FSDP；而在 Axolotl 中，您可以同时添加 `dp_shard_size` 和 `tensor_parallel_size` 配置字段。

### 全分片数据并行 + 上下文并行

这是一种结合了 FSDP 和 CP 的二维并行策略，虽然并不常用，因为 CP 本身已与 FSDP 结合（更多原因请参阅 `accelerate` 概念指南），但在某些情况下可能有用，例如需要处理长序列长度，从而需要较大的 `cp_size`。如果这仍然超出您的内存预算，您可以在此基础上应用 FSDP，进一步减少内存使用。

在 Accelerate 中，您可以通过在 `ParallelismConfig` 中同时定义 `dp_shard_size` 和 `cp_size` 来结合 CP 和 FSDP；而在 Axolotl 中，您可以同时添加 `dp_shard_size` 和 `context_parallel_size` 配置字段。

### 混合分片数据并行 + 张量并行

在足够大的世界大小下（注意：虽然 3D 并行的最小世界大小为 8，但在更大规模下最有效），我们可以考虑将 HSDP 与 TP 结合，从而创建一个层次结构：DP 首先跨节点组复制模型，FSDP 然后在每个组内分片模型，TP 在每个节点内拆分各个层。当面临我们上面提到的所有扩展限制时，您可能会考虑这种方法，因为它通过在内存使用和吞吐量之间进行权衡，提供了最大的灵活性来适应您特定的训练设置。

在 Accelerate 中，您可以通过在 `ParallelismConfig` 中同时定义 `dp_shard_size`、`dp_replicate_size` 和 `tp_size` 来结合 HSDP 和 TP。同样，在 Axolotl 中，您可以同时添加 `dp_shard_size`、`dp_replicate_size` 和 `tensor_parallel_size` 配置字段。

## 使用说明

还有其他组合多种并行策略的方式我们尚未涵盖，例如使用 HSDP + TP + CP 的 4D 并行，但它们的操作方式与我们已介绍的技术非常相似。最重要的是，我们鼓励您尝试不同的技术和配置——这是获得对不同内存/吞吐量权衡方式直觉的最佳途径。

以下是一些在分布式环境中工作时可能有用的额外提示：

-   当使用FSDP处理因模型过大而无法装入单个设备的情况时，启用CPU内存高效加载和分片状态字典检查点技术至关重要。您可以通过Accelerate的`FullyShardedDataParallelPlugin`中的`cpu_ram_efficient_loading`和`state_dict_type`参数来启用此功能：

```python
fsdp2_plugin = FullyShardedDataParallelPlugin(
    fsdp_version=2,
    auto_wrap_policy="transformer_based_wrap",
    transformer_cls_names_to_wrap=["LlamaDecoderLayer"],
    state_dict_type="SHARDED_STATE_DICT", 
    cpu_ram_efficient_loading=True
)
```

或者通过Axolotl中`fsdp_config`内的`cpu_ram_efficient_loading`和`state_dict_type`配置字段来启用：

```yaml
fsdp_version: 2
fsdp_config:
  auto_wrap_policy: TRANSFORMER_BASED_WRAP
  transformer_layer_cls_to_wrap: LlamaDecoderLayer
  state_dict_type: SHARDED_STATE_DICT
  cpu_ram_efficient_loading: True
```

-   训练期间使用的总批次大小对训练稳定性、内存使用和数据吞吐量有重要影响。当使用DP和/或FSDP时，有效批次大小的计算公式为：

    `effective_batch_size = micro_batch_size * gradient_accumulation_steps * dp_world_size`

    其中 `dp_world_size = (dp_shard_size * dp_replicate_size) / tp_size`。您可以通过增加训练循环中的总微批次大小或梯度累积步数，或在Axolotl中设置`micro_batch_size`和`gradient_accumulation_steps`配置字段，或通过添加更多GPU来增加总的`dp_world_size`，从而增大批次大小。如上所述，这强制了一个`dp_world_size`的最小总批次大小——当使用纯DP/FSDP时，这将是您的总世界大小，如果这个值过高，减小总批次大小的唯一方法是引入张量并行。最后，在GPU数量固定且内存受限的场景下，我们建议增加`gradient_accumulation_steps`而不是`micro_batch_size`来实现更大的有效批次大小，反之亦然。

-   相应地，当您的有效批次大小因引入数据并行而增加时，您应该调整学习率以保持训练稳定性。常见的方法包括线性缩放 `scaled_lr = base_lr * (effective_batch_size / base_batch_size)` 或平方根缩放 `scaled_lr = base_lr * sqrt(effective_batch_size / base_batch_size)`。

-   即使采用了并行策略，如果内存限制仍然存在，梯度检查点可以通过以计算换取内存来提供额外的内存节省。在前向传播过程中，只有一部分激活值被保留在内存中（通常在Transformer块边界处），而中间激活值在反向传播过程中重新计算。此技术与上述所有并行策略都能无缝协作。在Accelerate中，您可以通过在`FullyShardedDataParallelPlugin`中设置`activation_checkpointing=true`来启用它：

```python
fsdp2_plugin = FullyShardedDataParallelPlugin(
    fsdp_version=2,
    auto_wrap_policy="transformer_based_wrap",
    transformer_cls_names_to_wrap=["LlamaDecoderLayer"],
    state_dict_type="SHARDED_STATE_DICT", 
    cpu_ram_efficient_loading=True,
    activation_checkpointing=True
)
```

在Axolotl中类似：

```yaml
fsdp_version: 2
fsdp_config:
  auto_wrap_policy: TRANSFORMER_BASED_WRAP
  transformer_layer_cls_to_wrap: LlamaDecoderLayer
  state_dict_type: SHARDED_STATE_DICT
  cpu_ram_efficient_loading: True
  activation_checkpointing: True
```

请注意，由于激活重计算，梯度检查点通常会使训练时间增加约20-30%，但可以将激活内存减少60-80%，这使得它在训练非常大的模型或使用长序列长度时特别有价值。
