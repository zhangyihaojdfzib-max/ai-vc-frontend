# 我们如何构建Cloudflare的数据平台及其上的AI Agent（智能体）

2026-05-28

- Brian Brunner
- Dmitry Alexeenko
- Matt Moen

![](/images/posts/07ff4934333f.png)

Cloudflare每秒处理超过十亿个事件。我们的网络覆盖120多个国家的330多个城市。在每个HTTP请求、每个Worker调用、每个R2读取操作的背后，都有数据，而且是大量的数据。

多年来，这些数据并不容易访问。它们分布在数十个生产数据库、ClickHouse集群、Kafka流、Google Cloud存储桶、BigQuery数据集以及大量管道中。要回答像"今天注册的域名中有多少进入了流量排名前100？"这样的简单问题，Cloudflare的分析师必须知道该查询哪个系统、使用什么凭证、编写什么查询语言，以及他们查看的数据是采样数据、实时数据还是已过期七天的数据。因此，很难从数据中获取有见地的洞察。

为了解决这个问题，我们构建了两个内部工具：Town Lake——Cloudflare的统一数据分析平台，以及Skipper——运行在该平台之上的AI数据Agent（智能体）。Town Lake是Cloudflare所有数据的统一SQL接口，而Skipper让Cloudflare的任何人都可以用自然语言提问，并在几秒钟内获得准确、可审计的答案。

这就是我们构建这两个工具的故事。

### 问题的形态

如果你曾在经历过超高速增长期的公司工作过，你就会知道数据蔓延是什么样子。我们的情况有几个具体症状：

1. **系统过于分散。** 想要调查客户问题的产品工程师可能需要查询Postgres获取账户元数据、ClickHouse获取分析事件、BigQuery获取使用量汇总、R2获取原始日志，以及Kafka主题获取实时信号。每个系统都有自己的凭证、自己的语言和自己的保留策略。
2. **采样数据。** 这对仪表盘来说没问题，但对计费等场景不适用。我们的分析管道会进行降采样以处理每秒超过7亿个事件。当你希望分析仪表盘加载时，这是正确的行为，但当你试图计算某人的使用量以开具发票时，这恰恰是错误的做法。
3. **内部数据依赖外部。** 我们之前的内部报告堆栈部分由外部供应商提供支持。除了成本问题，我们的一些关键数据还严重依赖其他云服务。
4. **没人能找到数据。** 即使你拥有所有正确的凭证，你也需要知道"按账户计费的Workers请求"的正确表位于特定的ClickHouse集群、特定的模式中，需要关联到特定的Postgres维度表，而且关联需要一个晦涩的客户ID转换。这需要太多隐性知识。

我们还面临文化挑战：数据基础设施历来被视为服务于业务的后台功能，而不是其本身应有的关键基础设施。

### 我们的目标

我们希望创建一个地方，公司内任何拥有适当权限且有知情需求的人都能获得关于Cloudflare问题的答案："显示上季度收入排名前100的客户"、"列出过去48小时内来自特定ASN、评分>0.9的所有Bot Management ML评分事件"、"查找消费超过100美元的客户中排名前100的计费支持工单"等。

我们希望这个地方能够为需要精确数据的查询（如计费或安全调查）提供新鲜、准确、未采样的数据，而为不需要精确数据的查询（如仪表盘或探索）提供快速、降采样的数据。

我们希望内置安全性和治理功能，能够自动检测个人身份信息（PII），默认锁定敏感表。所有访问都应可审计，并具有时间限制的权限授予，以便用户仅在积极处理需要数据的任务时才能访问数据。

我们希望它构建在Cloudflare自己的平台上：R2用于存储、Workers用于计算、Cloudflare Access用于认证、Workflows用于编排。如果我们要对数据基础设施进行重大投资，那就要构建在我们销售给客户的同一套产品之上。

最终，我们希望有一个不需要了解任何SQL的接口。目标是让公司内任何拥有适当权限且有知情需求的人都能查看流经我们网络的数据流，而不仅仅是分析师。

最后一个需求就是Skipper的由来。

### Town Lake，数据平台

我们的数据平台架构核心是一个**数据湖仓**：一个从对象存储读取数据的查询引擎，带有一个使存储行为类似数据库的元数据层。我们称之为Town Lake，以德克萨斯州奥斯汀的同名湖泊命名。

其最重要的组件包括：

**查询引擎。** 我们选择了Apache Trino：一个SQL查询就可以关联Postgres表、ClickHouse表和R2上的Iceberg表，无需将中间结果物化到另一个系统。一个询问"本周按Workers请求计算的前100名付费客户是什么"的查询会被编译成一个计划，将过滤器推送到ClickHouse，关联Postgres中的账户维度，并在R2中对比计费汇总进行排名，一气呵成。

**R2 Data Catalog**，我们的托管Apache Iceberg服务，是冷数据和温数据的存储位置。Iceberg提供了模式演化、时间旅行、分区演化以及随数据老化进行压缩的能力。上周的每分钟使用量变为每小时，上季度的每小时变为每天，以此类推。存储成本随数据新鲜度降低而降低，同时数据保持可查询性。与将相同数据保存在OLAP数据库中相比，R2中的Parquet文件成本低得多。

**DataHub**是我们的元数据目录。每个表、列、所有者、血缘边和术语都存储在那里。当用户问"townlake.dim.accounts里有什么"时，DataHub会提供答案，包括表描述、列描述、所属团队、提供数据的上游表以及消费数据的下游表。

**Lifeguard**是我们的访问控制服务：它在D1中存储访问规则，从我们的内部访问管理系统动态拉取用户和组成员信息，并生成一个组合的JSON策略，Trino通过HTTP读取该策略。Lifeguard还会向Skipper和Gateway提供基本访问信息，因此用户在查询之前就会在前端被拦截，而不是在查询时。

Skimmer 是一个 PII 检测扫描器。它持续运行，对每个表中的每一列进行行采样，并使用 Workers AI 对每列是否包含 PII 进行分类。它通过两轮扫描完成：第一轮是快速的逐列分类器；如果发现任何标记，则进入第二轮基于 Agent 的扫描，获取完整的表上下文，并可直接查询 Trino 进行验证。检测结果会流入 DataHub 和 Lifeguard 的允许列表中，以便进行人工审核。

Transformer 是我们基于 Workflows 构建的 ELT（提取、加载、转换）引擎。用户通过 YAML 前置元数据（目标表、物化模式、依赖关系、调度计划）定义 SQL 转换的有向无环图（DAG）。Transformer 编译该图并在 Trino 上运行，状态由 Durable Objects 管理，定义存储在 R2 中，运行历史记录在 D1 中。

Ingestion 是连接运营系统与数据湖的桥梁。一个编排器作为长期运行的 Kubernetes 部署运行，读取管道配置，并生成短期运行的 Worker 任务，从 Postgres 或 ClickHouse 提取数据，转换为 Parquet 格式，然后作为 Iceberg 表加载到 R2 中。每个管道以全量替换或增量追加模式运行。

### 默认关闭：通过构建实现治理

当你构建一个统一的数据平台时，一个真实的担忧是你刚刚构建了一个庞大的敏感数据面。对此的传统答案是：默认开放，例外限制。允许访问所有内容，然后在有人注意到时审计并锁定敏感表。

Town Lake 采取了相反的方法。表在审核完成之前无法被查询。当一个新的数据库连接到 Trino 或创建了一个新表时，Skimmer 会扫描它，对其列进行分类，并将其注册到中央允许列表中，状态为待审核。在审核者批准该表及其中的特定列之前，用户无法查询它。这听起来很痛苦，实际上也确实如此，但有两件事例外。

首先，它是自动化的。Skimmer 的分类器相当不错：它能捕获明显的 PII（电子邮件、IP、姓名、电话号码）以及长尾的非明显敏感数据（匹配特定前缀的 API Token、可追溯到用户的不透明 ID）。审核者会看到检测到的内容，并可以选择批准、覆盖或拒绝。大多数审核只需几秒钟。

其次，工作流是自助式的。如果你查询一个没有权限的表，错误信息不会是“权限被拒绝”。而是“此表需要审核，点击此处申请审核。”AI Agent Skipper 甚至会建议合适的 RBAC 组供你申请，并直接链接到该组。

我们将模式发现与数据访问分开。用户可以看到存在哪些表，但未审核的列会从 `DESCRIBE`、`SHOW COLUMNS` 和 `SELECT *` 中隐藏。这种微妙的区别很重要：这意味着一个新的未审核列不会破坏基于已批准表其余部分构建的现有仪表板。

PII 是按会话选择加入的。默认情况下，Trino 会在敏感列到达你的屏幕之前对其进行编辑。如果你有正当理由需要原始 PII（例如，欺诈调查），你可以翻转会话中的开关，系统会检查你的权限，然后解除编辑。每次翻转和每次查询都会被记录。

### Skipper：AI 数据 Agent

如今，仅靠一个查询引擎是不够的。SQL 仍然是一个障碍，同样，知道要查询成千上万个表中的哪一个也是一个障碍——你需要知道规范的模式。

Skipper 是我们对对话式 AI Agent 的诠释，它从自然语言问题出发，得到经过验证的答案，并基于公司的实际数据、代码和机构知识。我们将其构建在 Town Lake 之上，也构建在我们的开发者平台之上：Workers、Workers AI、Durable Objects、D1、R2、Workflows、KV。

界面是一个聊天框。提出一个问题：

显示过去 30 天内按 R2 存储成本排名的前 10 位客户，以及与之前 30 天相比的变化。

Skipper 会找到正确的表（通过 DataHub 搜索），拉取它们的模式和血缘关系，编写 SQL，提交给 Trino，轮询结果，然后向你展示一个表格或图表。接着追问：

现在按区域细分，并忽略 Cloudflare 内部账户。

它会携带上下文，优化查询，并重新运行。如果某些内容看起来有误，例如，一个连接产生了零行，或者一个过滤器排除了你期望的内容，那么 Skipper 会进行调查、调整并重试，这是一个闭环推理过程。困难的部分在于拥有正确的上下文。

Skipper 还可以将图表打包成仪表板，这些仪表板可以在内部共享并嵌入到其他内部应用程序中。它还具有通过 Transformer 构建转换图以及通过 Lifeguard 检查访问和权限的工具。

Skipper 可以在用户所在的任何地方为其提供服务。所有这些工具都可以通过一个 Worker 获得，该 Worker 由一个由 Workers AI 驱动的内置 Agent 框架支持。另一方面，我们的许多内部用户通过本地 Agent 工作流工作，Skipper 的工具还可以通过 MCP 服务器获得。

### 上下文层次

给定一个 SQL 提示和一系列表名，LLM 可能会幻觉出连接、误用列，并自信地产生一个完全错误的数字。我们在早期的实验中通过艰难的方式学到了这一点。解决方案是多层次的、基于上下文的上下文，模型可以在检索时从中提取信息。

第 1 层：模式和使用元数据。DataHub 知道每个表的每一列、每个类型、每个主键、每个外键。它还知道哪些表基于历史查询模式经常被连接在一起。Skipper 的 `search_datasets` 和 `get_entity_details` 工具直接提供这些信息。

第 2 层：人工注释。当拥有 `dim.accounts` 的团队编写了诸如“账户级别实体。每个 account_id 一行。每个账户恰好属于一个客户（通过 customer_id 外键）”这样的描述时，该描述会存在于 DataHub 中，并最终进入 Skipper 的上下文。像 `curated` 这样的标签标记了 Skipper 应优先于临时空间的已验证表。

第 3 层：代码衍生知识。一些最有价值的上下文不在任何目录中：它存在于生成表的 SQL 中。Transformer 管道在每次成功运行时都会向 DataHub 发出每个节点的 `.meta.json` 文档。因此，当 Skipper 查看 `fct.billings_allocated` 时，它看到的不仅仅是模式；它看到这是一个从 `dim.accounts`、`dim.customers` 和 `seed.product_classification` 构建的预连接事实表，其 `alloc_amount` 列计算为 `billed_amount / 12 for annual; billed_amount for monthly`。这种细微差别正是区分正确答案和自信错误答案的关键。

第 4 层：精选数据模型。我们维护一小部分“数据模型”页面：简短的人工编写的文档，描述如何理解计费、客户、账户和区域。“优先选择标记为‘curated’的表。避免使用 `scratch_r2` 和标记为‘internal’的表。使用数据模型术语（例如，‘billing product revenue’）进行搜索，而不是自然语言。”这些作为 MCP 资源提供，当问题匹配时，Agent 可以拉取这些资源。

第 5 层：运行时内省。当所有其他方法都失败时，Skipper 可以向 Trino 发出实时查询：`DESCRIBE table`、`SELECT DISTINCT col LIMIT 20`、`SELECT COUNT(*)`。它谨慎使用这些查询，因为运行时上下文成本很高，但它是使系统其余部分健壮的安全网。

### Skipper 作为 MCP：代码模式

有一个具体的实现细节值得单独提出来，因为它是一个独特的 Cloudflare 形态的解决方案。

当你使用工具构建 AI Agent 时，标准模式是在你的提示中定义工具，让模型一次调用一个工具，解析响应，执行，并返回结果。这没问题，但很啰嗦：一个五工具的工作流需要五次模型往返，每次都必须重新建立上下文。

对于我们的 MCP 服务器，我们使用**代码模式**。我们不定义 30 个单独的工具，而是暴露两个工具：`search` 和 `execute`。模型编写一个 JavaScript 代码片段，以编程方式调用我们的整个工具集：

```JavaScript
const datasets = await skipper.search_datasets({ query: "billing product revenue" })
const queryId = await skipper.start_query({ sql: "SELECT ..." })
const results = await skipper.fetch_results({ queryId, mode: "inject" })
return skipper.create_chart({ chartType: "bar", data: results.rows, ... })
```

这段 JavaScript 代码通过 `WorkerLoader` 在沙箱化的 Dynamic Worker 隔离环境中运行。模型能够用其已熟练掌握的语言，在一次往返中表达复杂的多步骤工作流。这种方式更快、更便宜，且生成的工作流可作为代码进行审计。

### 安全模型即数据模型

Skipper 的所有操作都以调用用户的身份运行。如果你没有某个表的访问权限，Skipper 就无法为你查询该表。如果你请求 PII（个人身份信息），系统会检查你的权限。如果你保存的查询与团队成员共享，他们的访问权限会在查看时而非保存时进行检查，因为群组成员关系会发生变化。

共享仪表盘有其独特之处。它们可以通过一个占位符 div 和一个 script 标签嵌入到任何 Cloudflare 内部工具中：

```HTML
<div data-skipper-dashboard="dash-123"></div>
<script src="https://skipper.cloudflare.com/embed.js" async></script>
```

iframe 会自动调整大小以适应内容。Content Security Policy (CSP) 的 `frame-ancestors` 会阻止从企业域名之外的任何位置嵌入。Cloudflare Access 仍然对 iframe 内容进行访问控制，因此未认证的查看者会在 iframe 中看到 Access 登录页面，而非数据本身。非所有者查看者会基于底层表进行权限检查：如果他们没有访问权限，系统会引导他们向正确的群组申请权限。

### 它驱动的能力：极速答案

**计费。** 这是最初的使用场景。我们的可计费用量仪表盘（面向客户的仪表盘，向按量付费用户展示其确切欠费金额）由计量管道驱动，其数据源是 R2 中的一组 Iceberg 表，通过 Trino 查询。该仪表盘的 API 提取与计费系统相同的紧凑行（date, account_id, metric_name, usage），因此仪表盘上的数字与账单上的数字完全一致。

计费相关查询占 Town Lake 所有查询的 53%：在最近的一个测量周期中，来自 324 名不同的 Cloudflare 员工，共计 91,760 次查询。过去用于计算按客户收入汇总的 200-300 行传统 SQL 查询，现在仅需五行。

**商业智能。** 现在，在 Skipper 中查询"按收入排名的前 100 名客户"大约需要三秒钟。"今天注册的域名中有多少进入了前 100 名"也是如此。我们过去需要提交 Jira 工单的大多数数据相关问题，现在都能快速解决。

**安全分析。** 我们的 Bot Management 团队使用 Town Lake 查询过去 48 小时内 ML 评分事件中分数 > 0.9 的数据，并按 ASN 和地理位置进行过滤。威胁研究人员在其基础上构建了自己的查询工具包。Trust & Safety 团队提取信号以协助打击滥用行为。

**客户支持。** "查找来自消费超过 100 美元的客户的前 100 个计费支持工单"过去是一个需要多天的项目。现在，它只是一个 Skipper 查询。

### 我们的经验教训

有几件事让我们感到意外。

**更少的提示词反而更好。** 早期版本的 Skipper 有详尽、指令性的系统提示词："首先，使用 search_datasets。然后，使用 get_entity_details。然后，如果需要，使用 list_schema_fields……" 质量反而下降了。模型擅长推理分析工作流，不需要被微观管理。我们用高层指导替换了指令性提示词，让模型自行选择路径。结果变得更好了。

**工具重叠是毒药。** 我们最初暴露了每个工具的每个变体：三个不同的"获取结果"工具，两个"搜索"工具，几个"列表"工具。模型感到困惑并调用了错误的工具。我们进行了整合。现在，`fetch_results` 有一个 `mode` 参数（inject / display / both），而不是三个独立的工具。每个工具都有唯一的存在理由。

**代码而非元数据能捕捉含义。** 最大的准确性提升来自于我们开始摄取生成表的实际 SQL，而不仅仅是其模式。一个包含 `contract`、`paygo`、`free` 值的 `customer_type` 列在不同上下文中看起来完全相同，但 SQL 告诉你，当 Salesforce 数据缺失时，`customer_type` 默认为 `paygo`。这种上下文永远不会存在于列描述中。

**记忆比我们预期的更重要。** 存在一个长尾的修正，例如"你必须像这样过滤 X"或"忽略标记为 Y 的表"。没有记忆层，Agent（智能体）会在每次对话中重新发现并重新学习这些内容。有了记忆层，它会在团队实际提出的重复性问题中单调地变得更好。

**枯燥的基础设施才是难点。** Trino + Iceberg 并非新技术。难点在于那些枯燥的工作：按行访问控制、默认关闭的表白名单、查询审计、有时限的凭证、PII 检测、幂等摄取、模式演化。这些才是让数据平台安全可用的关键。
