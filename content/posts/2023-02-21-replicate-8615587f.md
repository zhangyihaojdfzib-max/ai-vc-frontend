# 机器学习需要更好的工具

- bfirsh

Replicate 在云端运行机器学习模型。我们拥有一个开源模型库，您只需几行代码即可运行这些模型。如果您正在构建自己的机器学习模型，Replicate 能让您轻松地大规模部署它们。了解更多。

机器学习曾经是一项学术追求。如果您想从事这方面的工作，您可能需要加入实验室或拥有博士学位。

2021 年初，情况发生了变化。Advadnoun 发布了一个名为 The Big Sleep 的 Colab 笔记本。RiversHaveWings 随后发布了 VQGAN+CLIP 笔记本。这些笔记本通过使用 CLIP 引导 GAN，将文本描述转化为图像。

这些项目并非将某个准确率指标提升零点几个百分点。它们没有作为学术论文发表。它们只是一些能实现有趣功能的软件片段。

![由 The Big Sleep 生成的图像，大约在 2021 年](/images/posts/6aafc25433c3.webp)

人们复制了这些笔记本并在此基础上进行改进。我们有了 pixray、dalle-mini、Disco Diffusion。感觉每周都有新东西出现。

这些人并非隶属于某个实验室。他们通常是自学成才，只是在业余时间摆弄。或者他们是软件工程师，将零散的机器学习代码拼凑在一起。

使这一切成为可能的部分原因是预训练的基础模型。个人无需花费巨大代价从头开始训练模型，而是可以直接选用现成的模型，并以有趣的方式将它们组合起来。这有点像您可以从 npm 导入几个包，然后将它们拼接起来创造新东西。

这一切感觉更像是开源软件，而不是机器学习研究。

然后，Stable Diffusion 出现了。

![现在经典的“骑马的宇航员”，由 Stable Diffusion 生成](/images/posts/1020643f0c7b.webp)

DALL-E 2 是几个月前发布的、质量相似的文本到图像模型，但它是闭源的，并且处于内测阶段。我们习惯了普通软件的进步是开源的，并且在 GitHub 上。DALL-E 是多年来最非凡的软件作品，但软件工程师却无法使用它。

这为 Stable Diffusion 创造了肥沃的土壤。Stable Diffusion 是开源的，并且比上一代开源文本到图像模型的质量好得多。

它激发了黑客们的想象力，并引发了分叉的爆炸式增长：图像修复、动画、纹理生成、微调等等。

这些开源的图像生成模型突然变得足够好用了。人们正在构建大量的东西。有从文本生成图像的移动应用，仅用几张训练图像就能创建个人头像的应用，以及程序化生成的游戏。

然而，有一个问题。

## 机器学习太难用了

如果您尝试用这些机器学习模型实际构建一些东西，您会发现它们实际上都不太管用。您整天都在与混乱的 Python 脚本、损坏的 Colab 笔记本、令人困惑的 CUDA 错误、形状错误的张量作斗争。真是一团糟。

普通软件曾经也是这样。如果您 20 年前想建一个网站，感觉就像今天尝试使用机器学习一样。您必须构建 Web 服务器、认证系统、用户界面组件。您手动拼接 HTML 和 SQL，希望自己不会被攻击。要部署，您将文件上传到 FTP 服务器，然后等待并祈祷一切顺利。

但后来我们有了 Ruby on Rails。还有 Django。还有 React。还有 Heroku。还有 Next.js。还有 Vercel。以及所有其他的工具。这些工具让软件开发变得更简单、更有趣。

在他关于 Software 2.0 的文章中，Andrej Karpathy 提出了一个观点：深度学习正变得足够先进，以至于它正在取代普通软件的很大一部分。但他也指出，这是一种奇怪的软件，需要一套新的工具来使用。他论证了我们需要一个新的工具栈：

我们看待趋势的视角很重要。如果您将 Software 2.0 视为一种新兴的编程范式，而不是仅仅将神经网络视为机器学习技术中一个相当不错的分类器，那么其推论就变得更加明显，并且很明显还有更多工作要做。
……
谁会开发第一个 Software 2.0 IDE，帮助完成积累、可视化、清理、标注和采购数据集的所有工作流程？
……
传统的包管理器及相关服务基础设施，如 pip、conda、docker 等，帮助我们更轻松地部署和组合二进制文件。我们如何有效地部署、共享、导入和使用 Software 2.0 二进制文件？

机器学习之所以难以使用，并不是因为它本身很难。我们只是还没有好的工具和抽象层。您不应该为了使用机器学习而必须了解 GPU，就像您不必为了建网站而了解 TCP/IP 一样。

## 机器学习在 Spotify 很难用

Andreas 是我的老朋友。他曾是 Spotify 的机器学习工程师。他是一个有点奇怪的机器学习工程师，因为他从基础研究一直做到构建 Spotify 主要的深度学习管道之一。涵盖了从数学、产品到服务器的方方面面。

我的背景是开发者工具，最近在 Docker 工作，在那里我创建了 Docker Compose。我还创立过几家现已倒闭的开发者工具初创公司，偶然写了一本关于命令行界面的书，并且很久以前曾是网络浏览器的滥用者。

Andreas 告诉我 Spotify 存在一系列相关问题：

- 运行开源机器学习模型很困难。所有这些进步都锁在 PDF 中的文字、GitHub 上的代码片段、Google Drive 上的权重文件里（如果幸运的话！）。如果您想在这些研究的基础上构建，或者将其应用于现实世界的问题，您必须从头开始全部实现。
- 将机器学习模型部署到生产环境很困难。通常，研究人员需要与工程师坐下来决定 API、编写服务器代码、打包依赖项、与 CUDA 作斗争、使其高效运行等等。让某些东西在生产环境中运行需要数周时间。

以我 Docker 的视角来看，Andreas 描述的这些问题都是 Docker 形状的问题。

如果我们能为机器学习模型定义一个标准盒子，那么研究人员就可以把他们的模型放进去。它们可以与他人共享。它们可以部署到生产环境。它们可以在任何地方运行，并且始终保持运行。

## 机器学习就是软件

很多人想用机器学习构建东西，但他们没有使用它的专业知识。阻碍机器学习普及的不是技术，而是您需要所有这些专业知识才能使用它。

软件工程师的数量大约比机器学习工程师多两个数量级（约3000万对50万）。我们认为，通过构建优秀的工具，软件工程师将能够像使用普通软件一样使用机器学习。

你应该能够像导入npm包一样导入音频转录器。你应该能够像在GitHub上复刻项目一样轻松地微调GPT。

二十年前，人们可能会说他们要构建一个"互联网应用"。现在我们不再这样说了。互联网只是做事的方式。

很快，机器学习也会如此。它将成为软件开发的方式。一切事物都将融入智能。

而这一切将由软件工程师来构建。
