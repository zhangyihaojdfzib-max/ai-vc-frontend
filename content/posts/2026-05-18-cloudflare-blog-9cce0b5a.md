# Project Glasswing：Mythos 给我们的启示

2026-05-18

- Grant Bourzikas

![](/images/posts/d5e95e7f6857.png)

过去几个月，我们一直在自有基础设施上测试一系列专注于安全的 LLM。这些 LLM 帮助我们识别自身系统中的潜在漏洞，以便我们进行修复——同时它们也向我们展示了攻击者能够利用最新模型做到什么。

这些 LLM 中，没有哪个比 Anthropic 的 Mythos Preview 更受关注。几周前，我们受邀在 Project Glasswing 中使用 Mythos Preview。我们很快将其指向了五十多个自有代码仓库——看看它能发现什么，以及它是如何工作的。

本文分享了我们的观察结果，模型做得好和不好的地方，以及围绕它们的架构和流程需要如何改变，才能使其得以大规模使用。

## Mythos Preview 带来了哪些变化

Mythos Preview 确实向前迈进了一大步，在讨论其他内容之前有必要先明确这一点。我们已经用模型分析代码有一段时间了，从之前通用前沿模型所能做到的水平，到如今 Mythos Preview 的表现，这不仅仅是此前成果的改进。

这是一种不同类型的工具，执行着不同类型的工作，这使得与早期模型进行纯粹的同类比较变得困难。因此，与其试图将 Mythos Preview 与通用前沿模型进行基准测试，不如描述它实际能做什么，以及我们在使用 Mythos Preview 的工作中突出的两个特性更有意义：

- **漏洞利用链构建**——真正的攻击很少只使用一个漏洞。它会将多个小型攻击原语串联成一个可用的利用程序。例如，它可能将一个释放后使用漏洞转化为任意读写原语，劫持控制流，并使用返回导向编程（ROP）链来完全控制系统。Mythos Preview 可以获取多个这样的原语，并推理如何将它们组合成一个可用的证明。它在此过程中展示的推理过程看起来像是一位资深研究员的工作，而非自动化扫描器的输出。
- **证明生成**——发现漏洞和证明其可利用性是两回事，而 Mythos Preview 两者都能做到。它会编写能够触发疑似漏洞的代码，在临时环境中编译并运行该代码。如果程序按模型预期运行，那就是证明。如果没有，模型会读取失败信息，调整假设，然后重试。这个循环与它发现的漏洞同样重要，因为一个没有可用证明的可疑缺陷只是猜测，而 Mythos Preview 能自行弥合这一差距。

漏洞利用链构建——真正的攻击很少只使用一个漏洞。它会将多个小型攻击原语串联成一个可用的利用程序。例如，它可能将一个释放后使用漏洞转化为任意读写原语，劫持控制流，并使用返回导向编程（ROP）链来完全控制系统。Mythos Preview 可以获取多个这样的原语，并推理如何将它们组合成一个可用的证明。它在此过程中展示的推理过程看起来像是一位资深研究员的工作，而非自动化扫描器的输出。

证明生成——发现漏洞和证明其可利用性是两回事，而 Mythos Preview 两者都能做到。它会编写能够触发疑似漏洞的代码，在临时环境中编译并运行该代码。如果程序按模型预期运行，那就是证明。如果没有，模型会读取失败信息，调整假设，然后重试。这个循环与它发现的漏洞同样重要，因为一个没有可用证明的可疑缺陷只是猜测，而 Mythos Preview 能自行弥合这一差距。

我们上面描述的某些能力并非 Mythos Preview 独有。当我们通过同样的测试框架运行其他前沿模型时，它们也发现了相当数量的相同底层漏洞，在某些情况下，它们在推理方面也比我们预期的走得更远。它们的不足之处在于将各个部分拼接在一起。一个模型可能会识别出一个有趣的漏洞，写出关于其重要性的详尽描述，然后就停止了，留下未完成的利用链和关于可利用性的未解问题。Mythos Preview 带来的变化是，模型现在可以将这些低严重性漏洞（传统上会默默积压在待办事项中）串联成一个更严重的利用程序。

## 模型在合法漏洞研究中的拒绝行为

Anthropic 作为 Project Glasswing 一部分提供的 Mythos Preview 模型，不具备公开可用模型（如 Opus 4.7 或 GPT-5.5）中存在的额外安全防护措施。

尽管如此，该模型会自发地对某些请求进行抵制——就像使其在漏洞搜寻中有用的网络能力一样，该模型有其自身涌现出的护栏，有时会导致它拒绝合法的安全研究请求。但正如我们所发现的，这些自发的拒绝并不一致——同样的任务，以不同的方式表述或在不同的上下文中呈现，可能会产生完全不同的结果，如下面的例子所示。

例如，该模型最初拒绝在一个项目上进行漏洞研究，然后在项目环境发生无关更改后，同意对同一代码进行同样的研究。被分析的代码没有任何变化。

在另一个案例中，该模型发现并确认了一个代码库中的几个严重内存漏洞，然后拒绝编写演示利用程序。同样的请求，以不同方式表述，得到了不同的答案，甚至相同的请求在不同运行中也可能因模型的概率特性而产生不同结果。语义上等价的任务，根据向模型呈现的方式和时间，可能产生相反的结果。

这一点很重要，因为虽然模型自发的拒绝/护栏是真实存在的，但它们不够一致，无法单独作为完整的安全边界。这正是为什么任何未来公开可用的、有能力的网络前沿模型，都必须在此基线行为之上包含额外的安全防护措施——使其适合在像 Project Glasswing 这样的受控研究环境之外更广泛地使用。

## 信噪比问题

安全漏洞分类中最困难的部分之一是判断哪些漏洞是真实的，哪些是可利用的，以及哪些需要立即修复。即使在 AI 出现之前，这也是一个难题。AI 漏洞扫描器和 AI 生成的代码使情况变得更糟，在 Cloudflare，我们构建了多个后验证阶段来处理这个问题。

有两个因素主导了噪声率：

- **编程语言**——C 和 C++ 提供了直接的内存控制，随之而来的是内存安全语言（如 Rust）在编译时消除的漏洞类别——缓冲区溢出、越界读写。我们看到，使用内存不安全语言编写的项目产生的误报始终更多。
- **模型偏差**——优秀的人类研究员会告诉你他们发现了什么以及他们的信心程度。模型不会。让模型寻找漏洞，它就会找到漏洞，无论代码中是否真的有。发现结果会带有"可能"、"潜在"、"理论上可以"等修饰词，而这些带有修饰词的发现数量远远超过可靠的发现。对于探索性工具来说，这是一个合理的偏差。但对于分类队列来说，这是一个毁灭性的偏差，因为每一个推测性的发现都需要消耗人力和 Token 来排除，而这种成本在数千个发现中会不断累积。

编程语言——C和C++赋予你直接的内存控制能力，同时也带来了缓冲区溢出、越界读写等漏洞类型，而Rust这类内存安全语言在编译阶段就能消除这些问题。我们发现，使用内存不安全语言编写的项目始终会产生更多误报。

模型偏差——优秀的人类研究员会告诉你发现了什么以及他们的置信度。但模型不会。让模型寻找漏洞，无论代码中是否存在漏洞，它都会给出结果。返回的结果中充斥着"可能"、"潜在"、"理论上可以"等模糊表述，而这些模糊发现的数量远超可靠发现。对于探索性工具而言，这种偏差尚可接受；但对于分类队列来说却是灾难性的——每个推测性发现都需要消耗人力和Token来排除，而这一成本会在数千个发现中不断累积。

Mythos Preview在这方面实现了显著改进，尤其是在组合原语的能力上——将多个漏洞整合为可运行的概念验证，而非孤立地报告它们。附带PoC的发现意味着你可以直接采取行动，也大大减少了"这到底是不是真的？"的验证时间。

我们的测试框架特意调整为过度报告，因此我们能看到更多（同时漏报更少），但这也会带来更多噪声。不过在分类阶段，Mythos Preview的输出质量明显更高：模糊表述更少，复现步骤更清晰，做出修复或排除决策所需的工作量也更少。

## 为什么通用编码Agent无法直接用于代码仓库

去年我们首次尝试AI辅助漏洞研究时，本能反应很直接：让通用编码Agent分析任意代码仓库并发现漏洞。这种方法确实能让模型产出发现，但无法实现对真实代码库的有意义覆盖并识别有价值的发现。主要原因有两个：

- **上下文**——编码Agent针对单一聚焦的工作流进行优化：构建功能、修复缺陷、编写重构代码。它们会摄入大量源代码，每次只持有一个假设并迭代验证。这与漏洞研究的本质需求恰好相反——漏洞研究需要窄聚焦且并行推进。人类研究员会选择某个具体方向深入调查，这个方向可能是单个复杂功能、跨安全边界的转换，或是特定漏洞类型（如命令注入——攻击者输入最终被当作shell命令执行）。然后他们会针对不同功能、安全边界或漏洞类型重复这个过程，在代码库中执行数千次。单个Agent会话（即使包含子Agent）面对十万行代码的仓库，在模型上下文窗口填满并触发压缩之前，可能只能有效覆盖不到千分之一的攻击面——而压缩过程可能丢弃早期的重要发现。

- **吞吐量**——单流Agent一次只能处理一件事，但真实代码库需要同时对多个组件提出大量假设，并在发现有趣线索时具备进一步扩展的能力。你可以让单个Agent更努力地工作，但到某个节点后，限制因素不再是模型本身，而是交互形态本身。在研究员已有线索并希望获得第二视角时，直接在编码Agent中使用模型进行手动调查是可行的。然而，对于实现高覆盖率而言，这是错误的工具。一旦我们认识到这一点，就不再试图让Mythos Preview执行错误的任务，而是围绕它构建测试框架。

## 测试框架真正解决了什么

通过大规模运行工作，我们总结出四条经验，每一条都指向需要能够管理整体执行的测试框架：

- **窄范围产生更优发现**——告诉模型"在这个仓库中查找漏洞"会让它漫无目的。告诉它"在这个特定函数中查找命令注入，上方有信任边界，这里是架构文档和该区域的先前覆盖情况"则会让它更接近研究员的实际工作方式。

- **对抗性审查降低噪声**——在初始发现与分类队列之间添加第二个Agent——使用不同的提示词、不同的模型，且不具备生成自身发现的能力——能够捕获第一个Agent在自我检查时可能遗漏的大量噪声。事实证明，让两个Agent刻意保持分歧比单纯告诉一个Agent要谨慎有效得多。

- **跨Agent拆分推理链提升推理质量**——询问"这段代码有缺陷吗？"和"攻击者能否从系统外部实际触发这个缺陷？"是两个不同的问题，分别提问时模型在每个问题上的表现都更好，因为每个问题都比组合版本更聚焦。

- **并行窄任务优于单一穷举Agent**——当多个Agent同时处理范围严格限定的问题，随后对结果进行去重时，覆盖率会得到提升，这比要求单个Agent穷举所有可能性效果更好。

将任务拆分给多个Agent（智能体）能产生更好的推理——“这段代码有bug吗？”和“攻击者能从系统外部实际触及这个bug吗？”是两个不同的问题，当分别提问时，模型对每个问题的回答效果更好，因为每个问题都比合并后的版本更聚焦。

并行处理窄任务优于单个穷举式Agent（智能体）——让多个Agent（智能体）处理范围明确的问题，然后对结果去重，比让一个Agent（智能体）穷举所有情况能获得更好的覆盖率。

上述每一点观察都关乎模型行为，综合起来，它们描述的已不再是聊天界面。而是一种能帮助你达成最终结果的工具框架。构建这种框架的第一步很简单，你可以让模型提供帮助，我们正是这样做的。我们利用Mythos Preview来构建、定制并优化原有的框架，使其充分发挥优势。

## 这对安全团队意味着什么

其他安全领导者对Mythos Preview最强烈的反应集中在速度上——扫描更快、修补更快、压缩响应周期。我们接触过的多个团队目前正遵循从CVE发布到生产环境打补丁两小时SLA的流程。这种本能可以理解：当攻击者的时间线缩短时，防御者的时间线也必须随之缩短。但仅靠更快是不够的，我们认为许多团队即将花费大量时间、精力和金钱，通过惨痛教训认识到这一点。

更快的修补并不能改变产生补丁的流程本身。如果回归测试需要一天时间，那么不跳过它就无法达到两小时的SLA，而跳过回归测试所引入的bug往往比原本试图修补的bug更严重。当我们尝试让模型自行编写补丁时，就曾经历过类似教训——有些补丁修复了原始bug，却悄悄破坏了代码依赖的其他功能。

更棘手的问题在于漏洞周围的架构应该是什么样子。原则是即使存在bug，也要让攻击者更难利用它，从而降低漏洞披露到修补完成这段时间的风险。这意味着要在应用程序前端部署防御措施，阻止攻击者触及bug；意味着设计应用程序时，确保代码某一部分的缺陷不会让攻击者获得其他部分的访问权限；还意味着能够同时将修复方案部署到所有运行该代码的位置，而不是等待各个团队单独部署。

我们也认识到这个话题具有两面性。帮助我们发现自己代码中bug的相同能力，若落入不当之手，将加速对互联网上所有应用程序的攻击。Cloudflare位于数百万个此类应用程序的前端，上述架构原则正是我们产品为客户所应用的核心。未来几周，我们将进一步分享这对客户的意义。
