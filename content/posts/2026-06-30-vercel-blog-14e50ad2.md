今天，我们推出 **Vercel Services**，它允许你在一个 Vercel 项目中运行多个框架。这实现了：

- **原子化部署**：你的前端、后端及其他服务保持同步，并一同部署或回滚
- **共享预览部署**：查看任何变更如何影响你的所有服务
- **内部服务通信**：服务之间可以相互通信，无需经过公共互联网路由

### 链接到标题服务绑定

服务可以在内部相互通信，无需经过公共互联网路由。以下示例展示了新的 `bindings` 配置键如何实现这一点：

```
1{
2  "services": {
3    "my_frontend": {
4      "root": "frontend/",
5      "framework": "nextjs",
6      "bindings": [
7        {
8          "type": "service",
9          "service": "my_backend",
10          "format": "url",
11          "env": "BACKEND_INTERNAL_URL"      
12        }
13      ]
14    },
15    "my_backend": { ... }
16  },
17  "rewrites": [ ... ]
18}
```

绑定将 `BACKEND_INTERNAL_URL` 注入到前端，指向通过 Vercel 内部网络的后端

现在，JavaScript 前端代码可以通过存储在 `BACKEND_INTERNAL_URL` 环境变量中的 URL 在内部与 Python 服务通信：

```
1export async function GET() {
2  const url = new URL("/users", process.env.BACKEND_INTERNAL_URL);
3  const res = await fetch(url);
4  const users = await res.json();
5  return Response.json(users);
6}
```

前端使用注入的内部 URL 调用后端。流量永远不会离开 Vercel 的网络。

服务到服务的流量保持在 Vercel 网络上，而不是出口到公共互联网。许多独立服务可以成为一个通过相同线路连接的应用，而不是你在不同主机间拼接的独立部署。

### 链接到标题框架定义的基础设施

大多数框架在 Vercel 上零配置即可运行。**框架定义的基础设施**意味着每个服务的框架会被自动检测和自动配置，从 Python 的 **FastAPI** 和 **Flask**，到 TypeScript 的 Express 和 **Hono**，并对 **Go** 和 **Rust** 服务器提供一流支持。

服务运行在 Fluid compute 上，随流量自动扩缩容，而你只需为活跃的 CPU 时间付费。但框架定义的优化更进一步。例如，使用 Django 时，我们会自动检测静态资源的位置，并从 CDN 提供它们。

## 链接到标题全栈平台

Vercel Services 为 Vercel 提供了一种结构化的方式来构建和运行你的应用。

这些服务运行在一个包含任何后端所需一切的平台上，无论你是在构建 API、Agent（智能体）还是后台工作进程：计算、数据、网络、后台工作，以及与外部服务的安全连接。

### 链接到标题为 Agent（智能体）服务提供的隔离计算

Agent（智能体）可以读取文件、运行命令和编写代码。但这种能力需要是安全的。**Vercel Sandbox** 为每个 Agent（智能体）提供其自己的 Linux 计算机：文件系统、Shell、**Docker 支持**以及其自己的内核，与你的部署完全隔离。Agent（智能体）可以执行代码、grep 文件，并启动 Redis 或 Postgres 作为依赖项。沙箱内的任何内容都无法触及你的生产环境。

通过**自动持久化**，状态可在会话之间保持，在 Pro 计划中，沙箱最长可运行 24 小时。

### 链接到标题支持 WebSocket 的实时后端

Vercel Functions 在每个 Vercel 运行时中处理持久的 WebSocket 连接，无论你的后端是用 Node.js、Python 还是 Go 编写的。它们与标准 WebSocket 库（如 **Socket.IO**）配合使用，并且由于连接运行在采用活跃 CPU 计费的 Fluid compute 上，你只需为处理消息的时间付费，而无需为连接空闲的时间付费。

### 链接到标题使用 Vercel Connect 访问外部服务

Agent（智能体）和后端需要访问项目外部的服务（Slack、GitHub、托管数据库），这通常意味着在你的环境中存储一个长期有效的密钥。**Vercel Connect** 用你的应用在运行时请求的短期凭证（范围限定于其面临的任务）来替代它。没有长期有效的密钥会泄露。

### 链接到标题为你的服务提供的数据库和存储

你可以从 **Vercel Marketplace** 中点击几下配置一个数据库，凭证会自动注入，提供商包括 Neon、Supabase，以及来自 AWS 的 Aurora PostgreSQL、Aurora DSQL 和 DynamoDB。**Amazon OpenSearch Serverless** 提供全文和向量搜索，**Vercel Blob** 处理对象存储。你可以在不离开 Vercel 的情况下**查询和管理**其中的任何内容，并且 Blob 和 AWS 集成都使用短期 OIDC 令牌进行身份验证，而不是存储的密钥。

### 链接到标题运行持久化工作流和后台任务

**Vercel Queues** 在请求路径之外处理后台任务。**Vercel Workflows** 处理能够承受崩溃和重新部署的持久化多步骤流程。**Vercel Cron** 按计划运行任务。这是后端中生命周期超过请求的部分，它与其他所有内容运行在同一个平台上，因此你后端中长时间运行的部分无需单独的基础设施。

### 链接到标题私有连接并在合适的计算资源上运行

当后端需要访问私有数据库或内部网络（而非公共服务）时，**Secure Compute**、**静态 IP** 和 **VPC 对等连接** 打开了这条路径。

在 Pro 和 Enterprise 计划中，函数最长可运行 **30 分钟**，Python 后端部署时最多可包含 500mb 的依赖项。

在这一切之下，**Fluid compute** 是新项目的默认选项，而活跃 CPU 计费按代码实际运行的时间（而非等待时间）收费，这对于空闲密集型后端和 AI 工作来说是合适的选择。
