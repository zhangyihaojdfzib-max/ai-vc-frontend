*Unity Catalog 现已成为最全面、可互操作且可投入生产的 Apache Iceberg 目录，Managed Iceberg、Iceberg v3 和 Foreign Iceberg 均已正式发布。*五大能力使其脱颖而出：开放 API、目录联邦、跨引擎访问控制、零拷贝安全共享以及 AI 驱动的优化。*展望未来，Iceberg v4 和 Delta 5.0 将融合为统一的元数据结构，终结互操作性与生产级性能之间的取舍。

开放湖仓的下一个阶段将由目录来定义。开放表格式使众多引擎能够处理同一份数据，但目录决定了这些数据能否跨系统进行一致的治理、优化和共享。随着越来越多的工作负载（包括 AI 和 Agent 应用）依赖于跨多个系统的受控数据访问，企业需要一个既能提供互操作性、卓越性能，又能提供企业级治理能力的 Iceberg 目录。

正因如此，今天我们宣布推出任何湖仓目录中功能最全面的 Iceberg 能力集。在本文中，我们将讨论 Unity Catalog 中 Iceberg 支持的新增强功能，并解析使 Unity Catalog 成为当今市场上互操作性最强的 Iceberg 目录的五大要素。

## 使 Unity Catalog 成为互操作性最强的 Iceberg 目录的五大要素

要交付一个完全开放的湖仓，Iceberg 目录必须超越基本的元数据追踪。它需要为您提供跨不同引擎、供应商和治理模型的绝对灵活性。我们相信，评估一个开放的 Iceberg 目录，关键在于它能在多大程度上满足五个基本运营需求：提供开放 API、跨外部资产联邦、实施跨引擎治理、实现安全开放的共享，以及持续的性能和格式创新。

Unity Catalog 是唯一能够满足全部五项需求的目录。

![Iceberg 目录对比](/images/posts/80e568ffaa66.png)

### 1. 开放 API 与凭证分发

客户应该能够使用最适合工作负载的引擎，无论是 Spark、Trino、Flink、Snowflake、DuckDB、pandas 还是其他兼容 Iceberg 的客户端，而无需复制数据或为每个引擎授予广泛的存储权限。

随着 Managed Iceberg 在 Databricks 上正式可用，客户可以使用 UC 的 Iceberg REST Catalog API，从任何引擎在 Unity Catalog 中创建、读取和写入 Iceberg 表。

UC 的 Iceberg REST Catalog API 现在也扩展到了 Managed Iceberg 表之外。UC 还为联邦 Iceberg 表分发凭证，通过开放 API 为即使在外部目录中管理的表提供安全访问。此外，在当前的受限公开预览中，客户可以在 Databricks 中创建物化视图，并将其作为 Iceberg 表暴露给下游消费者。随着未来几周更广泛的可用性，客户将能够直接使用 `CREATE MATERIALIZED VIEW my_mv USING ICEBERG` 创建兼容 Iceberg 的物化视图。

### 2. 目录联邦：您的整个 Iceberg 资产尽收眼底

许多大型企业在他们的湖仓中拥有多个目录。例如，他们的数据可能分布在 Unity Catalog、AWS Glue、Snowflake Horizon 和 Hive Metastore 中。随着 Foreign Iceberg 现已正式发布，Unity Catalog 可以治理在其他目录中管理的 Iceberg 表。客户可以通过 Databricks 发现、保护、查询和共享外部 Iceberg 表，同时将数据和源目录保留在原位。

Unity Catalog 现在支持广泛且不断增长的 Iceberg 目录集成，包括 AWS Glue、Google Cloud Lakehouse Runtime Catalog、Snowflake Horizon、Palantir、Salesforce 和 Workday。这些集成使企业能够将 Unity Catalog 视为其 Iceberg 资产的单一管理视图，即使数据是在其他地方产生或管理的。

### 3. 跨引擎基于属性的访问控制

以往，行级和列级控制是在单个引擎内部实施的。在开放湖仓中，同一个表可以被多个引擎访问。这引入了一个难题：治理需要在数据可被访问的任何地方发挥作用。

随着跨引擎基于属性的访问控制（ABAC）现已进入 Beta 阶段，Unity Catalog 使用 Iceberg REST Catalog Scan API，将基于属性的访问控制扩展到 Iceberg 客户端。

工作原理：管理员在 UC 中一次性定义策略，包括列掩码、行过滤器和基于标签的策略。当外部 Iceberg 引擎请求访问时，UC 在服务端扫描计划期间评估适用的策略。然后 UC 返回一个过滤后的扫描计划，使得引擎在处理查询时只读取授权数据。

这通过开放标准将细粒度治理带到了外部 Iceberg 引擎。任何实现了 Iceberg REST 目录扫描计划客户端（在 Iceberg 1.11 版本中添加）的引擎，例如 Apache Spark 或 DuckDB，都可以在实施 ABAC 的情况下访问数据。客户可以为每个工作负载使用最佳引擎，同时在湖仓中维护一个统一的治理模型。

### 4. 用于外部和跨域协作的零拷贝安全共享

跨域共享常常迫使数据提供者做出糟糕的取舍：将数据复制到另一个平台，构建复杂的外部认证机制，或者要求每个接收者使用相同的供应商生态。Databricks 通过 Delta Sharing 开创了安全开放的数据共享，这是数据与 AI 共享领域采用最广泛的开源协议——同时支持 Databricks 到 Databricks 以及 Databricks 到开放平台的共享。

我们很高兴地宣布，Iceberg 现在已成为 Databricks DeltaSharing 中的一等公民，既可作为源格式，也可作为目标格式。随着向 Iceberg 客户端共享现已正式可用，Databricks 客户可以与任何支持 Iceberg REST Catalog API 的接收者外部共享实时数据。接收者可以从兼容 Iceberg 的客户端（如 Snowflake、Trino、Flink 和 Spark）查询共享数据，无需手动摄取或复制。提供者继续通过 Unity Catalog 管理访问、审计和治理。

我们还宣布了外部Iceberg共享的公测版。客户可以共享在Databricks外部管理或编目、但在Unity Catalog中注册和治理的Iceberg表。这意味着UC可以作为托管和外部Iceberg表的共享层，同时保持数据原位存储和治理集中化。

### 5. 性能与格式创新：无需手动调优即可实现更快的开放表

开放互操作性只有在表在生产规模下保持高性能时才有效。Unity Catalog是唯一使用AI优化表以加快查询速度并降低运营开销的目录。预测性优化可确定哪些表需要维护、运行哪些优化以及运行频率，并根据工作负载模式调整表的数据布局。这减少了随着使用变化而保持Iceberg表快速且成本效益所需的运营工作，并且这些优化使所有引擎受益——例如，数据布局优化技术可改善在Databricks外部（如Apache Spark）运行的查询的数据跳过能力。我们持续在客户体验上进行创新——并且是唯一能够根据历史访问模式智能选择聚类键以实现最佳性能，或自动用最新创新升级开放表的目录。

Databricks也在推进Iceberg标准本身。随着Iceberg v3现已在Databricks上全面可用，客户可以在托管Iceberg表、外部Iceberg表和启用UniForm的托管表中获得对删除向量、行追踪和VARIANT的支持。这些能力填补了性能与互操作性之间的重要差距：删除向量加速更新、合并和删除操作；行追踪支持更高效的增量处理；VARIANT为半结构化数据提供了标准表示。这些功能在Delta和Iceberg表之间无缝运行，实现了无需重写数据的互操作性。

这些投入指向同一个目标：开放表不应迫使客户在生态系统互操作性与生产工作负载所需的性能能力之间做出选择。

综合来看，这五项能力使Unity Catalog成为Apache Iceberg的最佳目录。UC为客户提供对Iceberg表的开放访问、跨目录的统一视图、跨引擎的细粒度治理、跨域的安全共享以及生产工作负载的自动优化。
