我们最近发布了 Omnigent，一个面向 AI Agent（智能体）的开源元框架。它让你能够继续使用你喜欢的 Agent（智能体）框架，包括 Claude Code、Codex 以及自定义 Agent（智能体），同时增加一个用于协作、组合和策略的共享层。

为了安全性和成本管理，Omnigent 引入了一个强大的新工具：**上下文策略**。如今的 Agent（智能体）框架只有简单的控制手段来限制 Agent（智能体）能做什么，例如允许、拒绝或询问用户各种工具调用的规则。但这使得创建既安全又对用户便捷的策略变得困难。相比之下，Omnigent 的上下文策略能够记住会话中已发生的事件（例如，Agent（智能体）读取了什么，或者到目前为止花费了多少美元），并利用这些状态来决定下一步行动是否应该执行。这使得一系列丰富的策略成为可能，对用户来说既更安全也更便捷：从动态追踪会话的风险等级，到实现最小权限安全模型，再到允许用户为单个任务设定预算以管理支出。

## 为什么带上下文的 Agent（智能体）控制更好

AI Agent（智能体）为企业带来了新型风险。例如，由于 Agent（智能体）可能被不可信内容**提示注入**并要求执行有害操作，因此需要防止同一个 Agent（智能体）读取不可信内容、访问敏感数据以及与外部世界通信（这被 Simon Willison 的“致命三重奏”和 Meta 的“Agent（智能体）二选一规则”所普及）。一个操作是否“安全”，部分取决于之前发生了什么：如果一个编码 Agent（智能体）刚刚为工程师处理了一个功能，那么它推送到 GitHub 通常是没问题的；但如果该 Agent（智能体）之前下载了一个可能包含注入攻击的不可信网页，那么同样的 GitHub 推送就存在风险。

不幸的是，目前大多数 Agent（智能体）软件只提供简单的白名单控制或针对单个操作的防护栏，例如是否允许 Git 推送或网页搜索。为了防止注入攻击，我们不得不完全阻止这些操作中的至少一个，但这对于许多无害的用例来说限制性太强。要求用户批准每个操作效果也不好，因为用户会被批准提示弄得疲惫不堪。

同样的情况也适用于其他场景。也许让销售代表的 Agent（智能体）向客户发送一封邮件没问题，但让它发送数千封邮件则表明存在入侵或错误。一个 Agent（智能体）编辑它自己创建的文档是可以的，但同一个 Agent（智能体）编辑数千个内部文档则可能需要审查。事实上，许多面向人类用户的安全工具也会考虑他们的历史记录，而不仅仅是当前操作（这被称为上下文安全）。

## 引入上下文策略

在 Omnigent 中，一个**策略**可以监听 Agent（智能体）正在执行的事件（例如，工具调用和响应，以及 LLM 的输入和输出），并决定是允许、拒绝、转换消息，还是请求用户许可，这与传统的 Agent（智能体）防护栏类似。然而，该策略还可以更新关于会话的**状态**：仅对该策略可见的任意变量。这包括追踪 Agent（智能体）使用特定工具的次数、它读取了哪些文档等。Omnigent 服务器会记住每个策略和会话的状态，并在下次调用策略处理程序时将其传递给它。要编写一个上下文策略，你只需编写一个函数，该函数接收旧状态和 Agent（智能体）尝试执行的新事件，并返回状态更新和决策。Omnigent 也附带了许多已经包含在内的有用策略。

此外，由于 Omnigent 是一个元框架，它可以以相同的方式将你的上下文策略应用于使用任何框架的 Agent（智能体）。Omnigent 支持广泛使用的**编码 Agent（智能体）**，如 Claude Code、Codex、Antigravity、Pi、OpenCode 和 Hermes，以及 OpenAI Agents SDK 和 Claude Agents SDK 等框架中的**自定义 Agent（智能体）**。只需通过 Omnigent 启动 Agent（智能体），Omnigent 服务器就会拦截它们的工具调用来应用策略。

## 示例用例

以下是 Omnigent 目前内置的三个示例策略。每个策略都依赖于一种不同的会话状态：Agent（智能体）到目前为止读取的内容、累积的会话风险评分，或当前会话的总成本。

### 1. Google Drive 策略：将 Agent（智能体）的访问权限限定在正确的文档

Google Drive 策略管理 Agent（智能体）在文档、表格和幻灯片中可以读取和修改的内容。默认情况下，写入操作仅限于 Agent（智能体）在此会话期间创建的文档。因此，Agent（智能体）可以创建一个新文档并自由编辑，但不能悄悄修改一个它本不该触碰的已有文件。这种行为无法通过简单的白名单来实现：我们不想完全允许或拒绝“写入文档”工具，我们只想允许它在 Agent（智能体）**在同一会话中**创建的文档上执行。

作为该策略中上下文行为的第二个示例，你可以将一组文档标记为机密：一旦 Agent（智能体）打开其中一个，策略就会收紧，使得写入操作仅限于该组文档。即使是 Agent（智能体）一分钟前创建的文档也变得不可访问，因为将机密材料附加到该文档会将内容泄露到一个保护较弱的文件中。在经典安全中，这实现了带有“禁止向下写入”规则的 Bell-LaPadula 模型。

### 2. 风险评分策略：当 Agent（智能体）接触敏感材料时更加谨慎

风险评分通常被安全团队用于管理人类用户的访问权限。在 Omnigent 中，风险评分策略为当前会话维护一个动态评分，这是一个单一数字，用于追踪 Agent（智能体）工作时累积的风险程度。用户可以配置哪些操作会提高评分以及提高多少：一个常规的工具调用可能会增加一两分，而读取你标记为高度机密的文档则会增加更多分数。只要评分保持较低，Agent（智能体）就可以不间断地工作。一旦超过阈值，发送邮件或共享文件等操作将返回“询问”而不是“允许”，并提示用户批准。因此，在会话早期本可以自动发出的同一封邮件，在 Agent（智能体）处理了足够多的敏感材料导致评分升高后，可能需要人工批准。

### 3. 成本策略：控制支出

预算策略会追踪会话在模型调用上的累计花费。当超过软阈值时，系统会暂停并询问是否继续。当花费达到硬上限时，该策略会阻止对昂贵模型的进一步调用，直到Agent（智能体）切换到更便宜的模型，从而让会话得以继续而非中断。同样的思路也适用于单个会话之外。平台团队可以在每个会话的预算上限之上叠加每个用户的每日上限，这样任何人都无法通过多次独立对话累积高昂成本。在这两种情况下，上下文都是累计花费：策略并非评判某一次模型调用，而是监控累计总额，并在会话或用户花费过多时介入。

### 4. 基于意图的授权：根据目的限制权限

基于意图的授权根据用户对Agent（智能体）的初始提示词来设置其权限，这样即使遭遇提示词注入攻击，Agent（智能体）也无法使用大部分工具造成危害。例如，如果你通过要求Agent（智能体）更新Google Slides演示文稿来启动会话，策略会允许该Agent（智能体）读取和写入该演示文稿，但如果Agent（智能体）突然尝试使用GitHub，则会阻止访问。该策略只是将会话开始时用户实际请求的内容作为状态记录下来，然后根据原始目标检查每一次工具调用，应用最小权限原则。这是一种由上下文状态实现的非常简单但强大的策略：你可以默认给Agent（智能体）配置许多工具，它们会在每个会话中自动被锁定。同一个工具调用在一个会话中可能没问题，但在另一个会话中可能需要你的批准，具体取决于你要求Agent（智能体）执行的任务。

## 要点

随着Agent（智能体）承担更多实际工作，难点转向了如何控制它们。Agent（智能体）可能因恶意输入或诚实的错误而开始造成危害，而简单的白名单策略（仅检查单个操作）不够灵活，无法让Agent（智能体）既可用又安全。上下文策略通过动态追踪会话内的状态，并仅在Agent（智能体）累计风险足够高时阻止操作，从而实现了更可用且更安全的Agent（智能体）。它们还提供了一种强大而简单的工具，用于在会话级别而非仅按用户每日级别管理预算。Omnigent是开源的，并与最流行的编码Agent（智能体）和Agent（智能体）框架集成，因此你可以将这些策略应用于现有的Agent（智能体）。
