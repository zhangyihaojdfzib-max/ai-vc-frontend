# 如何将核心服务器启动时间从数小时缩短至数分钟

2026-06-01

- Giovanni Pereira Zantedeschi

- Omar Sheik-Omar

![](/images/posts/ceb0305c06a7.png)

Cloudflare 的核心是运行我们控制平面、计费和分析功能的集中式数据中心——与处理用户流量的全球分布式边缘节点不同。核心服务器采用裸金属架构，当重启过程中出现问题时，后果可能迅速蔓延。

它们的启动序列由 UEFI 编排，这是现代固件标准，负责初始化硬件并将控制权移交给操作系统。该交接过程中的微小异常可能产生巨大影响。

在一次常规固件更新后，我们部分核心服务器需要**四小时**才能恢复在线，而此前只需数分钟。原本应在一日内完成的整个机群部署，变成了持续数日的苦战。新节点在首次启动时就要经历完整的超时考验。维护窗口被大幅延长。工程团队不得不全程监控本应无人值守的升级。

此问题影响了整个 **Gen12 机群**——近 2000 台设备。升级过程中的每一次意外失败都意味着重启整个周期，而新容量只能闲置等待超时考验结束。

这就是我们如何将原因追踪到固件异常和过于激进的线性搜索（遍历每个可用网络启动接口），以及如何将总启动和升级时间从数小时缩短回数分钟的故事。在此过程中，我们将分享关于 UEFI 内部机制、供应商特定异常以及最终解决问题的自动化策略的经验。

### 网络启动接口

网络启动接口允许服务器通过网络而非本地存储启动操作系统。这对于集中化、自动化、可扩展的机器启动控制至关重要，尤其是在服务于不同工作负载的全球分布式机群中。由于我们的服务器位于不同环境并服务于不同目的，它们对特定网络启动接口有不同需求。两个主要接口是**预启动执行环境 (PXE)** 和**统一可扩展固件接口 (UEFI) HTTPS 启动**。

作为重启流程的一部分，出于各种自动化原因，我们的服务器通常通过 PXE 启动。在 Cloudflare，我们使用**开源 iPXE**，这是一种支持 HTTP 和 HTTPS 等现代协议的开源网络启动固件。这使得计算机能够直接从 Web 服务器、云或企业存储网络启动操作系统，速度显著更快，可靠性更高。

对于组织而言，iPXE 将启动过程转变为可编程工作流。它提供高级脚本功能，使 IT 团队能够自动化复杂部署，例如基于特定硬件配置配置服务器或管理安全的无盘工作站。

我们部分硬件支持基于 HTTPS 的 UEFI 网络启动，这使得计算机主板固件能够原生安全地下载操作系统文件。

### 线性搜索

我们的故事始于那次决定性的固件更新。更新后，内部渠道传来了首批报告：服务器无法恢复在线。监控仪表板显示机器长时间停留在预操作系统状态，远超预期。我们最初怀疑是固件回归：也许更新本身引入了导致启动过程挂起的错误。

为排除此可能性，我们调取了受影响机器的串行控制台，实时观察了一次启动周期。固件加电自检 (POST) 正常完成，硬件初始化看起来健康。但随后，系统并未快速进入网络启动阶段并下载操作系统镜像，而是停滞等待。等待。

控制台输出了真相：系统尝试 IPv4 HTTPS 网络启动，数分钟后超时，然后尝试 IPv4 iPXE，再次超时，然后重复两者——最终才到达实际能成功的 IPv6 HTTPS 启动接口。

每次失败的网络启动尝试大约浪费五分钟等待超时响应。在到达正确接口之前，四次尝试累积起来，单次启动周期浪费约二十分钟。对于常规重启，这已令人痛苦。对于需要多次顺序重启（每个组件一次）的固件升级自动化而言，这些二十分钟的惩罚累积成每台服务器近四小时的空闲等待。

## 无需搜索游戏：声明我的启动接口

在追踪启动序列并隔离超时模式后，根本原因变得清晰：服务器盲目地逐一搜索每个可用网络启动接口，等待每个接口失败后才继续。解决方案是完全消除猜测——预先声明正确的启动接口，使系统永远不会在永远不会响应的接口上浪费时间。

但将其付诸实践远非易事。正如我们接下来解释的，我们遇到了几个障碍：启动自动化工作流的顺序、一个我们无法更改的设置，以及来自不同网卡供应商的不同字符串格式。

### 我们的启动自动化工作流

我们的启动自动化流程分为三大阶段：固件初始化、预启动和内核启动。加电后，UEFI 固件执行一些硬件和外设初始化，随后进入 PXE 预启动环境。预启动阶段设置网卡并执行一个名为引导加载程序的小程序，该程序启动内核。正是在这个 PXE 阶段，系统探测各种网络接口以找到正确的接口。在首次启动时，固件升级包含在我们的启动自动化工作流中。

由于每次固件升级都需要一次重启（及其伴随的网络启动尝试序列），这就导致了总启动时间接近四小时的情况。

通过重构自动化序列，在预启动 PXE 阶段早期为每个硬件/用例声明网络启动接口顺序，我们成功将总时间缩短了约一小时，因为启动过程不再需要为每次固件升级花费 20 分钟进行探测。

尝试声明网络启动接口顺序引入了两个特定限制：

1.  **传统支持**：较旧版本的 UEFI 不支持启动顺序设置
2.  **持久性**：UEFI 固件升级后，配置设置通常会被重置

传统支持：较旧版本的 UEFI 不支持启动顺序设置

持久性：UEFI 固件升级后，配置设置通常会被重置

为处理这些边缘情况，我们实现了一个状态验证步骤。固件自动化现在会在变更后验证配置：如果检测到设置已被修改，它会重新应用配置并触发重启。

虽然首次启动可能稍长，但这一改变大幅减少了所有后续启动所需的时间，从约 20 分钟降至每次后续启动不到一分钟。

### 供应商禁用的启动顺序设置

网络启动设置的内部数据结构是一个 `EFI_IFR_REF3` 数据结构，该结构采用惰性加载，意味着数据只有在通过 GUI 回调显式访问时才会被实例化：

```C++
typedef struct _EFI_IFR_REF3 {
  EFI_IFR_OP_HEADER          Header;
  EFI_IFR_QUESTION_HEADER    Question;
  EFI_QUESTION_ID            QuestionId;
  EFI_GUID                   FormSetId;
} EFI_IFR_REF3;

```

虽然这是加速 BIOS 启动时间的行业标准做法，但它使得“网络启动接口”对我们的程序化扫描不可见。由于该结构尚未被“加载”，我们的自动化无法发现优先级设置。

我们与供应商合作，在固定的"启动顺序模块"中启用了特定Token。这使得在启动过程中无需手动进行图形界面交互即可发现网络启动接口。

来自设备制造商的UEFI固件中有一个不可更改的设置`Force Priority Httpv4 Httpv6 Pxev4 Pxev6`，这阻止了我们修改启动顺序。

这需要供应商提供新的BIOS版本，并在设置启动顺序时进行一次调试会话。

### 不同网卡厂商的不同字符串

根据网卡（NIC）厂商的不同，字符串也会有所不同，导致通过iPXE配置启动顺序时出现不匹配。

示例：

UEFI: HTTPS IPv4 Ethernet Network Adapter XXX-XXX-Y for OCP 3.0 P1
UEFI: HTTPS IPv4 Network Adapter - 50:00:E6:8F:4F:32 P1

为了解决这个问题，我们不得不为CfHIIConfig_App工具增加一项额外功能，使其能够在没有完整字符串的情况下进行配置：

.*HTTP.*IPv4.*P1

然后，配置将与可接受的配置字符串进行匹配，并选择正确的启动顺序。我们目前正在与UEFI供应商合作，标准化网络接口字符串，使其仅包含相关信息（例如协议、传输类型、端口号和物理插槽索引），并去掉MAC地址等产品细节。如果需要，可以从网卡内置的重要产品详细信息中读取这些产品细节。这样，我们既消除了配置漂移，也避免了使用通配符。

#### 无法通过iPXE检查配置

由于iPXE将此变量读取为十六进制，因此它将以十六进制形式读取字符串输出。为了检查网络启动设置是否已修改并减少启动时间（这样我们就不必在设置之前打印变量），我们实现了一个布尔标志`uefi-same-hex`，用于指示配置是否已更改。

这使我们能够执行单个`set`命令，而不是先运行`show`进行比较，然后在配置未达到预期状态时再执行`set`。

```Shell
# 构建路径以读取更新变量
set buffer-var-guid 91468514-75bc-4bb5-8f33-91efff9e9b1f
set var-upd-path efivar/CfHIIVarUpd-${buffer-var-guid}

# 运行配置更改命令
imgexec <signed CF UEFI configuration App> set ${uefi-setting}=${uefi-value}

# 如果更新变量已更改，则将其与预期值进行比较。
# 如果已更改，则设置本地变量以重启系统
iseq ${uefi-same-hex} ${${var-upd-path}} || set has-changed ${uefi-diff-hex}

```
