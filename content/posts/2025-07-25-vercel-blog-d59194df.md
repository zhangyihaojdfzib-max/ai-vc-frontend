模型上下文协议（MCP）是一种帮助标准化大语言模型（LLM）访问数据和系统的新方式，扩展了它们基于训练数据所能执行的能力。它标准化了开发者如何向模型和Agent（智能体）暴露数据源、工具和上下文，从而实现安全、可预测的交互，并充当AI与应用之间的通用连接器。

开发者无需为每个AI平台构建定制集成，只需创建一次MCP服务器，即可在各处使用。

## 什么是MCP？

MCP将AI模型（如Claude、GPT-4等）连接到外部工具和系统。这可以是你的应用API、产品数据库、代码库，甚至是桌面环境。

MCP允许你以模型能够理解的结构化方式暴露这些能力。MCP不是一个库或SDK。它是一个规范，类似于REST或GraphQL，但专为AI Agent设计。

模型继续依赖其自身的训练知识和推理能力，但现在它们可以通过MCP服务器访问专用工具来填补空白。当模型自身理解达到极限时，它可以调用真实函数、获取真实数据，并保持在您定义的防护栏内，而不是编造自己的答案（产生幻觉）。

因此，当它需要参考库存水平时，它不会做出假设。它是在查询你实际的库存系统。它获取的文档不是最后一次爬取的内容，而是你15分钟前合并到生产环境的最新版本。

## 为什么我要创建MCP服务器？

如果你希望让AI模型以受控方式访问你的应用或服务，你就会想要创建一个MCP服务器。这可能意味着让模型下订单、获取用户数据、写入文件或运行搜索。

其核心在于通过安全的方式让AI模型访问你的系统，从而扩展AI的能力。你无需寄希望于模型理解你的API文档，而是可以精确地定义它能做什么以及如何做。

## 我应该构建自己的MCP服务器，还是直接在应用中创建工具？

如果你只是为自己的应用或单个AI提供商构建功能，可以直接在LLM调用中定义你的工具。这更简单、更快，并且让你对执行拥有完全控制权。

MCP增加了一层：你的工具托管在你的应用之外，在一个独立的服务器上。当你调用LLM时，首先连接到MCP服务器，向模型提供上下文，然后使用模型选择的输入调用MCP服务器来执行。

当你希望你的上下文和工具能在多个应用、模型或环境中共享时，MCP服务器就很有意义。

## MCP服务器工具与API有什么区别？

API是为应用直接调用而设计的。开发者编写请求，将其发送到已知的端点，并获取可预测的响应。它是为确切知道自己要做什么以及何时去做的人类和程序构建的。

MCP服务器是为模型设计的。它提供元数据和结构化描述，帮助模型理解哪些工具可用、它们做什么以及何时使用它们。与需要硬编码调用的API不同，模型利用这个上下文来决定要做什么。

你可能已经拥有API、内部工具和数据库。MCP服务器以标准化的方式让AI能够访问所有这些。只需编写一次集成。任何兼容MCP的Agent都可以使用它。无需为每个模型编写自定义逻辑。没有供应商锁定。

## 创建MCP服务器最简单的方法是什么？

AI SDK内置了对MCP的支持，让你可以快速定义函数和工具。

你也可以使用我们的`mcp-handler`，这是一个开源SDK，可以轻松地将MCP服务器暴露给Next.js、Nuxt和Svelte应用。它很轻量，只需要几行代码，并且适用于大多数框架。

无论哪种方式，你都不需要理解MCP规范的每一个细节。你定义模型可以做什么以及可以访问什么，MCP服务器处理其余的事情。AI SDK和mcp-handler提供的抽象使得在几小时（或更短）内创建MCP服务器成为可能（我们用一个下午就给Grep添加了MCP服务器）。

## 如何使用MCP服务器？

你将一个客户端（如你的模型或AI宿主，例如Claude或Cursor）连接到MCP服务器。该连接允许客户端列出服务器暴露的能力、它们接受什么输入以及返回什么输出。

模型看到这些信息后，规划其调用，并在需要服务器时发送请求。这就像给模型一个可以从中挑选的工具箱，它根据提示词选择正确的工具。你不需要管理这个过程。模型读取MCP规范并决定使用什么。

## MCP如何为工具工作？

从高层次看，MCP服务器以机器可读的格式（通常是JSON）向模型暴露一个工具列表。每个工具都有名称、描述、输入模式和输出模式。这被称为工具清单。

当模型看到这个清单时，它可以通过调用这些工具来规划一系列操作，根据目标一次调用一个或循环调用。

每个工具调用都是从模型到MCP服务器的普通HTTP请求（或其他传输方式）。服务器运行工具逻辑并返回响应，模型随后可以使用该响应来决定下一步做什么。

MCP的核心是将请求视为函数调用。模型规划下一步。服务器运行工具并返回结果。这个循环持续进行，直到任务完成。

## MCP的"传输方式"是什么？

传输方式是模型与你的MCP服务器通信的方式。目前，StreamableHTTP是主要方式。模型使用标准HTTP访问URL并建立连接。可选地，它还允许在侧边使用服务器发送事件（SSE）进行实时通知。

另一种流行的传输方式是stdio，用于本地或命令行界面（CLI）环境，其中模型和工具共享同一个进程。

传输方式的选择取决于交互类型：一次性调用还是操作流。

像`mcp-handler`这样的包支持多种传输方式，因此很容易切换或支持多种方式。

## AI Agent和MCP服务器有什么区别？

AI Agent是进行规划和采取行动的模型（使用模型的客户端或系统）。MCP服务器是定义可能操作的环境。Agent决定做什么。MCP服务器告诉它如何做。

可以这样理解：

- 厨师决定做什么（AI Agent）
- 厨房决定可用的工具和食材（MCP服务器）
- 厨师只能使用厨房提供的东西工作

你需要两者兼备。Agent理解意图。MCP服务器确保它保持在正轨上并使用批准的工具。服务器不采取行动。它只是向模型展示什么是可能的。

## 本地和远程MCP服务器有什么区别？

区别在于它们运行的位置以及彼此通信的方式。

本地MCP在与模型相同的机器或环境中运行。这在沙盒环境、开发工具、原型设计、实验，或在本地机器上同时开发模型和MCP服务器时很常见。当你不想公开暴露数据时，本地MCP服务器很有用。它快速且私密。

远程MCP在服务器上运行，与访问它的AI模型分离。这是大多数生产应用的工作方式。它让运行在OpenAI、Claude或你的IDE上的模型能够访问你平台的工具，实现集中和共享的访问。

你可以在开发时从本地开始，上线后再迁移到远程环境。

## MCP 服务器的应用场景有哪些？

MCP 服务器适用于任何需要模型安全、可预测地与真实系统交互的场景。

*   在电子商务领域，这可能意味着开放产品搜索、购物车更新、订单历史和结账等工具。模型可以帮助用户查找商品、管理购物车，甚至完成购买。
*   在金融领域，模型可以在遵守权限和限制的前提下，查询账户余额、对交易进行分类或生成报告。
*   在营销领域，可以利用结构化工作流运行受众查询、发送外发消息或安排营销活动。

此外，还有一些元应用场景：MCP 服务器可以帮助一个 AI Agent 与另一个 Agent 协调，或者管理 AI 原生应用内部的工具。

以上只是几个例子。你还会看到 MCP 被用于客户支持机器人、物流跟踪、临床辅助工具等等。任何时候，当你希望模型真正采取行动（搜索、更新、预订、创建）时，MCP 都能为其提供结构和控制。

## MCP 将如何帮助普通互联网用户？

他们不会直接看到 MCP。但他们会注意到自己使用的网站和应用上出现更智能、更有帮助的 AI 功能。聊天机器人不再给出模糊或笼统的答案，而是能提供基于真实数据的精确回应。

他们将能够通过 AI 代其执行操作和完成任务。让 AI 重新安排会议，它就会实际移动日程、通知与会者并发送确认信息。说“帮我找一件下个月毕业典礼穿的蓝色连衣裙”，它就会检查商店库存、考虑运输时间并下单。计划旅行？AI 会直接预订航班，而不仅仅是列出网站。

对于企业而言，这意味着客户可以通过简单的对话完成更多事情。他们无需学习新的界面或应用。AI 能理解他们的意图并处理后续事宜。

## MCP 服务器安全吗？

是的，但这取决于你如何构建它们。MCP 服务器不会绕过你现有的身份验证或业务逻辑。它们只开放你选择暴露的部分。模型只能看到并使用你定义的工具。如果某个工具需要用户身份验证，你可以像对待任何其他 API 一样强制执行。

MCP 是关于暴露能力，而非授予全面访问权限。并且由于你控制着每个工具背后的逻辑，你可以加入检查、速率限制、日志记录等。模型无法执行你允许范围之外的任何操作。

`mcp-handler` 支持 MCP 授权规范。这使你能够保护 MCP 端点并在工具中访问身份验证信息。

## MCP 由谁创建和支持？是开源还是闭源的？

MCP 由 Anthropic 创建，并以 MIT 许可证开源发布。所有内容，包括规范、SDK 和示例实现，都托管在 GitHub 上。

现在，它已发展成为一个不断壮大的社区项目。Zed 将其用于编码工具。Sourcegraph 将其用于代码搜索和导航。独立开发者们已经构建了用于智能家居和研究工具等领域的服务器。

虽然没有正式的委员会，但有一个核心维护者小组和社区。人们在公开场合讨论想法，并通过实际使用来改进协议。这使其能专注于真正有效的东西。

AI SDK 和 v0 都开箱即用地支持 MCP 规范。

## MCP 有哪些替代方案？

根据你的需求，还有其他几种将 AI 模型连接到工具的方法。

简单语言开放协议使用接受自然语言的普通 HTTP 端点。它易于设置，但需要 AI 在运行时做更多工作来理解意图。对于简单的用例，这种权衡可能是合理的。

平台特定解决方案，如 OpenAI 的函数调用，与其自身生态系统紧密集成。它们提供了良好的性能和“开箱即用”的工具，但你必须为每个平台重建。这可能导致供应商锁定。

编排框架，如 LangChain，有助于管理跨多个 LLM 和工具的工作流。这些框架通常支持 MCP，并且可以结合使用。

根据你要构建的内容选择工具：

*   对于快速实验，SLOP 很快捷
*   对于与单一提供商的深度集成，平台工具很合适
*   对于复杂的工作流，编排框架很有帮助

然而，若要在众多 AI 中使用一套工具设置，并具备清晰的结构和安全性，MCP 对大多数情况来说是合适的选择。

## 为什么 MCP 很重要

MCP 改变了我们对 AI 集成的思考方式。开发者无需为每个平台构建一次性的解决方案，而是可以创建能在整个生态系统中工作的工具。

这类似于网络的演进方式。早期的网站需要针对特定浏览器的代码。然后像 HTTP 和 REST 这样的标准使网络更加一致和强大。MCP 为 AI 带来了这种结构。

对于开发者来说，集成成本很低。你已经拥有能执行有用功能的 API 和系统。MCP 为你提供了一种标准方式，使这些工具可供 AI 使用。随着更多平台采用它，单一集成的价值也会增长。

最令人兴奋的部分是它所释放的潜力。当 AI 能够采取行动，而不仅仅是给出建议时，新型应用便成为可能。知与行之间的界限变得更模糊。

例如，我们最近为我们的快速代码搜索应用 Grep 添加了一个 MCP 服务器。它本来就有 API，但通过添加 MCP 服务器，你现在可以将 Grep 集成到 Cursor 和 Claude Code 等工具中。AI Agent 现在可以参考超过一百万个开源项目中使用的编码模式和解决方案，来解决你在构建应用时遇到的问题和疑问。
