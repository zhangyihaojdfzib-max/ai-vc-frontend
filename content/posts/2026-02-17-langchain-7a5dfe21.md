太长不看版：我们的编程智能体在 Terminal Bench 2.0 上的排名从 Top 30 提升到了 Top 5。我们只改变了控制框架。以下是我们进行控制框架工程的方法（预告：自我验证与追踪帮助很大）。

## 控制框架工程的目标

控制框架的目标是塑造模型固有的、不稳定的智能，使其适用于我们关心的任务。**控制框架工程**是关于系统的，你围绕模型构建工具，以优化任务性能、Token 效率、延迟等目标。设计决策包括系统提示词、工具选择和执行流程。

但是，你应该如何改变控制框架来改进你的智能体呢？

在 LangChain，我们使用 **Trace** 来大规模地理解智能体的失败模式。如今的模型在很大程度上是黑盒，其内部机制难以解释。但我们可以看到它们在文本空间中的输入和输出，然后将其用于我们的改进循环中。

我们使用一个简单的流程，将 **deepagents-cli**（我们的编程智能体）在 Terminal Bench 2.0 上的得分从 **52.8** 迭代提升了 **13.7 分**，达到 **66.5**。我们只调整了控制框架，并保持模型 **gpt-5.2-codex** 不变。

## 实验设置与控制框架的可调节项

我们使用了 **Terminal Bench 2.0**，这是一个目前用于评估智能体编程能力的标准基准测试。它包含 89 个跨领域任务，如机器学习、调试和生物学。我们使用 **Harbor** 来编排运行。它会启动沙箱（Daytona），与我们的智能体循环交互，并运行验证和评分。

每个智能体操作都存储在 **LangSmith** 中。它还包括延迟、Token 计数和成本等指标。

### 我们可以调节的选项

一个智能体控制框架有很多可调节项：系统提示词、工具、钩子/中间件、技能、子智能体委托、记忆系统等等。我们有意压缩了优化空间，专注于三个方面：**系统提示词**、**工具**和**中间件**（我们对模型和工具调用周围的钩子的称呼）。

我们从默认提示词和标准工具+中间件开始。这在使用 GPT-5.2-Codex 时获得了 52.8% 的分数。这是一个不错的分数，刚好在目前排行榜的 Top 30 之外，但有提升空间。

![](/images/posts/731ffae8989d.png)

### 追踪分析器技能

我们希望追踪分析是可重复的，因此我们将其做成了一个智能体技能。这作为我们**分析跨运行错误并改进控制框架**的流程。流程如下：

1.  从 LangSmith 获取实验追踪数据。
2.  生成并行错误分析智能体 → 主智能体综合发现和建议。
3.  汇总反馈并对控制框架进行有针对性的更改。

这与 **boosting** 类似，后者专注于先前运行中的错误。在步骤 3 中，人类可以非常有帮助（虽然不是必需的）来验证和讨论提议的更改。过度拟合某个任务的更改不利于泛化，并可能导致其他任务出现性能倒退。

自动化的追踪分析节省了大量时间，并使快速尝试实验变得容易。我们将很快发布这个技能，目前我们正在对其进行通用提示词优化的测试。

## 真正提升智能体性能的因素

自动化的追踪分析使我们能够**调试智能体出错的地方**。问题包括推理错误、未遵循任务指令、缺少测试和验证、超时等。我们将在下面的部分更详细地介绍这些改进。

### 构建与自我验证

如今的模型是卓越的自我改进机器。

自我验证允许智能体在单次运行中通过反馈进行自我改进。然而，它们并没有自然倾向进入这种**构建-验证**循环。

最常见的失败模式是：智能体编写了一个解决方案，重新阅读自己的代码，确认看起来没问题，然后停止。测试是自主智能体编程的关键部分。它有助于测试整体正确性，同时为智能体提供信号以进行局部优化。

我们在系统提示词中增加了关于如何解决问题的指导。

1.  规划与探索：阅读任务，扫描代码库，并根据任务规范以及如何验证解决方案来制定初步计划。
2.  构建：在考虑验证的情况下实施计划。构建测试（如果不存在），并测试正常路径和边界情况。
3.  验证：运行测试，阅读完整输出，与要求的内容进行比较（而不是与你自己的代码比较）。
4.  修复：分析任何错误，重新审视原始规范，并修复问题。

我们真正专注于测试，因为它驱动着每次迭代的更改。我们发现，除了提示词之外，确定性的上下文注入也有助于智能体验证其工作。我们使用一个 **PreCompletionChecklistMiddleware**，它在智能体退出前进行拦截，并提醒其根据任务规范运行验证步骤。这类似于 **Ralph Wiggum Loop**，即一个钩子强制智能体在退出时继续执行，我们将其用于验证。

![](/images/posts/9b0e95022277.png)

### 为智能体提供关于其环境的上下文

控制框架工程的一部分是**为上下文工程构建一个良好的传递机制**。Terminal Bench 任务附带目录结构、内置工具和严格的超时限制。

1.  目录上下文与工具：一个 **LocalContextMiddleware** 在智能体启动时运行，以映射 `cwd` 和其他父/子目录。我们运行 `bash` 命令来查找如 `Python` 安装等工具。上下文发现和搜索容易出错，因此注入上下文可以减少这种错误面，并帮助**智能体适应其环境**。
2.  教导智能体编写可测试的代码：智能体不知道它们的代码需要如何变得可测试。我们添加提示词，说明它们的工作将根据程序化测试来衡量，类似于提交代码时的情况。例如，提到文件路径的任务规范应被严格遵守，以便解决方案在自动化评分步骤中有效。强调边界情况的提示词有助于智能体避免只检查“正常路径”情况。强制模型符合测试标准是避免随时间推移产生“马虎积累”的强大策略。
3.  时间预算：我们注入时间预算警告，以促使智能体完成工作并转向验证。众所周知，智能体不擅长时间估计，因此这种启发式方法在这种环境下很有帮助。现实世界的编程通常没有严格的时间限制，但如果不添加任何约束知识，智能体将无法在时间限制内工作。

智能体对其环境、约束和评估标准了解得越多，它们就越能自主地指导自己的工作。

控制框架工程师的目的：准备和传递上下文，以便智能体能够自主完成工作。

### 鼓励智能体退一步并重新考虑计划

智能体一旦决定了计划，可能会变得短视，从而导致“毁灭循环”，即对同一个错误的方法进行微小变化（在某些追踪中超过 10 次）。

我们使用一个 **LoopDetectionMiddleware**，它通过工具调用钩子跟踪每个文件的编辑次数。在对同一文件进行 N 次编辑后，它会添加上下文，如“……考虑重新评估你的方法”。这可以帮助智能体从毁灭循环中恢复，尽管如果模型认为自己是正确的，它可能会继续沿着相同的路径前进。

重要说明：这是一种设计启发式方法，旨在围绕当前感知到的模型问题进行工程化。随着模型的改进，这些防护措施可能变得不必要，但在今天有助于智能体正确且自主地执行。

### 选择在推理上花费多少计算资源

推理模型可以自主运行数小时，因此我们必须决定在每个子任务上花费多少计算资源。你可以在每个任务上使用最大的推理预算，但大多数工作可以通过优化推理计算支出来受益。

Terminal Bench 的超时限制造成了权衡。更多的推理有助于智能体评估每个步骤，但可能会消耗超过 **2 倍** 的 Token/时间。`gpt-5.2-codex` 有 4 种推理模式：`low`、`medium`、`high` 和 `xhigh`。

我们发现，推理有助于规划以完全理解问题，一些 Terminal Bench 任务非常困难。一个好的计划有助于更快地找到可行的解决方案。

后期验证阶段也能从更多推理中受益，以发现错误并提交解决方案。作为一种启发式方法，我们选择以 xhigh-high-xhigh 的“推理三明治”结构作为基线。

![在规划和验证上投入更多推理算力](/images/posts/6510fa4a258e.png)

仅在 xhigh 设置下运行时，由于 Agent 超时问题，得分仅为 53.9%，远低于 high 设置下的 63.6%。在不同推理预算分配方案的试运行中未发现显著差异，因此我们坚持原有方法，最终将得分提升至 66.5%。

模型自然采用的方法是**自适应推理**，这在 Claude 和 Gemini 模型中可见一斑，模型会自行决定投入多少算力进行推理。

在多模型系统中，平衡推理预算可以表现为：使用大模型进行规划，然后将任务**移交**给更小的模型执行。

## 构建 Agent 系统的实用要点

Agent 的设计空间非常广阔。以下是我们通过实验和构建深度 Agent 系统总结出的一些通用原则：

1.  **为 Agent 进行上下文工程**：当前 Agent 在上下文整合方面仍面临困难，尤其是在陌生环境中。通过提供目录结构、可用工具、编码最佳实践和问题解决策略等上下文信息来引导模型，有助于减少因搜索不当和规划错误导致的失误。
2.  **帮助 Agent 自我验证工作成果**：模型往往倾向于其首个看似合理的解决方案。应积极通过提示词引导它们运行测试并优化方案来验证工作成果。这在没有人工介入的自主编码系统中尤为重要。
3.  **将执行轨迹作为反馈信号**：轨迹能让 Agent 进行自我评估和调试。必须同时调试工具使用和推理过程（例如：模型可能因缺乏特定工具或操作指引而误入歧途）。
4.  **短期内检测并修正不良模式**：现有模型并不完美。系统设计者的职责是围绕当前缺陷进行设计，同时为未来更智能的模型做好规划。盲目重试和不验证工作成果就是典型的反面案例。这些防护措施未来很可能会逐渐淘汰，但对于当前构建稳健的 Agent 应用而言，它们仍是值得尝试的有效工具。
5.  **为不同模型定制系统**：Codex 和 Claude 的提示词指南表明，不同模型需要不同的提示方式。使用早期系统版本对 Claude Opus 4.6 进行测试时得分为 59.6%，虽然具有竞争力但低于 Codex，因为我们未对 Claude 运行相同的改进循环。许多原则具有普适性，例如良好的上下文准备和注重验证，但针对具体任务进行几轮系统迭代，有助于最大化 Agent 在不同任务中的表现。

在系统设计方面仍有大量开放研究空间。值得探索的方向包括：多模型系统（协同使用 Codex、Gemini 和 Claude）、支持持续学习的记忆原语（使 Agent 能自主改进任务表现），以及跨模型的系统变更效果评估。

对于提升 Agent 的外部循环，我们正在研究如 RLMS 等方法以更高效地挖掘轨迹数据。我们将持续改进系统并公开分享研究成果。
