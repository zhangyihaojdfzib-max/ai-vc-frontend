# 让任何大语言模型成为更好的诗人

- joehoover

在这篇文章中，我们讨论了我们刚刚发布的一个名为 Poet Vicuna-13B 的 Vicuna-13B 版本。该模型是一个早期阶段项目的一部分，专注于增强开源大语言模型。Poet Vicuna-13B 是 Vicuna-13B 的一个实现，经过修改后能够生成具有特定音节模式的诗歌和歌词。你可以用它来改写《小星星》，或者生成充满大量优美空白、行长度由你选择的现代主义诗歌。

例如，我们要求 Poet Vicuna-13B 根据以下音节计数 [3, 3, 0, 2, 0, 5, 0, 3, 2, 1, 0, 4] 写一首八行诗，以回应这个提示词：

写一首诗，探讨时间的概念及其对我们生活的影响——怀旧、时间的流逝、瞬间的短暂：

它生成了这首诗：

在瞬间  
过去又消失  
它们的时光  
流逝得如此之快。在  
此刻  
我们静止不动  
但能持续多久？

但是，首先，我们是如何走到这一步的？

当我们想要改进大语言模型的行为时，提示词工程和训练往往是我们首先想到的解决方案。毕竟，它们是让这些模型按照我们意愿行事的最常见方法。但它们并不是修改语言模型行为的唯一途径，而且在某些情况下，它们远非最佳解决方案。

例如，即使是像 GPT-4 这样最先进的语言模型，诗歌和歌词生成仍然相当困难。正如我们将讨论的，提示词和训练可能不是开发复杂诗歌生成器的最短路径。

创作一首好诗是件难事，即使对人类来说也是如此。你必须兼顾意义、情感、节奏，而最重要的是，还有音节的问题。现在想象一下让机器来做这件事。听起来很棘手，对吧？即使现有的大语言模型能够生成相当令人信服的文本，让它们严格遵守诗歌的精确规则——尤其是那些有严格音节计数的诗歌——又是另一回事了。

例如，考虑我们最近与 GPT-4 的一次交流：

[我们] 写一首关于狗的五行诗，每行 4 个音节。
[ChatGPT] 毛茸茸的朋友心真诚，
喜悦吠叫眼中情。
你的摇摆烦恼散，
狗狗你是家所在。
你的爱是纯净新礼。
[我们] 我要求你写一首关于狗的五行诗，每行 4 个音节，但你写了一首每行 6 个或更多音节的五行诗。
请更仔细地遵循我的指示：
写一首关于狗的五行诗，每行 4 个音节。
[ChatGPT] 爪在泥，鼻向天，
眼中忠诚凝视坚。
黎明吠，尾欢腾，
在你温暖中飞翔。
狗的爱，无可比。

GPT-4 不仅未能遵循我们的指示，甚至无法生成一行符合四个音节限制的诗歌。

语言模型虽然非常强大，但本质上并不理解音节这样的概念。它们不像我们那样“理解”语言；相反，它们在训练所用的文本数据中寻找模式。对于 GPT-4 和其他基于 Transformer 的模型，它们编码的最小单元是 Token。一个 Token 可以短至一个字符，或者在某些情况下长至一个或多个单词。分词器，即用于将输入分解为这些 Token 的工具，本身没有内置的音节概念。

因此，虽然模型可以生成种类繁多的响应，但它本质上并不“计数”音节或保持特定的格律模式。相反，它根据训练期间学到的模式做出决策。如果训练数据没有强调某个特定特征（如严格的音节计数），模型就不太可能自发地以这种方式生成文本。

# 从散文到诗歌：改进语言模型的诗歌生成

幸运的是，关于诗歌生成已经有很多出色的工作。我们还没有一份完整的文献综述可以分享（目前还没有！），但我们会重点介绍一些与我们正在做的工作特别相关的有趣内容。许多关于诗歌生成的研究都集中在创建训练方案上，以生成更擅长创作诗歌的模型。

例如，Ormazabal 等人（2022）引入了 PoeLM，这是一种使用控制码来编码一行中的音节数和结尾音节的方法，这有助于决定押韵方案。他们将这些控制码注入训练数据，并发现经过微调的模型能够更好地生成结构化的诗句。Chakrabarty 等人（2022）的工作也值得注意。他们开发了一个经过指令微调的诗歌写作助手 CoPoet，利用了各种 T5 模型的能力。

然而，其他研究则侧重于开发基于规则的机制来指导语言模型的生成过程。这种方法不是试图训练模型生成具有特定结构的诗句，而是提供引导框架，确保生成的文本满足用户指定的约束条件。例如，Roush 等人（2022）设计了一种方法，通过在生成文本之前对模型的词汇表应用过滤函数来操纵语言模型的输出。他们在“约束文本生成工作室”（CTGS）工具中展示了这项技术，该工具显示出比传统微调更好的性能。

我们发现这种方法特别令人兴奋，原因有几个。首先，它很有道理。文本生成本质上是概率性的，训练一个模型完美地遵循复杂的格式模式（例如，参见 jsonformer 了解不同领域的例子）充其量仍然很困难。此外，这种方法突显了开源语言模型的一个优势。我们可以用闭源模型根本无法实现的方式来修改它们，这为我们打开了一个充满创造力和实验的整个世界。

# 介绍 bragi 和 Poet Vicuna-13B

我们以 Roush 等人的工作为起点。我们的第一个目标是实现一个最小化的实现，使我们能够将行级音节约束应用于任何 Hugging Face Transformers 文本生成器的生成过程。

我们将其实现为一个名为 bragi 的库，以北欧诗歌之神命名。Bragi 目前处于非常早期的 Alpha 阶段，但将其简单的机制应用于 Vicuna-13B 产生了非常酷的体验，我们想与社区分享。

那么，它是如何工作的呢？

bragi 项目的核心是 MetricGenerator 类，它实现了一个 logit 扭曲器，在生成过程中约束模型词汇表。logit 扭曲器通过跟踪行级音节预算和行数预算来约束输出的格律结构。
这种机制依赖于计算每个 Token 的音节成本，我们使用 pronouncing 库来实现这一点。

换句话说，bragi 根据指定的音节模式，在生成过程的每一步动态调整可用的 Token。例如，如果一行只剩下一个音节的空间，bragi 将只允许选择一个音节的 Token。这种方法有助于在保持创造性生成的同时，维持所需的格律结构。

要初始化目标格律结构，我们可以指定一个整数列表，表示每行应包含的音节数，或者提供一个初始化文本。例如，“生日快乐”歌遵循 6、6、7、6 的音节模式，可以用来初始化格律结构。

而且，正如你已经看到的，你可以通过提示词来引导生成过程！这意味着你可以按照特定的音节模式生成一首新歌或新诗，同时加入你自己的创意风格。
