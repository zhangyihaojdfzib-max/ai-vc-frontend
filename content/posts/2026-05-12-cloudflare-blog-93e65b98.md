# 当“空闲”并非空闲：Linux内核优化如何成为QUIC的Bug

2026-05-12

- Esteban Carisimo
- Antonio Vicente

![](/images/posts/818ec6502091.png)

CUBIC，在RFC 9438中标准化，是Linux中的默认拥塞控制器，因此决定了公共互联网上大多数TCP和QUIC连接如何探测可用带宽、在检测到丢包时如何回退以及之后如何恢复。在Cloudflare，我们开源的QUIC实现quiche使用CUBIC作为其默认拥塞控制器，这意味着这段代码对我们所服务的相当一部分流量处于关键路径中。

在这篇文章中，我们将讲述一个Bug的故事——CUBIC的拥塞窗口（cwnd）被永久锁定在最小值，并且从未从拥塞崩溃事件中恢复。

故事始于一项Linux内核变更，旨在使CUBIC符合RFC 9438 §4.2-12中描述的应用程序受限排除规则——这是对TCP中一个真实问题的修复，当移植到我们的QUIC实现时，暴露了quiche中意想不到的行为。故事有一个圆满的结局：一个优雅的（近乎）单行修复打破了这一循环。

## CUBIC的逻辑简述

在深入核心问题之前，快速回顾一下CCA可能有助于铺垫背景。

CCA控制的核心旋钮是拥塞窗口（cwnd）：发送端对任意时刻可以有多少字节在传输中（已发送但尚未确认）的上限。较大的cwnd允许发送端在每个往返中推送更多数据；较小的cwnd则会限制发送速率。每个基于丢包的CCA（包括CUBIC）本质上都是一套策略，规定了在网络看起来健康时如何增大cwnd，以及在网络不健康时如何减小cwnd。

本质上，CCA旨在通过推断网络的"可用带宽"来最大化数据传输；因为没有人愿意为1 Gbps的订阅付费却只使用其中一小部分。CUBIC所属的基于丢包算法家族基于一个基本前提运作：（1）如果没有丢包，则增加发送速率（即提高带宽利用率）；（2）如果有丢包，基于丢包的算法假设网络容量已被超出，发送端必须回退（即降低带宽利用率）。

这一逻辑建立在多个假设之上，这些假设多年来已被重新审视。不过，我们将把这一讨论留到其他时候。

## 症状：一个61%概率失败的测试

我们的调查始于入口代理集成测试管道中出现意外失败的报告。这种不稳定的行为出现在CUBIC在连接早期经历严重丢包场景的测试中。

拥塞崩溃后的恢复是一种不常见的场景，但这正是拥塞控制器存在要处理的情况。大多数拥塞控制测试会测试算法的稳态和增长阶段；很少有测试会探究在连接被压制后，最小cwnd状态下会发生什么。状态空间中这一角落的Bug在吞吐量仪表板中不可见，无法通过静态审查检测到，只有当你故意将CCA驱动到该状态并观察它能否重新爬升时才会浮出水面——这正是这个测试所做的。

模拟测试设置包括以下细节：

- Quiche HTTP/3客户端和服务器在本地运行（localhost）
- RTT = 10ms（在配置中设置）
- 通过HTTP/3下载10 MB文件
- 使用CUBIC拥塞控制
- 在前两秒内注入30%的随机丢包
- 两秒后，丢包完全停止
- 测试有宽裕的10秒超时时间来完成下载，预计在四到五秒内完成

预期的行为很简单：CUBIC应在丢包阶段受到一些打击，减小其拥塞窗口，一旦丢包停止，稳步提升并在超时前顺利完成下载。然而，我们在多次100次运行中观察到，大约60%的测试无法在宽裕的10秒超时内完成下载。

## 异常：零丢包下的999次状态转换

我们使用丢包事件对quiche的qlog输出进行了检测，并构建了可视化图表以了解拥塞控制器内部发生的情况：

失败测试的连接概览。在T=2s之后，丢包完全停止——然而cwnd仍被锁定在最小值，拥塞状态每约14ms在恢复和拥塞避免之间振荡。

在两秒（2000 ms）标记之后，丢包完全停止。然而，传输中的字节数保持平稳，这与CUBIC算法的核心逻辑相矛盾：在没有丢包的情况下，应加大油门以提高速率（在我们的世界中增加字节数）。这就提出了一个问题：如果网络不再丢包，为什么拥塞窗口无法增长？

当我们放大该区域时，我们的分析显示CUBIC进入了一种快速振荡，在我们的图中表现为一个延长的恢复阶段，在拥塞避免状态（运行阶段）和恢复状态（丢包恢复阶段）之间——在大约6.7秒内发生了999次转换。这相当于每约14ms一次转换——可疑地接近连接的RTT（10ms）。在整个这段时间内，cwnd被锁定在最小值：2700字节，即两个完整大小的数据包。

显然，CUBIC逻辑中的某些东西错误地解释了连接的状态。关键线索是振荡周期：约14ms与RTT匹配。任何触发恢复/避免翻转的事件每个往返发生一次，与连接的ACK时钟同步；这是一种自时钟节奏，其中每个往返中来自客户端的ACK触发服务器的下一次发送。由于这是一个下载（服务器到客户端），相关的ACK是从客户端到服务器，而CUBIC的状态机在服务器端运行：每次这些ACK到达时，传输中的字节数降至零，服务器发送下一个两数据包突发，这正是触发Bug的原因。

为了确认这种行为是CUBIC特有的，我们使用Reno（同一基于丢包家族的另一个成员，但增长率不同）运行了相同的测试。结果具有决定性：100%通过率，显示Reno在丢包阶段后干净地恢复，并揭示了这是一个CUBIC相关的Bug。

Reno在T=2s丢包阶段结束后干净地恢复，并在约5s时完成下载

## 追溯根本原因

基于丢包的算法有两个踏板，油门和刹车，区别在于它们如何加速。嗯，CUBIC带有一些额外功能。这里我们将重点关注传输中的字节数 == 0。

### TCP CUBIC空闲后（Linux，2017）

要理解这个Bug，我们首先需要理解它源自的优化。2017年，Linux内核的CUBIC实现中发现了一个问题。提交信息解释道：

epoch仅在初始时和遇到丢包时更新/重置。在应用程序空闲后，"t"（now - epoch_start）的增量可能任意大，bic_target也是如此。因此斜率（ca->cnt的倒数）会非常大，最终ca->cnt会被下限约束为2，以具有延迟ACK的慢启动行为。

这在禁用slow_start_after_idle时尤为明显，因为在几秒空闲时间后会出现危险的cwnd膨胀（1.5 x RTT）。

纪元是CUBIC用于锚定其增长曲线的参考时间戳：`W_cubic(delta_t)` 由 `delta_t = now - epoch_start` 参数化，每当CUBIC重新启动其增长函数时（最显著的是在丢包事件导致 `cwnd` 减小之后），纪元会被重置。在重置之间，`delta_t` 随挂钟时间单调增长。

当应用程序空闲（停止发送）一段时间后恢复时，CUBIC增长函数 `W_cubic(delta_t)` 将 `delta_t` 计算为 `now - epoch_start`，如下图所示。由于空闲期间纪元未更新，`delta_t` 会非常大，从而产生一个巨大的目标窗口——CUBIC会立即尝试将 `cwnd` 膨胀到一个不合理的值。

Jana Iyengar 最初的修复方案是在应用程序恢复发送时重置 `epoch_start`。但 Neal Cardwell 指出了该方法中的缺陷：

“……它会要求 CUBIC 算法重新计算曲线，以便我们再次从当前 `cwnd` 的位置开始陡峭增长（就像 CUBIC 在丢包后所做的那样）。理想情况下，我们希望 `cwnd` 增长曲线的形状保持不变，只是在时间上向后平移空闲持续时间的长度。”

由 Eric Dumazet、Yuchung Cheng 和 Neal Cardwell 提出的优雅解决方案是**将纪元向前平移空闲持续时间**，而不是重置它。这保留了 CUBIC 增长曲线的形状——只是在时间上滑动它，使算法从它中断的地方继续。

### 移植到 quiche（2020年）

当 CUBIC 首次在 quiche 中实现时，这个空闲期调整被移植了过来。然而，运行在用户空间的 QUIC 没有 TCP 内核级别的 `CA_EVENT_TX_START` 回调。相反，quiche 实现在 `on_packet_sent()` 内部检查空闲条件：

```Rust
// cubic.rs —— on_packet_sent()（简化版）
/// 当发送数据包时更新状态。
fn on_packet_sent(&mut self, bytes_in_flight: usize, now: Instant, ...) {
    // 如果发送突发正在重新启动（即，在此次发送前 bytes_in_flight 为零），
    // 调整拥塞恢复开始时间以考虑发送间隔。
    if bytes_in_flight == 0 {
        let delta = now - self.last_sent_time;
        self.congestion_recovery_start_time += delta;
    }
    // 记录此次发送事件的时间。
    self.last_sent_time = now;
}
```

### 问题所在：QUIC 的差异

移植到 quiche 的修复包含了一个原始内核更改中的错误，该错误在大约一周后通过对内核 cubic 模块的**后续更改**得到了修复。第二次修复的提交信息解释道：

tcp_cubic: 不要将 `epoch_start` 设置为未来的值
	在 `bictcp_cwnd_event()` 中跟踪空闲时间是不精确的，因为 `epoch_start` 通常在 ACK 处理时设置，而不是在发送时设置。

进行适当的修复需要添加一个额外的状态变量，
	并且似乎不值得这么麻烦，因为 CUBIC 的 bug 在 Jana 注意到之前已经存在很久了。

我们干脆不要将 `epoch_start` 设置为未来的值，否则 `bictcp_update()` 可能会溢出，CUBIC 会再次过快增长 `cwnd`。

正如提交信息中提到的，恢复开始时间是在 ACK 处理期间设置的，而基于发送时间计算调整量可能会将恢复开始时间推到未来。这解释了我们在测试中观察到的恢复与拥塞避免之间的振荡。只有当每个传入的 ACK 都将 `bytes_in_flight` 一直驱动到零时，这个陷阱才会持续触发——这实际上意味着 `cwnd` 已经坍缩到其最小值（两个数据包），并且应用程序有数据准备好在 ACK 到达时立即发送另一个完整的窗口。在这种情况之外，`bytes_in_flight == 0` 不太可能在每次发送时都成立，因此触发该 bug 的可能性较小。

为什么连接启动时不会发生这种情况？该 bug 仅在连接退出慢启动并切换到拥塞避免时触发。在退出慢启动之前，`congestion_recovery_start_time` 未被设置，因此 `on_packet_sent` 中有问题的分支没有恢复边界可以推进。在慢启动期间，CUBIC 的 `cwnd` 通过所有基于丢包的 CCA 共享的相同的 Reno 风格基于 ACK 的规则增长——cubic 曲线及其对 `congestion_recovery_start_time` 的敏感性只有在连接进入拥塞避免后才会发挥作用，这意味着触发陷阱需要同时满足三个条件：一个设置恢复边界的真实丢包事件、正在运行的拥塞避免，以及 `cwnd` 坍缩到两个数据包的最低点。

**自我延续的恢复陷阱。** 在最小 `cwnd` 下，每个 ACK 周期都会触发空闲期调整，并带有膨胀的 delta。

在最小 `cwnd`（两个数据包）下，连接的动态进入一个“死亡螺旋”，其中空闲期优化变成了自我实现的预言。这个陷阱在一个连续循环中运行：

1. **发送和确认数据包：** 发送方传输整个两个数据包的窗口。经过一个 RTT（约 14ms）后，两个数据包都被 ACK 确认，导致 `bytes_in_flight` 降至零。
2. **虚假的空闲检测：** 当发送下一个突发时，`on_packet_sent()` 看到 `bytes_in_flight == 0` 并假设连接处于空闲状态，但实际上它是受拥塞限制的。
3. **膨胀的 delta：** 计算使用 `now - last_sent_time` 来确定空闲持续时间。当拥塞窗口（`cwnd`）处于最小值时，`last_sent_time` 是前一个 RTT 周期**开始**的时间戳。因此，得到的 delta 大约是 **14ms**（连接的 RTT 加上额外的舍入误差）。这个 RTT 大小的 delta 被错误地当作“空闲”时间。连接实际空闲的时间（最后一个 ACK 到达与下一个数据包发送之间的处理间隙）实际上是 0。通过测量完整的 RTT 而不是真实间隙，delta 被**显著膨胀**，激进地将恢复开始时间向前推移，可能推到未来。
4. **感知到的恢复：** 因为恢复开始时间现在在未来，`in_congestion_recovery()` 检查对每个传入的 ACK 都返回 true。处理下一个 ACK 会退出恢复并将恢复开始时间设置为 ACK 时间，该时间大于 `last_sent_time`，使得拥塞控制器在下次发送时很可能将恢复时间推到未来。
5. **停滞：** 由于 CUBIC 对任何被认为处于恢复期的数据包跳过 `cwnd` 增长，窗口仍然固定在两个数据包——确保管道在下一个 ACK 时完全排空，并重新开始循环。

发送和确认数据包：发送方传输整个两个数据包的窗口。经过一个 RTT（约 14ms）后，两个数据包都被 ACK 确认，导致 `bytes_in_flight` 降至零。

虚假的空闲检测：当发送下一个突发时，`on_packet_sent()` 看到 `bytes_in_flight == 0` 并假设连接处于空闲状态，但实际上它是受拥塞限制的。

膨胀的 delta：计算使用 `now - last_sent_time` 来确定空闲持续时间。当拥塞窗口（`cwnd`）处于最小值时，`last_sent_time` 是前一个 RTT 周期开始的时间戳。因此，得到的 delta 大约是 14ms（连接的 RTT 加上额外的舍入误差）。这个 RTT 大小的 delta 被错误地当作“空闲”时间。连接实际空闲的时间（最后一个 ACK 到达与下一个数据包发送之间的处理间隙）实际上是 0。通过测量完整的 RTT 而不是真实间隙，delta 被显著膨胀，激进地将恢复开始时间向前推移，可能推到未来。

感知到的恢复：因为恢复开始时间现在在未来，`in_congestion_recovery()` 检查对每个传入的 ACK 都返回 true。处理下一个 ACK 会退出恢复并将恢复开始时间设置为 ACK 时间，该时间大于 `last_sent_time`，使得拥塞控制器在下次发送时很可能将恢复时间推到未来。

停滞：由于 CUBIC 对任何被认为处于恢复期的数据包跳过 `cwnd` 增长，窗口仍然固定在两个数据包——确保管道在下一个 ACK 时完全排空，并重新开始循环。

这个循环会重复数千次，直到来自调度器抖动和ACK处理偏差的微小偏差累积，使得`in_congestion_recovery()`中的`<=`边界落后于下一个数据包的发送时间，从而打破这个循环。

## 修复方法：从正确的时间点测量空闲状态

修复这种死亡螺旋的方法涉及测量空闲持续时间，该时间从实际转换为零的飞行中字节数（最后一个ACK被处理时）开始计算，而不是从最后一个数据包发送时开始。

### 代码变更

1. 向CUBIC状态添加`last_ack_time`时间戳。
2. 在ACK到达时更新该时间戳。
3. 将其用于空闲增量计算：

向CUBIC状态添加`last_ack_time`时间戳。

在ACK到达时更新该时间戳。

将其用于空闲增量计算：

```Rust
// cubic.rs —— on_packet_sent()
fn on_packet_sent(&mut self, bytes_in_flight: usize, now: Instant, ...) {
    // 在此数据包发送前检查连接是否处于空闲状态。
    if bytes_in_flight == 0 {
        if let Some(recovery_start_time) = r.congestion_recovery_start_time {
            // 从最近的活动开始测量空闲时间：要么是
            // 最后一个ACK（近似表示飞行中字节数变为0的时刻），要么是最后一个数据
            // 发送时间，取两者中较晚的一个。仅使用last_sent_time
            // 会在cwnd较小且飞行中字节数在ACK和发送之间短暂变为0时，
            // 将增量膨胀整整一个RTT。
            let idle_start = cmp::max(cubic.last_ack_time, cubic.last_sent_time);

            if let Some(idle_start) = idle_start {
                if idle_start < now {
                    let delta = now - idle_start;
                    r.congestion_recovery_start_time =
                        Some(recovery_start_time + delta);
                }
            }
        }
}
```

由于增量现在反映了自最后一个ACK以来的实际时间间隔，恢复边界不再追赶发送时间：

旧代码：边界每个循环前进一个RTT，总是落在下一次发送时间上或之前。

修复后：边界几乎不动；下一次发送时间落在其之前，cwnd得以增长。

对于真正空闲的连接，`last_ack_time`在遥远的过去，相同的表达式捕获了完整的空闲持续时间，原始的纪元偏移行为得以保留。

## 验证

应用修复后，我们的quiche测试套件恢复了100%的通过率。

修复后，cwnd沿着预期的CUBIC曲线增长，下载在大约4-5秒内完成。

我们不担心连接末端的丢包——这是预期的，因为我们完全利用了路由器分配的缓冲区。换句话说，在此测试用例中，我们完全利用了可用带宽。

## 经验教训
