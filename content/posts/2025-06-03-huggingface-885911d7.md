# 不浪费任何GPU：通过TRL中的共置vLLM释放效率

## 🚀 引言

TRL支持使用GRPO训练LLM（大语言模型），GRPO是DeepSeekMath论文中最近提出的一种在线学习算法。在GRPO中，模型从其自身输出中学习：它在训练过程中生成响应，接收反馈，并利用该反馈随时间不断改进自身。

这使得生成成为训练循环中的关键步骤——同时也成为一个主要瓶颈。为了加速生成，TRL集成了vLLM。这种组合使您能在GRPO设置中更高效地训练强大的模型。然而，这里有一个问题。

## 🧨 问题所在

在TRL v0.18.0之前，vLLM仅支持在**服务器模式**下运行，作为一个独立的进程在与训练任务不同的GPU上运行。它通过HTTP与训练脚本通信，这使得设置模块化且易于使用——但也引入了GPU效率低下的问题。

以下是发生的情况：

- 在训练期间，模型需要频繁生成补全内容。
- 训练器向vLLM服务器发送请求，该服务器运行在其专用的GPU上。
- 当vLLM生成时，**训练GPU闲置**并等待。
- 一旦生成完成，**vLLM GPU变为闲置**，训练恢复。

这种训练和生成之间的“乒乓效应”导致：

- 双方GPU时间浪费
- 对**额外GPU**的需求增加，仅用于运行推理
- 整体**吞吐量降低和成本升高**

在像GRPO这样持续进行生成的在线学习方法中——这种低效问题变得更加严重。您在硬件上花费更多，却得不到预期的性能。

因此，关键问题变成了：**我们能否让训练和生成共享相同的GPU，而不是将它们分开？**

## 💡 机遇

主要问题在于训练和推理运行在独立的GPU上，导致了闲置时间和利用率不足。自然的解决方案是什么？让两者在同一GPU上运行。与其让vLLM作为独立服务器在其自己的进程和设备中运行，不如让vLLM与训练代码一起，在同一个分布式进程组内运行？这将使我们能够启动一个单一的分布式任务，其中训练和推理共享相同的设备，高效地在任务间切换而不浪费资源。

这种方法就是我们所说的**共置**。训练和推理共置在相同的GPU上，并通过相同的进程组进行协调，使它们能够顺利轮换——无需额外的硬件。

此前，这在TRL中是不可能的，因为TRL依赖vLLM作为外部HTTP服务器。随着我们的**PR #3394**，这种情况发生了改变，该PR增加了对vLLM外部启动器的支持，并实现了与训练过程的真正集成。

### 它带来的能力

- **统一执行**：通过将vLLM嵌入到同一个进程组中，训练和推理任务可以共享相同的GPU，轮流执行而不是相互等待。这减少了闲置时间，提高了整体效率。
- **跳过HTTP通信**：无需REST API调用或网络通信——vLLM与训练循环内联运行，避免了开销和延迟。
- **Torchrun兼容性**：与`torchrun`无缝协作，因此只需最少的配置更改即可轻松跨节点扩展。
- **TP和DP支持**：兼容张量并行和数据并行，适用于大规模训练运行。
- **SPMD执行模式**：使用单程序多数据模型，其中每个GPU同步运行其自己的引擎实例。非常适合分布式多GPU、多节点设置。
- **简化部署**：您不再需要维护单独的服务器脚本——vLLM直接在您的训练作业内部启动和控制。
- **增强的吞吐量**：通过避免GPU闲置和消除进程间通信，系统提供更快的训练和生成速度，这在像GRPO这样的在线学习设置中尤其重要。
- **稳健的进程间通信**：这更加稳健，因为它避免了在服务器模式下所需的、在独立进程之间设置分布式进程组的复杂性。

得益于这一特性，共置的训练和推理不再是权宜之计——它现在已成为**一流的、可扩展的、可用于生产**的功能。

## 🧩 设计：从独立服务器到共享GPU

从服务器TRL到共置TRL的转变，核心在于更智能地使用GPU。下图展示了区别：

![gpus-design](/images/posts/dcd445475524.png)

### 服务器TRL设置（上图）

在服务器TRL设置中，训练和推理运行在独立的GPU上。例如：

- GPU 0到2用于训练。
- GPU 3完全专用于作为独立服务器运行vLLM。

在训练步骤期间，**GPU 3闲置**。
在生成步骤期间，**GPU 0–2闲置**，而GPU 3生成输出。

这导致：

- GPU使用效率低下，设备频繁相互等待
- 额外配置GPU仅用于推理
- 成本和复杂性增加

### 共置TRL设置（下图）

相比之下，共置TRL设置在**相同的GPU**上同时运行训练和vLLM。每个GPU：

- 运行训练循环
- 在**同一进程内**启动一个vLLM引擎

训练和推理**轮流使用**GPU的资源——无需专用设备或独立进程。

这种设计：

- 减少了闲置时间
- 最小化了进程间和HTTP通信
- 充分利用了可用的GPU内存和计算能力
- 在不增加硬件要求的情况下提供**更快的吞吐量**

## 🛠️ 实现说明

现在，训练器不再将vLLM作为服务器启动，而是使用外部启动器**在进程内启动vLLM**，如下所示：

```python
self.llm = LLM(
    model=model.name_or_path,
    tensor_parallel_size=args.vllm_tensor_parallel_size,
    gpu_memory_utilization=self.vllm_gpu_memory_utilization,
    max_num_seqs=self.args.per_device_train_batch_size
        * self.vllm_tensor_parallel_size
        * self.args.gradient_accumulation_steps,
    max_model_len=self.max_prompt_length + self.max_completion_length,
    distributed_executor_backend="external_launcher",
    
    seed=self.accelerator.process_index // self.vllm_tensor_parallel_size,
)
```

共置的vLLM遵循torch.distributed进程组和等级结构。这使得vLLM可以与训练一起初始化而不会冲突，并使TP/DP设置无缝工作：

```python
if self.vllm_tensor_parallel_size > 1:
    
    self.tp_group, _ = torch.distributed.new_subgroups_by_enumeration(
        [
            list(range(i * self.vllm_tensor_parallel_size, (i + 1) * self.vllm_tensor_parallel_size))
            for i in range(self.accelerator.num_processes // self.vllm_tensor_parallel_size)
        ]
    )
```

共置式 vLLM 不再依赖 REST API——它直接在内存中运行，并通过原生 Python 调用进行通信：

```python
if self.vllm_tensor_parallel_size > 1:
    orig_size = len(prompts_text)
    gathered_prompts = [None for _ in range(self.vllm_tensor_parallel_size)]
    torch.distributed.all_gather_object(gathered_prompts, prompts_text, group=self.tp_group)
    all_prompts_text = [p for sublist in gathered_prompts for p in sublist]
else:
    all_prompts_text = prompts_text

with profiling_context(self, "vLLM.generate"):
    all_outputs = self.llm.generate(all_prompts_text, sampling_params=sampling_params, use_tqdm=False)

completion_ids = [output.token_ids for outputs in all_outputs for output in outputs.outputs]

if self.vllm_tensor_parallel_size > 1:
    local_rank_in_group = torch.distributed.get_rank(group=self.tp_group)
    tp_slice = slice(local_rank_in_group * orig_size, (local_rank_in_group + 1) * orig_size)
    completion_ids = completion_ids[tp_slice]

```

要使用此设置，只需在 GRPO 配置中设置 `vllm_mode="colocate"`：

```python
training_args = GRPOConfig(
    ...,
    use_vllm=True,
    vllm_mode="colocate",
)

```

注意：根据模型大小和训练所需的总体 GPU 内存，您可能需要调整 `GRPOConfig` 中的 `vllm_gpu_memory_utilization` 参数，以避免利用率不足或内存不足错误。

## 📊 展示：共置式 vs 普通 TRL 性能

为了衡量共置的影响，我们进行了一系列实验，将传统的**服务器模式**（vLLM 在单独的 GPU 上作为独立服务器运行）与新的**共置模式**（训练和推理共享相同的 GPU）进行了比较。

在**服务器模式**下，只有 7 个 GPU 用于训练，因为 1 个 GPU 完全专用于 vLLM 推理服务器。

在**共置模式**下，所有 8 个 GPU 都用于训练——默认情况下增加了有效批次大小。

为了确保公平比较，我们将**服务器模式**下的吞吐量按 8/7 的系数进行了**归一化**。此调整考虑了共置模式下更大的训练容量，使我们能够在相同的训练条件下比较两种设置。

### 实验 1：1.5B 模型 —— 不同批次大小

- 随着批次大小的增加，两种设置下的吞吐量都有所提高。
- 在最大批次大小时，**共置设置实现了高达 1.43 倍的加速**。
- 更大的批次在共置模式下能更好地利用共享的 GPU 内存。

![small-b](/images/posts/b8150810c75f.png)

### 实验 2：1.5B 模型 —— 不同张量并行度

- 在共置设置中，增加 TP **会降低性能**。
- 更多的分片会引入更多的通信开销——这对于较小的模型**并不理想**。
- 结论：对于小模型，在共置模式下应避免过度分片。

![small-tp](/images/posts/3cad516b3808.png)

### 实验 3：7B 模型 —— 不同批次大小

- 同样，共置模式**随着批次大小的增加扩展性更好**。
- 在测试的最大批次下，**增益达到 1.35 倍加速**。

![med-b](/images/posts/fe91913d817a.png)

### 实验 4：7B 模型 —— 不同张量并行度

- 与 1.5B 模型相反的趋势。
- 对于 7B 模型，**更多的 TP 提高了吞吐量**，达到**高达 1.73 倍的加速**。
- 较大的模型在共置设置中**受益于分片**。

![med-tp](/images/posts/b2701abdfc2d.png)

## 📊 扩展到 72B 模型

在训练像 **Qwen2.5-Math-72B** 这样的大型模型时，使用正确的策略非常重要，以使训练在多个 GPU 和节点上高效、可扩展且稳定。在我们的设置中，我们结合了**共置式 vLLM** 和几项关键优化，以高效地实现这一目标。

### vLLM 中的睡眠模式

在使用共置训练时，管理 GPU 内存至关重要，以便训练和推理都能在同一设备上平稳运行。为了支持这一点，我们将 vLLM 的 `sleep()` API 添加到了 GRPO 训练循环中。

`sleep()` 函数会暂时暂停 vLLM 引擎并释放 GPU 内存。它支持两个级别：

- **级别 1**：从 GPU 卸载模型权重（将其保留在 CPU 内存中）并清除 KV 缓存。
适用于同一模型将很快被重用的场景。
- **级别 2**：完全卸载模型权重和 KV 缓存。
最适合模型将更改或不会立即重用的场景。

适用于同一模型将很快被重用的场景。

**级别 2**：完全卸载模型权重和 KV 缓存。
最适合模型将更改或不会立即重用的场景。

在 GRPO 中，模型在每一步之后都会更新——因此我们使用**级别 2 睡眠**。

**级别 2 睡眠的好处**：

- **最大化训练可用的空闲 GPU 内存**
- **避免训练和生成之间的内存争用**
- 即使对于像 Qwen2.5-72B 这样的大型模型，也能保持共置效率

这个小小的添加在实现平稳且可扩展的共置训练方面**带来了巨大改变**。

### DeepSpeed 优化

为了训练像 Qwen2.5-72B 这样的大型模型，我们依赖 **DeepSpeed ZeRO Stage 3**，这与普通 TRL 中使用的设置相同。

ZeRO 通过在 GPU 之间分配内存来帮助扩展大型模型。Stage 3 更进一步，对以下内容进行分区：

- 模型权重
- 梯度
- 优化器状态

这对于无法装入单个 GPU 的模型至关重要。使用 ZeRO Stage 3，每个 GPU 只处理模型的一部分。

我们启用的其他选项：

- `"offload_optimizer": {"device": "cpu"}` 将优化器状态移动到 CPU 以释放 GPU 内存——在共置设置中至关重要。
- `"overlap_comm": true` 启用通信与计算的重叠，加速训练。
- `"contiguous_gradients": true` 在单个内存块中分配梯度，改善内存访问并减少碎片。

这些优化有助于**高效训练 72B 模型**，并确保在严格的内存限制下共置保持稳定。

### Accelerate 集成

正如 TRL 所推荐的，我们使用 **Accelerate**，这是一个简化分布式训练的轻量级库。它处理：

- 多 GPU 和多节点作业启动
- 数据并行
- 梯度累积
- 分布式数据加载

这使得设置简洁、可扩展且易于维护。

## 🎓 挑战、经验教训与后续步骤

通过我们在使用共置式 vLLM 扩展 GRPO 训练方面的工作，我们面临了几个关键挑战，并学到了关于训练大型模型时的效率、灵活性和系统设计的重要经验。

### 挑战

- **vLLM ≥ 0.8.0 中的张量并行性错误**。在 vLLM 0.8.0 及以上版本中，使用 `external_launcher` 的张量并行性（TP）停止工作。此问题在 Issue#15895 下被追踪。为了定位问题点，我们遵循了这篇 vLLM 开发者博客文章中描述的方法，该文章为每次提交提供了 wheel 包。经过二分查找，我们确定导致问题的提交是 `cc10281`。根本原因是确定性——新版本要求显式设置随机种子。一旦设置了种子，问题就消失了。
- **二级休眠缓冲区错误**。最初，当我们尝试使用 `load_weights` 重新加载权重时，二级休眠无法正常工作。此问题在 Issue #16564 中被追踪。问题在于模型缓冲区（如 BatchNorm 中的 running mean/var）在从休眠唤醒后未能恢复。修复随 PR#16889 而来，该 PR 增加了在从二级休眠唤醒时显式恢复缓冲区的逻辑。我们现在保留原始缓冲区的副本，并在加载新权重后手动重新应用它们。
- **退出时的段错误**。vLLM 休眠在训练结束时关闭进程时导致段错误的问题仍然存在。此问题在 Issue#16993 中被报告。此崩溃发生在关闭期间，但不会破坏训练本身，因此我们能够完成本博客中分享的所有演示和实验。然而，在将 `sleep()` 完全集成到 TRL 上游之前，我们正在等待官方的修复。

这些挑战并非阻碍，但它们需要仔细的调试、版本控制以及对 vLLM 底层如何管理内存和并行性的更深入理解。

### 经验教训

- **共置推理显著提高了 GPU 利用率**。通过允许训练和生成共享相同的 GPU，我们消除了空闲时间并减少了硬件需求——即使使用更少的 GPU 也能实现更高的吞吐量。
- **vLLM 的 `sleep()` 功能对于大规模共置至关重要**。它实现了对内存使用的细粒度控制，允许训练在生成步骤之间完全回收 GPU 内存——这是像 Qwen2.5-72B 这样模型的关键赋能因素。
- **DeepSpeed ZeRO 第三阶段对于训练大模型至关重要**。它通过将模型权重、梯度和优化器状态分布在多个 GPU 上，使得极大的网络能够放入内存。根据我们的经验，启用 `contiguous_gradients` 有助于减少内存碎片，而将优化器卸载到 CPU 则释放了关键的 GPU 内存——这两者在共置设置中都特别有帮助。
- **共置功能强大但需要权衡**。当 GPU 内存得到精心管理时效果最佳，通常需要手动调整内存使用参数，如 `vllm_gpu_memory_utilization`。虽然它提供了明显的吞吐量优势并减少了 GPU 空闲时间，但对于内存预算紧张或内存碎片控制不佳的模型，共置可能不是理想选择。然而，如果操作得当，它能释放显著的效率提升。
- **TP/DP 兼容性、Accelerate 和 torchrun 支持使部署无缝衔接**。尽管底层架构复杂，但整个系统可以使用标准的分布式工具启动和扩展。
- **共置训练保持了模型质量**。在多个基准测试（Math500, AIME24）中，共置设置和普通设置产生了可比较的结果，验证了性能并未因效率而牺牲。

## ✅ 结论

这篇博客文章探讨了将 vLLM 与 GRPO 训练共置如何能在训练大语言模型（包括像 Qwen2.5-72B 这样大的模型）时释放显著的效率提升。

传统上，TRL 仅支持服务器模式下的 vLLM，这需要单独的进程和 GPU 进行推理，导致计算浪费和空闲时间。随着 vLLM 的 `external_launcher` 和 TRL 中 PR #3394 共置 PR 的引入，我们现在可以在相同的分布式进程组、相同的 GPU 上运行训练和推理，并完全支持 TP、DP 和 Accelerate。

```python
from datasets import load_dataset
from trl import GRPOConfig, GRPOTrainer


dataset = load_dataset("trl-lib/tldr", split="train")


def reward_len(completions, **kwargs):
    return [-abs(20 - len(completion)) for completion in completions]
```

training_args = GRPOConfig(
    output_dir="Qwen2-0.5B-GRPO",
    logging_steps=1,
    use_vllm=True,
    vllm_mode="colocate",
    vllm_tensor_parallel_size=1,
    vllm_gpu_memory_utilization=0.3,
    max_prompt_length=512,
    max_completion_length=1024,
    max_steps=2,
    num_generations=4,
    num_train_epochs=1,
    per_device_train_batch_size=4,
    push_to_hub=False,
    report_to=None
)


trainer = GRPOTrainer(
    model="Qwen/Qwen2-0.5B-Instruct",
    reward_funcs=reward_len,
    args=training_args,
    train_dataset=dataset,
)

trainer.train()
