# 10亿次分类任务

你已经优化了模型。你的流水线运行顺畅。但现在，你的云服务账单暴涨。每天处理10亿次以上的分类或嵌入任务不仅仅是技术挑战——更是财务挑战。如何在不超出预算的情况下处理如此规模的任务？无论你是在运行大规模文档分类，还是为检索增强生成（RAG）构建批量嵌入流水线，你都需要具备成本效益、高吞吐量的推理能力，而这来自于精心优化的配置。

这些任务通常使用编码器模型，它们比现代LLM小得多，但在10亿次推理请求的规模下，这仍然是一项相当艰巨的任务。需要明确的是，这相当于处理了144倍于英文维基百科的数据量。我很少看到关于如何在考虑成本的情况下处理这个问题的信息，我想解决这个问题。这篇博客将详细解析如何计算大规模分类和嵌入任务的成本和延迟。我们将分析不同的模型架构，对比不同硬件选择的成本基准，并为你提供一个清晰的框架来优化你自己的配置。此外，如果你不想自己走完整个流程，我们也应该能帮你建立起一些直觉。

## 方法

要评估成本和延迟，我们需要4个关键组件：

- 硬件选项：多种硬件用于成本对比
- 部署工具：一种按我们选择的设置部署模型的方式
- 负载测试：一种发送请求并测量性能的方式
- 推理服务器：一种在选定硬件上高效运行模型的方式

我将使用推理端点作为我的硬件选项，因为它允许我从广泛的硬件选择中进行选择。请注意，你可以将其替换为你选择/考虑的GPU。对于部署工具，我将使用非常有用的Hugging Face Hub库，它允许我以编程方式轻松部署模型。

对于推理服务器，我还将使用Infinity，这是一个用于服务基于编码器的模型（现在支持更多！）的优秀库。我已经写过关于TEI的文章，这是另一个优秀的库。在处理你的用例时，你绝对应该考虑TEI，尽管这篇博客侧重于方法论而非框架对比。Infinity有许多关键优势，例如服务多模态嵌入、支持不同硬件（AMD、Nvidia、CPU和Inferentia），以及运行任何包含远程代码但尚未集成到Hugging Face Transformer库中的新模型。对我来说，其中最重要的是大多数模型默认兼容。

对于负载测试，我将使用Grafana的k6，这是一个用Go编写、带有JavaScript接口的开源负载测试工具。它易于配置，性能高，开销低。它有许多内置的执行器，非常有用。它还预分配虚拟用户（VU），这比你自己随便测试要真实得多。

### 负载测试参数

VU和批量大小很重要，因为它们影响我们如何充分利用GPU中所有可用的计算资源。足够大的批量大小确保我们充分利用流式多处理器和显存。有些情况下，我们还有剩余显存，但带宽成本阻碍了吞吐量的提升。因此实验可以帮助我们。VU使我们能够确保我们充分利用了可用的批量大小。这些是我将要测试的主要参数：

- INFINITY_BATCH_SIZE
  - 这是模型在批次中进行一次前向传播的文档数量
  - 太低则无法充分利用GPU
  - 太高则GPU无法处理大量输入
- VUs
  - 这是发送到K6的模拟并行客户端请求的虚拟用户数量
  - 模拟大量用户可能很困难，每台机器也会有所不同
- GPUs
  - 我们在推理端点上有多种GPU可用
  - 我优先选择了那些性价比最高的
- CPUs
  - 我忽略了这些，因为Nvidia-T4如此便宜，以至于在初步测试中CPU似乎没有吸引力。如果用户感兴趣，我留下了一些代码用于测试

根据你的模型，你可能需要考虑：

- 你正在使用的Docker镜像。['michaelf34/infinity:0.0.75-trt-onnx','michaelf34/infinity:0.0.75']
  - 有许多Infinity镜像可以支持不同的后端。你应该考虑哪些最适合你的硬件/模型/配置
- INFINITY_COMPILE
  - 是否要使用torch.compile()文档
- INFINITY_BETTERTRANSFORMER
  - 是否要让torch使用Better Transformer

K6很棒，因为它允许你预分配VU并防止一些隐蔽的错误。它在发送请求的方式上非常灵活。我决定以一种特定的方式使用它。

当我调用一个k6实验时，我主要想知道请求的吞吐量和平均延迟，并对所选的负载测试参数进行一些合理性检查。我还想进行一次扫描，这意味着要进行许多实验。

我使用shared-iterations执行器（文档），这意味着K6在VU数量之间共享迭代次数。一旦k6执行完所有迭代，测试结束。这使我能够设置合理的超时时间，同时也能发送足够多的请求，以便在扫描中有足够的信心区分不同的负载测试参数选择。与其他执行器相比，这使我能够确信我正在模拟客户端在每个VU上尽可能努力地工作，这应该能向我展示最便宜的选项。

我使用10_000个请求†，最大实验时间为1分钟。因此，如果10_000个请求在1分钟内没有完成，那么该实验就结束。

```
export const options = {
    scenarios: {
        shared_load_test: {
            executor: 'shared-iterations',
            vus: {{ pre_allocated_vus }}, 
            iterations: 10000, 
            maxDuration: '1m',
        },
    },
};

```

## 分类

### 简介

文本分类在大规模场景下有多种用例，如邮件过滤、预训练数据集中的毒性检测等。经典架构是BERT，随后涌现出许多其他架构。需要注意的是，与流行的解码器模型不同，这些模型在使用前需要针对任务进行微调。目前我认为以下架构†最为突出：

- DistilBERT：任务性能优秀，工程性能出色。它对原始BERT进行了架构调整，且无法与TEI兼容用于分类。
- DeBERTa-v3：任务性能出色，但工程性能††非常慢，因其独特的注意力机制难以优化。
- ModernBERT：采用序列打包和Flash-Attention-2技术，任务性能和工程性能††均表现出色。

† 请注意，这些模型通常需要针对你的数据进行微调才能表现良好。†† 我用"工程性能"指代预期的延迟/吞吐量。

## 视觉嵌入

ColQwen2是一种基于Qwen2-VL-2B-Instruct的视觉检索器，采用ColBERT风格的多向量表示处理文本和图像。与上述编码器相比，其架构更为复杂。

![](/images/posts/4090b0cba03a.png)

大规模场景下有许多用例可从中受益，如电商搜索、多模态推荐、企业多模态RAG等。

ColBERT风格与我们之前的嵌入用例不同，它将输入拆分为多个Token，并为每个Token返回一个向量，而非为整个输入返回单一向量。你可以在Jina AI找到优秀教程。这种方法能实现更优的语义编码和检索效果，但速度更慢、成本更高。

我对这个实验充满期待，因为它探索了两个较少被认知的概念：视觉嵌入和ColBERT风格嵌入†。关于ColQwen2/VLM有几点需要注意：

- 2B参数规模比本文其他模型大约15倍
- ColQwen2架构复杂，包含多个模型，其中解码器比编码器更慢
- 图像容易消耗大量Token
- API成本：通过API传输图像比传输文本更慢，若使用云服务还会产生更高的出站流量费用

† 如果这对你来说新颖且有趣，请查看这篇更详细的博客。

我尝试使用nvidia-l4这样的小型现代GPU，因为它既能容纳2B参数模型，又因成本低廉而具有良好的扩展性。与其他嵌入模型一样，我调整了batch_size和vus参数。

## 结论

每天处理10亿+次分类或嵌入任务是一项重大挑战，但通过合理优化可以实现成本效益。从我的实验中总结出几个关键模式：

- 硬件至关重要 – NVIDIA L4（$0.80/小时）在性能和成本之间提供了最佳平衡，是现代工作负载优于T4的首选。CPU在大规模场景下不具备竞争力。
- 批处理大小是关键 – 批处理大小的最优值因任务而异，但总体而言，在不超出GPU内存和带宽限制的前提下最大化批处理大小是提升效率的关键。分类任务的最佳批处理大小为64，嵌入任务为32。
- 并行性至关重要 – 找到合适的虚拟用户数可确保GPU充分利用。指数增长+二分搜索的方法有助于高效收敛到最佳虚拟用户设置。
- ColBERT风格视觉嵌入成本高昂 – 每10亿次嵌入成本超过44,000美元，基于图像的检索比文本任务昂贵两个数量级。

你的数据、硬件、模型等可能有所不同，但我希望本文提供的方法和代码对你有所启发。获取估算的最佳方式是在自己的任务上使用自己的配置运行测试。让我们开始探索吧！

## 附录

随着复杂度的提升，同步扩展合理性检查非常重要。由于我们使用`subprocess`和`jinja`来调用`k6`，感觉离实际测试较远，因此我设置了几项检查。

#### 任务性能

任务性能的目标是：对于特定配置，我们的表现与预期相近。任务性能在不同任务中含义不同。对于分类任务，我选择了准确率；对于嵌入任务，我跳过了这一指标。我们可以考察平均相似度及其他类似指标。对于ColBERT风格的任务，由于每次请求会得到多个向量，情况会稍微复杂一些。

尽管对于某些三分类任务来说，58%并不算差，但这并不重要。目标是确保我们获得了预期的任务性能。如果看到显著变化（增加或减少），我们应该保持怀疑并尝试理解原因。

#### 成本与虚拟用户数和批处理大小的等高线图

最后，当我们尝试这些不同设置时，建立直观理解非常重要。我们是否得到了漂亮的理想化图表？是否看到了意外的梯度？是否需要在某个区域进行更多探索？我们的设置是否存在问题，比如没有隔离环境？所有这些都是很好的问题，有些可能难以回答。

等高线图是通过在由三个维度定义的空间中插值中间点来构建的。有几个现象值得理解：

- 颜色渐变：显示成本水平，颜色越深代表成本越高，颜色越浅代表成本越低。
- 等高线：代表成本水平，有助于识别成本效益高的区域。
- 紧密的簇：（等高线）表明成本随着批处理大小或虚拟用户数的微小调整而快速变化。
