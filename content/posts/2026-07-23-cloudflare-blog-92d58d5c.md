今天我们很高兴地宣布推出缓存响应规则。这是一类新的规则类型，在源服务器响应之后、Cloudflare 缓存内容之前生效。

如果你曾因看到本应轻松从缓存中输出的内容，却被一个多余的 `Set-Cookie` 或错误的 `Cache-Control` 头（这些头部有时在源服务器上很难甚至无法剥离或修改）拖回源服务器而感到恼火，那么缓存响应规则正是解决这一问题的方案，并且在最恰当的时机生效。

## 缓存决策的时机与方式

CDN 缓存与源服务器是成对工作的。它们的目标是尽可能从缓存中响应，仅当边缘节点无法响应时才回源。每一个缓存命中率百分点都源于正确分配这一分工。在不该检查缓存时检查，就会浪费一次注定会未命中的查找。检查得太少，源服务器就会承担本应由边缘节点吸收的流量，性能优势也随之消失。

重要的是，源服务器引导着缓存。当它返回一个可缓存的资源时，其响应头部会告知 Cloudflare 可以安全提供该资源多长时间、何时以及如何重新验证，甚至是否要缓存它。缓存的效率完全取决于源服务器的配合。如果源服务器配置错误，缓存就会形同虚设，而源服务器基础设施成本则会飙升。

大多数缓存资格问题并非在请求阶段决定，而是在源服务器响应后才显现出来。

访客请求 `/static/app.js`。Cloudflare 检查缓存，未命中，便将请求转发至源服务器。源服务器返回文件。在这些响应头部的某处，静悄悄地存在一个 `Set-Cookie` 头部。本应在每个 Cloudflare 数据中心都被缓存的资源，现在变得不可缓存。将这种情况乘以每个站点上的每一位访客，再加上同样的意外头部，就会导致缓存命中率下降，源服务器带宽泄漏，性能受损，基础设施成本攀升。

同样的问题还有一长串变体。源服务器在完全可以在 CDN 上安全缓存的资源上发送 `Cache-Control: no-cache`。源服务器发送了正确的指令，但这些指令是针对浏览器而非 Cloudflare 的。或者源服务器附加了一个过于激进的 `ETag`（特定资源版本的标识符），导致每次条件请求都触发重新验证的震荡。通常，尤其是在大型团队中，管理源服务器响应的人员与管理 CDN 的人员并非同一批人。这使得修改一行头部信息变成了一场长达数周的谈判。

这些问题都无法在请求阶段解决。当 Cloudflare 在 `/app.js` 上看到 `Set-Cookie` 时，请求阶段已经结束，响应已经在传输中。

因此，我们在正确的位置设置了解决方案。

缓存响应规则在源服务器的响应到达 Cloudflare 之后、写入缓存之前运行。借助它们，你可以在 Cloudflare 的缓存看到源服务器响应之前，重写 `Cache-Control` 指令、管理缓存标签，并剥离 `Set-Cookie`、`ETag` 和 `Last-Modified` 等头部。这一修复完全在 Cloudflare 上完成，无需修改源服务器代码。

## 缺失的一环

如果你使用 Cloudflare 已有一段时间，你会看到缓存控制功能在不断演进。几年前，这些功能大多存在于页面规则中，页面规则作为一个单一但过载的原始工具运行，将缓存与重定向、安全以及十几种其他行为混合在一起，全部在请求时进行评估。后来我们将这些规则拆分，使得只有相关的行为变更才会针对请求进行评估，从而减少不必要的延迟，并允许在行为之间进行复杂的规则叠加。缓存规则成为了一种专用且富有表现力的规则类型，用于缓存决策，并加入了 CDN-Cache-Control、源缓存控制、自定义缓存键以及其他控制手段，让你能够精确地告诉 Cloudflare 哪些内容可以安全缓存、缓存多久以及在什么条件下缓存。

这些控制手段中的许多都有一个共同特点：它们在请求上运行。

虽然这可能不太直观，但这是有道理的。Cloudflare 必须做出的最重要的缓存决策是：我们是否应该在缓存中查找这个内容，以及使用什么键？这个问题必须在 Cloudflare 与源服务器通信之前回答。如果答案在应该为"是"时却是"否"，那么请求已经付出了源服务器往返的代价，而响应无法挽回这部分延迟。请求时规则利用当时唯一可用的信息来回答这个问题：URL、请求文件的扩展名、请求头部、地理位置、设备类型等。利用这些请求参数以及用于更改这些请求参数的规则，我们判断某个内容是否可能在缓存中，并在与源服务器通信之前先在那里查找。

但有些缓存决策无法在请求时做出。源服务器的 `Cache-Control` 指令（`cache-control: max-age=3600`）是 Cloudflare 从源服务器接收到的响应的一部分。状态码、`ETag`、`Last-Modified` 时间戳、`Set-Cookie` 以及源服务器选择的缓存标签格式，都是源服务器传递给缓存的响应中的内容。在请求时规则运行时，这些信息都不可用。

来自源服务器的响应是事实来源，因此如果需要在 Cloudflare 上更改某些在请求时不可用的内容，之前你只有三种变通方案：

1.  更改源服务器。
2.  编写一个 Worker 来重新获取并重写响应。
3.  接受更低的命中率。

每一种方案都会耗费工程时间、增加延迟或消耗资金。缓存响应规则为你提供了第四种选择：一个规则集，你可以在源服务器的响应进入 Cloudflare 缓存之前对其进行修改。

## 两个阶段，两个问题

理解缓存规则与新的缓存响应规则之间区别的最清晰方式，是将它们视为两个阶段，每个阶段回答各自的问题。

*   **缓存规则**在请求阶段运行，在 Cloudflare 与源服务器通信之前。它们回答：给定这个请求，Cloudflare 是否应该缓存响应，以及使用什么缓存键？这是可以在请求时通过缓存规则做出的三个决策：**是否**缓存（符合条件 vs. 绕过）、对象**是什么**（缓存键以及未来如何识别存储的对象），以及**如何**缓存（边缘 TTL、浏览器 TTL、提供过期内容等）。所有这些问题的答案都必须在获取源服务器内容之前确定，并且只能使用请求中的信息。

*   **缓存响应规则**在响应阶段运行，在源服务器响应之后、响应写入 Cloudflare 缓存之前。缓存响应规则回答：既然源服务器已经响应，我们是否应该调整缓存方式？缓存响应规则可以通过**剥离**会使响应不符合缓存条件的头部、**更改**告知 Cloudflare 是否以及如何缓存的源 `cache-control` 指令，或**设置**用于清除内容的缓存标签，来重写来自请求阶段的"如何"缓存。当缓存规则与缓存响应规则相互冲突时，缓存响应规则优先。

响应阶段无法做到请求阶段能做的一切。它无法更改"对象是什么"，因为键已经固定。不过它可以更改 Cloudflare 缓存的"是否"和"如何"。例如，响应规则可以设置 `no-store` 使可缓存对象变为不可缓存，或者剥离 `Set-Cookie` 使不可缓存内容符合缓存条件。然而，缓存响应规则无法通过响应阶段使先前不符合条件的请求变得可缓存，因为为时已晚。

因此，缓存响应规则并不能取代缓存规则。缓存规则在请求时决定我们缓存的"是否"、"对象是什么"和"如何"。缓存响应规则则在 Cloudflare 在一个此前不存在的阶段看到源服务器的响应后，对"如何"和"是否"拥有最终决定权。

## 你能做什么

缓存响应规则目前支持三种操作。

1.  剥离破坏缓存的头部

`set_cache_settings` 操作会在 Cloudflare 评估缓存之前，从源站响应中移除 `Set-Cookie`、`ETag` 或 `Last-Modified` 等内容：

```
"action_parameters": {
  "strip_etags": true,
  "strip_set_cookie": true,
  "strip_last_modified": true
}
```

这是针对“静态资源上的 Set-Cookie”问题的修复方案。源站框架经常为每个响应附加会话 Cookie（用于负载均衡或其他目的），包括用户不希望与会话关联的资源响应。在响应阶段移除 `Set-Cookie` 可使这些资源重新变为可缓存状态，而无需要求源站、负载均衡器或任何上游组件进行任何更改。

缓存响应规则也适用于完全不符合缓存条件的响应。因此，例如当你从动态响应中移除 Set-Cookie 时，无论该对象是否被存储，规则都会触发。这使你可以控制客户端看到的内容，即使响应未被存储在缓存中。你还可以移除 ETag 和 `Last-Modified`，当这些头部在源站配置错误并导致缓存抖动时，此功能非常有用。但这需要权衡：同时移除 ETag 和 Last-Modified 会为该响应启用智能边缘重新验证。如果缓存响应规则随后添加了新的验证器，Cloudflare 将不会为浏览器条件请求启用智能边缘重新验证。

因此，即使在动态请求上，移除和修改头部也是一种强大的新模式，这在之前的 Cloudflare 上是不存在的，但如果你进行这些更改，需要注意一些额外的配置。

1. 管理缓存标签

`set_cache_tags` 允许你在响应上 `添加`、`移除` 或 `设置` 用于“按标签清除”的缓存标签。标签可以是静态的：

```
"action_parameters": {
  "operation": "set",
  "values": ["product-catalog", "storefront"]
}
```

或者根据响应头部表达式计算得出：

```
"action_parameters": {
  "operation": "add",
  "expression": "split(http.response.headers[\"Surrogate-Keys\"][0], \",\", 64)"
}
```

第二种形式在 CDN 迁移期间尤其有价值。如果你之前的 CDN 使用类似 `Surrogate-Keys` 的头部（以逗号分隔）将代理键附加到响应，你可以直接在响应阶段将它们转换为 Cloudflare 的 `Cache-Tag` 格式。

`split()` 的第三个参数是限制：结果数组中的最大元素数量可以在 1 到 128 之间。使用一个比每个响应实际标签数量稍大的值。值为 1 会将整个头部作为一个标签返回。一旦响应上存在标签，“按标签清除”功能即可正常工作（这里的“正常工作”意味着通常在全局范围内耗时低于 150 毫秒）。

1. 修改 Cache-Control 指令

`set_cache_control` 是承担最繁重工作的操作。你可以设置或移除单个指令：

- 持续时间指令：`max-age`、`s-maxage`、`stale-if-error`、`stale-while-revalidate`
- 限定指令：`private`、`no-cache`（带有可选的头部名称限定符）
- 布尔指令：`no-store`、`no-transform`、`must-revalidate`、`proxy-revalidate`、`must-understand`、`public`、`immutable`

对于每个指令，你还可以设置 `cloudflare_only: true`，这通常是让人感到意外的部分：

```
"action_parameters": {
  "s-maxage": {
    "operation": "set",
    "value": 86400,
    "cloudflare_only": true
  }
}
```

当 `cloudflare_only` 为 true 时，该指令会影响 Cloudflare 缓存响应的方式，但发送给浏览器的下游 Cache-Control 值保持不变。这是 `CDN-Cache-Control` 在源站所提供功能的响应阶段版本：在 Cloudflare 缓存资源 24 小时，但告诉浏览器其他内容。区别在于，你现在是从 Cloudflare 仪表盘的规则界面进行操作，而不是要求源站设置一个单独的头部。

## 值得借鉴的示例

以下是一些我们认为可以帮助你展示缓存响应规则强大功能的示例。尝试它们，进行混合搭配，并与社区中的其他人分享，这样你就可以了解如何使用缓存规则和缓存响应规则实现强大的缓存自定义。更多示例可以在文档中找到。

### 从静态资源扩展名中移除 Set-Cookie

```
表达式:  http.request.uri.path.extension in {"js" "css" "woff2" "woff" "ttf" "png" "jpg" "svg"}
操作:    set_cache_settings
         strip_set_cookie: true
```

工作原理：导致“本应可缓存但实际不可缓存”的最常见原因是源站上的会话中间件为每个响应都附加了 `Set-Cookie`。对于已知的静态扩展名移除它，可以将这些响应转换为可缓存状态，而无需更改源站。

注意事项：仅对 Cookie 在语义上不是必需的资源类型执行此操作。如果你的源站使用 Cookie 来驱动这些 URL 上的变体行为（非常罕见但可能），请有选择地移除或按路径限定移除这些头部。

### 在 Cloudflare 长期缓存静态资源，在浏览器短期缓存

```
表达式: http.request.uri.path.extension in {"js" "css" "woff2"}
操作:  set_cache_control
         s-maxage: set 2592000 (30 days), cloudflare_only=true
         immutable: set
         max-age: set 86400  (1 day), cloudflare_only=false
```

工作原理：Cloudflare 持有该资源一个月并从缓存中提供。浏览器看到 `max-age=86400` 并在一天后重新验证。你无需接触源站即可解耦两个缓存生命周期。

注意事项：`immutable` 告诉浏览器即使在显式刷新时也不要重新验证。仅与版本化/哈希文件名配对使用。

### 覆盖已知静态路径上的 no-cache

```
表达式: starts_with(http.request.uri.path, "/static/") and http.response.code eq 200
操作:  set_cache_control
         no-cache:  remove
         s-maxage:  set 3600, cloudflare_only=true
```

工作原理：框架默认有时会为每个响应附加 `no-cache`。如果你知道 `/static/*` 可以安全缓存，你可以移除该指令并施加你自己的 TTL（仅在 Cloudflare 上），而无需更改源站或任何下游缓存看到的内容。

注意事项：要诚实判断什么是真正的静态内容。如果 `/static/` 有时用于提供特定于用户的内容，请缩小匹配范围（扩展名、响应头部信号、内容类型）。

### 在 CDN 迁移期间转换缓存标签

```
表达式:  any(http.response.headers.names[*] == "Surrogate-Keys")
操作:  set_cache_tags
         operation:  add
         expression: split(http.response.headers["Surrogate-Keys"][0], ",", 64)
```

工作原理：许多 CDN 迁移的痛苦源于源站工具以其他供应商的格式发出缓存标签头部。与其要求源站团队发布一个添加 Cloudflare 的 `Cache-Tag` 的版本，不如在响应阶段转换现有的头部。Cloudflare 上的“按标签清除”功能立即开始工作。

注意事项：`split()` 的第三个参数是结果数组大小的限制（1–128），与分隔符无关。

## 如何使用缓存响应规则

仪表盘

1. 转到 Cloudflare 仪表盘中的 **缓存** > **缓存规则**。
2. 选择 **创建规则**，然后选择 **缓存响应规则**。
3. 选择名称和表达式。表达式构建器同时公开请求字段和响应字段。
4. 选择一个操作：**修改缓存控制指令**、**修改缓存标签** 或 **移除头部**。
5. 对于指令，当你希望更改仅适用于 Cloudflare 对资源的视图时，切换 **仅 Cloudflare**。
6. 保存为草稿以进行迭代，或直接部署。

此阶段的规则位于：

```
/zones/{zone_id}/rulesets/phases/http_response_cache_settings/entrypoint
```

有关如何使用缓存响应规则的更多信息，包括 API 和 Terraform 示例，请参阅文档。
