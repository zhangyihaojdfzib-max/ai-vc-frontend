# 你会亲手提交的那个PR

使用Skill和测试工具链让Transformer模型在mlx-lm中可用

## 摘要

我们提供了一个**Skill**和一个**测试工具链**，帮助将语言模型从transformers移植到mlx-lm，使它们在添加到transformers后（几乎）立即可用。该Skill旨在作为助手支持贡献者和审阅者，而非实现自动化。我们将解释为何这样做、如何实现，并探讨在Agent时代如何有意义地为开源项目做贡献。

## 代码Agent时代的到来

2026年，代码Agent开始真正发挥作用。曾经只是编辑器侧边的自动补全功能，如今已演变为能够根据简要说明一次性生成合理解决方案的系统。生成的代码通常开箱即用，涵盖了你所要求的内容，并对你未明确的细节做出了合理假设。这非常棒。正如黄仁勋所言，**全球程序员数量瞬间从3000万激增至10亿**。创造力得以释放。

但这迫使我们重新思考开源。

以transformers库为例。它拥有数百名贡献者，被用于数千个项目，下载量超过十亿次。突然之间，任何拥有Agent的人都可以指示它找到某个未解决的问题，修复并提交PR。而这正是正在发生的事情。这些人感到高兴，因为他们正在为一个伟大的库做贡献，但可悲的现实是，大多数时候，他们并没有意识到自己实际上并非如此。

![来源：@ClementDelangue](/images/posts/5e617cf1ceea.png)

![来源：a16z / Sensor Tower](/images/posts/4ac5d70a3957.jpg)

为什么不是呢？Agent生成的PR通常忽略了两个关键假设：

*   **像transformers这样的代码库非常重视代码本身。** 构建那些代码外观无关紧要的项目很酷，但transformers不是其中之一。由于被成千上万人使用，transformers本质上是通过代码构建的一种人与人之间的沟通方式。模型文件从上到下阅读，因为我们希望实践者无需跳过多层复杂抽象就能理解它们。这种理念**贯穿整个库的设计**，也是我们偏爱扁平化层级结构的原因之一。
*   **Agent缺乏这种上下文。** 由于设计决策并非显式说明，Agent会建议遵循"最佳实践"来"改进"代码库，却没有意识到它们正在破坏库与其用户之间的隐性契约。它们冗长、过早泛化、未能注意到变更如何影响其他领域、引入难以察觉的错误、破坏性能。它们还善于奉承，接受任何想法并勤奋执行，包括那些维护者本会通过简短评论早早驳回的想法。

少数维护者仍然需要阅读每一个PR，理解它，判断设计方向是否正确，识别副作用，并撰写反馈。PR数量增长了十倍，但维护者的数量却没有（也无法）相应增长，因为团队协作无法线性扩展。

## 这与MLX有何关系？

Transformers是首批因巨大体量而感受到这种压力的项目之一，但同样的动态正在各处发生。以另一个领域为例，App Store的审核人员不堪重负，因为现在任何人都可以构建并提交应用，所以很多人都在这样做。

同样的逻辑也适用于MLX：其维护者非常重视代码，并仔细阅读每一个PR。我们想看看Agent是否能**帮助贡献者**快速完成高质量的模型移植，同时**支持审阅者**的工作。我们不仅致力于产出堪比细心人工提交的PR，还提供额外的产出物来增强信息量：生成示例、数值比较，以及一个独立的、非Agent的测试工具链以确保可复现性。

Transformers和MLX之间的另一个联系是，大多数时候，mlx-lm的模型都是从transformers的实现移植而来。由于transformers注重清晰性和可读性，它**已成为模型定义的事实标准**。下游贡献者会等到transformers的实现就绪后，再移植到其他框架。作为副作用，这对Agent来说是一个绝佳的环境，因为它自然地限定了范围：Agent无需从头创建实现，而是依赖transformers代码作为事实来源。

这种方法支持我们的目标：当一个模型登陆transformers后，它应该很快就能在MLX上可用。

## 我们的工作

我们构建了一个Skill，供mlx-lm贡献者用来将模型从transformers移植到MLX。给定类似"将olmo_hybrid架构转换为MLX"的提示，该Skill会设置一个虚拟工作环境，从Hub发现并下载相关模型，读取transformers的建模代码，编写MLX实现，并运行一系列测试。如果结果看起来不对，它会进行调试和迭代，直到满意才宣告成功。

我们设计它时，考虑了对审阅者和贡献者同样有用。

对于贡献者，该Skill当然处理了所有基础工作：在Hub上查找模型变体、比较配置以识别不同变体间的参数差异、下载检查点、设置mlx-lm和transformers的可编辑安装。但它也处理了更困难的建模任务。它会关注重要的架构细节，并验证敏感区域，例如可能导致难以发现错误的RoPE配置。它能检测配置中未声明的数据类型，并从safetensors元数据头中推断出来。它会在transformers和MLX之间进行逐层比较，以精确定位差异发生的位置。这些都是只有具备移植经验的人才会想到要进行的检查。

对于审阅者，该Skill生成的PR会明确说明是Agent辅助的，但看起来确实像一份细心的人工提交。审阅者会看到代码遵循mlx-lm的惯例：地道的解决方案、没有不必要的注释、没有推测性的抽象、未经明确批准不修改共享工具。鉴于代码是Agent辅助的，我们尝试提供比普通PR**更多**的数据，以提供尽可能多的信息。PR正文包含一份报告，总结了变体及其架构差异、生成示例、数值比较、数据类型验证、与transformers基线的逐层比较。PR始终会披露其是Agent辅助的，并且Skill在贡献者接受结果之前不会提交它。

为了验证，该Skill会为独立的、非Agent的测试工具链生成一个测试清单。该工具链在设计上易于复现，且不受LLM幻觉或自满情绪影响（下文详述）。

## 我们是如何做到的

技能（Skill）是智能体（Agent）的配方：它们是指导模型完成复杂任务的简单文本文件，包含操作指南。它们并非魔法；通过提示词（Prompt）和迭代，你也能达到同样的效果。但它们提供了**一致性**（每次运行都遵循相同流程，而不同的人会给出不同的提示词），最大限度地减少了歧义，并可作为文档使用：任何人都可以通过阅读技能来理解其功能，识别缺失的情况并提出改进建议。

我们通过与 Claude 对话，自行移植一个模型来启动这项技能的创建。我要求它将 GLM 4.7 从 transformers 移植到 mlx-lm，并像在正常会话中一样给出指令。有一个小技巧：我让 Claude 查看一个我删除了已有实现的 mlx-lm 代码库副本，这样我就能将其输出与真实情况进行比较。经过几次迭代，我得到了一个可运行的实现、一段揭示了 Claude 如何解决问题的对话记录，以及由 Claude 作为过程总结而创建的第一版技能草案。我对其进行了大量编辑，并融入了来自 @gabegoodhart 的经验教训，他慷慨地分享了他们为另一个模型进行的移植对话记录 🙌。

我们多次重复这个循环，技能也随之完善。在技术层面，我们涵盖了诸如可能导致长序列输出质量下降的 **RoPE 错误**、会悄无声息地降低推理（Inference）速度的 float32 精度污染（你可能会惊讶于这些事情发生的频率！）、不同模型变体之间必须以特定方式处理的配置字段差异，以及超大型模型无法在单机上运行时的分布式推理等问题。我们教会它如何调用 `hf` CLI 来发现和下载模型。最重要的是，我们指导它运行经验丰富的移植者会进行的测试，并且只有在测试通过后才宣布成功。

在文化层面，我们涵盖了**软性**特征，并解释了那些能让代码审查（PR）更容易的惯例：不要用注释来解释代码（审查者必须同时解析注释**和**代码 🤦‍♂️），永远不要提议重构，未经询问不要修改共享工具。这些规则对智能体来说毫无成本，却能为审查者节省大量时间。

最终结果是：贡献者输入一个提示词，技能就会生成一个像**这个**一样的 PR，以及一份供外部测试框架使用的测试清单。

## 测试框架

该技能会作为 PR 的一部分，分享一份全面的结果报告。所有这些都来自智能体在转换过程中运行的测试，但我们不希望审查者仅凭信任就接受它们。为了更进一步，我们创建了一个独立的、非智能体的测试框架，对转换后的代码进行系统性测试。这带来了几个好处：

*   消除了对大语言模型（LLM）产生幻觉（Hallucination）或对结果过于自满的不确定性。
*   保证了可复现性：任何人都可以下载测试框架代码库并运行测试。
*   文档化和透明度。所有结果都保存在不同层面：**总结报告**、**每个模型的详细信息**、保存为 JSON 文件的**原始输入/输出**。**测试**也会被复制到结果文件夹中，这样即使将来我们对框架进行修改，也能知道我们运行了什么。

测试框架不是一个 CI 关卡。有些检查是直接的（输出数据类型正确吗？），但大多数是定性的。预训练模型在长序列中重复自身是正常的吗？与 transformers 基线相比，4% 的相对 logits 差异可以接受吗？这些都是基于对类似架构经验的判断。框架提供了有用的信号，但最终仍需由审查者和贡献者做出决定。

## 如何使用该技能

该技能是为那些已经在为 mlx-lm 提交模型 PR，或者会手动进行此操作的人设计的。它并非面向大众，因为 mlx-lm 的 PR 很少会被直接接受。典型的周期是：贡献者提交 PR，审查者指出改进点，双方迭代直到达到质量标准。如果这对专家提交的 PR 成立，那么对智能体辅助的 PR 也将同样成立。

如果你不准备参与这个周期，你可能不应该提交 PR。审查者会努力理解你的代码（即使知道它是智能体辅助的），你也应该这样做。对代码负责，并准备好采纳他们的反馈。特别是，不要简单地将审查者的评论丢回给智能体，然后发布它生成的任何内容。大语言模型会固执己见，偏离主题，并且无法有效地提出反驳。一旦你与审查者互动，这就变成了人与人之间的对话，所以轮到你进行讨论，并尊重他们投入的时间。

你也可以使用该技能来学习；在建立信心和经验之前，你不需要提交任何东西。阅读技能以识别你未曾意识到的问题领域：技能文件、参考文档和实用脚本加起来有近 1.5 万字。将其指向你自己的 mlx-lm 分支，尝试一次转换，并将你的输出与官方代码库中已接受的实现进行比较。如果你这样做几次，你会学到很多关于 Transformer、MLX 和语言模型架构的知识。

如果你准备好了：

```bash
uv run https://raw.githubusercontent.com/huggingface/transformers-to-mlx/main/install_skill.py
uvx hf skills add --claude

```

我们使用 Claude Code 开发和测试了该技能。同样的方法也适用于 Codex 或其他编码智能体，但我们尚未测试它们。如果你在不同的环境中尝试该技能，请告诉我们效果如何！

## 后续步骤与已知不足

该技能在 mlx-lm 的大语言模型上效果良好，但仍有很大的改进空间。

### 下一步计划

*   **mlx-vlm**。视觉语言模型（VLM）位于一个**独立的代码库**中，遵循不同的惯例。除了建模代码，mlx-vlm 还需要**处理器**（processor）在大语言模型看到输入之前处理图像预处理。我们期待与 Prince Canuma 合作，帮助他完成他的工作。
*   **llama.cpp**。其中一些相同的挑战也适用。处理器需要在 C++ 中复现图像处理算法，并且数值差异不可避免。这是一个范围界定明确的智能体可能有所帮助的领域。
*   **测试框架**。我们希望扩展测试套件，并可能探索安全的自动化方案，以便在我们的基础设施上自动运行测试。

### 目前尚不支持的方面

*   **mlx-lm 中的共享工具**。mlx-lm 在将通用模式提取到共享函数方面不如 transformers 严格。该技能有意偏向于自包含的模型文件（与 transformers 相同），但审查者经常要求进行重构，将重复代码移到共享模块中。
*   **视觉语言模型及其他架构**，如上所述。
*   **量化模型上传**。该技能测试量化，但不会将量化模型上传到 Hub。我们认为在 PR 审查期间上传没有意义，但我们可以创建一个流程在之后进行。
*   **思维测试**。目前尚未设计针对思维链（Chain-of-Thought）的特定测试。该技能会转换并验证这些模型的生成结果，但不会验证思维结构。

## 结论

开源（Open Source）的瓶颈不在于打字速度，而在于理解代码库，以便在不破坏与用户的显式和隐式契约的前提下进行修改。如果我们教会智能体什么才是重要的，它们就能在这个过程中提供帮助。我们探索了这在 mlx-lm 的背景下是怎样的，并希望它能帮助贡献者和审查者更快地完成高质量的模型转换！
