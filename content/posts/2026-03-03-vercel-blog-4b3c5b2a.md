重定向在小规模时很简单，但当数量达到百万级时，延迟和成本就成了真正的系统问题。

此前在 Vercel 上，重定向是通过路由规则和中间件处理的。路由规则最多支持 2000 条包含通配符的复杂重定向，它们作为一个有序列表按顺序执行。每条规则都可能涉及正则表达式匹配，这意味着单个请求可能触发多次昂贵的计算。这对于几千条路由规则来说尚可接受，但随着数量增长，每个请求的工作量会线性增加。

中间件提供了更大的灵活性，但它通过在每次请求上运行额外代码而增加了延迟。为了以低延迟服务数百万次重定向，我们需要一个专用的查找路径，使每个请求的时间复杂度接近常数或对数级。基于我们之前使用布隆过滤器加速全局路由的工作，我们找到了一种扩展到数百万次重定向的方法。

## 我们的优化目标

- **规模**：每个项目支持数百万条静态重定向
- **运行时行为**：未配置重定向的项目不产生额外延迟成本；快速的“无重定向”路径，因为大多数请求不会被重定向；低进程内存使用，依赖外部存储和缓存层
- **工程价值观**：简单性和可调试性优于过早优化；迭代演进而非追求首次完美

考虑到这些目标，我们从能想到的最简单设计开始，将重定向和布隆过滤器结合在单个文件中。由于重定向数据已经是 JSON 格式，且我们的布隆过滤器已支持 JSON 导出，我们决定使用 JSONL 文件格式来存储这些信息。

## JSON 与布隆过滤器 vs. 粗略估算

布隆过滤器是一种概率数据结构，用于测试一个元素是否属于某个集合。布隆过滤器可能返回假阳性，但绝不会返回假阴性，因此它的回答是“肯定不在集合中”或“可能在集合中”。通过首先检查一个小的、缓存的布隆过滤器，我们可以为不匹配的请求完全跳过重定向查找，使得常见的“无重定向”路径成本极低。只有在出现阳性匹配时，我们才会解析 JSON 文件。

方案很简单，但它能扩展吗？粗略估算表明不能。一百万条重定向很容易产生一个数百兆字节的文件，获取和解析如此大的文件会严重超出我们的延迟和内存预算。我们需要避免一次性加载整个数据集。

## 分片与布隆过滤器保持低内存和快速查找

解决方案是分片。我们不再使用一个庞大的 JSONL 文件，而是对重定向路径进行哈希处理，将条目分布到许多小分片中。这使我们能够为特定请求加载一小片数据，从而将负担从进程内存转移到外部存储和文件系统缓存。布隆过滤器仍然位于前端，为绝大多数流量短路查找。但现在，当一个请求确实通过了布隆过滤器检查时，我们只需要获取并解析单个小分片，而不是整个重定向集合。

## 分片结构

每个分片包含 3 个部分：
- 编码布隆过滤器属性的标题行
- Base64 编码的布隆过滤器
- 以源路径为键的重定向 JSON 对象

示例如下：

```
1{"version":"bulk-redirects","bloom":{"n":3,"p":1e-7,"m":102,"k":23,"s":0}}2"Mec7FxGVcJ0fHdj8HA=="3{"/old-path":{"destination":"/new-path", ...},"/another-old-path":{"destination":"/another-new-path", ...}, ...}
```

包含标题、布隆过滤器和重定向映射的分片格式

在构建时，我们生成所有分片及其布隆过滤器，并将它们上传到外部存储。在运行时，服务器在收到请求时只需要知道哪个数据集和分片数量适用于给定的项目或部署。

在请求时，批量重定向查找的工作流程如下：
- 检查项目或部署是否配置了批量重定向。如果没有，跳过所有步骤，按常规流程处理。
- 根据传入的请求计算重定向键，并对其进行哈希以确定分片。
- 从缓存或源检索分片，并检查布隆过滤器。
    - 如果键不在布隆过滤器中，我们不解析分片的 JSON 主体。
    - 如果键可能在布隆过滤器中，我们加载分片的 JSON 主体，并在该对象中查找确切的重定向。

这个设计有一些很好的特性：
- **快速的否定查找**：布隆过滤器非常快，并且可以调整到具有极低的假阳性率。
- **人类可读的分片**：分片只是 JSONL 文件。如果出现问题，很容易转储一个分片并查看其确切内容。
- **低实现风险**：JSON 解析和布隆过滤器很简单，因此可以快速上线，使我们能够收集真实世界的数据。

## JSON 解析在阳性查找时成为瓶颈

我们怀疑 JSON 解析可能会成为瓶颈，而我们的内部使用（dogfooding）证实了这一点。当布隆过滤器指示可能存在重定向时，解析相关分片的完整 JSON 主体需要相当长的时间。我们还观察到在高 CPU 负载下出现巨大的延迟峰值，因为 JSON 解析是 CPU 密集型的，并且与节点上的其他所有资源竞争。

减小分片大小有助于提高解析速度，但更小的分片会增加基数（需要管理的分片数量）和缓存未命中率。这就形成了一个权衡。大分片意味着解析带来的CPU开销更高，而小分片则意味着因缓存未命中导致的I/O延迟更多。我们需要一种能够在不解析整个分片的情况下检索单个值的数据格式。

## 通过键排序进行二分查找，避免解析整个分片

我们没有将重定向存储在JSON blob中，而是实现了一种基于重定向路径进行键控的二分查找。每个分片按排序顺序存储其重定向键，因此我们可以对这些键执行对数时间复杂度的搜索。一旦找到键，我们只需要解析该特定重定向的JSON。这完全绕过了分片大小问题。查找成本不再随分片中数据总量而增加，因此我们可以保持足够大的分片以获得良好的缓存命中率，而无需支付完整JSON解析的代价。

```
1{"version":"bulk-redirects","bloom":{"n":3,"p":1e-7,"m":102,"k":23,"s":0}}2"Mec7FxGVcJ0fHdj8HA=="3"/old-path"4{"destination":"/new-path", ...}5"/another-old-path"6{"destination":"/another-new-path", ...}
```

排序的键使得无需解析完整分片即可进行二分查找

## 为常见情况设计

重定向本身很简单。挑战在于将这种简单的抽象与庞大、大部分是冷数据的数据集以及在边缘的严格延迟期望结合起来。路由规则不适合这项工作。

相反，我们为批量重定向构建了一条专用路径：

- 对重定向数据进行分片，使每个部分保持较小
- 使用布隆过滤器，使常见的“无重定向”情况保持低成本
- 以支持对键进行二分查找的布局存储重定向
